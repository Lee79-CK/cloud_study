Chapter 01 · Motivation

LLM 추론은 왜 그렇게 어려울까?

"모델을 그냥 GPU에 올리고 generate()를 호출하면 되지 않나?" — 학습 단계에선 통하던 이 발상이, 수천 명이 동시에 요청을 던지는 운영 환경에선 곧바로 무너집니다. 그 이유를 알면 TensorRT-LLM이 왜 태어났는지가 자연스럽게 보입니다.

LLM 추론의 세 가지 본질적 어려움

일반적인 딥러닝 모델 추론과 LLM 추론은 결이 다릅니다. ResNet 같은 이미지 분류 모델은 입력이 들어오면 한 번의 forward pass로 답이 나옵니다. 하지만 LLM은 토큰을 하나씩 순차적으로 생성합니다. 이 자기회귀(autoregressive) 구조가 모든 어려움의 출발점입니다.

Problem 01

메모리 대역폭 병목

토큰을 하나 생성할 때마다 수십~수백 GB의 가중치를 GPU 메모리에서 읽어 옵니다. 연산이 아니라 메모리 읽기 속도가 진짜 병목이죠.

Problem 02

가변 길이 시퀀스

요청마다 입력·출력 길이가 다릅니다. 배치를 단순하게 묶으면 짧은 요청도 가장 긴 요청이 끝날 때까지 GPU 안에서 기다려야 합니다.

Problem 03

KV Cache 폭발

이전에 생성한 토큰들의 Key/Value 텐서를 캐싱해야 합니다. 문맥이 길어질수록 GPU 메모리를 잡아먹어서 동시 처리 가능한 요청 수가 급격히 줄어듭니다.

현실 감각 잡기

Llama 3 70B 모델을 FP16으로 GPU에 올리면 가중치만 약 140GB입니다. A100 80GB 한 장에는 들어가지도 않죠. 토큰 하나 만들 때마다 이 거대한 가중치를 읽어야 한다는 사실을 기억해 두세요.

그래서 우리에게 필요한 것

초당 더 많은 토큰을 뽑아내려면(throughput), 그리고 사용자가 답을 더 빨리 받게 하려면(latency) — 단순히 "좋은 GPU"만으로는 부족합니다. GPU의 모든 SM(Streaming Multiprocessor)을 쥐어짜는 커널 최적화, 메모리를 정교하게 다루는 캐시 관리, 가변 길이 요청을 효율적으로 처리하는 스케줄링이 한꺼번에 필요합니다.

바로 이 지점에서 TensorRT-LLM이 등장합니다.

Chapter 02 · Definition

TensorRT-LLM이란 무엇인가

한 줄로 정의하면, TensorRT-LLM은 NVIDIA GPU 전용 LLM 추론 최적화 라이브러리입니다. PyTorch로 학습한 LLM을 받아서, GPU의 성능을 마지막 한 방울까지 짜내도록 컴파일·최적화해 주는 도구라고 생각하면 됩니다.

이름이 두 단어인 이유

TensorRT는 NVIDIA가 오래전부터 만들어 온 범용 딥러닝 추론 최적화 엔진입니다. 이미지, 음성, 추천 모델 등 어떤 모델이든 받아서 GPU에 맞게 컴파일하는 도구죠. 그런데 LLM은 KV Cache, 가변 길이 시퀀스, Attention 같은 특수성 때문에 일반 TensorRT만으론 부족했습니다. 그래서 LLM에 특화된 확장판을 따로 만든 게 TensorRT-LLM입니다.

TensorRT범용 추론 엔진
CUDA / cuBLAS / cuDNN저수준 GPU 라이브러리
↑ 위에 얹는다
TensorRT-LLMLLM 특화 컴파일러 + 런타임
↑ 위에 얹는다
Triton Inference Server프로덕션 서빙 (선택)
OpenAI 호환 서버간이 서빙 (선택)

컴파일 기반이라는 핵심 철학

TensorRT-LLM을 한 단어로 표현하라면 "컴파일"입니다. PyTorch처럼 코드를 실행할 때마다 연산 그래프를 해석(interpret)하는 게 아니라, 모델을 미리 GPU 전용 바이너리(엔진)로 컴파일해 둔 다음 그 엔진을 실행합니다.

초보를 위한 비유

Python은 인터프리터 언어, C는 컴파일러 언어입니다. TensorRT-LLM은 LLM을 위한 "C 컴파일러"라고 생각하세요. 한 번 컴파일하는 데 시간이 좀 걸리지만, 한 번 만들어 두면 실행이 훨씬 빠릅니다.

이 컴파일 과정에서 TensorRT-LLM은 다음과 같은 마법을 부립니다:

  • Layer Fusion — 여러 개의 작은 연산(LayerNorm + GELU + MatMul 등)을 하나의 거대한 CUDA 커널로 합칩니다. 메모리 왕복이 줄어듭니다.
  • Precision Optimization — 가능한 곳마다 FP32 → FP16 → FP8 → INT4로 정밀도를 낮춰 메모리와 연산량을 줄입니다.
  • Kernel Auto-Tuning — 우리 GPU(H100? A100? L40S?)에 가장 빠른 커널 구현을 자동으로 골라줍니다.
  • Memory Planning — 어느 텐서가 언제까지 살아있어야 하는지 분석해 메모리를 최소한으로 잡습니다.
Chapter 03 · Components

핵심 컴포넌트 한 장으로 보기

TensorRT-LLM이 어떻게 생긴 시스템인지 구조부터 머리에 그려 봅시다. 큰 그림을 먼저 익혀두면 뒤에서 다룰 최적화 기법들이 "아, 이게 그 컴포넌트에서 일어나는 일이구나" 하고 자연스럽게 연결됩니다.

Component / 01

Model Definition API

PyTorch와 닮은 Python API로 LLM 구조를 선언합니다. Llama, Qwen, Mistral, GPT-J 등 주요 모델은 이미 구현이 포함되어 있어 바로 가져다 쓸 수 있습니다.

Component / 02

Builder & Engine

모델 정의 + 가중치 + 빌드 옵션을 받아서 .engine 파일을 만들어 냅니다. 이 파일이 바로 GPU에서 실행될 컴파일된 바이너리입니다.

Component / 03

Runtime / Executor

완성된 엔진을 GPU에 로드하고 실제 추론을 돌립니다. 배치 스케줄링, KV Cache 관리, 스트리밍 응답까지 책임지는 심장부입니다.

Component / 04

Quantization Toolkit

모델을 FP8 / INT8 / INT4로 압축하는 도구. AWQ, GPTQ, SmoothQuant 같은 최신 양자화 알고리즘이 통합돼 있습니다.

Component / 05

Plugins

FlashAttention, Paged KV Cache, RMSNorm 같은 LLM 특화 연산을 고도로 최적화한 CUDA 커널 모음. 필요할 때 빌드 옵션으로 켜고 끕니다.

Component / 06

Multi-GPU 분산

Tensor Parallelism, Pipeline Parallelism을 통해 한 모델을 여러 GPU에 쪼개서 올립니다. Llama 405B 같은 초대형 모델 서빙에 필수입니다.

Component / 07

Triton 백엔드

NVIDIA Triton Inference Server와 통합되는 tensorrtllm_backend를 통해 프로덕션급 HTTP/gRPC 서빙을 제공합니다.

Component / 08

High-Level LLM API

최근 추가된 상위 API. from tensorrt_llm import LLM 한 줄로 빌드부터 추론까지 vLLM과 비슷한 경험을 제공합니다.

가장 짧은 예시 코드

전체 그림이 머리에 들어왔다면, 실제로 코드가 어떻게 생겼는지 한번 보고 갑시다. 최근 추가된 상위 API를 쓰면 vLLM만큼 간결합니다.

quickstart.pyPython
# pip install tensorrt-llm
from tensorrt_llm import LLM, SamplingParams

# 1) 모델 로드 — Hugging Face 모델을 그대로 받아 자동으로 엔진을 빌드합니다
llm = LLM(model="meta-llama/Llama-3.1-8B-Instruct")

# 2) 샘플링 파라미터
sampling = SamplingParams(temperature=0.7, max_tokens=256)

# 3) 추론 — 배치로 여러 요청을 동시에 처리할 수도 있습니다
prompts = ["한국의 수도는?", "TensorRT-LLM을 한 문장으로 설명해줘."]
outputs = llm.generate(prompts, sampling)

for out in outputs:
    print(out.outputs[0].text)
알아두기

이 코드 한 줄 뒤에서는 (1) Hugging Face 가중치를 받아 (2) TensorRT-LLM 포맷으로 변환하고 (3) GPU에 맞는 엔진으로 컴파일하고 (4) Executor를 띄우는 일이 모두 일어납니다. 개발자 친화적이긴 하지만, 내부에서 무슨 일이 일어나는지 아는 것이 운영에선 매우 중요합니다.

Chapter 04 · Optimization

속도를 만들어내는 최적화 기법들

아래 다섯 가지는 TensorRT-LLM이 자랑하는 핵심 무기들입니다. 하나하나가 그 자체로 논문이 될 만한 주제지만, 우리는 "무엇이고 왜 빠른가"를 직관적으로 잡는 데 집중합니다.

Quantization · 정밀도를 낮춰 메모리를 줄인다

가중치를 FP16(16비트) 대신 INT8(8비트)이나 INT4(4비트)로 저장하면 메모리 사용량과 메모리 대역폭이 그만큼 줄어듭니다. 앞서 봤듯이 LLM 추론은 메모리 병목이 본질이므로, 양자화는 곧바로 처리량 증가로 이어집니다.

TensorRT-LLM이 지원하는 주요 양자화 방식:

  • FP8 (Hopper, Blackwell) — H100/B100급 GPU에서 사용 가능. 정확도 손실이 거의 없으면서 FP16 대비 2배 가까운 속도.
  • INT8 SmoothQuant — 가중치와 활성값(activation)을 모두 INT8로. 약간의 보정을 거치면 정확도 손실 최소화.
  • INT4 AWQ / GPTQ — 가중치만 INT4로 압축, 활성값은 FP16 유지. 70B 모델을 소비자급 GPU에서 돌릴 수 있게 해 줍니다.
주의

양자화는 공짜 점심이 아닙니다. 정확도 손실, 캘리브레이션 데이터 준비, 지원되는 GPU 세대 제한 등을 함께 따져야 합니다. 운영 전에는 반드시 자체 데이터셋으로 품질 평가를 하세요.

In-flight Batching · 끝난 요청은 바로 빼고, 새 요청은 바로 넣는다

예전 방식의 정적 배치(static batching)는 16개 요청을 한 번에 묶었다면, 그중 가장 긴 요청이 끝날 때까지 나머지는 자기 자리에서 멍하니 기다려야 했습니다. GPU는 텅텅 비고요.

In-flight Batching(다른 이름: Continuous Batching, Iteration-level Scheduling)은 매 토큰 생성 스텝마다 배치를 재구성합니다. 끝난 요청은 즉시 빼내고, 그 자리에 새 요청을 밀어 넣습니다. 결과적으로 GPU 활용률이 극적으로 올라갑니다.

시각화: 매 스텝마다 배치가 다이내믹하게 바뀐다
Step T요청 A, B, C
→
Step T+1A 완료 · D 투입
→
Step T+2B 완료 · E 투입

Paged KV Cache · 메모리를 페이지 단위로 쪼개 관리

운영체제가 메모리를 페이지로 관리하듯, KV Cache도 고정 크기 블록(페이지)으로 잘라서 관리합니다. 이게 왜 중요할까요? 기존 방식은 "이 요청은 최대 4K 토큰까지 갈 수 있으니 4K짜리 연속 공간을 미리 잡아 두자"였습니다. 100개의 요청이 평균 500 토큰만 쓴다 해도 4K×100만큼을 잡아 둬서 메모리가 낭비됐죠.

Paged KV Cache는 필요할 때만, 페이지 단위로(예: 16/32토큰) 메모리를 할당합니다. 파편화가 없고, 같은 prefix를 공유하는 요청끼리 페이지를 공유할 수도 있어 시스템 프롬프트가 긴 챗봇에서 특히 효과적입니다.

vLLM과의 친척 관계

이 아이디어를 처음 대중적으로 알린 건 UC Berkeley의 vLLM(PagedAttention)이었습니다. TensorRT-LLM도 같은 철학의 구현을 자체적으로 갖고 있고, 최근 버전에선 prefix caching, KV reuse 등 더 진화한 기능을 제공합니다.

Speculative Decoding · 작은 모델이 먼저 추측하고, 큰 모델이 검산한다

토큰을 한 개씩 만드는 게 답답하니, 이런 아이디어가 등장합니다. 작은 draft 모델이 앞으로 나올 토큰 4~8개를 빠르게 추측해 놓으면, 큰 target 모델이 그 추측이 맞는지 한 번에 검증합니다. 맞으면 한 번의 forward pass로 여러 토큰을 통과시킨 셈이 됩니다.

TensorRT-LLM은 다양한 변형을 지원합니다:

  • Draft-Target Speculative Decoding — 별도 draft 모델 사용
  • Medusa — 추가 head를 붙여 한 모델 안에서 여러 토큰을 예측
  • EAGLE / Lookahead — 더 진화된 변형들

잘 적용하면 latency를 1.5~3배 개선할 수 있지만, 워크로드 특성에 따라 효과 편차가 큽니다.

Multi-GPU · 한 모델을 여러 장에 나눠 올린다

Llama 70B를 H100 한 장에 다 못 올릴 때, 우리는 두 가지 분산 방식을 사용합니다.

  • Tensor Parallelism (TP) — 같은 레이어를 GPU 여러 장에 가로로 쪼갬. GPU 간 통신이 잦지만 latency에 유리.
  • Pipeline Parallelism (PP) — 모델을 세로로 잘라(예: 1~16층은 GPU0, 17~32층은 GPU1) 단계별로 처리. 통신은 적지만 파이프라인 버블이 생김.

TensorRT-LLM은 빌드 시점에 --tp_size, --pp_size 옵션으로 분산 토폴로지를 지정하면 엔진이 그에 맞춰 컴파일됩니다. NCCL 기반 통신이 내부에 통합되어 있어 별도 설정이 거의 필요 없습니다.

실무 팁

일반적으로 같은 노드 안에선 NVLink로 묶인 GPU끼리 TP를, 노드 경계를 넘을 땐 PP를 쓰는 게 통신 비용 면에서 유리합니다. 8장짜리 H100 노드에서 TP=8을 쓰는 구성이 가장 흔합니다.

Chapter 05 · Workflow

모델을 띄우기까지의 5단계

이론은 충분히 들었으니, 실제로 모델을 서빙하기까지의 흐름을 한눈에 정리해 봅시다. 이 다이어그램이 머리에 박히면 어떤 단계에서 문제가 생겼는지 진단하기 쉬워집니다.

01
모델 준비
Hugging Face에서 모델 가중치 다운로드 또는 자체 학습한 체크포인트 준비.
02
체크포인트 변환
HF 포맷 → TensorRT-LLM 체크포인트(JSON + safetensors)로 변환.
03
엔진 빌드
trtllm-build 명령어로 GPU별 최적화된 .engine 파일을 생성.
04
런타임 실행
Executor API 또는 Triton 백엔드로 엔진을 로드해 추론 서비스 가동.

좀 더 깊이 들어가 보기

TensorRT-LLM vs vLLM — 가장 자주 듣는 질문+

속도 — 같은 H100에서 같은 모델을 잘 튜닝해서 비교하면 TensorRT-LLM이 대체로 약간 우위에 있는 편입니다. 다만 vLLM도 매 릴리스마다 격차를 좁히고 있어 워크로드에 따라선 비슷하거나 vLLM이 빠른 경우도 있습니다.

유연성 — vLLM이 압도적으로 편합니다. pip install vllm 하고 모델 경로만 주면 바로 OpenAI 호환 서버가 뜹니다. TensorRT-LLM은 엔진 빌드 단계가 필수라서 처음엔 진입 장벽이 있습니다.

새 모델 지원 — 새 아키텍처가 나오면 vLLM 쪽이 보통 더 빨리 지원합니다. TensorRT-LLM은 NVIDIA의 릴리스 사이클을 따라야 해서 며칠~몇 주 늦을 수 있습니다.

한 줄 결론 — 빠른 실험·다양한 모델·여러 클라우드 환경이라면 vLLM, NVIDIA GPU에 집중된 안정적 대규모 운영이라면 TensorRT-LLM.

TensorRT-LLM vs TGI(Text Generation Inference)+

TGI는 Hugging Face가 만든 서빙 솔루션입니다. HF 생태계(transformers, accelerate, PEFT 등)와 긴밀하게 통합되어 있고 Rust로 작성된 서버라 안정성이 좋습니다.

다만 순수 추론 속도에선 TensorRT-LLM이 대체로 앞섭니다. TGI는 "HF 생태계 안에서 모든 걸 해결하고 싶은 팀"에게 자연스러운 선택이고, TensorRT-LLM은 "성능을 위해 컴파일 단계를 감수할 수 있는 팀"에게 적합합니다.

TensorRT-LLM vs SGLang — 새로 떠오르는 강자+

SGLang은 LMSys(Vicuna 만든 곳)에서 출발한 비교적 새로운 프로젝트입니다. RadixAttention 같은 독자적인 KV 캐시 공유 기법, 구조화된 출력, 복잡한 프롬프트 체이닝에 강점이 있습니다.

RAG·에이전트처럼 시스템 프롬프트가 길고 prefix가 공유되는 워크로드에선 SGLang이 의외로 빠른 성능을 보입니다. TensorRT-LLM과 SGLang은 경쟁자라기보단 다른 영역에서 빛나는 도구라고 보는 게 정확합니다.

TensorRT-LLM vs llama.cpp / Ollama+

llama.cpp는 C++로 작성된 경량 LLM 추론 엔진이고, Ollama는 그 위에 사용자 친화적 래퍼를 얹은 도구입니다. CPU와 Apple Silicon, 저사양 GPU까지 폭넓게 지원합니다.

이들은 로컬·개인용·엣지 추론이 목적이고, TensorRT-LLM은 데이터센터급 대규모 서빙이 목적입니다. 같은 카테고리의 도구가 아니라고 보는 게 맞습니다. 둘 다 LLM을 띄우지만 추구하는 목표가 다릅니다.

그래서 우리 회사는 뭘 써야 할까?+

NVIDIA GPU(H100, A100, L40S 등)로 대규모 서빙을 한다면 — TensorRT-LLM 또는 vLLM. 모델·트래픽 패턴이 안정적이고 마지막 한 방울 성능까지 짜내야 한다면 TensorRT-LLM 우위.

여러 모델을 빠르게 갈아 끼우며 실험한다면 — vLLM 또는 SGLang. 빌드 과정이 없어 반복 속도가 빠릅니다.

온프레미스·하이브리드·멀티 클라우드라면 — vLLM. NVIDIA가 아닌 가속기도 지원.

RAG·구조화 출력·에이전트 중심이라면 — SGLang을 진지하게 평가해 보세요.

로컬·개발 환경·데모라면 — Ollama·llama.cpp가 가장 편합니다.

실무에서는 여러 개를 동시에 쓰는 경우도 많습니다. 개발은 vLLM으로 빠르게, 프로덕션 대용량은 TensorRT-LLM으로 — 같은 식의 조합이 흔합니다.

Chapter 07 · Use Cases

어디에 쓰이고 있는가

TensorRT-LLM은 단순히 벤치마크 위에서 빛나는 도구가 아닙니다. 이미 많은 회사가 프로덕션에서 이 엔진으로 실제 트래픽을 받고 있습니다. 어떤 시나리오에서 빛을 발하는지 살펴봅시다.

Use Case / 01

대화형 챗봇 서비스

수만 명이 동시에 채팅하는 환경에서 TTFT(첫 토큰 latency)와 TPOT(토큰 간 latency)을 동시에 잡아야 할 때. In-flight Batching이 진가를 발휘합니다.

Use Case / 02

코드 생성 / 코드 어시스턴트

입력 컨텍스트(코드베이스)가 길고 출력도 긴 워크로드. KV cache reuse와 chunked prefill로 long-context 비용을 절감합니다.

Use Case / 03

RAG 시스템

검색된 문서를 매번 prefix로 붙이는 RAG에서 Paged KV Cache의 prefix sharing이 메모리 사용량을 크게 줄여 줍니다.

Use Case / 04

멀티모달 추론

LLaVA, Qwen-VL 같은 비전-언어 모델 서빙. 이미지 인코더와 LLM이 한 엔진 안에서 효율적으로 돌아가도록 지원.

Use Case / 05

온디바이스 / 엣지 추론

NVIDIA Jetson, RTX 워크스테이션 등 엣지 GPU에서 INT4 양자화로 7B~13B 모델을 실시간 서빙. 보안·프라이버시가 중요한 시나리오.

Use Case / 06

대규모 배치 처리

야간 배치로 수백만 건 문서 요약·분류·임베딩 생성. throughput이 곧 비용이라 TensorRT-LLM의 강점이 직접적인 비용 절감으로 이어집니다.

실무 시나리오 한 가지

전형적인 한국 통신사 사내 AI 어시스턴트 구성 — H100 노드 위에서 Llama 3.1 70B를 FP8로 양자화하여 TensorRT-LLM으로 빌드하고, Triton Inference Server에 올린 뒤 Kubernetes 위에서 HPA(GPU 활용률 기반)와 함께 운영. KV 캐시 reuse를 켜서 사내 시스템 프롬프트가 공유되도록 하면 같은 GPU로 처리 가능한 동시 사용자 수가 크게 늘어납니다.

Chapter 08 · Wrap-up

정리, 용어, 그리고 작은 퀴즈

수고하셨습니다. 마지막으로 핵심 메시지를 한 번 더 정리하고, 자주 등장하는 용어들과 함께 배운 내용을 점검하는 짧은 퀴즈로 강의를 마무리합니다.

오늘 강의의 핵심 5문장

  1. LLM 추론은 자기회귀 구조 때문에 메모리 대역폭이 본질적 병목이다.
  2. TensorRT-LLM은 LLM을 GPU 전용 엔진으로 컴파일해서 그 병목을 공략하는 NVIDIA의 라이브러리다.
  3. 핵심 컴포넌트는 Model Definition API, Builder/Engine, Runtime, Plugins, Quantization Toolkit, Multi-GPU 분산이다.
  4. 가장 강력한 무기는 FP8/INT4 양자화 + In-flight Batching + Paged KV Cache + Speculative Decoding이다.
  5. vLLM이 유연함의 챔피언이라면, TensorRT-LLM은 NVIDIA GPU 위에서 성능의 챔피언이다.

꼭 외워야 할 용어집

TTFT
Time To First Token — 사용자가 요청 후 첫 토큰을 받기까지 걸리는 시간.
TPOT
Time Per Output Token — 두 번째 토큰 이후의 토큰 간 평균 간격.
Throughput
단위 시간당 시스템이 생성하는 총 토큰 수. 대규모 서빙의 핵심 KPI.
KV Cache
이전에 생성한 토큰들의 Key/Value 텐서. 매 토큰마다 재계산을 피하려고 캐싱.
Prefill
사용자가 보낸 입력 프롬프트를 한 번에 처리해 KV Cache를 만드는 단계.
Decode
새로운 출력 토큰을 한 개씩 만들어 가는 단계. LLM의 진짜 병목 구간.
Engine
TensorRT-LLM이 컴파일해 만든 GPU 전용 바이너리 파일(.engine).
Plugin
FlashAttention 같은 특화 CUDA 커널을 엔진에 끼워 넣는 모듈.
TP / PP
Tensor / Pipeline Parallelism — 모델을 여러 GPU에 가로/세로로 쪼개는 방식.
AWQ / GPTQ
가중치만 INT4로 압축하는 대표적 양자화 알고리즘. 정확도 손실이 작다.
FP8
8비트 부동소수점. H100·B100급 GPU에서 사용 가능, FP16 대비 약 2배 빠름.
Triton
NVIDIA Triton Inference Server. TensorRT-LLM 백엔드와 통합되어 운영 서빙에 사용.
