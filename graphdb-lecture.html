01 — Motivation

왜 그래프인가?

데이터베이스를 고를 때 우리는 보통 "데이터를 어떻게 저장할까"를 고민합니다. 하지만 진짜 중요한 질문은 따로 있습니다 — "이 데이터들이 서로 어떻게 연결되어 있나?"

관계형 데이터베이스(RDB)는 데이터를 표(table)에 넣고, 표와 표 사이의 연결은 JOIN이라는 연산으로 그때그때 계산합니다. 데이터가 얕게 연결될 때는 훌륭합니다. 그런데 연결이 깊어지면 이야기가 달라집니다.

JOIN 지옥 (The JOIN Hell)

"A의 친구의 친구의 친구가 좋아하는 상품"을 SQL로 구하려면, 관계를 한 단계 탈 때마다 테이블을 한 번씩 더 JOIN해야 합니다. 관계가 깊어질수록 JOIN이 기하급수적으로 쌓이고, 데이터가 커지면 쿼리 한 방에 DB가 몇 초씩 멈춥니다. 이것이 바로 개발자들이 말하는 JOIN 지옥입니다.

관계형 DB — 3단계 친구 찾기 (SQL)
SELECT f3.name
FROM person p
JOIN friendship a ON a.from_id = p.id
JOIN person   f1 ON f1.id = a.to_id
JOIN friendship b ON b.from_id = f1.id
JOIN person   f2 ON f2.id = b.to_id
JOIN friendship c ON c.from_id = f2.id
JOIN person   f3 ON f3.id = c.to_id
WHERE p.name = 'Alice';
-- 한 단계 늘 때마다 JOIN 2개 추가 😵
그래프 DB — 같은 질문 (Cypher)
MATCH (a:Person {name: 'Alice'})
      -[:FRIEND]->()-[:FRIEND]->()
      -[:FRIEND]->(f3)
RETURN f3.name

// 관계를 '그림 그리듯' 표현.
// 깊이가 늘어도 화살표만 이어붙이면 끝 ✨
한 줄 요약

RDB는 "관계를 매번 다시 계산"하고, GraphDB는 "관계를 이미 저장"해 둡니다. 그래서 연결을 깊게 탐색할수록 GraphDB가 압도적으로 빨라집니다.

02 — Fundamentals

그래프의 3요소

GraphDB에서 가장 널리 쓰이는 모델은 속성 그래프(Property Graph)입니다. 딱 세 가지 개념만 이해하면 됩니다. ML에서 다루는 그래프 자료구조와 똑같습니다.

① 노드 (Node / Vertex) — 사물

세상의 "것"을 나타냅니다. 사람, 영화, 상품, 계좌, 논문, 단어 — 명사라면 대부분 노드입니다. 노드에는 라벨(Label)로 타입을 붙이고(:Person, :Movie), 속성(Property)으로 세부 정보를 담습니다(name: 'Alice', age: 30).

② 관계 (Relationship / Edge) — 연결

노드와 노드를 잇는 "동사"입니다. 그리고 여기가 핵심인데, GraphDB에서 관계는 방향(direction)과 타입(type), 심지어 자기만의 속성까지 가질 수 있는 1급 데이터입니다. 관계형 DB에서 관계가 "외래 키"라는 그림자 취급을 받는 것과 대조적입니다.

  • (Alice)-[:FOLLOWS]->(Bob) — Alice가 Bob을 팔로우 (방향이 중요!)
  • (User)-[:RATED {stars: 5}]->(Movie) — 관계 자체가 별점 5점이라는 속성을 가짐

③ 속성 (Property) — 세부 정보

노드와 관계 모두에 붙일 수 있는 키-값 쌍입니다. RDB의 컬럼과 비슷하지만, 스키마를 미리 고정하지 않아 노드마다 다른 속성을 가질 수 있습니다(스키마 유연성).

ML 개발자를 위한 대응표

ML에서 배운 그래프 이론과 1:1로 매칭됩니다 — 노드 = Vertex(V), 관계 = Edge(E), 속성 = Feature. 즉 GraphDB는 G = (V, E)에 풍부한 feature를 얹어 영구 저장하는 시스템입니다. 나중에 GNN(그래프 신경망)의 입력으로 그대로 쓸 수 있습니다.

// :Person 라벨을 가진 노드 2개 생성
CREATE (:Person {name: 'Alice', age: 30})
CREATE (:Person {name: 'Bob', age: 27})
// Alice가 Bob을 팔로우하는 관계
MATCH (a:Person {name:'Alice'}), (b:Person {name:'Bob'})
CREATE (a)-[:FOLLOWS {since: 2024}]->(b)
// 노드와 관계를 한 문장에 '그림처럼' 선언
CREATE (alice:Person {name:'Alice'})
       -[:RATED {stars: 5}]->
       (:Movie {title: 'Interstellar'})

// 괄호()는 노드, 대괄호[]는 관계, -->는 방향

04 — Landscape

NoSQL 4대 유형

GraphDB를 제대로 이해하려면 이웃들을 알아야 합니다. NoSQL은 하나의 기술이 아니라 "관계형이 아닌" 여러 종류의 DB를 묶어 부르는 말입니다. 크게 4가지 유형으로 나뉩니다.

Key-Value

키-값 저장소

사물함처럼 키를 넣으면 값이 나옵니다. 구조를 따지지 않아 초고속입니다. 캐시·세션 저장에 최적.

대표: Redis, DynamoDB
약점 값 내부로 질의 불가
Document

문서 저장소

JSON 같은 문서 단위로 저장. 한 문서 안에 중첩 구조를 담아 유연합니다. 앱 데이터·카탈로그에 인기.

대표: MongoDB, Couchbase
약점 문서 간 관계 탐색 약함
Column-Family

와이드 컬럼

행마다 컬럼이 유연하고, 엄청난 쓰기량을 수평 확장으로 감당. 로그·시계열·IoT에 강합니다.

대표: Cassandra, HBase
약점 복잡한 조인·관계 부적합
Graph

그래프 DB

노드와 관계로 연결 그 자체를 저장. 깊은 관계 탐색이 압도적으로 빠릅니다. 추천·사기탐지·지식그래프.

대표: Neo4j, Neptune
약점 단순 대량 집계는 비효율
직관적 비유

Key-Value는 사물함, Document는 서류 폴더, Column-Family는 거대한 엑셀 시트, Graph는 지하철 노선도입니다. "어디서 어디로 어떻게 갈아타나"를 묻는다면 노선도(그래프)가 정답입니다.

05 — Decision Guide

언제 무엇을 쓸까

"그래프가 무조건 최고"가 아닙니다. 도구는 문제에 맞춰 고르는 것. 아래 표로 한눈에 비교해 봅니다.

유형데이터 모델가장 잘 맞는 질문대표 사례
Relational 행 × 열 (테이블)
고정 스키마, ACID
 "조건에 맞는 정형 데이터 집계·트랜잭션" 회계, 재고, 주문
Key-Value 키 → 값
스키마 없음
 "이 키의 값을 가장 빠르게 줘" 캐시, 세션, 랭킹
Document 중첩 JSON 문서
유연한 스키마
 "이 객체 하나를 통째로 저장·조회" 사용자 프로필, 상품 카탈로그
Column-Family 넓은 행 + 컬럼군
쓰기 최적
 "초당 수백만 건을 쓰고 시간축으로 조회" 센서 로그, 이벤트 스트림
Graph 노드 + 관계
관계가 1급 시민
 "어떻게 연결되나? 몇 다리 건너나? 누가 중심인가?" 추천, 사기탐지, 지식그래프, SNS

그래프를 선택해야 하는 신호

  • 데이터끼리의 연결·경로 자체가 핵심 질문일 때 (친구의 친구, 자금 흐름 추적)
  • 가변 깊이 탐색이 필요할 때 ("몇 단계든 연결된 모두 찾기")
  • 관계의 패턴을 찾을 때 (삼각관계, 순환 거래, 커뮤니티)
  • 스키마가 계속 진화하고, 새로운 관계 타입이 수시로 추가될 때
현실 조언 · Polyglot Persistence

실무에서는 하나만 쓰지 않습니다. 주문은 RDB, 캐시는 Redis, 추천 관계는 GraphDB 식으로 각 문제에 맞는 DB를 조합하는 것이 정석입니다. 이를 다중 저장소(Polyglot Persistence)라고 부릅니다.

06 — Under the Hood

왜 이렇게 빠를까: 내부 동작 원리

GraphDB가 깊은 관계 탐색에서 빠른 비결은 인덱스-프리 인접성(Index-Free Adjacency)이라는 한 문장에 담겨 있습니다. 개념만 잡으면 됩니다.

관계형 DB의 방식 — 매번 찾아보기

RDB에서 "Alice의 친구"를 찾으려면, friendship 테이블 전체에서 from_id = Alice인 행을 인덱스로 검색합니다. 데이터가 N개면 이 검색 비용은 대략 O(log N). 한 단계는 괜찮지만, 친구의 친구의 친구… 단계마다 이 검색을 다시 해야 합니다. 데이터가 커질수록 매 단계가 무거워집니다.

그래프 DB의 방식 — 이미 손에 쥐고 있기

GraphDB는 각 노드가 자신의 관계들을 가리키는 물리적 포인터를 직접 들고 있습니다. Alice 노드에서 출발하면, 친구 목록은 검색 없이 곧바로 따라갈 수 있습니다. 한 단계 이동 비용이 데이터 전체 크기 N과 무관합니다 — 그 노드의 이웃 수에만 비례하죠.

핵심 대비

깊이 k만큼 탐색할 때 —
· RDB: 단계마다 인덱스 검색 → 전체 데이터가 커지면 함께 느려짐
· GraphDB: 포인터를 따라갈 뿐 → 전체 데이터 크기와 무관하게 일정한 지역 비용
그래서 "친구의 친구의 친구" 같은 다단계 탐색에서 격차가 폭발적으로 벌어집니다.

비유하자면, RDB는 매번 전화번호부를 다시 펼쳐 이름을 찾는 것이고, GraphDB는 각 사람의 손에 이미 친구들의 연락처가 링크로 꿰어져 있어 그냥 손을 잡고 이동하는 것과 같습니다.

07 — Query Language

Cypher: 그림을 그리는 쿼리 언어

Cypher는 Neo4j가 만든 그래프 쿼리 언어로, 지금은 GQL이라는 이름의 국제 표준(ISO)으로도 자리 잡았습니다. 가장 큰 매력은 ASCII 아트로 그래프 패턴을 그린다는 점입니다.

패턴 문법 — 눈으로 읽힌다

  • () 소괄호는 노드 — 얼굴처럼 동그란 모양
  • [] 대괄호는 관계 — 화살표의 몸통
  • --> 화살표는 방향 — 관계가 향하는 쪽
  • (a)-[:FOLLOWS]->(b) 를 그대로 읽으면 "a가 b를 FOLLOWS"
// Alice가 팔로우하는 모든 사람의 이름
MATCH (a:Person {name: 'Alice'})-[:FOLLOWS]->(friend)
RETURN friend.name
// 30세 이상인 사람을 나이 내림차순으로
MATCH (p:Person)
WHERE p.age >= 30
RETURN p.name, p.age
ORDER BY p.age DESC
LIMIT 5
// 1~3단계 안에 연결된 모든 사람 (가변 깊이!)
MATCH (a:Person {name: 'Alice'})
      -[:FOLLOWS*1..3]->(reachable)
RETURN DISTINCT reachable.name
// *1..3 이 SQL로는 표현조차 힘든 부분!
// 가장 인기 있는(팔로워 많은) 사람 Top 3
MATCH (p:Person)<-[:FOLLOWS]-(follower)
RETURN p.name, count(follower) AS fans
ORDER BY fans DESC LIMIT 3
가장 중요한 문법 하나

[:FOLLOWS*1..3] 의 별표(*)가 그래프 쿼리의 진짜 무기입니다. "1~3단계 사이 어떤 깊이든" 을 뜻하며, 관계형 SQL로는 재귀 CTE를 동원해야 겨우 흉내 내는 것을 한 줄로 표현합니다.

08 — Side by Side

같은 질문, 두 언어

"Alice와 같은 영화를 좋아한 사람들이 본 다른 영화를 추천" — 전형적인 협업 필터링 질문을 두 언어로 나란히 보면 그래프의 표현력이 확 와닿습니다.

SQL — 다중 JOIN + 서브쿼리
SELECT DISTINCT m2.title
FROM ratings r1
JOIN ratings r2
  ON r1.movie_id = r2.movie_id
 AND r1.user_id <> r2.user_id
JOIN ratings r3
  ON r3.user_id = r2.user_id
JOIN movies m2
  ON m2.id = r3.movie_id
WHERE r1.user_id = 1   -- Alice
  AND r3.movie_id NOT IN (
    SELECT movie_id FROM ratings
    WHERE user_id = 1);
Cypher — 경로를 그대로 서술
MATCH (me:Person {name:'Alice'})
      -[:RATED]->(m)<-[:RATED]-(other)
      -[:RATED]->(rec:Movie)
WHERE NOT (me)-[:RATED]->(rec)
RETURN DISTINCT rec.title



// 화살표를 따라 읽으면 곧 논리가 됨:
// 나→(본 영화)←남→(그가 본 다른 영화)
차이의 본질

SQL은 "어떻게 테이블을 이어붙일지"를 명령합니다. Cypher는 "어떤 모양의 연결을 찾고 싶은지"를 그림으로 선언합니다. 사람의 머릿속 질문("나와 취향이 겹치는 사람이 본 다른 영화")에 더 가까운 쪽이 어느 쪽인지 느껴지시나요?

10 — For ML Engineers

GraphDB × 머신러닝

여기가 ML 개발자에게 가장 중요한 대목입니다. 그래프는 단순 저장소를 넘어, 모델의 입력·문맥·특성을 만들어내는 강력한 원천이 됩니다.

🧠

Knowledge Graph

엔티티와 사실을 관계로 엮은 지식 그래프. 사람·회사·개념 사이의 검증된 사실을 구조화해 LLM의 사실 근거로 씁니다.

🔎

GraphRAG

일반 RAG가 텍스트 조각을 벡터로만 찾는다면, GraphRAG는 지식 그래프의 관계를 함께 탐색해 다단계 추론이 필요한 질문에 훨씬 정확한 문맥을 제공합니다.

🕸️

GNN 입력

그래프 신경망(GCN, GraphSAGE, GAT)은 노드·엣지 구조를 그대로 학습합니다. GraphDB는 이 학습 데이터를 저장·추출하는 자연스러운 백엔드입니다.

🎯

추천 시스템

"함께 산 상품", "친구가 좋아한 콘텐츠" 같은 협업 필터링을 관계 탐색만으로 실시간 계산. 콜드스타트 완화에도 유리.

🛡️

사기 탐지

여러 계좌를 순환하는 자금, 공유된 기기·주소로 얽힌 계정 링(ring)을 패턴/커뮤니티 탐지로 잡아냅니다.

📊

그래프 피처 엔지니어링

PageRank(중요도), 중심성(centrality), 커뮤니티, 노드 임베딩(node2vec) 같은 그래프 알고리즘 결과를 피처로 뽑아 기존 모델 성능을 끌어올립니다.

실무 파이프라인 예시

① 원천 데이터를 GraphDB에 적재 → ② 그래프 알고리즘으로 PageRank·커뮤니티 등 피처 계산 → ③ 이 피처를 벡터화해 XGBoost/GNN 학습 → ④ LLM 서비스에 GraphRAG 문맥으로 연결. LLMOps 관점에서 그래프는 "특성 저장소 + 근거 저장소" 두 역할을 동시에 합니다.

// Neo4j GDS(Graph Data Science)로 영향력 점수 계산
CALL gds.pageRank.stream('socialGraph')
YIELD nodeId, score
RETURN gds.util.asNode(nodeId).name AS person,
       score AS influence
ORDER BY influence DESC
// 이 influence 값을 그대로 ML 모델의 피처로 사용
// LLM에게 넘길 '관계 문맥'을 그래프에서 추출
MATCH (d:Disease {name: '당뇨병'})
      -[r:TREATED_BY|CAUSES|RELATED_TO]-(n)
RETURN d.name, type(r), n.name
// → "당뇨병 -TREATED_BY-> 메트포르민" 같은
//    사실 트리플을 프롬프트 근거로 주입
11 — Ecosystem

주요 GraphDB 제품들

그래프 DB도 종류가 다양합니다. 처음 배운다면 자료가 가장 풍부한 Neo4j로 시작하길 권합니다.

Neo4j
Cypher / GQL
그래프 DB의 사실상 표준. 학습 자료·커뮤니티·GDS(그래프 알고리즘 라이브러리)가 가장 풍부해 입문에 최적. 무료 Neo4j Aura Free와 데스크톱 버전 제공.
Amazon Neptune
Gremlin / SPARQL / openCypher
AWS 완전관리형 그래프 DB. Property Graph와 RDF를 모두 지원. 클라우드 인프라와의 통합이 필요한 프로덕션 환경에 적합.
ArangoDB
AQL
그래프 + 문서 + 키-값을 한 엔진에서 다루는 멀티모델 DB. 여러 모델을 오가야 하는 경우 유용.
TigerGraph
GSQL
초대규모·고성능 분산 그래프 분석에 초점. 수십억 개 엣지의 딥 링크 분석이 필요한 대기업 환경.
JanusGraph
Gremlin
오픈소스 분산 그래프 DB. Cassandra·HBase 등을 스토리지 백엔드로 붙여 수평 확장. 인프라를 직접 통제하고 싶을 때.
쿼리 언어 지형

그래프 쿼리 언어는 크게 Cypher/openCypher/GQL 계열과 Gremlin(Apache TinkerPop) 계열, 그리고 RDF용 SPARQL로 나뉩니다. 최근 GQL이 ISO 국제 표준으로 채택되면서 Cypher 계열이 사실상의 공통어로 굳어지는 흐름입니다.

13 — Wrap Up

핵심 정리

01

관계가 주인공 — GraphDB는 관계를 외래 키 그림자가 아닌 방향·타입·속성을 가진 1급 데이터로 저장합니다.

02

3요소 — 노드(사물), 관계(연결), 속성(세부). ML의 V, E, Feature와 정확히 대응합니다.

03

인덱스-프리 인접성 — 관계를 물리적 포인터로 미리 연결해 두어, 다단계 탐색이 데이터 전체 크기와 무관하게 빠릅니다.

04

Cypher — ()-[]->() 로 그래프 패턴을 그림처럼 서술. *1..3 같은 가변 깊이 탐색이 결정적 무기.

05

도구 선택 — 연결·경로·패턴이 질문의 핵심이면 그래프. 그 외엔 RDB/문서/KV/컬럼과 조합(Polyglot).

06

ML 연결고리 — 지식그래프·GraphRAG·GNN 입력·추천·사기탐지·그래프 피처. 그래프는 특성 저장소이자 근거 저장소입니다.
