엣지의 끝에서 돌아가는 쿠버네티스

데이터센터가 아니라 기지국 옆 캐비닛, 공장 한켠, 위성 지상국에서 돌아가는 쿠버네티스가 있습니다. Wind River Cloud Platform은 오픈소스 StarlingX를 기반으로 한 상용 엣지 클라우드 플랫폼으로, 서버 1대부터 수천 개의 분산 사이트까지 하나의 화면에서 관리합니다. 이 강의는 초보 클라우드 엔지니어의 눈높이에서 개념 → 아키텍처 → 사용법 → 타 플랫폼 비교 → 실전 사례까지 차근차근 다룹니다.

Part 1 — 01

Wind River Kubernetes란 무엇인가?

흔히 "Wind River K8s"라고 부르는 것의 정식 명칭은 Wind River Cloud Platform(제품 라인에 따라 Wind River Studio Cloud Platform)입니다. 한 문장으로 정의하면 이렇습니다.

One-line Definition

Wind River Cloud Platform = 오픈소스 StarlingX 프로젝트의 상용(commercially supported) 배포판으로, 쿠버네티스와 컨테이너 기반 아키텍처 위에 지리적으로 분산된 엣지 인프라를 통합 관리하는 기능을 얹은 프로덕션급 엣지 클라우드 플랫폼입니다.

여기서 두 가지 이름의 관계를 먼저 정리해야 합니다. 초보자가 가장 많이 헷갈리는 부분입니다.

Open Source
StarlingX

Open Infrastructure Foundation(구 OpenStack Foundation)이 호스팅하는 오픈소스 프로젝트. 쿠버네티스, Ceph, Keystone 등 검증된 오픈소스를 조합해 엣지용 분산 클라우드 인프라 스택 전체를 제공합니다. 누구나 무료로 다운로드해 쓸 수 있습니다.

Commercial
Wind River Cloud Platform

StarlingX를 상용화한 제품. 기업용 테스트·하드닝·보안 패치·24/7 기술지원·수명주기 관리가 붙습니다. RHEL과 CentOS/Fedora의 관계, 또는 Rancher(SUSE)와 오픈소스 RKE2의 관계와 비슷하다고 생각하면 쉽습니다.

🌬️

비유로 이해하기 — 일반적인 쿠버네티스 배포판(EKS, OpenShift 등)이 "대형 마트 물류센터"라면, Wind River Cloud Platform은 "전국 편의점 체인 관리 시스템"입니다. 점포(엣지 사이트) 하나하나는 작지만 수천 개가 전국에 흩어져 있고, 각 점포에 상주 직원(운영자)을 둘 수 없습니다. 그래서 본사(중앙 System Controller)에서 발주·재고·POS 업데이트(배포·업그레이드·장애 복구)를 원격·무인·일괄로 처리해야 하죠. 이 "본사 시스템" 역할이 이 플랫폼의 핵심입니다.

1.1Wind River라는 회사, 그리고 이 제품의 탄생

Wind River는 원래 화성 탐사 로버, 항공기, 통신 장비에 들어가는 실시간 운영체제(RTOS) VxWorks로 유명한 임베디드 소프트웨어 명가입니다. 즉 "절대 죽으면 안 되는 시스템"을 수십 년간 만들어온 회사죠. 2019년 KubeCon에서 이 회사는 자신들의 미션 크리티컬 DNA를 클라우드에 이식한 쿠버네티스 기반 엣지 인프라 관리 제품으로 Cloud Platform을 발표했습니다. 이후 5G 시대가 열리면서 vRAN/O-RAN(가상화 기지국) 인프라의 사실상 대표 플랫폼 중 하나로 자리잡았습니다.

1.2이 플랫폼이 해결하는 문제

일반 쿠버네티스가 잘 못하는, 그러나 엣지에서는 반드시 필요한 것들입니다.

📡
수천 개 사이트 관리

기지국·공장·매장 등 지리적으로 흩어진 수천 노드를 단일 화면(Single Pane of Glass)에서 관리

🤖
Zero-Touch 운영

현장에 엔지니어 없이 원격 프로비저닝, 자동 장애 감지·복구, 무중단(in-service) 업그레이드

⚡
초저지연 성능

저지연 커널, CPU 격리, HugePages, SR-IOV/DPDK, PTP 시각 동기화 등 통신급 성능 튜닝 내장

🧱
풀스택 제공

OS(리눅스)부터 K8s, 스토리지(Ceph), 네트워킹, 모니터링, GUI까지 인프라 전체를 한 덩어리로

🔌
단절 내성

중앙과 연결이 끊겨도 엣지 사이트는 독립 동작, 재연결 시 자동 재동기화

🖥️
VM + 컨테이너 공존

OpenStack 애드온으로 레거시 VM 워크로드와 컨테이너를 같은 노드에서 함께 운영

초보자 체크포인트

"쿠버네티스 배포판"이 아니라 "쿠버네티스를 포함한 인프라 운영체제"에 가깝다는 점을 기억하세요. EKS/AKS는 K8s 클러스터를 "제공"하지만, Wind River Cloud Platform은 베어메탈 서버의 BIOS 옆 단계부터 OS 설치, 하드웨어 장애 감지, K8s, 그 위의 앱 배포까지 전 계층을 직접 책임집니다.

Part 1 — 02

왜 엣지 클라우드인가 — 배경 지식

Wind River K8s를 이해하려면 먼저 "엣지 컴퓨팅이 왜 필요한가"를 이해해야 합니다. 클라우드 엔지니어라면 대부분 중앙 데이터센터(리전) 중심으로 사고하는 데 익숙하지만, 물리 법칙과 비즈니스 요구가 그걸 허락하지 않는 영역이 있습니다.

2.1빛의 속도는 협상이 안 된다

서울에서 미국 서부 리전까지 왕복 네트워크 지연은 아무리 잘해도 130ms 이상입니다. 국내 리전이라도 수 ms~수십 ms가 걸리죠. 그런데 5G 기지국의 무선 신호 처리(vRAN)는 마이크로초(µs) 단위의 정확도를, 공장 로봇 제어나 자율주행 판단은 수 ms 이내의 응답을 요구합니다. 답은 하나뿐입니다. 연산을 데이터가 발생하는 곳 바로 옆으로 옮기는 것 — 이것이 엣지 컴퓨팅입니다.

2.2엣지의 운영 현실: "서버 2대, 상주 인력 0명"

  • 공간·전력 제약 — 기지국 함체나 공장 캐비닛에는 랙 반 칸 정도의 공간만 있습니다. Wind River Cloud Platform이 서버 1대, 최소 2코어부터 동작하도록 설계된 이유입니다.
  • 인력 부재 — 수천 개 사이트에 운영자를 배치할 수 없으므로 설치(Day 1)와 운영(Day 2) 모두 원격 자동화가 전제 조건입니다.
  • 불안정한 연결 — 백홀 회선이 끊겨도 현장 서비스는 계속되어야 하고, 복구되면 상태가 자동으로 다시 맞춰져야 합니다.
  • 높은 SLA — 통신·산업·국방 영역은 "잠깐 재부팅"이 허용되지 않아, 무중단 업그레이드와 초 단위 페일오버가 필요합니다.
🛰️

현업 관점 — 위성 지상국, 해상 플랜트, 군 전술망처럼 망 분리·간헐적 연결·무인 운영이 기본인 환경을 다뤄본 적이 있다면, "중앙 클라우드 콘솔에 항상 접속 가능하다"는 퍼블릭 클라우드의 전제가 얼마나 사치스러운지 아실 겁니다. 엣지 플랫폼은 그 전제가 무너진 세계를 위한 설계입니다.

2.3용어 미리 정리하기

Edge / Far Edge

데이터 발생 지점과 가까운 소규모 인프라. Far Edge는 기지국처럼 가장 말단의 초소형 사이트를 말합니다.

vRAN / O-RAN

기지국 신호처리를 전용 장비 대신 범용 서버 + 소프트웨어(가상화/컨테이너)로 구현하는 방식. O-RAN은 그 개방형 표준.

Day 0 / 1 / 2

Day 0 설계 → Day 1 설치·초기구성 → Day 2 운영(패치, 확장, 장애대응). 엣지에서는 Day 1·2 자동화가 생존 조건.

Zero-Touch Provisioning

현장 작업 없이 서버 전원만 넣으면 원격에서 OS 설치부터 K8s 구성까지 자동 완료되는 것.

Single Pane of Glass

수천 개 분산 사이트를 하나의 관리 화면/API로 통합 조회·제어하는 것.

PTP (Precision Time Protocol)

µs~ns급 시각 동기화 프로토콜. 5G 기지국 신호 동기화에 필수이며 플랫폼이 기본 지원.

Part 1 — 03

핵심 개념과 구성요소 뜯어보기

Wind River Cloud Platform(=StarlingX)은 크게 두 층으로 이해하면 깔끔합니다. ① 검증된 오픈소스를 조합한 클라우드 인프라 층과, 그 위에 StarlingX가 직접 개발해 얹은 ② 엣지 특화 관리 서비스 층입니다. 후자가 이 플랫폼의 진짜 차별점입니다.

3.1층 ① — 오픈소스 인프라 스택

구성요소역할초보자 설명
Linux (StarlingX OS)호스트 운영체제저지연 커널 옵션을 갖춘 리눅스. 플랫폼이 OS 설치·패치까지 관리합니다.
Kubernetes + containerd컨테이너 오케스트레이션우리가 아는 그 K8s. kubectl, Helm 그대로 사용 가능.
Calico / Multus / SR-IOV CNI네트워킹기본 Calico에 더해, 파드에 물리 NIC를 직결하는 고성능 네트워킹(SR-IOV)을 표준 지원.
Ceph분산 스토리지블록/오브젝트 스토리지를 자체 제공. 엣지엔 외장 스토리지가 없으니 내장이 필수.

Horizon GUI + REST API + CLI관리 인터페이스웹 GUI, CLI(system 명령), REST API, SNMP까지 다중 접근 지원.

3.2층 ② — StarlingX 고유의 엣지 관리 서비스 6총사

이 여섯 개가 시험에도(?) 나오는 핵심입니다. 각각이 독립 서비스로 돌면서 인프라 전 계층을 자동 관리합니다.

sysinv · system
① Configuration Management

노드 검색(discovery)부터 CPU·메모리·NIC·디스크 인벤토리 관리, 설정 변경까지. CLI의 system ... 명령이 바로 이 서비스와 대화합니다. "서버의 프로파일 관리자"입니다.

mtce
② Host Management (Maintenance)

호스트 생존 감시, 하드웨어/프로세스 장애 감지, 자동 격리 및 복구 개시. 노드가 이상하면 사람보다 먼저 알아채고 조치하는 "당직 근무자"입니다.

sm
③ Service Management

컨트롤러 이중화(active/standby)와 서비스 페일오버 담당. 컨트롤러 한 대가 죽어도 수 초 내 반대편으로 넘어가게 하는 HA 엔진입니다.

fm
④ Fault Management

플랫폼 전체의 알람/이벤트를 표준화해 수집·조회(fm alarm-list). 알람 억제(suppression)로 알람 폭풍도 제어합니다. "통합 관제 게시판"입니다.

USM · software
⑤ Software Management

커널·OS·K8s·플랫폼 서비스까지 전 스택의 패치와 메이저 업그레이드를 오케스트레이션. 롤링/무중단(in-service) 업그레이드와 롤백을 지원합니다.

dcmanager
⑥ Distributed Cloud

중앙 System Controller가 원격 Subcloud 수백~수천 개를 등록·배포·감시·일괄 업그레이드. 이 플랫폼의 정체성이자 최대 무기입니다.

개념 연결하기

일반 K8s 운영에서 우리가 Ansible, Prometheus+Alertmanager, ArgoCD, Cluster API, 벤더 펌웨어 툴을 조합해 힘들게 만드는 것들을, 이 플랫폼은 처음부터 하나로 통합된 제품으로 제공한다고 보면 됩니다. 그 대가는 "정해진 방식"을 따라야 한다는 것 — 이 트레이드오프는 6장 비교에서 자세히 다룹니다.

3.3OpenStack 애드온 — VM과 컨테이너의 동거

특이하게도 이 플랫폼에서는 OpenStack이 K8s 위에 컨테이너로 배포됩니다. 순서가 거꾸로 같지만(보통 VM 위에 K8s를 올리죠) 이렇게 하면 OpenStack 자체의 업그레이드·수명주기 관리가 K8s 애플리케이션 관리와 동일해져 운영이 단순해집니다. 결과적으로 같은 노드에서 레거시 VM 워크로드와 최신 컨테이너 워크로드를 함께 돌릴 수 있어, VMware 등에서 넘어오는 마이그레이션 시나리오에서 강력한 카드가 됩니다.

🏗️

비유 — 보통은 "아파트(VM) 안에 조립식 방(컨테이너)"을 넣는데, 여기서는 "조립식 구조물(K8s) 위에 아파트 관리사무소(OpenStack)를 세워서 아파트(VM)를 관리"합니다. 관리사무소도 조립식이라 교체·보수가 쉬워지는 게 포인트입니다. KubeVirt와 지향점이 비슷하지만, 성숙한 OpenStack API 전체(Nova/Neutron/Cinder...)를 그대로 제공한다는 차이가 있습니다.

Part 2 — 04

아키텍처와 4가지 배포 모델

이 플랫폼의 노드는 역할에 따라 Controller(제어부: K8s 컨트롤플레인 + StarlingX 관리 서비스), Worker(워크로드 실행), Storage(Ceph 전용) 세 종류로 나뉩니다. 이 역할들을 어떻게 조합하느냐에 따라 네 가지 표준 배포 모델이 나옵니다.

배포 모델노드 구성HA대표 용도
AIO Simplex (AIO-SX)1대 (Controller+Worker+Storage 겸직)없음기지국 Far Edge, PoC, 소규모 현장

가장 중요한 배포 모델인 Distributed Cloud의 구조입니다. 각 박스에 마우스를 올리면 설명이 나타납니다.

💡 박스에 마우스를 올리면(모바일: 터치) 각 구성요소의 설명이 여기에 표시됩니다.
Distributed Cloud 아키텍처 — 하나의 System Controller가 형태가 제각각인 Subcloud들을 통합 관리
여기서 핵심 통찰

각 Subcloud는 K8s "워커 노드 묶음"이 아니라 자체 컨트롤플레인을 가진 완전한 클러스터입니다. 그래서 중앙과의 회선이 끊겨도 스스로 살아갑니다. "멀티클러스터 페더레이션 + 인프라 수명주기 관리"가 제품 수준으로 통합된 형태라고 이해하면 정확합니다.

설치와 기본 사용법

5.1Day 1 — 설치 흐름 한눈에 보기

부팅 이미지로 OS 설치

StarlingX ISO로 서버를 부팅해 호스트 OS를 설치합니다. 설치 후 sysadmin 계정으로 로그인합니다.

Ansible Bootstrap — 플랫폼 초기화

단 하나의 Ansible 플레이북이 K8s 클러스터 생성, 로컬 Docker 레지스트리, 인증, 관리 서비스까지 부트스트랩합니다. 설정은 localhost.yml 파일 하나에 담습니다.

호스트 구성 — OAM 네트워크, 디스크, 역할 지정

system CLI로 관리(OAM) 네트워크 인터페이스와 Ceph용 OSD 디스크를 지정합니다.

Unlock — 노드를 서비스 상태로 전환

system host-unlock controller-0 한 줄로 노드가 재부팅되며 모든 서비스가 기동됩니다. Lock/Unlock은 이 플랫폼 특유의 안전한 정비 모드 개념입니다.

애플리케이션 배포 — 이제부터는 그냥 K8s

kubectl과 Helm으로 평소처럼 워크로드를 배포합니다. GUI(Horizon)로도 동일 작업이 가능합니다.

Part 3 — 06

6.21:1 상세 비교 — 궁금한 플랫폼을 선택하세요

Wind River Cloud Platform vs 순정 Kubernetes (kubeadm)

kubeadm은 K8s "클러스터"를 만들어줄 뿐, OS 설치·HW 감시·업그레이드 오케스트레이션·멀티사이트는 전부 여러분의 몫입니다. 결국 Ansible+Prometheus+ArgoCD+α를 직접 조립해야 하죠. WRCP는 그 조립 완제품입니다.

Wind River가 유리
  • OS부터 앱까지 수명주기 일원화, 무중단 업그레이드 내장
  • HW 장애 자동 감지·복구, 통합 알람(fm)
  • 수천 사이트 Zero-Touch 배포·일괄 패치
  • 통신급 성능 튜닝(PTP, SR-IOV, CPU 격리) 기본값
순정 K8s가 유리
  • 완전한 자유도 — 모든 컴포넌트 직접 선택
  • 라이선스 비용 없음, 학습 자료 압도적
  • 소수의 클러스터라면 오버엔지니어링 회피

Wind River Cloud Platform vs Red Hat OpenShift

가장 자주 비교되는 상대입니다. OpenShift는 개발자 경험과 엔터프라이즈 PaaS(빌드, 파이프라인, 서비스메시, 콘솔)가 강점이고, WRCP는 통신·엣지 인프라 운영이 강점입니다. 실제로 통신사들은 코어망에 OpenShift, RAN 엣지에 WRCP를 병행하는 경우도 많습니다.

Wind River가 유리
  • Far Edge 초소형(1노드 2코어) 풋프린트 — SNO보다 가벼움
  • 분산 클라우드 관리가 코어 기능(dcmanager) — ACM 같은 별도 제품 불필요
  • vRAN 실적: 대규모 5G RAN 상용망에서 검증
  • OpenStack 통합으로 성숙한 VM 관리 API 제공
OpenShift가 유리
  • 개발자 플랫폼(S2I, 파이프라인, Operator 생태계) 압도적
  • 범용 엔터프라이즈 앱·하이브리드 클라우드 생태계
  • 국내외 인력 풀과 파트너 생태계가 넓음
  • 일반 IT 워크로드 중심이라면 학습 자료 풍부

Wind River Cloud Platform vs Rancher / RKE2

Rancher도 "여러 클러스터를 한 화면에서"를 표방하지만, 관리 대상이 K8s 계층부터입니다. 그 아래의 서버·OS·펌웨어·하드웨어 장애는 범위 밖이죠. WRCP는 베어메탈부터 책임집니다.

Wind River가 유리
  • 베어메탈 수명주기(설치·장애·교체)까지 자동화
  • 통신급 결정성 성능과 캐리어 검증 실적
  • 단절 내성 있는 계층형 분산 아키텍처
Rancher가 유리
  • 이기종 클러스터(EKS+온프렘+K3s 혼합) 통합 관리
  • 가볍고 유연한 도입 — 기존 클러스터도 편입 가능
  • 오픈소스 중심으로 진입 비용 낮음

Wind River Cloud Platform vs K3s / MicroShift

같은 "엣지"라도 급이 다릅니다. K3s·MicroShift는 디바이스급 엣지(라즈베리파이, 매장 단말)를 위한 초경량 K8s이고, WRCP는 인프라급 엣지(기지국, 공장 서버룸)를 위한 캐리어그레이드 플랫폼입니다.

Wind River가 유리
  • HA, 자동복구, 무중단 업그레이드 등 통신급 SLA 기능
  • 수천 사이트 중앙 오케스트레이션 내장
  • Ceph 스토리지·SR-IOV 등 인프라 기능 통합
K3s/MicroShift가 유리
  • 메모리 수백 MB급 초저사양 디바이스 지원
  • 설치 1분, 단일 바이너리의 압도적 단순함
  • IoT 게이트웨이·POS 등 단말형 워크로드에 적합

Wind River Cloud Platform vs 관리형 K8s (EKS·AKS·GKE)

관리형 K8s는 컨트롤플레인 운영을 CSP에 맡기는 대신 CSP의 리전 안에서만 삽니다. 내 기지국, 내 공장, 폐쇄망에는 들어올 수 없죠(Outposts류 확장은 비용·제약이 큼). 반면 WRCP는 처음부터 "내 하드웨어, 내 현장"이 전제입니다.

Wind River가 유리
  • 온프렘·폐쇄망·오프라인 환경 — 주권(sovereign) 요구 충족
  • 물리적 초저지연 (연산이 현장에 있음)
  • 종량 과금 없이 예측 가능한 비용 구조
관리형이 유리
  • 인프라 운영 부담 최소 — 팀이 작아도 됨
  • CSP 생태계(IAM, DB, AI 서비스)와의 즉시 통합
  • 수요 변화에 따른 탄력적 확장
선택 가이드 요약

"지리적으로 흩어진 다수의 무인 사이트 + 베어메탈 + 고성능/고가용 요구" — 세 조건이 겹치면 WRCP(StarlingX)가 1순위 후보입니다. 개발자 플랫폼이 필요하면 OpenShift, 이기종 클러스터 통합이면 Rancher, 디바이스급이면 K3s, 리전 안에서 편하게 가려면 관리형 — 이렇게 기억하세요.

Part 3 — 07

실전 응용 사례

7.1대표 사례: 5G vRAN / O-RAN

이 플랫폼의 최대 실적 영역입니다. 5G 기지국의 신호처리 소프트웨어(vDU/vCU)를 전국 수만 개 사이트의 범용 서버에서 컨테이너로 돌리는 것 — Wind River는 초기 5G 시장에서 vRAN 배포의 다수를 뒷받침한 대표 주자였고, 미국 대형 통신사의 상용 5G vRAN 망 구축에 핵심 클라우드 인프라로 채택되었습니다. 기지국마다 AIO-SX Subcloud를 두고, 지역 System Controller가 이를 묶는 전형적 Distributed Cloud 구조입니다.

  • 왜 이 플랫폼인가 — vDU는 µs급 시각동기(PTP), 전용 CPU 코어, SR-IOV 패킷 처리 없이는 동작 자체가 불가능합니다. 이 요구를 K8s 표준 인터페이스로 감싸 제공하는 것이 결정적이었습니다.
  • 규모의 운영 — 신규 기지국 서버는 전원 연결만으로 Zero-Touch 설치되고, 소프트웨어 패치는 중앙에서 전략(동시 적용 수, 시간대)을 정해 일괄 배포됩니다.

7.2사례 카드 모음

Telecom · Migration
📶 통신사 전체망 VMware 탈출

미국 Boost Mobile은 코어망·엔터프라이즈·O-RAN까지 2만 개 이상 사이트 전체의 클라우드 인프라를 VMware에서 StarlingX 기반 WRCP로 전환했습니다. VM은 OpenStack으로, 컨테이너는 K8s로 — 한 플랫폼에서 수용하며 라이선스 비용 이슈와 벤더 종속을 해소한 대표 사례입니다.

Wrap-up — 08

이해도 점검 퀴즈

총 6문항입니다. 보기를 클릭하면 즉시 정답과 해설이 표시됩니다.

Q1Wind River Cloud Platform과 StarlingX의 관계로 가장 올바른 것은?
RHEL↔CentOS 관계처럼, StarlingX(Open Infrastructure Foundation 오픈소스)에 상용 테스트·하드닝·기술지원을 더한 것이 Wind River Cloud Platform입니다.
Q2기지국 함체에 서버 1대만 놓을 수 있는 Far Edge 사이트에 적합한 배포 모델은?
AIO Simplex(AIO-SX)는 서버 1대가 Controller/Worker/Storage 역할을 모두 수행하는 최소 구성으로, 최소 2코어부터 동작합니다.
Q3수백 개의 원격 Subcloud를 등록·감시하고 일괄 패치 전략을 실행하는 서비스/CLI는?
Distributed Cloud의 중앙 System Controller에서 dcmanager가 Subcloud의 등록(Zero-Touch 설치), 상태 감시, 일괄 업그레이드 전략을 담당합니다.
Q4중앙 System Controller와 네트워크가 단절되면 Subcloud는 어떻게 되나?
각 Subcloud는 자체 컨트롤플레인을 가진 완전한 클러스터라서 단절 중에도 로컬 서비스가 계속 동작하며, 연결 복구 시 중앙과 자동으로 다시 동기화됩니다.
Q5이 플랫폼에서 OpenStack의 배포 방식으로 옳은 것은?
OpenStack이 K8s 위에서 컨테이너로 돌기 때문에 OpenStack 자체의 수명주기 관리가 단순해지고, 같은 노드에서 VM(레거시)과 컨테이너(신규)를 함께 운영할 수 있습니다.
Q6"디바이스급 초경량 K8s가 필요한 매장 POS 단말"에는 어떤 선택이 더 합리적인가?
WRCP는 인프라급 엣지(기지국·공장 서버)를 위한 캐리어그레이드 플랫폼입니다. 수백 MB 메모리의 단말급 디바이스라면 K3s/MicroShift 같은 초경량 배포판이 적합합니다. 도구는 문제의 급에 맞게!
Wrap-up — 09

정리 및 다음 단계

9.1오늘의 핵심 다섯 줄 요약

  • 정체 — 오픈소스 StarlingX의 상용판. K8s 배포판이 아니라 K8s를 포함한 엣지 인프라 운영체제다.
  • 구조 — 오픈소스 스택(K8s/Ceph/Calico...) + StarlingX 고유 관리 서비스 6종(sysinv·mtce·sm·fm·USM·dcmanager).
  • 배포 — AIO-SX(1대) → AIO-DX(2대 HA) → Standard → Distributed Cloud(수천 사이트)로 같은 소프트웨어가 확장된다.
  • 강점 — Zero-Touch 무인 운영, 단절 내성, 무중단 업그레이드, 통신급 성능(PTP/SR-IOV/CPU 격리), VM+컨테이너 공존.
  • 포지션 — "흩어진 무인 사이트 + 베어메탈 + 고SLA"의 교집합에서 최강. 범용 PaaS는 OpenShift, 디바이스급은 K3s가 맞다.

9.2직접 해보기 로드맵

StarlingX AIO-SX를 VM에 설치

vCPU 8 / RAM 32GB / 디스크 500GB VM 하나면 충분합니다. 공식 문서의 "Virtual AIO-SX" 가이드를 따라 부트스트랩부터 unlock까지 경험해 보세요.

두 개의 CLI 세계 익히기

system host-list, fm alarm-list로 인프라를 보고, kubectl/helm으로 앱을 배포하며 계층 구분 감각을 잡습니다.

가상 Distributed Cloud 구성

VM 2~3대로 System Controller + Subcloud를 꾸며 dcmanager로 등록·패치 전략을 실습하면 이 플랫폼의 정수를 체험할 수 있습니다.

엣지 AI 시나리오 얹기

Subcloud에 경량 추론 서비스를 배포하고 중앙에서 버전 롤아웃/롤백을 해보면, 엣지 MLOps 아키텍처 설계에 바로 쓸 수 있는 감각이 생깁니다.
