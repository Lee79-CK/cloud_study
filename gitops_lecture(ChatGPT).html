GitOps 기초 개념과
사용·응용 방법

GitOps는 “운영 환경의 원하는 상태를 Git에 선언하고, 자동화 도구가 실제 환경을 그 상태와 계속 맞추는 방식”입니다. 이 강의는 Git, CI/CD, Kubernetes를 막 배우기 시작한 개발자도 이해할 수 있도록 쉬운 비유와 실습 중심으로 구성했습니다.

Lesson 1

GitOps를 한 줄로 이해하기

GitOps는 Git을 “운영 환경의 설계도이자 변경 이력 장부”로 사용하는 운영 방식입니다.

쉬운 비유

Git은 “주문서”, 클러스터는 “주방”

식당에서 주문서에 “김치찌개 1개”라고 적으면 주방은 그 주문서대로 음식을 만듭니다. GitOps에서 Git 저장소는 주문서입니다. 저장소에 “Deployment 3개, Service 1개, Ingress 1개”라고 적으면 GitOps 도구가 Kubernetes 클러스터를 그 상태로 맞춥니다.

핵심: 운영자가 직접 주방에 들어가서 조리하지 않습니다. 주문서를 바꾸면 자동으로 반영됩니다.
핵심 정의

GitOps란?

애플리케이션과 인프라의 원하는 상태를 Git에 선언하고, 자동화 에이전트가 실제 환경과 비교하여 차이를 줄이는 운영 모델입니다.

  • 변경은 Git Pull Request로 제안합니다.
  • 승인된 변경만 운영 환경에 반영합니다.
  • 실제 환경이 Git과 달라지면 자동으로 감지합니다.
  • 문제가 생기면 Git 이력으로 빠르게 되돌립니다.

Lesson 2

GitOps와 DevOps의 관계

GitOps는 DevOps를 대체하는 것이 아니라, DevOps 원칙을 Git 중심으로 실행하는 구체적인 방법입니다.

DevOps는 문화와 원칙

DevOps는 개발과 운영이 서로 분리되지 않고 함께 빠르고 안정적으로 서비스를 제공하기 위한 문화, 협업 방식, 자동화 원칙입니다.

  • 작게 자주 배포
  • 자동화
  • 모니터링과 피드백
  • 개발·운영 공동 책임

GitOps는 DevOps의 실행 패턴

GitOps는 DevOps의 자동화와 협업 방식을 Git, Pull Request, 선언형 설정, 자동 동기화로 구현합니다.

  • 변경 요청: Pull Request
  • 검토와 승인: Code Review
  • 감사 추적: Git Commit History
  • 배포 자동화: Argo CD / Flux
구분DevOpsGitOps
성격문화, 조직 운영 방식, 자동화 철학Git 기반 운영 자동화 구현 방식
핵심 질문어떻게 개발과 운영이 함께 빠르고 안전하게 일할까?어떻게 Git을 기준으로 운영 환경을 자동으로 맞출까?
변경 관리CI/CD 파이프라인, Change Process, 협업 도구Pull Request, Git 이력, 선언형 Manifest
운영 반영파이프라인이 직접 배포하거나 운영자가 실행클러스터 내부 에이전트가 Git을 Pull하여 반영
대표 도구Jenkins, GitHub Actions, GitLab CI, Azure DevOpsArgo CD, Flux, Helm, Kustomize, External Secrets
정리: DevOps가 “일하는 방식”이라면, GitOps는 “운영 변경을 Git으로 통제하고 자동 반영하는 방식”입니다.
Lesson 3

GitOps의 4대 핵심 원칙

OpenGitOps는 GitOps를 설명하는 대표 원칙으로 선언형, 버전 관리, 자동 Pull, 지속적 조정을 제시합니다.

1

Declarative

운영 환경을 명령어 순서가 아니라 “최종 상태”로 표현합니다.

예: nginx를 3개 띄워라.

2

Versioned

원하는 상태는 Git에서 버전 관리되고 변경 이력이 남아야 합니다.

예: 누가, 언제, 무엇을 바꿨는지 확인.

3

Pull Automatically

배포 도구가 환경 안에서 Git을 확인하고 변경을 가져옵니다.

예: 클러스터 안의 Argo CD가 Git을 읽음.

4

Continuous Reconcile

실제 상태가 원하는 상태와 다르면 계속 비교하고 맞춥니다.

예: 수동 변경 감지 후 원복 또는 알림.

명령형은 “A를 하고 B를 하고 C를 하라”입니다. 선언형은 “최종적으로 C 상태가 되어 있어야 한다”입니다. Kubernetes와 GitOps는 선언형 방식과 잘 맞습니다.

전통적인 Push 방식은 CI 서버가 클러스터 권한을 들고 외부에서 배포합니다. Pull 방식은 클러스터 내부 에이전트가 필요한 권한만 가지고 Git을 읽어 반영하므로 네트워크와 권한 경계를 더 단순하게 설계할 수 있습니다.

Reconcile은 “원하는 상태”와 “실제 상태”를 계속 비교하고 차이를 줄이는 과정입니다. 예를 들어 Git에는 replica가 3인데 실제 Pod가 2개라면, 컨트롤러가 다시 3개가 되도록 조정합니다.

초보자가 자주 헷갈리는 부분: GitOps에서 CI는 보통 “이미지를 만들고 검증”합니다. CD는 GitOps 도구가 “Git에 선언된 배포 상태를 클러스터에 반영”합니다.
Lesson 5

필수 용어를 쉽게 이해하기

처음 GitOps를 접할 때 가장 많이 만나는 용어입니다. 카드를 클릭하면 설명이 뒤집힙니다.

Desired State원하는 상태
Git 저장소에 선언된 최종 목표 상태입니다. “운영 환경은 이렇게 되어야 한다”라는 설계도입니다.
Live State실제 상태
현재 클러스터에 실제로 존재하는 상태입니다. 장애, 수동 변경, 실패 배포 등으로 Git과 다를 수 있습니다.
Drift상태 불일치
Git의 원하는 상태와 클러스터의 실제 상태가 달라진 상황입니다. GitOps 도구가 이를 감지합니다.
Sync동기화
Git에 있는 원하는 상태를 실제 환경에 적용하는 행위입니다. 수동 Sync 또는 자동 Sync가 가능합니다.
Rollback되돌리기
문제가 있는 Git 변경을 이전 커밋으로 되돌리거나 이전 Manifest 버전으로 복구하는 방식입니다.
Manifest선언 파일
Kubernetes 리소스를 YAML/JSON으로 표현한 파일입니다. Deployment, Service, Ingress 등이 대표적입니다.
Lesson 6

GitOps 저장소 구조 설계

좋은 저장소 구조는 운영 안정성과 팀 협업 수준을 결정합니다.

app-repo/ ├─ src/ ├─ tests/ ├─ Dockerfile └─ .github/workflows/ci.yaml gitops-repo/ ├─ base/ ├─ overlays/dev/ ├─ overlays/stg/ └─ overlays/prd/

왼쪽 저장소 트리의 각 파일/폴더가 어떤 역할을 하는지 표시됩니다.

환경별 분리 전략

초기에는 dev/stg/prd를 폴더로 나누는 방식이 이해하기 쉽습니다. 조직이 커지면 환경별 저장소, 팀별 저장소, 클러스터별 저장소로 분리할 수 있습니다.

예: Kustomize overlays
gitops-repo/
├─ base/
│  ├─ deployment.yaml
│  ├─ service.yaml
│  └─ kustomization.yaml
└─ overlays/
   ├─ dev/
   │  ├─ kustomization.yaml
   │  └─ patch-replica.yaml
   ├─ stg/
   │  └─ kustomization.yaml
   └─ prd/
      ├─ kustomization.yaml
      └─ patch-resource.yaml

App of Apps 패턴

Argo CD에서 여러 애플리케이션을 하나의 상위 Application으로 관리하는 방식입니다. 클러스터에 필요한 공통 컴포넌트와 업무 애플리케이션을 한 번에 부트스트랩할 때 유용합니다.

  • 공통: ingress-nginx, cert-manager, external-secrets, monitoring
  • 업무: portal, api, batch, admin
  • 환경: dev, stg, prd
권장: 애플리케이션 소스 저장소와 배포 Manifest 저장소를 분리하면 권한, 감사, 승인 절차를 더 명확히 나눌 수 있습니다.
Lesson 7

Argo CD로 GitOps 실습하기

아래 실습은 Kubernetes 클러스터가 있다는 가정의 예시입니다. AKS, EKS, GKE, kind, minikube에서 흐름은 거의 동일합니다.

1단계: Argo CD 설치

Argo CD는 Kubernetes에 설치되는 컨트롤러입니다. Git 저장소와 클러스터 상태를 비교합니다.

install-argocd.sh
kubectl create namespace argocd
kubectl apply -n argocd --server-side --force-conflicts \
  -f https://raw.githubusercontent.com/argoproj/argo-cd/stable/manifests/install.yaml

kubectl get pods -n argocd
운영 환경에서는 설치 전 네트워크, 인증, RBAC, Ingress, TLS, SSO, 백업 정책을 함께 설계해야 합니다.

2단계: Application 선언

Argo CD Application은 “어느 Git 저장소의 어느 경로를 어느 클러스터/네임스페이스에 반영할지” 정의합니다.

application.yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: hello-gitops
  namespace: argocd
spec:
  project: default
  source:
    repoURL: https://github.com/example/gitops-repo.git
    targetRevision: main
    path: overlays/dev
  destination:
    server: https://kubernetes.default.svc
    namespace: hello
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
    syncOptions:
      - CreateNamespace=true

3단계: 변경하고 동기화 확인

이미지 태그를 바꾸거나 replica 수를 바꾼 뒤 Pull Request를 Merge합니다. Argo CD가 OutOfSync를 감지하고 Sync합니다.

deployment patch 예시
apiVersion: apps/v1
kind: Deployment
metadata:
  name: hello-gitops
spec:
  replicas: 3
  template:
    spec:
      containers:
        - name: app
          image: ghcr.io/example/hello-gitops:v1.2.3
  • OutOfSync: Git과 클러스터가 다름
  • Synced: Git과 클러스터가 같음
  • Healthy: 리소스가 정상 동작함

4단계: 문제 발생 시 롤백

GitOps의 롤백은 “클러스터에서 수동 수정”보다 “Git 변경을 되돌리는 것”이 기본입니다.

rollback 예시
# 문제가 된 커밋을 되돌리는 PR 생성
git revert <bad-commit-sha>
git push origin rollback/hello-v1.2.2

# PR 승인 후 merge
# Argo CD가 이전 안정 상태로 자동 동기화
Lesson 8

GitOps 응용 방법

GitOps는 단순 배포 자동화를 넘어 플랫폼 운영, 보안 정책, 멀티 클러스터 관리에 활용됩니다.

1. 환경 승격

dev에서 검증한 이미지 태그를 stg, prd로 순차 반영합니다. 태그 변경 PR 자체가 배포 승인 문서가 됩니다.

2. 정책 as Code

NetworkPolicy, ResourceQuota, Pod Security, Kyverno/OPA 정책을 Git으로 관리하여 표준을 강제합니다.

3. 공통 컴포넌트

Ingress Controller, External Secrets, Monitoring, Logging Agent 등을 클러스터 부트스트랩 구성으로 관리합니다.

Lesson 9

보안과 운영 체크포인트

GitOps의 장점은 Git에 모든 것을 올리는 것이 아니라, 안전하게 통제 가능한 변경 경로를 만드는 것입니다.

도입 전 체크리스트

Secret 관리 방식

방식설명
SOPS암호화된 Secret 파일을 Git에 저장하고 클러스터에서 복호화합니다.
Sealed Secrets클러스터 공개키로 암호화한 Secret만 Git에 저장합니다.
External SecretsAWS Secrets Manager, Azure Key Vault, HashiCorp Vault 등 외부 저장소에서 Secret을 가져옵니다.
Vault 연동중앙 Secret 관리와 동적 Secret 발급이 필요한 조직에 적합합니다.
금지에 가까운 안티패턴: 운영 장애가 났다고 클러스터에서 직접 YAML을 고치고 Git에는 반영하지 않는 방식. 나중에 GitOps 도구가 다시 원래 상태로 덮어쓸 수 있습니다.
운영 팁: 운영 자동 Sync는 편리하지만, 초기에는 수동 Sync 또는 승인 기반 Sync로 시작한 뒤 안정화되면 자동화 범위를 넓히는 것이 좋습니다.
Lesson 10

GitOps 도입 로드맵

한 번에 모든 시스템을 GitOps로 바꾸기보다, 작게 시작해서 표준을 넓히는 접근이 안전합니다.

1단계: 읽기 전용 가시화

Argo CD를 설치하고 기존 Manifest를 연결합니다. 처음에는 자동 Sync를 끄고 OutOfSync/Health를 관찰합니다.

2단계: 개발 환경부터 자동화

dev 환경에 자동 Sync를 적용합니다. PR → Merge → 자동 배포 흐름을 팀이 익숙하게 만듭니다.

3단계: 스테이징 승인 절차 연결

stg는 리뷰어 승인과 테스트 결과를 조건으로 승격합니다. 배포 실패 알림을 붙입니다.

4단계: 운영 환경 표준화

prd는 Change 승인, Branch Protection, 보안 스캔, 수동 Sync 또는 제한적 자동 Sync로 운영합니다.

5단계: 플랫폼·정책까지 확장

공통 컴포넌트, 보안 정책, 모니터링, 멀티 클러스터 구성을 GitOps로 관리합니다.

권장 시작점: 영향도가 낮은 내부 서비스 1개를 선택해 “소스 저장소 + 배포 저장소 + Argo CD Application + PR 승인” 흐름을 먼저 경험하세요.

강의 요약

GitOps는 Git을 기준으로 운영 환경의 원하는 상태를 선언하고, 자동화 도구가 실제 환경을 지속적으로 맞추는 방식입니다. DevOps의 협업·자동화·피드백 원칙을 Kubernetes와 클라우드 네이티브 환경에 구체적으로 적용하는 실전 운영 모델입니다.

이 문서는 인쇄 모드에서도 전체 내용이 표시되도록 구성되었습니다.
