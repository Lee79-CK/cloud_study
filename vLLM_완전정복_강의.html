LLM 서빙은 왜 어려운가

모델을 학습하는 것과 배포하는 것은 완전히 다른 문제입니다. "Hello world"부터 시작해서, 왜 단순히 PyTorch로 model.generate()를 호출하면 안 되는지부터 살펴봅시다.

1) 토큰을 만드는 과정은 본질적으로 순차적이다

LLM은 한 번에 한 토큰씩 생성합니다. 새 토큰을 만들려면 이전에 생성된 모든 토큰을 다시 봐야 하죠. 이 "이전 토큰들의 attention 결과"를 매번 다시 계산하면 너무 느리기 때문에, KV Cache(Key-Value Cache)라는 임시 저장소에 저장해 둡니다.

▮ 핵심 개념: KV Cache
Transformer는 매 토큰마다 자기 자신과 이전 모든 토큰들 사이의 attention을 계산합니다. 이전 토큰들의 Key와 Value 벡터를 캐싱해 두면, 새 토큰 하나 만들 때 행렬 곱셈이 O(n²) → O(n)으로 줄어듭니다. 대신 GPU 메모리를 굉장히 많이 먹습니다.

2) KV Cache는 메모리 폭식가입니다

예시를 봅시다. Llama 3.1 70B 모델로 한 사용자에게 2048 토큰의 컨텍스트를 서빙한다고 가정하면:

◉

모델 가중치

FP16 기준 약 140GB. H100 80GB GPU 한 장으로는 불가능. (FP8 양자화시 ~70GB)

◉

KV Cache (사용자 1명)

한 토큰당 약 320KB × 2048 토큰 = ~650MB. 32명이면 20GB.

◉

활성 메모리

중간 계산 결과 buffer, gradient 없음에도 추가 메모리 필요. 여기에 PyTorch 자체 오버헤드까지.

3) 단순한 배치(static batching)는 GPU를 놀게 만든다

가장 단순한 방법은 8개 요청을 모아 한 번에 처리하는 것입니다. 그런데 이게 큰 문제를 만듭니다:

  • 길이가 다 다르다: 한 요청은 50 토큰, 다른 요청은 500 토큰. 짧은 요청이 끝나도 긴 요청 끝날 때까지 슬롯이 비어있음
  • 메모리 단편화: "최대 길이"만큼 KV Cache 슬롯을 미리 잡아두니, 실제로는 안 쓰는 메모리가 60~80%
  • 대기 시간 증가: 새 요청이 와도 현재 배치가 끝날 때까지 기다려야 함

이런 문제들 때문에, 단순한 추론 코드로 production을 운영하면 GPU 사용률이 20~30% 수준에 머무릅니다. 값비싼 H100/A100을 사놓고 70%를 놀리는 셈이죠. vLLM은 바로 이 문제를 풀기 위해 태어났습니다.

vLLM이란?

UC Berkeley Sky Computing Lab에서 2023년 시작된 오픈소스 LLM 추론·서빙 엔진. 현재는 PyTorch 생태계의 핵심 프로젝트로 자리잡았고, 업계 표준이라 부를 만한 사실상의 디폴트입니다.

Apache 2.0 Open-Source PyTorch

"Virtual" Large Language Model

이름의 'v'는 virtual memory에서 왔습니다. 운영체제가 RAM을 페이지 단위로 가상화해 관리하듯, GPU의 KV Cache를 가상화해 관리하자는 발상이 vLLM의 출발점입니다.

SOSP 2023 Paper

학술적으로도 검증된 시스템

"Efficient Memory Management for Large Language Model Serving with PagedAttention" (Kwon et al., 2023) 논문이 SOSP에 발표되며 학계와 산업계 양쪽에서 인정받았습니다.

vLLM이 해결하려는 4가지 핵심 문제

01

메모리 단편화

외부·내부 단편화로 낭비되던 KV Cache 공간을 95%+ 활용률로 끌어올림

02

요청별 토큰 단위 스케줄링으로 GPU가 토큰 1개도 놓치지 않게 만듦

03

대기 지연

새 요청이 진행 중인 배치에 즉시 합류 가능 (continuous batching)

04

이식성

OpenAI API 호환 서버 제공. Hugging Face Hub의 거의 모든 모델 지원

▮ 2026년 현재 위상
Meta · DeepSeek · Qwen · Mistral 등 주요 오픈웨이트 모델 출시 시 1순위로 지원되는 엔진입니다. Hugging Face의 Inference Endpoints도 기본값이 vLLM으로 바뀌었고, 직전 표준이던 TGI는 2025년 12월부터 유지보수 모드에 들어갔습니다.

핵심 1: PagedAttention

"GPU 메모리를 운영체제 메모리처럼 다루자." 이 한 줄짜리 아이디어가 LLM 서빙을 다시 썼습니다.

운영체제에서 빌려온 아이디어

리눅스 커널을 만져본 분이라면 친숙할 겁니다. OS는 프로세스 메모리를 page라는 작은 고정 크기 블록으로 잘라서, 물리 RAM의 아무 데나 흩어놓고 page table로 매핑합니다. 덕분에 메모리 단편화가 거의 없죠.

vLLM은 똑같은 짓을 GPU에서 합니다. KV Cache를 16 토큰짜리 페이지(block)로 잘라서, GPU 메모리에 흩뿌리고 block table로 매핑합니다.

기존 방식 (vLLM 이전)

  • 요청 1개당 "최대 길이" 만큼 연속된 메모리 슬롯을 미리 할당
  • 실제로 200 토큰만 써도, 2048 토큰 슬롯이 점유됨
  • 슬롯 사이에 외부 단편화 발생 → 실제 사용률 20~40%

PagedAttention

  • 16토큰짜리 페이지를 필요할 때마다 1개씩 할당
  • 물리 페이지는 흩어져 있어도, 논리 시퀀스는 연속처럼 동작
  • 외부 단편화 거의 0, 내부 단편화도 페이지 크기 이하 → 사용률 96%+

인터랙티브 시각화: 페이지가 할당되는 모습을 직접 보세요

아래 버튼들을 눌러보세요. 요청이 들어오면 vLLM이 16토큰짜리 페이지를 GPU 메모리의 빈 곳 아무 데나 할당합니다.

▮ 핵심 통찰
페이지는 16토큰짜리 고정 블록입니다. 요청이 짧으면 페이지 끝에 살짝 낭비(내부 단편화)가 생기지만, 최대 16토큰 × 320KB ≈ 5MB가 한 페이지의 최대 낭비량입니다. 기존 방식에서 수백 MB씩 버려지던 것과 비교가 안 되죠.

덤: Prefix Sharing (프리픽스 공유)

페이지 단위 관리의 부수적 효과가 있습니다. 여러 요청이 똑같은 시스템 프롬프트로 시작하면? 같은 물리 페이지를 가리키게(reference) 만들면 됩니다. 운영체제의 copy-on-write와 동일한 발상이에요.

예: 챗봇 서비스에서 모든 요청이 "당신은 친절한 어시스턴트입니다..." 같은 5000자짜리 시스템 프롬프트를 공유하면, 이 부분의 KV Cache는 모든 사용자가 같은 메모리를 공유합니다. 메모리도 아끼고, 첫 토큰 응답 시간(TTFT)도 단축됩니다. 이를 Automatic Prefix Caching (APC)라고 부릅니다.

핵심 2: Continuous Batching

배치를 "요청 단위"가 아니라 "토큰 단위"로 묶는다 — 23배의 처리량 증가를 만든 두 번째 비밀.

Static Batching의 한계

아래 그림에서 4개 요청을 static batching으로 처리한다고 생각해 봅시다. 모든 요청은 가장 긴 요청이 끝날 때까지 자리를 비울 수 없습니다.

A, B는 토큰 생성이 일찍 끝났는데도 GPU 슬롯을 계속 점유. D는 들어오지도 못함. GPU 사용률 손해.

매 토큰 step마다 스케줄러가 결정: "이 자리에 D를 넣자." GPU가 한 순간도 놀지 않음.

iteration-level scheduling

vLLM의 스케줄러는 "한 요청이 끝날 때까지 기다리지" 않습니다. 매 추론 iteration(1 토큰 생성)마다 다음을 결정합니다:

  • RUNNING — 지금 토큰을 생성 중인 요청들
  • WAITING — 아직 시작 못한, 또는 메모리 부족으로 preempt된 요청들
  • SWAPPED — KV Cache가 CPU로 옮겨져 있는 요청들

토큰 한 번 생성될 때마다 끝난 요청을 빼고, 새 요청을 채우고, 메모리가 모자라면 우선순위가 낮은 요청을 잠시 CPU로 swap합니다. OS의 process scheduler를 그대로 LLM에 적용한 셈이에요.

05 / 13

알아두면 좋은 추가 기능

PagedAttention과 Continuous Batching이 양대 기둥이지만, vLLM에는 production에서 빛을 발하는 기능들이 더 있습니다.

◇ Chunked Prefill

긴 프롬프트(예: 32K 토큰 RAG 컨텍스트)를 한 번에 처리하면 TTFT가 폭증합니다. vLLM은 prefill을 청크로 잘라서 디코딩과 함께 인터리브합니다. 결과: 긴 프롬프트가 있어도 짧은 요청의 대기 시간이 안정됨.

◇ Speculative Decoding

작은 "draft 모델"이 토큰 여러 개를 예측하고, 큰 "target 모델"이 검증만 하는 방식. 디코딩이 1.5~3배 빨라집니다. vLLM은 EAGLE-2, MLP-Speculators 등 다양한 방식 지원.

◇ Tensor / Pipeline Parallelism

큰 모델(예: Llama 70B, DeepSeek-V3)을 여러 GPU에 분산. --tensor-parallel-size 4 한 줄로 H100 4장에 가중치를 쪼개 올릴 수 있어요.

◇ Quantization 지원

FP8 (Hopper 이상), AWQ, GPTQ, BitsAndBytes, Marlin 등 거의 모든 양자화 포맷을 흡수. 70B 모델을 단일 H100 80GB에 올릴 수 있는 비결.

◇ OpenAI API 호환

/v1/chat/completions, /v1/embeddings 등 OpenAI SDK가 그대로 동작. 코드 수정 없이 GPT-4 호출을 사내 vLLM 서버로 바꿀 수 있음.

◇ Structured Output (Guided Decoding)

JSON 스키마, 정규식, 문맥 자유 문법으로 출력 제약. Function calling, RAG, 에이전트 만들 때 환각·포맷 오류 방지에 필수.

◇ Multi-LoRA Serving

베이스 모델 1개 + LoRA 어댑터 N개를 동시에 서빙. 도메인별로 fine-tune한 어댑터들을 메모리 효율적으로 운영. --enable-lora.

◇ 멀티모달 지원

Qwen2.5-VL, Llama 3.2 Vision, InternVL 등 VLM 지원. 이미지/비디오를 텍스트와 함께 추론하는 멀티모달 워크로드도 같은 엔진으로 서빙.

실제로 요청 1건이 들어오면 vLLM 내부에서 어떤 일이 벌어지는지, 컴포넌트 단위로 따라가 봅시다.

구성 요소 살펴보기

1. AsyncLLMEngine

비동기 메인 루프. 매 step마다 스케줄러에게 "이번에 처리할 요청 묶음을 줘"라고 요청하고, GPU worker에게 forward를 시키고, 결과를 클라이언트에게 스트리밍합니다.

2. Scheduler

vLLM의 두뇌. WAITING / RUNNING / SWAPPED 큐를 관리합니다. KV Cache가 부족하면 우선순위가 낮은 요청을 preempt(중단)해서 CPU로 swap하거나, 아예 재계산(recompute) 모드로 전환합니다.

3. Block Manager (PagedAttention의 핵심)

고정 크기 페이지 풀을 관리. 매 토큰 step마다 각 요청에게 페이지를 할당/해제하고, 시퀀스가 페이지를 공유하는 경우(prefix caching) 참조 카운팅합니다.

4. Worker

GPU 한 장당 하나씩 떠 있는 프로세스. Tensor parallel일 때는 GPU 개수만큼 worker가 NCCL로 통신합니다. 실제 모델 추론을 담당.

5. Model Runner + Custom CUDA Kernels

표준 PyTorch attention 대신 vLLM의 paged-attention 커널을 호출. block table을 GPU에 올려두고 흩어진 KV 페이지에서 attention을 계산합니다.

▮ V1 아키텍처 (2025+)
2025년 vLLM은 "v1" 리팩토링을 거치며 더 깔끔한 코어로 재탄생했습니다. 스케줄러가 단순해지고, prefill과 decode가 통합 스케줄링되어 chunked prefill이 default가 되었습니다. 최신 버전을 쓰는 게 좋습니다.

Step 2. 오프라인 배치 추론 (가장 간단한 사용법)

python · offline_inference.py 
from vllm import LLM, SamplingParams

# 1) 모델 로드 (Hugging Face Hub에서 자동 다운로드)
llm = LLM(
    model="Qwen/Qwen2.5-7B-Instruct",
    gpu_memory_utilization=0.9,     # GPU 메모리의 90% 사용
    max_model_len=8192,             # 최대 컨텍스트 길이
    dtype="bfloat16",
)

# 2) 샘플링 파라미터
params = SamplingParams(
    temperature=0.7,
    top_p=0.9,
    max_tokens=256,
)

# 3) 한 번에 여러 프롬프트를 batch로 처리 (continuous batching이 알아서)
prompts = [
    "vLLM이 빠른 이유를 3가지로 요약해줘.",
    "Kubernetes에서 GPU 노드를 라벨링하는 명령어는?",
    "PagedAttention을 OS 비유로 설명해줘.",
]

outputs = llm.generate(prompts, params)
for out in outputs:
    print(out.outputs[0].text)

유사 라이브러리 전수 비교

LLM 서빙 엔진은 vLLM 외에도 많습니다. 2026년 현재 살아있는 옵션들을 한자리에 놓고 따져봅시다.

vLLM

표준

UC Berkeley 발. PagedAttention의 원조. 가장 큰 커뮤니티, 가장 빠른 신규 모델 지원. 멀티 클라우드/멀티 하드웨어(NVIDIA/AMD/TPU/Intel). 안전한 디폴트.

SGLang

신흥강자

LMSYS 발. RadixAttention(트리 기반 프리픽스 공유)으로 멀티턴 채팅·RAG·에이전트 워크로드에서 vLLM 대비 +29% 처리량. 구조화 출력도 강함.

TensorRT-LLM

NVIDIA 전용

NVIDIA 공식. 모델별로 28분짜리 컴파일 단계가 필요한 대신, 같은 H100에서 가장 빠른 절대 throughput. 모델 자주 바꾸는 환경에는 부적합.

HF TGI

유지보수 모드

Hugging Face Text Generation Inference. 2025년 12월부터 신규 기능 동결. HF 자체 Inference Endpoints도 디폴트가 vLLM으로 변경됨. 신규 프로젝트는 vLLM/SGLang 권장.

LMDeploy

중국 발

상하이 AI Lab. InternLM/Qwen 등 중국 모델에 강함. TurboMind 엔진은 일부 워크로드에서 vLLM보다 빠르나 모델 지원 범위가 좁음.

llama.cpp / Ollama

로컬/CPU

C++ 기반, GGUF 양자화. CPU·맥북·라즈베리파이까지 돌아감. 단일 사용자, 개발 환경, 데스크톱 도구에 최적. production 동시성 처리에는 부적합.

Triton Inference Server

오케스트레이션

NVIDIA의 범용 모델 서버. LLM·CV·ASR 등 다양한 모델을 한 GPU에 멀티 호스팅. vLLM/TensorRT-LLM을 backend로 끼워 쓸 수 있음 (위 레이어).

Ray Serve

오케스트레이션

Ray 기반 서빙 레이어. vLLM 엔진 N개를 Ray cluster 위에 띄우고 LoRA 어댑터·라우팅·오토스케일링을 묶음. 멀티 모델 운영시 강력.

Kubernetes 배포

실전 LLMOps. GPU 노드 라벨링부터 readiness probe, 오토스케일링까지 — production-ready vLLM 매니페스트를 만들어 봅시다.

전제: 클러스터 준비

성능 튜닝 체크리스트

같은 GPU, 같은 모델인데 처리량이 2~3배 차이가 나는 이유는 보통 튜닝입니다. 가장 자주 손대는 노브들.

1. --gpu-memory-utilization

기본 0.9. KV cache가 가져갈 GPU 메모리 비율. 너무 높으면 OOM, 너무 낮으면 동시 처리 능력↓. 모델 로드 후 nvidia-smi로 여유 메모리 확인 후 조정.

2. --max-model-len

실제 사용할 최대 컨텍스트로 제한. 4K만 쓰는데 32K로 잡으면 KV cache 슬롯이 4배 작게 만들어짐. 워크로드의 p99 길이로 설정.

3. --max-num-seqs

동시에 처리할 최대 시퀀스 수. 너무 크면 메모리 압박, 너무 작으면 처리량 손해. 보통 64~256.

4. --enable-prefix-caching

시스템 프롬프트 공유 워크로드(챗봇·RAG)에서 필수. TTFT가 극적으로 개선됨. 거의 켜두는 게 좋음.

5. --enable-chunked-prefill

긴 프롬프트가 짧은 요청들을 막지 않도록 prefill을 청크로 잘라 디코딩과 인터리브. 긴 컨텍스트 운영시 필수.

6. --tensor-parallel-size

큰 모델 분산. 단, GPU간 통신(NCCL) 비용이 추가되므로 모델이 진짜 큰 경우(70B+)에만. 작은 모델은 데이터 병렬(replica 늘리기)이 더 빠름.

7. 양자화 (FP8 / AWQ / GPTQ)

70B를 H100 80GB 한 장에 올리려면 FP8/AWQ가 거의 필수. 정확도 손실은 보통 1~3% 이내. Hopper(H100/H200)에서는 FP8이 가장 빠름.

8. CUDA Graph (자동)

vLLM이 알아서 CUDA Graph로 디코딩 step을 캡처. --enforce-eager로 끄지 마세요 (디버그시에만).
