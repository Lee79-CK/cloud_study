CUDA, 병렬 컴퓨팅의
가장 친절한 첫걸음

딥러닝이 가능해진 결정적 이유는 GPU와 CUDA입니다. 이 강의는 "이름은 많이 들었지만 한 번도 제대로 배워본 적은 없는" AI 엔지니어가 CUDA의 본질을 머릿속에 또렷한 그림으로 그릴 수 있게 만드는 데에 초점을 둡니다. 수식을 외우는 강의가 아니라, 왜 그렇게 설계되었는가를 따라가는 강의입니다.

대상 : AI 입문 ~ 중급 예상 학습 시간 : 60–90분 선행 지식 : C 문법 기초
02 · 개념

CUDA란 무엇인가

CUDA(Compute Unified Device Architecture)는 NVIDIA가 2007년에 공개한 병렬 컴퓨팅 플랫폼이자 프로그래밍 모델입니다. 한 단어로 정의하기 어려운 이유는, CUDA가 여러 층(layer)으로 이루어진 스택이기 때문입니다.

한 줄로 말하면

"GPU를 그래픽 출력이 아니라, 범용 수치 계산에 쓰기 위해 NVIDIA가 만든 표준 인터페이스 + 컴파일러 + 런타임 + 라이브러리 묶음" 이 CUDA입니다.

CUDA가 등장하기 전과 후

2007년 이전에도 사람들은 GPU의 막대한 연산 자원을 일반 계산에 쓰고 싶어 했습니다. 문제는, GPU가 본래 "픽셀을 칠하는 기계"였다는 점입니다. OpenGL/DirectX 셰이더를 우회해 행렬 곱을 시키는 식의 트릭(GPGPU)을 썼지만, 코드는 이해할 수 없는 형태였고 디버깅도 불가능했습니다.

CUDA는 이 상황을 바꿔놓았습니다. C 언어 확장으로 GPU 코드를 작성하고, nvcc 컴파일러가 그것을 GPU 명령어로 번역해 주는 모델을 만들었기 때문입니다. 이후 10여 년 동안 cuDNN·cuBLAS 같은 고성능 라이브러리가 쌓이고, 그 위에 PyTorch와 TensorFlow가 올라가면서, 오늘날 우리가 아는 딥러닝 생태계가 형성되었습니다.

핵심 통찰

CUDA는 단순한 라이브러리가 아니라, "하드웨어–컴파일러–런타임–라이브러리–프레임워크"가 한 줄로 정렬된 스택(stack)입니다. 우리가 torch.cuda.is_available()을 부를 때, 사실은 이 스택 전체가 함께 움직이고 있습니다.

03 · 직관

왜 GPU인가 — CPU와 무엇이 다른가

"GPU는 CPU보다 빠르다"는 말은 절반만 맞습니다. 정확히는 "GPU는 같은 연산을 동시에 수만 번 반복하는 일에 특화"되어 있습니다. 반대로 복잡한 분기나 순차적 의사 결정은 여전히 CPU가 압도적으로 빠릅니다. 구조 자체가 다르기 때문입니다.

비유 — 교수님 4명 vs 조교 수천 명

CPU는 박사급 교수님 몇 명이 책상에 앉아 있는 것과 같습니다. 한 명 한 명이 어떤 어려운 문제든 빠르게 풀 수 있지만, 머릿수가 적습니다. GPU는 대학원생 수천 명이 한 강당에 모여 있는 것과 같습니다. 한 명의 능력은 교수보다 떨어지지만, "각자 1번부터 1024번까지 동일한 계산을 해라"라는 지시에 동시에 응답할 수 있습니다. 딥러닝의 행렬 곱은 정확히 이 두 번째 패턴입니다.

04 · 실행 모델

SIMT — CUDA의 심장

CPU가 SIMD(Single Instruction, Multiple Data)로 한 명령에 여러 데이터를 처리한다면, CUDA는 SIMT(Single Instruction, Multiple Threads)를 채택합니다. 한 명령을 여러 스레드가 동시에 실행하되, 스레드 각각이 자신만의 인덱스를 가지고 자기 몫의 데이터를 처리합니다.

왜 SIMT가 강력한가

예를 들어 길이 1,000,000인 두 벡터를 더한다고 합시다. CPU에서는 for 루프로 100만 번 반복합니다. CUDA에서는 100만 개의 스레드를 띄워서, 각 스레드가 자기 인덱스 i의 한 쌍만 더하고 종료합니다. 스레드 사이에는 의존 관계가 없으므로, 하드웨어가 가능한 만큼 동시에 실행합니다. 이 개념을 임베러싱리 패럴렐(embarrassingly parallel)이라고 부릅니다.

SIMT의 단위 — Warp

NVIDIA GPU는 32개의 스레드를 묶은 단위(warp, 워프)로 명령을 실행합니다. 한 워프 안의 32개 스레드는 완전히 같은 명령을 동시에 실행합니다. 만약 한 워프 안에서 if로 갈래가 갈리면, 두 갈래를 차례로 실행해야 하므로 속도가 절반으로 떨어집니다. 이를 warp divergence(워프 분기)라 합니다.

시뮬레이션 — 벡터 덧셈을 SIMT가 어떻게 처리하는가

▶ 실행을 누르면 8개 스레드가 동시에 한 칸씩 처리합니다. CPU 루프와 달리 8칸이 한 단계에서 모두 켜집니다.

입력 A
입력 B

대기 중
Q. 한 워프(warp)는 몇 개의 스레드로 구성될까요?
정답입니다. NVIDIA 아키텍처는 일관되게 32개를 한 워프로 묶어 동기적으로 실행합니다. 이 숫자는 외워두면 좋습니다 — 블록 크기를 32의 배수로 잡는 이유가 됩니다.
05 · 핵심 모델

스레드 계층 — Thread → Block → Grid

CUDA에서 가장 중요한 추상은 스레드의 3단 계층입니다. 한 번 머릿속에 그려지면, 이후 모든 CUDA 코드가 자연스럽게 읽힙니다.

  • Thread (스레드) — 가장 작은 실행 단위. 자기 인덱스 threadIdx를 가짐.
  • Block (블록) — 스레드의 묶음. 같은 블록의 스레드는 shared memory로 통신 가능. 한 블록은 항상 같은 SM에서 실행.
  • Grid (그리드) — 블록의 묶음. 한 커널 호출 = 하나의 그리드.

왜 이렇게 나누는가

하드웨어가 GPU마다 SM(Streaming Multiprocessor) 개수가 다르기 때문입니다. 프로그래머는 "이 작업을 1,000,000번 해라"라고 큰 그리드를 던지기만 하면 됩니다. 그러면 스케줄러가 자동으로 그리드의 블록을 사용 가능한 SM에 배치합니다. 같은 코드가 RTX 4060이든 H100이든 그대로 동작하는 핵심 비결이 이것입니다.

인터랙티브 — 슬라이더로 블록과 스레드를 조절해 보세요

색은 워프 단위로 다릅니다. 같은 색의 32개 스레드가 한 워프이며, 동시에 같은 명령을 수행합니다.

코드로 보면

// 1차원 벡터 덧셈 커널
__global__ void vecAdd(float* a, float* b, float* c, int n) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;  // 전역 인덱스 계산
    if (i < n) c[i] = a[i] + b[i];                  // 자기 몫만 처리
}

// 호스트(CPU)에서 호출
int threadsPerBlock = 256;
int blocks = (n + threadsPerBlock - 1) / threadsPerBlock;  // 올림
vecAdd<<<blocks, threadsPerBlock>>>(a, b, c, n);  // 커널 런치

<<<blocks, threadsPerBlock>>> 라는 독특한 문법은 CUDA의 트레이드마크인 커널 실행 구성(execution configuration)입니다. "이 함수를 GPU에서 〈블록 수, 블록당 스레드 수〉 형태로 띄워라"라는 뜻입니다.

실무 팁

블록 크기는 32의 배수로, 보통 128 또는 256으로 잡습니다. 너무 작으면 SM 점유율(occupancy)이 낮아지고, 너무 크면 레지스터 부족으로 성능이 떨어집니다. 의심스럽다면 256부터 시작하세요.

06 · 메모리

메모리 계층 — 빠른 만큼 작다

CUDA 성능 최적화의 90%는 메모리입니다. GPU에는 여러 종류의 메모리가 있고, 각각 속도와 크기, 가시 범위가 다릅니다. 아래는 빠른 순서대로 정렬한 계층입니다 (각 행을 클릭해 자세히 확인하세요).

실전 — Coalesced Access

"인접한 스레드가 인접한 메모리 주소에 접근하면 한 번에 읽힌다"가 CUDA 메모리 최적화의 황금 법칙입니다. 같은 32바이트를 32개 스레드가 각자 한 바이트씩 가져가면 한 번의 트랜잭션으로 끝납니다. 반대로 스레드 0이 0번지, 스레드 1이 1000번지를 읽으면 32번의 트랜잭션이 발생합니다. 

// ✅ 좋은 접근 — coalesced
int i = blockIdx.x * blockDim.x + threadIdx.x;
output[i] = input[i] * 2.0f;

// ❌ 나쁜 접근 — strided
output[i * 32] = input[i * 32] * 2.0f;  // 같은 워프가 듬성듬성 접근
07 · 생태계

CUDA 스택 — 우리는 어느 층에 있는가

AI 엔지니어가 평소 다루는 것은 PyTorch입니다. 하지만 "왜 GPU 드라이버 버전이 안 맞는다"라는 메시지를 보고 당황한 적이 있다면, 그것은 우리가 사실 거대한 스택의 가장 윗층에만 서 있기 때문입니다. 한 번쯤 스택 전체를 그려보면 모든 오류 메시지가 다르게 보입니다.

CUDA Stack — 위에서 아래로
① 프레임워크 — PyTorch, TensorFlow, JAX
Python으로 모델을 정의 · 자동 미분 · 우리가 가장 자주 쓰는 층
② 고수준 라이브러리 — cuDNN, cuBLAS, NCCL, TensorRT
conv, matmul, all-reduce, 추론 최적화 등 자주 쓰는 연산을 NVIDIA가 미리 작성·튜닝
③ CUDA Runtime API + nvcc
우리가 직접 CUDA C++ 코드를 짤 때 보는 층 · cudaMalloc, 커널 런치 등
④ CUDA Driver API
더 저수준 · 가상 메모리, 컨텍스트, 모듈 로딩 · 보통 직접 다루지 않음
⑤ GPU 드라이버 (kernel-mode)
OS와 GPU 하드웨어 사이 · nvidia-smi가 보여주는 그 드라이버
⑥ GPU 하드웨어 (SM, HBM, NVLink, …)
실제 실리콘 — H100, A100, GB10(DGX Spark) 등
실무 직관

"CUDA out of memory"는 ②~③에서, "CUDA driver version is insufficient"는 ⑤에서, "no kernel image is available"은 sm_xx 호환 문제(③↔⑥)에서 발생합니다. 오류 메시지가 어느 층에서 났는지 짐작할 수 있으면 디버깅이 훨씬 빨라집니다.

08 · 컴포넌트

CUDA 주요 라이브러리 — 클릭해서 살펴보기

NVIDIA가 제공하는 라이브러리들은 단순히 "편의 기능"이 아니라, 같은 알고리즘을 아키텍처별로 손튜닝한 결과물입니다. 직접 구현한 행렬 곱이 cuBLAS의 1/10 속도밖에 안 나오는 게 정상입니다. AI 엔지니어가 자주 마주치는 라이브러리들을 정리합니다.

cuBLAS
선형대수 · BLAS
행렬 곱 · 벡터 곱 · GEMM 등 BLAS 표준 연산을 GPU에서.
역할. CPU 세계의 BLAS(LAPACK 등) 대응. PyTorch의 @나 torch.matmul은 결국 cuBLAS의 GEMM을 호출합니다.

변형. cuBLASLt는 더 새로운 API로 mixed-precision과 텐서 코어 활용에 유리합니다.
cuDNN
딥러닝 프리미티브
합성곱 · pooling · BN · RNN 등 딥러닝 빌딩블록 모음.
역할. PyTorch/TensorFlow가 가장 의존하는 라이브러리. 같은 conv2d라도 입력 크기·dtype에 따라 cuDNN이 7가지 알고리즘 중 하나를 자동 선택합니다 (torch.backends.cudnn.benchmark = True가 그 선택을 가속).

버전. CUDA 버전과 cuDNN 버전은 정밀하게 매칭되어야 합니다.
NCCL
멀티 GPU 통신
all-reduce · broadcast 등 분산 학습의 필수 통신 라이브러리.
역할. 여러 GPU 사이 / 여러 노드 사이의 통신을 NVLink·InfiniBand 위에서 최적화. DDP(DistributedDataParallel)의 그래디언트 동기화가 NCCL의 all-reduce입니다.

MLOps 관점. 멀티노드 학습에서 가장 자주 문제가 생기는 지점. 네트워크 토폴로지에 매우 민감합니다.
TensorRT
추론 가속
학습된 모델을 추론용으로 그래프 최적화 · 양자화 · 커널 융합.
역할. PyTorch 모델 → ONNX → TensorRT 엔진으로 변환하면 FP16/INT8 양자화와 layer fusion으로 2–10배 추론 가속이 가능합니다.

최근. TensorRT-LLM은 LLM 전용 가속(KV-cache, paged attention 등).
cuFFT
신호 처리
고속 푸리에 변환 · 신호처리·음성·물리 시뮬에서 사용.
FFT는 컨볼루션을 가속하는 데에도 쓰입니다(cuDNN의 FFT 기반 알고리즘이 내부적으로 호출).
Thrust

C++ STL과 비슷한 인터페이스로 sort / reduce / scan 등.
thrust::sort, thrust::reduce 같은 함수로 손쉽게 GPU 알고리즘 사용 가능. 교육용·프로토타이핑에 좋습니다.

Thrust보다 한 단계 낮은 라이브러리. 커스텀 커널 안에서 효율적인 collective 연산이 필요할 때 사용.

09 · 실전

첫 CUDA 코드 — 손에 익히기

이론을 다 봤으니, 가장 작은 CUDA 프로그램을 직접 읽어 봅시다. 세 가지 예제를 탭으로 준비했습니다. 각 예제의 "왜 이렇게 쓰는가"를 함께 적었습니다. 

#include <cstdio>

__global__ void hello() {
    printf("Hello from thread %d of block %d\n",
           threadIdx.x, blockIdx.x);
}

int main() {
    hello<<<2, 4>>>();          // 2 블록 × 4 스레드 = 총 8개 스레드
    cudaDeviceSynchronize();      // GPU 작업 끝날 때까지 대기
    return 0;
}

읽는 법. __global__은 "GPU에서 실행되되 CPU에서 호출되는 함수" (= 커널)임을 알리는 키워드입니다. <<<2, 4>>>가 그리드 차원 × 블록 차원. cudaDeviceSynchronize가 없으면 CPU가 먼저 끝나버려 출력이 안 보일 수 있습니다.

빌드. nvcc hello.cu -o hello && ./hello 

__global__ void vecAdd(float* a, float* b, float* c, int n) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i < n) c[i] = a[i] + b[i];
}

int main() {
    int n = 1 << 20;                          // 100만 개
    size_t bytes = n * sizeof(float);

    // 1. 호스트에서 메모리 할당 & 초기화
    float *hA = (float*)malloc(bytes), *hB = (float*)malloc(bytes), *hC = (float*)malloc(bytes);
    for (int i=0; i<n; ++i) { hA[i] = 1.0f; hB[i] = 2.0f; }

    // 2. GPU 메모리 할당
    float *dA, *dB, *dC;
    cudaMalloc(&dA, bytes); cudaMalloc(&dB, bytes); cudaMalloc(&dC, bytes);

    // 3. CPU → GPU 전송
    cudaMemcpy(dA, hA, bytes, cudaMemcpyHostToDevice);
    cudaMemcpy(dB, hB, bytes, cudaMemcpyHostToDevice);

    // 4. 커널 실행
    int tpb = 256, blk = (n + tpb - 1) / tpb;
    vecAdd<<<blk, tpb>>>(dA, dB, dC, n);

    // 5. GPU → CPU 복사
    cudaMemcpy(hC, dC, bytes, cudaMemcpyDeviceToHost);

    // 6. 정리
    cudaFree(dA); cudaFree(dB); cudaFree(dC);
    free(hA); free(hB); free(hC);
}

핵심 패턴. 모든 CUDA 프로그램은 사실상 이 여섯 단계의 변주입니다 — ① CPU 할당 → ② GPU 할당 → ③ H2D 복사 → ④ 커널 실행 → ⑤ D2H 복사 → ⑥ 해제. PyTorch에서 .to("cuda")가 ②③을, 연산이 ④를, .cpu()가 ⑤를 자동으로 합니다. 

// 한 블록 안에서 합을 구하는 간단한 reduce (트리 방식)
__global__ void blockReduce(float* in, float* out, int n) {
    __shared__ float s[256];          // 블록 공유 메모리
    int tid = threadIdx.x;
    int i = blockIdx.x * blockDim.x + tid;
    s[tid] = (i < n) ? in[i] : 0.0f;
    __syncthreads();                        // 블록 내 모든 스레드 동기화

    // 트리: 절반씩 짝지어 더함
    for (int s2 = blockDim.x/2; s2 > 0; s2 >>= 1) {
        if (tid < s2) s[tid] += s[tid + s2];
        __syncthreads();
    }
    if (tid == 0) out[blockIdx.x] = s[0];
}

여기서 새로 보이는 것들. __shared__ = 블록 안 모든 스레드가 같이 쓰는 작은 SRAM. __syncthreads() = "모든 스레드가 여기까지 와야 다음으로 가자"는 배리어. 트리 reduce는 O(log N) 깊이로 합을 구하는 고전적인 GPU 패턴입니다. cuBLAS·CUB가 내부적으로 다 이런 패턴을 정교화한 것입니다.

학습 권장 경로

① 위 세 예제를 직접 컴파일·실행 → ② Nsight Compute로 vecAdd 프로파일링 → ③ blockReduce를 warpReduce(__shfl_down_sync)로 개선해 보기. 이 세 단계만 끝내도 CUDA 시야가 확연히 넓어집니다.

10 · 프레임워크

PyTorch와 CUDA — 한 줄 뒤에 무엇이 있나

AI 엔지니어가 매일 쓰는 한 줄을 해부해 봅시다. 

import torch
x = torch.randn(1024, 1024, device="cuda")
y = x @ x.T

이 세 줄이 실제로 호출하는 것:

  1. device="cuda" → CUDA Runtime의 cudaSetDevice 호출, GPU 컨텍스트 확보.
  2. torch.randn → cudaMalloc으로 VRAM 할당 후, PyTorch 내부 RNG 커널을 실행.
  3. x @ x.T → cuBLAS의 cublasSgemm (또는 텐서 코어 활용 시 cublasGemmEx) 호출.
  4. 모든 작업은 CUDA stream에 비동기로 큐잉되며, CPU는 결과를 실제로 읽기 전까지 기다리지 않음.

PyTorch에서 자주 마주치는 CUDA 개념

  • torch.cuda.synchronize() — 비동기 큐의 모든 작업이 끝날 때까지 대기. 벤치마킹 전에 반드시 호출. 안 부르면 측정 시간이 사실상 의미 없음.
  • torch.cuda.Stream — 여러 스트림을 만들어 계산과 데이터 전송을 겹쳐 실행(overlapping).
  • torch.cuda.amp.autocast — FP16/BF16 자동 혼합 정밀도. 텐서 코어를 활용해 2–3배 가속.
  • torch.compile — 그래프 단위 컴파일. 내부적으로 Inductor가 Triton/CUDA 커널을 생성.
  • torch.backends.cudnn.benchmark = True — cuDNN이 입력 크기에 맞는 최적 알고리즘을 1회 탐색 후 캐싱.
자주 만나는 함정

"CPU 시간 측정 결과 GPU가 더 느리다" — 거의 항상 synchronize()를 안 불러서 발생합니다. 커널 런치는 비동기이므로, time.time() 사이의 시간은 런치 큐잉 시간일 뿐 실행 시간이 아닙니다. torch.cuda.Event를 쓰는 게 정석입니다.

CUDA와 다른 ML/GPU 프레임워크

CUDA는 사실상 표준이지만 유일한 선택지는 아닙니다. 각 대안이 어떤 자리에서 어떤 트레이드오프를 갖는지 정리합니다.

한눈에 — 어떤 자리에 어떤 도구를 쓰나

가장 흔한 조합 (95%의 경우)

PyTorch (모델 정의) → cuDNN/cuBLAS (자동 호출) → CUDA Runtime → NVIDIA GPU

최고 성능이 필요할 때 (FlashAttention 같은 커스텀 커널)

PyTorch → Triton 또는 CUDA C++로 커널 작성 → 직접 cuBLASLt/cuDNN 우회

NVIDIA가 아닌 환경

Apple Silicon → PyTorch MPS / MLX. AMD → PyTorch ROCm. TPU → JAX/XLA.

왜 CUDA가 표준이 되었나

기술적 우위만이 아니라 10년 누적된 생태계가 결정적입니다. 모든 논문이 CUDA로 코드를 공개하고, cuDNN의 conv가 가장 빠르며, NCCL이 가장 안정적이고, 모든 클라우드의 GPU 인스턴스가 NVIDIA입니다. 이 관성을 뒤집는 데는 시간이 더 필요할 것입니다.

12 · 정리

정리하며 — 머릿속에 남겨야 할 그림

이 강의에서 다룬 것들을 한 장면으로 압축해 보면:

CUDA = 〈하드웨어 · 컴파일러 · 런타임 · 라이브러리〉의 정렬된 스택

  • 실행 모델은 SIMT — 한 명령, 여러 스레드, 32개 워프 단위.
  • 스레드 계층은 Thread→Block→Grid — 블록 안에서만 통신, 그리드는 자유 배치.
  • 성능의 90%는 메모리 — coalesced access, shared memory 활용이 핵심.
  • 직접 짜기 전에 라이브러리부터 — cuBLAS·cuDNN·NCCL이 거의 항상 더 빠르다.
  • PyTorch 한 줄 뒤에는 이 모든 것이 있다 — 오류 메시지를 다르게 읽을 수 있게 된다.

다음 단계 — 실습으로 확장하기

  1. 본인 환경 점검. nvidia-smi, nvcc --version, python -c "import torch; print(torch.version.cuda)"의 세 값을 일치시켜 두기. DGX Spark처럼 ARM64 + sm_121 환경이면 PyTorch nightly·CUDA 13 호환 휠을 확인하기.
  2. Hello World → 벡터 덧셈 → reduce 순으로 직접 컴파일·실행.
  3. Nsight Compute로 자기 커널의 메모리 throughput·occupancy 보기.
  4. PyTorch의 한 연산 (예: F.scaled_dot_product_attention)에 nsys를 붙여서, 어떤 CUDA 커널들이 어떤 순서로 실행되는지 관찰.
  5. Triton으로 첫 커스텀 커널을 작성해 보기 — Python으로 작성하지만 CUDA 위에서 동작.

추가 학습 자료

  • 공식 문서 — "CUDA C++ Programming Guide"가 가장 정확. 길지만 처음 4장만 읽어도 충분히 도움됨.
  • Programming Massively Parallel Processors (Kirk & Hwu) — 정석 교과서.
  • NVIDIA Deep Learning Performance Guide — 텐서 코어와 정밀도 튜닝을 위한 실용 가이드.
  • Triton tutorials — Python으로 GPU 커널을 짜는 가장 빠른 길.
맺으며

CUDA를 안다는 것은 단순히 한 라이브러리를 안다는 것이 아닙니다. 그것은 현대 AI 인프라가 무엇 위에 서 있는지를 안다는 뜻입니다. 스레드 계층과 메모리 계층, 단 두 그림만 머릿속에 또렷이 남아 있다면, 이 강의는 충분히 제 몫을 다한 것입니다.
