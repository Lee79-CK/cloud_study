01

Proxmox VE란 무엇인가

한 문장으로: Proxmox VE(Virtual Environment)는 물리 서버 한 대 위에 여러 개의 가상 컴퓨터를 만들고 관리하는 무료 오픈소스 운영체제입니다. 웹 브라우저로 접속해서 클릭 몇 번으로 서버를 찍어내는 도구라고 생각하면 됩니다.

아파트 건물로 이해하기

물리 서버 한 대를 넓은 빈 땅이라고 해봅시다. 보통은 그 땅에 집 한 채(서버 1대 = OS 1개)만 짓습니다. 하지만 Proxmox는 그 땅 위에 아파트 건물을 올립니다. 한 건물 안에 여러 세대(가상머신)가 각자의 현관문과 살림살이를 갖고 독립적으로 살죠. Proxmox는 이 건물의 관리사무소입니다. 누가 전기를 얼마나 쓰는지, 빈 집은 없는지, 새 세대를 들일지를 한 곳에서 통제합니다.

왜 가상화를 하는가

서버 한 대에 애플리케이션 하나만 설치하면 CPU와 메모리가 대부분 놀게 됩니다. 가상화는 이 남는 자원을 쪼개서 여러 워크로드에 나눠줍니다. 실무에서 얻는 이점은 다음과 같습니다.

  • 자원 효율 — 유휴 자원을 모아 한 대의 물리 서버에서 10~50개의 워크로드를 돌릴 수 있습니다.
  • 격리 — A 서비스가 죽어도 B 서비스는 멀쩡합니다. 각 VM이 독립된 OS를 갖기 때문입니다.
  • 유연성 — 서버를 켜고 끄고 복제하고 옮기는 일이 몇 초 만에 끝납니다. 물리 서버라면 며칠 걸릴 일이죠.
  • 스냅샷·백업 — 특정 시점으로 되돌리는 "타임머신"을 무료로 씁니다.

Proxmox VE의 3가지 핵심 정체성

키워드의미초보자 해설
Debian 기반안정적인 리눅스 배포판 위에서 동작익숙한 apt 명령어를 그대로 씁니다. 별도 신기술 학습 부담이 적습니다.

웹 기반 관리브라우저로 모든 작업 수행 (8006 포트)에이전트 설치, 별도 관리 서버가 필요 없습니다. 노트북에서 바로 접속합니다.
참고

Proxmox VE는 "타입 1 하이퍼바이저"입니다. 즉 가상화 소프트웨어가 하드웨어 바로 위에서 동작합니다(VMware ESXi와 같은 급). VirtualBox처럼 윈도우/맥 위에 얹어 쓰는 "타입 2"보다 훨씬 빠르고 자원 낭비가 적습니다. 이 차이는 다음 섹션에서 그림으로 설명합니다.

03

가상머신(VM) vs 컨테이너(LXC)

Proxmox의 가장 큰 강점은 두 가지 가상화 방식을 모두 쓸 수 있다는 점입니다. 둘의 차이를 이해하는 것이 Proxmox 학습의 절반입니다. 아래 토글을 눌러 직접 비교해 보세요.

부팅 속도
메모리 사용
격리 수준(보안)
OS 자유도

호텔 vs 셰어하우스

VM(가상머신)은 호텔 객실입니다. 방마다 독립된 화장실·주방·보일러(=각자의 OS 커널)를 갖춰 완벽하게 분리되지만, 그만큼 자재(자원)가 많이 들고 입실(부팅)에 시간이 걸립니다. LXC(컨테이너)는 셰어하우스 방입니다. 화장실·주방(=호스트의 리눅스 커널)을 공유해서 방을 가볍게 추가할 수 있고 즉시 입주(부팅)할 수 있지만, 공용 시설을 쓰는 만큼 격리는 호텔보다 약합니다.

언제 무엇을 쓸까

상황추천이유
Windows 서버를 돌려야 함VM (KVM)컨테이너는 리눅스 커널 공유라 Windows 불가
커널 모듈 수정·특수 드라이버 필요VM (KVM)독립 커널이 있어야 자유롭게 변경 가능
외부 노출 서비스 (강한 보안 격리)VM (KVM)커널까지 완전 분리되어 탈취 시 영향 최소화
가벼운 리눅스 서비스 수십 개LXC메모리·부팅 오버헤드가 거의 없음
홈랩에서 자원 빠듯할 때LXC같은 하드웨어로 훨씬 많이 돌릴 수 있음
GPU 패스스루로 AI 추론 서버상황에 따라전용 GPU는 VM, 공유 분할은 LXC가 유리할 수 있음
헷갈리기 쉬운 점

여기서 말하는 LXC는 시스템 컨테이너로, 그 안에서 완전한 리눅스 환경이 돌아갑니다(SSH 접속, systemd 등). 반면 Docker는 애플리케이션 컨테이너로 하나의 프로세스를 패키징합니다. 실무에서는 흔히 Proxmox에 VM 하나를 만들고 그 안에서 Docker/Kubernetes를 돌립니다. LXC와 Docker는 경쟁 관계가 아니라 층위가 다릅니다.

한 줄 요약

  • VMware vSphere — 업계 표준이자 기능 최강이지만 Broadcom 인수 이후 라이선스 비용이 크게 올라 탈(脫)VMware 흐름이 강해졌습니다. Proxmox는 그 대표적 대안입니다.
  • Hyper-V — Windows Server에 내장. 마이크로소프트 생태계(Active Directory 등) 중심 조직에 자연스럽습니다.
  • XCP-ng — Citrix Hypervisor(Xen)의 오픈소스 후예. Proxmox의 직접 경쟁자지만 컨테이너 내장이 없습니다.
  • OpenStack — 가상화 도구가 아니라 "클라우드를 통째로 만드는 프레임워크"입니다. AWS 같은 셀프서비스 클라우드를 직접 구축할 때 쓰며, 규모와 복잡도가 차원이 다릅니다.
  • Proxmox VE — 무료·간단·VM과 컨테이너 통합. 홈랩과 중견 규모 사내 인프라에서 가성비가 압도적입니다.
실무 팁

최근 수년간 VMware 라이선스 정책 변화로 많은 기업이 마이그레이션 대안을 찾고 있습니다. Proxmox VE는 ESXi 가상디스크(VMDK) 가져오기 기능을 제공해 이전 부담을 낮춰주며, 그 결과 채택 사례가 빠르게 늘고 있습니다. 클라우드 엔지니어 이력서에 적어두면 좋은 키워드입니다.

05

설치와 웹 UI 둘러보기

설치는 USB로 부팅해 화면을 따라가면 끝납니다. 별도의 리눅스 지식 없이 15분이면 첫 화면을 만날 수 있습니다.

설치 4단계

  1. ISO 다운로드 — 공식 사이트에서 Proxmox VE 설치 ISO를 받습니다.
  2. 부팅 USB 제작 — Rufus(Windows)나 dd(리눅스/맥)로 USB에 굽습니다.
  3. 설치 — 서버를 USB로 부팅 → 디스크 선택, 비밀번호, 고정 IP 입력 → 자동 설치.
  4. 웹 접속 — 같은 네트워크의 PC 브라우저에서 https://서버IP:8006 으로 접속.
자주 막히는 부분

주소는 http가 아니라 https, 포트는 8006입니다. 자체 서명 인증서라 브라우저가 "안전하지 않음" 경고를 띄우는데, 내 서버이므로 "고급 → 계속 진행"을 누르면 됩니다. 로그인 시 Realm은 Linux PAM, 사용자는 root를 선택합니다.

웹 UI의 구조

화면은 크게 세 부분입니다. 왼쪽 트리에서 대상을 고르고, 위쪽 메뉴에서 작업을 선택하면, 가운데에 결과가 표시됩니다.

영역위치역할
Resource Tree왼쪽데이터센터 → 노드 → VM/컨테이너 계층 구조. 클릭으로 대상 선택
콘텐츠 패널가운데선택한 대상의 요약·설정·콘솔·백업 탭이 표시됨
Task Log아래실행한 모든 작업의 성공/실패 로그가 실시간으로 쌓임
알아두면 편한 것

VM과 컨테이너는 각각 VMID라는 고유 숫자를 가집니다(보통 100부터 시작). CLI 명령에서 이 숫자로 대상을 지정하므로 기억해두면 좋습니다. 예: 100번 VM 시작 → qm start 100.

06

VM·컨테이너 직접 만들기

웹 UI로도 만들 수 있지만, 실무 자동화에서는 CLI를 씁니다. 아래는 가장 자주 쓰는 명령들입니다. 탭을 눌러 비교해 보세요.

create-vm.sh
# 1) ISO 업로드 후, 100번 ID로 새 VM 생성
root@pve:~# qm create 100 --name web-server --memory 2048 --cores 2 \
  --net0 virtio,bridge=vmbr0

# 2) 디스크 32GB 추가 (local-lvm 스토리지에)
root@pve:~# qm set 100 --scsi0 local-lvm:32

# 3) 설치용 ISO를 CD 드라이브에 연결
root@pve:~# qm set 100 --ide2 local:iso/ubuntu-24.04.iso,media=cdrom

# 4) VM 시작 → 웹 콘솔에서 OS 설치 진행
root@pve:~# qm start 100
# vmbr0 은 기본 가상 네트워크 브리지(스위치)입니다
create-ct.sh
# 컨테이너는 ISO 설치 없이 "템플릿"에서 즉시 생성됩니다
# 1) 사용 가능한 OS 템플릿 목록 갱신
root@pve:~# pveam update
root@pve:~# pveam available --section system

# 2) Debian 12 템플릿 내려받기
root@pve:~# pveam download local debian-12-standard_*.tar.zst

# 3) 101번 컨테이너 생성 (메모리 512MB면 충분!)
root@pve:~# pct create 101 local:vztmpl/debian-12-standard_*.tar.zst \
  --hostname dns-server --memory 512 --cores 1 \
  --net0 name=eth0,bridge=vmbr0,ip=dhcp --rootfs local-lvm:8

# 4) 즉시 시작 (1초 컷) 후 내부 진입
root@pve:~# pct start 101 && pct enter 101

VM은 OS 설치에 몇 분이 걸리지만, 컨테이너는 템플릿에서 1초 만에 완성됩니다. 이것이 LXC의 매력입니다.

clone.sh
# 잘 만든 VM을 "골든 이미지" 템플릿으로 굳히기
root@pve:~# qm template 100

# 템플릿에서 새 VM을 순식간에 찍어내기 (linked clone)
root@pve:~# qm clone 100 200 --name web-02
root@pve:~# qm clone 100 201 --name web-03

# Cloud-Init 으로 IP·SSH키·계정을 자동 주입하면
# 완전 무인 프로비저닝이 가능 → IaC의 시작점
root@pve:~# qm set 200 --ipconfig0 ip=192.168.0.200/24,gw=192.168.0.1

템플릿 + 복제 + Cloud-Init 조합은 Terraform·Ansible 같은 IaC 도구와 연결되어 "코드로 인프라 찍어내기"의 토대가 됩니다.

07

스토리지와 네트워크

VM은 결국 "어디에 저장되고(스토리지) 어떻게 통신하느냐(네트워크)"로 귀결됩니다. 초보가 가장 헷갈리는 두 영역을 단순하게 정리합니다.

스토리지 — 어디에 디스크를 둘까

Proxmox는 여러 종류의 저장소를 지원합니다. 각각 용도가 다릅니다.

스토리지특징추천 용도
local (Directory)그냥 디렉터리. ISO·백업·템플릿 보관입문용, 단일 서버
LVM-Thin블록 단위, 빠르고 스냅샷 지원VM 디스크 기본값
ZFS스냅샷·압축·체크섬·자가복구데이터 안정성 중시
Ceph여러 노드 디스크를 묶은 분산 스토리지클러스터·HA 환경
NFS / iSCSI외부 NAS·SAN 연결기존 스토리지 인프라 활용
선택 가이드

혼자 공부하는 홈랩이라면 처음엔 기본값(local-lvm)으로 충분합니다. 데이터 무결성이 중요하면 ZFS, 여러 서버로 무중단 운영을 하려면 Ceph를 공부하세요. Ceph는 보통 노드 3대 이상부터 의미가 있습니다.

네트워크 — 가상 스위치 이해하기

Proxmox는 리눅스 브리지를 가상 네트워크 스위치로 씁니다. 기본 이름은 vmbr0입니다.

건물 안의 멀티탭

vmbr0는 건물(서버) 안에 설치된 네트워크 멀티탭(스위치)입니다. 물리 랜선(서버의 NIC)이 이 멀티탭에 꽂혀 있고, 각 VM의 가상 랜카드도 같은 멀티탭에 연결됩니다. 그래서 VM들은 서로, 그리고 바깥 네트워크와 통신할 수 있습니다. 격리가 필요하면 멀티탭을 여러 개(vmbr1, vmbr2…) 만들거나 VLAN으로 칸막이를 칩니다.

  • 브리지(Bridge) — VM이 물리 네트워크에 직접 붙은 것처럼 동작. 가장 흔함.
  • VLAN — 하나의 물리 회선을 논리적으로 분리해 트래픽을 격리.
  • SDN(Software-Defined Networking) — 최신 버전에서 제공. 코드로 가상 네트워크를 정의하고 클러스터 전체에 적용.
08

클러스터 · 고가용성 · 백업

서버 한 대를 넘어서는 순간 진짜 클라우드 엔지니어링이 시작됩니다. Proxmox가 "장난감"이 아니라 실무 도구인 이유가 여기 있습니다.

클러스터(Cluster) — 여러 서버를 하나처럼

Proxmox 노드 여러 대를 묶으면 하나의 웹 화면에서 모두 관리됩니다. 설정은 자동으로 전 노드에 동기화됩니다.

cluster.sh
# 첫 번째 노드에서 클러스터 생성
root@pve1:~# pvecm create my-cluster

# 두 번째 노드를 클러스터에 합류시키기
root@pve2:~# pvecm add 192.168.0.11

# 클러스터 상태 확인
root@pve1:~# pvecm status

라이브 마이그레이션 — 무중단 이사

실행 중인 VM을 끄지 않고 다른 노드로 옮깁니다. 서버 점검·하드웨어 교체 시 서비스를 멈추지 않아도 됩니다. 마치 달리는 기차에서 승객을 옆 기차로 옮기는 것과 같습니다.

고가용성(HA) — 자동 부활

HA를 설정한 VM은 해당 노드가 다운되면 살아있는 다른 노드에서 자동으로 재시작됩니다. 새벽에 서버가 죽어도 사람이 깨어날 필요 없이 서비스가 복구됩니다.

전제 조건

HA가 동작하려면 ① 노드 최소 3대(과반수 투표로 "진짜 죽었는지" 판단), ② 공유 스토리지(Ceph/NFS 등 — VM 디스크가 모든 노드에서 보여야 함)가 필요합니다. 2대만으로는 "누가 진짜 살았는지" 판단이 어려워(split-brain) 권장되지 않습니다.

백업과 Proxmox Backup Server(PBS)

Proxmox는 강력한 백업을 무료로 제공합니다. 별도 제품인 PBS를 함께 쓰면 한 단계 위로 갑니다.

  • 스냅샷 — 특정 시점 상태를 즉시 저장/복원 (수정 전 안전장치).
  • 스케줄 백업 — "매일 새벽 3시 전체 백업" 같은 자동화.
  • 증분·중복제거 백업(PBS) — 바뀐 부분만 저장해 용량을 획기적으로 절약하고, 백업 속도를 높입니다.
backup.sh
# 100번 VM을 지금 즉시 백업 (압축, 스냅샷 모드)
root@pve:~# vzdump 100 --mode snapshot --compress zstd --storage backup-nfs
09

응용 분야

Proxmox는 어디에 쓰일까요? 개인 학습실부터 기업 인프라, 그리고 AI 인프라까지 활용 폭이 넓습니다.

🏠

홈랩 / 자기 학습

중고 PC 한 대로 K8s 클러스터, NAS, DNS, 미디어 서버를 동시에 운영. 클라우드 엔지니어 실습 환경으로 최적.

🏢

사내 프라이빗 클라우드

VMware 라이선스 비용 절감 대안. 중견 기업의 사내 서버 가상화 통합 플랫폼으로 채택 증가.

🧪

개발 · 테스트 환경

스냅샷으로 즉시 되돌리는 샌드박스. CI 파이프라인용 일회성 VM을 코드로 찍어내기.

🌐

엣지 컴퓨팅

지점·공장·매장에 소형 서버를 배치해 현장 데이터를 처리. 중앙 클러스터와 연계.

🤖

AI · MLOps 인프라

GPU 패스스루로 VM에 그래픽카드를 통째로 할당해 모델 학습·추론 서버 구축. vLLM·TensorRT-LLM 서빙 노드를 VM으로 격리 운영.

🛡️

보안 · 망분리

업무망/인터넷망을 VLAN+VM으로 분리. 침해 시 영향 범위를 가상머신 단위로 한정.

클라우드 엔지니어에게

Proxmox 위에 VM을 여러 개 만들고 그 안에 Kubernetes 클러스터를 구성하는 것은 가장 가성비 좋은 K8s/LLMOps 실습 방법입니다. 베어메탈 GPU 노드는 GPU 패스스루로 추론 서버에 할당하고, 컨트롤 플레인은 가벼운 VM으로 두는 식의 하이브리드 구성이 실무에서 흔합니다. Terraform Proxmox Provider를 붙이면 인프라를 코드로 선언적으로 관리할 수 있습니다.

실전 CLI 치트시트

웹 UI는 편하지만, 자동화와 트러블슈팅은 결국 명령어입니다. 외우지 말고 필요할 때 꺼내 쓰세요. 핵심은 단 세 개의 명령군입니다.

명령군담당외우는 법
qmVM(KVM) 제어qemu manage
pct컨테이너(LXC) 제어proxmox container
pve*노드/클러스터/스토리지pvecm · pveam · pvesm
cheatsheet.sh
# ── VM 다루기 (qm) ──────────────────────
qm list                  # 모든 VM 목록
qm start 100             # 시작
qm shutdown 100          # 정상 종료 (게스트에 신호)
qm stop 100              # 강제 종료 (전원 차단)
qm config 100            # 설정 확인
qm migrate 100 pve2 --online  # 무중단 마이그레이션

# ── 컨테이너 다루기 (pct) ───────────────
pct list                 # 컨테이너 목록
pct start 101            # 시작
pct enter 101            # 컨테이너 내부 쉘 진입
pct exec 101 -- apt update  # 내부에서 명령 실행

# ── 노드 · 클러스터 · 스토리지 ──────────
pvecm status            # 클러스터 상태
pvesm status            # 스토리지 상태/용량
pveam available         # 받을 수 있는 OS 템플릿
pveversion             # Proxmox 버전 확인
기억할 단 하나

qm = 가상머신, pct = 컨테이너, pve로 시작 = 시스템 전체. 이 세 가지만 머리에 넣으면 나머지는 --help로 그때그때 찾으면 됩니다.
