CHAPTER 01 — MOTIVATION

왜 RAG에는 "다른 종류의 DB"가 필요한가?

LLM은 모르는 걸 그럴듯하게 지어내는 경향이 있습니다(이른바 환각, hallucination). 해결책은 단순합니다 — 답변에 필요한 문서를 먼저 찾아서 모델에게 같이 보여주는 것. 문제는 "찾기"의 정의입니다.

전통적 검색의 한계

RDB의 WHERE name = '홍길동' 같은 정확 일치 검색이나 전문 검색 엔진의 키워드 매칭(Elasticsearch의 BM25)은 표현이 다른 같은 의미를 잡아내지 못합니다.

예시: 다음 두 질문을 같은 답변으로 연결할 수 있는가?

질문 A   "쿠버네티스에서 GPU 노드를 어떻게 추가하나요?"

질문 B   "K8s 클러스터에 NVIDIA 워커를 붙이는 방법이 궁금해"

두 질문은 단어가 거의 겹치지 않습니다. 키워드 검색은 실패합니다. 하지만 의미는 동일합니다. 사람은 알지만 — DB는 의미를 모릅니다. 의미를 좌표로 바꿔서 저장하는 DB가 필요한 이유입니다.

핵심 아이디어

문장을 고차원 공간의 점(벡터)으로 바꾸면, 의미가 비슷한 문장은 공간상에서도 가까이 모입니다. 그러면 검색은 "가장 가까운 점 찾기" 문제가 됩니다. 이것이 벡터DB의 출발점입니다.

CHAPTER 02 — EMBEDDING

임베딩 — 의미를 좌표로 바꾸는 마법

임베딩(embedding)이란 텍스트·이미지·오디오 같은 비정형 데이터를 고정된 길이의 숫자 배열(벡터)로 변환하는 것입니다. 이 변환은 임베딩 모델이 담당합니다 (예: OpenAI text-embedding-3-small, BAAI bge-m3, Sentence-Transformers).

1차원으로 단순화한 예시

이해를 돕기 위해 차원을 극단적으로 줄여봅시다. "차량 관련성"이라는 단 하나의 축에 단어들을 놓는다고 상상해보세요: 

의미축 →   0 ────── 0.3 ────── 0.6 ────── 0.9 ────── 1.0
                                                          
       "사과"   "고양이"   "자전거"   "버스"   "트럭"
        0.05     0.12      0.65      0.88    0.95

"버스(0.88)"와 "트럭(0.95)"은 가깝고, "사과(0.05)"는 멀리 떨어져 있습니다. 실제 임베딩 모델은 이런 의미축을 수백~수천 개 동시에 만듭니다. OpenAI의 text-embedding-3-small은 1536차원, bge-m3는 1024차원입니다.

차원이 많으면 뭐가 좋은가?

표현할 수 있는 미묘한 의미 차이가 풍부해집니다. "강아지 vs 고양이", "차가운 vs 시원한", "기쁜 vs 즐거운" 같은 세밀한 구분이 가능해지죠.

그럼 무한정 늘리면 되나?

아닙니다. 차원이 늘수록 저장 비용·검색 비용·메모리가 선형으로 증가하고, "차원의 저주(curse of dimensionality)"로 거리 함수가 변별력을 잃기도 합니다. 보통 384~3072 사이에서 균형을 잡습니다.

실전 팁

한국어 RAG에서는 bge-m3, multilingual-e5-large, jhgan/ko-sroberta-multitask 같은 다국어/한국어 특화 모델이 영어 위주 모델보다 검색 품질이 월등히 높습니다. 임베딩 모델 선택은 RAG 품질의 70%를 결정합니다.

CHAPTER 03 — COMPARISON

RDB vs 벡터DB — 무엇이 어떻게 다른가

둘 다 "데이터를 저장하고 검색한다"는 점은 같지만, 철학과 자료구조 자체가 다릅니다. 같은 일을 하는 다른 도구가 아니라, 다른 일을 하는 보완재로 보는 것이 정확합니다.

항목 관계형 DB (RDB) 벡터 DB (Vector DB)
데이터 모델 행과 열의 테이블, 정형 스키마 벡터(숫자 배열) + 메타데이터, 비정형 콘텐츠
저장 단위 레코드 (예: {id, name, age}) 벡터 + 페이로드 (예: {id, [0.12, -0.4, …], text, source})
쿼리 방식 SQL — 정확 일치, 범위, JOIN
WHERE price < 100		 유사도 검색 — "이 벡터에 가까운 것 top-K"
search(query_vec, k=10)
인덱스 B-Tree, Hash — 정확한 위치 찾기 HNSW, IVF, PQ — 근사적으로 가까운 이웃 찾기
일관성 모델 ACID, 강한 일관성 최종적 일관성, 정확도-속도 트레이드오프
스케일링 수직 확장 + 읽기 복제본 위주 샤딩 기반 수평 확장이 기본 설계
대표 제품 PostgreSQL, MySQL, Oracle Pinecone, Milvus, Qdrant, Weaviate, Chroma
적합한 질문 "지난 분기 매출이 1억 이상인 고객 목록은?" "이 사용자 질문과 가장 의미가 비슷한 문서는?"

결정적 차이: "Exact" vs "Approximate"

RDB의 SELECT는 정확히 일치하는 행을 모두 찾아냅니다 — 1개의 누락도 없습니다. 벡터DB의 검색은 근사 최근접 이웃 검색(ANN, Approximate Nearest Neighbor)입니다. 100만 개 중 진짜로 가장 가까운 1등이 살짝 누락되어도 괜찮다고 받아들이는 대신, 검색 속도를 수천 배 빠르게 얻습니다.

흔한 오해

"PostgreSQL에 pgvector 확장을 설치하면 벡터DB가 된다"는 말은 반은 맞고 반은 틀립니다. pgvector는 훌륭한 도입 옵션이지만, 수억 벡터·고QPS 환경에서는 전용 벡터DB(Milvus, Qdrant 등)의 분산 ANN 인덱스·GPU 가속·메모리 매핑 최적화를 따라잡기 어렵습니다. "규모"가 선택의 분기점입니다.

혼동하기 쉬운 개념 정리

RDB의 "검색"

이미 아는 정확한 값(혹은 범위)으로 행을 찾는 행위. "id가 42인 사용자", "price가 1000–2000 사이인 상품". 결정론적·완전 일치.

벡터DB의 "검색"

주어진 벡터와 가장 비슷한 벡터들을 거리 순으로 정렬해 K개 반환. "이 질문과 의미가 가까운 문서 5개". 확률적·유사도 기반.

범위 −1 ~ 1 높을수록 유사
L2 / EUCLIDEAN

실제 직선거리

두 점 사이의 자(尺)로 잰 거리. 직관적이지만 크기에 민감해 정규화되지 않은 벡터에는 부적합. 이미지·임베딩이 정규화된 경우 코사인과 동치.

범위 0 ~ ∞ 낮을수록 유사
DOT PRODUCT

방향 × 크기

범위 −∞ ~ ∞ 높을수록 유사
CHAPTER 05 — INDEXING

인덱싱 알고리즘 — 1억 벡터를 1초 안에 검색하는 비결

저장된 모든 벡터와 일일이 거리를 계산하면(이를 Brute Force, Flat이라 부름), 벡터가 100만 개만 돼도 매 쿼리에 수 초가 걸립니다. ANN(Approximate Nearest Neighbor) 인덱스는 검색 공간을 미리 잘라두어 속도를 비약적으로 높입니다.

FLAT (Brute Force)

모든 벡터와 비교

가장 정확하지만 가장 느림. 데이터셋이 작거나(<10만) 정확도가 100% 필요할 때만 사용.

정확도 100% 속도 느림
IVF (Inverted File)

K-Means로 동네 나누기

전체 벡터를 K개 클러스터로 분할. 쿼리는 가장 가까운 몇 개 클러스터만 탐색. "도시 전체가 아니라 가까운 동네 3곳만 본다"는 발상.

정확도 높음 속도 빠름
HNSW

계층적 작은 세계 그래프

고속도로(상위층)→국도(중간층)→골목(하위층) 식의 다층 그래프를 만들어 상위층에서 빠르게 좁힌 뒤 하위층에서 정밀 탐색. 현재 가장 인기 있는 방식.

정확도 매우 높음 속도 매우 빠름
PQ (Product Quantization)

벡터 압축

벡터를 작은 조각으로 쪼개 각 조각을 코드북에 양자화 — 메모리를 1/10 이상 절약. IVF·HNSW와 결합해 IVF-PQ, HNSW-PQ 형태로 자주 쓰임.

정확도 중간 메모리 매우 적음
LSH (Locality Sensitive Hashing)

비슷한 것끼리 같은 버킷

비슷한 벡터가 같은 해시값을 갖도록 설계된 해시 함수. 구현이 간단하고 분산 친화적이지만 정확도는 HNSW보다 낮음. 학습 자료에서 자주 등장.

정확도 중간 구현 간단
DiskANN / SPANN

SSD 활용 대규모 인덱스

전체 인덱스가 RAM에 안 들어가는 수십억 벡터급 환경용. 그래프 일부는 디스크에 두고 필요한 부분만 메모리로 가져와 검색.

규모 1B+ 벡터 비용 저렴

실전 권장 조합

~100만 벡터: HNSW (메모리 충분하면 무조건)
100만~1억 벡터: IVF-PQ 또는 HNSW-PQ
1억 벡터 이상: 샤딩 + DiskANN/SPANN, 혹은 Milvus/Vespa 같은 분산 엔진
초기 PoC: pgvector + HNSW로 시작 → 규모 커지면 전용 DB로 이전

CHAPTER 06 — COMPONENTS

벡터DB의 주요 컴포넌트

벡터DB는 단순히 "벡터를 저장하는 통"이 아닙니다. 프로덕션에서 RAG를 운영하려면 아래 컴포넌트들이 유기적으로 동작해야 합니다.

1. Collection (컬렉션)

RDB의 테이블과 비슷한 개념. 같은 임베딩 차원·같은 거리 함수를 공유하는 벡터들의 묶음. 예: documents_kr 컬렉션과 code_snippets 컬렉션은 별개로 관리합니다. 멀티 테넌시를 위해 namespace나 partition 단위로 더 잘게 나누기도 합니다.

2. Embedding (임베딩 벡터)

실제로 저장되는 핵심 데이터. 차원 수는 컬렉션 생성 시 고정되며, 나중에 변경할 수 없습니다. 임베딩 모델을 교체하려면 컬렉션을 다시 만들고 전체 재인덱싱이 필요합니다.

3. Payload / Metadata (메타데이터)

벡터와 함께 저장되는 부가 정보. JSON 형태로 자유롭게 넣을 수 있습니다. 검색 후 LLM에게 전달할 원문 텍스트, 문서 출처, 작성일, 권한 정보 등이 들어갑니다. 

{
  "id": "doc-2026-0142",
  "vector": [0.012, -0.456, 0.789, /* ... 1533 more */],
  "payload": {
    "text": "쿠버네티스에서 NVIDIA GPU를 사용하려면 ...",
    "source": "k8s-guide-v1.4.md",
    "chunk_index": 7,
    "created_at": "2026-04-12",
    "team": "platform",
    "acl": ["engineer", "admin"]
  }
}

4. Index (인덱스)

앞 장에서 다룬 HNSW·IVF 등의 자료구조. 컬렉션마다 하나의 인덱스를 갖는 것이 일반적이며, 인덱스 빌드는 비용이 크기 때문에 보통 비동기로 진행됩니다.

5. Distance Function (거리 함수)

컬렉션 생성 시 결정 (cosine / L2 / dot product 중 택1). 임베딩 모델이 정규화된 벡터를 출력한다면 셋 중 어느 것을 써도 동등하지만, 모델 문서를 반드시 확인해야 합니다.

6. Filter (메타데이터 필터)

벡터 유사도 검색에 RDB의 WHERE와 비슷한 필터 조건을 결합하는 기능. "최근 3개월 이내 문서 중에서, platform 팀 자료만, 의미가 가장 비슷한 5개"처럼 실무에서 가장 자주 쓰입니다. 

client.search(
    collection_name="documents_kr",
    query_vector=query_emb,
    limit=5,
    query_filter=Filter(
        must=[
            FieldCondition(key="team", match=MatchValue(value="platform")),
            FieldCondition(key="created_at", range=Range(gte="2026-02-01")),
        ]
    )
)

7. Hybrid Search (하이브리드 검색)

벡터 검색(의미 기반) + 키워드 검색(BM25, 정확 일치)을 가중합으로 결합. 고유명사·제품코드·버전번호 같은 임베딩이 약한 부분을 키워드가 보강합니다. 실무 RAG에서 정확도를 끌어올리는 가장 효과적인 기법 중 하나입니다.

8. Reranker (재정렬)

벡터DB는 보통 top-K를 빠르게 가져오지만 그 순서가 완벽하진 않습니다. Cross-Encoder(예: bge-reranker-v2-m3)로 상위 결과를 다시 정밀 채점해 최종 LLM 입력 품질을 끌어올립니다. 검색 단계 → 재정렬 단계의 2-stage가 정석.

CHAPTER 07 — RAG PIPELINE

RAG에서 벡터DB는 어떻게 동작하는가

RAG는 크게 두 단계로 나뉩니다 — 인덱싱(미리 준비)과 질의(런타임). 벡터DB는 양쪽 모두의 중심에 자리합니다.

인덱싱 파이프라인 (Offline)

STEP 01
Load
PDF, 마크다운, DB, 웹페이지 등 원본 문서 수집
STEP 02
Chunk
긴 문서를 500~1000 토큰 단위로 분할(오버랩 포함)
STEP 03
Embed
각 청크를 임베딩 모델로 벡터화
STEP 04
Store
벡터 + 원문 + 메타데이터를 벡터DB에 적재
STEP 05
Index
HNSW 등 ANN 인덱스 빌드

질의 파이프라인 (Online)

STEP 01
Query
사용자 질문 수신
STEP 02
Embed Query
동일 임베딩 모델로 질문을 벡터화
STEP 03
Search
벡터DB에서 top-K + 필터 검색
STEP 04
Rerank
Cross-Encoder로 정밀 재정렬
STEP 05
Generate
상위 문서를 프롬프트에 주입 → LLM 호출

중요 — 임베딩 모델은 양쪽이 같아야 한다

저장할 때 쓴 임베딩 모델과 검색할 때 쓰는 임베딩 모델은 반드시 동일해야 합니다. 서로 다른 모델의 벡터 공간은 좌표계 자체가 달라 거리 계산이 무의미해집니다. 모델을 바꾸려면 전체 컬렉션을 다시 임베딩해야 하므로, 초기 모델 선택에 시간을 충분히 투자하세요.

청킹(Chunking)의 중요성

청크가 너무 작으면 문맥이 잘리고, 너무 크면 한 벡터가 여러 주제를 평균화해 검색 정확도가 떨어집니다. 보통 500~1000 토큰, 50~150 토큰의 오버랩이 시작점입니다. 문서 종류(코드, 산문, 표)에 따라 다른 청킹 전략(Semantic Chunking, Recursive Splitter)을 적용합니다.

CHAPTER 08 — SOLUTIONS

주요 벡터DB 솔루션 비교

2026년 현재 활발히 쓰이는 벡터DB들을 정리합니다. Kubernetes 친화성, 셀프호스팅 여부, GPU 가속 지원 등이 선택의 핵심 기준입니다.

제품 특징 적합한 상황
Milvus
OSSK8sGPU		 Zilliz가 개발한 분산형 OSS. K8s Operator 제공, GPU 인덱스(RAFT) 지원, 수십억 벡터급 운영. 대규모·온프레미스·K8s 환경, GPU 가속이 필요한 LLMOps
Qdrant
OSSK8s		 Rust 기반 고성능. 필터링 + 벡터검색 결합이 강점. API가 깔끔하고 단일 바이너리. 중규모 RAG, 강력한 메타데이터 필터링이 필요한 경우
Weaviate
OSSK8s		 GraphQL 인터페이스, 모듈식 임베더 통합, 하이브리드 검색 내장. 스키마가 중요한 도메인, 멀티모달 RAG
Pinecone
SaaS		 완전 관리형 서비스. 인프라 고민 없이 즉시 사용. 서버리스 모드 제공. 운영 인력이 부족한 팀, 빠른 PoC, 초기 스타트업
Chroma
OSS		 임베디드 사용이 쉬워 LangChain 튜토리얼의 기본값. 단일 노드 위주. 로컬 개발, 프로토타입, 소규모 RAG
pgvector
OSS		 PostgreSQL 확장. 기존 DB에 그대로 붙임 → 트랜잭션·JOIN·벡터검색을 하나의 SQL로. 이미 Postgres 쓰는 팀, 수천만 벡터까지의 중규모
Elasticsearch / OpenSearch
OSSK8s		 전통적인 텍스트 검색 엔진에 벡터 검색을 결합. 하이브리드 검색에 강점. 기존 로그/검색 인프라를 RAG로 확장하는 경우
Vespa
OSSK8s		 Yahoo 출신의 분산 서빙 엔진. 텐서 표현·랭킹·하이브리드 검색이 일급 시민. 검색+추천+RAG가 한 엔진에 필요한 대규모 서비스

K8s 기반 LLMOps 환경에서의 선택 가이드

이종 아키텍처(ARM64 + x86_64) 클러스터를 운영한다면 멀티 아키텍처 컨테이너 이미지 제공 여부가 중요합니다. Qdrant·Milvus·Weaviate 모두 공식 멀티아치 이미지를 제공하며, docker buildx로 직접 빌드가 가능합니다. GPU 가속이 필요하면 Milvus의 GPU 인덱스 또는 NVIDIA cuVS를 활용한 커스텀 빌드를 고려하세요.

CHAPTER 09 — CODE

실전 예제 — 최소 RAG를 30줄로

지금까지 배운 개념을 코드로 묶어봅시다. Qdrant + Sentence-Transformers 기반, 로컬에서 바로 돌릴 수 있는 최소 예제입니다.

1. 벡터DB 컬렉션 생성

from qdrant_client import QdrantClient
from qdrant_client.models import Distance, VectorParams, PointStruct
from sentence_transformers import SentenceTransformer

# 1) 임베딩 모델 로딩 (한국어 다국어 모델)
embedder = SentenceTransformer("BAAI/bge-m3")

# 2) 벡터DB 클라이언트 (Qdrant 로컬)
client = QdrantClient(host="localhost", port=6333)

# 3) 컬렉션 생성 — 차원과 거리함수는 한번 정하면 못 바꿈
client.recreate_collection(
    collection_name="docs_kr",
    vectors_config=VectorParams(size=1024, distance=Distance.COSINE),
)

2. 문서 임베딩 후 적재

docs = [
    "쿠버네티스에서 GPU 노드를 추가하려면 NVIDIA Device Plugin을 설치해야 한다.",
    "Helm Chart는 values.yaml로 환경별 설정을 분리한다.",
    "Argo Workflows는 DAG 구조로 ML 파이프라인을 정의한다.",
]

# 한 번에 배치 임베딩이 효율적
vectors = embedder.encode(docs, normalize_embeddings=True)

points = [
    PointStruct(id=i, vector=v.tolist(), payload={"text": t})
    for i, (v, t) in enumerate(zip(vectors, docs))
]
client.upsert(collection_name="docs_kr", points=points)

3. 유사도 검색 → LLM 호출

question = "K8s에서 GPU 어떻게 붙여요?"
q_vec = embedder.encode(question, normalize_embeddings=True).tolist()

hits = client.search(
    collection_name="docs_kr",
    query_vector=q_vec,
    limit=3,
)

context = "\n\n".join([h.payload["text"] for h in hits])

prompt = f"""아래 문서를 참고해서 질문에 답변하세요.

[문서]
{context}

[질문]
{question}

[답변]"""

# LLM (OpenAI, Anthropic, vLLM 등) 호출 → 답변 생성
answer = llm.complete(prompt)
print(answer)

이 코드의 의미

단 30여 줄이지만 RAG의 본질이 모두 들어있습니다 — 임베딩 → 저장 → 검색 → 컨텍스트 주입 → 생성. 실전에서는 여기에 청킹·재정렬·필터·캐싱·관측성이 더해지며, 그 모든 것의 중심에 벡터DB가 있습니다.
