SECTION 01

Redis란 무엇인가

Redis(REmote DIctionary Server)는 데이터를 디스크가 아닌 메모리(RAM)에 저장하는 키-값(Key-Value) 데이터 저장소입니다. "세상에서 가장 빠른 데이터베이스"라는 별명이 있을 만큼 압도적인 속도를 자랑하며, 캐시·세션 저장소·메시지 브로커·실시간 랭킹 등 현대 서비스의 거의 모든 곳에서 쓰입니다.

왜 이렇게 빠를까?

일반적인 데이터베이스(MySQL, PostgreSQL 등)는 데이터를 디스크에 저장합니다. 디스크는 용량은 크지만 접근 속도가 느립니다. Redis는 모든 데이터를 메모리에 올려두기 때문에, 디스크 I/O 없이 나노초 단위로 데이터에 접근합니다.

접근 속도 비교 (대략적인 감각)
저장 매체접근 시간사람 시간으로 환산하면
CPU L1 캐시~1 ns1초
메모리 (Redis)~100 ns약 2분
SSD 랜덤 읽기~100 µs약 1일
HDD 탐색~10 ms약 4개월
같은 데이터센터 내 네트워크 왕복~500 µs약 5일

속도의 비결은 메모리만이 아닙니다. Redis의 설계 철학 세 가지가 함께 만들어낸 결과입니다.

인메모리 저장

모든 데이터가 RAM에 상주합니다. 디스크 I/O가 응답 경로에 없어서 읽기/쓰기 모두 마이크로초 미만입니다.

싱글 스레드 이벤트 루프

명령 처리는 단일 스레드가 순차 실행합니다. 락(lock) 경합, 컨텍스트 스위칭 비용이 없고 모든 명령이 원자적(atomic)입니다.

I/O 멀티플렉싱

epoll 기반 이벤트 루프로 수만 개의 클라이언트 연결을 한 스레드가 동시에 처리합니다. 초당 10만+ 요청도 거뜬합니다.

💡

"싱글 스레드인데 왜 빠르죠?" — 처음 배울 때 가장 많이 나오는 질문입니다. 각 명령이 메모리 연산이라 마이크로초 안에 끝나기 때문에, 스레드를 여러 개 두고 락을 관리하는 비용이 오히려 더 큽니다. 단, Redis 6.0부터 네트워크 I/O(읽기/쓰기)는 멀티 스레드로 처리할 수 있게 되었습니다(io-threads 설정). 명령 실행 자체는 여전히 싱글 스레드입니다.

⚠️

싱글 스레드의 뒷면: 하나의 느린 명령이 전체를 멈춥니다. 예를 들어 수백만 개 키를 가진 서버에서 KEYS *를 실행하면 그 명령이 끝날 때까지 다른 모든 요청이 대기합니다. 이것이 뒤에서 배울 "운영 금지 명령"의 근본 이유입니다.

Redis는 어디에 쓰일까?

캐시 (Cache)

가장 대표적인 용도. DB 조회 결과를 Redis에 저장해두고 재사용하여 DB 부하를 90% 이상 줄입니다. TTL로 자동 만료를 관리합니다.

세션 저장소

여러 대의 웹 서버가 로그인 세션을 공유해야 할 때. 서버가 스케일 아웃되어도 어느 서버로 접속하든 같은 세션을 봅니다.

실시간 랭킹 / 리더보드

Sorted Set 자료구조가 점수 기반 정렬을 O(log N)에 처리합니다. 게임 랭킹, 인기 검색어에 최적입니다.

분산 락 / 카운터

원자적 연산(INCR, SETNX)으로 여러 서버가 동시에 접근하는 자원을 안전하게 제어합니다. API Rate Limiting도 여기 속합니다.

메시지 브로커 / 큐

Pub/Sub, List, Stream으로 서비스 간 비동기 메시징을 구현합니다. 가벼운 작업 큐라면 Kafka 없이도 충분합니다.

AI/ML 인프라

피처 스토어(Feature Store), LLM 시맨틱 캐시, 벡터 검색(Redis Stack)까지 — MLOps 파이프라인에서도 존재감이 커지고 있습니다.

RDBMS와 무엇이 다른가

구분RedisRDBMS (MySQL 등)
저장 위치메모리 (선택적 디스크 영속화)디스크 (메모리는 버퍼)
데이터 모델키-값 + 풍부한 자료구조테이블 / 관계형 스키마
질의키 기반 명령어 (SQL 없음)SQL, 조인, 인덱스
응답 속도수십 µs ~ 1ms 미만수 ms ~ 수백 ms
용량 한계장비 RAM 크기 (클러스터로 확장)디스크 크기 (수 TB 용이)
주 용도캐시, 세션, 실시간 처리원본 데이터(Source of Truth)

중요한 것은 대체 관계가 아니라 보완 관계라는 점입니다. 실무에서는 "원본은 RDBMS, 빠른 접근은 Redis"라는 조합이 표준입니다.

📌

라이선스 이야기 (2024~2025) — Redis는 7.4부터 라이선스를 변경했고, 이에 반발해 커뮤니티가 포크한 Valkey(리눅스 재단 주관)가 등장했습니다. AWS ElastiCache, GCP Memorystore 등이 Valkey를 지원합니다. 이후 Redis 8.0은 다시 오픈소스(AGPLv3) 옵션을 추가했습니다. 명령어와 프로토콜은 사실상 호환되므로 이 강의 내용은 두 쪽 모두에 적용됩니다.

SECTION 02

설치와 첫 실행

Redis를 만나는 가장 빠른 길은 Docker입니다. 로컬 실습, Kubernetes 배포, 패키지 설치까지 환경별로 정리했습니다.

실행 방법 선택

terminal
# Redis 7 컨테이너 실행 (ARM64/x86_64 멀티아치 이미지 제공)
docker run -d --name redis \
  -p 6379:6379 \
  redis:7-alpine

# CLI 접속
docker exec -it redis redis-cli

# 살아있는지 확인
127.0.0.1:6379> PING
PONG
docker-compose.yml
services:
  redis:
    image: redis:7-alpine
    container_name: redis
    ports:
      - "6379:6379"
    command: redis-server --appendonly yes --requirepass "changeme"
    volumes:
      - redis-data:/data
    healthcheck:
      test: ["CMD", "redis-cli", "-a", "changeme", "ping"]
      interval: 5s

volumes:
  redis-data:
terminal
# 공식 APT 저장소 등록 후 설치
sudo apt-get install -y lsb-release curl gpg
curl -fsSL https://packages.redis.io/gpg | sudo gpg --dearmor -o /usr/share/keyrings/redis-archive-keyring.gpg
echo "deb [signed-by=/usr/share/keyrings/redis-archive-keyring.gpg] https://packages.redis.io/deb $(lsb_release -cs) main" | sudo tee /etc/apt/sources.list.d/redis.list
sudo apt-get update && sudo apt-get install -y redis

sudo systemctl enable --now redis-server
redis-cli ping   # → PONG
terminal
# Bitnami 차트로 단일 노드(standalone) 설치
helm repo add bitnami https://charts.bitnami.com/bitnami
helm install my-redis bitnami/redis \
  --set architecture=standalone \
  --set auth.password=changeme \
  -n redis --create-namespace

# 클러스터 내부 접속 테스트
kubectl run redis-client --rm -it --restart=Never \
  --image redis:7-alpine -n redis -- \
  redis-cli -h my-redis-master -a changeme ping

redis-cli 기본기

redis-cli는 Redis 서버와 대화하는 공식 커맨드라인 도구입니다. 자주 쓰는 옵션만 알아도 운영이 편해집니다.

명령설명
redis-cli -h <host> -p 6379 -a <pw>원격 서버에 접속 (호스트/포트/비밀번호)
redis-cli --stat초당 처리량, 메모리, 클라이언트 수를 실시간 표시
redis-cli --bigkeys큰 키(빅 키)를 샘플링으로 탐지 — 운영 필수 도구
redis-cli --latency서버 응답 지연을 지속 측정
redis-cli monitor서버로 들어오는 모든 명령을 실시간 출력 (부하 큼, 짧게만!)
redis-cli info서버 상태 전체 리포트 (메모리, 복제, 통계…)

꼭 알아야 할 초기 설정 (redis.conf)

redis.conf — 최소한의 프로덕션 기본값
# 접속 제어 — 기본값은 localhost만 허용. 외부 오픈 시 반드시 비밀번호와 함께!
bind 0.0.0.0
protected-mode yes
requirepass "StrongPassword!"

# 메모리 상한과 축출 정책 (4장에서 자세히)
maxmemory 4gb
maxmemory-policy allkeys-lru

# 영속성 (5장에서 자세히)
appendonly yes

# 위험 명령 비활성화 — 실수로 FLUSHALL 치는 사고 방지
rename-command FLUSHALL ""
rename-command KEYS ""
🚨

인터넷에 그냥 열면 몇 시간 안에 해킹당합니다. 비밀번호 없이 bind 0.0.0.0으로 공인 IP에 노출된 Redis는 자동화된 봇이 스캔해 채굴 악성코드를 심는 단골 표적입니다. 반드시 ① 사설 네트워크에만 배치, ② requirepass/ACL 설정, ③ 보안그룹·방화벽으로 6379 포트 차단 — 세 가지를 지키세요.

SECTION 03

자료구조 완전 정복

Redis가 "단순한 캐시"를 넘어서는 이유는 값(Value)이 다양한 자료구조일 수 있기 때문입니다. 키는 항상 문자열이지만, 값은 String·Hash·List·Set·Sorted Set 등 목적에 맞는 구조를 선택합니다. 자료구조 선택이 곧 Redis 설계의 절반입니다.

① String — 만능 기본형

가장 단순한 타입으로, 텍스트·숫자·직렬화된 JSON·바이너리까지 최대 512MB를 담을 수 있습니다. 숫자를 담으면 원자적 증감 연산이 가능해 카운터로도 활약합니다.

redis-cli
127.0.0.1:6379> SET user:1001:name "철수"
OK
127.0.0.1:6379> GET user:1001:name
"철수"
127.0.0.1:6379> SET page:views 0
OK
127.0.0.1:6379> INCR page:views          # 원자적 +1 → 동시성 걱정 없는 카운터
(integer) 1
127.0.0.1:6379> SET session:abc "data" EX 3600   # 1시간 뒤 자동 삭제
OK
127.0.0.1:6379> SETNX lock:job42 "worker-1"     # 키가 없을 때만 SET → 분산 락의 기초
(integer) 1

② Hash — 객체를 담는 그릇

하나의 키 아래 필드-값 쌍의 묶음을 저장합니다. "사용자 프로필"처럼 여러 속성을 가진 객체에 딱 맞고, 특정 필드만 읽고 쓸 수 있어 JSON 문자열 통째 저장보다 효율적입니다.

redis-cli
127.0.0.1:6379> HSET user:1001 name "철수" role "engineer" level 7
(integer) 3
127.0.0.1:6379> HGET user:1001 role
"engineer"
127.0.0.1:6379> HINCRBY user:1001 level 1     # 필드 단위 증감
(integer) 8
127.0.0.1:6379> HGETALL user:1001
1) "name"  2) "철수"  3) "role"  4) "engineer"  5) "level"  6) "8"

③ List — 순서가 있는 큐/스택

삽입 순서를 유지하는 문자열 목록입니다. 양쪽 끝 삽입/삭제가 O(1)이라 작업 큐, 최근 활동 타임라인에 적합합니다. BLPOP은 데이터가 올 때까지 대기하는 블로킹 소비를 지원해 간단한 메시지 큐가 됩니다.

redis-cli
127.0.0.1:6379> LPUSH queue:jobs "job-1" "job-2"   # 왼쪽에 넣고
(integer) 2
127.0.0.1:6379> RPOP queue:jobs                       # 오른쪽에서 꺼내면 = FIFO 큐
"job-1"
127.0.0.1:6379> LRANGE queue:jobs 0 -1              # 전체 조회 (0부터 끝까지)
1) "job-2"
127.0.0.1:6379> LTRIM recent:user1 0 99            # 최근 100개만 유지 (타임라인 패턴)
OK

④ Set — 중복 없는 집합

순서 없이 유일한 값만 저장합니다. 멤버 존재 확인이 O(1)이고, 교집합·합집합·차집합 연산을 서버가 직접 계산해줍니다. 태그, 팔로워 목록, "오늘 방문한 순 사용자" 같은 유니크 집계에 씁니다.

redis-cli
127.0.0.1:6379> SADD tags:post1 redis cache devops
(integer) 3
127.0.0.1:6379> SISMEMBER tags:post1 redis      # 존재 확인 O(1)
(integer) 1
127.0.0.1:6379> SADD tags:post2 redis kubernetes
(integer) 2
127.0.0.1:6379> SINTER tags:post1 tags:post2    # 교집합 → 공통 태그
1) "redis"

⑤ Sorted Set (ZSet) — 점수로 정렬되는 집합

각 멤버가 score(실수)를 가지며 항상 점수순으로 정렬 상태를 유지합니다. 내부적으로 스킵 리스트를 사용해 삽입·조회가 O(log N)입니다. 리더보드, 우선순위 큐, 시간순 이벤트 인덱스의 최강자입니다.

redis-cli
127.0.0.1:6379> ZADD leaderboard 3200 "철수" 4100 "영희" 2800 "민수"
(integer) 3
127.0.0.1:6379> ZREVRANGE leaderboard 0 2 WITHSCORES   # TOP 3
1) "영희" 2) "4100" 3) "철수" 4) "3200" 5) "민수" 6) "2800"
127.0.0.1:6379> ZINCRBY leaderboard 500 "민수"          # 점수 갱신도 원자적
"3300"
127.0.0.1:6379> ZRANK leaderboard "철수"                # 내 순위는? (0부터, 오름차순)
(integer) 1

SECTION 04

키 관리 · TTL · 메모리 정책

메모리는 유한합니다. Redis 운영의 핵심은 "무엇을, 얼마나 오래, 얼마나 많이" 담을지 통제하는 것입니다. 키 네이밍, 만료(TTL), 축출(Eviction) 정책이 그 세 축입니다.

키 네이밍 컨벤션

Redis에는 스키마가 없기 때문에 키 이름이 곧 스키마입니다. 콜론(:)으로 계층을 구분하는 것이 사실상 표준입니다.

naming convention
# {서비스}:{엔터티}:{ID}:{속성}
user:1001:profile          # 사용자 1001의 프로필 Hash
session:ab3f9c             # 세션
cache:product:42           # 상품 42 캐시
rank:daily:2026-07-10      # 날짜별 랭킹 ZSet
lock:order:9981            # 주문 처리 분산 락
📌

키는 짧고 일관되게. 팀 위키에 네이밍 규칙을 문서화해 두면, 장애 상황에서 SCAN MATCH "cache:product:*" 같은 패턴 조회로 문제 범위를 빠르게 좁힐 수 있습니다.

TTL — 유통기한 관리

키에 만료 시간을 지정하면 시간이 지나 자동 삭제됩니다. 캐시가 무한정 쌓이는 것을 막는 가장 기본적인 장치입니다.

redis-cli
SET cache:home "<html>…" EX 300    # 저장과 동시에 300초 TTL (권장)
EXPIRE user:1 3600                  # 기존 키에 TTL 부여
TTL user:1                          # 남은 시간 조회 (-1: 무기한, -2: 없음)
PERSIST user:1                      # TTL 제거 → 영구 보관

만료된 키는 두 가지 방식으로 실제 삭제됩니다.

  • 지연 삭제(Lazy) — 누군가 그 키에 접근하는 순간 "아, 만료됐네" 하고 지웁니다.
  • 능동 삭제(Active) — 백그라운드에서 초당 10회, 만료 키를 무작위 샘플링해 청소합니다.
⚠️

같은 시각에 대량 만료 금지. 자정 정각에 100만 개 키가 동시에 만료되도록 설계하면 능동 삭제 사이클이 이벤트 루프를 점유해 지연 스파이크가 발생합니다. TTL에 무작위 지터(jitter)를 ±10% 섞어 만료 시점을 분산하세요.

maxmemory와 축출(Eviction) 정책

메모리가 maxmemory 한도에 도달하면 Redis는 설정된 정책에 따라 행동합니다. 이 설정 하나가 "캐시로서의 Redis"와 "저장소로서의 Redis"를 가릅니다.

정책동작언제 쓰나
noeviction쓰기 명령을 에러로 거부 (기본값)데이터 유실이 절대 안 되는 저장소 용도
allkeys-lru모든 키 중 가장 오래 안 쓴 것부터 삭제순수 캐시 — 가장 일반적인 선택
volatile-lruTTL 있는 키 중에서만 LRU 삭제캐시와 영구 데이터가 섞여 있을 때
allkeys-lfu사용 빈도가 가장 낮은 키부터 삭제인기 편차가 큰 캐시 (핫 아이템 보호)
volatile-ttl남은 TTL이 가장 짧은 키부터 삭제만료 임박 데이터를 우선 정리
allkeys-random / volatile-random무작위 삭제접근 패턴이 완전히 균등할 때 (드묾)
🚨

실무 단골 장애: maxmemory를 설정하지 않으면 Redis는 OS 메모리를 계속 먹다가 OOM Killer에게 강제 종료당합니다. 반대로 noeviction 상태에서 메모리가 차면 모든 쓰기가 OOM command not allowed 에러로 실패합니다. 캐시 용도라면 maxmemory = 장비 RAM의 60~70% + allkeys-lru가 안전한 출발점입니다. (나머지 30~40%는 5장에서 배울 fork 기반 스냅샷과 복제 버퍼의 몫입니다.)

SECTION 05

영속성 — RDB와 AOF

메모리는 전원이 꺼지면 사라집니다. Redis는 재시작 후에도 데이터를 되살릴 수 있도록 두 가지 디스크 기록 방식을 제공합니다. RDB(스냅샷)와 AOF(명령 로그) — 각각의 트레이드오프를 이해하는 것이 이 장의 목표입니다.

RDB — 특정 시점의 스냅샷

지정한 조건이 되면 메모리 전체를 바이너리 파일(dump.rdb) 한 장으로 찍어냅니다. 사진처럼 "그 순간"의 데이터만 담깁니다.

redis.conf — RDB
save 900 1      # 900초 동안 1개 이상 변경되면 스냅샷
save 300 10     # 300초 동안 10개 이상 변경되면
save 60 10000   # 60초 동안 10,000개 이상 변경되면
dbfilename dump.rdb
dir /data

# 수동 실행: BGSAVE (백그라운드, 권장) / SAVE (블로킹, 금지)

핵심 메커니즘은 fork + Copy-on-Write(COW)입니다. BGSAVE가 실행되면 Redis는 자식 프로세스를 fork하고, 자식이 그 시점의 메모리를 파일로 씁니다. 부모(본체)는 계속 요청을 처리하되, 스냅샷 중 변경되는 메모리 페이지만 복사됩니다.

⚠️

COW의 함정: 스냅샷 도중 쓰기가 폭주하면 변경 페이지가 모두 복제되어 순간 메모리 사용량이 최대 2배까지 치솟을 수 있습니다. maxmemory를 RAM의 60~70%로 잡으라는 이유가 바로 이것입니다.

AOF — 모든 쓰기 명령의 일기장

서버가 받은 모든 쓰기 명령을 순서대로 로그 파일에 기록합니다. 재시작 시 로그를 처음부터 재생(replay)해 데이터를 복원합니다. 얼마나 자주 디스크에 동기화(fsync)할지가 내구성과 성능의 균형점입니다.

redis.conf — AOF
appendonly yes
appendfsync everysec   # 매초 fsync — 최대 1초 유실, 성능/내구성 균형 (권장 기본)
# appendfsync always   → 매 명령 fsync. 유실 0에 가깝지만 처리량 크게 하락
# appendfsync no       → OS에 위임(보통 ~30초). 빠르지만 유실 범위 큼

# AOF가 커지면 자동으로 압축 재작성 (rewrite)
auto-aof-rewrite-percentage 100
auto-aof-rewrite-min-size 64mb

같은 키에 INCR counter를 100만 번 실행하면 AOF에는 100만 줄이 쌓이지만, 결과는 SET counter 1000000 한 줄과 같습니다. AOF rewrite는 이렇게 로그를 현재 상태 기준으로 압축해 다시 쓰는 과정으로, RDB처럼 fork를 사용합니다.

RDB vs AOF — 그리고 정답

비교 항목RDBAOF
데이터 유실 범위마지막 스냅샷 이후 전부 (수 분)최대 1초 (everysec)
파일 크기작음 (압축 바이너리)큼 (텍스트 명령 로그)
재시작 복구 속도빠름느림 (명령 재생)
런타임 부하fork 시점에 집중매초 fsync로 분산
백업/재해복구파일 하나 — 원격 복사에 최적가능하지만 무거움
📌

실무 정답은 "둘 다"입니다. Redis 7부터는 AOF rewrite 시 본문을 RDB 포맷으로 쓰는 하이브리드 모드(aof-use-rdb-preamble yes)가 기본이라, RDB의 빠른 복구와 AOF의 낮은 유실을 동시에 얻습니다. 반대로 순수 캐시 용도라면 영속성을 아예 끄는 것도 훌륭한 선택입니다 — 어차피 원본은 DB에 있으니까요.

SECTION 06

트랜잭션 · 파이프라인 · Pub/Sub · Lua

단일 명령을 넘어 여러 명령을 묶고, 왕복 지연을 줄이고, 실시간 메시지를 뿌리는 방법을 배웁니다. 이 장의 도구들은 애플리케이션 설계의 표현력을 크게 넓혀줍니다.

트랜잭션 — MULTI / EXEC

여러 명령을 큐에 모았다가 한 덩어리로 순차 실행합니다. 실행 중 다른 클라이언트의 명령이 끼어들지 않습니다. 단, RDBMS와 달리 롤백이 없습니다 — 중간 명령이 실패해도 나머지는 그대로 실행됩니다.

redis-cli
127.0.0.1:6379> MULTI
OK
127.0.0.1:6379(TX)> DECRBY account:A 100
QUEUED
127.0.0.1:6379(TX)> INCRBY account:B 100
QUEUED
127.0.0.1:6379(TX)> EXEC       # 두 명령이 원자적으로 함께 실행
1) (integer) 900
2) (integer) 1100

WATCH key를 먼저 걸면 낙관적 락(Optimistic Lock)이 됩니다. 감시 중인 키가 EXEC 전에 다른 클라이언트에 의해 변경되면 트랜잭션 전체가 무효화되어, "읽고-계산하고-쓰는" 패턴을 안전하게 만들 수 있습니다.

파이프라인 — 왕복 지연(RTT) 줄이기

명령 1개당 네트워크 왕복 1회가 기본입니다. RTT가 1ms라면 1만 개 명령에 10초가 걸립니다. 파이프라인은 명령 여러 개를 한 번에 보내고 응답을 한 번에 받는 기법으로, 같은 작업을 수십 배 빠르게 만듭니다.

pipeline.py
import redis
r = redis.Redis(host="localhost", port=6379)

pipe = r.pipeline(transaction=False)   # 순수 파이프라인
for i in range(10000):
    pipe.set(f"key:{i}", i)
results = pipe.execute()               # 왕복 1회로 1만 건 처리
terminal
# 파일의 명령을 파이프 모드로 대량 주입
cat commands.txt | redis-cli --pipe

Pub/Sub — 실시간 방송

발행자(Publisher)가 채널에 메시지를 보내면, 그 채널을 구독(Subscribe) 중인 모든 클라이언트가 즉시 받습니다. 채팅, 실시간 알림, 서버 간 이벤트 브로드캐스트에 쓰입니다.

터미널 2개로 실습
# [터미널 1] 구독
127.0.0.1:6379> SUBSCRIBE news
Reading messages...

# [터미널 2] 발행
127.0.0.1:6379> PUBLISH news "Redis 8 출시!"
(integer) 1    # ← 수신자 수
⚠️

Pub/Sub은 "지금 접속해 있는 사람에게만" 전달됩니다(fire-and-forget). 구독자가 잠시 끊겨 있으면 그 사이 메시지는 영영 사라집니다. 유실이 허용되지 않는 메시징이라면 Stream(소비자 그룹 + ACK + 재처리)을 쓰세요.

Lua 스크립트 — 서버에서 실행되는 원자적 로직

"읽고 → 판단하고 → 쓰는" 복합 로직을 클라이언트에서 하면 그 사이에 다른 요청이 끼어들 수 있습니다. Lua 스크립트는 서버 안에서 통째로, 원자적으로 실행되므로 이 문제를 근본적으로 해결합니다. 대표 사례가 Rate Limiter와 분산 락 해제입니다.

rate_limiter.lua — 분당 N회 제한
-- KEYS[1]=카운터 키, ARGV[1]=제한 횟수, ARGV[2]=윈도(초)
local current = redis.call('INCR', KEYS[1])
if current == 1 then
  redis.call('EXPIRE', KEYS[1], ARGV[2])
end
if current > tonumber(ARGV[1]) then
  return 0   -- 차단
end
return 1     -- 허용
terminal
redis-cli EVAL "$(cat rate_limiter.lua)" 1 ratelimit:user:1001 100 60
(integer) 1
SECTION 07

복제 (Replication) — 고가용성의 첫걸음

서버 한 대는 언젠가 반드시 죽습니다. 고가용성(High Availability)의 출발점은 같은 데이터를 가진 서버를 여러 대 두는 것 — 즉 복제입니다. Redis 복제는 마스터(Master) 1대에 레플리카(Replica) N대가 붙는 구조입니다.

Master–Replica 기본 토폴로지
Master
읽기 + 쓰기
──▶
Replica 1
읽기 전용
──▶ └──▶
Replica 2
읽기 전용

복제 설정은 단 한 줄

replica 노드에서
# redis.conf 방식
replicaof 10.0.1.10 6379
masterauth "StrongPassword!"
replica-read-only yes        # 기본값 — 레플리카 직접 쓰기 방지

# 또는 런타임에
127.0.0.1:6379> REPLICAOF 10.0.1.10 6379
OK
127.0.0.1:6379> INFO replication   # role, 연결된 replica 수, offset 확인

복제는 어떻게 동작하나

  1. 전체 동기화 (Full Sync) — 레플리카가 처음 연결되면 마스터는 BGSAVE로 RDB 스냅샷을 만들어 통째로 전송합니다. 레플리카는 자기 데이터를 비우고 이 스냅샷을 로드합니다.
  2. 명령 스트림 (Command Propagation) — 이후 마스터에 들어오는 모든 쓰기 명령이 실시간으로 레플리카에 흘러갑니다. 레플리카는 같은 명령을 재실행해 상태를 따라갑니다.
  3. 부분 재동기화 (Partial Resync) — 네트워크가 잠깐 끊겼다 붙으면, 마스터의 복제 백로그 버퍼(replication backlog)에 남아 있는 구간만 이어받습니다. 백로그를 넘어서면 다시 Full Sync — 대형 인스턴스에서는 이게 큰 부하이므로 repl-backlog-size(기본 1MB)를 트래픽에 맞게 수십~수백 MB로 키우는 것이 좋습니다.
⚠️

Redis 복제는 비동기(async)입니다. 마스터는 레플리카의 수신 확인을 기다리지 않고 클라이언트에 OK를 반환합니다. 즉 마스터가 죽는 순간, 아직 레플리카에 전달되지 않은 최근 몇 밀리초~수 초의 쓰기는 유실될 수 있습니다. 이 특성은 Sentinel/Cluster로 장애조치를 해도 사라지지 않습니다. "Redis를 계좌 원장의 유일한 저장소로 쓰면 안 되는 이유"가 바로 여기 있습니다.

복제로 얻는 것

읽기 확장 (Read Scaling)

읽기 트래픽을 레플리카로 분산합니다. 읽기:쓰기 = 9:1인 전형적 캐시 워크로드에서 매우 효과적입니다.

장애 대비 (Standby)

마스터가 죽으면 레플리카를 승격(promote)해 서비스를 이어갑니다. 단, 복제 자체는 승격을 자동으로 해주지 않습니다 — 그건 다음 장 Sentinel의 일입니다.

SECTION 08

Sentinel — 자동 장애조치(Failover)

복제만 있는 상태에서 마스터가 죽으면, 새벽 3시에 엔지니어가 일어나 수동으로 레플리카를 승격해야 합니다. Sentinel은 이 과정을 자동화하는 감시자입니다. 마스터를 지켜보다가, 죽었다고 판단되면 투표를 거쳐 레플리카를 새 마스터로 승격합니다.

Sentinel의 세 가지 역할

감시 (Monitoring)

마스터·레플리카에 주기적으로 PING을 보내 생존을 확인합니다.

장애조치 (Failover)

마스터 사망이 확정되면 최적의 레플리카를 골라 새 마스터로 승격하고, 나머지를 재구성합니다.

구성 제공 (Config Provider)

클라이언트는 Sentinel에게 "지금 마스터가 누구야?"라고 물어 항상 올바른 주소로 접속합니다.

왜 Sentinel은 3대 이상, 홀수여야 할까

Sentinel이 1대뿐이면 그 Sentinel 자신이 단일 장애점(SPOF)이 됩니다. 더 위험한 건 오판입니다 — 네트워크가 잠깐 끊긴 것을 "마스터 사망"으로 착각해 승격하면, 잠시 후 마스터가 2대(split-brain)가 되어 데이터가 갈라집니다.

그래서 Sentinel은 합의(quorum)로 동작합니다:

  • SDOWN (주관적 다운) — Sentinel 1대가 "내가 보기엔 마스터가 죽었어"라고 판단한 상태.
  • ODOWN (객관적 다운) — quorum 수(예: 3대 중 2대) 이상의 Sentinel이 동의한 상태. 이때부터 장애조치가 시작됩니다.
  • 이후 Sentinel들끼리 리더 선출(과반수 필요)을 하고, 리더가 승격을 집행합니다. 과반수 개념 때문에 3, 5 같은 홀수 구성이 표준입니다.
sentinel.conf (3대 모두 동일)
port 26379
# "mymaster"라는 이름으로 10.0.1.10:6379 감시, quorum=2
sentinel monitor mymaster 10.0.1.10 6379 2
sentinel auth-pass mymaster "StrongPassword!"
sentinel down-after-milliseconds mymaster 5000   # 5초 무응답 → SDOWN
sentinel failover-timeout mymaster 60000
sentinel parallel-syncs mymaster 1   # 승격 후 레플리카 재동기화 동시 수

클라이언트는 어떻게 접속해야 하나

장애조치 후 마스터의 IP가 바뀌므로, 애플리케이션은 마스터 주소를 하드코딩하면 안 됩니다. Sentinel에게 물어보는 방식으로 접속합니다.

Python — Sentinel-aware 클라이언트
from redis.sentinel import Sentinel

sentinel = Sentinel([
    ("10.0.1.21", 26379),
    ("10.0.1.22", 26379),
    ("10.0.1.23", 26379),
], socket_timeout=0.5)

master = sentinel.master_for("mymaster", password="...")   # 쓰기용
replica = sentinel.slave_for("mymaster", password="...")   # 읽기 분산용

master.set("hello", "world")
print(replica.get("hello"))
💡

Sentinel의 한계: Sentinel은 가용성을 지켜주지만 데이터를 여러 노드에 나누지는 못합니다. 전체 데이터는 여전히 마스터 1대의 RAM에 다 들어가야 하고, 쓰기 처리량도 1대의 한계를 넘지 못합니다. 데이터와 쓰기를 수평 분산하려면 다음 장의 Redis Cluster가 필요합니다.

SECTION 09

Redis Cluster — 데이터를 쪼개서 확장하기

Sentinel이 "죽지 않게" 만드는 기술이라면, Cluster는 "커질 수 있게" 만드는 기술입니다. 데이터를 여러 마스터에 자동 분산(샤딩)하고, 각 마스터에 레플리카를 붙여 고가용성까지 내장합니다. 별도의 Sentinel이 필요 없습니다.

핵심 개념: 16384개의 해시 슬롯

Redis Cluster는 전체 키 공간을 16384개(0~16383)의 슬롯으로 나눕니다. 모든 키는 다음 공식으로 정확히 하나의 슬롯에 속합니다:

슬롯 결정 공식
slot = CRC16(key) mod 16384

각 마스터 노드는 슬롯 범위를 나눠 가집니다. 예를 들어 마스터 3대라면 대략 5461개씩 담당합니다. 키가 어느 노드에 있는지는 계산만으로 알 수 있으므로 중앙 조회 서버(코디네이터)가 필요 없습니다 — 이것이 Redis Cluster 설계의 백미입니다.

🧮 해시 슬롯 계산기 — 직접 확인해보세요

키를 입력하면 실제 CRC16 알고리즘으로 슬롯 번호를 계산하고, 3-마스터 클러스터에서 어느 노드로 가는지 보여줍니다. {}가 포함된 키는 해시 태그 규칙(중괄호 안만 해싱)이 적용되는 것도 관찰해보세요. 예: user:1001, {user:1001}:cart, {user:1001}:orders

user:1001 user:1002 {user:1001}:cart {user:1001}:orders session:abc123

노드들은 가십(gossip) 프로토콜로 서로의 생존과 슬롯 배치를 공유합니다(클라이언트 포트 + 10000번, 예: 16379가 클러스터 버스). 마스터 하나가 죽으면 남은 마스터들의 과반수 합의로 해당 레플리카가 자동 승격됩니다 — Sentinel의 역할이 클러스터 안에 내장된 셈입니다. 과반수가 필요하므로 마스터는 최소 3대, 홀수가 원칙입니다.

클러스터 만들어보기

terminal — 6노드 클러스터 생성
# 각 노드의 redis.conf
cluster-enabled yes
cluster-config-file nodes.conf
cluster-node-timeout 5000
appendonly yes

# 6개 노드를 하나의 클러스터로 묶기 (마스터 3 + 레플리카 3 자동 배치)
redis-cli --cluster create \
  10.0.1.10:6379 10.0.1.11:6379 10.0.1.12:6379 \
  10.0.1.13:6379 10.0.1.14:6379 10.0.1.15:6379 \
  --cluster-replicas 1

# 상태 확인
redis-cli -c -h 10.0.1.10 cluster info     # cluster_state:ok
redis-cli -c -h 10.0.1.10 cluster nodes    # 노드별 슬롯 배치
💡

-c 옵션은 클러스터 모드입니다. 다른 노드 소관의 키를 요청하면 서버가 MOVED 3999 10.0.1.11:6379처럼 올바른 노드를 알려주는 리다이렉트를 보내는데, -c가 이를 자동으로 따라갑니다. 실제 애플리케이션에서는 클러스터 지원 클라이언트(redis-py-cluster, Lettuce 등)가 슬롯 맵을 캐시해 처음부터 올바른 노드로 직행합니다.

리샤딩 — 무중단으로 슬롯 옮기기

노드를 추가하면 기존 노드들의 슬롯 일부를 새 노드로 이관해야 합니다. 이 과정은 서비스 중단 없이 진행됩니다.

terminal — 노드 추가와 리샤딩
# 1) 새 마스터 노드 합류
redis-cli --cluster add-node 10.0.1.16:6379 10.0.1.10:6379

# 2) 슬롯 4096개를 새 노드로 이동 (기존 노드들에서 골고루 차출)
redis-cli --cluster reshard 10.0.1.10:6379 \
  --cluster-to <새노드ID> --cluster-slots 4096 --cluster-from all

# 3) 슬롯 분포 자동 균형
redis-cli --cluster rebalance 10.0.1.10:6379

이관 중인 슬롯의 키를 요청하면 ASK 리다이렉트로 임시 위치를 안내받습니다. 클라이언트 라이브러리가 자동 처리하므로 애플리케이션 코드는 신경 쓸 필요가 없습니다.

대규모 클러스터 설계 — 사이징부터 Kubernetes까지

"어떤 구성을 몇 대로, 어디에 배치할 것인가?" 이 장은 아키텍처 선택 기준, 용량 산정 공식, 배치 원칙, Kubernetes 배포까지 — 실제 설계 문서에 그대로 옮길 수 있는 내용을 다룹니다.

단순함도 설계 역량입니다. 데이터 20GB, 5만 ops/s 수준이라면 Sentinel 구성이 Cluster보다 운영 복잡도·제약(멀티 키, DB 번호)이 훨씬 적습니다. Cluster는 "필요해질 때" 도입하세요.

용량 산정 (Sizing) 공식

  1. 순수 데이터 크기 추정 — 키 수 × (평균 키 길이 + 평균 값 크기 + 오버헤드 약 50~90바이트). 예: 1억 키 × 평균 200B ≈ 약 25GB. 실측이 최선입니다: 샘플 데이터를 넣고 INFO memory의 used_memory를 확인하세요.
  2. 안전 계수 적용 — 복제 버퍼, 클라이언트 버퍼, fork COW, 파편화(fragmentation)를 고려해 노드 RAM의 60~70%만 데이터에 사용합니다. 25GB 데이터라면 노드당 가용 메모리 총합은 약 40GB 필요.
  3. 샤드 수 결정 — 노드 1대당 데이터 10~25GB 수준을 권장합니다. 인스턴스가 너무 크면 Full Sync·fork·장애조치 시간이 모두 길어집니다. 25GB 데이터라면: 마스터 3대 × 노드당 ~8.3GB가 무난. "큰 노드 소수"보다 "적당한 노드 다수"가 운영에 유리합니다.
  4. 처리량 검증 — redis-benchmark -h <host> -c 100 -n 1000000 -t get,set -P 16으로 목표 ops/s를 실측 검증합니다. 필요 처리량이 노드 한계를 넘으면 샤드를 늘립니다.
  5. 총 노드 수 = 마스터 수 × (1 + 레플리카 수) — 프로덕션 최소 권장: 마스터 3 × 레플리카 1 = 6노드.

📐 사이징 예제: "일 1억 요청, 데이터 100GB" 서비스

  • 데이터 100GB ÷ 노드당 15GB(권장선) ≈ 마스터 7대 → 여유를 두어 8대
  • 레플리카 1씩 → 총 16노드, 노드당 RAM = 15GB ÷ 0.65 ≈ 24GB → 32GB 인스턴스 선택
  • 피크 트래픽 초당 약 5,000 ops(1억/일 기준 평균의 4배 가정) → 노드당 처리량 여유 충분
  • 결론: 32GB RAM × 16노드, 3개 가용영역(AZ)에 분산 배치

배치 원칙 — 장애 도메인 분리

  • 마스터와 그 레플리카는 반드시 다른 물리 서버/AZ에. 같은 랙에 두면 랙 장애 = 샤드 전체 유실입니다. Cluster 생성 시 --cluster-replicas가 기본적으로 안티어피니티를 고려하지만, 반드시 cluster nodes로 검증하세요.
  • 3 AZ 배치 — 마스터 3대를 AZ마다 1대씩. 한 AZ가 통째로 죽어도 과반수(2/3)가 살아 있어 자동 장애조치가 가능합니다. 2 AZ 구성은 한쪽이 죽으면 과반수 미달로 클러스터가 멈출 수 있습니다.
  • 네트워크 지연 주의 — 클러스터 노드 간 RTT가 크면(예: 리전 간 배치) gossip과 복제가 불안정해집니다. 클러스터는 단일 리전 내에, 리전 간에는 별도 복제 도구를 사용합니다.

Kubernetes에 배포하기

Redis는 상태를 가진(Stateful) 워크로드이므로 StatefulSet + PVC + Headless Service 조합이 기본입니다. 직접 매니페스트를 짜기보다 Helm 차트 또는 Operator를 쓰는 것이 표준 관행입니다.

terminal
# 마스터 1 + 레플리카 2 + Sentinel 3 (파드마다 사이드카)
helm install redis-ha bitnami/redis \
  --set architecture=replication \
  --set sentinel.enabled=true \
  --set replica.replicaCount=3 \
  --set auth.password="StrongPassword!" \
  --set master.persistence.size=8Gi \
  -n redis --create-namespace
terminal
# Redis Cluster (기본: 마스터 3 + 레플리카 3 = 파드 6개)
helm install redis-cluster bitnami/redis-cluster \
  --set cluster.nodes=6 \
  --set cluster.replicas=1 \
  --set password="StrongPassword!" \
  --set persistence.size=10Gi \
  -n redis --create-namespace

kubectl exec -it redis-cluster-0 -n redis -- \
  redis-cli -c -a StrongPassword! cluster info
values.yaml — 프로덕션 체크포인트
# 1) 파드 분산 — 노드/AZ 단위 안티어피니티
podAntiAffinityPreset: hard
topologySpreadConstraints:
  - maxSkew: 1
    topologyKey: topology.kubernetes.io/zone
    whenUnsatisfiable: DoNotSchedule

# 2) 리소스 — requests=limits로 QoS Guaranteed 권장
resources:
  requests: { memory: 8Gi, cpu: "2" }
  limits:   { memory: 8Gi, cpu: "2" }

# 3) maxmemory는 limit의 약 70%로 (fork/버퍼 여유)
commonConfiguration: |-
  maxmemory 5gb
  maxmemory-policy allkeys-lru

# 4) 이기종(ARM64/x86_64) 클러스터라면 아키텍처 고정 배치
nodeSelector:
  kubernetes.io/arch: amd64   # 멀티아치 이미지라면 생략 가능
⚠️

Kubernetes 특유의 함정들: ① 메모리 limit을 넘으면 OOMKilled — maxmemory를 반드시 limit보다 낮게. ② PodDisruptionBudget 없이 노드 드레인하면 마스터·레플리카가 동시에 내려갈 수 있음. ③ Cluster 모드에서 파드 IP가 재시작마다 바뀌므로 차트/오퍼레이터의 IP 추적 로직에 의존 — 직접 매니페스트로 구성할 때 가장 많이 실패하는 지점입니다. 이런 이유로 프로덕션에서는 검증된 Helm 차트나 Redis Operator(OT-Container-Kit, Spotahome 등)를 권장합니다.

SECTION 11

캐싱 패턴과 운영 실무

아키텍처를 잘 세워도 사용하는 패턴이 틀리면 장애가 납니다. 캐싱 전략, 3대 캐시 장애(스탬피드·핫 키·빅 키), 모니터링, 보안 — 현장에서 바로 쓰는 지식을 정리합니다.

캐싱 전략 3형제

Cache-Aside (Lazy)

앱이 캐시를 먼저 조회 → 없으면(miss) DB에서 읽어 캐시에 채우고 반환. 가장 보편적인 패턴. 필요한 데이터만 캐시되는 장점, 첫 요청이 느린 단점.

Write-Through

쓰기 시 캐시와 DB를 함께 갱신. 캐시가 항상 최신이지만 쓰기 지연이 늘고, 안 읽힐 데이터도 캐시됩니다.

Write-Behind

캐시에만 쓰고 DB 반영은 비동기 배치로. 쓰기 폭주 흡수에 탁월하지만 캐시 유실 = 데이터 유실 위험.

cache_aside.py — 표준 구현
import json, random

def get_product(product_id: int):
    key = f"cache:product:{product_id}"
    cached = r.get(key)
    if cached:                          # ① Cache Hit
        return json.loads(cached)

    product = db.query_product(product_id)  # ② Miss → DB 조회
    ttl = 300 + random.randint(0, 60)       # ③ TTL 지터로 동시 만료 방지
    r.set(key, json.dumps(product), ex=ttl)
    return product

보안 체크리스트

  • 네트워크 격리 — 사설 서브넷 배치, 보안그룹/NetworkPolicy로 애플리케이션 대역만 6379/16379 허용.
  • 인증 — requirepass는 기본, Redis 6+ 라면 ACL로 계정별 권한 분리: ACL SETUSER app on >pw ~cache:* +get +set 처럼 "이 계정은 cache:* 키에 GET/SET만" 수준까지 제한 가능합니다.
  • 암호화 — 전송 구간 TLS(tls-port 6379), 저장 데이터는 디스크/볼륨 암호화로 보완.
  • 위험 명령 봉인 — FLUSHALL, CONFIG, DEBUG는 rename 또는 ACL로 차단.
  • 백업 — RDB 스냅샷을 주기적으로 오브젝트 스토리지(S3/MinIO)로 반출, 복원 리허설까지가 백업입니다.
📌

업그레이드는 무중단으로: 레플리카부터 새 버전으로 교체 → 수동 장애조치(CLUSTER FAILOVER 또는 Sentinel SENTINEL FAILOVER)로 역할 전환 → 구 마스터 교체. Redis는 상위 버전 레플리카가 하위 버전 마스터를 따라갈 수 있어 롤링 업그레이드가 가능합니다.
