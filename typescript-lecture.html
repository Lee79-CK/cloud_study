01 — 왜 배우나요?

TypeScript는 결국 "JavaScript + 타입"

한 줄 요약: 실행 전에 실수를 잡아주는 JavaScript

TypeScript(TS)는 마이크로소프트가 만든 언어로, JavaScript의 모든 문법을 그대로 포함합니다. 즉 유효한 JS 코드는 거의 그대로 유효한 TS 코드예요. 여기에 "타입(type)"이라는 추가 정보를 붙일 수 있게 해줍니다.

🤔 JavaScript의 흔한 함정

아래 JS 코드는 문법상 완벽히 정상입니다. 브라우저가 에러 없이 실행하죠. 하지만 결과는 의도와 다릅니다.

function 합계(a, b) {
  return a + b;
}

합계(10, 20);      // 30  ✅ 의도한 대로
합계(10, "20");    // "1020"  😱 문자열로 이어붙여짐!
합계(10);          // NaN  😱 b가 undefined
문제점 JS는 실행하기 전까지 이런 실수를 알려주지 않습니다. 실제 서비스에서 사용자가 버튼을 눌렀을 때 비로소 터지죠. 디버깅에 몇 시간이 날아갈 수 있습니다.

같은 함수를 TypeScript로 쓰면, 코드를 작성하는 순간 에디터가 빨간 줄로 경고합니다.

function 합계(a: number, b: number): number {
  return a + b;
}

합계(10, 20);      // 30 ✅
합계(10, "20");    // ❌ '"20"'은 number가 아닙니다 (실행 전에 경고!)
합계(10);          // ❌ 인자 2개가 필요합니다
핵심 TS는 실행 전(컴파일 시점)에 타입을 검사합니다. 버그가 사용자에게 도달하기 전에, 내 화면에서 미리 잡힙니다.

TypeScript를 쓰면 좋은 점

  • 실수를 미리 잡음 — 오타, 잘못된 인자, 없는 속성 접근을 작성 시점에 발견.
  • 자동완성이 똑똑해짐 — 객체에 어떤 속성이 있는지 에디터가 정확히 알려줌(IntelliSense).
  • 코드가 곧 문서 — 타입만 봐도 "이 함수는 뭘 받고 뭘 돌려주는지" 한눈에 파악.
  • 리팩토링이 안전 — 이름을 바꾸거나 구조를 고쳐도, 영향받는 곳을 TS가 전부 짚어줌.
  • 협업에 강함 — 팀원이 내 함수를 잘못 쓰면 즉시 에러로 알려줌.
중요한 사실 브라우저는 TypeScript를 직접 실행하지 못합니다. TS는 마지막에 평범한 JavaScript로 변환(컴파일)되어 실행됩니다. 즉 타입은 "개발할 때만 존재하는 도우미"이고, 배포되는 결과물은 결국 JS예요.
02 — 한눈에 비교

JavaScript vs TypeScript

둘은 경쟁 관계가 아닙니다. TS는 JS 위에 얹는 "개발 도구"에 가깝습니다.

구분JavaScriptTypeScript
타입 검사실행 중에만 (런타임)작성/컴파일 시점에 미리
파일 확장자.js / .jsx.ts / .tsx
브라우저 실행바로 가능JS로 컴파일 후 실행
에러 발견 시점사용자가 만난 뒤 😱내가 코딩할 때 ✅
자동완성제한적매우 강력함
학습 부담낮음JS를 알면 약간 더
대표 사용처모든 웹React, Vue, Angular, Node 등 실무 표준
비유 JavaScript가 안전벨트 없는 자동차라면, TypeScript는 안전벨트 + 후방카메라 + 차선이탈 경고를 단 같은 자동차입니다. 차(=실행 결과)는 똑같지만, 사고를 미리 막아줍니다.
03 — 시작하기

설치부터 첫 컴파일까지

Node.js가 설치되어 있다는 가정 하에, 가장 표준적인 시작 방법입니다.

TypeScript 설치

프로젝트 폴더에서 컴파일러(tsc)를 설치합니다.

npm install -D typescript
# 또는 전역 설치
npm install -g typescript

설정 파일 만들기 (tsconfig.json)

TS가 어떻게 동작할지 정하는 설정 파일을 생성합니다.

npx tsc --init

처음엔 아래 정도만 알면 충분합니다.

{
  "compilerOptions": {
    "target": "ES2020",      // 어떤 JS 버전으로 변환할지
    "module": "ESNext",
    "strict": true,          // ⭐ 엄격 모드 (강력 추천)
    "outDir": "./dist",      // 컴파일 결과 JS 저장 위치
    "esModuleInterop": true
  }
}
strict: true 초보자일수록 strict 모드를 켜세요. 처음엔 빨간 줄이 많이 보여 답답할 수 있지만, TS가 주는 보호를 100% 받는 길입니다. 여기서 가장 중요한 건 null/undefined를 깐깐하게 체크해주는 것이에요.

코드 작성 → 컴파일

hello.ts 파일을 만들고 컴파일하면, hello.js가 생성됩니다.

npx tsc            # 전체 컴파일
npx tsc --watch    # 파일 저장할 때마다 자동 컴파일

컴파일이 무슨 뜻인가요?

왼쪽 TS 코드를 tsc가 오른쪽 JS로 바꿔줍니다. 타입 표기(: number 등)는 깔끔하게 사라지고, 브라우저가 실행할 수 있는 순수 JS만 남습니다.

greet.ts (작성한 코드)
function greet(name: string): string {
  return `안녕, ${name}!`;
}
const msg: string = greet("철규");
greet.js (컴파일 결과)
function greet(name) {
  return `안녕, ${name}!`;
}
const msg = greet("철규");
바로 해보기 설치가 번거롭다면 TypeScript Playground(typescriptlang.org/play)에서 브라우저만으로 바로 실험할 수 있습니다. 왼쪽에 TS를 쓰면 오른쪽에 변환된 JS가 실시간으로 보입니다.
04 — 기본 타입

타입을 "표기"하는 법

핵심 문법은 딱 하나예요. 이름 뒤에 콜론(:)을 찍고 타입을 적는다. 이걸 "타입 어노테이션(주석)"이라고 부릅니다.

let 이름: string = "철규";        // 문자열
let 나이: number = 34;            // 숫자 (정수/실수 구분 없음)
let 활성화: boolean = true;       // 참/거짓
let 빈값: null = null;            // 비어있음(의도적)
let 미정: undefined = undefined;  // 아직 값 없음

// 배열: '타입[]' 형태
let 점수들: number[] = [90, 85, 100];
let 친구들: string[] = ["민수", "영희"];

// 어떤 값이든 (가능하면 피하세요)
let 아무거나: any = 42;
아무거나 = "이제 문자열";  // any는 검사를 꺼버림 ⚠️
기억하세요 JS에는 int, float 같은 구분이 없습니다. 숫자는 전부 number 하나입니다.

JS와 나란히 보기

위 탭을 눌러 같은 코드를 JS와 TS로 비교해보세요. 차이는 오직 콜론 뒤의 타입 표기뿐입니다.

변수.ts
변수.js
let user: string = "kim";
let level: number = 5;
let isAdmin: boolean = false;
let tags: string[] = ["a", "b"];
let user = "kim";
let level = 5;
let isAdmin = false;
let tags = ["a", "b"];
심화: any는 왜 위험한가요?

any를 쓰면 그 변수에 대한 모든 타입 검사가 꺼집니다. TS를 쓰는 의미가 사라지는 거죠. "JS로 도망치는 비상구"라고 생각하세요. 정말 타입을 모를 때는 any 대신 뒤에서 배울 unknown을 쓰는 게 안전합니다.

let x: any = "글자";
x.toFixed(2);   // ❌ 런타임 에러나지만 TS는 경고 안 함 (any라서)
05 — 타입 추론

매번 타입을 적지 않아도 됩니다

TS의 가장 친절한 기능. 값을 보고 타입을 자동으로 알아냅니다. 이를 "타입 추론(Type Inference)"이라고 해요.

let 도시 = "서울";        // TS가 자동으로 'string'으로 추론
// 도시 = 123;            // ❌ string에 number 못 넣음

let 인구 = 9700000;       // 자동으로 'number'
let 광역시 = true;        // 자동으로 'boolean'
실무 팁 변수 초기화처럼 값이 명확할 땐 타입을 생략하고 추론에 맡기세요. 코드가 깔끔해집니다. 반대로 함수의 매개변수는 추론이 안 되므로 꼭 직접 적어야 합니다.
inference.ts● TypeScript
let city = "서울";
// TS는 city를 string으로 기억합니다
city = 42;
'number' 형식은 'string' 형식에 할당할 수 없습니다. — TS가 알아서 막아줍니다 (위 빨간 물결선에 마우스를 올려보세요)
06 — 객체의 모양

interface 와 type — 가장 많이 쓰는 도구

실무에서 데이터는 대부분 객체입니다(사용자, 상품, 게시글...). 그 객체가 "어떤 속성을 가져야 하는지"를 미리 정의하는 것이 인터페이스입니다.

interface로 객체 모양 정의하기

interface User {
  id: number;
  name: string;
  email: string;
  age?: number;        // '?' = 있어도 되고 없어도 됨 (선택적)
  readonly 가입일: string; // readonly = 한번 정하면 못 바꿈
}

const u: User = {
  id: 1,
  name: "철규",
  email: "ck@kt.com",
  가입일: "2026-01-01"
  // age는 선택적이라 생략 OK
};

u.name = "철규2";   // ✅ 가능
u.가입일 = "변경";  // ❌ readonly라서 못 바꿈
자주 하는 실수 인터페이스에 없는 속성을 넣거나, 필수 속성을 빠뜨리면 즉시 에러입니다. 이게 바로 TS가 지켜주는 부분이에요.
user.ts● TypeScript
const u: User = {
id: 1,
name: "철규",
emial: "ck@kt.com" // 오타!
};
개체 리터럴은 알려진 속성만 지정할 수 있습니다. 'emial'은(는) 'User'에 없습니다. 'email'을(를) 쓰려고 했나요? — 오타까지 잡아줍니다!

interface vs type — 뭘 써야 하죠?

type 키워드로도 거의 같은 일을 할 수 있습니다.

interface 방식
interface Product {
  name: string;
  price: number;
}
// 나중에 확장 가능
interface Product {
  stock: number;
}
type 방식
type Product = {
  name: string;
  price: number;
};
// 유니온 등 더 유연함
type ID = string | number;
type Status = "ON" | "OFF";
초보자용 결론 너무 고민 마세요. 객체 모양엔 interface, 그 외(유니온·별칭 등)엔 type으로 시작하면 충분합니다. 둘 다 쓰다 보면 감이 옵니다.
07 — 함수

함수에 타입 입히기

함수야말로 타입의 효과가 가장 큽니다. 매개변수(입력)와 반환값(출력)에 타입을 달아줍니다.

// (매개변수: 타입): 반환타입
function multiply(a: number, b: number): number {
  return a * b;
}

// 화살표 함수도 동일
const greet = (name: string): string => `Hi ${name}`;

// 선택적 매개변수 '?'  / 기본값
function log(msg: string, level?: string): void {
  // void = 아무것도 반환하지 않음
  console.log(level ?? "INFO", msg);
}

log("저장 완료");           // level 생략 OK
log("에러 발생", "ERROR");  // level 전달 OK
void 란? 함수가 값을 반환하지 않을 때의 타입입니다. console.log처럼 "동작만 하고 돌려주는 건 없는" 함수에 씁니다.

JS에서 자주 터지던 버그가 사라집니다

JavaScript — 조용히 실패
TypeScript — 즉시 경고
function getDiscount(price, rate) {
  return price * rate;
}
// 인자 순서를 헷갈림 — JS는 그냥 실행됨 😱
getDiscount(0.1, 10000);   // 1000 (의도와 다름, 경고 없음)
getDiscount(10000);        // NaN (rate가 undefined)
function getDiscount(price: number, rate: number): number {
  return price * rate;
}
getDiscount(10000, 0.1);   // 1000 ✅
getDiscount(10000);        // ❌ 인자 2개가 필요합니다
getDiscount(10000, "0.1"); // ❌ string은 number가 아닙니다
08 — 유니온 & 리터럴

"이것 또는 저것" 표현하기

현실의 값은 한 종류가 아닐 때가 많죠. 파이프(|)로 여러 타입 중 하나임을 표현합니다.

// 유니온 타입: number 또는 string
let id: number | string;
id = 101;       // ✅
id = "A-101";   // ✅
id = true;      // ❌ boolean은 안 됨

// 리터럴 타입: 정해진 값만 허용 (오타 방지에 최고!)
type Theme = "light" | "dark";
let mode: Theme = "dark";   // ✅
let mode2: Theme = "darks"; // ❌ 오타 즉시 발견!
실무에서 자주 씀 버튼 상태("idle" | "loading" | "done"), 사이즈("sm" | "md" | "lg")처럼 정해진 값만 받아야 할 때 리터럴 유니온이 빛납니다. 자동완성도 이 값들만 제안해줍니다.

타입 좁히기 (Narrowing)

유니온 값을 다룰 땐, if로 "지금은 어떤 타입인지" 확인해줘야 합니다. 그러면 TS가 그 블록 안에서 타입을 좁혀줍니다.

function printId(id: number | string) {
  if (typeof id === "string") {
    // 이 안에서 id는 'string'으로 확정됨
    console.log(id.toUpperCase());  // 문자열 메서드 사용 OK
  } else {
    // 여기선 id가 'number'
    console.log(id.toFixed(2));     // 숫자 메서드 사용 OK
  }
}
09 — null 안전성

"undefined의 속성을 읽을 수 없음" 박멸하기

프론트엔드 최악의 런타임 에러 1위. TS의 strict 모드는 이걸 코딩 단계에서 막아줍니다.

null-check.ts (strict 모드)● TypeScript
const el = document.getElementById("btn");
// 반환 타입: HTMLElement | null (없을 수도 있으니까)
el.addEventListener("click", fn);
'el'은(는) 'null'일 수 있습니다. — 요소가 없을 경우를 처리하라고 강제합니다

올바른 처리

const el = document.getElementById("btn");

if (el) {                       // null이 아닐 때만
  el.addEventListener("click", fn);   // ✅ 안전
}

// 또는 옵셔널 체이닝 '?.'
el?.addEventListener("click", fn);    // el이 null이면 그냥 넘어감
이것 하나만 기억해도 ?.(옵셔널 체이닝)과 ??(널 병합 연산자)는 실무 필수입니다. a ?? b는 "a가 null/undefined면 b를 쓴다"는 뜻이에요.
심화: unknown vs any vs never
  • any — 모든 검사를 끔. 위험. 되도록 피하기.
  • unknown — "뭔지 모름". 쓰기 전에 타입 확인을 강제 → any보다 안전한 대안.
  • never — "절대 일어나지 않음". 에러를 던지는 함수나, 모든 경우를 처리했음을 보장할 때 등장.
function fail(msg: string): never {
  throw new Error(msg);   // 정상적으로 끝나지 않음 → never
}

let data: unknown = JSON.parse("...");
// data.length;          // ❌ unknown은 바로 못 씀
if (typeof data === "string") data.length;  // ✅ 확인 후 사용
10 — 제네릭

타입을 "변수처럼" 다루기

처음엔 가장 낯선 개념이지만, 비유로 보면 쉽습니다. 제네릭은 "타입을 나중에 정하는 빈칸"입니다.

왜 필요할까?

"배열의 첫 요소를 돌려주는 함수"를 만든다고 합시다. 숫자 배열, 문자 배열... 타입마다 따로 만들 순 없죠. 빈칸(<T>)을 두면 됩니다.

// T는 '아직 모르는 타입'을 담는 빈칸 (이름은 자유, 보통 T)
function first<T>(arr: T[]): T {
  return arr[0];
}

const n = first<number>([10, 20, 30]);  // n은 number
const s = first(["a", "b"]);           // 호출만 봐도 T=string 추론
// n.toFixed() ✅ ,  s.toUpperCase() ✅  ← 타입이 유지됨!
비유 제네릭은 택배 상자예요. 상자(Box<T>)는 같지만, 안에 책을 넣으면 "책 상자", 신발을 넣으면 "신발 상자"가 됩니다. 꺼낼 때도 넣은 그대로의 타입으로 나옵니다.

이미 매일 쓰고 있었습니다

사실 여러분은 제네릭을 이미 사용 중입니다. 배열과 Promise가 대표적이에요.

let nums: Array<number> = [1, 2, 3];     // number[] 와 동일
let names: Array<string> = ["a", "b"];

// API 호출 결과 — Promise도 제네릭
async function getUser(): Promise<User> {
  const res = await fetch("/api/user");
  return res.json();
}
11 — 유틸리티 타입

기존 타입을 재활용하는 마법

이미 만든 타입에서 변형된 타입이 필요할 때, 처음부터 다시 쓰지 마세요. TS가 미리 만들어 둔 도구가 있습니다.

interface User {
  id: number;
  name: string;
  email: string;
}

// Partial: 모든 속성을 선택적으로 (수정 폼에 유용)
type UserEdit = Partial<User>;
// → { id?: number; name?: string; email?: string }

// Pick: 원하는 속성만 골라내기
type UserCard = Pick<User, "id" | "name">;
// → { id: number; name: string }

// Omit: 특정 속성만 제외
type UserNoEmail = Omit<User, "email">;
// → { id: number; name: string }

// Record: 키-값 쌍의 객체 타입
type Scores = Record<string, number>;
const s: Scores = { kor: 90, eng: 85 };
유틸리티역할주 사용처
Partial<T>모든 속성을 선택적으로수정/업데이트 폼
Required<T>모든 속성을 필수로검증 후 데이터
Pick<T,K>일부 속성만 추출요약 카드, 목록
Omit<T,K>일부 속성 제외비밀번호 제거 등
Record<K,V>키-값 객체사전, 매핑 데이터
12 — 실전 활용

React + API 타이핑 맛보기

프론트엔드에서 TS가 가장 빛나는 두 곳: 컴포넌트 Props와 API 응답 데이터입니다.

① 컴포넌트 Props에 타입 달기

어떤 props를 받아야 하는지 명확해지고, 빠뜨리면 즉시 경고합니다.

interface ButtonProps {
  label: string;
  variant?: "primary" | "ghost";   // 정해진 값만
  onClick: () => void;             // 인자 없고 반환 없는 함수
}

function Button({ label, variant = "primary", onClick }: ButtonProps) {
  return <button className={variant} onClick={onClick}>{label}</button>;
}

// 사용처
<Button label="저장" onClick={save} />            // ✅
<Button label="삭제" variant="danger" />          // ❌ "danger" 없음 + onClick 누락

② useState에 타입 주기

// 추론으로 충분한 경우
const [count, setCount] = useState(0);        // number로 추론

// 초기값이 null이거나 복잡할 땐 직접 명시
const [user, setUser] = useState<User | null>(null);

// user.name; // ❌ null일 수 있음
if (user) console.log(user.name);  // ✅ 좁히기 후 안전

③ API 응답 타이핑

서버가 주는 데이터의 모양을 인터페이스로 정의해두면, 그 뒤 모든 사용처에서 자동완성과 검사를 받습니다.

interface Post {
  id: number;
  title: string;
  author: { name: string; email: string };
}

async function fetchPosts(): Promise<Post[]> {
  const res = await fetch("/api/posts");
  return res.json() as Promise<Post[]>;
}

const posts = await fetchPosts();
posts[0].title;          // ✅ 자동완성됨
posts[0].author.name;    // ✅ 중첩 객체까지 안전
posts[0].titel;          // ❌ 오타 즉시 발견
실무 효과 백엔드 응답 구조를 한 번만 정의하면, 프론트 전체에서 "이 데이터에 뭐가 들어있더라?"를 다시 찾아볼 필요가 없습니다. 점(.)만 찍으면 에디터가 다 알려줍니다.
13 — 초보 탈출 팁

자주 하는 실수 & 좋은 습관

실수 1 — any 남용 에러를 빨리 없애려고 any를 붙이면 TS를 끄는 것과 같습니다. 모르면 unknown 또는 정확한 타입을 찾으세요.
실수 2 — 타입 단언(as) 과신 값 as 타입은 "내가 확실하니 믿어"라고 TS를 강제로 설득하는 것. 틀리면 런타임에 터집니다. 꼭 필요할 때만.
습관 1 — 추론을 믿기 변수엔 타입을 생략하고, 함수 경계(매개변수·반환)에만 명시하면 깔끔합니다.
습관 2 — strict 끄지 않기 답답해도 strict를 유지하세요. 빨간 줄은 적이 아니라 미래의 나를 구하는 알림입니다.
습관 3 — 에러 메시지 끝까지 읽기 TS 에러는 길지만 친절합니다. 보통 마지막 줄에 "무엇이 무엇에 할당될 수 없다"는 핵심이 있어요.
14 — 확인 퀴즈

제대로 이해했는지 점검!

선택지를 눌러보세요. 정답과 해설이 바로 나타납니다.

Q1. TypeScript 코드는 브라우저에서 어떻게 실행되나요?
TS는 브라우저가 못 읽습니다. tsc가 순수 JavaScript로 변환하고, 타입 표기는 모두 제거됩니다.
Q2. 다음 중 "정해진 문자열 값만 허용"하려면?
리터럴 유니온입니다. 정해진 값 외엔 막아주고, 자동완성으로 그 값들만 제안해줍니다.
Q3. 함수에서 타입을 반드시 직접 적어야 하는 곳은?
매개변수는 추론이 안 되므로 명시 필수. 반환값은 추론되지만, 명확성을 위해 적는 걸 권장합니다.
Q4. el?.click() 에서 ?.의 의미는?
옵셔널 체이닝. null 안전성을 지키는 핵심 문법으로, "있으면 실행, 없으면 통과"입니다.
Q5. 기존 User 타입의 모든 속성을 선택적으로 만든 새 타입은?
Partial<User>는 모든 속성에 ?를 붙인 타입을 생성합니다. 수정 폼에서 특히 유용합니다.
🎉 축하합니다 — 이제 TypeScript의 기초가 잡혔습니다

핵심은 단 하나, "이 값은 어떤 모양이어야 하는지 미리 약속한다"는 것. 나머지는 모두 그 변주일 뿐입니다.
