NVIDIA NemoClaw 완벽 가이드
— AI 에이전트를 "안전하게" 돌리는 법

사상 최고 속도로 성장한 오픈소스 AI 에이전트 OpenClaw. 하지만 "내 컴퓨터를 마음대로 조작하는 AI"를 그냥 믿고 돌려도 될까요? NVIDIA가 내놓은 답이 바로 NemoClaw입니다. 이 강의에서 개념부터 설치, 명령어, 네트워크 정책까지 차근차근 배워봅니다.

난이도: 입문 소요 시간: 약 40분 대상: 초보 AI/클라우드 엔지니어 기준: NemoClaw docs (2026)

01OpenClaw 먼저 이해하기

NemoClaw을 이해하려면 그 기반이 되는 OpenClaw부터 알아야 합니다.

OpenClaw는 Peter Steinberger가 만든 오픈소스 개인용 AI 에이전트(personal AI agent)입니다. ChatGPT 같은 "채팅창 속 AI"와 달리, OpenClaw는 내 컴퓨터 위에서 24시간 상주하며 실제로 일을 하는 AI입니다. 파일을 만들고, 명령어를 실행하고, 브라우저를 조작하고, 메신저(WhatsApp, Telegram 등)로 나와 대화하면서 작업을 처리합니다.

🍞 쉬운 비유

일반 챗봇이 "전화 상담원"이라면(말로만 도와줌), OpenClaw는 "우리 집에 들어와 사는 가사도우미"입니다. 직접 청소하고 요리하고 심부름까지 해주죠. 훨씬 유용하지만 — 집 열쇠를 통째로 맡겨야 한다는 점이 문제입니다.

왜 폭발적으로 성장했나?

  • 진짜 "행동"하는 AI — 답변 생성에서 끝나지 않고 셸 명령 실행, 파일 관리, 브라우저 자동화까지 수행합니다.
  • 항상 켜져 있음(always-on) — 서버나 PC에 상주하며 메시지 한 통으로 어디서든 일을 시킬 수 있습니다.
  • 자기 진화(self-evolving) — 스스로 스킬을 추가하고 설정을 바꾸며 능력을 확장합니다.
  • 오픈소스 — GitHub 스타 30만 개를 넘기며 역사상 가장 빠르게 성장한 오픈소스 프로젝트로 꼽힙니다.

그런데… 무엇이 문제였을까?

강력함은 곧 위험이기도 합니다. OpenClaw를 호스트 머신에 그대로 설치하면:

🔓

무제한 권한

에이전트가 내 홈 디렉터리, SSH 키, 자격 증명 파일까지 접근할 수 있습니다.

🌐

무제한 네트워크

프롬프트 인젝션에 당하면 데이터를 외부로 유출하거나 악성 사이트에 접속할 수 있습니다.

☁️

데이터 프라이버시

내 개인 파일과 대화 내용이 클라우드 LLM API로 그대로 전송됩니다.

⚠️

핵심 문제 정의: "에이전트를 작동시키는 것"과 "에이전트를 안전하게, 기업 환경에서, 대규모로 작동시키는 것"은 완전히 다른 문제입니다. NemoClaw은 바로 이 두 번째 문제를 풀기 위해 탄생했습니다.

02NemoClaw이란?

한 줄 요약: "OpenClaw를 안전한 격리 공간(샌드박스)에서 돌려주는 NVIDIA의 오픈소스 스택"

NVIDIA NemoClaw™은 OpenClaw 같은 상시 구동(always-on) AI 에이전트를 NVIDIA OpenShell 샌드박스 안에서 더 안전하게 실행하도록 도와주는 오픈소스 레퍼런스 스택입니다. 2026년 GTC에서 발표되었고, Apache 2.0 라이선스로 무료 공개되어 있습니다.

가장 큰 특징은 "단 한 줄의 명령"으로 전체 스택이 설치된다는 점입니다. OpenShell 런타임(격리 샌드박스), Nemotron 오픈 모델, 추론 라우팅, 네트워크 보안 정책이 한 번에 세팅됩니다.

terminal — 설치는 이 한 줄
$ curl -fsSL https://www.nvidia.com/nemoclaw.sh | bash
🍞 쉬운 비유

OpenClaw가 "우리 집에 사는 가사도우미"라면, NemoClaw은 도우미 전용 별채(샌드박스)를 지어주고, 출입 가능한 방 목록(정책)을 정하고, 외출할 때마다 허락을 받게 하고(egress 승인), CCTV 기록(감사 로그)까지 남기는 "안전 관리 시스템"입니다. 도우미의 능력은 그대로 쓰면서 집 열쇠는 지키는 거죠.

NemoClaw이 해결하는 것

기존의 고통NemoClaw의 해법
복잡한 설치드라이버 호환성 체크, 의존성 해결, 보안 도구 구성을 단일 명령 + 온보딩 마법사로 자동화
호스트 무제한 접근OpenShell 샌드박스(격리 컨테이너) 안에서만 에이전트 실행
무분별한 외부 통신기본 차단(default-deny) 네트워크 정책 + 운영자 승인 기반 egress 제어
데이터 유출 우려Nemotron 로컬 모델 또는 프라이버시 라우터를 통한 로컬/클라우드 하이브리드 추론
운영·재현성 부재블루프린트 기반 재현 가능한 셋업, 스냅샷, 라이프사이클 관리
💡

용어 정리: NVIDIA 생태계에서 상시 구동 자율 에이전트를 "claw(클로)"라고 부르기도 합니다. NemoClaw = NeMo(NVIDIA의 AI 모델/툴 브랜드) + Claw(에이전트)의 합성어라고 이해하면 쉽습니다.

🖥️

지원 하드웨어: GeForce RTX PC/노트북, RTX PRO 워크스테이션, DGX Station, 그리고 DGX Spark까지 — 에이전트에게 전용 로컬 컴퓨트를 주는 것이 컨셉입니다. 클라우드나 온프레미스 서버에서도 물론 동작합니다.

034대 핵심 구성 요소

NemoClaw = OpenShell + Nemotron + Privacy Router + Network Policy. 하나씩 뜯어봅시다.

📦

① OpenShell — 격리 샌드박스 런타임

NVIDIA Agent Toolkit의 일부인 샌드박스 런타임입니다. 에이전트는 호스트가 아닌 격리 컨테이너 안에서 실행되며, 파일시스템·네트워크·추론 경로가 모두 정책으로 통제됩니다. 게이트웨이가 샌드박스와 바깥세상 사이의 유일한 통로 역할을 합니다.

🧠

② Nemotron — 오픈 모델 (로컬 추론)

NVIDIA의 오픈소스 LLM 패밀리입니다. 기본 온보딩에서는 Nemotron 3 Super 120B를 build.nvidia.com API로 연결하고, 내 GPU에서 vLLM 등으로 완전 로컬 추론도 가능합니다. 로컬로 돌리면 토큰 비용 0원, 데이터 유출 0, 오프라인 동작까지 가능합니다.

🔀

③ Privacy Router — 로컬/클라우드 지능형 라우팅

모든 작업에 거대 모델이 필요한 건 아닙니다. 프라이버시 라우터는 민감한 데이터는 로컬 모델로, 고난도 추론은 클라우드 프론티어 모델로 정책에 따라 요청을 배분합니다. 로컬의 프라이버시 + 클라우드의 성능을 동시에 잡는 구조입니다.

🚦

④ Network Policy — 기본 차단 + 승인 기반 허용

샌드박스의 외부 통신(egress)은 기본적으로 전부 차단(default-deny)됩니다. 허용 목록에 있는 도메인만 통과하고, 그 외 접속 시도는 운영자(사람)의 승인을 거쳐야 합니다. 프롬프트 인젝션으로 인한 데이터 유출을 구조적으로 막는 핵심 장치입니다.

🧩

헷갈리지 마세요: NemoClaw은 OpenClaw를 대체하지 않습니다. OpenClaw(에이전트 본체)는 그대로 두고, 그 아래에 보안·추론·운영 인프라 계층을 깔아주는 래퍼(wrapper)이자 배포판(distribution)입니다. OpenClaw 외에 Hermes, LangChain Deep Agents 같은 다른 에이전트도 지원합니다.

04아키텍처 & 동작 원리

아래 다이어그램의 각 구성 요소를 클릭해 보세요. 요청이 어떻게 흐르는지 설명이 나타납니다.

👆 다이어그램의 박스를 클릭하면 설명이 여기에 표시됩니다.

요청 한 건의 여정 (Life of a Request)

사용자가 명령을 보냄

nemoclaw <이름> connect로 접속하거나 메신저로 "이번 주 일정 정리해줘"라고 요청합니다.

에이전트가 샌드박스 안에서 작업

OpenClaw는 격리된 /sandbox 파일시스템 안에서만 파일을 읽고 쓰고 명령을 실행합니다. 호스트의 SSH 키나 개인 파일에는 손댈 수 없습니다.

추론 요청은 게이트웨이를 경유

LLM 호출은 샌드박스에서 직접 나가지 않고 반드시 OpenShell 게이트웨이를 통과합니다. 게이트웨이가 설정된 추론 경로(로컬 Nemotron / 클라우드)로 라우팅합니다.

외부 접속은 정책 검사

에이전트가 웹사이트에 접속하려 하면 정책 엔진이 확인합니다. 허용 목록에 없으면 차단하거나 운영자에게 승인을 요청합니다.

모든 활동이 기록됨

어떤 명령이 실행되고 어디에 접속했는지 감사 로그로 남아, OpenShell TUI에서 실시간 모니터링할 수 있습니다.

📐

재현 가능한 셋업(블루프린트): NemoClaw은 샌드박스 구성을 블루프린트로 정의합니다. nemoclaw onboard를 다시 실행하면 동일한 블루프린트와 정책 정의로부터 샌드박스가 그대로 재생성됩니다 — IaC(Infrastructure as Code)와 같은 철학입니다.

05OpenClaw vs NemoClaw 비교

버튼을 눌러 두 가지 실행 방식을 전환하며 비교해 보세요.

OpenClaw를 호스트에 직접 설치하면

  • 설치: npm 설치 → 모델 API 키 설정 → 채널 연동 → 권한 구성… 단계마다 직접 판단해야 합니다.
  • 실행 위치: 호스트 OS 위에서 직접 실행 — 에이전트가 내 계정 권한을 그대로 갖습니다.
  • 파일 접근: 홈 디렉터리, ~/.ssh, 브라우저 프로필 등 사실상 전부 접근 가능.
  • 네트워크: 제한 없음. 프롬프트 인젝션에 당하면 임의의 외부 서버로 데이터 전송 가능.
  • 추론: 설정한 클라우드 API로 대화·파일 내용이 전송됨.
  • 운영: 백업·복구·정책·감사는 전부 사용자 몫.

적합한 경우: 개인 실험, 빠른 프로토타이핑, 위험을 이해하고 감수할 수 있는 파워 유저.

NemoClaw으로 실행하면

  • 설치: 한 줄 설치 + nemoclaw onboard 마법사가 게이트웨이·모델·샌드박스·정책을 일괄 구성.
  • 실행 위치: OpenShell 격리 샌드박스(컨테이너) 내부. 호스트와 분리됩니다.
  • 파일 접근: /sandbox 격리 파일시스템만. 필요한 것만 share mount로 명시적 공유.
  • 네트워크: default-deny. 허용 도메인 + 운영자 승인 기반 egress만 통과.
  • 추론: 로컬 Nemotron / 클라우드 / 프라이버시 라우터 중 선택 — 민감 데이터는 로컬에 남길 수 있음.
  • 운영: 스냅샷, 블루프린트 재현, 감사 로그, TUI 모니터링, 라이프사이클 명령 제공.

적합한 경우: 기업·팀 환경, 민감 데이터 취급, 상시 구동 에이전트를 "믿고 맡기고" 싶은 모든 경우.

🐳

Docker

Node.js

Node.js 20+ (없으면 nvm으로 자동 설치)

Step 1 — CLI 설치

07주요 CLI 명령어

NemoClaw은 두 개의 CLI 표면을 제공합니다 — 언제 무엇을 쓸지가 중요합니다.

CLI역할이럴 때 사용
nemoclaw호스트 측 관리형 워크플로샌드박스 생성/재생성, 접속, 스냅샷, 모델 전환 등 일상 작업 대부분

openclaw nemoclaw … / /nemoclaw샌드박스 내부 플러그인에이전트 채팅 중 상태 확인 등 인-샌드박스 빠른 액션
🚫

주의: NemoClaw이 관리하는 환경에서는 openshell sandbox create, openshell self-update 등을 직접 실행하지 마세요. 관리 상태가 어긋납니다. 샌드박스·게이트웨이의 생성/재생성은 항상 nemoclaw onboard로 하세요.

샌드박스 라이프사이클
# 새 설치 / 정책·구성 변경 후 재생성 (마법사)
$ nemoclaw onboard
$ nemoclaw onboard --resume   # 중단된 온보딩 이어하기
$ nemoclaw onboard --fresh    # 저장된 세션 버리고 처음부터

# 에이전트 접속 (대화형)
$ nemoclaw my-agent connect

# 작업 공간 보존 — 스냅샷
$ nemoclaw my-agent snapshot create
$ nemoclaw my-agent snapshot restore
단발 실행 & 파일 공유
# 대화형 셸 없이 샌드박스 안에서 명령 한 번만 실행
# (샌드박스 사용자 권한, HOME=/sandbox 로 실행됨)
$ nemoclaw my-agent exec "ls -la /sandbox/workspace"

# 호스트 디렉터리를 샌드박스에 마운트해 공유
$ nemoclaw my-agent share mount

# 정밀한 경로 제어가 필요한 수동 복사는 openshell 사용
$ openshell sandbox upload   ./report.md
$ openshell sandbox download /sandbox/out.csv
런타임 모델 전환
# 모델/공급자 조회와 전환은 nemoclaw 명령을 사용해야
# OpenShell 라우트와 에이전트 설정이 일관되게 유지됩니다.
$ nemoclaw my-agent model list
$ nemoclaw my-agent model switch <model-id>

# 빌드 타임 설정(에이전트 자체 구성)을 바꾸려면 → 온보딩 재실행
$ nemoclaw onboard

클라우드 Nemotron ↔ 로컬 vLLM 사이를 상황에 맞게 오갈 수 있는 것이 관리형 추론의 장점입니다.

상태 · 로그 · TUI
# 호스트에서 샌드박스 목록/상태 확인
$ openshell sandbox list

# 블루프린트 실행 · 샌드박스 로그 스트리밍 (플러그인 명령)
$ openclaw nemoclaw logs -f -n 100

# OpenShell TUI로 샌드박스 활동 실시간 모니터링
$ openshell tui

# 에이전트 채팅 안에서는 슬래시 명령으로 빠른 액션
> /nemoclaw status

08네트워크 정책 체험 시뮬레이터

NemoClaw의 핵심인 "default-deny + 승인 기반 egress"가 어떻게 동작하는지 직접 눌러보며 체험해 보세요.

시나리오: 샌드박스 안의 에이전트가 아래 도메인들에 접속을 시도합니다. 정책 모드를 바꿔가며 결과가 어떻게 달라지는지 확인해 보세요.

🚦 Egress 정책 시뮬레이터

허용 목록(allowlist): build.nvidia.com, pypi.org, github.com

위 도메인 버튼을 눌러 에이전트의 접속 시도를 시뮬레이션하세요…
✅

default-deny의 가치

"나쁜 곳을 막는다"가 아니라 "허락한 곳만 연다"는 발상의 전환입니다. 미지의 위협(제로데이 유출 경로)까지 구조적으로 차단됩니다.

🙋

운영자 승인(operator approval)

에이전트가 새 도메인이 필요하면 요청이 사람에게 전달됩니다. 승인하면 통과, 거부하면 차단 — "AI에게 자율성을 주되 최종 결정권은 사람"이라는 설계 철학입니다.

🔧

실무 팁: 자주 승인하게 되는 도메인은 정책 파일에 영구 등록하세요(Customize the Network Policy 가이드 참고). 정책 변경 후에는 nemoclaw onboard로 샌드박스를 재생성하면 블루프린트에 반영된 상태로 일관성이 유지됩니다.

어떤 추론 구성을 골라야 할까?

☁️

클라우드 (build.nvidia.com)

가장 쉬운 시작. GPU가 약한 머신도 OK. 단, 데이터가 외부 API로 나갑니다.

🖥️

로컬 (vLLM + Nemotron)

토큰 비용 0, 데이터 유출 0, 오프라인 가능. 충분한 GPU 메모리가 필요합니다.

🔀

Privacy Router (하이브리드)

민감 데이터는 로컬, 고난도 추론은 클라우드로 자동 배분. 실무 추천 구성.

⚡

DGX Spark 사용자라면: 128GB 통합 메모리를 가진 DGX Spark는 NemoClaw이 공식 지원하는 "에이전트 전용 로컬 컴퓨트" 대표 기기입니다. Nemotron급 대형 모델을 로컬 vLLM으로 서빙하면서 상시 구동 에이전트를 24/7 돌리기에 이상적인 구성입니다. ARM64(aarch64) 환경이므로 커스텀 Dockerfile을 쓸 때는 이미지의 아키텍처 호환성만 확인하세요.

보안 관점 체크리스트

  • 최소 허용 원칙: allowlist는 실제 필요한 도메인만. "일단 다 열어두기"는 NemoClaw을 쓰는 의미를 없앱니다.
  • 공유 마운트 최소화: share mount는 편리하지만 격리 경계를 넓힙니다. 꼭 필요한 디렉터리만 공유하세요.
  • 감사 로그 확인 습관: OpenShell TUI로 에이전트가 실제로 무엇을 했는지 주기적으로 리뷰하세요.
  • 자격 증명은 온보딩 위저드에: API 키를 에이전트 채팅에 붙여넣지 마세요. 위저드/설정 경로로만 전달하는 것이 원칙입니다.

10이해도 체크 퀴즈

5문제로 오늘 배운 내용을 점검해 봅시다.

QUESTION 1 / 5
NemoClaw과 OpenClaw의 관계로 가장 정확한 설명은?
✅ NemoClaw은 OpenClaw(에이전트 본체)를 그대로 두고 OpenShell 샌드박스, 관리형 추론, 네트워크 정책 등 보안·운영 계층을 더해주는 래퍼/배포판입니다. NVIDIA는 OpenClaw 프로젝트에도 기여하고 있습니다.
QUESTION 2 / 5
NemoClaw에서 에이전트가 격리 실행되는 런타임의 이름은?
✅ OpenShell은 NVIDIA Agent Toolkit의 일부인 샌드박스 런타임으로, 에이전트를 격리 컨테이너 안에서 실행하고 게이트웨이를 통해 추론·네트워크를 통제합니다. Nemotron은 모델 패밀리 이름입니다.
QUESTION 3 / 5
NemoClaw 기본 네트워크 정책의 동작 방식은?
✅ "나쁜 곳을 막는다"가 아니라 "허락한 곳만 연다"는 default-deny 방식입니다. 목록 밖 접속은 차단되거나 운영자 승인 절차를 거칩니다.
QUESTION 4 / 5
NemoClaw 관리 환경에서 샌드박스를 재생성하는 올바른 방법은?
✅ 생성/재생성의 공식 진입점은 nemoclaw onboard입니다. 블루프린트와 정책 정의로부터 동일한 구성이 재현되며, openshell 명령으로 직접 만들면 NemoClaw 관리 상태와 어긋납니다.
QUESTION 5 / 5
Privacy Router의 역할로 옳은 것은?
✅ Privacy Router는 민감한 데이터는 로컬 Nemotron으로, 고난도 작업은 클라우드 프론티어 모델로 라우팅해 프라이버시와 성능을 동시에 잡는 구성 요소입니다.
문제를 풀면 점수가 여기에 표시됩니다.

📌 3줄 요약
