cuDNN, 도대체 무엇인가

cuDNNCUDA Deep Neural Network library

cuDNN은 NVIDIA가 만든 딥러닝 전용 GPU 가속 라이브러리입니다. 합성곱(convolution), 풀링, 정규화, 어텐션 같은 신경망의 핵심 연산을 NVIDIA GPU에서 가능한 가장 빠른 속도로 실행할 수 있도록 고도로 최적화된 커널(kernel)을 제공합니다.

한 문장으로 정리하면

cuDNN은 "딥러닝의 표준 연산을 GPU에서 가장 빠르게 돌릴 수 있도록 만든 NVIDIA의 무료 라이브러리"입니다. 여러분이 PyTorch, TensorFlow, JAX 같은 프레임워크에서 nn.Conv2d나 tf.nn.conv2d를 호출하면, 그 호출은 결국 cuDNN의 함수로 내려갑니다. 우리가 직접 다루는 일은 거의 없지만, 딥러닝 모델 학습/추론 속도의 상당 부분은 cuDNN이 책임집니다.

CUDA vs cuDNN — 가장 헷갈리는 부분

"둘 다 NVIDIA 거고, 둘 다 GPU에서 뭔가 빠르게 한다"는 점에서 입문자에게 가장 혼동되는 영역입니다. 명확히 구분해 봅시다.

CUDA
general-purpose GPU computing platform
  • 범용 병렬 컴퓨팅 플랫폼. GPU에서 어떤 종류의 계산이든 돌릴 수 있게 해주는 기반.
  • 저수준(low-level). 스레드, 블록, 그리드, 공유 메모리를 직접 제어합니다.
  • C/C++ 기반 확장 언어(CUDA C++)와 드라이버, 런타임 API, 컴파일러(nvcc)를 포함합니다.
  • 도메인 무관. 물리 시뮬레이션, 암호화, 영상 처리, 금융 계산 어디든 사용 가능.
  • 커널을 직접 작성할 수 있는 자유 — 그리고 책임도 있음.
  • 예시: __global__ void myKernel(...) 함수를 직접 짜서 GPU에 실행
cuDNN
deep learning specialized library
  • 딥러닝 전용 고수준 라이브러리. CUDA 위에서 동작하는 도메인 특화 계층.
  • 고수준(high-level). "이 텐서를 합성곱해라"라고 명령만 하면 됩니다.
  • 이미 컴파일된 최적화 커널을 제공. 우리는 함수 호출만 합니다.
  • 딥러닝 도메인 한정. Convolution, Attention, RNN, Pooling, BatchNorm 등.
  • 커널은 NVIDIA가 작성 — 우리는 가장 빠른 구현을 그냥 받아 씁니다.
  • 예시: cudnnConvolutionForward(handle, ...) 한 번 호출이면 끝

같은 일을 두 방식으로 — 코드 비교

"입력 텐서에 3×3 합성곱을 적용한다"는 같은 목표를, CUDA로 직접 짤 때 vs cuDNN을 쓸 때 어떻게 달라지는지 봅시다 (개념 설명용 의사 코드).

CUDA C++ (직접 작성)low-level · 수십~수백 줄
// 합성곱 커널을 직접 작성해야 함
__global__ void conv2dKernel(
    const float* input, const float* kernel,
    float* output,
    int H, int W, int K) {

    int x = blockIdx.x * blockDim.x + threadIdx.x;
    int y = blockIdx.y * blockDim.y + threadIdx.y;
    if (x >= W || y >= H) return;

    float sum = 0.0f;
    for (int i = 0; i < K; i++) {
      for (int j = 0; j < K; j++) {
        sum += input[(y+i)*W + (x+j)] * kernel[i*K + j];
      }
    }
    output[y*W + x] = sum;
}

// 그리드/블록 크기 결정, 공유 메모리 활용, 타일링,
// 메모리 코얼레싱, 텐서 코어 활용… 모두 직접 튜닝해야 함
conv2dKernel<<<grid, block>>>(d_input, d_kernel, d_output, H, W, K);
cuDNN (라이브러리 사용)high-level · 핵심 호출 1줄
// 텐서와 컨볼루션 설명자만 정의하고 함수 호출
cudnnTensorDescriptor_t inputDesc, outputDesc;
cudnnFilterDescriptor_t kernelDesc;
cudnnConvolutionDescriptor_t convDesc;
// ... descriptor 초기화 (생략)

// 단 한 줄로 GPU에서 최적화된 합성곱 실행
cudnnConvolutionForward(
    handle, &alpha,
    inputDesc, inputData,
    kernelDesc, kernelData,
    convDesc, algo, workspace, workspaceSize,
    &beta,
    outputDesc, outputData);

// NVIDIA가 아키텍처별로 튜닝한 최적 알고리즘이 자동 선택됨
// Tensor Core, WMMA, FFT-based conv 등 모두 내부적으로 처리
기억할 비유

CUDA는 "식재료와 주방 도구"이고, cuDNN은 "미슐랭 셰프가 미리 만들어둔 레시피 모음집"입니다. 둘 다 필요하지만, 일반적인 요리(딥러닝)는 셰프의 레시피를 그대로 따라 하는 게 훨씬 빠르고 맛있습니다.

그래서 결론

cuDNN은 CUDA의 대체재가 아니라, CUDA 위에서 돌아가는 특화 라이브러리입니다. 딥러닝을 한다면 두 가지를 모두 설치해야 합니다. 실제로는 nvidia-smi로 드라이버를 확인하고, CUDA Toolkit과 cuDNN을 각각 설치한 뒤, PyTorch/TensorFlow가 둘을 찾아서 알아서 씁니다.

cuDNN의 주요 컴포넌트

cuDNN은 단일 기능이 아니라 여러 "원시 연산(primitive)"의 집합입니다. 딥러닝에서 자주 쓰는 연산들을 카테고리별로 살펴봅시다. 탭을 클릭해 보세요.

cudnnConvolutionForward / Backward

합성곱(Convolution) — 가장 핵심적인 연산

CNN의 심장이자, cuDNN이 가장 신경 써서 최적화한 부분입니다. 같은 합성곱이라도 입력 크기, 채널 수, 필터 크기, 데이터 타입에 따라 cuDNN은 여러 알고리즘 중 최적의 것을 자동 선택합니다.

  • IMPLICIT_GEMM — 합성곱을 행렬곱(GEMM)으로 풀어내는 가장 일반적인 방식
  • WINOGRAD — 3×3 같은 작은 필터에 효과적. 곱셈 횟수를 줄이는 수학적 변환 사용
  • FFT-based — 큰 커널에 유리. 주파수 영역에서 합성곱이 곱셈이 되는 성질 이용
  • Tensor Core(WMMA) — FP16/BF16/INT8에서 텐서 코어 직접 활용. 가장 빠름

cudnnPoolingForward / Backward

풀링(Pooling) — 공간 차원 축소

Max Pooling, Average Pooling을 GPU에서 효율적으로 수행합니다. 이미지를 다운샘플링하면서 특징을 보존하는 역할을 합니다. 합성곱만큼 무겁진 않지만, 모든 CNN에서 빈번하게 사용되므로 잘 최적화되어 있어야 합니다.

  • CUDNN_POOLING_MAX — 최댓값 풀링
  • CUDNN_POOLING_AVERAGE_COUNT_INCLUDE_PADDING — 패딩 포함 평균
  • CUDNN_POOLING_AVERAGE_COUNT_EXCLUDE_PADDING — 패딩 제외 평균
  • CUDNN_POOLING_MAX_DETERMINISTIC — 결정적(deterministic) 버전, 재현성이 필요할 때

cudnnActivationForward / Backward

활성화 함수(Activation) — 비선형성 부여

ReLU, Sigmoid, Tanh, ELU, Softplus 같은 elementwise 비선형 함수를 GPU에서 처리합니다. 연산 자체는 단순하지만 텐서가 크기 때문에 메모리 대역폭이 병목입니다. cuDNN은 이 작업을 다른 연산(예: convolution + bias + ReLU)과 퓨전(fusion)해서 메모리 왕복을 줄이는 최적화를 제공합니다.

Fused operation의 위력

Conv → Bias → ReLU를 따로 호출하면 GPU 메모리에 세 번 쓰고 읽지만, fused 버전은 한 번의 패스로 끝납니다. 이게 추론 속도 향상의 큰 비결입니다.

cudnnBatchNormalization / LayerNormalization

정규화(Normalization)

Batch Normalization: 학습 안정성과 속도를 크게 향상시킨 기법. cuDNN은 forward/backward 모두를 단일 커널로 처리합니다.

Layer Normalization: 트랜스포머 시대의 표준 정규화. cuDNN 8 이후 본격 지원되고, 트랜스포머 워크로드에서 핵심적입니다.

Group Normalization, Instance Normalization도 최신 버전에서 지원됩니다.

cudnnRNN* / cudnnLSTMForward

순환 신경망(RNN/LSTM/GRU)

RNN 계열은 시퀀스를 따라 순차적으로 계산해야 해서 GPU 병렬화가 어려운 영역입니다. cuDNN은 여러 시점/여러 레이어의 RNN을 한 번에 묶어 실행하는 fused 구현을 제공합니다. 트랜스포머 등장 이후 사용 빈도는 줄었지만, 음성 인식, 시계열 예측에서는 여전히 중요합니다.

실전 팁

PyTorch에서 nn.LSTM(..., batch_first=True)가 자동으로 cuDNN의 fused LSTM 커널을 호출합니다. 직접 for loop으로 LSTM cell을 굴리는 것보다 수십 배 빠릅니다.

cudnnMultiHeadAttn / Scaled Dot-Product Attention

어텐션(Attention) — 트랜스포머의 심장

cuDNN 8 후반부부터 Multi-Head Attention과 Scaled Dot-Product Attention(SDPA)이 정식 지원됩니다. cuDNN 9에서는 FlashAttention 스타일의 fused attention이 표준화되어, LLM 학습/추론에서 메모리 효율과 속도 모두 큰 폭의 개선을 가져왔습니다.

  • Causal mask, padding mask, ALiBi bias 등 다양한 변형 지원
  • FP16/BF16/FP8 데이터 타입 최적화 — Hopper의 FP8 텐서 코어 활용
  • 긴 시퀀스에서도 O(N) 메모리로 동작 (FlashAttention 알고리즘)
LLM 시대의 cuDNN

"cuDNN은 CNN용"이라는 인상은 옛말입니다. 현재 cuDNN의 가장 활발한 최적화 영역은 트랜스포머와 어텐션이며, PyTorch의 F.scaled_dot_product_attention은 이미 cuDNN 백엔드를 사용합니다.

cuDNN Graph API (v8.x+)

그래프 API — 연산을 묶어서 더 빠르게

전통적인 cuDNN은 "한 번에 한 연산"을 호출하는 방식이었습니다. 하지만 최신 GPU에서는 여러 연산을 묶어 한 번에 실행하는 게 훨씬 효율적입니다. cuDNN Graph API는 사용자가 연산 그래프를 정의하면, cuDNN이 자동으로 fusion하고 최적의 실행 계획을 생성합니다.

  • 예: Conv → Bias → BatchNorm → ReLU를 단일 fused 커널로 컴파일
  • JIT 컴파일과 휴리스틱 기반 알고리즘 선택
  • 커널 호출 오버헤드를 제거 → 작은 배치/짧은 시퀀스에서 큰 이득

PyTorch의 torch.compile이나 TensorRT 같은 컴파일러형 백엔드가 내부적으로 cuDNN Graph API를 적극 활용합니다.

프레임워크는 cuDNN을 어떻게 쓸까

우리는 cuDNN을 직접 호출할 일은 거의 없습니다. 하지만 어떻게 사용되는지 알면 "왜 첫 epoch가 느리다가 두 번째부터 빨라지지?" 같은 현상을 이해할 수 있습니다.

PyTorch에서 cuDNN 컨트롤하기

PyTorch는 cuDNN 사용 여부와 동작 방식을 다음 플래그로 제어합니다.

Python · PyTorchtorch.backends.cudnn
import torch

# 1. cuDNN 사용 자체를 켜고 끄기 (기본: True)
torch.backends.cudnn.enabled = True

# 2. 알고리즘 자동 탐색 (benchmark 모드)
#    같은 입력 크기가 반복되면 첫 회 여러 알고리즘을 측정 후 가장 빠른 것 선택
torch.backends.cudnn.benchmark = True

# 3. 결정적(deterministic) 동작 보장 — 재현 가능한 실험용
#    성능은 살짝 떨어지지만 항상 같은 결과
torch.backends.cudnn.deterministic = True

# 4. TF32 허용 여부 (Ampere 이상에서 FP32를 TF32로 가속)
torch.backends.cudnn.allow_tf32 = True
실전 팁 — benchmark의 양면성

cudnn.benchmark = True는 입력 크기가 일정한 학습 루프(이미지 분류 등)에서 큰 속도 향상을 줍니다. 하지만 입력 크기가 자주 바뀌면(가변 길이 시퀀스 등) 매번 새 알고리즘을 탐색하느라 오히려 느려집니다. 워크로드에 맞게 켜고 끄세요.

TensorFlow / Keras

TensorFlow는 GPU를 사용하면 자동으로 cuDNN을 활용합니다. 사용자가 명시적으로 제어할 일은 거의 없습니다. 환경 변수 TF_CUDNN_DETERMINISTIC=1로 결정적 동작을 강제할 수 있습니다.

JAX

JAX도 XLA 컴파일러를 통해 cuDNN을 호출합니다. 트랜스포머 어텐션 같은 연산은 cuDNN의 fused kernel로 매핑됩니다. XLA 자체가 패턴 매칭으로 cuDNN 호출을 생성하므로, 사용자 코드는 거의 그대로입니다.

자주 받는 질문

"PyTorch에서 cuDNN 버전을 어떻게 확인하지?"
print(torch.backends.cudnn.version()) — 정수로 반환됩니다. 예: 90100은 cuDNN 9.1.0.

[ 06 / 08 ]

cuDNN이 쓰이는 실제 분야

cuDNN은 우리가 의식하지 못한 채 거의 모든 딥러닝 워크로드에서 동작합니다. 구체적인 활용 영역을 살펴봅시다.

CV

컴퓨터 비전

ResNet, EfficientNet 같은 CNN 학습/추론. 객체 탐지(YOLO, DETR), 세그멘테이션(U-Net, Mask R-CNN), 이미지 분류 — Convolution 커널이 핵심.

LLM

대규모 언어 모델

GPT, Llama, Qwen 등 트랜스포머 기반 모델의 학습과 추론. Fused Attention, LayerNorm, GEMM 최적화가 학습 시간을 좌우합니다.

SR

음성 인식

Whisper, Conformer 같은 모델. RNN-T, 어텐션 기반 ASR 모두 cuDNN의 RNN/Attention 커널을 활용합니다.

GEN

생성형 AI

Stable Diffusion, Flux 같은 디퓨전 모델. U-Net의 Convolution, Cross-Attention 모두 cuDNN에서 가속됩니다.

RL

강화학습

AlphaGo, AlphaFold, 로보틱스 — 정책망/가치망의 forward/backward에서 cuDNN이 동작합니다.

REC

추천 시스템

DLRM, Wide & Deep 같은 대규모 추천 모델. 임베딩 룩업과 MLP 부분에서 cuDNN(또는 cuBLAS와 함께)이 작동합니다.

BIO

과학·바이오

AlphaFold 단백질 구조 예측, 분자 시뮬레이션, 의료 영상(2D/3D 합성곱) — 3D conv에 특화된 cuDNN 커널이 필수.

EDGE

엣지/추론 최적화

TensorRT가 cuDNN을 백엔드로 활용해 모델을 더 빠른 형태로 컴파일합니다. Jetson, DGX, 클라우드 GPU 모두 동일.

왜 이렇게 빠른가

cuDNN의 속도는 단순히 "GPU를 쓰니까"가 아닙니다. 여러 최적화 기법이 누적된 결과입니다.

①

하드웨어 직접 최적화

NVIDIA 내부 엔지니어가 SM(Streaming Multiprocessor), 텐서 코어, 메모리 계층을 정확히 알고 커널을 작성합니다.

②

알고리즘 자동 선택

같은 합성곱이라도 입력 크기에 따라 IMPLICIT_GEMM, Winograd, FFT 중 최적의 것을 선택합니다.

③

커널 퓨전(Fusion)

Conv + Bias + ReLU 같은 연산을 한 번의 GPU 호출로 묶어 메모리 왕복을 제거합니다.

④

저정밀도 가속

FP16, BF16, FP8, INT8을 활용해 텐서 코어의 처리량을 극대화합니다. 최신 Hopper GPU의 FP8은 FP16 대비 2배 빠릅니다.

이해도 확인 퀴즈

여기까지 읽은 내용을 짧은 퀴즈로 정리해 봅시다. 클릭해서 답을 선택하세요.

Q1cuDNN과 CUDA의 관계를 가장 정확히 설명한 것은?
AcuDNN은 CUDA의 대체재이며, cuDNN만 설치하면 된다
BCUDA는 cuDNN의 일부이며, cuDNN 안에 CUDA가 포함되어 있다
CcuDNN은 CUDA 위에서 동작하는 딥러닝 전용 고수준 라이브러리다
D둘은 완전히 별개의 기술이며 서로 무관하다
정답: C. CUDA는 범용 GPU 컴퓨팅 플랫폼이고, cuDNN은 그 위에서 동작하는 딥러닝 도메인 특화 라이브러리입니다. 딥러닝 환경에는 둘 다 필요합니다.
Q2다음 중 cuDNN이 직접 제공하는 연산이 아닌 것은?
A2D Convolution forward/backward
B데이터셋의 셔플링(shuffle)과 배치 로딩
CBatch Normalization
DMulti-Head Attention
정답: B. 데이터 로딩과 셔플링은 cuDNN의 영역이 아니라 프레임워크(예: PyTorch의 DataLoader)나 NVIDIA DALI 같은 별도 라이브러리의 역할입니다. cuDNN은 텐서 위에서의 신경망 연산에 집중합니다.
Q3torch.backends.cudnn.benchmark = True는 언제 가장 도움이 될까?
A입력 텐서 크기가 학습 내내 일정한 이미지 분류 작업
B매 배치마다 시퀀스 길이가 크게 달라지는 NLP 작업
C완전히 재현 가능한 결과가 필요한 학술 실험
DCPU만 사용하는 환경
정답: A. benchmark 모드는 첫 호출 시 여러 알고리즘을 시험해 최적을 캐싱합니다. 입력 크기가 일정하면 캐시 적중률이 높아 큰 속도 향상을 얻지만, 크기가 자주 바뀌면 매번 탐색해 오히려 느려집니다. 재현성이 중요하면 deterministic=True를 써야 합니다.
Q4cuDNN이 같은 합성곱 연산에 대해 여러 알고리즘(IMPLICIT_GEMM, Winograd, FFT 등) 중 하나를 선택하는 이유는?
A알고리즘마다 결과가 미세하게 다르기 때문
BNVIDIA가 알고리즘을 무작위로 선택하기 때문
C사용자에게 선택의 자유를 주기 위해
D입력 크기·필터 크기·데이터 타입에 따라 최적 알고리즘이 다르기 때문
정답: D. 작은 3×3 필터에는 Winograd가, 큰 필터에는 FFT가, 일반적인 경우엔 IMPLICIT_GEMM이 유리합니다. cuDNN은 휴리스틱과 벤치마크로 가장 빠른 알고리즘을 선택합니다.
Q5최신 LLM 시대에 cuDNN의 역할이 가장 두드러진 영역은?
A합성곱(Convolution)만이 여전히 유일한 핵심 영역이다
BcuDNN은 LLM에는 거의 쓰이지 않는다
CFused Attention, LayerNorm, GEMM 최적화로 트랜스포머 워크로드를 가속한다
DcuDNN은 추론에만 쓰이고 학습에는 사용되지 않는다
정답: C. cuDNN 9는 FlashAttention 스타일의 fused attention, 정규화, 행렬곱 최적화를 통해 LLM의 학습과 추론 모두에서 핵심적인 역할을 합니다. "cuDNN = CNN용"은 더 이상 맞지 않는 인상입니다.
마무리

cuDNN은 우리가 일상적으로 직접 다루지는 않지만, 작성한 모든 모델의 속도와 효율성을 좌우하는 딥러닝 스택의 가장 깊은 핵심 계층입니다. CUDA와의 관계, 주요 컴포넌트, 프레임워크에서의 동작 방식을 알면 — 모델이 느릴 때 어디를 의심해야 할지, 어떤 플래그를 켜야 할지, 새 GPU의 어떤 기능을 활용해야 할지가 보입니다.
