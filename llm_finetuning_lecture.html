CHAPTER 01

Fine-tuning 이란 무엇인가

Fine-tuning은 이미 막대한 양의 텍스트로 사전 학습(Pre-training)된 거대 언어 모델을, 훨씬 작은 양의 특정 데이터로 한 번 더 학습시켜 우리의 목적에 맞게 “미세하게 조정”하는 기법입니다. 모델을 처음부터 학습시키는 것이 아니라, 이미 똑똑한 모델을 특정한 방향으로 다듬는 작업이라는 점이 핵심입니다.

대학원 졸업생을 신입사원으로 채용한 상황을 떠올려 보세요. 그 사람은 이미 일반 지식과 글쓰기·논리·언어 능력을 갖추고 있습니다 (= 사전 학습). 그러나 우리 회사의 사내 용어, 보고서 양식, 고객 응대 톤은 모릅니다. 그래서 우리는 그를 회사 자료로 며칠~몇 주간 교육시킵니다. 이 단기 교육이 바로 Fine-tuning입니다 — 이미 갖춰진 일반 능력 위에 우리 도메인의 특수성을 얹는 과정입니다.

LLM 학습의 3단계

STAGE 01
Pre-training
수조 토큰의 인터넷 텍스트로 “다음 토큰 예측”을 학습. 수백~수천 장의 GPU와 수개월이 필요한 단계. 결과물 = Base Model (예: Llama 3 Base, Qwen2.5 Base)
STAGE 02
Fine-tuning (SFT)
특정 작업·도메인에 맞는 데이터(보통 수천~수십만 건)로 추가 학습. 대화 형식·도메인 지식·스타일을 주입. 결과물 = Instruct/Chat Model
STAGE 03
Alignment
RLHF·DPO 등으로 모델의 응답을 “사람이 선호하는 방향”으로 정렬. 유해성·정직성·도움됨을 향상. 결과물 = 우리가 일상적으로 쓰는 ChatGPT·Claude 같은 모델.
용어 정리
좁은 의미의 Fine-tuning은 STAGE 02(SFT)만 가리키지만, 넓은 의미로는 STAGE 02 + 03 전체를 묶어 “Post-training”이라고도 부릅니다. 이 강의에서는 두 단계를 모두 다룹니다.

왜 Fine-tuning이 필요한가

사전 학습된 모델은 “일반인”입니다. 다음과 같은 경우에 Fine-tuning이 강력해집니다.

USE CASE 01
도메인 적응

의료·법률·금융·반도체 등 일반 모델이 모르는 전문 용어와 추론 패턴을 주입합니다.

USE CASE 02
출력 형식 통제

특정 JSON 스키마, 보고서 양식, 회사 톤앤매너로 항상 응답하도록 만듭니다.

USE CASE 03
새 능력 부여

Function calling, tool use, 새로운 언어, 도메인 특화 추론 패턴을 학습시킵니다.

USE CASE 04
비용·지연·프라이버시

작은 모델을 fine-tune하여 큰 API를 대체 → 추론 비용·지연·데이터 유출 위험을 줄입니다.

CHAPTER 02

언제 Fine-tuning을 써야 하는가

“모델이 우리가 원하는 답을 안 해요” 라는 문제에는 항상 세 가지 도구가 있습니다. Prompt Engineering · RAG · Fine-tuning. 이 세 가지는 경쟁 관계가 아니라 해결하는 문제의 종류가 다른 도구입니다. 아래 탭으로 각 도구의 강점·약점·궁합을 비교해 봅시다.

핵심 아이디어

모델 가중치는 그대로 두고, 입력 프롬프트만 정교하게 설계해서 원하는 출력을 끌어냅니다. Few-shot 예시, role/system 지시, Chain-of-Thought 등이 여기에 속합니다.

강점

  • 비용 0원, 즉시 적용 가능
  • 모델 학습 자원·인프라 불필요
  • 실험·반복이 가장 빠름

한계

  • 긴 프롬프트는 추론 비용·지연을 늘림
  • 모델의 “기본 성격”을 바꾸진 못함
  • 대규모로 일관된 형식 강제가 어려움

이럴 때 적합

새로운 작업의 PoC, 간단한 일회성 작업, 명확한 지시로 해결 가능한 문제.

핵심 아이디어

모델은 그대로, 외부 지식 베이스에서 관련 문서를 검색해 프롬프트에 끼워 넣습니다. “모델 + 검색 엔진” 구조입니다.

강점

  • 최신·사실 기반 응답 (사내 위키, 매뉴얼 등)
  • 출처(citation) 제공이 쉬움
  • 문서가 바뀌면 즉시 반영 — 재학습 불필요

한계

  • 모델의 추론 패턴·스타일은 바꾸지 못함
  • 검색 품질이 응답 품질의 상한선
  • 벡터 DB·임베딩·청킹 인프라 필요

이럴 때 적합

지식이 자주 바뀌는 도메인, 사실성·출처가 중요한 응답, 사내 문서 Q&A.

핵심 아이디어

모델 가중치 자체를 추가 학습해, 모델의 “본능”을 바꿉니다. 도메인 추론 패턴, 스타일, 출력 형식이 가중치에 새겨집니다.

강점

  • 프롬프트 없이도 일관된 톤·형식
  • 도메인 특화 추론 능력 향상
  • 작은 모델로 큰 모델 수준 응답 가능 (비용 절감)

한계

  • GPU·시간·전문 인력 필요
  • 지식 업데이트마다 재학습 필요
  • 잘못 학습하면 능력이 “퇴화”할 수 있음 (catastrophic forgetting)

이럴 때 적합

특정 형식의 대규모 생성, 스타일 일관성, 도메인 추론, 작은 모델로 비용 최적화.

CHAPTER 03

학습 패러다임: SFT · RLHF · DPO

Fine-tuning은 “무엇을 학습하느냐”에 따라 여러 종류로 나뉩니다. 가중치를 어떻게 업데이트할지(=PEFT 기법)와는 별개의 축입니다. 여기서는 가장 중요한 세 가지 학습 신호 — 정답 모방(SFT), 사람의 선호 강화(RLHF), 선호 직접 최적화(DPO) — 를 살펴봅니다.

SFT (Supervised Fine-Tuning)

가장 기본적이고 가장 흔한 방식. (입력, 정답 출력) 쌍을 모델에게 보여주고, 모델이 그 정답을 그대로 생성하도록 cross-entropy loss로 학습합니다. “모범 답안을 따라하라”는 지도학습입니다.

SFT 데이터 형식 (ShareGPT / Alpaca 스타일) 
{
  "instruction": "다음 한국어 문장을 영어로 번역하세요.",
  "input": "오늘 회의는 30분 지연될 예정입니다.",
  "output": "Today's meeting will be delayed by 30 minutes."
}

# 또는 대화 형식
{
  "messages": [
    {"role": "system", "content": "당신은 친절한 어시스턴트입니다."},
    {"role": "user", "content": "Kubernetes Pod와 Container의 차이는?"},
    {"role": "assistant", "content": "Pod는 하나 이상의 컨테이너를 묶는..."}
  ]
}

Instruction Tuning은 SFT의 특수 케이스로, 다양한 작업의 “지시문 + 정답” 쌍을 대량으로 학습시켜 모델이 새로운 지시도 잘 따르도록 만드는 단계입니다. ChatGPT, Llama-Instruct, Qwen-Instruct 등이 모두 이 과정을 거칩니다.

RLHF (Reinforcement Learning from Human Feedback)

SFT만으로는 “정답이 여러 개”인 상황을 다루기 어렵습니다. 예: 같은 질문에 대해 모델이 두 개의 답변 A, B를 만들 때, 어느 쪽이 “더 좋은지”는 라벨러의 선호도(preference)로만 정의됩니다. RLHF는 이 선호 데이터를 강화학습으로 모델에 주입합니다.

PHASE 01
SFT 베이스 만들기
먼저 instruction-tuning된 모델 πSFT를 준비합니다.
PHASE 02
Reward Model 학습
사람이 (응답 A vs B) 중 어느 쪽을 선호했는지 라벨링한 데이터로, 응답 점수를 매기는 보상 모델 rφ를 학습합니다.
PHASE 03
PPO로 정책 최적화
모델이 보상을 최대화하면서도 πSFT에서 너무 벗어나지 않도록 KL 페널티를 걸고 강화학습합니다.
maxπ 𝔼[ rφ(x, y) − β · KL( π(y|x) ∥ πSFT(y|x) ) ]
RLHF의 어려움
Reward Model 학습 + PPO 강화학습 + 4개 모델(policy, ref, reward, value)을 동시에 메모리에 올려야 해서 구현이 매우 복잡하고 불안정합니다. 하이퍼파라미터에 민감하고 reward hacking(보상 해킹)도 자주 일어납니다.

DPO (Direct Preference Optimization)

RLHF의 복잡성을 해결한 2023년의 혁신. 핵심 통찰은 “보상 모델이 명시적으로 필요 없다”는 것입니다. 선호 데이터 (x, ychosen, yrejected) 만 있으면, 이를 직접 분류 손실 형태로 변환해 강화학습 없이 SFT처럼 학습할 수 있습니다.

ℒDPO = − log σ ( β · log [ πθ(yw|x)/πref(yw|x) ] − β · log [ πθ(yl|x)/πref(yl|x) ] )

해석은 단순합니다: “선호된 응답의 확률은 올리고, 거절된 응답의 확률은 내려라. 단 reference 모델 대비 비율로.”

DPO 외에도 IPO, KTO, ORPO, SimPO 등 변형들이 활발히 제안되고 있고, 수학·코딩 추론 능력 강화에는 GRPO(Group Relative Policy Optimization, DeepSeek-R1)나 RFT(Reinforcement Fine-Tuning, OpenAI)처럼 검증 가능한 보상을 쓰는 RL 방법이 주목받고 있습니다.
CHAPTER 04

Full Fine-tuning의 한계

가장 단순한 fine-tuning은 모델의 모든 파라미터를 학습시키는 것입니다(Full FT). 이론적으로 가장 강력하지만, LLM 시대에는 실용적으로 거의 불가능해졌습니다. 무엇이 문제일까요?

메모리의 벽

7B 파라미터 모델을 FP16(2바이트)으로 학습한다고 가정해 봅시다. 메모리는 단순히 “모델 크기 × 2” 가 아닙니다. 옵티마이저(Adam) 상태와 그라디언트까지 합산해야 합니다.

항목7B 모델 (FP16/FP32 mixed)설명
모델 가중치14 GBFP16 × 7B
그라디언트14 GB가중치와 동일 크기
Adam 1st moment (m)28 GBFP32 × 7B
Adam 2nd moment (v)28 GBFP32 × 7B
활성화(activations)10~30 GBbatch size·seq len에 따라
합계약 100 GB+A100 80GB 단일 GPU로 학습 불가

파라미터 효율의 직관

“모든 가중치를 학습해야 정말 좋아질까?” 라는 질문에서 PEFT가 시작됩니다. 경험적으로 fine-tuning 시 가중치 변화 ΔW = Wfinetuned − Wpretrained는 원본 W에 비해 매우 저차원(low-rank)이라는 사실이 알려져 있습니다. 즉 “변화량”이 모델의 풀 랭크 전체에 퍼져 있지 않다는 것입니다. 이 통찰이 LoRA의 출발점입니다.

아래 계산기로 모델 크기·학습 방식별 학습 가능 파라미터 수를 직접 비교해 보세요.

※ LoRA 파라미터 수는 Transformer attention의 Q·K·V·O 행렬에만 적용(전형적인 설정)했을 때의 근사치입니다. Prompt Tuning은 가상 토큰 20개를 가정했습니다.

결론
LoRA는 학습 가능 파라미터를 보통 0.1% ~ 1% 수준으로 줄이면서도, 많은 작업에서 Full Fine-tuning과 거의 동등한 성능을 냅니다. 그래서 오늘날 “fine-tuning” = “LoRA 계열”이라고 봐도 무방할 정도입니다.
CHAPTER 05

PEFT 핵심: LoRA

LoRA (Low-Rank Adaptation)는 2021년 마이크로소프트에서 제안된 PEFT 기법으로, 오늘날 사실상 모든 오픈소스 fine-tuning의 표준입니다. 그 핵심 아이디어는 놀라울 만큼 단순합니다: “학습할 때는 가중치 변화량 ΔW를 두 개의 작은 행렬의 곱으로 근사하자.”

수학적 원리

Transformer의 어떤 가중치 행렬 W ∈ ℝd×k가 있다고 합시다. Fine-tuning은 W를 W + ΔW로 바꾸는 작업입니다. LoRA는 ΔW를 두 개의 저차원 행렬로 분해합니다:

W' = W + ΔW = W + B · A,   B ∈ ℝd×r,  A ∈ ℝr×k,  r ≪ min(d, k)

학습 시 W는 동결(frozen)하고, 작은 행렬 A, B만 업데이트합니다. 추론 시에는 BA를 계산해 W에 더해버리면(merge) 원래 모델과 동일한 구조가 되므로 추론 지연이 전혀 없습니다.

직접 만져보는 LoRA

왜 작동하는가

LoRA의 핵심 가설은 “fine-tuning 시 가중치 변화는 본질적으로 저차원(low intrinsic rank)이다”입니다. 예를 들어 4096×4096 행렬의 변화량이라도, 그 변화가 r=8 차원의 subspace 안에서 일어난다면 BA 분해로 거의 완벽히 표현할 수 있다는 뜻입니다. 실험적으로 r=4~16 사이에서도 Full FT와 거의 동등한 성능이 나옵니다.

하이퍼파라미터 가이드

RANK · r
4 ~ 64

대부분의 작업에서 r=8 또는 r=16으로 시작. 더 어렵고 큰 변화가 필요한 작업(예: 새 언어 학습)은 r=32~64.

ALPHA · α
α = 2r

실제 가중치는 (α/r)·BA 만큼 더해집니다. α=2r이 기본 권장. α를 크게 하면 LoRA 영향력이 커집니다.

TARGET MODULES
q_proj, v_proj 우선

Attention의 Query·Value 투영에 LoRA를 적용. 더 강한 학습이 필요하면 K, O, MLP까지 확장(all-linear).

DROPOUT
0.05 ~ 0.1

과적합 방지용. 데이터가 많을 땐 0, 적을 땐 0.1까지.

CHAPTER 06

QLoRA · 4-bit 시대의 표준

LoRA가 “학습 가능 파라미터”를 줄였다면, QLoRA(2023, University of Washington)는 한 발 더 나아가 베이스 모델 자체를 4-bit로 양자화해 메모리를 또 한 번 1/4로 줄였습니다. 이 기술 덕분에 단일 24GB 소비자 GPU에서도 7B–13B 모델 fine-tuning이 가능해졌습니다.

QLoRA의 세 가지 혁신

INNOVATION 01
4-bit NormalFloat (NF4)

정규분포를 따르는 가중치에 최적화된 4-bit 데이터 타입. INT4보다 정보 손실이 적습니다. 가중치 값의 분위수(quantile) 기반으로 양자화 격자를 설계합니다.

INNOVATION 02
Double Quantization

양자화 시 필요한 “스케일 상수”까지 한 번 더 양자화. 추가로 약 0.37 bits/parameter를 절약합니다.

INNOVATION 03
Paged Optimizers

GPU 메모리가 부족할 때 옵티마이저 상태를 자동으로 CPU RAM으로 옮기는 NVIDIA Unified Memory 활용. OOM을 막아줍니다.

INNOVATION 04
학습은 BF16, 저장은 4-bit

역전파 시에만 dequantize → BF16 계산 → 그라디언트는 LoRA의 A, B만. 추론 정확도는 거의 손상 없음.

주의
QLoRA는 학습 속도는 LoRA보다 다소 느립니다(역전파 시 dequantize 비용). 또한 4-bit 양자화로 인한 미세한 성능 저하가 있을 수 있어, 매우 민감한 작업(예: 수치 추론)에서는 LoRA + BF16을 권장합니다.
CHAPTER 07

기타 PEFT 기법들

LoRA가 압도적으로 많이 쓰이지만, PEFT 가족에는 다른 기법들도 있습니다. 각각은 다른 관점에서 “어디에 학습 가능한 파라미터를 끼워 넣을 것인가”를 답합니다. 핵심만 빠르게 짚어봅시다.

Adapter Tuning

LoRA보다 먼저 등장한 PEFT의 원조(2019, Houlsby et al.). Transformer 각 블록 사이에 작은 bottleneck MLP(어댑터)를 끼워 넣고, 어댑터만 학습합니다. 구조: down_projection → nonlinearity → up_projection + residual.

장점: 단순·강력. 단점: 추론 시에도 어댑터가 그대로 남아 추론 지연이 약간 증가합니다. 이것이 LoRA가 결국 더 인기를 얻은 이유입니다.

Prompt Tuning

모델은 완전히 동결하고, 입력 임베딩 앞에 학습 가능한 “가상 토큰” 임베딩을 몇 개(보통 20개) 붙입니다. 학습되는 건 이 가상 토큰의 임베딩 벡터들 뿐이라 파라미터가 극도로 적습니다(수만 개 수준).

장점: 가장 가벼움. 단점: 모델이 매우 클 때만 잘 작동(10B+에서 효과적). 작은 모델에선 표현력이 부족합니다.

Prefix Tuning / P-Tuning v2

Prompt Tuning이 입력에만 가상 토큰을 붙인다면, Prefix Tuning은 모든 Transformer 레이어의 attention key/value에 학습 가능한 prefix를 붙입니다. 더 많은 위치에 개입하므로 표현력이 좋고, 중·소형 모델에도 잘 작동합니다.

IA³

Infused Adapter by inhibiting and Amplifying inner Activations. Attention의 key·value 와 MLP의 중간 활성화에 학습 가능한 벡터를 곱해 “확대/축소”만 시킵니다. 파라미터가 LoRA보다도 적습니다.

실무 권장
잘 모르겠다면 LoRA(또는 QLoRA)로 시작하세요. 오픈소스 생태계의 도구·튜토리얼·체크포인트가 압도적으로 LoRA 중심입니다. Prompt/Prefix Tuning은 “여러 작업에 대응하는 어댑터를 한 모델에 잔뜩 붙이고 싶을 때” 정도에서 고려합니다.

최신 변형: DoRA, rsLoRA, LoRA+

2024년 이후로도 LoRA를 개선한 변형이 활발히 등장하고 있습니다.

  • DoRA (Weight-Decomposed LoRA) — 가중치를 “방향(magnitude) × 크기(direction)” 으로 분해하고 방향에만 LoRA 적용. Full FT와의 격차를 더 줄임.
  • rsLoRA — α 스케일링을 √r 기반으로 바꿔 고-rank에서도 안정적 학습.
  • LoRA+ — A와 B에 서로 다른 학습률을 줘서 수렴 개선.
  • VeRA — A, B 행렬을 모든 레이어가 공유하고 작은 스케일 벡터만 학습. 파라미터 추가 절감.
CHAPTER 08

데이터 · 평가 · 도구

알고리즘은 절반에 불과합니다. 실전 fine-tuning의 품질은 데이터 품질에서 80%가 결정되고, 재현성·이력 관리는 MLOps 도구에서 결정됩니다. 이 챕터는 학습 “외부”의 핵심을 다룹니다.

데이터 준비 원칙

  1. 양보다 질. 잘 정제된 1,000건이 노이즈 섞인 100,000건보다 낫습니다. LIMA(Less Is More for Alignment) 연구가 보여준 사실입니다.
  2. 일관된 형식. 시스템 프롬프트, 응답 스타일, 종결 토큰을 학습 데이터 전반에 걸쳐 통일하세요.
  3. 다양성. 같은 패턴 반복은 overfitting을 부릅니다. 의도(intent), 길이, 난이도를 분산시킵니다.
  4. 대표성. 실제 서빙 시 들어올 입력 분포와 학습 분포가 비슷해야 합니다.
  5. 안전성 데이터 혼합. 일반 instruction 데이터 일부(예: 10%)를 섞어 학습해야 “바보가 되는” catastrophic forgetting을 막을 수 있습니다.

평가 방법

METHOD 01
자동 벤치마크

MMLU, GSM8K, HumanEval, KoBEST, KMMLU 등. 일반 능력 보존 여부 확인용. 도메인 학습 후 일반 능력이 얼마나 떨어졌는지 모니터링.

METHOD 02
도메인 홀드아웃 셋

학습에 쓰지 않은 도메인 평가 데이터로 정확도·BLEU·ROUGE 측정. 가장 직접적인 지표.

METHOD 03
LLM-as-a-Judge

GPT-4·Claude 등 강력한 모델로 응답 품질을 페어와이즈 비교. 빠르지만 편향(verbosity bias 등) 주의.

METHOD 04
사람 평가

가장 신뢰할 수 있지만 가장 비쌈. 최종 출시 전 샘플 50~200건에 대해 라벨러 검토 권장.

Kubernetes 환경에서의 인프라 팁

로컬 클러스터에서 LLM fine-tuning을 운영할 때 자주 만나는 실무 포인트입니다.

  • NCCL 통신 — 다중 GPU/노드 학습 시 InfiniBand나 최소 100GbE가 권장. RDMA 미지원이면 학습이 통신 병목에 막힙니다.
  • NVIDIA GPU Operator — 드라이버·CUDA·MIG·DCGM exporter를 클러스터 차원에서 관리. GPU 노드의 표준 셋업입니다.
  • 스토리지 — 체크포인트가 수십~수백 GB. NFS도 가능하지만, 가능하면 NVMe 기반 분산 파일시스템(JuiceFS, Ceph) 권장.
  • 오케스트레이션 — 학습 잡 자체는 Kubeflow Training Operator, Volcano, KubeRay 중 선택. 하이퍼파라미터 탐색은 Argo Workflows·Katib과 결합.
  • 실험 추적 — MLflow, Weights & Biases로 하이퍼파라미터·메트릭·체크포인트 버전 관리. fine-tuning은 한 번 잘못 돌리면 며칠을 날리므로 추적이 필수.
  • 이종(heterogeneous) 노드 — ARM64 + x86_64 혼합 클러스터라면, 학습 컨테이너는 멀티아키 빌드(docker buildx) 후 nodeSelector로 GPU 노드에 명시적으로 스케줄.

CHAPTER 11

정리 및 다음 단계

여기까지 따라오셨다면 이제 “Fine-tuning이 뭐냐”는 질문에 자신 있게 답하실 수 있을 것입니다. LLM fine-tuning의 두 축 — “학습 신호” × “파라미터 효율” — 만 기억하면, 새로 등장하는 어떤 기법도 이 좌표 위에 위치시킬 수 있습니다.

핵심 정리 10줄

  1. Fine-tuning은 사전 학습된 모델을 우리 목적에 맞게 추가 학습하는 작업.
  2. Prompt Engineering / RAG / Fine-tuning은 경쟁이 아닌 보완 관계.
  3. 학습 신호에 따라 SFT (정답 모방) / RLHF·DPO (선호 정렬) 로 나뉜다.
  4. Full Fine-tuning은 메모리 문제로 LLM 시대에는 비현실적.
  5. PEFT는 “학습 가능 파라미터를 1% 미만으로 줄이는” 기법들의 총칭.
  6. LoRA: ΔW를 두 저차원 행렬 BA의 곱으로 근사. 사실상 표준.
  7. QLoRA: LoRA + 4-bit 양자화로 소비자 GPU에서도 7B–13B 학습.
  8. DPO는 reward model 없이 선호 데이터를 직접 최적화 → RLHF의 실용적 대체.
  9. 품질의 80%는 데이터에서, 20%는 알고리즘에서 결정된다.
  10. 학습은 한 번이 아니라 반복이다. 실험 추적·재현성·평가 파이프라인이 곧 경쟁력.
