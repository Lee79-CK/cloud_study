Java의 진화를 한눈에

Java는 6개월마다 새 버전이 나오지만, 2~3년에 한 번 LTS (Long Term Support, 장기 지원 버전)가 발표됩니다. 기업에서 안정적으로 쓸 수 있는 버전은 LTS 뿐. 그래서 우리는 8 → 11 → 17 → 21 이렇게 LTS 라인을 따라가게 됩니다.

JAVA 8 · 2014
Lambda & Stream API
반복문 대신 흐름(stream)으로 데이터를 다루는 시대
REVIEW

Lambda는 "이름 없는 함수"를 한 줄로 표현하는 문법입니다. Stream은 그런 람다를 이용해 데이터를 가공하는 파이프라인이고요.

java · 이름 목록에서 5글자 이상만 골라 대문자로 Java 8+ 
List<String> names = List.of("alice", "bob", "charlie", "dan");

List<String> result = names.stream()
    .filter(n -> n.length() >= 5)        // 람다 표현식
    .map(String::toUpperCase)            // 메서드 참조
    .toList();                              // Java 16+에서 추가

// 결과: [ALICE, CHARLIE]
기억할 점

Stream의 흐름은 stream() → 중간연산(filter/map/...) → 종결연산(toList/collect/...) 순서입니다. 종결연산이 호출돼야 실제로 실행돼요 (lazy evaluation).

JAVA 8 · 2014
Optional
NullPointerException과의 영원한 작별을 위한 첫걸음
REVIEW

Optional<T>는 "값이 있을 수도, 없을 수도 있다"를 명시적으로 표현합니다. null을 직접 다루는 대신 안전한 래퍼로 감싸는 것이죠.

java · null 대신 Optional Java 8+ 
// ❌ 옛날 방식
User user = userRepo.findById(1);
if (user != null) {
    System.out.println(user.getName());
}

// ✅ Optional 방식
Optional<User> userOpt = userRepo.findById(1);
userOpt.map(User::getName)
       .ifPresent(System.out::println);

// 기본값이 필요할 때
String name = userOpt.map(User::getName)
                     .orElse("이름없음");
var
JAVA 11 · 2018 (실제로는 10부터)
지역 변수 타입 추론 (var)
반복적인 타입 선언을 줄이는 작지만 강력한 변화

JavaScript의 let이나 TypeScript의 const와 비슷해 보이지만, Java의 var는 여전히 정적 타입입니다. 컴파일러가 타입을 추론해줄 뿐, 한 번 정해진 타입은 바꿀 수 없어요.

java · 타입 선언 다이어트 Java 11 
// 😩 Java 8: 타입을 두 번씩 쓰는 고통
Map<String, List<User>> usersByDept = new HashMap<>();

// 😊 Java 11+: var로 깔끔하게
var usersByDept = new HashMap<String, List<User>>();

var count = 10;            // int로 추론
var name = "Claude";       // String으로 추론
var price = 9.99;          // double로 추론

// ⚠️ 주의: 초기값이 반드시 필요
var nothing;                // 컴파일 에러!
var x = null;             // 컴파일 에러! (타입 추론 불가)
언제 var를 쓰면 안 될까?

오른쪽 표현식만 봐서 타입이 한눈에 안 들어올 때는 var를 피하세요. 예를 들어 var result = service.process(data);는 result 타입을 알기 어렵죠. 가독성이 떨어지면 명시적 타입을 쓰는 게 낫습니다.

Java 17 — 데이터 클래스의 시대

Java 17이 정말 큰 영향을 미친 부분은 "코드를 짧고 표현력 있게"입니다. Record 하나만 봐도 이전엔 50줄 걸리던 클래스가 1줄이 됩니다. 패턴 매칭은 instanceof 캐스팅 지옥에서 우리를 구해주고요.

R
FEATURE 01
Record (레코드)
DTO를 한 줄로 — getter, equals, hashCode, toString 자동 생성
JAVA 17

Record는 "데이터만 담는 불변(immutable) 클래스"를 위한 새로운 키워드입니다. 한 줄만 쓰면 컴파일러가 다음을 자동으로 만들어줍니다:

  • 모든 필드의 private final 선언
  • 모든 필드를 받는 생성자
  • 각 필드의 접근자 (getter, 단 이름이 getName()이 아닌 name())
  • equals(), hashCode(), toString() 자동 구현
코드 비교 · DTO 만들기
동일한 User 데이터 클래스를 각 버전에서 만들 때
java · 약 40줄 😩Java 8
public class User {
    private final String name;
    private final int age;

    public User(String name, int age) {
        this.name = name;
        this.age = age;
    }

    public String getName() { return name; }
    public int getAge()    { return age; }

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof User)) return false;
        User user = (User) o;
        return age == user.age &&
               Objects.equals(name, user.name);
    }

    @Override
    public int hashCode() {
        return Objects.hash(name, age);
    }

    @Override
    public String toString() {
        return "User{name=" + name + ", age=" + age + "}";
    }
}
java · Lombok 사용 시Java 8
@Value  // @Getter, @AllArgsConstructor, @EqualsAndHashCode, @ToString 포함
public class User {
    String name;
    int age;
}

// 짧지만 외부 라이브러리(Lombok) 의존 + 어노테이션 프로세서 필요
java · 1줄! 라이브러리 의존 없음 ✨Java 17
public record User(String name, int age) {}

// 사용
var u = new User("Cheolkyu", 30);
u.name();    // "Cheolkyu" (getName 아님!)
u.age();     // 30
u.toString();// "User[name=Cheolkyu, age=30]"

Switch Expressions (스위치 표현식)
switch가 값을 반환할 수 있게 됐어요 + break 지옥 탈출
JAVA 17

옛 switch의 가장 큰 함정은 break를 빼먹으면 다음 케이스로 떨어진다(fall-through)는 것이었습니다. 새 switch는 화살표(->) 문법으로 이를 해결하고, 표현식으로 값까지 돌려줍니다.

요일에서 부서 코드로 매핑
java · break 빼먹기 쉬움Java 8
String type;
switch (day) {
    case "MON":
    case "TUE":
    case "WED":
    case "THU":
    case "FRI":
        type = "WEEKDAY";
        break;  // 잊으면 큰일
    case "SAT":
    case "SUN":
        type = "WEEKEND";
        break;
    default:
        type = "UNKNOWN";
}
java · 표현식 + 화살표 + 콤마Java 17
String type = switch (day) {
    case "MON", "TUE", "WED", "THU", "FRI" -> "WEEKDAY";
    case "SAT", "SUN"                       -> "WEEKEND";
    default                                  -> "UNKNOWN";
};

// 복잡한 로직은 블록으로, yield로 반환
int price = switch (size) {
    case "S" -> 5000;
    case "M" -> {
        System.out.println("기본 사이즈 선택");
        yield 8000;
    }
    default -> throw new IllegalArgumentException();
};
04 · JAVA 21

Java 21 — 동시성 혁명

Java 21의 진짜 주인공은 Virtual Threads(가상 스레드)입니다. 수십 년간 Java 개발자의 골치였던 "스레드 비용 문제"를 우아하게 해결합니다. LLM API 호출이나 마이크로서비스 같은 I/O 중심 워크로드에서 게임 체인저예요.

∞
FEATURE 06 · ⭐ 가장 중요
Virtual Threads (가상 스레드)
백만 개의 스레드를 거의 공짜로 만드는 마법
JAVA 21

핵심 한 줄: 운영체제 스레드(Platform Thread)는 비싸지만(약 1MB 메모리), 가상 스레드는 매우 가볍습니다(KB 단위). 같은 메모리로 수천 배 많은 동시 작업을 다룰 수 있어요.

비유로 이해하기

  • Platform Thread(전통적 스레드) = 각자 차를 가진 직원. 차 자체가 비싸고, 주차장도 좁음.
  • Virtual Thread(가상 스레드) = 공유 택시(Carrier Thread) 시스템. 필요할 때만 차를 잡고, 일이 끝나면 다른 사람에게 양보. 차는 그대로지만 처리할 수 있는 손님 수가 폭발적으로 늘어남.
Platform Thread (제한 200) 0 / 200
활성: 0 대기: 0 완료: 0
Virtual Thread (제한 없음) 0 / ∞
활성: 0 대기: 0 완료: 0

※ 1,000개의 HTTP 호출을 동시에 처리한다고 가정한 시각화입니다

코드로 보기

1,000개의 외부 API를 동시에 호출하기
java · 스레드풀로 분산Java 8
// 200개 스레드로 1000개 작업을 처리. 800개는 대기해야 함.
ExecutorService executor = Executors.newFixedThreadPool(200);

for (int i = 0; i < 1000; i++) {
    int id = i;
    executor.submit(() -> {
        // HTTP 호출 (응답 대기 중에도 OS 스레드 점유)
        callApi(id);
    });
}
executor.shutdown();

// 메모리: 200 × 1MB = 200MB 추가 점유
java · 사실상 코드 한 줄 변경Java 21
// newVirtualThreadPerTaskExecutor — 각 작업마다 가상 스레드 생성
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    for (int i = 0; i < 1000; i++) {
        int id = i;
        executor.submit(() -> callApi(id));
    }
} // try-with-resources가 자동으로 모두 종료 대기

// 메모리: 1000 × 수 KB = 수 MB만 추가
// 1000개 동시 진행, 대기열 없음!
언제 안 좋은가?

CPU 계산이 많은 작업(예: 행렬 곱셈, 이미지 처리)에는 효과가 거의 없습니다. 가상 스레드는 I/O 대기 시간(DB 쿼리, HTTP, 파일 입출력)이 많은 워크로드에서 빛납니다. 그리고 synchronized 블록 안에서 블로킹되면 carrier thread가 "고정(pinning)"되는 문제가 있으니, ReentrantLock을 쓰는 게 좋습니다.

LLMOps/MLOps에서의 의미

여러 LLM API를 병렬로 호출해 결과를 모으는 RAG 게이트웨이, 모델 서빙 앞단의 fan-out 패턴, 대량 추론 요청 fan-in — 모두 가상 스레드가 큰 효과를 내는 패턴입니다. Spring Boot 3.2+, Tomcat 10.1+에서 spring.threads.virtual.enabled=true 한 줄로 활성화 가능합니다.

String describe(Payment p) {
    return switch (p) {
        case Card c   -> "카드: " + c.number();
        case Cash ca  -> "현금: " + ca.amount();
        case Crypto cr -> "크립토: " + cr.coin();
        // default 없어도 OK — sealed라서 컴파일러가 완전성 보장!
    };
}

// 가드(when) 조건도 가능
String level(Payment p) {
    return switch (p) {
        case Cash c when c.amount() > 1_000_000 -> "고액 현금";
        case Cash c                            -> "일반 현금";
        case Card c, Crypto c                  -> "디지털";
        case null                              -> "미선택";
    };
}

String summary(Object obj) {

        case Line(Point(var x1, var y1),
                  Point(var x2, var y2))
            -> "선분 (" + x1 + "," + y1 + ")→(" + x2 + "," + y2 + ")";
        case Point(var x, var y) -> "점 (" + x + "," + y + ")";
        default -> "알 수 없음";
    };
}
