컨테이너 이미지를 위한
항구, Harbor

컨테이너 이미지를 안전하게 보관하고, 스캔하고, 배포하는 오픈소스 레지스트리. 항구에 화물 컨테이너가 드나들듯, Harbor를 통해 이미지가 클러스터로 흘러갑니다. 초보 클라우드 엔지니어를 위한 가장 친절한 안내서.

대상 입문 ~ 초급 Cloud/DevOps 소요 약 35분 키워드 Registry · OCI · Trivy · RBAC
SECTION 00

이미지를 어디에 보관할까

레지스트리가 왜 필요한지부터 짚고 갑니다. 여기서 출발해야 Harbor가 풀어주는 문제가 또렷이 보입니다.

도커로 애플리케이션을 만들면 컨테이너 이미지가 생깁니다. 이미지는 앱 코드, 라이브러리, 실행 환경을 통째로 묶어 놓은 압축 꾸러미입니다. 그런데 이 이미지를 내 노트북에만 두면 동료도, 서버도, 쿠버네티스 클러스터도 그걸 가져다 쓸 수가 없습니다.

그래서 이미지를 한곳에 모아 두고 누구나 약속된 방식으로 올리고(push) 내려받는(pull) 중앙 저장소가 필요합니다. 이 저장소가 바로 컨테이너 레지스트리(Container Registry)입니다.

항구 비유

이미지는 규격이 똑같은 화물 컨테이너입니다. 레지스트리는 그 컨테이너가 드나드는 항구(Harbor)죠. 배(개발자·CI 서버)는 항구에 컨테이너를 내려놓고(push), 다른 배(쿠버네티스 노드)는 같은 항구에서 컨테이너를 실어 갑니다(pull). 규격이 표준화되어 있으니 어떤 배든 어떤 항구든 문제없이 주고받을 수 있습니다.

→가장 단순한 선택: Docker Hub

처음엔 보통 Docker Hub 같은 공개 클라우드 레지스트리를 씁니다. 회원가입만 하면 바로 쓸 수 있어 편하죠. 하지만 회사 환경에서는 곧 한계에 부딪힙니다.

공개 레지스트리만 쓸 때 생기는 문제들

① 보안 — 내부 소스가 담긴 이미지를 외부 서버에 두기 부담스럽다. ② 접근망 — 폐쇄망(에어갭) 환경에서는 인터넷 자체가 막혀 있다. ③ 취약점 관리 — 어떤 이미지에 보안 결함이 있는지 회사 차원에서 알기 어렵다. ④ 속도·비용 — 같은 이미지를 매번 외부에서 끌어오면 느리고 트래픽 비용이 든다. ⑤ 권한 — 누가 어떤 이미지에 접근할지 팀 단위로 통제하기 어렵다.

이 다섯 가지를 한 번에 풀어주는 것이 바로 회사가 직접 운영하는 사설(private) 레지스트리이고, 그중 가장 널리 쓰이는 오픈소스가 Harbor입니다. 본격적으로 만나기 전에, 레지스트리 세계의 기본 단어부터 정리하겠습니다.

SECTION 01

레지스트리 기본 용어

Registry · Repository · Image · Tag · Digest. 비슷해 보이지만 역할이 다릅니다. 한 문장씩 분해해 봅시다.

이미지를 가리킬 때 우리는 harbor.mycompany.com/team-ai/llm-server:1.2.0 같은 긴 주소를 씁니다. 이 한 줄 안에 모든 개념이 들어 있습니다. 아래에서 각 조각을 눌러 의미를 확인해 보세요.

image_reference_decoder.sh

Interactive
Registryharbor.mycompany.com / Projectteam-ai / Repositoryllm-server : Tag1.2.0
위 조각을 클릭해 보세요 👆

각 부분이 이미지 주소에서 어떤 역할을 하는지 설명이 나타납니다.

→한눈에 정리

🏢

Registry

이미지를 보관하는 서버 그 자체. 항구 전체에 해당. Harbor·Docker Hub·ECR 등이 여기 들어간다.

📁

Project (Harbor 고유)

레포지토리를 묶는 네임스페이스·폴더. 팀·서비스 단위로 권한과 정책을 건다. 부두 구역에 해당.

📦

Repository

하나의 앱에 대한 이미지 모음. llm-server라는 이름표가 붙은 선반.

🏷️

Tag

같은 레포 안 여러 버전을 구분하는 꼬리표. 1.2.0, latest 등. 옮겨 붙일 수 있다.

🔒

Digest

sha256:... 형태의 내용 지문. 이미지가 1비트라도 바뀌면 값이 달라진다. 절대 변하지 않는 진짜 주소.

📜

Manifest

이미지를 이루는 레이어 목록과 설정을 담은 명세서. 멀티 아키텍처일 땐 여러 manifest를 묶는 index가 생긴다.

Tag vs Digest — 초보가 가장 많이 헷갈리는 부분

Tag는 옮겨 붙일 수 있는 별명입니다. 오늘 latest가 가리키던 이미지가 내일은 다른 이미지일 수 있죠. 반면 Digest는 내용에서 계산된 고정 지문이라 한 번 정해지면 영원히 같은 이미지만 가리킵니다. 그래서 운영 환경에서는 재현성을 위해 @sha256:... Digest로 배포하는 것이 안전합니다.

OCI 표준 — 모든 도구가 통하는 이유

이미지 포맷과 레지스트리 통신 규약은 OCI(Open Container Initiative) 표준으로 정해져 있습니다. 그래서 Docker로 만든 이미지를 Harbor에 올리고, 쿠버네티스의 containerd가 그걸 받아 실행하는 일이 매끄럽게 됩니다. Harbor도 OCI 호환 레지스트리라서 이미지뿐 아니라 Helm 차트·서명·SBOM 같은 OCI 아티팩트까지 같은 방식으로 저장합니다.

SECTION 02

Harbor란 무엇인가

단순 저장소를 넘어, 보안·정책·거버넌스를 갖춘 "엔터프라이즈급" 레지스트리.

Harbor는 컨테이너 이미지와 OCI 아티팩트를 저장·서명·스캔하는 오픈소스 레지스트리입니다. 핵심 한 줄로 요약하면 이렇습니다.

한 줄 정의

Harbor = 오픈소스 컨테이너 레지스트리 + 취약점 스캔 + 이미지 서명 + 세밀한 권한 관리(RBAC) + 레지스트리 간 복제를 한 패키지로 묶은 자체 호스팅(self-hosted) 솔루션.

→어디서 왔고, 얼마나 믿을 만한가

Harbor는 VMware에서 시작해 2018년 CNCF(Cloud Native Computing Foundation)에 기증되었고, 2020년 Graduated(졸업) 프로젝트 등급에 올랐습니다. Kubernetes·Prometheus·Envoy와 같은 등급으로, CNCF가 인정하는 가장 성숙하고 안정적인 단계입니다. 즉 "검증된 표준 도구"라는 뜻이라 회사에서 채택하기에 부담이 적습니다.

🆓

완전 오픈소스

Apache 2.0 라이선스. 라이선스 비용 없이 무제한으로 자체 운영할 수 있다.

🎓

CNCF Graduated

성숙도 최고 등급. 대규모 운영 사례와 활발한 커뮤니티가 뒷받침한다.

🧩

OCI 네이티브

이미지·Helm 차트·Cosign 서명·SBOM을 표준 방식으로 한곳에 저장한다.

🏠

Self-hosted

사내·온프레미스·폐쇄망 어디든 직접 설치. 데이터가 내 통제 아래 있다.

비유로 다시

그냥 이미지를 쌓아 두기만 하는 창고가 Docker Registry(distribution)라면, Harbor는 입출항 기록·세관 검사(취약점 스캔)·통행증 발급(RBAC)·세관 인장(서명)·다른 항구로의 화물 이송(복제)까지 갖춘 운영되는 항만청입니다.

SECTION 03

내부 아키텍처

Harbor는 사실 여러 컨테이너의 묶음입니다. 어떤 부품이 무슨 일을 하는지 큰 그림을 봅니다.

Harbor를 설치하면 단일 프로그램이 아니라 여러 개의 컨테이너(마이크로서비스)가 함께 뜹니다. 사용자는 그걸 의식하지 않고 하나의 레지스트리처럼 쓰지만, 내부 구조를 알아 두면 장애 대응과 튜닝이 쉬워집니다.

Harbor 핵심 컴포넌트 개요 — 사용자는 Proxy를 통해 들어와 Core/Registry와 대화한다

→부품별 역할

🚪

Proxy (nginx)

모든 요청이 들어오는 정문. 웹 UI·API·이미지 전송을 알맞은 내부 컴포넌트로 라우팅한다.

🧠

Core

Harbor의 두뇌. 인증, RBAC, 프로젝트·정책 관리, API를 담당한다.

🖥️

Portal

우리가 보는 웹 대시보드(Angular). 프로젝트·사용자·스캔 결과를 GUI로 다룬다.

📥

Registry

실제 이미지 레이어를 받고 내보내는 엔진. CNCF Distribution(구 Docker Registry) 기반.

⚙️

JobService

스캔·복제·가비지 컬렉션·보존 정책 같은 무거운 비동기 작업을 백그라운드로 처리한다.

🛡️

Trivy

이미지 안 라이브러리의 알려진 취약점(CVE)을 찾아내는 기본 스캐너.

🗄️

PostgreSQL

프로젝트·사용자·스캔 결과 등 메타데이터를 저장하는 데이터베이스.

⚡

Redis

세션·캐시 저장과 JobService의 작업 큐 역할을 맡는다.

이미지 본체는 어디에?

이미지 레이어(실제 데이터)는 DB가 아니라 Storage에 저장됩니다. 로컬 디스크는 물론 S3, GCS, Azure Blob, MinIO 같은 오브젝트 스토리지를 백엔드로 붙일 수 있어, 용량이 커져도 확장하기 쉽습니다. DB에는 "어떤 이미지가 어디 있는지"에 대한 정보(메타데이터)만 들어갑니다.

SECTION 04

핵심 기능 9가지

Harbor가 단순 저장소와 갈리는 지점. 이 기능들이 "거버넌스"를 만듭니다.

①Project & RBAC — 팀 단위 권한 관리

Harbor의 모든 이미지는 Project라는 폴더 안에 들어갑니다. 프로젝트는 공개(public)·비공개(private)로 나뉘고, 사용자마다 프로젝트별로 다른 역할(Role)을 줄 수 있습니다. 아래에서 역할을 클릭해 권한 차이를 확인해 보세요.

rbac_role_explorer

Interactive

현업 팁 — Robot Account

사람이 아니라 CI 파이프라인이 푸시할 때는 개인 계정 대신 로봇 계정(Robot Account)을 발급합니다. 권한 범위를 좁게 묶고 만료일을 설정할 수 있어, 토큰이 유출돼도 피해를 최소화할 수 있습니다.

②취약점 스캔 (Trivy)

이미지를 올리면 Harbor가 내장 Trivy로 내부 패키지를 검사해 알려진 보안 취약점(CVE)을 목록으로 보여줍니다. "위험도 Critical 이상이면 배포(pull) 자체를 막기" 같은 차단 정책도 걸 수 있어, 위험한 이미지가 운영에 흘러가는 걸 입구에서 차단합니다.

③이미지 서명 & 검증 (Cosign)

이미지가 "내가 만든 그 이미지가 맞는지, 중간에 바꿔치기 되진 않았는지" 보장하려면 서명이 필요합니다. Harbor는 Sigstore의 Cosign 서명을 OCI 아티팩트로 저장하고, 서명이 없는 이미지의 배포를 막는 정책을 지원합니다. 이른바 공급망 보안(supply chain security)의 핵심입니다.

④~⑨나머지 핵심 기능

🔁

④ Replication (복제)

다른 Harbor·Docker Hub·ECR·ACR·GCR 등과 이미지를 자동 동기화. 멀티 리전·재해 복구·폐쇄망 반입에 쓴다.

🪞

⑤ Proxy Cache

Docker Hub 등 외부 레지스트리를 통과 캐시(pull-through). 한 번 받은 이미지는 사내에 저장돼 다음부터 빠르고, Docker Hub 다운로드 제한도 회피한다.

🧹

⑥ 보존 & 불변 태그

오래된 태그를 규칙대로 자동 삭제(Retention)하고, 운영 태그는 덮어쓰기 금지(Immutability)로 잠근다.

📊

⑦ Quota (용량 할당)

프로젝트별 저장 용량 상한을 설정. 한 팀이 디스크를 독식하는 사고를 방지한다.

🗑️

⑧ Garbage Collection

어떤 태그도 가리키지 않는 버려진 레이어를 정리해 디스크를 회수한다.

🔔

⑨ Webhook & Helm/OCI

푸시·스캔 완료 시 Slack·CI로 알림 전송. Helm 차트도 OCI 방식으로 함께 저장·배포한다.

멀티 아키텍처(ARM64 + x86_64) 환경이라면

Harbor는 OCI 이미지 인덱스(manifest list)를 그대로 저장하므로, linux/amd64와 linux/arm64를 한 태그로 묶어 올릴 수 있습니다. 그러면 x86 노드는 amd64 레이어를, ARM 노드는 arm64 레이어를 같은 image:tag로 자동 선택해 받습니다. 이기종 클러스터를 하나의 레지스트리로 깔끔하게 운영할 수 있습니다.

SECTION 05

설치하기

테스트는 docker-compose로, 운영은 Kubernetes(Helm)로. 두 가지 길을 나란히 봅니다.

Harbor 설치는 크게 두 갈래입니다. 한 대 서버에서 빠르게 띄우는 docker-compose 방식과, 쿠버네티스 위에 올리는 Helm 방식입니다. 학습·소규모는 전자, 운영·고가용성은 후자를 권합니다.

# 1) 오프라인 설치 패키지 다운로드 (버전은 릴리스 페이지 참고)
curl -LO https://github.com/goharbor/harbor/releases/download/v2.11.0/harbor-offline-installer-v2.11.0.tgz
tar xzvf harbor-offline-installer-v2.11.0.tgz
cd harbor

# 2) 설정 템플릿 복사 후 수정 (다음 탭 참고)
cp harbor.yml.tmpl harbor.yml
vi harbor.yml

# 3) 설치 — Trivy(스캔)까지 함께 활성화
sudo ./install.sh --with-trivy

# 4) 확인 — 모든 컴포넌트 컨테이너가 떠 있어야 한다
docker compose ps
# 기본 접속: https://<호스트>  /  admin / Harbor12345
# 1) Harbor 공식 Helm 저장소 추가
helm repo add harbor https://helm.goharbor.io
helm repo update

# 2) 네임스페이스 생성
kubectl create namespace harbor

# 3) 설치 — 도메인·관리자 비밀번호·스토리지 지정
helm install harbor harbor/harbor -n harbor \
  --set expose.type=ingress \
  --set expose.ingress.hosts.core=harbor.mycompany.com \
  --set externalURL=https://harbor.mycompany.com \
  --set harborAdminPassword="ChangeMe!" \
  --set persistence.persistentVolumeClaim.registry.size=200Gi

# 4) 파드 상태 확인
kubectl get pods -n harbor
# harbor.yml — 가장 먼저 손대야 할 항목들
hostname: harbor.mycompany.com   # 접속 도메인 (필수)

https:                           # 운영은 반드시 HTTPS
  port: 443
  certificate: /data/cert/harbor.crt

HTTPS는 선택이 아니라 기본

도커는 기본적으로 HTTP 레지스트리에 접속을 거부합니다. 테스트로 HTTP를 쓰려면 클라이언트마다 insecure-registries 예외를 등록해야 해서 번거롭고 위험합니다. 운영에서는 처음부터 사내 인증서나 Let's Encrypt로 HTTPS를 구성하는 편이 훨씬 깔끔합니다.

SECTION 07

CI/CD · 쿠버네티스 연동

사람이 직접 푸시하는 일은 드뭅니다. 실제로는 파이프라인이 올리고, 클러스터가 받아 갑니다.

현업에서 Harbor는 두 흐름의 한가운데 있습니다. CI 파이프라인이 이미지를 빌드해 푸시하고, 쿠버네티스가 그 이미지를 풀해 파드로 실행합니다. 각각의 연결 고리를 봅니다.

# .gitlab-ci.yml
build-image:
  stage: build
  image: gcr.io/kaniko-project/executor:debug
  script:
    # Harbor 로봇 계정으로 인증 정보 구성
    - mkdir -p /kaniko/.docker
    - echo "{\"auths\":{\"$HARBOR_HOST\":{\"auth\":\"$HARBOR_AUTH\"}}}" \
        > /kaniko/.docker/config.json
    # 빌드 후 Harbor로 직접 푸시
    - /kaniko/executor \
        --context $CI_PROJECT_DIR \
        --destination $HARBOR_HOST/team-ai/llm-server:$CI_COMMIT_TAG
# 1) 비공개 프로젝트라면 클러스터에 인증 정보 등록
kubectl create secret docker-registry harbor-cred \
  --docker-server=harbor.mycompany.com \
  --docker-username='robot$k8s-puller' \
  --docker-password='<robot-token>' \
  -n ai-serving

# 2) 파드에서 그 secret으로 이미지 풀
apiVersion: apps/v1
kind: Deployment
spec:
  template:
    spec:
      imagePullSecrets:
        - name: harbor-cred
      containers:
        - name: llm-server
          image: harbor.mycompany.com/team-ai/llm-server:1.2.0
CI에서는 Kaniko·Buildah처럼 데몬 없이 빌드하는 도구가 쿠버네티스 친화적입니다.
전체 그림으로 정리

개발자가 코드를 push → GitLab CI가 이미지 build → 로봇 계정으로 Harbor에 push → Harbor가 자동 scan → 정책 통과 시 → 쿠버네티스가 imagePullSecret으로 pull → 파드 실행. Harbor는 이 사슬에서 "신뢰할 수 있는 이미지만 통과시키는 관문" 역할을 합니다.

비용 없이 자체 운영하고 싶고, 스캔·서명·RBAC·복제 같은 보안·거버넌스가 필요하며, 온프레미스·폐쇄망·이기종 클러스터를 다룰 때.

클라우드 매니지드가 나을 때

핵심 구분

Harbor는 "보안과 거버넌스를 갖춘, 공짜로 자체 운영하는 컨테이너 레지스트리"라는 자리에 정확히 들어맞습니다. Distribution보다 풍부하고, 클라우드 매니지드보다 자유로우며, Artifactory보다 가볍고 저렴합니다.

SECTION 09

실전 응용 분야

Harbor가 실제 현장에서 어떤 문제를 푸는지, 대표적인 네 가지 시나리오.

①폐쇄망(에어갭) 환경의 이미지 반입

금융·국방·공공처럼 인터넷이 차단된 폐쇄망에서는 외부 레지스트리에서 직접 풀할 수 없습니다. 이때 인터넷 구간의 Harbor에서 필요한 이미지를 모아 두고, 복제(Replication)나 이미지 export/import로 내부망 Harbor에 옮긴 뒤, 클러스터는 내부 Harbor만 바라보게 합니다.

②멀티 클러스터 · 멀티 리전 배포

여러 지역·여러 클러스터에 같은 이미지를 빠르게 깔아야 할 때, 중앙 Harbor에 한 번 올리고 각 리전의 Harbor로 자동 복제합니다. 노드는 가장 가까운 레지스트리에서 풀하므로 지연과 트래픽 비용이 줄어듭니다.

③공급망 보안 게이트

"스캔에서 Critical 취약점이 없고, 서명이 검증된 이미지만 배포 허용"이라는 정책을 Harbor에 걸면, 위험한 이미지가 운영에 들어가는 것을 입구에서 막을 수 있습니다. CVE 대응과 SBOM 관리의 출발점이 됩니다.

④Docker Hub 풀 제한 회피 & 캐시 가속

Docker Hub는 익명·무료 사용자에게 다운로드 횟수 제한을 둡니다. CI가 자주 같은 베이스 이미지를 받으면 금방 한도에 걸리죠. Harbor의 Proxy Cache를 앞에 두면, 한 번 받은 이미지는 사내에 캐시돼 이후엔 빠르게 제공되고 한도 문제도 사라집니다.

MLOps/LLM 서빙 관점에서

vLLM·Triton 같은 추론 서버 이미지는 수 GB로 무겁고 GPU 드라이버·CUDA 버전에 민감합니다. Harbor로 이런 무거운 이미지를 사내에서 빠르게 배포하고, ARM(예: Grace Blackwell)·x86 노드용 멀티 아키텍처 이미지를 한 태그로 묶어 두면, 이기종 GPU 클러스터에 일관되게 모델 서버를 굴릴 수 있습니다. 스캔·서명까지 더하면 추론 파이프라인 전체의 신뢰성이 올라갑니다.

SECTION 10

정리 & 퀴즈

핵심을 다시 짚고, 다섯 문제로 이해도를 확인합니다.

Q1.Tag와 Digest의 차이로 가장 옳은 것은?
정답: Digest(sha256:...)는 이미지 내용의 지문이라 변하지 않습니다. 운영 배포에 재현성을 위해 Digest를 쓰는 이유죠.
Q2.Harbor에서 이미지 실제 레이어 데이터가 저장되는 곳은?
정답: DB에는 메타데이터만, 실제 이미지 레이어는 Storage에 저장됩니다. S3·GCS·MinIO 등을 백엔드로 붙일 수 있습니다.
Q3.Harbor에 기본 내장된 취약점 스캐너는?
정답: Trivy입니다. Cosign은 이미지 서명, Prometheus는 모니터링 도구죠.
Q4.다음 중 Harbor의 자리(포지션)를 가장 잘 설명한 것은?
정답: 범용 저장소는 Nexus·Artifactory, 매니지드 클라우드는 ECR·ACR 등입니다. Harbor는 self-hosted 컨테이너 레지스트리에 보안 기능을 더한 위치입니다.
Q5.이기종(ARM64 + x86_64) 클러스터에서 한 태그로 양쪽 노드를 모두 지원하려면?
정답: buildx --platform linux/amd64,linux/arm64 --push로 만들면 manifest list가 생겨, 각 노드가 자기 아키텍처 레이어를 자동 선택합니다.

한 장으로 끝내는 요약

Harbor를 다시 만났을 때 떠올릴 다섯 가지.

WHAT오픈소스·CNCF 졸업 컨테이너 레지스트리. 이미지를 위한 항구.
WHY보안 스캔·서명·RBAC·복제·캐시로 단순 저장소를 넘어선다.
HOWcompose로 테스트, Helm으로 운영. login→tag→push→scan→pull.
VSDistribution보다 풍부, 클라우드보다 자유, Artifactory보다 가볍다.
WHERE폐쇄망·멀티클러스터·공급망 보안·이기종 GPU 서빙에 강하다.
