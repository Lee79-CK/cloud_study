SECTION 01

OpenShift란 무엇인가

한 문장으로 정의하면 이렇습니다. OpenShift는 Red Hat이 Kubernetes 위에 기업(엔터프라이즈)에서 바로 쓸 수 있도록 필요한 것들을 미리 얹어서 파는 "완성품 플랫폼"입니다. 정식 명칭은 Red Hat OpenShift Container Platform(OCP)이고, 커널에 해당하는 부분은 100% 쿠버네티스입니다. 즉, OpenShift를 배우면 쿠버네티스를 버리는 게 아니라 쿠버네티스를 포함해서 배우는 것입니다.

비유로 이해하기: 리눅스 커널과 배포판

리눅스 커널(kernel)만 다운로드해서 쓰는 사람은 거의 없습니다. 대부분 Ubuntu, RHEL 같은 배포판(distribution)을 씁니다. 배포판은 커널에 패키지 관리자, 보안 정책, 데스크톱, 지원 서비스를 얹은 것이죠. Kubernetes가 커널이라면 OpenShift는 RHEL 같은 "쿠버네티스 배포판"입니다. 실제로 업계에서는 OpenShift를 "Kubernetes distribution"이라고 부릅니다.

바닐라(Vanilla) Kubernetes란?

이 강의에서 계속 등장할 용어부터 정리합시다. 바닐라 쿠버네티스란 아무 토핑도 얹지 않은 바닐라 아이스크림처럼, CNCF가 배포하는 순정 업스트림(upstream) 쿠버네티스를 그대로 kubeadm 등으로 직접 설치해서 쓰는 형태를 말합니다.

바닐라 K8s를 설치하고 나면 실무에 필요한 많은 것들이 "비어" 있습니다. 컨테이너 이미지를 빌드하는 CI, 이미지를 저장할 레지스트리, 모니터링, 로깅, 인증(로그인), 네트워크 플러그인(CNI), 인그레스 컨트롤러… 이걸 하나하나 직접 고르고, 설치하고, 연동하고, 업그레이드까지 책임져야 합니다. OpenShift는 이 과정을 통째로 대신해 준 것입니다.

SECTION 02

왜 K8s만으로는 부족하다고 말할까

"쿠버네티스가 이미 오케스트레이션을 다 해주는데 왜 굳이 돈 내고 OpenShift를 쓰지?"라는 의문은 아주 자연스럽습니다. 답은 쿠버네티스가 해결하는 문제의 범위가 생각보다 좁기 때문입니다. 쿠버네티스는 "컨테이너를 어디에, 몇 개, 어떻게 띄울 것인가"라는 오케스트레이션 문제를 풀어줄 뿐, 그 앞뒤 단계 — 코드를 이미지로 만들고, 이미지를 안전하게 보관하고, 사용자를 인증하고, 장애를 관측하고, 규제 감사에 대응하는 문제 — 는 전부 사용자의 몫으로 남겨둡니다.

바닐라 K8s 운영자가 마주치는 현실

CNI(네트워크)만 해도 Calico, Cilium, Flannel, Weave... 인그레스는 NGINX, Traefik, HAProxy, Kong... 모니터링, 로깅, 인증, 스토리지(CSI)까지 조합하면 경우의 수가 폭발합니다. 문제는 선택 자체보다 조합 간 호환성 검증과 버전 업그레이드 책임이 전부 팀에 귀속된다는 것입니다. OpenShift는 이 조합을 Red Hat이 미리 검증(테스트 매트릭스)해서 하나의 버전으로 릴리스합니다.

쿠버네티스는 대략 4개월 주기로 마이너 버전이 나오고, 각 버전의 패치 지원은 약 14개월입니다. 바닐라 환경에서는 컨트롤 플레인, kubelet, CNI, 애드온을 순서 맞춰 직접 올려야 하고 실패 시 롤백 계획도 스스로 세워야 합니다. OpenShift는 Cluster Version Operator(CVO)가 클러스터 전체(노드 OS 포함!)를 하나의 단위로 오케스트레이션하며 업그레이드합니다. 웹 콘솔에서 버튼 클릭으로 마이너 업그레이드를 수행하는 수준입니다.

바닐라 K8s에서 아무 설정 없이 Pod를 배포하면 컨테이너는 이미지에 정의된 사용자(대개 root)로 실행됩니다. 보안 정책(Pod Security Admission, NetworkPolicy 등)은 알아서 켜야 합니다. OpenShift는 반대로 기본이 "제한(restricted)"입니다. SCC(Security Context Constraints)가 루트 실행을 기본 차단하고 임의의 UID를 부여합니다. 처음엔 불편하지만, 금융·공공처럼 감사가 있는 환경에서는 이 기본값이 큰 자산이 됩니다.

바닐라 K8s는 커뮤니티 지원(GitHub 이슈, Slack)이 전부입니다. 엔터프라이즈에서 새벽 3시에 etcd가 깨졌을 때 SLA를 걸고 대응해 줄 주체가 없습니다. OpenShift 서브스크립션은 Red Hat의 기술지원과 보안 패치(CVE 대응), 장기 지원(EUS 버전)을 포함합니다. 기업이 OpenShift에 돈을 내는 가장 큰 이유 중 하나입니다.

바닐라 K8s에서 개발자가 앱을 올리려면 Dockerfile, Deployment, Service, Ingress YAML을 모두 이해해야 합니다. OpenShift는 oc new-app https://github.com/... 한 줄이나 웹 콘솔의 "+Add" 버튼으로 소스 코드 → 이미지 빌드 → 배포 → 외부 URL 발급까지 자동으로 처리하는 개발자 워크플로우(S2I)를 제공합니다. PaaS(Platform as a Service)의 편의성을 K8s 위에 얹은 것이죠.

공정한 시선: OpenShift가 항상 정답은 아닙니다

통합과 검증의 대가는 비용(서브스크립션)과 자유도 감소입니다. 최신 K8s 기능을 가장 먼저 쓰고 싶거나, 팀에 강한 플랫폼 엔지니어링 역량이 있고 경량 클러스터가 많이 필요하다면 바닐라/매니지드 K8s가 더 맞을 수 있습니다. 이 판단 기준은 Section 14에서 체계적으로 다룹니다.

SECTION 03

역사와 제품 계보 — OpenShift는 K8s보다 먼저 태어났다

의외의 사실 하나. OpenShift는 쿠버네티스보다 3년 먼저(2011년) 세상에 나왔습니다. 초기 OpenShift(v1~v2)는 자체 기술("Gear"와 "Cartridge")로 만든 PaaS였습니다. 2014년 구글이 쿠버네티스를 공개하자 Red Hat은 과감하게 자체 엔진을 버리고 2015년 OpenShift 3부터 아키텍처 전체를 Kubernetes + Docker 기반으로 갈아엎었습니다. 지금의 OpenShift 4는 여기에 "Operator로 스스로를 운영하는 클러스터"라는 철학을 더한 세대입니다.

2011
OpenShift v1/v2 출시
자체 엔진 기반 PaaS. Heroku와 경쟁하던 시절. 쿠버네티스는 아직 존재하지 않음.
2014
Kubernetes 공개
구글이 내부 시스템 Borg의 경험을 담아 오픈소스로 공개. Red Hat은 초기부터 최상위 기여 기업으로 참여.
2015
OpenShift 3 — 쿠버네티스로 전면 재설계
자체 엔진을 폐기하고 K8s를 코어로 채택. "K8s 배포판"으로서의 정체성 확립.
2019
OpenShift 4 — Operator 시대
CoreOS 인수 성과를 반영. 불변 OS(RHCOS), Operator 기반 자가 관리 클러스터, 자동화된 설치(IPI) 도입.
현재
하이브리드 클라우드 & AI 플랫폼으로 확장
가상화(OpenShift Virtualization), 서비스 메시, 서버리스, OpenShift AI(구 RHODS)까지 포트폴리오 확대.

이름이 비슷해서 헷갈리는 제품들 정리

OpenShift라는 이름이 붙은 제품이 많아서 초보자가 가장 혼란스러워하는 부분입니다. 딱 정리해 드립니다.

이름정체비용이런 경우에
OCP
OpenShift Container Platform		 이 강의의 주인공. 자체 인프라(온프렘/클라우드 IaaS)에 설치하는 상용 제품 유료 기업 프로덕션, 프라이빗 클라우드
OKD OpenShift의 업스트림 오픈소스 커뮤니티 배포판 (Fedora ↔ RHEL 관계와 유사) 무료 학습, 커뮤니티 지원으로 충분한 환경
ROSA
Red Hat OpenShift Service on AWS		 AWS 위에서 Red Hat+AWS가 공동 운영하는 매니지드 OpenShift 유료 AWS 중심 조직, 운영 위임 원할 때
ARO
Azure Red Hat OpenShift		 Azure 위 매니지드 OpenShift (MS+Red Hat 공동 운영) 유료 Azure 중심 조직
OpenShift Local
구 CRC(CodeReady Containers)		 노트북에서 돌리는 단일 노드 미니 OpenShift 무료* 개인 학습, 로컬 개발 (*개발자 계정 필요)
SNO
Single Node OpenShift		 컨트롤 플레인+워커를 1대에 합친 프로덕션급 단일 노드 구성 유료 엣지, 통신 기지국 등 공간 제약 환경
MicroShift 엣지 디바이스용 초경량 OpenShift (RHEL for Edge 위에서 동작) 유료 IoT, 차량, 초소형 엣지
Red Hat 생태계의 반복 패턴

Red Hat 제품은 대부분 "커뮤니티 업스트림 → 상용 다운스트림" 구조를 가집니다. Fedora → RHEL, OKD → OCP처럼요. 상용 버전은 업스트림에서 검증된 코드에 기술지원·인증·장기 유지보수를 더한 것입니다. 이 패턴을 알면 Red Hat 포트폴리오 전체가 쉽게 읽힙니다.

SECTION 04

아키텍처 한눈에 보기

OpenShift 4의 아키텍처는 "쿠버네티스를 가운데 두고 아래로는 OS까지, 위로는 개발자 경험까지 확장한 스택"으로 이해하면 정확합니다. 아래 다이어그램의 각 계층을 클릭해 보세요. 파란 계층은 바닐라 K8s에도 있는 부분, 빨간 계층은 OpenShift가 얹은 부분, 초록 계층은 OS 영역입니다.

바닐라 K8s와의 구조적 차이 요약

바닐라 Kubernetes
  • K8s 코어만 제공 — 나머지는 레고 조립
  • 노드 OS는 범용 리눅스, 관리 주체는 별도(나)
  • 런타임: containerd가 사실상 표준
  • 업그레이드: 컴포넌트별로 순서 맞춰 수동/반자동
  • 구성 자유도 최상 — 원하는 CNI/CSI/애드온 마음대로
OpenShift 4
  • K8s 코어 + 플랫폼/개발자 서비스가 완성품으로 통합
  • 노드 OS(RHCOS)까지 클러스터가 선언적으로 관리
  • 런타임: CRI-O 고정 (K8s와 버전 동기화)
  • 업그레이드: CVO가 OS 포함 전체를 원자적으로 진행
  • 구성은 Red Hat 검증 조합 내에서 — 자유도 대신 안정성

아닙니다. OpenShift는 업스트림 쿠버네티스를 그대로 포함하는 CNCF 인증(Certified Kubernetes) 배포판입니다. Red Hat은 K8s 프로젝트의 최상위 기여 기업 중 하나이고, OpenShift 고유 기능(Route, SCC 등)은 K8s를 수정한 것이 아니라 CRD와 API 확장으로 얹은 것입니다. 그래서 표준 K8s 워크로드가 그대로 동작합니다.

SECTION 05

Project vs Namespace — 뭐가 다른가

Project는 Namespace를 감싼 껍데기입니다. Project를 만들면 내부적으로 동일한 이름의 Namespace가 생성되고, 여기에 (1) 생성자에게 관리자 권한 자동 부여, (2) 기본 네트워크 정책/쿼터 템플릿 적용, (3) 표시 이름·설명 어노테이션 지원이 더해집니다. 중요한 차이 하나: 일반 사용자는 Namespace를 직접 만들 권한이 없어도 Project는 셀프서비스로 만들 수 있습니다 (self-provisioner 역할). 멀티테넌시 환경에서 팀별로 안전하게 공간을 나눠주기 위한 설계입니다. 

# 네임스페이스 생성 (cluster 권한 필요)
kubectl create namespace team-a

# 컨텍스트에 네임스페이스 지정
kubectl config set-context --current --namespace=team-a
# 프로젝트 생성 — 일반 사용자도 셀프서비스 가능
oc new-project team-a --display-name="A팀 개발환경" \
  --description="A팀 마이크로서비스 개발용"

# 생성과 동시에 해당 프로젝트로 전환됨
oc project team-a        # 프로젝트 전환
oc projects              # 내가 접근 가능한 프로젝트 목록

Route vs Ingress — 외부 노출의 두 가지 방식

Route는 Ingress가 표준화되기 전에 OpenShift가 먼저 만든 외부 노출 리소스입니다. HAProxy 기반 Router가 트래픽을 받아 서비스로 전달하며, TLS 종료 방식 3종(edge/passthrough/re-encrypt), 가중치 기반 트래픽 분할(카나리 배포!), 자동 호스트네임 발급을 리소스 스펙만으로 지원합니다. 최신 OpenShift는 표준 Ingress도 지원하며, Ingress를 만들면 내부적으로 Route가 자동 생성됩니다. 

# 전제: Ingress Controller(nginx 등)를 미리 설치해 두어야 함
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: my-app
spec:
  ingressClassName: nginx
  rules:
  - host: myapp.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: my-app
            port: {number: 8080}
# Router가 내장되어 있어 별도 컨트롤러 설치 불필요
# 명령 한 줄로도 가능: oc expose service my-app
apiVersion: route.openshift.io/v1
kind: Route
metadata:
  name: my-app
spec:
  host: myapp.apps.cluster.example.com  # 생략 시 자동 발급
  to:
    kind: Service
    name: my-app
  port: {targetPort: 8080}
  tls:
    termination: edge  # edge | passthrough | reencrypt
# Route 하나로 v1:v2 = 90:10 카나리 트래픽 분할
spec:
  to:
    kind: Service
    name: my-app-v1
    weight: 90
  alternateBackends:
  - kind: Service
    name: my-app-v2
    weight: 10
# 바닐라 K8s에서 같은 걸 하려면 nginx annotation,
# Istio VirtualService, 또는 Argo Rollouts 등이 필요

ImageStream — OpenShift만의 이미지 추상화

바닐라 K8s에서 Deployment는 nginx:1.25처럼 레지스트리의 태그를 직접 참조합니다. 문제는 태그가 움직이는 표적이라는 것 — 같은 latest 태그라도 내용물이 바뀔 수 있죠. ImageStream은 이미지 태그에 대한 "클러스터 내부의 안정적인 포인터"입니다. 실제 이미지의 SHA 다이제스트를 기록해 두고, 새 이미지가 push되거나 원본 태그가 갱신되면 이를 감지해서 빌드나 배포를 자동으로 트리거할 수 있습니다.

  • 재현성 — 태그가 아닌 다이제스트를 기록하므로 "어제의 latest"로 롤백 가능
  • 자동화 — 베이스 이미지가 패치되면 의존하는 앱 이미지를 자동 리빌드 (보안 패치 전파!)
  • 디커플링 — 외부 레지스트리 주소가 바뀌어도 매니페스트는 ImageStream 이름만 참조
SECTION 06

보안 모델 — SCC 깊이 보기

OpenShift로 넘어온 엔지니어가 가장 먼저, 가장 세게 부딪히는 벽이 바로 SCC(Security Context Constraints)입니다. Docker Hub에서 잘 돌던 이미지가 OpenShift에서는 CrashLoopBackOff가 나는 상황, 십중팔구 SCC 때문입니다. 원리를 이해하면 벽이 아니라 방패가 됩니다.

SCC가 하는 일

SCC는 "이 Pod가 어떤 보안 권한을 가지고 실행될 수 있는가"를 심사하는 관문(Admission)입니다. Pod가 생성될 때 서비스어카운트에 부여된 SCC 목록과 Pod의 요구사항(securityContext)을 대조해서, 허용 범위 안이면 통과시키고 아니면 거부합니다. 기본값인 restricted-v2 SCC는:

  • 루트(UID 0) 실행 금지 — 컨테이너는 프로젝트별로 할당된 임의의 높은 UID(예: 1000690000)로 실행됨
  • 권한 상승(privilege escalation) 금지, 대부분의 Linux capabilities 제거
  • 호스트 네트워크/PID/IPC/경로 접근 금지
  • Seccomp 프로파일 RuntimeDefault 강제
그래서 이미지가 죽는 겁니다

많은 공개 이미지는 root로 실행되며 /var/log, 80번 포트 같은 특권 자원을 가정합니다. OpenShift에서 임의 UID로 실행되는 순간 쓰기 권한이 없어 죽는 것이죠. 해결책은 SCC를 푸는 게 아니라 이미지를 "임의 UID에서도 동작하도록" 고치는 것이 정석입니다: 1024 이상 포트 사용, 쓰기가 필요한 디렉터리를 그룹 쓰기 가능(GID 0)으로 설정.

주요 SCC 프로파일

SCC허용 수준용도
restricted-v2가장 엄격기본값. 모든 일반 워크로드의 표준
nonroot-v2엄격루트만 아니면 특정 UID 지정 허용
anyuid완화이미지에 정의된 UID(루트 포함) 허용. 레거시 이미지 구제용
hostnetwork-v2완화호스트 네트워크 사용 허용
privileged최대 개방모든 권한. CNI·스토리지 드라이버 등 시스템 컴포넌트 전용

🎮 인터랙티브: SCC 심사 시뮬레이터

Pod의 요구사항을 선택하면, 기본 restricted-v2 SCC의 심사 결과와 처방을 보여줍니다.

이 Pod는 무엇을 요구하나요?
✅ 통과 (restricted-v2) — 프로젝트에 할당된 임의 UID(예: 1000690000)로 실행됩니다. 가장 이상적인 상태입니다.

SCC vs Pod Security Admission(PSA)

바닐라 K8s의 표준 보안 관문은 PSA입니다. PSA는 네임스페이스 단위로 privileged / baseline / restricted 3단계 프로파일을 라벨로 지정하는 방식이라 단순하지만 거칠고, SCC는 서비스어카운트 단위로 UID 범위·capabilities·볼륨 타입까지 지정하는 세밀한 다이얼입니다. 최신 OpenShift는 PSA도 함께 동작시키며 SCC와 동기화합니다.

특정 서비스어카운트에만 anyuid SCC를 부여하는 정석 패턴
# 1. 워크로드 전용 서비스어카운트 생성 (default SA에 풀지 말 것!)
oc create sa legacy-app-sa -n my-project

# 2. 해당 SA에만 anyuid SCC 부여
oc adm policy add-scc-to-user anyuid -z legacy-app-sa -n my-project

# 3. Deployment에서 이 SA를 사용하도록 지정
#    spec.template.spec.serviceAccountName: legacy-app-sa

# 어떤 SCC가 적용되었는지 확인
oc get pod mypod -o yaml | grep "openshift.io/scc"
실무 원칙 3줄 요약

① SCC는 낮추는 게 아니라 이미지를 고치는 것이 우선. ② 어쩔 수 없다면 전용 SA에만 최소 SCC 부여. ③ privileged는 시스템 컴포넌트가 아니면 사실상 금지 — 감사에서 반드시 지적됩니다.

SECTION 07

Operator와 OperatorHub — OpenShift 4의 심장

OpenShift 4를 한 단어로 압축하면 "Operator"입니다. Operator는 "사람 운영자의 운영 지식(설치, 업그레이드, 백업, 장애 복구)을 코드로 옮겨 쿠버네티스 컨트롤러로 만든 것"입니다. 개념 자체는 CoreOS(현 Red Hat)가 2016년에 제창했고 지금은 K8s 생태계 전체의 표준 패턴이 되었지만, OpenShift는 이를 플랫폼의 뼈대로까지 밀어붙였습니다.

Operator 패턴 3분 이해

쿠버네티스의 기본 동작 원리는 "선언한 상태(desired state)와 실제 상태(current state)를 끊임없이 비교해서 맞추는 조정 루프(reconcile loop)"입니다. Deployment 컨트롤러가 "replica 3개"라는 선언을 지키듯이, Operator는 CRD(Custom Resource Definition)로 새로운 리소스 종류를 정의하고 그것을 지키는 커스텀 컨트롤러입니다.

예: PostgreSQL Operator가 있다면, DB 구축이 이렇게 바뀝니다
# "PostgreSQL 클러스터 하나 주세요"를 YAML로 선언
apiVersion: postgresql.example.com/v1
kind: PostgresCluster        # ← Operator가 정의한 커스텀 리소스
metadata:
  name: orders-db
spec:
  replicas: 3                 # HA 구성
  version: "16"
  backup:
    schedule: "0 2 * * *"     # 매일 새벽 2시 백업
# → Operator가 StatefulSet, Service, 복제 설정, 백업 CronJob,
#   장애 시 failover까지 알아서 처리. 이것이 "Day-2 운영의 자동화"

OpenShift에서 Operator가 특별한 이유

🏗️
플랫폼 자체가 Operator로 구성

네트워크, 인증, 콘솔, 모니터링, 스토리지 등 클러스터의 모든 구성요소가 각각 Cluster Operator로 관리됩니다. oc get clusteroperators로 클러스터 건강 상태를 한눈에 확인합니다.

🎼
CVO — 오케스트라의 지휘자

Cluster Version Operator가 전체 Cluster Operator의 버전과 상태를 지휘합니다. 업그레이드 시 정해진 순서로 각 Operator를 갱신하고, 문제가 생기면 스스로 복구를 시도합니다.

🛒
OperatorHub — 앱스토어

웹 콘솔에 내장된 Operator 카탈로그입니다. Red Hat 인증 Operator, 파트너 Operator, 커뮤니티 Operator를 클릭 몇 번으로 설치합니다. GPU Operator, Kafka(AMQ Streams), 서비스 메시 등.

🔄
OLM — 생명주기 관리자

Operator Lifecycle Manager가 Operator 자체의 설치·의존성·업그레이드 채널을 관리합니다. "Operator를 관리하는 Operator"인 셈입니다. 바닐라 K8s에도 설치 가능하지만 OpenShift에는 기본 내장.

바닐라 K8s에서 소프트웨어 설치
  • 주로 Helm 차트로 설치 — 설치 시점의 템플릿 렌더링
  • 설치 후 운영(백업, 장애 복구, 버전 업)은 별도 자동화 필요
  • Operator 쓰려면 OLM부터 직접 설치
OpenShift에서 소프트웨어 설치
  • OperatorHub에서 클릭 설치 → Day-2 운영까지 Operator가 지속 관리
  • 업그레이드 채널(stable, fast...) 구독으로 자동/수동 갱신
  • Helm도 물론 지원 (콘솔에 Helm 카탈로그 내장)
Helm vs Operator, 언제 뭘 쓰나

Helm은 "설치 도구", Operator는 "운영 자동화"입니다. 상태 없는 웹앱처럼 설치만 하면 끝인 것은 Helm으로 충분하고, 데이터베이스·메시지 큐처럼 지속적인 운영 판단(failover, 리밸런싱, 백업)이 필요한 것은 Operator가 진가를 발휘합니다. 실무에서는 둘을 섞어 씁니다.

SECTION 08

설치 옵션과 배포 형태

"OpenShift 설치는 어렵다"는 말은 절반만 맞습니다. OpenShift 4는 설치 방식을 두 갈래로 표준화했고, 학습용이라면 노트북에서 20분 만에 띄울 수도 있습니다.

IPI vs UPI — 두 가지 설치 철학

IPI — Installer Provisioned Infrastructure
  • 설치 프로그램(openshift-install)이 인프라(VM, 네트워크, LB)까지 직접 생성
  • AWS, Azure, GCP, vSphere, 베어메탈 등 지원
  • 설정 파일 하나(install-config.yaml)로 완전 자동 설치
  • 👉 권장 기본값. 가장 빠르고 실수 여지가 적음
UPI — User Provisioned Infrastructure
  • 인프라(VM, DNS, LB)를 사용자가 미리 준비하고 그 위에 설치
  • 망분리, 기존 인프라 표준 준수 등 제약이 있는 환경용
  • 국내 금융·공공의 폐쇄망(air-gapped) 설치에서 자주 등장
  • 👉 유연하지만 준비 항목이 많음 (DNS 레코드, LB 규칙...)

학습 환경 만들기 — 지금 바로 시작하는 3가지 방법

💻
① OpenShift Local

노트북에 단일 노드 클러스터를 띄우는 공식 도구. RAM 16GB 이상 권장. Red Hat 개발자 계정(무료)으로 pull secret 발급 후 crc setup && crc start

🧪
② Developer Sandbox

Red Hat이 제공하는 무료 공유 클러스터. 설치 없이 브라우저로 바로 실습. 30일 후 갱신 가능. 콘솔·oc 체험에 최적.

🆓
③ OKD

커뮤니티 배포판을 홈랩/VM에 직접 설치. 상용 지원은 없지만 구조는 거의 동일. 설치 과정 자체를 학습하고 싶을 때.

초보자 추천 경로

Developer Sandbox로 콘솔과 oc 감각 익히기 → OpenShift Local로 관리자 작업(Operator 설치, SCC) 실습 → 필요 시 OKD/클라우드 체험판으로 다중 노드 경험 순서를 추천합니다. 처음부터 다중 노드 설치에 도전하면 설치에서 지쳐버립니다.

배포 지형도 — 어디에나 있는 OpenShift

OpenShift의 세일즈 포인트는 "하이브리드 클라우드 일관성"입니다. 온프렘 베어메탈, vSphere, AWS/Azure/GCP, 엣지(SNO, MicroShift)까지 동일한 플랫폼, 동일한 API, 동일한 운영 방식을 유지합니다. 여러 클라우드에 흩어진 클러스터는 RHACM(Red Hat Advanced Cluster Management)으로 중앙에서 정책·배포를 통합 관리합니다. 멀티/하이브리드 클라우드 전략을 가진 대기업이 OpenShift를 선택하는 핵심 이유입니다.

SECTION 09

oc CLI 마스터하기

oc는 kubectl의 상위집합(superset)입니다. kubectl로 하던 모든 것(get, describe, apply, logs, exec...)이 그대로 되고, 여기에 OpenShift 전용 명령이 추가됩니다. kubectl에 익숙하다면 학습 비용은 거의 없습니다.

SECTION 10

웹 콘솔 둘러보기 — 두 개의 얼굴

OpenShift 웹 콘솔은 단순한 대시보드가 아니라 실제 운영과 개발이 가능한 작업 공간입니다. 가장 큰 특징은 사용자의 역할에 따라 최적화된 두 가지 관점(Perspective)을 제공한다는 점입니다.

🛠️ 관리자 뷰 (Administrator)
  • 클러스터 상태 대시보드, 노드·머신 관리
  • OperatorHub에서 Operator 설치
  • 사용자·RBAC·쿼터 관리, 클러스터 설정
  • 모니터링(메트릭·알림), 클러스터 업그레이드 버튼
👩‍💻 개발자 뷰 (Developer)
  • 토폴로지 뷰 — 내 앱들의 관계를 그래프로 시각화
  • "+Add" — Git 저장소·이미지·카탈로그에서 앱 배포
  • 빌드/파이프라인 진행 상황, 파드 로그 실시간 확인
  • YAML을 몰라도 배포 가능한 폼 기반 UI

바닐라 K8s의 Kubernetes Dashboard와 비교하면 차이가 확연합니다. Dashboard는 리소스를 "조회"하는 데 중점을 둔 별도 설치형 도구인 반면, OpenShift 콘솔은 OAuth 인증과 RBAC이 통합된 상태로 처음부터 제공되고, Operator 설치·업그레이드·빌드까지 콘솔에서 수행합니다. 특히 토폴로지 뷰는 "마이크로서비스가 서로 어떻게 연결되어 있는지"를 개발자가 직관적으로 파악하게 해주는, 다른 곳에서 보기 힘든 기능입니다.

콘솔에서 이런 것까지 됩니다

웹 터미널(브라우저에서 oc 셸 실행), YAML 에디터의 스키마 자동완성, 리소스별 이벤트·메트릭 통합 뷰, Helm 차트 카탈로그, 그리고 다크 모드까지. 초보자라면 콘솔에서 폼으로 리소스를 만들어 보고 생성된 YAML을 열어 학습하는 방법이 대단히 효과적입니다.

SECTION 11

S2I로 앱 배포하기 — OpenShift 개발자 경험의 정수

S2I(Source-to-Image)는 "Dockerfile 없이 소스 코드에서 곧바로 컨테이너 이미지를 만드는" OpenShift의 대표 기능입니다. 언어별 빌더 이미지(Java, Python, Node.js, Go...)가 소스를 받아 의존성 설치와 빌드를 수행하고 실행 가능한 이미지를 만들어 내장 레지스트리에 push, 곧바로 배포까지 이어집니다.

단 한 줄로 Git 저장소의 앱을 배포하고 외부 노출까지
# 언어를 자동 감지해 빌더 선택 → 빌드 → 배포까지
oc new-app https://github.com/sclorg/django-ex.git --name=myapp

# 외부 URL 발급 (Route 생성)
oc expose service/myapp

# 빌드 로그 실시간 확인
oc logs -f buildconfig/myapp

# 발급된 URL 확인
oc get route myapp -o jsonpath='{.spec.host}'

바닐라 K8s에서 같은 일을 하려면?

바닐라 K8s 워크플로우
  • ① Dockerfile 작성 (베이스 이미지, 보안 설정 직접 책임)
  • ② 외부 CI 구축 (GitHub Actions / Jenkins / GitLab CI)
  • ③ 외부 레지스트리 준비 (Harbor / ECR / Docker Hub)
  • ④ Deployment + Service + Ingress YAML 작성
  • ⑤ CI에서 kubectl/ArgoCD로 배포 연결
OpenShift 워크플로우
  • oc new-app <git-url> + oc expose — 이상 끝
  • 빌드(BuildConfig), 레지스트리, 배포, URL이 클러스터 안에서 완결
  • Git 웹훅 연결 시 push → 자동 재빌드·재배포
  • 본격 CI/CD가 필요하면 Pipelines(Tekton)·GitOps(Argo CD)로 확장
균형 잡힌 시각

S2I는 시작을 빠르게 해주지만, 조직의 CI/CD가 이미 GitHub Actions·GitLab 중심으로 표준화되어 있다면 굳이 S2I를 쓸 필요는 없습니다. 실제 현업에서도 "빌드는 외부 CI, 배포만 OpenShift"인 조직이 많습니다. S2I는 선택지이지 강제가 아닙니다 — 표준 Dockerfile 빌드와 외부 이미지 배포 모두 완벽 지원됩니다.

SECTION 12

Route로 외부 노출하기 — 트래픽의 흐름 이해

앱을 배포했으면 사용자가 접근할 수 있어야죠. OpenShift에서 HTTP(S) 트래픽이 사용자에서 파드까지 도달하는 경로를 이해하면 네트워크 트러블슈팅의 절반이 해결됩니다.

사용자 브라우저 → DNS (*.apps.cluster.example.com) → 외부 LB
→ Router 파드 (HAProxy, infra 노드) → Service → Pod
  • 와일드카드 DNS — 클러스터 설치 시 *.apps.<cluster-domain>가 Router를 가리키도록 설정됩니다. 그래서 Route를 만들면 DNS 작업 없이 즉시 접속 가능한 URL이 발급되는 것입니다.
  • TLS 3종 세트 — edge(Router에서 TLS 종료), passthrough(파드까지 암호화 유지), re-encrypt(Router에서 풀고 내부용 인증서로 재암호화). 규제 환경에서는 re-encrypt가 자주 요구됩니다.
  • Ingress/Gateway API와의 관계 — 표준 Ingress를 만들면 Route가 자동 생성되어 호환되고, 차세대 표준인 Gateway API도 지원이 확대되고 있습니다. Route를 배우는 것이 갈라파고스가 될 걱정은 하지 않아도 됩니다.
Route 생성과 확인 — 실무 기본기
# HTTP Route 생성 (호스트네임 자동 발급)
oc expose service my-app

# TLS edge Route 생성
oc create route edge my-app-tls --service=my-app --port=8080

# Route 목록과 URL 확인
oc get routes
# NAME     HOST/PORT                                PATH  SERVICES  PORT
# my-app   my-app-myproject.apps.ocp.example.com          my-app    8080
Service 타입과의 관계 정리

바닐라 K8s에서 외부 노출 수단으로 배우는 NodePort, LoadBalancer는 OpenShift에서도 그대로 쓸 수 있습니다. 다만 HTTP(S) 워크로드라면 Route가 관용적(idiomatic) 방식이고, TCP 등 비HTTP 프로토콜은 LoadBalancer 타입 Service나 NodePort를 사용합니다.

SECTION 13

적용 분야와 실제 사례 — 누가 왜 쓰는가

OpenShift의 고객층을 보면 이 플랫폼의 성격이 보입니다. 규제가 강하고, 온프렘/하이브리드 요구가 있고, 장애 시 책임 주체가 필요한 산업이 핵심 시장입니다.

🏦
금융 (은행·카드·증권)

망분리·감사 대응이 필수인 환경. SCC 기반의 엄격한 기본 보안, 폐쇄망(air-gapped) 설치 지원, 벤더 기술지원이 결정적 요인. 코어뱅킹 주변계와 채널계 시스템의 컨테이너 전환에 널리 채택.

국내외 대형 은행 다수가 프라이빗 클라우드 표준 플랫폼으로 채택
📡
통신 (Telco / 5G)

5G 코어와 RAN의 네트워크 기능이 CNF(컨테이너화된 네트워크 기능)로 전환되면서, 기지국 엣지까지 커버하는 SNO·MicroShift와 저지연 튜닝(SR-IOV, DPDK, PTP) 지원이 강점. 통신은 OpenShift의 대표 수직 시장.

글로벌 주요 통신사 5G 코어 인프라 / 국내 통신사 클라우드 전환
🏛️
공공·국방

인터넷 차단 환경 설치, 장기 지원(EUS), 컴플라이언스 인증 문서화가 요구되는 영역. 오픈소스이면서 상용 지원이 있는 조합이 조달 요건과 잘 맞음.

정부·공공기관 프라이빗 클라우드, 국방 분야 미션 시스템
🏭
제조·엣지

공장 라인, 매장, 차량 등 현장에 소형 클러스터(SNO/MicroShift)를 배포하고 RHACM으로 수백~수천 사이트를 중앙 관리하는 패턴. 레거시 VM은 OpenShift Virtualization으로 함께 수용.

스마트 팩토리 MES 주변계, 리테일 매장 엣지 컴퓨팅
🤖
AI/ML 플랫폼 (MLOps)

OpenShift AI(구 RHODS)가 Jupyter 워크벤치, 모델 서빙(KServe/vLLM), 파이프라인(Kubeflow 기반), GPU Operator를 통합 제공. 사내 데이터가 밖으로 못 나가는 조직의 온프렘 LLM 서빙·파인튜닝 플랫폼으로 부상.

금융·공공의 사내 sLLM 서빙, GPU 팜 멀티테넌시 운영
🖥️
VM 통합 (가상화 전환)

OpenShift Virtualization(KubeVirt 기반)으로 컨테이너와 VM을 한 플랫폼에서 운영. 가상화 라이선스 비용 이슈 이후 VMware 대안으로 검토하는 조직이 급증한 영역.

기존 가상화 인프라의 단계적 전환, VM+컨테이너 혼합 워크로드
MLOps 엔지니어 관점의 포인트

GPU 노드 운영 시 NVIDIA GPU Operator + Node Feature Discovery가 OperatorHub에서 클릭 설치되고, time-slicing·MIG 설정도 Operator의 커스텀 리소스로 선언 관리됩니다. 바닐라 K8s 대비 "GPU 드라이버·툴킷·디바이스 플러그인 수동 설치"의 고통이 크게 줄어드는 지점이며, 온프렘 LLM 서빙 플랫폼 구축 제안에서 OpenShift AI가 경쟁 구도에 자주 등장합니다.

OpenShift 스킬은 쿠버네티스 스킬 위에 쌓이는 것이라 상충하지 않습니다. K8s 표준 지식(CKA 수준)을 기반으로 OpenShift 운영 경험을 더하면 엔터프라이즈 시장(금융·통신·공공 SI/MSP)에서 특히 희소가치가 높습니다.

프로젝트/쿼터 관리, RBAC(oc adm policy), SCC 트러블슈팅, OperatorHub에서 Operator 설치, ClusterOperator 상태 읽기.

전문 트랙 선택
CI/CD(Pipelines·GitOps), 가상화(OpenShift Virtualization), AI/ML(OpenShift AI + GPU Operator), 네트워크 심화(서비스 메시) 중 커리어 방향에 맞게.
