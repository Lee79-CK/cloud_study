Confidential Containers (CoCo)

클라우드 관리자도, 하이퍼바이저도, 심지어 쿠버네티스 자신도 내 워크로드의 메모리를 들여다볼 수 없다면? 초보 클라우드 엔지니어를 위한 CoCo 기초 개념부터 Kata Containers와의 비교, 설치·사용법, 실전 응용 사례까지 한 번에 정리합니다.

난이도 입문 ~ 중급 대상 주니어 Cloud/Platform 엔지니어 선수 지식 K8s Pod · 컨테이너 런타임 기초 읽는 시간 약 40분

01왜 기밀 컨테이너인가 — 마지막 남은 빈틈, "사용 중 데이터"

보안 이야기를 할 때 데이터는 항상 세 가지 상태로 나눠 생각합니다. 저장 중(At Rest), 이동 중(In Transit), 그리고 사용 중(In Use). 앞의 두 가지는 이미 표준적인 해법이 있지만, 마지막 하나는 오랫동안 비어 있던 퍼즐 조각이었습니다.

✗ 전통적으로 보호 수단 없음 → 기밀 컴퓨팅이 해결

아무리 디스크를 암호화하고 TLS로 통신해도, 프로그램이 데이터를 처리하는 순간에는 메모리에 평문으로 존재합니다. 이때 메모리를 읽을 수 있는 사람은 누구일까요?

  • 호스트의 root 권한을 가진 사람 — 클라우드 사업자의 운영 인력, 침해당한 관리 계정
  • 하이퍼바이저 — VM의 메모리를 언제든 덤프할 수 있는 위치
  • 쿠버네티스 노드에 침투한 공격자 — 노드 하나가 뚫리면 그 위의 모든 Pod 메모리가 노출
  • 같은 물리 서버의 다른 테넌트 — 사이드 채널 공격 등
🏦
비유로 이해하기 — 은행 금고와 회의실

저장 중 데이터 암호화는 금고에 돈을 넣어두는 것이고, 이동 중 암호화는 현금수송차로 옮기는 것입니다. 그런데 돈을 세려면 결국 금고에서 꺼내 책상 위에 펼쳐야 하죠. 이때 건물 관리인(클라우드 사업자)이 회의실 유리창 너머로 다 볼 수 있다면? 기밀 컴퓨팅은 이 회의실을 불투명한 방탄 유리로 밀폐하고, 열쇠는 나만 갖는 것입니다.

클라우드 시대의 근본 질문 — "나는 클라우드 사업자를 믿어야만 하는가?"

퍼블릭 클라우드를 쓴다는 것은 지금까지 "인프라 제공자를 무조건 신뢰한다"는 암묵적 전제를 깔고 있었습니다. AWS, Azure, GCP의 내부 직원이 내 VM 메모리를 안 본다는 걸 기술적으로 보장할 방법은 없었고, 계약서와 컴플라이언스 인증(ISO 27001 등)으로 제도적으로 믿는 수밖에 없었죠.

하지만 다음과 같은 워크로드라면 이야기가 다릅니다.

  • 금융권의 고객 개인정보·신용정보 처리 (전자금융감독규정, 마이데이터)
  • 의료 데이터 분석 (HIPAA, 국내 의료법·개인정보보호법)
  • LLM 추론 서비스 — 모델 가중치(수백억 원짜리 자산)와 사용자 프롬프트(민감 정보) 모두 보호 필요
  • 여러 회사가 데이터를 합쳐 분석하되 서로의 원본은 못 보게 해야 하는 경우
ℹ️ 핵심 정리

기밀 컨테이너(Confidential Containers, CoCo)는 하드웨어 기반 신뢰 실행 환경(TEE) 안에서 컨테이너를 실행하여, 호스트·하이퍼바이저·클라우드 관리자로부터도 워크로드의 메모리와 데이터를 보호하는 기술입니다. 신뢰의 뿌리를 "클라우드 사업자의 선의"에서 "CPU 하드웨어"로 옮기는 패러다임 전환입니다.

02기밀 컴퓨팅과 TEE — CoCo를 떠받치는 하드웨어 기술

CoCo를 이해하려면 먼저 그 토대인 기밀 컴퓨팅(Confidential Computing)과 TEE(Trusted Execution Environment, 신뢰 실행 환경)를 알아야 합니다. 어렵게 들리지만 핵심은 두 가지뿐입니다.

ℹ️ TEE의 두 가지 약속

① 메모리 암호화 + 격리: TEE 내부의 메모리는 CPU 안에서만 복호화됩니다. 메모리 버스를 물리적으로 도청해도, 하이퍼바이저가 메모리를 덤프해도 암호문만 보입니다.

② 원격 증명(Remote Attestation): "지금 이 코드가 진짜 TEE 안에서, 변조 없이 실행되고 있다"는 사실을 하드웨어가 서명한 증거로 외부에 증명할 수 있습니다.

✉️
비유로 이해하기 — 봉인된 투표소

TEE는 안이 보이지 않는 투표소와 같습니다. 밖에서는 안에서 무슨 일이 일어나는지 볼 수 없고(메모리 암호화), 투표소 입구에는 "이 투표소는 규격대로 지어졌고 몰래카메라가 없음"을 보증하는 공인 봉인 스티커(원격 증명)가 붙어 있습니다. 유권자(데이터 소유자)는 이 봉인을 확인한 후에만 투표용지(비밀 키·데이터)를 건네줍니다.

주요 하드웨어 TEE 기술

TEE는 CPU 제조사가 실리콘 레벨에서 제공하는 기능입니다. 접근 방식에 따라 크게 프로세스 단위와 VM 단위로 나뉘며, CoCo는 주로 VM 단위 TEE를 사용합니다.

기술제조사격리 단위특징CoCo 지원
SEV-SNP
Secure Nested Paging		 AMD (EPYC 3세대+) VM 전체 VM 메모리 암호화 + 무결성 보호 + 하이퍼바이저의 메모리 재매핑 공격 차단. 애플리케이션 수정 불필요 ✓ 지원

💡 초보자를 위한 감 잡기

이름이 많아 복잡해 보이지만 이렇게 기억하세요. "AMD는 SEV-SNP, Intel은 TDX, 이 둘은 VM을 통째로 암호화하는 형제 같은 기술이고, CoCo가 주로 올라타는 토대"입니다. SGX는 더 오래된 프로세스 단위 방식으로, 앱을 고쳐야 해서 CoCo에서는 별도 트랙(enclave-cc)으로 다룹니다.

측정(Measurement)과 원격 증명 — 신뢰는 어떻게 만들어지는가

TEE가 부팅될 때, 하드웨어는 로딩되는 펌웨어·커널·초기 구성 요소의 해시값을 차곡차곡 기록합니다. 이를 측정(measurement)이라 합니다. 이후 외부에서 "너 정말 믿을 수 있는 상태니?"라고 물으면, TEE는 이 측정값들을 CPU 제조사의 키로 서명한 증거(evidence)로 만들어 제출합니다.

검증자는 이 증거를 받아 ① 서명이 진짜 AMD/Intel 하드웨어에서 왔는지, ② 측정값이 내가 기대한 값(내가 배포한 그 커널, 그 이미지가 맞는지)과 일치하는지 확인합니다. 둘 다 통과해야만 비밀(암호 키 등)을 건네줍니다. 이 과정이 6장에서 자세히 볼 Attestation 흐름의 뼈대입니다.

⚠️ 흔한 오해 바로잡기

기밀 컴퓨팅은 애플리케이션 자체의 취약점을 막아주지 않습니다. TEE 안에서 돌아가는 앱에 SQL 인젝션 구멍이 있으면 여전히 뚫립니다. TEE가 막는 것은 "바깥(호스트·하이퍼바이저·물리적 접근)에서 안을 들여다보거나 조작하는 행위"입니다. 보호 대상과 위협 모델을 정확히 구분하는 것이 중요합니다.

03CoCo 프로젝트 소개 — 기밀 컴퓨팅을 쿠버네티스 네이티브로

Confidential Containers(줄여서 CoCo)는 2022년 CNCF 샌드박스 프로젝트로 출범한 오픈소스 프로젝트로, 목표는 단 하나입니다. "쿠버네티스 사용자가 YAML 한 줄만 바꿔서 기밀 컴퓨팅을 쓸 수 있게 하자."

TEE 하드웨어 자체는 이미 존재했지만, 이를 실제로 쓰려면 하이퍼바이저 설정, 게스트 이미지 빌드, 증명 서버 구축, 키 배포까지 전부 직접 엮어야 했습니다. CoCo는 이 복잡한 조립을 쿠버네티스의 표준 추상화(RuntimeClass, Operator, Pod Spec) 뒤로 숨깁니다.

사용자 입장에서 CoCo를 쓴다는 것 — 딱 이 한 줄
apiVersion: v1
kind: Pod
metadata:
  name: my-secret-workload
spec:
  runtimeClassName: kata-qemu-tdx   # ← 이 한 줄이 전부!
  containers:
  - name: app
    image: registry.example.com/secret-app:1.0

CoCo가 약속하는 것 — 위협 모델

CoCo의 위협 모델은 명확합니다. "워크로드(Pod) 입장에서, 인프라 전체를 신뢰하지 않는다."

주체일반 컨테이너에서CoCo에서
클라우드 사업자 / 호스트 관리자Pod 메모리·환경변수·볼륨 전부 열람 가능메모리 암호화로 열람 불가
하이퍼바이저 / 호스트 커널신뢰 대상 (뚫리면 끝)신뢰하지 않음 (신뢰 경계 밖)
쿠버네티스 컨트롤 플레인Secret, exec 등으로 내부 접근 가능기본적으로 신뢰하지 않음 (exec/log 차단 가능)
같은 노드의 다른 Pod네임스페이스 격리 (커널 공유)VM + 메모리 암호화로 격리
CPU 하드웨어 (AMD/Intel)신뢰유일한 신뢰의 뿌리(Root of Trust)
📦
비유로 이해하기 — 남의 창고에 놓는 금고형 컨테이너

일반 컨테이너는 남의 창고(클라우드)에 놓아둔 투명한 상자입니다. 창고 주인은 마음만 먹으면 내용물을 볼 수 있죠. CoCo는 같은 창고에 놓지만 내용물이 보이지 않고, 열쇠는 원격의 나만 가진 금고입니다. 심지어 금고에는 "정품 인증 장치"가 붙어 있어서, 내가 원격으로 "이 금고는 변조되지 않은 진짜"임을 확인한 뒤에만 열쇠를 꽂아 줍니다.

CoCo의 설계 철학 3가지

  • ① 무수정 배포 (Lift & Shift) — 기존 컨테이너 이미지를 코드 수정 없이 그대로 기밀 환경에서 실행. SGX처럼 앱을 다시 짜지 않아도 됩니다.
  • ② 쿠버네티스 네이티브 — Operator로 설치하고, RuntimeClass로 선택하고, Pod Spec으로 사용. 새로운 CLI나 별도 플랫폼을 배우지 않아도 됩니다.
  • ③ 하드웨어 중립 — AMD SEV-SNP든 Intel TDX든 IBM SE든, 동일한 사용자 경험. 증명 프로토콜과 컴포넌트를 공통화했습니다.
ℹ️ 프로젝트 계보 한눈에 보기

CoCo는 맨땅에서 시작한 프로젝트가 아닙니다. Kata Containers(VM 기반 격리 런타임)를 실행 기반으로 삼고, 그 위에 기밀 VM + 증명(Attestation) + 이미지 보호를 얹은 것입니다. Intel, IBM, Red Hat, Alibaba, Microsoft 등이 주요 기여사이며, Red Hat OpenShift Sandboxed Containers, Azure Confidential Containers(AKS) 같은 상용 제품의 상류(upstream)이기도 합니다.

04Kata vs CoCo — 화살표의 방향이 다르다

CoCo를 가장 빠르게 이해하는 방법은 Kata Containers와 비교하는 것입니다. 결론부터 말하면, CoCo는 Kata를 대체하는 게 아니라 Kata 위에 지어진 확장이며, 두 기술은 보호의 방향이 다릅니다.

🛡️ Kata Containers
"호스트를 컨테이너로부터 보호한다"
  • 컨테이너를 경량 VM 안에서 실행 → 커널을 공유하지 않음
  • 악성/취약 컨테이너가 탈출해도 호스트와 이웃 Pod이 안전
  • 위협의 화살표: 컨테이너 → 호스트 를 차단
  • 전제: 호스트(인프라)는 여전히 신뢰. 호스트 관리자는 VM 메모리를 볼 수 있음
  • 대표 용례: 멀티테넌트 CaaS, 신뢰할 수 없는 코드 실행(CI, FaaS)
🔐 Confidential Containers
"컨테이너를 호스트로부터도 보호한다"
  • Kata의 VM을 기밀 VM(TEE)으로 교체 → 메모리 암호화
  • 호스트·하이퍼바이저·클라우드 관리자가 Pod 내부를 볼 수 없음
  • 위협의 화살표: 호스트 → 컨테이너 까지 차단 (양방향 보호)
  • 원격 증명 + 암호화 이미지 + 봉인 시크릿으로 데이터·코드 기밀성 보장
  • 대표 용례: 규제 데이터 처리, 기밀 AI 추론, 제로 트러스트 클라우드
격리 스펙트럼: runc(커널 공유) → Kata(VM 격리) → CoCo(VM 격리 + 메모리 암호화 + 증명)

💡 한 문장 공식

CoCo = Kata Containers + 기밀 VM(TEE) + 원격 증명 + 게스트 이미지 풀 + 키/시크릿 보호. Kata가 "격리의 뼈대"라면 CoCo는 그 위에 "기밀성의 근육"을 붙인 것입니다. 실제로 CoCo의 런타임 바이너리와 RuntimeClass 이름에 kata- 접두사가 그대로 남아 있는 이유입니다.

⚠️ 자주 하는 질문 — "Kata만으로는 왜 부족한가요?"

Kata의 VM은 일반 VM입니다. 하이퍼바이저(QEMU/KVM)와 호스트 root는 그 VM의 메모리를 자유롭게 읽을 수 있습니다. 즉 Kata는 "이웃과 집주인을 내 세입자로부터 지키는 기술"이고, CoCo는 거기에 더해 "내 방을 집주인으로부터도 지키는 기술"입니다. 집주인(클라우드 사업자)을 100% 신뢰한다면 Kata로 충분하지만, 그렇지 않다면 CoCo가 필요합니다.

05CoCo 아키텍처 — 무엇이 어디에서 돌아가는가

CoCo의 구성 요소는 "신뢰 경계의 안(게스트)이냐 밖(호스트)이냐"로 나누면 깔끔하게 정리됩니다. 신뢰 경계 안에 있는 것만 믿고, 밖에 있는 것은 전부 의심한다는 원칙만 기억하세요.

CoCo 아키텍처: 초록 경계(기밀 VM) 안이 신뢰 영역, 노드의 나머지는 전부 비신뢰. Trustee는 데이터 소유자가 통제하는 "열쇠 보관소"

구성 요소 하나씩 뜯어보기

① 호스트 쪽 (신뢰하지 않는 영역)

  • Kata Runtime (containerd-shim-kata-v2) — containerd의 요청을 받아 QEMU/Cloud Hypervisor로 기밀 VM을 부팅합니다. Kata Containers의 그 런타임에 TEE 옵션이 추가된 것입니다.
  • CoCo Operator — 노드에 Kata 런타임·설정·RuntimeClass를 설치·관리하는 쿠버네티스 오퍼레이터. 관리자는 이것만 배포하면 됩니다.

② 게스트 쪽 (기밀 VM 내부 = 신뢰 영역)

  • kata-agent — VM 안의 총감독. 호스트의 shim과 통신하며 컨테이너 생성·라이프사이클을 관리합니다. CoCo에서는 위험한 API(exec 등)를 정책으로 잠글 수 있습니다.
  • image-rs (게스트 풀) — CoCo의 가장 중요한 차별점 중 하나. 컨테이너 이미지를 호스트가 아니라 VM 안에서 직접 레지스트리로부터 pull 합니다. 덕분에 호스트는 복호화된 이미지 레이어를 구경도 못 합니다. 서명 검증(cosign 등)과 암호화 이미지 복호화도 여기서 처리합니다.
  • Attestation Agent (AA) — TEE 하드웨어로부터 증거(evidence)를 뽑아 KBS에 제출하는 담당자.
  • Confidential Data Hub (CDH) — 봉인된 시크릿(sealed secret) 해제, KMS 연동, 블록 볼륨 암호화 등 "비밀 자원 사용"의 창구.

③ Trustee (데이터 소유자의 진영)

  • KBS (Key Broker Service) — 열쇠 보관소의 문지기. "증명에 통과한 TEE에게만" 키·시크릿을 내어줍니다. 클러스터 밖(온프레미스, 별도 클라우드)에 둘 수 있다는 점이 핵심입니다.
  • AS (Attestation Service) — 제출된 증거의 하드웨어 서명과 측정값을 실제로 검증하는 감정사.
  • RVPS (Reference Value Provider Service) — "정답지" 보관소. 내가 배포한 커널·rootfs의 올바른 해시값(기준값)을 관리합니다.
🎭
비유로 이해하기 — 공연장 출입 시스템

AA는 신분증(하드웨어 증거)을 들고 온 관객, AS는 신분증 위조 여부를 감별하는 검표원, RVPS는 초대 명단(기준 측정값), KBS는 검표원의 OK 사인을 받은 뒤에야 공연장 열쇠(암호 키)를 건네는 매표소장입니다. 넷이 합쳐진 것이 Trustee라는 하나의 출입 관리 사무소입니다.

ℹ️ 왜 "게스트 풀"이 그렇게 중요한가요?

일반 쿠버네티스에서 이미지는 노드의 containerd가 받아서 노드 디스크에 풀어놓습니다. 호스트 관리자는 /var/lib/containerd만 뒤져도 이미지 내용물(코드, 모델 가중치, 임베디드 설정)을 전부 볼 수 있죠. CoCo는 이미지를 기밀 VM 안에서 직접 받기 때문에, 암호화 이미지라면 호스트는 시작부터 끝까지 암호문만 봅니다. "코드와 모델도 데이터다"라는 관점에서 필수적인 설계입니다.

06Attestation 흐름 — 열쇠는 어떻게 안전하게 전달되는가

CoCo의 심장은 원격 증명입니다. "암호화된 이미지를 쓰려면 키가 필요한데, 그 키를 믿을 수 있는 TEE에게만 주려면 어떻게 해야 할까?"라는 문제를 푸는 과정을 단계별로 따라가 봅시다.

사용자 → K8s
Pod 배포 요청

runtimeClassName: kata-qemu-snp가 붙은 Pod을 apply 합니다. 스케줄러가 TEE 지원 노드에 배치합니다.

Kata Runtime → 하드웨어
기밀 VM 부팅 + 측정

Kata 런타임이 기밀 VM을 띄웁니다. 부팅 과정에서 CPU가 펌웨어·커널·initrd의 해시를 측정(measure)하여 하드웨어 레지스터에 기록합니다. 이 순간부터 VM 메모리는 암호화 상태입니다.

kata-agent → image-rs
게스트 풀 시작 → "어? 키가 필요하네"

VM 안의 image-rs가 레지스트리에서 이미지를 pull 합니다. 암호화된 레이어를 만나면 복호화 키가 필요하다는 것을 알게 되고, CDH/AA에게 키를 요청합니다.

AA → KBS
증거 수집과 제출 (RCAR 핸드셰이크)

Attestation Agent가 KBS에 접속하면 KBS는 일회용 챌린지(nonce)를 던집니다. AA는 TEE 하드웨어에게 "nonce를 포함한 증거를 만들어줘"라고 요청하고, CPU가 측정값 + nonce를 제조사 키로 서명한 증거를 생성해 KBS에 제출합니다.

KBS → AS → RVPS
검증 — 진짜인가? 기대한 상태인가?

AS가 ① 서명이 진짜 AMD/Intel 하드웨어의 것인지(인증서 체인), ② 측정값이 RVPS의 기준값과 일치하는지(내가 배포한 그 커널·rootfs가 맞는지) 확인합니다. 하나라도 어긋나면 키 배포 거부.

KBS → 기밀 VM
키 전달 → 이미지 복호화 → 컨테이너 기동

검증 통과 시 KBS가 복호화 키를 암호화된 채널로 VM 안에만 전달합니다. image-rs가 이미지를 복호화해 컨테이너를 시작합니다. 이 모든 과정에서 호스트는 암호문과 통신 트래픽만 봤을 뿐입니다.

⚠️ 왜 nonce(일회용 값)가 필요한가요?

nonce가 없다면 공격자가 예전에 정상 TEE가 만들었던 증거를 녹화해 두었다가 재생(replay)할 수 있습니다. 매번 새로운 nonce를 증거에 포함시키도록 강제하면, 증거는 "지금 이 순간, 이 요청에 대한" 것임이 보장됩니다. TLS 핸드셰이크의 랜덤값과 같은 원리입니다.

💡 봉인 시크릿(Sealed Secret)도 같은 원리

암호화 이미지의 키뿐 아니라, 애플리케이션이 쓰는 DB 비밀번호·API 키도 같은 흐름으로 보호할 수 있습니다. 쿠버네티스 Secret에는 가짜 봉투(sealed secret 참조값)만 넣어두고, 진짜 값은 KBS에 보관합니다. Pod이 뜨면 CDH가 증명을 거쳐 진짜 값으로 바꿔치기하므로, etcd가 털려도 진짜 시크릿은 안전합니다.

07설치와 사용 방법 — Operator 배포부터 암호화 이미지까지

CoCo의 설치는 크게 ① Operator 설치 → ② CcRuntime 생성 → ③ RuntimeClass로 Pod 실행의 3단계입니다. TEE 하드웨어가 없어도 coco-dev 런타임으로 전체 흐름을 연습할 수 있으니, 학습용 클러스터에서 그대로 따라 해보세요.

사전 요구사항

항목학습/개발용 (coco-dev)운영용 (실제 TEE)
CPU가상화 지원 (VT-x / AMD-V)AMD SEV-SNP 또는 Intel TDX 지원 CPU + BIOS 활성화
노드 OSUbuntu 22.04+ 등 (커널이 해당 TEE 지원 필요)
K8s1.24+ 권장, containerd 런타임
중첩 가상화VM 위 K8s라면 nested virtualization 활성화 필요베어메탈 권장 (또는 클라우드 기밀 VM 인스턴스 + peer-pods)

Step 1 — CoCo Operator 설치

operator-install.sh
# 1) 워커 노드에 라벨 부여 (Operator가 이 라벨을 보고 설치 대상 선정)

Step 2 — CcRuntime 생성 (런타임 실제 설치)

Operator는 관리자일 뿐, 실제 Kata/CoCo 런타임 설치는 CcRuntime이라는 커스텀 리소스를 만들어야 시작됩니다. 설치가 끝나면 여러 개의 RuntimeClass가 자동 생성됩니다.

ccruntime-and-verify.sh
# 1) 기본 CcRuntime 배포 (샘플 매니페스트 사용)

# 2) 설치 완료 후 생성된 RuntimeClass 확인
kubectl get runtimeclass
# NAME                 HANDLER              AGE
# kata                 kata-qemu            2m   ← 일반 Kata (기밀 아님)
# kata-qemu            kata-qemu            2m
# kata-qemu-coco-dev   kata-qemu-coco-dev   2m   ← TEE 없이 CoCo 흐름 체험 (학습용!)

# kata-qemu-tdx        kata-qemu-tdx        2m   ← Intel TDX
💡 kata-qemu-coco-dev의 정체

이름 그대로 개발·학습용입니다. 실제 TEE 하드웨어 없이 일반 VM 위에서 게스트 풀, 증명 흐름(샘플 attester), 봉인 시크릿 등 CoCo의 소프트웨어 스택 전체를 체험할 수 있습니다. 단, 메모리 암호화가 없으므로 실제 보안성은 없습니다. "운전 연습장"이라고 생각하세요. 운영 환경에서는 반드시 -snp, -tdx 클래스를 사용해야 합니다.

    ports:
    - containerPort: 80

Step 4 — Trustee(KBS) 배포와 암호화 이미지 사용

여기서부터가 진짜 CoCo입니다. 데이터 소유자 입장에서 ① 이미지를 암호화해 올리고, ② 키를 KBS에 보관하고, ③ 증명을 통과한 Pod만 복호화하도록 구성합니다.

deploy-trustee.sh — 열쇠 보관소 구축
# Trustee(KBS+AS+RVPS) 저장소 클론
git clone https://github.com/confidential-containers/trustee.git
cd trustee/kbs/config/kubernetes

# KBS 인증용 키 쌍 생성 (관리자만 KBS에 리소스 등록 가능하도록)
openssl genpkey -algorithm ed25519 > base/kbs.key
openssl pkey -in base/kbs.key -pubout -out base/kbs.pem

# KBS 배포 (기본: coco-tenant 네임스페이스)
kubectl apply -k base/

kubectl get pods -n coco-tenant
# kbs-xxxxxxxxx   1/1   Running
encrypt-image.sh — skopeo + ocicrypt로 이미지 암호화
# 1) 32바이트 대칭 키 생성
head -c 32 /dev/urandom | base64 > image_key.b64

# 2) ocicrypt 설정: 키 프로바이더가 KBS 리소스 경로를 가리키도록 구성
#    (coco_keyprovider 사용 — KBS 리소스 URI: kbs:///default/image-key/nginx1)

# 3) skopeo로 암호화하며 레지스트리에 push
skopeo copy --encryption-key provider:attestation-agent:keypath=... \
  docker://nginx:1.27 \
  docker://registry.example.com/team/nginx-encrypted:1.27

# 4) 결과 확인 — 레이어 미디어타입이 +encrypted 로 바뀜
skopeo inspect docker://registry.example.com/team/nginx-encrypted:1.27 \
  | grep MIMEType
# "application/vnd.oci.image.layer.v1.tar+gzip+encrypted"
register-key.sh — KBS에 복호화 키 등록
# KBS 클라이언트로 키를 리소스 경로에 등록
# 경로 규칙: {repository}/{type}/{tag} 3단 구조
kbs-client --url http://kbs.coco-tenant:8080 \
  config --auth-private-key kbs.key \
  set-resource \
  --path default/image-key/nginx1 \

클라우드에서는? — Peer Pods (cloud-api-adaptor)

지금까지는 "TEE 지원 베어메탈 노드"를 전제했지만, 현실의 많은 클러스터는 클라우드 VM 위에서 돕니다. VM 안에서 또 기밀 VM을 만들 수는 없으니(중첩 제약), CoCo는 Peer Pods라는 우회로를 제공합니다.

Peer Pods: 노드 내부에 VM을 만드는 대신, 클라우드 API로 노드와 나란히 서는 기밀 VM 인스턴스를 만들어 그 안에서 Pod을 실행

Pod 하나가 곧 클라우드의 기밀 VM 인스턴스 하나가 되는 방식으로, Azure·AWS·IBM Cloud·libvirt 등 다양한 프로바이더를 지원합니다. Azure의 "Confidential Containers on AKS"가 이 계열의 대표적인 관리형 구현입니다.

08응용 사례 — CoCo는 어디에 쓰이는가

사례 1 · 기밀 AI/LLM 추론 (Confidential AI Inference)

요즘 CoCo가 가장 뜨겁게 논의되는 영역입니다. LLM 서비스에는 지켜야 할 자산이 둘 있습니다.

  • 모델 가중치 — 수십~수백억 원을 들여 학습한 회사의 핵심 자산. GPU 클라우드에 올리는 순간 인프라 관리자에게 노출될 수 있음
  • 사용자 프롬프트/응답 — 의료 상담, 사내 기밀이 섞인 질문 등 민감 정보의 집합체

CoCo 구성: 모델 가중치를 암호화 이미지(또는 암호화 볼륨)로 배포하고, vLLM 등의 추론 서버를 기밀 VM에서 실행합니다. NVIDIA H100/H200의 Confidential Computing 모드를 CPU TEE와 연계하면 GPU 메모리까지 보호 범위에 들어옵니다. 결과적으로 "GPU 클라우드 사업자도 모델과 프롬프트를 볼 수 없는 추론 서비스"가 가능해집니다.

💡 MLOps 엔지니어의 시선

온프레미스 GPU가 부족해 퍼블릭 클라우드로 추론을 확장(burst)하고 싶은데 "모델 유출이 걱정돼서 못 나간다"는 조직이 많습니다. CoCo + GPU CC는 정확히 이 딜레마를 푸는 조합입니다. 사내 KBS는 온프레미스에 두고, 클라우드의 기밀 Pod이 증명을 통과할 때만 모델 복호화 키를 내주는 구조를 만들 수 있습니다.

사례 2 · 금융·의료 등 규제 데이터 처리

금융권의 신용평가 모델 실행, 병원의 유전체 분석 파이프라인처럼 "클라우드로 가고 싶지만 규제·감사 때문에 못 가는" 워크로드가 대상입니다. CoCo를 쓰면 "클라우드 사업자조차 데이터에 접근할 수 없음"을 기술적으로 증명할 수 있어, 컴플라이언스 논의의 성격이 "믿어주세요"에서 "검증하세요"로 바뀝니다. 원격 증명 로그 자체가 감사 증적이 됩니다.

사례 3 · 다자간 데이터 협력 (Multi-Party Computation 스타일)

🤝
시나리오 — 서로 못 믿는 두 회사의 공동 분석

보험사 A와 병원 B가 데이터를 합쳐 질병 예측 모델을 만들고 싶지만, 서로에게 원본 데이터를 줄 수는 없습니다. 해법: 중립 클라우드에 CoCo Pod을 띄우고, 그 안에서만 두 데이터를 합쳐 분석합니다. A와 B는 각자 원격 증명으로 "합의된 분석 코드만 돌아가는 TEE"임을 확인한 뒤 각자의 KBS에서 데이터 복호화 키를 내줍니다. 결과(모델)만 나오고, 원본은 누구도 못 봅니다. 클라우드 사업자조차도요.

사례 4 · SaaS 멀티테넌시의 신뢰 업그레이드

B2B SaaS에서 대기업 고객이 가장 자주 하는 질문: "당신네 운영자가 우리 데이터를 볼 수 있나요?" 테넌트별 워크로드를 CoCo Pod으로 격리하면 "우리 운영진도 기술적으로 볼 수 없습니다"라고 답할 수 있고, 이는 강력한 영업 차별화 포인트가 됩니다. 고객이 직접 KBS를 운영하며 키 주권을 갖는 BYOK를 넘어선 'Hold Your Own Key' 모델도 가능합니다.

사례 5 · 엣지·통신 인프라

물리 보안이 취약한 엣지 사이트(기지국, 공장, 매장)에서는 서버 자체를 도난·물리 접근당할 위험이 있습니다. 통신사(telco) 환경의 5G 코어 함수, 매장의 결제 처리 워크로드를 CoCo로 실행하면 메모리 덤프·콜드부트 공격에도 데이터가 암호문으로만 존재합니다. 통신·엣지 사업자에게 특히 매력적인 이유입니다.

사례보호 대상핵심 CoCo 기능키 소유자
기밀 AI 추론모델 가중치, 프롬프트암호화 이미지/볼륨 + GPU CC 연계모델 소유 기업 (온프레미스 KBS)
규제 데이터 처리개인정보, 의료·금융 데이터증명 기반 키 배포, 감사 증적데이터 관리 조직
다자간 협력각 사의 원본 데이터코드 측정값 상호 검증참여사 각자 (복수 KBS)
SaaS 멀티테넌시테넌트 데이터테넌트별 기밀 Pod + 봉인 시크릿고객 (HYOK)
엣지/통신현장 처리 데이터물리 공격 대응 메모리 암호화중앙 운영 조직

09한계와 운영 고려사항 — 도입 전에 알아야 할 것들

CoCo는 만능이 아닙니다. 실무에서 마주칠 제약과 트레이드오프를 솔직하게 정리합니다. 이 부분을 알고 도입해야 "기대와 현실의 간극"으로 고생하지 않습니다.

① 성능과 리소스 오버헤드

  • 기동 시간 — VM 부팅 + 원격 증명 + Pod마다 게스트 풀(이미지 다시 받기). 노드 이미지 캐시를 활용할 수 없어, 큰 이미지일수록 콜드 스타트가 길어집니다. 스케일아웃이 빈번한 워크로드는 이미지 경량화가 필수입니다.
  • 메모리 — Pod마다 게스트 커널 + 에이전트가 올라가는 VM 하나씩. 밀도(density)가 runc 대비 낮아집니다.
  • CPU/IO — 메모리 암호화 자체의 오버헤드는 최신 CPU에서 크지 않지만, virtio 경유 IO와 부족한 페이지 공유로 워크로드에 따라 체감 차이가 있습니다.

② 운영 경험의 변화 — "디버깅이 불편해지는 것은 버그가 아니라 기능"

🚫 exec·logs가 안 되는 이유

kubectl exec가 된다는 것은 클러스터 관리자가 기밀 Pod 내부에 들어갈 수 있다는 뜻이고, 이는 CoCo의 위협 모델 전체를 무너뜨립니다. 그래서 운영용 CoCo에서는 kata-agent 정책으로 exec/log API를 차단하는 것이 권장됩니다. 디버깅은 앱 레벨 로깅(외부 전송), 관측용 사이드카, 개발 환경의 coco-dev 런타임으로 해결해야 합니다. 운영 방식 자체를 다시 설계해야 한다는 뜻입니다.

③ 신뢰 경계에 대한 정확한 이해

  • 앱 취약점은 그대로 — TEE는 외부 관찰·변조를 막을 뿐, 앱의 RCE·인젝션 취약점을 막지 않습니다. 취약한 앱을 TEE에 넣으면 "공격자까지 보호받는" 역설이 생깁니다.
  • 사이드 채널 — 캐시 타이밍, 전력 분석 등 하드웨어 사이드 채널 연구는 계속되고 있으며, CPU 마이크로코드 패치 관리가 중요합니다.
  • 가용성은 보장 안 됨 — 호스트는 여전히 VM을 끌 수 있습니다. TEE가 보장하는 것은 기밀성·무결성이지 가용성이 아닙니다.
  • 네트워크는 별개 — Pod 밖으로 나가는 트래픽 보호는 여전히 TLS/mTLS의 몫입니다.

④ 인프라·생태계 제약

  • TEE 지원 CPU(EPYC Milan+/Xeon Sapphire Rapids+)와 BIOS·커널·펌웨어 버전 정합성 관리가 필요합니다.
  • 버전에 따라 GPU 패스스루, 일부 CSI/CNI 조합 등 기능 제약이 있을 수 있어 PoC로 자사 스택 호환성 검증이 선행되어야 합니다.
  • Trustee(KBS)는 새로운 단일 장애점이자 최고 등급 보안 자산이 됩니다. KBS의 HA 구성과 접근 통제 설계가 CoCo 도입의 절반입니다.
ℹ️ 도입 판단 체크리스트

다음 중 하나라도 "예"라면 CoCo 검토 가치가 충분합니다. ① 인프라 관리자(내부자 포함)로부터 데이터를 보호해야 하는 규제·계약 요건이 있다 ② 클라우드 사업자를 신뢰 경계에서 제외해야 한다 ③ 모델·코드 같은 IP를 남의 인프라에서 실행해야 한다 ④ 고객에게 "우리도 못 봅니다"를 기술적으로 증명해야 한다. 반대로 단순히 "컨테이너 격리를 강화하고 싶다"가 목적이라면 Kata만으로 충분하고 운영 부담도 훨씬 적습니다.

10확인 퀴즈 — 10문제로 점검하기

각 문항의 보기를 선택한 뒤 맨 아래 채점하기 버튼을 누르세요. 정답과 해설이 표시됩니다.

Q1
데이터의 세 가지 상태 중, 기밀 컴퓨팅(그리고 CoCo)이 새롭게 보호하는 상태는?
저장 중은 디스크 암호화, 이동 중은 TLS로 이미 해결되어 있었습니다. 기밀 컴퓨팅은 마지막 빈틈인 "사용 중(메모리에서 처리 중)" 데이터를 하드웨어 메모리 암호화로 보호합니다.
Q2
TEE가 제공하는 두 가지 핵심 능력의 올바른 조합은?
TEE의 두 약속은 ① 내부 메모리를 밖에서 볼 수 없게 하는 암호화·격리, ② "변조 없는 진짜 TEE"임을 하드웨어 서명으로 증명하는 원격 증명입니다.
Q3
Kata Containers와 CoCo의 관계·차이에 대한 설명으로 가장 정확한 것은?
Kata의 위협 모델은 "컨테이너 → 호스트" 방향 차단(호스트는 신뢰), CoCo는 여기에 기밀 VM·증명을 더해 "호스트 → 컨테이너" 방향까지 차단합니다. CoCo는 Kata를 실행 기반으로 사용하는 확장입니다.
Q4
CoCo가 컨테이너 이미지를 호스트(containerd)가 아니라 기밀 VM 안에서 직접 pull(게스트 풀)하는 가장 중요한 이유는?
호스트가 이미지를 받으면 관리자가 노드 디스크에서 내용물을 그대로 볼 수 있습니다. 게스트 풀은 복호화가 신뢰 경계(TEE) 안에서만 일어나게 만들며, 성능 면에서는 오히려 캐시를 못 써서 손해입니다(보안을 위한 트레이드오프).
Q5
Attestation 흐름에서 KBS(Key Broker Service)의 역할은?
KBS는 문지기입니다. 실제 증거 검증은 AS(Attestation Service)가, 기준값 관리는 RVPS가 하고, KBS는 그 결과에 따라 키를 내줄지 말지 결정·배포합니다.
Q6
RuntimeClass kata-qemu-coco-dev의 올바른 용도는?
coco-dev는 일반 VM 위에서 게스트 풀·증명 흐름 등을 체험하는 학습·개발용입니다. 메모리 암호화가 없으므로 실제 보안성은 없고, 운영에서는 -snp/-tdx를 써야 합니다.
Q7
AMD와 Intel의 "VM 단위" 기밀 컴퓨팅 기술을 올바르게 짝지은 것은?
AMD는 SEV-SNP, Intel은 TDX가 VM 전체를 암호화하는 형제 기술입니다. SGX는 Intel의 프로세스(엔클레이브) 단위 기술, CCA는 ARM 진영입니다.
Q8
Trustee 구성 요소 중 "내가 배포한 커널·rootfs의 올바른(기대되는) 측정값"을 관리하는 것은?
RVPS는 정답지 보관소입니다. AS는 제출된 증거의 측정값을 RVPS의 기준값과 대조하여 "기대한 소프트웨어 상태인지"를 판정합니다.
Q9
운영 환경의 CoCo에서 kubectl exec를 정책으로 차단하기를 권장하는 이유는?
CoCo의 목표는 "인프라 관리자도 못 들어오게" 하는 것입니다. exec/log가 열려 있으면 컨트롤 플레인을 장악한 공격자(또는 내부자)가 그대로 내부에 접근할 수 있으므로, kata-agent 정책으로 차단하는 것이 권장됩니다.
Q10
Peer Pods(cloud-api-adaptor)가 해결하는 문제는?
일반 클라우드 VM 위의 노드에서는 그 안에 기밀 VM을 또 만들 수 없습니다. Peer Pods는 클라우드 API로 노드 "옆에" 기밀 VM 인스턴스를 생성해 Pod을 그 안에서 실행합니다. Azure Confidential Containers on AKS가 대표 사례입니다.

11핵심 요약 & 참고자료

TAKEAWAY 01

CoCo는 사용 중 데이터(Data in Use)를 보호하는 마지막 퍼즐. 신뢰의 뿌리를 클라우드 사업자에서 CPU 하드웨어로 옮긴다.

TAKEAWAY 02

Kata는 호스트를 컨테이너로부터, CoCo는 컨테이너를 호스트로부터도 보호. CoCo = Kata + TEE + 증명 + 게스트 풀 + 키 보호.

TAKEAWAY 03

하드웨어 기반: AMD SEV-SNP, Intel TDX(VM 단위)가 주력. 앱 수정 없이 기존 이미지를 그대로 실행한다.

TAKEAWAY 04

심장은 원격 증명: AA가 증거 제출 → AS가 검증(RVPS 기준값 대조) → KBS가 키 배포. 통과 못 하면 키도 없다.

TAKEAWAY 05

사용법은 Operator 설치 + RuntimeClass 한 줄. 학습은 coco-dev로, 운영은 -snp/-tdx로, 클라우드는 Peer Pods로.

TAKEAWAY 06

대가도 있다: 콜드 스타트·밀도 저하, exec/디버깅 차단, KBS라는 새 핵심 자산. 위협 모델이 맞을 때만 도입하라.
