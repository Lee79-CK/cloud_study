Part 1 — 01

왜 AIOps가 필요해졌을까?

AIOps를 이해하는 가장 빠른 방법은 "왜 등장했는가"를 먼저 이해하는 것입니다. 답은 간단합니다. 클라우드가 사람이 눈으로 감당할 수 있는 규모를 넘어섰기 때문입니다.

1.1운영 환경이 폭발적으로 복잡해졌다

과거 온프레미스 시절에는 서버 수십 대를 담당자가 "잘 아는 시스템"으로 관리할 수 있었습니다. 하지만 지금은 상황이 완전히 다릅니다.

🏢

과거: 모놀리식

서버 수십 대, 애플리케이션 몇 개. 담당자가 시스템 전체를 머릿속에 그릴 수 있었습니다.

☁️

현재: 멀티 클라우드

AWS + Azure + KT Cloud처럼 여러 클라우드에 걸친 하이브리드 환경이 기본이 되었습니다.

📦

마이크로서비스 + K8s

하나의 서비스가 수백 개의 Pod로 쪼개지고, 컨테이너는 수시로 생성·소멸합니다.

🔄

배포 속도

월 1회 배포가 하루 수십 회 배포로. 변경이 잦을수록 장애 원인 후보도 늘어납니다.

1.2사람의 눈은 이미 한계에 도달했다

복잡해진 만큼 시스템이 쏟아내는 텔레메트리 데이터(메트릭, 로그, 이벤트)의 양도 폭증했습니다. 중견 규모 서비스 하나가 하루에 수 테라바이트의 로그를 만들어내는 것이 흔한 시대입니다.

현장의 고통: 알람 피로 (Alert Fatigue)

운영자가 하루 수백 건의 알람에 노출되면, 정말 중요한 알람을 놓치기 시작합니다. "늑대가 나타났다"를 너무 많이 들으면 진짜 늑대를 못 잡는 것과 같습니다. 이것이 AIOps가 해결하려는 첫 번째 문제입니다.

1.3그래서 나온 답: 운영에 AI를 붙이자

사람이 규칙(Rule)을 하나하나 만들어 감시하는 방식은 이 규모를 따라갈 수 없습니다. 그래서 기계가 데이터에서 스스로 패턴을 학습해 이상을 찾고, 원인을 좁혀주고, 가능하면 복구까지 해주는 접근이 등장했습니다. 이것이 바로 AIOps입니다.

Part 1 — 02

AIOps란 무엇인가

AIOps(Artificial Intelligence for IT Operations)는 2016년경 Gartner가 정의한 개념으로, 빅데이터와 머신러닝을 IT 운영에 적용하여 이벤트 분석·이상 탐지·원인 분석·자동 대응을 수행하는 것을 말합니다.

💡
한 문장 정의

AIOps = 운영 데이터(로그·메트릭·트레이스·이벤트)를 AI가 대신 보고, 판단하고, 조치를 돕는 것. "모니터링의 자동화"가 아니라 "운영 판단의 자동화"에 가깝습니다.

2.1이름이 비슷한 것들과 헷갈리지 마세요

초보자가 가장 많이 혼동하는 지점입니다. 아래 표로 확실히 구분해 둡시다.

용어핵심 질문한 줄 설명
AIOps"AI로 운영을 어떻게 잘할까?"AI를 도구로 사용해 IT 인프라/서비스 운영을 자동화·지능화
MLOps"ML 모델을 어떻게 잘 배포·운영할까?"머신러닝 모델의 개발→배포→모니터링 생명주기 관리
LLMOps"LLM 서비스를 어떻게 잘 운영할까?"MLOps의 하위 분야. 프롬프트, RAG, 토큰 비용, 평가 등 LLM 특화 운영
DevOps"개발과 운영을 어떻게 합칠까?"개발-운영 협업 문화와 CI/CD 자동화 방법론
Observability"시스템 내부 상태를 어떻게 볼까?"메트릭·로그·트레이스로 시스템을 관측 가능하게 만드는 것. AIOps의 입력 데이터 공급원
🧭
쉽게 외우는 법

MLOps는 "AI를 위한 운영", AIOps는 "운영을 위한 AI"입니다. 방향이 정반대입니다. Observability는 AIOps에게 밥(데이터)을 주는 식당이라고 생각하면 됩니다.

2.2AIOps가 하는 일 (Gartner의 3단계 모델)

Observe
관찰

메트릭·로그·트레이스·이벤트를 수집하고, 이상 탐지·상관 분석으로 "무슨 일이 일어나고 있는지"를 파악

Engage
참여

ITSM(티켓 시스템)과 연동해 인시던트를 생성·라우팅하고, 담당자에게 원인 후보와 컨텍스트를 제공

Act
행동

런북 자동 실행, 스케일 아웃, 재시작, 롤백 등 자동 복구(Auto-remediation)를 수행

중요한 점: AIOps는 도구 하나가 아니라 이 세 단계를 아우르는 접근 방식(플랫폼)입니다. 처음부터 Act까지 갈 필요는 없으며, 대부분의 조직은 Observe부터 시작합니다.

Part 1 — 03

전통 MSP 운영 vs AIOps

MSP(Managed Service Provider)는 고객의 클라우드 인프라를 대신 운영해주는 사업입니다. 전통적인 MSP 운영 방식과 AIOps 기반 운영이 어떻게 다른지 비교하는 것이 이 강의의 핵심입니다.

3.1전통적인 MSP 운영은 이렇게 돌아갑니다

  • 임계치(Threshold) 기반 감시 — "CPU 80% 넘으면 알람"처럼 사람이 정한 고정 규칙으로 감시합니다.
  • 24/7 관제 인력 — 관제 요원이 모니터링 화면을 눈으로 보다가 알람이 오면 매뉴얼(런북)에 따라 대응합니다.
  • 티켓 중심 프로세스 — 알람 → 티켓 발행 → L1 확인 → L2/L3 에스컬레이션 → 조치 → 종결. 단계마다 사람의 판단과 대기 시간이 들어갑니다.
  • 사후 대응(Reactive) — 장애가 터진 뒤에 대응이 시작됩니다. 고객이 먼저 장애를 발견해 전화하는 경우도 있습니다.

3.2한눈에 보는 비교표

관점전통 MSP 운영AIOps 기반 운영
감시 방식고정 임계치 + 사람이 만든 규칙ML이 학습한 동적 베이스라인으로 이상 탐지
알람 처리알람 수백 건을 사람이 하나씩 확인상관 분석으로 관련 알람을 1개 인시던트로 묶음 (노이즈 감소)
원인 분석엔지니어 경험에 의존, 여러 콘솔을 오가며 수동 추적토폴로지·시계열 기반 RCA로 원인 후보를 자동 제시
대응 시점사후 대응 (장애 발생 후)예측·조기 탐지로 사전 대응 가능 (Proactive)
복구런북을 사람이 읽고 수동 실행검증된 시나리오는 자동 복구 (재시작, 스케일 아웃 등)
확장성고객·시스템이 늘면 인력도 비례해 증가플랫폼이 흡수 — 인력 증가 없이 관리 범위 확대
산출물월간 운영 보고서 (수작업)실시간 대시보드 + 자동 생성 리포트
지식 관리베테랑의 머릿속 (퇴사하면 소실)과거 인시던트 데이터로 축적 — 모델·지식베이스로 자산화
🤝
오해 금지: AIOps는 MSP의 적이 아닙니다

AIOps는 MSP를 대체하는 것이 아니라 MSP의 운영 방식을 업그레이드하는 도구입니다. 실제로 글로벌 MSP들은 AIOps 플랫폼을 자사 관제에 도입해 "같은 인력으로 더 많은 고객"을 관리하는 방향으로 진화하고 있습니다. 반복 업무는 AI가, 판단과 고객 커뮤니케이션은 사람이 맡는 구조입니다.

3.3인터랙티브: 같은 장애, 다른 대응

동일한 장애 시나리오 — "결제 서비스 응답 지연 발생" — 를 두 방식이 어떻게 처리하는지 직접 비교해 보세요. ▶ 시뮬레이션 시작 버튼을 눌러보세요.

🎬 장애 대응 시뮬레이터

시나리오: 02:00 AM, DB 커넥션 풀 고갈로 결제 API 응답 지연 시작
🏢 전통 MSP 운영00:00
🤖 AIOps 기반 운영00:00

Part 2 — 04

데이터: AIOps의 연료

AI는 데이터가 없으면 아무것도 못 합니다. AIOps의 품질은 얼마나 좋은 관측 데이터(Observability Data)를 공급하느냐에서 결정됩니다. 4가지 핵심 데이터 유형을 알아봅시다. 흔히 MELT라고 부릅니다.

M — Metrics

메트릭

CPU 사용률, 응답 시간, 초당 요청 수처럼 숫자로 된 시계열. 가볍고 집계가 쉬워 이상 탐지의 주 재료입니다. 예: Prometheus.

E — Events

이벤트

배포 완료, Pod 재시작, 설정 변경 같은 "무슨 일이 있었다"는 기록. 장애 시점과 변경 이력을 연결하는 열쇠입니다.

L — Logs

로그

애플리케이션이 남기는 텍스트 기록. 가장 상세하지만 양이 방대합니다. 로그 패턴 분석·클러스터링이 AIOps의 주요 기법입니다.

T — Traces

트레이스

하나의 요청이 여러 마이크로서비스를 거쳐간 경로와 소요 시간. "어느 서비스에서 느려졌나"를 알려줍니다. 예: OpenTelemetry.

⚠️
초보자가 꼭 알아야 할 원칙: Garbage In, Garbage Out

수집이 누락되거나 시간이 안 맞는(타임스탬프 불일치) 데이터로는 어떤 훌륭한 AI도 제대로 판단할 수 없습니다. AIOps 도입의 첫걸음이 항상 Observability 체계 정비(OpenTelemetry 표준화 등)인 이유입니다.

4.1한 가지 더: 토폴로지 (Topology)

MELT에 더해, 좋은 AIOps 플랫폼은 토폴로지 — "무엇이 무엇에 연결되어 있는가"라는 구성 관계 지도를 유지합니다. 결제 API → 주문 DB → 스토리지 같은 의존 관계를 알아야, DB 장애 시 그 위에서 발생한 API 알람들이 같은 원인의 결과임을 추론할 수 있기 때문입니다.

Part 2 — 05

AIOps의 5대 핵심 기능

AIOps 플랫폼이 공통적으로 제공하는 5가지 기능입니다. 면접에서도, 실무 도입 검토에서도 이 5개가 기준이 됩니다.

5.1이상 탐지 (Anomaly Detection)

"CPU 80% 초과 시 알람" 같은 고정 임계치의 문제는 정상의 기준이 시간마다 다르다는 것입니다. 쇼핑몰의 낮 트래픽과 새벽 트래픽은 완전히 다르니까요. 이상 탐지는 ML이 시간대·요일별 정상 패턴(베이스라인)을 학습하고, 거기서 벗어난 순간을 잡아냅니다.

  • 정적 임계치: 새벽에 CPU 20%→60% 급증해도 80% 미만이라 침묵 → 조기 징후를 놓침
  • 동적 베이스라인: "이 시간대 정상은 15~25%인데 60%다" → 즉시 탐지

5.2이벤트 상관 분석 & 노이즈 감소 (Event Correlation)

DB 하나가 죽으면 그 DB를 쓰는 서비스 30개가 동시에 알람을 쏟아냅니다. 상관 분석은 시간·토폴로지·텍스트 유사도를 근거로 관련 알람들을 하나의 인시던트로 묶어 운영자가 봐야 할 대상을 극적으로 줄여줍니다. 아래 슬라이더로 직접 체험해 보세요.

5.3근본 원인 분석 (RCA, Root Cause Analysis)

"어디가 아픈지"를 넘어 "왜 아픈지"를 찾는 기능입니다. 토폴로지 상 의존 관계, 이벤트 발생 순서, 최근 변경(배포) 이력을 종합해 원인 후보를 확률 순으로 제시합니다. 예: "23:47 결제서비스 v2.3.1 배포 직후 DB 커넥션 오류 급증 — 배포가 원인일 확률 높음".

5.4예측 분석 (Predictive Analytics)

과거 추세를 학습해 미래를 내다봅니다. "디스크가 이 속도로 차면 12일 후 가득 참", "다음 주 프로모션 트래픽에는 노드 3대 증설 필요" 같은 사전(Proactive) 대응을 가능하게 합니다. 용량 계획(Capacity Planning)과 비용 최적화에 직결됩니다.

5.5자동 복구 (Auto-Remediation)

탐지·분석 결과에 따라 정해진 조치를 자동 실행합니다. 디스크 정리 스크립트 실행, Pod 재시작, 오토스케일 트리거, 카나리 배포 자동 롤백 등이 대표적입니다. 처음부터 전면 자동화하지 않고, 사람 승인 후 실행(Human-in-the-loop) → 검증된 시나리오만 완전 자동화 순서로 신뢰를 쌓아가는 것이 정석입니다.

🧠
최근 트렌드: LLM 기반 AIOps

2023년 이후 LLM이 결합되며 새 국면이 열렸습니다. 장애 요약을 자연어로 자동 작성하고, "지난주 결제 장애 원인이 뭐였지?"라고 채팅으로 질의하고, 런북을 LLM 에이전트가 수행하는 방식입니다. LLMOps 역량이 AIOps와 만나는 지점으로, 여러분과 같은 클라우드 엔지니어에게 가장 유망한 교차 영역입니다.

Part 2 — 06

AIOps 아키텍처 & 데이터 파이프라인

AIOps 플랫폼의 내부는 결국 데이터 파이프라인입니다. 5개 계층으로 이해하면 어떤 상용/오픈소스 제품을 봐도 구조가 보입니다.

Layer 1
수집 (Collect)

에이전트·익스포터가 MELT 데이터 수집. OpenTelemetry, Fluent Bit, Prometheus Exporter

Layer 2
저장·처리 (Store)

시계열 DB, 로그 스토어, 스트림 처리. Prometheus/Mimir, OpenSearch, Kafka

Layer 3
분석 (Analyze)

이상 탐지, 상관 분석, RCA 모델이 동작하는 ML 엔진 계층

Layer 4
참여 (Engage)

인시던트 생성, 알림 라우팅, ITSM 연동. PagerDuty, Slack, Jira/ServiceNow

Layer 5
실행 (Act)

런북 자동화, 오케스트레이션. Ansible, Argo Workflows, StackStorm

6.1K8s 환경 예시로 보는 실제 흐름

여러분이 운영하는 쿠버네티스 클러스터에 적용하면 이런 그림이 됩니다.

# 하나의 이상 징후가 흘러가는 경로 (예시)
[Pod: payment-api]
   │  OpenTelemetry SDK/Agent가 메트릭·트레이스 수집
   ▼
[Collector: OTel Collector / Fluent Bit]
   │  Kafka 토픽으로 스트리밍
   ▼
[저장: Prometheus(메트릭) + OpenSearch(로그) + Tempo(트레이스)]
   │
   ▼
[ML 엔진] p95 latency가 학습된 베이스라인 이탈 → 이상 탐지
   │  동일 시간대 DB 알람과 토폴로지 연결 → 상관 분석 → 인시던트 1건 생성
   │  10분 전 배포 이벤트 발견 → RCA: "v2.3.1 배포 의심"
   ▼
[Engage] Slack 채널에 요약 + 원인 후보 + 대시보드 링크 발송
   ▼
[Act] 승인 클릭 → Argo Workflow가 자동 롤백 실행 ✅
🎯
기억할 것

이 5계층 중 Layer 1~2는 이미 여러분이 아는 Observability 스택입니다. AIOps는 그 위에 Layer 3(ML 분석)을 얹는 것에서 시작됩니다. 즉, Prometheus·Grafana·OpenSearch를 다룰 줄 안다면 AIOps의 절반은 이미 알고 있는 셈입니다.

Part 3 — 07

도구 생태계

AIOps 도구는 크게 오픈소스 조합형, 상용 통합 플랫폼, 클라우드 네이티브 서비스 세 갈래입니다. 탭을 눌러 비교해 보세요.

수집·저장

Prometheus + OpenTelemetry + OpenSearch

사실상 표준 Observability 스택. 메트릭·트레이스·로그의 수집과 저장을 담당합니다.

시각화·알림

Grafana + Alertmanager

대시보드와 알림 라우팅. Grafana ML 플러그인으로 기초적인 이상 탐지도 가능합니다.

스트림 처리

Apache Kafka + Flink

대량 이벤트를 실시간 파이프라인으로 처리. 대규모 환경의 중추입니다.

ML 분석

Prophet / PyOD / Merlion

시계열 예측(Prophet), 이상 탐지 라이브러리(PyOD), Salesforce의 시계열 ML 툴킷(Merlion)으로 직접 분석 계층을 구축합니다.

자동화

탐지 결과를 받아 런북을 자동 실행하는 Act 계층. StackStorm은 이벤트 기반 자동화(IFTTT for Ops)에 특화되어 있습니다.

단점: 직접 통합·운영해야 하며 ML 계층은 자체 구축 필요 — 엔지니어링 역량이 요구됩니다.

결정론적 AI 엔진 Davis가 토폴로지 기반 자동 RCA를 제공. "원인 후보 제시"가 가장 성숙한 제품군으로 꼽힙니다.

APM 계열

APM 계열

New Relic AI와 Elastic의 ML 기능. 기존 스택에 따라 자연스러운 선택지가 됩니다.

이벤트 상관 특화

BigPanda / Moogsoft

여러 모니터링 도구의 알람을 한데 모아 상관 분석·노이즈 감소에 특화. 대형 관제센터(NOC)와 MSP에서 인기가 많습니다.

티켓·CMDB와 강하게 결합된 AIOps. 전통 MSP 프로세스를 유지하면서 지능화할 때 유리합니다.

특징

장점 vs 단점

장점: 빠른 도입, 검증된 ML, 벤더 지원.
단점: 라이선스 비용이 상당하며(데이터량 과금 주의), 벤더 종속 리스크가 있습니다.

AWS

DevOps Guru / CloudWatch Anomaly Detection

ML 기반 이상 탐지와 운영 인사이트를 관리형으로 제공. AWS 리소스에 한정되지만 설정이 매우 간단합니다.

Azure

Azure Monitor + Smart Detection

Application Insights의 Smart Detection이 자동 이상 탐지를 수행. Azure MSP 운영 시 기본으로 활용할 수 있습니다.

GCP

Cloud Operations Suite

구 Stackdriver. 로그 기반 메트릭과 이상 탐지 기능을 제공합니다.

Part 3 — 08

도입 로드맵 & 첫 실습

AIOps는 "제품 하나 사면 끝"이 아니라 단계적 성숙 여정입니다. 업계에서 통용되는 Crawl → Walk → Run 모델로 정리합니다.

Stage 1 · Crawl

🐢 데이터 기반 다지기

목표: 관측 데이터 통합.
OpenTelemetry 표준화, 메트릭·로그 중앙화, 태깅/라벨 규칙 정립, 대시보드 정비. 이 단계 없이는 그 어떤 AI도 무용지물입니다. (기간 감각: 수개월)

Stage 2 · Walk

🚶 탐지·분석 지능화

목표: 사람이 보는 양 줄이기.
동적 베이스라인 이상 탐지 적용, 알람 상관 분석으로 노이즈 감소, RCA 후보 자동 제시. 여기까지만 해도 관제 부담이 크게 줄어듭니다.

Stage 3 · Run

🏃 자동 대응

목표: 사람 개입 최소화.
승인 기반 반자동 복구 → 검증된 시나리오 완전 자동화 → 예측 기반 사전 조치. 신뢰가 쌓인 만큼만 자동화 범위를 넓힙니다.

🚧
흔한 실패 패턴

① 데이터 정비(Crawl) 없이 비싼 플랫폼부터 구매 ② 첫날부터 전면 자동 복구를 시도하다 자동화가 장애를 키움 ③ 운영팀 프로세스 변화 없이 도구만 도입. AIOps는 기술 프로젝트이기 이전에 운영 프로세스 개선 프로젝트입니다.

8.1실습: 30줄로 만드는 미니 이상 탐지기

거창한 플랫폼 없이도 원리는 직접 만져볼 수 있습니다. 탭에서 단순 통계 방식(Z-Score)과 ML 방식(Prophet) 두 가지 접근을 비교해 보세요.

# pip install numpy — 이동 평균 대비 표준편차로 이상을 판정하는 가장 단순한 방식
import numpy as np

def detect_anomalies(values, window=30, threshold=3.0):
    """직전 window개 구간의 평균·표준편차 대비 z-score가 threshold를 넘으면 이상"""
    anomalies = []
    for i in range(window, len(values)):
        baseline = values[i-window:i]          # 최근 30개 관측치 = 동적 베이스라인
        mu, sigma = np.mean(baseline), np.std(baseline)
        if sigma == 0:
            continue
        z = abs(values[i] - mu) / sigma       # 정상 범위에서 몇 σ 벗어났나
        if z > threshold:
            anomalies.append((i, values[i], round(z, 1)))
    return anomalies

# CPU 사용률 예시: 평소 20% 안팎 → 마지막에 65% 스파이크
cpu = [20+np.random.randn() for _ in range(100)] + [65]
print(detect_anomalies(cpu))   # → [(100, 65.0, 44.3)]  z=44σ, 명백한 이상!

핵심: 고정 임계치("80% 넘으면") 대신 "평소 대비 얼마나 튀었나"를 봅니다. 단, 주기성(낮/밤 패턴)은 반영하지 못하는 한계가 있습니다. 그 한계를 푸는 것이 ②번입니다.

# pip install prophet pandas — Meta의 시계열 예측 라이브러리로 주기성까지 학습
import pandas as pd
from prophet import Prophet

# 1) 과거 2주치 메트릭 로드 (ds=시각, y=값 컬럼명은 Prophet 규약)
df = pd.read_csv("cpu_metrics.csv")   # columns: ds, y

# 2) 일간/주간 주기성을 학습 → "화요일 새벽 3시의 정상"을 앎
model = Prophet(interval_width=0.99, daily_seasonality=True)
model.fit(df)

# 3) 미래 24시간의 정상 범위(신뢰구간) 예측
future = model.make_future_dataframe(periods=24, freq="H")
fcst = model.predict(future)

# 4) 실측값이 예측 신뢰구간(yhat_lower~yhat_upper)을 벗어나면 이상
merged = df.merge(fcst[["ds","yhat_lower","yhat_upper"]], on="ds")
anomalies = merged[(merged.y < merged.yhat_lower) |
                   (merged.y > merged.yhat_upper)]
print(anomalies[["ds", "y"]])

핵심: Prophet은 "이 시간대의 정상 범위"를 스스로 계산합니다. 새벽의 작은 급증도 잡고, 낮 피크 트래픽은 정상으로 인식합니다. 이것이 상용 AIOps의 동적 베이스라인과 같은 원리입니다.

# ML 없이 PromQL만으로 "평소 대비 이탈"을 근사하는 실전 트릭
# 지난주 같은 시각 대비 현재 트래픽이 3배 이상이면 알람

( sum(rate(http_requests_total[5m]))
  /
  sum(rate(http_requests_total[5m] offset 1w)) ) > 3

# z-score를 recording rule로 직접 계산하는 방법
# (avg_over_time과 stddev_over_time 활용)
(
  node_cpu_utilization
  - avg_over_time(node_cpu_utilization[1h])
) / stddev_over_time(node_cpu_utilization[1h]) > 3

핵심: 이미 운영 중인 Prometheus에서 오늘 바로 시도할 수 있는 방법입니다. offset으로 과거와 비교하는 것만으로도 고정 임계치보다 훨씬 똑똑한 알람이 됩니다.

Part 3 — 09

응용 범위: AIOps는 어디까지 쓰이나

AIOps는 장애 대응만을 위한 것이 아닙니다. 운영 데이터가 흐르는 모든 곳이 응용 범위입니다.

🚨

인시던트 관리 (가장 대표적)

이상 탐지 → 노이즈 감소 → RCA → 자동 복구로 이어지는 장애 대응 전체 사이클의 지능화. MTTD(탐지 시간)와 MTTR(복구 시간) 단축이 핵심 지표입니다.

💰

비용 최적화 (FinOps 연계)

사용 패턴을 학습해 과다 프로비저닝된 리소스를 찾고, 적정 인스턴스 크기(Right-sizing)를 추천하고, 예약 인스턴스 구매 시점을 제안합니다. MSP의 고객 비용 절감 리포트에 직결됩니다.

📈

용량 계획 (Capacity Planning)

트래픽·스토리지 증가 추세를 예측해 "언제 얼마나 증설해야 하는가"에 데이터 기반 답을 제공합니다. 예측적 오토스케일링도 이 범주입니다.

🛡️

보안 운영 (AISecOps)

비정상 로그인 패턴, 이상 트래픽, 권한 오남용 탐지 등 보안 이벤트에 같은 기법을 적용합니다. SIEM + UEBA(사용자 행위 분석)가 대표 사례입니다.

⚡

성능 최적화

트레이스 데이터를 분석해 병목 구간을 자동 식별하고, 느린 쿼리·비효율 API 호출 패턴을 찾아냅니다.

🤖

LLM 결합 운영 (차세대)

장애 보고서 자동 작성, 자연어 운영 질의("어제 새벽 알람 요약해줘"), RAG 기반 런북 검색, LLM 에이전트의 자율 대응까지 — AIOps와 LLMOps가 융합되는 최전선입니다.

9.1MSP 비즈니스 관점에서의 의미

MSP 소속 엔지니어라면 AIOps를 비즈니스 언어로도 설명할 수 있어야 합니다.

  • 수익성: 관제 인력이 고객 수에 비례해 늘지 않으므로, 고객이 늘수록 마진이 개선되는 구조를 만들 수 있습니다.
  • 차별화: "24/7 사람이 지켜봅니다"는 모든 MSP가 말하지만, "AI가 90% 노이즈를 걸러내고 예측적으로 대응합니다"는 차별화된 제안이 됩니다.
  • SLA 경쟁력: MTTR 단축은 곧 더 공격적인 SLA 제시 능력입니다.
  • 지식 자산화: 인시던트 이력이 사람의 기억이 아닌 데이터·모델로 축적되어 회사의 자산이 됩니다.

11.1오늘 배운 것 한 장 요약

  • 왜: 클라우드·MSA의 복잡도와 알람 폭증이 사람의 눈을 넘어섰기 때문에 AIOps가 등장했습니다.
  • 무엇: AIOps는 운영 데이터에 ML을 적용해 탐지 → 분석 → 대응을 지능화하는 접근입니다. (MLOps는 "AI를 위한 운영", AIOps는 "운영을 위한 AI")
  • vs MSP: 고정 임계치·티켓·사후 대응 중심의 전통 운영을, 동적 베이스라인·노이즈 감소·RCA·자동 복구·사전 대응으로 업그레이드합니다. 대체가 아니라 증강입니다.
  • 어떻게: MELT 데이터 → 5계층 파이프라인(수집-저장-분석-참여-실행), 도입은 Crawl → Walk → Run 순서로.
  • 어디에: 인시던트 관리, FinOps, 용량 계획, 보안(AISecOps), 성능 최적화, 그리고 LLM 결합 운영까지.

11.2추천 다음 학습 경로

Step 1

Observability 기초 다지기

Prometheus + Grafana + OpenTelemetry. AIOps의 Layer 1~2를 손으로 구축해 보세요. 홈랩 K8s 클러스터가 최고의 실습장입니다.

Step 2

이상 탐지 직접 구현

본 강의의 Prophet 예제를 실제 클러스터 메트릭에 적용해 보세요. Kafka로 스트림 파이프라인까지 얹으면 미니 AIOps 완성입니다.

Step 3

LLM 결합 실험

알람 요약 봇, RAG 기반 런북 검색 등 LLMOps 역량과 결합해 보세요. AIOps + LLMOps 교차 역량은 현재 가장 희소한 스킬셋입니다.
