CHAPTER 01

DB는 어떻게 동작하는가

튜닝을 잘하려면 "DB가 데이터를 어떻게 저장하고 읽는지"부터 알아야 합니다. 복잡한 내부 구조 대신, 튜닝에 꼭 필요한 5가지 핵심 개념만 정리합니다.

디스크 vs 메모리 — 속도 차이 100,000배

DB 성능을 이해하는 가장 중요한 사실은 이것입니다: 디스크는 메모리보다 압도적으로 느립니다. 숫자로 보면 더 충격적입니다.

저장 장치1회 접근 시간비유
CPU 레지스터~ 0.3 나노초지금 들고 있는 책
L1/L2 캐시~ 1~10 나노초책상 위
RAM (메모리)~ 100 나노초책장
SSD~ 100,000 나노초 (0.1ms)도서관 다른 층
HDD~ 10,000,000 나노초 (10ms)다른 도시의 도서관
핵심 통찰

DB 튜닝의 본질은 "디스크 접근을 최소화하고, 메모리(=버퍼)에 데이터를 오래 머물게 하는 것"입니다. 앞으로 나올 모든 기법(인덱스, 캐시, 파티셔닝...)은 사실상 이 한 문장에서 파생됩니다.

페이지(Page) — DB가 데이터를 다루는 기본 단위

DB는 한 줄(row)씩 읽지 않습니다. 대신 페이지(또는 블록)라는 일정 크기의 묶음 단위로 읽고 씁니다. MySQL InnoDB는 16KB, PostgreSQL은 8KB가 기본값입니다.

이게 왜 중요할까요? 한 번의 디스크 I/O는 1행이 아니라 페이지 전체를 가져오기 때문입니다. 1,000건을 찾으려고 1,000번 I/O를 한다면, 각 행이 다른 페이지에 흩어져 있을 때 1,000 페이지(16MB)를 읽어야 합니다. 같은 페이지에 모여 있다면? 단 몇 페이지면 끝납니다. 이것이 클러스터링 인덱스가 강력한 이유입니다.

버퍼 풀(Buffer Pool) — 메모리 위의 DB 사본

자주 쓰는 페이지를 메모리에 캐싱해 두는 영역을 버퍼 풀이라고 부릅니다. PostgreSQL에서는 shared_buffers, MySQL에서는 innodb_buffer_pool_size가 이 영역의 크기를 결정합니다.

실무에서 가장 먼저 확인해야 할 지표인 버퍼 히트율(Buffer Hit Ratio)은 "요청한 페이지가 메모리에 이미 있었을 확률"입니다. 99% 이상이 건강한 상태입니다. 이 값이 떨어지면, 메모리가 부족하거나 쿼리가 너무 많은 데이터를 훑고 있다는 신호입니다.

트랜잭션 로그(WAL/REDO) — 안전의 대가

DB는 변경 사항을 디스크에 영구 기록하기 전에, 먼저 로그 파일에 순차적으로 기록합니다. PostgreSQL은 WAL(Write-Ahead Log), MySQL은 Redo Log, Oracle도 Redo Log라고 부릅니다. 이 덕분에 서버가 갑자기 죽어도 데이터가 복구됩니다. 하지만 이는 모든 쓰기 작업에 추가 I/O가 발생한다는 뜻이기도 합니다.

옵티마이저(Optimizer) — 보이지 않는 결정자

같은 결과를 내는 SQL이라도 실행 방법은 수십 가지가 있습니다. 인덱스를 쓸지, 어떤 JOIN 알고리즘을 쓸지, 어느 테이블부터 읽을지 — 이 모든 걸 결정하는 게 옵티마이저입니다. 옵티마이저는 통계(statistics)를 기반으로 가장 비용이 적을 것 같은 계획을 선택합니다. 튜닝의 절반은 옵티마이저가 좋은 선택을 하도록 정보를 주고, 잘못된 선택을 했을 때 힌트로 바로잡는 일입니다.

이 챕터의 핵심 정리

① 디스크는 메모리보다 압도적으로 느리다. ② DB는 페이지 단위로 읽고 쓴다. ③ 자주 쓰는 페이지는 버퍼 풀에 캐싱된다. ④ 쓰기 작업은 항상 로그를 동반한다. ⑤ 실행 방법은 옵티마이저가 결정한다.

CHAPTER 02

성능 측정의 언어

"느리다"는 주관입니다. 튜닝은 측정 가능한 지표로 시작해서 측정 가능한 지표로 끝납니다. DBA·SRE가 매일 이야기하는 핵심 지표 7가지를 정리합니다.

지표의미나쁠 때 시사점
TPS / QPS 초당 트랜잭션/쿼리 처리량 처리 한계 도달. 락 경합, CPU 포화 의심
Latency (p50/p95/p99) 응답 시간의 백분위수 p99가 튀면 일부 쿼리가 풀스캔 중일 가능성
Buffer Hit Ratio 요청 페이지가 메모리에 있었던 비율 99% 미만이면 메모리 부족 또는 쿼리 비효율
IOPS 초당 I/O 횟수 스토리지 한계 임박, 인덱스 부재 가능성
Lock Wait 락 대기 시간 / 횟수 긴 트랜잭션, 부적절한 격리 수준
Connection Count 활성 연결 수 커넥션 풀 부족, 코드의 누수
Replication Lag 복제 지연 시간 읽기 슬레이브가 최신 데이터 못 줄 위험

p50보다 p99를 보라

평균 응답 시간(p50, 즉 중앙값)만 보면 항상 빨라 보입니다. 하지만 실제 사용자 경험을 망치는 건 p99 — 상위 1%의 가장 느린 쿼리입니다. 어떤 쿼리는 0.5ms로 끝나고 어떤 쿼리는 5초가 걸린다면 평균은 행복해도, 그 5초를 만난 사용자는 떠납니다.

실무 팁

모니터링 대시보드에는 반드시 p50, p95, p99를 함께 표시하세요. 평균만 보면 문제를 놓칩니다. "꼬리 지연(tail latency)"이라는 표현은 바로 이 p99의 분포를 가리킵니다.

CHAPTER 03

인덱스의 모든 것

DB 튜닝의 80%는 인덱스 이야기입니다. 인덱스를 제대로 이해하면 대부분의 성능 문제는 해결됩니다. 이 챕터는 가장 길고, 가장 중요합니다.

인덱스란 무엇인가 — 책의 색인 비유

500페이지짜리 요리책에서 "마늘"이라는 단어가 나오는 페이지를 찾는다고 합시다. 두 가지 방법이 있습니다.

  1. 처음부터 끝까지 읽기 (Full Scan) — 확실하지만 끔찍하게 느립니다.
  2. 맨 뒤의 색인(Index) 보기 — "마늘 ... 23, 47, 198" 한 번에 위치를 찾습니다.

DB의 인덱스도 똑같습니다. 특정 컬럼의 값들을 미리 정렬해 둔 별도의 자료구조로, 실제 데이터가 있는 위치(rowid 또는 PK)를 가리킵니다. 인덱스가 있으면 "어디에 있는지" 거의 즉시 찾을 수 있습니다.

B-Tree — 99%의 인덱스가 쓰는 자료구조

상용 DB의 기본 인덱스는 거의 모두 B-Tree(정확히는 B+Tree)입니다. 트리 구조이기 때문에 데이터가 백만 건이든 십억 건이든 검색에 필요한 단계 수가 로그 스케일로만 증가합니다. 천만 건 테이블에서 인덱스로 한 행을 찾는 데 보통 3~4단계면 충분합니다.

인덱스의 종류

Primary Key Index · Unique Index

PK 인덱스는 테이블 생성 시 자동으로 만들어지며, 중복도 허용하지 않습니다. MySQL InnoDB에서는 클러스터 인덱스로 동작합니다 — 즉, 실제 데이터 자체가 PK 순서로 정렬되어 저장됩니다. 이는 PK로 검색할 때 가장 빠르고, PK로 정렬할 때 추가 정렬 비용이 없다는 뜻입니다.

Unique Index는 중복을 막으면서 검색도 빠르게 해 줍니다. 이메일·전화번호처럼 유일해야 하는 컬럼에 적합합니다.

sql
-- PK는 테이블 생성 시 함께 지정
CREATE TABLE users (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  email VARCHAR(255) NOT NULL,
  created_at TIMESTAMP
);

-- 유일성을 보장하면서 검색 인덱스 효과
CREATE UNIQUE INDEX idx_users_email
  ON users(email);

Composite Index (복합 인덱스)

여러 컬럼을 묶어 하나의 인덱스로 만든 것입니다. 컬럼 순서가 매우 중요합니다. 인덱스 (a, b, c)는 "a 정렬 → 같은 a 안에서 b 정렬 → 같은 b 안에서 c 정렬" 입니다. 이름 순으로 정렬된 전화번호부를 떠올리세요. "성"부터 모르면 "이름"으로 찾을 수 없습니다.

다음 쿼리에서 인덱스 (user_id, status, created_at)가 어떻게 쓰이는지 봅시다:

WHERE 조건인덱스 활용이유
user_id = 5✅ 완전 활용맨 앞 컬럼
user_id = 5 AND status = 'A'✅ 완전 활용접두사 일치
user_id = 5 AND created_at > ?⚠️ 부분 활용status 건너뜀, 비효율
status = 'A'❌ 활용 불가맨 앞 컬럼 누락
created_at > ?❌ 활용 불가맨 앞 컬럼 누락
컬럼 순서 정하는 원칙

① 등호(=) 조건을 먼저, 범위(>,<,BETWEEN) 조건은 뒤에. ② 선택도(Selectivity)가 높은(=고유한 값이 많은) 컬럼을 앞에. ③ ORDER BY/GROUP BY 컬럼을 뒤쪽에 배치하면 정렬도 같이 해결됩니다.

Covering Index (커버링 인덱스)

쿼리가 필요로 하는 모든 컬럼이 인덱스에 포함되어 있어, 실제 테이블 데이터를 읽지 않고 인덱스만 보고 답을 줄 수 있는 경우입니다. 가장 빠른 형태의 인덱스 활용입니다.

sql
-- 이 인덱스가 있다면...
CREATE INDEX idx_cover
  ON orders(user_id, status, amount);

-- 다음 쿼리는 테이블을 전혀 안 읽음 (Index Only Scan)
SELECT status, SUM(amount)
FROM orders
WHERE user_id = 100
GROUP BY status;

PostgreSQL은 INCLUDE 절로 인덱스 키가 아닌 컬럼도 인덱스에 끼워 넣을 수 있어, 불필요한 정렬 비용 없이 커버링 효과를 누립니다.

Partial Index (부분 인덱스)

WHERE 절을 가진 인덱스입니다. 전체 행 중 일부만 인덱싱하므로 인덱스 크기가 작아지고 갱신 비용도 줍니다. PostgreSQL이 강력하게 지원합니다.

sql / postgresql
-- '활성' 상태 주문만 인덱싱
-- 전체 주문 1억 건 중 활성은 10만 건이라면, 인덱스가 1000배 작아짐
CREATE INDEX idx_active_orders
  ON orders(user_id)
  WHERE status = 'ACTIVE';

Full-text Index (전문 검색 인덱스)

"내용에 '머신러닝'이 들어간 글" 같은 검색을 빠르게 하는 인덱스입니다. LIKE '%머신러닝%'은 인덱스를 못 쓰지만, Full-text 인덱스는 단어 단위로 미리 토큰화해 두기 때문에 즉시 답이 나옵니다.

한국어는 형태소 분석이 필요해서 영어보다 까다롭습니다. MySQL은 N-gram 파서를, PostgreSQL은 외부 확장(예: pg_bigm)을 자주 씁니다. 복잡한 검색이라면 차라리 OpenSearch/Elasticsearch 같은 검색 전용 엔진을 분리하는 것이 정공법입니다.

Hash / Bitmap / GIN / GiST

  • Hash Index: 등호(=) 검색만 가능, 범위 검색 불가. 해시 충돌이 없으면 O(1)에 가까움. PostgreSQL과 MySQL Memory 엔진에서 지원.
  • Bitmap Index: 카디널리티(고유값 수)가 낮은 컬럼(예: 성별, 상태)에 강력. 비트 연산으로 AND/OR가 빠름. Oracle, ClickHouse 등.
  • GIN (Generalized Inverted Index): PostgreSQL의 배열·JSONB·전문검색용 역색인. JSON 필드 검색의 핵심.
  • GiST (Generalized Search Tree): PostgreSQL의 지리정보(PostGIS)·범위 타입 검색에 사용되는 일반화 트리.
  • HNSW / IVF (벡터 인덱스): 임베딩 벡터의 유사도 검색(ANN). pgvector, Milvus, Pinecone 등.

인덱스가 동작하지 않는 7가지 경우

인덱스를 만들어 놓고도 풀스캔이 일어난다면 다음을 의심해 보세요. 실무에서 가장 흔한 함정들입니다.

인덱스를 못 쓰는 코드
WHERE YEAR(created_at) = 2024

인덱스 컬럼에 함수를 씌우면 인덱스가 무력화됩니다.

인덱스를 활용하는 코드
WHERE created_at >= '2024-01-01'
  AND created_at < '2025-01-01'

범위 조건으로 풀어 쓰면 인덱스가 정상 동작합니다.

LIKE 앞에 와일드카드
WHERE name LIKE '%kim%'

앞이 %로 시작하면 인덱스를 못 씁니다 (전문검색 필요).

접두 일치는 OK
WHERE name LIKE 'kim%'

뒤쪽 %는 인덱스 사용 가능합니다.

암묵적 형 변환
-- phone은 VARCHAR인데 숫자로 비교
WHERE phone = 01012345678

타입이 안 맞으면 DB가 모든 행에 형 변환을 적용해 풀스캔됩니다.

타입 일치
WHERE phone = '01012345678'

저장된 타입과 동일하게 비교해야 인덱스가 동작합니다.

나머지 흔한 함정들

  • OR 조건: 각 조건이 다른 인덱스를 가리키면 옵티마이저가 풀스캔을 택할 때가 많습니다. UNION으로 분리하면 해결됩니다.
  • NOT, !=: 부정 조건은 일반적으로 인덱스 효율이 떨어집니다. "있는 것을 찾는" 쪽으로 쿼리를 재작성하세요.
  • IS NULL: 일부 DB는 NULL을 인덱싱하지 않습니다 (Oracle B-Tree). 함수 기반 인덱스나 부분 인덱스로 우회합니다.
  • 옵티마이저 오판: 통계가 오래되었거나 분포가 왜곡되면 옵티마이저가 인덱스를 무시합니다. ANALYZE로 통계를 갱신하세요.

인덱스의 비용 — 공짜가 아니다

"그럼 인덱스를 무조건 많이 만들면 되는 거 아닌가요?" — 흔한 함정입니다. 인덱스는 다음과 같은 비용을 동반합니다.

  • 저장 공간: 큰 테이블이면 인덱스 하나가 수 GB를 차지할 수 있습니다.
  • 쓰기 성능 저하: INSERT/UPDATE/DELETE 때마다 모든 관련 인덱스도 같이 갱신됩니다. 인덱스 10개면 쓰기가 10배 무거워집니다.
  • 버퍼 풀 잠식: 인덱스 페이지도 메모리를 차지하므로, 정작 필요한 데이터 페이지가 캐시에서 밀려날 수 있습니다.
  • 옵티마이저 혼란: 비슷한 인덱스가 여럿이면 옵티마이저가 잘못된 것을 고르기도 합니다.
인덱스 설계의 황금률

① 읽기/쓰기 비율을 파악하라 — 쓰기가 많은 테이블엔 인덱스를 보수적으로. ② 실제 자주 쓰는 쿼리에 맞춰 만들어라 — "혹시 모르니까"는 금물. ③ 중복 인덱스를 정기적으로 청소하라 — (a)와 (a,b)가 있으면 전자는 보통 불필요.

CHAPTER 04

쿼리 튜닝

같은 결과를 내는 쿼리도 작성법에 따라 성능이 1000배 차이 납니다. 가장 자주 만나는 5가지 패턴을 비교 코드로 익혀 봅시다.

1. SELECT * 를 쓰지 말 것

필요한 컬럼만 명시하세요. 불필요한 컬럼까지 가져오면 네트워크 비용, 메모리 비용, 그리고 커버링 인덱스 활용 기회를 모두 잃습니다. "필요할지도 모르니까"라는 생각은 가장 빠르게 시스템을 느리게 만드는 마인드입니다.

2. N+1 문제 — ORM의 가장 흔한 함정

"사용자 100명을 가져오고, 각 사용자의 주문을 가져옵니다." 이게 어떻게 구현되어 있는지가 핵심입니다.

N+1 (쿼리 101번)
# Python ORM 의사 코드
users = User.query.all()  # 쿼리 1번
for u in users:
    print(u.orders)  # 쿼리 N번

사용자가 1만 명이면 쿼리가 1만 1번 날아갑니다.

Eager Loading / JOIN (쿼리 1번)
users = User.query \
    .options(joinedload(User.orders)) \
    .all()
# 단 1번의 JOIN 쿼리로 모든 데이터 획득

JOIN 또는 IN 절을 활용해 한 번에 해결합니다.

3. EXISTS vs IN — 의미는 같아도 성능은 다르다

"주문이 있는 사용자만"을 찾을 때, IN 서브쿼리보다 EXISTS가 빠른 경우가 많습니다. EXISTS는 행 하나만 발견하면 즉시 멈추기 때문입니다.

sql
-- 비효율적 (전체 서브쿼리를 다 만든 뒤 매칭)
SELECT * FROM users u
WHERE u.id IN (SELECT user_id FROM orders);

-- 효율적 (한 건만 발견하면 즉시 종료)
SELECT * FROM users u
WHERE EXISTS (
  SELECT 1 FROM orders o WHERE o.user_id = u.id
);

참고로, 최신 옵티마이저(PostgreSQL 12+, MySQL 8+)는 둘을 같은 계획으로 변환하기도 합니다. 실행 계획을 직접 확인해 보고 더 빠른 쪽을 쓰세요.

4. COUNT(*) — 큰 테이블에서는 비싸다

SELECT COUNT(*) FROM big_table은 PostgreSQL에서 전체 풀스캔이 됩니다 (MVCC 때문에 정확한 카운트가 비쌈). 웹페이지의 페이지네이션에서 "총 1,234,567건" 같은 정확한 숫자가 정말 필요한지 다시 생각해 보세요.

  • "많은 결과가 있습니다" 또는 "1만 건 이상" 같은 대략적 표시로 대체
  • 주기적으로 카운트를 계산해 별도 테이블에 저장 (요약 테이블)
  • Approximate count: PostgreSQL의 reltuples, ClickHouse의 uniq() 같은 근사 함수

5. 페이지네이션 — OFFSET은 깊을수록 느려진다

OFFSET 1000000 LIMIT 20은 100만 건을 모두 읽어서 버린 뒤 20건을 줍니다. 끔찍하죠. 해결책은 커서 기반 페이지네이션(Keyset Pagination)입니다.

OFFSET 방식 — 깊은 페이지에서 폭사
SELECT * FROM posts
ORDER BY created_at DESC
LIMIT 20 OFFSET 1000000;

OFFSET이 클수록 직선적으로 느려집니다.

Keyset — 항상 일정한 속도
-- 이전 페이지 마지막 행의 created_at, id를 알고 있을 때
SELECT * FROM posts
WHERE (created_at, id) < ('2024-01-15', 12345)
ORDER BY created_at DESC, id DESC
LIMIT 20;

인덱스를 타고 정확히 시작점부터 20건만 읽습니다.

6. JOIN 알고리즘 — 옵티마이저는 무엇을 고르나

알고리즘동작 방식유리한 상황
Nested Loop Join 바깥 테이블의 각 행에 대해 안쪽 테이블을 (인덱스로) 검색 한쪽이 작고, 다른 쪽엔 인덱스가 있음
Hash Join 작은 쪽으로 해시 테이블을 만들고, 큰 쪽을 스캔하며 매칭 양쪽 모두 크고, 등호 조건이며, 메모리가 충분
Merge Join 양쪽을 정렬해 둔 뒤 병합 양쪽 모두 조인 키로 이미 정렬되어 있음

MySQL InnoDB는 오랫동안 Nested Loop만 지원하다가 8.0부터 Hash Join을 도입했습니다. PostgreSQL/Oracle/SQL Server는 세 가지를 모두 자유롭게 씁니다. 큰 테이블끼리의 JOIN이라면 Hash Join이 가능한 DB에서 훨씬 빠릅니다.

CHAPTER 05

실행 계획 읽기

추측 대신 증거로 튜닝합시다. EXPLAIN은 옵티마이저가 어떤 길을 갈 것인지 알려주는 지도입니다. MySQL과 PostgreSQL의 실행 계획을 실제 예시로 해석해 봅니다.

EXPLAIN의 기본 사용법

sql
-- MySQL — 실행 계획만 보기
EXPLAIN SELECT * FROM orders WHERE user_id = 100;

-- MySQL — 실제 실행하면서 통계까지 (8.0+)
EXPLAIN ANALYZE SELECT * FROM orders WHERE user_id = 100;

-- PostgreSQL — 가장 자주 쓰는 형태 (실제 실행 + 버퍼 통계)
EXPLAIN (ANALYZE, BUFFERS, VERBOSE)
SELECT * FROM orders WHERE user_id = 100;

MySQL EXPLAIN의 주요 컬럼

컬럼의미주목할 값
type조인/접근 방식const, eq_ref, ref, range = 좋음 / ALL = 풀스캔, 위험
key실제 사용한 인덱스NULL이면 인덱스를 안 쓴 것
rows옵티마이저가 읽을 거라고 추정한 행 수실제와 크게 다르면 통계가 낡았다는 신호
Extra부가 정보Using filesort, Using temporary 보이면 튜닝 여지

PostgreSQL 실행 계획 해석 — 실제 예시

postgresql output
Seq Scan on orders  (cost=0.00..18334.00 rows=1 width=72)
                    (actual time=2.345..823.451 rows=1 loops=1)
  Filter: (user_id = 100)
  Rows Removed by Filter: 999999
  Buffers: shared hit=8000 read=10334
Planning Time: 0.234 ms
Execution Time: 823.512 ms

이 계획에서 읽어야 할 신호들:

  • Seq Scan: 풀스캔이 일어났습니다. 인덱스가 없거나, 옵티마이저가 풀스캔이 낫다고 판단했거나.
  • Rows Removed by Filter: 999999: 100만 건을 읽어서 99만9999건을 버렸습니다. 매우 나쁜 신호.
  • cost ↔ actual time: 추정 cost는 18334인데 실제 시간은 823ms. 비교 기준으로 활용.
  • Buffers: shared hit=8000 read=10334: 18334 페이지 중 1만 개를 디스크에서 읽음. 캐시 효과가 약함.
  • 해결책: CREATE INDEX ON orders(user_id); 한 줄이면 cost가 수십 단위로 떨어집니다.

좋은 계획 vs 나쁜 계획

나쁜 신호
  • Seq Scan / Type: ALL (풀스캔)
  • Using filesort (메모리에 정렬)
  • Using temporary (임시 테이블 생성)
  • estimated rows ≪ actual rows (통계 부정확)
  • Nested Loop × 큰 테이블
좋은 신호
  • Index Scan / Index Only Scan
  • Type: const, eq_ref, range
  • buffers hit ≫ read
  • Hash Join with small build side
  • estimated ≈ actual rows
실무 체크리스트

① 의심 쿼리는 무조건 EXPLAIN ANALYZE로 확인. ② actual time이 가장 큰 노드부터 공략. ③ rows 추정치와 실제값이 10배 이상 차이 나면 ANALYZE 테이블;로 통계 갱신. ④ 시각화 도구(https://explain.dalibo.com, pev2)를 활용하면 큰 계획도 한눈에 보입니다.

CHAPTER 06

스키마와 데이터 모델

쿼리 튜닝은 미세 수술이고, 스키마 설계는 외과 수술입니다. 잘못된 스키마는 어떤 인덱스로도 구할 수 없습니다.

정규화 vs 비정규화 — 균형의 미학

학교에서는 "3정규형까지 정규화하라"고 가르치지만, 실무에서는 의도적인 비정규화가 흔합니다. JOIN 비용을 피하기 위해 데이터를 일부러 중복 저장하는 것이죠.

관점정규화비정규화
저장 공간적음많음 (중복)
일관성 유지쉬움어려움 (모든 사본 갱신)
읽기 성능JOIN 비용한 번에 읽음 (빠름)
쓰기 성능한 곳만 수정여러 곳 수정 (느림)
적합한 워크로드OLTP (트랜잭션 위주)OLAP / 읽기 위주

데이터 타입 선택 — 작을수록 빠르다

같은 정보라면 더 작은 타입이 무조건 유리합니다. 페이지에 더 많은 행이 들어가고, 캐시 효율이 오르고, 인덱스도 작아집니다.

  • BIGINT가 정말 필요한가? 21억까지는 INT(4byte)면 충분합니다. BIGINT는 8byte입니다.
  • CHAR(255) 대신 VARCHAR를 — 단, MySQL VARCHAR도 길이가 길면 인덱스 비용이 큽니다.
  • UUID는 신중하게: 16byte라 BIGINT의 2배. InnoDB 클러스터 인덱스에선 랜덤 삽입으로 페이지 분할 폭증. UUID v7이나 ULID로 시간 정렬 보장하는 것을 추천.
  • TIMESTAMP vs DATETIME: MySQL TIMESTAMP는 4byte, DATETIME은 8byte. 단 2038년 문제가 있음.
  • ENUM vs TINYINT: 의미 있는 상수는 ENUM 또는 작은 정수로. 문자열은 비싸다.

파티셔닝 — 큰 테이블을 작은 테이블처럼

수십억 건짜리 테이블 하나는 다루기 어렵습니다. 파티셔닝은 큰 테이블을 논리적으로 하나처럼 보이게 두면서 실제로는 여러 작은 테이블(파티션)로 나누는 기법입니다.

Range Partitioning
시간 / 범위 기준
월별 / 연도별 로그 테이블에 가장 흔함
오래된 파티션 통째로 DROP하면 즉시 삭제
쿼리에 WHERE date BETWEEN... 이면 해당 파티션만 스캔
List Partitioning
목록 기준
지역, 카테고리 등 분류가 명확할 때
한국/일본/미국별로 나눠 저장 가능
조회 시 특정 지역만 검색하면 매우 빠름
Hash Partitioning
해시 기준
데이터 분포를 균등하게 분산
user_id 같은 균등 분포 키에 적합
단일 파티션 쿼리 시 풀스캔 위험

샤딩(Sharding) — 파티셔닝의 분산 버전

파티셔닝이 한 DB 안에서 테이블을 쪼개는 것이라면, 샤딩은 여러 DB 서버에 데이터를 나눠 저장하는 것입니다. 단일 서버 한계를 넘는 데이터·트래픽 규모에서 마지막 카드처럼 꺼냅니다. 샤딩은 운영 복잡도가 폭증하기 때문에 가능하면 미루는 것이 좋습니다 — 그 전에 인덱스·캐시·읽기 슬레이브로 충분히 갈 수 있는 경우가 많습니다.

CHAPTER 07

파라미터 튜닝

DB는 수백 개의 설정 파라미터를 가지고 있지만, 90%의 성능은 5~10개의 핵심 파라미터에서 결정됩니다. 절대 만지면 안 되는 것과, 꼭 점검해야 할 것을 정리합니다.

MySQL (InnoDB) 핵심 파라미터

파라미터의미권장
innodb_buffer_pool_size 가장 중요한 캐시. 데이터·인덱스 페이지가 머무는 메모리 전용 서버라면 RAM의 50~75%
innodb_log_file_size Redo Log 파일 크기. 쓰기 성능과 복구 시간의 트레이드오프 1~2 GB (쓰기 많은 시스템은 더 크게)
innodb_flush_log_at_trx_commit 커밋 시 로그를 디스크로 강제 동기화할지 금융 = 1 (안전) / 일반 = 2 (성능)
max_connections 동시 접속 가능 클라이언트 수 너무 크면 메모리 폭주. 보통 200~500
innodb_io_capacity InnoDB가 사용할 수 있는 IOPS 한계 힌트 SSD = 2000~20000

PostgreSQL 핵심 파라미터

파라미터의미권장
shared_buffers 공유 메모리 캐시. MySQL의 buffer pool과 유사 RAM의 25% 정도 (PG는 OS 캐시도 활용)
work_mem 정렬·해시 같은 작업당 메모리 한도 4MB~64MB (커넥션 × 쿼리 × 노드 만큼 곱해짐, 주의!)
maintenance_work_mem VACUUM, CREATE INDEX 같은 유지 작업용 256MB~2GB
effective_cache_size 옵티마이저에게 "OS 캐시까지 합치면 이만큼 있다"고 알려주는 힌트 RAM의 50~75%
checkpoint_timeout / max_wal_size 체크포인트 빈도 조절. 너무 잦으면 쓰기 폭주 15min / 4GB부터 시작

커넥션 풀 — 잊혀진 영웅

"max_connections를 늘리면 더 많이 처리할 수 있겠지!" — 흔한 오해입니다. DB 커넥션은 메모리(보통 5~10MB)와 CPU 컨텍스트 스위치를 소모합니다. 실제로는 적정 커넥션 수 × 빠른 처리가 훨씬 빠릅니다.

애플리케이션과 DB 사이에 커넥션 풀(예: HikariCP, pgbouncer)을 두세요. PostgreSQL의 권장 동시 활성 커넥션은 (CPU 코어 × 2 ~ 4) 정도, 많아도 100을 잘 안 넘깁니다. 수천 개의 클라이언트가 있어도 pgbouncer 같은 풀러가 받아서 적은 수의 DB 커넥션으로 다중화합니다.

조심해야 할 점

파라미터는 한 번에 하나씩, 측정과 함께 바꾸세요. 여러 개를 동시에 바꾸면 어느 게 효과를 냈는지 알 수 없습니다. 그리고 운영 DB의 파라미터를 변경하기 전에는 항상 워크로드 시뮬레이션과 백업이 선행되어야 합니다.

CHAPTER 08

캐싱 전략

가장 빠른 쿼리는 실행하지 않는 쿼리입니다. 캐시는 DB 앞에 놓는 강력한 방패이지만, "캐시 무효화는 컴퓨터 과학의 두 가지 어려운 문제 중 하나"라는 농담의 무게도 함께 옵니다.

3단계 캐시 계층

  1. DB 내부 캐시: 버퍼 풀, 쿼리 결과 캐시 (단, MySQL은 8.0에서 query cache 제거).
  2. 애플리케이션 캐시: 로컬 메모리(Guava Cache, Caffeine 등) 또는 분산 캐시(Redis, Memcached).
  3. CDN / Edge 캐시: HTTP 레벨에서 정적 응답 캐싱.

캐시 패턴 4가지

패턴동작적합한 상황
Cache-Aside (Lazy Loading) 읽기: 캐시 확인 → 없으면 DB 읽고 캐시에 저장. 쓰기: DB만 수정 + 캐시 무효화 가장 흔한 기본 패턴. 읽기 위주 워크로드
Write-Through 쓰기: 캐시와 DB를 동시에 갱신 데이터 일관성이 중요할 때
Write-Behind (Write-Back) 쓰기: 캐시만 갱신, 비동기로 DB 반영 쓰기 처리량이 매우 높을 때 (유실 위험 있음)
Refresh-Ahead 만료 직전에 미리 갱신 예측 가능한 인기 데이터

캐시 안티패턴

  • Stampede(돌진) 문제: 인기 키가 만료되는 순간 수많은 요청이 동시에 DB로 쏟아짐. 해결: 짧은 lock 또는 stale-while-revalidate 패턴.
  • Hot Key 문제: 특정 키 하나에 트래픽이 집중. 해결: 키를 N개로 샤딩하거나(예: top_post:0~top_post:9), 로컬 캐시 병행.
  • 캐시 일관성 실패: DB 업데이트 직후 캐시 무효화에 실패하면 stale data가 영구화. 해결: TTL을 반드시 설정, 트랜잭션 후 무효화, CDC(Debezium) 활용.
  • Big Key: 캐시에 수 MB짜리 값을 넣으면 네트워크와 Redis 단일 스레드가 막힘. 분할 필수.
실무 팁

모든 캐시 키에는 반드시 TTL을 두세요. "절대 만료 안 되게" 두는 순간, 잘못된 데이터가 영구히 남을 수 있습니다. 그리고 캐시는 "성능 최적화"이지 "데이터 저장소"가 아닙니다 — 캐시가 비어도 시스템은 정상 동작해야 합니다.

CHAPTER 10

안티패턴 모음집

현장에서 가장 많이 발견되는 "DB를 느리게 만드는" 코드 패턴 10가지. 각 항목을 펼쳐 자세한 설명과 해결책을 확인하세요.

① 모든 컬럼에 인덱스 걸기 — "혹시 모르니까 다 걸어두자" +

인덱스가 많을수록 INSERT/UPDATE/DELETE가 느려집니다. 인덱스 5개면 쓰기가 5배 무거워지고, 메모리도 그만큼 차지합니다.

해결: pg_stat_user_indexes 또는 sys.schema_unused_indexes로 사용 안 되는 인덱스를 주기적으로 삭제하세요.

② 트랜잭션 안에 외부 API 호출 +

BEGIN ... 외부 API 호출 ... COMMIT 패턴은 락을 매우 오래 잡습니다. 외부가 느려지면 DB 전체가 마비됩니다.

해결: 외부 호출은 트랜잭션 밖에서, 결과만 받아 짧은 트랜잭션으로 반영. 또는 Outbox 패턴 활용.

③ COUNT(*)로 페이지 수 정확히 보여주기 +

큰 테이블에서 COUNT(*)는 매우 느립니다. 정확한 숫자가 정말 필요한지 PM과 협의하세요.

해결: "1만+" 같은 근사치, 별도 카운터 테이블, 또는 무한 스크롤 UI 채택.

④ 인덱스 컬럼에 함수 / 연산 적용 +

WHERE UPPER(name) = 'KIM', WHERE created_at + INTERVAL ... — 모두 인덱스를 무력화합니다.

해결: 함수 기반 인덱스(PostgreSQL) 또는 보조 컬럼에 미리 변환된 값을 저장, 또는 쿼리를 다시 쓰기.

⑤ ORM의 N+1 모르고 사용 +

가장 흔하고 가장 비싼 안티패턴. 정작 개발 환경에서는 데이터가 적어서 안 보입니다.

해결: Eager loading 명시(select_related, joinedload), 슬로우 쿼리 로그 상시 모니터링, 개발 환경에 충분한 데이터 시드.

⑥ 대용량 IN 절 — WHERE id IN (1, 2, 3, ... , 50000) +

수만 개 항목의 IN 절은 옵티마이저를 헷갈리게 하고, 쿼리 파싱도 느려집니다.

해결: 임시 테이블에 넣고 JOIN, 또는 청크로 나눠 여러 번 쿼리, 또는 VALUES 절 활용.

⑦ 컬럼 타입 불일치로 인한 묵시적 변환 +

phone(VARCHAR)에 숫자로 비교하면 DB가 알아서 변환하느라 인덱스가 깨집니다. JOIN 키 타입이 다른 것도 같은 문제.

해결: 스키마 설계 시 JOIN 키 타입을 통일. 애플리케이션에서 명시적으로 같은 타입으로 전달.

⑧ 트리거 남용 +

트리거는 쓰기 트랜잭션의 일부로 동작하므로, 무거운 트리거는 모든 쓰기를 느리게 만듭니다. 디버깅도 힘듭니다.

해결: 비동기 처리(메시지 큐), CDC, 또는 애플리케이션 레이어로 이동.

⑨ 슈퍼 와이드 테이블 — 컬럼 100개 이상 +

한 페이지에 들어가는 행 수가 줄어 캐시 효율 하락. 일부 컬럼만 자주 쓴다면 더더욱 낭비.

해결: 자주 쓰는 컬럼 / 거의 안 쓰는 컬럼을 1:1 관계로 수직 분할.

⑩ DELETE 후 즉시 빈 공간 회수 기대 +

PostgreSQL은 MVCC라서 DELETE해도 공간이 즉시 안 줄어듭니다. VACUUM이 동작해야 회수됩니다.

해결: autovacuum이 잘 동작하는지 확인, 대량 삭제 시 VACUUM FULL 또는 파티션 DROP 활용.

모니터링과 진단

측정하지 않으면 개선할 수 없습니다. 실무에서 매일 보게 될 도구와 명령어들을 정리합니다.

슬로우 쿼리 로그 활성화 — 가장 먼저 할 일

mysql · my.cnf
# 1초 이상 걸린 쿼리만 로그
slow_query_log = 1
slow_query_log_file = /var/log/mysql/slow.log
long_query_time = 1
log_queries_not_using_indexes = 1
postgresql · postgresql.conf
# 500ms 이상 걸린 쿼리 로그
log_min_duration_statement = 500
log_lock_waits = on
log_temp_files = 0

실시간 활동 보기

sql
-- MySQL: 지금 돌아가는 쿼리
SHOW PROCESSLIST;
-- 더 자세히
SELECT * FROM performance_schema.events_statements_current;

-- PostgreSQL: 지금 돌아가는 쿼리와 락 대기
SELECT pid, state, wait_event, query, now() - query_start AS duration
FROM pg_stat_activity
WHERE state <> 'idle'
ORDER BY duration DESC;

쿼리 통계의 보석 — pg_stat_statements

PostgreSQL을 운영한다면 pg_stat_statements 확장을 무조건 켜세요. 모든 쿼리의 호출 횟수, 총 시간, 평균 시간이 누적됩니다. "어떤 쿼리가 가장 비싼가"를 단 한 줄로 알 수 있습니다.

postgresql
-- 총 누적 시간 기준 TOP 10 비싼 쿼리
SELECT
  substring(query, 1, 80) AS query,
  calls,
  round(total_exec_time::numeric, 2) AS total_ms,
  round(mean_exec_time::numeric, 2) AS mean_ms,
  rows
FROM pg_stat_statements
ORDER BY total_exec_time DESC
LIMIT 10;

CHAPTER 12

케이스 스터디

현장에서 자주 마주치는 3가지 시나리오. 증상을 보고 진단부터 해결까지의 흐름을 따라가 봅시다.

케이스 1. "갑자기 모든 쿼리가 느려졌어요"

증상

어제까지 잘 돌던 서비스. 오늘 오전부터 API 응답 시간이 2~10배 증가. 코드 배포는 없었음.

  1. 가장 먼저: 모니터링 대시보드에서 변곡점 시간 확인. 그 시간에 무슨 이벤트가 있었는가?
  2. 활성 쿼리 확인: pg_stat_activity / SHOW PROCESSLIST로 지금 대기 중인 쿼리들 확인. 같은 쿼리가 락 대기 중이라면? → 누군가 긴 트랜잭션을 가지고 있을 가능성.
  3. 슬로우 쿼리 로그: 새로 들어온 무거운 쿼리가 있는지. 야간 배치가 평소보다 길어졌을 수도.
  4. 통계 확인: 자동 통계 갱신이 동작했는데 분포가 바뀌어 옵티마이저가 잘못된 계획을 선택할 수도. ANALYZE 수행.
  5. 디스크 / 메모리: 디스크가 가득 차서 임시 파일을 못 만드는 경우. 또는 OOM Kill로 캐시가 비워진 경우.

케이스 2. "특정 사용자만 페이지 로딩이 느려요"

증상

대부분 사용자는 50ms 이내 응답. 일부 헤비유저(주문 수만 건)는 5초 이상.

  1. 데이터 편향 의심: 한 사용자가 가진 데이터량이 평균의 1000배라면, 그 사용자에 대한 쿼리만 풀스캔처럼 동작.
  2. 실행 계획 비교: 일반 사용자와 헤비유저로 같은 쿼리를 EXPLAIN. 옵티마이저가 다른 계획을 택했을 수 있음 (스큐 통계).
  3. 해결: 인덱스 (user_id, created_at) 형태로 정렬 컬럼까지 묶기. 또는 사용자별 페이지네이션 시 keyset 도입.
  4. 그래도 안 되면: 해당 사용자의 데이터를 별도 파티션으로, 또는 비동기 집계 + 캐시.

케이스 3. "쓰기가 갑자기 폭증하면서 DB가 멈춰요"

증상

이벤트 트래픽 시간대만 되면 INSERT가 큐에 쌓이고 DB CPU 100%. 평소엔 멀쩡.

  1. 인덱스 갯수 점검: 인덱스가 10개라면 INSERT 1번이 인덱스 갱신 10번. 정말 필요한 인덱스만 남기기.
  2. 커밋 정책: row 단위로 commit하지 말고 배치 commit. innodb_flush_log_at_trx_commit=2 같은 안전성 트레이드오프 고려.
  3. Bulk insert: 1만 건 1만 번 INSERT 말고, 한 번에 1000건씩 multi-value INSERT.
  4. 비동기 버퍼링: Kafka 같은 큐를 앞단에 두고, DB는 일정 속도로 소비하게.
  5. 장기적: 쓰기 전용 샤딩, 또는 OLTP / OLAP 분리.
진단의 황금률

① 추측 금지, 측정만. ② 변화의 시점을 먼저 특정. ③ 한 번에 하나의 변수만 바꾸기. ④ 모든 변경은 롤백 계획과 함께.
