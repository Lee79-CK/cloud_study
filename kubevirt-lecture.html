00 — 문제의식

왜 쿠버네티스 안에서 VM을 돌리려 할까?

결론부터 말하면 "인프라 스택을 두 개 운영하기 싫어서"입니다. 대부분의 조직은 100% 컨테이너로 가지 못했고, 앞으로도 그럴 가능성이 큽니다.

현실의 데이터센터에는 컨테이너로 옮기기 어려운 워크로드가 늘 남아 있습니다. 전체 운영체제를 가정하고 만들어진 레거시 애플리케이션, 컨테이너 안에서는 돌릴 수 없는 윈도우 워크로드, 하드웨어에 직접 접근해야 하는 통신 네트워크 기능(NFV)·데이터베이스 어플라이언스·특수 ML 툴킷 등이 그렇습니다.

그래서 오랫동안 답은 스택을 둘로 나누는 것이었습니다. 컨테이너용 쿠버네티스 클러스터 하나, VM용 하이퍼바이저 플랫폼(주로 VMware vSphere) 하나. 결과적으로 도구도 둘, 운영 팀도 둘, 과금 모델도 둘이 됩니다.

🧱 스택이 둘일 때

  • kubectl과 vCenter, 서로 다른 두 콘솔을 오간다
  • VM 팀과 컨테이너 팀이 분리되어 협업 비용 증가
  • 네트워크·스토리지·인증을 두 번 설계·운영
  • 라이선스·서포트 계약이 이중으로 발생

🧩 KubeVirt로 합치면

  • kubectl 하나로 Pod와 VM을 동일하게 다룬다
  • 같은 RBAC·네트워크·스토리지·모니터링 재사용
  • VM 옆 컨테이너를 같은 네임스페이스·Service로 연결
  • 오픈소스 기반으로 벤더 종속·라이선스 부담 완화
시장 배경

Broadcom의 VMware 인수 이후 라이선스 비용이 크게 오르면서, 많은 조직이 독점 하이퍼바이저의 대안을 적극적으로 찾고 있습니다. 이미 쿠버네티스를 쓰고 있다면, 그 위에서 VM까지 돌릴 수 있는 KubeVirt가 유력한 후보로 떠오른 이유입니다.

KubeVirt는 쿠버네티스를 대체하지 않습니다. 컨테이너가 잘하는 일은 컨테이너에 맡기고, 컨테이너가 할 수 없는 일(전통 VM)을 쿠버네티스가 다룰 수 있게 능력을 더해주는 확장 기능입니다. KubeVirt 위의 VM은 같은 클러스터 네트워크로 Pod와 통신할 수 있어서, VM으로 띄운 데이터베이스와 컨테이너로 띄운 API 서버를 같은 네임스페이스에 두고 하나의 Service로 묶을 수도 있습니다.

01 — 기초 개념

VM과 컨테이너, 그리고 KubeVirt의 자리

먼저 가상머신과 컨테이너의 차이를 짧게 복습하고, KubeVirt가 정확히 무엇을 하는지 한 문장으로 정리합니다.

VM vs 컨테이너, 무엇이 다른가

둘 다 "격리된 실행 환경"을 만들지만 격리하는 층(layer)이 다릅니다.

🖥️ 가상머신(VM)

  • 하이퍼바이저가 하드웨어를 가상화한다
  • VM마다 독립된 게스트 OS 커널을 가진다
  • 격리가 강하지만 무겁고 부팅이 느리다
  • 윈도우·레거시·커널 의존 워크로드에 적합

📦 컨테이너

  • 호스트 OS 커널을 공유한다
  • 프로세스 단위로 가볍게 격리(namespace·cgroup)
  • 매우 가볍고 시작이 빠르며 밀도가 높다
  • 상태 없는 마이크로서비스에 이상적
한 줄 핵심

컨테이너는 OS 위의 "프로세스 격리", VM은 OS 자체를 통째로 돌리는 "하드웨어 격리"입니다. 그래서 강한 격리나 다른 커널/OS가 필요하면 여전히 VM이 답입니다.

그래서 KubeVirt는 무엇인가

KubeVirt는 쿠버네티스에 VM을 1급 객체로 다루는 능력을 더하는 오픈소스 확장입니다. 쿠버네티스를 대체하거나 별도 플랫폼을 세우는 대신, CRD(Custom Resource Definition)를 추가해 VirtualMachine 같은 새로운 리소스를 쿠버네티스 API에 등록합니다.

그 결과, 우리가 이미 아는 방식 그대로 VM을 정의·시작·정지·관리할 수 있습니다.

# 컨테이너를 다루던 그대로…
kubectl get pods
kubectl apply -f my-app.yaml

# …VM도 똑같이 다룬다 (kubectl + CRD)
kubectl get virtualmachines        # 별칭: vm
kubectl apply -f my-vm.yaml
kubectl get virtualmachineinstances  # 별칭: vmi (실제 실행 인스턴스)
VM이 Pod와 동일한 API 표면에 올라온다 — 이것이 KubeVirt의 핵심 가치
# KubeVirt는 검증된 리눅스 가상화 스택 위에 서 있다
Kubernetes      # 오케스트레이션 / 스케줄링 / API
   └─ KubeVirt   # VM을 쿠버네티스 리소스로 변환·관리
        └─ libvirt   # VM 수명주기 관리 데몬
             └─ QEMU      # 하드웨어 에뮬레이션
                  └─ KVM   # 리눅스 커널의 하드웨어 가속 가상화
새 가상화 기술을 만든 게 아니라, 성숙한 KVM/QEMU/libvirt를 쿠버네티스와 연결한다
CNCF 현황 (2026)

KubeVirt는 2017년 Red Hat에서 시작해 2019년 CNCF Sandbox, 2022년 Incubating 단계로 올라섰습니다. 아직 Graduated(졸업) 전이지만 NVIDIA·Intel·AMD·IBM 등 폭넓은 후원과 활발한 커뮤니티를 갖춘, VM을 쿠버네티스에서 돌리는 사실상의 표준 도구입니다.

02 — 아키텍처

안을 들여다보기: VM은 어떻게 Pod가 되는가

KubeVirt의 구조는 쿠버네티스의 표준 패턴(컨트롤러 · DaemonSet · CRD)을 그대로 따릅니다. 오퍼레이터를 이해해 봤다면 익숙하게 느껴질 겁니다.

핵심 아이디어 한 가지만 기억하세요. 모든 VM은 결국 하나의 Pod(virt-launcher) 안에서 실행됩니다. 그래서 VM이 스케줄링·CNI 네트워킹·CSI 스토리지·네임스페이스·RBAC 같은 쿠버네티스의 모든 혜택을 그대로 누립니다. 아래 다이어그램의 각 구성 요소를 눌러 역할을 확인해 보세요.

컨트롤 플레인 (클러스터 레벨)
워커 노드 (각 노드마다)
virt-launcher Pod 내부 (실제 가상화)
구성 요소를 선택하세요
Interactive
왼쪽 박스를 누르면 각 컴포넌트가 무슨 일을 하는지 설명이 여기에 나타납니다. 위에서 아래로 "요청 → 결정 → 실행" 흐름으로 읽으면 이해가 쉽습니다.
파랑 = 쿠버네티스/커널 영역 · 호박색 = KubeVirt 영역

전체 흐름을 한 문장으로

사용자가 kubectl apply로 VM을 만들면 → virt-api가 받아 검증하고 → virt-controller가 그 VM을 실행할 virt-launcher Pod를 만들고 → 쿠버네티스 스케줄러가 적당한 노드에 Pod를 배치하고 → 그 노드의 virt-handler가 Pod 안의 libvirt에게 "이 스펙대로 VM을 띄워라"라고 지시하고 → libvirt/QEMU가 KVM 가속을 써서 실제 VM을 부팅합니다.

왜 "Pod 안의 VM"이 좋은가

VM이 Pod라는 것은, 쿠버네티스가 이미 잘하는 모든 것 — 스케줄링, 헬스 체크, 네트워크 정책, 스토리지 연결, 권한 관리 — 을 VM에도 공짜로 적용한다는 뜻입니다. KubeVirt가 바퀴를 다시 발명하지 않는 이유입니다.

03 — 핵심 리소스(CRD)

알아야 할 네 가지 리소스

KubeVirt를 쓴다는 것은 결국 몇 가지 새로운 쿠버네티스 리소스를 다루는 일입니다. 가장 자주 만나는 것들만 추렸습니다.

① VirtualMachine vs ② VirtualMachineInstance

가장 헷갈리는 한 쌍입니다. 비유하면 Deployment와 Pod의 관계와 비슷합니다.

  • VirtualMachine (VM) — VM의 "명세서이자 스위치"입니다. "이런 VM이 켜져 있어야 한다/꺼져 있어야 한다"는 의도(desired state)를 담습니다. 꺼다 켜도 정의는 유지됩니다.
  • VirtualMachineInstance (VMI) — 실제로 지금 돌고 있는 한 번의 실행 인스턴스입니다. VM을 켜면 VMI가 생기고, 끄면 사라집니다. 보통 직접 만들기보다 VM이 자동으로 만들게 둡니다.
apiVersion: kubevirt.io/v1
kind: VirtualMachine
metadata:
  name: testvm
spec:
  runStrategy: Halted        # 처음엔 꺼진 상태로 생성
  template:                    # 이 안이 VMI(실행체)의 설계도
    spec:
      domain:
        devices:
          disks:
            - name: rootdisk
              disk: { bus: virtio }
        resources:
          requests: { memory: 1Gi }
      volumes:
        - name: rootdisk
          containerDisk:          # OS 이미지를 컨테이너 이미지로 배포
            image: quay.io/containerdisks/fedora:latest
runStrategy로 켜짐/꺼짐 의도를 표현 · template 아래가 실제 VM 스펙
# instancetype/preference로 스펙을 재사용 — 최신 권장 방식
apiVersion: kubevirt.io/v1
kind: VirtualMachine
metadata:
  name: fedora-vm
spec:
  runStrategy: Always
  instancetype: { name: u1.medium }   # CPU/메모리 규격(T셔츠 사이즈)
  preference:   { name: fedora }      # OS별 권장 설정(버스·기능 등)
  template:
    spec:
      domain: { devices: {} }
      volumes:
        - name: rootdisk
          containerDisk: { image: quay.io/containerdisks/fedora:latest }
스펙을 매번 손으로 쓰지 않고 표준 규격을 골라 쓴다 — 일관성·재사용성↑
# CDI(Containerized Data Importer)가 외부 이미지를 PVC로 가져온다
apiVersion: cdi.kubevirt.io/v1beta1
kind: DataVolume
metadata:
  name: ubuntu-disk
spec:
  source:
    http:                         # URL의 cloud 이미지를 가져옴
      url: "https://.../ubuntu-22.04.img"
  storage:
    resources:
      requests: { storage: 10Gi }
URL·레지스트리·기존 PVC 등에서 디스크를 받아 영구 스토리지(PVC)로 준비

③ DataVolume / CDI — 디스크는 어디서 오나

VM에는 부팅할 디스크가 필요합니다. KubeVirt의 동반 프로젝트 CDI(Containerized Data Importer)가 외부 클라우드 이미지(URL, 컨테이너 레지스트리, 기존 PVC 등)를 받아와 쿠버네티스 영구 볼륨(PVC)으로 만들어 줍니다. 이렇게 준비한 디스크를 VM에 연결합니다.

④ instancetype / preference — 스펙의 표준화

매번 CPU·메모리·버스 설정을 손으로 쓰면 실수도 많고 번거롭습니다. instancetype은 "u1.medium" 같은 T셔츠 사이즈식 규격(CPU·메모리)을, preference는 게스트 OS별 권장 설정(디스크 버스, 기능 게이트 등)을 모아둔 재사용 가능한 묶음입니다. 클라우드의 인스턴스 타입과 비슷한 개념이라고 보면 됩니다.

용어 정리

VM은 의도(켜짐/꺼짐), VMI는 실행체, DataVolume은 디스크 공급, instancetype/preference는 스펙 템플릿. 이 네 가지만 잡으면 KubeVirt 리소스의 8할은 이해한 것입니다.

04 — 직접 해보기

설치하고 첫 VM을 부팅하기

개념을 봤으니 실제 명령을 따라가 봅시다. 크게 세 단계입니다: ① KubeVirt 설치 → ② VM 정의 → ③ 켜고·접속하고·끄기.

전제 조건

워커 노드가 하드웨어 가상화(KVM)를 지원해야 합니다. 클라우드 VM 위라면 "중첩 가상화(nested virtualization)"가 켜져 있어야 합니다. 검증이 어려운 환경에서는 소프트웨어 에뮬레이션 모드로도 테스트할 수 있습니다(성능은 느림).

# 최신 릴리스 버전을 변수로 (예: v1.8 계열 — 실제 최신 태그 확인)
export VERSION=$(curl -s https://storage.googleapis.com/kubevirt-prow/release/kubevirt/kubevirt/stable.txt)

# 1) KubeVirt 오퍼레이터 설치
kubectl apply -f https://github.com/kubevirt/kubevirt/releases/download/${VERSION}/kubevirt-operator.yaml
# 2) KubeVirt CR 배포(오퍼레이터가 나머지 컴포넌트를 폄침)
kubectl apply -f https://github.com/kubevirt/kubevirt/releases/download/${VERSION}/kubevirt-cr.yaml

# 3) 준비될 때까지 대기
kubectl -n kubevirt wait kv kubevirt --for condition=Available --timeout=10m
오퍼레이터 패턴 — CR 하나로 virt-api/controller/handler가 자동 배치된다
# 컴포넌트가 다 떴는지 확인
kubectl get pods -n kubevirt
# virt-api / virt-controller / virt-handler 가 Running 이어야 정상

# 노드의 가상화 지원 여부 점검(선택)
kubectl get nodes -o json | grep -i devices.kubevirt.io/kvm

# virtctl CLI 설치(VM 전용 보조 명령)
# krew 사용 시:
kubectl krew install virt
virtctl은 console·vnc·migrate 등 VM 특화 명령을 제공하는 보조 CLI
# 정의 적용(아직 꺼진 상태)
kubectl apply -f testvm.yaml

# 켜기 / 끄기 / 재시작
virtctl start   testvm
virtctl stop    testvm
virtctl restart testvm

# 콘솔·화면·SSH 접속
virtctl console testvm      # 시리얼 콘솔
virtctl vnc     testvm      # 그래픽 콘솔
virtctl ssh     user@testvm

# 무중단 이주(라이브 마이그레이션)
virtctl migrate testvm
Pod를 못 다루던 영역 — 콘솔·VNC·라이브 마이그레이션을 virtctl이 채운다

05 — 비교

다른 가상화 솔루션과 무엇이 다른가

KubeVirt의 위치를 이해하려면 전통 하이퍼바이저·IaaS 플랫폼·순수 컨테이너와 나란히 놓고 보는 게 가장 빠릅니다.

고르는 기준 한 줄

이미 쿠버네티스가 운영의 중심이고 거기에 VM까지 얹고 싶다면 → KubeVirt. 순수 대규모 VM 전용 데이터센터라면 → vSphere/OpenStack이 여전히 강점. 새 워크로드가 상태 없는 앱이라면 → 그냥 컨테이너가 답입니다.

오해 주의

KubeVirt는 "VM을 컨테이너로 변환"하지 않습니다. VM은 그대로 진짜 VM(KVM)으로 돌고, 다만 그 VM을 관리하는 방식을 쿠버네티스로 통일하는 것입니다. 따라서 컨테이너 수준의 가벼운 오버헤드를 기대하면 안 됩니다.

06 — 응용 사례

실제로 어디에 쓰는가

KubeVirt가 빛나는 상황은 대부분 "컨테이너만으로는 안 되는데, 인프라는 쿠버네티스로 통일하고 싶을 때"입니다.

CASE 01

레거시 애플리케이션 이전

전체 OS를 가정하고 만든 오래된 앱을 다시 작성하지 않고, VM 그대로 쿠버네티스로 옮겨 한 플랫폼에서 운영합니다.

CASE 02

VM + 컨테이너 혼합 워크로드

VM으로 띄운 DB와 컨테이너로 띄운 API를 같은 네임스페이스·Service로 묶어, 한 애플리케이션처럼 운영합니다.

CASE 03

윈도우 / 특수 워크로드

컨테이너화가 불가능한 윈도우 서버, 통신 네트워크 기능(NFV), 어플라이언스성 소프트웨어를 수용합니다.

CASE 04

GPU · AI 워크로드

v1.8의 PCIe NUMA 토폴로지 인식으로, 가상화된 AI 워크로드가 베어메탈에 가까운 저지연으로 GPU에 접근합니다. vLLM 같은 LLM 서빙도 VM에서 가능합니다.

CASE 05

CI/CD용 일회성 VM

빌드·테스트마다 깨끗한 VM을 띄웠다 버리는 식으로, 격리된 일회성 환경을 쿠버네티스 워크플로 안에서 자동화합니다.

CASE 06

개발 · 테스트 환경

다양한 OS·커널 버전이 필요한 개발/QA 환경을, 셀프서비스로 빠르게 만들고 회수합니다.

CASE 07

VMware 대안 (탈-독점)

라이선스 비용이 급등한 독점 하이퍼바이저를 단계적으로 대체하며, 운영을 쿠버네티스로 수렴시킵니다.

CASE 08

컨피덴셜 컴퓨팅 · 엣지

Intel TDX 기반 신뢰 실행 VM, 그리고 엣지 노드에서의 격리 실행 등 보안·분산 시나리오로 확장됩니다.

AI 인프라 관점

최근 KubeVirt는 NVIDIA·Intel 등과 함께 GPU 패스스루, NUMA 인식, 라이브 마이그레이션을 강화해 왔습니다. 이종(ARM64/x86) 노드가 섞인 클러스터에서도 VM을 노드 아키텍처에 맞춰 스케줄링할 수 있어, 실험용 홈랩부터 사설 클라우드 AIOps까지 같은 컨트롤 플레인으로 다룰 수 있습니다.

07 — 실전 팁 & 주의점

현장에서 자주 막히는 것들

개념을 넘어 실제로 운영하기 전에 알아두면 좋은 포인트를 압축했습니다.

스토리지 — VM 디스크는 PVC다

VM 디스크는 결국 쿠버네티스 영구 볼륨(PVC) 위에 놓입니다. 라이브 마이그레이션을 하려면 여러 노드에서 동시에 붙일 수 있는 ReadWriteMany 또는 블록 모드를 지원하는 CSI 드라이버가 필요합니다. 디스크 공급은 CDI(DataVolume)로 표준화하세요.

네트워킹 — 기본은 Pod 네트워크, 고급은 Multus

가장 단순한 구성은 VM이 Pod 네트워크를 그대로 쓰는 masquerade 방식입니다. VM에 별도 VLAN·전용 NIC가 필요하면 Multus로 보조 네트워크를 붙입니다. v1.8에서는 passt 바인딩이 코어로 승격되어 네트워킹 선택지가 넓어졌습니다.

라이브 마이그레이션 — 무중단 이주

노드 점검·롤링 업데이트 시 VM을 끄지 않고 다른 노드로 옮기는 기능입니다. 공유 스토리지와 노드 간 네트워크 조건이 맞아야 하며, 모든 워크로드가 마이그레이션 가능한 것은 아니므로(예: 로컬 디스크·특정 패스스루) 사전에 정책을 정해 두는 것이 좋습니다.

멀티 아키텍처(ARM64 / x86)

KubeVirt는 ARM64를 포함한 멀티 아키텍처를 지원합니다. 단, VM 이미지(게스트 OS·containerDisk)는 노드의 CPU 아키텍처와 일치해야 합니다. 이종 클러스터에서는 nodeSelector나 affinity로 VM을 올바른 아키텍처의 노드에 고정하세요.

초보가 흔히 겪는 함정

① 노드에 KVM/중첩 가상화가 꺼져 있어 VMI가 안 뜸 → 노드 가상화 지원 확인. ② running:true로 켰는데 의도대로 동작 안 함 → 최신 권장은 runStrategy. ③ 디스크 PVC가 RWO라 마이그레이션 실패 → 공유 스토리지 검토. ④ ARM 노드에 x86 이미지를 스케줄 → 아키텍처 불일치.

09 — 마무리

정리, 그리고 다음 단계

KubeVirt는 "VM이냐 컨테이너냐"의 논쟁을 "둘 다, 같은 컨트롤 플레인에서"로 바꿔놓는 도구입니다.

핵심을 다시 묶으면: KubeVirt는 쿠버네티스에 CRD로 VirtualMachine을 더해, 검증된 KVM/QEMU/libvirt 스택으로 VM을 돌립니다. 모든 VM은 virt-launcher Pod 안에서 실행되므로 쿠버네티스의 스케줄링·네트워크·스토리지·권한을 그대로 물려받습니다. 그래서 레거시·윈도우·GPU/AI 워크로드를 컨테이너 옆에서 하나의 도구로 운영할 수 있습니다.

STEP 01
로컬에서 띄워보기
minikube / kind 같은 테스트 클러스터에 오퍼레이터를 설치하고 Fedora VM 하나를 부팅해 console로 들어가 보세요.
STEP 02
스토리지·네트워크 붙이기
CDI로 DataVolume을 만들고, Multus로 보조 네트워크를 붙여 실제 운영 형태에 가깝게 구성해 봅니다.
STEP 03
혼합 워크로드 설계
VM(DB) + 컨테이너(API)를 같은 Service로 묶고, 라이브 마이그레이션·instancetype 표준화까지 실험합니다.
한 걸음 더

공식 문서(kubevirt.io/user-guide), CDI·Multus·HyperConverged Operator(HCO) 묶음, 그리고 OKD/OpenShift Virtualization(KubeVirt 기반 상용 배포판)을 살펴보면 운영 수준의 그림이 완성됩니다.
