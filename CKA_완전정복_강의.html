강의 안내 · 시험 개요

START HERE · 시험이 뭐고 어떻게 준비하나

CKA(Certified Kubernetes Administrator)는 실제로 쿠버네티스 클러스터를 운영할 수 있는가를 검증하는 자격증입니다. 객관식이 아니라, 진짜 리눅스 터미널에서 클러스터를 만지며 문제를 푸는 실기 시험이에요.

▸CKA는 이런 시험입니다

항목내용
주관CNCF(Cloud Native Computing Foundation) + 리눅스 재단
형식100% 실기 — 실제 클러스터에서 kubectl로 문제 해결 (객관식 없음)
문항/시간약 15~20문제 · 2시간
합격선66% (부분 점수 인정 — 일부만 맞아도 점수 획득)
K8s 버전현재 v1.34 기반 (최신 마이너 버전에 4~8주 내 정렬, v1.35로 전환 중)
오픈북시험 중 kubernetes.io/docs, /blog, helm.sh/docs 한 탭 열람 가능
유효기간합격일로부터 2년
응시료약 $445 (Killer.sh 모의 시뮬레이터 2회 포함)
시험 전략의 핵심

합격선이 66%라는 건, 완벽할 필요가 없다는 뜻입니다. 막히는 문제는 표시(flag)해두고 넘어가서, 풀 수 있는 문제를 빠르게 다 푸는 게 합격의 정석이에요.

▸2025년 개편 — 5개 도메인과 배점

2025년 2월 개편으로 도메인이 재구성됐고, Helm·Kustomize·CRD/Operator·Gateway API 같은 최신 운영 주제가 들어왔습니다. 배점이 곧 공부 시간 배분이라고 생각하세요.

도메인배점핵심 키워드
① 클러스터 아키텍처 · 설치 · 구성25%kubeadm, RBAC, Helm, Kustomize, CRD/Operator, etcd 백업, 업그레이드
② 워크로드 · 스케줄링15%Deployment, ConfigMap/Secret, HPA, affinity, taint/toleration
③ 스토리지10%PV, PVC, StorageClass, 동적 프로비저닝, access mode
④ 서비스 · 네트워킹20%Service, Ingress, Gateway API, NetworkPolicy, CoreDNS
⑤ 트러블슈팅30%Pod/노드/컨트롤플레인/네트워크 장애 진단, 로그, 이벤트
공부 순서 추천

트러블슈팅(30%) + 클러스터 아키텍처(25%) = 전체의 55%. 시간이 부족하면 이 둘에 공부 시간의 절반 이상을 쓰세요. 단, 트러블슈팅은 다른 도메인을 다 알아야 풀리므로, 학습은 ①→⑤ 순서로 하고 복습은 ⑤·①에 집중하는 게 좋습니다.

▸이 강의를 보는 법

  • 개념 → 비유 → 그림 → 명령어 → 실전 팁 순서로 반복됩니다. 비유 박스(보라색)는 "이게 현실로 치면 뭔데?"를 잡아줍니다.
  • 코드 블록의 탭을 눌러 명령어형(kubectl)과 선언형(YAML)을 비교하세요. 복사 버튼으로 바로 실습할 수 있습니다.
  • 각 도메인 끝의 퀴즈로 바로 점검하고, 맨 아래 터미널 연습실에서 명령어를 손에 익히세요.
실습 환경부터 만드세요

CKA는 머리로 외우는 시험이 아니라 손이 기억하는 시험입니다. killer.sh(응시료에 포함), kind(로컬 도커 클러스터), minikube, 또는 두 대 이상의 VM으로 kubeadm 클러스터를 직접 만들어 두고 강의를 따라가세요.

01

쿠버네티스가 뭐죠?

FOUNDATION · 시험 전에 반드시 잡아야 할 기초

쿠버네티스(Kubernetes, 줄여서 K8s — k와 s 사이에 글자가 8개라서)는 수많은 컨테이너를 자동으로 배치·관리·복구해 주는 시스템입니다. 그리스어로 "조타수(helmsman)"라는 뜻이라, 로고도 배의 키(타륜)예요.

🚢
컨테이너 하나하나가 화물 컨테이너라면, 쿠버네티스는 그 컨테이너들을 어느 배(서버)에 실을지, 하나가 바다에 빠지면(죽으면) 어떻게 다시 채울지를 결정하는 항만 관제 시스템 + 조타수입니다. CKA는 바로 그 관제 시스템을 운영하는 자격증이고요.

▸왜 컨테이너, 왜 쿠버네티스?

가상머신(VM)은 OS를 통째로 띄우기 때문에 무겁습니다. 컨테이너는 OS 커널을 공유하고 앱과 의존성만 격리해서 가볍고 빠르게 뜨죠. 그런데 컨테이너가 수백 개가 되면 "어디에 띄울지, 죽으면 누가 살릴지, 트래픽은 어떻게 나눌지"를 사람이 못 합니다. 그걸 대신해 주는 게 쿠버네티스의 오케스트레이션입니다.

가상머신 (VM)컨테이너
격리 단위OS 전체프로세스 + 의존성
크기GB 단위MB 단위
부팅수십 초~분초 단위
커널각자 따로호스트와 공유

▸가장 중요한 개념: 선언형(Declarative)

쿠버네티스의 철학은 "어떻게(how)"가 아니라 "무엇을(what)"을 말하는 것입니다. "nginx 컨테이너를 3개 띄워줘"라고 원하는 상태(desired state)를 선언하면, 쿠버네티스가 알아서 현재 상태를 거기에 맞춥니다. 하나가 죽으면? 자동으로 하나를 또 띄웁니다. 이걸 조정 루프(reconciliation loop)라고 부릅니다.

🌡️
온도 조절기(thermostat)와 같습니다. "방 온도를 22도로 유지해"라고 목표만 정하면, 추워지면 난방을 켜고 더워지면 끄는 걸 알아서 합니다. "지금 난방 3분 켜"라고 일일이 명령하지 않죠. 쿠버네티스도 똑같이 목표 상태를 향해 끊임없이 스스로 조정합니다.

▸핵심 오브젝트 미리보기

쿠버네티스에서 다루는 모든 것은 오브젝트(리소스)이고, 대부분 YAML로 정의합니다. 시험 내내 만지게 될 주연들입니다.

오브젝트한 줄 설명
Pod컨테이너를 담는 가장 작은 배포 단위. 보통 컨테이너 1개를 품음
ReplicaSet같은 Pod를 정해진 개수만큼 유지
DeploymentReplicaSet을 관리하며 무중단 업데이트·롤백 담당 (가장 많이 씀)
Service여러 Pod에 고정된 접속 주소를 부여 (Pod IP는 계속 바뀌므로)
ConfigMap / Secret설정값 / 민감정보를 Pod와 분리해서 주입
Namespace리소스를 묶는 가상의 폴더. 팀·환경별로 격리
PV / PVC영구 저장소(PV)와 그 요청서(PVC)
YAML의 뼈대는 항상 같습니다

모든 K8s 오브젝트 YAML은 apiVersion → kind → metadata → spec 네 칸으로 시작합니다. 이 네 글자만 기억하면 어떤 리소스든 절반은 쓴 거예요.

apiVersion: v1                  # Pod는 핵심 그룹이라 v1
kind: Pod                       # 만들 오브젝트의 종류
metadata:
  name: my-nginx               # 이름표
  labels:
    app: web                   # 라벨 = 검색/연결용 꼬리표
spec:                            # 원하는 상태(desired state)
  containers:
  - name: nginx
    image: nginx:1.27          # 어떤 이미지를
    ports:
    - containerPort: 80
실전에선 YAML을 외워 쓰지 않습니다

시험에서는 명령형으로 뼈대를 뽑아 수정합니다: kubectl run my-nginx --image=nginx:1.27 --dry-run=client -o yaml > pod.yaml — 이렇게 만든 뒤 필요한 부분만 고치는 게 가장 빠릅니다. (뒤 치트시트에서 자세히)

02

클러스터 구조 한눈에

FOUNDATION · 컨트롤 플레인 vs 워커 노드

쿠버네티스 클러스터는 크게 두뇌(컨트롤 플레인)와 일꾼(워커 노드)으로 나뉩니다. 아래 다이어그램의 각 부품을 눌러보세요 — 시험 트러블슈팅의 절반은 "어느 부품이 고장났는지" 찾는 일입니다.

부품을 클릭하면 역할과 시험 포인트가 여기에 나옵니다.
▲ 단일 클러스터 구조 — 실제 운영에선 컨트롤 플레인을 3대로 이중화(HA)합니다

▸두뇌 — 컨트롤 플레인 4인방

  • kube-apiserver — 모든 요청이 거쳐가는 유일한 관문. kubectl도, 다른 컴포넌트도 전부 API 서버하고만 대화합니다. 여기가 죽으면 클러스터를 조작할 수 없습니다.
  • etcd — 클러스터의 모든 상태를 저장하는 키-값 데이터베이스. "클러스터의 기억"입니다. etcd 백업은 CKA 단골 문제이니 꼭 손에 익히세요.
  • kube-scheduler — 새 Pod를 어느 노드에 올릴지 결정합니다. 자원(CPU/메모리), affinity, taint 등을 보고 점수를 매겨 배치합니다.
  • kube-controller-manager — "현재 상태 ≠ 원하는 상태"면 맞추는 조정 루프들의 집합. Pod가 죽으면 다시 만드는 게 이 친구들 일입니다.

▸일꾼 — 워커 노드 3인방

  • kubelet — 각 노드의 현장 감독. API 서버에게 "내 노드에 이 Pod 올려야 한다"는 지시를 받아 실제로 컨테이너를 띄우고, 상태를 보고합니다. kubelet이 죽으면 그 노드는 NotReady가 됩니다.
  • kube-proxy — Service로 들어온 트래픽을 적절한 Pod로 전달하는 네트워크 규칙(iptables/IPVS)을 관리합니다.
  • 컨테이너 런타임 (containerd) — 실제로 컨테이너를 실행하는 엔진. 도커는 옛말이고 지금은 containerd가 표준입니다. 디버깅엔 crictl을 씁니다.
트러블슈팅 첫 질문은 항상 이것

"정적 Pod(static pod)로 뜨는 컨트롤 플레인 컴포넌트의 manifest는 어디에?" → /etc/kubernetes/manifests/. kubelet 설정은 /var/lib/kubelet/config.yaml, kubelet 서비스 상태는 systemctl status kubelet, 로그는 journalctl -u kubelet. 이 경로들은 통째로 외우세요.

정적 Pod: /etc/kubernetes/manifests/ kubeconfig: /etc/kubernetes/admin.conf kubelet 설정: /var/lib/kubelet/config.yaml PKI 인증서: /etc/kubernetes/pki/
D1

클러스터 아키텍처 · 설치 · 구성

DOMAIN 1 · 배점 25% · kubeadm · RBAC · Helm · etcd · 업그레이드

두 번째로 큰 도메인입니다. 클러스터를 직접 세우고(kubeadm), 권한을 나누고(RBAC), 패키지를 설치하고(Helm/Kustomize), 백업·업그레이드하는 운영의 정석을 다룹니다.

1.1RBAC — 누가 무엇을 할 수 있는가

RBAC(Role-Based Access Control)은 "누가(주체) 무엇을(리소스) 어떻게(동사) 할 수 있는가"를 정하는 권한 체계입니다. 시험 단골이며, 부품 4개의 조합으로 이해하면 쉽습니다.

🏢
회사 출입증과 같습니다. Role은 "3층 서버실 출입 가능"이라는 권한 묶음이고, RoleBinding은 그 권한을 특정 직원에게 발급하는 행위입니다. 권한 정의와 발급이 분리돼 있어, 한 Role을 여러 명에게 재사용할 수 있죠.
오브젝트범위역할
Role네임스페이스권한 묶음 정의 (예: pods를 get/list)
ClusterRole클러스터 전체노드·PV 등 비네임스페이스 리소스, 또는 전 네임스페이스 권한
RoleBinding네임스페이스Role/ClusterRole을 주체에게 연결
ClusterRoleBinding클러스터 전체ClusterRole을 클러스터 범위로 연결
# 1) Role 생성: dev 네임스페이스에서 pods를 읽을 수 있는 권한
kubectl create role pod-reader -n dev \
  --verb=get,list,watch --resource=pods

# 2) ServiceAccount 생성
kubectl create serviceaccount app-sa -n dev

# 3) RoleBinding: 위 Role을 SA에게 연결
kubectl create rolebinding read-pods -n dev \
  --role=pod-reader --serviceaccount=dev:app-sa

# 4) 권한 검증 — 시험에서 꼭 확인!
kubectl auth can-i list pods -n dev \
  --as=system:serviceaccount:dev:app-sa
yes
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: pod-reader
  namespace: dev
rules:
- apiGroups: [""]            # "" = 핵심 그룹(pods 등)
  resources: ["pods"]
  verbs: ["get", "list", "watch"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: read-pods
  namespace: dev
roleRef:
  kind: Role
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io
subjects:
- kind: ServiceAccount
  name: app-sa
  namespace: dev
auth can-i 를 습관처럼

RBAC 문제는 만들고 끝이 아니라 반드시 검증까지 해야 점수가 안전합니다. kubectl auth can-i <verb> <resource> --as=<user> 로 yes가 나오는지 확인하세요.

1.2kubeadm — 클러스터를 직접 세우기

kubeadm은 프로덕션급 클러스터를 부트스트랩하는 공식 도구입니다. 흐름은 ① 컨트롤 플레인 init → ② CNI 설치 → ③ 워커 join 세 단계입니다.

# 컨트롤 플레인 초기화 (Pod 네트워크 대역 지정)
sudo kubeadm init --pod-network-cidr=10.244.0.0/16

# kubeconfig를 일반 사용자가 쓸 수 있게 복사
mkdir -p $HOME/.kube
sudo cp -i /etc/kubernetes/admin.conf $HOME/.kube/config
sudo chown $(id -u):$(id -g) $HOME/.kube/config

# CNI(네트워크 플러그인) 설치 — 이게 없으면 노드가 NotReady
kubectl apply -f https://raw.githubusercontent.com/.../calico.yaml

# join 토큰을 잃어버렸다면 새로 발급
kubeadm token create --print-join-command
# 워커 노드에서 실행 (init 출력의 join 명령을 그대로)
sudo kubeadm join 172.30.1.2:6443 \
  --token abcdef.0123456789abcdef \
  --discovery-token-ca-cert-hash sha256:1234...

# 컨트롤 플레인에서 확인
kubectl get nodes
NAME           STATUS   ROLES           AGE   VERSION
controlplane   Ready    control-plane   5m    v1.34.0
worker-1       Ready    <none>          1m    v1.34.0

1.3클러스터 업그레이드 — 단골 시험 문제

업그레이드는 한 마이너 버전씩, 컨트롤 플레인 먼저, 워커 나중이 철칙입니다. 노드를 drain해 비우고, 업그레이드하고, uncordon으로 되살립니다.

# 1) kubeadm 업그레이드
sudo apt-mark unhold kubeadm
sudo apt-get update && sudo apt-get install -y kubeadm=1.35.0-1.1
sudo apt-mark hold kubeadm

# 2) 계획 확인 후 적용
sudo kubeadm upgrade plan
sudo kubeadm upgrade apply v1.35.0

# 3) 노드 비우기 → kubelet/kubectl 업그레이드 → 복귀
kubectl drain controlplane --ignore-daemonsets
sudo apt-mark unhold kubelet kubectl
sudo apt-get install -y kubelet=1.35.0-1.1 kubectl=1.35.0-1.1
sudo apt-mark hold kubelet kubectl
sudo systemctl daemon-reload && sudo systemctl restart kubelet
kubectl uncordon controlplane
# 컨트롤 플레인에서 먼저 워커를 drain
kubectl drain worker-1 --ignore-daemonsets --delete-emptydir-data

# 워커 노드에 SSH 접속 후
sudo apt-mark unhold kubeadm && sudo apt-get install -y kubeadm=1.35.0-1.1
sudo apt-mark hold kubeadm
sudo kubeadm upgrade node            # 워커는 'node' (apply 아님!)
sudo apt-mark unhold kubelet kubectl
sudo apt-get install -y kubelet=1.35.0-1.1 kubectl=1.35.0-1.1
sudo apt-mark hold kubelet kubectl
sudo systemctl daemon-reload && sudo systemctl restart kubelet

# 컨트롤 플레인에서 복귀
kubectl uncordon worker-1
자주 틀리는 포인트

컨트롤 플레인은 kubeadm upgrade apply, 워커는 kubeadm upgrade node 입니다. 또 drain 시 DaemonSet 때문에 막히면 --ignore-daemonsets, emptyDir 데이터 때문이면 --delete-emptydir-data를 붙이세요.

2.1Deployment — 무중단 배포의 핵심

Pod를 직접 만들면 죽었을 때 끝입니다. 그래서 거의 항상 Deployment로 감쌉니다. Deployment는 ReplicaSet을 만들고, ReplicaSet이 Pod 개수를 유지합니다. 업데이트하면 새 ReplicaSet으로 점진 교체(rolling update)하고, 문제가 생기면 롤백할 수 있습니다.

🍞
빵집 진열대 같습니다. "항상 빵 5개 진열"이 목표(replicas=5)면, 하나가 팔리면(Pod 사망) 직원이 새로 채웁니다. 레시피를 바꿀 땐(이미지 업데이트) 진열대를 통째로 비우지 않고 한두 개씩 신상으로 교체해 손님이 끊기지 않게 하죠. 이게 롤링 업데이트입니다.
# Deployment 생성 (명령형 — 시험에서 가장 빠름)
kubectl create deployment web --image=nginx:1.27 --replicas=3

# 스케일 조정
kubectl scale deployment web --replicas=5

# 이미지 업데이트 → 롤링 업데이트 트리거
kubectl set image deployment/web nginx=nginx:1.28

# 자원 요청/제한 추가
kubectl set resources deployment web \
  --requests=cpu=100m,memory=128Mi --limits=cpu=250m,memory=256Mi
# 롤아웃 진행 상황 보기
kubectl rollout status deployment/web

# 변경 이력 (--record 없어도 revision 추적)
kubectl rollout history deployment/web

# 직전 버전으로 롤백
kubectl rollout undo deployment/web

# 특정 revision으로 롤백
kubectl rollout undo deployment/web --to-revision=2
워크로드언제 쓰나
Deployment무상태(stateless) 앱. 대부분의 웹/API 서버
StatefulSet고정 신원·순서가 필요한 상태유지 앱 (DB, Kafka). Pod 이름이 app-0,app-1로 안정적
DaemonSet모든 노드에 하나씩 떠야 하는 것 (로그 수집기, 모니터링 에이전트)
Job / CronJob한 번 실행하고 끝나는 작업 / 주기적 작업

2.2ConfigMap & Secret — 설정 분리

설정값을 이미지에 박아넣으면 환경마다 다시 빌드해야 합니다. ConfigMap(일반 설정)과 Secret(민감정보)으로 분리해 두고 Pod에 주입하세요. Secret은 base64로 인코딩될 뿐 암호화는 아니라는 점이 시험 포인트입니다.

# ConfigMap — 키-값으로 직접
kubectl create configmap app-config \
  --from-literal=APP_MODE=prod --from-literal=LOG_LEVEL=info

# ConfigMap — 파일에서
kubectl create configmap nginx-conf --from-file=./nginx.conf

# Secret — 자동 base64 인코딩
kubectl create secret generic db-cred \
  --from-literal=username=admin --from-literal=password=s3cr3t
spec:
  containers:
  - name: app
    image: myapp:1.0
    envFrom:                    # ConfigMap 전체를 환경변수로
    - configMapRef:
        name: app-config
    env:                        # Secret의 특정 키만
    - name: DB_PASSWORD
      valueFrom:
        secretKeyRef:
          name: db-cred
          key: password
    volumeMounts:                # 파일로 마운트도 가능
    - name: conf
      mountPath: /etc/nginx/conf.d
  volumes:
  - name: conf
    configMap: {name: nginx-conf}

2.3스케줄링 제어 — Pod를 어디에 올릴까

기본적으로 스케줄러가 알아서 배치하지만, 특정 노드에만 올리거나 피하게 만들 수 있습니다. 비슷해 보이는 개념이 많아 헷갈리니 "끌어당기는 것 vs 밀어내는 것"으로 구분하세요.

기법방향설명
nodeSelector끌어당김라벨이 맞는 노드에만 (가장 단순)
Node Affinity끌어당김nodeSelector의 강화판 (required/preferred, 연산자)
Taint / Toleration밀어냄노드가 Pod를 거부. toleration 있는 Pod만 허용
Pod Affinity/Anti-affinityPod끼리특정 Pod 곁에 모으기 / 떨어뜨리기
🎟️
Taint는 노드에 붙인 "출입 금지" 팻말이고, Toleration은 그 팻말을 무시할 수 있는 특별 출입증입니다. 컨트롤 플레인 노드엔 보통 taint가 걸려 일반 Pod가 안 올라가는데, 시스템 Pod는 toleration을 가지고 있어 올라갈 수 있죠.
# 노드에 taint 추가 (NoSchedule = 새 Pod 거부)
kubectl taint nodes worker-1 gpu=true:NoSchedule

# taint 제거 (끝에 - 붙임)
kubectl taint nodes worker-1 gpu=true:NoSchedule-

# 노드에 라벨 추가 (nodeSelector/affinity용)
kubectl label nodes worker-1 disktype=ssd

# cordon = 새 Pod 차단(기존 유지), drain = 비우기
kubectl cordon worker-1
spec:
  tolerations:                 # 위 taint를 견디는 출입증
  - key: "gpu"
    operator: "Equal"
    value: "true"
    effect: "NoSchedule"
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
        - matchExpressions:
          - key: disktype
            operator: In
            values: [ssd]
  nodeSelector:                 # 또는 단순하게
    disktype: ssd
cordon vs drain 헷갈리지 마세요

cordon = 새 Pod만 막음(기존 Pod 유지). drain = 막고 + 기존 Pod까지 다른 노드로 쫓아냄(노드 점검·업그레이드 전 필수). uncordon = 다시 스케줄 가능하게 복귀.

3.1핵심 3종: PV · PVC · StorageClass

🏠
부동산에 비유하면, PV(PersistentVolume)는 실제 매물(창고)이고, PVC(PersistentVolumeClaim)는 "방 20평, 화장실 딸린 곳 구함"이라는 세입자의 요청서입니다. 둘이 조건이 맞으면 바인딩(계약)되죠. StorageClass는 "요청이 오면 자동으로 창고를 지어주는 건설사"입니다(동적 프로비저닝).
오브젝트누가 만드나역할
PersistentVolume (PV)관리자 (또는 자동)실제 스토리지 자원. 클러스터 범위
PersistentVolumeClaim (PVC)사용자/개발자"이만큼 필요해"라는 요청. 네임스페이스 범위
StorageClass관리자PVC 요청 시 PV를 자동 생성하는 템플릿

3.2Access Mode & Reclaim Policy — 암기 필수

Access Mode약어의미
ReadWriteOnceRWO한 노드에서 읽기/쓰기 (가장 흔함)
ReadOnlyManyROX여러 노드에서 읽기 전용
ReadWriteManyRWX여러 노드에서 동시 읽기/쓰기 (NFS 등)
ReadWriteOncePodRWOP단 하나의 Pod만 읽기/쓰기
Reclaim PolicyPVC 삭제 후
RetainPV·데이터 보존 (수동 정리 필요) — 중요 데이터에
DeletePV와 실제 스토리지까지 삭제 (동적 프로비저닝 기본값)
apiVersion: v1
kind: PersistentVolume
metadata: {name: pv-data}
spec:
  capacity: {storage: 5Gi}
  accessModes: [ReadWriteOnce]
  persistentVolumeReclaimPolicy: Retain
  hostPath: {path: /mnt/data}   # 실습용(단일노드)
---
apiVersion: v1
kind: PersistentVolumeClaim
metadata: {name: pvc-data}
spec:
  accessModes: [ReadWriteOnce]
  resources: {requests: {storage: 5Gi}}
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata: {name: fast}
provisioner: kubernetes.io/aws-ebs  # 환경별 프로비저너
parameters: {type: gp3}
reclaimPolicy: Delete
volumeBindingMode: WaitForFirstConsumer
---
# PVC가 storageClassName만 지정하면 PV가 자동 생성됨
kind: PersistentVolumeClaim
spec:
  storageClassName: fast
  accessModes: [ReadWriteOnce]
  resources: {requests: {storage: 10Gi}}
spec:
  containers:
  - name: app
    image: nginx
    volumeMounts:
    - name: storage
      mountPath: /usr/share/nginx/html
  volumes:
  - name: storage
    persistentVolumeClaim:
      claimName: pvc-data     # 위에서 만든 PVC 연결
PVC가 Pending에서 안 넘어갈 때

① 맞는 PV가 없음(용량/accessMode 불일치), ② StorageClass 이름 오타, ③ WaitForFirstConsumer라서 Pod가 떠야 바인딩됨. kubectl describe pvc <name> 의 Events를 먼저 보세요 — 원인이 그대로 적혀 있습니다.

4.1Service — 사라지는 Pod에 고정 주소를

Pod는 죽고 다시 뜰 때마다 IP가 바뀝니다. 그래서 Service가 라벨로 Pod 묶음을 가리키는 고정된 가상 IP + DNS 이름을 제공합니다. 트래픽은 Service가 살아있는 Pod들에게 부하 분산합니다.

📞
Service는 대표번호입니다. 상담원(Pod)이 출근/퇴근하며 바뀌어도, 고객은 항상 같은 대표번호로 전화하면 비어있는 상담원에게 연결됩니다. 상담원 개인 휴대폰(Pod IP)으로 직접 걸면 그 사람이 퇴사하는 순간 끊기죠.
Service 타입접근 범위쓰임새
ClusterIP클러스터 내부만기본값. 내부 서비스 간 통신
NodePort노드IP:포트(30000~32767)외부에서 노드 포트로 직접 접근
LoadBalancer클라우드 LB 통해 외부퍼블릭 클라우드 외부 노출
ExternalNameDNS CNAME외부 서비스를 내부 이름으로 매핑
# Deployment를 ClusterIP Service로 노출
kubectl expose deployment web --port=80 --target-port=80

# NodePort로
kubectl expose deployment web --port=80 --type=NodePort

# 연결 확인 — Endpoints가 비어있으면 selector 불일치!
kubectl get svc,endpoints web
NAME          TYPE        CLUSTER-IP     PORT(S)
service/web   ClusterIP   10.96.140.2    80/TCP
NAME            ENDPOINTS
endpoints/web   10.244.1.3:80,10.244.2.4:80
apiVersion: v1
kind: Service
metadata: {name: web}
spec:
  type: ClusterIP
  selector:
    app: web            # Pod 라벨과 정확히 일치해야 함!
  ports:
  - port: 80            # Service 포트
    targetPort: 80      # 컨테이너 포트
트러블슈팅 1순위: Endpoints

"Service에 접속이 안 돼요" 문제의 90%는 selector가 Pod 라벨과 안 맞아서 Endpoints가 비어있는 경우입니다. kubectl get endpoints <svc> 가 비었으면 라벨을 의심하세요.

4.2Ingress & Gateway API — L7 라우팅

Service만으로는 도메인/경로 기반 라우팅(/api→A, /web→B)이 어렵습니다. Ingress가 그 역할을 하고, 동작하려면 Ingress Controller(nginx 등)가 설치돼 있어야 합니다. 2025년부터는 더 유연한 Gateway API가 GA가 되어 시험 범위에 들어왔습니다.

IngressGateway API
리소스Ingress 하나GatewayClass → Gateway → HTTPRoute (역할 분리)
유연성annotation 의존, 제한적표준화된 트래픽 분할·헤더 매칭 등 풍부
위치networking.k8s.iogateway.networking.k8s.io (CRD)
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: app-ingress
spec:
  ingressClassName: nginx
  rules:
  - host: app.example.com
    http:
      paths:
      - path: /api
        pathType: Prefix
        backend:
          service:
            name: api-svc
            port: {number: 80}
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata: {name: app-route}
spec:
  parentRefs:
  - name: my-gateway     # 어느 Gateway에 붙을지
  hostnames: ["app.example.com"]
  rules:
  - matches:
    - path: {type: PathPrefix, value: /api}
    backendRefs:
    - name: api-svc
      port: 80

4.3NetworkPolicy — Pod 방화벽

기본적으로 모든 Pod는 서로 통신할 수 있습니다(allow-all). NetworkPolicy로 "누가 누구에게 접속 가능한지"를 제한합니다. 단, 이를 강제하려면 NetworkPolicy를 지원하는 CNI(Calico, Cilium 등)가 필요합니다.

화이트리스트 방식 이해가 핵심

특정 Pod에 NetworkPolicy가 하나라도 적용되면, 그 Pod는 정책에 명시된 트래픽만 허용하고 나머지는 모두 차단됩니다. podSelector: {}(빈 셀렉터)는 "네임스페이스의 모든 Pod"를 의미하니 주의하세요.

# db Pod로는 app=frontend 라벨을 가진 Pod의 3306 포트만 허용
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata: {name: db-allow-frontend}
spec:
  podSelector:
    matchLabels: {app: db}    # 이 정책이 적용될 대상
  policyTypes: [Ingress]
  ingress:
  - from:
    - podSelector:
        matchLabels: {app: frontend}  # 허용 출처
    ports:
    - {protocol: TCP, port: 3306}

같은 NS: web 다른 NS: web.dev 전체 FQDN: web.dev.svc.cluster.local

5.2Pod 상태별 원인 사전

STATUS의미먼저 확인할 것
Pending스케줄 안 됨자원 부족? taint? PVC Pending? → describe Events
ImagePullBackOff이미지 못 가져옴이미지 이름 오타, 태그 없음, 레지스트리 인증(imagePullSecret)

5.3노드 NotReady 진단

노드가 NotReady면 그 노드의 모든 Pod가 위태롭습니다. 원인은 거의 항상 kubelet 또는 네트워크(CNI)입니다.

kubectl describe node <node>        # Conditions: MemoryPressure/DiskPressure?

# 해당 노드에 SSH로 들어가서 kubelet부터
sudo systemctl status kubelet       # active 인가?
sudo journalctl -u kubelet -f        # 실시간 로그로 에러 추적
sudo systemctl restart kubelet      # 죽었으면 재시작

# 설정/인증서/디스크도 의심
cat /var/lib/kubelet/config.yaml
df -h                                # 디스크 가득 차면 NotReady
crictl ps                            # 컨테이너 런타임 상태

5.4컨트롤 플레인 컴포넌트 장애

API 서버, scheduler, controller-manager, etcd는 정적 Pod(static pod)로 떠 있습니다. kubectl이 안 먹히면 crictl로 직접 봐야 합니다.

# manifest의 오타(YAML 들여쓰기, 잘못된 flag)가 가장 흔한 원인
ls /etc/kubernetes/manifests/
# kube-apiserver.yaml  kube-scheduler.yaml  ...  etcd.yaml

# API 서버가 죽어 kubectl이 안 되면 crictl로 직접
sudo crictl ps -a                     # 죽은 컨테이너 포함 조회
sudo crictl logs <container-id>      # 해당 컴포넌트 로그

# manifest를 수정하면 kubelet이 자동으로 Pod를 재생성
sudo vim /etc/kubernetes/manifests/kube-apiserver.yaml
static pod는 kubectl로 못 지웁니다

정적 Pod는 manifest 파일이 곧 정의입니다. kubectl delete해도 kubelet이 즉시 되살립니다. 멈추려면 manifest 파일을 디렉터리 밖으로 옮기세요(mv etcd.yaml /tmp/).

Q2 · 워크로드
Deployment를 이전 버전으로 되돌리는 명령은?
정답: B. kubectl rollout undo deploy/web로 직전 리비전으로 롤백합니다. --to-revision=N으로 특정 버전 지정도 가능합니다.
Q3 · 스케줄링
노드를 점검하려고 합니다. 기존 Pod를 안전하게 다른 노드로 옮기며 비우는 명령은?
정답: B. drain은 노드를 스케줄 불가로 표시(cordon)하고 + 기존 Pod를 퇴거(evict)합니다. cordon은 새 Pod만 막고 기존 건 그대로 둡니다.
Q4 · 스토리지
여러 노드의 Pod가 동시에 읽고 쓰는 볼륨이 필요합니다. 필요한 accessMode는?
정답: C. RWX는 여러 노드에서 동시 읽기·쓰기를 허용합니다(NFS·CephFS 등). RWO는 한 노드, RWOP는 단일 Pod로 제한됩니다.
Q5 · 네트워킹
Service에 접속이 안 됩니다. Endpoints가 비어 있다면 1순위로 의심할 것은?
정답: A. Endpoints가 비었다는 건 Service의 selector가 어떤 Pod의 label과도 안 맞는다는 뜻입니다. 라벨/셀렉터부터 맞춰보세요.
Q6 · 네트워킹
같은 네임스페이스의 Service web에 접근하는 클러스터 내부 FQDN은?
정답: B. Service DNS는 <svc>.<namespace>.svc.cluster.local 형식입니다. 같은 ns 안에서는 그냥 web만으로도 됩니다.
Q7 · 트러블슈팅
노드가 NotReady입니다. 그 노드에 ssh로 들어가 가장 먼저 확인할 것은?
정답: A. 노드 NotReady의 1순위 용의자는 그 노드의 kubelet입니다. 서비스 상태와 journalctl -u kubelet으로 원인을 좁힙니다.
Q8 · 트러블슈팅
컨트롤 플레인 컴포넌트(static pod)의 매니페스트가 있는 경로는?
정답: B. kubeadm 클러스터의 apiserver·etcd·scheduler·controller-manager는 /etc/kubernetes/manifests/의 static pod YAML로 떠 있습니다. 여기 파일을 고치면 kubelet이 자동 반영합니다.
Q9 · 권한
내가 default ns에서 pod를 만들 수 있는지 확인하는 명령은?
정답: B. kubectl auth can-i create pods -n default로 권한을 즉시 확인합니다. 다른 사용자 기준으로 보려면 --as=jane을 붙입니다.
Q10 · 전략
시험 중 한 문제에서 5분째 막혔습니다. 가장 현명한 행동은?
정답: A. 부분 점수가 있고 배점이 다른 문제가 남아있습니다. 막히면 북마크하고 넘어간 뒤, 시간이 남으면 돌아오는 게 합격 확률을 높입니다. (구글 검색은 부정행위!)
