Identity & Access Management

로그인은 왜 이렇게
어려울까?

"아이디랑 비밀번호만 확인하면 되는 거 아니야?" — 실제로 로그인 기능을 직접 만들어 본 개발자라면 이 말이 함정이라는 걸 안다. 이 강의는 그 함정을 Keycloak이라는 도구로 어떻게 벗어나는지, 웹 개발을 막 시작한 사람도 이해할 수 있도록 처음부터 설명한다.

🔑 인증/인가 기초 📜 OAuth 2.0 · OIDC 🏛 Realm · Client · Role 🐳 Docker 실습 ⚖️ Auth0 · Cognito 비교

먼저 "로그인 기능을 직접 만들면 무엇을 해야 하는지" 목록을 보자. 겁주려는 게 아니라, Keycloak이 왜 존재하는지를 이해하기 위한 준비 운동이다.

직접 만들면 해야 하는 일들

사용자가 있는 모든 서비스는 결국 세 가지 질문에 답해야 한다. 너는 누구인가(인증), 너는 무엇을 할 수 있는가(인가), 그리고 어떻게 그 상태를 계속 기억할 것인가(세션/토큰). 이걸 코드로 옮기면 아래처럼 끝없이 늘어난다.

  • 비밀번호를 안전하게 저장 (평문 저장은 사고, 반드시 해시 + 솔트)
  • 회원가입 · 이메일 인증 · 비밀번호 재설정 · 계정 잠금
  • 로그인 세션 유지, 자동 로그아웃, "다른 기기에서 로그아웃"
  • 소셜 로그인 (구글/카카오/네이버) 각각의 OAuth 연동
  • 2단계 인증(OTP), 비밀번호 정책, 무차별 대입 공격 방어
  • 여러 서비스를 만들면 → 한 번 로그인으로 전부 접근(SSO)
  • 보안 취약점(CVE)이 터질 때마다 직접 패치
⚠️ 핵심 문제

이 목록의 어느 하나라도 잘못 구현하면 사용자 전체의 계정이 위험해진다. 인증은 "기능"이 아니라 "보안 인프라"다. 그래서 대부분의 회사는 이걸 직접 만들지 않고, 검증된 전문 도구에 맡긴다. Keycloak이 바로 그런 도구다.

🏢
건물 로비 비유 — 사무실마다 각자 경비원을 두는 대신, 건물 1층에 중앙 안내 데스크를 하나 두는 것과 같다. 방문자는 데스크에서 신분증을 한 번 확인받고 출입증(토큰)을 받는다. 이후 모든 사무실은 "우리가 확인할게" 하지 않고 출입증만 보면 된다. Keycloak은 바로 그 중앙 안내 데스크다.
Part 1 · 기초 개념

인증 vs 인가, 그리고 SSO

Keycloak을 배우기 전에 딱 네 개의 용어만 확실히 잡고 가자. 이 네 개를 헷갈리면 공식 문서가 외계어처럼 보인다.

① 인증(Authentication) — "너 누구야?"

신원을 증명하는 과정이다. 아이디/비밀번호, 지문, OTP 등으로 "이 사람이 진짜 그 사람이 맞는지" 확인한다. 영어로 AuthN이라고 줄여 쓴다.

② 인가(Authorization) — "너 이거 해도 돼?"

신원이 확인된 사용자가 무엇을 할 수 있는지 결정하는 과정이다. 예를 들어 로그인은 됐지만(인증 OK), 관리자 페이지는 못 들어간다(인가 실패). 영어로 AuthZ.

직접 해보기 · 인증과 인가는 다르다

아래에서 신분을 확인(인증)하고, 문에 접근을 시도(인가)해 보세요.

③ SSO(Single Sign-On) — "한 번만 로그인"

여러 개의 서비스를 쓰는데 로그인은 한 번만 하면 되는 것. 구글에 로그인하면 유튜브·지메일·드라이브가 전부 로그인되어 있는 경험이 바로 SSO다. 회사에서 여러 사내 시스템을 쓸 때 특히 중요하다.

④ IAM(Identity & Access Management)

위의 인증·인가·SSO·사용자 관리를 통합해서 다루는 시스템 전체를 IAM이라 부른다. Keycloak의 정식 분류가 바로 "오픈소스 IAM 솔루션"이다.

용어질문예시영어
인증너 누구야?비밀번호·OTP로 본인 확인Authentication (AuthN)
인가이거 해도 돼?관리자만 삭제 버튼 사용Authorization (AuthZ)
SSO또 로그인해?한 번 로그인 → 모든 앱 접근Single Sign-On
IAM이걸 뭐가 관리해?위 전부를 다루는 시스템Identity & Access Mgmt
📌 기억할 한 문장

인증은 신분증 검사, 인가는 출입 권한 검사. 인증을 통과했다고 아무 데나 들어갈 수 있는 건 아니다. Keycloak은 이 둘을 모두 처리한다.

Part 1 · 기초 개념

OAuth 2.0 · OIDC · SAML

Keycloak 문서에는 이 세 단어가 계속 나온다. 셋 다 "인증·인가를 위한 표준 규칙(프로토콜)"인데, 역할이 조금씩 다르다. 초보자는 OAuth 2.0과 OIDC 두 개만 확실히 알면 충분하다.

OAuth 2.0 — "권한을 빌려주는" 표준

OAuth 2.0은 원래 인가(권한 위임)를 위한 표준이다. 핵심 아이디어: 비밀번호를 넘기지 않고도 다른 앱에게 "내 데이터에 접근할 권한"만 안전하게 빌려주는 것.

🚗
발렛 키 비유 — 호텔에 차를 맡길 때 발렛 전용 키를 준다. 이 키는 시동은 걸리지만 트렁크는 안 열리고 최고 속도도 제한된다. 내 진짜 키(비밀번호)는 절대 넘기지 않는다. OAuth의 액세스 토큰이 바로 이 발렛 키다 — 딱 필요한 권한만, 정해진 시간 동안.

OIDC (OpenID Connect) — OAuth 위에 얹은 "로그인"

OAuth 2.0은 "권한 빌려주기"는 잘하지만 "이 사람이 누구인지"를 알려주는 표준이 아니었다. 그래서 OAuth 2.0 위에 신원 정보(로그인) 계층을 얹은 것이 OpenID Connect(OIDC)다. "구글로 로그인" 같은 게 전부 OIDC다.

  • OAuth 2.0 → "이 앱이 내 캘린더를 읽어도 됩니다" (권한)
  • OIDC → "이 사람은 hong@example.com 이고 이메일 인증됨" (신원)
💡 초보자 결론

웹 앱에 로그인을 붙일 거라면 OIDC를 쓰면 된다. Keycloak은 OIDC를 기본으로 지원하고, 대부분의 라이브러리도 OIDC 기준으로 만들어져 있다.

SAML — 기업/기관에서 쓰는 오래된 표준

SAML은 OIDC보다 먼저 나온 XML 기반의 SSO 표준이다. 주로 대기업·대학·정부 기관의 사내 시스템에서 아직 널리 쓰인다. 새 프로젝트에서는 보통 OIDC를 쓰지만, 기존 기업 시스템과 연동해야 할 때 SAML이 필요할 수 있다. Keycloak은 SAML도 지원한다.

프로토콜주 역할형식언제 쓰나
OAuth 2.0권한 위임(인가)JSON/토큰API 접근 권한 부여
OIDC로그인(신원 + 인가)JSON/JWT현대 웹 앱 기본
SAML기업용 SSOXML레거시·엔터프라이즈 연동
Part 2 · Keycloak

Keycloak란 무엇인가

Keycloak은 "우리 서비스의 인증·인가를 통째로 대신 처리해 주는, 직접 설치해서 쓰는 오픈소스 로그인 서버"다. 이 한 문장을 조각내서 이해해 보자.

키워드의미
오픈소스무료. 소스가 공개되어 있고 원하는 대로 커스터마이징 가능
직접 설치내 서버·회사 서버·클라우드에 내가 올려서 운영 (self-hosted). 데이터가 우리 손에 있음
로그인 서버회원가입·로그인·소셜로그인·2FA·SSO·권한관리를 전부 처리
표준 지원OIDC · OAuth 2.0 · SAML 을 모두 기본 제공

어디서 왔나

Keycloak은 원래 Red Hat(현재 IBM 소속)이 2014년에 처음 공개했고, 지금은 클라우드 네이티브 재단인 CNCF의 인큐베이팅 프로젝트다(2023년 4월 합류). 쿠버네티스·Prometheus 같은 프로젝트들과 같은 재단 소속이라는 뜻이고, 그만큼 널리 검증된 오픈소스 IAM 표준으로 자리잡았다. 이 강의는 26.7.0(2026년 7월) 버전을 기준으로 한다.

📌 self-hosted가 왜 중요한가

Auth0·Firebase 같은 서비스는 "남의 서버(SaaS)"에 사용자 데이터를 맡긴다. 반면 Keycloak은 우리가 직접 운영하므로, 사용자 정보가 외부로 나가면 안 되는 금융·의료·공공·폐쇄망 환경에서 특히 선호된다. (뒤의 비교 섹션에서 자세히 다룬다.)

Keycloak이 대신 해주는 일

  • 로그인 화면 · 회원가입 · 비밀번호 재설정 화면 기본 제공 (테마 커스터마이징 가능)
  • 구글·깃허브·카카오 등 소셜 로그인 연동을 클릭 몇 번으로
  • 2단계 인증(OTP·WebAuthn/패스키) 내장
  • 여러 앱을 하나의 로그인으로 묶는 SSO
  • 사용자·역할·그룹 관리용 관리자 콘솔(웹 UI)
  • 기존 회사 LDAP/Active Directory 사용자 그대로 연동

Part 2 · Keycloak

로그인은 어떻게 동작하나

"로그인 버튼을 누르면 실제로 무슨 일이 벌어질까?" 현대 웹 앱의 표준인 Authorization Code Flow를 한 단계씩 애니메이션으로 따라가 보자. 이걸 이해하면 Keycloak 연동의 90%가 끝난다.

단계별 애니메이션 · 로그인 여정

등장하는 3개의 토큰

로그인이 끝나면 앱은 세 종류의 "토큰"을 받는다. 토큰은 출입증이라고 생각하면 된다.

토큰역할비유수명
Access TokenAPI 호출 시 "나 권한 있어" 증명출입증짧음(분)
ID Token"이 사람 누구"라는 신원 정보 (OIDC)신분증짧음
Refresh Token만료된 Access Token 재발급용출입증 갱신권김(일/주)
🤔 왜 이렇게 복잡하게?

Access Token을 일부러 짧게(예: 5분) 만든다. 혹시 토큰이 탈취돼도 금방 만료되기 때문이다. 대신 Refresh Token으로 조용히 새로 발급받아 사용자는 다시 로그인할 필요가 없다. 보안과 편의성을 동시에 잡는 설계다.

🔐 PKCE — 요즘은 필수

브라우저·모바일 앱처럼 비밀을 숨길 수 없는 환경에서는 PKCE(픽시)라는 추가 보안 장치를 함께 쓴다. 중간에 인가 코드를 가로채도 악용하지 못하게 막는 잠금장치다. Keycloak 최신 라이브러리는 기본으로 켜준다.

Part 3 · 실습

5분 만에 띄우기

이론은 충분하다. 이제 실제로 내 컴퓨터에서 Keycloak을 켜보자. Docker만 설치돼 있으면 명령어 한 줄로 끝난다. (Docker가 없다면 Docker Desktop을 먼저 설치)

1단계 · Keycloak 실행

터미널에 아래 명령어를 붙여넣자. 개발용 모드(start-dev)로, 관리자 계정을 admin / admin으로 만들어 8080 포트에 띄운다.

# Keycloak 26.7 을 개발 모드로 실행
docker run -p 8080:8080 \
  -e KC_BOOTSTRAP_ADMIN_USERNAME=admin \
  -e KC_BOOTSTRAP_ADMIN_PASSWORD=admin \
  quay.io/keycloak/keycloak:26.7.0 \
  start-dev
# docker-compose.yml — docker compose up 으로 실행
services:
  keycloak:
    image: quay.io/keycloak/keycloak:26.7.0
    command: start-dev
    environment:
      KC_BOOTSTRAP_ADMIN_USERNAME: admin
      KC_BOOTSTRAP_ADMIN_PASSWORD: admin
    ports:
      - "8080:8080"
실행 후 브라우저에서 http://localhost:8080 접속 → admin / admin 로 로그인
⚠️ start-dev 는 개발 전용

start-dev는 HTTPS를 끄고 임시 메모리 DB를 쓰는 연습용 모드다. 실제 서비스(프로덕션)에서는 절대 이렇게 쓰면 안 되고, PostgreSQL 같은 외부 DB + HTTPS + start 모드로 운영해야 한다. (실무 체크리스트 참고)

2단계 · 관리자 콘솔에서 할 일 (클릭만)

브라우저로 접속하면 웹 UI가 나온다. 코드 없이 마우스 클릭만으로 아래를 순서대로 만든다.

  1. Realm 생성 — 좌측 상단 드롭다운 → "Create realm" → 이름 myapp
  2. Client 등록 — Clients → Create → Client ID my-frontend, 타입은 OpenID Connect
  3. Redirect URI 설정 — 우리 앱 주소 등록 (예: http://localhost:3000/*)
  4. User 생성 — Users → Add user → 사용자 만들고 Credentials 탭에서 비밀번호 설정
  5. Role 생성 & 할당 — Realm roles → admin 만들고, 유저에게 부여

여기까지 하면 서버 쪽 준비는 끝이다. 이제 우리 웹 앱에서 이 Keycloak을 향해 "로그인 좀 처리해줘"라고 부르기만 하면 된다.

Part 3 · 실습

웹 앱에 붙이기

직접 로그인 로직을 짜지 않는다. Keycloak이 주는 표준 라이브러리를 쓴다. 프론트엔드(React)와 백엔드(Node/Spring) 예시를 각각 보자. 탭을 눌러 비교해 보세요.

프론트엔드 — "로그인 시키고 토큰 받기"

import Keycloak from 'keycloak-js';

// 어느 Keycloak의 어느 Realm·Client에 연결할지 지정
const keycloak = new Keycloak({
  url: 'http://localhost:8080',
  realm: 'myapp',
  clientId: 'my-frontend',
});

// 앱 시작 시 로그인 상태 확인 (PKCE 자동 적용)
await keycloak.init({

백엔드 — "받은 토큰이 진짜인지 검증하기"

프론트가 API를 부를 때 Access Token을 같이 보낸다. 백엔드는 이 토큰이 Keycloak이 발급한 진짜인지, 만료되지 않았는지, 어떤 권한인지를 검증한다. 여기서도 직접 검증 로직을 짜기보다 검증된 미들웨어/라이브러리를 쓴다.

import { expressjwt } from 'express-jwt';
import jwksRsa from 'jwks-rsa';

// Keycloak의 공개키로 토큰 서명을 검증하는 미들웨어
const checkToken = expressjwt({
  secret: jwksRsa.expressJwtSecret({
    jwksUri: 'http://localhost:8080/realms/myapp' +
             '/protocol/openid-connect/certs',
  }),
  algorithms: ['RS256'],
});

// 이 API는 유효한 토큰이 있어야만 접근 가능
app.get('/api/profile', checkToken, (req, res) => {
  res.json({ user: req.auth.preferred_username });
});
// application.yml — Keycloak을 인증 서버로 지정
spring:
  security:
    oauth2:
      resourceserver:
        jwt:
          issuer-uri: http://localhost:8080/realms/myapp

// SecurityConfig.java
http
  .authorizeHttpRequests(a -> a
    .requestMatchers("/api/admin/**").hasRole("admin")
    .anyRequest().authenticated())
  .oauth2ResourceServer(o -> o.jwt(Customizer.withDefaults()));
issuer-uri / jwksUri 만 알려주면, 나머지 서명 검증은 라이브러리가 자동으로 처리한다.
💡 전체 그림

① 프론트가 Keycloak으로 로그인 → ② 토큰 받음 → ③ 그 토큰을 백엔드 API에 첨부 → ④ 백엔드가 Keycloak 공개키로 검증. 우리가 짠 코드는 "어디에 연결할지"와 "어떤 권한이 필요한지"뿐이다.

Keycloak이 주는 토큰(JWT)은 사실 누구나 열어볼 수 있는 정보다. 암호화가 아니라 서명이기 때문이다. 아래에서 직접 디코딩해 보며 안에 뭐가 들었는지 확인하자.

JWT 디코더 · 토큰을 눌러 열어보기
🚨 매우 중요한 오해 바로잡기

JWT는 암호화가 아니다. 누구나 내용을 읽을 수 있다(Base64일 뿐). 그래서 토큰 안에 비밀번호·주민번호 같은 민감정보를 넣으면 안 된다. JWT가 지키는 건 "내용 비밀"이 아니라 "위조 방지" — 서명 덕분에 내용을 바꾸면 즉시 들통난다는 점이다.

Part 4 · 선택과 운영

다른 인증 도구와 비교

Keycloak만 있는 게 아니다. 인증 도구를 고를 때 가장 중요한 갈림길은 "내가 서버를 직접 운영할 것인가(self-hosted), 아니면 남에게 맡길 것인가(SaaS)"다. 대표 주자들을 비교해 보자.

우선순위 선택 · 나에게 맞는 도구는?

주요 도구 한눈에 비교

도구방식비용데이터 소유대표 강점
Keycloak Self-hosted 무료(오픈소스) 우리 서버 완전한 통제·커스터마이징·폐쇄망
Auth0 SaaS 사용량 과금 Okta 서버 가장 쉬운 시작·풍부한 문서
AWS Cognito SaaS MAU 과금 AWS AWS 생태계 연동
Firebase Auth SaaS 넉넉한 무료 Google 모바일·프로토타입 초고속
Okta SaaS 기업용 고가 Okta 서버 대기업 SSO·규정준수
Supabase Auth 둘 다 가능 플랜별 선택 가능 DB와 통합·스타트업 친화

Keycloak을 골라야 할 때 / 피해야 할 때

✅ Keycloak이 정답일 때
  • 데이터를 외부에 못 보내는 폐쇄망/금융/의료/공공
  • 사용자가 많아 SaaS 과금이 부담될 때
  • 로그인 흐름을 깊게 커스터마이징해야 할 때
  • 사내에 운영 인력(DevOps)이 있을 때
  • 기존 LDAP/AD와 연동이 필요할 때
⚠️ 다른 걸 고려할 때
  • 서버 운영이 부담인 1인/소규모 팀
  • 당장 프로토타입을 빠르게 띄우고 싶을 때 → Firebase
  • 이미 AWS에 올인 → Cognito
  • 운영·업데이트·보안패치를 맡기고 싶을 때
📌 결정적 차이 한 줄

Keycloak = 내가 운영하는 대신 무료 & 완전한 통제. Auth0/Cognito/Firebase = 편한 대신 유료 & 데이터를 남의 서버에 맡김. 정답은 없다. "데이터를 어디에 둬야 하는가"와 "운영 인력이 있는가" 두 질문이 대부분의 선택을 결정한다.

실무 체크리스트

연습(start-dev)과 실제 서비스는 완전히 다르다. 프로덕션에 올리기 전 반드시 확인해야 할 항목들을 체크박스로 정리했다. 눌러서 진행 상황을 확인하세요.

프로덕션 체크리스트 · 항목을 눌러 완료 표시
🚨 초보자가 가장 많이 하는 실수
  • HTTPS 없이 운영 — 토큰이 평문으로 노출된다. 반드시 TLS.
  • admin/admin 그대로 — 기본 관리자 계정은 즉시 변경/삭제.
  • Access Token을 localStorage에 — XSS에 취약. 가급적 메모리/HttpOnly 쿠키.
  • Redirect URI에 와일드카드 남발 — *는 공격 표면을 넓힌다. 정확히 지정.
  • 토큰 만료를 무한대로 — 짧게 두고 Refresh로 갱신하는 게 원칙.

한 장으로 정리

🎯 이 강의의 핵심 5가지
  • 인증(누구야?)과 인가(해도 돼?)는 다르다. Keycloak은 둘 다 처리한다.
  • 현대 웹 로그인의 표준은 OIDC(OAuth 2.0 위에 신원 계층).
  • Keycloak의 뼈대는 Realm → Client · User → Role · Group.
  • 로그인 결과로 Access · ID · Refresh 세 토큰을 받는다. JWT는 암호화가 아니라 서명이다.
  • self-hosted(무료·통제) vs SaaS(편함·유료) — 데이터 소유와 운영 인력이 선택의 핵심.
