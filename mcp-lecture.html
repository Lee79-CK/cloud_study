Prologue · 00

이 강의에서
다루는 것

ChatGPT나 Claude 같은 AI 비서에게 "내 캘린더 일정 확인해줘" 라고 부탁한 적이 있나요? 또는 "이 GitHub 저장소의 코드를 분석해줘" 라고 말했을 때, AI가 정말로 그 작업을 수행했을까요?

오래된 LLM은 "말은 잘하지만 손이 없는 천재" 와 같았습니다. 아는 것은 많지만, 외부 세계의 무엇도 만질 수 없었습니다. 파일을 읽을 수 없었고, 데이터베이스에 질의할 수 없었으며, 누구에게도 메시지를 보낼 수 없었습니다. 그저 학습된 지식 안에서만 대답했습니다.

Model Context Protocol(MCP) 은 이 한계를 깨뜨리기 위해 만들어진 약속(프로토콜)입니다. 이 강의에서는 다음을 배웁니다:

  1. 왜 우리는 새로운 표준이 필요했는가 — N×M 문제의 정체
  2. MCP가 무엇이고, 왜 "AI를 위한 USB-C" 라 불리는가
  3. 일반 LLM 사용 / Function Calling / MCP의 실제 차이
  4. MCP의 세 가지 핵심 구성 요소: Tools, Resources, Prompts
  5. 요청 한 번이 어떻게 흘러가는지 — 라이프사이클
  6. 현실에서 MCP가 쓰이는 곳들과, 직접 만드는 첫 서버
사전 지식
이 강의는 코딩 경험이 거의 없어도 따라올 수 있도록 만들었습니다. 마지막 실습 코드 예제만 Python에 익숙하면 더 좋지만, 개념 이해에는 필요하지 않습니다.
Chapter 01 · 문제

없는 것 빼고
모두 만들던 시대

MCP 이전, AI에게 도구를 연결한다는 것은 매번 새로운 어댑터를 손으로 깎는 일 이었습니다.

2024년, AI 비서가 외부 도구와 연결되는 방식은 혼란 그 자체였습니다. OpenAI의 ChatGPT에 Slack을 연결하려면 OpenAI 방식으로 코드를 짜야 했고, 같은 Slack을 Anthropic의 Claude에 연결하려면 Anthropic 방식으로 처음부터 다시 짜야 했습니다. Google Gemini도 마찬가지였습니다.

이것을 우리는 N×M 문제 라고 부릅니다. N개의 AI 모델과 M개의 도구가 있다면, 둘을 연결하는 어댑터는 무려 N × M개 가 필요했습니다.

Before MCP
통합의 카오스
N × M
AI 3개 × 도구 4개 = 어댑터 12개 필요.
모델이 늘거나 도구가 늘면 어댑터는 곱하기로 늘어남.
With MCP
통합의 질서
N + M
AI 3개 + 도구 4개 = 연결 7개 면 충분.
모델이 늘어도 도구는 그대로, 도구가 늘어도 모델은 그대로.

숫자가 작을 때는 차이가 별로 안 느껴지지만, 모델 10개 × 도구 100개 라면? 1,000개의 어댑터 가 됩니다. 모델이 새로 나오거나 API가 살짝 바뀌면 모두 깨집니다. 회사 전체 개발팀이 평생 어댑터만 짜야 할 판이죠.

"모든 AI 회사와 모든 데이터 소스 사이마다 새로운 어댑터를 만드는 것은 지속 가능하지 않다." — Anthropic, MCP 발표문, 2024년 11월

Anthropic은 이 문제를 해결하기 위해 MCP라는 공통 표준 을 제안했고, 이것을 누구나 쓸 수 있도록 오픈 소스로 공개 했습니다. 그리고 놀라운 일이 벌어졌습니다 — 2025년 3월 OpenAI가, 그 후 Google이 차례로 MCP를 채택했습니다. 경쟁사들이 한 표준 위에서 만나는 드문 풍경입니다.

Chapter 02 · 정의

그래서, MCP가
정확히 뭔가요?

한 줄로: LLM 애플리케이션이 외부 데이터·도구와 연결되는 방법을 표준화한 공개 프로토콜.

조금 더 풀어서 설명하면, MCP는 약속 입니다. "AI 클라이언트(Claude Desktop 같은)와 외부 서비스(예: GitHub)가 서로 대화할 때, 이런 형식으로 메시지를 주고받기로 하자" 는 약속입니다. 약속을 지키기만 하면, 누가 만들었든 호환됩니다.

가장 좋은 비유: AI를 위한 USB-C

예전에 노트북 충전기, 휴대폰 충전기, 카메라 충전기가 모두 다른 시절을 기억하나요? 여행 가방에 충전기만 가득했죠. USB-C가 등장한 후, 하나의 케이블로 거의 모든 기기를 충전하고 연결할 수 있게 되었습니다.

MCP는 AI 세계의 USB-C입니다. 한 번 만든 MCP 서버는 Claude Desktop에서도, ChatGPT에서도, Cursor IDE에서도, VS Code Copilot에서도 그대로 작동합니다. 새로운 AI가 나와도 MCP만 지원하면 자동으로 호환됩니다.

— MCP의 기술적 정체 —
기억할 단어들
JSON-RPC 2.0 : 함수 호출을 JSON 텍스트로 주고받는 표준 방식. "이 함수를 이 인자로 실행해줘 → 결과는 이거야" 를 정해진 형식대로 표현합니다.

stdio (Standard I/O) : 같은 컴퓨터 안에서 프로그램끼리 표준 입출력으로 대화. 로컬 도구에 적합.

HTTP + SSE (Server-Sent Events) : 인터넷 너머의 원격 서버와 대화. 클라우드/SaaS 도구에 적합.

한 줄 정리

MCP는 LLM과 외부 세계를 잇는 "공통 콘센트"입니다. 콘센트 모양만 맞으면, 어떤 전자제품도 꽂힙니다. AI도 마찬가지입니다.

  1. MCP는 공개 표준이며, Anthropic이 만들었지만 2025년 말 Linux Foundation 산하 AAIF에 기증되어 중립적으로 운영됩니다.
  2. 기술적으로는 JSON-RPC 2.0 위에서 동작하는 메시지 약속입니다.
  3. "USB-C for AI" — 한 번 만들면 모든 호환 클라이언트에서 작동.
  4. OpenAI, Google, Microsoft, Cloudflare 등 주요 AI 플레이어들이 모두 채택했습니다.
Chapter 03 · 비교

같은 질문,
세 가지 방식

같은 질문 "내일 서울 날씨 어때?" 를 세 가지 방식으로 처리해 봅시다. 차이가 한눈에 보일 겁니다.

아래 탭을 눌러 각 방식의 흐름을 비교해 보세요. 같은 질문이지만, 처리 방식과 결과의 신뢰도는 완전히 다릅니다.

사용자 : "내일 서울 날씨 어때?"
1
LLM은 학습 데이터 안에서만 답을 찾습니다. "내일"이 언제인지, 지금이 언제인지조차 모릅니다.
2
LLM이 답합니다: "서울은 보통 이맘때 영하 5도 정도이며..." 그럴듯하지만 실제 내일과는 무관한 일반론. 환각(hallucination) 가능성도 큼.
3
사용자는 답을 받지만, 믿을 수 없습니다. 실제 날씨 정보가 아닌, 통계적으로 그럴 법한 문장의 조합.
외부 연결 없음 실시간 정보 불가 행동 수행 불가능 신뢰도 낮음
사용자 : "내일 서울 날씨 어때?"
1
개발자가 미리 OpenAI 전용 형식으로 함수를 정의해 둠. get_weather(city, date) 를 OpenAI Functions 스펙대로 작성.
2
LLM이 판단: "이 질문에는 get_weather 함수를 부르자" {"city": "Seoul", "date": "2026-05-12"} 같은 인자 생성.
3
앱이 실제 날씨 API를 호출 → 결과를 LLM에 다시 전달. "기온 18도, 맑음" 같은 진짜 데이터가 LLM 손에 들어옴.
4
LLM이 자연어로 답: "내일 서울은 맑고 최고 18도입니다." 좋은 답변! 하지만... 이 코드는 OpenAI 모델에서만 작동.
외부 연결 가능 실시간 정보 가능 행동 수행 가능 재사용성 낮음 (벤더 종속)
사용자 : "내일 서울 날씨 어때?"
1
어딘가의 누군가가 weather-mcp 서버를 한 번 만들어 공개해 둠. 표준 MCP 명세대로 작성. 어떤 클라이언트와도 호환.
2
Claude Desktop이 시작될 때 weather-mcp 서버와 핸드셰이크. "넌 어떤 도구들이 있어?" → "get_weather(city, date) 있어요"
3
사용자 질문이 들어오면 LLM이 자동으로 get_weather 호출. 표준 형식이라 모델별 코드 작성 불필요.
4
결과 반환, LLM이 자연어로 답변: "내일 서울은 맑고 최고 18도입니다." + 사용자가 다음 날 Cursor IDE에서 같은 질문을 해도, 같은 weather-mcp 서버가 그대로 작동.
외부 연결 가능 실시간 정보 가능 행동 수행 가능 재사용성 매우 높음 (표준)
자주 헷갈리는 점
Function Calling과 MCP는 경쟁자가 아닙니다. Function Calling은 모델이 "이 함수를 부르고 싶다"고 의사를 표현하는 모델 API의 기능이고, MCP는 그 함수를 어디서 어떻게 가져올 것인가를 표준화한 위층의 프로토콜입니다. 실제로는 둘이 같이 씁니다. MCP가 도구 카탈로그를 제공하고, Function Calling이 그것을 호출하는 식이죠.

Chapter 04 · 아키텍처

누가 누구와
대화하는가

MCP는 세 종류의 등장인물이 협업합니다: Host, Client, Server. 각자의 역할을 알면, 전체 그림이 단번에 보입니다.

아래 다이어그램에서 각 노드를 클릭해 보세요. 그 역할이 무엇이고, 실제로 어떤 프로그램들이 그 역할을 하는지 보여드립니다.

▶ 노드를 클릭하세요
각 등장인물의 역할 살펴보기
위 다이어그램에서 Host, Client, Server 영역을 클릭하면 자세한 설명이 여기에 표시됩니다.
중요한 1:1 관계
Host 안에는 여러 개의 Client가 존재할 수 있고, 각 Client는 정확히 하나의 Server와 짝을 이룹니다. 비유하자면 Host는 멀티 콘센트 본체이고, 각 Client 슬롯이 각각의 외부 기기(Server) 와 연결되는 식입니다.

왜 이렇게 분리했을까?

각 등장인물의 책임을 명확히 나누었기 때문입니다:

  • Host는 사용자와 LLM을 책임집니다. 권한 동의, UI, 모델과의 대화는 모두 Host의 몫.
  • Client는 Server와의 1대1 통신을 책임집니다. 메시지 직렬화, 상태 추적.
  • Server는 도구의 실제 구현을 책임집니다. "GitHub Issue를 어떻게 가져올까"는 Server만 알면 됩니다.

이 분리 덕분에, Server를 만든 사람은 어떤 Host가 자기 Server를 쓸지 신경 쓸 필요가 없습니다. 약속만 지키면 끝입니다.

Chapter 05 · 구성요소

세 개의 원시 요소,
그게 전부입니다

MCP 서버가 제공할 수 있는 것은 Tools, Resources, Prompts 단 세 가지입니다. 단순하지만, 거의 모든 상호작용을 표현할 수 있습니다.

PRIMITIVE 01
Tools
실행 가능한 작업

LLM이 호출해 실제로 무언가를 일으키는 함수. 파일을 만들고, 메일을 보내고, DB 쿼리를 실행합니다. 부작용(side-effect)이 있는 모든 행동.

name: "send_email" desc: "이메일 발송" input: { to, subject, body }

예시: create_issue, run_query, send_slack_message, deploy_to_k8s

PRIMITIVE 02
Resources
읽기 전용 데이터

LLM이 참조하지만 변경하지 않는 콘텐츠. 파일, 문서, DB 스냅샷, 로그. URI로 식별되며, 부작용이 없습니다.

uri: "file:///docs/api.md" mime: "text/markdown" text: "# API 문서..."

예시: 프로젝트 README, 시스템 로그, 데이터베이스 스키마, 회의록

PRIMITIVE 03
Prompts
재사용 가능한 템플릿

자주 쓰는 작업 패턴을 이름과 인자가 있는 템플릿으로 미리 만들어 두는 것. 사용자가 슬래시 명령처럼 부를 수 있음.

name: "code_review" args: { language, file } template: "다음 {lang} 코드를 검토해줘: {file}"

예시: /summarize-pr, /explain-code, /write-tests, /generate-changelog

기억하는 법
셋의 차이를 "누가 통제하는가"로 외우면 쉽습니다.

• Tools — 모델이 통제. LLM이 알아서 부름.
• Resources — 애플리케이션이 통제. Host/사용자가 어떤 자료를 모델에 넣을지 결정.
• Prompts — 사용자가 통제. 사용자가 명시적으로 선택해서 실행.

실제 예: 같은 GitHub 서버가 셋 모두 제공

GitHub MCP 서버 하나가 어떻게 셋 모두를 노출하는지 보세요:

Tool
create_pull_request()

실제로 PR을 만듦. 부작용 발생.

Resource
github://repo/README.md

README 본문을 그저 읽기.

Prompt
/review-this-pr

"이 PR을 리뷰해줘" 템플릿.

Chapter 06 · 동작

요청 한 번,
네 번의 무대

"내 GitHub 이슈 보여줘" 한 마디가 들어왔을 때, 무대 뒤에서 어떤 일이 벌어지는지 따라가 봅시다.

아래 "다음" 버튼을 눌러 단계별로 진행하세요. 누가 누구에게 무엇을 보내는지, 실제 메시지와 함께 봅니다.

Step 0 / 5 — 시작 전
HOST
Claude Desktop

MCP Connector

GitHub MCP
▶ "다음 단계" 버튼을 눌러 시작하세요.
총 5단계로 진행됩니다: 초기화 → 능력 발견 → 사용자 입력 → 도구 호출 → 응답. 각 단계에서 어떤 메시지가 오가는지, 누가 활성화되는지 봅니다.
왜 핸드셰이크가 필요한가
클라이언트와 서버는 만나자마자 "넌 어떤 버전 MCP야? 어떤 기능 지원해?" 를 묻습니다. 이걸 capability negotiation(능력 협상) 이라 부릅니다. 양쪽이 공통으로 지원하는 기능만 쓰기로 합의해야 안전하게 통신할 수 있습니다.

Chapter 08 · 실습

30줄로 만드는
나의 첫 MCP 서버

개념을 가장 빨리 익히는 방법은 직접 만들어 보는 것. 날씨를 알려주는 단순한 MCP 서버를 Python으로 작성해 봅니다.

Anthropic이 공식 SDK를 제공하므로, 실제로는 거의 데코레이터 몇 줄이면 끝납니다. 우리가 신경 쓸 것은 "이 서버가 제공할 도구의 로직"뿐입니다.

1단계 — 환경 준비

$ shell
# Python 3.10+ 가 필요합니다 pip install mcp httpx # 또는 권장: uv 패키지 매니저 사용 uv init weather-mcp && cd weather-mcp uv add mcp httpx

2단계 — 서버 작성 (server.py)

server.py
from mcp.server.fastmcp import FastMCP import httpx # 서버 이름과 함께 인스턴스 생성 mcp = FastMCP('weather-mcp') @mcp.tool() async def get_weather(city: str) -> str: '''주어진 도시의 현재 날씨를 가져옵니다. Args: city: 영문 도시명 (예: 'Seoul', 'Tokyo') ''' url = f'https://wttr.in/{city}?format=j1' async with httpx.AsyncClient() as client: r = await client.get(url, timeout=10.0) data = r.json() cur = data['current_condition'][0] return ( f'{city}: {cur["temp_C"]}°C, ' f'{cur["weatherDesc"][0]["value"]}, ' f'습도 {cur["humidity"]}%' ) if __name__ == '__main__': mcp.run(transport='stdio')

이게 전부입니다. 핵심을 짚어 보면:

  • @mcp.tool() 데코레이터 하나가 이 함수를 MCP Tool로 노출합니다.
  • 함수의 docstring이 도구의 설명이 됩니다 — LLM은 이 설명을 읽고 언제 부를지 결정합니다.
  • 타입 힌트(city: str) 가 자동으로 입력 스키마가 됩니다.
  • transport='stdio' 로 로컬 실행 (Claude Desktop이 자식 프로세스로 띄우는 방식).

3단계 — Claude Desktop에 연결

Claude Desktop 설정 파일(claude_desktop_config.json)에 추가합니다:

claude_desktop_config.json
{ "mcpServers": { "weather": { "command": "uv", "args": ["--directory", "/path/to/weather-mcp", "run", "server.py"] } } }

Claude Desktop을 재시작하면, 우리가 만든 get_weather 도구가 자동으로 발견됩니다. 이제 Claude에게 "도쿄 날씨 어때?" 라고 물으면, 진짜로 우리 서버를 호출해서 답을 가져옵니다.

엔지니어 노트 — 멀티 아키텍처 환경
DGX Spark(ARM64)와 ThinkStation(x86_64) 처럼 이종 노드 클러스터에 MCP 서버를 배포한다면, multi-arch 이미지 빌드를 잊지 마세요. docker buildx build --platform linux/amd64,linux/arm64 로 한 번에 양쪽 이미지를 만들고, 노드 셀렉터(kubernetes.io/arch)로 적절히 배치하면 같은 MCP 서버를 어디서든 띄울 수 있습니다.

JavaScript/TypeScript 버전도 있나요?

물론입니다. Python, TypeScript, C#, Java, Kotlin, Go, Rust 등 공식 SDK가 다국어로 제공됩니다. TypeScript 버전의 같은 예시는 이렇게 짧습니다:

server.ts (TypeScript SDK)
import { McpServer } from '@modelcontextprotocol/sdk/server/mcp.js'; import { StdioServerTransport } from '@modelcontextprotocol/sdk/server/stdio.js'; import { z } from 'zod'; const server = new McpServer({ name: 'weather-mcp', version: '1.0.0' }); server.tool( 'get_weather', { city: z.string().describe('영문 도시명') }, async ({ city }) => { const r = await fetch(`https://wttr.in/${city}?format=j1`); const data = await r.json(); const cur = data.current_condition[0]; return { content: [{ type: 'text', text: `${city}: ${cur.temp_C}°C` }] }; } ); const transport = new StdioServerTransport(); await server.connect(transport);
  1. MCP 서버 작성은 "평범한 함수 + 데코레이터" 수준으로 단순합니다.
  2. SDK가 핸드셰이크·메시지 파싱·스키마 생성 등 지루한 부분을 모두 처리합니다.
  3. 한 번 만든 서버는 Claude Desktop, Cursor, VS Code 등 모든 호환 호스트에서 그대로 작동.
  4. 실무에서는 입력 검증(Pydantic/Zod), 인증, 에러 처리 등을 추가로 신경 써야 합니다.

거버넌스 — 누가 표준을 관리하는가

2025년 12월, Anthropic은 MCP를 Linux Foundation 산하의 Agentic AI Foundation (AAIF)에 기증했습니다. Anthropic, Block, OpenAI가 공동 창립자로 참여한 중립 기구입니다. 즉, MCP는 더 이상 한 회사의 소유가 아닙니다.

스펙은 SEP(Spec Enhancement Proposal) 프로세스로 발전합니다. 누구나 제안할 수 있고, Working Group이 검토합니다. 2026년 로드맵은 전송 확장성, 에이전트 간 통신, 거버넌스 성숙, 엔터프라이즈 준비 네 가지를 우선순위로 두고 있습니다.

찾는 법 — MCP 서버 레지스트리

공식 레지스트리(modelcontextprotocol/registry)에 수백 개의 서버가 등록돼 있습니다. 자신이 필요한 도구가 있는지 먼저 검색해 보고, 없다면 직접 만들어 기여하는 것이 자연스러운 흐름입니다.

Chapter 10 · 보안

강력한 만큼,
조심해야 합니다

MCP는 LLM에게 실제 시스템에 영향을 미치는 권한을 줍니다. 그래서 보안은 부가 기능이 아니라 설계의 핵심입니다.

MCP 명세는 보안 원칙을 명문화하고 있습니다. Host 구현자는 다음을 반드시 고려해야 합니다:

필수 보안 원칙
1. 명시적 사용자 동의(consent) — Host는 도구를 호출하기 전 사용자에게 확인을 받아야 합니다. "이 도구를 부르려는데 괜찮으세요?" 라고요. LLM이 마음대로 실행해선 안 됩니다.

2. 도구 설명은 신뢰하지 말 것 — 악의적인 서버가 도구 설명을 조작해 LLM을 속일 수 있습니다(프롬프트 인젝션). 신뢰된 출처가 아니면 어노테이션을 모델에 그대로 넘기지 않아야 합니다.

3. 입력 검증 — LLM이 만들어내는 인자는 항상 스키마 검증을 거쳐야 합니다(Pydantic, Zod 등). 환각된 파라미터는 실제 발생하는 실패 모드입니다.

4. 자격 증명 격리 — API 키는 환경 변수에, 절대로 도구 스키마나 응답 페이로드에 노출하지 마세요.

5. 최소 권한 — 토큰 스코프를 좁히고, Resource Indicators (RFC 8707) 로 토큰이 다른 곳에서 재사용되는 것을 막아야 합니다.

인증 — OAuth 2.1

원격 MCP 서버(HTTP 기반)는 OAuth 2.1 을 인증 표준으로 채택합니다. 2025년 6월 스펙 업데이트는 MCP 서버를 OAuth Resource Server로 명시적으로 분류했고, 토큰 오용을 막기 위한 Resource Indicators 사용을 의무화했습니다. 이는 토큰이 의도하지 않은 다른 서버에서 재사용되는 공격을 차단합니다.

현실의 함정
가장 흔한 실수는 "이 LLM은 똑똑하니까 알아서 잘하겠지" 라며 가드레일 없이 운영하는 것입니다. 프로덕션에서는: 도구 호출 감사 로그, 위험 도구(예: delete_*)에 추가 승인 단계, 샌드박스(Docker, gVisor) 안에서 서버 실행, 그리고 정기적인 권한 회수를 권장합니다.

Epilogue · 12

여기까지
오신 분께

이제 MCP를 들었을 때, 막연한 buzzword가 아니라 구체적인 그림이 떠오를 겁니다.

오늘 우리는 왜 MCP가 필요한지 (N×M 문제), 무엇인지 (JSON-RPC 기반 오픈 표준), 어떻게 동작하는지 (Host/Client/Server, 세 가지 프리미티브, 라이프사이클), 그리고 어디에 쓰이는지 (수많은 실전 시나리오)를 살펴봤습니다. 작은 서버를 직접 만들어 보기도 했고, 보안의 함정도 짚어봤습니다.

AI 비서가 정말 비서답게 일하려면, 세상과 닿아 있어야 합니다. MCP는 그 닿음을 표준화한 약속입니다. 한 번 만든 도구가 모든 곳에서 작동하고, 한 번 만든 호스트가 모든 도구를 쓸 수 있는 — 그런 미래의 인프라를 우리는 지금 만들고 있습니다.

"2024년에 우리는 모두 프롬프트 엔지니어였다. 2025년엔 RAG 아키텍트가 되었다.
2026년의 컨텍스트 엔지니어는, 지능을 행동으로 잇는 인프라를 만드는 사람들이다." — 업계의 한 정리

  • 1. Claude Desktop을 설치하고, 공식 레지스트리의 서버 한두 개를 직접 연결해 봅니다.
  • 2. 이 강의의 weather 서버 예제를 따라 만들어, 실제로 동작하는 것을 눈으로 확인합니다.
  • 3. 본인 업무에서 반복되는 작업 하나를 골라 MCP 서버로 만들어 봅니다.
  • 4. 공식 명세(modelcontextprotocol.io)를 한번 훑어보면, 더 깊은 기능(Sampling, Elicitation, MCP Apps 등)이 보입니다.
  1. MCP는 N×M 통합 문제를 N+M으로 줄이는 오픈 표준입니다.
  2. 핵심은 단순합니다: JSON-RPC 메시지로 Host·Client·Server가 대화합니다.
  3. 서버가 노출할 수 있는 것은 단 셋: Tools(행동) · Resources(읽기) · Prompts(템플릿).
  4. SDK 덕분에 몇 십 줄 코드로 첫 서버를 만들 수 있습니다.
  5. 강력한 만큼 사용자 동의·입력 검증·OAuth 2.1·최소 권한이 필수.
  6. 한 번 만들면 Claude · ChatGPT · Cursor · VS Code 등 모든 호환 호스트에서 그대로 작동합니다.
