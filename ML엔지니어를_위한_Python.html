왜 "그냥 Python"으론 부족한가

문법을 아는 것과, 실무 ML 코드를 읽고 쓰는 것은 다른 능력이다. 그 차이가 무엇인지부터 짚는다.

당신은 이미 for i in range(10)를 쓸 줄 안다. 그런데 PyTorch 데이터 로더나 전처리 스크립트를 열면 이런 코드가 보인다.

@lru_cache(maxsize=128)
def load_features(path: str) -> np.ndarray:
    with open(path) as f:
        rows = (line.strip().split(",") for line in f)   # 제너레이터
        data = [[float(x) for x in r] for r in rows]      # 중첩 컴프리헨션
    return np.array(data)

# 정규화: 반복문 한 줄도 없이 전체 배열을 한 번에
X = load_features("train.csv")
X_norm = (X - X.mean(axis=0)) / X.std(axis=0)

여기엔 우리가 이 강의에서 다룰 거의 모든 것이 들어 있다. 데코레이터(@lru_cache), 타입 힌트(-> np.ndarray), 컨텍스트 매니저(with), 제너레이터(괄호 안의 for), 컴프리헨션, 그리고 벡터화(반복문 없이 배열 전체 연산). 문법책에는 잘 안 나오지만, 실무 코드에는 거의 항상 나온다.

⌘ 이 강의의 약속

각 주제마다 (1) 왜 필요한가 → (2) 가장 쉬운 예제 → (3) ML 실무에서 어떻게 쓰이나 순서로 간다. 외우지 말고, "아, 이래서 이렇게 쓰는구나"를 느끼는 게 목표다.

두 가지 마음가짐

1. "Pythonic"이란 단어에 익숙해지자. Python에는 "문법적으로 맞다"를 넘어 "Python답다"는 미감이 있다. 같은 결과라도 더 짧고, 더 읽기 쉽고, 더 빠른 방식이 선호된다. ML 코드 리뷰에서 자주 듣게 될 말이다.

2. "배열처럼 사고하기"를 연습하자. ML은 본질적으로 숫자 배열(텐서)을 다루는 일이다. 한 개씩 처리하는 for 루프 사고에서, 배열 전체를 한 번에 처리하는 벡터화 사고로 넘어가는 것이 가장 큰 도약이다. (그래서 이 강의의 섹션 번호도 배열 인덱스 arr[0]로 매겼다.)

컴프리헨션, 언패킹, zip/enumerate, f-string, 바다코끼리 연산자. 코드를 절반으로 줄이는 도구들.

리스트 컴프리헨션 (List Comprehension)

"리스트를 만드는 for문"을 한 줄로 압축하는 문법이다. ML 코드에서 데이터 변환할 때 끊임없이 쓰인다.

# 0~9의 제곱을 담은 리스트
squares = []
for i in range(10):
    squares.append(i * i)

# 짝수만 골라서 제곱
even_sq = []
for i in range(10):
    if i % 2 == 0:
        even_sq.append(i * i)
# 같은 일을 한 줄로
squares = [i * i for i in range(10)]

# 조건(필터)은 뒤에 if를 붙인다
even_sq = [i * i for i in range(10) if i % 2 == 0]

# dict / set 컴프리헨션도 있다
word_len = {w: len(w) for w in ["cat", "dog"]}   # {'cat':3, 'dog':3}
unique   = {x % 3 for x in range(10)}            # {0, 1, 2}
# squares
[0, 1, 4, 9, 16, 25, 36, 49, 64, 81]
# even_sq
[0, 4, 16, 36, 64]
✓ 읽는 법

[ 결과  for 변수 in 반복대상  if 조건 ] — 항상 이 순서다. "각 i에 대해 i*i를, 단 i가 짝수일 때만"이라고 읽으면 된다. 영어 어순과 비슷하다.

중첩 컴프리헨션 (행렬 다룰 때)

2차원 데이터를 평탄화(flatten)하거나 변환할 때 쓴다. ML에서 배치(batch) 데이터를 다룰 때 자주 등장한다.

matrix = [[1, 2, 3], [4, 5, 6]]

# 평탄화: [1, 2, 3, 4, 5, 6]
flat = [x for row in matrix for x in row]
# 순서 주의! 바깥 for가 먼저, 안쪽 for가 나중 (왼→오른쪽으로 중첩 풀기)

# 각 원소에 함수 적용 (2차원 유지)
doubled = [[x * 2 for x in row] for row in matrix]
⚠ 가독성 한계

컴프리헨션이 한 줄에 for 두 개 + if까지 들어가면 오히려 읽기 어렵다. 이럴 땐 그냥 일반 for문이 낫다. 그리고 숫자 배열이라면 — 곧 배울 NumPy 벡터화가 컴프리헨션보다 훨씬 빠르고 깔끔하다.

언패킹(Unpacking)과 별표(*)

여러 값을 한 번에 꺼내는 문법. 함수가 여러 값을 반환할 때, 데이터를 분리할 때 핵심이다.

point = (3.5, 7.2)
x, y = point                      # x=3.5, y=7.2  (튜플 언패킹)

a, b = b, a                       # 변수 교환 (임시변수 불필요)

# 별표(*)로 "나머지 전부"를 묶기
first, *rest = [1, 2, 3, 4, 5]    # first=1, rest=[2, 3, 4, 5]
*init, last = [1, 2, 3, 4, 5]     # init=[1,2,3,4], last=5
head, *mid, tail = [1, 2, 3, 4]   # head=1, mid=[2,3], tail=4

# 딕셔너리/리스트 펼치기 (함수 인자, 합치기에 자주 씀)
defaults = {"lr": 0.01, "epochs": 10}
config = {**defaults, "epochs": 50}   # lr 유지, epochs 덮어쓰기
✓ ML 실무

train_test_split()은 4개를 반환한다: X_train, X_test, y_train, y_test = train_test_split(...). 이게 언패킹이다. 설정(config) 병합에는 {**base, **override} 패턴이 표준처럼 쓰인다.

⌘ zip의 두 얼굴

zip(*matrix)는 행렬을 전치(transpose)한다 — 별표 언패킹과 결합한 관용구다. 또 zip은 가장 짧은 리스트에 맞춰 멈추므로, 길이가 다르면 데이터가 조용히 잘릴 수 있다. 길이 검증이 필요하면 zip(a, b, strict=True)(Python 3.10+)를 쓰자.

함수를 깊이 있게

가변 인자(*args/**kwargs), 기본 인자의 함정, 람다와 고차 함수, 클로저. 함수가 "객체"라는 사실에서 데코레이터가 출발한다.

*args 와 **kwargs

"인자를 몇 개 받을지 미리 모를 때" 쓴다. ML 라이브러리 래핑이나 콜백 함수에서 거의 필수다.

def train(model, *args, **kwargs):
    # args   : 위치 인자들을 담은 튜플   (1, 2, 3)
    # kwargs : 키워드 인자들을 담은 딕셔너리  {'lr': 0.01}
    print(args)      # (32, 10)
    print(kwargs)    # {'lr': 0.01, 'shuffle': True}

train(model, 32, 10, lr=0.01, shuffle=True)
# 반대로, 이미 만든 자료구조를 인자로 "펼쳐서" 전달
batch_sizes = [32, 64]
options = {"lr": 0.01, "shuffle": True}

train(model, *batch_sizes, **options)
# == train(model, 32, 64, lr=0.01, shuffle=True)

# 래퍼 함수의 전형: 받은 인자를 그대로 안쪽으로 넘긴다
def my_fit(*args, **kwargs):
    log("학습 시작")
    return model.fit(*args, **kwargs)
⌘ 별표(*)의 의미는 위치로 결정

정의할 때 *args는 "흩어진 인자를 모은다(pack)". 호출할 때 *list는 "모인 것을 흩뿌린다(unpack)". 같은 기호, 반대 동작. 이 대칭만 잡으면 끝이다.

⚠ 가장 악명 높은 함정: 가변 기본 인자

ML 엔지니어가 실제로 가장 많이 당하는 버그 중 하나다. 기본값으로 리스트나 딕셔너리를 쓰면 안 된다.

# ❌ 절대 이렇게 하지 말 것
def add_sample(x, batch=[]):
    batch.append(x)
    return batch

add_sample(1)   # [1]
add_sample(2)   # [1, 2]  ← 어?! 이전 호출이 남아있다
add_sample(3)   # [1, 2, 3]
# ✅ 기본값은 None으로 두고 함수 안에서 생성
def add_sample(x, batch=None):
    if batch is None:
        batch = []
    batch.append(x)
    return batch

add_sample(1)   # [1]
add_sample(2)   # [2]   ← 매번 새 리스트
add_sample(3)   # [3]
# 함정: 기본 리스트가 함수에 "단 한 번" 생성되어 공유됨
[1]
[1, 2]
[1, 2, 3]

# 정상: 호출마다 새로 생성

✗ 원인

기본 인자 값은 함수가 정의되는 순간 단 한 번 만들어져 함수에 붙어 있다. 매 호출마다 새로 만들어지지 않는다. 그래서 가변 객체(list, dict, set)를 기본값으로 두면 호출 간에 상태가 새어 나간다. 규칙: 기본값으로는 불변 객체(None, 숫자, 문자열, 튜플)만 쓴다.

람다 · 고차 함수

람다는 "이름 없는 한 줄 함수"다. 정렬 기준이나 콜백처럼 작은 함수를 인자로 넘길 때 쓴다.

samples = [{"name": "a", "loss": 0.5}, {"name": "b", "loss": 0.2}]

# loss 기준 정렬 — key에 "어떤 값으로 비교할지" 함수를 넘긴다
samples.sort(key=lambda s: s["loss"])

# 가장 좋은(작은) loss
best = min(samples, key=lambda s: s["loss"])

# map/filter — 다만 컴프리헨션이 보통 더 읽기 쉽다
names  = list(map(lambda s: s["name"], samples))   # ['b', 'a']
names2 = [s["name"] for s in samples]              # 같은 결과, 더 Pythonic
⚠ 람다는 작을 때만

람다에 로직이 두 줄 이상 들어가고 싶어지면 그냥 def로 이름 있는 함수를 만들자. 람다는 "정렬 키" 정도의 초소형 함수에 어울린다.

클로저(Closure) — 함수가 상태를 기억한다

함수 안에서 함수를 만들어 반환하면, 안쪽 함수는 바깥 변수를 "기억"한다. 데코레이터의 작동 원리이자, 콜백·팩토리 패턴의 기반이다.

def make_multiplier(factor):
    def multiply(x):
        return x * factor      # 바깥의 factor를 기억한다
    return multiply

double = make_multiplier(2)
triple = make_multiplier(3)

print(double(10))   # 20
print(triple(10))   # 30
20

이터레이터 & 제너레이터

대용량 데이터를 메모리에 다 올리지 않고 "필요할 때 하나씩" 흘려보내는 기술. ML 데이터 파이프라인의 심장이다.

10GB짜리 CSV를 학습에 쓴다고 하자. data = list(open("huge.csv"))처럼 전부 메모리에 올리면 터진다. 제너레이터는 한 줄씩 읽어서 처리하고 버린다. 이게 PyTorch DataLoader, TensorFlow tf.data가 내부에서 하는 일의 본질이다.

제너레이터: yield 한 줄의 마법

return 대신 yield를 쓰면, 함수는 값을 "하나 내놓고 그 자리에서 멈췄다가" 다음에 호출되면 이어서 실행된다.

# 전부 만들어서 한꺼번에 반환 → 메모리에 1억 개가 동시에 존재
def squares(n):
    result = []
    for i in range(n):
        result.append(i * i)
    return result

data = squares(100_000_000)   # 💥 수 GB 메모리 점유
# yield: 하나 내놓고 멈춤 → 메모리엔 "지금 그 한 개"만
def squares(n):
    for i in range(n):
        yield i * i           # 여기서 멈췄다가, 다음 요청 때 이어서 실행

gen = squares(100_000_000)    # 아직 아무것도 계산 안 됨 (게으름)
for x in gen:                 # 요청할 때마다 하나씩 생성
    if x > 100:
        break                 # 필요한 만큼만 계산하고 중단 가능
# 리스트 방식 메모리 (1억 개 정수)
~ 4,000 MB

# 제너레이터 방식 메모리
~ 0.0001 MB  (한 번에 한 개씩)
▶ 인터랙티브: 메모리 점유 비교 (n개 정수 처리)
list 방식

제너레이터 표현식 (괄호만 바꾸면 된다)

컴프리헨션의 대괄호 [ ]를 소괄호 ( )로 바꾸면 리스트가 아니라 제너레이터가 된다. sum() 같은 함수에 바로 넣으면 중간 리스트 없이 처리된다.

# [ ] 리스트: 전체를 메모리에 만든 뒤 합산
total = sum([x * x for x in range(10_000_000)])   # 중간 리스트 생성

# ( ) 제너레이터: 하나씩 더하고 버림 → 메모리 절약
total = sum(x * x for x in range(10_000_000))     # 함수 인자라 괄호 생략 가능

데코레이터

@ 기호로 시작하는 그것. 함수의 코드를 건드리지 않고 기능을 "감싸서" 추가한다. 타이밍 측정·캐싱·재시도가 한 줄로 끝난다.

앞에서 "함수도 객체이고, 함수를 받아 함수를 돌려줄 수 있다"고 했다. 데코레이터는 정확히 그것이다. 함수를 입력받아, 그 함수를 감싼 새 함수를 돌려준다. @는 그저 그 과정을 예쁘게 적는 문법 설탕(syntactic sugar)이다.

def timer(func):                      # 함수를 받아서
    def wrapper(*args, **kwargs):     # 감싸는 새 함수를 만들고
        start = time.perf_counter()
        result = func(*args, **kwargs)   # 원래 함수 실행
        print(f"{func.__name__}: {time.perf_counter()-start:.3f}s")
        return result
    return wrapper                    # 새 함수를 돌려준다

def train():
    ...

train = timer(train)     # ← 수동으로 감싸기. 이게 데코레이터의 본질
train()
def timer(func):
    def wrapper(*args, **kwargs):
        start = time.perf_counter()
        result = func(*args, **kwargs)
        print(f"{func.__name__}: {time.perf_counter()-start:.3f}s")
        return result
    return wrapper

@timer                   # ← train = timer(train) 과 정확히 동일
def train():
    ...

train()    # 자동으로 시간 측정됨
⌘ 한 문장 요약

@timer는 "바로 아래 함수를 timer()로 감싸서 같은 이름에 다시 넣어줘"라는 뜻이다. *args, **kwargs로 받는 이유는, 어떤 시그니처의 함수든 그대로 통과시키기 위해서다.

functools.wraps — 잊지 말 것

감싸면 원래 함수의 이름(__name__)과 docstring이 wrapper로 덮인다. @wraps가 이를 복원해준다. 디버깅·로깅에서 중요하다.

from functools import wraps

def timer(func):

    def wrapper(*args, **kwargs):
        ...
        return func(*args, **kwargs)

인자를 받는 데코레이터 (재시도 예제)

@retry(times=3)처럼 데코레이터에 설정을 주려면 한 겹 더 감싼다. 데이터 다운로드·API 호출처럼 가끔 실패하는 작업에 유용하다.

from functools import wraps

def retry(times=3):                       # ① 설정을 받는 바깥 함수
    def decorator(func):                  # ② 함수를 받는 진짜 데코레이터
        @wraps(func)
        def wrapper(*args, **kwargs):     # ③ 실제 동작
            for attempt in range(1, times + 1):
                try:
                    return func(*args, **kwargs)
                except Exception as e:
                    print(f"시도 {attempt} 실패: {e}")
            raise RuntimeError(f"{times}회 모두 실패")
        return wrapper
    return decorator

@retry(times=3)
def download(url):
    ...

내가 직접 만들기: @contextmanager

가장 쉬운 방법은 @contextmanager 데코레이터 + 제너레이터다. yield 이전은 "들어갈 때", 이후는 "나올 때" 코드다. (앞서 배운 데코레이터+제너레이터가 여기서 만난다!)

from contextlib import contextmanager
import time

@contextmanager
def timed(label):
    start = time.perf_counter()      # ── 들어갈 때 (setup)
    try:
        yield                        # ── 여기서 with 블록 본문이 실행됨
    finally:
        dt = time.perf_counter() - start
        print(f"[{label}] {dt:.3f}s")   # ── 나올 때 (teardown, 에러 나도 실행)

with timed("전처리"):
    preprocess(dataset)

with timed("학습 1 에폭"):
    train_one_epoch(model)
[전처리] 0.842s
[학습 1 에폭] 12.317s

@dataclass — 설정 객체의 표준

하이퍼파라미터·설정을 딕셔너리로 넘기면 오타가 나도 모른다. @dataclass는 보일러플레이트(__init__ 등) 없이 타입이 명시된 설정 클래스를 만들어준다.

# dict: 자동완성 없음, 오타 잡기 어려움, 기본값 관리 번거로움
config = {"lr": 0.01, "epochs": 10, "batch_size": 32}
config["learnig_rate"]   # 오타 → KeyError를 런타임에야 발견
from dataclasses import dataclass, field

@dataclass
class TrainConfig:
    lr: float = 0.01
    epochs: int = 10
    batch_size: int = 32
    layers: list = field(default_factory=list)  # 가변 기본값은 이렇게!

cfg = TrainConfig(lr=0.001)
print(cfg)            # TrainConfig(lr=0.001, epochs=10, batch_size=32, layers=[])
print(cfg.lr)         # 자동완성·타입 힌트 지원, 오타 시 에러
✗ 또 그 함정

dataclass에서 리스트·딕셔너리를 기본값으로 줄 때 layers: list = []라고 쓰면 에러가 난다. Python이 아예 막아준다. 대신 field(default_factory=list)를 써야 한다. 함수 기본 인자 함정의 dataclass 버전이다.

매직 메서드 — Dataset을 만드는 법

이름이 __xxx__인 메서드들. 이걸 정의하면 내 객체가 Python 기본 문법(len(), obj[i], for)과 자연스럽게 동작한다. PyTorch Dataset의 정체가 바로 이것이다.

        return len(self.paths)

        return image, self.labels[idx]

        return f"ImageDataset(n={len(self)})"

NumPy로 사고하기

ML의 모든 데이터는 숫자 배열이다. 반복문 대신 배열 전체를 한 번에 — 이 "벡터화 사고"가 가장 큰 도약이다. PyTorch·TensorFlow 텐서도 전부 이 사고의 확장이다.

벡터화: for 루프를 지워라

같은 연산이라도 NumPy 배열에 한 번에 적용하면 C로 구현된 내부 루프가 돌아 수십~수백 배 빠르다. 그리고 코드도 짧다.

result = []
for a, b in zip(list_a, list_b):
    result.append(a * b)
dot = sum(result)

# 정규화 — 한 원소씩
out = []
for x in data:
    out.append((x - mean) / std)
import numpy as np

a = np.array(list_a)
b = np.array(list_b)

# 정규화 — 배열 전체를 한 줄로 (반복문 없음)

⌘ 사고 전환

"각 원소에 대해 무엇을 할까?"(for 사고)에서 "배열 전체에 어떤 연산을 적용할까?"(벡터 사고)로. a + b, a * 2, np.exp(a)는 모두 배열 전체에 동시 적용된다. ML 코드에 for가 거의 없는 이유다.

브로드캐스팅 (Broadcasting)

shape가 다른 배열끼리 연산할 때, NumPy가 작은 쪽을 "자동으로 늘려서" 맞춰주는 규칙이다. 데이터를 실제로 복제하지 않으면서(메모리 절약) 동작한다. 배치에 편향을 더하거나, 각 특성을 정규화할 때 매 순간 쓰인다.

▶ 인터랙티브: 브로드캐스팅이 일어나는 방식
(3, 3)
+
(3,)
=
(3, 3)
X = np.ones((100, 3))      # 100개 샘플, 3개 특성  shape (100, 3)
bias = np.array([1, 2, 3]) # 특성별 편향           shape (3,)

X + bias    # (100,3) + (3,) → bias가 100개 행에 자동 적용. for 불필요

# 특성별 표준화: mean/std는 (3,), X는 (100,3)
X_norm = (X - X.mean(axis=0)) / X.std(axis=0)

# 열 방향으로 더하려면 (100,1)로 모양을 맞춘다
col = np.arange(100).reshape(100, 1)   # shape (100, 1)
X + col                                 # 각 행에 서로 다른 값
# 브로드캐스팅 규칙 (오른쪽 차원부터 비교)
# (100, 3)
#      (3,)   ← 차원을 오른쪽 정렬
# ---------
# 각 차원이 ① 같거나  ② 한쪽이 1 이면 OK
#
# (100, 3) + (3,)    → OK  (3==3, 100은 1로 간주되어 늘어남)
# (100, 3) + (100,1) → OK  (1이 3으로, 3 그대로)
# (100, 3) + (100,)  → 에러! (3 vs 100, 둘 다 1 아님)
⚠ shape 에러는 1순위 디버깅 대상

ML 코드 에러의 절반은 shape 불일치다. 막히면 일단 print(x.shape)를 곳곳에 찍어라. .reshape(-1, 1), .squeeze(), np.newaxis로 차원을 맞추는 게 일상이다.

인덱싱·슬라이싱·불리언 마스크

X = np.arange(12).reshape(3, 4)    # 3행 4열

X[0]          # 첫 번째 행 (1차원)
X[:, 1]       # 모든 행의 두 번째 열
X[0:2, 1:3]   # 0~1행, 1~2열 부분 행렬
X[-1]         # 마지막 행

# 불리언 마스크 — 조건에 맞는 원소만 (전처리·필터링 핵심)
mask = X > 5
X[mask]            # 5보다 큰 원소들만 1차원으로
X[X > 5] = 0       # 5보다 큰 값을 0으로 한 번에

# 팬시 인덱싱 — 원하는 인덱스만 골라서
X[[0, 2]]         # 0행과 2행만
✗ view vs copy 함정

슬라이싱 y = X[0:2]는 대부분 원본을 공유하는 view다. y[0] = 99를 하면 X도 바뀐다! 독립 복사가 필요하면 X[0:2].copy()를 명시하라. 반대로 불리언/팬시 인덱싱은 복사본을 만든다. 헷갈리면 .copy()로 안전하게.

axis 개념 — 어느 방향으로 줄일까

합·평균 등을 구할 때 axis가 "사라지는 축"이다. 2차원에서 axis=0은 행 방향으로 내려가며(→ 열별 결과), axis=1은 열 방향으로 가로지른다(→ 행별 결과).

X = np.array([[1, 2, 3],
              [4, 5, 6]])        # shape (2, 3)

X.sum()           # 전체 합: 21
X.sum(axis=0)     # 열별 합 (행이 사라짐): [5, 7, 9]   shape (3,)
X.sum(axis=1)     # 행별 합 (열이 사라짐): [6, 15]      shape (2,)

# 분류 모델 출력에서 가장 확률 높은 클래스 고르기

동시성 & 병렬성

GIL이 무엇인지, 언제 thread를 쓰고 언제 process를 쓰는지. DataLoader(num_workers=4)가 내부에서 하는 일.

먼저 GIL을 이해하자

CPython에는 GIL(Global Interpreter Lock)이 있어, 한 프로세스 안에서는 한 번에 하나의 스레드만 Python 바이트코드를 실행한다. 즉 여러 스레드를 띄워도 순수 Python 계산은 동시에 빨라지지 않는다. 이 한 문장이 아래 모든 선택을 결정한다.

⌘ 핵심 분기

I/O 바운드(파일 읽기, 네트워크 다운로드 — 기다리는 시간이 큼) → threading / asyncio가 효과적. 기다리는 동안 GIL을 놓아주기 때문.
CPU 바운드(순수 Python 계산, 이미지 디코딩 등) → multiprocessing으로 진짜 여러 코어를 써야 한다.

concurrent.futures — 가장 쉬운 진입점

스레드/프로세스를 직접 다루지 말고 Executor로 추상화하자. 인터페이스가 거의 같아서 둘을 바꿔 끼우기 쉽다.

from concurrent.futures import ThreadPoolExecutor

# 여러 URL을 동시에 다운로드 (I/O 바운드 → 스레드가 효과적)
urls = ["http://...", "http://...", ...]

with ThreadPoolExecutor(max_workers=8) as ex:
    results = list(ex.map(download, urls))
# 한 다운로드가 응답을 기다리는 동안 다른 다운로드가 진행됨
from concurrent.futures import ProcessPoolExecutor

# CPU 무거운 전처리를 여러 코어로 (CPU 바운드 → 프로세스)
def heavy_preprocess(path):
    img = decode_and_resize(path)   # 순수 계산 비용 큼
    return augment(img)

with ProcessPoolExecutor(max_workers=4) as ex:
    processed = list(ex.map(heavy_preprocess, paths))
# 진짜로 4개 코어가 동시에 일한다 (GIL 우회)
⚠ 프로세스의 비용

프로세스는 메모리를 공유하지 않아, 함수와 데이터가 각 프로세스로 복사(pickle)된다. 그래서 ① 인자/반환값이 직렬화 가능해야 하고(람다·로컬 함수 불가), ② 작은 작업을 잘게 쪼개면 통신 비용이 이득을 잡아먹는다. "충분히 무거운 작업을 적당한 덩어리로"가 원칙이다.

asyncio — 한 스레드로 수천 개 대기 다루기

많은 네트워크 요청처럼 "대부분 기다리는" 작업에 적합하다. LLM API를 동시에 여러 개 호출할 때 특히 유용하다.

import asyncio

async def call_llm(prompt):           # async def: 코루틴 함수
    resp = await client.chat(prompt)  # await: 응답 기다리는 동안 양보
    return resp

async def main(prompts):
    # gather: 여러 코루틴을 동시에 진행시키고 모두 모으기
    return await asyncio.gather(*(call_llm(p) for p in prompts))

results = asyncio.run(main(prompts))  # 100개 요청을 순차 아닌 동시에

그래서 DataLoader가 하는 일

DataLoader(dataset, num_workers=4)는 위 개념의 응용이다. GPU가 현재 배치를 학습하는 동안, 여러 워커 프로세스가 다음 배치들을 미리 디코딩·증강해 준비한다. 그래서 GPU가 데이터를 기다리며 노는 시간이 줄어든다. 방금 배운 ProcessPool + 미리읽기(prefetch) 조합이다.
