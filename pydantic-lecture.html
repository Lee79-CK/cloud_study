거친 데이터를
믿을 수 있는 타입으로

바깥세상에서 들어오는 데이터(JSON, 폼, 환경변수, LLM 출력)는 언제나 거칠고 믿을 수 없습니다. Pydantic은 그 데이터가 우리 코드 안으로 들어오는 관문(gate)에 서서, 타입을 검사하고·변환하고·잘못된 것은 분명한 이유와 함께 돌려보냅니다. 파이썬 기본 기능과 하나하나 비교하며, 왜 필요한지부터 ML 실전 활용까지 차근차근 배워봅니다.

00 : 왜 Pydantic인가

"이 데이터가 정말 내가 생각한 모양일까?"

ML 코드는 거의 항상 외부 데이터로 시작합니다. API 요청의 JSON, 설정 파일, 환경변수, 데이터셋의 한 줄, 그리고 요즘은 LLM이 뱉어낸 출력까지. 문제는 이 데이터들이 내가 코드를 짤 때 머릿속에 그린 모양과 다를 수 있다는 점입니다.

파이썬에서 데이터를 받는 가장 흔한 방법은 dict입니다. 그런데 dict는 "무엇이든 담는 자루"입니다. 키가 있는지, 값의 타입이 맞는지, 아무도 보장해 주지 않습니다. 

# 외부에서 받았다고 가정한 데이터
user = {"name": "미나", "age": "30"}   # age 가 문자열!

# 코드는 age 가 숫자라고 "믿고" 계산한다
next_year = user["age"] + 1          # 💥 "30" + 1 → TypeError
discount  = user["aeg"]              # 💥 오타 → KeyError
city      = user["city"]             # 💥 없는 키 → KeyError
# 진짜 무서운 점: 에러가 "한참 뒤"에 터진다
def train_pipeline(records):
    for r in records:
        # ... 수십 줄의 전처리 ...
        score = r["age"] * 0.1       # 여기서 갑자기 터진다
        # 어떤 레코드가 문제였는지, 왜인지 알 수가 없다

물론 직접 검사할 수 있습니다. if "age" not in user, isinstance(...), try/except... 하지만 필드가 10개만 되어도 검사 코드가 데이터 본질보다 길어집니다. 그리고 모든 함수마다 같은 검사를 반복해야 하죠.

Pydantic 의 한 줄 요약 "데이터의 모양(스키마)을 클래스로 한 번 선언해 두면, 그 뒤로는 검증·변환·에러 메시지·문서화·직렬화를 Pydantic이 알아서 해 준다." — 검사 코드를 데이터 정의로 바꾸는 것입니다.
✕ 파이썬 기본 — 수작업 검증
def make_user(d):
    if "name" not in d:
        raise ValueError("name 필요")
    if not isinstance(d["name"], str):
        raise TypeError("name 은 str")
    if "age" not in d:
        raise ValueError("age 필요")
    age = int(d["age"])  # 변환도 직접
    # ... 필드마다 끝없이 ...
    return d
✓ Pydantic — 모양만 선언
from pydantic import BaseModel, EmailStr

class User(BaseModel):
    name: str
    age: int
    email: EmailStr

# 끝. 검증·변환·에러는 자동.
user = User(**d)
# age 가 "30" 이면 30 으로 변환,
# 형식이 틀리면 친절한 에러.
01 : BaseModel 첫걸음

모델 하나 = 데이터 한 종류의 설계도

Pydantic의 모든 것은 BaseModel을 상속하는 클래스에서 시작합니다. 클래스 본문에 필드 이름 : 타입 형태로 적기만 하면, 그게 곧 "이 데이터는 이렇게 생겨야 한다"는 계약서가 됩니다. 

from pydantic import BaseModel

class Movie(BaseModel):
    title: str          # 필수 필드 (기본값 없음)
    year: int
    rating: float = 0.0   # 기본값이 있으면 선택 필드

# 키워드 인자로 생성
m = Movie(title="기생충", year=2019, rating=8.5)

# dict 로 받은 데이터는 ** 로 펼쳐서
data = {"title": "올드보이", "year": 2003}
m2 = Movie(**data)        # rating 은 기본값 0.0
print(m2)
# title='올드보이' year=2003 rating=0.0
# 일반 객체처럼 점(.) 으로 접근 — dict 의 [""] 보다 안전
print(m.title)      # 기생충
print(m.year)       # 2019

# 속성 수정도 가능 (기본적으로 이때도 검증된다)
m.rating = 9.0

# 모델을 dict / json 으로 되돌리기
m.model_dump()       # {'title': '기생충', 'year': 2019, ...}
m.model_dump_json()  # '{"title":"기생충","year":2019,...}'
# 잘못된 데이터를 넣으면 — 생성하는 순간 막힌다
from pydantic import ValidationError

try:
    Movie(title="인셉션", year="이천십")  # year 가 숫자 아님
except ValidationError as e:
    print(e)
# 1 validation error for Movie
# year
#   Input should be a valid integer ... [type=int_parsing]
핵심 감각 파이썬 @dataclass는 "필드를 편하게 적는" 도구일 뿐 검증은 하지 않습니다. Pydantic의 BaseModel은 같은 문법으로 적되, 객체를 만드는 순간 검증까지 합니다. 이 "생성 = 검증" 이 가장 큰 차이입니다.
02 : 타입 강제와 자동 변환 (Coercion)

타입 힌트는 '주석', Pydantic은 '검문'

파이썬의 타입 힌트(age: int)는 사람과 IDE를 위한 안내문일 뿐, 실행할 때 아무것도 강제하지 않습니다. 문자열을 넣어도 파이썬은 군말 없이 받습니다. Pydantic은 같은 힌트를 실제 규칙으로 바꿉니다.

✕ 그냥 타입 힌트
def f(age: int):
    return age

f("30")   # 아무 에러 없음
# '30'  ← 문자열 그대로 통과
# 힌트는 검사하지 않는다
✓ Pydantic
class P(BaseModel):
    age: int

P(age="30").age
# 30   ← int 로 자동 변환
P(age="서른")
# ValidationError (숫자 아님)

coercion똑똑한 변환 vs 무리한 변환

기본 모드(lax)에서 Pydantic은 합리적인 변환은 해 줍니다. "30" → 30, 3 → 3.0, "true" → True처럼요. 하지만 의미가 통하지 않는 변환("서른" → int)은 거부합니다. "엄격하게" 받고 싶다면 strict 모드를 쓸 수 있습니다.

▶ 변환 실험실 — 값을 입력하고 목표 타입을 골라보세요
자주 헷갈리는 지점 bool 변환은 직관과 다를 수 있습니다. Pydantic은 "true"/"yes"/"1"/1 을 True 로, "false"/"no"/"0"/0 을 False 로 봅니다. 하지만 "아무말" 같은 임의 문자열은 에러입니다. 환경변수(항상 문자열)를 다룰 때 특히 유용합니다.
03 : dict · dataclass · TypedDict 와의 비교

비슷해 보이는 네 가지, 무엇이 다른가

파이썬에는 "구조화된 데이터"를 표현하는 방법이 여럿입니다. 입문자가 가장 헷갈리는 부분이라 한 표로 정리합니다. 핵심 질문은 단 하나: "실행 시점에 값을 검사·변환해 주는가?"

기능 dict @dataclass TypedDict Pydantic
필드 이름 자동완성 ✕○△○
실행 시 타입 검증 ✕✕✕○
자동 타입 변환 ✕✕✕○
제약조건(범위/길이/정규식) ✕수작업✕○
JSON 직렬화/역직렬화 json 모듈수작업json 모듈내장
JSON 스키마 자동 생성 ✕✕✕○
속도(검증 포함) 빠름*빠름*빠름*빠름(Rust 코어)

* dict·dataclass·TypedDict 가 빠른 이유는 단순히 검증을 안 하기 때문입니다. "검증까지 포함"한 공정한 비교라면, Pydantic v2의 Rust 기반 코어(pydantic-core)가 손으로 짠 파이썬 검증 코드보다 보통 더 빠릅니다. 

from dataclasses import dataclass

@dataclass
class Point:
    x: int
    y: int

p = Point(x="왼쪽", y=None)   # 아무 불평 없음 😶
print(p)
# Point(x='왼쪽', y=None)  ← 쓰레기 값이 그대로 들어감
# 나중에 p.x + 1 할 때 비로소 터진다
from typing import TypedDict

class Point(TypedDict):
    x: int
    y: int

p: Point = {"x": "왼쪽", "y": None}
# 정적 타입검사기(mypy)는 경고하지만
# 실행하면 그냥 평범한 dict 일 뿐 — 막지 못한다
print(type(p))   # <class 'dict'>
# 데이터가 "내부에서만" 돌고, 신뢰할 수 있다면
  → dataclass (가볍다)

# 데이터가 "경계를 넘어" 들어온다면 (API/파일/LLM/env)
  → Pydantic (검증이 필요하다)

# 한 줄 기준:
# "이 데이터의 출처를 내가 100% 믿는가?"
# 아니라면 — Pydantic.
참고 dataclass를 이미 쓰고 있다면 pydantic.dataclasses.dataclass로 바꾸기만 해도 검증 기능을 얹을 수 있습니다. 기존 코드를 크게 안 바꾸고 검증만 더하고 싶을 때 좋은 다리입니다.
04 : Field 와 제약조건

타입만으로는 부족할 때 — 규칙을 덧붙이기

"나이는 int"만으로는 충분하지 않을 때가 많습니다. 음수면 안 되고, 150을 넘으면 이상하고, 이름은 비어 있으면 안 됩니다. 이런 세부 규칙은 Field()로 타입 옆에 덧붙입니다. 

from pydantic import BaseModel, Field

class Account(BaseModel):
    name: str   = Field(min_length=1, max_length=20)
    age: int    = Field(ge=0, le=150)     # 0 ≤ age ≤ 150
    balance: float = Field(gt=0)        # 0 보다 큼
    tag: str    = Field(pattern=r"^[a-z_]+$") # 정규식

Account(name="", age=-5, balance=0, tag="BAD!")
# 4개의 에러가 한 번에 모여서 보고된다:
#   name → too_short, age → greater_equal,
#   balance → greater_than, tag → string_pattern_mismatch
from datetime import datetime

class Job(BaseModel):
    status: str = "pending"           # 단순 기본값
    # ⚠ 리스트/딕트 같은 가변 기본값은 default_factory 로!
    tags: list[str] = Field(default_factory=list)
    created: datetime = Field(default_factory=datetime.now)

j1 = Job()
j2 = Job()
j1.tags.append("urgent")
print(j2.tags)   # []  ← j1 과 안 섞인다 (각자 새 리스트)
class Hyperparam(BaseModel):
    lr: float = Field(
        1e-3,
        gt=0, lt=1,
        description="학습률 (0~1)",   # 문서/스키마에 노출
        examples=[1e-3, 3e-4],
    )
    epochs: int = Field(10, ge=1, le=1000)

# 이 메타데이터는 FastAPI 문서·JSON 스키마로 그대로 전파된다
Hyperparam.model_json_schema()
파이썬 함정과 직접 연결 파이썬에서 def f(x=[]) 같은 가변 기본 인자는 모든 호출이 같은 리스트를 공유하는 악명 높은 버그입니다. Pydantic은 default_factory로 이 문제를 깔끔히 차단합니다 — 매번 새 객체를 만들어 줍니다.

Annotated현대적 표기 — 타입과 규칙을 한 곳에

최신 스타일에서는 Annotated를 써서 "타입 + 제약"을 하나로 묶습니다. 여러 모델에서 같은 규칙을 재사용하기 좋아집니다. 

from typing import Annotated
from pydantic import Field

# 0~1 사이 확률값 타입을 한 번 정의
Probability = Annotated[float, Field(ge=0, le=1)]

class Prediction(BaseModel):
    confidence: Probability      # 재사용
    dropout: Probability         # 재사용
05 : 검증 에러 다루기

실패는 '깔끔한 데이터'다 — ValidationError

검증이 실패하면 Pydantic은 ValidationError를 던집니다. 중요한 점은 이 에러가 단순한 문자열이 아니라, 구조화된 데이터라는 것입니다. 어느 필드가·왜·어떤 값 때문에 실패했는지를 프로그램이 읽어서 처리할 수 있습니다. API의 에러 응답을 만들 때 결정적입니다. 

from pydantic import BaseModel, ValidationError

class Signup(BaseModel):
    username: str
    age: int

try:
    Signup(username=123, age="열살")
except ValidationError as e:
    for err in e.errors():
        print(err["loc"], "→", err["type"])
# ('age',) → int_parsing
# username=123 은 "123" 으로 변환되어 통과(lax)
# e.errors() : 에러 리스트(dict)
# e.json()   : JSON 문자열로 한 방에
# 사용자에게 보여줄 한국어 메시지로 가공
FIELD_KR = {"username": "아이디", "age": "나이"}

def to_messages(e: ValidationError) -> list[str]:
    msgs = []
    for err in e.errors():
        field = err["loc"][0]
        name = FIELD_KR.get(field, field)
        msgs.append(f"{name}: {err['msg']}")
    return msgs
# → ["나이: Input should be a valid integer ..."]
왜 한 번에 모아 줄까 수작업 검증은 보통 첫 에러에서 멈춥니다. 사용자는 고치고, 또 다음 에러를 보고, 또 고치는 핑퐁을 반복하죠. Pydantic은 모든 필드를 검사해 에러를 한꺼번에 돌려줍니다. 폼 검증·API 검증에서 사용자 경험이 크게 좋아집니다.
06 : 커스텀 검증기

"숫자인지"를 넘어 "말이 되는지"까지

타입과 범위로 표현할 수 없는 규칙도 있습니다. "비밀번호와 확인값이 같아야 한다", "끝 날짜는 시작 날짜보다 뒤여야 한다" 같은 것이죠. 이럴 때 커스텀 검증 함수를 모델에 붙입니다. 

from pydantic import BaseModel, field_validator

class User(BaseModel):
    name: str

    @field_validator("name")
    @classmethod
    def strip_and_title(cls, v: str) -> str:
        v = v.strip()
        if not v:
            raise ValueError("이름은 공백일 수 없습니다")
        return v.title()      # 값을 가공해 반환 가능

User(name="  mina kim  ").name
# 'Mina Kim'  ← 정리 + 검증을 동시에
from pydantic import BaseModel, model_validator

class DateRange(BaseModel):
    start: int
    end: int

    @model_validator(mode="after")
    def check_order(self):
        if self.end < self.start:
            raise ValueError("end 는 start 이후여야 합니다")
        return self

DateRange(start=10, end=3)   # ValidationError
# before: 원본 입력을 "변환 전에" 손본다
@field_validator("tags", mode="before")
@classmethod
def split_csv(cls, v):
    if isinstance(v, str):
        return v.split(",")   # "a,b,c" → ["a","b","c"]
    return v

# after: 타입 변환이 끝난 "정제된 값"을 검사 (기본)
# 규칙: 입력을 고치려면 before, 결과를 확인하려면 after
ML 에서 특히 데이터셋의 라벨을 정규화("Yes"/"y"/"1" → True), 단위를 통일(ms → s), 카테고리 화이트리스트 검사 같은 전처리 규칙을 모델 안에 모아두면, 학습·추론·평가 어디서 데이터를 받아도 같은 규칙이 적용됩니다. 전처리 로직이 코드 곳곳에 흩어지는 것을 막아 줍니다.
07 : 중첩 · 복합 구조

모델 안에 모델 — 현실의 데이터는 깊다

실제 데이터는 평평하지 않습니다. 주문 안에 여러 상품이 있고, 사용자 안에 주소가 있죠. Pydantic은 모델을 레고처럼 중첩할 수 있고, 중첩된 부분까지 전부 검증합니다. 

class Address(BaseModel):
    city: str
    zipcode: str

class Customer(BaseModel):
    name: str
    address: Address          # 모델을 타입으로

# 중첩 dict 를 그대로 넘기면 안쪽도 자동 변환·검증
c = Customer(name="진수", address={"city": "서울", "zipcode": "04524"})
print(c.address.city)   # 서울 (Address 객체로 변환됨)
from typing import Optional, Union

class Order(BaseModel):
    items: list[str]            # 문자열 리스트
    coupon: Optional[str] = None # 있어도 되고 없어도 됨
    pay: Union[int, str]        # 둘 중 하나 (int 우선 시도)

# 최신 문법: Optional[str] == str | None
#           list[str] 처럼 소문자 제네릭 사용 권장
from typing import Literal
from enum import Enum

class Split(str, Enum):
    train = "train"
    valid = "valid"
    test  = "test"

class Sample(BaseModel):
    split: Split                       # 정해진 값만
    device: Literal["cpu", "cuda"] = "cpu"  # 가벼운 대안

Sample(split="valdi")   # 오타 → ValidationError
감각 Literal["cpu","cuda"]나 Enum을 쓰면 "허용된 값"이 코드와 문서·자동완성에 한꺼번에 박힙니다. ML 설정에서 옵티마이저·스케줄러·디바이스 같은 "고정된 선택지"를 다룰 때 오타 버그를 원천 차단합니다.
08 : 직렬화 / 역직렬화

모델 ⇄ JSON, 양방향 다리

ML 서비스는 결국 데이터를 주고받습니다. 들어올 때는 JSON → 모델(역직렬화), 나갈 때는 모델 → JSON(직렬화). Pydantic은 이 두 방향을 모두 한 줄로 처리합니다. 

# dict 로부터: model_validate
User.model_validate({"name": "수아", "age": 28})

# JSON 문자열로부터: model_validate_json (가장 빠름)
raw = '{"name": "수아", "age": "28"}'
u = User.model_validate_json(raw)   # 파싱+검증 한 번에
print(u.age, type(u.age))      # 28 <class 'int'>
# 모델 → dict
u.model_dump()
# {'name': '수아', 'age': 28}

# 모델 → JSON 문자열 (datetime 등도 알아서 직렬화)
u.model_dump_json(indent=2)
# {
#   "name": "수아",
#   "age": 28
# }
# 일부 필드만 / 비밀 필드 제외 / None 제거
u.model_dump(include={"name"})        # 화이트리스트
u.model_dump(exclude={"password"})    # 블랙리스트
u.model_dump(exclude_none=True)        # None 인 필드 빼기
u.model_dump(by_alias=True)            # 별칭(camelCase 등)으로

# 응답 페이로드를 상황별로 깎아낼 때 유용
v1 → v2 이름 바뀜 주의 오래된 글에서는 .dict(), .json(), parse_obj()를 씁니다. 이는 Pydantic v1 문법입니다. v2에서는 각각 model_dump(), model_dump_json(), model_validate()로 바뀌었습니다. 구버전 예제를 따라 하다 막히는 가장 흔한 원인입니다.
09 : 설정 관리 (Settings)

환경변수와 비밀값을, 검증된 객체로

ML 시스템은 수많은 설정으로 굴러갑니다. 모델 경로, GPU 개수, API 키, 배치 크기... 이를 os.environ["..."]로 흩어 읽으면 오타·누락·타입 문제가 산발합니다. pydantic-settings는 환경변수·.env 파일을 검증된 설정 객체로 모아 줍니다.

설치 v2에서 Settings 기능은 별도 패키지로 분리되었습니다 — pip install pydantic-settings. 
from pydantic_settings import BaseSettings
from pydantic import Field

class Settings(BaseSettings):
    model_path: str
    batch_size: int = Field(32, gt=0)
    use_gpu: bool = True

settings = Settings()
# 환경변수에서 자동으로 읽음 (대소문자 무시):
#   MODEL_PATH=/models/v3  BATCH_SIZE=64  USE_GPU=false
# "64"→64, "false"→False 로 변환+검증까지 끝.
from pydantic_settings import BaseSettings, SettingsConfigDict
from pydantic import SecretStr

class Settings(BaseSettings):
    model_config = SettingsConfigDict(env_file=".env")
    openai_key: SecretStr     # 로그에 값이 안 찍힘

s = Settings()
print(s.openai_key)                 # **********
print(s.openai_key.get_secret_value()) # 진짜 값은 명시적으로만
왜 좋은가 (12-Factor) 설정을 코드에서 분리하고 환경변수로 주입하는 것은 현대 배포의 표준입니다. Pydantic Settings는 여기에 타입 검증을 더합니다. BATCH_SIZE=abc처럼 잘못 주입하면 컨테이너가 뜨는 즉시 명확한 에러로 실패합니다 — 학습이 한참 진행된 뒤 터지는 것보다 훨씬 낫죠.
10 : 실전 · FastAPI 추론 API

모델을 API의 '입과 출력'으로

Pydantic이 ML 진영에서 폭발적으로 퍼진 계기가 바로 FastAPI입니다. FastAPI는 요청·응답 스키마를 Pydantic 모델로 선언하게 하고, 나머지(검증·에러응답·문서화)를 자동으로 처리합니다. 추론 서버를 만들 때 보일러플레이트가 극적으로 줄어듭니다. 

from fastapi import FastAPI
from pydantic import BaseModel, Field

class PredictIn(BaseModel):
    text: str = Field(min_length=1, max_length=2000)
    top_k: int = Field(3, ge=1, le=20)

class PredictOut(BaseModel):
    label: str
    score: float

app = FastAPI()

@app.post("/predict", response_model=PredictOut)
def predict(req: PredictIn):       # 들어오는 순간 검증 완료
    # req.text, req.top_k 는 이미 안전한 타입
    return {"label": "긍정", "score": 0.97}
# 위 30 줄로 공짜로 얻는 것들:
#  ✓ 잘못된 요청 → 자동 422 응답 (어느 필드가 왜 틀렸는지 JSON 으로)
#  ✓ /docs  : 인터랙티브 Swagger UI 문서 자동 생성
#  ✓ OpenAPI 스키마 자동 생성 → 클라이언트 SDK 생성 가능
#  ✓ 응답도 PredictOut 모양으로 검증 후 직렬화
#
# 직접 짰다면: 요청 파싱 + 검증 + 에러포맷 + 문서 = 수백 줄
현장 패턴 요청 모델(PredictIn)과 응답 모델(PredictOut)을 분리하세요. 내부 처리용 모델과 외부 노출용 모델을 나누면, 민감 정보 유출을 막고 API 계약을 안정적으로 유지할 수 있습니다.
11 : 실전 · LLM 구조화 출력

"제발 JSON으로만 답해줘"의 정석

LLM의 출력은 본질적으로 자유로운 텍스트입니다. 하지만 우리는 종종 정해진 모양의 데이터가 필요합니다 — 분류 결과, 추출된 엔티티, 평가 점수 같은 것들요. Pydantic 모델을 "원하는 출력의 스키마"로 LLM에 건네면, 그 모양으로 받아 검증까지 할 수 있습니다. 이것이 LLMOps에서 Pydantic이 핵심이 된 이유입니다. 

from pydantic import BaseModel, Field
from typing import Literal

class Review(BaseModel):
    """리뷰 한 건의 분석 결과."""
    sentiment: Literal["positive", "negative", "neutral"]
    score: float = Field(ge=0, le=1, description="확신도")
    keywords: list[str] = Field(max_length=5)

# 이 모델 자체가 LLM 에게 줄 "출력 명세서"가 된다.
# docstring·description 은 프롬프트로 전달되어 품질을 높인다.
from langchain_openai import ChatOpenAI

llm = ChatOpenAI(model="gpt-4o-mini")
structured = llm.with_structured_output(Review)

result = structured.invoke("배송 느렸지만 제품은 최고예요!")
print(result.sentiment)   # 'positive'
print(type(result))       # <class 'Review'> ← 검증된 객체!
# LLM 이 형식을 어기면 재시도/에러로 잡힌다
# Pydantic AI — Pydantic 팀이 만든 에이전트 프레임워크
from pydantic_ai import Agent

agent = Agent("openai:gpt-4o-mini", output_type=Review)
out = agent.run_sync("배송 느렸지만 제품은 최고예요!")
print(out.output)        # Review 객체

# instructor 라이브러리도 같은 철학:
#   client.chat.completions.create(..., response_model=Review)
왜 게임체인저인가 Pydantic 모델은 JSON 스키마로 자동 변환됩니다. 라이브러리들은 이 스키마를 LLM의 function calling / structured output 기능에 그대로 먹입니다. 즉, 한 번 정의한 모델이 ① 프롬프트 명세 ② 출력 파싱 ③ 검증 ④ 타입 안전성을 동시에 책임집니다. "텍스트를 정규식으로 긁어 파싱"하던 시대를 끝낸 것이죠.

habit몸에 붙이면 좋은 습관

  • 경계에서만 검증. 데이터가 시스템에 처음 들어오는 곳(API·파일·LLM·env)에서 한 번 검증하고, 내부에서는 검증된 모델 객체를 신뢰하며 넘깁니다. 함수마다 재검증하지 마세요.
  • 입력/출력 모델 분리. 같은 엔티티라도 받을 때와 내보낼 때 모양이 다릅니다. 섞지 마세요.
  • 가변 기본값엔 default_factory. 리스트·딕트·datetime 기본값은 무조건 factory로.
  • 제약은 Field·Annotated로, 의미 규칙은 validator로. 역할을 나누면 읽기 쉽습니다.
  • 성능 걱정은 대부분 기우. v2 코어는 Rust(pydantic-core)라 충분히 빠릅니다. 정말 뜨거운 루프에서 같은 데이터를 수백만 번 재검증하지만 않으면 됩니다.
한 문장으로 기억하기 거친 데이터가 관문(BaseModel)을 지나면 믿을 수 있는 타입이 된다. — 이 한 장면만 기억하면, 나머지 기능은 "그 관문을 어떻게 더 정교하게 만들 것인가"의 변주일 뿐입니다.
