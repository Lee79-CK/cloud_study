CHAPTER 01

네트워크 기초 다지기

IP 주소, CIDR, 서브넷 — AWS를 배우기 전에 꼭 필요한 최소한의 기초

1.1 IP 주소: 네트워크 세계의 주소 체계

IP 주소는 네트워크에 연결된 장비를 식별하는 고유한 번호입니다. IPv4 주소는 192.168.10.25처럼 0~255 사이의 숫자 4개를 점으로 구분해 표현하며, 내부적으로는 32비트(bit) 이진수입니다.

🏢 비유로 이해하기 — 아파트 주소

IP 주소를 아파트 주소라고 생각해 보세요. "서울시 강남구 테헤란로 아파트 101동 1204호"에서 앞부분(시/구/동)은 네트워크 부분, 뒷부분(호수)은 호스트 부분에 해당합니다.

같은 동(같은 네트워크)에 사는 이웃끼리는 엘리베이터로 바로 만날 수 있지만(로컬 통신), 다른 동으로 가려면 단지 정문(라우터/게이트웨이)을 거쳐야 합니다.

1.2 CIDR: IP 대역을 표현하는 표기법

CIDR(Classless Inter-Domain Routing, "사이더"라고 읽습니다)는 IP 주소의 범위(대역)를 간결하게 표현하는 방법입니다. 10.0.0.0/16에서 /16은 "앞의 16비트가 네트워크 부분으로 고정된다"는 뜻입니다.

  • / 뒤의 숫자가 작을수록 → 고정되는 부분이 적음 → 더 넓은 대역 (IP가 많음)
  • / 뒤의 숫자가 클수록 → 고정되는 부분이 많음 → 더 좁은 대역 (IP가 적음)
  • 사용 가능한 IP 개수 = 2(32 − prefix). 예: /24 → 28 = 256개

1.3 서브넷: 네트워크를 쪼개는 이유

서브넷(Subnet)은 큰 네트워크(VPC)를 작은 구역으로 나눈 것입니다. 왜 굳이 나눌까요? 크게 세 가지 이유가 있습니다.

🛡️
보안 경계

인터넷에 노출되어도 되는 구역과, 절대 노출되면 안 되는 구역(DB 등)을 물리적으로 분리합니다.

🏥
가용성(AZ) 분산

AWS에서 서브넷은 하나의 가용 영역(AZ)에 속합니다. 여러 AZ에 서브넷을 만들어 장애에 대비합니다.

📐
트래픽 제어

서브넷 단위로 라우팅 규칙을 다르게 적용해 "이 구역은 인터넷 가능, 저 구역은 불가"를 설계합니다.

🏬 비유로 이해하기 — 백화점의 구역 설계

VPC가 백화점 건물 전체라면, 서브넷은 층/구역입니다. 1층 매장(Public 서브넷)은 정문으로 누구나 드나들 수 있지만, 지하 금고와 물류창고(Private 서브넷)는 직원 전용 통로로만 접근할 수 있습니다. 건물은 하나지만 구역마다 출입 규칙이 다르죠. 그 "출입 규칙"이 바로 다음 장에서 배울 라우팅 테이블입니다.

QUIZ 1
Q. 10.0.1.0/24 서브넷에서 EC2 인스턴스에 실제로 할당할 수 있는 IP는 몇 개일까요?
정답: 251개. 일반 네트워크에서는 2개(네트워크/브로드캐스트)만 빼지만, AWS는 VPC 라우터·DNS·예비 용도로 총 5개를 예약합니다. 이 차이는 실무 IP 설계에서 자주 등장하는 함정입니다.
CHAPTER 02

VPC 핵심 구성 요소

AWS 네트워크의 뼈대를 이루는 7가지 부품

VPC(Virtual Private Cloud)는 AWS 안에 만드는 나만의 격리된 가상 네트워크입니다. 같은 리전의 다른 고객과 완전히 분리되어 있으며, IP 대역·서브넷·라우팅·보안을 모두 내 마음대로 설계할 수 있습니다. VPC를 구성하는 핵심 부품들을 하나씩 살펴봅시다.

그림 2-1. 2개 AZ에 걸친 표준 VPC 구조 — 이 그림 하나가 이 강의 전체의 지도입니다

2.1 일곱 가지 핵심 부품

① VPC — 나만의 가상 네트워크
리전 단위로 생성하는 격리된 네트워크 공간입니다. 생성 시 CIDR 대역(/16 ~ /28)을 지정하며, 이후 확장이 번거로우므로 처음부터 넉넉하게 /16을 잡는 것이 관례입니다. 특히 EKS를 쓸 계획이라면 Pod들이 IP를 대량 소비하므로 더욱 그렇습니다.
② Subnet — VPC를 나눈 구역 (AZ에 종속)
VPC CIDR의 부분집합으로 만드는 작은 대역입니다. 중요한 특징: 하나의 서브넷은 정확히 하나의 AZ에만 속합니다. 고가용성을 위해 최소 2개 AZ에 같은 역할의 서브넷을 쌍으로 만드는 것이 표준입니다.
③ Route Table — 트래픽의 내비게이션
"목적지가 X이면 Y로 보내라"는 규칙의 목록입니다. 모든 서브넷은 반드시 하나의 라우팅 테이블에 연결되며, 이 테이블의 내용이 서브넷의 성격(Public/Private)을 결정합니다. 3장에서 집중적으로 다룹니다.
④ Internet Gateway (IGW) — 인터넷으로 통하는 정문
VPC와 인터넷을 연결하는 관문입니다. VPC당 1개만 연결할 수 있으며, 양방향(인터넷→VPC, VPC→인터넷) 통신을 모두 지원합니다. 수평 확장되는 관리형 서비스라 병목이나 장애 지점이 되지 않습니다.
⑤ NAT Gateway — 나갈 수만 있는 뒷문
Private 서브넷의 리소스가 인터넷으로 나가기만(Outbound) 할 수 있게 해주는 관문입니다. 외부에서 먼저 들어오는 것은 불가능합니다. Public 서브넷에 위치하며 Elastic IP가 필요합니다. 시간당 요금 + 데이터 처리 요금이 발생하는 유료 서비스입니다.
⑥ Security Group / NACL — 방화벽 2중 구조
Security Group은 인스턴스(ENI) 단위, NACL은 서브넷 단위의 방화벽입니다. 5장에서 상세히 비교합니다.
⑦ VPC Endpoint — AWS 서비스로 가는 전용 통로
S3, ECR, DynamoDB 같은 AWS 서비스에 인터넷을 거치지 않고 접근하게 해주는 통로입니다. Private 서브넷에서 NAT Gateway 비용을 아끼면서 AWS 서비스를 쓰는 핵심 수단으로, EKS 아키텍처(7장)에서 중요하게 등장합니다.
✅
실무 기본 공식: "2 AZ × (Public + Private) = 서브넷 4개" 대부분의 프로덕션 VPC는 최소 2개 AZ에 Public/Private 서브넷을 한 쌍씩, 총 4개로 시작합니다. DB 계층까지 분리하면 6개(Public 2 + Private App 2 + Private DB 2)가 됩니다. 이 패턴을 기억해 두면 어떤 아키텍처 다이어그램도 쉽게 읽힙니다.
QUIZ 2
Q. 하나의 서브넷은 몇 개의 가용 영역(AZ)에 걸칠 수 있을까요?
정답: 정확히 1개. 서브넷은 AZ에 종속됩니다. 그래서 고가용성 설계에서는 "AZ마다 같은 역할의 서브넷을 하나씩" 만들어야 합니다. AZ 하나가 통째로 장애가 나도 다른 AZ의 서브넷이 살아있게 하기 위함입니다.
CHAPTER 03

Public vs Private 서브넷 — 핵심 비교

이 강의의 심장부. 단 한 줄의 라우팅 규칙이 만드는 결정적 차이

3.1 충격적일 만큼 단순한 진실

많은 입문자가 "Public 서브넷"과 "Private 서브넷"이 AWS 콘솔에서 체크박스로 선택하는 별개의 상품이라고 오해합니다. 그렇지 않습니다. 두 서브넷은 태어날 때는 완전히 동일하며, 유일한 차이는 연결된 라우팅 테이블의 단 한 줄입니다.

🔑
정의 (이것이 전부입니다) Public 서브넷 = 라우팅 테이블에 0.0.0.0/0 → igw-xxxx 경로가 있는 서브넷
Private 서브넷 = 라우팅 테이블에 IGW로 가는 경로가 없는 서브넷 (보통 0.0.0.0/0 → nat-xxxx)
🏨 비유로 이해하기 — 호텔의 로비층과 직원 전용 구역

Public 서브넷은 호텔 로비입니다. 정문(IGW)과 직접 연결되어 있어 외부 손님이 걸어 들어올 수 있고, 안에 있는 사람도 정문으로 바로 나갈 수 있습니다.

Private 서브넷은 주방과 사무실입니다. 정문과 직접 연결된 복도가 없어 손님은 절대 들어올 수 없습니다. 하지만 직원은 물품 반입용 뒷문(NAT Gateway)을 통해 밖으로 나가 재료를 사올 수 있죠. 단, 이 뒷문은 밖에서는 열 수 없는 일방향 문입니다.

3.2 인터랙티브 데모: 라우팅 한 줄이 바꾸는 운명

🔀 서브넷 타입 전환 시뮬레이터
버튼을 눌러 같은 서브넷이 라우팅 규칙에 따라 어떻게 성격이 바뀌는지 확인해 보세요.
🌐
Public 모드 라우팅 테이블에 0.0.0.0/0 → IGW 경로가 있습니다. EC2에 공인 IP만 붙어 있다면 인터넷에서 직접 들어올 수도(Inbound), 직접 나갈 수도(Outbound) 있습니다. 웹서버, 로드밸런서, Bastion Host에 적합합니다.

3.4 자주 하는 오해 바로잡기

오해 ① "Private 서브넷은 인터넷을 아예 못 쓴다?"
아닙니다. 들어오는 것이 불가능할 뿐, NAT Gateway를 연결하면 나가는 것은 자유롭습니다. OS 패치, 외부 API 호출, 컨테이너 이미지 다운로드는 모두 Private 서브넷에서 일상적으로 일어납니다. NAT조차 없는 완전 폐쇄망(Isolated Subnet)은 별도의 설계 선택입니다.
오해 ② "공인 IP를 붙이면 무조건 인터넷에서 접근 가능하다?"
아닙니다. Private 서브넷의 인스턴스에 공인 IP를 억지로 붙여도, 라우팅 테이블에 IGW 경로가 없으면 그 IP는 장식품입니다. 응답 패킷이 인터넷으로 돌아갈 길이 없기 때문입니다. "공인 IP + IGW 라우팅 + SG 허용" 3박자가 모두 맞아야 통신이 성립합니다.
오해 ③ "Public 서브넷의 인스턴스는 항상 위험하다?"
Public 서브넷에 있어도 공인 IP를 할당하지 않으면 인터넷에서 직접 접근할 수 없습니다. 또한 Security Group이 기본적으로 모든 Inbound를 차단하므로, "Public 서브넷 = 즉시 노출"은 아닙니다. 다만 실수 한 번(IP 자동할당 + SG 0.0.0.0/0 오픈)으로 노출될 수 있는 위험 구역이므로, 꼭 필요한 리소스만 두는 것이 원칙입니다.
오해 ④ "RDS를 Public 서브넷에 두고 SG로만 막으면 되지 않나?"
기술적으로는 가능하지만 절대 권장하지 않습니다. 방어는 여러 겹(Defense in Depth)이어야 합니다. SG 설정 실수 한 번이 곧바로 DB 노출로 이어지는 구조 대신, Private 서브넷에 두면 라우팅 계층에서 이미 한 겹이 막혀 있어 실수의 파급이 줄어듭니다. 컴플라이언스(ISMS-P, PCI-DSS 등) 심사에서도 DB의 Private 배치는 사실상 필수입니다.
QUIZ 3
Q. 서브넷을 "Public"으로 만드는 결정적 요소는 무엇일까요?
정답: 라우팅 테이블의 IGW 경로. AWS에는 서브넷 타입을 고르는 옵션 자체가 없습니다. 공인 IP 할당은 개별 인스턴스의 속성일 뿐이며, IGW 경로가 없으면 공인 IP가 있어도 통신이 불가능합니다.

라우팅, IGW, NAT의 이해

패킷 하나가 인터넷까지 갔다 오는 여정을 추적한다

4.1 라우팅 테이블 읽는 법

라우팅 테이블은 "목적지(Destination)가 이 대역이면, 이 관문(Target)으로 보내라"는 규칙의 목록입니다. 실제 테이블을 읽어 봅시다.

DestinationTarget해석
10.0.0.0/16localVPC 내부끼리의 통신. 모든 라우팅 테이블에 자동 포함되며 삭제 불가
0.0.0.0/0igw-0abc..."그 외 모든 목적지"는 인터넷 게이트웨이로 → Public 서브넷의 증표
0.0.0.0/0nat-0def..."그 외 모든 목적지"는 NAT 게이트웨이로 → Private 서브넷의 전형
pl-xxxx (S3)vpce-...S3로 가는 트래픽은 VPC 엔드포인트로 (인터넷 우회 없이)
💡
가장 구체적인 경로가 이긴다 (Longest Prefix Match) 목적지가 여러 규칙에 동시에 해당하면, 프리픽스가 가장 긴(구체적인) 규칙이 적용됩니다. 예를 들어 10.0.5.4로 가는 패킷은 0.0.0.0/0보다 10.0.0.0/16 → local에 매칭됩니다. /16이 /0보다 구체적이기 때문입니다.

4.3 패킷의 여정 따라가기

시나리오 A — Private 서브넷의 EC2가 외부 API를 호출할 때

EC2(10.0.10.5)가 api.example.com(203.0.113.9)으로 요청 전송. 목적지가 VPC 밖이므로 라우팅 테이블 조회.
0.0.0.0/0 → nat-xxxx 규칙에 매칭 → 패킷이 Public 서브넷의 NAT Gateway로 이동.
NAT GW가 출발지 주소를 10.0.10.5 → NAT의 Elastic IP로 바꿔치기(Source NAT)하고, 이 연결을 변환 테이블에 기록.
패킷은 IGW를 거쳐 인터넷으로. 외부 API 입장에서는 NAT의 공인 IP에서 온 요청으로 보임.
응답이 NAT의 공인 IP로 돌아오면, NAT GW가 변환 테이블을 보고 원래 주인인 10.0.10.5에게 전달. 외부가 먼저 시작한 연결은 테이블에 없으므로 폐기 — 이것이 "일방향"의 정체입니다.

시나리오 B — 인터넷 사용자가 Public 서브넷의 ALB에 접속할 때

사용자가 ALB의 공인 IP(DNS로 해석됨)로 HTTPS 요청 → IGW 도착.
IGW가 공인 IP를 ALB의 사설 IP로 변환해 Public 서브넷으로 전달. NACL과 SG 검사를 통과.
ALB가 local 라우팅(10.0.0.0/16 → local)으로 Private 서브넷의 대상(EC2/Pod)에 요청을 전달 — 이 구간은 인터넷을 전혀 거치지 않는 VPC 내부 통신입니다.
응답은 왔던 길을 역순으로. 사용자는 Private 서브넷의 존재조차 알 수 없습니다.
QUIZ 4
Q. NAT Gateway 뒤의 Private EC2에 인터넷에서 SSH 접속을 시도하면 어떻게 될까요?
정답: 실패. NAT GW는 내부→외부로 시작된 연결을 변환 테이블에 기록하고, 그 연결의 응답만 내부로 통과시킵니다. 외부에서 먼저 시작한 연결은 테이블에 없으므로 버려집니다. SG를 열어도 패킷이 EC2까지 도달할 방법 자체가 없습니다. Private EC2 접속은 Bastion Host나 SSM Session Manager를 사용합니다.
CHAPTER 05

보안 계층: Security Group vs NACL

라우팅이 "길"이라면, 보안 그룹과 NACL은 그 길 위의 "검문소"

라우팅 테이블이 트래픽의 경로를 결정한다면, Security Group(SG)과 Network ACL(NACL)은 그 경로 위에서 트래픽을 허용할지 차단할지 결정합니다. 같은 방화벽이지만 작동 위치와 방식이 다릅니다.

🏢 비유로 이해하기 — 건물 출입 시스템

NACL은 건물(서브넷) 1층의 공용 게이트입니다. 층 전체에 일괄 적용되고, 출입 명단(허용)과 블랙리스트(거부)를 모두 운영합니다. 들어갈 때와 나올 때 매번 검사합니다(Stateless).

Security Group은 각 사무실(인스턴스) 문 앞의 지문 인식기입니다. 허용 명단만 존재하고, 한 번 정상 출입한 사람의 퇴장은 다시 검사하지 않습니다(Stateful).

항목Security GroupNetwork ACL
적용 단위인스턴스(정확히는 ENI) 단위서브넷 단위
상태 추적Stateful — 허용된 요청의 응답은 자동 허용Stateless — 요청과 응답을 각각 검사
규칙 유형Allow(허용)만 가능Allow + Deny(명시적 거부) 모두 가능
평가 방식모든 규칙을 종합 평가규칙 번호가 낮은 순서대로, 먼저 매칭되면 종료
기본 동작Inbound 전부 차단 / Outbound 전부 허용기본 NACL은 전부 허용 / 커스텀은 전부 차단
주 용도일상적인 접근 제어의 주력 수단특정 IP 차단, 서브넷 차원의 안전망
✅
실무 황금 패턴: SG 참조(Chaining) SG의 소스에는 IP뿐 아니라 다른 SG를 지정할 수 있습니다. 예: DB의 SG에 "WAS의 SG에서 오는 3306만 허용". 이렇게 하면 WAS 인스턴스가 늘어나거나 IP가 바뀌어도 규칙을 수정할 필요가 없습니다. IP 대신 역할(role)로 보안을 설계하는 것 — 이것이 클라우드다운 방화벽 설계입니다.

Stateless가 만드는 함정: Ephemeral Port

NACL로 "80번 포트 Inbound만 허용"하고 Outbound를 잠그면 웹서버가 응답을 못 보냅니다. 클라이언트는 요청을 보낼 때 임시 포트(1024~65535)로 응답을 받기 때문에, Stateless인 NACL에서는 Outbound에서 임시 포트 대역을 열어줘야 응답이 나갈 수 있습니다. Stateful인 SG에서는 이런 고민이 필요 없습니다. 이 차이 때문에 실무에서는 SG를 주력으로 쓰고 NACL은 기본값(전부 허용) 그대로 두거나 특정 IP 차단용 안전망으로만 쓰는 경우가 많습니다.

QUIZ 5
Q. Security Group에서 Inbound 80 포트를 허용했습니다. 웹서버의 응답이 나가려면 Outbound 규칙을 어떻게 해야 할까요?
정답: 아무것도 안 해도 됩니다. SG는 Stateful이므로 허용된 Inbound 연결의 응답 트래픽은 Outbound 규칙과 무관하게 자동으로 나갑니다. 임시 포트를 신경 써야 하는 것은 Stateless인 NACL의 이야기입니다.
CHAPTER 06

EC2 기반 3-Tier 아키텍처 설계

배운 부품들을 조립해 실제 서비스 네트워크를 만든다

3-Tier 아키텍처는 웹 서비스의 가장 고전적이면서도 여전히 유효한 설계 패턴입니다. Web(접점) → App(로직) → DB(데이터)의 세 계층을 네트워크로 분리하여, 바깥 계층이 뚫려도 안쪽 계층이 바로 노출되지 않게 합니다.

그림 6-1. EC2 기반 3-Tier 표준 아키텍처 — 계층이 내려갈수록 격리가 강해진다

6.1 계층별 설계 의도

Tier 1 (Public) — 인터넷과 닿는 유일한 계층. ALB는 두 AZ의 Public 서브넷에 걸쳐 배치되어 트래픽을 받아 App 계층으로 넘깁니다. NAT GW와 Bastion도 이곳에 삽니다. 애플리케이션 서버는 절대 이 계층에 두지 않습니다.
Tier 2 (Private App) — 실제 비즈니스 로직이 도는 곳. Auto Scaling Group으로 두 AZ에 분산 배치. Inbound는 ALB의 SG에서 오는 트래픽만, Outbound는 NAT GW를 통해서만 인터넷에 나갑니다(패키지 설치, 외부 API 호출).
Tier 3 (Private DB) — 가장 깊은 격리 계층. 라우팅 테이블에 NAT 경로조차 없는 완전 폐쇄(Isolated) 구성이 이상적입니다. DB는 인터넷에 나갈 일이 없기 때문입니다. Inbound는 App 계층 SG에서 오는 DB 포트만 허용합니다.

6.2 이 설계가 방어하는 시나리오

  • 웹 취약점 공격 — 공격자가 ALB까지는 도달해도, App 서버의 사설 IP를 직접 겨냥할 수 없습니다.
  • App 서버 침해 — 최악의 경우 App 서버가 장악되어도, DB 서브넷은 App-SG에서 오는 DB 포트만 허용하므로 측면 이동(lateral movement)이 크게 제한됩니다.
  • AZ 장애 — 한 AZ가 통째로 죽어도 다른 AZ의 ALB 노드·EC2·RDS Standby가 서비스를 이어갑니다. NAT GW를 AZ당 하나씩 둔 것도 같은 이유입니다.
  • 운영자 실수 — SG 하나를 잘못 열어도 라우팅 계층의 격리가 남아 있어 단일 실수가 곧바로 사고로 이어지지 않습니다.
🔐
Bastion 대신 SSM Session Manager 전통적으로는 Public 서브넷에 Bastion Host(점프 서버)를 두고 SSH로 접근했지만, 최근에는 AWS Systems Manager Session Manager가 표준입니다. 인바운드 포트를 하나도 열지 않고, 공인 IP도 없이, IAM 권한만으로 Private EC2에 셸 접속이 가능하며 모든 세션이 감사 로그로 남습니다. 신규 설계라면 Bastion 없이 시작하는 것을 권장합니다.
QUIZ 6
Q. 3-Tier 설계에서 DB 서브넷의 라우팅 테이블에 NAT Gateway 경로조차 두지 않는 이유는?
정답: 공격 표면 최소화. 만에 하나 DB가 침해되어도 데이터를 외부로 유출할 경로(Outbound) 자체가 없다면 피해를 크게 줄일 수 있습니다. "필요 없는 길은 만들지 않는다"는 최소 권한 원칙의 네트워크 버전입니다.
CHAPTER 07

EKS 네트워크 아키텍처

Kubernetes가 VPC 위에 올라오면 네트워크는 어떻게 달라지는가

7.1 EKS 네트워크의 큰 그림

EKS는 크게 두 부분으로 나뉩니다. AWS가 관리하는 컨트롤 플레인(API 서버, etcd)과 내 VPC에서 도는 데이터 플레인(워커 노드와 Pod)입니다. 컨트롤 플레인은 AWS 소유의 별도 VPC에 살고 있으며, 내 VPC와는 ENI(탄력적 네트워크 인터페이스)를 통해 연결됩니다. 그래서 EKS 클러스터를 만들 때 "서브넷을 지정하라"고 하는 것입니다 — 그 서브넷에 컨트롤 플레인의 ENI가 꽂히기 때문입니다.

그림 7-1. EKS 네트워크 구조 — 컨트롤 플레인은 ENI로, Pod는 VPC IP로 연결된다

7.2 VPC CNI: EKS 네트워크의 최대 특징

일반적인 Kubernetes(예: 온프레미스의 Calico, Flannel)에서는 Pod가 노드 내부의 가상(오버레이) 네트워크 IP를 받습니다. 하지만 EKS의 기본 CNI인 Amazon VPC CNI는 다릅니다 — Pod가 VPC의 실제 IP를 직접 할당받습니다.

✅
장점: Pod가 1급 시민

Pod IP가 VPC IP이므로 RDS, EC2, 온프레미스(VPN 경유)와 오버레이 변환 없이 직접 통신합니다. SG를 Pod 단위로 붙이는 것(Security Groups for Pods)도 가능합니다.

⚠️
단점: IP 대량 소모

Pod 하나가 VPC IP 하나를 먹습니다. 노드 100대 × Pod 30개 = IP 3,000개. /24 서브넷(251개)로는 어림도 없습니다. EKS 서브넷은 /20~/19처럼 크게 잡아야 합니다.

🔢
노드당 Pod 수 제한

인스턴스 타입별 ENI 수 × ENI당 IP 수가 노드의 최대 Pod 수를 결정합니다. 예: m5.large는 기본 29개. Prefix Delegation을 켜면 이 한계를 크게 늘릴 수 있습니다.

🧮
EKS IP 고갈은 실제로 자주 겪는 장애입니다 "Pod가 Pending에서 멈춰요" 문의의 단골 원인이 서브넷 IP 고갈입니다. 예방책: ① 서브넷을 /20 이상으로 크게, ② Prefix Delegation 활성화(ENI당 /28 프리픽스 단위 할당), ③ 부족해지면 VPC에 Secondary CIDR(예: 100.64.0.0/16)을 추가하고 Pod 전용 서브넷으로 분리(Custom Networking). MLOps처럼 학습 Job이 수백 Pod를 일시에 띄우는 환경이라면 처음부터 여유 있게 설계하세요.

7.3 클러스터 엔드포인트: Public / Private / 혼합

"EKS의 Public/Private"이라고 하면 서브넷 외에 하나 더 있습니다 — API 서버 엔드포인트의 공개 범위입니다. kubectl이 접속하는 그 주소를 어디에 노출할지 정하는 설정입니다.

모드kubectl 접근 경로노드 → API 서버권장 대상
Public인터넷 어디서나 (IP 제한 가능)인터넷 경유 (NAT/IGW 필요)학습/테스트 환경
Public + Private인터넷 + VPC 내부 모두VPC 내부 ENI로 직접과도기, 일반적 시작점
PrivateVPC 내부에서만 (VPN/Bastion/SSM 필요)VPC 내부 ENI로 직접프로덕션 표준, 금융/공공 폐쇄망

7.4 서비스 노출: Pod는 Private에, LB는 Public에

EKS의 표준 트래픽 흐름은 3-Tier의 철학을 그대로 계승합니다. 워커 노드와 Pod는 전부 Private 서브넷에 두고, 외부 노출은 AWS Load Balancer Controller가 Public 서브넷에 만들어주는 ALB/NLB가 담당합니다.

개발자가 Ingress 리소스를 배포하면, LB Controller가 이를 감지해 Public 서브넷에 ALB를 자동 생성합니다. 어느 서브넷에 만들지는 서브넷의 태그로 판단합니다.
태그 규칙: Public 서브넷에는 kubernetes.io/role/elb = 1, Private 서브넷에는 kubernetes.io/role/internal-elb = 1. 이 태그가 없으면 "서브넷을 찾을 수 없다"는 에러로 LB 생성이 실패합니다 — EKS 입문자가 가장 자주 만나는 함정입니다.
ALB는 IP 타겟 모드로 Private 서브넷의 Pod IP에 직접 트래픽을 전달합니다(VPC CNI 덕분에 가능). 사용자 → ALB(Public) → Pod(Private)의 흐름이 완성됩니다.

7.5 VPC Endpoint로 NAT 비용 잡기

Private 서브넷의 노드는 컨테이너 이미지를 받을 때(ECR), 로그를 보낼 때(CloudWatch), IAM 인증을 할 때(STS) 기본적으로 NAT Gateway를 경유합니다. 노드가 많아질수록 이 비용이 커지는데, VPC Endpoint를 만들면 이 트래픽이 AWS 내부망으로 직행합니다.

  • Gateway Endpoint (S3, DynamoDB) — 라우팅 테이블에 경로가 추가되는 방식. 무료이므로 무조건 만드는 것이 이득입니다.
  • Interface Endpoint (ECR, STS, CloudWatch 등) — 서브넷에 ENI가 생기는 방식. 시간당 요금이 있지만 NAT 처리 요금보다 저렴한 경우가 많고, 완전 폐쇄망 구성의 필수 요소입니다.
  • EKS 폐쇄망(Private Only) 클러스터의 최소 세트: ecr.api, ecr.dkr, s3(Gateway), sts, ec2, logs
QUIZ 7
Q. EKS에서 Ingress를 배포했는데 ALB가 생성되지 않고 "couldn't auto-discover subnets" 에러가 발생합니다. 가장 유력한 원인은?
정답: 서브넷 태그 누락. AWS Load Balancer Controller는 태그(kubernetes.io/role/elb=1 또는 internal-elb=1)로 LB를 배치할 서브넷을 자동 탐색합니다. eksctl로 만들면 자동으로 붙지만, Terraform이나 수동 생성 VPC에서는 직접 태깅해야 합니다. 실무에서 매우 흔한 트러블슈팅 케이스입니다.

중요한 사실부터: EKS도 결국 VPC 위에서 돕니다. Public/Private 서브넷, IGW, NAT, SG의 원리는 완전히 동일합니다. 달라지는 것은 그 위에 얹히는 추상화 계층과 IP 소비 패턴, 그리고 누가 네트워크 리소스를 만드느냐입니다.

8.1 그래도 변하지 않는 것

  • "노출은 Public, 워크로드는 Private" 원칙 — EC2든 Pod든 동일하게 적용됩니다.
  • 라우팅 테이블이 서브넷의 성격을 정한다 — EKS 노드가 인터넷에 나가는 것도 결국 NAT 경로 덕분입니다.
  • SG 참조 패턴 — "노드 SG → RDS SG 허용"처럼 역할 기반 설계는 그대로 유효합니다.
  • Multi-AZ 분산 — 노드 그룹을 여러 AZ의 서브넷에 걸치는 것 역시 같은 이유(가용성)입니다.

8.2 어떤 것을 선택할까?

🖥️
EC2 기반이 맞는 경우

단일 모놀리식 앱, 소수의 서비스, 컨테이너 운영 경험이 없는 팀, 라이선스가 인스턴스에 묶인 상용 SW, 빠르게 단순한 구조가 필요할 때.

☸️
EKS 기반이 맞는 경우

마이크로서비스 다수, 배포 빈도가 높음, MLOps/LLMOps 파이프라인(Kubeflow, Airflow, vLLM 서빙 등), 멀티 팀이 하나의 플랫폼을 공유, 오토스케일링이 격렬한 워크로드.

🎯
면접 단골 질문으로 정리하면 "EC2와 EKS의 네트워크 차이를 설명하세요"라는 질문에 대한 모범 답변 뼈대: ① 둘 다 동일한 VPC 원리 위에서 동작하지만, ② EKS는 VPC CNI로 Pod가 VPC IP를 직접 소비하므로 IP 설계 규모가 달라지고, ③ LB·ENI 같은 네트워크 리소스를 컨트롤러가 자동 생성하므로 태그 기반 규약이 중요해지며, ④ API 엔드포인트 공개 범위라는 추가적인 Public/Private 결정이 생긴다.

IaC 실습: 직접 만들어보기

9.1 실습 목표 구조

2개 AZ에 Public/Private 서브넷 한 쌍씩(총 4개), IGW 1개, NAT GW 1개(비용 절약을 위해 실습은 1개만), 라우팅 테이블 2개로 구성된 표준 VPC를 만듭니다. 탭을 눌러 도구별 코드를 비교해 보세요 — 같은 개념이 도구마다 어떻게 표현되는지 보는 것이 핵심입니다. 

# 1. VPC 생성
aws ec2 create-vpc --cidr-block 10.0.0.0/16 \
  --tag-specifications 'ResourceType=vpc,Tags=[{Key=Name,Value=lecture-vpc}]'
# 반환된 VpcId를 환경변수로 저장했다고 가정: $VPC_ID

# 2. 서브넷 4개 생성 (AZ 2개 × Public/Private)
aws ec2 create-subnet --vpc-id $VPC_ID --cidr-block 10.0.0.0/24 \
  --availability-zone ap-northeast-2a   # Public-a
aws ec2 create-subnet --vpc-id $VPC_ID --cidr-block 10.0.1.0/24 \
  --availability-zone ap-northeast-2c   # Public-c
aws ec2 create-subnet --vpc-id $VPC_ID --cidr-block 10.0.10.0/24 \
  --availability-zone ap-northeast-2a   # Private-a
aws ec2 create-subnet --vpc-id $VPC_ID --cidr-block 10.0.11.0/24 \
  --availability-zone ap-northeast-2c   # Private-c

# 3. IGW 생성 및 VPC에 연결
aws ec2 create-internet-gateway
aws ec2 attach-internet-gateway --internet-gateway-id $IGW_ID --vpc-id $VPC_ID

# 4. Public 라우팅 테이블 — 여기가 핵심!
aws ec2 create-route-table --vpc-id $VPC_ID
aws ec2 create-route --route-table-id $PUB_RT_ID \
  --destination-cidr-block 0.0.0.0/0 --gateway-id $IGW_ID
# 이 한 줄이 서브넷을 'Public'으로 만든다
aws ec2 associate-route-table --route-table-id $PUB_RT_ID --subnet-id $PUB_A_ID
aws ec2 associate-route-table --route-table-id $PUB_RT_ID --subnet-id $PUB_C_ID

# 5. NAT Gateway (Public 서브넷에, EIP 필요)
aws ec2 allocate-address --domain vpc
aws ec2 create-nat-gateway --subnet-id $PUB_A_ID --allocation-id $EIP_ALLOC_ID

# 6. Private 라우팅 테이블 — IGW 대신 NAT로
aws ec2 create-route-table --vpc-id $VPC_ID
aws ec2 create-route --route-table-id $PRI_RT_ID \
  --destination-cidr-block 0.0.0.0/0 --nat-gateway-id $NAT_ID
aws ec2 associate-route-table --route-table-id $PRI_RT_ID --subnet-id $PRI_A_ID
aws ec2 associate-route-table --route-table-id $PRI_RT_ID --subnet-id $PRI_C_ID
# main.tf — 커뮤니티 표준 VPC 모듈 사용 (실무 권장)

  }

  }

설계 체크리스트 & 총정리

새 프로젝트의 네트워크를 설계할 때 순서대로 점검할 것들

10.1 네트워크 설계 체크리스트

CIDR 계획 — VPC는 /16으로 넉넉하게. 다른 VPC·온프레미스와 대역이 겹치지 않게(향후 Peering/VPN/TGW 연결 대비). EKS 예정이라면 Pod IP 소비량까지 계산.
AZ 전략 — 최소 2개 AZ. 각 AZ에 Public/Private(필요시 DB) 서브넷 쌍. 프로덕션 NAT GW는 AZ당 1개.
배치 원칙 — Public에는 ALB·NAT·(필요시) Bastion만. 애플리케이션과 데이터는 전부 Private. DB 서브넷은 Outbound 경로도 제거 검토.
보안 계층 — SG는 역할 기반 참조(Chaining)로 설계. 0.0.0.0/0 인바운드는 ALB의 80/443 외에 금지. NACL은 기본값 유지 + 필요 시 차단용.
접속 경로 — 운영자 접근은 SSM Session Manager 우선. SSH가 필요하면 Bastion + IP 제한.
비용 최적화 — S3/DynamoDB Gateway Endpoint는 무조건 추가(무료). NAT 트래픽이 큰 서비스(ECR, CloudWatch)는 Interface Endpoint 검토. 동일 AZ 내 통신 우선 설계로 AZ 간 전송 비용 절감.
EKS 추가 항목 — 서브넷 /20 이상 + Prefix Delegation, LB용 서브넷 태그, API 엔드포인트는 Private(+필요시 Public 임시 허용), 폐쇄망이면 Endpoint 6종 세트.
관측성 — VPC Flow Logs 활성화(S3 또는 CloudWatch). 장애·보안 분석의 출발점이 됩니다.

최종 확인 퀴즈

전 챕터를 아우르는 5문항 — 실무 시나리오 기반

FINAL 1
Q. Private 서브넷의 EC2에서 yum update가 타임아웃됩니다. 라우팅 테이블에는 10.0.0.0/16 → local 만 있습니다. 해결책은?
정답: NAT 경로 추가. 라우팅 테이블에 외부로 나가는 경로 자체가 없는 상태(Isolated)입니다. EIP는 IGW 경로 없이는 무의미하고, SG Outbound는 기본이 전체 허용입니다. 문제는 방화벽이 아니라 길이 없다는 것 — 증상을 보고 라우팅/보안 중 어느 계층의 문제인지 구분하는 것이 실전 감각입니다.
FINAL 2
Q. EKS 클러스터에서 대규모 배포 시 Pod가 Pending 상태로 멈추고 "failed to assign an IP address" 이벤트가 보입니다. 근본 원인은?
정답: 서브넷 IP 고갈. VPC CNI의 특성상 Pod 수 = 소비되는 VPC IP 수입니다. 해결책은 Prefix Delegation 활성화, Secondary CIDR 추가(Custom Networking), 더 큰 서브넷으로의 노드 그룹 이전 등입니다.
FINAL 3
Q. 보안 감사에서 "Private 서브넷의 노드가 ECR 이미지를 당길 때 트래픽이 인터넷(NAT)을 경유한다"는 지적을 받았습니다. 폐쇄망 요건을 만족시키면서 이미지를 받는 방법은?
정답: VPC Endpoint 구성. ECR 이미지 풀은 ecr.api(인증) + ecr.dkr(레지스트리) + S3(실제 레이어 저장소) 세 경로를 모두 사용하므로 3종 세트가 필요합니다. 이렇게 하면 트래픽이 AWS 내부망으로만 흐르고 NAT 비용도 사라집니다.
FINAL 4
Q. ALB는 Public 서브넷에, EC2는 Private 서브넷에 있습니다. ALB → EC2 구간의 트래픽은 어떤 경로로 흐를까요?
정답: local 라우팅. ALB와 EC2는 모두 같은 VPC(10.0.0.0/16) 안에 있으므로, 모든 라우팅 테이블에 기본 포함된 10.0.0.0/16 → local 규칙으로 직접 통신합니다. Public/Private은 "인터넷과의 관계"일 뿐, VPC 내부 통신에는 아무런 장벽이 아닙니다(SG만 허용한다면).
FINAL 5
Q. 프로덕션 EKS의 API 엔드포인트를 Private 전용으로 전환했습니다. 재택 근무 중인 개발자가 kubectl을 쓰려면?
정답: VPC 내부 경로 확보. Private 엔드포인트는 VPC 내부에서만 해석/접근됩니다. SG를 아무리 열어도 인터넷에서 도달할 경로 자체가 없습니다. Client VPN, Site-to-Site VPN, SSM 포트 포워딩, CI/CD 러너를 VPC 안에 두는 방법 등이 실무 해법입니다.
