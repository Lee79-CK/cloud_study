01

오브젝트 스토리지란?

MinIO를 이해하려면 먼저 "오브젝트 스토리지"가 무엇인지 알아야 합니다. 우리가 평소 쓰는 폴더·파일 구조와는 생각보다 많이 다릅니다.

데이터를 저장하는 방식에는 크게 세 가지가 있습니다. 각각이 어떻게 다른지 비유로 시작해 보겠습니다.

블록 스토리지는 빈 공책 한 권입니다. 페이지(블록)들이 있을 뿐, 내용을 어떻게 채울지는 전적으로 사용자(운영체제) 몫입니다. → 하드디스크, SSD, EBS

파일 스토리지는 서류 캐비닛입니다. 폴더 안에 폴더, 그 안에 파일… 트리 구조로 정리합니다. 경로(/home/user/docs/a.txt)로 찾습니다. → NAS, NFS

오브젝트 스토리지는 거대한 무인 택배 보관함입니다. 물건(오브젝트)마다 고유 번호표(키)를 붙여 통째로 넣어 두고, 번호표만 대면 꺼내 줍니다. 폴더 계층이 "진짜로는" 존재하지 않습니다.

왜 오브젝트 스토리지인가

클라우드 시대에 데이터는 폭발적으로 늘어났습니다. 사진, 동영상, 로그, 백업, 그리고 요즘은 머신러닝 모델 가중치와 데이터셋까지. 이런 "구조 없는(unstructured) 대용량 데이터"를 다루는 데 파일 시스템은 한계가 명확합니다. 폴더 안 파일이 수억 개가 되면 느려지고, 여러 서버로 확장하기도 어렵습니다.

오브젝트 스토리지는 이 문제를 정면으로 해결합니다. 계층 구조 대신 평평한(flat) 주소 공간을 쓰고, 각 오브젝트에 데이터·메타데이터·고유 ID를 함께 담아, 수십억 개도 거의 무한히 확장하며 저장합니다.

02

MinIO란 무엇인가

MinIO는 어디서나 실행할 수 있는, S3 호환 오브젝트 스토리지 서버입니다. 내 노트북, 사내 서버, 쿠버네티스, 엣지 디바이스 어디든 올릴 수 있습니다.

2014년에 시작된 MinIO의 핵심 철학은 단순합니다. "AWS S3는 훌륭하지만 AWS 안에만 있다. 그 경험을 내 인프라 어디서든 쓰게 하자." 그래서 MinIO는 다음 세 가지를 강하게 밀어붙였습니다.

🪶

단일 바이너리

Go로 작성된 실행 파일 하나. 의존성 지옥 없이 ./minio server /data 한 줄로 시작.

🔌

100% S3 호환

S3용으로 짠 코드(boto3, aws-cli 등)를 한 줄도 안 바꾸고 그대로 사용.

⚡

고성능

NVMe·고속 네트워크에 최적화. AI/분석 워크로드의 처리량을 위해 설계.

비유하자면 MinIO는 "동네에 차린 S3 매장"입니다. 본사(AWS S3)와 똑같은 메뉴판(S3 API)을 쓰기 때문에, 손님(애플리케이션)은 본사인지 동네 매장인지 구분하지 못한 채 똑같이 주문합니다. 다만 가게는 내 건물(내 서버) 안에 있죠.

⚠️

먼저 짚고 갈 점. MinIO는 2025~2026년에 오픈소스 정책이 크게 바뀌었습니다. 기술 자체는 여전히 훌륭하지만, "지금 새 프로젝트에 도입해도 되나?"는 별개의 문제입니다. 이 부분은 9장에서 솔직하게 다룹니다. 우선은 개념부터 확실히 익히겠습니다.

03

핵심 개념 4가지

MinIO(그리고 모든 S3 호환 스토리지)를 쓰려면 딱 네 가지 단어만 알면 됩니다.

① 버킷 (Bucket)

오브젝트를 담는 최상위 컨테이너입니다. 택배 보관함 비유에서 "보관함 구역"에 해당합니다. 이름은 전역적으로 유일해야 하고, 소문자·숫자·하이픈만 쓰는 규칙이 있습니다. 예: ml-models, raw-datasets, langfuse-backups

② 오브젝트 (Object)

실제 저장되는 데이터 한 덩어리입니다. 파일 하나가 보통 오브젝트 하나가 됩니다. 오브젝트는 (1) 실제 데이터 + (2) 메타데이터 + (3) 고유 키, 이 세 가지로 이루어집니다.

③ 키 (Key)

오브젝트의 고유 이름(경로처럼 보이는 문자열)입니다. 여기가 초보자가 가장 자주 헷갈리는 지점입니다.

🔑

핵심: models/llama3/8b/weights.safetensors 라는 키에서 슬래시(/)는 진짜 폴더가 아닙니다. 그냥 긴 이름의 일부일 뿐이며, MinIO 내부는 평평한 구조입니다. 다만 도구들이 models/로 시작하는 키들을 "폴더처럼" 묶어 보여줄 뿐입니다. 이를 프리픽스(prefix)라고 부릅니다.

④ 액세스 키 / 시크릿 키

S3 API에 접근하기 위한 아이디·비밀번호 쌍입니다. Access Key는 공개되는 식별자, Secret Key는 절대 노출되면 안 되는 비밀번호라고 생각하면 됩니다. 모든 요청은 이 키로 서명(signature)되어 인증됩니다.

한 장으로 정리

버킷 = 보관함 구역 오브젝트 = 보관된 물건 키 = 물건의 번호표(이름) 프리픽스 = 가짜 폴더 Access/Secret Key = 아이디/비번

정식 주소 형태: s3://ml-models/llama3/8b/weights.safetensors → 버킷 ml-models, 키 llama3/8b/weights.safetensors

04

아키텍처와 데이터 보호

"디스크가 고장 나면 데이터는 어떻게 되나요?" MinIO가 분산 환경에서 데이터를 지키는 핵심 기술, Erasure Coding(소거 부호)을 직접 만져 보며 이해해 봅시다.

두 가지 실행 모드

🧪 Standalone (단일 노드)

서버 한 대, 디스크 한두 개로 실행. 개발·테스트·학습용으로 적합. 데이터 보호 기능은 제한적.

🏭 Distributed (분산)

여러 노드·여러 디스크를 하나의 풀로 묶음. Erasure Coding으로 디스크나 노드가 죽어도 데이터를 복구. 운영(production)용.

Erasure Coding이란?

단순 복제(같은 데이터를 3벌 복사)는 용량을 3배 잡아먹습니다. Erasure Coding은 더 똑똑합니다. 데이터를 여러 조각(데이터 샤드)으로 쪼갠 뒤, 수학적으로 계산한 패리티 샤드를 추가로 만듭니다. 그러면 일부 조각이 사라져도 나머지로 원본을 복원할 수 있습니다.

RAID와 비슷한 아이디어입니다. 시험 답안지를 친구 8명에게 나눠 맡기되, "각자 답을 합치면 빠진 친구 몫을 추측할 수 있게" 힌트(패리티)를 4명에게 더 준다고 생각해 보세요. 12명 중 4명이 빠져도 답안 전체를 복원할 수 있습니다.

언제 무엇을 쓰나

AWS S3가 유리한 경우

  • 이미 AWS에 인프라가 있다
  • 운영 인력을 쓰고 싶지 않다
  • 트래픽이 들쭉날쭉해 탄력적 확장이 필요
  • 11 nine(99.999999999%) 내구성을 직접 책임지기 싫다

MinIO가 유리한 경우

  • 사내·폐쇄망·엣지에서 S3 API가 필요
  • 대용량 데이터를 오래 두면서 비용을 줄이고 싶다
  • 데이터를 외부에 둘 수 없는 규제 환경
  • GPU 옆 고속 로컬 스토리지로 학습 데이터를 빠르게 공급
🔁

핵심 통찰: S3 API가 표준이기 때문에, 처음엔 MinIO로 개발하다가 나중에 AWS S3로(또는 그 반대로) 코드 수정 없이 엔드포인트만 바꿔 옮길 수 있습니다. 이 "이식성"이 MinIO를 배우는 가장 실용적인 이유 중 하나입니다.

06

설치와 실행

가장 빠른 방법부터 운영용 쿠버네티스 배포까지. 환경에 맞춰 탭을 골라 보세요.

단일 노드 · 가장 빠른 시작
# API는 9000, 웹 콘솔은 9001 포트
$ docker run -d --name minio \
    -p 9000:9000 -p 9001:9001 \
    -e "MINIO_ROOT_USER=admin" \
    -e "MINIO_ROOT_PASSWORD=change-me-please" \
    -v /mnt/data:/data \
    minio/minio server /data --console-address ":9001"

# 브라우저에서 http://localhost:9001 접속 → 객체 브라우저 확인
docker-compose.yml
services:
  minio:
    image: minio/minio:latest
    command: server /data --console-address ":9001"
    ports:
      - "9000:9000"
      - "9001:9001"
    environment:
      MINIO_ROOT_USER: admin
      MINIO_ROOT_PASSWORD: change-me-please
    volumes:
      - minio_data:/data
    restart: unless-stopped
volumes:
  minio_data:

# ⚠️ 2026년 현재 공식 바이너리 배포가 중단됨 → 9장 참고
# (예시: 과거 방식 또는 소스 빌드 / 커뮤니티 포크 사용)

07

사용법: mc 클라이언트와 SDK

MinIO를 다루는 두 갈래: 사람이 터미널에서 쓰는 mc(MinIO Client)와, 코드에서 부르는 SDK(보통 S3 SDK 그대로).

mc — 명령줄 클라이언트

2026년 현재 커뮤니티 에디션은 웹 콘솔의 관리 기능이 빠졌기 때문에, 대부분의 관리 작업을 mc로 합니다. 버킷·정책·사용자 관리가 전부 여기 있습니다. 아래 버튼을 눌러 명령어가 무슨 일을 하는지 확인해 보세요.

SDK — 코드에서 접근

MinIO는 S3 호환이므로 AWS의 boto3를 그대로 쓸 수 있습니다. 엔드포인트만 MinIO 주소로 바꾸면 끝입니다. 전용 minio 파이썬 패키지도 있습니다.

S3 코드를 그대로 — endpoint만 변경
import boto3

s3 = boto3.client(
    "s3",
    endpoint_url="http://192.168.0.21:9000",   # ← MinIO 주소

08

응용 분야 — 특히 MLOps

MinIO가 가장 빛나는 곳은 AI/ML 파이프라인의 데이터 백본입니다. 클라우드 네이티브 스택의 거의 모든 도구가 S3를 "저장 백엔드"로 받아들이기 때문입니다.

🧠

모델 레지스트리 / 아티팩트

MLflow·Kubeflow의 아티팩트 저장소를 MinIO로 지정. 모델 가중치·체크포인트를 버전별로 보관.

📚

데이터셋 레이크

학습용 Parquet·이미지·텍스트 코퍼스를 버킷에 보관. Spark·Ray가 S3 경로로 직접 읽음.

🔎

벡터 DB 백엔드

Milvus는 오브젝트 스토리지에 세그먼트·인덱스를 저장. MinIO가 기본 백엔드로 자주 쓰임.

📊

관측성 장기 저장

Loki(로그)·Thanos/Mimir(메트릭)·Tempo(트레이스)가 S3에 청크를 저장. MinIO로 사내 보관.

🗃️

백업 / 스냅샷 타깃

Velero·DB 덤프·Langfuse 백업의 대상. 폐쇄망에서도 S3 워크플로 유지.

⚙️

워크플로 산출물

Argo Workflows의 아티팩트 패싱(스텝 간 파일 전달)을 S3 호환 저장소로 처리.

실전 시나리오: 온프레미스 LLM 서빙 파이프라인

예를 들어 GPU 서버에서 LLM을 서빙하는 환경을 생각해 봅시다. MinIO가 "데이터 허브" 역할을 합니다.

모델 보관

파인튜닝·양자화한 가중치를 ml-models 버킷에 키 규칙으로 정리해 저장.

서빙 노드가 당겨오기

vLLM·Triton 파드가 시작될 때 MinIO에서 해당 모델을 내려받아 로드. 사내망이라 속도가 빠름.

RAG 자료 보관

문서 원본은 MinIO에, 임베딩 벡터는 Milvus(백엔드도 MinIO)에 저장.

관측·백업

Langfuse 트레이스 백업, Loki 로그를 같은 MinIO 클러스터에 장기 보관.

🎯

왜 GPU 옆에 두면 좋은가. 수십 GB짜리 모델·데이터셋을 매번 외부 클라우드에서 당겨오면 이그레스 비용과 지연이 큽니다. 같은 사내망(또는 같은 랙)의 MinIO에서 당기면 고속 네트워크의 대역폭을 그대로 활용해 학습·서빙 준비 시간을 크게 줄일 수 있습니다.

09

2026년 현황과 대안 (꼭 읽기)

기술을 배우는 것과, 그것을 지금 새 프로젝트에 채택하는 것은 다른 문제입니다. MinIO 오픈소스의 최근 변화를 사실 그대로 정리합니다.

🚨

요약: MinIO 오픈소스(커뮤니티 에디션)는 2025~2026년에 걸쳐 핵심 관리 기능이 제거되고, 공식 바이너리·도커 이미지 배포가 중단됐으며, 결국 유지보수 모드를 거쳐 GitHub 저장소가 아카이브되었습니다. 새로운 기능 개발은 멈춘 상태입니다.

무슨 일이 있었나 (타임라인)

여러 매체와 커뮤니티 보도를 종합하면 다음과 같은 흐름입니다.

2021 · 라이선스 전환

핵심 라이선스를 Apache 2.0에서 AGPLv3로 변경. 클라우드 사업자의 "무임승차"를 막으려는 방어적 조치로 설명됨.

2025년 5~6월 · 콘솔 기능 제거

커뮤니티 에디션 웹 콘솔에서 계정·정책·버킷·라이프사이클 등 관리 기능을 제거하고 단순 "객체 브라우저"만 남김. LDAP/OIDC 외부 로그인도 제거. 관리 기능은 상용 AIStor로 이동.

2025년 10월 · 배포 중단

Docker Hub·Quay.io에 대한 이미지 게시와 사전 빌드 바이너리 배포를 중단. 같은 달 보안 취약점(CVE)이 공개되며 자동 배포 파이프라인에 혼란.

2025년 12월 3일 · 유지보수 모드

GitHub에서 오픈소스 프로젝트가 "maintenance mode"로 전환됨을 공지. 신규 기능 중단, 보안 패치만 제한적 제공.

2026년 · 저장소 아카이브

상태가 "더 이상 유지되지 않음"으로 바뀌고 저장소가 읽기 전용으로 아카이브됨(아카이브/해제가 반복된 정황도 보도됨). 사실상 오픈소스 개발 종료.

💰

상용 가격. 보도에 따르면 상용 제품 AIStor는 보안·관리 기능을 포함하지만, 진입 가격이 연 약 $96,000(최소 구성)부터 시작하는 것으로 알려져 많은 소규모·중간 규모 팀에게 부담이라는 평가가 있습니다. (가격 정책은 변동될 수 있으니 도입 전 공식 확인 필수)

그래서 지금 어떻게 해야 하나

기존 운영 중이라면

  • 이미 돌아가는 배포는 계속 동작합니다 — 급히 바꿀 필요는 없음
  • 보안 패치 적용 경로(소스 빌드 또는 커뮤니티 포크)를 마련
  • 이미지·바이너리를 사내 레지스트리에 미러링해 공급망 단절 대비

새로 시작한다면

  • 장기 유지보수·기능 추가가 멈춘 점을 반드시 고려
  • AGPLv3 라이선스의 컴플라이언스 영향을 검토
  • 아래 대안을 함께 평가한 뒤 결정

S3 호환 대안들

다행히 "S3 API 호환"이라는 표준 덕분에 갈아탈 선택지가 여럿 있습니다. 보도·커뮤니티에서 자주 언급되는 후보입니다.

학습 가치는 여전히 유효. 위 대안 대부분이 같은 S3 API를 따릅니다. 이 강의에서 배운 버킷·키·프리픽스·presigned URL·boto3 사용법은 어떤 S3 호환 스토리지로 가더라도 그대로 통합니다. MinIO 자체의 미래와 무관하게, S3 모델을 이해하는 것은 클라우드 엔지니어의 핵심 역량입니다.
