Chapter One · The Definition

Langfuse란 무엇인가

Langfuse는 LLM(대규모 언어 모델)을 사용하는 애플리케이션을 위한 오픈소스 관측(Observability) 및 엔지니어링 플랫폼입니다. 쉽게 말해, LLM 애플리케이션의 "Datadog + APM + 실험 노트"를 하나로 합쳐놓은 도구라고 생각하면 됩니다.

전통적인 마이크로서비스는 요청과 응답이 결정론적이지만, LLM은 그렇지 않습니다. 같은 질문에 다른 답이 나오고, 토큰 단위로 비용이 발생하며, 응답 품질을 단순히 HTTP 200으로 판단할 수 없습니다. Langfuse는 이 본질적으로 다른 워크로드를 위한 도구입니다.

한 줄 정의

✦
In Essence
Langfuse는 LLM 호출의 입출력·메타데이터·비용·지연시간을 추적하고, 프롬프트를 버전 관리하며, 모델 응답의 품질을 평가할 수 있게 해주는 오픈소스 플랫폼입니다. Self-host 가능하며, MIT 라이선스 코어를 제공합니다.

Langfuse가 제공하는 6가지 기능

i.
Tracing
관측의 핵심
LLM 호출과 그 주변의 모든 단계(검색·도구·체인)를 시계열 트리로 기록합니다.
ii.
Prompt Management
Git-like 버전 관리
코드 배포 없이 프롬프트를 수정·배포·롤백할 수 있습니다.
iii.
Evaluations
자동 + 수동
LLM-as-a-judge, 휴먼 어노테이션, 커스텀 메트릭으로 응답 품질을 점수화합니다.
iv.
Datasets
회귀 테스트셋
실제 트래픽에서 골든셋을 추출하고, 새 프롬프트/모델을 일괄 테스트합니다.
v.
Playground

UI에서 프롬프트와 모델 파라미터를 바꿔가며 즉시 결과를 비교합니다.
vi.
Metrics & Cost
대시보드

Chapter Two · The Motivation

왜 LLM에는 별도 관측이 필요한가

엔지니어들이 흔히 던지는 질문: "Prometheus + Grafana + Loki로 충분하지 않아?" 답은 단호하게 아니오입니다. 몇 가지 이유를 보겠습니다.

① 비결정성 (Non-determinism)

같은 입력에 다른 출력이 나옵니다. temperature가 0이라도 모델 버전, 시스템 부하, 토크나이저 미세 차이로 답이 갈립니다. 따라서 "성공/실패" 이진 메트릭이 무의미하고, 응답 자체를 저장해야만 문제를 재현할 수 있습니다.

② 비용의 가변성 (Variable Cost)

전통 API는 호출당 비용이 고정이지만, LLM은 입력·출력 토큰 수에 비례합니다. 한 명의 사용자가 무심코 긴 컨텍스트를 붙여넣으면 비용이 100배가 됩니다. 토큰 단위 추적이 필수입니다.

③ 다단계 체인 (Multi-step Reasoning)

RAG, Agent, Tool-use는 LLM을 한 번이 아니라 5~50번 호출합니다. 어느 단계에서 실패했는지, 어떤 retrieval이 잘못된 chunk를 가져왔는지를 보려면 분산 추적(distributed trace)과 동일한 사고방식이 필요합니다.

④ 품질 평가의 주관성

"답이 맞나?"라는 질문에 200/500 같은 답이 없습니다. 정답이 여러 개고, 문체나 안전성·근거 인용 같은 차원이 함께 평가되어야 합니다. 따라서 휴먼 평가, LLM-as-a-judge, 데이터셋 기반 회귀 테스트가 통합되어야 합니다.

⌘
사고 실험
"어제 챗봇이 이상하게 답한다는 민원이 들어왔다. 어떤 사용자가, 어떤 질문에, 어떤 retrieval로, 어떤 모델/프롬프트 버전으로, 얼마나 걸려서, 얼마의 비용으로 답했는지 1분 안에 재현할 수 있는가?"

이 질문에 답할 수 없다면 LLM Observability가 필요한 것입니다.

Chapter Three · The Vocabulary

핵심 개념 7가지

Langfuse를 다루기 전에 데이터 모델을 익혀야 합니다. 모든 SDK·API는 이 어휘를 그대로 씁니다.

1. Trace — 최상위 단위

하나의 사용자 요청(또는 워크플로 실행) 전체를 묶는 컨테이너입니다. 예: "유저가 챗봇에 질문 하나를 보냄" 전체가 1개의 Trace. ID 하나로 그 안의 모든 단계를 끌어올 수 있습니다.

2. Observation — Trace 안의 단위 작업

3가지 하위 타입이 있습니다.

  • Span — 시작·끝이 있는 일반 작업 (예: 검색, 파싱, 도구 실행)
  • Generation — LLM 호출 전용. 모델명·input·output·prompt tokens·completion tokens·cost 자동 기록
  • Event — 시점(point-in-time) 이벤트. 길이가 없는 마커

Observation은 중첩이 가능합니다. Span 안에 Generation, 그 안에 또 Span... 트리 구조가 트레이스 뷰의 본질입니다.

3. Session — 여러 Trace를 묶는 상위 그룹

멀티턴 대화에서 한 대화 세션의 모든 메시지를 하나의 Session으로 묶을 수 있습니다. UI에서 "이 사용자가 한 대화 전체"를 한 화면에서 봅니다.

4. User — 종단 사용자 식별자

익명 ID라도 좋습니다. 같은 user_id로 들어온 모든 트래픽을 묶어 비용·품질을 분석할 수 있습니다.

5. Score — 평가 점수

Trace 또는 Observation에 붙는 수치/카테고리/부울 평가입니다. 출처는 (a) 휴먼 어노테이션, (b) LLM-as-a-judge, (c) 사용자 피드백(👍/👎), (d) 코드 기반 메트릭 등 다양합니다.

6. Prompt — 버전 관리되는 프롬프트 템플릿

name + version + labels로 관리됩니다. production 라벨을 갈아끼우는 방식으로 코드 배포 없이 프롬프트를 교체할 수 있습니다.

7. Dataset — 입력·기대출력 쌍의 모음

회귀 테스트용 셋입니다. 새 프롬프트나 새 모델을 데이터셋 전체에 돌려 점수를 비교합니다. 프로덕션 트레이스를 그대로 dataset item으로 변환할 수 있어 골든셋 구축이 쉽습니다.

시각적으로 보면 — Trace 트리의 예

▶ TRACE chatbot-query-9f3a
2,410ms
└─ SPAN retrieval
420ms
└─ SPAN vector-search
280ms
└─ GEN qwen2.5-7b · system+rag
1,810ms
└─ SPAN post-process
180ms

※ 보라색 = Trace 전체, 시안 = 일반 Span, 앰버 = LLM Generation. Generation이 전체 지연의 75%를 차지하는 전형적인 RAG 패턴입니다.

Chapter Four · The Internals

Langfuse v3 아키텍처

Langfuse는 v2(2024)까지 PostgreSQL 단일 백엔드였지만, v3에서 대규모 트래픽을 위한 이벤트 기반 + 분리된 저장소 아키텍처로 진화했습니다. self-host에서 마주칠 컴포넌트는 다음과 같습니다.

Langfuse v3 — Component Topology

각 컴포넌트의 역할

  • Langfuse Web — 사용자가 보는 대시보드와, SDK가 이벤트를 POST하는 API ingest 엔드포인트. 받은 이벤트는 바로 처리하지 않고 Redis 큐에 넣고 200을 반환합니다(빠른 응답).
  • Langfuse Worker — 큐에서 이벤트를 꺼내 ClickHouse에 적재하고, 비용 계산·LLM-as-a-judge 평가·웹훅 같은 비동기 작업을 수행합니다.
  • PostgreSQL — 사용자·프로젝트·API 키·프롬프트·데이터셋 등 트랜잭션성 데이터.
  • ClickHouse — 트레이스/관측/점수 같은 수억 건 규모의 분석 데이터. 대시보드의 모든 집계가 여기서 나옵니다.
  • Redis — BullMQ 큐 백엔드, 레이트리밋 카운터, API 키 캐시.
  • S3/MinIO — 원본 이벤트 JSON과 멀티모달 미디어(이미지·오디오) 블롭. ClickHouse에는 가공된 데이터만 들어가고, 원본은 객체 스토리지에 보관합니다.
!
v2 vs v3 차이점
v2에서는 Postgres 하나로 모든 걸 처리했지만, 트레이스 수가 늘면 인덱스 비대화로 성능이 무너졌습니다. v3는 트랜잭션은 Postgres, 분석은 ClickHouse, 큐는 Redis, 블롭은 S3로 책임을 분리한 것이 핵심입니다. self-host 시 Helm 차트가 이 5개 의존성을 모두 띄워줍니다.

Chapter Five · Practicum Begins

실습 ① — 이종 클러스터 준비

이제 손에 잡히는 실습으로 들어갑니다. 우리 클러스터는 두 노드로 구성됩니다.

A
node-spark
ARM64 · GB10 · 128GB
NVIDIA DGX Spark. GPU 워크로드 전담. Langfuse 자체는 여기 안 올려도 무방.
B
node-ws
x86_64 · Workstation
Lenovo ThinkStation. Langfuse 본체 + 의존성 스택을 여기에 배치.

배치 전략 (Placement Strategy)

Langfuse 스택은 GPU가 필요 없는 일반 워크로드입니다. 그러므로:

  • Langfuse 본체(web/worker) + Postgres/Redis/MinIO → node-ws (x86_64)
  • ClickHouse → node-ws (x86_64, 메모리 여유 있는 쪽). 단, 운영 규모가 크면 별도 노드 권장.
  • node-spark의 GPU 자원은 LLM 추론 워크로드 전용으로 남겨둠 (vLLM, Triton, NIM 등)

단, ARM64에서도 대부분의 이미지(postgres, redis, clickhouse/clickhouse-server, minio/minio, langfuse/langfuse)가 multi-arch이므로 강제 배치가 가능합니다. 학습 목적으로는 node-spark에 일부를 띄워봐도 좋습니다.

Step 1. 노드 라벨링

kubectl로 두 노드에 라벨을 붙여 향후 nodeSelector로 명시적으로 배치합니다.

bash
# 현재 노드 확인
kubectl get nodes -o wide

# 노드 라벨 부여
kubectl label node node-spark node.kt.com/arch=arm64 node.kt.com/role=gpu
kubectl label node node-ws    node.kt.com/arch=amd64 node.kt.com/role=general

# (선택) GPU 노드에는 일반 워크로드가 못 들어가도록 taint
kubectl taint node node-spark workload=gpu:NoSchedule

Step 2. Namespace와 StorageClass 확인

bash
# 네임스페이스 생성
kubectl create namespace langfuse

# StorageClass 확인 (PVC를 만들 수 있어야 함)
kubectl get sc

# 기본 SC가 없다면 local-path-provisioner 추천 (홈랩 표준)
kubectl apply -f https://raw.githubusercontent.com/rancher/local-path-provisioner/v0.0.30/deploy/local-path-storage.yaml
kubectl patch storageclass local-path -p '{"metadata": {"annotations":{"storageclass.kubernetes.io/is-default-class":"true"}}}'

Step 3. Helm 준비

bash
# Helm 3.x 확인
helm version

# Langfuse 공식 차트 리포 추가
helm repo add langfuse https://langfuse.github.io/langfuse-k8s
helm repo update

# 차트 검색 — 버전 확인용
helm search repo langfuse
i
Sanity check
여기까지 했으면 kubectl get nodes --show-labels로 두 노드에 라벨이 잘 붙었는지, helm search repo langfuse가 langfuse/langfuse 차트를 찾는지 확인하세요. 둘 다 OK면 다음 챕터로 갑니다.

Chapter Six · Deployment

실습 ② — Helm으로 Langfuse 배포

Langfuse 공식 Helm 차트는 web/worker + Postgres + ClickHouse + Redis + S3(MinIO) 의존성을 모두 묶어 배포합니다. 전체 values를 한 번에 보면 어렵기 때문에, 섹션별로 끊어서 설명합니다.

1

values.yaml — 기본 골격

핵심은 두 가지입니다. ① 모든 컴포넌트를 node-ws로 보내고, ② 인증 시크릿을 32바이트 hex로 직접 지정.

yaml — values.yaml
langfuse:
  replicas: 1
  nodeSelector:
    node.kt.com/arch: amd64
  salt:
    value: "<openssl rand -hex 32 결과>"
  nextauth:
    secret:
      value: "<openssl rand -hex 32 결과>"
    url: "https://langfuse.lab.local"
  encryptionKey:
    value: "<openssl rand -hex 32 결과>"
  telemetryEnabled: false

worker:
  replicas: 1
  nodeSelector:
    node.kt.com/arch: amd64

# --- 의존성: Postgres, ClickHouse, Redis, MinIO ---
postgresql:
  deploy: true
  auth:
    password: "changeMe-pg"
  primary:
    nodeSelector:
      node.kt.com/arch: amd64
    persistence:
      size: 20Gi

clickhouse:
  deploy: true
  auth:
    password: "changeMe-ch"
  shards: 1
  replicaCount: 1
  nodeSelector:
    node.kt.com/arch: amd64
  persistence:
    size: 30Gi

redis:
  deploy: true
  auth:
    password: "changeMe-redis"
  master:
    nodeSelector:
      node.kt.com/arch: amd64

s3:
  deploy: true        # MinIO 자동 배포
  auth:
    rootPassword: "changeMe-minio"
  nodeSelector:
    node.kt.com/arch: amd64
2

시크릿 값 생성

위 salt, nextauth.secret, encryptionKey는 각각 다른 32바이트 hex여야 합니다.

bash
# 3개를 각각 생성
openssl rand -hex 32   # → salt
openssl rand -hex 32   # → nextauth.secret
openssl rand -hex 32   # → encryptionKey
⚠
중요
encryptionKey는 한 번 배포 후 변경하면 기존 데이터(API 키 등 암호화 컬럼)를 못 읽습니다. 반드시 처음에 안전하게 백업하세요. Vault·SealedSecret·SOPS 사용을 권장합니다.
3

설치 실행

bash
helm install langfuse langfuse/langfuse \
  -n langfuse \
  -f values.yaml \
  --version 1.5.0   # 강의 시점 기준, helm search repo로 최신 확인

Chapter Seven · Hello, Trace

실습 ③ — 첫 Trace 만들기

이제 가장 짜릿한 순간 — 실제 LLM 호출을 Langfuse가 잡아오는 것을 보겠습니다.

환경 변수 세팅

bash
export LANGFUSE_PUBLIC_KEY="pk-lf-..."
export LANGFUSE_SECRET_KEY="sk-lf-..."
export LANGFUSE_HOST="http://langfuse.lab.local"

# SDK 설치
pip install langfuse openai

3가지 통합 패턴을 비교해봅시다

가장 권장하는 방식. 함수에 데코레이터만 붙이면 자동으로 trace/span이 만들어집니다.

python · observe_example.py
from langfuse import observe, get_client
from langfuse.openai import openai  # 자동 instrument된 OpenAI wrapper

langfuse = get_client()

@observe()
def retrieve_docs(query: str) -> list[str]:
    # 실제로는 vector DB 호출
    return ["relativity is...", "time dilation occurs..."]

@observe()
def answer(question: str) -> str:
    docs = retrieve_docs(question)
    context = "\n".join(docs)

    rsp = openai.chat.completions.create(
        model="gpt-4o-mini",
        messages=[
            {"role": "system", "content": f"Context:\n{context}"},
            {"role": "user", "content": question},
        ],
    )
    return rsp.choices[0].message.content

if __name__ == "__main__":
    print(answer("느린 빛이란 무엇입니까?"))
    langfuse.flush()   # 종료 전 큐 비우기

⇒ Langfuse UI의 Traces 탭으로 가면 answer가 root span, 그 아래 retrieve_docs span과 OpenAI generation이 자동 중첩되어 보입니다.

코드 변경 최소화 방식. import openai를 from langfuse.openai import openai로 한 줄만 바꿉니다.

python · drop_in.py

# After:

    name="greeting-call",
    user_id="u-1234",
    session_id="sess-abc",
    tags=["prod", "v1"],
)

로컬에서 vLLM/Ollama로 OpenAI 호환 엔드포인트를 띄웠다면 openai.base_url = "http://node-spark:8000/v1" 만 추가하면 됩니다.

LangChain/LangGraph 체인을 쓴다면 콜백 핸들러를 붙입니다.

Chapter Eight · Versioned Prompts

실습 ④ — 프롬프트를 버전 관리하기

하드코딩된 프롬프트는 LLMOps의 적입니다. 프롬프트 한 줄 바꾸려고 컨테이너를 다시 빌드/배포하는 건 비효율이고, 어제와 오늘 프롬프트가 어떻게 달랐는지 추적할 방법도 없습니다. Langfuse의 Prompt Management는 이걸 Git처럼 풀어줍니다.

모델: name + version + label

  • name — 프롬프트 식별자 (예: rag-answer-v1)
  • version — 자동 증가 정수 (1, 2, 3...)
  • label — 사람이 붙이는 태그. production, staging, latest 등. 한 라벨은 한 버전에만.

코드는 이름 + 라벨로 가져옵니다. 운영 중 새 버전을 만들고 production 라벨을 새 버전으로 옮기면 — 그게 곧 무중단 프롬프트 배포입니다.

① UI에서 프롬프트 만들기

좌측 메뉴 Prompts → New prompt:

  • Name: sf-novelist-system
  • Type: Chat 또는 Text
  • Content (예시):
    prompt template
    당신은 한국어 하드 SF 소설가입니다.
- 상대성이론과 시간 지연을 정확히 다룹니다.
- 톤: 차분하고 명상적
- 길이: {{paragraphs}}문단

주제: {{topic}}
  • Variables: topic, paragraphs (자동 감지됨)
  • 저장 후 production 라벨 부여

② SDK에서 가져와 사용하기

python · use_prompt.py
from langfuse import get_client
from langfuse.openai import openai

langfuse = get_client()

# production 라벨이 가리키는 버전을 가져옴
prompt = langfuse.get_prompt("sf-novelist-system", label="production")

# 변수 치환
compiled = prompt.compile(topic="느린 빛", paragraphs=2)

rsp = openai.chat.completions.create(
    model="gpt-4o-mini",
    messages=[
        {"role": "system", "content": compiled},
        {"role": "user", "content": "써주세요"},
    ],
    langfuse_prompt=prompt,   # ← 이 호출이 어떤 프롬프트 버전을 썼는지 trace에 링크
)

langfuse_prompt=prompt를 넘기면 그 generation이 어떤 프롬프트 이름/버전에서 나왔는지가 Langfuse UI에 자동 연결됩니다. 나중에 "v3 프롬프트로 호출된 트레이스만 보기" 필터가 가능해집니다.

③ 캐시와 무중단 배포

get_prompt는 클라이언트 측 캐시(기본 60초 TTL)를 가집니다. 새 버전 라벨을 옮기면 캐시 만료 후 자동으로 새 프롬프트를 받습니다 — 코드 재시작 불필요.

✦
실무 팁
긴급 롤백은 Langfuse UI에서 production 라벨을 이전 버전으로 한 번 클릭해 옮기면 끝납니다. cache_ttl_seconds를 짧게(예: 10초) 설정하거나 SDK 재시작으로 즉시 반영도 가능합니다.

Chapter Nine · How Good Was It?

실습 ⑤ — 평가 (Evaluations)

"우리 챗봇이 잘 작동하는가?"에 답할 수 없으면 개선도 불가능합니다. Langfuse는 4가지 평가 경로를 제공합니다.

i.
User Feedback
👍 / 👎
엔드 유저가 보낸 좋아요/싫어요를 score로 기록. 코드 한 줄.
ii.
Human Annotation
UI 큐
QA 팀이 UI에서 trace를 직접 보고 채점.
iii.
LLM-as-a-Judge
자동 평가
템플릿화된 평가 프롬프트로 LLM이 LLM의 답을 채점.
iv.
Custom Code
SDK
BLEU, ROUGE, 정규식 매치 등 결정론적 메트릭을 점수로 push.

경로 ① — 사용자 피드백을 score로 기록

python · user_feedback.py
from langfuse import get_client
langfuse = get_client()

# 사용자가 👍 눌렀을 때 (예: FastAPI 엔드포인트 내부)
langfuse.create_score(
    name="user_thumb",
    value=1,                       # -1 (👎) or 1 (👍)
    trace_id=trace_id_from_client,
    data_type="NUMERIC",
    comment="답변이 정확함",
)

경로 ② — LLM-as-a-Judge를 UI에서 설정

좌측 메뉴 Evaluation → Templates → New evaluator에서:

  1. 평가 모델 선택 (예: GPT-4o, 또는 사내 Qwen2.5 OpenAI-compatible endpoint)
  2. 평가 프롬프트 작성:
    judge prompt
    아래 사용자 질문과 챗봇 답변을 평가해줘.
- 사실성(factuality): 답변이 질문에 사실적으로 정확한가? 0~1 사이 점수.
- 친절도(friendliness): 톤이 친절한가? 0~1 사이 점수.
- 이유를 한 문장으로 설명해줘.

Question: {{input}}
Answer: {{output}}

응답은 JSON으로: {"factuality": 0.x, "friendliness": 0.x, "reason": "..."}
  3. 변수 매핑: input ← trace.input, output ← trace.output
  4. 샘플링 비율 설정 (예: 새 trace의 20%만 평가)
  5. 저장 → Worker가 큐에서 꺼내 자동으로 score 부여

경로 ③ — Dataset 기반 회귀 테스트

"새 프롬프트가 기존보다 좋은가?"를 데이터 기반으로 판단하는 방법.

from langfuse import get_client
langfuse = get_client()

dataset = langfuse.get_dataset("sf-novel-eval")

for item in dataset.items:
    with item.run(run_name="sf-system-v3") as run:
        # 여기서 실제로 새 프롬프트/모델로 호출
        answer = my_pipeline(item.input)
        run.update(output=answer)

        run.score(
            name="length_ok",
            value=1 if len(answer) > 100 else 0,
        )

Chapter Ten · Going to Production

운영 베스트 프랙티스

실습이 끝났으니 — 운영하면서 누구나 겪는 함정과 권장 패턴을 정리합니다.

1. 시크릿 관리

  • nextauth.secret, salt, encryptionKey는 SealedSecret 또는 SOPS + age로 Git에 안전하게 보관.
  • 특히 encryptionKey는 한 번 분실하면 DB의 암호화된 API 키 컬럼을 복호화할 수 없습니다.

2. 백업

  • PostgreSQL — pgBackRest 또는 CronJob + pg_dump.
  • ClickHouse — BACKUP DATABASE 명령어 또는 clickhouse-backup 도구.
  • S3/MinIO — 외부 S3(혹은 다른 노드 MinIO)로 mirror.
  • Postgres만 있어도 사용자/프로젝트는 살아납니다. ClickHouse 손실 = 트레이스 손실.

3. 토큰 비용 단가 등록

Langfuse는 OpenAI/Anthropic의 공개 단가는 내장하지만, 사내 호스팅 모델(Qwen2.5 등)은 직접 등록해야 비용이 계산됩니다. UI Settings → Models에서 모델명 패턴과 토큰당 단가를 입력하세요.

4. 샘플링과 PII 마스킹

모든 호출을 다 기록하면 ClickHouse가 폭증합니다. 운영에선 보통:

  • 샘플링 — 비용 분석은 100% 유지, LLM-as-a-judge는 10~20%만.
  • PII 마스킹 — SDK의 mask 콜백으로 전화번호·이메일 등을 사전 치환.
python · masking.py
import re
from langfuse import Langfuse

def mask_pii(data):
    if isinstance(data, str):
        return re.sub(r"\d{3}-\d{4}-\d{4}", "[PHONE]", data)
    return data

langfuse = Langfuse(mask=mask_pii)

5. DGX Spark GPU 워크로드와의 통합

node-spark에 vLLM/Triton으로 Qwen2.5-7B를 띄웠다고 가정하면:

  • vLLM의 OpenAI-compatible endpoint(http://node-spark.svc:8000/v1)를 openai.base_url로 가리킴.
  • from langfuse.openai import openai가 그대로 동작 → 추론 호출이 자동 trace됨.
  • Langfuse Settings → Models에서 Qwen2.5-7B 단가를 등록(예: input/output 모두 0원이면 cost 0으로 표시).

6. 자주 묻는 문제

① langfuse.flush()를 호출했는지 확인 (스크립트 종료 시 누락 흔함). ② LANGFUSE_HOST에 오타가 없는지, https 인증서가 유효한지. ③ Worker 파드 로그(kubectl logs -n langfuse deploy/langfuse-worker)에서 ingest 에러 확인.
기본 retention이 무제한입니다. Langfuse는 retention 정책 UI를 제공하므로 Settings → Data Retention에서 trace 보관 기간(예: 30일)을 설정하세요. OSS 버전 제약이 있으면 ClickHouse TTL을 직접 ALTER로 거는 방법도 있습니다.
SAML/Enterprise SSO는 Langfuse Cloud 또는 Enterprise edition 기능입니다. OSS self-host에서는 OAuth(Google/GitHub/Azure AD) 통합이 환경변수로 가능합니다. values.yaml의 langfuse.additionalEnv로 AUTH_GOOGLE_CLIENT_ID 등을 넘기세요.
대부분의 핵심 이미지(langfuse/langfuse, postgres, redis, minio/minio, clickhouse/clickhouse-server)는 multi-arch입니다. 단, Bitnami 차트의 일부 sidecar(metric exporter 등)는 ARM64 빌드가 없을 수 있으니 kubectl describe pod로 "no matching manifest" 에러가 나면 해당 sidecar를 비활성화하거나 대체 이미지를 사용하세요. 안전한 기본값은 전부 x86_64인 node-ws에 배치입니다.
① 샘플링 비율을 5~10%로 낮추세요. ② 평가 모델을 GPT-4가 아닌 사내 Qwen2.5로 돌리면 비용 0. ③ 평가 프롬프트를 짧게 만들어 토큰을 줄이세요. ④ 결정론적 메트릭(길이·정규식)으로 1차 필터 후 LLM 평가 2차 적용.
