Claude Code는 코드를 읽고, 고치고, 명령을 실행하고, 도구와 연결하는 에이전틱(agentic) 코딩 도구입니다. 단순히 코드를 추천하는 자동완성과는 다릅니다. 이 강의는 "바이브 코딩"과의 차이부터 실제 설치·사용·응용까지, 코딩에 익숙하지 않은 사람도 따라올 수 있도록 차근차근 설명합니다.

01 — 기초 개념

Claude Code란 무엇인가

"코드를 대신 쳐주는 도구"가 아니라 "일을 대신 해주는 동료"라는 관점의 전환

Claude Code는 Anthropic이 만든 에이전틱 코딩 도구입니다. 여기서 "에이전틱(agentic)"이라는 단어가 핵심인데, 풀어 말하면 스스로 목표를 이해하고 → 계획을 세우고 → 여러 파일을 직접 수정하고 → 명령을 실행해 결과를 확인하는, 일종의 자율적인 일꾼이라는 뜻입니다.

비유하자면, 기존 자동완성 도구가 "다음 단어 받아쓰기를 도와주는 보조"라면, Claude Code는 "옆자리에 앉아 '인증 모듈에 테스트 좀 짜고 돌려서 깨지는 거 고쳐줘'라고 말하면 알아서 해내는 동료"에 가깝습니다.

↺핵심 동작 루프

Claude Code는 한 번에 답을 뱉는 게 아니라, 사람처럼 반복(loop)하면서 일합니다. 이 루프를 이해하면 도구의 본질이 보입니다.

STEP 1
읽기코드베이스와 관련 파일을 탐색해 맥락을 파악
STEP 2
계획무엇을 어떤 순서로 바꿀지 접근법을 설계
STEP 3
실행여러 파일을 수정하고 명령·테스트를 직접 실행
STEP 4
검증결과를 확인하고 실패하면 다시 1번으로 반복

당신은 이 과정의 감독관입니다. 무엇을 할지 자연어로 말하고, 제안된 변경을 검토·승인하고, 방향을 잡아주죠. 키보드를 두드리는 사람에서 의도를 전달하고 결과를 책임지는 사람으로 역할이 바뀝니다.

▤어디서나 같은 두뇌

Claude Code는 한 곳에만 묶여 있지 않습니다. 터미널·IDE·데스크톱 앱·브라우저 어디서 쓰든 동일한 엔진이 동작하고, 당신의 CLAUDE.md 설정과 연결한 도구(MCP)가 모든 환경에서 그대로 따라옵니다.

◆ 한눈 정리

Claude Code = "코드베이스 전체를 이해하고, 여러 파일에 걸쳐 다단계 작업을 자율적으로 수행하되, 모든 변경은 당신이 검토·승인하는" 에이전틱 코딩 도구. 자동완성과의 결정적 차이는 '추천'이 아니라 '실행'한다는 점입니다.

02 — 기초 개념

바이브 코딩 vs 에이전틱 엔지니어링

같은 AI 코딩이라도 "기대고 흘러가기"와 "설계하고 가속하기"는 전혀 다른 행위입니다

바이브 코딩(vibe coding)은 2025년 2월 Andrej Karpathy가 만든 말입니다. 핵심은 단순합니다 — 원하는 걸 자연어로 말하면 AI가 코드를 쓰고, 돌려보고, 깨지면 에러를 그대로 다시 붙여넣어 고치는 식의 흐름이죠. Karpathy의 표현을 빌리면 "완전히 바이브에 몸을 맡기고, 코드가 존재한다는 사실조차 잊는" 방식입니다.

그런데 1년 뒤인 2026년, Karpathy는 모델이 충분히 똑똑해지면서 이제 "바이브 코딩"이라는 말이 한물갔다고 보고, 전문가용 워크플로를 가리키는 새 이름으로 에이전틱 엔지니어링(agentic engineering)을 제안했습니다. 그의 구분은 날카롭습니다:

Vibe Coding · 바닥을 끌어올린다

바이브 코딩

  • "원하는 걸 말하고, 나온 걸 받아들인다"
  • 코드를 깊이 검토하지 않고 직관에 기댐
  • 프로토타입·탐색·학습·일회용 스크립트에 적합
  • 접근성이 목적 — 비전문가도 만들 수 있게
  • 빠르고 재밌지만, 프로덕션 워크플로는 아님
Agentic Engineering · 천장을 끌어올린다

에이전틱 엔지니어링

  • "시스템을 설계하고, 제약을 명시하고, 이미 추론을 끝낸 구현을 AI로 가속한다"
  • 스펙·테스트·검토를 통해 품질 기준을 지킴
  • 코드를 직접 안 쓰는 대신 에이전트를 지휘·감독
  • 품질 저하 없는 처리량이 목적 — 전문가용
  • 배워서 잘하게 되는 하나의 기술(art & science)

한 줄로 요약하면 — 바이브 코딩은 "표현(expression)", 에이전틱 엔지니어링은 "공학(engineering)"입니다. 바이브 코딩이 초보자의 바닥(floor)을 끌어올린다면, 에이전틱 엔지니어링은 전문가의 천장(ceiling)을 끌어올립니다.

직접 비교해 보기 — "결제 API에 속도 제한(rate limit)을 추가해줘"

Claude Code는 두 방식 모두 가능하지만, 진짜 위력은 에이전틱 엔지니어링 쪽에 있습니다. 주말 해커톤용 장난감을 만들 땐 바이브 코딩처럼 빠르게 흘러가도 되지만, 회사 코드에는 스펙·검토·테스트를 갖춘 규율 있는 워크플로가 필요합니다. Karpathy의 경고처럼 — "바이브 코딩 중이었다는 이유로 보안 취약점을 넣어도 되는 건 아닙니다." 소프트웨어에 대한 책임은 여전히 당신에게 있습니다.

⚠ 초보자가 빠지기 쉬운 함정

"AI가 다 해주니 검토는 필요 없다"는 태도가 가장 위험합니다. 바이브 코딩으로 만든 결과물을 그대로 프로덕션에 올리는 것이 2026년 현재 조직들이 비싼 대가를 치르는 대표적 실수입니다. 둘을 구분하고, 지금 내가 무엇을 하고 있는지 아는 것이 첫걸음입니다.

03 — 사용 방법

설치하기

운영체제에 맞는 방법 하나만 고르면 됩니다. 가장 권장되는 건 네이티브 설치입니다.

먼저 준비물: Claude 구독(claude.ai의 Pro/Max 등) 또는 Anthropic Console 계정이 필요합니다. 터미널 CLI와 VS Code는 서드파티 제공자도 지원합니다. 아래에서 본인 OS 탭을 고르세요.

⤓네이티브 설치 (권장)

네이티브 설치는 백그라운드에서 자동 업데이트되어 늘 최신 버전을 유지합니다. 가장 손이 덜 갑니다.

bash
curl -fsSL https://claude.ai/install.sh | bash
PowerShell
irm https://claude.ai/install.ps1 | iex
cmd
curl -fsSL https://claude.ai/install.cmd -o install.cmd && install.cmd && del install.cmd

⊞패키지 관리자로 설치

terminal
# 안정 채널
brew install --cask claude-code

# 최신 채널을 원하면
brew install --cask claude-code@latest
PowerShell
winget install Anthropic.ClaudeCode
terminal
npm install -g @anthropic-ai/claude-code

Linux에서는 apt, dnf, apk(Debian/Fedora/RHEL/Alpine)로도 설치할 수 있습니다. IDE를 쓴다면 VS Code·Cursor 확장 검색창에서 "Claude Code"를 찾아 설치하거나, JetBrains 마켓플레이스에서 플러그인을 받을 수 있습니다.

✓ 윈도우 팁

네이티브 윈도우에서는 Git for Windows를 함께 설치하면 Claude Code가 Bash 도구를 쓸 수 있어 권장됩니다. 없으면 PowerShell을 셸로 사용합니다. (WSL 환경은 별도 설치 불필요)

▶실행

설치가 끝나면 작업할 프로젝트 폴더로 이동해 claude만 입력하면 됩니다. 처음 실행 시 로그인 안내가 뜹니다.

첫 실행
cd your-project
claude
04 — 사용 방법

첫 세션과 기본 루프

"무엇을 원하는지 말한다 → Claude가 쓰고 실행한다 → 검토하고 반복한다"

세션을 시작하면 작업 디렉터리·사용 모델·힌트가 보이는 환영 화면이 뜹니다. 인터페이스는 대화창이에요. 메시지를 입력하고 Enter를 누르면 됩니다. 처음엔 거창하게 시작하지 말고, 코드베이스를 파악하는 탐색 질문부터 던져보세요.

claude session
> 이 프로젝트가 뭐 하는 건지 구조를 요약해줘
# Claude가 파일들을 읽고 스택과 아키텍처를 설명

> @src/pages/Home.tsx 의 헤더 문구를 'Welcome'으로 바꿔줘
# 제안된 변경(diff)을 보여주고 → 승인 후 수정

> auth 모듈에 테스트를 짜고, 돌려서, 깨지면 고쳐줘
# 계획 → 작성 → 실행 → 실패 분석 → 수정의 루프 반복

핵심 습관 — 구체적으로 말하기. "데이터 분석해줘"는 나쁜 프롬프트입니다. "employment.csv를 불러와 부문별 월간 성장률을 계산하고, 시계열 그래프를 그려줘"가 좋은 프롬프트죠. AI는 마음을 읽지 못합니다. 의도·제약·기대 결과를 명시할수록 결과가 좋아집니다.

@파일 참조와 핵심 단축키

@파일명으로 특정 파일을 직접 가리키면, Claude가 폴더를 헤매며 검색하는 대신 정확히 그 파일을 봅니다(토큰도 절약). 외워두면 좋은 단축키:

키동작
Tab자동완성
↑ 방향키이전 입력 기록 불러오기
?모든 단축키 보기
/모든 슬래시 명령어 보기
!터미널 명령을 실행하고 그 출력을 Claude에게 전달
Esc작업 중인 Claude를 즉시 중단
◆ 변경은 항상 당신의 승인을 거칩니다

Claude Code는 파일을 고치기 전에 diff(변경 미리보기)를 보여주고 승인을 요청합니다. "AI가 멋대로 내 코드를 망가뜨리면 어쩌지?"라는 걱정은 접어두세요 — 운전대는 당신이 쥐고 있습니다.

05 — 사용 방법

CLAUDE.md와 메모리

매 세션마다 같은 설명을 반복하지 않으려면, 프로젝트의 "온보딩 문서"를 한 번 써두세요.

CLAUDE.md는 프로젝트 루트에 두는 마크다운 파일로, Claude Code가 매 세션 시작 시 자동으로 읽습니다. 새로 온 AI 팀원에게 건네는 온보딩 가이드라고 생각하면 됩니다. 빌드 명령, 코드 스타일 규칙, 아키텍처 결정, 선호 라이브러리, 리뷰 체크리스트 같은 걸 담죠.

직접 처음부터 쓸 필요는 없습니다. 세션 안에서 /init을 실행하면 Claude가 프로젝트를 분석해 빌드 시스템·테스트 프레임워크·코드 패턴을 감지하고 초안을 만들어줍니다. 단, 생성된 초안은 과감히 다듬으세요. 대부분은 코드만 읽어도 알 수 있는 내용이라 군더더기입니다.

CLAUDE.md
# 프로젝트 규칙

## 빌드 & 테스트
- 빌드: `npm run build`
- 테스트: `npm test` (커밋 전 반드시 통과)

## 코드 스타일
- 함수형 컴포넌트만 사용, class 컴포넌트 금지
- 타입은 명시적으로, any 사용 금지
- 들여쓰기 2칸

## 아키텍처
- API 호출은 src/lib/api 에만 둔다
- 상태 관리는 Zustand 사용
claude session
> /init
# 프로젝트를 분석해 CLAUDE.md 초안을 생성

> /memory
# CLAUDE.md 편집 인터페이스 열기

⊙자동 메모리(Auto Memory)

당신이 직접 쓰지 않아도, Claude는 작업하면서 스스로 배운 것을 메모리에 저장합니다. 빌드 명령, 디버깅 인사이트 같은 것들이 세션을 넘어 누적되죠. 같은 실수를 두 번 설명하지 않아도 된다는 뜻입니다.

✓ 핵심 원칙: 짧게 유지하라

CLAUDE.md는 매 세션 컨텍스트에 통째로 로드됩니다. 그래서 비대해지면 오히려 Claude가 지시를 무시합니다. 200줄 이내로 유지하고, 상세 내용은 별도 파일로 빼서 @파일명으로 불러오세요. 팀이라면 git에 커밋해 모두가 같은 규칙을 공유하게 하면 됩니다.

06 — 사용 방법

컨텍스트 관리

긴 세션에서 답변 품질이 떨어진다면, 대부분 "컨텍스트 창"이 꽉 찼기 때문입니다.

LLM과의 대화는 사람과의 대화와 다릅니다. 메시지를 보낼 때마다 지금까지의 전체 기록(내 프롬프트, Claude의 응답, 읽은 파일, 실행한 명령, 출력 결과)이 통째로 묶여 한 번에 전달됩니다. 이 묶음을 컨텍스트 창(context window)이라 하고, 토큰 단위로 정해진 한계가 있습니다. (토큰 ≈ 단어의 4분의 3)

기본 컨텍스트 창은 20만 토큰이고, Sonnet·Opus는 베타로 100만 토큰까지 쓸 수 있습니다. 많아 보이지만 생각보다 빨리 찹니다. 그리고 중요한 점 — 한계에 닿기 전부터 품질이 떨어집니다. 30번째 턴의 Claude는 3번째 턴보다 눈에 띄게 둔해집니다.

컨텍스트 창 시뮬레이터 — 직접 채우고 비워 보기
18%
컨텍스트 사용량: 18% — 쾌적합니다. 작업을 진행해 보세요.

⌗두 가지 도구: /clear 와 /compact

명령어하는 일언제 쓰나
/clear대화를 완전히 비우고 새 출발 (CLAUDE.md는 유지)관련 없는 다른 작업으로 전환할 때
/compact핵심을 보존하며 오래된 메시지를 요약·압축긴 작업을 이어가되 자리를 잃기 싫을 때
/context현재 컨텍스트 사용량 확인슬슬 무거워졌나 점검할 때

/compact에는 무엇을 남길지 지시할 수도 있습니다: /compact Focus on the API changes 처럼요. Claude는 한계에 가까워지면 알아서 자동 압축(auto-compact)도 하지만, 능동적으로 관리하는 편이 훨씬 좋습니다.

✓ 실전 습관

① 관련 없는 작업 사이엔 무조건 /clear — 버그 수정 끝내고 새 기능 들어갈 땐 비우세요. 옛 맥락은 새 작업을 방해합니다. ② 중요한 결정은 대화가 아니라 파일(README, 플랜 파일)에 적어두기 — 파일은 남지만 대화 맥락은 사라집니다. ③ 두 번 고쳐도 안 되면 세션을 새로 시작하세요.

07 — 사용 방법

플랜 모드와 워크플로우

코드를 건드리기 전에 "스펙 → 계획 → 검토 → 실행" 순서를 지키면 실수가 눈덩이처럼 불어나지 않습니다.

초보자가 가장 흔히 저지르는 실수는 "바로 코드부터 짜게 하는 것"입니다. 중요한 파일을 건드리기 전엔 플랜 모드(Plan Mode)로 먼저 계획을 세우게 하세요. 계획을 바꾸는 데는 비용이 거의 안 들지만, 코드를 바꾸는 데는 몇 시간이 듭니다.

▦견고한 5단계 워크플로

STEP 1 — 스펙
무엇을 만들지 명세Claude가 범위·제약·저장 방식을 되묻고, 합의된 내용을 설계 문서로 기록
  1. 스펙 작성 — "API에 속도 제한을 추가하고 싶어"라고 하면, Claude가 범위·한도·저장 백엔드(Redis vs 인메모리 등)를 되묻고 설계 문서를 씁니다.
  2. 스펙 검토 — 읽고, 가정을 의심하고, "X는 어떻게 처리?"라고 물으세요. 스펙을 고치는 건 공짜에 가깝습니다.
  3. 계획 수립 — Claude가 스펙을 아주 작은 작업들의 순서 있는 목록으로 변환합니다(파일 경로·작성할 코드·검증 명령 포함).
  4. 계획 검토 — 코드가 만들어지기 전 마지막 기회. 순서는 맞나? 빠진 건? 불필요한 단계는?
  5. 실행 — 각 작업마다 새 서브에이전트가 구현·테스트·커밋하고, 리뷰어 에이전트가 스펙과 대조해 점검합니다.
◆ /goal — 손 떼고 맡기는 자율 작업

완료 조건을 정해두면 Claude가 충족될 때까지 알아서 반복합니다. 예: /goal test/auth의 모든 테스트가 통과하고 lint가 깨끗할 것. 매 턴 후 작고 빠른 모델이 조건 충족 여부를 평가하고, 아니면 다음 턴을 자동으로 시작합니다. 다단계 무인 작업의 핵심 기능입니다.

08 — 사용 방법

슬래시 명령어 · 단축키

90%는 한 줌의 명령어로 끝납니다. 칩을 눌러 각 명령어가 뭘 하는지 확인해 보세요.

슬래시 명령어는 세션 안에서 /로 시작해 입력하는 단축 동작입니다. /를 치면 자동완성 메뉴가 뜨고, /help는 항상 현재 버전에서 실제로 쓸 수 있는 명령을 보여주는 "진실의 원천"입니다.

슬래시 명령어 탐색기
/ 명령어를 눌러보세요

위 칩 중 하나를 클릭하면 설명이 여기 표시됩니다.

참고로 명령어 외에 두 계층이 더 있습니다. CLI 플래그는 실행 전에 세션을 설정하고(예: claude --continue, claude -c로 최근 세션 이어가기), 단축키는 실시간 조작을 담당합니다(Esc로 중단 등).

✓ 가장 많이 쓰는 작은 집합

전체를 외울 필요 없습니다. 실제로 90% 이상은 이것들입니다: /init(프로젝트 시작 시) · /clear(작업 전환) · /compact(긴 작업 이어가기) · /model(모델 전환) · /review(코드 리뷰) · /resume(세션 복구). 이것부터 습관 들이세요.

09 — 응용 기법

MCP — 외부 도구 연결

Claude Code가 당신의 코드뿐 아니라 Google Drive 문서, Jira 티켓, Slack 메시지까지 다룰 수 있게 하는 표준 규약

MCP(Model Context Protocol)는 AI 도구를 외부 데이터·서비스에 연결하기 위한 오픈 표준입니다. MCP 서버를 연결하면 Claude Code가 Google Drive의 설계 문서를 읽고, Jira 티켓을 업데이트하고, Slack에서 데이터를 가져오고, 당신이 직접 만든 사내 도구까지 호출할 수 있습니다.

비유하자면, MCP는 전기 콘센트 규격과 같습니다. 콘센트 규격(MCP)이 표준화되어 있으면, 어떤 가전(Jira, Slack, GitHub, 사내 DB...)이든 같은 방식으로 꽂아 쓸 수 있죠. 각 서비스마다 따로 통합 코드를 짤 필요가 없습니다.

⌥왜 LLMOps/MLOps 관점에서 중요한가

인프라·운영을 다루는 입장에서 MCP는 특히 강력합니다. 모니터링 대시보드, 티켓 시스템, 사내 위키, 벡터 DB 같은 운영 컨텍스트를 에이전트에 직접 연결하면, "어제 발생한 알람을 분석하고 관련 런북을 찾아 조치 PR을 올려줘" 같은 흐름이 가능해집니다. 표준 인터페이스이므로 도구를 갈아끼워도 워크플로는 유지됩니다.

◆ 시작점

공식 MCP 퀵스타트가 첫 서버를 처음부터 끝까지 연결하는 과정을 안내합니다. 한 번 연결한 MCP 서버는 터미널·IDE·데스크톱 등 모든 환경에서 공유됩니다. CI에서는 GitHub Actions·GitLab CI/CD로 코드 리뷰와 이슈 분류를 자동화할 수 있습니다.

⚓Hooks — 보장된 자동 실행

Hooks는 Claude Code의 특정 동작 전/후에 셸 명령을 실행하게 합니다. 예를 들어 파일을 수정할 때마다 자동 포맷팅을 돌리거나, 커밋 전에 lint를 실행하는 식이죠. 프롬프트와의 결정적 차이 — 프롬프트는 "제안"이지만 Hooks는 "보장된 실행"입니다. 매번 빠짐없이 lint를 돌리고 싶다면 프롬프트가 아니라 Hooks가 맞는 도구입니다.

hook 예시
# 파일 편집 후(after edit) → 자동 포맷
edit  →  prettier --write {file}

# 커밋 전(before commit) → lint 통과해야 진행
commit →  npm run lint
✓ 설계 원칙

부작용(배포·삭제 등)이 있는 명령에는 disable-model-invocation: true를 붙이세요. "코드가 준비된 것 같으니 배포하자"고 Claude가 스스로 판단하는 건 곤란하니까요. 또 안전한 도구(일상적 git 명령, 빌드, 읽기 작업)는 /permissions에서 미리 허용해 권한 묻는 횟수를 줄이세요.

11 — 응용 기법

서브에이전트 · 병렬 작업

큰 일을 여러 일꾼에게 나눠 동시에 처리하되, 메인 세션은 "조율"에만 집중하게

모든 병렬 기능이 풀려는 근본 문제는 하나입니다 — 컨텍스트 창이 꽉 찬다는 것. 파일 읽기, 검색 결과, 탐색 출력이 모두 예산을 갉아먹죠. 작업을 분리된 일꾼에게 맡기면, 그들의 장황한 출력이 메인 세션을 어지럽히지 않습니다.

A
서브에이전트(Subagents)단일 집중 작업을 위해 띄우는 가벼운 격리 일꾼. 각자 독립된 컨텍스트를 가져 메인 세션을 비대하게 만들지 않음. "쏘고 잊는(fire-and-forget)" 방식.
B
에이전트 팀(Agent Teams)공유 작업 목록으로 조율되는 완전한 독립 세션들. 리드 에이전트가 작업을 배분하고 결과를 병합. 더 복잡한 협업에 적합.
C
백그라운드 에이전트여러 풀세션을 동시에 돌리며 한 화면에서 관찰(agent view). 장시간 작업을 병렬로.
D
git worktreeclaude -w feature-auth 로 격리된 작업 트리 생성. 브랜치 전환 충돌 없이 여러 작업을 동시에.
⚠ 올바른 모델 고르기

상황에 맞는 모델을 고르는 것이 깔끔한 병렬 실행과 "조율 악몽"을 가르는 차이입니다. 단순하고 독립적인 작업이면 서브에이전트, 서로 의존하며 조율이 필요하면 에이전트 팀을 쓰세요. 무조건 많이 띄운다고 빨라지지 않습니다.

자동화 · CI/CD

Claude Code는 유닉스 철학을 따릅니다 — 파이프로 잇고, 스크립트에 넣고, CI에서 돌릴 수 있습니다.

프린트 모드(-p 또는 --print)는 대화창 없이 한 방에 실행하는 비대화형 모드입니다. 로그를 파이프로 흘려보내거나, CI 파이프라인에 끼워 넣거나, 다른 도구와 체이닝할 수 있죠.

bash
# 최근 로그를 분석해 이상 징후가 있으면 슬랙으로
tail -200 app.log | claude -p "이상 징후 있으면 슬랙으로 알려줘"

# 바뀐 파일들만 보안 관점에서 리뷰
git diff main --name-only | claude -p "이 변경들 보안 점검"
CI (예산·도구 제한)
claude -p "이 PR을 보안 관점에서 리뷰" \
  --model sonnet \
  --max-budget-usd 2.00 \
  --max-turns 10 \
  --output-format json \
  --allowedTools "Read,Bash(git *)"

7
장황한 작업은 위임테스트 실행·로그 분석은 서브에이전트로. 출력이 메인 세션을 어지럽히지 않게.
8
검토는 당신 몫AI가 썼어도 책임은 사람에게. 모든 변경 diff를 확인하고 승인하는 습관.
⚠ 가장 중요한 한 가지

"AI가 다 했으니 검토는 생략"이 아니라 — "AI가 가속해줬으니, 나는 더 높은 수준의 설계와 검증에 집중한다"가 올바른 마음가짐입니다. 키보드에서 손을 떼는 것이지, 책임에서 손을 떼는 게 아닙니다.
