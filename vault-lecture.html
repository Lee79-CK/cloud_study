Cloud Engineer 입문 시리즈 · Lecture

비밀번호를 코드에
심지 않는 법, Vault

API 키, DB 비밀번호, 인증서. 이 "시크릿"들이 깃 저장소와 환경변수에 흩어져 있다면 그건 사고가 일어날 시간 문제일 뿐입니다. HashiCorp Vault는 이 모든 비밀을 한곳에서 안전하게 보관하고, 누가·언제·얼마 동안 접근하는지 통제하는 도구입니다. 개념부터 실습, 클라우드 네이티브 응용까지 차근차근 짚어봅니다.

난이도 입문 예상 학습 40분 기준 버전 Vault 2.0.x 업데이트 2026.06
01 — The Problem

왜 시크릿 관리가 필요한가

코드를 짜다 보면 어디엔가 비밀번호를 적어야 합니다. 문제는 "어디에" 적느냐입니다.

애플리케이션은 데이터베이스에 접속하려면 비밀번호가 필요하고, 외부 API를 호출하려면 키가 필요합니다. 이런 민감한 값들을 통틀어 시크릿(Secret) 이라고 부릅니다. 초보 개발자가 가장 흔히 하는 일은 이 시크릿을 코드나 설정 파일에 그냥 적어 넣는 것입니다.

# config.py — 절대 이렇게 하면 안 됩니다
DB_PASSWORD = "P@ssw0rd_2024!"   # 평문 비밀번호
AWS_SECRET_KEY = "wJalrXUtnFEMI/K7MDENG/bPxRfiCY"
STRIPE_API_KEY = "sk_live_51H8x...실제 결제 키"

# 이 파일이 git에 커밋되는 순간,
# 모든 히스토리에 비밀번호가 영구히 박제됩니다.

이렇게 시크릿이 코드·환경변수·CI 설정·위키·슬랙 메시지 등 여기저기 흩어져 통제 불능이 되는 현상을 시크릿 스프롤(Secret Sprawl) 이라고 합니다. 구체적으로 어떤 문제가 생길까요?

깃 히스토리 박제

한 번 커밋된 시크릿은 나중에 지워도 커밋 히스토리에 남습니다. 공개 저장소라면 봇이 수 분 안에 긁어갑니다.

회수 불가능

키가 유출돼도 누가 어디서 쓰는지 모르면 폐기할 수 없습니다. 모든 곳을 동시에 바꾸지 못하면 서비스가 멈춥니다.

감사 추적 불가

누가 언제 그 비밀번호를 읽었는지 기록이 없습니다. 보안 사고가 나도 추적할 단서가 없습니다.

⚠️ 현실의 사고

매년 수만 건의 API 키·자격증명이 공개 깃허브 저장소에서 자동으로 탐지됩니다. 실수로 올린 AWS 키 하나로 수천만 원의 부정 사용 요금이 발생한 사례는 셀 수 없이 많습니다. Vault는 바로 이 "흩어진 비밀"을 한 금고로 모으는 데서 출발합니다.

02 — The Concept

Vault란 무엇인가

Vault를 한 문장으로 정의하면: "시크릿을 위한 단일 출처(Single Source of Truth)이자, 정체성 기반으로 접근을 통제하는 금고"입니다.

은행 금고를 떠올려 보세요. 모든 귀중품은 금고 안에 보관되고, 들어가려면 신분 확인을 거쳐야 하며, 출입 기록이 남고, 일부 물품은 정해진 시간만 빌릴 수 있습니다. Vault가 디지털 시크릿에 대해 정확히 이 역할을 합니다.

중앙 집중 저장 (Centralize)

흩어진 모든 시크릿을 한곳에 모읍니다. 저장 시 항상 암호화되어, 디스크를 통째로 훔쳐도 내용을 읽을 수 없습니다.

정체성 기반 접근 (Authenticate)

"이 사람/이 서버는 누구인가"를 먼저 확인한 뒤, 그 정체성에 허용된 시크릿만 내줍니다. 사람·앱·쿠버네티스 Pod 모두 가능합니다.

시간 제한 임대 (Lease)

시크릿을 영구히 주지 않고 "1시간짜리"처럼 임대(Lease)합니다. 만료되면 자동 폐기되어 유출돼도 곧 무용지물이 됩니다.

완전한 감사 로그 (Audit)

누가·언제·무엇을 요청했는지 모든 접근이 기록됩니다. 사고 발생 시 추적과 컴플라이언스 대응이 가능합니다.

💡 핵심 통찰

Vault의 진짜 가치는 "비밀번호를 암호화해서 저장한다"가 아닙니다. 정체성을 확인하고, 짧은 수명의 자격증명을 그때그때 발급하며, 모든 것을 기록한다는 점입니다. 단순한 암호화 저장소를 넘어선 "시크릿 운영 플랫폼"이라는 관점이 중요합니다.

03 — Architecture

핵심 아키텍처와 Seal/Unseal

Vault를 켜면 처음엔 잠겨(Sealed) 있습니다. 이 "봉인" 개념이 Vault 보안의 출발점입니다.

Vault의 모든 데이터는 배리어(Barrier) 라는 암호화 장벽 뒤에 저장됩니다. 데이터를 암호화하는 키는 암호화 키(Encryption Key)이고, 이 암호화 키 자체를 다시 암호화하는 키가 루트 키(Root Key)입니다. Vault가 켜진 직후엔 이 루트 키가 메모리에 없어서 아무것도 복호화할 수 없습니다. 이 상태가 바로 Sealed(봉인) 입니다.

샤미르의 비밀 분산 (Shamir's Secret Sharing)

그렇다면 루트 키는 누가 가지고 있을까요? Vault는 한 사람에게 전권을 주지 않습니다. 루트 키를 여러 조각(Key Share)으로 쪼개어 서로 다른 담당자에게 나눠줍니다. 기본 설정은 5조각으로 나누고, 그중 3조각이 모여야 금고를 열 수 있게(threshold) 합니다. 아래에서 직접 열어보세요.

🔓 Unseal 시뮬레이터 — 5개 중 3개를 모으세요

SEALED
Key Share 1
Key Share 2
Key Share 3
Key Share 4
Key Share 5
수집한 조각: 0 / 3 · 상태: 봉인됨

이렇게 하면 한 사람이 키를 잃어버리거나 악의를 품어도 단독으로 금고를 열 수 없습니다. 임계값(3)을 넘는 조각이 모여야만 루트 키가 메모리에서 재조합되고, 그제서야 Vault가 시크릿을 복호화해 줍니다.

Auto-Unseal — 사람 손 없이 깨우기

매번 사람이 세 명씩 모여 조각을 입력하는 건 자동화 환경에서 비현실적입니다. 그래서 프로덕션에서는 클라우드 KMS(AWS KMS, GCP KMS, Azure Key Vault) 또는 HSM에 루트 키 보관을 위임하는 Auto-Unseal을 씁니다. Vault가 재시작되면 KMS에 "이 키 좀 풀어줘"라고 요청해 스스로 봉인을 해제합니다.

Storage Backend

암호화된 데이터가 실제로 저장되는 곳. 현재는 Vault에 내장된 Raft 통합 스토리지가 표준입니다(과거엔 Consul도 사용).

Barrier

저장소로 드나드는 모든 데이터가 통과하는 암호화 장벽. 봉인 상태에서는 이 장벽이 닫혀 있습니다.

HA & Raft

여러 노드를 묶어 고가용성(HA)을 구성합니다. Raft 합의로 1개 Active 노드와 여러 Standby 노드가 동작합니다.

04 — Core Concepts

반드시 알아야 할 4대 핵심 개념

Vault를 이해한다는 건 사실상 이 네 가지 — 인증, 시크릿 엔진, 정책, 토큰 — 의 관계를 이해하는 것입니다.

① 인증 방법 (Auth Methods) — "너 누구야?"

Vault에 무언가를 요청하려면 먼저 자신이 누구인지 증명해야 합니다. Vault는 다양한 신원 증명 방식을 플러그인처럼 지원합니다.

  • Token — 가장 기본. 다른 인증의 결과로도 발급되는 출입증입니다.
  • AppRole — 사람이 아닌 애플리케이션·머신용. role_id(아이디)와 secret_id(비밀번호) 조합으로 로그인합니다.
  • Kubernetes — Pod의 ServiceAccount 토큰(JWT)으로 인증. 쿠버네티스 워크로드가 별도 비밀번호 없이 Vault에 로그인하는 핵심 방식입니다.
  • OIDC / LDAP / GitHub — 사람 사용자를 위한 SSO 연동. 기존 사내 계정으로 로그인합니다.

② 시크릿 엔진 (Secrets Engines) — "무엇을 보관·생성하나"

시크릿 엔진은 시크릿을 저장하거나 생성하는 모듈입니다. 경로(path)에 마운트하여 사용합니다.

KV (Key-Value)

가장 단순한 정적 시크릿 저장소. v2는 버전 관리를 지원해 이전 값을 되돌릴 수 있습니다.

Database

요청할 때마다 DB에 임시 계정을 즉석 생성. 동적 시크릿의 대표 사례입니다.

PKI

내부 인증서(TLS) 자동 발급·폐기. 짧은 수명 인증서를 자동화합니다.

Transit

데이터를 저장하지 않고 암호화/복호화만 대행하는 "서비스형 암호화(Encryption as a Service)".

③ 정책 (Policies) — "무엇을 허용하나"

정체성을 확인했다고 모든 걸 줄 순 없습니다. 정책은 "어떤 경로에 어떤 동작(읽기/쓰기/삭제)을 허용할지"를 HCL 문법으로 정의합니다. 기본은 모두 거부(deny) 이며, 명시적으로 허용한 것만 가능합니다(최소 권한 원칙).

# app-policy.hcl — 결제 앱은 자기 시크릿만 읽기 가능
path "secret/data/payment/*" {
  capabilities = ["read", "list"]
}

# DB 자격증명은 발급(읽기)만 허용, 설정 변경은 불가
path "database/creds/payment-role" {
  capabilities = ["read"]
}

④ 토큰 · 임대 · TTL (Token · Lease · TTL) — "얼마나 오래"

인증에 성공하면 Vault는 토큰을 발급합니다. 이후 모든 요청은 이 토큰으로 이뤄집니다. 토큰과 동적 시크릿에는 임대(Lease)가 붙고, TTL(Time To Live, 유효시간)이 지나면 자동 만료됩니다. 필요하면 만료 전 갱신(renew)하거나, 즉시 폐기(revoke)할 수 있습니다.

🧩 네 개념의 흐름

인증(누구야?) → 정책(뭘 줄까?) → 토큰(출입증 발급) → 시크릿 엔진(시크릿 제공) → 임대/TTL(시한부) — 이 사슬을 머릿속에 그릴 수 있다면 Vault의 절반은 이해한 것입니다.

05 — Hands-on

실습: 처음부터 끝까지

개발 모드로 띄워서 시크릿을 저장하고 읽는 가장 짧은 여정입니다. 운영용이 아닌 학습용 흐름입니다.

1단계 — 개발 서버 실행

학습용 -dev 모드는 자동으로 봉인이 해제된 상태로 떠서 바로 실습할 수 있습니다. (운영에서는 절대 사용 금지)

# 개발 모드 실행 (메모리 저장, 봉인 자동 해제)
vault server -dev

# 다른 터미널에서 주소와 토큰 설정
export VAULT_ADDR='http://127.0.0.1:8200'
export VAULT_TOKEN='dev-only-token'

vault status   # Sealed: false 확인
# OpenBao도 명령어/사용법이 거의 동일 (bao 바이너리)
bao server -dev

export BAO_ADDR='http://127.0.0.1:8200'
export BAO_TOKEN='dev-only-token'

bao status

2단계 — KV 시크릿 저장하고 읽기

가장 흔한 작업. CLI, REST API, 그리고 앱에서의 사용을 모두 비교해 보세요.

# 저장
vault kv put secret/payment/db \
    username=app_user password=s3cr3t-from-vault

# 읽기
vault kv get secret/payment/db

# 특정 필드만 추출 (스크립트에 유용)
vault kv get -field=password secret/payment/db
# 저장 (KV v2는 data 안에 data로 감쌈)
curl -H "X-Vault-Token: $VAULT_TOKEN" \
  -X POST "$VAULT_ADDR/v1/secret/data/payment/db" \
  -d '{"data":{"username":"app_user","password":"s3cr3t"}}'

# 읽기
curl -H "X-Vault-Token: $VAULT_TOKEN" \
  "$VAULT_ADDR/v1/secret/data/payment/db"
# pip install hvac
import hvac

client = hvac.Client(url="http://127.0.0.1:8200",
                     token="dev-only-token")

secret = client.secrets.kv.v2.read_secret_version(
    path="payment/db")

db_pw = secret["data"]["data"]["password"]
# 코드 어디에도 비밀번호 평문이 없습니다 👍

3단계 — 정책 적용과 AppRole 로그인

앞서 만든 정책을 등록하고, 앱이 그 권한으로 로그인하도록 연결합니다.

# 1) 정책 등록
vault policy write app-policy app-policy.hcl

# 2) AppRole 인증 활성화 후 역할 생성
vault auth enable approle
vault write auth/approle/role/payment-app \
    token_policies="app-policy" token_ttl=1h

# 3) 앱은 role_id + secret_id 로 로그인 → 토큰 획득
vault read auth/approle/role/payment-app/role-id
vault write -f auth/approle/role/payment-app/secret-id
06 — Killer Feature

동적 시크릿 — Vault의 진짜 무기

정적 시크릿은 그냥 "금고에 둔 비밀번호"입니다. 동적 시크릿은 "필요할 때 즉석에서 만들고 곧 폐기하는 일회용 비밀번호"입니다. 이 차이가 보안 수준을 바꿉니다.

예를 들어 결제 앱이 DB에 붙어야 한다고 합시다. 정적 방식이라면 모든 인스턴스가 같은 비밀번호를 공유하고, 그 비밀번호는 오래 살아 있습니다. 동적 방식이라면 Vault가 요청 시점에 DB에 그 인스턴스만의 임시 계정을 만들어 주고, TTL이 지나면 그 계정을 DB에서 삭제합니다.

🔁 정적 vs 동적 — 자격증명의 일생

# DB 엔진 활성화 & PostgreSQL 연결 등록
vault secrets enable database
vault write database/config/app-db \
    plugin_name=postgresql-database-plugin \
    connection_url="postgresql://{{username}}:{{password}}@db:5432/app" \
    allowed_roles="payment-role"

# "1시간짜리 읽기 계정"을 만드는 역할 정의
vault write database/roles/payment-role \
    db_name=app-db default_ttl=1h max_ttl=24h \
    creation_statements="CREATE ROLE \"{{name}}\" ..."

# 앱이 요청 → 매번 새로운 임시 DB 계정 발급 ✨
vault read database/creds/payment-role
# username: v-approle-payment-x9f2..  password: A1b2C3..  lease: 1h
💡 왜 강력한가

자격증명이 유출돼도 길어야 1시간이면 자동 폐기됩니다. 인스턴스마다 계정이 달라서 어떤 계정이 사고를 냈는지 즉시 특정할 수 있고, 비밀번호 회전(rotation)을 사람이 신경 쓸 필요가 없습니다. "유출 시 피해 시간"을 영구에서 분 단위로 줄이는 것이 동적 시크릿의 본질입니다.

07 — Comparison

다른 보안 툴과의 비교

Vault만 시크릿을 다루는 건 아닙니다. 각 도구는 잘하는 영역이 다릅니다. "언제 무엇을 쓸지"의 감을 잡아봅시다.

도구유형동적 시크릿멀티 클라우드운영 부담가장 잘 맞는 곳
HashiCorp Vault 전용 시크릿 플랫폼 (자체 운영/HCP) 강력함 예 높음 멀티클라우드·동적 시크릿·서비스형 암호화가 필요한 조직
AWS Secrets Manager 매니지드 (AWS 종속) 자동 회전 (Lambda 기반) 아니오 낮음 AWS만 쓰고, 운영 최소화가 우선인 팀
Azure Key Vault 매니지드 (Azure 종속) 제한적 아니오 낮음 Azure 중심, HSM 기반 키 관리가 필요한 팀
GCP Secret Manager 매니지드 (GCP 종속) 아니오 아니오 낮음 GCP에서 단순 KV + 버전 관리만 필요할 때
Kubernetes Secrets K8s 내장 객체 아니오 — 없음 임시/저민감 값. 기본은 base64일 뿐 암호화 아님
SOPS / Sealed Secrets GitOps형 암호화 파일 아니오 부분 낮음 시크릿을 깃에 암호화해 저장하는 GitOps 워크플로
CyberArk 엔터프라이즈 PAM 계정 중심 부분 높음 사람의 특권 계정 접근 통제가 핵심인 대기업

핵심 구분선

매니지드 클라우드 시크릿 매니저

AWS/Azure/GCP 제공 서비스는 운영이 거의 필요 없고 IAM과 깔끔히 통합됩니다. 대신 해당 클라우드에 묶입니다. 단일 클라우드라면 가장 합리적인 출발점입니다.

Vault / OpenBao

멀티클라우드 중립성, 동적 시크릿, 서비스형 암호화, PKI 등 기능 폭이 가장 넓습니다. 대신 직접 운영(HA·백업·업그레이드)하는 부담이 큽니다.

Kubernetes Secrets의 함정

K8s 기본 Secret은 암호화가 아니라 base64 인코딩일 뿐입니다. etcd 암호화를 별도로 켜지 않으면 사실상 평문에 가깝습니다. 진짜 시크릿은 Vault 등 외부에 두고 동기화하는 패턴이 권장됩니다.

SOPS / Sealed Secrets

"시크릿을 깃에 안전하게 커밋"하려는 GitOps 친화 도구입니다. 가볍지만 동적 시크릿·중앙 감사·즉시 폐기 같은 Vault의 운영 기능은 없습니다.

🧭 의사결정 한 줄 요약

단일 클라우드 + 운영 최소화 → 클라우드 네이티브 매니저. 멀티클라우드 + 동적 시크릿/암호화 서비스 → Vault/OpenBao. 깃 기반 선언적 배포 → SOPS·Sealed Secrets(또는 ESO로 Vault 연동). 사람의 특권 계정 통제가 본질 → CyberArk.

08 — Applications

클라우드 엔지니어를 위한 응용 분야

개념을 알았으니, 실무에서 Vault가 어디에 꽂히는지 봅시다. 특히 쿠버네티스 연동이 핵심입니다.

쿠버네티스 시크릿 주입 — 세 가지 패턴

Vault Agent Injector

Mutating Webhook으로 Pod에 사이드카를 자동 주입. 시크릿을 파일로 마운트해 줍니다. 앱 코드 수정이 거의 필요 없습니다.

Secrets Store CSI Driver

CSI 볼륨으로 시크릿을 Pod 파일시스템에 마운트. 쿠버네티스 표준 인터페이스를 따릅니다.

External Secrets Operator

Vault의 값을 표준 K8s Secret 객체로 동기화. 멀티 백엔드를 통일된 방식으로 다루는 커뮤니티 표준입니다.

CI/CD 파이프라인

GitLab CI·GitHub Actions·Argo가 OIDC/JWT로 Vault에 로그인해, 빌드 시점에 단명 자격증명을 받아 씁니다. 파이프라인 변수에 비밀번호를 박을 필요가 없습니다.

동적 DB 자격증명

MySQL·PostgreSQL·MongoDB·클라우드 RDS에 대해 단명 계정을 자동 발급·폐기. 자격증명 회전을 운영에서 사라지게 합니다.

PKI / 내부 인증서 자동화

서비스 간 mTLS에 필요한 짧은 수명 인증서를 Vault가 자동 발급. 인증서 만료 사고를 구조적으로 예방합니다.

서비스형 암호화 (Transit)

앱이 암호화 로직을 직접 구현하지 않고 Vault에 위임. 주민번호·카드번호 같은 민감 필드를 안전하게 암복호화합니다.

09 — 2026 Landscape

꼭 알아야 할 라이선스 & OpenBao

Vault를 도입하기 전 반드시 짚어야 할 주제입니다. 2023년 이후 Vault의 라이선스와 소유 구조가 크게 바뀌었습니다.

HashiCorp는 2023년 8월, Vault를 포함한 핵심 제품의 라이선스를 오픈소스 MPL 2.0에서 BSL(Business Source License) 1.1로 변경했습니다. BSL에서는 사용·수정·재배포가 대부분 허용되지만, Vault를 가지고 HashiCorp의 상용 제품과 경쟁하는 호스팅 서비스를 제공하는 것은 금지됩니다. 또한 각 릴리스는 발표일로부터 4년이 지나면 MPL 2.0으로 전환됩니다.

그리고 IBM은 2025년 2월 27일 약 64억 달러 규모의 HashiCorp 인수를 완료했고, 이제 Vault는 IBM의 소프트웨어 제품입니다. IBM 체제에서 릴리스 주기는 "IBM Support Cycle-2"로 바뀌어, 메이저 릴리스가 대략 연 1회로 정렬되며 v1.21에서 v2.0으로 점프했습니다.

OpenBao — 커뮤니티 포크

BSL 변경에 대응해 만들어진 것이 OpenBao입니다. Vault의 마지막 MPL 2.0 버전인 1.14.0에서 분기했고, 바이너리 이름은 bao로 바뀌었습니다. OpenBao는 Linux Foundation 산하에서 OSI 승인 오픈소스 라이선스인 MPL 2.0로 운영되며, 상업적 사용 제한이 없습니다.

항목HashiCorp VaultOpenBao
라이선스BSL 1.1 (4년 후 MPL 전환)MPL 2.0 (진정한 OSS)
거버넌스HashiCorp / IBMLinux Foundation
API 호환—Vault와 사실상 동일
Namespaces(멀티테넌시)Enterprise 전용무료 포함
재해복구(DR) 복제Enterprise 전용미지원(우회 필요)
상용 지원HashiCorp / IBM커뮤니티 / 서드파티

두 제품은 시크릿 엔진과 API 수준에서 사실상 같은 소프트웨어이며, Vault를 다뤄봤다면 OpenBao도 동일하게 느껴집니다. 따라서 선택의 핵심은 기능 차이가 아니라 라이선스와 거버넌스입니다. OSI 승인 라이선스나 특정 벤더에 종속되지 않는 중립적 거버넌스가 요구사항이라면 OpenBao가 답이고, 그렇지 않다면 Vault의 성숙도와 엔터프라이즈 기능이 실질적 장점입니다.

📌 2026년 6월 기준 현황

Vault Community 최신 라인은 2.0.x(2.0.3, 2026-06 릴리스), OpenBao 최신 라인은 2.5.x(2.5.0, 2026-02 릴리스)입니다. 학습은 어느 쪽으로 시작해도 명령어와 개념이 그대로 통용됩니다.

10 — Production

프로덕션 베스트 프랙티스

실습 모드와 운영은 다릅니다. 실서비스에 올리기 전 최소한 이것들은 챙기세요.

① 최소 권한 원칙

정책은 기본 거부에서 출발해 꼭 필요한 경로만 허용하세요. root 토큰은 초기 설정 후 즉시 폐기하고 평소엔 쓰지 않습니다.

② Auto-Unseal 구성

클라우드 KMS나 HSM으로 자동 봉인 해제를 설정해 재시작 시 사람이 개입하지 않도록 합니다. 복구 키는 안전하게 분산 보관합니다.

③ HA + 정기 백업

Raft 통합 스토리지로 최소 3노드 HA를 구성하고, 스냅샷 백업을 자동화합니다. 복구 절차도 실제로 리허설하세요.

④ 감사 로그 활성화

운영에서는 반드시 Audit Device를 켜고 로그를 SIEM으로 수집합니다. 단, 감사 로그 저장소 장애 시 Vault가 멈출 수 있으니 다중화하세요.

⑤ 동적 시크릿 우선

가능한 곳은 정적 KV 대신 동적 시크릿/회전을 사용해 "유출 시 노출 시간"을 최소화합니다.

⑥ 짧은 TTL과 갱신

토큰·임대의 TTL을 짧게 잡고 자동 갱신(renew) 패턴을 씁니다. 떠나간 워크로드의 자격증명은 즉시 revoke 합니다.

⚠️ 흔한 실수

-dev 모드를 운영에 쓰는 것, root 토큰을 앱에 박아두는 것, Unseal 키 조각을 한 사람이 전부 보관하는 것 — 이 셋은 Vault를 쓰는 의미를 무력화합니다. 절대 피하세요.
