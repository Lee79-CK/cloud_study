Cisco ACI 기초 완전 정복

스위치마다 CLI로 명령어를 치던 시대는 끝났습니다. ACI는 네트워크를 "장비의 집합"이 아니라 "애플리케이션이 필요로 하는 정책의 집합"으로 바라봅니다. 이 강의는 전통적인 네트워크 구성과 하나하나 비교하면서, ACI의 개념을 그림·비유·인터랙티브 데모로 풀어냅니다.

난이도 입문 예상 학습 40–55분 사전지식 VLAN · IP · 스위치 기초 키워드 SDN · Spine-Leaf · EPG · Contract
Chapter 01 · Motivation

왜 ACI가 등장했나

ACI를 배우기 전에, "기존 방식의 무엇이 불편했는가"를 먼저 이해해야 합니다. 문제를 알아야 해법이 보입니다.

전통 네트워크의 세 가지 통증

🧰

장비마다 수작업

스위치 50대면 50번 SSH 접속해서 VLAN·ACL을 손으로 넣습니다. 오타 하나가 장애로 이어지고, "누가 언제 무엇을 바꿨는지" 추적이 어렵습니다.

🕸️

VLAN·STP의 한계

VLAN은 4096개가 한계, Spanning Tree는 링크의 절반을 놀립니다. 서버가 늘수록 브로드캐스트 도메인이 커지고 장애 파급 범위도 커집니다.

🐢

느린 provisioning

앱 하나 올리려면 네트워크 팀에 티켓을 넣고 며칠 기다립니다. 클라우드는 몇 초 만에 배포되는데 네트워크만 발목을 잡습니다.

🏙️
비유로 이해하기 — 전통 네트워크는 집집마다 찾아가 수도 밸브를 손으로 잠그는 방식입니다. ACI는 중앙 관제실에서 "3동 501호는 물을 A동과만 주고받게 하라"는 정책을 한 번 내리면, 시스템이 알아서 모든 밸브를 맞춰주는 방식입니다. 사람은 "무엇을 원하는지(의도, intent)"만 말하고, "어떻게 설정할지(장비별 명령)"는 시스템이 계산합니다.
Key Term · SDN

ACI는 시스코가 만든 SDN(Software-Defined Networking) 구현체입니다. SDN의 핵심은 "제어(control plane)"와 "전달(data plane)"을 분리하는 것. 즉 어디로 보낼지 정하는 두뇌를 스위치에서 떼어내 중앙(APIC)으로 모으고, 스위치는 빠르게 패킷을 전달하는 근육 역할만 하도록 나눕니다.

한 문장 요약

ACI = "애플리케이션이 원하는 통신 정책"을 중앙에서 선언하면, 물리 패브릭 전체가 자동으로 그에 맞게 구성되는 데이터센터 네트워크 시스템. 이름 그대로 Application Centric(애플리케이션 중심) Infrastructure입니다.

Chapter 02 · Comparison

전통 네트워크 vs ACI

가장 큰 사고방식의 차이부터 잡고 갑니다. 뒤 챕터의 모든 개념은 이 표에서 파생됩니다.

LEGACY 전통 3-Tier 네트워크

  • Core–Aggregation–Access 3계층 구조
  • 장비별 CLI로 개별 설정 (박스 바이 박스)
  • VLAN·서브넷·IP가 곧 "위치"를 의미
  • 보안은 방화벽·ACL을 경로마다 배치
  • STP로 루프 방지 → 링크 절반 낭비
  • 변경 = 여러 장비를 순서대로 손댐
  • "어떻게(How)"를 사람이 직접 지정
VS

ACI 정책 기반 패브릭

  • Spine–Leaf 2계층 (모든 리프가 동등)
  • APIC 한 곳에서 정책 선언 → 자동 배포
  • 위치와 정책을 분리 (EPG = 논리 그룹)
  • 보안은 EPG 간 Contract로 내장
  • ECMP로 모든 링크를 동시에 활용
  • 변경 = 정책 객체 한 번 수정
  • "무엇을 원하는가(What)"만 선언
가장 중요한 관점 전환

전통 네트워크에서 "IP 주소 = 그 장비가 무엇인지(정체성)"였습니다. 웹서버는 10.1.10.0/24, DB는 10.1.20.0/24 식으로 서브넷이 곧 역할이었죠. ACI는 이 둘을 분리합니다. "이 엔드포인트가 어느 그룹(EPG)에 속하는가"로 정체성을 정하고, IP·VLAN·위치는 부차적인 정보가 됩니다. 이 개념 하나가 ACI 전체를 관통합니다.

선언형 vs 명령형 — 한눈에

이 "선언형(declarative)" 개념이 익숙하신가요? 맞습니다 — Kubernetes의 YAML 매니페스트, Terraform의 .tf와 정확히 같은 철학입니다. "원하는 상태(desired state)"를 선언하면 컨트롤러가 현재 상태를 그에 맞게 조정(reconcile)하죠. ACI의 APIC는 네트워크판 kube-controller-manager라고 생각하면 감이 잡힙니다.

Chapter 03 · Physical Fabric

Spine-Leaf 패브릭 구조

먼저 "물리적으로 어떻게 생겼는가"를 봅니다. ACI 하드웨어는 놀랄 만큼 단순한 규칙을 따릅니다.

3-Tier에서 Spine-Leaf(2-Tier)로

전통 데이터센터는 Core→Aggregation→Access 3계층이었고, 서버-서버 통신이 위아래로 여러 번 오르내렸습니다. 가상화로 인해 동-서(East-West, 서버↔서버) 트래픽이 폭증하자 이 구조는 병목이 됐습니다. ACI는 Clos 토폴로지 기반의 Spine-Leaf 2계층으로 이를 해결합니다.

인터랙티브 토폴로지 각 노드를 클릭해 역할을 확인하세요
💡 어떤 두 서버든 항상 "리프→스파인→리프" 3홉
노드를 선택하세요

위 다이어그램의 스파인·리프·APIC·서버를 클릭하면 각 구성요소의 역할이 여기에 표시됩니다.

패브릭의 절대 규칙

  • 모든 리프는 모든 스파인에 연결된다 — 하지만 리프끼리, 스파인끼리는 절대 직접 연결하지 않습니다. (full-mesh between tiers)
  • 서버·방화벽·라우터 등 모든 장비는 리프에만 붙는다 — 스파인에는 오직 리프만 연결됩니다.
  • 어떤 두 엔드포인트든 거리가 균일 — 항상 "리프 → 스파인 → 리프" 최대 3홉. 예측 가능한 지연시간(latency).
  • 모든 링크를 동시에 사용(ECMP) — STP처럼 링크를 막지 않아 대역폭을 100% 활용합니다.
  • 확장은 리프 추가 — 서버가 부족하면 리프를 늘리고, 대역폭이 부족하면 스파인을 늘립니다.
Underlay & Overlay

패브릭 내부는 VXLAN으로 동작합니다. 물리 배선(underlay)은 IS-IS 라우팅으로 리프·스파인을 연결하고, 그 위에 VXLAN 터널(overlay)을 씌워 논리 네트워크를 만듭니다. 덕분에 VLAN 4096개 한계를 넘어 1600만 개(24bit VNID)의 세그먼트를 만들 수 있고, 서버가 어느 리프에 붙든 같은 네트워크에 속할 수 있습니다. 즉 "물리 위치"와 "논리 소속"이 완전히 분리됩니다.

🛄
VXLAN 비유 — underlay는 고속도로(물리 도로망), overlay는 그 위를 달리는 택배 상자입니다. 출발지·목적지가 다른 도시(다른 리프)라도, 상자(VXLAN 캡슐)에 "실제 수취인 정보"를 넣어두면 고속도로는 그저 상자를 빠르게 옮기기만 하면 됩니다. 도로는 상자 내용물(원래 이더넷 프레임)을 알 필요가 없습니다.
Chapter 04 · Controller

APIC — 패브릭의 두뇌

ACI에서 사람이 직접 대화하는 유일한 대상, APIC를 이해합니다.

APIC(Application Policy Infrastructure Controller)는 ACI의 중앙 컨트롤러입니다. 네트워크 관리자는 개별 스위치에 로그인하지 않고 오직 APIC와만 대화합니다. APIC가 모든 정책을 저장하고, 물리 스위치들에게 "이렇게 동작하라"고 자동으로 밀어 넣습니다(push down).

🧠

단일 정책 저장소

모든 논리 객체(테넌트·EPG·Contract)가 APIC의 객체 트리에 저장됩니다. 진실의 원천(single source of truth)입니다.

🔁

클러스터로 운영

보통 3대(홀수) 이상으로 클러스터를 구성해 고가용성을 확보합니다. APIC가 다 죽어도 패브릭 통신은 계속됩니다.

🔌

API 우선 설계

GUI·CLI가 하는 모든 일은 REST API로도 가능합니다. 자동화·IaC·CI/CD 파이프라인에 그대로 연결됩니다.

흔한 오해 바로잡기

APIC는 데이터 경로에 끼어들지 않습니다. 즉 서버 A→B 패킷이 APIC를 거치지 않습니다. APIC는 정책을 만들고 배포·감시할 뿐, 실제 패킷 전달은 리프·스파인 스위치가 합니다. 그래서 APIC 클러스터가 전부 다운돼도 기존 트래픽은 멈추지 않습니다(다만 새 정책 변경만 불가). 컨트롤 플레인과 데이터 플레인 분리의 실제 이점입니다.

같은 작업, 세 가지 인터페이스

"App EPG에 Contract를 붙인다"는 동일한 작업을 GUI·REST·자동화 도구로 각각 표현하면 이렇습니다. 초보자는 GUI로 시작하고, 운영이 커지면 API로 넘어갑니다.

# APIC GUI에서의 클릭 순서 (선언형 사고)
1. Tenants  →  Prod 테넌트 선택
2. Application Profiles  →  ShopApp
3. EPGs  →  EPG-App 선택
4. Contracts  →  "Add Consumed Contract"
5. Contract 이름 = web-to-app 선택  →  Submit

# 끝. 관련된 모든 리프 스위치가 자동으로 설정됨.
// POST /api/mo/uni/tn-Prod.json  — 원하는 "상태"를 그대로 선언
{
  "fvAEPg": {
    "attributes": { "name": "EPG-App" },
    "children": [{
      "fvRsCons": {           // Consumed contract 관계
        "attributes": { "tnVzBrCPName": "web-to-app" }
      }
    }]
  }
}
// APIC가 이 객체를 받으면 패브릭 전체를 알아서 맞춤
# 클라우드 엔지니어에게 가장 익숙한 방식 — IaC
- name: App EPG가 web-to-app 계약을 소비하도록 보장
  cisco.aci.aci_epg_to_contract:
    host: apic.example.com
    tenant: Prod
    ap: ShopApp
    epg: EPG-App
    contract: web-to-app
    contract_type: consumer
    state: present   # 선언형: "있어야 한다"
# 전통 방식이라면? — 관련 스위치마다 반복 (명령형)
# switch-1
interface Vlan10
  ip access-group WEB_TO_APP in
ip access-list WEB_TO_APP
  permit tcp 10.1.10.0/24 10.1.20.0/24 eq 8080
# switch-2 ... switch-N 에서 또 반복 (사람이 일관성 책임)

▲ 같은 의도, 다른 표현. ACI는 "무엇을" 선언하고, 전통 방식은 "어떻게"를 장비마다 지시합니다.

Chapter 05 · Logical Model

핵심 논리 객체 6가지

ACI를 "말하다"는 것은 곧 이 객체들을 조합하는 것입니다. 위에서 아래로 계층 구조를 이룹니다.

계층 구조 (외우기)

Tenant → VRF → Bridge Domain → Subnet 이 "네트워크 골격"이고, Application Profile → EPG 가 "애플리케이션 그룹"입니다. EPG는 BD 위에 놓입니다. 비유하면 Tenant는 건물, VRF는 층, BD는 방, EPG는 방 안의 사람 그룹입니다.

🏢

① Tenant (테넌트)

정책·설정을 담는 최상위 논리 컨테이너. 부서·고객·환경(운영/개발)별로 격리합니다. 멀티테넌시의 단위이며, 클라우드의 "프로젝트/구독"과 유사합니다.

🗺️

② VRF (Context)

독립된 라우팅 테이블 = L3 격리 공간. 하나의 테넌트 안에서 서로 다른 IP 대역이 겹쳐도 됩니다. 전통의 VRF와 동일 개념, "누구와 라우팅되는가"를 정의합니다.

🚪

③ Bridge Domain (BD)

L2 경계(브로드캐스트 도메인). 전통 VLAN과 비슷하지만 훨씬 유연합니다. 하나의 BD 안에 여러 서브넷을 둘 수 있고, VRF에 소속됩니다.

📮

④ Subnet (게이트웨이)

BD에 설정하는 IP 대역과 게이트웨이 IP. ACI는 "애니캐스트 게이트웨이"라, 같은 게이트웨이 IP를 모든 리프가 동시에 제공합니다(=서버가 어디 있든 최적 경로).

👥

⑤ EPG (Endpoint Group) ★

ACI의 심장. "같은 정책을 받아야 하는 엔드포인트들의 묶음". 웹서버 10대가 IP·VLAN이 달라도 하나의 EPG-Web으로 묶입니다. 정책은 EPG 단위로 적용됩니다.

📦

⑥ Application Profile (AP)

하나의 애플리케이션을 이루는 EPG들의 묶음. 예: ShopApp = { EPG-Web, EPG-App, EPG-DB }. 앱 관점으로 네트워크를 문서화합니다.

EPG를 다시, 천천히

EPG가 낯설다면 이렇게 생각하세요. 전통 방식에서는 "이 서버는 VLAN 10에 있으니까 웹서버다"라고 위치(VLAN)로 정체성을 추론했습니다. ACI는 반대로 "이 서버는 EPG-Web에 속한다. 그래서 웹 정책을 받는다"고 명시적으로 그룹에 배정합니다. 배정 기준은 매우 유연합니다:

  • 붙어 있는 물리 포트 기준 (Leaf-1의 eth1/5)
  • VLAN 태그 기준 (전통과 호환)
  • 가상머신이 붙은 vCenter 포트그룹 기준 (VMM 연동)
  • IP·MAC 등 속성(attribute) 기준 (μSeg EPG)
쿠버네티스로 치면

EPG는 쿠버네티스의 Label로 묶은 파드 그룹 + NetworkPolicy의 대상 selector와 매우 닮았습니다. "이 라벨을 가진 것들에 이 정책을 적용" = "이 EPG에 이 Contract를 적용". 실제로 ACI의 쿠버네티스 CNI는 네임스페이스·Deployment를 EPG로 매핑합니다(챕터 10).

Chapter 06 · Policy

Contract — ACI의 방화벽 규칙

EPG를 만들었으면, 이제 "누가 누구와 통신할 수 있는가"를 정해야 합니다. 그게 Contract입니다.

화이트리스트 모델 (매우 중요)

ACI의 기본값은 "모두 차단(default deny)"입니다. 같은 EPG 안의 엔드포인트끼리는 자유롭게 통신하지만, 서로 다른 EPG는 Contract를 명시적으로 맺기 전까지 절대 통신할 수 없습니다. 전통 네트워크가 "일단 통하게 하고 위험한 것만 막는(blacklist)" 방식이었다면, ACI는 "일단 다 막고 허용할 것만 여는(whitelist)" 제로 트러스트 방식입니다.

Provider와 Consumer

Contract는 방향이 있는 관계입니다. 서비스를 제공하는 쪽(Provider)과 소비하는 쪽(Consumer)이 있습니다. 웹→앱 통신이라면, App 서비스를 제공하는 EPG-App이 Provider, 그 서비스를 호출하는 EPG-Web이 Consumer입니다.

Contract의 내부 구조

Contract는 러시아 인형처럼 겹겹이 들어 있습니다. 아래에서 위로 조립됩니다.

기억할 규칙

Contract는 EPG 사이(inter-EPG)에만 필요합니다. 같은 EPG 내부(intra-EPG)는 기본적으로 자유롭게 통신합니다. 따라서 "같은 정책을 받아야 할 것들을 하나의 EPG로 묶는" 설계가 곧 보안 설계가 됩니다.

Chapter 08 · Integration

가상화 · L4-L7 서비스 통합

ACI가 진짜 강력해지는 지점 — 물리 네트워크가 가상 세계와 방화벽·LB까지 하나의 정책으로 엮입니다.

VMM Domain — 가상화 연동

ACI는 VMware vCenter, Microsoft SCVMM, OpenStack, Red Hat 등과 API로 연동됩니다. 이를 VMM(Virtual Machine Manager) Domain이라 합니다. 연동하면 APIC에서 EPG를 만들 때 vCenter에 포트그룹이 자동으로 생성되고, VM을 그 포트그룹에 붙이는 순간 해당 VM이 EPG에 소속됩니다. 즉 서버 담당자와 네트워크 담당자의 경계가 자동화로 이어집니다.

  • vMotion을 해도 정책이 따라간다 — VM이 다른 호스트(다른 리프)로 이동해도 EPG 소속이 유지되어 보안 정책이 끊기지 않습니다.
  • 포트그룹 수동 생성 불필요 — EPG ↔ 포트그룹이 자동 동기화됩니다.
  • μSeg(마이크로 세그멘테이션) — 같은 서브넷 안에서도 VM 이름·OS·IP 속성으로 더 잘게 격리할 수 있습니다.

Service Graph — 방화벽·LB를 정책에 삽입

전통 방식에서 "웹→앱 트래픽을 방화벽에 통과시키려면" 경로·VLAN·라우팅을 수작업으로 꺾어야 했습니다(트롬본잉). ACI는 Service Graph로 이를 정책화합니다. Contract에 "이 통신은 방화벽을 거쳐라"를 선언하면, APIC가 트래픽 리다이렉션(PBR)을 자동 구성합니다.

PBR (Policy-Based Redirect)

Service Graph의 핵심 기술. "EPG-Web → EPG-App 트래픽은 방화벽 클러스터로 우회시켜라"를 Contract 안에 넣으면, 네트워크 경로를 다시 설계하지 않고도 보안 장비를 정책적으로 삽입할 수 있습니다. 방화벽을 추가·교체해도 서버·라우팅 설정은 그대로입니다.

🧱

Physical Domain

베어메탈 서버·물리 방화벽 등 가상화되지 않은 장비를 EPG에 연결할 때 사용하는 도메인. VLAN Pool과 함께 물리 포트를 EPG에 매핑합니다.

☁️

VMM Domain

하이퍼바이저 매니저와 API 연동. EPG↔포트그룹 자동 생성, VM 이동 시 정책 추적, μSeg 지원.

Chapter 09 · Hands-on

실습 시나리오: 3-tier 앱 배포

지금까지 배운 객체를 순서대로 조립해 "쇼핑몰 앱(Web–App–DB)"을 배포하는 전체 흐름을 따라갑니다.

목표 아키텍처

인터넷 → EPG-Web → EPG-App → EPG-DB. 각 계층은 바로 옆 계층과만 통신하고, Web은 DB에 직접 접근할 수 없어야 합니다(제로 트러스트). 이 "이웃하고만 대화" 규칙이 Contract로 강제됩니다.

STEP 1  Tenant 생성 ................ "Prod"
STEP 2  VRF 생성 ................... "Prod-VRF"  # L3 격리
STEP 3  Bridge Domain 생성 ......... "BD-Shop" (VRF에 연결)
STEP 4  Subnet 추가 ............... 10.10.0.1/24 (애니캐스트 GW)
STEP 5  Application Profile ....... "ShopApp"
STEP 6  EPG 3개 생성 .............. EPG-Web / EPG-App / EPG-DB
STEP 7  Filter + Contract ......... web→app(tcp/8080), app→db(tcp/3306)
STEP 8  Contract 결선 ............. Provider/Consumer 지정
STEP 9  도메인 바인딩 ............. VMM 또는 물리 포트에 EPG 연결
STEP 10 검증 ..................... 엔드포인트 학습·연결성 확인
# 핵심만 추린 선언형 플레이북 (반복 태스크는 생략)
- cisco.aci.aci_tenant: { tenant: Prod, state: present }

- cisco.aci.aci_vrf:
    tenant: Prod
    vrf: Prod-VRF

- cisco.aci.aci_bd:
    tenant: Prod
    bd: BD-Shop
    vrf: Prod-VRF

- cisco.aci.aci_bd_subnet:
    tenant: Prod
    bd: BD-Shop
    gateway: 10.10.0.1
    mask: 24

- cisco.aci.aci_ap:
    tenant: Prod
    ap: ShopApp

# EPG 3종 (loop로 생성)
- cisco.aci.aci_epg:
    tenant: Prod
    ap: ShopApp
    epg: "{{ item }}"
    bd: BD-Shop
  loop: [EPG-Web, EPG-App, EPG-DB]

# 이후 filter → contract → provider/consumer 결선
# 엔드포인트가 잘 "학습"됐는지 확인 (핵심 진단)
APIC> show endpoints tenant Prod
   MAC              IP           EPG        Encap   Leaf
   00:50:56:aa..    10.10.0.11   EPG-Web    vlan-.. Leaf-1
   00:50:56:bb..    10.10.0.21   EPG-App    vlan-.. Leaf-2

# 통신이 안 될 때 체크리스트
1) Contract가 Provider/Consumer 양쪽에 걸렸나?  # 한쪽만 X
2) Filter의 포트·프로토콜이 맞나?
3) EPG가 올바른 BD에 속했나?
4) 도메인(VMM/물리)이 EPG에 바인딩됐나?
5) Health Score / Faults 화면에 경고가 있나?
초보가 가장 많이 겪는 장애

"Contract를 한쪽에만 걸었다." Contract는 반드시 한 EPG가 Provider, 다른 EPG가 Consumer로 양쪽 모두 결선되어야 통신이 열립니다. Provider만 걸고 Consumer를 빠뜨리면 트래픽은 계속 막힙니다. 두 번째로 흔한 것은 도메인 바인딩 누락 — EPG는 만들었는데 물리/가상 포트에 연결을 안 해서 엔드포인트가 학습되지 않는 경우입니다.

Chapter 10 · Cloud & K8s

클라우드 · 쿠버네티스 연계

클라우드 엔지니어에게 가장 실전적인 부분. ACI가 컨테이너·멀티사이트로 어떻게 확장되는지 봅니다.

ACI CNI — 쿠버네티스와 네트워크를 하나로

ACI는 쿠버네티스용 CNI 플러그인을 제공합니다. 이를 쓰면 파드 네트워크와 물리 패브릭이 분리되지 않고 하나로 통합됩니다.

  • 네임스페이스·Deployment → EPG 자동 매핑 — 쿠버네티스 오브젝트가 ACI 정책 객체로 반영됩니다.
  • K8s NetworkPolicy → ACI Contract — 쿠버네티스에서 선언한 네트워크 정책이 하드웨어 레벨에서 강제됩니다.
  • 가시성 통합 — 파드 단위 트래픽을 APIC에서 관찰·트러블슈팅할 수 있습니다.
  • 하드웨어 로드밸런싱 — Service(LoadBalancer)를 패브릭이 직접 분산 처리합니다.
🔗
왜 좋은가 — 보통 쿠버네티스 네트워크(오버레이 CNI)와 물리 네트워크는 서로를 모르는 두 세계입니다. 문제가 생기면 "파드 문제냐, 네트워크 문제냐"로 팀이 싸웁니다. ACI CNI는 이 둘을 같은 정책 언어(EPG·Contract)로 통일해 단일 창(single pane)에서 보게 해줍니다.

Multi-Pod · Multi-Site — 여러 데이터센터로 확장

🏬

Multi-Pod

지연이 낮은(같은/인접 캠퍼스) 여러 패브릭을 하나의 APIC 클러스터가 관리. 단일 관리 도메인처럼 동작합니다. IPN(Inter-Pod Network)으로 연결.

🌐

Multi-Site

지리적으로 떨어진 데이터센터를 Nexus Dashboard Orchestrator(NDO)가 총괄. 각 사이트는 독립 APIC를 갖고, 정책만 상위에서 조율. 재해복구·확장에 사용.

퍼블릭 클라우드로

온프레미스 ACI 정책을 AWS·Azure까지 확장하는 Cloud Network Controller(구 Cloud APIC)도 있습니다. 온프레미스의 EPG·Contract 개념을 클라우드 네이티브 구성(보안그룹·VPC 등)으로 번역·동기화해, 하이브리드 환경에서 일관된 정책을 유지하도록 돕습니다. 온프레미스 대규모 인프라를 클라우드로 확장·마이그레이션하는 시나리오에서 검토 가치가 큽니다.

이해도 점검 퀴즈

Q1.APIC 클러스터가 전부 다운되면 서버 간 기존 트래픽은 어떻게 되는가?
정답 해설: APIC는 컨트롤 플레인으로 정책을 만들 뿐, 데이터 경로에 끼어들지 않습니다. 따라서 컨트롤러가 다 죽어도 리프·스파인은 이미 배포된 정책대로 계속 패킷을 전달합니다.
Q2.ACI에서 서로 다른 두 EPG가 통신하려면 기본적으로 무엇이 필요한가?
정답 해설: ACI는 기본 차단(default deny) 화이트리스트 모델입니다. 서로 다른 EPG는 Provider/Consumer로 Contract를 맺어야만 통신이 열립니다. 같은 EPG 내부는 기본 허용입니다.
Q3.전통 네트워크에서 "VLAN이 곧 서버의 정체성"이었다면, ACI에서 그 정체성을 담당하는 것은?
정답 해설: ACI는 위치(VLAN·IP)와 정체성을 분리합니다. 엔드포인트가 어느 EPG에 속하는지가 정체성이며, 정책은 EPG 단위로 적용됩니다.
Q4.Spine-Leaf 패브릭에서 서버끼리 통신할 때의 최대 홉 수와 그 이유로 옳은 것은?
정답 해설: 모든 리프가 모든 스파인에 연결되므로 어떤 두 서버든 "리프→스파인→리프" 경로를 거칩니다. 거리가 균일해 지연시간이 예측 가능하고, ECMP로 모든 링크를 동시에 씁니다.
Q5.방화벽을 웹↔앱 트래픽 경로에 "정책적으로" 끼워 넣는 ACI 기능은?
정답 해설: Service Graph는 Policy-Based Redirect(PBR)로 특정 Contract의 트래픽을 방화벽·LB 등 L4-L7 장비로 우회시킵니다. 경로·라우팅을 수작업으로 꺾지 않아도 됩니다.
최종 정리 · 3문장

① ACI는 장비별 명령형 설정을 중앙 선언형 정책으로 바꾼 시스코 SDN이다. ② 물리는 Spine-Leaf + VXLAN, 논리는 Tenant→VRF→BD→EPG + Contract(기본 deny)로 구성된다. ③ 위치와 정체성을 분리하는 EPG 개념 하나가 자동화·보안·클라우드 연계의 출발점이다.
