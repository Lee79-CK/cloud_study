00 · 기초 개념

디자인 패턴은 레시피북입니다

요리를 처음 배울 때 우리는 백종원의 레시피를 따라 합니다. 누군가 수없이 실패하며 찾아낸 "이렇게 하면 잘 된다"는 검증된 절차죠. 코드도 똑같습니다.

소프트웨어를 만들다 보면 "객체를 딱 하나만 두고 싶다", "상황에 따라 다른 알고리즘을 갈아끼우고 싶다", "한 곳이 바뀌면 여러 곳에 자동으로 알려주고 싶다" 같은 비슷한 문제가 계속 반복됩니다. 디자인 패턴은 이런 단골 문제에 대한 이름 붙은 해결책입니다.

◈ 어디서 왔나요?

1994년 네 명의 저자(흔히 GoF, Gang of Four라 부릅니다)가 23개의 패턴을 책으로 정리하면서 널리 알려졌습니다. 지금도 거의 모든 프레임워크(Spring, React, Django, NestJS...) 내부에 이 패턴들이 녹아 있습니다.

왜 알아야 할까요?

  • 공통 언어가 됩니다. "여기 옵저버 쓰자"는 한마디면 팀원 모두가 구조를 머릿속에 그립니다. 긴 설명이 필요 없죠.
  • 검증된 구조라 안전합니다. 수십 년간 다듬어진 설계라 함정을 미리 피할 수 있습니다.
  • 프레임워크가 보입니다. React의 상태 구독, Spring의 의존성 주입... 패턴을 알면 도구의 동작 원리가 한눈에 들어옵니다.

세 가지(+한 가지) 분류

GoF 패턴은 "무엇을 다루는가"에 따라 크게 셋으로 나뉩니다. 여기에 웹/앱 현업에서 빼놓을 수 없는 아키텍처 패턴을 더해 네 묶음으로 안내합니다.

분류한 줄 요약대표 패턴
Creational 생성객체를 어떻게 만들지를 다룸싱글톤 · 팩토리 · 빌더
Structural 구조객체들을 어떻게 조립할지를 다룸어댑터 · 데코레이터 · 퍼사드 · 프록시
Behavioral 행위객체들이 어떻게 협력·소통할지를 다룸전략 · 옵저버 · 커맨드
Architectural 구조설계앱 전체 골격을 다룸MVC/MVVM · 리포지토리 · DI
01 · 마음가짐

패턴은 목적이 아니라 도구입니다

초보자가 가장 흔히 빠지는 함정 — 패턴을 외워서 "어디든 끼워 맞추려는" 태도입니다. 못이 보이면 다 망치로 보이는 거죠.

⚠ 가장 중요한 한 가지

"문제가 먼저, 패턴은 나중"입니다. 코드가 단순하면 그냥 두세요. 패턴은 변화·확장이 예상되는 복잡함을 길들일 때 비로소 값을 합니다. 단순한 곳에 패턴을 욱여넣으면 오히려 읽기 어려운 코드가 됩니다(이를 over-engineering이라 합니다).

패턴을 떠받치는 한 가지 원칙

거의 모든 패턴은 결국 이 한 문장을 실현하려는 시도입니다.

"변하는 것과 변하지 않는 것을 분리하라."

자주 바뀌는 부분(예: 결제 수단, 알림 방식)을 따로 떼어내 갈아끼울 수 있게 만들면, 나머지 안정적인 코드는 건드리지 않아도 됩니다. 이 강의의 모든 패턴을 이 렌즈로 바라보면 훨씬 쉽게 이해됩니다.

◈ 코드 읽는 법

예제는 JavaScript와 Python 탭으로 함께 제공합니다. 둘 다 웹/앱 현업에서 가장 많이 쓰는 언어죠. 한 언어만 봐도 충분하니 익숙한 쪽을 고르세요. 문법보다 "무엇을 분리했는가"에 집중하면 됩니다.

02 · Creational 생성 패턴

객체를 잘 만드는 법

new Something() 한 줄이면 객체가 만들어집니다. 하지만 "딱 하나만", "조건 따라 다른 종류로", "복잡한 조립 과정을 거쳐" 만들고 싶다면? 그 고민을 깔끔하게 푸는 것이 생성 패턴입니다.

싱글톤 Singleton

생성
앱 전체에서 인스턴스를 단 하나만 만들어 공유한다.
🏢
회사에 대표이사는 한 명입니다. 부서마다 "우리 대표"를 따로 뽑지 않죠. 누가 "대표님"을 찾든 항상 같은 사람을 가리킵니다. 싱글톤은 이 "한 명뿐인 존재"를 코드로 보장합니다.
이럴 때 씁니다
  • DB 커넥션 풀, 설정(Config), 로거(Logger)처럼 앱 전체에서 하나로 공유해야 하는 자원
  • 매번 새로 만들면 비용이 크거나, 여러 개면 상태가 어긋나는 경우
logger.singleton 
// 모듈 스코프에 인스턴스를 가둬 단 하나만 유지
class Logger {
  constructor() { this.logs = []; }
  log(msg) {
    this.logs.push(msg);
    console.log(`[LOG] ${msg}`);
  }
}
// 한 번만 생성하고 export → 모두가 같은 객체를 import
const instance = new Logger();
export default instance;

// 어디서 import하든 동일한 Logger
logger.log("서버 시작");
class Logger:
    _instance = None
    def __new__(cls):           # 객체 생성 가로채기
        if cls._instance is None:
            cls._instance = super().__new__(cls)
            cls._instance.logs = []
        return cls._instance   # 항상 같은 인스턴스 반환
    def log(self, msg):
        self.logs.append(msg)
        print(f"[LOG] {msg}")

a = Logger()
b = Logger()
print(a is b)   # True — 같은 객체!
장점
  • 전역에서 하나의 상태/자원을 일관되게 공유
  • 생성 비용이 큰 객체를 재사용해 절약
주의
  • 사실상 전역 변수 → 어디서 바뀌는지 추적이 어려움
  • 테스트하기 까다롭고, 멀티스레드에선 동기화 필요
◈ 현업 팁

JS/TS에서는 모듈 자체가 캐싱되어 한 번만 실행되므로, 인스턴스를 만들어 export하면 그게 곧 싱글톤입니다. 별도의 복잡한 코드가 거의 필요 없습니다.

팩토리 Factory Method

생성
어떤 객체를 만들지의 결정을 별도의 "공장 함수"에 맡긴다.
🏭
카페에서 "아메리카노 주세요"라고 하면 됩니다. 원두를 갈고 물을 붓는 조립 과정은 바리스타(공장)가 알아서 하죠. 손님은 무엇을 원하는지만 말합니다. 팩토리는 "어떻게 만드는지"를 호출자에게서 숨깁니다.
이럴 때 씁니다
  • 입력(타입 문자열, 설정값)에 따라 다른 종류의 객체를 만들어야 할 때
  • 객체 생성 로직이 흩어지면 곤란할 때(결제: 카드/카카오페이/페이팔 객체 등)
payment.factory 
class CardPay   { pay(a){ return `카드 ${a}원`; } }
class KakaoPay  { pay(a){ return `카카오 ${a}원`; } }

// 공장: 타입만 받아 알맞은 객체를 돌려준다
function createPayment(type) {
  switch (type) {
    case "card":  return new CardPay();
    case "kakao": return new KakaoPay();
    default: throw new Error("미지원 결제");
  }
}
// 호출자는 구체 클래스를 몰라도 된다
const p = createPayment("kakao");
p.pay(10000);
class CardPay:
    def pay(self, a): return f"카드 {a}원"
class KakaoPay:
    def pay(self, a): return f"카카오 {a}원"

def create_payment(kind):
    table = {"card": CardPay, "kakao": KakaoPay}
    if kind not in table:
        raise ValueError("미지원 결제")
    return table[kind]()      # 알맞은 객체 생성

p = create_payment("kakao")
print(p.pay(10000))
장점
  • 생성 로직이 한곳에 모여 관리가 쉬움
  • 새 종류 추가 시 호출하는 쪽 코드를 안 고쳐도 됨
주의
  • 종류가 단 1~2개뿐이면 굳이 만들 필요가 없음
  • 클래스가 늘어나 구조가 다소 장황해질 수 있음

빌더 Builder

생성
옵션이 많은 복잡한 객체를 단계별로 조립한다.
🍔
수제버거를 주문할 때 "패티 추가, 치즈 빼고, 소스는 바비큐"처럼 한 단계씩 고릅니다. 한 번에 20가지 옵션을 외쳐 헷갈릴 일이 없죠. 빌더는 이렇게 단계별 조립 → 마지막에 완성하는 방식입니다.
이럴 때 씁니다
  • 생성자 인자가 너무 많아 new X(a,b,null,null,true,...) 처럼 읽기 힘들 때
  • SQL 쿼리 빌더, HTTP 요청 빌더, 복잡한 설정 객체 등
query.builder 
class QueryBuilder {
  constructor(){ this.q = { table:"", wheres:[], limit:null }; }
  from(t){ this.q.table = t; return this; }  // this 반환 → 체이닝
  where(c){ this.q.wheres.push(c); return this; }
  take(n){ this.q.limit = n; return this; }
  build(){                                    // 마지막에 완성
    let s = `SELECT * FROM ${this.q.table}`;
    if(this.q.wheres.length) s += ` WHERE `+this.q.wheres.join(" AND ");
    if(this.q.limit) s += ` LIMIT ${this.q.limit}`;
    return s;
  }
}
const sql = new QueryBuilder()
  .from("users").where("age > 20").take(10).build();
class QueryBuilder:
    def __init__(self):
        self.table, self.wheres, self.limit = "", [], None
    def frm(self, t):  self.table = t;        return self
    def where(self, c): self.wheres.append(c); return self
    def take(self, n):  self.limit = n;        return self
    def build(self):
        s = f"SELECT * FROM {self.table}"
        if self.wheres: s += " WHERE " + " AND ".join(self.wheres)
        if self.limit:  s += f" LIMIT {self.limit}"
        return s

sql = QueryBuilder().frm("users").where("age > 20").take(10).build()
◈ 핵심 트릭

각 메서드가 this(자기 자신)를 반환하기 때문에 .from().where().take()처럼 점으로 줄줄이 이을 수 있습니다. 이를 메서드 체이닝(fluent interface)이라 부르며, 빌더의 트레이드마크입니다.

03 · Structural 구조 패턴

객체를 조립하는 법

레고 블록을 끼워 맞추듯, 이미 있는 객체들을 어떻게 연결하고 감싸서 더 큰 구조를 만들지 다룹니다. "끼우기 어려운 걸 끼우기(어댑터)", "기능을 덧입히기(데코레이터)", "복잡함을 가리기(퍼사드)", "대신 내세우기(프록시)"가 핵심 동작입니다.

어댑터 Adapter

구조
호환되지 않는 두 인터페이스 사이에 변환기를 끼운다.
🔌
해외여행 갈 때 챙기는 110V↔220V 변환 플러그가 바로 어댑터입니다. 콘센트(외부 라이브러리)도, 우리 기기(내 코드)도 바꾸지 않고, 그 사이에 끼워 서로 맞물리게 합니다.
이럴 때 씁니다
  • 외부 라이브러리·구버전 코드의 인터페이스가 내 코드와 안 맞을 때
  • 결제사 A는 .pay(), 결제사 B는 .charge() 처럼 메서드 이름이 제각각일 때 하나로 통일
payment.adapter 
// 우리 앱은 모두 pay(amount) 형태를 기대한다
class OldPayPal {            // 그런데 외부 SDK는 형식이 다름
  sendPayment(dollars){ return `PayPal $${dollars}`; }
}
// 어댑터: 우리 규격(pay)을 외부 규격(sendPayment)으로 변환
class PayPalAdapter {
  constructor(sdk){ this.sdk = sdk; }
  pay(won){
    const usd = won / 1300;          // 단위까지 변환
    return this.sdk.sendPayment(usd);
  }
}
const pay = new PayPalAdapter(new OldPayPal());
pay.pay(13000);   // 우리 앱은 통일된 pay()만 호출
class OldPayPal:            # 외부 SDK (형식이 다름)
    def send_payment(self, dollars):
        return f"PayPal ${dollars}"

class PayPalAdapter:       # 우리 규격 pay() 로 감싸기
    def __init__(self, sdk):
        self.sdk = sdk
    def pay(self, won):
        usd = won / 1300         # 단위 변환
        return self.sdk.send_payment(usd)

pay = PayPalAdapter(OldPayPal())
print(pay.pay(13000))
장점
  • 기존 코드와 외부 코드를 둘 다 안 고치고 연결
  • 변환 로직이 한곳에 모임
주의
  • 어댑터 계층이 늘면 추적이 복잡해짐
  • 근본적으로 안 맞는 설계를 미봉책으로 가릴 위험

데코레이터 Decorator

구조
원본을 건드리지 않고 기능을 한 겹씩 덧입힌다.
☕
기본 에스프레소에 우유 한 겹, 휘핑 한 겹을 더하면 카페라떼·카푸치노가 됩니다. 커피 자체는 그대로고, 위에 얹는 방식이죠. 데코레이터는 객체를 감싸 새 기능을 더합니다 — 그것도 여러 겹으로요.
이럴 때 씁니다
  • 로깅·캐싱·인증 같은 부가 기능을 선택적으로, 조합 가능하게 얹고 싶을 때
  • 서버 미들웨어(Express, NestJS), 함수 래퍼(Python @decorator)가 모두 이 패턴
api.decorator 
// 원본 함수
function fetchUser(id){ return `user#${id}`; }

// 데코레이터: 함수를 받아 기능 추가한 새 함수를 반환
function withLog(fn){
  return (...args) => {
    console.log("호출:", args);   // 덧입힌 기능
    return fn(...args);
  };
}
function withCache(fn){
  const mem = new Map();
  return (id) => mem.has(id) ? mem.get(id)
            : (mem.set(id,fn(id)), mem.get(id));
}
// 여러 겹으로 쌓기 (조합!)
const smart = withLog(withCache(fetchUser));
smart(1); smart(1);   // 두 번째는 캐시에서
import functools

def with_log(fn):            # 데코레이터
    @functools.wraps(fn)
    def wrapper(*args):
        print("호출:", args)   # 덧입힌 기능
        return fn(*args)
    return wrapper

@with_log                  # 파이썬은 @ 문법으로 한 겹 입히기
@functools.lru_cache       # 캐시까지 또 한 겹!
def fetch_user(id):
    return f"user#{id}"

fetch_user(1); fetch_user(1)  # 두 번째는 캐시
장점
  • 원본 수정 없이 기능 추가(개방-폐쇄 원칙)
  • 레고처럼 자유롭게 조합·중첩
주의
  • 겹이 많아지면 디버깅 시 흐름 추적이 어려움
  • 적용 순서에 따라 결과가 달라질 수 있음

퍼사드 Facade

구조
복잡한 하위 시스템들 앞에 단순한 창구 하나를 둔다.
🏨
호텔에서 택시·식당 예약·세탁을 각 부서에 따로 전화하지 않습니다. 프런트 데스크에 말하면 알아서 처리해 주죠. 퍼사드(=건물 정면)는 이런 단일 창구입니다. 내부가 아무리 복잡해도 손님은 쉬운 입구 하나만 보면 됩니다.
이럴 때 씁니다
  • 여러 모듈을 순서대로 호출해야 하는 복잡한 작업을 한 함수로 묶고 싶을 때
  • 주문 처리(재고 확인→결제→배송→알림)처럼 다단계 절차를 감출 때
order.facade 
// 복잡한 내부 시스템들
const stock   = { check:(i)=> true };
const billing = { charge:(a)=> `결제 ${a}` };
const ship    = { send:(i)=> `배송 ${i}` };
const notify  = { mail:(m)=> `메일 ${m}` };

// 퍼사드: 한 메서드 뒤로 복잡한 절차를 숨김
class OrderFacade {
  placeOrder(item, amount){
    if(!stock.check(item)) return "품절";
    billing.charge(amount);
    ship.send(item);
    notify.mail("주문 완료");
    return "성공";
  }
}
new OrderFacade().placeOrder("책", 15000);  // 호출은 단 한 줄
class Stock:   def check(self,i): return True
class Billing: def charge(self,a): return f"결제 {a}"
class Ship:    def send(self,i): return f"배송 {i}"

class OrderFacade:           # 단일 창구
    def __init__(self):
        self.stock, self.bill, self.ship = Stock(), Billing(), Ship()
    def place_order(self, item, amount):
        if not self.stock.check(item): return "품절"
        self.bill.charge(amount)
        self.ship.send(item)
        return "성공"

OrderFacade().place_order("책", 15000)  # 한 줄
◈ 현업에서

서비스 계층(Service Layer)이 사실상 퍼사드인 경우가 많습니다. 컨트롤러는 orderService.place() 한 줄만 부르고, 내부의 복잡한 조율은 서비스가 책임지죠.

프록시 Proxy

구조
실제 객체 앞에 대리인을 세워 접근을 제어한다.
🛎️
바쁜 CEO에게 직접 연락하지 않고 비서를 거칩니다. 비서는 일정을 거르고(접근 제어), 자주 묻는 건 자기가 답하고(캐싱), 정말 필요할 때만 CEO를 부릅니다(지연 로딩). 프록시는 이 "비서" 역할을 합니다.
이럴 때 씁니다
  • 캐싱(같은 요청은 저장된 답으로), 지연 로딩(무거운 객체를 실제 필요할 때 생성)
  • 접근 제어(권한 검사 후 통과), 호출 로깅/측정
api.proxy 
class RealApi {
  get(url){ console.log("네트워크 요청!"); return `data:${url}`; }
}
// 프록시: 같은 인터페이스(get)지만 앞단에서 캐시를 처리
class CachingProxy {
  constructor(real){ this.real = real; this.cache = new Map(); }
  get(url){
    if(this.cache.has(url)) return this.cache.get(url); // 가로채기
    const r = this.real.get(url);
    this.cache.set(url, r);
    return r;
  }
}
const api = new CachingProxy(new RealApi());
api.get("/me"); api.get("/me");  // 두 번째는 요청 없음
class RealApi:
    def get(self, url):
        print("네트워크 요청!")
        return f"data:{url}"

class CachingProxy:           # 같은 get(), 앞단에서 캐시
    def __init__(self, real):
        self.real, self.cache = real, {}
    def get(self, url):
        if url in self.cache:
            return self.cache[url]   # 가로채기
        self.cache[url] = self.real.get(url)
        return self.cache[url]

api = CachingProxy(RealApi())
api.get("/me"); api.get("/me")   # 두 번째는 캐시
장점
  • 실제 객체를 수정하지 않고 제어 로직 추가
  • 호출하는 쪽은 프록시인지 모르고 그대로 사용
주의
  • 한 단계가 더해져 응답이 살짝 느려질 수 있음
  • 캐시 무효화 등 부가 관리가 필요
04 · Behavioral 행위 패턴

객체끼리 소통하는 법

객체를 만들고 조립했다면, 이제 서로 어떻게 일을 주고받을지 정할 차례입니다. "알고리즘을 갈아끼우고(전략)", "변화를 자동으로 알리고(옵저버)", "요청을 객체로 포장하기(커맨드)" — 웹/앱에서 가장 자주 만나는 세 가지입니다.

전략 Strategy

행위
같은 일을 하는 여러 방법을 객체로 만들어 자유롭게 갈아끼운다.
🗺️
길찾기 앱에서 자동차 / 대중교통 / 도보를 고릅니다. "A에서 B로 간다"는 목표는 같지만 방법(전략)만 교체되죠. 코드에서도 알고리즘을 통째로 갈아끼우고 싶을 때 전략 패턴을 씁니다.
이럴 때 씁니다
  • 똑같은 작업을 상황에 따라 다른 방식으로 처리할 때(정렬 방식, 할인 정책, 인증 방식)
  • if/else로 분기가 끝없이 늘어나는 코드를 정리하고 싶을 때
discount.strategy 
// 전략들 — 각각 같은 모양(calc)을 가진 함수/객체
const noDiscount  = (p) => p;
const rate10      = (p) => p * 0.9;
const vip        = (p) => p * 0.7;

// 컨텍스트: 어떤 전략이든 주입받아 그대로 실행
class Cart {
  constructor(strategy){ this.strategy = strategy; }
  setStrategy(s){ this.strategy = s; }   // 런타임 교체!
  checkout(price){ return this.strategy(price); }
}
const cart = new Cart(rate10);
cart.checkout(10000);     // 9000
cart.setStrategy(vip);
cart.checkout(10000);     // 7000 — 방법만 교체
# 전략들 — 같은 시그니처를 가진 함수
no_discount = lambda p: p
rate10      = lambda p: p * 0.9
vip         = lambda p: p * 0.7

class Cart:
    def __init__(self, strategy):
        self.strategy = strategy
    def set_strategy(self, s):
        self.strategy = s          # 런타임 교체
    def checkout(self, price):
        return self.strategy(price)

cart = Cart(rate10)
print(cart.checkout(10000))   # 9000
cart.set_strategy(vip)
print(cart.checkout(10000))   # 7000
장점
  • 거대한 if/else 분기를 깔끔하게 제거
  • 실행 중에도 전략을 바꿀 수 있음
주의
  • 전략이 2~3개뿐이고 안 바뀌면 과한 설계
  • 호출자가 어떤 전략을 쓸지는 알아야 함

옵저버 Observer

행위
한 객체가 바뀌면 구독 중인 모두에게 자동으로 알린다.
🔔
유튜브 채널을 구독해 두면, 새 영상이 올라올 때마다 알림이 자동으로 옵니다. 구독자가 1명이든 100만이든 유튜버는 신경 쓰지 않죠. "상태 변화 → 구독자 전원 통지"가 옵저버의 전부입니다.
이럴 때 씁니다
  • 한 곳의 데이터가 바뀌면 여러 화면/모듈이 자동 갱신되어야 할 때
  • 이벤트 시스템, 실시간 알림, React/Vue의 상태 반응성, pub-sub 메시징
weather.observer 
class Subject {            // 관찰 대상(유튜버)
  constructor(){ this.observers = []; }
  subscribe(fn){ this.observers.push(fn); }   // 구독
  notify(data){                          // 전원 통지
    this.observers.forEach(fn => fn(data));
  }
}
const weather = new Subject();
weather.subscribe(t => console.log(`폰: ${t}°`));  // 구독자1
weather.subscribe(t => console.log(`시계: ${t}°`)); // 구독자2
weather.notify(26);   // 모든 구독자에게 한 번에 전달
class Subject:               # 관찰 대상
    def __init__(self):
        self.observers = []
    def subscribe(self, fn):
        self.observers.append(fn)     # 구독
    def notify(self, data):
        for fn in self.observers:   # 전원 통지
            fn(data)

weather = Subject()
weather.subscribe(lambda t: print(f"폰: {t}°"))
weather.subscribe(lambda t: print(f"시계: {t}°"))
weather.notify(26)   # 모두에게 전달
◈ 아래에서 직접 체험

이 패턴은 07. 라이브 데모에서 버튼을 눌러 직접 동작을 확인할 수 있습니다. 날씨를 바꾸면 구독한 기기들이 어떻게 동시에 반응하는지 눈으로 보세요.

장점

  • 구독자를 자유롭게 추가/제거
주의
  • 구독 해제를 잊으면 메모리 누수 위험
  • 통지가 연쇄되면 흐름 추적이 어려움

커맨드 Command

행위
"요청"을 객체로 포장해 저장·전달·취소할 수 있게 한다.
📝
식당에서 주문하면 주문서(전표)가 적힙니다. 이 종이는 주방에 전달되고, 보관되고, 잘못되면 취소도 됩니다. 커맨드는 "~를 해라"라는 행동 자체를 종이(객체)로 만들어 다룰 수 있게 합니다.
이럴 때 씁니다
  • 실행 취소(Undo/Redo)가 필요한 에디터·그래픽 툴
  • 작업 큐, 예약 실행, 매크로(여러 명령 묶음), 트랜잭션 로그
editor.command 
// 각 명령은 실행(do)과 취소(undo)를 안다
class AddText {
  constructor(doc, text){ this.doc = doc; this.text = text; }
  do(){   this.doc.content += this.text; }
  undo(){ this.doc.content =
      this.doc.content.slice(0, -this.text.length); }
}
class History {                     // 명령들을 쌓아 둠
  constructor(){ this.stack = []; }
  run(cmd){ cmd.do(); this.stack.push(cmd); }
  undo(){ this.stack.pop()?.undo(); }   // 되돌리기!
}
const doc = { content:"" }, h = new History();
h.run(new AddText(doc, "안녕"));
h.undo();   // content 다시 ""
class AddText:                 # 실행/취소를 아는 명령
    def __init__(self, doc, text):
        self.doc, self.text = doc, text
    def do(self):
        self.doc["c"] += self.text
    def undo(self):
        self.doc["c"] = self.doc["c"][:-len(self.text)]

class History:
    def __init__(self): self.stack = []
    def run(self, cmd):
        cmd.do(); self.stack.append(cmd)
    def undo(self):
        if self.stack: self.stack.pop().undo()

doc, h = {"c": ""}, History()
h.run(AddText(doc, "안녕")); h.undo()
장점
  • 실행 취소·재실행·로그를 자연스럽게 구현
  • 요청을 큐에 넣어 나중에/원격으로 실행 가능
주의
  • 명령마다 클래스가 생겨 코드량 증가
  • 단순 호출에는 과한 구조
05 · Architectural 실전 골격

앱 전체를 떠받치는 큰 패턴

GoF 패턴이 "방 안 가구 배치"라면, 아키텍처 패턴은 "건물 전체의 골조"입니다. 웹/앱 프로젝트를 시작할 때 가장 먼저 마주치는, 그래서 실무에서 가장 자주 입에 오르내리는 세 가지를 짚습니다.

MVC · MVVM 관심사의 분리

아키텍처
데이터·화면·중간 조율 역할을 세 덩어리로 나눈다.
🍽️
식당은 주방(데이터·로직=Model), 손님 테이블(화면=View), 서버 직원(중간 조율=Controller)로 역할이 나뉩니다. 주방이 직접 손님과 흥정하지 않죠. 역할을 나누면 한쪽을 고쳐도 다른 쪽이 흔들리지 않습니다.

세 글자의 정체

  • Model(모델) — 데이터와 비즈니스 규칙. "회원가입은 이메일 중복이면 거절" 같은 핵심 로직.
  • View(뷰) — 사용자가 보는 화면. 버튼·목록·폼. 데이터를 어떻게 "보여줄지"만 담당.
  • Controller(컨트롤러) — 둘 사이의 중개자. 입력을 받아 모델을 부르고, 결과를 뷰에 전달.
◈ MVC vs MVVM

MVC는 서버 웹(Spring, Django, Rails)에서 전통적으로 쓰입니다. MVVM은 컨트롤러 자리를 ViewModel로 바꾼 변형으로, 화면과 데이터를 자동으로 연결(데이터 바인딩)하는 프론트엔드(Vue, 안드로이드, SwiftUI)에서 인기입니다. 핵심 정신은 같습니다 — "화면, 데이터, 조율을 섞지 말 것."

todo.mvc 
// Model: 데이터 + 규칙
class TodoModel {
  constructor(){ this.items = []; }
  add(t){ if(t.trim()) this.items.push(t); }
}
// View: 보여주기만
class TodoView {
  render(items){ return items.map(i => `• ${i}`).join("\n"); }
}
// Controller: 입력을 받아 모델·뷰를 조율
class TodoController {
  constructor(m, v){ this.m = m; this.v = v; }
  addTodo(text){
    this.m.add(text);                 // 모델 갱신
    return this.v.render(this.m.items); // 뷰 갱신
  }
}
const c = new TodoController(new TodoModel(), new TodoView());
c.addTodo("우유 사기");
class TodoModel:           # 데이터 + 규칙
    def __init__(self): self.items = []
    def add(self, t):
        if t.strip(): self.items.append(t)

class TodoView:            # 보여주기만
    def render(self, items):
        return "\n".join(f"• {i}" for i in items)

class TodoController:      # 조율
    def __init__(self, m, v): self.m, self.v = m, v
    def add_todo(self, text):
        self.m.add(text)
        return self.v.render(self.m.items)

c = TodoController(TodoModel(), TodoView())
print(c.add_todo("우유 사기"))

리포지토리 Repository

아키텍처
데이터 저장소 접근을 한 계층 뒤로 숨겨, 나머지 코드가 DB를 모르게 한다.
📚
도서관에서 책이 어느 서가·어느 창고에 있는지 직접 찾지 않습니다. 사서에게 "이 책 주세요"라고 하면 끝이죠. 리포지토리는 이 "사서"입니다. 비즈니스 코드는 userRepo.findById(1)만 부르고, 그게 MySQL인지 MongoDB인지 캐시인지 몰라도 됩니다.
이럴 때 씁니다
  • 비즈니스 로직과 DB 접근 코드를 분리하고 싶을 때(거의 모든 백엔드 서비스)
  • 나중에 DB를 바꾸거나, 테스트에서 가짜 저장소로 교체하고 싶을 때
user.repository

◈ 테스트의 비밀 무기

테스트할 때 진짜 DB 대신 메모리에서 동작하는 가짜 리포지토리를 끼워 넣을 수 있습니다. 그래서 DB 없이도 비즈니스 로직을 빠르게 검증할 수 있죠. 이것이 다음 패턴 DI와 환상의 짝꿍인 이유입니다.

의존성 주입 Dependency Injection

아키텍처
필요한 부품을 객체가 직접 만들지 않고, 밖에서 끼워 넣어 준다.
🔋
리모컨은 건전지를 스스로 만들지 않습니다. 우리가 밖에서 넣어 주죠. 덕분에 알카라인이든 충전지든 자유롭게 갈아 끼웁니다. DI는 객체가 필요한 의존 부품을 직접 new 하지 않고 외부에서 받도록 만드는 것입니다.
이럴 때 씁니다
  • 객체끼리 단단히 붙어(강결합) 테스트·교체가 어려울 때 → 느슨하게 풀기

// ❌ 강결합: 내부에서 직접 생성 → 교체·테스트 어려움

}

// ✅ 의존성 주입: 밖에서 받음 → 자유롭게 교체
class GoodService {
  constructor(repo){ this.repo = repo; }   // 주입!
  getUser(id){ return this.repo.findById(id); }
}
// 운영에선 진짜 DB, 테스트에선 가짜를 끼운다
const prod = new GoodService(new MySQLRepo());
const test = new GoodService(new FakeRepo());
# ❌ 강결합
class BadService:
    def __init__(self):
        self.repo = MySQLRepo()   # 못 바꿈

# ✅ 의존성 주입
class GoodService:
    def __init__(self, repo):     # 밖에서 받음
        self.repo = repo
    def get_user(self, id):
        return self.repo.find_by_id(id)

test = GoodService(FakeRepo())    # 가짜

06 · 핵심 비교

비슷해 보여 헷갈리는 패턴들

초보자가 가장 자주 혼동하는 지점입니다. 구조가 닮았어도 "왜 쓰는가(의도)"가 다릅니다. 의도를 기준으로 구분하면 절대 헷갈리지 않습니다.

어댑터 vs 데코레이터 vs 프록시 vs 퍼사드

넷 다 "객체를 감싼다"는 점이 같아 가장 많이 헷갈립니다. 차이는 감싸는 이유입니다.

패턴감싸는 이유(의도)인터페이스 변화한마디
어댑터안 맞는 것을 맞게바꿈(A→B)"규격을 변환"
데코레이터기능을 추가유지(같은 모양)"기능을 덧칠"
프록시접근을 제어(캐싱/권한)유지(같은 모양)"대리인을 세움"
퍼사드복잡함을 단순화새 단순 인터페이스"창구 하나로"
한 문장 정리: 어댑터는 맞추고, 데코레이터는 더하고, 프록시는 통제하고, 퍼사드는 줄인다.
전략 vs 팩토리
전략 (Strategy)

객체를 어떻게 쓸지(행동·알고리즘)를 교체. "할인을 어떤 방식으로 계산할까?"

팩토리 (Factory)

객체를 어떻게 만들지(생성)를 위임. "어떤 결제 객체를 생성할까?"

구분법: 팩토리는 "만드는 순간"의 고민, 전략은 "쓰는 순간"의 고민입니다. 종종 함께 쓰입니다 — 팩토리로 전략 객체를 만들기도 하죠.
옵저버 vs 전략
옵저버 (Observer)

일대다 통지. 하나가 바뀌면 여럿이 자동 반응. "누구에게 알릴까"의 문제.

전략 (Strategy)

일대일 교체. 하나의 동작 방식을 골라 끼움. "어떻게 처리할까"의 문제.

구분법: 옵저버는 "알림(이벤트)", 전략은 "방법 선택(알고리즘)"에 관한 것입니다.

날씨(관찰 대상)를 바꾸면, 이 날씨를 구독한 기기들이 어떻게 동시에 반응하는지 직접 확인해 보세요. 구독을 켜고 끄며 "느슨한 연결"을 체감할 수 있습니다.

구독을 해제한 기기는 통지를 받지 않습니다. Subject는 누가 듣고 있는지 일일이 신경 쓰지 않고 그저 "변했다!"고 외칠 뿐입니다. 이 느슨한 연결이 옵저버의 핵심 가치입니다.

08 · 마무리

이제 어디로 가야 할까

패턴 13개를 훑었습니다. 외우는 것이 목표가 아닙니다. 코드를 짜다 "어디서 본 구조인데?" 싶을 때 이 강의를 다시 펼치세요. 다음 단계를 제안합니다.

1

코드에서 패턴을 "찾아보기"

새로 짜기 전에, 익숙한 프레임워크(React·Express·Spring) 안에서 이 패턴들을 먼저 발견해 보세요. React의 상태 구독은 옵저버, 미들웨어는 데코레이터입니다.

2

SOLID 원칙 익히기

패턴은 SOLID라는 5가지 설계 원칙의 구체적 실현입니다. 특히 "개방-폐쇄 원칙(OCP)"과 "의존성 역전(DIP)"을 알면 패턴이 왜 그 모양인지 납득됩니다.

3

작은 프로젝트에 한 개씩 적용

토이 프로젝트에서 전략·옵저버·리포지토리를 직접 써 보세요. 한 번 손으로 짜 본 패턴은 절대 잊히지 않습니다.

4

"안 쓰는 판단"도 연습하기

고수는 패턴을 많이 쓰는 사람이 아니라, 필요 없을 때 쓰지 않는 사람입니다. 단순한 코드를 단순하게 두는 절제도 실력입니다.

이해도 점검 퀴즈

Quiz 01
DB 커넥션을 앱 전체에서 하나만 공유하고 싶습니다. 가장 알맞은 패턴은?
정답: 싱글톤. "인스턴스를 단 하나만 두고 공유"가 싱글톤의 정의입니다. DB 풀·설정·로거가 대표적인 사용처죠.
Quiz 02
"한 곳의 데이터가 바뀌면 여러 화면이 자동으로 갱신"되도록 하려면?
정답: 옵저버. 상태 변화를 구독자 전원에게 자동 통지하는 일대다 패턴입니다. React/Vue의 반응형 상태가 이 원리로 동작합니다.
Quiz 03
외부 결제 SDK의 메서드 이름이 우리 코드 규격과 다릅니다. 둘 다 안 고치고 연결하려면?
정답: 어댑터. 호환되지 않는 두 인터페이스 사이에 변환기를 끼우는 패턴입니다. 110V↔220V 변환 플러그를 떠올리세요.
Quiz 04
데코레이터와 프록시는 둘 다 객체를 감쌉니다. 가장 핵심적인 차이는?
정답: 의도의 차이. 둘 다 같은 인터페이스를 유지하며 감싸지만, 데코레이터는 기능을 더하려고, 프록시는 접근(캐싱·권한·지연)을 통제하려고 감쌉니다.
Quiz 05
초보 개발자가 가장 흔히 저지르는 실수는?
정답: 과한 적용(over-engineering). 패턴은 변화·복잡함을 길들이는 도구입니다. 단순한 곳엔 그냥 단순한 코드가 최선입니다. "문제가 먼저, 패턴은 나중."

"패턴을 아는 것은 시작일 뿐, 언제 쓰지 않을지 아는 것이 완성이다."
