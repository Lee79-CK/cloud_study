Maven과 Gradle로
Java 빌드의 정석을 익히다

"내 코드는 잘 도는데, 왜 빌드는 항상 어렵게 느껴질까?" 빌드 도구는 단순한 컴파일러 래퍼가 아니라, 의존성 · 테스트 · 배포 · 자동화의 모든 길목을 책임지는 인프라입니다. 이 강의는 초보 Java 개발자가 Maven과 Gradle의 핵심 개념을 손에 잡히듯 익히고, 다른 언어 빌드 도구와의 큰 그림까지 함께 이해할 수 있도록 설계했습니다.

예상 학습 시간: 약 90분 난이도: 입문 → 중급 전제 지식: Java 기초
01 / SECTION

왜 빌드 도구가 필요한가

아주 작은 Java 프로그램을 떠올려 봅시다. 파일 하나에 main 메서드가 있고, javac Hello.java로 컴파일하고 java Hello로 실행하면 끝입니다. 여기까지는 빌드 도구가 필요 없습니다. 그런데 현실 프로젝트는 이렇게 단순하지 않습니다.

현실 프로젝트가 직면하는 7가지 골칫거리

  1. 수십~수천 개의 소스 파일 — 일일이 javac에 넘겨주면 명령어가 끝없이 길어집니다.
  2. 외부 라이브러리(JAR) 의존 — Spring, Jackson, Hibernate… 각 JAR을 어디서 받아 어디에 두고 클래스패스에 어떻게 추가할지 일일이 관리해야 합니다.
  3. 의존성의 의존성 — A 라이브러리가 B를 쓰고, B는 C와 D를 씁니다. 이 전이 의존성(transitive dependency)까지 사람이 추적하기는 거의 불가능합니다.
  4. 버전 충돌 — A는 Jackson 2.10을, B는 Jackson 2.15를 요구합니다. 어떤 버전을 쓸지 결정하는 정책이 필요합니다.
  5. 다양한 산출물 — JAR, WAR, fat-JAR, native image, 컨테이너 이미지… 같은 코드에서 여러 형태의 산출물을 만들어야 합니다.
  6. 테스트 · 정적분석 · 코드 커버리지 — 단순 컴파일을 넘어 자동화된 검증 단계가 빌드의 일부가 됩니다.
  7. 재현 가능성(Reproducibility) — "내 컴퓨터에선 됐는데" 문제를 막으려면, 누가 어디서 빌드해도 동일한 결과가 나와야 합니다.
i
핵심 통찰 빌드 도구는 "컴파일 자동화 도구"가 아니라 소프트웨어 공급망(supply chain)을 관리하는 도구입니다. MLOps에서 Docker · Kubernetes · Helm이 인프라 공급망을 관리하듯, Maven · Gradle은 코드와 의존성의 공급망을 관리합니다.
02 / SECTION

"빌드"라는 단어의 정확한 의미

한국어로 "빌드한다"는 막연하게 "프로젝트를 돌릴 수 있는 형태로 만든다" 정도의 뜻으로 쓰입니다. 하지만 빌드 도구의 세계에서 빌드(build)는 잘 정의된 일련의 단계(phases/tasks)를 의미합니다. 일반적으로 다음 단계가 모두 또는 일부 수행됩니다.

빌드 도구는 이 파이프라인의 각 단계가 언제, 어떤 순서로, 어떤 입력으로, 어떤 산출물을 만들며 진행되는지 정의하고, 그 결과를 캐싱해 빠르게 재실행할 수 있게 해 줍니다.

빌드 도구가 책임지는 4가지 본질적 업무

Dependency
Resolution
선언된 라이브러리와 그 전이 의존성을 원격 저장소(Maven Central 등)에서 받아오고, 버전 충돌을 해소합니다.
Compilation
Orchestration
변경된 파일만 재컴파일하는 incremental build를 수행하고, 멀티 모듈/멀티 소스셋(main, test 등)을 올바른 순서로 빌드합니다.
Task
Automation
테스트 실행, 코드 검사, 문서 생성, 컨테이너 이미지 빌드 등 빌드 전후의 작업을 일관된 방식으로 자동화합니다.
Artifact
Publishing
완성된 JAR/WAR를 사내 Nexus, Artifactory, Maven Central 등에 게시(publish)합니다.
03 / SECTION · MAVEN

Maven의 철학과 등장 배경

Maven은 2004년 Apache 재단에서 처음 정식 릴리즈된, Java 진영의 1세대 표준 빌드 도구입니다. 그 이전에는 Apache Ant가 주류였지만, Ant는 "무엇을 어떻게 할지" 개발자가 일일이 명령형(imperative)으로 작성해야 했습니다. Maven은 발상을 뒤집어, 관례(convention)를 정해 두고 개발자는 차이점만 선언(declare)하면 된다는 "Convention over Configuration" 철학을 들고 나왔습니다.

★
Convention over Configuration (관례 우선) Maven은 "Java 프로젝트라면 으레 이렇게 생겼을 것"이라는 표준 디렉터리 구조를 가정합니다. 이 가정을 따르기만 하면 설정이 거의 필요 없습니다. 모든 Maven 프로젝트가 비슷한 구조를 가지므로, 누가 만든 프로젝트든 익숙하게 읽을 수 있다는 큰 장점이 있습니다.

Maven 표준 디렉터리 구조

project-root/tree
my-app/
├── pom.xml                  # 빌드 설정 파일 (Maven의 심장)
├── src/
│   ├── main/
│   │   ├── java/             # 운영 소스코드
│   │   │   └── com/example/App.java
│   │   └── resources/        # 운영 리소스 (yml, properties 등)
│   │       └── application.yml
│   └── test/
│       ├── java/             # 테스트 소스코드
│       │   └── com/example/AppTest.java
│       └── resources/        # 테스트 리소스
└── target/                   # 빌드 산출물 (gitignore 대상)
    ├── classes/              # 컴파일된 .class 파일
    └── my-app-1.0.0.jar      # 최종 패키지

이 구조를 그대로 지키면 Maven에게 "어디에 소스가 있는지, 어디에 결과를 둘지" 따로 알려줄 필요가 없습니다. 이 작은 약속 하나가, Java 생태계 전체의 코드 가독성을 끌어올린 제도적 혁신이었습니다.

04 / SECTION · MAVEN

pom.xml 한 줄씩 해부하기

POM은 Project Object Model의 약자로, 프로젝트의 모든 정보를 담은 XML 파일입니다. 한 마디로 "이 프로젝트는 무엇이고, 누구에게 의존하며, 어떻게 빌드되어야 하는가"를 기술하는 프로젝트의 신분증입니다.

pom.xmlXML
<?xml version="1.0" encoding="UTF-8"?>
<project xmlns="http://maven.apache.org/POM/4.0.0">
    <modelVersion>4.0.0</modelVersion>

    <!-- ① 프로젝트의 좌표 (GAV) -->
    <groupId>com.example</groupId>
    <artifactId>my-app</artifactId>
    <version>1.0.0</version>
    <packaging>jar</packaging>

    <!-- ② 빌드 설정 속성 -->
    <properties>
        <maven.compiler.source>21</maven.compiler.source>
        <maven.compiler.target>21</maven.compiler.target>
        <project.build.sourceEncoding>UTF-8</project.build.sourceEncoding>
    </properties>

    <!-- ③ 외부 라이브러리 선언 -->
    <dependencies>
        <dependency>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-starter-web</artifactId>
            <version>3.3.0</version>
        </dependency>
        <dependency>
            <groupId>org.junit.jupiter</groupId>
            <artifactId>junit-jupiter</artifactId>
            <version>5.10.0</version>
            <scope>test</scope>
        </dependency>
    </dependencies>

    <!-- ④ 빌드 동작을 바꾸는 플러그인 -->
    <build>
        <plugins>
            <plugin>
                <groupId>org.springframework.boot</groupId>
                <artifactId>spring-boot-maven-plugin</artifactId>
            </plugin>
        </plugins>
    </build>
</project>

① GAV — 프로젝트의 좌표

Maven 세계에서 모든 산출물은 GAV 좌표 세 개로 유일하게 식별됩니다.

  • groupId — 조직/패키지 이름. 일반적으로 도메인 역순(com.example).
  • artifactId — 산출물 이름. 한 조직 안에서 유일해야 합니다.
  • version — 버전. 1.0.0, 2.3.1-SNAPSHOT 등.

이 세 개의 좌표는 Maven Central 같은 글로벌 저장소에서 어떤 라이브러리든 정확히 찾아낼 수 있게 해 줍니다. 예: org.springframework.boot:spring-boot-starter-web:3.3.0은 전 세계에서 유일한 한 개의 JAR을 가리킵니다.

② 속성(Properties)

Java 버전, 인코딩 등을 한 곳에 모아두는 변수 선언부입니다. POM 안 어디서든 ${maven.compiler.source}처럼 참조할 수 있습니다.

③ 의존성(Dependencies)

프로젝트가 사용하는 외부 라이브러리 목록입니다. 각 의존성은 GAV 좌표와 함께 scope(적용 범위)를 가집니다.

Scope의미예시
compile (기본)컴파일 · 테스트 · 실행 모두에서 사용Spring, Jackson
provided컴파일 시에만 필요. 런타임에는 컨테이너가 제공Servlet API, Lombok
runtime컴파일에는 불필요, 실행 시에만 필요JDBC 드라이버
test테스트 코드에서만 사용JUnit, Mockito
system로컬 JAR 직접 지정 (사용 권장X)레거시 라이브러리

④ 플러그인(Plugins)

Maven 자체는 의외로 작습니다. 실제 컴파일, 테스트, 패키징 같은 작업은 모두 플러그인이 수행합니다. 뒤에서 자세히 다룹니다.

05 / SECTION · MAVEN

Maven 빌드 생명주기(Build Lifecycle)

Maven에서 가장 자주 헷갈리는 개념입니다. 다음 세 단어를 구분할 수 있으면 Maven의 절반은 이해한 것입니다.

Lifecycle
빌드의 큰 흐름. 기본 lifecycle은 default, clean, site 세 종류입니다.
Phase
lifecycle 안의 정해진 단계. compile, test, package 같이 순서가 정해진 마디들.
Goal
실제로 일을 수행하는 단위. 플러그인이 제공하며, phase에 묶여 실행됩니다. 예: compiler:compile.

아래는 default lifecycle의 핵심 phase입니다. 아무 카드나 클릭해 보세요.

// Maven Default Lifecycle

01
validate
검증
02
compile
컴파일
03
test
테스트
04
package
패키징
05
verify
검사
06
install
로컬 설치
07
deploy
원격 배포
// phase를 선택하세요
위의 단계 중 하나를 클릭하면 상세 설명이 표시됩니다.

중요한 약속: 앞 단계는 자동 실행됩니다

Maven의 모든 phase는 이전 phase를 모두 거친 뒤에야 실행됩니다. 예를 들어 mvn package를 실행하면 Maven은 자동으로 validate → compile → test → package를 차례로 수행합니다. mvn install은 거기에 verify → install까지 추가합니다. 따라서 보통은 phase 이름만 외워 두면 됩니다.

terminalbash
# 자주 쓰는 Maven 명령어
$ mvn clean                # target/ 삭제 (이전 빌드 잔재 제거)
$ mvn compile              # 컴파일까지만
$ mvn test                 # 컴파일 + 테스트
$ mvn package              # 컴파일 + 테스트 + JAR 생성
$ mvn install              # 위 + 로컬 저장소(~/.m2)에 설치
$ mvn clean package        # 이전 산출물 삭제 후 새로 패키징 (실무에서 가장 자주 사용)
$ mvn package -DskipTests  # 테스트 건너뛰기 (긴급 배포 시)
$ mvn dependency:tree      # 의존성 트리 출력 (디버깅 필수 명령)
!
초보자가 자주 하는 실수 mvn install은 라이브러리를 "내 컴퓨터에 설치"하는 것이지 운영 서버에 배포하는 게 아닙니다. 운영 배포는 mvn deploy이며, 보통 사람이 직접 실행하지 않고 CI/CD가 수행합니다.
06 / SECTION · MAVEN

의존성 관리의 본질

Maven의 의존성 시스템을 이해하려면 다음 그림을 머릿속에 그려야 합니다. 하나의 라이브러리를 추가했을 뿐인데, 실제로는 그 라이브러리가 의존하는 다른 라이브러리들까지 줄줄이 따라옵니다. 이를 전이 의존성(transitive dependency)이라 합니다.

의존성 충돌 해결 — Nearest Wins

같은 라이브러리가 서로 다른 경로에서, 서로 다른 버전으로 들어올 수 있습니다. Maven은 이때 "내 프로젝트에서 가장 가까운 경로의 버전이 이긴다"(nearest-wins)는 규칙을 씁니다. 트리에서 깊이가 더 얕은 쪽이 우선합니다. 두 경로의 깊이가 같으면 pom.xml에 먼저 선언된 쪽이 이깁니다.

⌘
꼭 외워 둘 진단 명령어 mvn dependency:tree는 의존성 트리를 출력하고, 어떤 라이브러리가 어떤 버전으로 결정되었는지 보여 줍니다. "분명히 2.15를 썼는데 2.10이 들어왔다" 같은 미스터리는 거의 모두 이 명령어로 해결됩니다.

BOM(Bill of Materials) — 버전 관리의 베스트 프랙티스

Spring Boot 같은 큰 생태계는 수십 개의 라이브러리를 함께 씁니다. 각 라이브러리의 호환 버전을 외워서 관리하기는 불가능에 가깝습니다. BOM은 "이 버전 집합은 함께 검증되었다"는 메타 의존성입니다. dependencyManagement에 BOM을 등록하면, 그 안에 포함된 라이브러리들의 버전을 일일이 적지 않아도 됩니다.

pom.xml — BOM 사용 예XML
<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-dependencies</artifactId>
            <version>3.3.0</version>
            <type>pom</type>
            <scope>import</scope>
        </dependency>
    </dependencies>
</dependencyManagement>

<dependencies>
    <!-- 버전을 적지 않아도 BOM이 결정해 줌 -->
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter-web</artifactId>
    </dependency>
</dependencies>
07 / SECTION · MAVEN

Maven은 사실 "플러그인 실행기"다

여기서 충격적인 사실을 하나 알려드립니다. Maven은 스스로는 거의 아무 일도 하지 않습니다. Maven 코어는 phase의 순서를 관리하고, 어떤 phase에 어떤 플러그인 goal이 묶일지 결정할 뿐입니다. 실제로 컴파일, 테스트, 패키징을 하는 것은 모두 플러그인입니다.

플러그인역할묶이는 Phase
maven-compiler-pluginJava 소스 컴파일compile test-compile
maven-surefire-plugin단위 테스트 실행test
maven-failsafe-plugin통합 테스트 실행verify
maven-jar-pluginJAR 패키징package
maven-install-plugin로컬 저장소 설치install
maven-deploy-plugin원격 저장소 배포deploy
spring-boot-maven-plugin실행 가능한 fat-JAR 생성, Spring Boot 앱 구동package
jib-maven-pluginDockerfile 없이 컨테이너 이미지 빌드커스텀

즉, Maven은 텅 빈 무대(phase)를 제공하고, 플러그인들이 배우처럼 올라와 각자의 역할을 연기합니다. 이 구조 덕분에 Maven은 핵심을 작게 유지하면서도 어떤 새로운 작업이든 플러그인으로 확장할 수 있습니다.

08 / SECTION · GRADLE

Gradle의 등장과 새로운 패러다임

Maven이 자리 잡고 약 7년이 흐른 2012년, Gradle 1.0이 정식 릴리즈됩니다. Maven의 "관례 우선" 철학은 훌륭했지만, 시간이 흐르며 두 가지 한계가 드러났습니다.

  1. XML의 한계 — XML은 사람이 읽고 쓰기에는 너무 장황합니다. 간단한 조건문 하나, 작은 동적 로직 하나도 XML 안에서는 매우 어색합니다.
  2. 커스터마이징의 어려움 — Maven은 "표준 흐름"을 따르는 데는 최적이지만, 그 표준을 살짝 비트는 일은 의외로 까다롭습니다. 새로운 일을 추가하려면 결국 플러그인을 만들어야 합니다.

Gradle은 이 두 가지를 동시에 해결하려 했습니다. 빌드 스크립트를 프로그래밍 언어(처음에는 Groovy, 지금은 Kotlin이 권장)로 작성하게 한 것이죠. 덕분에 빌드 로직 자체에 변수, 함수, 조건문, 반복문을 자연스럽게 쓸 수 있습니다.

★
Gradle의 세 가지 핵심 약속 ① 빠르다 (incremental build, build cache, daemon) ② 유연하다 (DSL 기반 스크립트) ③ Maven 호환 (같은 의존성 저장소와 GAV 좌표 사용)

같은 프로젝트, 두 가지 표현

pom.xmlXML · 28 lines

Groovy DSL vs Kotlin DSL

Gradle 빌드 스크립트는 두 가지 언어로 작성할 수 있습니다.

  • build.gradle — Groovy DSL. Gradle의 원조 언어. 더 짧고 동적.
  • build.gradle.kts — Kotlin DSL. 2019년 정식 도입. 타입 안전, IDE 지원 우수.

2026년 현재 신규 프로젝트에서는 Kotlin DSL을 권장합니다. Gradle 공식 문서, Android Studio, IntelliJ의 자동완성, 빌드 캐시 직렬화 등 모든 면에서 Kotlin DSL이 1급 시민으로 우대받기 때문입니다.

// Groovy DSL

build.gradleGroovy
plugins {
    id 'java'
}

dependencies {
    implementation 'org.apache.commons:commons-lang3:3.14.0'
}

tasks.named('test') {
    useJUnitPlatform()
}

// Kotlin DSL

build.gradle.ktsKotlin
plugins {
    java
}

dependencies {
    implementation("org.apache.commons:commons-lang3:3.14.0")
}

tasks.named<Test>("test") {
    useJUnitPlatform()
}

문법 차이는 다음과 같이 기억하면 됩니다.

Gradle의 심장: 태스크(Task) 시스템

Maven이 phase 중심이라면, Gradle은 태스크 중심입니다. Gradle 빌드는 방향성 비순환 그래프(DAG, Directed Acyclic Graph)의 형태로 태스크들이 연결된 구조입니다. Airflow나 Argo Workflows의 DAG와 정확히 같은 개념입니다.

⇄
MLOps 엔지니어를 위한 비유 Gradle 태스크 그래프는 Airflow DAG, Argo Workflow의 step 그래프와 본질적으로 같은 구조입니다. 각 태스크는 입력과 출력을 가지고, 의존성에 따라 실행 순서가 결정되며, 입력이 바뀌지 않으면 캐시된 출력이 재사용됩니다.

커스텀 태스크 만들기

Gradle의 가장 큰 매력은 사용자가 직접 태스크를 정의할 수 있다는 점입니다. 아래는 빌드 시 현재 시각을 build-info.txt로 기록하는 간단한 태스크입니다.

build.gradle.ktsKotlin DSL
tasks.register("writeBuildInfo") {
    group = "documentation"
    description = "빌드 시각을 build-info.txt에 기록합니다"

    val outputFile = layout.buildDirectory.file("build-info.txt")

    outputs.file(outputFile)

    doLast {
        outputFile.get().asFile.writeText(
            "Built at: " + java.time.Instant.now()
        )
        println("✓ build-info.txt 생성 완료")
    }
}

// build 태스크가 끝나면 자동으로 실행되도록 연결
tasks.named("build") {
    finalizedBy("writeBuildInfo")
}

이제 터미널에서 다음과 같이 사용할 수 있습니다.

terminalbash
$ ./gradlew writeBuildInfo
> Task :writeBuildInfo
✓ build-info.txt 생성 완료

$ ./gradlew tasks --group documentation
writeBuildInfo - 빌드 시각을 build-info.txt에 기록합니다
⚡
Incremental Build의 비밀 outputs.file()로 출력 파일을 선언했기 때문에, Gradle은 다음 실행 때 출력 파일이 입력보다 새것이면 "UP-TO-DATE"라며 태스크를 건너뜁니다. 이것이 Gradle이 Maven보다 훨씬 빠른 핵심 비결 중 하나입니다.

Gradle의 의존성 설정(configuration)

Gradle은 Maven의 5가지 scope보다 더 세밀한 의존성 설정을 제공합니다. 그중 핵심은 implementation과 api의 구분입니다.

Configuration의미Maven 대응
implementation내부 구현용. 의존하는 라이브러리가 외부로 노출되지 않음.≈ compile
api공개 API. 의존성이 모듈 사용자에게 전파됨.≈ compile
compileOnly컴파일 시에만 필요. Lombok 같은 도구.≈ provided
runtimeOnly실행 시에만 필요. JDBC 드라이버 등.≈ runtime
testImplementation테스트 코드에서만 사용.≈ test
annotationProcessor컴파일 시 어노테이션 처리기로 동작.플러그인 설정 필요

implementation vs api — 왜 중요한가

당신이 lib-A를 만들고 있고, 내부적으로 Guava를 사용합니다. 당신의 라이브러리를 쓰는 app-X가 있다고 합시다.

  • implementation으로 Guava를 선언하면, app-X는 Guava를 직접 import할 수 없습니다 (캡슐화됨). 컴파일은 더 빠릅니다.
  • api로 선언하면, app-X도 Guava를 자유롭게 쓸 수 있습니다 (전이 노출). 단, lib-A가 Guava를 바꾸면 모든 사용자가 영향을 받습니다.
!
실무 가이드 특별한 이유가 없다면 항상 implementation을 사용하세요. api는 의존성을 외부로 노출하기로 의도적으로 결정한 경우에만 씁니다. 이 한 가지 규칙만 지켜도 빌드 시간과 의존성 그래프가 훨씬 깔끔해집니다.
12 / SECTION · GRADLE

Gradle이 빠른 이유 — 3대 성능 기술

① Gradle Daemon

Gradle은 JVM 위에서 동작합니다. JVM 시동 시간(JIT 워밍업 포함)이 짧지 않기 때문에, 매번 새 프로세스를 띄우면 손해입니다. Gradle Daemon은 백그라운드에서 계속 살아 있는 JVM 프로세스로, 두 번째 빌드부터는 시동 비용 없이 곧바로 작업을 시작합니다.

② Incremental Build

각 태스크는 입력(inputs)과 출력(outputs)을 선언합니다. Gradle은 모든 입력 파일의 해시를 기록해 두고, 이전 실행 이후 변화가 없으면 그 태스크를 UP-TO-DATE로 건너뜁니다.

③ Build Cache

Incremental build가 "같은 머신에서의 캐시"라면, Build Cache는 팀 전체가 공유하는 캐시입니다. 동료가 어제 빌드한 결과물을 오늘 내가 그대로 재사용할 수 있습니다. 큰 모노레포에서 빌드 시간을 극적으로 줄여 줍니다.

gradle.propertiesconfig
# Daemon 활성화 (기본값 true, 명시 권장)
org.gradle.daemon=true

# 병렬 빌드 (멀티 모듈에서 효과 큼)
org.gradle.parallel=true

# Configuration Cache (Gradle 8+ 정식)
org.gradle.configuration-cache=true

# Build Cache
org.gradle.caching=true

# JVM 힙 메모리
org.gradle.jvmargs=-Xmx2g -XX:+UseG1GC
⚡
벤치마크 감각 Spring Boot 정도 규모의 프로젝트에서, "변경 없음" 상태의 재빌드는 Maven은 보통 10~30초가 걸리는 반면, Gradle은 캐시가 켜져 있으면 1초 미만으로 끝납니다. 큰 모노레포라면 이 차이가 수십 분 vs 수 분 단위로 벌어집니다.

다른 언어의 빌드 도구와 한눈에 비교

Maven과 Gradle만의 개념이라고 생각했던 것들은 사실 다른 언어 생태계에도 거의 그대로 있습니다. 이름과 문법만 다를 뿐, 본질은 같습니다. 아래 탭에서 각 언어를 클릭해 보세요.

Node.js — npm / yarn / pnpm

Node.js의 빌드 도구는 npm 계열입니다. 흥미롭게도 Java 진영과 매우 유사한 구조를 가집니다.

package.jsonJSON
{
  "name": "my-app",            // artifactId
  "version": "1.0.0",           // version
  "dependencies": {              // implementation
    "react": "^18.3.0"
  },
  "devDependencies": {           // testImplementation
    "jest": "^29.7.0"
  },
  "scripts": {                   // 커스텀 태스크
    "build": "tsc && vite build",
    "test": "jest"
  }
}

Python — pip / Poetry / uv

Python 진영은 오랫동안 빌드 도구가 파편화되어 있었지만, 최근 Poetry와 uv가 표준화를 이끌고 있습니다. pyproject.toml은 pom.xml의 Python 버전이라 볼 수 있습니다.

pyproject.tomlTOML
[project]
name = "my-ml-app"
version = "1.0.0"
requires-python = ">=3.11"
dependencies = [
    "fastapi>=0.115.0",
    "pydantic>=2.7",
]

[tool.poetry.group.dev.dependencies]
pytest = "^8.2"
ruff = "^0.5"

[build-system]
requires = ["poetry-core"]
build-backend = "poetry.core.masonry.api"

MLOps 관점에서 흥미로운 점은, Python에는 conda처럼 OS 레벨 의존성까지 관리하는 도구가 별도로 존재한다는 것입니다. Java는 JVM이라는 추상화 덕분에 OS 의존성을 빌드 도구가 직접 신경 쓸 일이 거의 없습니다.

Rust — Cargo

Cargo는 많은 사람이 "현존 최고의 빌드 도구"로 꼽는 사례입니다. 패키지 관리, 빌드, 테스트, 문서화, 배포가 한 도구에 통합되어 있습니다. Gradle이 지향하는 모습에 가장 가까운 도구라고 볼 수 있습니다.

Cargo.tomlTOML
[package]
name = "my-app"
version = "1.0.0"
edition = "2021"

[dependencies]
tokio = { version = "1", features = ["full"] }
serde = { version = "1", features = ["derive"] }

[dev-dependencies]
criterion = "0.5"

Java 개발자가 Cargo에서 배워야 할 점은 "단일 도구로 모든 것"이라는 사용자 경험입니다. cargo new, cargo build, cargo test, cargo publish — 명령어 한 줄로 처리됩니다.

Go — go modules

Go는 의도적으로 빌드 도구를 단순하게 유지합니다. go.mod 파일과 go build, go test 명령만으로 대부분의 작업이 끝납니다. 컴파일러 자체가 의존성을 가져오고 빌드합니다.

go.modGo
module github.com/example/my-app

go 1.22

require (
    github.com/gin-gonic/gin v1.10.0
    github.com/stretchr/testify v1.9.0
)

Go의 접근법은 "빌드 도구는 컴파일러의 일부여야 한다"는 입장입니다. Java도 JDK가 점차 jpackage, jlink 같은 도구를 내장하면서 비슷한 방향으로 움직이고 있지만, 의존성 관리만큼은 여전히 Maven/Gradle의 영역입니다.

C/C++ — Make, CMake, Bazel

가장 오래된 빌드 도구 계보입니다. Make는 1976년에 등장한 원조 빌드 도구로, "파일 의존성과 명령"을 연결하는 단순한 아이디어로 시작했습니다. 오늘날 우리가 쓰는 Maven, Gradle, npm 모두 그 후손이라 할 수 있습니다.

MakefileMake
# 가장 원시적인 빌드 도구
CC = gcc
CFLAGS = -Wall -O2

my-app: main.o utils.o
	$(CC) $(CFLAGS) -o my-app main.o utils.o

main.o: main.c
	$(CC) $(CFLAGS) -c main.c

clean:
	rm -f *.o my-app

흥미로운 점은 Gradle의 태스크 모델이 Make의 "타깃과 의존성" 개념을 그대로 계승했다는 것입니다. 다만 Gradle은 그 위에 JVM, DSL, 캐시, 분산 빌드를 얹어 현대화시켰을 뿐입니다.

실전 워크플로 — 0에서 배포까지

이론을 살펴봤으니, 이제 실제 프로젝트의 처음부터 끝까지 흐름을 따라가 봅시다. Spring Boot 기반 간단한 REST API를 만든다고 가정합니다.

STEP 1 — 프로젝트 생성

가장 쉬운 방법은 Spring Initializr(start.spring.io)를 이용하는 것입니다. Maven이든 Gradle이든 한 번에 선택할 수 있습니다. 명령줄로도 가능합니다.

terminalbash
# Gradle Kotlin DSL · Spring Boot · Java 21
$ curl https://start.spring.io/starter.zip \
    -d type=gradle-project-kotlin \
    -d language=java \
    -d javaVersion=21 \
    -d bootVersion=3.3.0 \
    -d groupId=com.example \
    -d artifactId=demo \
    -d dependencies=web,actuator \
    -o demo.zip

$ unzip demo.zip -d demo
$ cd demo

STEP 2 — Wrapper 사용하기

프로젝트에는 Maven Wrapper(./mvnw) 또는 Gradle Wrapper(./gradlew)가 포함되어 있습니다. Wrapper는 빌드 도구 자체를 프로젝트에 내장시키는 표준 방식입니다. 팀원이 Maven/Gradle을 설치하지 않아도 됩니다.

★
실무 황금률 커맨드를 mvn, gradle이 아니라 항상 ./mvnw, ./gradlew로 실행하세요. Wrapper는 프로젝트가 사용해야 할 정확한 버전을 자동으로 받아 실행하므로, "내 컴퓨터에선 됐는데" 문제의 가장 흔한 원인을 차단합니다.

STEP 3 — 빌드 · 실행 · 테스트

terminalmaven
# 컴파일 + 테스트 + JAR 생성
$ ./mvnw clean package

# Spring Boot 앱 실행
$ ./mvnw spring-boot:run

# 테스트만 실행
$ ./mvnw test

# 특정 테스트 클래스만
$ ./mvnw test -Dtest=UserControllerTest

# 의존성 트리 확인
$ ./mvnw dependency:tree

# 사용하지 않는 의존성 분석
$ ./mvnw dependency:analyze
terminalgradle
# 컴파일 + 테스트 + JAR 생성
$ ./gradlew build

# Spring Boot 앱 실행
$ ./gradlew bootRun

# 테스트만 실행
$ ./gradlew test

# 특정 테스트 클래스만
$ ./gradlew test --tests UserControllerTest

# 의존성 트리 확인
$ ./gradlew dependencies

# 특정 라이브러리가 왜 들어왔는지 추적
$ ./gradlew dependencyInsight --dependency jackson-core

# 사용 가능한 모든 태스크 보기
$ ./gradlew tasks

STEP 4 — CI/CD 연동

이제 코드를 GitHub에 푸시했다고 합시다. GitHub Actions로 자동 빌드를 돌리는 워크플로는 다음과 같이 작성합니다.

.github/workflows/build.ymlYAML
name: Build & Test

on:
  push:
    branches: [main, develop]
  pull_request:

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Set up JDK 21
        uses: actions/setup-java@v4
        with:
          distribution: temurin
          java-version: '21'
          cache: gradle   # 또는 'maven'

      - name: Build with Gradle
        run: ./gradlew build --no-daemon

      - name: Upload artifacts
        uses: actions/upload-artifact@v4
        with:
          name: jar
          path: build/libs/*.jar

STEP 5 — Kubernetes 환경에서의 빌드

인프라 엔지니어의 관점에서, 빌드 도구의 마지막 관문은 컨테이너화입니다. Jib 플러그인을 쓰면 Dockerfile 없이도 OCI 이미지를 만들 수 있고, BuildKit을 쓰면 캐시 효율을 극대화할 수 있습니다.

build.gradle.kts — Jib 사용 예Kotlin
plugins {
    id("com.google.cloud.tools.jib") version "3.4.3"
}

jib {
    from {
        image = "eclipse-temurin:21-jre"
    }
    to {
        image = "registry.example.com/my-app"
        tags = setOf("latest", version.toString())
    }
    container {
        ports = listOf("8080")
        jvmFlags = listOf("-Xms256m", "-Xmx512m")
    }
}

이제 ./gradlew jib 한 줄로 멀티 아키텍처(ARM64/x86_64) OCI 이미지가 빌드되고 레지스트리에 푸시됩니다.

⌘
이종 노드 클러스터에서의 팁 DGX Spark(ARM64) + ThinkStation(x86_64) 같은 이종 노드 환경에서 Java 애플리케이션을 배포한다면, Jib의 platforms 설정이나 docker buildx를 활용해 멀티 아키텍처 매니페스트를 한 번에 푸시할 수 있습니다. JVM 자체가 OS/아키텍처를 추상화해 주기 때문에, Java는 멀티 아키텍처 배포에서 가장 운영이 편한 언어 중 하나입니다.

자가 점검 퀴즈

강의에서 배운 내용을 점검해 봅시다. 답을 고르면 즉시 해설이 표시됩니다.

Maven의 GAV 좌표 중 artifactId의 역할로 가장 정확한 설명은?
GAV(groupId · artifactId · version)는 Maven 좌표계입니다. artifactId는 한 조직 안에서 그 산출물을 구분하는 이름이며, groupId·version과 결합되어 Maven Central에서 유일한 JAR을 가리킵니다.
다음 중 Gradle이 Maven보다 빠른 이유가 아닌 것은?
XML이냐 DSL이냐는 빌드 속도에 거의 영향을 주지 않습니다. Gradle의 속도는 Daemon · Incremental Build · Build Cache 세 기술의 결합에서 나옵니다.
팀에서 만드는 라이브러리에 Guava를 내부 구현용으로만 사용하고 외부에는 노출하지 않으려고 합니다. Gradle에서 가장 적절한 설정은?
implementation으로 선언하면 의존성이 라이브러리 사용자에게 노출되지 않아 캡슐화가 보장되고, 컴파일 시간도 짧아집니다. api는 의존성을 의도적으로 전파하고 싶을 때만 사용합니다.
Maven에서 mvn package를 실행하면 자동으로 함께 실행되는 단계는?
Maven의 phase는 순차 실행됩니다. package 단계를 호출하면 그 이전 단계인 validate → compile → test → package가 모두 실행됩니다. install, deploy는 명시적으로 호출해야 실행됩니다.
Python의 pyproject.toml은 Java 생태계의 무엇과 가장 직접적으로 대응되는가?
pyproject.toml은 프로젝트 메타데이터 · 의존성 · 빌드 설정을 모두 담는 파일이므로, Java의 pom.xml 또는 build.gradle.kts에 정확히 대응됩니다. 패키지 저장소(Maven Central ↔ PyPI)와 혼동하지 않도록 주의하세요.
