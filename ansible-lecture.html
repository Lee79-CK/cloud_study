01 / 기초

Ansible란 무엇인가

한 문장으로: "여러 서버에 똑같은 설정을 자동으로, 안전하게 적용해 주는 도구"입니다. 사람이 SSH로 하나씩 접속해 명령어를 치던 일을, 코드 한 벌로 대신합니다.

왜 필요할까 — 손으로 하면 생기는 문제

클라우드에 서버 10대를 띄웠다고 합시다. 각 서버에 Nginx를 깔고, 방화벽을 열고, 설정 파일을 올려야 합니다. 손으로 하면 이런 일이 벌어집니다.

!

휴먼 에러

3번 서버에서만 오타로 포트를 잘못 열었습니다. 나중에 장애가 나서야 발견합니다.

≠

설정 드리프트

서버마다 미묘하게 상태가 다릅니다. "내 PC에선 됐는데"가 서버 단위로 일어납니다.

⏱

반복 노동

10대면 같은 작업을 10번. 100대면 100번. 사람의 시간이 그대로 녹습니다.

Ansible은 "어떤 상태가 되어야 하는가"를 YAML 파일에 한 번만 적어두면, 그 파일을 대상 서버 수와 무관하게 똑같이 적용합니다. 1대든 1,000대든 명령은 동일합니다.

핵심 정의

Ansible은 구성 관리(Configuration Management)와 애플리케이션 배포, 오케스트레이션을 담당하는 오픈소스 자동화 엔진입니다. 2012년 Michael DeHaan이 만들었고, 2015년 Red Hat이 인수해 현재는 Red Hat 생태계의 핵심 도구입니다.

다른 도구와의 가족 관계

자동화 도구는 크게 둘로 나뉩니다. 이 구분을 머릿속에 박아두면 뒤의 Terraform 비교가 훨씬 쉬워집니다.

  • 프로비저닝(Provisioning) — 서버, 네트워크, 스토리지 같은 인프라 자원을 "만드는" 일. 대표주자 Terraform.
  • 구성 관리(Configuration Management) — 이미 만들어진 서버 "안에" 들어가 소프트웨어를 깔고 설정하는 일. 대표주자 Ansible. (Chef, Puppet, SaltStack도 같은 가족)

실무에서는 둘을 함께 씁니다. Terraform으로 서버를 만들고, Ansible로 그 서버를 설정하는 조합이 가장 흔합니다. 12장에서 자세히 다룹니다.

02 / 기초

핵심 개념 3가지

Ansible을 다른 도구와 구별 짓는 세 가지 성질입니다. 이것만 이해해도 절반은 끝났습니다.

① Agentless — 대상 서버에 아무것도 안 깔아도 된다

Puppet이나 Chef 같은 옛 도구는 관리할 서버마다 전용 프로그램(에이전트)을 설치해야 했습니다. Ansible은 다릅니다. SSH와 Python만 있으면 끝입니다. 대부분의 리눅스 서버에는 이미 둘 다 있으니, 사실상 추가 설치가 없습니다.

왜 좋은가

관리 대상이 늘어도 에이전트를 깔고 유지보수할 필요가 없습니다. 에이전트가 죽거나 버전이 안 맞아 생기는 문제 자체가 없습니다. 네트워크 장비나 클라우드 API처럼 에이전트를 못 까는 대상도 다룰 수 있습니다.

② 멱등성(Idempotency) — 몇 번 돌려도 결과가 같다

가장 중요한 개념입니다. 같은 플레이북을 100번 실행해도, 서버의 최종 상태는 1번 실행했을 때와 똑같다는 뜻입니다.

비결은 Ansible이 "명령"이 아니라 "상태"를 다룬다는 점입니다. "Nginx를 설치하라"가 아니라 "Nginx가 설치된 상태를 보장하라"고 말합니다. 이미 설치돼 있으면 Ansible은 아무 일도 하지 않고 ok라고 보고합니다. 안 깔려 있을 때만 설치하고 changed라고 보고합니다.

상황
멱등하지 않은 셸 스크립트
멱등한 Ansible
첫 실행
유저 생성됨 ✓
유저 생성됨 → changed
두 번째 실행
"이미 존재함" 에러로 중단
아무 일 안 함 → ok (안전)
실무 의미

멱등성 덕분에 "지금 서버 상태가 정확히 뭔지 몰라도" 일단 플레이북을 돌리면 됩니다. 부족한 부분만 채워지고, 이미 맞는 부분은 건드리지 않습니다. 이것이 셸 스크립트와의 결정적 차이입니다.

③ Push 모델 — 중앙에서 밀어 넣는다

당신의 노트북이나 관리 서버(컨트롤 노드)에서 명령을 내리면, Ansible이 SSH로 각 대상 서버에 접속해 작업을 "밀어 넣습니다(push)". 대상 서버가 중앙 서버에 "할 일 있나요?" 하고 물으러 오는 Pull 방식(Puppet 등)과 반대입니다.

그래서 별도의 마스터 서버를 항상 띄워둘 필요가 없습니다. 필요할 때, 명령을 내리는 그 컴퓨터가 곧 컨트롤 노드입니다.

03 / 기초

구조와 동작 원리

Ansible이 일하는 그림을 한 장으로 봅니다. 등장인물은 다섯입니다.

컨트롤 노드 (Control Node)

Ansible이 설치되어 명령을 내리는 곳. 보통 내 노트북이나 CI 서버. 여기서만 Ansible이 필요합니다.

관리 노드 (Managed Node)

설정 대상 서버들. SSH 접속과 Python만 되면 충분. Ansible을 따로 깔지 않습니다.

인벤토리 (Inventory)

"어떤 서버들을 관리할지" 적어둔 목록. IP나 호스트명을 그룹으로 묶습니다.

모듈 (Module)

실제 작업을 수행하는 부품. 패키지 설치, 파일 복사, 서비스 재시작 등 수천 개가 기본 제공됩니다.

플레이북 (Playbook)

"무엇을 어떤 순서로 할지" 적은 YAML 시나리오. 모듈들을 엮어 하나의 작업 흐름을 만듭니다.

플러그인 · 팩트

연결·출력 등을 확장하는 플러그인, 그리고 대상 서버에서 자동 수집한 정보(facts)도 함께 동작합니다.

한 번에 외우기

인벤토리(누구에게) + 플레이북(무엇을) → 컨트롤 노드가 SSH로 모듈을 밀어 넣어 관리 노드를 원하는 상태로 만든다. 이게 Ansible의 전부입니다.

04 / 실습

설치와 첫 연결

설치는 내 컴퓨터(컨트롤 노드)에만 하면 됩니다. 대상 서버에는 손대지 않습니다.

control node 
# Homebrew로 설치
brew install ansible

# 버전 확인
ansible --version
# 공식 PPA 추가 후 설치
sudo apt update
sudo apt install -y software-properties-common
sudo add-apt-repository --yes --update ppa:ansible/ansible
sudo apt install -y ansible
# 어떤 OS든 Python만 있으면 가능
python3 -m pip install --user ansible

# ansible-core만 가볍게 원하면
pip install ansible-core

대상 서버에 SSH 키로 접속 준비

Ansible은 SSH로 접속합니다. 매번 비밀번호를 묻지 않도록 공개키를 대상 서버에 등록해 둡니다.

bash
# 키가 없으면 생성
ssh-keygen -t ed25519

# 대상 서버에 공개키 복사 (이때 한 번만 비밀번호 입력)
ssh-copy-id ubuntu@192.168.0.11

# 이제 비밀번호 없이 접속되면 준비 완료
ssh ubuntu@192.168.0.11
연결 테스트

인벤토리(다음 장)를 만든 뒤 ansible all -m ping을 실행하면 모든 대상 서버에 핑을 던져 SSH·Python 연결이 정상인지 한 번에 확인할 수 있습니다. 응답이 pong이면 성공입니다.

05 / 실습

인벤토리 — 누구를 관리할지

관리할 서버 목록입니다. 서버를 역할별 그룹으로 묶으면, 플레이북에서 "웹 서버 그룹에만 적용" 같은 일이 쉬워집니다.

inventory.ini 
# 그룹 이름은 대괄호로
[web]
web-01 ansible_host=192.168.0.11
web-02 ansible_host=192.168.0.12

[db]
db-01 ansible_host=192.168.0.21

# 그룹들을 묶는 상위 그룹
[production:children]
web
db

# 그룹 공통 변수
[web:vars]
ansible_user=ubuntu
ansible_python_interpreter=/usr/bin/python3
all:
  children:
    web:
      hosts:
        web-01:
          ansible_host: 192.168.0.11
        web-02:
          ansible_host: 192.168.0.12
      vars:
        ansible_user: ubuntu
    db:
      hosts:
        db-01:
          ansible_host: 192.168.0.21
# 클라우드에서는 서버가 수시로 생기고 사라집니다.
# 목록을 손으로 적는 대신, AWS/Azure/GCP에
# 실시간으로 물어보는 "동적 인벤토리"를 씁니다.

# 예: AWS EC2 동적 인벤토리 플러그인
# aws_ec2.yml
plugin: amazon.aws.aws_ec2
regions:
  - ap-northeast-2
keyed_groups:
  - key: tags.Role      # 태그로 자동 그룹핑
    prefix: role

# 실행: 태그가 Role=web 인 인스턴스가
# 자동으로 role_web 그룹에 들어갑니다.
ansible-inventory -i aws_ec2.yml --graph
클라우드 엔지니어 포인트

고정 IP를 INI에 박아두는 방식은 학습용입니다. 실무의 클라우드 환경에서는 동적 인벤토리가 표준입니다. 오토스케일링으로 인스턴스가 늘었다 줄어도, 태그만 잘 붙여두면 인벤토리가 알아서 따라옵니다.

06 / 실습

애드혹 명령 — 한 줄짜리 즉석 작업

플레이북을 쓰기 전에, 명령줄에서 즉시 한 가지 작업만 돌리는 방법입니다. "지금 모든 웹 서버 디스크 용량만 빨리 보고 싶다" 같은 일에 씁니다.

bash
# 형식:  ansible <대상> -m <모듈> -a "<인자>"

# 1) 연결 확인
ansible all -m ping

# 2) web 그룹의 디스크 용량 확인 (셸 명령 실행)
ansible web -m shell -a "df -h"

# 3) 모든 서버 패키지 캐시 업데이트 (관리자 권한 필요 → --become)
ansible all -m apt -a "update_cache=yes" --become

# 4) 특정 서비스 재시작
ansible web -m service -a "name=nginx state=restarted" --become

# 5) 서버 정보(facts) 한눈에 보기
ansible web-01 -m setup
팁

-m은 모듈(module), -a는 인자(arguments), --become은 sudo로 권한 상승입니다. 애드혹은 일회성·조회용으로만 쓰고, 반복하거나 공유할 작업은 반드시 플레이북으로 만드세요.

07 / 실습

플레이북 — 작업의 시나리오

Ansible의 본체입니다. "어떤 대상에, 어떤 작업들을, 어떤 순서로" 적은 YAML 파일입니다. 음악의 악보처럼, 한 번 써두면 누구나 똑같이 재생할 수 있습니다.

구조 이해하기

플레이북은 하나 이상의 플레이(play)로 이뤄지고, 각 플레이는 대상(hosts)과 작업 목록(tasks)을 가집니다. 각 작업은 모듈을 하나씩 호출합니다.

site.yml 
# --- 가 YAML 문서의 시작 표시
---
- name: 첫 번째 플레이      # 사람이 읽을 설명
  hosts: web              # 어떤 그룹에 적용?
  become: true             # sudo 권한으로 실행

  tasks:                    # 위에서 아래로 순서대로 실행
    - name: 패키지가 설치된 상태 보장
      ansible.builtin.apt:   # 모듈 이름
        name: nginx
        state: present
---
- name: Nginx 웹 서버 구성
  hosts: web
  become: true

  tasks:
    - name: Nginx 설치
      ansible.builtin.apt:
        name: nginx
        state: present
        update_cache: true

    - name: 홈페이지 파일 복사
      ansible.builtin.copy:
        src: files/index.html
        dest: /var/www/html/index.html

    - name: Nginx 실행 + 부팅 시 자동 시작
      ansible.builtin.service:
        name: nginx
        state: started
        enabled: true
# 실행
ansible-playbook -i inventory.ini site.yml

# 실제로 바꾸기 전에 "무엇이 바뀔지" 미리보기 (강력 추천)
ansible-playbook -i inventory.ini site.yml --check --diff

# 특정 호스트에만
ansible-playbook -i inventory.ini site.yml --limit web-01

# 문법 검사만
ansible-playbook site.yml --syntax-check
--check 모드를 사랑하세요

--check는 실제로는 아무것도 바꾸지 않고 "이대로 돌리면 무엇이 changed 될지"만 보여주는 드라이런(dry-run)입니다. 운영 서버에 적용하기 전 안전벨트입니다. --diff를 붙이면 파일이 어떻게 바뀌는지까지 보여줍니다.

08 / 실습

플레이북 실행 체험

실제 ansible-playbook 실행을 흉내 낸 시뮬레이터입니다. 처음 실행과 다시 실행 버튼을 눌러, 멱등성이 출력으로 어떻게 드러나는지 직접 보세요.

site.yml — Nginx 웹 서버 구성

대상: web-01, web-02
Playbook
--- - hosts: web become: true tasks: - name: Nginx 설치 apt: { name: nginx, state: present } - name: index.html 배포 copy: { src: index.html, dest: /var/www/html/ } - name: Nginx 시작 service: { name: nginx, state: started }
출력 (Output)
버튼을 눌러 실행해 보세요 →
출력 읽는 법

각 작업 결과는 ok(이미 원하는 상태), changed(변경 발생), skipping(조건 불충족으로 건너뜀)으로 보고됩니다. 맨 아래 PLAY RECAP에 호스트별 집계가 나옵니다. 두 번째 실행에서 changed가 0이 되는 것 — 그것이 멱등성입니다.

09 / 심화

변수 · 팩트 · 모듈

플레이북을 재사용 가능하게 만드는 세 요소입니다.

변수(Variables) — 값을 한 곳에서 관리

포트 번호나 패키지 이름 같은 값을 변수로 빼두면, 환경마다 값만 바꿔 같은 플레이북을 재사용할 수 있습니다. {{ 중괄호 두 개 }}로 변수를 참조합니다.

vars
- hosts: web
  vars:
    http_port: 8080
    app_pkg: nginx
  tasks:
    - name: "{{ app_pkg }} 설치"
      ansible.builtin.apt:
        name: "{{ app_pkg }}"
        state: present
# 같은 변수가 여러 곳에 있으면 아래로 갈수록 우선
#  1. 역할 기본값 (roles/x/defaults)   ← 가장 약함
#  2. 인벤토리 group_vars
#  3. 인벤토리 host_vars
#  4. 플레이북 vars
#  5. -e 로 넘긴 추가 변수            ← 가장 강함

ansible-playbook site.yml -e "http_port=9090"

팩트(Facts) — 서버가 스스로 알려주는 정보

플레이북이 시작되면 Ansible은 각 서버에서 OS 종류, IP, CPU 수, 메모리 같은 정보를 자동 수집합니다. 이를 팩트라 하며, 변수처럼 바로 쓸 수 있습니다.

yaml
- name: OS에 따라 다른 패키지 설치
  ansible.builtin.debug:
    msg: "이 서버는 {{ ansible_distribution }}
          {{ ansible_distribution_version }},
          메모리 {{ ansible_memtotal_mb }}MB"

# 자주 쓰는 팩트들
#   ansible_hostname     호스트명
#   ansible_default_ipv4.address   기본 IP
#   ansible_os_family    RedHat / Debian ...
#   ansible_processor_vcpus   vCPU 수

모듈(Modules) — 작업의 부품

각 작업은 모듈 하나를 호출합니다. 클라우드 엔지니어가 가장 자주 만나는 것들만 추렸습니다.

📦

apt / yum / dnf

패키지 설치·제거. OS 패키지 매니저를 멱등하게 감쌉니다.

📄

copy / template

파일을 올리거나, 변수를 채운 설정 파일을 생성합니다.

⚙

service / systemd

서비스 시작·중지·재시작, 부팅 시 자동 실행 설정.

👤

user / group

계정과 그룹을 생성·관리합니다.

$

command / shell

임의 명령 실행. 단 멱등하지 않으니 최후의 수단으로.

☁

amazon.aws.* 등

클라우드 자원 제어 모듈. 컬렉션으로 제공됩니다.

전체 모듈 이름 규칙

요즘은 ansible.builtin.apt처럼 네임스페이스.컬렉션.모듈 형태(FQCN)로 부르는 것을 권장합니다. 짧게 apt로도 되지만, 충돌을 막으려면 전체 이름이 안전합니다.

10 / 심화

조건 · 반복 · 핸들러

단순 나열을 넘어, 상황에 따라 분기하고 반복하며 변경 시에만 반응하게 만드는 장치들입니다.

when — 조건부 실행

yaml
- name: 데비안 계열에서만 apt 사용
  ansible.builtin.apt:
    name: nginx
    state: present
  when: ansible_os_family == "Debian"

- name: 레드햇 계열에서만 dnf 사용
  ansible.builtin.dnf:
    name: nginx
    state: present
  when: ansible_os_family == "RedHat"
- name: 여러 패키지를 한 번에
  ansible.builtin.apt:
    name: "{{ item }}"
    state: present
  loop:
    - nginx
    - git
    - htop

- name: 계정 여러 개 생성
  ansible.builtin.user:
    name: "{{ item.name }}"
    groups: "{{ item.group }}"
  loop:
    - { name: alice, group: dev }
    - { name: bob,   group: ops }
# 핸들러: "변경이 일어났을 때만" 실행되는 작업
# 설정 파일이 바뀐 경우에만 서비스를 재시작 → 불필요한 재시작 방지

  tasks:
    - name: Nginx 설정 배포
      ansible.builtin.template:
        src: nginx.conf.j2
        dest: /etc/nginx/nginx.conf
      notify: Restart nginx   # 변경 시 핸들러 호출

  handlers:
    - name: Restart nginx
      ansible.builtin.service:
        name: nginx
        state: restarted
핸들러의 묘미

핸들러는 플레이의 모든 작업이 끝난 뒤, 실제로 changed가 발생한 경우에만 한 번 실행됩니다. 설정이 그대로면 서비스는 재시작되지 않습니다. 이것이 "바뀐 게 없으면 건드리지 않는다"는 멱등성 철학의 실천입니다.

11 / 심화

템플릿 · 롤 · Vault

규모가 커질 때 코드를 깔끔하게 유지하고, 비밀을 안전하게 다루는 도구들입니다.

Jinja2 템플릿 — 변수가 채워지는 설정 파일

서버마다 조금씩 다른 설정 파일이 필요할 때, 템플릿(.j2)에 변수를 박아두면 Ansible이 각 서버에 맞게 값을 채워 생성합니다.

jinja2
# 변수와 반복문이 들어간 설정 템플릿
server {
    listen {{ http_port }};
    server_name {{ ansible_hostname }};

    {% for path in proxy_paths %}
    location {{ path }} {
        proxy_pass http://backend;
    }
    {% endfor %}
}

롤(Roles) — 재사용 가능한 작업 묶음

플레이북이 길어지면 관리가 어렵습니다. 롤은 관련된 작업·변수·템플릿·파일을 정해진 폴더 구조로 묶은 패키지입니다. "nginx 롤", "docker 롤"처럼 만들어 여러 프로젝트에서 재사용합니다.

roles/
roles/
└── nginx/
    ├── tasks/main.yml        # 핵심 작업
    ├── handlers/main.yml     # 핸들러
    ├── templates/nginx.conf.j2
    ├── files/index.html
    ├── defaults/main.yml     # 기본 변수
    └── vars/main.yml         # 내부 변수
---
- hosts: web
  become: true
  roles:
    - common      # 공통 기본 설정 롤
    - nginx       # 웹 서버 롤
    - monitoring  # 모니터링 에이전트 롤

# 검증된 롤은 Ansible Galaxy에서 받아 쓸 수도 있습니다
#   ansible-galaxy install geerlingguy.nginx
Ansible Galaxy

Galaxy는 커뮤니티가 만든 롤과 컬렉션을 공유하는 저장소입니다. 잘 만들어진 롤(예: geerlingguy 시리즈)을 받아 쓰면 바퀴를 다시 발명할 필요가 없습니다.

Ansible Vault — 비밀번호를 코드에 안전하게

DB 비밀번호, API 키 같은 민감 정보를 평문으로 저장소에 올릴 수는 없습니다. Vault는 이런 값을 암호화해 보관하고, 실행 시점에만 복호화합니다.

bash
# 암호화된 변수 파일 생성 (편집기로 비밀 작성)
ansible-vault create secrets.yml

# 기존 파일 암호화 / 편집 / 복호화
ansible-vault encrypt vars/prod.yml
ansible-vault edit secrets.yml
ansible-vault view secrets.yml

# 플레이북 실행 시 비밀번호 입력받아 자동 복호화
ansible-playbook site.yml --ask-vault-pass
12 / 비교

Ansible vs Terraform

둘 다 IaC(Infrastructure as Code) 도구지만 역할이 다릅니다. 경쟁자가 아니라 역할 분담하는 동료에 가깝습니다. 핵심 차이를 항목별로 비교합니다.

한 줄 요약

Terraform = 집을 짓는 도구(인프라를 만든다). Ansible = 집에 가구를 들이고 꾸미는 도구(서버 안을 설정한다). 영역이 겹치기도 하지만 각자 가장 잘하는 곳이 다릅니다.

대부분의 현장에서는 두 도구를 파이프라인으로 연결합니다. 역할이 겹치지 않으니 충돌이 없습니다.

Terraform이 인프라를 만든다

VPC, 서브넷, 보안 그룹, EC2 인스턴스, 로드밸런서를 선언적으로 프로비저닝합니다.

출력으로 서버 정보를 넘긴다

생성된 인스턴스의 IP·태그를 Ansible 동적 인벤토리가 읽어 들입니다.

Ansible이 서버를 설정한다

OS 패치, 미들웨어 설치, 앱 배포, 모니터링 에이전트 구성을 멱등하게 적용합니다.

운영 중 변경도 Ansible로

설정 변경·재배포는 Ansible 플레이북 재실행으로 처리합니다. 인프라 자체의 증감은 다시 Terraform으로.

결론

"Ansible이냐 Terraform이냐"는 잘못된 질문입니다. Terraform으로 만들고(provision), Ansible로 길들인다(configure) — 이 조합이 가장 널리 쓰이는 정석입니다.

13 / 응용

실전 응용 분야

클라우드 엔지니어가 Ansible을 실제로 어디에 쓰는지, 대표 시나리오 여섯 가지입니다.

🖥

서버 초기 구성 (Provisioning 후속)

새로 띄운 서버에 표준 패키지, 보안 설정, 사용자 계정, 시간 동기화를 일괄 적용해 "골든 베이스라인"을 만듭니다.

🚀

애플리케이션 배포

코드 가져오기 → 의존성 설치 → 설정 주입 → 서비스 재시작을 한 플레이북으로. serial 옵션으로 무중단 롤링 배포도 구현합니다.

🔁

CI/CD 파이프라인 연동

GitLab CI·Jenkins·GitHub Actions에서 빌드 후 배포 단계로 ansible-playbook을 호출합니다. 배포 로직을 코드로 버전 관리합니다.

🛡

보안 · 컴플라이언스 강제

CIS 벤치마크 같은 보안 기준을 플레이북으로 정의하고 주기적으로 실행해, 설정 드리프트를 자동 교정합니다.

🌐

네트워크 장비 자동화

에이전트를 못 까는 스위치·라우터·방화벽도 전용 컬렉션으로 설정을 일괄 관리합니다. agentless의 진가가 드러나는 영역입니다.

📊

관측·MLOps 스택 구성

Prometheus·Grafana·Loki 같은 모니터링 스택이나 모델 서빙 노드의 환경을 표준화된 롤로 재현 가능하게 배포합니다.

한 걸음 더

여러 플레이북을 GUI·RBAC·스케줄링과 함께 운영하고 싶다면 AWX(오픈소스)나 상용 Ansible Automation Platform으로 확장할 수 있습니다. 팀 단위 운영에서 감사 로그와 권한 관리가 필요할 때 고려합니다.

14 / 응용

베스트 프랙티스

초급에서 중급으로 넘어갈 때 몸에 배어 있어야 할 습관들입니다.

① 모든 task에 name을 단다

출력이 사람이 읽을 수 있게 되고, 디버깅이 쉬워집니다. name 없는 작업은 미래의 나를 괴롭힙니다.

② 운영 적용 전 --check --diff

무엇이 바뀔지 먼저 확인하는 습관. 운영 서버에서의 실수를 가장 많이 막아줍니다.

③ command/shell은 최후의 수단

전용 모듈이 있으면 그것을 쓰세요. 셸 명령은 멱등성을 깨뜨립니다. 꼭 써야 하면 creates·changed_when으로 보완합니다.

④ 비밀은 반드시 Vault로

비밀번호·키를 평문으로 Git에 올리지 않습니다. ansible-vault로 암호화합니다.

⑤ 롤로 구조화한다

플레이북이 길어지면 롤로 쪼갭니다. 재사용성과 가독성이 함께 올라갑니다.

⑥ ansible-lint로 점검

린터로 안티패턴을 자동 검출합니다. CI에 넣어두면 품질이 일정하게 유지됩니다.

⑦ 변수는 group_vars / host_vars로

변수를 플레이북에 흩뿌리지 말고 인벤토리 옆 폴더에 모아 환경별로 관리합니다.

⑧ 멱등성을 늘 의식한다

"이 작업을 두 번 돌려도 안전한가?"를 작성할 때마다 자문합니다. 이것이 Ansible 사고방식의 핵심입니다.

15 / 점검

이해도 점검 퀴즈

핵심 개념을 제대로 잡았는지 확인해 보세요. 정답을 누르면 해설이 나옵니다.

Q1
Ansible이 "agentless"라는 말의 의미로 가장 정확한 것은?
대상 서버에 에이전트(전용 프로그램)를 깔지 않고, 이미 있는 SSH와 Python만으로 작업을 수행합니다. 컨트롤 노드에만 Ansible을 설치합니다.
Q2
같은 플레이북을 두 번째 실행했을 때 "changed=0, ok=3"이 나왔습니다. 무엇을 뜻하나요?
ok=3은 세 작업 모두 "이미 원하는 상태"임을 확인했고, changed=0은 실제 변경이 없었다는 뜻입니다. 멱등성의 전형적인 결과입니다.
Q3
"VPC와 EC2 인스턴스 50대를 새로 만든다"는 작업에 더 적합한 도구는?
인프라 자원을 "만드는" 일은 Terraform의 영역입니다. 상태 파일로 자원을 추적하고 destroy로 회수까지 깔끔합니다. 만든 뒤 서버 내부 설정은 Ansible이 맡습니다.
Q4
핸들러(handler)는 언제 실행되나요?
핸들러는 플레이 끝에서, 자신을 notify한 작업이 changed 되었을 때만 실행됩니다. 설정이 그대로면 재시작 등 부수 작업을 건너뜁니다.
Q5
민감한 DB 비밀번호를 플레이북에서 안전하게 다루는 올바른 방법은?
Ansible Vault로 변수 파일을 암호화하고, 실행 시점에만 --ask-vault-pass 등으로 복호화합니다. 평문 비밀은 절대 저장소에 올리지 않습니다.
