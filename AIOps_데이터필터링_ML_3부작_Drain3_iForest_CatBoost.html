Section 01 — Motivation

왜 "필터링"이 AIOps의 첫 번째 문제인가

AIOps를 "로그를 LLM에 넣으면 장애 원인을 알려주는 시스템"으로 오해하기 쉽습니다. 실제 현장의 첫 번째 벽은 훨씬 단순하고 물리적입니다. 데이터가 너무 많다는 것입니다.

1.1 규모의 문제를 숫자로 보기

중형 규모의 서비스 인프라(VM 수백 대 + K8s 클러스터 몇 개)만 되어도 로그는 초당 수만~수십만 건, 하루 수억~수십억 건 규모로 쏟아집니다. 위성 관제, 통신망 관제처럼 장비 텔레메트리가 붙는 도메인이라면 여기에 시계열 메트릭까지 더해집니다. 이 데이터를 LLM에 그대로 흘려보내는 것은 세 가지 이유로 불가능합니다.

비용·물리 한계

10억 라인 ≈ 수백 GB/일. 토큰으로 환산하면 수천억 토큰입니다. H200을 수백 장 붙여도 실시간 처리는 불가능하고, 가능하더라도 GPU 비용이 장애 손실보다 커집니다.

신호 대 잡음비

로그의 99.9% 이상은 "정상 동작 기록"입니다. 정상 로그로 컨텍스트를 채우면 LLM은 정작 중요한 한 줄을 놓칩니다. 건초더미를 통째로 주면 바늘은 못 찾습니다.

알람 피로 (Alert Fatigue)

필터링 없는 탐지는 오탐(False Positive)을 양산합니다. 하루 수천 건의 알람을 받는 운영자는 결국 알람을 무시하게 되고, 진짜 장애도 함께 묻힙니다.

1.2 해법: 계단식(Cascade) 필터링

해법은 단일 만능 모델이 아니라, 싸고 빠른 모델부터 비싸고 똑똑한 모델 순서로 계단을 쌓는 것입니다. 공항 보안 검색과 같은 원리입니다. 전원이 금속탐지기(저비용)를 통과하고, 걸린 사람만 정밀 검색(중비용)을 받고, 그중 극소수만 심층 조사(고비용)를 받습니다.

단계모델학습 방식역할 (공항 비유)처리량건당 비용
1단계Drain3온라인 클러스터링 (모델 없음)수하물 X-ray — 전수 검사, 구조화수십만 line/s (CPU 1코어)≈ 0
2단계Isolation Forest비지도 (라벨 불필요)금속탐지기 — "평소와 다름" 감지수만 sample/s (CPU)매우 낮음
3단계CatBoost지도학습 (운영자 라벨)보안 요원 심사 — 진짜/가짜 판별수천 sample/s낮음
4단계sLLM / LLM사전학습 + RAG심층 조사실 — 원인 분석·조치 제안수 건/s (GPU)높음
이 조합이 사실상의 표준인 이유

세 모델은 각자 다른 종류의 "모름"을 해결합니다. Drain3는 비정형 텍스트를 정형 데이터로 바꾸는 문제(파싱), Isolation Forest는 라벨이 하나도 없는 초기 상태에서의 탐지 문제(콜드 스타트), CatBoost는 운영자 피드백이 쌓인 뒤의 정밀 분류 문제(오탐 억제)를 풉니다. 즉 이 3부작은 "모델 3개"가 아니라 AIOps 시스템의 성장 단계 그 자체입니다.

2.1 한 줄 요약으로 미리 보는 세 모델

Drain3 = 문장 틀 추출기

"user 42 logged in"과 "user 87 logged in"은 사실 같은 문장 틀 user <*> logged in입니다. Drain3는 흘러가는 로그에서 이런 틀(템플릿)을 실시간으로 찾아내 수백만 줄을 수천 개 패턴으로 접습니다.

iForest = 스무고개 이상 탐지

"무리에서 떨어진 점일수록 적은 질문으로 고립시킬 수 있다"는 직관 하나로 동작합니다. 정상이 무엇인지 정의하지 않아도, 라벨이 없어도, 남들과 다른 놈을 찾아냅니다.

CatBoost = 경력직 심사관

운영자가 "이건 진짜 장애 / 이건 노이즈"라고 매긴 판정 이력을 학습해, 이상 후보 중 진짜만 통과시킵니다. 범주형 피처(장비명, 에러코드)를 전처리 없이 잘 다루는 게 특기입니다.

학습 로드맵

3장은 각 모델을 ① 직관 → ② 알고리즘 원리 → ③ 코드 사용법 → ④ 유사 모델 비교 순서로 다룹니다. 6장에서 세 모델을 하나의 파이프라인으로 엮고, 7장에서 Kubernetes 위의 실제 배치 아키텍처로 구현합니다. 급하다면 각 장의 인터랙티브 데모만 먼저 돌려보고 돌아와도 좋습니다.

Section 03 — Stage 1 · Log Template Mining

Drain3 — 로그를 "문장 틀"로 접는 기술

ML 파이프라인의 가장 큰 적은 비정형 텍스트입니다. Drain3는 자유 텍스트 로그를 템플릿 ID + 파라미터라는 정형 데이터로 바꿔주는 온라인 로그 파서입니다. 뒤따르는 모든 모델(iForest, CatBoost, LLM)의 입력 품질이 여기서 결정됩니다.

3.1 직관: 로그 파싱이란 무엇인가

개발자가 코드에 이렇게 썼다고 합시다.

java
logger.info("Connection from {} closed, session {} lasted {} ms", ip, sid, dur);

운영 환경에서 이 한 줄의 코드는 하루에 수백만 번 실행되며 매번 다른 로그를 남깁니다.

log
Connection from 10.0.3.17 closed, session a9f2 lasted 3021 ms
Connection from 10.0.8.44 closed, session 77b0 lasted 1288 ms
Connection from 10.0.1.02 closed, session c3d1 lasted 45802 ms

로그 파싱(log parsing / template mining)은 이 역과정입니다. 즉, 출력된 로그만 보고 원래의 print 문(템플릿)을 복원하는 작업입니다.

template
Template #17 : Connection from <*> closed, session <*> lasted <*> ms
Parameters  : ["10.0.3.17", "a9f2", 3021]

이렇게 바꾸면 세 가지 강력한 효과가 생깁니다.

차원 붕괴

수억 개의 서로 다른 문자열이 수백~수천 개의 템플릿 ID로 접힙니다. 이제 로그는 "이벤트 타입의 시계열"이라는 카운트 가능한 데이터가 됩니다.

피처화 가능

"템플릿 17이 분당 몇 번 발생했나" 같은 수치 피처를 만들 수 있습니다. 이것이 바로 다음 단계 iForest의 입력이 됩니다.

신규 패턴 감지

"지금까지 없던 템플릿의 첫 등장" 자체가 강한 이상 신호입니다. 배포 직후 새 에러 패턴을 즉시 포착할 수 있습니다.

3.2 원리: 고정 깊이 파스 트리 (Fixed-Depth Parse Tree)

Drain(2017, He et al.)의 핵심 아이디어는 "비슷한 로그를 빨리 찾기 위해 트리를 이정표로 쓰자"입니다. 새 로그가 도착하면 전체 템플릿과 일일이 비교하는 대신, 트리를 따라 내려가 후보 그룹만 빠르게 좁힙니다. 트리의 각 층은 다음 기준으로 분기합니다.

층분기 기준직관
Root → L1토큰 개수 (로그 길이)단어 수가 다르면 같은 print 문일 가능성이 낮다
L1 → L2...앞쪽 토큰들 (기본: 첫 토큰)로그의 첫 단어(모듈명, 동사)는 대개 상수다. 숫자가 섞인 토큰은 <*>로 취급해 분기 폭발 방지
Leaf유사도 비교리프에 모인 소수의 템플릿 후보와만 토큰 단위 유사도(simTh, 기본 0.4)를 계산

리프에서의 판정은 단순합니다. 같은 위치의 토큰이 일치하는 비율 sim = 일치 토큰 수 / 전체 토큰 수가 임계값(sim_th) 이상이면 기존 클러스터에 병합하고, 불일치 위치의 토큰을 <*>로 바꿔 템플릿을 갱신합니다. 임계값 미만이면 새 클러스터를 만듭니다. 트리 덕분에 로그 1건 처리 비용이 전체 클러스터 수와 무관하게 거의 상수 시간(O(depth + 리프 내 후보 수))이라, 스트리밍 환경에서도 CPU 한 코어로 수십만 line/s를 처리합니다.

Drain vs Drain3 — 이름이 비슷한데 뭐가 다른가

Drain은 2017년 논문의 알고리즘 이름이고, Drain3는 IBM이 이를 프로덕션용 Python 3 라이브러리로 재구현·확장한 오픈소스입니다. Drain3가 추가한 것: ① 마스킹 전처리(IP·숫자·경로를 정규식으로 미리 <IP> 등으로 치환), ② 스냅샷 영속화(Kafka/Redis/파일에 상태 저장 → 재시작해도 학습한 템플릿 유지), ③ 템플릿 변경 콜백, 캐시 크기 제한(LRU), 유사도 파라미터 튜닝 등 운영 기능입니다. 실무에서는 사실상 Drain3만 사용합니다.

Section 04 — Stage 2 · Unsupervised Anomaly Detection

Isolation Forest — "다른 놈은 빨리 고립된다"

라벨이 한 장도 없는 시스템 오픈 첫날부터 동작해야 하는 것이 이상 탐지입니다. Isolation Forest(iForest, Liu et al. 2008)는 "정상"을 모델링하는 대신 "이상한 점은 무작위 질문 몇 번이면 혼자 남는다"는 발상의 전환으로, 비지도 이상 탐지의 기본기가 되었습니다.

4.1 직관: 스무고개로 사람 찾기

운동장에 100명이 모여 있고, 99명은 가운데에 뭉쳐 있고 1명만 구석에 서 있다고 합시다. "동쪽 절반에 있나요?", "북쪽 절반에 있나요?"처럼 무작위로 공간을 반씩 잘라가며 특정 인물을 고립시키는 게임을 한다면 — 구석의 1명은 두세 번의 질문이면 혼자 남지만, 무리 한가운데의 사람은 수십 번을 잘라야 합니다. iForest는 이 직관을 그대로 알고리즘으로 만든 것입니다.

기존 접근 (밀도/거리 기반)

"정상 데이터의 분포를 먼저 배우고, 그 분포에서 먼 점을 이상으로 판정" — 정상을 모델링하는 비용이 크고, 고차원에서 거리 개념이 무너집니다(차원의 저주).

iForest의 역발상

"정상은 몰라도 된다. 이상만 직접 노린다" — 이상치는 ① 수가 적고(few) ② 값이 다르다(different)는 성질 덕에 무작위 분할로 빨리 고립됩니다. 고립 속도 자체가 점수가 됩니다.

주황 선은 무작위 축·무작위 위치의 분할입니다. 선택한 점(테두리 강조)이 혼자 남을 때까지의 분할 횟수가 경로 길이 h(x)입니다. 구석의 이상치는 평균 2~5회, 군집 중심은 8회 이상 걸리는 것을 확인해 보세요. 점수 s(x)가 1에 가까울수록 이상, 0.5 근처면 평범입니다.

4.3 원리를 조금 더 정확히

학습은 놀랄 만큼 단순합니다. 데이터에서 소량(기본 256개)을 서브샘플링해, 무작위 피처·무작위 분할값으로 트리를 만들고(점이 전부 고립되거나 최대 깊이 도달까지), 이를 100그루(기본) 반복해 숲을 만듭니다. 예측 시에는 각 트리에서 해당 점이 도달한 리프까지의 경로 길이 h(x)를 평균 내고, 다음처럼 0~1 점수로 정규화합니다.

s(x, n) = 2−E[h(x)] / c(n)   ·   c(n) = 표본 n개 이진탐색트리의 평균 경로 길이(정규화 상수)
s → 1 : 경로가 매우 짧음 = 뚜렷한 이상  ·  s ≈ 0.5 : 평균적 = 정상  ·  s → 0 : 오히려 고립이 어려움 = 밀집 중심부
왜 서브샘플 256개?

표본이 작을수록 이상치끼리 뭉치는 masking, 정상이 이상을 가리는 swamping이 줄어듭니다. "적게 볼수록 이상이 더 잘 보인다"는 iForest의 반직관적 매력 포인트입니다. 덕분에 학습이 극도로 가볍습니다(선형 시간, 저메모리).

차원의 저주에 강함

거리 계산이 아예 없습니다. 축 하나씩 자르기 때문에 피처 수십~수백 개에서도 동작합니다. 단, 무의미한 피처가 많으면 성능이 희석되므로 피처 선택은 여전히 중요합니다.

한계도 분명

축에 평행한 분할만 하므로 대각선 경계·국소 밀도 차이에 약하고(→ Extended iForest, LOF 참고), 범주형 피처를 직접 못 먹으며, "왜 이상인가"의 설명력이 약합니다. 그래서 3단계 CatBoost가 필요해집니다.

CatBoost — 운영자의 판단을 학습하는 심사관

iForest가 "평소와 다르다"고 올린 후보에는 배포 이벤트, 배치 작업, 트래픽 급증 같은 무해한 이상이 잔뜩 섞여 있습니다. CatBoost는 운영자가 매긴 "진짜 장애 / 무해 이벤트" 라벨을 학습해 이 마지막 오탐을 걸러내는 지도학습 분류기입니다. 이름 그대로 Categorical 피처 처리에 특화된 Gradient Boosting 라이브러리(Yandex, 2017)입니다.

5.1 배경 지식: 그래디언트 부스팅 3분 요약

부스팅은 "약한 모델을 순서대로 쌓되, 각 모델이 이전까지의 오답(잔차)을 보정"하는 앙상블 기법입니다. 시험 채점에 비유하면: 1번 채점자가 대충 점수를 매기고 → 2번 채점자는 1번이 틀린 만큼만 교정하고 → 3번은 그 나머지를 교정하고… 를 수백 번 반복해 최종 점수는 모두의 교정치 합이 됩니다. 결정트리를 약한 모델로 쓰는 것이 GBDT(Gradient Boosted Decision Trees)이며, 정형 데이터(테이블 데이터)에서는 여전히 딥러닝보다 GBDT가 기본 선택지입니다. 3대 구현체가 XGBoost, LightGBM, CatBoost입니다.

5.2 CatBoost만의 두 가지 발명

발명 ① Ordered Target Statistics — 범주형 피처의 안전한 수치화

AIOps 데이터의 핵심 피처는 대부분 범주형입니다: 호스트명, 서비스명, 에러코드, 템플릿ID… 원-핫 인코딩은 카디널리티가 수천이면 폭발하고, 흔히 쓰는 타깃 인코딩(카테고리별 타깃 평균값으로 치환)은 치명적 함정이 있습니다. 자기 자신의 정답을 인코딩에 쓰는 타깃 누수(target leakage)가 생겨, 학습 점수는 좋은데 운영에선 무너지는 모델이 됩니다.

CatBoost의 해법은 시간의 화살을 도입하는 것입니다. 데이터를 무작위로 줄 세운 뒤, 각 행의 인코딩 값은 "그 행보다 앞에 있는 행들"의 통계만으로 계산합니다. 미래(자기 자신 포함)를 훔쳐볼 수 없게 만든 것이죠. 아래 데모로 차이를 직접 확인해 보세요.

INTERACTIVE타깃 인코딩 누수 실험 — Ordered TS는 무엇이 다른가
#host_type (범주형)y (장애=1)인코딩 값계산 근거
Ordered TS: enc(i) = ( Σj<i, 같은 카테고리 yj + a·P ) / ( countj<i + a )   — a=1(prior 강도), P=전체 평균(prior)
방식을 "일반 타깃 인코딩(누수)"으로 전환해 보세요. 각 행의 인코딩에 자기 자신의 y가 포함되어 값이 정답을 노골적으로 흉내 내는 것을 볼 수 있습니다(빨간 값). 표본이 적은 카테고리일수록 누수가 심해집니다. Ordered TS는 같은 데이터에서 미래 정보를 차단합니다.

발명 ② Ordered Boosting + 대칭 트리(Oblivious Tree)

Ordered Boosting

전통 부스팅은 잔차를 계산할 때도 같은 데이터를 재사용해 미세한 낙관 편향(prediction shift)이 누적됩니다. CatBoost는 여기에도 "앞 데이터로만 뒤 데이터의 잔차를 추정"하는 순서 원칙을 적용해, 작은 데이터셋에서 특히 과적합에 강한 특성을 얻었습니다. 운영자 라벨이 수백~수천 건뿐인 AIOps 초기에 이 성질이 대단히 유용합니다.

대칭 트리

트리의 같은 깊이에서는 모든 노드가 동일한 분할 조건을 사용합니다. 트리 하나가 사실상 "질문 k개짜리 체크리스트"가 되어, 추론이 비트 연산 수준으로 단순해집니다(→ 추론 속도 매우 빠름, 결과 재현성·안정성 향상). 정규화 효과도 있어 기본값만으로도 잘 동작하는 이유 중 하나입니다.

주의: 필터 파이프라인의 3대 리스크

① 침묵 장애(silent failure) — CatBoost가 진짜 장애를 "무해"로 오판하면 알람 자체가 사라집니다. 반드시 안전밸브를 두세요: iForest 점수 극단값(상위 0.01%)은 CatBoost 판정과 무관하게 무조건 통보, 그리고 "무해 판정" 건도 대시보드에는 남겨 감사 가능하게. ② 라벨 편향 — 운영자가 바쁠 때 대충 판정한 라벨이 모델을 오염시킵니다. "보류" 옵션을 제공하고 보류는 학습에서 제외하세요. ③ 피드백 루프 고착 — 필터가 특정 유형을 계속 걸러내면 그 유형의 라벨이 더는 쌓이지 않습니다. 걸러진 건의 일부(예: 5%)를 무작위 샘플링해 운영자에게 노출하는 탐사(exploration) 트래픽을 유지하세요.

# iforest-scorer: features.window 소비 → 임계 초과분만 anomaly.candidates 발행
import joblib, orjson, numpy as np

bundle = joblib.load("/models/iforest/current/model.joblib")
clf, thr, cols = bundle["model"], bundle["threshold_q999"], bundle["feature_columns"]

def handle(window: dict):
    x = np.array([[window["features"][c] for c in cols]])
    score = float(-clf.score_samples(x)[0])
    if score > thr:
        producer.produce("anomaly.candidates", value=orjson.dumps({
            **window, "iforest_score": score}))
    # 안전밸브: 극단값은 CatBoost와 무관하게 즉시 통보
    if score > thr * 1.8:

Section 09 — Wrap-up

정리: 기억해야 할 것

Drain3

로그를 템플릿 ID + 파라미터로 접는 온라인 파서. 고정 깊이 파스 트리로 상수 시간 처리. 마스킹 설계가 튜닝의 80%, 상태 영속화(Redis/Kafka)는 필수. 이후 모든 단계의 피처 원천.

Isolation Forest

"이상은 빨리 고립된다." 라벨 없이, 거리 계산 없이, 서브샘플 256개로 가볍게 동작하는 비지도 탐지의 기본값. predict()보다 score_samples() + 분위수 임계로 운용. 오탐은 숙명 — 다음 단계가 있는 이유.

CatBoost

Ordered TS로 타깃 누수 없이 범주형을 네이티브 처리하고, Ordered Boosting으로 소량 라벨에서도 과적합에 강한 GBDT. 운영자 피드백을 학습해 오탐을 걸러내고, SHAP 근거를 LLM에 물려준다.

마지막 한 문장

AIOps에서 LLM은 마지막 1%를 빛나게 하는 마무리이고, 앞의 99%를 접어내는 것은 오늘 배운 세 개의 "작고 오래된" 모델입니다. 파이프라인의 지능은 모델 크기가 아니라 단계 설계에서 나옵니다.
