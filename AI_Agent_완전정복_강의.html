01Chapter One

AI Agent는 스스로 일을 끝내는 AI입니다

"오늘 서울 날씨 어때?"라는 질문에 답만 해 주면 LLM, "주말 비 오면 우산 사 두라고 알려줘"를 듣고 날씨를 확인하고 캘린더에 알림까지 등록해 주면 AI Agent입니다.

한 줄 정의

AI Agent는 사용자의 목표(Goal)를 받아서, 스스로 계획을 세우고, 도구를 사용하며, 여러 단계를 거쳐 일을 끝내는 AI 시스템입니다. 핵심은 "한 번 답하고 끝"이 아니라 여러 차례 생각·행동을 반복한다는 점입니다.

쉽게 비유하면

LLM은 똑똑한 도서관 사서 — 질문하면 정확한 답을 빠르게 알려줍니다.
AI Agent는 똑똑한 비서 — "이번 주 출장 준비해 줘"라고 하면 항공편 검색, 호텔 예약, 일정 추가까지 알아서 처리합니다.

Agent의 네 가지 구성 요소

모든 AI Agent는 다음 네 가지 부품으로 이루어져 있습니다. 이 네 가지가 어떻게 조립되느냐에 따라 Agent의 성능과 성격이 달라집니다.

① Perception · 인지

사용자의 입력, 외부 환경(웹페이지, API 응답, 파일)을 읽어 들이는 능력입니다. 텍스트뿐 아니라 이미지·음성·표·코드까지 받아들일 수 있다면 멀티모달 에이전트가 됩니다.

② Memory · 기억

대화 흐름을 기억하는 단기 기억(Short-term)과 사용자 정보·과거 경험을 저장하는 장기 기억(Long-term, 보통 Vector DB)으로 나뉩니다. 기억이 없으면 매 턴마다 처음부터 다시 설명해야 합니다.

③ Planning · 계획

목표를 작은 단계로 쪼개는 능력입니다. "출장 준비"는 ⓐ 항공편 검색 → ⓑ 호텔 검색 → ⓒ 캘린더 등록 같은 하위 작업으로 분해됩니다. 이 부분에서 ReAct, Plan-and-Execute, Tree of Thoughts 같은 패턴이 등장합니다.

④ Tool Use · 도구 사용

웹 검색, 계산기, DB 쿼리, 코드 실행, 외부 API 호출처럼 LLM 외부의 능력을 빌려 쓰는 것입니다. 도구가 없으면 Agent는 자기 머릿속만 뒤지지만, 도구가 있으면 실제 세계와 연결됩니다.

ReAct 루프 — Agent가 일하는 방식

가장 널리 쓰이는 패턴이 ReAct (Reasoning + Acting)입니다. Agent는 다음 사이클을 목표 달성까지 반복합니다.

FIGURE 1.1 — REACT LOOP: THOUGHT → ACTION → OBSERVATION → REFLECT

이 사이클이 1~2번으로 끝날 수도 있고, 복잡한 작업이면 수십 번 반복될 수도 있습니다. 그래서 Agent는 본질적으로 "여러 번 LLM을 호출하는 시스템"이라고 이해해도 됩니다.

개발자 노트

실무에서는 LangChain·LangGraph·LlamaIndex·CrewAI·AutoGen 같은 프레임워크가 이 루프를 추상화해 줍니다. 하지만 "무엇을 호출하고, 결과를 어떻게 다음 호출에 넘기는가"가 본질이라는 점만 기억하면 됩니다.

02Chapter Two

LLM과 Agent, 무엇이 다른가

많은 분들이 헷갈리는 부분입니다. 차이를 한 마디로 정리하면: "LLM은 한 번 부르면 한 번 답한다. Agent는 한 번 부르면 알아서 여러 번 일한다."

03Chapter Three

Agent를 평가하는 여덟 가지 지표

Agent는 "답이 맞았나"만 보면 안 됩니다. 어떻게 답에 도달했는지, 얼마나 비싼지, 얼마나 안전한지까지 함께 봐야 합니다. 각 카드를 눌러 자세히 살펴보세요.

왜 LLM 평가 지표로는 부족한가

LLM은 BLEU·ROUGE·Perplexity·정확도 같은 출력 텍스트 중심 지표로 평가합니다. 그런데 Agent는 여러 단계의 행동으로 일을 끝내기 때문에, 출력만 봐서는 "이번엔 운이 좋았던 건지, 실력으로 푼 건지" 구별이 어렵습니다. 그래서 Agent 전용 지표가 필요합니다.

01
Task Success Rate (TSR)작업 성공률 — 가장 기본이 되는 지표
Outcome
+

무엇을 측정하나? Agent에게 N개의 작업을 줬을 때, 최종 목표를 달성한 비율을 측정합니다. 중간에 몇 번 헛돌든, "결국 일을 끝냈는가"가 기준입니다.

TSR = 성공한 task 수 / 전체 task 수 × 100 (%)
예시
"이 CSV에서 매출 상위 5개 지점을 찾아 막대그래프로 그려라" 100개 작업 → 87개 정상 완료 → TSR = 87%

주의점: "성공"의 정의를 명확히 해야 합니다. 그래프가 그려졌어도 데이터가 틀렸으면 실패로 봐야 합니다.

02
Tool Use Accuracy도구 선택·사용 정확도
Process
+

무엇을 측정하나? Agent가 적절한 도구를 골랐는지, 인자(파라미터)를 맞게 넘겼는지를 측정합니다. 보통 두 가지를 따로 봅니다.

Tool Selection Acc = 올바른 도구 호출 수 / 전체 도구 호출 수 Tool Arg Acc = 인자가 맞은 호출 수 / 전체 도구 호출 수
예시
"내일 서울 날씨" 질문에 web_search() 대신 weather_api(city="Seoul")를 골랐다면 도구 선택 정답 — 거기에 city="seoul" 같이 인자도 정확히 넘겼는지를 별도로 봅니다.

중요한 이유: 도구 선택이 틀리면 그 뒤 모든 단계가 헛돕니다. Agent의 "초반 실수"를 잡아내는 핵심 지표입니다.

03
Trajectory Quality경로 품질 — 어떻게 답에 도달했나
Process
+

무엇을 측정하나? Agent가 거쳐 간 모든 단계의 순서가 합리적이었는지를 봅니다. 결과만 보면 "운으로 맞춘 답"과 "제대로 추론한 답"을 구별할 수 없습니다.

보통 정답 경로(reference trajectory)를 사람이 정의해 두고, Agent의 실제 경로와 비교합니다.

Trajectory Score = f( step_match, order_match, redundancy ) · step_match : 필수 단계를 모두 거쳤는가? · order_match : 순서가 맞았는가? · redundancy : 불필요한 단계는 없었는가?
예시
"주가 분석" 정답 경로: ⓐ 종목 검색 → ⓑ 가격 데이터 조회 → ⓒ 지표 계산 → ⓓ 차트 생성
Agent가 ⓐ→ⓑ→ⓒ→ⓓ로 갔다면 만점. ⓐ→ⓓ로 건너뛰었다면 step_match 감점.
04
Plan Quality계획의 합리성
Process
+

무엇을 측정하나? Agent가 처음에 세운 계획(plan)이 얼마나 논리적이고 완전한지를 봅니다. Plan-and-Execute 패턴을 쓰는 Agent에서 특히 중요합니다.

주요 평가 항목은:

· 완전성(Completeness) — 목표 달성에 필요한 단계를 다 포함했나?
· 정확성(Correctness) — 각 단계가 실현 가능하고 올바른가?
· 효율성(Efficiency) — 불필요한 단계 없이 최단 경로인가?
· 의존성(Dependency) — 단계 간 순서·의존 관계가 맞나?

평가 방법
보통 LLM-as-Judge(GPT-4·Claude 등 강한 모델로 평가)로 1~5점 척도 채점하거나, 사람이 직접 채점합니다.
05
Latency & Cost응답 시간과 토큰 비용
Operational
+

무엇을 측정하나? 정답을 내도 너무 느리거나 너무 비싸면 실서비스에 못 씁니다. 운영 관점에서 가장 중요한 지표입니다.

Total Latency = Σ ( LLM_call_time + tool_exec_time + network_overhead ) Total Cost = Σ ( input_tokens × in_price + output_tokens × out_price ) + Σ ( tool_call_cost )
실무 기준
챗봇용 Agent: p95 latency < 8초, 건당 $0.05 이하 목표.
백그라운드 작업(리서치·분석)은 분 단위도 허용되지만, 비용 모니터링이 필수.

병목 분석: "어느 단계가 가장 느린가?", "어떤 도구 호출이 비용을 많이 잡아먹나?" 를 Trace로 확인해야 최적화가 가능합니다.

06
Hallucination Rate환각 발생률
Quality
+

무엇을 측정하나? Agent가 사실이 아닌 정보를 만들어내는 비율입니다. 도구 사용 결과를 무시하고 "지어낸" 답변, 존재하지 않는 함수·API를 호출하는 경우, 출처 없는 주장을 포함하면 환각으로 봅니다.

Hallucination Rate = 환각 포함 응답 수 / 전체 응답 수

측정 방법:

· Retrieval-grounded check — 답변의 각 주장을 도구 결과와 대조
· Fact-checking 모델 — 별도 LLM이 사실 여부 판정
· 사람 검수 — 도메인 전문가가 표본 검토

예시
웹 검색 도구가 "2025년 매출 1000억"을 가져왔는데 Agent가 "1500억"이라고 답하면 환각.
07
Safety & Alignment안전성·정렬성
Risk
+

무엇을 측정하나? Agent가 해서는 안 되는 행동을 거부하는지, 사용자 의도와 정렬되어 있는지를 봅니다. 단순 응답이 아니라 실제 행동(이메일 발송, DB 수정 등)이 가능하기 때문에 LLM보다 훨씬 중요해집니다.

주요 항목:

· 권한 준수 — 허용되지 않은 도구·자원을 호출하지 않는가?
· 데이터 유출 방지 — 개인정보·기밀을 외부로 보내지 않는가?
· Prompt Injection 저항 — 외부 입력에 의해 지시가 뒤집히지 않는가?
· Refusal 비율 — 위험 요청을 적절히 거절하는가? (단, 과도한 거절도 감점)

평가 도구
NeMo Guardrails, Garak, PyRIT 등 red-teaming 프레임워크로 적대적 입력을 자동 생성해 측정.
08
User Satisfaction (CSAT)사용자 만족도
Outcome
+

무엇을 측정하나? 결국 최종 사용자가 만족했는가입니다. 자동 지표가 다 좋아도 사용자가 떠나면 실패한 시스템입니다.

측정 방법:

· 명시적 — 👍/👎 버튼, 1~5점 별점, 설문
· 암묵적 — 재사용률, 세션 시간, 작업 완료율, 이탈률
· 정성적 — 자유 코멘트 분석(토픽 모델링)

CSAT = 긍정 응답 수 / 전체 응답 수 × 100 (%) NPS = % 추천자(9-10점) − % 비추천자(0-6점)

주의: 자동 지표(TSR 등)와 사용자 만족도는 꼭 일치하지 않습니다. 둘 다 봐야 합니다.

04Chapter Four

실제로 어떻게 평가하는가

지표가 무엇인지 알았으니 이제 "어떻게 측정하는가"입니다. 평가 방법은 크게 네 가지 — 벤치마크, LLM-as-Judge, 휴먼 평가, 운영 모니터링으로 나뉩니다.

① 표준 벤치마크(Benchmark) 활용

이미 잘 만들어진 데이터셋과 평가 환경을 가져다 쓰는 방법입니다. 학계·업계가 인정하는 객관적 비교가 가능합니다.

② LLM-as-Judge — AI가 AI를 채점하기

강력한 LLM(GPT-4, Claude Opus 등)을 평가자로 활용해 다른 Agent의 출력을 채점하게 하는 방법입니다. 사람이 직접 보는 것보다 훨씬 빠르고 저렴합니다.

전형적인 프로세스

1. 평가하려는 Agent로 100~1000개 작업 실행 → trajectory 수집
2. Judge LLM에게 평가 기준(rubric)과 trajectory를 함께 제공
3. Judge가 각 항목(plan quality, tool use, final answer)에 점수·근거 출력
4. 결과를 집계해 평균 점수·분포 분석

함정 주의

Judge LLM도 편향이 있습니다. ① 자기 모델 편향(GPT-4가 GPT-4 출력을 더 좋게 평가), ② 길이 편향(긴 답을 더 좋게), ③ 위치 편향(먼저 본 답을 더 좋게)이 대표적입니다. 평가 프롬프트 설계와 휴먼 평가 교차 검증이 필요합니다.

③ 휴먼 평가 — 사람이 직접 보기

가장 비싸지만 가장 신뢰할 수 있는 방법입니다. 도메인 전문가가 trajectory를 직접 보고 채점합니다.

· Pairwise Comparison — A/B 두 Agent 결과를 나란히 보여주고 어느 쪽이 더 좋은지 선택
· Likert Scale — 각 항목 1~5점 척도 평가
· Error Analysis — 실패 케이스를 모아서 원인 유형별 분류

실무 팁: 모든 케이스를 휴먼 평가하기엔 비싸므로, 중요·민감 케이스 또는 무작위 샘플만 휴먼 평가하고 나머지는 LLM-as-Judge로 자동화하는 하이브리드 방식이 일반적입니다.

④ 운영 단계 모니터링 — 실서비스 평가

벤치마크는 출시 전 평가, 운영 모니터링은 출시 후 평가입니다. 실제 사용자 트래픽에서 지속적으로 측정해야 진짜 품질이 보입니다.

핵심 모니터링 항목

· Trace 수집 — 모든 단계(LLM 호출·도구 호출·중간 결과)를 기록 (LangSmith, Langfuse, Phoenix, Arize)
· 실시간 지표 — TSR, latency, error rate, cost를 시계열로 트래킹
· 회귀 탐지 — 모델·프롬프트 변경 시 성능 하락 자동 감지
· 피드백 루프 — 사용자 👎 응답을 수집해 fine-tuning 또는 프롬프트 개선에 활용
· A/B 테스트 — 새 버전을 10% 트래픽에 먼저 노출해 비교

전체 그림

이상적인 평가는 "벤치마크로 기본 능력 확인 → LLM-as-Judge로 대량 평가 → 휴먼 평가로 신뢰성 검증 → 운영 모니터링으로 지속 추적" 의 네 단계를 모두 거치는 것입니다. 다만 모든 팀이 4단계를 다 할 수는 없으니, 위험도·중요도에 따라 깊이를 조절하면 됩니다.

Agent 도입 시 체크리스트

모든 작업에 Agent가 답은 아닙니다. 도입 전에 다음 5가지를 자문해 보세요.

5-Question Check

① 다단계인가?   한 번의 LLM 호출로 끝나면 굳이 Agent가 아니어도 됩니다.
② 외부 도구가 필요한가?   모델 내부 지식만으로 풀린다면 RAG가 더 간단합니다.
③ 실패 비용은 감당 가능한가?   잘못된 행동이 큰 손실로 이어진다면 휴먼-인-더-루프 설계가 필수.
④ 평가 체계는 준비되었나?   측정하지 못하면 개선할 수 없습니다.
⑤ 비용·지연을 견딜 수 있나?   응답이 분 단위여도 괜찮은 시나리오인가?

앞으로의 방향

2026년 현재 Agent 분야는 다음 방향으로 빠르게 움직이고 있습니다.

· 더 긴 컨텍스트와 더 깊은 메모리 — 며칠·몇 주 단위 작업을 끊김 없이 수행
· Computer Use — 화면 보고 마우스·키보드 조작하는 범용 Agent
· On-device Agent — 프라이버시·저지연을 위한 로컬 실행 (소형 모델 + 도구)
· Multi-Agent Orchestration — 수십~수백 개 Agent의 협업·합의 알고리즘
· Verifiable Agent — 모든 행동에 감사 추적·근거 첨부가 표준화
