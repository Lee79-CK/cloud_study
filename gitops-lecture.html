Git을 진실의 원천으로.
이것이 GitOps다.

서버에 직접 접속해서 명령어를 치던 시대는 끝났습니다. GitOps는 "시스템이 어떤 상태여야 하는가"를 Git 저장소에 적어두고, 나머지는 자동화에 맡기는 운영 방식입니다. DevOps와 무엇이 다른지, 어떻게 시작하는지, 한 걸음씩 짚어 봅니다.

01 개념

GitOps란 무엇인가

GitOps는 두 단어의 합성어입니다. Git(버전 관리 시스템)과 Ops(운영, Operations). 풀어 쓰면 이렇게 됩니다.

시스템의 원하는 상태(desired state)를 전부 Git에 선언해 두고, 실제 시스템이 항상 그 선언과 똑같아지도록 자동화 도구가 계속 맞춰주는 운영 방식.

핵심 문장 하나만 기억하세요. "Git에 적힌 것이 정답이고, 실제 환경은 그 정답을 따라간다." Git 저장소가 곧 시스템의 설계도이자 명령서이며, 무엇이 배포되어 있어야 하는지에 대한 단 하나의 진실(single source of truth)입니다.

요리 레시피로 이해하기

주방에서 매번 즉흥적으로 요리하면, 같은 음식도 만들 때마다 맛이 달라집니다. 누가 만들었는지, 그날 기분이 어땠는지에 따라 결과가 흔들리죠. 대신 정확한 레시피(분량·순서·온도)를 문서로 고정해 두면, 누가 따라 해도 같은 음식이 나옵니다.

GitOps에서 Git 저장소가 바로 이 레시피입니다. 그리고 주방장 대신 자동화 도구가 레시피를 보고, 실제 음식(=운영 중인 시스템)이 레시피와 다르면 즉시 다시 맞춰줍니다.

한 줄로 요약하면

  • 무엇을 — 배포·인프라 설정 같은 운영 작업을
  • 어디에 — Git 저장소에 코드(주로 YAML)로 적어두고
  • 어떻게 — 사람이 서버를 직접 만지는 대신, 자동화 도구가 Git을 보고 시스템을 똑같이 만든다
핵심 직관

GitOps는 새로운 마법이 아닙니다. 우리가 이미 애플리케이션 소스 코드를 Git으로 관리하던 습관을, 인프라와 배포 작업까지 똑같이 확장한 것입니다. "코드처럼 다루자"는 생각이 전부입니다.

02 동기

왜 필요한가 — 기존 방식의 문제

GitOps가 왜 등장했는지 알려면, 그 전에 사람들이 어떻게 운영했는지를 봐야 합니다. 전통적인 방식에는 몇 가지 고질적인 문제가 있었습니다.

① 손으로 만지는 운영 (수동 배포)

예전에는 서버에 직접 접속(ssh)해서 명령어를 입력하고, 설정 파일을 손으로 고쳤습니다. 문제는 "누가, 언제, 무엇을, 왜 바꿨는지"가 기록되지 않는다는 점입니다. 새벽에 급하게 고친 설정 하나가 다음 날 장애의 원인이 되어도 추적할 수 없습니다.

② 설정 드리프트 (Configuration Drift)

가장 무서운 단어입니다. 드리프트란 "문서나 설계도에 적힌 상태"와 "실제 운영 중인 상태"가 조금씩 어긋나는 현상을 말합니다.

'드리프트'를 집 청소로 이해하기

처음 이사 왔을 때 집 구조도(설계도)와 실제 집이 똑같았습니다. 그런데 살다 보니 가구를 옮기고, 선반을 달고, 임시로 박스를 쌓습니다. 어느 순간 설계도와 실제 집이 전혀 다른 모습이 됩니다. 누군가 "설계도대로 복구해줘"라고 해도 불가능하죠. 무엇이 바뀌었는지 아무도 모르니까요.

운영 환경도 똑같습니다. 여러 사람이 급하게 손대다 보면 실제 서버가 원래 의도와 멀어집니다. 그리고 그 서버가 죽으면, 똑같이 복구할 방법이 없습니다.

③ 재현 불가능 (Snowflake Server)

손으로 조금씩 다르게 설정한 서버들은 눈송이처럼 전부 모양이 다릅니다(스노우플레이크 서버). 한 대가 장애 나도 똑같은 서버를 다시 만들 수 없어서, 복구에 며칠이 걸리기도 합니다.

전통적 운영의 문제GitOps가 주는 해답
변경 이력이 남지 않음모든 변경이 Git 커밋으로 기록됨
누가 무엇을 바꿨는지 모름커밋 author·메시지·PR로 명확
실제 상태가 설계도와 어긋남(드리프트)자동으로 감지하고 원래대로 복구
장애 시 복구에 오랜 시간Git을 이전 커밋으로 되돌리면 끝
서버마다 설정이 제각각같은 선언 → 항상 같은 결과

요약하면, GitOps는 운영을 "기억과 손기술"에서 "기록과 자동화"로 옮기는 일입니다.

03 관계

DevOps와 GitOps의 관계

가장 많이 헷갈리는 부분입니다. 결론부터 말하면 둘은 경쟁 관계가 아닙니다. GitOps는 DevOps를 실현하는 구체적인 한 가지 방법입니다.

DevOps는 '문화이자 철학'

DevOps는 개발(Development)과 운영(Operations) 팀의 벽을 허물어, 빠르고 안정적으로 소프트웨어를 전달하자는 철학·문화·일하는 방식입니다. "자동화하자", "협업하자", "빠르게 자주 배포하자" 같은 지향점을 말하지만, "구체적으로 어떻게?"에 대한 정답을 딱 정해주지는 않습니다.

GitOps는 그 철학을 이루는 '구체적 실천법'

DevOps가 "자동화로 빠르고 안정적으로 배포하자"라고 말한다면, GitOps는 "그럼 Git을 중심으로 이렇게 하자"라는 구체적인 실행 레시피를 제시합니다.

'건강하게 살자' vs '주 3회 달리기'

DevOps는 "건강하게 살자"라는 큰 목표(철학)입니다. 좋은 말이지만 추상적이죠. GitOps는 "매주 3회, 5km를 달리고 식단을 기록하자"라는 구체적인 실천 계획입니다. 달리기는 건강해지는 여러 방법 중 하나이듯, GitOps도 DevOps를 실현하는 여러 방법 중 하나입니다.

구분DevOpsGitOps
성격문화 · 철학 · 일하는 방식구체적인 운영 기법(방법론)
범위조직·협업·전체 파이프라인배포·인프라 상태 관리에 집중
중심 도구특정하지 않음(다양)Git + 조정 도구(Argo CD·Flux)
핵심 질문"어떻게 더 잘 협업·전달할까?""상태를 어떻게 Git으로 관리할까?"
관계GitOps ⊂ DevOps — GitOps는 DevOps를 구현하는 한 방식

CI/CD와는 어떻게 연결될까?

DevOps의 대표 실천이 CI/CD 파이프라인입니다. 여기서 역할을 나눠보면 GitOps의 위치가 또렷해집니다.

  • CI(지속적 통합) — 코드를 빌드·테스트하고 컨테이너 이미지를 만들어 등록합니다. GitOps는 보통 이 영역엔 관여하지 않습니다.
  • CD(지속적 배포) — 만들어진 결과물을 실제 환경에 올립니다. GitOps는 바로 이 'CD' 영역을 Git 기반으로 새롭게 정의합니다.
한 문장 정리

CI는 "좋은 결과물을 만드는 일", GitOps는 "그 결과물을 Git을 진실의 원천으로 삼아 안전하게 배포·유지하는 일". 그래서 흔히 "GitOps = Git 중심의 새로운 CD"라고도 부릅니다.

04 원칙

GitOps 4대 원칙

GitOps에는 업계가 합의한 네 가지 핵심 원칙이 있습니다(OpenGitOps 기준). 이 네 가지를 만족해야 "진짜 GitOps"라고 부를 수 있습니다.

원칙 01

선언적이다 (Declarative)

"이렇게 해라(명령)"가 아니라 "이런 상태여야 한다(선언)"로 기술합니다. 과정이 아니라 결과를 적습니다.

원칙 02

버전 관리·불변 (Versioned)

모든 상태는 Git에 저장되어 버전·이력이 남고, 함부로 바뀌지 않으며 언제든 되돌릴 수 있습니다.

원칙 03

자동으로 가져온다 (Pulled)

자동화 에이전트가 Git의 변경을 스스로 감지해 끌어옵니다(pull). 사람이 밀어 넣지 않습니다.

원칙 04

지속적으로 맞춘다 (Reconciled)

에이전트가 끊임없이 실제 상태와 Git을 비교하고, 어긋나면 자동으로 다시 맞춥니다(self-heal).

가장 중요한 한 가지 — '선언적'이라는 말

입문자가 꼭 잡고 가야 할 개념입니다. 명령형(imperative)과 선언형(declarative)의 차이를 봅시다.

# "이 순서대로 직접 실행해라" — 무엇을 '하는지'를 적음
docker run -d --name web nginx:1.25
docker scale web=3
# 컨테이너가 죽으면? 사람이 다시 실행해야 함.
# 지금 몇 개 떠 있는지? 직접 확인해야 함.
# "최종 상태는 이래야 한다" — 무엇이 '되어야 하는지'를 적음
kind: Deployment
spec:
  replicas: 3            # nginx 3개가 항상 떠 있어야 한다
  template:
    spec:
      containers:
        - image: nginx:1.25
# 하나가 죽으면? 시스템이 알아서 3개로 다시 채운다.
선언형은 "어떻게"가 아니라 "무엇"만 적습니다. 그 "무엇"을 Git에 두는 것이 GitOps의 출발점입니다.

선언형이 GitOps에 필수인 이유는 명확합니다. 결과(상태)만 적혀 있으면, 자동화 도구가 "지금 상태"와 "원하는 상태"를 단순 비교해서 차이만 메우면 되기 때문입니다. 과정이 적혀 있으면 비교가 불가능합니다.

05 방식

Push 방식 vs Pull 방식

배포를 자동화하는 방식은 크게 둘로 나뉩니다. 누가 누구에게 변경을 전달하느냐의 차이이며, GitOps는 'Pull 방식'을 권장합니다. 탭을 눌러 비교해 보세요.

Git/CI 서버 ──▶ 바깥에서 클러스터로
밀어 넣음(push) ──▶ 운영 클러스터

외부의 CI 도구가 클러스터에 직접 명령을 보냅니다.

CI 파이프라인이 빌드를 끝낸 뒤, kubectl apply 같은 명령으로 운영 환경에 변경을 밀어 넣습니다(push).

단점: ① 외부 도구가 클러스터를 조작할 강력한 접근 권한(자격증명)을 가져야 해서 보안 위험이 큽니다. ② 배포 후 누군가 클러스터를 손대도(드리프트) 알아채지 못합니다. CI는 "한 번 밀어 넣고" 끝이니까요.

Git 저장소 ◀── 에이전트가 안에서
끌어옴(pull) ──▶ 운영 클러스터

클러스터 '안'의 에이전트가 Git을 직접 지켜봅니다.

클러스터 내부에 설치된 GitOps 에이전트(Argo CD·Flux)가 Git을 주기적으로 확인해, 변경을 스스로 끌어옵니다(pull).

장점: ① 외부에 클러스터 자격증명을 노출할 필요가 없어 훨씬 안전합니다. ② 에이전트가 항상 깨어 비교하므로, 누가 몰래 손대도(드리프트) 즉시 감지하고 되돌립니다. 이것이 GitOps가 pull을 선호하는 이유입니다.

기억하기

Push는 "택배기사가 집까지 와서 문을 따고 물건을 놓고 감"(권한이 필요), Pull은 "내가 직접 택배함을 확인하고 가져옴"(내가 통제). GitOps는 클러스터가 스스로 가져오는 방식을 택합니다.

06 핵심 원리

작동 원리 — 조정 루프

GitOps의 심장은 조정 루프(Reconciliation Loop)입니다. 에이전트가 쉬지 않고 돌리는 단순한 3단계 반복입니다.

  1. 관찰(Observe) — Git의 "원하는 상태"와 클러스터의 "실제 상태"를 읽는다.
  2. 비교(Diff) — 둘을 비교해 차이가 있는지 확인한다.
  3. 조정(Act) — 차이가 있으면 실제 상태를 Git에 맞게 고친다. 차이가 없으면 그대로 둔다.

초록 버튼 = Git을 바꾸면 클러스터가 따라옵니다(정상 배포). 빨강 버튼 = 클러스터만 몰래 바꾸면 에이전트가 감지해 원상복구합니다(self-heal).

방금 본 두 가지가 GitOps의 전부입니다.

  • 정상 배포 — 사람은 Git만 수정합니다. 클러스터는 절대 직접 만지지 않습니다. 에이전트가 차이를 보고 클러스터를 v2.0으로 올립니다.
  • 자가 치유(self-heal) — 누가 클러스터를 몰래 바꿔도, Git이 정답이므로 에이전트가 원래대로 되돌립니다. 드리프트가 자동으로 사라집니다.
그래서 황금률은

"운영 환경을 직접 만지지 마라. 항상 Git을 통해라." 클러스터에 직접 가한 변경은 어차피 다음 조정 루프에서 지워집니다. 바꾸고 싶으면 Git에 커밋(또는 PR)하세요.

07 도구

대표 도구 — Argo CD & Flux

조정 루프를 직접 만들 필요는 없습니다. 이미 검증된 오픈소스 에이전트가 있습니다. Kubernetes 생태계의 양대 도구는 Argo CD와 Flux입니다(둘 다 CNCF 졸업 프로젝트).

구분Argo CDFlux (CD)
한 줄 소개강력한 웹 UI 중심의 GitOps 도구가볍고 모듈형인 GitOps 도구
웹 대시보드풍부한 시각화 UI 기본 제공UI는 별도(예: Weave GitOps)
강점한눈에 보는 동기화 상태, 직관적경량, 쿠버네티스 친화적 구성
입문자 추천처음엔 보통 Argo CD가 쉬움(UI)CLI·자동화에 익숙하면 Flux

입문자는 Argo CD부터 시작하는 것을 추천합니다. 동기화 상태가 색으로 보이는 웹 화면 덕분에 "지금 무슨 일이 일어나는지"를 눈으로 이해하기 쉽기 때문입니다. Argo CD는 배포 단위를 Application이라는 객체로 관리합니다. 아래가 그 핵심 정의입니다.

apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: payment-api
  namespace: argocd
spec:
  project: default
  source:
    repoURL: https://github.com/my-org/k8s-manifests  # 정답이 있는 Git
    path: apps/payment-api     # 그 안에서 볼 폴더
    targetRevision: main      # 추적할 브랜치
  destination:
    server: https://kubernetes.default.svc  # 배포 대상 클러스터
    namespace: payment
  syncPolicy:
    automated:
      prune: true       # Git에서 지운 리소스는 클러스터에서도 삭제
      selfHeal: true    # 드리프트 발생 시 자동 복구 (앞 데모의 그것!)
apiVersion: kustomize.toolkit.fluxcd.io/v1
kind: Kustomization
metadata:
  name: payment-api
  namespace: flux-system
spec:
  interval: 5m            # 5분마다 Git을 확인(조정 루프 주기)
  path: ./apps/payment-api
  prune: true
  sourceRef:
    kind: GitRepository
    name: k8s-manifests   # 미리 등록한 Git 소스
두 도구 모두 "어느 Git의, 어느 경로를, 어느 클러스터에 맞춘다"를 선언할 뿐입니다. 조정은 에이전트가 알아서 합니다.
한번 더 음미할 점

위 selfHeal: true가 바로 6장 데모에서 본 자가 치유 스위치입니다. 한 줄 켜두면, 누가 운영 환경을 손대도 Git 상태로 자동 복구됩니다.

08 실습

실습 — 처음부터 시작하기

개념을 잡았으니 흐름을 따라가 봅시다. 실제 GitOps 워크플로는 놀랍도록 단순합니다.

  1. Git 저장소 준비 (매니페스트 분리)

    애플리케이션 소스 코드 저장소와 별도로, 배포 설정(YAML)만 모으는 저장소를 만듭니다. 흔히 app-repo(코드)와 config-repo(매니페스트)로 나눕니다.

  2. 클러스터에 GitOps 에이전트 설치

    Argo CD를 클러스터에 설치합니다. 이 에이전트가 앞으로 Git을 지켜보는 역할을 합니다.

  3. Application 등록 (Git ↔ 클러스터 연결)

    7장의 Application YAML을 적용해, "이 Git의 이 경로를 이 클러스터에 맞춰라"라고 선언합니다. 이 순간부터 조정 루프가 돕니다.

  4. 배포하기 = Git에 커밋하기

    이제 배포는 서버 접속이 아니라 Git 커밋/PR입니다. 이미지 태그를 v1.0 → v2.0으로 바꿔 커밋하면, 에이전트가 감지해 자동 배포합니다.

  5. 롤백하기 = Git 되돌리기

    문제가 생기면 git revert로 이전 커밋으로 되돌립니다. 에이전트가 그 상태로 클러스터를 다시 맞춥니다. 롤백이 곧 커밋입니다.

전형적인 배포 명령 흐름

# 1) Argo CD 설치
kubectl create namespace argocd
kubectl apply -n argocd -f \
  https://raw.githubusercontent.com/argoproj/argo-cd/stable/manifests/install.yaml

# 2) 위에서 만든 Application 선언 적용
kubectl apply -f payment-api-application.yaml
# 배포 = 매니페스트 저장소에서 이미지 태그만 바꿔 커밋
# deployment.yaml:  image: payment-api:v1.0  ->  v2.0
git add apps/payment-api/deployment.yaml
git commit -m "deploy payment-api v2.0"
git push

# 끝. 서버에 접속할 일이 없다.
# Argo CD가 변경을 감지하고 클러스터를 v2.0으로 동기화한다.
# 롤백 = 방금 커밋을 되돌리는 것이 전부
git revert HEAD
git push

# Argo CD가 다시 v1.0 상태로 클러스터를 맞춘다.
# "누가 무엇을 왜 되돌렸는지"가 Git 이력에 그대로 남는다.
배포·롤백이 모두 Git 작업으로 통일됩니다. 이것이 GitOps가 주는 가장 큰 실무적 편안함입니다.
09 응용

응용 — 실무 패턴

기초가 잡혔다면, 실무에서 GitOps가 빛을 발하는 대표 패턴들을 알아둘 차례입니다. 지금은 "이런 게 있구나" 정도로만 익혀도 충분합니다.

① 환경 분리 (dev / staging / prod)

각 환경을 폴더 또는 브랜치로 분리해 같은 앱을 여러 환경에 다르게 배포합니다. 보통 차이(레플리카 수·리소스·도메인)만 덮어쓰는 Kustomize의 오버레이를 씁니다.

# 매니페스트 저장소 구조 예시
k8s-manifests/
├── base/                 # 공통 정의(한 번만 작성)
│   └── deployment.yaml
└── overlays/
    ├── dev/              # 개발: replicas 1
    ├── staging/          # 스테이징: replicas 2
    └── prod/             # 운영: replicas 5, 더 큰 리소스
환경마다 다른 Application을 두어 각 폴더를 가리키게 합니다. 차이는 코드로 명시되고, 모두 Git에 남습니다.

② 점진적 배포 (Progressive Delivery)

새 버전을 한 번에 전부 바꾸지 않고 조금씩, 안전하게 내보내는 기법입니다. Argo Rollouts·Flagger 같은 도구를 GitOps와 결합합니다.

  • 카나리(Canary) — 새 버전을 트래픽의 5% → 25% → 100% 순으로 천천히 늘립니다. 문제가 보이면 자동 중단·롤백.
  • 블루-그린(Blue-Green) — 기존(blue)과 신규(green)를 동시에 띄워두고, 검증 후 트래픽을 한 번에 전환합니다.

③ 멀티 클러스터 / 멀티 환경 관리

클러스터가 수십 개로 늘어나도, 하나의 Git 저장소가 전부의 진실의 원천이 됩니다. Argo CD의 ApplicationSet을 쓰면 "이 앱을 모든 클러스터에 배포"를 한 번의 선언으로 처리합니다. 클러스터가 늘어도 운영 복잡도가 폭증하지 않는 것이 큰 장점입니다.

④ 비밀 정보(Secrets) 다루기

주의가 필요한 부분입니다. 비밀번호·토큰 같은 민감 정보를 Git에 평문으로 절대 올리면 안 됩니다. GitOps에서는 암호화해서 안전하게 다룹니다.

  • Sealed Secrets — 비밀 값을 공개키로 암호화해 Git에 올리고, 클러스터 안에서만 복호화합니다.
  • External Secrets — 실제 값은 Vault·클라우드 비밀 저장소에 두고, Git에는 "그 값을 참조하라"는 정보만 둡니다.
입문자가 가장 흔히 하는 실수

편하다고 secret.yaml에 비밀번호를 그대로 적어 커밋하는 것. Git 이력은 영원히 남기 때문에, 한 번 올라간 비밀은 사실상 유출된 것으로 봐야 합니다. 반드시 암호화 도구를 쓰세요.

⑤ 인프라까지 — GitOps for Infrastructure

GitOps는 애플리케이션 배포에만 쓰이지 않습니다. Terraform·Crossplane 등과 결합하면 클라우드 인프라(네트워크·DB·스토리지)까지 Git으로 선언해 관리할 수 있습니다. "코드형 인프라(IaC)"와 GitOps가 만나는 지점입니다.

10 균형 잡기

장점·한계·주의점

GitOps는 강력하지만 만병통치약은 아닙니다. 균형 있게 봐야 올바르게 도입할 수 있습니다.

분명한 장점

+

완벽한 감사 추적

모든 변경이 Git 커밋. 누가·언제·왜를 100% 추적할 수 있습니다.

+

빠르고 안전한 롤백

이전 커밋으로 되돌리면 끝. 검증된 안정 상태로 즉시 복귀.

+

드리프트 자동 차단

실제 상태가 어긋나면 자동 복구. 환경이 항상 의도대로.

+

높은 보안성

Pull 방식으로 외부에 클러스터 자격증명을 노출하지 않음.

+

재현 가능성

Git만 있으면 동일 환경을 어디서든 똑같이 재구축.

+

쉬운 협업

배포를 PR·코드 리뷰로. 익숙한 Git 워크플로 그대로.

알아둘 한계와 주의점

한계 / 함정대응
초기 학습 곡선이 있다작은 앱 하나로 먼저 연습하며 익힌다
주로 Kubernetes 중심 생태계K8s 밖에서도 가능하나 도구가 적다
비밀 정보 관리가 까다롭다Sealed/External Secrets를 반드시 사용
긴급 상황에 직접 손대고 싶은 유혹그래도 Git을 통한다(직접 변경은 곧 되돌려짐)
Git이 곧 단일 장애 지점이 될 수 있음저장소 백업·접근 권한·다중화 관리
언제 GitOps가 잘 맞나

Kubernetes를 쓰고, 배포가 잦고, 여러 명이 협업하며, 안정성과 추적성이 중요한 환경일수록 효과가 큽니다. 반대로 아주 단순한 1인 프로젝트라면 오히려 과할 수 있습니다. 도구는 문제에 맞게 고르는 것입니다.

11 점검

이해도 점검 퀴즈

지금까지 배운 내용을 확인해 봅시다. 정답을 누르면 해설이 나옵니다.

Q1
GitOps에서 "단일 진실의 원천(single source of truth)" 역할을 하는 것은?
정답: Git 저장소. GitOps에서는 Git에 선언된 상태가 "정답"이며, 실제 환경은 항상 이 정답을 따라가도록 맞춰집니다.
Q2
DevOps와 GitOps의 관계를 가장 잘 설명한 것은?
정답: 3번. DevOps는 문화·철학이고, GitOps는 그것을 Git 중심으로 실현하는 구체적 실천법입니다. 특히 CD 영역을 새롭게 정의합니다.
Q3
"명령어 순서를 적는" 방식이 아니라 "원하는 최종 상태를 적는" 방식을 무엇이라 하나?
정답: 선언적 방식. "무엇이 되어야 하는가(결과)"만 적습니다. 덕분에 도구가 현재 상태와 단순 비교하여 차이만 메울 수 있습니다.
Q4
GitOps가 'Pull 방식'을 선호하는 가장 큰 이유는?
정답: 2번. 클러스터 내부 에이전트가 스스로 Git을 끌어오므로 외부에 권한을 줄 필요가 없고, 항상 비교 중이라 드리프트도 잡아냅니다.
Q5
누군가 운영 클러스터를 직접 수정해 Git 상태와 달라졌을 때, GitOps 에이전트(self-heal 켜짐)는?
정답: 3번. 이것이 자가 치유(self-heal)입니다. Git이 정답이므로, 직접 가한 변경(드리프트)은 다음 조정 루프에서 원래대로 되돌려집니다.
12 마무리

정리와 다음 단계

핵심만 다시 모으면 이렇습니다.

  • GitOps = 원하는 상태를 Git에 선언하고, 에이전트가 실제 환경을 거기에 계속 맞추는 운영 방식.
  • Git이 진실의 원천 — 배포도 롤백도 전부 Git 작업으로 통일된다.
  • DevOps의 구체적 실천법 — 특히 CD 영역을 Git 중심으로 다시 정의한다.
  • 4대 원칙 — 선언적 · 버전 관리 · 자동 pull · 지속 조정.
  • 조정 루프(관찰 → 비교 → 조정)가 심장이며, 드리프트를 자동으로 치유한다.
  • 대표 도구는 Argo CD와 Flux.

직접 해보고 싶다면

  1. 로컬에 가벼운 Kubernetes(minikube·kind·k3s)를 띄운다.
  2. Argo CD를 설치하고 웹 UI에 접속해 본다.
  3. 간단한 nginx 매니페스트를 Git에 올리고 Application으로 연결한다.
  4. 이미지 태그를 바꿔 커밋하고, UI에서 동기화가 일어나는 것을 눈으로 확인한다.
  5. 클러스터를 일부러 kubectl edit으로 바꿔 보고, 자동 복구되는지 관찰한다.
마지막 한마디

GitOps의 본질은 기술이 아니라 습관의 전환입니다. "서버를 직접 만지는 손"을 "Git에 커밋하는 손"으로 바꾸는 것. 이 습관 하나가 운영을 추적 가능하고, 재현 가능하고, 안전하게 만듭니다.
