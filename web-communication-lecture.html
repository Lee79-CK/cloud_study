01 Why · 왜 여러 통신 방식이 필요할까

하나의 길로는 부족합니다

웹은 30여 년간 “브라우저가 요청하고, 서버가 응답한다”는 단순한 모델로 작동해 왔습니다. 하지만 채팅, 알림, 실시간 주가, AI 챗봇 스트리밍, 게임, 대규모 마이크로서비스가 등장하면서 "요청-응답" 모델만으로는 풀 수 없는 문제들이 생겼습니다.

① 서버가 먼저 말하고 싶을 때

새 메시지, 가격 변동, 알림처럼 이벤트가 발생한 쪽은 서버인 경우, 브라우저가 계속 "있어요? 있어요?" 물어볼 수는 없습니다.

② 양쪽이 동시에 떠들어야 할 때

온라인 게임, 화상 회의, 협업 도구. 클라이언트와 서버가 거의 동시에 메시지를 주고받아야 합니다.

③ 한 응답이 길고 점진적일 때

LLM 답변처럼 결과가 토큰 단위로 흘러나오는 경우, 다 만들어질 때까지 기다리지 않고 일부씩 받고 싶습니다.

핵심 통찰

네 가지 프로토콜은 서로 경쟁 관계가 아니라, 해결하려는 문제가 다른 도구입니다. 망치와 드라이버를 비교하지 않듯, 상황에 맞는 도구를 고르는 감각이 핵심입니다.

02 기준점 · The Baseline

REST API · 묻고 답하기의 정석

REST는 가장 친숙한 통신 방식입니다. 클라이언트가 HTTP로 요청을 보내면 서버가 응답을 돌려주고 끝. 이 단순함이 강점이지만, 동시에 한계이기도 합니다.

통신 흐름
클라이언트가 요청을 보낼 때마다 새 연결이 열렸다가 닫힙니다.

핵심 특징

• 요청 한 번 → 응답 한 번 (1:1 모델)
• HTTP 메서드(GET/POST/PUT/DELETE)로 의도 표현
• 데이터는 보통 JSON
• 상태를 서버에 저장하지 않음 (stateless)
• 캐시·CDN과 잘 어울림

장점

  • 모든 언어/도구에서 지원
  • 디버깅·테스트 쉬움
  • HTTP 인프라(캐시,게이트웨이) 활용

한계

  • 서버가 먼저 알려줄 수 없음
  • 실시간성이 떨어짐 (폴링 필요)
  • 매번 헤더 오버헤드
코드로 보기 — 가장 기본적인 REST 호출
브라우저(JavaScript)에서 서버 API를 호출하는 가장 흔한 패턴
// 사용자 정보를 가져온다
const res = await fetch('https://api.example.com/users/42');
const user = await res.json();
console.log(user.name);

// 새 주문을 만든다
await fetch('/orders', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ item: 'book', qty: 2 })
});
import express from 'express';
const app = express();
app.use(express.json());

app.get('/users/:id', (req, res) => {
  res.json({ id: req.params.id, name: 'Lee' });
});

app.post('/orders', (req, res) => {
  // req.body 로 클라이언트가 보낸 JSON 접근
  res.status(201).json({ ok: true });
});

app.listen(3000);
REST의 한계가 드러나는 순간

"채팅방에 새 메시지가 오면 알려줘" — REST로 구현하려면 클라이언트가 1초마다 GET /messages를 호출해야 합니다. 이를 폴링(polling)이라 부르며, 대부분의 호출이 "변화 없음"으로 끝나기 때문에 매우 비효율적입니다. 여기서 SSE와 WebSocket이 등장합니다.

03 High Performance RPC

gRPC · 함수 호출처럼 통신하기

Google이 만든 고성능 통신 프레임워크. 서버의 함수를, 마치 내 코드에 있는 함수인 것처럼 호출하게 해 줍니다. 백엔드 마이크로서비스 사이에서 가장 빛납니다.

3줄 요약

① Protocol Buffers(.proto)라는 정의 파일에 "어떤 함수가 있고, 입출력은 어떤 모양인지"를 적습니다.
② 그 정의로부터 클라이언트/서버 코드가 자동 생성됩니다.
③ 통신은 HTTP/2 + 바이너리라서 JSON보다 훨씬 작고 빠릅니다.

왜 빠른가

• HTTP/2의 멀티플렉싱: 한 연결로 여러 요청을 동시에
• 바이너리 직렬화: JSON 텍스트 대신 압축된 바이트
• 스키마 기반: 필드명이 매번 전송되지 않음 (번호로 식별)
• 스트리밍 내장: 한 연결로 연속적인 메시지 교환

gRPC의 4가지 통신 모드
단방향 요청부터 양방향 스트리밍까지, 한 프레임워크 안에서 모두 지원합니다.
// 이 파일이 "계약서" 역할을 한다.
// 클라이언트와 서버가 이 파일을 공유해서, 양쪽 코드가 자동 생성된다.
syntax = "proto3";

message UserRequest  { int32 id = 1; }
message UserResponse { string name = 1; int32 age = 2; }

service UserService {
  // ① unary
  rpc GetUser(UserRequest) returns (UserResponse);

  // ② server streaming — 한 번 요청, 여러 응답
  rpc WatchUser(UserRequest) returns (stream UserResponse);
}
# Python gRPC 서버 (자동 생성된 stub 사용)
import grpc, time
from concurrent import futures
import user_pb2, user_pb2_grpc

class UserServicer(user_pb2_grpc.UserServiceServicer):
    def GetUser(self, request, context):
        return user_pb2.UserResponse(name="Lee", age=30)

    def WatchUser(self, request, context):    # 서버 스트리밍
        for i in range(5):
            yield user_pb2.UserResponse(name=f"update-{i}", age=30)
            time.sleep(1)

server = grpc.server(futures.ThreadPoolExecutor(max_workers=10))
user_pb2_grpc.add_UserServiceServicer_to_server(UserServicer(), server)
server.add_insecure_port("[::]:50051")
server.start(); server.wait_for_termination()
# 클라이언트는 마치 로컬 함수처럼 호출한다
import grpc, user_pb2, user_pb2_grpc

channel = grpc.insecure_channel("localhost:50051")
stub = user_pb2_grpc.UserServiceStub(channel)

# ① unary 호출
res = stub.GetUser(user_pb2.UserRequest(id=42))
print(res.name)        # → "Lee"

# ② 서버 스트림 받기
for update in stub.WatchUser(user_pb2.UserRequest(id=42)):
    print(update.name)   # update-0, update-1, ...

gRPC의 강점

  • JSON 대비 5~10배 작은 페이로드
  • 스키마 기반 → 타입 안정성, 자동 코드 생성
  • HTTP/2 멀티플렉싱으로 낮은 지연
  • 4가지 스트리밍 모드 내장
  • 다양한 언어 지원 (Go, Python, Java, ...)

gRPC의 약점

  • 브라우저 직접 사용 불가 (gRPC-Web 필요)
  • 바이너리라 디버깅이 어렵다
  • .proto 학습·관리 필요
  • HTTP 캐시·CDN과 안 맞음
MLOps 관점 · Cheolkyu님 환경과의 접점

NVIDIA Triton Inference Server, KServe, TorchServe 같은 모델 서빙 시스템은 내부 통신을 gRPC로 노출하는 경우가 많습니다. REST 엔드포인트도 함께 제공하지만, 추론 클라이언트(특히 배치/스트리밍)에서는 gRPC가 처리량·지연 면에서 우위입니다. Kubernetes 안의 service-to-service 호출에도 자주 쓰입니다.

04 Server → Client, One Way

Server-Sent Events · 서버의 라디오 방송

서버가 일방적으로 클라이언트에게 데이터를 흘려보내는 가장 단순한 방법. ChatGPT, Claude의 타이핑 효과가 모두 SSE 위에서 동작합니다.

통신 흐름 — 한 번 열고, 계속 흘려보내기
연결은 한 번만 열립니다. 서버는 준비될 때마다 메시지를 push 합니다.

핵심 특징

• 단방향: 서버 → 클라이언트만 가능
• 그냥 HTTP 위에서 동작 (방화벽 친화적)
• 텍스트 기반 프로토콜 (가독성 ↑)
• 자동 재연결이 표준 스펙에 포함
• 브라우저는 EventSource API로 두세 줄이면 끝

장점

  • 구현이 매우 간단
  • HTTP라 프록시·CORS와 잘 맞음
  • 끊기면 자동 재연결

한계

  • 클라이언트→서버 메시지 불가
  • 한 연결당 텍스트(UTF-8)만
  • HTTP/1.1에서는 연결 수 제한 있음
코드로 보기
단 몇 줄로 실시간 스트리밍이 가능합니다.
// 브라우저 표준 API — EventSource
const es = new EventSource('/events');

es.onmessage = (e) => {
  const data = JSON.parse(e.data);
  console.log('새 메시지:', data);
};

es.onerror = () => {
  // 연결이 끊겨도 브라우저가 자동으로 재시도한다
};
import express from 'express';
const app = express();

app.get('/events', (req, res) => {
  res.setHeader('Content-Type', 'text/event-stream');
  res.setHeader('Cache-Control', 'no-cache');
  res.setHeader('Connection', 'keep-alive');

  const timer = setInterval(() => {
    const price = Math.random() * 100;
    res.write(`data: ${JSON.stringify({price})}\n\n`);
  }, 1000);

  req.on('close', () => clearInterval(timer));
});

app.listen(3000);

05 Full Duplex Channel

WebSocket · 전화선 같은 양방향 연결

한 번 연결하면 그 연결 위에서 클라이언트와 서버가 자유롭게, 동시에, 양방향으로 메시지를 주고받을 수 있습니다. 채팅, 게임, 협업 도구의 표준 선택지.

통신 흐름 — 핸드셰이크 후 자유 통신
처음에 HTTP "업그레이드" 요청으로 시작한 뒤, 별도의 양방향 채널로 전환됩니다.

핵심 특징

• 전이중(full-duplex): 양쪽이 동시에 송수신
• TCP 위의 별도 프로토콜 (포트는 보통 80/443)
• 텍스트와 바이너리 모두 전송 가능
• 연결이 한 번 열린 뒤로는 HTTP 헤더 오버헤드 없음
• 게임·금융처럼 낮은 지연이 중요한 곳에 강함

장점

  • 진정한 양방향 실시간
  • 오버헤드 매우 작음 (한번 핸드셰이크 후)
  • 바이너리 데이터 자유

한계

  • HTTP가 아니라 캐시/CDN 활용 어려움
  • 자동 재연결을 직접 구현해야 함
  • 연결 유지 → 서버 자원·스케일 고민
  • 일부 기업 프록시가 차단할 수 있음
코드로 보기
SSE와 비슷한 분량이지만 양방향이라는 점이 차이를 만듭니다.
// 브라우저 표준 API — WebSocket
const ws = new WebSocket('wss://example.com/chat');

ws.onopen = () => {
  ws.send(JSON.stringify({ type: 'hello', name: 'Cheolkyu' }));
};

ws.onmessage = (e) => {
  const msg = JSON.parse(e.data);
  console.log('서버에서 받음:', msg);
};

ws.onclose = () => console.log('연결 종료');
ws.onerror = (e) => console.error(e);
import { WebSocketServer } from 'ws';
const wss = new WebSocketServer({ port: 8080 });

wss.on('connection', (client) => {
  console.log('새 클라이언트 접속');

  client.on('message', (raw) => {
    const msg = JSON.parse(raw);

    // 모든 접속자에게 브로드캐스트 (간단한 채팅 서버)
    wss.clients.forEach(c => {
      if (c.readyState === 1) c.send(JSON.stringify(msg));
    });
  });
});

08 Glossary

기억해 두면 좋을 용어 사전

HTTP/1.1 vs HTTP/2
한 연결에 한 번에 한 요청만 흘릴 수 있던 HTTP/1.1과 달리, HTTP/2는 한 연결에 여러 요청을 동시에 보낼 수 있습니다(멀티플렉싱). gRPC는 이 능력을 사용해 빨라집니다.
폴링 (Polling)
클라이언트가 주기적으로 "변한 거 있어?"라고 묻는 방식. REST밖에 없을 때의 차선책이며, 대부분의 요청이 헛수고가 됩니다.
풀-듀플렉스 (Full Duplex)
양쪽이 동시에 송신·수신할 수 있는 통신. 전화 통화가 풀-듀플렉스, 무전기가 하프-듀플렉스의 비유.
스트리밍 (Streaming)
응답을 한 덩어리로 받지 않고, 조각 단위로 흘려가며 받는 방식. LLM 토큰 출력, 영상 재생 등이 대표 사례.
Protocol Buffers (Protobuf)
Google이 만든 직렬화 포맷. JSON처럼 데이터를 표현하지만, 스키마 기반 바이너리라서 더 작고 빠릅니다. gRPC의 기본 포맷.
핸드셰이크 (Handshake)
통신을 시작하기 전 양쪽이 규칙을 합의하는 과정. WebSocket은 처음에 HTTP Upgrade 헤더로 핸드셰이크 후 별도 채널로 전환됩니다.
EventSource
SSE를 사용하기 위한 브라우저 표준 JavaScript API. URL만 주면 자동으로 연결·수신·재연결까지 처리합니다.
gRPC-Web
브라우저에서 gRPC를 쓰기 위한 변형. HTTP/2 trailers를 브라우저가 지원하지 않아, Envoy 같은 프록시를 통해 트랜스코딩하는 방식이 일반적입니다.
09 Wrap up

기억할 한 문장씩

REST

"묻고 답하는" 일에 가장 단순하고 안정적인 정답.

gRPC

백엔드끼리, 함수 호출처럼, 빠르고 타입 안전하게.

SSE

서버가 일방적으로 흘려보내는, 가장 손쉬운 실시간.

WebSocket

양쪽이 동시에 떠들어야 한다면, 이 도구 한 가지.

마지막 조언

처음에는 모든 것을 REST로 시작해도 괜찮습니다. 폴링이 답답해질 때 SSE를 더하고, 양방향이 필요해질 때 WebSocket을 도입하고, 백엔드 사이의 빈번한 호출이 병목이 될 때 gRPC를 고려하세요. 도구는 문제를 만난 뒤 도입하는 것이 가장 좋은 학습 순서입니다.
