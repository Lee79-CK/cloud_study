00 — Motivation

왜 RAG를 "평가"해야 하는가

RAG(Retrieval-Augmented Generation, 검색 증강 생성)는 LLM에게 외부 문서를 검색해 함께 넣어줌으로써, 모델이 모르는 최신 지식이나 사내 데이터를 바탕으로 답하게 만드는 기법입니다.

RAG 파이프라인은 크게 두 단계로 나뉩니다. ① 검색기(Retriever)가 질문과 관련된 문서를 찾아오고, ② 생성기(LLM)가 그 문서를 근거로 답변을 만듭니다. 문제는 — 둘 중 어디서든 잘못될 수 있다는 점입니다.

INPUT
질문
사용자의 user_input
→
STEP 1
검색기
관련 문서 검색
→
STEP 2
생성기
LLM이 답변 작성
→
OUTPUT
답변
최종 response

예를 들어 챗봇이 엉뚱한 답을 했다고 가정해 봅시다. 원인은 무엇일까요?

  • 검색기가 관련 없는 문서를 가져왔을 수도 있고 (검색 문제)
  • 좋은 문서를 가져왔는데 LLM이 그걸 무시하고 지어냈을 수도 있고 (생성 문제 = 환각/Hallucination)
  • 답변이 사실이긴 한데 질문과 동떨어진 이야기일 수도 있습니다 (관련성 문제)
비유로 이해하기

RAG를 오픈북 시험을 보는 학생이라고 생각해 보세요. 검색기는 "참고서에서 필요한 페이지를 펼치는 능력"이고, 생성기는 "그 페이지를 읽고 답을 쓰는 능력"입니다. 학생이 틀렸다면, 엉뚱한 페이지를 펼쳤는지 아니면 맞는 페이지를 보고도 헛소리를 썼는지 구분해야 고칠 수 있습니다. RAGAS는 바로 이 "어디서 틀렸는지"를 단계별 점수로 알려주는 채점관입니다.

"느낌상 잘 되는 것 같다(vibe check)"는 평가는 위험합니다. 프롬프트 하나, 청크 크기 하나만 바꿔도 품질이 출렁이는데, 사람이 매번 수십 개 답변을 눈으로 검수할 수는 없죠. 측정할 수 없으면 개선할 수 없습니다. RAGAS는 이 반복적인 채점을 자동화합니다.

01 — Definition

RAGAS란 무엇인가

RAGAS는 Retrieval Augmented Generation Assessment의 약자로, RAG 파이프라인을 자동으로 평가하기 위한 오픈소스 파이썬 프레임워크입니다.

2023년 논문(arXiv:2309.15217)에서 처음 제안되었으며, 가장 큰 특징은 "참조(정답) 없이도(reference-free) 평가가 가능하다"는 점입니다. 전통적인 평가는 사람이 만든 정답 라벨이 반드시 필요했지만, RAGAS는 LLM을 심사위원(LLM-as-a-judge)으로 활용해 사람 라벨링 없이도 여러 품질 차원을 점수화합니다.

핵심 아이디어 3가지

아이디어설명
LLM을 심사위원으로"이 답변이 이 문서에 근거하고 있나?" 같은 판단을 강력한 LLM에게 시켜서, 사람 대신 일관되게 채점합니다.
단계별 분해 측정검색 품질과 생성 품질을 따로 점수 내기 때문에, 문제의 원인을 검색 vs 생성으로 분리해 진단할 수 있습니다.
프레임워크 통합LangChain, LlamaIndex, Haystack 등 주요 RAG 프레임워크 및 LangSmith·Arize 같은 관측 도구와 쉽게 연결됩니다.
알아두기

RAGAS는 처음에 explodinggradients 팀이 만들었고, 현재는 vibrantlabsai/ragas 저장소에서 유지보수되고 있습니다. 버전이 빠르게 올라가며 API가 바뀌어 왔으므로(v0.1 → v0.2 → v0.4), 인터넷의 오래된 예제 코드는 동작하지 않을 수 있습니다. 이 강의는 최신(v0.2+) API 기준입니다.

RAGAS ≠ RAG 라이브러리

RAGAS는 RAG를 만들어 주는 도구가 아니라 채점하는 도구입니다. RAG 자체는 LangChain·LlamaIndex 등으로 직접 구축하고, RAGAS는 그 결과물의 품질을 측정하는 데 씁니다. 둘의 역할을 헷갈리지 마세요.

02 — Data Model

평가 데이터의 4요소

RAGAS로 평가하려면 먼저 평가할 데이터를 정해진 형식으로 모아야 합니다. RAG 한 번 실행 = 한 개의 "샘플"이며, 샘플은 보통 다음 네 가지 필드로 구성됩니다.

user_input 필수
사용자가 던진 질문. 예) "상대성 이론을 제안한 사람은?"
retrieved_contexts 필수
검색기가 가져온 문서 조각들의 목록. 검색 품질 메트릭의 채점 대상입니다.
response 필수
LLM이 최종적으로 생성한 답변. 생성 품질 메트릭의 채점 대상입니다.
reference 선택
사람이 만든 정답(golden answer). Context Recall, Factual Correctness 같은 일부 메트릭에서만 필요합니다.

여기서 핵심은 메트릭마다 필요한 필드가 다르다는 점입니다. 어떤 메트릭은 정답(reference) 없이도 채점되고(reference-free), 어떤 메트릭은 정답이 있어야만 채점됩니다(reference-based). 이 차이를 알면 "데이터를 얼마나 준비해야 하는가"가 명확해집니다.

메트릭 예시필요한 필드정답 필요?
Faithfulnessresponse + retrieved_contexts아니오
Answer Relevancyuser_input + response아니오
Context Precision (LLM)user_input + retrieved_contexts (+reference)경우에 따라
Context Recallretrieved_contexts + reference예
Factual Correctnessresponse + reference예

RAGAS에서는 이 샘플들의 묶음을 EvaluationDataset 객체로 만듭니다. 파이썬 딕셔너리 리스트를 그대로 변환할 수 있어 직관적입니다.

build_dataset.py
from ragas import EvaluationDataset # RAG 실행 결과 1건 = 딕셔너리 1개 dataset = [ { "user_input": "상대성 이론을 제안한 사람은?", "retrieved_contexts": ["알베르트 아인슈타인은 상대성 이론을 제안했다."], "response": "아인슈타인이 상대성 이론을 제안했습니다.", "reference": "알베르트 아인슈타인", # 선택 }, # ... 더 많은 샘플 ] evaluation_dataset = EvaluationDataset.from_list(dataset)
실무 팁

처음에는 reference(정답)를 굳이 만들지 마세요. 정답 라벨링은 비용이 큽니다. Faithfulness·Answer Relevancy처럼 정답이 필요 없는 메트릭부터 시작하고, 검색 재현율(Context Recall)을 정밀하게 보고 싶을 때 정답을 추가하는 것이 효율적입니다.

03 — Core Metrics

핵심 메트릭 4가지

RAGAS에는 수십 개의 메트릭이 있지만, 입문자는 우선 이 4가지 "고전 RAG 점수"만 확실히 이해하면 됩니다. 2개는 생성(답변) 품질을, 2개는 검색(문서) 품질을 봅니다.

구분메트릭한 줄 질문
생성 품질Faithfulness답이 가져온 문서에 근거하나? (환각 여부)
생성 품질Answer Relevancy답이 질문에 제대로 답하나?
검색 품질Context Precision가져온 문서 중 쓸모 있는 게 위에 있나?
검색 품질Context Recall정답에 필요한 정보를 다 가져왔나?

아래 탐색기에서 각 메트릭을 직접 눌러보며 무엇을 묻는지, 어떻게 계산되는지, 예시 점수가 왜 그렇게 나오는지를 확인하세요.

Faithfulness — 충실도 (환각 탐지)

답변에 담긴 모든 주장(claim)이 검색된 문서로 뒷받침되는지를 봅니다. 문서에 없는 내용을 LLM이 지어냈다면(환각), 점수가 떨어집니다. 정답이 필요 없는 메트릭입니다.

Faithfulness = (문서로 뒷받침되는 주장 수) / (답변의 전체 주장 수)

예시 — 답변: "아인슈타인은 상대성 이론을 제안했고, 1921년 노벨 화학상을 받았다." 문서에는 상대성 이론 얘기만 있고 노벨상 언급은 없음.

→ 주장 2개 중 1개만 문서로 뒷받침됨. 두 번째 주장은 근거 없음(게다가 사실은 물리학상).

0.50

Answer Relevancy — 답변 관련성

답변이 질문에 얼마나 직접적으로 답하는지를 봅니다. 사실 여부가 아니라 "동문서답이 아닌지"를 보는 지표입니다. 답변에서 거꾸로 질문을 여러 개 생성해, 원래 질문과 얼마나 비슷한지를 비교합니다.

Answer Relevancy = 답변→역생성 질문들과 원본 질문의 평균 유사도

예시 — 질문: "프랑스의 수도는?" / 답변: "프랑스는 유럽에 있는 아름다운 나라입니다."

→ 사실이지만 질문(수도)에 답하지 않음. 답변에서 역생성되는 질문은 "프랑스는 어디 있나?"에 가까워 원본과 동떨어짐.

0.28

Context Precision — 문맥 정밀도

검색기가 가져온 문서들 중 실제로 관련 있는 문서가 얼마나 상위에 랭크되었는지를 봅니다. 관련 문서가 1등이면 높고, 쓸모없는 문서가 위에 깔려 있으면 낮아집니다. 순위(ranking)를 평가하는 지표입니다.

Context Precision@k = 관련 문서의 순위 가중 정밀도의 평균

예시 — 검색 결과 3건: [① 관련 ✓, ② 무관 ✗, ③ 무관 ✗]. 관련 문서가 1등에 위치.

→ 정작 필요한 문서가 맨 위에 있으므로 정밀도는 높음. 다만 아래에 잡음이 섞여 만점은 아님.

0.83

Context Recall — 문맥 재현율

정답(reference)을 만드는 데 필요한 정보가 검색된 문서 안에 얼마나 담겨 있는지를 봅니다. 정답의 각 문장이 검색 문맥으로 뒷받침되는지 따집니다. 정답 라벨이 반드시 필요한 메트릭입니다.

Context Recall = (문맥으로 뒷받침되는 정답 문장 수) / (정답의 전체 문장 수)

예시 — 정답이 3개 사실로 구성. 검색 문맥에는 그중 2개만 들어있고 1개는 누락.

→ 필요한 정보의 2/3만 검색됨. 검색기가 핵심 문서 하나를 놓친 상황.

0.67
정밀도 vs 재현율, 헷갈리지 않기

Precision(정밀도)은 "내가 가져온 것 중 진짜가 얼마나 되나?"(잡음을 적게 가져왔나), Recall(재현율)은 "있어야 할 것 중 내가 얼마나 가져왔나?"(빠뜨리지 않았나)입니다. 어망에 비유하면, 정밀도는 "잡은 것 중 물고기 비율", 재현율은 "바다의 물고기 중 잡은 비율"입니다. 검색기를 튜닝할 때 이 둘은 보통 트레이드오프 관계입니다.

점수 해석 시뮬레이터

비용을 줄이는 조합

모든 메트릭이 LLM을 호출하는 것은 아닙니다. BLEU·ROUGE·Exact Match 같은 전통 NLP 메트릭은 LLM 호출이 없어 거의 공짜이고 빠릅니다. CI 파이프라인에서는 전통 메트릭으로 1차 거름망을 만들고, 통과한 케이스만 LLM 기반 메트릭으로 정밀 평가하면 비용을 크게 아낄 수 있습니다.

커스텀 메트릭도 쉽게

"답변이 정중한가?", "위험한 의료 조언을 포함했나?" 같은 도메인 기준은 AspectCritic이나 DiscreteMetric으로 프롬프트만 적어 직접 만들 수 있습니다.

custom_metric.py
from ragas.metrics import AspectCritic # 예/아니오로 판정하는 커스텀 기준 politeness = AspectCritic( name="politeness", definition="답변이 정중하고 공손한 어조인가?", llm=evaluator_llm, )
05 — Installation

설치 방법

설치는 pip install ragas 한 줄이면 끝납니다. 다만 RAGAS는 평가에 LLM을 쓰므로, LLM/임베딩 제공자(OpenAI 등)와 API 키를 함께 준비해야 실제로 동작합니다.

# 안정 버전 설치 (권장) pip install ragas # 가상환경 사용을 강력 권장 python -m venv .venv source .venv/bin/activate # Windows: .venv\Scripts\activate pip install ragas
# 최신 기능을 쓰고 싶다면 main 브랜치에서 직접 설치 pip install git+https://github.com/vibrantlabsai/ragas.git # 코드를 수정/기여할 경우 (editable install) git clone https://github.com/vibrantlabsai/ragas.git cd ragas pip install -e .
# OpenAI를 평가 LLM으로 쓸 때, 버전 충돌 방지를 위해 # langchain 관련 패키지를 명시적으로 설치하는 것을 권장 pip install -U "langchain-core>=0.2,<0.3" \ "langchain-openai>=0.1,<0.2" \ openai # 다른 제공자: Bedrock, Gemini, Vertex AI, Azure 등도 지원
# API 키는 코드에 직접 쓰지 말고 환경변수로 (보안!) # Linux / macOS export OPENAI_API_KEY="sk-..." # 파이썬에서 .env 파일을 쓰려면 pip install python-dotenv # 코드: from dotenv import load_dotenv; load_dotenv()
설치 시 자주 겪는 함정

1. 버전 비호환. 인터넷의 옛 예제는 ragas.metrics.context_precision 같은 구버전 import를 씁니다. v0.2부터 API가 크게 바뀌었으니 반드시 설치된 버전의 공식 문서를 보세요. (pip show ragas로 버전 확인)

2. langchain 의존성 충돌. 이미 langchain을 쓰는 프로젝트라면 버전 핀이 충돌하기 쉽습니다. 가상환경을 분리하세요.

3. API 키 누락. 키 없이 evaluate()를 부르면 인증 오류가 납니다. 평가도 LLM을 호출한다는 점을 잊지 마세요.

06 — Hands On

실전: 처음부터 끝까지

이제 가장 작은 완전한 예제를 만들어 봅니다. ① 평가 LLM 준비 → ② 데이터셋 구성 → ③ 메트릭 선택 → ④ evaluate() 실행 → ⑤ 결과 분석의 5단계 흐름입니다.

# RAGAS는 "심사위원 LLM"으로 채점합니다. 먼저 그 LLM을 준비합니다. from langchain_openai import ChatOpenAI from ragas.llms import LangchainLLMWrapper # LangChain의 LLM을 RAGAS 래퍼로 감쌉니다 llm = ChatOpenAI(model="gpt-4o") evaluator_llm = LangchainLLMWrapper(llm)
from ragas import EvaluationDataset # 우리 RAG를 돌려 모은 결과들 (여기선 예시로 손으로 작성) dataset = [ { "user_input": "상대성 이론을 제안한 사람은?", "retrieved_contexts": [ "알베르트 아인슈타인은 상대성 이론을 제안해 " "시간·공간·중력에 대한 이해를 바꿨다." ], "response": "아인슈타인이 상대성 이론을 제안했습니다.", "reference": "알베르트 아인슈타인이 상대성 이론을 제안했다.", }, # ... 보통 수십~수백 개를 모읍니다 ] evaluation_dataset = EvaluationDataset.from_list(dataset)
from ragas import evaluate from ragas.metrics import ( LLMContextRecall, # 검색 재현율 (정답 필요) Faithfulness, # 충실도 (환각) FactualCorrectness, # 사실 정확성 (정답 필요) ) result = evaluate( dataset=evaluation_dataset, metrics=[LLMContextRecall(), Faithfulness(), FactualCorrectness()], llm=evaluator_llm, ) print(result) # → {'context_recall': 1.0000, 'faithfulness': 0.8571, # 'factual_correctness': 0.7280}
# 전체 평균뿐 아니라 샘플별 점수를 데이터프레임으로 봅니다 df = result.to_pandas() print(df.head()) # 점수가 낮은 케이스만 골라 원인 분석 low = df[df["faithfulness"] < 0.5] print(low[["user_input", "response", "faithfulness"]]) # 결과를 CSV로 저장해 이력 관리/회귀 테스트에 활용 df.to_csv("ragas_result.csv", index=False)
핵심은 "샘플별 점수"

평균 점수(faithfulness: 0.85)만 보면 "그럭저럭 괜찮네"로 끝나기 쉽습니다. 진짜 가치는 to_pandas()로 점수가 낮은 개별 사례를 찾아내 "왜 이 답변은 환각이 났나"를 들여다보는 데 있습니다. 평가는 점수 매기기가 아니라 디버깅 도구입니다.

비동기로 빠르게

샘플이 많으면 평가에 시간이 걸립니다. RunConfig로 동시성과 타임아웃을 조절하거나 aevaluate()(비동기)를 쓰면 처리 속도를 높일 수 있습니다.

tune_runtime.py
from ragas import evaluate, RunConfig # 동시 요청 수·재시도·타임아웃 제어 (Rate Limit 회피에 유용) config = RunConfig(max_workers=4, timeout=60, max_retries=5) result = evaluate(evaluation_dataset, metrics=[...], llm=evaluator_llm, run_config=config)
07 — Test Data Generation

테스트 데이터 자동 생성

"평가하려면 질문-정답 데이터가 있어야 하는데, 그걸 손으로 만들기가 제일 힘들다." — RAGAS의 강력한 기능 중 하나가 바로 여러분의 문서로부터 테스트셋을 자동 생성해 주는 것입니다.

원리는 이렇습니다. ① 문서들로 지식 그래프(Knowledge Graph)를 만들고, ② 문서 간 관계를 분석해 ③ 단순 질문(single-hop)부터 여러 문서를 종합해야 답할 수 있는 복합 질문(multi-hop)까지 다양한 난이도의 질문과 정답을 자동 생성합니다.

from ragas.testset import TestsetGenerator from ragas.llms import LangchainLLMWrapper from ragas.embeddings import LangchainEmbeddingsWrapper from langchain_openai import ChatOpenAI, OpenAIEmbeddings generator_llm = LangchainLLMWrapper(ChatOpenAI(model="gpt-4o")) generator_emb = LangchainEmbeddingsWrapper(OpenAIEmbeddings()) generator = TestsetGenerator(llm=generator_llm, embedding_model=generator_emb) # docs = 내가 로드한 LangChain Document 리스트 testset = generator.generate_with_langchain_docs(docs, testset_size=20) df = testset.to_pandas() # 질문/정답/문맥이 채워진 표
자동 생성 ≠ 검수 면제

자동 생성된 테스트셋은 출발점일 뿐입니다. LLM이 만든 질문·정답에는 부정확한 것이 섞일 수 있으니, 평가의 기준선으로 삼기 전에 사람이 한 번 훑어보며 이상한 항목을 거르는 것이 안전합니다. 한국어 문서라면 프롬프트 언어 적응(language adaptation) 설정도 확인하세요.

응용 분야와 통합

RAGAS는 단발성 채점을 넘어, RAG 시스템의 개발-운영 전 주기에 녹아들 때 진가를 발휘합니다.

생태계 통합

RAGAS는 주요 도구들과 곧바로 연결됩니다. 빌드 프레임워크로는 LangChain, LlamaIndex, Haystack, LangGraph, 관측 도구로는 LangSmith, Arize Phoenix, LLM 제공자로는 OpenAI, Amazon Bedrock, Google Gemini/Vertex AI, Azure 등을 지원합니다.

MLOps 관점에서

전통적인 ML에 정확도·F1 같은 지표와 회귀 테스트가 있듯, LLM/RAG 시스템에는 RAGAS 점수가 그 역할을 합니다. "느낌(vibe check)"을 "평가 루프(eval loop)"로 대체하는 것 — 이것이 LLMOps의 핵심 실천 중 하나이며, RAGAS 점수를 CI 파이프라인의 통과 기준으로 삼으면 RAG 품질을 코드 변경처럼 관리할 수 있습니다.

09 — Caveats

주의사항과 함정

RAGAS는 강력하지만 만능이 아닙니다. "평가자도 LLM"이라는 본질에서 비롯되는 한계를 반드시 이해하고 써야 점수를 신뢰할 수 있습니다.

1. 점수는 확정적이지 않다 (비결정성)

LLM 심사위원은 같은 입력에도 미묘하게 다른 점수를 낼 수 있습니다. 점수를 소수점 끝자리까지 절대시하지 말고, 추세와 상대 비교(A안 vs B안)로 해석하세요. 안정성을 높이려면 temperature=0에 가깝게 설정합니다.

2. 평가자 LLM의 품질에 종속된다

약한 모델로 채점하면 채점 자체가 부정확합니다. 평가에는 가급적 강한 모델(GPT-4o급)을 쓰고, 비용이 걱정이면 전통 NLP 메트릭이나 NVIDIA 경량 메트릭과 섞으세요.

3. 평가에도 비용과 시간이 든다

LLM 기반 메트릭은 샘플 하나당 여러 번 LLM을 호출합니다. 수천 건을 매번 GPT-4o로 평가하면 API 비용이 빠르게 쌓입니다. 대표 샘플로 평가 규모를 정하고, 캐싱(Cache)과 동시성(RunConfig)을 활용하세요.

꼭 피해야 할 실수

· 한 메트릭만 보기 — Faithfulness만 높다고 좋은 RAG가 아닙니다. "문서를 충실히 베꼈지만 질문엔 안 맞는" 답일 수 있죠. 생성·검색 메트릭을 함께 봐야 합니다.

· 평가 LLM = 생성 LLM 동일시 맹신 — 같은 모델이 생성하고 채점하면 자기 편향이 생길 수 있습니다. 가능하면 다른 계열 모델로 교차 검증하세요.

· 데이터 형식 불일치 — retrieved_contexts는 문자열이 아니라 문자열의 리스트여야 합니다. 필드명·타입이 어긋나면 조용히 잘못된 점수가 나옵니다.

4. 한국어·도메인 특수성

기본 프롬프트는 영어 기준입니다. 한국어 RAG를 평가할 때는 메트릭의 언어 적응(language adaptation) 기능으로 프롬프트를 한국어에 맞추면 채점 품질이 올라갑니다. 의료·법률처럼 전문 도메인이라면 커스텀 메트릭(Rubrics)으로 기준을 직접 정의하는 편이 정확합니다.

10 — Check Your Understanding

이해도 점검 퀴즈

개념이 손에 잡혔는지 확인해 봅시다. 보기를 클릭하면 정답과 해설이 나타납니다.

Q1.챗봇이 검색한 문서에 없는 내용을 지어내 답했습니다. 이 "환각"을 가장 잘 잡아내는 메트릭은?
정답: Faithfulness. 답변의 주장들이 검색된 문서로 뒷받침되는 비율을 측정하므로, 문서에 없는 내용을 지어내면 점수가 떨어집니다.
Q2.다음 중 평가 시 정답(reference) 라벨이 반드시 필요한 메트릭은?
정답: Context Recall. 정답을 구성하는 정보가 검색 문맥에 얼마나 들어있는지 보므로 정답이 있어야 합니다. Faithfulness·Answer Relevancy는 정답 없이도 측정됩니다(reference-free).
Q3.RAGAS의 본질에서 비롯되는 한계로 옳은 설명은?
정답: 첫 번째. RAGAS는 LLM-as-a-judge라서 비결정성과 평가자 모델 의존성이 있습니다. BLEU 같은 전통 메트릭은 LLM 호출이 없고, 다수 메트릭은 정답 없이도 동작합니다.
Q4."가져온 문서 중 관련 있는 것이 얼마나 상위에 랭크되었는가"를 보는 검색 품질 지표는?
정답: Context Precision. 검색 결과의 순위를 고려해 관련 문서가 상단에 잘 배치됐는지 평가합니다. 반대로 "필요한 정보를 빠뜨리지 않았나"는 Context Recall입니다.
11 — Summary

정리 & 실전 체크리스트

오늘 배운 내용을 한 장으로 압축하면 — RAGAS는 RAG의 검색·생성 품질을 LLM 심사위원으로 자동 채점해, 느낌이 아닌 숫자로 개선하게 해 주는 프레임워크입니다.

핵심 요약

  • RAGAS = RAG 채점관. 만드는 도구가 아니라 측정하는 도구입니다.
  • 데이터 4요소: user_input · retrieved_contexts · response · (선택) reference.
  • 핵심 4메트릭: 생성은 Faithfulness·Answer Relevancy, 검색은 Context Precision·Context Recall.
  • 흐름: 평가 LLM 준비 → EvaluationDataset 구성 → 메트릭 선택 → evaluate() → to_pandas()로 디버깅.
  • 한계: 비결정성, 평가자 모델 의존, 비용. 여러 메트릭을 함께 보고 추세로 해석하세요.

RAG 평가 시작 체크리스트

  • 01가상환경을 만들고 pip install ragas + 평가 LLM(API 키) 준비
  • 02정답 없이 시작 — Faithfulness · Answer Relevancy로 첫 베이스라인 측정
  • 03검색을 정밀 진단할 때만 정답을 추가해 Context Recall · Precision 도입
  • 04테스트셋이 부족하면 TestsetGenerator로 자동 생성 후 사람이 검수
  • 05to_pandas()로 점수 낮은 사례를 찾아 원인(검색 vs 생성) 분리 진단
  • 06구성 변경(청크·top-k·프롬프트)마다 재평가해 점수로 의사결정
  • 07안정되면 CI 파이프라인에 평가를 넣어 회귀 방지 게이트로 운영
