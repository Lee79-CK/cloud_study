Chapter 01

왜 우리는 추론 서버가 필요할까?

AI 모델을 만드는 것과, 그 모델을 수많은 사용자에게 안정적으로 서비스하는 것은 완전히 다른 문제입니다. Triton을 이해하려면, 먼저 우리가 어떤 문제를 풀고 있는지를 알아야 합니다.

가장 단순한 출발점: Flask + model.pt

AI 모델을 처음 배포할 때 떠올리는 가장 쉬운 방법은 이런 코드입니다.

naive_server.py 
from flask import Flask, request
import torch

app = Flask(__name__)
model = torch.load("resnet50.pt").cuda().eval()

@app.route("/predict", methods=["POST"])
def predict():
    img = preprocess(request.files["image"])
    with torch.no_grad():
        out = model(img.cuda())
    return {"class": out.argmax().item()}

깔끔해 보이지만, 사용자가 100명, 1,000명, 10,000명으로 늘어나기 시작하면 이 서버는 곧 한계에 부딪힙니다. 어떤 문제들이 생길까요?

PROBLEM 01

GPU가 한가하다

요청 1건당 GPU 커널 1번 실행 → GPU가 일하는 시간보다 노는 시간이 더 깁니다. H100급 GPU의 잠재력을 5%도 못 쓰는 상황이 발생합니다.

PROBLEM 02

여러 모델 = 여러 서버

이미지 분류·텍스트 분류·임베딩 모델을 각각 서비스하려면 컨테이너·포트·배포·모니터링이 모두 별도가 됩니다.

PROBLEM 03

프레임워크 lock-in

PyTorch 모델은 PyTorch 서버, TensorFlow는 TF Serving… ONNX·TensorRT 최적화 모델을 섞어 쓰려면 코드를 새로 짜야 합니다.

PROBLEM 04

운영 기능 부재

모델 버전 관리, A/B 테스트, 무중단 업데이트, Prometheus 메트릭, gRPC 지원… Flask로는 모두 직접 짜야 합니다.

PROBLEM 05

전처리·후처리가 분산

이미지 디코딩 → 모델 추론 → 후처리(NMS, softmax 등)가 클라이언트·서버에 흩어져 디버깅과 성능 측정이 어렵습니다.

PROBLEM 06

LLM·스트리밍 미지원

LLM은 토큰을 한 번에 하나씩 생성합니다(스트리밍). 연속 배칭(continuous batching)·KV 캐시 관리가 없으면 GPU 메모리가 곧 터집니다.

핵심 통찰

"모델을 띄우는 것"과 "모델을 서비스하는 것"은 다른 일입니다. 후자는 분산 시스템 엔지니어링의 영역이며, 매번 새로 만들 필요 없이 이미 검증된 인프라(=추론 서버)를 쓰는 게 합리적입니다.

NVIDIA Triton은 바로 이 6가지 문제를 한 번에 해결하기 위해 만들어진 추론 서버입니다.

Chapter 02

Triton이란 무엇인가

한 문장으로 정의하면: "어떤 프레임워크의 모델이든, 어떤 하드웨어 위에서든, 고성능 추론을 표준화된 방식으로 제공하는 오픈소스 서버"입니다.

한눈에 보는 Triton

정체
오픈소스 추론 서버 (BSD 3-Clause 라이선스). NVIDIA가 2018년에 처음 공개했고, 지금은 NVIDIA AI Enterprise의 핵심 구성요소입니다.
최신 버전
v2.66.0 (26.02 NGC 컨테이너, 2026년 초 기준). 매월 새로운 컨테이너가 NGC에 릴리즈됩니다.
최근 변화
2025년 3월, NVIDIA는 Triton을 NVIDIA Dynamo 플랫폼의 일부로 편입시키면서 공식 명칭을 "NVIDIA Dynamo Triton"으로 변경. 하지만 코어 소프트웨어는 동일한 오픈소스 프로젝트로 계속 유지·발전 중입니다.
지원 프레임워크
TensorRT, PyTorch, ONNX Runtime, TensorFlow, OpenVINO, Python, RAPIDS FIL, DALI, TensorRT-LLM, vLLM 등 10여 종 — 한 서버에서 동시 운영 가능.
지원 하드웨어
모든 NVIDIA GPU (Jetson 임베디드 ~ DGX/HGX 대형 클러스터) + x86 / ARM CPU + AWS Inferentia 가속기까지.
통신 프로토콜
HTTP/REST, gRPC, in-process C API — KServe v2 추론 프로토콜과 호환되어 대부분의 MLOps 도구와 곧바로 연동됩니다.
생태계에서의 위치
NVIDIA NIM 마이크로서비스의 기반 엔진이자, NeMo Framework·Riva·Maxine 등 NVIDIA AI 스택 전반의 추론 백본 역할을 합니다.
이름의 역사

Triton은 처음에 "TensorRT Inference Server"라는 이름으로 출발했습니다. 이후 TensorRT 외 다양한 백엔드를 지원하면서 "Triton Inference Server"로 이름이 바뀌었고, 2025년 3월에 다시 "NVIDIA Dynamo Triton"으로 리브랜딩되었습니다. 이 강의에서는 가장 널리 쓰이는 명칭인 "Triton"으로 계속 부르겠습니다.

헷갈리지 마세요: OpenAI Triton과는 다릅니다

OpenAI에도 "Triton"이라는 이름의 프로젝트가 있습니다. 이건 GPU 커널을 Python처럼 쓸 수 있게 해주는 커널 컴파일러 언어(예: FlashAttention 구현체)이며, NVIDIA Triton 추론 서버와는 이름만 같고 완전히 다른 도구입니다. 이 강의에서 다루는 것은 NVIDIA Triton, 즉 추론 서버 쪽입니다.

Chapter 03

Triton은 어떻게 동작하는가

Triton의 내부 구조를 알면, 왜 이렇게 빠른지 / 어떤 옵션을 어떻게 튜닝해야 하는지가 한 번에 이해됩니다. 4단계 데이터 흐름으로 정리해봅시다.

요청이 흘러가는 길

① Client

Python / C++ / Java

HTTP · gRPC · C API
② Frontend

Protocol Layer

KServe v2 호환
③ Scheduler

모델별 큐 & 배칭

Dynamic batcher
④ Backend

TRT · PyTorch · …

GPU 실제 추론
화살표는 양방향입니다. 백엔드가 결과를 만들어내면, 그 응답은 거꾸로 스케줄러 → 프론트엔드 → 클라이언트로 흘러갑니다. 모든 단계는 모델별로 독립이라, 한 모델이 느려도 다른 모델에는 영향이 없습니다.

각 단계를 더 자세히 들여다보면

① Client — 어떤 언어에서든 호출 가능

NVIDIA가 공식 제공하는 클라이언트 라이브러리는 Python·C++·Java가 있고, gRPC 코드 생성으로 Go·Scala도 사용 가능합니다. 또한 KServe v2 표준을 따르므로, 단순 curl로 HTTP/REST 호출도 됩니다. Triton 서버 자체에 직접 모델을 임베딩하는 in-process C API도 지원합니다 (예: 카메라 옆에서 추론하는 엣지 디바이스).

② Frontend — 프로토콜 변환 & 인증

HTTP/gRPC 요청을 받아 내부 표준 형식으로 변환합니다. 모델 메타데이터 조회(/v2/models/<name>), 헬스체크(/v2/health/ready), 통계(/v2/models/<name>/stats) 같은 관리 API도 모두 이 계층이 처리합니다.

③ Scheduler — Triton의 진짜 심장

들어오는 요청을 즉시 백엔드로 넘기지 않고, 모델별 큐에 잠시 모았다가 가장 효율적인 단위로 묶어서 백엔드에 전달합니다. 이 "잠시"의 시간이 뒤에서 다룰 Dynamic Batching의 핵심입니다. 스케줄러는 모델 단위로 4가지 정책을 가질 수 있습니다.

  • Default — 들어오는 즉시 1건씩 실행 (배칭 없음)
  • Dynamic Batching — 짧은 시간 안에 도착한 요청을 묶어서 한 번에 실행
  • Sequence Batching — 상태가 있는 모델(예: 스트리밍 ASR, RNN)을 위한 세션 유지 배칭
  • Ensemble Scheduling — 여러 모델을 파이프라인으로 묶어 한 번의 요청처럼 실행

④ Backend — 모델 형식별 실행 엔진

Triton은 각 프레임워크의 실행 엔진을 백엔드로 추상화합니다. PyTorch 모델은 LibTorch 백엔드가, ONNX 모델은 ONNX Runtime 백엔드가, TensorRT 모델은 TensorRT 런타임이 처리합니다. 백엔드는 C++ 플러그인 형태로 동적 로딩되므로, 커스텀 백엔드를 만들어 자체 추론 엔진을 끼워 넣을 수도 있습니다.

Triton 인스턴스 안에서 일어나는 일

하나의 Triton 서버 프로세스 안에는 다음 요소들이 동시에 존재합니다. 이것을 머릿속에 그릴 수 있으면 운영·디버깅이 훨씬 쉬워집니다.

Model Repository
파일시스템 기반 모델 저장소. 로컬 디렉토리 / S3 / GCS / Azure Blob을 지정할 수 있고, 서버 기동 시 자동 로딩되거나 API로 동적 로딩됩니다.
Model Manager
모델의 라이프사이클(load → ready → unload)을 관리. "explicit / polling / none" 세 가지 모드로 동작합니다.
Per-Model Scheduler
모델마다 독립적인 스케줄러 인스턴스가 있어, 동적 배칭·시퀀스 배칭 등을 모델 단위로 다르게 설정할 수 있습니다.
Instance Groups
한 모델을 GPU 0·1·2에 각각 1개씩 — 또는 같은 GPU에 N개씩 띄울 수 있습니다. Concurrent Model Execution의 단위입니다.
Metrics Endpoint
/metrics에서 Prometheus 포맷으로 추론 횟수, 지연시간, 큐 대기시간, GPU 메모리 사용량 등을 노출합니다.
Chapter 04

Triton의 7가지 핵심 기능

이 7가지는 "Triton이 왜 다른 서버보다 빠르고 운영하기 좋은가"를 설명하는 모든 것입니다. 각각은 독립적인 옵션이지만, 함께 쓰면 시너지가 폭발합니다.

FEATURE / 01

Multi-Framework

PyTorch, TensorFlow, ONNX, TensorRT, OpenVINO, Python, FIL, TensorRT-LLM, vLLM… 한 서버에서 동시 운영. 팀마다 좋아하는 프레임워크가 달라도 인프라는 하나로 통일됩니다.

→ 인프라 통합
FEATURE / 02

Dynamic Batching

짧은 시간(예: 5ms) 안에 도착한 개별 요청들을 자동으로 묶어서 한 번의 GPU 호출로 처리. 대부분의 워크로드에서 처리량 2~10배 향상.

→ 처리량 2~10x
FEATURE / 03

Concurrent Execution

같은 모델을 한 GPU에 N개 인스턴스로, 또는 여러 GPU에 분산 배치. CUDA Stream을 활용해 진짜 병렬로 추론합니다.

→ GPU 활용률 ↑
FEATURE / 04

Model Ensemble

"이미지 디코드 → 전처리 → 모델 A → 후처리 → 모델 B"를 한 번의 요청으로 실행. 네트워크 왕복이 사라지고 GPU/CPU 사이 데이터 이동이 최소화됩니다.

→ 파이프라인 최적화
FEATURE / 05

Business Logic Scripting

Python 코드로 조건문·반복문·외부 API 호출이 들어간 복잡한 파이프라인 작성. RAG·에이전트·multi-step LLM 호출에 필수입니다.

→ 복잡한 워크플로
FEATURE / 06

Model Management API

POST /v2/repository/models/<name>/load 한 줄로 무중단 모델 교체·A/B 테스트·카나리 배포. 운영팀이 사랑하는 기능입니다.

→ 무중단 배포
FEATURE / 07

Prometheus Metrics

추론 횟수·지연시간·큐 대기·GPU 메모리·캐시 적중률을 표준 Prometheus 포맷으로 노출. Grafana 대시보드·HPA 자동 스케일링과 즉시 연동됩니다.

→ 관측 가능성
FEATURE / 08

Streaming & LLM

gRPC bidirectional streaming으로 토큰 단위 응답 가능. TensorRT-LLM·vLLM 백엔드와 함께 쓰면 continuous batching까지 자동 적용됩니다.

→ LLM-ready
가장 큰 임팩트는 어디서?

처음 Triton을 도입할 때 한 가지 기능만 켠다면 답은 정해져 있습니다. Dynamic Batching입니다. 평균적으로 처리량 3~5배 향상이 거의 무료로 따라오기 때문이죠. 이 기능 하나를 정확히 이해하기 위해 별도 챕터를 마련했습니다. 다음 장으로 가시죠.

Chapter 05

한 발 깊이: Dynamic Batching

Triton의 모든 기능 중 가장 큰 성능 향상을 가져다주는 기능. 어떻게 동작하고, 어떤 트레이드오프가 있는지 시각적으로 살펴봅니다.

왜 배칭이 빨라질까?

GPU는 본질적으로 병렬 컴퓨터입니다. 100명이 동시에 일할 수 있는 공장과 같죠. 그런데 1명짜리 일감을 100번 보내면, 매번 공장을 가동·정지하는 비용(=CUDA 커널 런치 오버헤드)이 들고, 99명의 작업자는 매번 놀게 됩니다. 100명짜리 일감 하나로 묶어서 보내면 같은 시간에 100배 일이 됩니다.

배칭 없음 6개 요청 → 6번 GPU 호출

Dynamic Batching 5ms 윈도우 → 1번 호출

동작 원리: 요청이 도착해도 즉시 처리하지 않고, max_queue_delay_microseconds(예: 5,000μs = 5ms) 동안 추가 요청을 기다립니다. 그 시간 안에 들어온 요청들을 모아 preferred_batch_size(예: 8, 16, 32)에 가장 가까운 배치를 만들어 한 번에 GPU로 보냅니다.

config.pbtxt에서 켜는 법

모델 설정 파일에 단 몇 줄을 추가하면 됩니다. 정말 이게 전부입니다.

model_repository/resnet50/config.pbtxt 
name: "resnet50"
platform: "tensorrt_plan"
max_batch_size: 32

# ↓↓↓ 이 블록 하나로 Dynamic Batching 활성화
dynamic_batching {
  preferred_batch_size: [ 4, 8, 16 ]
  max_queue_delay_microseconds: 5000
}

input [
  { name: "input", data_type: TYPE_FP32, dims: [ 3, 224, 224 ] }
]
output [
  { name: "output", data_type: TYPE_FP32, dims: [ 1000 ] }
]

# ↓ 같은 GPU에 인스턴스 2개 (Concurrent Execution)
instance_group [
  { count: 2, kind: KIND_GPU, gpus: [ 0 ] }
]

핵심 트레이드오프: 처리량 vs 지연시간

max_queue_delay_microseconds를 늘리면 처리량은 올라가지만 지연시간도 늘어납니다. 이 둘의 균형점은 워크로드 특성에 따라 다릅니다.

  • 실시간 응답 (챗봇, 검색): 5,000~10,000μs (5~10ms) 권장
  • 배치 처리 (오프라인 임베딩, 분석): 50,000μs 이상도 OK
  • 스트리밍 LLM: 일반 동적 배칭 대신 continuous batching (vLLM/TRT-LLM 백엔드 자동 적용)
최적값 찾기: Model Analyzer

NVIDIA가 공식 제공하는 Model Analyzer 도구를 쓰면, 다양한 batch size·instance count·delay 조합을 자동으로 벤치마크하고 "이 워크로드에 가장 좋은 설정"을 제안해줍니다. 처음부터 손으로 튜닝할 필요가 없습니다.

Chapter 06

Triton의 백엔드 생태계

Triton이 "프레임워크 독립적"일 수 있는 비밀은 플러그형 백엔드에 있습니다. 각 백엔드는 특정 모델 형식을 실행하는 전문 엔진. 어떤 게 있고 언제 무엇을 쓰는지 살펴봅시다.

주요 백엔드 11종

탭을 눌러 각 백엔드의 특징을 확인하세요.

TensorRT — GPU 추론의 최강 카드

NVIDIA GPU 전용 가장 빠름 .plan / .engine

NVIDIA가 만든 GPU 추론 컴파일러의 출력물을 직접 실행합니다. FP16·INT8·FP8 양자화, 레이어 fusion, 커널 자동 튜닝 등 모든 최적화가 컴파일 타임에 적용된 .plan 파일을 그대로 로드합니다.

언제 쓰나: 비전 모델·중소형 트랜스포머 등 모델 구조가 안정적이고 최고 성능이 필요할 때. PyTorch보다 보통 2~5배 빠릅니다.

TensorRT-LLM — LLM 전용 TensorRT

LLM 특화 continuous batching paged KV cache tensor parallel

대규모 언어 모델만을 위한 TensorRT의 확장판. continuous batching(들어오고 나가는 시퀀스를 토큰 단위로 동적 재배치), paged KV cache(메모리 단편화 최소화), tensor/pipeline parallelism, speculative decoding 등 LLM 추론의 모든 최적화가 들어 있습니다.

언제 쓰나: Llama·Qwen·Mistral 등 대형 LLM을 NVIDIA GPU에서 최고 성능으로 서비스할 때. NIM 마이크로서비스의 기본 백엔드이기도 합니다.

vLLM — 가장 인기 있는 오픈소스 LLM 엔진

PagedAttention OpenAI API 호환 v1 엔진 기본

UC 버클리에서 시작된 LLM 추론 엔진. PagedAttention으로 KV 캐시를 OS의 가상 메모리처럼 관리해 메모리 효율이 매우 높습니다. Triton 25.10부터 vLLM V1 엔진이 기본값입니다.

언제 쓰나: TensorRT-LLM 컴파일 단계 없이 HuggingFace 모델을 즉시 서비스하고 싶을 때. OpenAI 호환 API가 필요할 때 특히 편리합니다.

PyTorch (LibTorch) — 가장 친숙한 옵션

.pt / TorchScript PyTorch 2.0+

PyTorch 모델을 torch.jit.script 또는 torch.jit.trace로 직렬화한 후 LibTorch C++ 런타임이 실행합니다. 정확도는 PyTorch와 동일하고, 추가 변환 작업이 거의 없습니다.

언제 쓰나: 빠른 프로토타이핑, 자주 바뀌는 모델, TensorRT로 변환하기 어려운 동적 그래프. 성능 < 편의성인 시나리오에 적합.

ONNX Runtime — 프레임워크 중립 표준

.onnx CPU + GPU 교차 플랫폼

ONNX는 PyTorch·TensorFlow·Sklearn 등 어떤 프레임워크에서도 export 가능한 중립 모델 형식입니다. ONNX Runtime은 이를 CUDA·TensorRT·OpenVINO·DirectML 등 다양한 실행 프로바이더로 실행합니다.

언제 쓰나: GPU와 CPU 모두 지원해야 하거나, 학습 프레임워크와 추론 환경을 분리하고 싶을 때. 클라우드와 엣지에서 같은 모델을 쓰는 시나리오에 강력합니다.

TensorFlow — SavedModel / GraphDef

SavedModel .pb

TF 1.x의 GraphDef와 TF 2.x의 SavedModel을 모두 지원. 내부적으로 TF C++ 런타임이 실행합니다. TF Serving을 별도로 띄울 필요가 없어집니다.

언제 쓰나: 기존에 TF Serving으로 운영하던 모델을 Triton으로 이전하면서 다른 프레임워크의 모델과 함께 운영하고 싶을 때.

OpenVINO — Intel CPU 최적화

Intel CPU 특화 .xml + .bin

Intel이 만든 CPU 추론 최적화 도구. GPU가 없는 환경에서도 AVX-512·VNNI·INT8 양자화를 활용해 빠른 추론이 가능합니다.

언제 쓰나: CPU만 있는 엣지/온프레미스 환경, 비용 절감용 인스턴스.

Python — 모든 것을 가능하게 하는 만능 백엔드

완전 자유 BLS 지원

model.py 안에 임의의 Python 코드를 작성할 수 있습니다. HuggingFace Transformers를 직접 import해서 쓰거나, 외부 API를 호출하거나, 다른 모델을 호출(BLS)하는 등 "표준화하기 어려운 로직"을 다 담을 수 있습니다.

언제 쓰나: 전처리/후처리, RAG 파이프라인, 에이전트 워크플로, 모델을 직접 정의해 쓰고 싶을 때. 유연성의 끝판왕.

FIL — 트리 기반 ML 모델 전용

XGBoost LightGBM sklearn RF GPU 가속

Forest Inference Library 백엔드. XGBoost·LightGBM·sklearn RandomForest·cuML 같은 트리 기반 ML 모델을 GPU에서 가속 실행합니다. 추천 시스템·금융 모델·이상 탐지에서 인기.

언제 쓰나: 대규모 트리 모델을 딥러닝 모델과 같은 서버에서 서비스하고 싶을 때. 실시간 추천·랭킹 파이프라인에서 자주 사용됩니다.

DALI — GPU 데이터 전처리 가속

GPU 전처리 이미지/비디오/오디오

JPEG/H.264 디코딩·resize·normalize 같은 전처리를 GPU에서 수행하는 NVIDIA 라이브러리. CPU 전처리가 GPU 추론보다 느려서 병목이 되는 경우를 해결합니다.

언제 쓰나: 4K 영상·고해상도 이미지 추론에서 CPU가 병목일 때. DALI → Triton Ensemble로 묶으면 네트워크 → GPU → 응답이 한 줄로 흐릅니다.

선택 가이드 (2026년 기준)

비전 모델이라면 → TensorRT (최고 성능) 또는 ONNX Runtime (이식성)
대형 LLM이라면 → TensorRT-LLM (NVIDIA 전용 최고 성능) 또는 vLLM (편의성)
트리 ML 모델이라면 → FIL
커스텀 로직·RAG가 필요하면 → Python + BLS
여러 모델을 파이프라인으로 묶고 싶으면 → 위 백엔드들을 Ensemble로 연결

Chapter 07 — Model Repository

모델 리포지토리 — Triton과 대화하는 유일한 방법

Triton에 모델을 올리는 방법은 딱 한 가지입니다 — 정해진 디렉터리 구조로 파일을 배치하기. 데이터베이스도, 별도 등록 API도 없습니다. 폴더가 곧 API입니다.

표준 디렉터리 구조

리포지토리 루트 아래에 모델명/버전번호/모델파일 형태로 배치하고, 모델명 폴더에 config.pbtxt를 두는 것이 기본 규칙입니다.

model_repository/ ├── resnet50/ # 모델명 = HTTP/gRPC 엔드포인트 이름 │ ├── config.pbtxt # 모델 설정 (필수) │ ├── 1/ # 버전 번호 (정수) │ │ └── model.plan # TensorRT 엔진 파일 │ └── 2/ # 새 버전을 추가하면 hot-reload │ └── model.plan ├── bert_qa/ │ ├── config.pbtxt │ └── 1/ │ └── model.onnx # ONNX Runtime 백엔드 ├── llama3_8b/ │ ├── config.pbtxt │ └── 1/ │ └── model.py # vLLM 또는 Python 백엔드 └── pipeline_rag/ # 앙상블 모델 (실제 가중치 없음) ├── config.pbtxt └── 1/ # 빈 폴더지만 반드시 존재해야 함
핵심 규칙 세 가지

① 모델명 = 엔드포인트: 폴더 이름이 그대로 POST /v2/models/resnet50/infer의 URL이 됩니다.
② 버전 번호는 정수: 1, 2, 3… Triton은 기본적으로 가장 높은 번호를 서비스합니다 (정책 변경 가능).
③ Hot reload 지원: 서버 실행 중에 새 버전 폴더를 추가하면 다운타임 없이 로드됩니다.

config.pbtxt 한 줄씩 뜯어보기

모델의 입출력 스펙·백엔드·실행 옵션을 모두 담는 설정 파일입니다. Protobuf 텍스트 형식으로 작성하고, 처음엔 길어 보이지만 익숙해지면 5분이면 씁니다. 

# resnet50/config.pbtxt
name: "resnet50"                           # 폴더명과 일치해야 함
platform: "tensorrt_plan"                 # 사용할 백엔드
max_batch_size: 128                       # 한 번에 처리 가능한 최대 배치

input [
  {
    name: "input_image"
    data_type: TYPE_FP32
    dims: [ 3, 224, 224 ]               # 배치 차원 제외
  }
]
output [
  {
    name: "probabilities"
    data_type: TYPE_FP32
    dims: [ 1000 ]                      # 1000 클래스
  }
]

# Dynamic Batching 활성화
dynamic_batching {
  preferred_batch_size: [ 8, 16, 32 ]    # 선호 배치 크기
  max_queue_delay_microseconds: 5000    # 최대 대기 5ms
}

# GPU 인스턴스 2개를 동시에 실행 (동시성 ↑)
instance_group [
  { count: 2, kind: KIND_GPU }
]

Ensemble — 모델 여러 개를 하나의 파이프라인으로

Ensemble은 실제 가중치가 없는 가상의 모델입니다. config.pbtxt 안에서 다른 모델들의 입출력을 연결하면, 클라이언트 입장에선 단 한 번의 요청으로 전처리 → 추론 → 후처리가 한 줄로 흐릅니다. 

# pipeline_rag/config.pbtxt — 임베딩 → 검색 → LLM 응답
name: "pipeline_rag"
platform: "ensemble"
max_batch_size: 16

input  [ { name: "user_query", data_type: TYPE_STRING, dims: [ 1 ] } ]
output [ { name: "answer",     data_type: TYPE_STRING, dims: [ 1 ] } ]

ensemble_scheduling {
  step [
    {
      model_name: "embedder"                          # 1단계
      input_map  { key: "text"      value: "user_query" }
      output_map { key: "embedding" value: "q_vec" }
    },
    {
      model_name: "vector_search"                     # 2단계
      input_map  { key: "query_vec" value: "q_vec" }
      output_map { key: "context"   value: "retrieved" }
    },
    {
      model_name: "llama3_8b"                          # 3단계
      input_map  { key: "prompt"    value: "retrieved" }
      output_map { key: "response"  value: "answer" }
    }
  ]
}

Chapter 09 — Real World

활용 사례 — 현장에선 이렇게 씁니다

Triton은 실험실에서 끝나는 도구가 아닙니다. Microsoft·Snap·Naver·Kakao·KT 등 수많은 조직이 프로덕션에서 운영 중입니다. 대표적인 4가지 패턴을 살펴봅니다.

대화형 LLM 서비스 (챗봇·코드 어시스턴트)

TensorRT-LLM vLLM 백엔드 스트리밍 Continuous Batching

Llama·Qwen·Mistral 같은 LLM을 Triton + TensorRT-LLM 백엔드로 서비스합니다. 토큰을 SSE/gRPC streaming으로 한 글자씩 흘려보내고, 동시에 들어오는 여러 사용자 요청을 continuous batching으로 묶어 GPU 활용률을 90% 이상 끌어올립니다.

→
토큰 스트리밍 응답

성능 포인트: KV 캐시 관리·In-flight Batching·Speculative Decoding 같은 LLM 전용 기법이 TRT-LLM 백엔드에 모두 내장되어 있어, 코드 변경 없이 활성화만 하면 됩니다.

실시간 컴퓨터 비전 (CCTV·자율주행·품질검사)

TensorRT DALI 전처리 Ensemble 멀티스트림

영상 프레임이 초당 수십 장씩 들어오는 환경. DALI(GPU 전처리) → ResNet/YOLO(검출) → 후처리를 Ensemble로 묶어 네트워크 진입부터 응답까지 한 번에 처리합니다. 동일 GPU에 여러 카메라 스트림을 동시에 태우려면 instance group으로 인스턴스를 늘립니다.

CCTV 프레임
→
DALI 디코드/리사이즈
→
YOLO (TensorRT)
→
NMS 후처리

왜 Triton: CPU 전처리가 GPU 추론보다 느린 병목을 DALI로 제거하고, 여러 모델을 GPU 메모리에 함께 올려 한 카드로 여러 카메라를 커버할 수 있습니다.

실시간 추천·랭킹 (이커머스·미디어·광고)

FIL 백엔드 XGBoost Python 백엔드 초저지연

추천 시스템은 트리 기반 ML 모델(XGBoost·LightGBM)과 딥러닝 모델(DLRM·BERT4Rec)이 함께 쓰이는 영역. Triton은 FIL 백엔드로 트리 모델을 GPU 가속하고, DL 모델은 TensorRT로 묶어 한 서버에서 1ms 단위 응답을 제공합니다.

사용자 컨텍스트
→
후보 임베딩(BERT)
→
XGBoost 랭커 (FIL)
→
Top-K 상품

왜 Triton: 트리 모델을 CPU에서 돌리던 시절보다 5~10배 빠른 응답이 가능하고, DL 임베딩과 같은 GPU에서 메모리를 공유합니다.

RAG 파이프라인 (사내 검색·문서 QA)

Ensemble BLS 임베딩 + LLM Vector DB 연동

임베딩 모델 → 벡터 검색 → LLM 응답으로 이어지는 RAG는 Triton의 Ensemble/BLS가 빛을 발하는 영역입니다. 벡터 검색은 보통 외부(Milvus·Qdrant)에서 수행하므로 Python 백엔드 + BLS로 호출 흐름을 만들고, 임베딩과 LLM은 같은 GPU에서 함께 돌립니다.

사용자 질문
→
임베딩(BGE)
→
벡터 DB 조회
→
LLM (TRT-LLM)
→
스트리밍 답변

왜 Triton: 임베딩 모델과 LLM이 한 서버에 있으면 네트워크 hop이 줄고, BLS로 분기·재시도 로직을 서버 안에서 처리해 클라이언트는 단 한 번의 요청만 보냅니다.

Chapter 10 — Production

Kubernetes 위에서 운영하기 — 이기종 클러스터 실전

프로덕션의 Triton은 거의 항상 Kubernetes 위에서 돕니다. 오토스케일·롤링 업데이트·관측성을 K8s에 맡기고, Triton은 추론 엔진에만 집중하게 하는 구조입니다. ARM(DGX Spark) + x86(ThinkStation) 같은 이기종 환경에서도 노드 라벨링으로 깔끔하게 운영할 수 있습니다.

① 사전 준비 — NVIDIA GPU Operator

K8s 노드에서 GPU를 쓰려면 드라이버·CUDA·container toolkit·device plugin이 모두 맞물려야 합니다. 이걸 하나의 Operator로 묶어주는 것이 NVIDIA GPU Operator입니다. Helm 한 줄로 설치 가능합니다. 

helm install gpu-operator nvidia/gpu-operator \
  -n gpu-operator --create-namespace \
  --set driver.enabled=true \
  --set toolkit.enabled=true

② Triton Helm Chart values.yaml

NVIDIA 공식 Helm 차트를 활용해 Triton을 배포합니다. 핵심은 nodeSelector·tolerations·resources로 어느 노드에 떨어질지를 명시하는 것입니다. 

image:
  repository: nvcr.io/nvidia/tritonserver
  tag: "26.02-py3"

modelRepositoryPath: "s3://my-models/repo"

replicaCount: 2

resources:
  limits:
    nvidia.com/gpu: 1
    memory: "32Gi"
  requests:
    cpu: "4"
    memory: "16Gi"

# DGX Spark에만 GPU 워크로드 배치
nodeSelector:
  node-role: "gpu-arm64"
  kubernetes.io/arch: "arm64"

tolerations:
  - key: "nvidia.com/gpu"
    operator: "Exists"
    effect: "NoSchedule"

# Prometheus 메트릭 노출
service:
  ports:
    http: 8000
    grpc: 8001
    metrics: 8002
이기종 클러스터(ARM + x86) 운영 팁

① 멀티 아키텍처 이미지: Triton의 NGC 공식 이미지는 arm64/amd64 매니페스트가 함께 제공됩니다. 같은 태그를 양쪽 노드에서 그대로 쓸 수 있어요.
② 노드 라벨: kubectl label node dgx-spark node-role=gpu-arm64 같은 식으로 GPU 노드를 명확히 표시하세요.
③ Tolerations + Taints: CPU 작업(Argo Workflows·Airflow scheduler)이 실수로 DGX에 떨어지지 않게 GPU 노드에 taint를 걸어두면 안전합니다.

③ KServe로 한 단계 더 — 선언적 추론 서비스

Helm 차트 대신 KServe를 쓰면 InferenceService CRD 하나로 모델을 선언할 수 있고, Knative 기반 0까지의 오토스케일·카나리 배포·트래픽 분배가 표준화됩니다. 

apiVersion: serving.kserve.io/v1beta1
kind: InferenceService
metadata:
  name: llama3-service
  namespace: ml-prod
spec:
  predictor:
    minReplicas: 1
    maxReplicas: 8
    nodeSelector:
      node-role: gpu-arm64
    triton:
      storageUri: "s3://my-models/llama3"
      runtimeVersion: "26.02-py3"
      resources:
        limits:
          nvidia.com/gpu: 1

④ 관측성 — Prometheus · Grafana · DCGM

Triton 메트릭 (포트 8002)

nv_inference_request_duration_us, nv_inference_queue_duration_us, nv_gpu_utilization 등 모델·요청 단위 지표를 Prometheus 포맷으로 직접 노출합니다.

DCGM Exporter

GPU 자체의 SM 사용률·메모리·전력·온도 등 하드웨어 지표. GPU Operator와 함께 설치되며, Triton 지표와 묶어 보면 병목이 모델인지 GPU인지가 보입니다.

Grafana 대시보드

NVIDIA가 공식 Triton 대시보드를 공개합니다(ID: 12239). 요청량·지연·큐·GPU 활용률을 한 화면에서 확인.

오토스케일 신호

HPA에 nv_inference_queue_duration_us 같은 커스텀 메트릭을 연결하면 큐 적체를 기준으로 스케일 아웃되어, CPU 사용률보다 훨씬 정확합니다.

Chapter 11 — The Next Chapter

NVIDIA Dynamo로의 진화 — 대규모 분산 LLM의 시대

2025년 3월, NVIDIA는 차세대 추론 플랫폼 NVIDIA Dynamo를 발표하고 Triton을 그 안에 통합했습니다. 공식 명칭은 이제 "NVIDIA Dynamo Triton"입니다. Triton이 사라진 게 아니라, 더 큰 그림의 한 부분이 된 것입니다.

NVIDIA Dynamo란?

수천 개 GPU를 가로지르는 분산 LLM 추론에 특화된 오픈소스 플랫폼. 단일 노드 중심으로 설계된 Triton의 한계를 넘어, 대규모 LLM 서비스에서 발생하는 새로운 문제들 (KV 캐시 라우팅·disaggregated serving·GPU 풀 관리)을 해결합니다.

30×
DeepSeek-R1 + Blackwell 처리량 향상 (NVIDIA 자체 측정)
2×
Hopper GPU에서 Llama 모델 처리량 향상
∞
노드 수에 따라 선형으로 확장

Dynamo가 새로 들고 온 것들

01

Disaggregated Serving

LLM 추론을 Prefill(긴 입력 처리) 단계와 Decode(토큰 생성) 단계로 나눠 각기 다른 GPU 그룹에 배치합니다. 두 단계의 GPU 사용 패턴이 다르기 때문에, 분리하면 처리량이 크게 오릅니다.

02

KV-Cache-Aware Routing

같은 사용자의 후속 요청을 이전 대화의 KV 캐시를 보유한 GPU로 라우팅합니다. 매번 캐시를 다시 계산하는 비용이 사라져, 멀티턴 대화에서 큰 이득을 봅니다.

03

GPU Resource Planner

트래픽 패턴을 보고 Prefill ↔ Decode GPU의 비율을 동적으로 조정합니다. 새벽엔 Decode 위주, 낮엔 Prefill 위주… 같은 식의 자동 재조정이 가능합니다.

04

NIXL — GPU-to-GPU 통신

Inference Transfer Library. GPU 간 KV 캐시·중간 텐서 전송을 NVLink/NVSwitch/RDMA로 최적화해, disaggregated 구조의 통신 오버헤드를 최소화합니다.

Triton과 Dynamo의 관계

혼란스러울 수 있는데, 정리하면 이렇습니다.

  • Triton은 여전히 살아 있습니다. 단일 노드/소규모 클러스터에서 다양한 모델을 서빙하기 위한 검증된 추론 엔진입니다.
  • Dynamo는 그 위에서 여러 노드에 걸친 LLM 분산 추론을 조율하는 상위 레이어입니다.
  • 실제로 Dynamo는 내부적으로 vLLM·SGLang·TensorRT-LLM 같은 엔진을 워커로 묶어 운영하며, Triton 또한 백엔드로 동작할 수 있습니다.
지금 무엇을 배워야 할까?

Triton부터 익히는 게 정답입니다. Dynamo는 수천 GPU 규모의 LLM 서비스를 운영하는 조직을 위한 도구이고, 그 안의 핵심 개념(model repository·config.pbtxt·dynamic batching·instance group·ensemble)은 모두 Triton에서 그대로 가져온 것입니다. Triton을 알면 Dynamo도 자연스럽게 이해됩니다.

Chapter 12 — Check Yourself

정리 & 퀴즈 — 5문제로 점검하기

여기까지 읽었다면 Triton의 큰 그림은 잡혔을 거예요. 짧은 퀴즈로 핵심 개념을 굳혀봅니다. 보기 클릭 → 해설 자동 표시.

Question 01 / 05
동시에 들어온 작은 요청들을 모아서 GPU에 한 번에 처리하게 하는 Triton의 기능은?
A
Instance Group
B
Dynamic Batching
C
Model Ensemble
D
Business Logic Scripting (BLS)
정답: B. Dynamic Batching은 짧은 대기 시간(보통 수 ms) 안에 도착한 요청을 자동으로 묶어 한 배치로 처리해, GPU의 병렬성을 최대한 활용합니다. config.pbtxt의 dynamic_batching 블록 한 줄로 활성화됩니다.
Question 02 / 05
Triton에 모델을 등록하는 표준 방법은?
A
REST API로 모델 파일을 업로드한다
B
관리자 콘솔에서 GUI로 등록한다
C
지정된 디렉터리 구조에 파일을 배치한다 (Model Repository)
D
전용 데이터베이스에 모델 메타데이터를 INSERT한다
정답: C. Triton은 별도의 등록 API가 없습니다. 모델명/버전/모델파일 구조의 폴더에 config.pbtxt만 두면 끝. S3·GCS·Azure Blob 같은 클라우드 스토리지도 리포지토리로 직접 지정 가능합니다.
Question 03 / 05
XGBoost·LightGBM 같은 트리 기반 모델을 Triton에서 GPU 가속으로 서비스하려면 어떤 백엔드를 써야 할까?
A
TensorRT
B
ONNX Runtime
C
Python Backend
D
FIL (Forest Inference Library)
정답: D. FIL 백엔드는 XGBoost·LightGBM·sklearn RandomForest·cuML 등 트리 기반 모델을 GPU에서 가속 실행합니다. 추천 시스템·금융 모델·이상 탐지에서 자주 쓰이며, 같은 서버에서 딥러닝 모델과 함께 서비스할 수 있습니다.
Question 04 / 05
아래 중 Triton과 vLLM의 관계를 가장 정확하게 설명한 것은?
A
vLLM이 Triton의 상위 호환이라 Triton은 곧 사라진다
B
Triton은 vLLM을 백엔드 중 하나로 사용할 수 있다 (조합 가능)
C
둘은 호환되지 않는 경쟁 솔루션이라 양자택일이다
D
vLLM은 Triton의 한 모듈로만 존재한다
정답: B. Triton은 다중 프레임워크 서버, vLLM은 LLM 전용 추론 엔진입니다. Triton의 vLLM 백엔드를 쓰면 vLLM의 PagedAttention 성능을 누리면서 Triton의 batching·앙상블·관측성을 함께 활용할 수 있어, 실무에서 둘을 묶어 쓰는 경우가 많습니다.
Question 05 / 05
NVIDIA Dynamo가 Triton 대비 새로 제공하는 핵심 기능이 아닌 것은?
A
Disaggregated Serving (Prefill/Decode 분리)
B
KV-Cache-Aware Routing
C
GPU Resource Planner (동적 GPU 그룹 재배분)
D
config.pbtxt 기반 모델 설정
정답: D. config.pbtxt 모델 설정은 원래 Triton의 핵심 개념입니다. Dynamo는 그 위에서 수천 GPU 분산 LLM 추론을 위한 상위 오케스트레이션 (disaggregated serving·KV cache routing·GPU planner·NIXL)을 추가했습니다.

최종 요약 — 한 페이지 치트시트

  • Triton은 다중 프레임워크 추론 서버: TensorRT·PyTorch·ONNX·vLLM·XGBoost 등 거의 모든 모델을 하나의 API로 서비스합니다.
  • 핵심 자산은 Model Repository: 모델명/버전/파일 + config.pbtxt 하나로 등록 끝.
  • GPU를 짜내는 3대 기능: Dynamic Batching · Concurrent Execution(Instance Group) · Ensemble.
  • K8s 위에서는 보통 KServe + Triton 조합으로 선언적 운영. 메트릭은 포트 8002로 자동 노출.
  • 대규모 LLM 분산 추론은 Dynamo가 차세대 답이지만, 그 안의 모든 개념은 Triton에서 출발합니다.
