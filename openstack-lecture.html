Chapter 01 · 개념

OpenStack이란 무엇인가?

한 문장으로: "내 서버실에 직접 깔아서 쓰는 오픈소스 클라우드 운영체제(Cloud OS)" 입니다.

여러분이 AWS 콘솔에서 버튼을 눌러 가상 서버(EC2)를 띄우고, 디스크(EBS)를 붙이고, 네트워크를 만드는 경험을 해봤다고 합시다. 그 모든 것을 가능하게 하는 소프트웨어 덩어리가 AWS 데이터센터 안에서 돌아가고 있습니다. OpenStack은 바로 그 "소프트웨어 덩어리"에 해당하는 것을, 누구나 무료로 가져다 자기 서버에 설치할 수 있게 만든 오픈소스 프로젝트입니다.

비유로 이해하기

물리 서버 수백 대가 있는 창고가 있다고 해봅시다. 그냥 두면 그냥 "쇳덩이 더미"입니다. OpenStack은 이 쇳덩이들 위에 깔리는 관리자(운영체제)입니다. 흩어진 CPU·메모리·디스크·네트워크를 하나의 거대한 자원 풀(pool)로 묶고, 사용자가 "VM 2GB짜리 하나 주세요"라고 요청하면 알아서 적당한 서버에 배치해 줍니다. 즉 Linux가 한 대의 PC를 관리하는 OS라면, OpenStack은 데이터센터 전체를 관리하는 OS입니다.

핵심 정체성 세 가지

IaaS

인프라를 서비스로

컴퓨팅·스토리지·네트워크라는 "인프라"를 API와 셀프서비스로 제공합니다. 클라우드의 가장 밑단 계층입니다.

Open

완전 오픈소스

Apache 2.0 라이선스. 코드가 공개돼 있어 특정 벤더에 묶이지 않고, 직접 수정·확장할 수 있습니다.

Private

주로 사설 클라우드

내 데이터센터·온프레미스에 구축하는 프라이빗 클라우드의 대표 주자. 데이터 주권과 규제 대응에 유리합니다.

※ 헷갈리기 쉬운 점: OpenStack은 "하이퍼바이저"가 아닙니다

OpenStack은 VM을 직접 만들지 않습니다. 실제 가상화는 KVM, Xen 같은 하이퍼바이저가 담당하고, OpenStack은 그 위에서 "어느 서버에, 어떤 사양으로, 누구를 위해 VM을 만들지"를 지휘·조율(orchestrate)하는 상위 관리 계층입니다. 지휘자와 연주자의 관계라고 보면 됩니다.

짧은 역사

2010년, 미국 항공우주국(NASA)의 컴퓨팅 프로젝트(Nova)와 클라우드 호스팅 기업 Rackspace의 스토리지 프로젝트(Swift)가 합쳐지며 OpenStack이 탄생했습니다. 이후 OpenInfra Foundation(구 OpenStack Foundation)이 운영을 맡아, 매년 두 차례씩 알파벳 순서의 코드네임(예: Antelope, Bobcat, Caracal, Dalmatian…)으로 새 버전을 발표하는 거대한 글로벌 커뮤니티 프로젝트로 성장했습니다.

Chapter 02 · 동기

왜 굳이 OpenStack을 쓸까?

"그냥 AWS 쓰면 되지 않나?" — 이 질문에 답하는 것이 OpenStack을 이해하는 핵심입니다.

퍼블릭 클라우드(AWS·Azure·GCP)는 편리하지만, 모든 조직에 정답은 아닙니다. 다음과 같은 상황에서 OpenStack 같은 프라이빗 클라우드가 선택됩니다.

①

데이터 주권 · 규제

금융·공공·의료·통신처럼 데이터를 외부에 둘 수 없는 산업. 물리적으로 내 건물 안에 데이터를 보관해야 하는 컴플라이언스 요구를 만족시킵니다.

②

비용 예측 가능성

대규모·장기 운영 시, 사용량 기반 과금보다 자체 데이터센터가 총소유비용(TCO) 면에서 유리해지는 손익분기점이 존재합니다.

③

벤더 종속 회피

특정 클라우드의 독점 API에 코드와 운영을 묶지 않습니다. 오픈 표준 API로 멀티·하이브리드 클라우드 전략을 짤 수 있습니다.

④

완전한 제어권

커널 파라미터, 네트워크 토폴로지, 하드웨어 선택까지 바닥부터 제어. 통신사 NFV처럼 극단적 성능·지연 요구에 맞춰 튜닝할 수 있습니다.

⚠ 솔직한 단점도 알아두기

OpenStack은 강력한 만큼 구축·운영 난이도가 높습니다. 30개가 넘는 서비스가 서로 얽혀 동작하므로, 설치·업그레이드·트러블슈팅에 깊은 전문성이 필요합니다. "내가 직접 AWS를 만들어 운영하는 것"과 같다고 생각하면 됩니다. 그래서 보통 전담 플랫폼 엔지니어 조직이나 MSP의 도움을 받습니다.

Chapter 03 · 구성 요소

핵심 컴포넌트 탐험기

OpenStack은 "하나의 프로그램"이 아니라, 각자 한 가지 일을 잘하는 여러 서비스의 모음입니다. 모든 서비스는 코드네임(사람 이름 느낌)을 갖습니다. 아래 칩을 눌러 각 컴포넌트가 무슨 일을 하는지 살펴보세요.

✓ 처음엔 이 7개만 기억하세요

전부 외울 필요 없습니다. 클라우드를 굴리는 데 거의 항상 쓰이는 핵심은 Keystone(인증) · Nova(서버) · Neutron(네트워크) · Cinder(블록스토리지) · Glance(이미지) · Swift(오브젝트스토리지) · Horizon(웹화면) 입니다. 나머지는 필요할 때 하나씩 붙여 나갑니다.

Chapter 04 · 아키텍처

물리적으로는 어떻게 생겼을까: 노드 구성

OpenStack을 실제로 깔면 서버들이 역할에 따라 나뉩니다. 각 박스를 클릭해 무슨 역할인지 확인해 보세요.

제어 평면 (두뇌) 컴퓨트 (VM 실행) 네트워크 (통신) 스토리지 (저장)
위 다이어그램에서 노드를 클릭하면 역할 설명이 여기에 표시됩니다.
◎ 두 개의 평면(Plane)으로 이해하기

컨트롤 플레인은 "무엇을 할지 결정하는 두뇌"(API·스케줄러·DB·메시지큐)이고, 데이터 플레인은 "실제로 일하는 손발"(VM이 도는 컴퓨트, 트래픽이 흐르는 네트워크, 데이터가 쌓이는 스토리지)입니다. 작은 실습 환경에서는 이 모든 역할을 서버 한 대(All-in-One)에 몰아넣기도 합니다.

Chapter 05 · 동작 원리

VM 한 대가 만들어지기까지

"VM 만들기" 버튼 하나를 누르면 내부에서 컴포넌트들이 어떻게 협력할까요? 한 단계씩 따라가 봅시다.

1

요청 & 인증

사용자가 Horizon(웹) 또는 CLI에서 "이 이미지로, 이 사양의 VM을 만들어줘"라고 요청합니다. 가장 먼저 Keystone이 "이 사람이 그럴 권한이 있는가?"를 확인하고 토큰을 발급합니다.

▸ 관여: Keystone (인증)
한 줄 요약

인증(Keystone) → 일감 접수(Nova API) → 배치 결정(Scheduler) → 메시지로 지시 전달(RabbitMQ) → 이미지 받아 디스크 만들고(Glance·Cinder) → 네트워크 연결(Neutron) → 부팅. 이 협주(協奏)가 보통 수십 초 안에 끝납니다.

Chapter 06 · 사용법

실제로는 어떻게 다룰까?

OpenStack을 조작하는 길은 크게 셋입니다. 웹 대시보드(Horizon), 명령줄(CLI), 그리고 코드형 인프라(Heat). 같은 "VM 만들기" 작업을 세 방식으로 비교해 보세요.

① Horizon (웹)
② CLI (명령줄)
③ Heat (코드형 인프라)
# Horizon 대시보드는 브라우저로 접속하는 웹 GUI입니다.
# 클릭만으로 자원을 만들 수 있어 입문·운영자 모두에게 친숙합니다.

1. 브라우저로 접속        → http://<컨트롤노드-IP>/dashboard
2. 로그인               → 도메인 / 사용자 / 비밀번호 (Keystone 인증)
3. 좌측 메뉴            → Compute > Instances > [Launch Instance]
4. 이미지 선택          → Ubuntu 22.04 (Glance에 등록된 이미지)
5. Flavor(사양) 선택    → m1.small (1 vCPU / 2GB RAM)
6. 네트워크 선택        → private-net (Neutron 네트워크)
7. 키페어·보안그룹 지정  → SSH 키 / 22·80 포트 허용
8. [Launch] 클릭       → 수십 초 후 VM 부팅 완료 ✓

# 장점: 직관적 / 단점: 반복 작업·자동화에는 비효율적
# OpenStack CLI는 모든 서비스를 하나의 'openstack' 명령으로 다룹니다.
# 먼저 인증정보를 환경변수로 불러옵니다 (RC 파일).
source admin-openrc.sh

# 사용 가능한 이미지 / 사양 / 네트워크 조회
openstack image list
openstack flavor list
openstack network list

# VM(인스턴스) 한 대 생성
openstack server create \
  --image "Ubuntu-22.04" \
  --flavor m1.small \
  --network private-net \
  --key-name my-key \
  --security-group default \
  web-server-01

# 상태 확인 (ACTIVE가 되면 부팅 완료)
openstack server list

# 외부 접속용 공인 IP(Floating IP) 붙이기
openstack floating ip create public
openstack server add floating ip web-server-01 203.0.113.42
# Heat은 인프라를 'YAML 코드'로 선언합니다 (AWS CloudFormation에 해당).
# 같은 파일을 재사용하면 동일한 환경을 몇 번이고 재현할 수 있습니다.
heat_template_version: 2021-04-16

description: 웹서버 1대 + 네트워크 자동 생성

resources:
  web_server:
    type: OS::Nova::Server
    properties:
      name: web-server-01
      image: Ubuntu-22.04
      flavor: m1.small
      key_name: my-key
      networks:
        - network: private-net

# 배포: 한 줄이면 위 인프라가 통째로 생성됩니다.
#   openstack stack create -t web.yaml my-stack
# 삭제도 한 줄:  openstack stack delete my-stack
✓ 실무 감각

처음 배울 땐 Horizon으로 개념을 잡고, 익숙해지면 CLI로 빠르게 작업하며, 운영 단계에서는 Heat(또는 Terraform OpenStack provider)로 코드화해 자동화·버전관리·재현성을 확보합니다. 이는 LLMOps/MLOps 파이프라인에서 GPU 노드 풀을 자동 프로비저닝할 때 그대로 적용되는 패턴입니다.

Chapter 07 · 비교

다른 가상화·클라우드 솔루션과 비교

OpenStack의 위치를 이해하려면 이웃들을 알아야 합니다. 각 솔루션이 "어느 계층의 문제를 푸는가"가 핵심입니다.

계층부터 정리하기

비교 전에 헷갈리는 용어를 정리합시다. 같은 "가상화"라도 푸는 문제가 다릅니다.

하이퍼바이저

VM을 만드는 엔진

KVM · VMware ESXi · Xen · Hyper-V. 한 대의 물리 서버를 여러 VM으로 쪼갭니다. 가장 아래 계층.

클라우드 OS

데이터센터를 묶는 관리자

OpenStack · VMware vCloud. 수많은 하이퍼바이저 위에서 셀프서비스·멀티테넌시·API를 제공.

컨테이너 오케스트레이터

컨테이너를 굴리는 관리자

Kubernetes. VM이 아니라 컨테이너(앱 단위)를 배치. 보통 OpenStack 위에서 함께 씁니다.

구분 OpenStack VMware vSphere Proxmox VE 퍼블릭(AWS 등)

OpenStack vs Kubernetes — 경쟁이 아니라 동거

흔한 오해가 "둘 중 뭘 써야 하나?"입니다. 사실 둘은 다른 층입니다. OpenStack은 인프라(VM·네트워크·스토리지)를 제공하고, Kubernetes는 그 위에서 컨테이너 애플리케이션을 굴립니다. 실무에서는 OpenStack이 만들어 준 VM 위에 Kubernetes 클러스터를 올리거나, OpenStack의 Magnum 서비스로 K8s 클러스터를 자동 프로비저닝합니다. "건물(OpenStack) 안에 입주한 회사(Kubernetes)"로 보면 정확합니다.

※ 한 문장 선택 가이드

데이터를 외부에 못 두고 대규모 인프라를 직접 소유·제어해야 하면 OpenStack, 기존 기업 가상화를 안정적으로 편하게 운영하려면 VMware, 소규모·실험·홈랩이면 Proxmox, 운영 부담 없이 빠르게 시작하려면 퍼블릭 클라우드입니다.

과거 전용 장비(박스)로 구현하던 라우터·방화벽·게이트웨이 같은 네트워크 기능을, 범용 서버 위 가상 함수(VNF/CNF)로 돌리는 것이 NFV입니다. OpenStack은 이 NFV 인프라(NFVI)의 사실상 표준 토대입니다. 5G 코어망, MEC(엣지)까지 폭넓게 쓰입니다.

기업·공공 프라이빗 클라우드

데이터 외부 반출이 어려운 금융·공공·국방·의료 기관이 내부에 AWS급 셀프서비스 환경을 구축. 부서별로 테넌트를 나눠 자원을 격리·과금하고, 규제(망분리·데이터 주권)를 충족합니다.

AI · HPC
🧪

연구 · 슈퍼컴퓨팅 · AI 인프라

대학·연구소가 GPU/HPC 자원을 연구자에게 셀프서비스로 분배. Ironic(베어메탈)으로 GPU 서버를 통째로 할당하고, 그 위에 ML/LLM 학습·서빙 파이프라인을 올립니다. 대규모 모델 학습용 사설 GPU 클라우드의 토대가 됩니다.

하이브리드
🌐

하이브리드 · 멀티 클라우드

민감 데이터는 OpenStack(사내)에, 변동 트래픽은 퍼블릭에 두는 하이브리드 전략의 사내 거점. 표준 API 덕분에 Terraform 같은 도구로 여러 클라우드를 한 코드로 관리할 수 있습니다.

◎ MLOps/LLMOps 관점의 연결

OpenStack이 깔린 사설 클라우드는 GPU 노드 풀 → Kubernetes(Magnum) → 모델 서빙(vLLM/TensorRT-LLM)·학습 파이프라인으로 이어지는 스택의 가장 아래 기반이 됩니다. Heat/Terraform으로 인프라를 코드화해 두면, 모델 실험마다 동일한 GPU 환경을 재현 가능하게 찍어낼 수 있습니다.

Chapter 09 · 설치

어떻게 설치하고 운영할까

30여 개 서비스를 손으로 일일이 까는 건 비현실적입니다. 그래서 목적별 배포 도구가 발달했습니다.

학습용

DevStack

스크립트 한 번으로 노트북·VM 한 대에 모든 서비스를 올리는 개발/학습 환경. 운영용은 아니지만 개념 익히기에 최고입니다. 가장 먼저 손대볼 도구.

컨테이너형

Kolla-Ansible

각 서비스를 Docker 컨테이너로 패키징해 Ansible로 배포. 업그레이드·롤백이 깔끔해 현재 운영 환경에서 널리 쓰입니다.

대규모

OpenStack-Ansible (OSA)

Ansible 기반으로 대규모·고가용성(HA) 구성을 자동화. 프로덕션급 멀티노드 클러스터에 적합합니다.

상용 배포판

벤더 디스트리뷰션

Red Hat OpenStack Platform, Canonical(Ubuntu) Charmed OpenStack, Mirantis 등. 상용 지원·검증된 라이프사이클 관리를 원할 때 선택합니다.

DevStack 맛보기
설치 후 첫 확인
# Ubuntu 한 대(또는 VM)에서 — 학습용 All-in-One 설치
git clone https://opendev.org/openstack/devstack
cd devstack

# 최소 설정 파일 작성
cat > local.conf <<'EOF'
[[local|localrc]]
ADMIN_PASSWORD=secret
DATABASE_PASSWORD=$ADMIN_PASSWORD
RABBIT_PASSWORD=$ADMIN_PASSWORD
SERVICE_PASSWORD=$ADMIN_PASSWORD
EOF

# 설치 시작 (커피 한 잔 시간이 걸립니다 ☕)
./stack.sh

# 완료되면 브라우저로 Horizon 접속:  http://<IP>/dashboard
# 설치가 끝났다면 CLI로 동작을 확인합니다.
source openrc admin admin

# 각 서비스가 정상 등록됐는지 확인
openstack service list
openstack endpoint list

# 컴퓨트(하이퍼바이저) 상태
openstack hypervisor list
openstack compute service list

# 첫 VM을 띄울 준비물 확인
openstack image list      # Glance에 이미지 있나?
openstack flavor list     # 사양 정의 있나?
openstack network list    # 붙일 네트워크 있나?
✓ 학습 로드맵 제안

1단계 DevStack으로 VM·네트워크·볼륨을 직접 만들어 보기 → 2단계 Keystone/Nova/Neutron의 역할을 로그로 추적하기 → 3단계 Heat로 인프라를 코드화 → 4단계 Kolla-Ansible로 멀티노드 HA 구성 도전. 보유하신 이종(ARM64 + x86_64) 랩 클러스터는 멀티 아키텍처 컴퓨트 노드를 실험하기에 훌륭한 환경입니다.

Chapter 10 · 마무리

핵심 정리 & 이해도 퀴즈

여기까지 왔다면 OpenStack의 큰 그림이 잡혔을 겁니다. 용어를 한 번 정리하고, 퀴즈로 확인해 봅시다.

꼭 기억할 용어

IaaS
컴퓨팅·스토리지·네트워크 인프라를 셀프서비스 API로 제공하는 클라우드 모델. OpenStack의 본질.
Tenant / Project
자원을 격리해 쓰는 단위(고객·부서). 멀티테넌시의 기본 단위.
Flavor
vCPU·RAM·디스크를 묶은 VM 사양 템플릿. AWS의 인스턴스 타입에 해당.
Image
VM의 출고 상태(OS가 설치된 디스크 스냅샷). Glance가 관리.
Instance
실제로 실행 중인 VM 한 대를 부르는 말.
Floating IP
VM에 동적으로 붙였다 뗐다 하는 외부 접속용 공인 IP.
Hypervisor
물리 서버를 여러 VM으로 쪼개는 가상화 엔진(KVM 등). OpenStack의 하부.
Control / Data Plane
결정을 내리는 두뇌(컨트롤) vs 실제 일하는 손발(데이터). 아키텍처 이해의 축.

◎ 한 장 요약

OpenStack = 내 데이터센터에 직접 깔아 AWS처럼 쓰는 오픈소스 클라우드 OS. 하이퍼바이저(KVM)를 지휘해 VM·네트워크·스토리지를 셀프서비스로 제공하고, 컨트롤·데이터 플레인으로 구성되며, Horizon·CLI·Heat로 다룬다. 통신사 NFV·기업/공공 사설 클라우드·AI 인프라의 토대로 쓰이고, Kubernetes와는 경쟁이 아니라 상하 계층으로 함께 간다.
