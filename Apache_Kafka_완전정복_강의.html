SECTION 01

왜 Kafka가 필요한가 — 스파게티가 된 데이터 파이프라인

서비스가 작을 때는 문제가 없습니다. 주문 서버가 결제 서버를 직접 호출하고, 결제 서버가 알림 서버를 호출하면 끝이죠. 하지만 서비스가 커지면서 데이터를 만드는 시스템(Source)과 데이터를 쓰는 시스템(Target)이 늘어나면 이야기가 달라집니다.

소스 4개, 타깃 6개라면 연결은 최대 4 × 6 = 24개. 각 연결마다 프로토콜, 데이터 포맷, 장애 처리, 재시도 로직을 따로 관리해야 합니다. 어느 한 타깃이 느려지면 소스까지 같이 느려지고(동기 호출의 저주), 새 시스템 하나를 붙이려면 기존 시스템 전부를 수정해야 합니다.

연결 복잡도: N×M(24개) → N+M(8개). 새 시스템 추가도 "Kafka에 한 줄 연결"로 끝.

2011년 LinkedIn은 바로 이 문제로 고통받고 있었습니다. 그래서 만든 것이 Kafka — 모든 데이터가 일단 한곳에 모이고, 필요한 쪽이 알아서 가져가는 중앙 이벤트 허브입니다. 현재는 하루 수십조 건의 메시지를 처리하는 사실상의 업계 표준이 되었습니다.

ANALOGY — 우체국이 아니라 신문사

일반 메시지 큐(RabbitMQ 등)가 우체국이라면 — 편지를 받는 사람에게 전달하면 임무 끝, 편지는 사라짐 — Kafka는 신문사입니다. 신문(이벤트)을 계속 발행하고 보관하며, 구독자는 각자 자기 속도로 읽습니다. 어제 신문을 못 읽었으면 오늘 몰아서 읽으면 되고, 새 구독자는 과거 신문부터 다시 읽을 수도 있습니다.

💡

한 줄 정의: Kafka는 "분산형, 고가용성, 고처리량의 이벤트 스트리밍 플랫폼"입니다. 단순 메시지 전달을 넘어 이벤트를 저장하고(로그), 재생하고(리플레이), 실시간 처리(Kafka Streams)까지 아우릅니다.

SECTION 02

핵심 개념 도감 — 8개 용어만 알면 절반은 끝

Kafka 문서를 읽다 보면 낯선 용어가 쏟아지지만, 핵심은 8개뿐입니다. 아래 카드를 하나씩 읽어보세요. 이후 모든 내용은 이 8개 위에 쌓입니다.

📮

PRODUCER프로듀서

메시지를 만들어 Kafka에 보내는 쪽. 주문 API, 웹 서버, IoT 센서 등. "어느 토픽에 보낼지"만 알면 되고, 누가 읽는지는 전혀 몰라도 됩니다.

📥

CONSUMER컨슈머

Kafka에서 메시지를 가져가 처리하는 쪽. 중요한 점 — Kafka가 밀어주는(push) 게 아니라 컨슈머가 자기 속도로 당겨갑니다(pull).

🗂️

TOPIC토픽

메시지를 분류하는 논리적 채널. 데이터베이스의 "테이블", 폴더의 "디렉터리"와 비슷합니다. orders, user-clicks처럼 주제별로 만듭니다.

🧩

PARTITION파티션

토픽을 물리적으로 쪼갠 단위이자 병렬 처리의 핵심. 파티션이 3개면 3명의 컨슈머가 동시에 읽을 수 있습니다. 각 파티션 안에서는 순서가 보장됩니다.

🔢

OFFSET오프셋

파티션 안에서 각 메시지에 붙는 일련번호(0, 1, 2…). 컨슈머는 "어디까지 읽었는지"를 오프셋으로 기억합니다. 책갈피와 같은 역할.

🖥️

BROKER브로커

Kafka 서버 한 대. 보통 3대 이상을 묶어 클러스터로 운영합니다. 파티션들이 브로커 여러 대에 분산 배치됩니다.

👥

CONSUMER GROUP컨슈머 그룹

같은 일을 하는 컨슈머들의 팀. 파티션을 팀원끼리 나눠 갖고, 한 파티션은 그룹 내에서 딱 한 명만 읽습니다. 팀원이 죽으면 자동으로 재분배(리밸런싱).

🛡️

REPLICATION복제

각 파티션을 여러 브로커에 복사해 두는 것. 리더(Leader) 1개 + 팔로워(Follower) N개. 리더 브로커가 죽어도 팔로워가 즉시 승격되어 데이터 유실을 막습니다.

그림으로 한 번에 보기

같은 토픽을 그룹 A와 그룹 B가 각자 독립적으로 읽습니다 — 이것이 신문사 모델의 힘.

메시지(레코드)의 구조

Kafka의 메시지 한 건은 단순한 문자열이 아니라 구조를 가집니다.

  • Key (선택): 파티션 배정의 기준. 같은 키 → 같은 파티션 → 순서 보장. 예: user_id를 키로 쓰면 한 사용자의 이벤트는 항상 순서대로 처리됩니다.
  • Value: 실제 데이터 본문. JSON, Avro, Protobuf 등 자유.
  • Timestamp / Headers: 생성 시각과 메타데이터.
🗝️

ZooKeeper는 이제 안녕: 과거 Kafka는 클러스터 메타데이터 관리를 ZooKeeper에 의존했지만, Kafka 3.x부터 KRaft(Kafka Raft) 모드가 도입되어 4.0부터는 ZooKeeper가 완전히 제거되었습니다. 새로 시작한다면 KRaft만 알면 됩니다 — 구성 요소가 하나 줄어 운영이 훨씬 단순해졌습니다.

SECTION 03 · INTERACTIVE

파티션 시뮬레이터 — 메시지가 어디로 가는지 직접 보내보세요

프로듀서가 메시지를 보내면 Kafka는 어느 파티션에 넣을지 결정합니다. 규칙은 단순합니다.

  • 키가 있으면: hash(key) % 파티션 수 → 같은 키는 언제나 같은 파티션
  • 키가 없으면: 라운드로빈/스티키 방식으로 골고루 분배

⚖️

병렬성의 상한 = 파티션 수

파티션 4개에 컨슈머 5명을 넣으면 1명은 반드시 논다(idle). 그래서 파티션 수를 처음부터 여유 있게 잡는 것이 일반적입니다.

🔄

리밸런싱은 자동

컨슈머가 죽거나 새로 들어오면 그룹 코디네이터가 파티션을 자동 재분배합니다. 장애 복구를 별도로 코딩할 필요가 없습니다.

📰

그룹이 다르면 각자 다 읽음

"주문 처리팀"과 "분석팀"이 서로 다른 그룹 ID로 같은 토픽을 구독하면, 두 팀 모두 전체 메시지를 받습니다. Pub/Sub과 큐의 장점을 동시에.

SECTION 05

Kafka는 왜 빠른가 — 초당 수백만 건의 비밀 4가지

Kafka는 평범한 디스크에 쓰면서도 초당 수백 MB~GB급 처리량을 냅니다. 마법이 아니라, OS와 하드웨어의 특성을 영리하게 이용한 결과입니다.

01

순차 I/O (Sequential I/O)

디스크가 느린 건 "임의 접근(random access)"일 때입니다. Kafka는 로그 파일 끝에 이어 붙이기(append-only)만 합니다. 순차 쓰기는 디스크에서도 메모리에 견줄 만큼 빠르며, HDD에서조차 랜덤 I/O 대비 수천 배 빠를 수 있습니다.

02

페이지 캐시 (Page Cache)

Kafka는 자체 캐시를 만들지 않고 OS의 페이지 캐시에 전부 위임합니다. 최근 쓴 데이터는 사실상 메모리에서 읽히고, JVM 힙 밖에 있으므로 GC 부담도 없습니다.

03

제로 카피 (Zero-Copy)

일반적인 파일 전송은 [디스크→커널→앱→커널→NIC]로 데이터를 4번 복사합니다. Kafka는 sendfile() 시스템 콜로 [디스크→커널→NIC] 경로를 사용해 애플리케이션 단 복사를 생략합니다.

04

배칭 + 압축 (Batching)

메시지를 한 건씩 보내지 않고 수백 건을 묶어 전송하며, 묶음 단위로 압축(lz4, zstd)합니다. 네트워크 왕복과 디스크 쓰기 횟수가 극적으로 줄어듭니다.

ANALOGY — 택배 상하차

박스를 하나씩 트럭에 싣는 사람(랜덤 I/O, 건별 전송)과, 컨베이어 벨트로 팔레트째 싣는 사람(순차 I/O + 배칭)의 차이입니다. 같은 트럭, 같은 도로라도 처리량은 비교가 되지 않습니다.

SECTION 06

RabbitMQ와 비교 — 우체국 vs 신문사, 제대로 따져보기

"Kafka랑 RabbitMQ 중 뭐 써요?"는 면접 단골 질문이자 실무 단골 고민입니다. 결론부터: 둘은 경쟁자라기보다 다른 문제를 푸는 도구입니다. 철학의 차이부터 봅시다.

철학의 차이: 똑똑한 브로커 vs 똑똑한 컨슈머

🐰

RABBITMQ — SMART BROKER브로커가 똑똑하다

브로커가 라우팅 규칙(Exchange)에 따라 메시지를 적절한 큐로 분배하고, 컨슈머에게 밀어주고(push), 소비가 확인(ack)되면 삭제합니다. 메시지 하나하나의 전달에 집중하는 "전통적 메시지 브로커".

🌊

KAFKA — SMART CONSUMER컨슈머가 똑똑하다

브로커는 그저 로그에 이어 쓰고 보관할 뿐. 어디까지 읽었는지(오프셋), 언제 읽을지는 전적으로 컨슈머가 관리하며 당겨갑니다(pull). 읽어도 메시지는 삭제되지 않습니다(retention 기간 보관).

RabbitMQ를 선택

① 작업 큐(이메일 발송, 이미지 변환 등 백그라운드 잡) ② 복잡한 라우팅 규칙 ③ 우선순위·지연 메시지 ④ 요청-응답(RPC) 패턴 ⑤ 낮은 지연이 중요한 건별 처리 ⑥ 소규모로 빠르게 시작

🤝

둘 다 쓰는 회사도 많습니다. 예: 사용자 행동 이벤트는 Kafka로 수집·분석하고, "결제 완료 시 영수증 메일 발송" 같은 작업은 RabbitMQ 작업 큐로 처리. 도구는 문제에 맞춰 고르는 것이지, 한쪽이 다른 쪽을 대체하는 관계가 아닙니다.

ANALOGY — 면접용 한 줄 요약

"RabbitMQ는 메시지를 전달하는 데 최적화된 우체국, Kafka는 이벤트를 기록하고 흘려보내는 데 최적화된 신문사입니다. 전달이 목적이면 RabbitMQ, 기록·재생·대용량 스트림이 목적이면 Kafka를 선택합니다."

SECTION 07

활용 방안 7가지 — 현장에서 Kafka가 일하는 모습

각 항목을 클릭해서 펼쳐보세요. 데이터 흐름도와 함께 "왜 하필 Kafka인지"를 설명합니다.

📋

로그 수집 파이프라인LOG AGGREGATION — 가장 흔한 입문 사례

▼

수백 대 서버의 애플리케이션 로그를 한곳에 모아 검색·분석합니다. 서버가 Elasticsearch에 직접 쓰면 ES 장애 시 로그가 유실되지만, Kafka가 중간에서 버퍼 역할을 하면 ES가 복구된 뒤 밀린 로그를 따라잡으면 됩니다.

App 서버들 → Filebeat/Fluent Bit → Kafka (topic: app-logs) → Logstash/Consumer → Elasticsearch → Kibana
🔗

마이크로서비스 비동기 통신EVENT-DRIVEN MSA — 서비스 간 결합도 제거

▼

주문 서비스가 "주문 생성됨" 이벤트만 발행하면, 결제·재고·알림 서비스가 각자 구독해 처리합니다. 주문 서비스는 누가 듣는지 몰라도 되고, 알림 서비스가 죽어 있어도 주문은 정상 진행됩니다. 새 서비스(예: 추천)를 추가할 때 기존 코드는 한 줄도 안 바꿉니다.

주문 서비스 → Kafka (topic: order-events) → 결제 / 재고 / 알림 / 추천 (각자 다른 Consumer Group)
🪞

CDC — 데이터베이스 변경 실시간 복제CHANGE DATA CAPTURE — Debezium + Kafka Connect

▼

운영 DB의 변경(INSERT/UPDATE/DELETE)을 트랜잭션 로그(binlog/WAL) 수준에서 캡처해 이벤트로 흘립니다. 운영 DB에 분석 쿼리를 날려 부하를 주는 대신, 검색엔진·캐시·DW를 거의 실시간으로 동기화할 수 있습니다. 코드를 작성하지 않고 Kafka Connect 설정만으로 구성 가능합니다.

MySQL binlog → Debezium (Kafka Connect Source) → Kafka → Sink Connector → Elasticsearch / Redis / BigQuery
⚡

실시간 스트림 처리STREAM PROCESSING — Kafka Streams / Flink

▼

흘러가는 이벤트를 "배치로 모아서 나중에"가 아니라 도착하는 즉시 집계·변환·조인합니다. 이상 거래 탐지(FDS), 실시간 대시보드, 5분 윈도우 집계 같은 작업이 대표적입니다. Kafka Streams는 별도 클러스터 없이 애플리케이션 라이브러리로 동작한다는 점이 매력입니다.

topic: payments → Kafka Streams (5분 윈도우 집계, 이상 패턴 필터) → topic: fraud-alerts → 알림 서비스
📜

이벤트 소싱EVENT SOURCING — 상태가 아닌 사건을 저장

▼

"현재 잔액 50,000원"이라는 상태 대신 [+100,000 입금, -30,000 출금, -20,000 출금]이라는 사건의 기록을 저장합니다. Kafka의 보관·재생 능력 덕분에 어느 시점의 상태든 이벤트를 재생해 복원할 수 있고, 완벽한 감사(audit) 추적이 공짜로 따라옵니다.

Command → Kafka (이벤트 = 진실의 원천) → 재생(replay) → 현재 상태 / 과거 임의 시점 상태 / 읽기 모델(CQRS)
🤖

ML/LLM 파이프라인MLOPS — 피처 스트림과 추론 로그

▼

MLOps 현장에서 Kafka는 두 군데서 일합니다. ① 사용자 행동 이벤트를 실시간 피처로 변환해 Feature Store에 공급(실시간 추천·이상탐지의 신선한 피처), ② LLM 서비스의 추론 요청/응답 로그를 수집해 모니터링·평가·재학습 데이터셋으로 적재. 추론 서버가 로깅 때문에 느려지지 않도록 비동기로 분리하는 것이 핵심입니다.

vLLM/Triton 추론 로그 → Kafka (topic: llm-traces) → ① Langfuse류 관측 도구 ② S3/MinIO 적재 → 평가·파인튜닝 데이터셋
📡

IoT / 텔레메트리 수집IOT INGESTION — 수백만 디바이스의 심장박동

▼

수십만 대 센서가 초 단위로 보내는 측정값을 받아내는 관문입니다. 디바이스 ID를 키로 쓰면 디바이스별 순서가 보장되고, 다운스트림(시계열 DB, 알람, 집계)이 일시 정지해도 데이터는 Kafka에 안전하게 쌓입니다. 엣지에서는 MQTT로 받고 브리지를 통해 Kafka로 합류시키는 구성이 일반적입니다.

센서 → MQTT 브로커 → MQTT-Kafka 브리지 → Kafka → InfluxDB/TimescaleDB + 실시간 알람

# pip install confluent-kafka  (librdkafka 기반, 실무 표준)
import json
from confluent_kafka import Producer

producer = Producer({
    "bootstrap.servers": "localhost:9092",
    "acks": "all",                # 모든 복제본 저장 후 응답 → 유실 방지
    "enable.idempotence": True,   # 재시도 시 중복 저장 방지
    "compression.type": "lz4",    # 배치 압축으로 네트워크 절약
    "linger.ms": 5,               # 5ms 모아서 배치 전송 (처리량↑)
})

def on_delivery(err, msg):
    if err:
        print(f"전송 실패: {err}")   # 실무에선 재시도/알림 처리
    else:
        print(f"OK → {msg.topic()}[{msg.partition()}] offset={msg.offset()}")

event = {"event": "order_created", "order_id": "A-1042", "amount": 35000}

producer.produce(
    topic="orders",
    key="user-1",                  # 같은 user는 같은 파티션 → 순서 보장
    value=json.dumps(event),
    on_delivery=on_delivery,
)
producer.flush()                   # 버퍼 비우고 종료 전 전송 보장
import json
from confluent_kafka import Consumer

consumer = Consumer({
    "bootstrap.servers": "localhost:9092",
    "group.id": "order-service",        # 같은 group.id = 한 팀 (파티션 분담)
    "auto.offset.reset": "earliest",    # 저장된 오프셋 없으면 처음부터
    "enable.auto.commit": False,        # 처리 완료 후 직접 커밋 → at-least-once
})
consumer.subscribe(["orders"])

try:
    while True:
        msg = consumer.poll(timeout=1.0)   # pull 모델 — 내 속도로 당겨온다
        if msg is None:
            continue
        if msg.error():
            print(f"에러: {msg.error()}")
            continue

        event = json.loads(msg.value())
        print(f"[P{msg.partition()}@{msg.offset()}] key={msg.key()} → {event}")

        # ... 비즈니스 로직 (DB 저장 등) ...

        consumer.commit(msg)   # 처리 성공 후 커밋 — 실패 시 재처리됨
except KeyboardInterrupt:
    pass
finally:
    consumer.close()           # 그룹에서 정상 탈퇴 → 빠른 리밸런싱

운영 베스트 프랙티스 — 장애를 미리 막는 7가지 습관

📏

1. 파티션 수는 처음에 넉넉히

파티션은 늘릴 수는 있어도 줄일 수 없고, 늘리면 키-파티션 매핑이 바뀌어 순서 보장이 깨집니다. 목표 처리량 ÷ 컨슈머 1개 처리량으로 계산하되 향후 확장분까지 고려하세요.

📊

2. Consumer Lag을 1순위 지표로

Lag = "최신 오프셋 − 커밋된 오프셋" = 밀린 일감. Lag이 계속 증가하면 컨슈머가 생산 속도를 못 따라가는 것. Prometheus의 kafka_exporter + Grafana 대시보드가 표준 조합입니다.

🛡️

3. acks=all + 멱등성 프로듀서

유실이 치명적인 데이터라면 acks=all(모든 ISR 복제 후 응답) + enable.idempotence=true(재시도 중복 방지)를 기본값으로. 처리량이 더 중요하면 acks=1로 타협합니다.

🔁

4. "최소 1회"를 전제로 멱등하게 설계

Kafka 기본은 at-least-once — 같은 메시지가 두 번 올 수 있습니다. 컨슈머 로직을 멱등(idempotent)하게 만드세요. 예: INSERT 대신 UPSERT, 이벤트 ID 기반 중복 체크.

🗓️

5. Retention은 디스크와의 협상

기본 7일(retention.ms). 토픽별로 데이터 가치에 맞게 조정하고, 크기 기준(retention.bytes)도 함께 걸어 디스크 폭발을 막으세요. 최신 상태만 필요하면 compaction도 검토.

📐

6. 스키마는 계약이다

프로듀서가 필드 하나 바꾸면 컨슈머 전체가 깨질 수 있습니다. Schema Registry + Avro/Protobuf로 스키마 호환성(BACKWARD 등)을 강제하는 것이 다인원 조직의 생존 전략입니다.

💀

7. 독약 메시지 대비 (DLQ)

파싱 불가능한 메시지 하나가 컨슈머를 무한 재시도 루프에 빠뜨릴 수 있습니다. N회 실패 시 별도 토픽(orders.DLQ)으로 보내고 본 흐름은 계속 진행하도록 설계하세요.

핵심 요약 — 이것만 기억하세요

🎯 10줄 요약

  1. Kafka는 메시지 큐가 아니라 이벤트 스트리밍 플랫폼 — 전달하고, 보관하고, 재생한다.
  2. N×M 직접 연결의 스파게티를 N+M 중앙 허브로 푼다.
  3. 핵심 8개 용어: Producer, Consumer, Topic, Partition, Offset, Broker, Consumer Group, Replication.
  4. 파티션 = 병렬성의 단위이자 순서 보장의 단위. 같은 키 → 같은 파티션 → 순서 보장.
  5. 컨슈머 그룹: 한 파티션은 그룹 내 한 컨슈머만 읽는다. 컨슈머 수 > 파티션 수면 남는 컨슈머는 논다.
  6. 빠른 이유 4가지: 순차 I/O · 페이지 캐시 · 제로카피 · 배칭.
  7. RabbitMQ는 우체국(전달 후 삭제, push, 정교한 라우팅), Kafka는 신문사(보관·재생, pull, 대용량 스트림).
  8. 작업 큐·우선순위·복잡한 라우팅 → RabbitMQ / 대용량 수집·다중 소비·재생·스트림 처리 → Kafka.
  9. 운영 3대 키워드: Consumer Lag 모니터링, acks=all + 멱등성, 파티션 수 사전 설계.
  10. 기본은 at-least-once — 컨슈머 로직은 항상 멱등하게 작성한다.
