이름이 곧 주소가 되는
클러스터의 비밀 K8S CoreDNS의 모든 것 — 일반 DNS부터 클러스터 내부 동작까지

Kubernetes 안에서는 Pod끼리 my-service.default.svc.cluster.local 같은 신기한 이름으로 서로를 부릅니다. 이 이름을 실제 IP로 바꿔주는 주인공이 바로 CoreDNS죠. 이 강의에서는 일반 DNS와 비교해가며, K8S 초보자도 CoreDNS의 동작 원리·구성·활용 방법을 끝까지 따라올 수 있도록 차근차근 풀어드립니다.

CHAPTER 01

먼저, 일반 DNS가 뭔지부터
다시 짚고 가자

CoreDNS를 이해하려면, 일단 일반적인 DNS가 무엇이고 어떻게 동작하는지부터 명확하게 잡고 가야 합니다. DNS는 한마디로 인터넷의 전화번호부예요. 우리가 www.google.com이라고 입력하면, 뒤에서는 누군가 그 이름을 142.250.207.46 같은 IP 주소로 바꿔주고 있죠.

왜 이름이 필요한가

컴퓨터끼리 통신할 때는 결국 IP 주소가 필요합니다. 그런데 사람은 숫자를 외우기 어렵고, 무엇보다 서버의 IP는 자주 바뀝니다 — 클라우드에서는 더더욱요. 그래서 변하지 않는 이름과 변하기 쉬운 IP를 분리해두고, 그 사이를 매번 매핑해주는 시스템이 필요해졌습니다. 그게 바로 DNS입니다.

DNS의 핵심 등장인물 5명

RESOLVER ① 스텁 리졸버 (Stub Resolver)

클라이언트(브라우저, 애플리케이션, Pod)에 내장된 작은 DNS 클라이언트입니다. 실제로 질의를 만들어내는 출발점이고, OS의 /etc/resolv.conf를 읽어서 어디로 물어볼지 결정합니다.

RECURSIVE ② 재귀 리졸버 (Recursive Resolver)

흔히 말하는 "DNS 서버". 8.8.8.8(Google), 1.1.1.1(Cloudflare), 회사 내부 DNS 등이 모두 여기에 해당합니다. 대신 발품을 팔아가며 답을 찾아주는 역할을 합니다.

ROOT ③ 루트 네임서버 (.)

DNS 계층의 최상위. 전 세계에 13개 그룹이 있고, ".com은 저쪽으로 가봐"처럼 위임 정보만 알려줍니다.

TLD ④ TLD 네임서버

.com, .kr, .io 같은 최상위 도메인을 관리. "google.com의 권한 서버는 여기야"라고 알려줍니다.

AUTHORITATIVE ⑤ 권한 네임서버 (Authoritative)

해당 도메인의 실제 답을 가지고 있는 서버. 도메인을 등록할 때 우리가 ns1.example.com처럼 지정하는 그 서버입니다. 여기까지 와야 비로소 "www.google.com은 142.250.207.46이야"라는 최종 답이 나옵니다.

전체 흐름 — 한 번에 보기

위 등장인물들이 모두 모여 한 번의 이름 해석이 어떻게 일어나는지, 도식으로 정리해봅시다. DNS의 본질은 결국 "위임의 연쇄"입니다.

실선: 질의 / 점선: 응답 · 재귀 리졸버 한 명이 위임을 따라가며 답을 찾아옴

포인트는 이거예요. 클라이언트는 단 한 번만 묻고, 나머지 발품은 재귀 리졸버가 다 처리합니다. 그리고 한 번 답을 찾은 결과는 TTL(Time-To-Live)이라는 유효기간 동안 캐시해두므로, 같은 이름을 반복해서 묻는 비용을 크게 줄여줍니다. 이 캐싱 개념은 CoreDNS에서도 똑같이 작동합니다.

DNS 레코드 타입 — 자주 보게 될 것들

  • A — 도메인 이름 → IPv4 주소. 가장 흔합니다.
  • AAAA — 도메인 이름 → IPv6 주소.
  • CNAME — 다른 도메인 이름으로의 별칭. (예: www.example.com → example.com)
  • SRV — 서비스의 호스트·포트 정보. K8S에서 Headless Service가 사용합니다.
  • PTR — IP → 도메인 이름 (역방향 조회).
  • NS / SOA — 위임·존(zone)의 권한 정보.
CHAPTER 02

Kubernetes에서는 왜
내부 DNS가 따로 있어야 할까?

K8S 클러스터에는 Pod가 수백, 수천 개 떠 있을 수 있습니다. 그리고 Pod는 죽었다 살아날 때마다 IP가 바뀝니다. 이런 환경에서 우리가 일반 인터넷 DNS만 쓴다면? 도저히 관리가 안 됩니다. 그래서 K8S는 클러스터 안에서만 통용되는 내부 DNS 시스템을 별도로 운영합니다.

Pod 입장에서 본 "이름이 필요한 순간"

시나리오

frontend Pod가 backend Service를 호출해야 합니다. 그런데:

  • backend Pod의 IP는 재시작될 때마다 바뀝니다.
  • backend Pod는 3개로 늘어났다가 1개로 줄기도 합니다(autoscaling).
  • 그렇다고 매번 frontend의 코드를 수정해서 IP를 바꿔줄 순 없죠.

해법은? "backend"라는 이름을 만들고, 그 이름으로만 호출하게 합니다. 뒤에서 Service가 변하는 Pod 집합을 추상화해주고, 그 Service 이름을 IP(ClusterIP)로 바꿔주는 일을 CoreDNS가 담당합니다.

K8S DNS가 풀어주는 이름 형식

K8S에서 Service에는 자동으로 FQDN(Fully Qualified Domain Name)이 생깁니다. 기본 형태는 이렇습니다:

FQDN 구조
<service-name>.<namespace>.svc.cluster.local # 실제 예시 redis-master.cache.svc.cluster.local mlflow-server.mlops.svc.cluster.local vllm-qwen.inference.svc.cluster.local

그리고 Pod 안에서는 위의 풀네임을 다 칠 필요가 없어요. 같은 네임스페이스라면 그냥:

짧은 이름 호출
# 같은 namespace에서 curl http://redis-master:6379 # 다른 namespace로 curl http://redis-master.cache:6379 # 완전한 FQDN curl http://redis-master.cache.svc.cluster.local:6379

이게 가능한 이유는 Pod의 /etc/resolv.conf에 search 도메인이 설정돼 있기 때문입니다. 직접 확인해볼게요.

$ kubectl exec POD -- cat /etc/resolv.conf
search default.svc.cluster.local svc.cluster.local cluster.local nameserver 10.96.0.10 # ← kube-dns Service의 ClusterIP options ndots:5

중요한 두 줄이 있습니다.

  • nameserver 10.96.0.10 — Pod가 보내는 모든 DNS 질의는 이 IP로 갑니다. 이 IP가 바로 CoreDNS Service의 ClusterIP입니다.
  • search 도메인 — redis-master만 적어도 자동으로 redis-master.default.svc.cluster.local을 시도해줍니다. 이걸 name suffix search라고 합니다.
  • ndots:5 — 점(.)이 5개 미만이면 일단 search 도메인을 모두 붙여보고, 안 되면 그제서야 외부 DNS로 나갑니다. 성능 이슈로 유명한 옵션입니다.
유명한 함정 ndots:5 외부 호출 지연 문제

api.openai.com처럼 외부 도메인을 호출할 때, 점이 2개라 search 도메인이 모두 붙은 채로 api.openai.com.default.svc.cluster.local 같은 질의가 먼저 실패하고, 그제서야 진짜 외부로 나갑니다. 그래서 외부 호출이 많은 워크로드(LLM API 호출, 외부 객체 스토리지 등)에서는 FQDN 끝에 점(.)을 명시하거나 Pod spec에 dnsConfig.options로 ndots:2를 주는 튜닝을 많이 합니다.

CHAPTER 03

드디어 등장, CoreDNS
그런데 그 전엔 누가 있었나?

K8S 클러스터의 내부 DNS를 담당하는 컴포넌트는 시대에 따라 바뀌어 왔습니다. 초보자가 가장 헷갈리는 부분이기도 한데요, 한 번에 정리하면 이렇습니다.

K8S 클러스터 DNS의 세대 변천

~2018 (LEGACY) 1세대 — kube-dns

초창기 K8S의 기본 DNS. 한 Pod 안에 kubedns + dnsmasq + sidecar 컨테이너 세 개를 묶어 돌렸습니다. C 기반 dnsmasq가 캐싱을 담당했는데, 메모리 누수·플러그인 한계·확장성 문제가 누적되었습니다.

2019 ~ 현재 (DEFAULT) 2세대 — CoreDNS

Kubernetes 1.13부터 정식 기본값이 되었고, 현재는 사실상 표준입니다. 단일 Go 바이너리이고, 플러그인 체인이라는 깔끔한 구조로 캐싱·k8s 연동·메트릭·재귀 질의 같은 모든 기능이 끼워맞춰집니다. CNCF Graduated 프로젝트이기도 합니다.

CoreDNS의 작동 컨셉 한 문장

"DNS 질의가 들어오면, Corefile에 정의된 플러그인을 위에서 아래로 순서대로 통과시키며, 처리할 수 있는 플러그인이 답하면 끝, 못하면 다음 플러그인으로 넘기는 체인 구조다."

이 구조 때문에 CoreDNS는 "Kubernetes DNS"이기 이전에 범용 DNS 서버입니다. K8S 안에서는 kubernetes 플러그인을 활성화해서 Service/Endpoint를 DNS 레코드로 노출하는 것뿐이고, 그 외에도 일반 권한 서버나 포워더로 얼마든지 쓸 수 있습니다. 이 점이 단순한 dnsmasq 래퍼였던 kube-dns와의 근본적인 차이입니다.

CoreDNS는 클러스터에서 어떻게 떠 있나

설치된 클러스터에서 직접 확인해 보세요. 거의 항상 kube-system 네임스페이스에 Deployment로 떠 있고, 그 앞에 kube-dns라는 Service가 ClusterIP 10.96.0.10(기본값)을 갖고 있습니다. Service 이름이 "kube-dns"인 건 역사적 호환 때문이지, 실제 동작 주체는 CoreDNS입니다.

$ kubectl -n kube-system get all -l k8s-app=kube-dns
NAME READY STATUS RESTARTS pod/coredns-5d78c9869d-abc12 1/1 Running 0 pod/coredns-5d78c9869d-xyz34 1/1 Running 0 NAME TYPE CLUSTER-IP PORT(S) service/kube-dns ClusterIP 10.96.0.10 53/UDP,53/TCP,9153/TCP NAME READY UP-TO-DATE AVAILABLE deployment.apps/coredns 2/2 2 2

Pod 2개가 떠 있는 게 보이시죠? HA를 위한 기본 레플리카 수가 2입니다. 대규모 클러스터에서는 보통 HPA를 붙이거나 노드 수에 맞춰 더 늘립니다(이건 챕터 6에서 자세히).

CHAPTER 04

CoreDNS의 심장,
Corefile 한 줄씩 뜯어보기

CoreDNS의 모든 동작은 Corefile 하나로 결정됩니다. nginx의 nginx.conf를 떠올리시면 좋아요. K8S 환경에서는 이 Corefile이 kube-system 네임스페이스의 ConfigMap으로 저장되어 있고, CoreDNS Pod가 마운트해서 읽습니다.

직접 확인해보기

$ kubectl -n kube-system get cm coredns -o yaml
apiVersion: v1 kind: ConfigMap metadata: name: coredns namespace: kube-system data: Corefile: | .:53 { errors health { lameduck 5s } ready kubernetes cluster.local in-addr.arpa ip6.arpa { pods insecure fallthrough in-addr.arpa ip6.arpa ttl 30 } prometheus :9153 forward . /etc/resolv.conf { max_concurrent 1000 } cache 30 loop reload loadbalance }

인터랙티브: Corefile 한 줄씩 클릭해보세요

아래 코드 영역에서 플러그인 이름을 클릭하면 오른쪽에 무슨 일을 하는지가 나옵니다.

.:53 { errors health { lameduck 5s } ready kubernetes cluster.local in-addr.arpa ip6.arpa { pods insecure fallthrough in-addr.arpa ip6.arpa ttl 30 } prometheus :9153 forward . /etc/resolv.conf cache 30 loop reload loadbalance }
선택 항목
플러그인을 클릭해보세요

왼쪽에서 kubernetes, forward, cache 등 플러그인 이름을 누르면, 각각이 어떤 역할을 하는지 여기에 설명이 나옵니다.

💡 CoreDNS는 위에서 아래 순서대로 플러그인을 통과시킵니다. 순서가 곧 동작 흐름이에요.

주요 플러그인 카테고리

CORE 데이터 소스

kubernetes, file, auto, etcd, route53, secondary. 실제 레코드의 출처가 되는 플러그인들.

FORWARD 전달·재귀

forward, proxy(deprecated). 자기가 모르는 도메인을 외부 DNS로 던지는 역할. K8S에서는 클러스터 외 도메인을 처리하는 핵심.

UTIL 캐싱·관측·안전장치

cache, prometheus, health, ready, log, errors, loop, reload, loadbalance.

실전 시나리오 — Corefile 커스터마이징

사내 DNS 서버(10.10.0.53)에 있는 corp.kt.com 도메인을 클러스터 내부 Pod에서도 빠르게 풀고 싶을 때:

Corefile - 사내 도메인 stub 추가
.:53 { # 기존 설정 ... kubernetes cluster.local in-addr.arpa ip6.arpa { ... } forward . /etc/resolv.conf cache 30 } corp.kt.com:53 { # 별도 zone 블록 추가 errors cache 30 forward . 10.10.0.53 # 사내 DNS로 직행 }

구버전 코드가 db.legacy를 호출하는데, 실제 Service는 postgres.data.svc.cluster.local일 때:

Corefile - rewrite 플러그인
.:53 { errors health rewrite name db.legacy postgres.data.svc.cluster.local kubernetes cluster.local in-addr.arpa ip6.arpa { ... } forward . /etc/resolv.conf cache 30 loop reload }

워크로드가 광고/추적 도메인으로 새는 것을 막고 싶을 때, template 플러그인으로 NXDOMAIN 응답:

Corefile - 차단 zone
doubleclick.net:53 googletagmanager.com:53 { template IN ANY { rcode NXDOMAIN } }

특정 클라이언트가 어떤 질의를 던지는지 일시적으로 로그를 켤 때:

Corefile - log 플러그인
.:53 { errors log . { class denial error } # 실패만 로그 log mysql.data.svc.cluster.local # 특정 이름만 추적 kubernetes cluster.local in-addr.arpa ip6.arpa { ... } forward . /etc/resolv.conf cache 30 reload }

💡 ConfigMap 수정 후 reload 플러그인 덕분에 별도 재시작 없이 적용됩니다(약 30초 이내).

CHAPTER 05

실제로 한 번 풀어보자
Pod가 이름을 부르면 무슨 일이?

이론은 충분하니, 직접 시뮬레이션해봅시다. 아래 입력창에 도메인 이름을 넣고 RESOLVE를 눌러보세요. CoreDNS가 resolv.conf의 search·ndots 규칙을 어떻게 적용하고, 어느 플러그인이 어떻게 답하는지 단계별로 보여드립니다.

PRESET ↓

이 시뮬레이터에서 꼭 보고 가야 할 포인트

  • 점(.) 개수에 따라 search 순서가 달라진다 — ndots:5 미만이면 search 도메인을 먼저 시도.
  • FQDN 끝에 점(.)을 붙이면 search를 건너뛴다 — api.openai.com.으로 입력해 비교해보세요.
  • kubernetes 플러그인이 먼저 시도되고, 못 풀면 forward로 넘어간다 — 플러그인 체인의 본질.
  • cache 플러그인이 동일 응답을 30초간 보관 — 반복 호출 시 K8S API 부하가 줄어듦.
CHAPTER 06

CoreDNS만 있는 게 아니다
구현체의 종류와 함께 쓰는 친구들

"K8S의 DNS = CoreDNS"라고 단순하게 외워두면 90%는 맞지만, 실무에서는 그 주변에 몇 가지 동반 컴포넌트가 더 등장합니다. 초보자가 가장 자주 혼동하는 다섯 가지를 정리해드립니다.

① CoreDNS — 클러스터 내부 DNS의 본체

여기까지가 우리가 4장까지 다뤘던 그 친구. kube-system 네임스페이스에 Deployment로 떠 있고, Service 이름은 (역사적 이유로) kube-dns입니다. 클러스터 안의 모든 Pod가 결국 여기로 DNS를 묻습니다.

② NodeLocal DNSCache — 노드마다 캐시 한 대씩

PERFORMANCE 왜 필요한가?

Pod → CoreDNS Service(ClusterIP) 통신은 사실 iptables(혹은 IPVS) NAT를 거치는 cross-node 트래픽입니다. 대규모 클러스터에서는 이게 conntrack 부하·UDP 5초 지연 버그 등 다양한 문제를 일으킵니다. NodeLocal DNSCache는 각 노드에 DaemonSet으로 캐시 DNS를 띄워, Pod가 169.254.20.10 같은 link-local IP로 같은 노드의 캐시에게 먼저 묻게 만듭니다.

→ 동일 노드 내 통신이므로 NAT 없음, conntrack 폭증 없음, 캐시 적중률 ↑. 1000개 노드 이상의 클러스터에서는 사실상 필수.

NodeLocal DNSCache · 노드마다 캐시를 두어 cross-node DNS 트래픽 최소화

③ ExternalDNS — K8S 리소스 → 외부 DNS에 자동 등록

이것도 자주 헷갈리는 친구입니다. ExternalDNS는 K8S의 DNS 서버가 아닙니다. Ingress·Service에 붙은 호스트명을 보고, 외부 DNS 제공자(Route53, Cloudflare, Google Cloud DNS, Azure DNS 등)에 레코드를 자동으로 만들어주는 컨트롤러입니다. 즉, CoreDNS는 "클러스터 안에서 이름을 푸는" 일을, ExternalDNS는 "클러스터 밖 세상에 이름을 알리는" 일을 합니다. 둘은 함께 쓰입니다.

Ingress + ExternalDNS 예시
apiVersion: networking.k8s.io/v1 kind: Ingress metadata: name: mlflow annotations: external-dns.alpha.kubernetes.io/hostname: mlflow.example.com spec: rules: - host: mlflow.example.com # ExternalDNS가 Route53에 자동 등록 http: paths: - path: / pathType: Prefix backend: service: name: mlflow-server port: number: 5000

④ headless Service와 SRV 레코드

일반 Service는 ClusterIP라는 가상 IP 하나를 받지만, Headless Service(clusterIP: None)는 IP가 없는 대신 CoreDNS가 각 Pod IP를 그대로 A 레코드로 반환합니다. 또한 SRV 레코드도 제공합니다. 이건 분산 시스템에서 클라이언트가 직접 모든 인스턴스를 알아야 할 때 — 예를 들어 StatefulSet의 각 멤버를 식별해야 하는 Kafka, Cassandra, etcd 같은 경우 — 필수입니다.

Headless Service - StatefulSet 패턴
apiVersion: v1 kind: Service metadata: name: kafka-broker spec: clusterIP: None # Headless selector: app: kafka ports: - port: 9092 # DNS 조회 결과: # kafka-broker.default.svc.cluster.local => 모든 Pod IP들이 나열됨 # kafka-0.kafka-broker.default.svc.cluster.local => 특정 Pod의 IP만

⑤ Pod의 dnsPolicy — Pod별로 다르게 풀고 싶을 때

Pod spec에는 dnsPolicy가 있고, 4가지 값이 있습니다. 이게 CoreDNS와 직접 통신할지 말지를 결정합니다.

dnsPolicy
동작
ClusterFirst
기본값. CoreDNS(10.96.0.10)를 먼저 거치고, 못 풀면 forward 플러그인이 외부로.
ClusterFirstWithHostNet
hostNetwork: true Pod에서 ClusterFirst를 강제. 호스트의 resolv.conf를 무시하고 CoreDNS를 씀.
Default
노드의 /etc/resolv.conf를 그대로 사용. CoreDNS를 거치지 않습니다.
None
아무것도 안 함. dnsConfig를 직접 명시해야 함 (특수 케이스).
CHAPTER 07

실전 활용과 자주 만나는 함정

여기까지 잘 따라오셨다면 이미 CoreDNS의 90%는 마스터한 셈입니다. 마지막으로, 현장에서 자주 마주치는 문제들과 LLMOps/MLOps 관점에서 특히 신경 써야 할 활용 포인트를 정리합니다.

실전 활용 1 — 외부 LLM API 호출 지연 줄이기

실제 사례 증상

Pod 안에서 api.openai.com이나 huggingface.co를 호출할 때마다, 첫 응답이 묘하게 느립니다(수백 ms). 그런데 curl --resolve로 IP를 직접 박으면 빨라요.

원인

ndots:5 때문에 외부 도메인도 일단 search 도메인을 다 붙여서 NXDOMAIN을 받은 다음에야 진짜로 외부에 묻습니다. Pod 한 번 호출에 DNS 질의가 4~5번 발생.

처방
Pod spec - dnsConfig 튜닝
spec: dnsConfig: options: - name: ndots value: "2" # 5 → 2로 낮추기 - name: single-request-reopen

실전 활용 2 — 대규모 추론 클러스터의 DNS 부하 대응

수십 개의 vLLM 워커 Pod가 mlflow, S3 호환 스토리지, Redis 등을 끊임없이 호출하는 환경이라면, CoreDNS 자체가 병목이 됩니다. 권장 조치:

  • CoreDNS HPA 설정 — CPU 기반 또는 cluster-proportional-autoscaler로 노드 수에 비례하게.
  • NodeLocal DNSCache 활성화 — 사실상 첫 번째 처방. UDP 5초 지연 문제도 함께 사라집니다.
  • cache TTL 늘리기 — Service IP는 자주 안 바뀌므로 cache 300 정도까지 늘려도 무방.
  • autopath 플러그인 — search 도메인 순회를 CoreDNS 쪽에서 처리해 Pod의 왕복 질의를 줄여줍니다.

트러블슈팅 — 진단의 5단계

STEP 1 — Pod에서 직접 nslookup
임시 디버그 Pod
kubectl run dnsutils --rm -it --image=registry.k8s.io/e2e-test-images/jessie-dnsutils:1.3 -- bash # Pod 안에서 nslookup kubernetes.default nslookup mlflow.mlops.svc.cluster.local cat /etc/resolv.conf
STEP 2 — CoreDNS Pod 상태와 로그
kubectl
kubectl -n kube-system get pods -l k8s-app=kube-dns kubectl -n kube-system logs -l k8s-app=kube-dns --tail=50 kubectl -n kube-system top pods -l k8s-app=kube-dns
STEP 3 — Prometheus 메트릭 확인 (:9153)

coredns_dns_request_duration_seconds, coredns_cache_hits_total, coredns_forward_request_duration_seconds 등이 핵심 지표. 캐시 적중률이 낮거나 forward 응답 시간이 튀면 외부 DNS가 의심됩니다.

STEP 4 — Corefile에 임시 log 켜기

앞서 본 log 플러그인을 잠깐 추가하고 reload가 적용되길 기다린 뒤, 실제 어떤 질의가 들어오는지 패턴을 확인합니다. 작업 끝나면 반드시 다시 끄세요(로그량 폭주).

STEP 5 — ServiceAccount/RBAC 확인

kubernetes 플러그인은 K8S API에서 Service/Endpoint를 watch합니다. RBAC가 깨지면 새 Service가 DNS에 안 보입니다. kubectl -n kube-system get clusterrole system:coredns -o yaml로 권한 확인.

모범 사례 체크리스트

  • CoreDNS Pod는 최소 2개 이상으로, anti-affinity로 다른 노드에 분산.
  • 중대형 클러스터(50노드+)에서는 NodeLocal DNSCache 도입을 표준으로.
  • 외부 API 호출이 많은 Pod에는 ndots:2 적용 또는 FQDN 끝에 명시적 . 사용.
  • Corefile은 ConfigMap이므로 GitOps로 관리(Argo CD/Flux). 직접 kubectl edit 금지.
  • Prometheus 메트릭 수집 + alert (응답 시간, 캐시 적중률, SERVFAIL 비율).
  • 업그레이드 전 항상 CoreDNS 변경 사항을 changelog로 확인 — 메이저 버전에서 deprecate되는 플러그인이 있음.
CHAPTER 08

이해도 확인 — 5문항 퀴즈

객관식입니다. 정답을 클릭하면 해설이 펼쳐집니다. 한 문제씩 차분히 답해보세요.

QUESTION 01K8S 클러스터에서 Pod가 DNS 질의를 보낼 때, 가장 먼저 도달하는 곳은?
정답: ② Pod의 /etc/resolv.conf에 nameserver 10.96.0.10이 박혀 있고, 이 IP가 바로 CoreDNS를 가리키는 kube-dns Service의 ClusterIP입니다. CoreDNS는 이름이 cluster.local 도메인이면 kubernetes 플러그인이 응답하고, 외부 도메인이면 forward 플러그인이 노드의 resolv.conf를 따라 외부로 던집니다.
QUESTION 02다음 중 kube-dns(legacy)와 비교한 CoreDNS의 가장 큰 구조적 차이는?
정답: ② kube-dns는 kubedns + dnsmasq + sidecar 세 컨테이너 조합이었고, CoreDNS는 단일 Go 바이너리에 플러그인을 Corefile로 체인 구성하는 깔끔한 구조입니다. K8S 외에서도 일반 권한 서버로 활용 가능합니다.
QUESTION 03Pod에서 api.openai.com 호출이 묘하게 느릴 때, 가장 가능성 높은 원인은?
정답: ② 점이 5개 미만이므로 default.svc.cluster.local, svc.cluster.local, cluster.local이 모두 붙어 시도된 후 NXDOMAIN을 받고서야 진짜 외부 질의가 나갑니다. 처방은 dnsConfig.options로 ndots:2 적용 또는 api.openai.com.처럼 끝에 점을 명시.
QUESTION 04NodeLocal DNSCache를 도입하는 가장 큰 이유는?
정답: ③ 노드 안에 캐시 DaemonSet을 두면 Pod → 같은 노드의 캐시(169.254.20.10)로 통신이 끝나므로, iptables NAT·conntrack을 우회하고 캐시 적중률이 크게 올라갑니다. 대규모 클러스터에서 UDP 5초 지연 버그 회피책으로도 표준입니다.
QUESTION 05ExternalDNS에 대한 설명으로 가장 정확한 것은?
정답: ② CoreDNS는 클러스터 안의 이름 해석을, ExternalDNS는 클러스터 밖 세상에 이름을 알리는 일을 합니다. 둘은 역할이 정반대이고, 보통 함께 씁니다. Ingress의 annotation external-dns.alpha.kubernetes.io/hostname을 활용합니다.
학습 후 여기까지 오셨다면, 다음은?

이제 다음 단계는 실제 클러스터(DGX Spark + ThinkStation 환경)에서 CoreDNS Corefile을 직접 만져보는 것입니다. 추천 실습 순서는: ①현재 Corefile 백업 → ②사내 도메인 stub 추가 → ③Prometheus 메트릭 수집 연동 → ④NodeLocal DNSCache 설치 → ⑤Headless Service로 StatefulSet 한 번 띄워보고 SRV 레코드 확인. 다섯 가지를 차례로 해보시면 CoreDNS는 더 이상 블랙박스가 아닙니다.
