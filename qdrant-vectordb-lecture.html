CH 01

왜 벡터 데이터베이스가 필요한가?

Qdrant를 배우기 전에, 애초에 "벡터 DB"라는 물건이 왜 세상에 등장했는지부터 이해해야 합니다. 이유를 알면 나머지 기능들이 전부 자연스럽게 이해됩니다.

1.1 전통 데이터베이스의 한계

PostgreSQL이나 MySQL 같은 관계형 DB는 정확히 일치하는 것을 찾는 데 최적화되어 있습니다. WHERE name = '김철수', WHERE price < 10000처럼요. 텍스트 검색도 LIKE '%쿠버네티스%'처럼 문자열이 포함되어 있는지만 봅니다.

그런데 사용자가 이렇게 검색한다면 어떨까요?

💬

"컨테이너 여러 대를 자동으로 관리해주는 도구가 뭐였지?"
→ 사용자가 원하는 답은 "쿠버네티스" 문서입니다. 하지만 질문 문장에는 '쿠버네티스'라는 단어가 한 글자도 없죠. LIKE 검색은 여기서 완전히 무력합니다.

사람은 "컨테이너 오케스트레이션"과 "컨테이너 여러 대를 자동으로 관리"가 같은 의미라는 걸 압니다. 이 "의미가 비슷하다"를 컴퓨터가 계산할 수 있게 만든 것이 바로 임베딩(Embedding)이고, 임베딩을 대규모로 저장하고 빠르게 검색하는 전용 시스템이 벡터 데이터베이스입니다.

Analogy · 도서관 비유

관계형 DB가 "책 제목의 가나다순 색인"이라면, 벡터 DB는 "비슷한 주제의 책들이 물리적으로 가까운 서가에 꽂혀 있는 도서관"입니다. "이 책과 비슷한 책 주세요"라고 하면 사서가 그 책이 꽂힌 서가 주변만 둘러보고 바로 건네줄 수 있죠. 전체 책을 다 뒤질 필요가 없습니다.

1.2 벡터 DB가 실제로 쓰이는 곳

🔍
시맨틱 검색

키워드가 아닌 "의미"로 문서를 찾습니다. 사내 위키, 고객센터 FAQ 검색 등.

🤖
RAG (검색 증강 생성)

LLM에게 관련 문서를 찾아 먹여주는 파이프라인의 핵심 저장소. 요즘 벡터 DB 수요의 1등 공신.

🎯
추천 시스템

"이 상품을 본 사람이 좋아할 상품", "비슷한 노래" 추천. 유저/아이템을 벡터로 표현해 이웃을 찾습니다.

🖼️
이미지/멀티모달 검색

이미지로 이미지 찾기, 텍스트로 이미지 찾기(CLIP 등). 벡터는 텍스트에만 국한되지 않습니다.

🚨
이상 탐지 / 중복 제거

정상 패턴 벡터들과 동떨어진 벡터를 찾아 이상 징후를 감지. AIOps 로그 분석에도 활용됩니다.

⚡
시맨틱 캐시

"의미상 같은 질문"에 대한 LLM 응답을 재사용해 비용과 지연시간을 절감합니다.

1.3 이 강의의 주인공, Qdrant

Qdrant(발음: 쿼드런트, quadrant에서 유래)는 Rust로 작성된 오픈소스 벡터 검색 엔진입니다. 2021년 첫 공개 이후 빠르게 성장해, 현재 Milvus·Weaviate와 함께 오픈소스 벡터 DB 3강으로 꼽힙니다.

🎯

Qdrant를 한 문장으로: "필터링에 강하고, 메모리를 아끼는 기술(양자화)이 뛰어나며, 운영이 단순한 Rust 기반 벡터 검색 엔진." 이 세 가지 강점을 이 강의 내내 반복해서 확인하게 될 겁니다.

CH 02

임베딩과 유사도 — 벡터 검색의 수학적 기초

벡터 DB를 다루려면 딱 두 가지 수학 개념만 알면 됩니다. "임베딩이 무엇인가"와 "가깝다는 걸 어떻게 재는가". 둘 다 고등학교 수준의 수학으로 충분합니다.

2.1 임베딩: 의미를 숫자 좌표로

임베딩은 텍스트, 이미지, 오디오 같은 데이터를 고정 길이의 숫자 배열(벡터)로 변환한 것입니다. 예를 들어 OpenAI의 text-embedding-3-small 모델은 어떤 문장이든 1,536개의 실수로 이루어진 벡터로 바꿔줍니다.

# 임베딩 모델은 "의미"를 좌표로 바꿔주는 함수입니다
embed("쿠버네티스는 컨테이너 오케스트레이션 도구다")
→ [0.12, -0.87, 0.44, ..., 0.03]  # 1536차원

embed("K8s는 여러 컨테이너를 자동 관리한다")
→ [0.11, -0.85, 0.47, ..., 0.05]  # 위와 매우 비슷한 좌표!

embed("오늘 점심은 김치찌개를 먹었다")
→ [-0.65, 0.31, -0.92, ..., 0.78]  # 완전히 다른 좌표

핵심 성질은 하나입니다: 의미가 비슷한 데이터는 벡터 공간에서 서로 가까운 곳에 놓인다. 임베딩 모델(BERT, E5, BGE, OpenAI embeddings 등)이 이 성질을 갖도록 학습되어 있습니다. 벡터 DB는 이 성질을 "믿고" 가까운 벡터를 빨리 찾는 데만 집중합니다.

Analogy · 지도 위의 도시

임베딩은 모든 문장에게 "의미 지도 위의 GPS 좌표"를 부여하는 일입니다. 서울과 인천이 지도에서 가깝듯, "고양이"와 "야옹이"는 의미 지도에서 가깝습니다. 다만 실제 지도는 2차원이지만 의미 지도는 384~4096차원이라는 점이 다를 뿐입니다.

두 벡터가 "가깝다"를 재는 방법은 크게 세 가지이고, Qdrant는 세 가지 모두를 지원합니다.

2.3 정확한 검색 vs 근사 검색 (kNN vs ANN)

벡터 1억 개 중에서 쿼리와 가장 가까운 10개를 찾으려면 어떻게 해야 할까요? 순진한 방법은 1억 번 전부 거리를 계산하는 것입니다(kNN, 완전 탐색). 정확하지만 느립니다.

그래서 실전에서는 ANN(Approximate Nearest Neighbor, 근사 최근접 이웃) 알고리즘을 씁니다. "100% 정확한 top-10은 포기하고, 99% 정도 정확한 top-10을 수천 배 빠르게 찾자"는 전략이죠. 이때 정확도를 리콜(recall)이라고 부릅니다.

🎯

벡터 DB의 본질: 결국 모든 벡터 DB는 "ANN 인덱스 + 저장 엔진 + API"의 조합입니다. Qdrant가 채택한 ANN 알고리즘이 바로 다음 장부터 다룰 HNSW입니다. 속도 ↔ 리콜 ↔ 메모리의 3각 트레이드오프를 조절하는 것이 벡터 DB 튜닝의 전부라고 해도 과언이 아닙니다.

CH 03

Qdrant 첫인상 — 무엇이고, 왜 선택하는가

이제 주인공을 자세히 살펴봅시다. Qdrant의 정체성, 아키텍처의 큰 그림, 그리고 다른 선택지 대신 Qdrant를 고르는 이유를 정리합니다.

3.1 Qdrant의 정체성

🦀
Rust 네이티브

GC(가비지 컬렉션)가 없는 Rust로 작성되어 지연시간이 예측 가능하고 메모리 효율이 높습니다. JVM 튜닝 같은 고민이 없습니다.

📦
단일 바이너리

의존성 없는 실행 파일 하나로 동작합니다. etcd, ZooKeeper, 별도 메타 스토어가 필요 없어 운영 부담이 매우 작습니다.

🔌
REST + gRPC API

모든 기능이 HTTP REST(6333 포트)와 gRPC(6334 포트)로 노출됩니다. 대량 처리엔 gRPC가 유리합니다.

🏷️
필터 가능한 HNSW

"카테고리=전자제품 AND 가격<5만원인 것 중에서 유사한 상품"처럼 조건 필터와 벡터 검색을 한 번에, 빠르게 처리하도록 설계됐습니다.

에어갭(폐쇄망) 환경 참고: Qdrant는 외부 의존성이 없는 단일 바이너리/이미지라 폐쇄망 프라이빗 클라우드에 반입하기 쉽습니다. 라이선스도 Apache 2.0이라 상용 제약이 없습니다. 온프레미스 K8s에 Helm 차트로 올리는 방법은 8장에서 다룹니다.

CH 04

Qdrant 핵심 구성 요소 — Collection, Point, Payload

Qdrant의 데이터 모델은 놀랄 만큼 단순합니다. 딱 세 단어만 기억하면 됩니다: Collection(컬렉션) → Point(포인트) → Payload(페이로드).

4.2 Point의 해부학

{
  "id": 42,                          // 정수 또는 UUID
  "vector": [0.12, -0.87, ...],       // 임베딩 벡터 (컬렉션의 차원과 일치해야 함)
  "payload": {                       // 자유 형식 JSON — 스키마 강제 없음!
    "title": "쿠버네티스 네트워킹 가이드",
    "category": "infra",
    "tags": ["k8s", "cni", "network"],
    "published_at": 1719800000,
    "view_count": 1523,
    "is_public": true,
    "location": { "lon": 126.97, "lat": 37.56 }  // 지리 좌표도 지원
  }
}

페이로드는 스키마리스입니다. 포인트마다 다른 필드를 가져도 되고, 중첩 JSON도 됩니다. 다만 6장에서 배울 "필터링"을 빠르게 하려면 자주 필터링하는 필드에 페이로드 인덱스를 만들어줘야 합니다.

4.3 하나의 포인트에 여러 벡터: Named Vectors

실무에서는 하나의 문서에 대해 여러 종류의 벡터가 필요한 경우가 많습니다. 예를 들어 상품 하나에 대해 "제목 임베딩", "이미지 임베딩", "희소(sparse) 벡터"를 모두 저장하고 싶을 수 있죠. Qdrant는 Named Vectors로 이를 지원합니다.

{
  "id": 7,
  "vector": {
    "text_dense":  [0.1, 0.3, ...],   // 1024차원, Cosine
    "image_clip":  [0.7, 0.2, ...],   // 512차원, Dot — 벡터마다 설정이 달라도 OK
    "text_sparse": { "indices": [12, 873], "values": [0.8, 0.4] }  // 희소 벡터
  },
  "payload": { "product": "노트북" }
}
📌

Dense vs Sparse 벡터: 지금까지 본 임베딩은 모든 자리에 값이 있는 밀집(dense) 벡터입니다. 반면 희소(sparse) 벡터는 대부분이 0이고 소수의 위치에만 값이 있는 벡터로, BM25나 SPLADE 같은 키워드 기반 검색을 벡터 연산으로 표현한 것입니다. 10장의 하이브리드 검색에서 이 둘을 결합합니다.

CH 05

HNSW — Qdrant의 심장

Qdrant의 벡터 검색 속도는 HNSW(Hierarchical Navigable Small World) 인덱스에서 나옵니다. 이름은 무섭지만, 원리는 "고속도로 → 국도 → 골목길" 비유 하나로 이해할 수 있습니다.

5.1 직관적 이해: 고속도로 내비게이션

Analogy · 서울에서 부산 골목집 찾아가기

서울에서 부산의 어느 골목집을 찾아간다면 어떻게 하나요? ① 먼저 고속도로로 부산 근처까지 빠르게 이동하고 → ② 국도로 해당 구까지 간 다음 → ③ 골목길을 걸어 정확한 집을 찾습니다. 처음부터 골목길만으로 이동하면 몇 달이 걸리겠죠. HNSW가 정확히 이 구조입니다: 위층은 롱점프용 성긴 그래프(고속도로), 아래층으로 내려갈수록 촘촘한 그래프(골목길)입니다.

CH 06

필터링과 페이로드 인덱스 — Qdrant의 최대 강점

실무 검색은 순수 벡터 검색만으로 끝나지 않습니다. "공개 문서 중에서, 최근 1년 내 작성된 것 중에서 유사한 문서"처럼 조건이 항상 붙죠. 이 "필터 + 벡터 검색" 조합이 Qdrant가 가장 자신 있는 영역입니다.

6.1 왜 필터링이 어려운 문제인가

ANN 인덱스와 필터를 결합하는 전통적 방식은 두 가지인데, 둘 다 함정이 있습니다.

🔻
Post-filtering (선 검색, 후 필터)

벡터로 top-100을 찾은 뒤 조건에 맞는 것만 남깁니다. 문제: 필터가 빡빡하면(예: 조건 만족 1%) 100개 중 1개만 남아 결과가 텅 빌 수 있습니다.

🔺
Pre-filtering (선 필터, 후 완전탐색)

조건에 맞는 후보를 먼저 추린 뒤 그 안에서 거리 계산. 문제: 후보가 수백만 개면 완전 탐색이 되어 느려집니다.

Qdrant는 제3의 길, filterable HNSW를 택했습니다. HNSW 그래프를 탐색하는 도중에 필터 조건을 실시간으로 평가하면서, 필터 때문에 그래프 연결이 끊어지는 문제를 추가 링크와 카디널리티 추정으로 보완합니다. 필터 통과 비율이 아주 낮다고 추정되면 자동으로 페이로드 인덱스 기반 완전 탐색으로 전환하기도 합니다. 사용자는 이 전략 선택을 신경 쓸 필요 없이 그냥 필터를 던지면 됩니다.

6.2 필터 문법: must / should / must_not

Qdrant의 필터는 불리언 논리의 세 절(clause)로 구성됩니다. Elasticsearch를 써봤다면 익숙할 구조입니다.

절논리의미
mustAND나열된 조건을 모두 만족해야 함
shouldOR나열된 조건 중 하나 이상 만족
must_notNOT나열된 조건을 모두 만족하지 않아야 함
from qdrant_client import models

# "infra 카테고리이면서, 조회수 1000 이상, 초안(draft)은 제외"
flt = models.Filter(
    must=[
        models.FieldCondition(key="category",
                              match=models.MatchValue(value="infra")),
        models.FieldCondition(key="view_count",
                              range=models.Range(gte=1000)),
    ],
    must_not=[
        models.FieldCondition(key="status",
                              match=models.MatchValue(value="draft")),
    ],
)
{
  "filter": {
    "must": [
      { "key": "category",   "match": { "value": "infra" } },
      { "key": "view_count", "range": { "gte": 1000 } }
    ],
    "must_not": [
      { "key": "status", "match": { "value": "draft" } }
    ]
  }
}

지원하는 조건 타입도 풍부합니다: 값 일치(match), 다중 값(MatchAny = SQL의 IN), 텍스트 부분 일치(MatchText), 숫자/날짜 범위(range), 지리 반경·박스·폴리곤(geo_radius 등), 배열 길이(values_count), 필드 존재 여부(is_empty, is_null), 중첩 객체 필터까지.

6.3 페이로드 인덱스 — 필터를 빠르게

필터를 자주 거는 필드에는 반드시 페이로드 인덱스를 생성하세요. 인덱스가 없으면 필터 평가가 세그먼트 전체 스캔이 되어 느려집니다. RDB에서 WHERE 절에 쓰는 컬럼에 인덱스를 거는 것과 같은 이치입니다.

client.create_payload_index(
    collection_name="docs",
    field_name="category",
    field_schema="keyword",   # keyword | integer | float | bool |
)                              # datetime | geo | text | uuid

# 전문(full-text) 인덱스: MatchText 필터가 빨라짐
client.create_payload_index(
    collection_name="docs",

한국어 주의: 텍스트 인덱스 토크나이저로 multilingual을 선택해야 한국어 형태소가 제대로 분리됩니다. 기본 word 토크나이저는 공백 기준이라 한국어에 부정확합니다.

🎯

핵심 요약: ① 필터는 벡터 검색과 "동시에" 처리된다(성능 저하 최소). ② 필터에 쓰는 필드엔 페이로드 인덱스 필수. ③ 이 필터링 성능이 Milvus·Pinecone 대비 Qdrant를 고르는 가장 흔한 이유 중 하나다.

CH 07

양자화(Quantization) — 메모리를 1/4에서 1/32까지

벡터 DB 운영 비용의 대부분은 메모리입니다. float32 벡터 1억 개(768차원)면 벡터만 약 286GB죠. 양자화는 이 비용을 극적으로 줄이는 Qdrant의 대표 무기입니다. LLM 양자화(INT8, AWQ 등)를 접해봤다면 개념이 바로 통합니다.

9.2 데이터 적재 (Upsert)

Qdrant의 쓰기는 upsert(있으면 갱신, 없으면 삽입) 하나로 통일됩니다. 임베딩 생성까지 포함한 현실적인 코드를 봅시다.

from sentence_transformers import SentenceTransformer

model = SentenceTransformer("BAAI/bge-m3")   # 한국어 지원 우수 (1024차원)

docs = [
    {"id": 1, "text": "쿠버네티스는 컨테이너 오케스트레이션 플랫폼이다", "category": "infra"},
    {"id": 2, "text": "vLLM은 PagedAttention으로 LLM 서빙 처리량을 높인다", "category": "llmops"},
    {"id": 3, "text": "Redis는 인메모리 키-값 저장소다", "category": "database"},
]

vectors = model.encode([d["text"] for d in docs], normalize_embeddings=True)

client.upsert(
    collection_name="tech_docs",
    points=[
        models.PointStruct(
            id=d["id"],
            vector=v.tolist(),
            payload={"text": d["text"], "category": d["category"]},
        )
        for d, v in zip(docs, vectors)
    ],
    wait=True,   # 디스크 반영까지 대기 (테스트 시 편리)
)
# pip install "qdrant-client[fastembed]"
# 임베딩 생성 + 업서트를 Qdrant 클라이언트가 알아서 처리 (ONNX, CPU로 동작)

client.add(
    collection_name="tech_docs_fe",
    documents=[
        "쿠버네티스는 컨테이너 오케스트레이션 플랫폼이다",

hits = client.query(collection_name="tech_docs_fe",
                    query_text="컨테이너 관리 도구", limit=3)

CH 10

하이브리드 검색 — 의미 검색과 키워드 검색의 결합

순수 벡터 검색의 아킬레스건은 고유명사·코드·모델명입니다. "GPT-4o-mini 요금"을 검색했는데 임베딩이 "GPT-4o 요금" 문서를 더 유사하다고 판단할 수 있죠. 이를 보완하는 것이 키워드 정확 매칭에 강한 희소 벡터와의 하이브리드 검색입니다.

10.1 Dense와 Sparse는 서로의 약점을 메운다

 Dense (의미 기반)Sparse (키워드 기반, BM25/SPLADE)
강점동의어, 패러프레이즈, 다국어 의미 매칭고유명사, 버전 번호, 에러 코드, 희귀 용어 정확 매칭
약점처음 보는 고유명사·코드에 약함동의어를 전혀 모름 ("K8s" ≠ "쿠버네티스")
예시 쿼리"컨테이너 관리 도구" → 쿠버네티스 문서 발견"CVE-2025-1234" → 정확히 그 문서 발견

10.2 Query API로 하이브리드 구현하기

Qdrant v1.10부터 도입된 Query API는 prefetch(후보 수집)와 fusion(결과 융합)으로 하이브리드 검색을 서버 한 번 호출로 처리합니다. 융합 방식은 RRF(Reciprocal Rank Fusion — 순위 기반, 점수 스케일 무관)가 표준입니다.

client.create_collection(
    collection_name="hybrid_docs",
    vectors_config={
        "dense": models.VectorParams(size=1024, distance=models.Distance.COSINE),
    },
    sparse_vectors_config={
        "sparse": models.SparseVectorParams(
            modifier=models.Modifier.IDF   # BM25 스타일 IDF 가중치를 서버가 계산
        ),
    },
)
results = client.query_points(
    collection_name="hybrid_docs",
    prefetch=[
        models.Prefetch(query=dense_vec,  using="dense",  limit=50),
        models.Prefetch(
            query=models.SparseVector(indices=sp_idx, values=sp_val),
            using="sparse", limit=50),
    ],
    query=models.FusionQuery(fusion=models.Fusion.RRF),  # 순위 융합
    limit=10,
)
# 실전 RAG 검색 파이프라인: 하이브리드 → 크로스인코더 리랭킹
# 후보를 넉넉히(50) 뽑고 → 정밀 모델로 최종 5개 선별

from sentence_transformers import CrossEncoder
reranker = CrossEncoder("BAAI/bge-reranker-v2-m3")

candidates = client.query_points(..., limit=50).points   # 위의 하이브리드
pairs = [(query_text, p.payload["text"]) for p in candidates]
scores = reranker.predict(pairs)

top5 = [p for _, p in sorted(zip(scores, candidates),
                              key=lambda x: -x[0])][:5]
💡

어디까지 해야 하나? 실무 체감 기준으로 ① dense 단독 → ② +sparse 하이브리드(RRF) → ③ +리랭커 순으로 검색 품질이 계단식으로 좋아집니다. 사내 문서처럼 고유명사·제품명·에러코드가 많은 도메인일수록 ②의 효과가 큽니다. ③은 지연시간이 늘어나므로 top-k 후보 수와 함께 SLA를 보며 결정하세요.

10.3 그 밖의 Query API 능력들

  • 추천(Recommend): positive=[id...], negative=[id...]로 "이건 좋고 저건 싫어" 스타일 검색. 쿼리 벡터 없이 기존 포인트 ID만으로 가능합니다.
  • Discovery: 컨텍스트 쌍(positive/negative)으로 탐색 공간을 제한하는 고급 탐색.
  • Multi-stage: prefetch를 중첩해 "바이너리 양자화로 1000개 → INT8로 100개 → 원본으로 10개" 같은 다단 정밀화도 서버 안에서 처리됩니다.
  • Grouping: group_by="doc_id"로 "문서당 최고 청크 1개씩"처럼 중복을 억제한 결과를 받을 수 있습니다. RAG에서 같은 문서의 청크가 top-k를 독식하는 문제를 해결합니다.
  • Random sampling / order_by: 페이로드 값 기준 정렬, 무작위 샘플링 등 유틸리티성 조회도 지원합니다.

12.2 청킹 설계 — 검색 품질의 절반

  • 청크 크기: 한국어 기술 문서 기준 300~800 토큰이 무난한 시작점. 너무 작으면 맥락 부족, 너무 크면 임베딩이 뭉개지고 LLM 컨텍스트 낭비.
  • 오버랩: 청크 경계에서 문맥이 잘리는 것을 막기 위해 10~20% 겹침을 둡니다.
  • 구조 존중: 마크다운 헤더, 코드 블록, 표 경계를 무시하고 자르면 품질이 급락합니다. 구조 인식 스플리터를 쓰세요.
  • 메타데이터 풍부하게: 청크마다 doc_id, title, section, source_url, updated_at을 페이로드에 넣어야 필터링·출처 표시·갱신이 가능해집니다.

12.3 핵심 코드: 적재부터 답변 생성까지

import uuid
from qdrant_client import QdrantClient, models
from sentence_transformers import SentenceTransformer
from fastembed import SparseTextEmbedding

dense_model  = SentenceTransformer("BAAI/bge-m3")
sparse_model = SparseTextEmbedding("Qdrant/bm25")
client = QdrantClient(url="http://qdrant.vector-db:6333", prefer_grpc=True)

def index_chunks(chunks: list[dict]):
    texts  = [c["text"] for c in chunks]
    dense  = dense_model.encode(texts, normalize_embeddings=True)
    sparse = list(sparse_model.embed(texts))

    client.upload_points(   # 배치·재시도·병렬 내장
        collection_name="kb_chunks",
        points=[
            models.PointStruct(
                id=str(uuid.uuid5(uuid.NAMESPACE_URL,
                       f"{c['doc_id']}#{c['chunk_no']}")),  # 결정적 ID → 재실행해도 중복 없음
                vector={
                    "dense": d.tolist(),
                    "sparse": models.SparseVector(
                        indices=s.indices.tolist(), values=s.values.tolist()),
                },
                payload={"text": c["text"], "doc_id": c["doc_id"],
                         "title": c["title"], "updated_at": c["updated_at"]},
            )
            for c, d, s in zip(chunks, dense, sparse)
        ],
    )
def answer(question: str) -> str:
    dq = dense_model.encode(question, normalize_embeddings=True)
    sq = next(sparse_model.embed([question]))

    hits = client.query_points(
        collection_name="kb_chunks",
        prefetch=[
            models.Prefetch(query=dq.tolist(), using="dense", limit=25),
            models.Prefetch(query=models.SparseVector(
                indices=sq.indices.tolist(), values=sq.values.tolist()),
                using="sparse", limit=25),
        ],
        query=models.FusionQuery(fusion=models.Fusion.RRF),
        limit=5,
        with_payload=True,
    ).points

    context = "\n\n---\n\n".join(
        f"[출처: {h.payload['title']}]\n{h.payload['text']}" for h in hits)

    prompt = f"""아래 컨텍스트만 근거로 질문에 답하세요.
컨텍스트에 답이 없으면 "문서에서 찾을 수 없습니다"라고 답하세요.

# 컨텍스트
{context}

# 질문
{question}"""
    return call_llm(prompt)   # vLLM / OpenAI 호환 엔드포인트 등

Q1 / 8 · Chapter 1~2
전통적인 RDB의 LIKE '%검색어%' 대신 벡터 DB를 쓰는 근본적인 이유는 무엇일까요?
해설. "노트북이 너무 뜨거워요"와 "랩탑 발열 문제"는 겹치는 단어가 거의 없지만 의미는 같습니다. 임베딩 모델이 의미를 좌표(벡터)로 바꾸고, 벡터 DB는 그 좌표 공간에서 가까운 이웃을 찾아줍니다. 속도나 트랜잭션은 본질이 아닙니다 — 오히려 트랜잭션은 RDB가 훨씬 강합니다.
Q2 / 8 · Chapter 2
정규화된 임베딩 벡터에서 가장 흔히 쓰이는 유사도 지표는?
해설. 임베딩에서 "의미"는 벡터의 방향에 담깁니다. 코사인 유사도는 두 벡터의 사잇각만 보므로 문장 길이 등에 따른 크기 차이에 둔감합니다. 참고로 벡터를 길이 1로 정규화하면 코사인 유사도와 내적(Dot)이 같아져서, 많은 시스템이 내부적으로는 더 계산이 싼 내적을 사용합니다.
Q3 / 8 · Chapter 4
Qdrant의 Payload를 RDB 개념에 대응시키면 가장 가까운 것은?
해설. Point = 행(row), Vector = 검색 키, Payload = 나머지 컬럼들입니다. Payload는 스키마리스 JSON이라 자유롭게 넣을 수 있고, must / should / must_not 필터의 대상이 됩니다. 자주 필터링하는 필드에는 페이로드 인덱스를 만들어야 성능이 나옵니다.
Q4 / 8 · Chapter 5
HNSW에서 검색 시점 파라미터 ef를 올리면 일반적으로 일어나는 일은?
해설. ef는 검색 중 유지하는 후보 리스트의 크기입니다. 더 넓게 탐색하니 정확도(리콜)는 오르고 지연은 늘어납니다. 그래프 크기·메모리를 결정하는 건 빌드 파라미터 m이고, 빌드 시간에 영향을 주는 건 ef_construct입니다. ef는 쿼리마다 바꿀 수 있는 유일한 "무료 손잡이"라는 점이 실무 포인트입니다.
Q5 / 8 · Chapter 6
"카테고리=노트북인 문서 중에서 유사한 것 top-10"을 검색할 때, Qdrant의 filterable HNSW가 해결하는 문제는?
해설. 순진한 post-filtering은 "일단 top-10을 찾고 → 필터로 거른다" 방식이라, 10개 중 노트북이 2개뿐이면 결과가 2개만 남습니다. Qdrant는 그래프 탐색 도중에 필터 조건을 평가해 조건에 맞는 이웃만 따라가므로, 필터가 있어도 온전한 top-k를 돌려줍니다. 이것이 Qdrant의 대표적인 차별점 중 하나입니다.
Q6 / 8 · Chapter 7
1억 개 × 1024차원 float32 벡터의 RAM이 부담될 때, 정확도 손실을 최소화하면서 메모리를 크게 줄이는 표준 전략은?
해설. Qdrant의 권장 패턴은 ① 압축 벡터(Scalar INT8은 ¼, Binary는 1/32)로 빠르게 후보를 넓게 뽑고(oversampling) ② 디스크의 float32 원본으로 정확한 점수를 다시 계산(rescore)하는 2단계입니다. 속도·메모리·정확도를 모두 챙기는 실무 표준입니다. gzip은 유사도 계산이 불가능하고, m 축소는 리콜을 희생합니다.
Q7 / 8 · Chapter 10
하이브리드 검색에서 RRF(Reciprocal Rank Fusion)가 하는 일은?
해설. dense(코사인 0~1)와 sparse(BM25류, 상한 없음)는 점수 스케일이 달라 단순 합산이 불가능합니다. RRF는 점수 대신 순위만 사용해(1/(k+rank)) 두 리스트를 융합하므로 스케일 문제를 우회합니다. Qdrant Query API에서는 prefetch + FusionQuery(RRF) 한 번의 호출로 처리됩니다.
Q8 / 8 · Chapter 11~13
이미 PostgreSQL 중심으로 운영 중인 팀이 수십만 건 규모의 소규모 시맨틱 검색을 붙이려 합니다. 가장 합리적인 첫 선택은?
해설. "최고의 DB"가 아니라 "지금 상황에 맞는 DB"가 정답입니다. 수십만~수백만 건에 이미 PostgreSQL 운영 역량이 있다면 pgvector가 운영 비용 최소의 선택입니다. 트래픽과 데이터가 커져 전용 엔진이 필요해지는 시점에 Qdrant로 이전해도 늦지 않습니다. 반대로 처음부터 수억 건 이상 + 고급 기능이 필요하면 전용 벡터 DB로 시작하는 것이 맞습니다.

15.1 한 장 요약

🧭
개념

임베딩은 의미를 좌표로 바꾸고, 벡터 DB는 그 좌표 공간의 이웃을 찾는다. 정확한 kNN 대신 HNSW 기반 ANN으로 속도와 정확도를 교환한다.

🧱
구조

Collection → Point(id + vector + payload). 필터는 must/should/must_not, 자주 쓰는 필드엔 페이로드 인덱스. Qdrant는 탐색 도중 필터를 평가하는 filterable HNSW.

⚖️
자원

RAM ≈ 벡터 수 × 차원 × 바이트 × 1.5. 부담되면 Scalar/Binary 양자화 + on_disk 원본 + rescoring. 손잡이는 m / ef_construct / ef 세 개.

🔀
응용

dense + sparse를 RRF로 융합하는 하이브리드 검색, 크로스인코더 리랭킹, 멀티테넌시(is_tenant), alias 무중단 모델 교체 — 프로덕션 RAG의 표준 부품들.
