모델은 똑똑해졌다.
그런데 왜 자꾸 실패할까?

에이전트 = 모델 + 하네스

프롬프트 엔지니어링은 “무엇을 물어볼까”를 다룹니다. 컨텍스트 엔지니어링은 “무엇을 보여줄까”를 다룹니다. 하네스 엔지니어링(Harness Engineering)은 그 위에서 “모델이 일하는 환경 전체를 어떻게 설계할까”를 다룹니다. 이 강의는 그 개념을 처음부터, 일반적인 LLM 사용과 나란히 놓고 설명합니다.

난이도: 입문 예상 30분 코드 예제 포함 실습형 데모
Chapter 01

왜 2026년에 갑자기 이 개념이 등장했나

먼저 흐름부터 잡아두면 나머지가 쉬워집니다. LLM을 다루는 기술은 지난 몇 년간 단계적으로 “모델 바깥쪽”으로 확장되어 왔습니다. 모델 성능이 올라갈수록, 정작 결과를 좌우하는 것은 모델 자체가 아니라 모델을 둘러싼 환경이 되었기 때문입니다.

2023–24

프롬프트 엔지니어링

질문 한 번, 답 한 번. 문구·예시·구조를 다듬어 단일 응답의 품질을 끌어올리던 시기.

2025 중반

컨텍스트 엔지니어링

RAG·MCP·메모리로 모델이 무엇을 보게 할지를 설계. 한 번의 응답을 넘어 시스템 차원의 정보 공급으로 확장.

2026.02

하네스 엔지니어링

HashiCorp 공동창업자 Mitchell Hashimoto가 2월 블로그 글에서 이름을 붙였고, 며칠 뒤 OpenAI·Anthropic이 실전 사례로 확장하면서 용어가 자리잡음.

핵심 전환점 에이전트가 “쓸 만큼은 똑똑해졌지만, 혼자 믿고 맡길 만큼 안정적이지는 않은” 지점에 도달한 것이 결정적이었습니다. 모델 성능이 더 이상 병목이 아니게 되자, 병목은 “실제 시스템 안에서 안정적으로 행동하게 만들 수 있는가”로 옮겨갔습니다.
Chapter 02

Harness Engineering이란 무엇인가

Harness는 원래 말에게 씌우는 마구(馬具)를 뜻합니다. 고삐, 안장, 재갈처럼 말의 힘을 올바른 방향으로 끌어내고, 제멋대로 달아나지 않게 잡아주는 장비 일체입니다.

직관을 잡는 비유 말(LLM)은 그 자체로 힘이 셉니다. 하지만 마구가 없으면 밭을 갈 수 없습니다. 고삐와 멍에가 있어야 그 힘을 “유용한 일”로 돌릴 수 있죠. 열 마리를 동시에 풀어놓으면 더 엉망이 됩니다. 하네스는 강력하지만 예측이 어려운 말을, 안정적인 일꾼으로 바꾸는 장치입니다.

정의는 의외로 단순합니다. 업계가 수렴한 공식은 이렇습니다.

에이전트(Agent) = 모델(Model) + 하네스(Harness)

하네스 = 모델 자신을 제외한 모든 코드·설정·실행 로직

즉 하네스는 모델을 감싸는 결정론적 실행 계층입니다. 모델이 “이 행동을 하겠다”고 제안하면, 하네스는 그것을 검증하고(schema), 허가하고(permission), 실행하고(execute), 기록합니다(log). 책임이 분리되어 있다는 점이 핵심입니다 — 모델은 제안하고, 하네스는 집행합니다.

그리고 Hashimoto가 제시한 운영 철학이 이 분야의 정체성을 만들었습니다.

하네스 엔지니어링의 한 문장 “에이전트가 실수를 할 때마다, 그 실수를 구조적으로 다시는 못 하게 환경에 영구적인 수정을 새겨 넣는다.”
— 같은 실수를 프롬프트로 “다시 부탁”하는 게 아니라, 시스템 문제로 보고 영구히 막는다는 발상입니다.
Chapter 03

일반 LLM 활용 vs 하네스 기반 에이전트

이 강의에서 가장 중요한 비교입니다. 우리가 흔히 하는 “LLM 활용”은 대부분 단발성 호출입니다. 질문을 넣으면 답이 나오고, 사람이 그 답을 읽고 판단합니다. 하네스 기반 에이전트는 그 답을 실행하고, 검증하고, 틀리면 스스로 고치게 만드는 환경입니다.

세 층위로 보면 더 명확합니다

서로 대체 관계가 아니라 겹겹이 쌓이는 관계입니다. 위로 갈수록 다루는 범위가 넓어집니다.

층위한 줄 정의다루는 범위
프롬프트 엔지니어링무엇을 물어볼지한 번의 대화 — 문구, 예시, 구조
컨텍스트 엔지니어링무엇을 보여줄지그 순간 컨텍스트 창에 무엇을 넣고/요약하고/버릴지
하네스 엔지니어링전체가 어떻게 돌아가는지도구·권한·상태·검증·재시도·로그 등 환경 전체
자동차 비유 모델은 엔진입니다 — 차를 사면 따라옵니다. 컨텍스트는 연료와 계기판 정보죠. 하네스는 나머지 차 전체입니다 — 핸들, 브레이크, 차선 유지, 경고등, 그리고 “고속도로에서 문짝이 떨어지지 않게 하는 것”. 엔진과 연료만 신경 쓰면, 여전히 형편없는 차를 만들 수 있습니다.

같은 일을 두 방식으로: 직접 토글해보기

예시 과제: “웹앱에 로그인 기능을 추가해줘.” 버튼을 눌러 두 방식의 진행을 비교해보세요.

실행 방식 비교
비교 축일반 LLM 활용하네스 기반 에이전트
실행 형태단발성 1회 호출도구를 쓰며 도는 반복 루프
상태(메모리)매 호출마다 백지에서 시작파일·git 등 모델 밖에 상태 보존
결과의 처리사람이 읽고 직접 실행/검증하네스가 실행·검증·재시도
“다 됐나?” 판단모델 말만 믿음 (대개 과신)별도 검증자가 증거로 판정
실패 대응프롬프트 다시 작성환경에 영구 수정 → 재발 차단
적합한 작업단순·단일 턴 작업다단계·장시간·자율 작업
기억할 것 단순한 한 번짜리 작업에는 하네스가 필요 없습니다. 하네스는 여러 단계를 거치거나 오래 도는 작업에서 비로소 값을 합니다. “항상 에이전트”가 아니라, 에이전트를 써서 손해 보는 순간을 아는 것도 실력입니다.
Chapter 04

하네스의 구성요소 — 무엇으로 이루어지나

1차 출처들(OpenAI 실험, Anthropic 엔지니어링 글, Hashimoto의 실천)에서 반복적으로 등장하는 구성 블록은 대략 여섯 가지입니다. 처음부터 전부 만들 필요는 없습니다. 필요한 곳부터 하나씩 붙입니다.

01

지침 파일

CLAUDE.md / AGENTS.md / .cursorrules. 에이전트가 작업을 시작할 때 읽는 문서. 프로젝트 구조, 빌드 명령, 코딩 규칙, 명명 규칙. “기록의 원천”으로 동작.

02

도구 / 도구 레지스트리

파일 읽기·쓰기, 코드 실행, 테스트 러너, 브라우저, git, HTTP. 각 호출을 스키마·권한·예산 기준으로 검증한 뒤 실행하고 결과를 돌려줌.

03

상태 / 메모리

모델은 매번 백지에서 시작하므로, 파일시스템을 외부 기억으로 사용. progress.txt(작업 노트), feature-list.json(계획), git 히스토리.

04

가드레일 / 권한 / 훅

“할 수 있는 행동”의 경계. 결정론적 린터·구조 테스트·enforcement hook으로 위반을 사람이 아니라 기계가 막음.

05

검증 게이트 / 평가

별도의 평가자(evaluator)가 “완료” 여부를 증거 기반으로 판정. 기본값은 실패(default-FAIL); 증거 없이는 통과로 못 바꿈.

06

오케스트레이션 루프

계획→실행→검증→리셋을 도는 바깥쪽 제어. initializer/coding 분리, planner–generator–evaluator 같은 패턴이 여기에 해당.

한 가지 통찰 하네스의 모든 구성요소는 “모델이 혼자서는 못 하는 일이 무엇인가”에 대한 가정을 담고 있습니다. 그래서 모델이 좋아지면 이 가정은 낡습니다. 뒤에서 다시 다룹니다.
Chapter 05

핵심 철학을 손으로: 실수 → 영구 수정 루프

말로만 들으면 추상적입니다. 직접 돌려봅시다. 아래 에이전트는 작업을 하다 같은 종류의 실수를 반복합니다. “하네스 수정” 버튼을 누를 때마다, 그 실수를 막는 규칙이 AGENTS.md에 영구히 새겨지고, 다음부터는 같은 실수가 사라집니다.

▶ “작업 실행”을 눌러 에이전트를 한 번 돌려보세요.
$ agent run --task "결제 모듈 리팩터링"
대기 중…
AGENTS.md — 누적되는 규칙
린터 규칙을 끄지 말 것. 코드를 고쳐서 통과시킨다. 완료 선언 전 반드시 end-to-end 테스트를 실제로 실행한다. 레이어 의존 방향 위반(상위→하위 역참조) 금지.
아직 새겨진 규칙이 없습니다.

포인트: 규칙은 한 번에 다 쓰는 게 아니라, 에이전트가 같은 자리에서 반복해서 넘어질 때 한 줄씩 늘어납니다. 각 줄은 “실제로 관찰된 실패 하나”에 대응합니다. 이게 Hashimoto가 말한 그 습관입니다.

Chapter 06

직접 시작해보기 — 최소한의 하네스

가장 빠르게 효과를 보는 출발점은 세 가지입니다. ① 프로젝트 루트에 AGENTS.md(또는 CLAUDE.md)를 만들고, ② 에이전트가 같은 곳에서 반복 실패할 때 규칙을 한 줄씩 추가하고, ③ 올바른 동작을 기계적으로 검증하는 도구를 붙입니다. 아래 탭에서 실제 형태를 보세요. 

# AGENTS.md  (프로젝트 루트에 배치 — 에이전트가 시작 시 읽음)

## Build
# 전체 빌드
./gradlew build
# 테스트
./gradlew test

## 코딩 규칙
# 패키지 의존 방향: domain -> application -> infrastructure
# infrastructure 는 domain 을 직접 참조하지 않는다
# 엔티티는 기본 lazy 로딩, N+1 은 fetch join 으로 해결

## 금지 (관찰된 실패에서 추가됨)
# - 린터 규칙을 끄지 말 것 (.eslintrc 수정 금지)
# - 테스트를 삭제하거나 수정하지 말 것
# - 완료 선언 전 e2e 테스트를 실제로 실행할 것

## Commit
# 커밋 메시지는 한국어로, 끝에 마침표 없이
# 가장 단순한 하네스: 모델 + 루프 + 도구 + 검증
def run_agent(task, max_steps=40):
    state = load_state("progress.txt")   # 외부 기억에서 복원
    rules = read("AGENTS.md")            # 지침 주입

    for step in range(max_steps):
        action = model.propose(task, state, rules)  # 모델은 "제안"만

        if not harness.validate(action):  # 스키마·권한·예산 검증
            state += f"거부됨: {action}"
            continue

        result = harness.execute(action)  # 하네스가 "집행"
        harness.log(action, result)       # 모든 행동 기록
        state = persist(state, result)     # 상태를 파일로 저장

        if evaluator.is_done(task, state):   # 별도 검증자가 판정
            return "완료"
    return "미완료 — 다음 세션으로 이어감"
#!/usr/bin/env bash
# pre-done.sh : "완료" 선언 전에 자동 실행되는 검증 게이트
# 하나라도 실패하면 에이전트는 완료로 넘어갈 수 없다 (default-FAIL)

set -e

# 1) 린터: 규칙을 끄지 않았는지 + 통과 여부
git diff --name-only | grep -q ".eslintrc" && {
  echo "FAIL: 린터 설정을 수정했습니다"; exit 1; }
npm run lint

# 2) 테스트가 삭제되지 않았는지
test $(find . -name "*.test.*" | wc -l) -ge $EXPECTED

# 3) e2e 를 실제로 실행 (모델 말만 믿지 않음)
npx playwright test

echo "PASS: 완료 조건 충족"
# progress.txt — 세션 간 인수인계 노트 (모델 밖 기억)
# 새 세션은 백지에서 시작하므로, 이 파일이 "교대 근무 인수인계서"

OpenAI Codex — 100만 줄 실험

3명으로 시작해 7명까지 늘어난 팀이 2025년 8월부터 약 5개월간, 빈 저장소에서 Codex 에이전트만으로 프로덕션 소프트웨어를 구축했습니다. 약 1,500개의 PR이 병합되었고, 사람이 직접 작성한 코드는 한 줄도 없었습니다.

중요한 점: 처음부터 잘 된 게 아니었습니다. 초기엔 환경 설정·도구 연동·복구 로직이 부실해 생산성이 낮았고, 하네스를 단계적으로 개선하면서 성능이 급격히 올랐습니다. 잘 돌 때는 수작업 대비 약 10배 빨랐다고 보고됐습니다.

Anthropic — 장시간 에이전트 하네스

핵심 문제는 “여러 컨텍스트 창에 걸친 작업에서 매 세션이 백지에서 시작한다”는 것. 해법은 처음 한 번만 도는 initializer 에이전트가 환경을 세팅하고(init.sh, 수백 개 요구사항으로 확장한 feature-list.json, 첫 git 커밋), 이후 coding 에이전트가 매 세션 한 기능씩 점진적으로 진행하며 claude-progress.txt에 인수인계를 남기는 구조입니다.

요약(compaction)만으로는 부족해, 아주 긴 작업에서는 세션을 허물고 구조화된 인수인계 파일에서 전체 컨텍스트 리셋 후 재구성했습니다. 사람이 신입을 온보딩하는 방식 그대로죠.

Anthropic — 계획·생성·평가 삼각 구조

2D 게임 엔진 제작 실험에서, Planner는 짧은 프롬프트를 제품 명세로 확장하되 구현 세부는 일부러 비워둡니다(이른 과잉 명세가 하류 오류로 번지기 때문). Generator는 스프린트 단위로 구현하되, 코드를 쓰기 전에 Evaluator와 “완료의 정의”를 담은 계약을 먼저 맺습니다.

Evaluator는 Playwright로 실제 사용자처럼 클릭해 UI·API·DB 동작을 검증합니다. 하나라도 실패하면 스프린트가 실패. 반면 단독 에이전트는 “실행은 되지만 내부 연결이 끊긴” 게임을 만들었고, 그 결함은 소스를 읽어야만 발견됐습니다. 스스로 채점하면 후하게 준다는 모델의 약점을 별도 검증자로 보완한 것입니다.

Anthropic Managed Agents — 자격증명 격리

오래 도는 에이전트는 API 키·클라우드 접근·셸 실행 권한을 갖기에 공격면이 큽니다. 그래서 “뇌”(모델+하네스)를 “손”(샌드박스·도구)과 “세션”에서 분리합니다. 자격증명은 볼트에 두고, 모델이 만든 코드가 도는 샌드박스에서는 토큰에 닿을 수 없게 합니다. git push/pull은 토큰을 에이전트가 직접 만지지 않고도 동작하도록 배선합니다.

Chapter 08

LLMOps / MLOps 엔지니어의 시선으로

이 그림이 낯익지 않으신가요? 2015년 Sculley 등의 고전적 도식 — “ML 코드는 프로덕션 ML 시스템의 아주 작은 검은 상자일 뿐, 정작 엔지니어링의 대부분은 데이터 수집·피처 추출·서빙·모니터링 같은 주변 상자들에 있다”. 에이전트 시스템은 같은 교훈을 더 빠르게 반복하고 있습니다.

■ 모델(LLM) 은 프로덕션 에이전트 시스템의 중심이지만, 그 비중은 작습니다. ■ 하네스 — 상태 관리, 거버넌스, 비용 통제, 관측가능성, CI/CD — 가 시스템을 실제로 작동하게 만드는 대부분의 엔지니어링입니다.

유출로 드러난 한 코딩 에이전트의 코드베이스가 약 50만 줄 규모였는데, 모델과의 상호작용 자체는 그중 작은 일부였고 나머지가 전부 하네스였다는 사실은 이 점을 분명히 보여줍니다.

Chapter 09

자주 빠지는 함정과 원칙

함정 1 · 하네스는 낡는다

하네스의 모든 조각은 “모델이 못 하는 것”에 대한 가정입니다. 모델이 좋아지면 그 가정은 불필요해집니다. 한 사례에서 이전 모델은 컨텍스트 한계가 다가오면 작업을 성급히 끝내는 버릇이 있어 “컨텍스트 리셋”을 넣었는데, 더 나은 모델에서는 그 버릇 자체가 사라졌습니다. 새 모델이 나올 때마다 하네스 조각을 하나씩 꺼보며 “아직 필요한가”를 점검하세요.

함정 2 · 과잉 설계

처음부터 6개 구성요소를 다 만들면 무겁고 느리고 비쌉니다. 원칙은 “가장 단순한 해법에서 시작해, 필요할 때만 복잡도를 늘린다”. 첫 결과가 좋아도 곧바로 단순화를 시도해 load-bearing(실제로 성능을 떠받치는) 조각만 남기세요.

함정 3 · 자기 채점 신뢰

모델에게 “다 됐어?”라고 물으면 실제보다 후하게 답합니다. 30%만 끝났는데 자신만만하게 “완료”라고 하죠. 빌드를 본 적 없는 새 컨텍스트의 별도 평가자가 증거를 열어 판정하게 하세요(default-FAIL).

함정 4 · 비용·보안 방치

예산·서킷 브레이커가 없으면 오래 도는 에이전트가 하루 만에 한 주치 API 예산을 태울 수 있습니다. 셸·키·클라우드 접근을 가진 에이전트는 공격면도 큽니다. 비용 상한과 자격증명 격리는 “나중”이 아니라 명시적 설계 단계입니다.

한 가지 더 에이전트가 다루기 어려운 신생 프레임워크 위에 하네스를 세우면, 하네스가 스스로와 싸웁니다. 예제가 풍부하고 문서가 안정적이며 API가 예측 가능한 기술을 선호하세요. 신뢰는 종종 해법 공간을 넓히는 게 아니라 좁힐 때 생깁니다.
Chapter 10

이해도 점검 퀴즈

정답을 고르면 즉시 해설이 나타납니다.

Q1“에이전트 = 모델 + 하네스”에서 하네스에 해당하지 않는 것은?
하네스는 모델 자신을 제외한 모든 것입니다. 모델 가중치는 “모델” 쪽이며, 하네스는 그 바깥의 코드·설정·실행 로직을 가리킵니다.
Q2프롬프트·컨텍스트·하네스 엔지니어링의 관계로 가장 정확한 것은?
세 층위는 대체가 아니라 누적입니다. 프롬프트(한 번의 대화) ⊂ 컨텍스트(무엇을 보여줄지) ⊂ 하네스(환경 전체). 컨텍스트 엔지니어링은 하네스 안에 들어갑니다.
Q3하네스 엔지니어링의 핵심 철학을 한 문장으로 옮기면?
Hashimoto가 이름 붙인 그 습관입니다 — 실수를 “시스템 문제”로 보고 환경에 영구 수정을 새겨, 같은 실수가 구조적으로 불가능해지게 만듭니다.
Q4다음 중 하네스가 꼭 필요하지 않은 상황은?
하네스는 다단계·장시간·자율 작업에서 값을 합니다. 단발성 질의응답에는 오히려 과한 장치입니다.
Q5새 모델이 출시됐을 때 하네스에 대해 권장되는 실천은?
하네스는 “모델이 못 하는 것”에 대한 가정의 집합이라, 모델이 좋아지면 일부 가정이 낡습니다. 조각을 제거해 보며 load-bearing 여부를 확인하는 게 정석입니다.
Chapter 11

한 장 요약 — 치트시트

공식

에이전트 = 모델 + 하네스. 하네스 = 모델을 뺀 모든 것.

세 층위

프롬프트(무엇을 묻나) ⊂ 컨텍스트(무엇을 보여주나) ⊂ 하네스(어떻게 돌아가나).

철학

실수 → 프롬프트 재시도가 아니라, 환경에 영구 수정을 새겨 재발을 구조적으로 차단.

6대 구성

지침파일 · 도구 · 상태/메모리 · 가드레일 · 검증게이트 · 오케스트레이션.

상태는 밖에

모델은 매번 백지. 파일시스템(progress·feature-list·git)이 외부 기억.

완료 판정

모델 자기 채점 금지. 새 컨텍스트의 별도 평가자가 증거로 판정(default-FAIL).

단순함 우선

최소에서 시작, 필요할 때만 복잡도 추가. 모델이 좋아지면 조각을 덜어낸다.

비용·보안

예산·서킷브레이커, 뇌/손 분리와 자격증명 격리는 처음부터 설계.

한 걸음 더 가려면 ① 작은 프로젝트에 AGENTS.md를 만들고 빌드·규칙을 적는다 → ② 에이전트를 돌리며 반복 실패를 규칙 한 줄로 박제한다 → ③ “완료 전 자동 실행되는 검증 훅”을 붙여 자기 채점을 없앤다. 이 세 가지만으로도 실전 하네스의 8할입니다.
