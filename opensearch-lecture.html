01

OpenSearch란 무엇인가

// 검색 + 분석 + 관측성을 한 번에

OpenSearch는 대량의 데이터를 저장하고, 거의 실시간으로 검색하고, 분석하는 오픈소스 분산 검색·분석 엔진입니다. Apache 2.0 라이선스로 누구나 자유롭게 쓸 수 있습니다.

이름만 보면 "구글 같은 검색창"을 떠올리기 쉽지만, 실제 쓰임새는 훨씬 넓습니다. OpenSearch는 크게 두 개의 소프트웨어로 이뤄진 한 가족입니다.

  • OpenSearch (엔진) — 데이터를 색인하고 검색·집계하는 본체. Java로 작성됐고 내부적으로 Apache Lucene이라는 검색 라이브러리를 사용합니다.
  • OpenSearch Dashboards — 저장된 데이터를 차트·지도·표로 시각화하고 탐색하는 웹 UI. (Elasticsearch 진영의 Kibana에 해당합니다.)

일반적인 데이터베이스(예: PostgreSQL)가 "정확한 키로 행을 찾는" 데 강하다면, OpenSearch는 "수많은 텍스트·로그·벡터 중에서 관련 있는 것을 빠르게 찾고 집계하는" 데 특화돼 있습니다. 그래서 검색창뿐 아니라 로그 분석, 보안 모니터링, AI의 의미 검색까지 폭넓게 쓰입니다.

📖
한 줄 비유

관계형 DB가 "주민번호로 한 사람을 찾는 주민센터"라면, OpenSearch는 "키워드만 던지면 관련 책을 모두 찾아주고, 분야별로 몇 권인지까지 세어주는 거대한 도서관 사서"입니다.

◆ 핵심 정리

OpenSearch = Lucene 기반의 분산 검색·분석 엔진(엔진) + 시각화 도구(Dashboards). 2025년 기준 소프트웨어 다운로드 10억 건을 넘긴, 검색 분야의 대표 오픈소스입니다.

02

탄생 배경 — '포크' 이야기

// 왜 Elasticsearch와 OpenSearch가 따로 존재할까

OpenSearch를 이해하려면 그 출생의 비밀, 즉 Elasticsearch와의 분리(fork)를 알아야 합니다. 이 이야기가 두 제품을 비교할 때 모든 차이의 뿌리가 되기 때문입니다.

2010 — Elasticsearch 등장

Elastic 사가 Lucene을 감싼 분산 검색 엔진 Elasticsearch를 Apache 2.0(완전 자유 오픈소스)으로 공개. 빠르게 업계 표준이 됩니다.

2021 — 라이선스 변경

Elastic이 Elasticsearch·Kibana의 라이선스를 Apache 2.0에서 SSPL + Elastic License 이중 라이선스로 전환. 핵심 의도는 "클라우드 사업자가 Elasticsearch를 그대로 매니지드 서비스로 팔지 못하게 막는 것"이었고, AWS가 주요 표적이었습니다.

2021 — AWS의 포크, OpenSearch 탄생

AWS가 마지막 Apache 2.0 버전인 Elasticsearch 7.10.2를 복제(fork)해 OpenSearch를, Kibana를 복제해 OpenSearch Dashboards를 만들고 Apache 2.0으로 계속 유지합니다. 첫해에만 1억 다운로드를 돌파했습니다.

2024 — 거버넌스 이관 & ES의 오픈소스 복귀

두 가지 큰 사건이 같은 해에 일어납니다. ① AWS가 OpenSearch 소유권을 Linux Foundation 산하 OpenSearch Software Foundation으로 넘겨 특정 회사에 종속되지 않는 중립 프로젝트로 만들었습니다. ② Elastic은 Elasticsearch에 OSI 승인 오픈소스 라이선스인 AGPLv3를 추가해 다시 "오픈소스"라는 표현을 되찾았습니다.

2025~2026 — 독자 노선

OpenSearch 3.0(2025년 4월, Lucene 10 기반)을 시작으로 벡터·하이브리드 검색과 'agentic AI' 기능을 강화하고, 2026년에는 장기 지원(LTS) 프로그램을 도입했습니다. 두 제품은 더 이상 같은 코드가 아니라 각자의 길을 걷고 있습니다.

▲ 알아두면 좋은 포인트

2024년 Elastic이 오픈소스로 "돌아왔지만", 이미 OpenSearch로 이전한 많은 조직은 되돌아가지 않았습니다. 한 번 깨진 신뢰와 마이그레이션 비용 때문입니다. 즉 두 제품의 공존은 당분간 계속됩니다.

03

핵심 개념 5가지

// 이 다섯 단어만 알면 절반은 끝

OpenSearch의 용어는 관계형 DB와 매핑해보면 훨씬 쉽습니다.

OpenSearch 용어비슷한 RDB 개념한 줄 설명
Document (문서)Row (행)저장되는 데이터 한 건. JSON 형태입니다.
Index (인덱스)Table (테이블)비슷한 문서들을 모아둔 논리적 묶음. products, logs-2026처럼 이름을 붙입니다.
Field (필드)Column (열)문서 안의 각 속성. title, price 등.
Mapping (매핑)Schema (스키마)각 필드의 타입(text, keyword, integer…)을 정의.
Shard / Replica파티션 / 복제본인덱스를 잘게 쪼갠 조각(shard)과 그 사본(replica). 분산·고가용성의 핵심.

특히 중요한 두 가지

① 샤드(Shard) — 큰 데이터를 나눠 담는 조각

인덱스가 너무 커서 한 서버에 다 안 들어가면? OpenSearch는 인덱스를 여러 개의 샤드로 쪼개 여러 노드에 분산 저장합니다. 검색 요청이 오면 각 샤드가 동시에 일을 처리하므로 빠릅니다. 이걸 Primary Shard(원본)라 부릅니다.

② 레플리카(Replica) — 안전과 속도를 위한 사본

Primary 샤드가 들어있는 노드가 죽으면 데이터가 사라질까요? 그래서 OpenSearch는 레플리카(복제본)를 다른 노드에 둡니다. 원본이 사라져도 복제본이 그 역할을 이어받고(고가용성), 평소엔 검색 부하를 나눠 받습니다(성능).

🍕
피자로 이해하는 샤드 & 레플리카

큰 피자(인덱스)를 8조각(샤드)으로 잘라 여러 접시(노드)에 나눠 담으면 여러 사람이 동시에 먹을 수 있습니다(병렬 검색). 그리고 각 조각을 한 판 더 구워 다른 접시에도 둔 것이 레플리카입니다 — 한 접시를 떨어뜨려도 피자는 무사합니다.

04

역색인을 직접 만들어보기

// 검색이 빠른 진짜 이유

OpenSearch가 수억 개 문서에서 단어를 1초 만에 찾는 비밀은 역색인(Inverted Index)입니다. 책 맨 뒤의 "찾아보기(색인)"와 똑같은 원리입니다.

일반적인 방식이라면 "검색어가 들어있는 문서"를 찾기 위해 모든 문서를 처음부터 끝까지 읽어야 합니다(느림). 역색인은 반대로 "단어 → 그 단어가 들어있는 문서 목록"을 미리 만들어 둡니다. 그래서 검색어를 던지면 해당 목록만 즉시 꺼내오면 됩니다.

아래에서 직접 단어를 검색해보세요. 왼쪽 문서가 어떻게 매칭되는지, 오른쪽에서 역색인 테이블이 어떻게 동작하는지 실시간으로 보입니다.

역색인 인터랙티브 데모 search the index →
빠른 검색:
📄 문서 (Documents)
🔁 역색인 (Inverted Index)
검색어를 입력하면 매칭 결과가 여기에 표시됩니다.
◆ 무슨 일이 일어났나

OpenSearch는 문서를 색인할 때 텍스트를 단어 단위로 잘게 쪼개고(이 과정을 분석/토크나이징이라 합니다), 각 단어가 어느 문서에 있는지 목록(posting list)을 만들어 둡니다. 검색 시에는 전체 문서를 훑지 않고 이 목록만 보면 되므로 데이터가 아무리 많아도 빠릅니다.

05

클러스터 아키텍처

// 노드들이 모여 하나처럼 동작하는 법

운영 환경의 OpenSearch는 한 대가 아니라 여러 대의 서버(노드, Node)가 모인 클러스터(Cluster)로 동작합니다. 각 노드는 역할을 나눠 맡습니다.

  • Cluster Manager 노드 (구 Master) — 클러스터 전체 상태를 관리·조율하는 지휘자. 어떤 샤드가 어느 노드에 있는지 등을 결정합니다.
  • Data 노드 — 실제 데이터(샤드)를 저장하고 검색·집계를 수행하는 일꾼.
  • Coordinating 노드 — 클라이언트 요청을 받아 적절한 데이터 노드로 분배하고 결과를 모아 돌려주는 안내데스크.
  • ML / Ingest 노드 — 머신러닝 작업이나 데이터 전처리를 전담하는 특화 노드(선택).

아래는 3개 노드로 이뤄진 클러스터가 인덱스 하나를 어떻게 분산 저장하는지 보여줍니다. P는 Primary 샤드, R은 그 Replica입니다. 같은 번호의 P와 R은 항상 서로 다른 노드에 놓여, 한 노드가 죽어도 데이터를 잃지 않습니다.

▲ 왜 이렇게 두는가

예를 들어 node-1이 통째로 다운돼 P0가 사라져도, node-2에 있던 R0가 즉시 Primary로 승격되어 서비스가 끊기지 않습니다. 이것이 고가용성(High Availability)의 작동 방식입니다.

06

기본 사용법 — REST API

// HTTP만 알면 누구나 쓸 수 있다

OpenSearch의 가장 큰 장점 중 하나는 모든 조작을 HTTP REST API로 한다는 점입니다. 별도 드라이버 없이 curl이나 어떤 언어의 HTTP 라이브러리로도 다룰 수 있습니다. 데이터를 넣고(색인), 찾고(검색), 지우는 흐름을 따라가 봅시다.

① 문서 색인 (데이터 넣기)

# products 인덱스에 문서 1건 저장 (PUT /인덱스/_doc/문서ID)
curl -XPUT "https://localhost:9200/products/_doc/1" \
  -H 'Content-Type: application/json' \
  -u admin:<password> --insecure \
  -d '{
    "name": "무선 기계식 키보드",
    "category": "주변기기",
    "price": 89000,
    "in_stock": true
  }'
from opensearchpy import OpenSearch

client = OpenSearch(
    hosts=[{"host": "localhost", "port": 9200}],
    http_auth=("admin", "<password>"),
    use_ssl=True, verify_certs=False,
)

client.index(
    index="products",
    id=1,
    body={
        "name": "무선 기계식 키보드",
        "category": "주변기기",
        "price": 89000,
        "in_stock": True,
    },
)

② 문서 검색 (데이터 찾기)

# name 필드에서 '키보드'를 검색
curl -XGET "https://localhost:9200/products/_search" \
  -H 'Content-Type: application/json' \
  -u admin:<password> --insecure \
  -d '{
    "query": {
      "match": { "name": "키보드" }
    }
  }'
resp = client.search(
    index="products",
    body={
        "query": {
            "match": {"name": "키보드"}
        }
    },
)

for hit in resp["hits"]["hits"]:
    print(hit["_score"], hit["_source"]["name"])

검색 결과에는 _score라는 값이 함께 옵니다. 이는 "이 문서가 검색어와 얼마나 관련 있는가"를 나타내는 관련도 점수로, OpenSearch는 기본적으로 점수가 높은 순서로 결과를 정렬합니다. 단순히 일치/불일치가 아니라 "얼마나 잘 맞는가"를 다룬다는 점이 검색 엔진의 특징입니다.

▲ 자주 쓰는 운영 API

GET /_cat/health?v — 클러스터 상태(green/yellow/red) 확인 · GET /_cat/indices?v — 인덱스 목록과 크기 · GET /products/_mapping — 매핑 확인. _cat API는 사람이 읽기 좋은 표로 응답을 주어 디버깅에 유용합니다.

07

쿼리 DSL 맛보기

// JSON으로 표현하는 검색 언어

OpenSearch의 검색은 Query DSL이라는 JSON 기반 질의 언어로 표현합니다. 처음엔 종류가 많아 보이지만, 입문자는 아래 네 가지만 알아도 대부분의 검색을 만들 수 있습니다. 탭을 눌러 차이를 비교해보세요.

match — 가장 흔한 전문 검색. 검색어를 분석(토크나이징)해서 관련 있는 문서를 점수와 함께 찾습니다. "키보드"를 검색하면 "무선 기계식 키보드"도 매칭됩니다.
{
  "query": {
    "match": { "name": "기계식 키보드" }
  }
}
term — 정확히 일치하는 값을 찾을 때. 분석을 거치지 않으므로 keyword 타입 필드(카테고리, 태그, ID, 상태값)에 적합합니다. 대소문자·띄어쓰기까지 정확히 같아야 매칭됩니다.
{
  "query": {
    "term": { "category": "주변기기" }
  }
}
bool — 여러 조건을 조합하는 만능 쿼리. must(반드시 충족, 점수 영향), filter(조건 필터, 점수 영향 없음·캐시됨), should(있으면 가점), must_not(제외)을 섞습니다. 실무 검색은 대부분 bool로 만듭니다.
{
  "query": {
    "bool": {
      "must":   [{ "match": { "name": "키보드" }}],
      "filter": [{ "term":  { "in_stock": true }}],
      "must_not":[{ "term":  { "category": "중고" }}]
    }
  }
}
range — 숫자·날짜의 범위 검색. gte(이상), lte(이하) 등을 씁니다. "5만~10만원 사이 상품" 또는 "최근 7일 로그"처럼 범위 필터에 필수입니다.
{
  "query": {
    "range": {
      "price": { "gte": 50000, "lte": 100000 }
    }
  }
}
◆ match vs term — 입문자가 가장 많이 헷갈리는 지점

match는 "비슷하면 OK"(분석 O, 전문 검색), term은 "정확히 같아야 함"(분석 X, 정확 일치). 카테고리·상태·태그 같은 값엔 term을, 본문·제목 같은 자연어엔 match를 쓴다고 기억하면 됩니다.

08

OpenSearch Dashboards

// 코드 없이 데이터를 보는 눈

모든 사람이 JSON 쿼리를 쓰진 않습니다. OpenSearch Dashboards는 브라우저에서 데이터를 탐색·시각화하는 웹 도구로, 보통 5601 포트로 접속합니다. 주요 기능은 다음과 같습니다.

  • Discover — 색인된 원본 문서를 검색하고 필터링하며 한 건씩 들여다보기. 로그 탐색의 출발점입니다.
  • Visualize / Dashboard — 막대·선·파이 차트, 지도, 표를 만들고 여러 개를 한 대시보드로 모아 모니터링.
  • Dev Tools — 위에서 배운 REST API를 자동완성과 함께 바로 실행해보는 콘솔. 학습·디버깅에 최고입니다.
  • Alerting / Anomaly Detection — 특정 조건(에러 급증 등)에 알림을 보내거나, 머신러닝으로 이상 징후를 자동 탐지.
🔭
엔진과 대시보드의 관계

OpenSearch(엔진)가 데이터를 담는 데이터베이스라면, Dashboards는 그 위에 얹는 BI·관제 화면입니다. 엔진만 있어도 API로 모든 게 가능하지만, 대시보드가 있으면 비개발자도 데이터를 다룰 수 있습니다.

09

4가지 핵심 응용 분야

// 현장에서 OpenSearch가 일하는 곳

① 로그 분석 & 관측성 (Observability)

가장 대표적인 용도입니다. 수십~수백 대의 서버·컨테이너가 쏟아내는 로그를 한곳에 모아 검색·집계합니다. Data Prepper나 Fluent Bit 같은 수집기가 로그를 OpenSearch로 보내고, 엔지니어는 "지난 1시간 동안 5xx 에러가 난 서비스"를 즉시 찾습니다. 이른바 OpenSearch가 ELK/EFK 스택의 핵심으로 쓰이는 영역입니다.

② 애플리케이션 검색 (Full-text Search)

쇼핑몰의 상품 검색, 사내 위키·문서 검색, 콘텐츠 플랫폼의 검색창. 오타 보정, 자동완성, 한국어 형태소 분석(Nori 플러그인), 관련도 기반 랭킹 등이 모두 OpenSearch의 영역입니다.

③ 벡터 검색 & RAG (AI 시대의 핵심)

이 부분이 ML 엔지니어에게 가장 중요합니다. OpenSearch는 텍스트·이미지를 임베딩 벡터로 저장하고, 의미가 가까운 것을 찾는 k-NN(최근접 이웃) 벡터 검색을 지원합니다. FAISS, NMSLIB, Lucene 엔진을 백엔드로 쓸 수 있고 최대 16,000 차원까지 지원합니다.

덕분에 OpenSearch는 LLM 애플리케이션의 RAG(검색 증강 생성)에서 벡터 데이터베이스 역할을 합니다. 키워드 검색(BM25)과 벡터 검색을 합친 하이브리드 검색까지 한 엔진에서 제공해, "검색 + 의미 이해"를 모두 처리합니다.

◆ ML 엔지니어를 위한 포인트

별도의 전용 벡터 DB(Milvus, Pinecone 등)를 도입하기 전에, 이미 로그·검색용으로 OpenSearch를 쓰고 있다면 같은 클러스터에서 벡터 검색까지 처리할 수 있습니다. 운영 스택을 단순화하는 강력한 선택지입니다.

④ 보안 분석 (SIEM)

보안 로그를 모아 위협을 탐지하는 SIEM 용도. OpenSearch는 Security Analytics 기능과 이상 탐지(Anomaly Detection)를 오픈소스로 기본 제공해, 비용 부담 없이 보안 모니터링 기반을 구축할 수 있습니다.

10

OpenSearch vs Elasticsearch

// 같은 뿌리, 다른 길

두 제품은 2021년까지 같은 코드였기에 기본 개념과 API가 매우 비슷합니다. 하지만 그 이후로 라이선스·기능·거버넌스에서 뚜렷이 갈라졌습니다. 핵심 차이를 정리하면 다음과 같습니다.

항목OpenSearchElasticsearch
라이선스 Apache 2.0 (완전 자유) ELv2 · SSPL · AGPLv3(2024 추가) 선택형
거버넌스 Linux Foundation 산하 중립 재단
(400+ 조직, 3,300+ 기여자)		 Elastic 사가 주도
보안(RBAC·필드 보안·감사로그) 무료 기본 포함 유료 티어
이상 탐지 / ML 일부 오픈소스 포함 유료 기능 다수
벡터 검색 FAISS·NMSLIB·Lucene, 최대 16,000차원 k-NN, 최대 4,096차원
매니지드 서비스 Amazon OpenSearch Service, Aiven 등 여러 사업자 주로 Elastic Cloud(구독)
시각화 도구 OpenSearch Dashboards Kibana
독점 고급 기능 상대적으로 적음 (대부분 오픈) Elastic Security, APM, ES|QL 등 풍부

성능은 어느 쪽이 빠를까?

민감한 주제입니다. Elastic은 자사 벤치마크에서 Elasticsearch가 OpenSearch보다 상당히 빠르다고 주장합니다(워크로드에 따라 큰 차이를 제시). 반면 독립적인 테스트에서는 워크로드에 따라 OpenSearch가 더 나은 경우도 보고됩니다.

결론은 명확합니다. 성능은 데이터·쿼리 패턴·설정에 크게 좌우되므로, 벤더 벤치마크를 그대로 믿기보다 본인의 실제 워크로드로 직접 측정하는 것이 정답입니다. 단순 로그 분석이나 중소 규모 검색에서는 두 제품 모두 충분히 빠르고, 차이는 극한의 처리량·복잡한 분석에서 드러나는 경향이 있습니다.

▲ 호환성 주의

OpenSearch는 Elasticsearch 7.10 API와 상당 부분 호환됩니다. 다만 최신 Elasticsearch 클라이언트 라이브러리는 OpenSearch 접속을 막아둔 경우가 있어, OpenSearch에는 전용 클라이언트(opensearch-py 등)를 쓰는 것이 안전합니다.

설치 & 배포

// 노트북에서 Kubernetes까지

학습용으로는 Docker 한 줄이면 충분합니다. 운영 환경에서는 Kubernetes Operator로 클러스터를 선언적으로 관리하는 것이 일반적입니다.

# 단일 노드로 빠르게 띄우기 (학습/실습 전용)
docker run -d \
  -p 9200:9200 -p 9600:9600 \
  -e "discovery.type=single-node" \
  -e "OPENSEARCH_INITIAL_ADMIN_PASSWORD=<StrongP@ss123>" \
  --name opensearch \
  opensearchproject/opensearch:latest

# 동작 확인
curl https://localhost:9200 -u admin:<StrongP@ss123> --insecure
# docker-compose.yml — 엔진 + 대시보드 함께
services:
  opensearch-node1:
    image: opensearchproject/opensearch:latest
    environment:
      - discovery.type=single-node
      - OPENSEARCH_INITIAL_ADMIN_PASSWORD=<StrongP@ss123>
    ports: ["9200:9200"]
  dashboards:
    image: opensearchproject/opensearch-dashboards:latest
    ports: ["5601:5601"]
    environment:
      - OPENSEARCH_HOSTS=["https://opensearch-node1:9200"]
# OpenSearch Kubernetes Operator로 선언적 클러스터 정의
apiVersion: opensearch.opster.io/v1
kind: OpenSearchCluster
metadata:
  name: my-cluster
spec:
  general:
    version: "3.0.0"
  nodePools:
    - component: managers
      replicas: 3
      roles: ["cluster_manager", "data"]
      diskSize: "30Gi"

# Operator 3.0(2026)은 quorum-safe 롤링 재시작,
# 멀티 네임스페이스, TLS 핫리로드 등을 지원합니다.
▲ 운영 전 체크리스트

① JVM 힙은 물리 메모리의 50% 이하 & 32GB 미만으로 · ② Cluster Manager 노드는 홀수 개(보통 3개)로 스플릿 브레인 방지 · ③ vm.max_map_count 등 OS 커널 파라미터 조정 · ④ 운영에선 단일 노드(single-node) 금지.

무엇을 선택할까?

// 상황별 의사결정 가이드

정답은 "조직의 우선순위"에 달려 있습니다. 아래 기준으로 판단해보세요.

OpenSearch가 유리할 때 FREE & OPEN

  • 벤더 종속(lock-in) 없이 완전 오픈소스를 원할 때
  • 보안(RBAC)·이상 탐지를 추가 비용 없이 쓰고 싶을 때
  • 이미 AWS 환경이고 매니지드 서비스를 원할 때
  • 로그 분석 + 검색 + 벡터를 한 스택으로 묶고 싶을 때
  • 여러 매니지드 사업자 중 골라 비용을 최적화하고 싶을 때

Elasticsearch가 유리할 때 ENTERPRISE

  • 이미 Elastic Stack에 깊이 투자해 있을 때
  • Elastic Security·APM·ES|QL 같은 고급 독점 기능이 필요할 때
  • 극한의 성능·복잡한 분석에서 최신 최적화를 우선할 때
  • 벤더의 공식 지원·SLA를 단일 창구로 받고 싶을 때
  • 최신 AGPL 라이선스 조건을 수용 가능할 때
◆ 한 문장 결론

"열려 있고, 비용 효율적이며, 커뮤니티 주도"를 원하면 OpenSearch. "풍부한 엔터프라이즈 기능과 단일 벤더 지원"을 원하면 Elasticsearch. 새로 시작하는 클라우드·ML 프로젝트라면, 무료 보안 기능과 벡터 검색을 갖춘 OpenSearch가 부담 없는 출발점입니다.

이해도 점검 퀴즈

// 5문항으로 확인하기

Q1.OpenSearch가 수억 건에서 단어를 빠르게 찾을 수 있는 핵심 자료구조는?
정답: C. 역색인은 "단어 → 그 단어가 포함된 문서 목록"을 미리 만들어 두어, 전체 문서를 훑지 않고도 즉시 검색 결과를 찾습니다.
Q2.OpenSearch가 Elasticsearch에서 분리(fork)된 직접적인 계기는?
정답: B. Elastic이 2021년 Apache 2.0에서 SSPL+Elastic License로 바꾸자, AWS가 마지막 오픈소스 버전(7.10.2)을 포크해 OpenSearch를 만들었습니다.
Q3.카테고리("주변기기")처럼 정확히 일치하는 값을 찾을 때 가장 적합한 쿼리는?
정답: D. term은 분석을 거치지 않고 정확히 일치하는 값을 찾습니다. 자연어 본문에는 match, 카테고리·태그·상태값에는 term을 씁니다.
Q4.다음 중 OpenSearch가 무료 기본 제공하지만 Elasticsearch에서는 주로 유료 티어인 기능은?
정답: A. OpenSearch는 RBAC·필드 수준 보안·이상 탐지 등을 오픈소스로 포함합니다. Elasticsearch에서는 이런 고급 보안·ML 기능 상당수가 유료입니다.
Q5.같은 인덱스의 Primary 샤드 P0와 그 Replica R0를 배치하는 올바른 방법은?
정답: C. P0와 R0를 다른 노드에 두어야, 한 노드가 죽어도 복제본이 살아남아 데이터 유실 없이 서비스가 이어집니다(고가용성).

수고하셨습니다. 다음 단계로는 Docker로 단일 노드를 직접 띄워 Dev Tools에서 위 쿼리들을 실행해보고, 작은 데이터셋으로 벡터 검색(k-NN)까지 시도해보는 것을 추천합니다.
