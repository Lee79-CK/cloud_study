H/W부터 Container까지 계층으로 꿰뚫기

"Pod IP는 어디서 오는 걸까?" "다른 노드의 Pod과는 어떻게 통신하지?" — 초보 클라우드 엔지니어가 가장 헷갈리는 질문들을, 물리 하드웨어부터 컨테이너까지 7개 계층을 바닥부터 쌓아 올리며 하나씩 해결합니다.

난이도 · 입문~초급 소요 시간 · 약 60분 패킷 여행 시뮬레이터 포함 퀴즈 8문항
INTRO

왜 "계층"으로 이해해야 할까?

쿠버네티스 네트워크가 어렵게 느껴지는 진짜 이유는, 여러 계층의 개념이 한꺼번에 등장하기 때문입니다.

처음 쿠버네티스를 배우면 이런 상황을 겪게 됩니다. kubectl get pods -o wide를 쳤더니 Pod마다 IP가 있습니다. 그런데 노드에도 IP가 있고, Service에도 IP가 있고, 심지어 VM에도, 물리 서버에도 IP가 있습니다. "도대체 IP가 몇 개인 거야?" 라는 혼란이 시작되죠.

정답은 간단합니다. 각 IP는 서로 다른 계층에 속해 있고, 각 계층은 자기 아래 계층 위에 쌓여 있습니다. 아래 계층을 이해하면 위 계층은 "아, 이건 아래 걸 한 번 더 추상화한 거구나"로 자연스럽게 이해됩니다. 반대로 계층 구분 없이 외우려고 하면 영원히 헷갈립니다.

🏢 비유: 대형 오피스 빌딩

이 강의 전체에서 쓸 비유 하나를 소개합니다. 쿠버네티스 클러스터를 대형 오피스 빌딩 단지라고 생각해 보세요.

  • H/W = 땅과 건물 뼈대, 단지 사이의 실제 도로
  • OS = 건물 내부의 복도·엘리베이터·우편 규칙
  • VM = 한 건물을 칸막이로 나눈 독립 임대 공간
  • Node = 쿠버네티스에 등록된 하나의 "지점 사무실"
  • Pod = 같은 방을 쓰는 팀 (방 번호 = Pod IP)
  • Container = 그 방 안에서 일하는 개별 직원
  • Cluster = 모든 지점을 묶는 "회사 전체 조직도 + 사내 전화망"

이 강의의 학습 순서

우리는 바닥(H/W)에서 시작해 위(Container)로 쌓아 올린 뒤, 마지막에 이 모든 걸 하나로 묶는 Cluster 네트워킹을 다룹니다. 실제 패킷이 이동하는 물리적 순서와 같기 때문에, 이 순서로 배우면 트러블슈팅할 때도 "어느 계층에서 끊겼는지"를 논리적으로 추적할 수 있습니다.

🎯 학습 목표
  • 7개 계층(H/W → OS → VM → Node → Pod → Container → Cluster)의 역할과 네트워크 책임을 구분한다
  • Pod IP가 어떻게 만들어지고, 노드를 넘어 어떻게 전달되는지 설명한다
  • 같은 Pod 내 / 같은 Node 내 / 다른 Node 간 통신 경로의 차이를 그림으로 그릴 수 있다
  • 장애 발생 시 어느 계층부터 확인해야 하는지 판단한다
OVERVIEW

7계층 스택 한눈에 보기

아래 스택의 각 계층을 클릭해 보세요. 아래에서 위로 갈수록 "물리적 실체"에서 "논리적 추상화"로 이동합니다.

💡 핵심 관점: "각 계층은 아래 계층의 자원을 나눠 쓰는 방법"

OS는 H/W의 NIC을 프로세스들이 나눠 쓰게 하고, 하이퍼바이저는 물리 NIC을 여러 VM이 나눠 쓰게 하고(vNIC), 쿠버네티스는 노드의 네트워크를 여러 Pod이 나눠 쓰게 합니다(veth + 네임스페이스). "나눠 쓰기(공유) + 격리(isolation)"가 모든 계층에서 반복되는 단 하나의 패턴입니다.

LAYER 1 · H/W

물리 네트워크 — 모든 것의 바닥

아무리 추상화해도 결국 모든 패킷은 구리선·광케이블·전파 위를 달리는 전기 신호입니다.

클라우드 시대에도 데이터센터 어딘가에는 실제 서버가 랙에 꽂혀 있습니다. 이 계층의 등장인물은 단순합니다.

NIC
네트워크 인터페이스 카드

서버가 네트워크에 연결되는 물리 포트. 전 세계에서 유일한 MAC 주소가 새겨져 있습니다. 요즘 서버는 10/25/100GbE NIC을 여러 장 꽂습니다.

SWITCH
스위치 (L2)

같은 네트워크 안에서 MAC 주소를 보고 프레임을 전달합니다. 데이터센터에서는 랙 상단의 ToR(Top of Rack) 스위치가 서버들을 1차로 묶습니다.

ROUTER
라우터 (L3)

서로 다른 네트워크(서브넷) 사이에서 IP 주소를 보고 패킷의 다음 목적지를 결정합니다. 인터넷은 라우터들의 거대한 연결망입니다.

전형적인 데이터센터 Leaf-Spine 구조의 단순화 모습

이 계층에서 기억할 것

  • MAC 주소는 L2(같은 스위치 영역), IP 주소는 L3(네트워크 간) 통신의 기준입니다. 이 구분은 위의 모든 가상 계층에서도 그대로 반복됩니다.
  • 물리 대역폭이 모든 상위 계층 성능의 상한선입니다. Pod 간 통신이 느리다면 최종적으로는 이 계층의 케이블·스위치 용량까지 내려와서 봐야 할 수 있습니다.
  • 퍼블릭 클라우드에서는 이 계층이 완전히 숨겨져 있습니다. AWS/GCP/Azure를 쓰면 우리는 이 계층을 만질 수 없고, 대신 VPC라는 "가상의 L3 네트워크"를 받습니다.
🏢 빌딩 비유로 보면

H/W 계층은 땅, 건물 뼈대, 단지 안팎의 실제 도로입니다. 위에서 아무리 조직을 바꿔도(팀 이동, 부서 개편) 도로 자체가 막히면 어떤 우편물도 배달될 수 없죠. 반대로 평소에 우리는 도로의 존재를 의식하지 않고 삽니다 — 클라우드에서 H/W를 의식하지 않는 것과 같습니다.

LAYER 2 · OS

리눅스 네트워킹 — 컨테이너 네트워크의 재료 창고

쿠버네티스 네트워크의 마법처럼 보이는 것들은 전부 리눅스 커널이 이미 갖고 있던 기능의 조합입니다.

이 계층이 이 강의에서 가장 중요합니다. Docker도, 쿠버네티스도 네트워크를 "새로 발명"하지 않았습니다. 리눅스 커널이 제공하는 4가지 재료를 조립했을 뿐입니다. 이 4가지만 이해하면 Pod 네트워크는 저절로 이해됩니다.

재료 ① network namespace
네트워크 네임스페이스

프로세스에게 "자기만의 독립된 네트워크 세상"을 줍니다. 자기만의 인터페이스 목록, 라우팅 테이블, iptables 규칙을 따로 가집니다. 나중에 배울 "Pod마다 IP가 따로 있는 이유"의 정체가 바로 이것입니다.

재료 ② veth pair
가상 이더넷 케이블

양 끝이 쌍으로 만들어지는 가상 랜선입니다. 한쪽 끝을 네임스페이스 안에, 다른 끝을 호스트에 두면 격리된 세상과 바깥 세상을 잇는 통로가 됩니다. 한쪽으로 넣으면 반드시 반대쪽으로 나옵니다.

재료 ③ bridge
가상 스위치

소프트웨어로 구현된 L2 스위치입니다. 여러 veth의 호스트 쪽 끝을 브리지에 꽂으면, 같은 호스트 안의 네임스페이스들이 서로 통신할 수 있게 됩니다. Docker의 docker0, K8s CNI의 cni0가 이것입니다.

재료 ④ iptables / IPVS / eBPF
패킷 규칙 엔진

커널을 지나는 패킷을 조건에 따라 바꾸고(NAT), 막고(필터), 다른 곳으로 보내는(DNAT) 규칙 시스템입니다. 나중에 배울 Service의 "가상 IP → 실제 Pod IP 변환"이 여기서 일어납니다.

네임스페이스 2개가 veth로 브리지에 연결된 모습 — 이것이 곧 "한 노드 안의 Pod 2개"의 실체입니다

직접 만들어 보기 — 쿠버네티스 없이 "미니 Pod 네트워크" 만들기

아래 명령어 몇 줄이면 쿠버네티스가 Pod 네트워크를 만들 때 하는 일을 그대로 재현할 수 있습니다. 이 실습이 이 강의 전체에서 가장 가치 있는 15분입니다.

# "미니 Pod" 역할을 할 네트워크 네임스페이스 2개 생성
$ sudo ip netns add pod-a
$ sudo ip netns add pod-b

# 확인 — 각자 완전히 격리된 네트워크 세상을 가짐
$ sudo ip netns list
pod-a
pod-b

# pod-a 안에서 인터페이스를 보면 lo(루프백)뿐, 바깥 세상과 단절 상태
$ sudo ip netns exec pod-a ip link
1: lo: <LOOPBACK> mtu 65536 ...
# 가상 스위치(브리지) 생성 — cni0와 같은 역할
$ sudo ip link add br0 type bridge
$ sudo ip link set br0 up
$ sudo ip addr add 10.244.1.1/24 dev br0   # 브리지 = 게이트웨이

# veth 케이블 2쌍 생성 → 한쪽 끝은 네임스페이스로, 반대쪽은 브리지로
$ sudo ip link add veth-a type veth peer name veth-a-br
$ sudo ip link set veth-a netns pod-a
$ sudo ip link set veth-a-br master br0 up

$ sudo ip link add veth-b type veth peer name veth-b-br
$ sudo ip link set veth-b netns pod-b
$ sudo ip link set veth-b-br master br0 up

# 각 "미니 Pod"에 IP 부여 — 이것이 곧 Pod IP의 정체!
$ sudo ip netns exec pod-a ip addr add 10.244.1.2/24 dev veth-a
$ sudo ip netns exec pod-a ip link set veth-a up
$ sudo ip netns exec pod-b ip addr add 10.244.1.3/24 dev veth-b
$ sudo ip netns exec pod-b ip link set veth-b up
# pod-a에서 pod-b로 ping — 브리지를 통해 통신 성공!
$ sudo ip netns exec pod-a ping -c 2 10.244.1.3
64 bytes from 10.244.1.3: icmp_seq=1 ttl=64 time=0.058 ms
64 bytes from 10.244.1.3: icmp_seq=2 ttl=64 time=0.041 ms

# 축하합니다. 방금 CNI 플러그인이 하는 일의 핵심을 손으로 재현했습니다.
# K8s에서 Pod가 생성될 때마다 CNI가 정확히 이 작업을 자동으로 수행합니다:
#   netns 생성 → veth 생성 → 브리지 연결 → IP 할당 → 라우팅 설정

# 실습 후 정리
$ sudo ip netns del pod-a && sudo ip netns del pod-b
$ sudo ip link del br0
🏢 빌딩 비유로 보면

OS 계층은 건물 내부의 복도·엘리베이터·우편 분류 규칙입니다. 네임스페이스는 "독립된 사무실 공간", veth는 "사무실 문과 복도를 잇는 통로", 브리지는 "층별 우편 분류대", iptables는 "우편물 배송 규정집"입니다.

⚠️ 초보자가 자주 헷갈리는 것

"컨테이너 네트워크는 특별한 기술이다" → 아닙니다. 위에서 본 것처럼 1990~2000년대부터 리눅스 커널에 있던 기능(netns는 2008년, iptables는 2001년)의 조합입니다. Docker(2013)와 K8s(2014)는 이를 자동화했을 뿐입니다. 그래서 트러블슈팅의 최종 병기도 언제나 ip, iptables, tcpdump 같은 리눅스 표준 도구입니다.

LAYER 3 · VM

가상머신 — 하드웨어를 나눠 쓰는 기술

클라우드의 "서버"는 대부분 물리 서버가 아니라 VM입니다. VM은 네트워크마저 통째로 가상화합니다.

물리 서버 한 대는 너무 크고 비쌉니다(CPU 128코어, 메모리 1TB…). 그래서 하이퍼바이저(Hypervisor)라는 소프트웨어로 한 대를 여러 대의 "가상 컴퓨터"로 쪼갭니다. 이때 네트워크도 함께 쪼개져야 합니다.

vNIC
가상 NIC

VM 입장에서는 진짜 랜카드로 보이지만, 실제로는 하이퍼바이저가 만든 소프트웨어 장치입니다. 자기만의 MAC 주소와 IP를 가집니다.

vSwitch
가상 스위치

하이퍼바이저 안의 소프트웨어 스위치. 같은 호스트 위 VM들의 vNIC을 연결하고, 물리 NIC과의 다리 역할을 합니다. (OVS, Linux bridge, VMware vSwitch 등)

VPC / 오버레이
가상 네트워크

클라우드의 VPC는 수많은 물리 서버에 흩어진 VM들을 "마치 한 네트워크에 있는 것처럼" 묶는 대규모 오버레이 네트워크입니다.

🔍 어디서 많이 본 구조 아닌가요?

vNIC ↔ vSwitch ↔ 물리 NIC 구조는, 방금 OS 계층에서 본 veth ↔ bridge ↔ 물리 NIC과 정확히 같은 패턴입니다. 격리 단위가 "프로세스(netns)"냐 "운영체제 전체(VM)"냐만 다를 뿐입니다. 이 패턴 인식이 되면 쿠버네티스 네트워크의 절반은 이미 이해한 것입니다.

구분VM 가상화컨테이너 (netns)
격리 단위OS 전체 (커널 포함, 게스트 OS 각자 부팅)프로세스 그룹 (호스트 커널 공유)
가상 랜카드vNIC (하이퍼바이저 에뮬레이션)veth (커널 가상 장치)
가상 스위치vSwitch / OVSLinux bridge (cni0 등)
기동 시간수십 초 ~ 분수 밀리초 ~ 초
격리 강도강함 (커널 분리)상대적으로 약함 (커널 공유)

온프레미스 vs 퍼블릭 클라우드에서의 이 계층

  • 온프레미스(예: OpenStack, VMware, Proxmox): 우리가 하이퍼바이저와 vSwitch를 직접 운영합니다. K8s 노드가 VM이라면, Pod 트래픽은 veth → 브리지 → vNIC → vSwitch → 물리 NIC이라는 이중 가상화 경로를 지납니다.
  • 퍼블릭 클라우드(예: AWS EC2): 하이퍼바이저는 클라우드 사업자가 관리하고, 우리에게는 "VPC 안의 인스턴스"만 보입니다. EKS 노드의 eth0 = VM의 vNIC = ENI(Elastic Network Interface)입니다.
  • 베어메탈 K8s: VM 계층이 아예 없을 수도 있습니다! 이 경우 노드 = 물리 서버이며, 이 계층은 건너뜁니다. 계층은 "항상 7개 전부"가 아니라 환경에 따라 생략될 수 있습니다.
🏢 빌딩 비유로 보면

VM은 한 건물을 칸막이 공사로 나눈 독립 임대 공간입니다. 각 임대 공간은 자기 현관(vNIC)과 우편함(IP)을 갖고, 건물 관리사무소(하이퍼바이저)가 공용 우편 시스템(vSwitch)으로 각 공간에 우편을 분배합니다. 세입자(VM 사용자)는 옆 공간이 존재하는지조차 모릅니다.

LAYER 4 · NODE

Node — 쿠버네티스에 등록된 일꾼 머신

VM(또는 물리 서버)에 쿠버네티스 구성 요소가 설치되는 순간, 그 머신은 "노드"가 됩니다.

노드는 새로운 하드웨어가 아닙니다. 앞 계층의 VM/물리 서버 그 자체이고, 다만 쿠버네티스의 일원으로서 3가지 네트워크 관련 구성 요소를 추가로 실행할 뿐입니다.

kubelet
노드 관리인

API 서버의 지시를 받아 Pod를 만들고, Pod 생성 시 CNI 플러그인을 호출해 네트워크를 붙입니다. "Pod 하나 만들어졌으니 네트워크 연결해줘"라고 시키는 주체입니다.

CNI plugin
네트워크 배선공

Container Network Interface. OS 계층 실습에서 우리가 손으로 한 일(netns·veth·브리지·IP 할당·라우팅)을 자동으로 수행하는 표준 인터페이스입니다. Calico, Cilium, Flannel 등이 구현체입니다.

kube-proxy
서비스 규칙 관리자

Service(뒤에서 배움)의 가상 IP가 실제 Pod IP로 변환되도록 노드의 iptables/IPVS 규칙을 계속 갱신합니다. Cilium 같은 eBPF 기반 CNI는 이 역할을 대체하기도 합니다.

노드의 두 얼굴: Node IP와 Pod CIDR

노드는 네트워크 관점에서 두 가지를 가집니다. 이 둘의 구분이 매우 중요합니다.

항목설명예시

노드는 회사 조직도에 정식 등록된 "지점 사무실"입니다. 지점 주소(Node IP)는 시청 지도(VPC/사내망)에 등록된 실제 주소이고, 지점 내부의 방 번호 체계(Pod CIDR)는 본사가 지점마다 겹치지 않게 배정해 준 것입니다. 지점장(kubelet)이 새 팀을 들일 때마다 시설 담당(CNI)이 방을 만들고 문패(Pod IP)를 답니다.

LAYER 5 · POD

Pod — 쿠버네티스 네트워크에서 IP를 가지는 단위

쿠버네티스 네트워크의 가장 중요한 규칙: IP는 컨테이너가 아니라 Pod에 부여됩니다.

드디어 핵심에 도착했습니다. Pod의 네트워크 정체를 한 문장으로 말하면 이렇습니다.

📌 이 강의에서 단 하나만 기억한다면

Pod = 하나의 network namespace. OS 계층에서 만든 netns가 바로 Pod의 실체입니다. Pod 안의 모든 컨테이너는 이 네임스페이스를 공유하므로, 같은 IP를 쓰고 서로를 localhost로 부릅니다.

pause 컨테이너 — 네임스페이스를 붙잡고 있는 닻

Pod가 만들어질 때 쿠버네티스는 가장 먼저 pause(infra) 컨테이너라는 아주 작은 컨테이너를 띄웁니다. 이 컨테이너가 하는 일은 사실상 하나 — 네트워크 네임스페이스를 만들어 붙잡고(유지하고) 있는 것입니다. 이후 우리가 정의한 앱 컨테이너들은 pause의 네임스페이스에 "합류"합니다.

  • 덕분에 앱 컨테이너가 재시작되어도 Pod IP가 바뀌지 않습니다 (네임스페이스의 주인은 pause니까).
  • 반대로 Pod 자체가 삭제·재생성되면 netns도 새로 만들어지므로 Pod IP는 언제든 바뀔 수 있습니다. → 그래서 Service(8장)가 필요합니다.

IP-per-Pod 모델: 쿠버네티스 네트워크의 헌법

쿠버네티스는 어떤 CNI를 쓰든 반드시 지켜야 하는 네트워크 규칙을 정해 두었습니다. 단 3줄입니다.

RULE 1
모든 Pod은 고유 IP를 가진다

클러스터 전체에서 겹치지 않는 자기만의 IP. 포트 매핑 없이 자기 포트를 그대로 씁니다.

RULE 2
모든 Pod은 NAT 없이 서로 통신한다

어느 노드에 있든 Pod A가 Pod B의 IP로 직접 패킷을 보낼 수 있어야 합니다.

RULE 3
노드와 Pod도 NAT 없이 통신한다

노드의 에이전트(kubelet 등)가 Pod에 바로 접근할 수 있어야 합니다.

이 규칙 덕분에 개발자는 "내 앱이 어느 노드에 뜰까"를 신경 쓰지 않아도 됩니다. Docker 단독 사용 시절의 골칫거리였던 포트 충돌과 포트 매핑(-p 8080:80) 지옥이 사라진 이유가 바로 이 모델입니다.

🏢 빌딩 비유로 보면

Pod은 같은 방을 쓰는 하나의 팀입니다. 방 번호(Pod IP)는 하나이고, 방 안의 팀원(컨테이너)들은 옆자리 동료를 부를 때 주소가 필요 없죠(localhost). 우편(외부 트래픽)은 방 번호로 오고, 방 안에서 누가 받을지는 내선 번호(포트)로 구분합니다. pause 컨테이너는 방 열쇠를 쥐고 상주하는 총무 — 팀원이 바뀌어도 방 번호가 유지되는 이유입니다.

# Pod IP는 어떤 노드에 뜨느냐에 따라 그 노드의 Pod CIDR에서 할당됩니다
$ kubectl get pods -o wide
NAME     READY   STATUS    IP           NODE
web-1    1/1     Running   10.244.1.2   worker-1   ← worker-1의 CIDR
web-2    1/1     Running   10.244.2.5   worker-2   ← worker-2의 CIDR

# 다른 노드의 Pod에도 IP로 직접 통신 (NAT 없음 = IP-per-Pod 모델)
$ kubectl exec web-1 -- curl -s 10.244.2.5:8080/healthz
ok
# 컨테이너 2개짜리 Pod에서, 두 컨테이너의 IP를 각각 확인해 보면?
$ kubectl exec mypod -c app -- hostname -i
10.244.1.7
$ kubectl exec mypod -c sidecar -- hostname -i
10.244.1.7          ← 완전히 동일! 같은 netns를 공유하기 때문

# sidecar에서 app(같은 Pod, 포트 8080)에 localhost로 접근
$ kubectl exec mypod -c sidecar -- curl -s localhost:8080
hello from app container

# 노드에서 pause 컨테이너 확인 (containerd 기준)
worker-1$ sudo crictl ps -a | grep -i pause | head -2
3f2a...   registry.k8s.io/pause:3.10   Running   ...
⚠️ 초보자가 자주 헷갈리는 것

"Pod IP로 서비스하면 되지 않나요?" → 안 됩니다. Pod은 언제든 죽고 다시 태어나며(스케일링, 롤링 업데이트, 노드 장애), 그때마다 IP가 바뀝니다. Pod IP는 "현재 순간의 주소"일 뿐, 영구적인 엔드포인트가 아닙니다. 영구적인 주소는 8장의 Service가 제공합니다.

LAYER 6 · CONTAINER

Container — 네트워크를 "빌려 쓰는" 최소 실행 단위

놀랍게도, 쿠버네티스에서 컨테이너는 자기 네트워크를 갖지 않습니다.

Docker를 먼저 배운 분들이 가장 크게 헷갈리는 지점입니다. Docker 단독 사용 시에는 컨테이너마다 IP가 있었는데(docker0 브리지에서 할당), 쿠버네티스에서는 네트워크의 주인이 Pod로 승격되고 컨테이너는 세입자가 됩니다.

관점Docker 단독Kubernetes
IP 소유자컨테이너 1개당 IP 1개Pod이 IP 소유, 컨테이너는 공유
netns컨테이너마다 생성Pod(pause)이 생성, 앱 컨테이너는 합류
컨테이너 간 통신브리지 경유 (IP 필요)같은 Pod이면 localhost
외부 노출-p 8080:80 포트 매핑Pod IP + Service (포트 매핑 불필요)

같은 Pod 안에서 컨테이너들이 지켜야 할 것

  • 포트 충돌 주의: 네임스페이스(=포트 공간)를 공유하므로, 한 Pod 안의 두 컨테이너가 같은 포트를 리슨할 수 없습니다. app이 8080을 쓰면 sidecar는 다른 포트를 써야 합니다.
  • localhost 통신: 사이드카 패턴(로그 수집기, 프록시, Envoy 등)이 성립하는 근거입니다. Istio의 사이드카가 앱 트래픽을 가로챌 수 있는 것도 같은 netns 안에 있기 때문입니다.
  • 네트워크는 공유, 파일시스템은 격리: 컨테이너들은 netns는 공유하지만 파일시스템·프로세스 공간은 기본적으로 각자입니다(공유하려면 volume, shareProcessNamespace 필요). "무엇이 공유되고 무엇이 격리되는지"를 구분하는 것이 중요합니다.
Pod 안: 네트워크의 주인은 pause, 나머지는 같은 방의 세입자
🏢 빌딩 비유로 보면

컨테이너는 방 안의 개별 직원입니다. 직원들은 각자 내선 번호(포트)를 하나씩 맡습니다. 두 직원이 같은 내선 번호를 쓸 수 없고(포트 충돌), 옆자리 동료에게는 그냥 말을 걸면 됩니다(localhost). 방 번호(Pod IP)는 직원 개인 소유가 아니라 팀(Pod) 공용입니다.

LAYER 7 · CLUSTER

Cluster — 흩어진 노드들을 하나의 네트워크로

개별 부품은 다 모였습니다. 이제 이들을 "하나의 거대한 평평한 네트워크"로 묶는 클러스터 차원의 장치들을 봅니다.

① CNI가 노드 사이를 잇는 두 가지 방법

같은 노드 안의 Pod끼리는 브리지로 통신한다는 걸 배웠습니다. 그럼 다른 노드의 Pod에게는 어떻게 갈까요? Pod IP(10.244.x.x)는 물리 네트워크가 모르는 주소인데 말이죠. CNI는 두 가지 전략 중 하나를 씁니다.

전략 A · 오버레이 (터널링)
캡슐화해서 나른다 — VXLAN 등

Pod의 패킷을 노드 IP로 된 겉봉투에 한 번 더 포장(캡슐화)해서 보냅니다. 물리망은 노드 IP만 보고 배달하고, 받는 노드가 봉투를 뜯어 안의 Pod 패킷을 꺼냅니다. 물리망 설정 변경이 필요 없어 어디서나 동작. (Flannel VXLAN, Calico IPIP/VXLAN, Cilium VXLAN)

전략 B · 라우팅 (네이티브)
길을 알려준다 — BGP 등

"10.244.2.0/24 대역은 192.168.10.12 노드로 보내라"는 경로 정보를 물리망/노드에 직접 심습니다. 캡슐화 오버헤드가 없어 빠르지만 네트워크 환경의 협조가 필요. (Calico BGP, AWS VPC CNI — Pod에 VPC IP를 직접 부여)

🏢 빌딩 비유로 보면

오버레이 = 지점 간 우편을 보낼 때, 방 번호가 적힌 내부 봉투를 지점 주소가 적힌 겉봉투에 넣어 일반 택배(물리망)로 보내는 것. 택배 기사는 겉봉투만 봅니다. 라우팅 = 아예 시청 지도(라우팅 테이블)에 "3층 3xx호 우편은 강남지점으로"라고 등록해 두는 것. 겉봉투가 필요 없지만 시청의 협조가 필요하죠.

② Service — Pod IP의 무상함을 이기는 고정 주소

Pod IP는 Pod이 재생성될 때마다 바뀝니다. 그래서 쿠버네티스는 절대 바뀌지 않는 가상 IP(ClusterIP)를 만들어 그 뒤에 Pod들을 세워 둡니다.

  • Service는 실체가 없는 가상 IP입니다. 어떤 인터페이스에도 붙어 있지 않습니다.
  • 패킷이 ClusterIP(예: 10.96.0.10)로 향하면, 노드의 iptables/IPVS/eBPF 규칙이 목적지를 실제 Pod IP로 바꿔치기(DNAT)합니다. 여러 Pod 중 하나가 랜덤/라운드로빈으로 선택되어 로드밸런싱도 됩니다.
  • 이 규칙을 매 순간 갱신하는 것이 노드 계층에서 본 kube-proxy의 일입니다. Pod가 죽거나 새로 생기면 EndpointSlice가 갱신되고, kube-proxy가 이를 규칙에 반영합니다.
Service 타입무엇을 만드나주 용도
ClusterIP클러스터 내부 전용 가상 IP (기본값)마이크로서비스 간 내부 통신
NodePort모든 노드의 특정 포트(30000-32767)를 개방간단한 외부 노출, 테스트
LoadBalancer클라우드 LB를 생성해 NodePort 앞에 배치운영 환경의 외부 노출
Headless (clusterIP: None)가상 IP 없이 Pod IP 목록을 DNS로 직접 반환StatefulSet, DB 클러스터

③ DNS — 이름으로 부르기

IP를 외우는 대신 이름을 씁니다. 클러스터 안의 CoreDNS(그 자체도 Pod!)가 서비스명.네임스페이스.svc.cluster.local 형식의 이름을 ClusterIP로 풀어 줍니다. 모든 Pod의 /etc/resolv.conf는 CoreDNS를 바라보도록 자동 설정됩니다.

# Deployment(Pod 3개) + Service 생성
$ kubectl create deployment web --image=nginx --replicas=3
$ kubectl expose deployment web --port=80
$ kubectl get svc web
NAME   TYPE        CLUSTER-IP     PORT(S)   AGE
web    ClusterIP   10.96.142.7    80/TCP    5s

# Service 뒤의 실제 Pod IP 목록 (EndpointSlice)
$ kubectl get endpointslices -l kubernetes.io/service-name=web
NAME        ADDRESSTYPE   PORTS   ENDPOINTS
web-x7k2p   IPv4          80      10.244.1.2,10.244.1.3,10.244.2.5

# ClusterIP로 호출하면 세 Pod 중 하나로 분배됨
$ kubectl run test --rm -it --image=busybox -- wget -qO- 10.96.142.7
<!DOCTYPE html> ... Welcome to nginx! ...
# 이름으로 호출 — 같은 네임스페이스면 서비스명만으로 충분
$ kubectl run test --rm -it --image=busybox -- nslookup web
Server:    10.96.0.10          ← CoreDNS의 ClusterIP
Name:      web.default.svc.cluster.local
Address:   10.96.142.7         ← web 서비스의 ClusterIP

# FQDN 구조: {svc}.{namespace}.svc.{cluster-domain}
#   web            → 같은 네임스페이스
#   web.payment    → payment 네임스페이스의 web 서비스
#   web.payment.svc.cluster.local → 완전한 이름
# "Service는 iptables 규칙일 뿐"임을 눈으로 확인 (iptables 모드 기준)
worker-1$ sudo iptables -t nat -L KUBE-SERVICES -n | grep 10.96.142.7
KUBE-SVC-XXXX  tcp  --  0.0.0.0/0  10.96.142.7  tcp dpt:80

KUBE-SEP-AAA ... statistic mode random probability 0.33333  → 10.244.1.2
KUBE-SEP-BBB ... statistic mode random probability 0.50000  → 10.244.1.3
KUBE-SEP-CCC ... (나머지)                                    → 10.244.2.5

④ 외부에서 클러스터로: Ingress와 Gateway

지금까지는 클러스터 내부 이야기였습니다. 외부 사용자의 HTTP 요청은 보통 Ingress(또는 최신 표준인 Gateway API)를 통해 들어옵니다. Ingress Controller(nginx, Traefik 등 — 이것도 Pod!)가 L7에서 도메인·경로 기반 라우팅을 수행한 뒤, 알맞은 Service로 넘깁니다.

🛬 외부 요청의 전형적인 여정

사용자 → DNS → 클라우드 LB(LoadBalancer Service) → 어느 노드의 NodePort → Ingress Controller Pod → 라우팅 규칙 매칭 → 대상 Service(ClusterIP) → DNAT → 애플리케이션 Pod. 계층을 배웠으니 이제 이 문장이 읽히실 겁니다.

⑤ NetworkPolicy — 기본은 "전부 허용", 잠그는 건 우리 몫

IP-per-Pod 모델의 대가로, 쿠버네티스는 기본적으로 모든 Pod이 모든 Pod과 통신 가능합니다. 운영 환경에서는 NetworkPolicy로 "payment 네임스페이스에는 api Pod만 접근 가능" 같은 방화벽 규칙을 걸어야 합니다. 이 규칙을 실제로 집행하는 것도 CNI입니다(Flannel 단독은 미지원, Calico/Cilium 지원 — CNI 선택이 중요한 이유).

세 가지 통신 범위 비교 요약

SCOPE 1 · INTRA-POD
같은 Pod 안

netns 공유. 네트워크 장비를 전혀 거치지 않는 커널 내 루프백 통신. 가장 빠릅니다.

app → localhost:9090 → sidecar
SCOPE 2 · INTRA-NODE
같은 노드, 다른 Pod

veth를 나와 브리지(또는 노드 라우팅)를 거쳐 상대 Pod의 veth로. 물리 NIC은 통과하지 않습니다.

PodA eth0 → veth → cni0 → veth → PodB eth0
SCOPE 3 · INTER-NODE
다른 노드의 Pod

브리지에서 답을 못 찾으면 노드 라우팅으로. CNI가 캡슐화(오버레이) 또는 라우팅(BGP)으로 상대 노드까지 배달합니다.

PodA → cni0 → NodeA eth0 → 물리망 → NodeB eth0 → cni0 → PodB

CHEATSHEET

계층별 트러블슈팅 명령어 치트시트

"통신이 안 돼요"라는 장애가 오면, 위(앱)에서 아래(물리)로 계층을 하나씩 내려가며 확인합니다.

# [1단계] Pod은 살아 있고 IP는 받았는가?
$ kubectl get pods -o wide                    # STATUS, IP, NODE 확인
$ kubectl describe pod <pod>                  # Events에서 CNI 오류 확인
$ kubectl logs <pod> -c <container>           # 앱이 정말 리슨 중인가

# [2단계] Pod 안에서 직접 확인
$ kubectl exec -it <pod> -- sh
# netstat -tlnp        # 어떤 포트를 리슨 중인가 (ss -tlnp도 가능)
# curl -v localhost:8080  # 자기 자신은 응답하는가
# cat /etc/resolv.conf    # DNS 설정은 정상인가

# 디버깅 도구가 없는 이미지라면 임시 디버그 컨테이너 주입
$ kubectl debug -it <pod> --image=nicolaka/netshoot --target=<container>
# [3단계] Service가 Pod을 제대로 바라보는가?
$ kubectl get svc <svc>                        # ClusterIP, 포트 확인
$ kubectl get endpointslices -l kubernetes.io/service-name=<svc>
#  → ENDPOINTS가 비어 있다면? selector와 Pod label 불일치가 90%!
$ kubectl get pods --show-labels               # label 대조

# [4단계] DNS가 이름을 풀어 주는가?
$ kubectl run dnstest --rm -it --image=busybox -- nslookup <svc>
$ kubectl get pods -n kube-system -l k8s-app=kube-dns   # CoreDNS 상태
$ kubectl logs -n kube-system -l k8s-app=kube-dns --tail=20
# [5단계] 노드와 CNI는 건강한가?
$ kubectl get nodes                             # NotReady = CNI 문제 단골
$ kubectl get pods -n kube-system -o wide       # CNI 데몬셋 상태 (calico-node 등)
$ kubectl get node <node> -o jsonpath='{.spec.podCIDR}'

# kube-proxy 상태와 모드
$ kubectl logs -n kube-system -l k8s-app=kube-proxy --tail=20
$ kubectl describe cm -n kube-system kube-proxy | grep mode
# [6단계] 노드 OS 레벨에서 직접 추적 (SSH 접속 후)
node$ ip addr && ip route                # 인터페이스와 라우팅 테이블
node$ ip link | grep veth               # Pod 수만큼 veth가 있는가
node$ sudo iptables -t nat -L KUBE-SERVICES -n | grep <ClusterIP>
node$ sudo tcpdump -i any host <PodIP> -nn   # 최종 병기: 패킷 직접 관찰

# [7단계] 물리/기반 네트워크
node$ ping <다른-노드-IP>                # 노드 간 기본 연결성
node$ ethtool eth0 | grep Speed          # 링크 속도 (H/W 계층!)
node$ ip -s link show eth0               # 드롭/에러 카운터
