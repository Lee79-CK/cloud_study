Apache Airflow,

매일 새벽 3시에 데이터를 모으고, 피처를 만들고, 모델을 학습시키고, 실패하면 알려주는 일 — 이걸 사람이 지키고 앉아 있을 수는 없습니다. Airflow는 이 모든 흐름을 Python 코드 한 파일로 정의하고, 스케줄링하고, 감시하는 워크플로 오케스트레이션의 사실상 표준(de facto standard)입니다.

Chapter 01 · What is Airflow

Airflow란 무엇인가 — "Workflows as Code"

Apache Airflow는 워크플로(workflow)를 프로그래밍 방식으로 작성하고, 스케줄링하고, 모니터링하는 오픈소스 플랫폼입니다. 2014년 Airbnb에서 시작되어 2019년 Apache 최상위 프로젝트(Top-Level Project)가 되었고, 현재 데이터 엔지니어링 분야에서 가장 널리 쓰이는 오케스트레이션 도구입니다.

핵심 철학은 단 하나, "Workflows as Code"입니다. 파이프라인을 GUI에서 마우스로 그리는 게 아니라 Python 코드로 정의합니다. 코드이기 때문에 얻는 것들이 곧 Airflow의 장점입니다.

🧬

버전 관리

파이프라인이 .py 파일이므로 Git으로 diff, review, rollback이 전부 가능합니다. "지난주에 누가 뭘 바꿨지?"에 즉시 답할 수 있습니다.

🔁

동적 생성

for 루프로 태스크 100개를 찍어내거나, 설정 파일을 읽어 DAG 자체를 동적으로 생성할 수 있습니다. GUI 도구로는 불가능한 영역입니다.

🧪

테스트 가능

일반 Python 코드처럼 단위 테스트를 작성하고 CI에서 검증합니다. 파이프라인도 소프트웨어 엔지니어링의 규율 안으로 들어옵니다.

#Airflow가 하는 일 vs 하지 않는 일

초보자가 가장 많이 혼동하는 지점입니다. Airflow는 오케스트레이터(지휘자)이지, 데이터 처리 엔진(연주자)이 아닙니다.

구분Airflow의 역할 ⭕Airflow의 역할이 아님 ❌
실행 제어"Spark 잡을 새벽 2시에 실행해"라고 명령하고 결과를 추적대용량 데이터를 Airflow 워커 안에서 직접 셔플/조인 (그건 Spark의 일)
데이터 이동S3 → DW 적재 작업을 트리거하고 성공/실패 관리스트리밍 데이터 실시간 처리 (그건 Kafka/Flink의 일)
ML 학습학습 잡을 GPU 클러스터에 제출하고 완료를 기다림Airflow 워커에서 직접 거대 모델 학습 (메모리 폭발)
처리 단위배치(batch) — 정해진 주기나 이벤트 기반 실행밀리초 단위 실시간 응답 (그건 API 서버의 일)
💡

한 줄 요약: Airflow는 무거운 짐을 직접 들지 않습니다. "누가, 언제, 어떤 순서로 들지"를 정하고, 들다가 넘어지면 다시 들게 하거나 사람을 부르는 현장 감독입니다.

Chapter 02 · Why Orchestration

왜 필요한가 — cron만으로 버티다 무너지는 순간

처음엔 누구나 cron으로 시작합니다. 0 2 * * * python train.py 한 줄이면 매일 새벽 2시에 학습이 돌아가니까요. 그런데 파이프라인이 자라면서 반드시 다음 질문들에 부딪힙니다.

  • "어제 학습 돌았어?" — cron은 실행 기록을 남기지 않습니다. 로그 파일을 직접 뒤져야 합니다.
  • "데이터 수집이 늦어졌는데 학습이 먼저 돌아버렸어" — cron에는 의존성 개념이 없습니다. A가 끝나야 B가 도는 보장이 없습니다.
  • "네트워크가 잠깐 끊겨서 실패했는데 아무도 몰랐어" — 재시도도, 알림도 직접 코드로 짜야 합니다.
  • "지난 한 달치를 다시 돌려야 해" — 과거 날짜 기준 재실행(backfill)은 cron으로는 수작업 지옥입니다.
  • "서버 한 대로는 이제 안 돼" — cron은 단일 머신에 묶여 있습니다. 분산 실행, 동시성 제어가 없습니다.

워크플로 오케스트레이터는 정확히 이 다섯 가지를 해결하기 위해 존재합니다.

요구사항cronAirflow
의존성 관리없음 시간차로 어림짐작DAG extract >> transform >> train 한 줄로 명시
실패 처리없음 조용히 죽음내장 재시도 횟수·간격, 실패 콜백, 이메일/Slack 알림
실행 이력없음메타데이터 DB 모든 실행의 상태·로그·소요시간 기록
과거 재실행수작업backfill 날짜 구간 지정 한 번으로 일괄 재실행
분산 실행단일 머신Executor Celery/Kubernetes로 수평 확장
가시성로그 파일Web UI 그래프·간트차트·로그를 브라우저에서 확인
🎯

판단 기준: 작업이 3개 이상이고 서로 순서 의존성이 있으며 실패하면 곤란한 파이프라인이라면, cron을 졸업할 때입니다.

Chapter 03 · Core Concepts

핵심 개념 사전 — 이 8개만 알면 대화가 됩니다

Airflow 문서와 동료들의 대화에 끊임없이 등장하는 용어들입니다. 각각을 비유와 함께 정리합니다.

DAG — 작업 흐름도 (악보)

Directed Acyclic Graph. 방향이 있고(→) 순환이 없는(한 바퀴 돌아 제자리로 못 옴) 그래프입니다. "무엇을, 어떤 순서로, 언제" 실행할지의 청사진. 하나의 .py 파일이 보통 하나의 DAG를 정의합니다.

Task — 작업 단위 (음표)

DAG를 구성하는 개별 노드. "데이터 내려받기", "모델 학습" 같은 하나의 작업입니다. Task의 1회 실행 인스턴스를 Task Instance라 부르며, 각각 queued → running → success/failed 상태를 가집니다.

Operator — 작업 템플릿 (악기)

Task를 만드는 틀. PythonOperator(파이썬 함수 실행), BashOperator(셸 명령), KubernetesPodOperator(K8s Pod 실행) 등. 커뮤니티 Provider 패키지로 AWS, GCP, Spark, Slack 등 수백 종이 제공됩니다.

Scheduler — 지휘자

Airflow의 심장. DAG 파일을 주기적으로 읽고(파싱), 실행 시각이 된 DAG Run을 생성하고, 의존성이 충족된 Task를 큐에 밀어넣습니다. 이 프로세스가 죽으면 아무것도 실행되지 않습니다.

Executor + Worker — 연주자

큐에 들어온 Task를 실제로 실행하는 주체. Executor는 "어디서 어떻게 실행할지"의 전략이고(7장에서 상세히), Worker는 그 전략에 따라 일하는 프로세스/Pod입니다.

Metadata DB — 기억 장치

모든 DAG/Task의 상태, 실행 이력, 변수, 연결 정보를 저장하는 데이터베이스(운영에서는 PostgreSQL 권장). Scheduler·Webserver·Worker가 모두 이 DB를 통해 상태를 공유합니다.

XCom — 쪽지 전달

"Cross-Communication". Task 사이에 작은 데이터를 주고받는 메커니즘입니다. 학습 결과 메트릭, 파일 경로 같은 것을 전달합니다. 메타데이터 DB에 저장되므로 대용량 데이터 전달용이 절대 아닙니다 — 데이터프레임은 S3 경로만 넘기세요.

Sensor & Hook — 감지기와 연결선

Sensor는 "조건이 충족될 때까지 기다리는" 특수 Operator(예: S3에 파일이 도착하면 진행). Hook은 외부 시스템(DB, 클라우드)과의 연결을 추상화한 인터페이스로, Operator 내부에서 쓰입니다.

⚠️

왜 하필 '비순환(Acyclic)'인가? A → B → C → A처럼 순환이 있으면 "A를 실행하려면 C가 끝나야 하고, C는 B가, B는 A가 끝나야" 하는 교착 상태에 빠집니다. 순환이 없어야 위상 정렬(topological sort)로 실행 순서를 항상 결정할 수 있습니다. 반복이 필요하면 순환 대신 DAG 자체를 주기 실행하는 방식으로 풉니다.

Chapter 04 · Architecture

아키텍처 해부 — 컴포넌트를 클릭해 보세요

Airflow는 단일 프로그램이 아니라 역할이 분리된 여러 컴포넌트의 협주입니다. 아래 다이어그램의 각 박스를 클릭하면 설명이 나타납니다.

👆 박스를 클릭하면 각 컴포넌트의 역할 설명이 여기에 표시됩니다.

#한 번의 실행이 흘러가는 길

  • ① 파싱 — Scheduler가 dags/ 폴더의 .py 파일을 주기적으로 읽어 DAG 구조를 메타데이터 DB에 등록합니다.
  • ② 트리거 — 스케줄 시각이 되면 Scheduler가 DAG Run(해당 시점의 실행 인스턴스)을 생성합니다.
  • ③ 큐잉 — 의존성(upstream)이 모두 성공한 Task Instance를 Executor의 큐에 넣습니다.
  • ④ 실행 — Worker가 큐에서 Task를 꺼내 실행하고, 진행 상태와 로그를 기록합니다.
  • ⑤ 상태 갱신 — 결과(success/failed)가 메타데이터 DB에 반영되고, 이를 본 Scheduler가 다음 Task를 큐잉합니다. Webserver는 이 DB를 읽어 UI에 보여줄 뿐입니다.
🔍

Airflow 3.0 참고: 2025년 4월 출시된 Airflow 3.0에서는 Worker가 메타데이터 DB에 직접 접근하지 않고 API Server를 경유하는 구조(Task Execution API)로 바뀌어 보안과 격리가 강화되었습니다. 또한 DAG 버전 관리, 이벤트 기반 스케줄링(Assets), React 기반 새 UI가 도입되었습니다. 개념 자체는 2.x와 동일하니 이 강의 내용은 그대로 유효합니다.

Chapter 05 · Writing a DAG

첫 DAG 작성하기 — 두 가지 문법

같은 파이프라인을 두 가지 스타일로 작성해 봅니다. 전통 방식(Operator 직접 사용)은 오래된 코드베이스에서 반드시 만나게 되고, TaskFlow API(@task 데코레이터, Airflow 2.0+)는 새 프로젝트의 권장 방식입니다. 탭을 눌러 비교해 보세요.

dags/ml_pipeline.py 
from airflow.decorators import dag, task
from datetime import datetime

@dag(
    dag_id="ml_pipeline_taskflow",
    schedule="0 2 * * *",              # 매일 새벽 2시 (cron 표현식)
    start_date=datetime(2026, 1, 1),
    catchup=False,                       # 과거분 자동 소급 실행 끔 (6장 참고)
    default_args={"retries": 2},       # 실패 시 2회 재시도
    tags=["ml", "daily"],
)
def ml_pipeline():

    @task
    def extract() -> str:
        # 실제로는 DB/API에서 데이터를 받아 S3에 저장
        path = "s3://my-bucket/raw/2026-06-12.parquet"
        return path                  # return 값이 자동으로 XCom에 저장됨

    @task
    def build_features(raw_path: str) -> str:
        print(f"피처 생성: {raw_path}")
        return "s3://my-bucket/features/2026-06-12.parquet"

    @task
    def train(feat_path: str) -> dict:
        # 실전에서는 KubernetesPodOperator로 GPU 잡 제출 (10장 참고)
        return {"auc": 0.91, "model_uri": "s3://models/v42"}

    @task
    def deploy(metrics: dict):
        if metrics["auc"] < 0.85:
            raise ValueError("품질 기준 미달 — 배포 중단")
        print(f"배포: {metrics['model_uri']}")

    # 함수 호출의 연쇄가 곧 의존성 그래프가 됩니다
    deploy(train(build_features(extract())))

ml_pipeline()
from airflow import DAG
from airflow.operators.python import PythonOperator
from airflow.operators.bash import BashOperator
from datetime import datetime

def _extract(**context):
    path = "s3://my-bucket/raw/2026-06-12.parquet"
    # XCom에 명시적으로 push
    context["ti"].xcom_push(key="raw_path", value=path)

def _train(**context):
    # XCom에서 명시적으로 pull
    path = context["ti"].xcom_pull(key="raw_path", task_ids="extract")
    print(f"학습 시작: {path}")

with DAG(
    dag_id="ml_pipeline_classic",
    schedule="0 2 * * *",
    start_date=datetime(2026, 1, 1),
    catchup=False,
) as dag:

    extract = PythonOperator(task_id="extract", python_callable=_extract)

    build_features = BashOperator(
        task_id="build_features",
        bash_command="python /opt/jobs/build_features.py",
    )

    train = PythonOperator(task_id="train", python_callable=_train)

    # 의존성을 비트시프트 연산자로 명시 — Airflow의 시그니처 문법
    extract >> build_features >> train

#의존성 표현 치트시트

코드의미
a >> ba가 성공하면 b 실행 (a는 b의 upstream)
a >> [b, c]a가 끝나면 b와 c를 병렬 실행 (fan-out)
[b, c] >> db와 c가 모두 성공해야 d 실행 (fan-in)
a >> b >> c직렬 체인
chain(a, b, c) / cross_downstream(...)복잡한 그래프를 함수형으로 표현
💡

TaskFlow의 매력: 함수의 return → 인자 전달이 자동으로 XCom + 의존성 설정으로 변환됩니다. xcom_push/pull 보일러플레이트가 사라지고, 그냥 Python 함수를 합성하듯 파이프라인이 짜집니다.

Chapter 06 · Scheduling

스케줄링의 함정 — 모든 입문자가 한 번씩 빠지는 곳

Airflow 스케줄링에는 입문자의 9할이 헷갈리는 규칙이 하나 있습니다: DAG는 "데이터 구간(data interval)이 끝난 뒤"에 실행됩니다.

예를 들어 매일 자정 스케줄(@daily)의 "6월 11일자 실행"은 6월 11일 00:00이 아니라 6월 12일 00:00에 시작됩니다. 왜냐하면 "6월 11일 하루치 데이터"는 11일이 다 지나야 완성되기 때문입니다. 이 실행의 logical_date(구 execution_date)는 6월 11일 — 즉 처리 대상 데이터의 날짜이지, 실제 실행 시각이 아닙니다.

🗓️ 인터랙티브: 스케줄 → 실제 실행 시각 계산기

#catchup과 backfill

  • catchup=True(과거 기본값) — start_date부터 현재까지 밀린 구간을 전부 자동 실행합니다. start_date를 1년 전으로 잡으면 DAG 활성화 순간 365개의 DAG Run이 쏟아집니다. 의도한 게 아니라면 재앙이므로, 대부분의 DAG에서 catchup=False를 명시합니다. (Airflow 3.0부터는 False가 기본값)
  • backfill — 반대로, 과거 구간을 의도적으로 다시 돌리는 기능. airflow dags backfill -s 2026-05-01 -e 2026-05-31 my_dag 한 줄로 5월 한 달치를 재실행합니다. "로직을 고쳤으니 지난달 데이터를 다시 처리해야 해"라는 요구의 정답입니다.
  • idempotency(멱등성) — backfill이 안전하려면 같은 날짜로 몇 번을 돌려도 결과가 같아야 합니다. INSERT 대신 날짜 파티션 DELETE 후 INSERT(또는 upsert), 파일은 덮어쓰기 — 이 원칙이 Airflow 파이프라인 설계의 1번 규칙입니다.
🚨

실수 단골 1위: start_date=datetime.now()는 절대 금지. 파싱할 때마다 값이 바뀌어 스케줄러가 혼란에 빠집니다. start_date는 항상 과거의 고정된 시각으로 지정하세요.

#비유로 잡는 포지셔닝

🏛️ Airflow — 종합 관제탑

가장 오래되고 가장 넓습니다. 어떤 시스템과도 연결되는 Provider 생태계, 방대한 레퍼런스, 채용 시장의 공용어. 대신 "스케줄 기반 배치"라는 태생적 세계관이 있어 이벤트 기반·초동적 워크플로는 상대적으로 부자연스럽습니다.

☸️ Argo Workflows — K8s의 원주민

워크플로 자체가 Kubernetes CRD입니다. 모든 스텝이 컨테이너이고, kubectl로 다룹니다. K8s 운영팀에게는 더없이 자연스럽지만, "Python으로 짜고 싶다"는 데이터 사이언티스트에게 YAML은 장벽입니다. KT처럼 K8s 인프라가 표준인 조직의 플랫폼 레이어에서 강세.

🌊 Prefect — 동적 흐름의 자유인

"워크플로는 그냥 Python 코드여야 한다"는 철학. if/for/while로 흐름을 바꾸고, 실행 시점에 구조가 결정되는 파이프라인이 자연스럽습니다. 가볍게 시작하기 가장 쉽지만 생태계와 레퍼런스는 Airflow보다 얇습니다.

🧱 Dagster — 데이터 자산 중심 사고

"태스크를 실행한다"가 아니라 "데이터 자산(asset)을 최신으로 유지한다"로 관점을 뒤집습니다. 테이블·모델·리포트가 일급 객체이고 리니지가 기본 제공됩니다. 데이터 플랫폼을 처음부터 설계한다면 유력한 대안.

🤖 Kubeflow Pipelines — ML 실험 특화

하이퍼파라미터 튜닝, 실험 비교, 모델 아티팩트 추적 등 ML 실험 루프에 특화. 단, Kubeflow 전체 스택의 운영 부담이 크기로 유명합니다. 이미 Kubeflow를 운영 중이라면 자연스럽고, 아니라면 도입 비용을 신중히 계산해야 합니다.

⏰ cron / Jenkins — 졸업 대상

cron은 2장에서 본 한계 그대로이고, Jenkins는 CI/CD 도구를 데이터 파이프라인에 오용하는 경우입니다. 의존성·backfill·데이터 인터벌 개념이 없어 규모가 커지면 반드시 한계에 부딪힙니다.

#실전 패턴: GPU 학습 잡을 K8s에 위임하기

"Airflow는 지휘만, 무거운 일은 위임" 원칙의 대표 코드입니다. Airflow 워커는 가볍게 유지하고, 학습은 GPU 노드의 Pod에서 수행합니다.

dags/gpu_training.py — KubernetesPodOperator
from airflow.providers.cncf.kubernetes.operators.pod import KubernetesPodOperator
from kubernetes.client import models as k8s

train_on_gpu = KubernetesPodOperator(
    task_id="train_on_gpu",
    name="llm-finetune",
    namespace="ml-jobs",
    image="registry.local/llm-train:v1.4",   # 학습 환경은 이미지로 격리
    cmds=["python", "train.py"],
    arguments=["--data", "s3://features/{{ ds }}"],  # {{ ds }} = logical_date 템플릿
    container_resources=k8s.V1ResourceRequirements(
        limits={"nvidia.com/gpu": "1", "memory": "32Gi"},
    ),
    node_selector={"gpu-type": "a100"},      # GPU 노드로 핀
    get_logs=True,                              # Pod 로그를 Airflow UI로
    is_delete_operator_pod=True,                # 종료 후 Pod 정리
)
🧪

홈랩 적용 아이디어: DGX Spark + ThinkStation 2노드 클러스터라면, Airflow를 x86 노드에 Helm으로 올리고 node_selector로 학습 Pod만 ARM64 GPU 노드(Spark)에 보내는 구성이 이 패턴의 축소판 실습으로 딱 맞습니다. 단, 이미지가 aarch64용으로 빌드되어야 한다는 점만 주의하세요.

Chapter 12 · Best Practices

베스트 프랙티스 & 안티패턴 — 현업의 흉터에서 배운 것

✅

멱등성을 설계의 1원칙으로

같은 logical_date로 몇 번을 재실행해도 결과가 동일하게. 날짜 파티션 overwrite, upsert를 기본으로 하면 retry와 backfill이 두려움 없는 도구가 됩니다.

✅

Top-level 코드는 가볍게

DAG 파일의 최상위는 스케줄러가 수십 초마다 반복 파싱합니다. 최상위에서 DB 조회·API 호출·무거운 import를 하면 스케줄러 전체가 느려집니다. 무거운 일은 반드시 task 함수 안으로.

✅

XCom에는 '참조'만

데이터프레임이 아니라 S3 경로를, 모델이 아니라 모델 레지스트리 URI를 전달하세요. 실데이터는 외부 스토리지, XCom은 그 주소록입니다.

✅

알림과 SLA를 처음부터

on_failure_callback으로 Slack 알림, 핵심 DAG에는 SLA를 설정해 "조용한 실패"를 구조적으로 차단하세요. 모르는 실패가 가장 비쌉니다.

🚫

Sensor 남용 (worker slot 점거)

기본 모드 Sensor는 기다리는 동안 워커 슬롯을 통째로 점유합니다. mode="reschedule" 또는 Deferrable Operator(Triggerer 활용)로 대기 비용을 0에 가깝게 만드세요.

🚫

Task 간 로컬 파일 공유

"이전 Task가 /tmp에 쓴 파일을 다음 Task가 읽기" — LocalExecutor에선 되다가 Celery/K8s로 가는 순간 깨집니다. Task는 서로 다른 머신/Pod에서 돈다고 항상 가정하세요. 공유는 객체 스토리지로.

🚫

거대 단일 Task

"하나의 PythonOperator에서 추출+가공+학습+배포 전부" — 실패 지점 파악도, 부분 재실행도 불가능해집니다. 재시도하고 싶은 단위 = Task의 단위입니다.

확인 퀴즈 — 8문제로 점검하기

각 문항의 선택지를 클릭하면 즉시 정답과 해설이 표시됩니다.

DAG에서 "Acyclic(비순환)"이 필수인 가장 근본적인 이유는?
해설: A→B→C→A 순환이 있으면 어느 것도 먼저 시작할 수 없습니다. 비순환이어야 위상 정렬로 실행 순서가 항상 유일하게 결정됩니다. 반복이 필요하면 DAG 자체를 주기 실행합니다.
@daily 스케줄 DAG에서 logical_date가 6월 11일인 DAG Run이 실제로 시작되는 시각은?
해설: Airflow는 "데이터 구간이 끝난 뒤" 실행합니다. 6월 11일 하루치 데이터는 12일 자정에야 완성되므로, logical_date=6/11 실행은 6/12 00:00에 시작됩니다. 입문자 혼란 1순위 개념입니다.
Task 사이에 10GB짜리 학습 데이터셋을 전달하는 올바른 방법은?
해설: XCom은 메타데이터 DB에 저장되므로 소량 데이터 전용입니다. 전역 변수와 /tmp는 Task가 서로 다른 프로세스/Pod에서 돌면 깨집니다. 실데이터는 외부 스토리지, XCom은 주소만 — 철칙입니다.
Task마다 요구 자원이 크게 다른(일부는 GPU 필요) ML 워크로드에 가장 자연스러운 Executor는?
해설: KubernetesExecutor는 Task마다 Pod을 새로 만들기 때문에 GPU 노드 셀렉터, 컨테이너 이미지, 자원 한도를 Task 단위로 격리 지정할 수 있습니다. 자원 이질성이 큰 ML 워크로드와 궁합이 가장 좋습니다.
catchup=True인 DAG의 start_date를 1년 전으로 설정하고 활성화하면 일어나는 일은?
해설: catchup=True는 start_date부터 현재까지의 모든 미실행 구간을 소급 실행합니다. 의도한 backfill이 아니라면 시스템에 부하 폭탄이 되므로, 대부분의 DAG는 catchup=False로 시작합니다.
"Airflow는 오케스트레이터다"라는 말의 올바른 해석은?
해설: Airflow는 "현장 감독"입니다. 데이터 처리 엔진(Spark/Flink), 학습 인프라(GPU 클러스터), 서빙(API 서버)을 대체하는 게 아니라 그들을 올바른 순서로 호출하고 결과를 추적합니다.
데이터 자산(테이블·모델)의 리니지와 최신성 관리가 핵심 요구라면, Airflow의 대안으로 가장 그 철학에 부합하는 도구는?
해설: Dagster는 "태스크 실행"이 아닌 "데이터 자산을 최신으로 유지"를 1급 개념으로 삼는 asset 중심 오케스트레이터로, 리니지·카탈로그가 기본 내장입니다. Argo는 K8s 네이티브 컨테이너 실행, cron/Jenkins는 오케스트레이션 도구가 아닙니다.
DAG 파일 최상위(top-level)에서 무거운 API 호출을 하면 안 되는 이유는?
해설: dags/ 폴더의 파일은 스케줄러가 지속적으로 import하며 파싱합니다. 최상위 코드는 그때마다 실행되므로, 무거운 연산·네트워크 호출은 반드시 task 함수 내부로 옮겨야 합니다.
