    검색으로 기억을 빌려주는 LLM,
    RAG 완전 정복

    LLM은 똑똑하지만 "내 회사 문서"는 모릅니다. RAG는 질문에 필요한 자료를 먼저 찾아서(Retrieval) LLM에게 건네주고 답을 쓰게(Generation) 하는 기법입니다. 개념부터 컴포넌트, 직접 구현, 다른 기법과의 비교까지 — 만지면서 배웁니다.

    검색 Retrieval 증강 Augmented 생성 Generation 예상 학습 · 40분 실습 위젯 4종
    01동기

    왜 RAG가 필요한가

    RAG를 이해하려면 먼저 "그냥 LLM"이 무엇을 못 하는지를 알아야 합니다. RAG는 새로운 능력을 더하는 게 아니라, LLM의 구조적 약점 네 가지를 메우는 처방전이기 때문입니다.

    1환각 (Hallucination)

    LLM은 "모른다"고 말하기보다 그럴듯한 답을 지어내는 경향이 있습니다. 학습 데이터의 통계적 패턴으로 다음 단어를 예측할 뿐, 사실을 조회하는 장치가 없기 때문입니다. RAG는 실제 근거 문서를 함께 주어 "지어내기"를 "읽고 답하기"로 바꿉니다.

    2지식 컷오프 (Knowledge Cutoff)

    모델은 학습이 끝난 시점까지의 지식만 가집니다. 어제 발표된 정책, 오늘 바뀐 사내 규정은 알 수 없습니다. RAG는 최신 문서를 그때그때 검색해 주입하므로, 모델을 재학습하지 않고도 지식을 갱신합니다.

    3도메인·사내 지식 부재 (Private Knowledge)

    공개 데이터로 학습한 모델은 우리 회사 위키, 계약서, 운영 매뉴얼을 본 적이 없습니다. 이런 비공개 자료는 애초에 학습에 포함될 수 없습니다. RAG는 사내 문서를 검색 대상으로 삼아 "우리 회사를 아는 LLM"을 만듭니다.

    4출처·추적성 부재 (No Citation)

    일반 LLM의 답은 "어디서 나온 말인지" 알 수 없어 검증이 어렵습니다. RAG는 답의 근거가 된 문서를 함께 보여줄 수 있어, 출처를 인용하고 신뢰성을 높입니다. 규제 산업·기업 환경에서 결정적인 장점입니다.

    한 줄 정의

    RAG(Retrieval-Augmented Generation, 검색 증강 생성) = 질문이 들어오면 ① 관련 문서를 검색해서 ② LLM 프롬프트에 근거로 끼워 넣고 ③ 그 근거를 바탕으로 답을 생성하게 하는 구조.

    02개념

    RAG를 한 장면으로

    가장 쉬운 비유는 "오픈북 시험"입니다. 일반 LLM은 외운 것만으로 답하는 폐쇄형 시험(closed-book)을 봅니다. 외운 게 틀렸으면 그냥 틀린 답을 자신 있게 씁니다.

    RAG는 시험장에 교재와 노트를 들고 들어가는 오픈북 시험입니다. 문제(질문)를 읽고 → 관련 페이지를 펼쳐 찾고(검색) → 그 내용을 보면서 답을 작성합니다(생성). 학생(LLM)의 머리가 좋아진 게 아니라, 참고할 자료를 손에 쥐어 준 것이 핵심입니다.

    자주 하는 오해

    "RAG는 모델을 다시 학습시키는 것"이라고 생각하기 쉽지만 아닙니다. RAG는 모델의 가중치를 전혀 건드리지 않습니다. 단지 입력 프롬프트에 검색한 문서를 덧붙일 뿐입니다. 그래서 빠르고, 싸고, 문서만 바꾸면 즉시 지식이 업데이트됩니다. (재학습 = 파인튜닝은 뒤에서 비교합니다.)

    ▹ RAG = 두 단계, 두 시점

    RAG는 시점이 다른 두 묶음의 작업으로 나뉩니다. 이 구분이 전체 그림의 뼈대입니다.

    1. 인덱싱(Indexing) — 미리, 오프라인. 문서를 잘게 나눠 → 숫자 벡터로 바꿔 → 검색 가능한 데이터베이스에 저장해 둡니다. 질문이 오기 전에 끝내 두는 "도서관 정리" 단계입니다.
    2. 검색·생성(Retrieval & Generation) — 실시간, 질문이 올 때. 사용자 질문을 받아 → 비슷한 문서를 찾아 → 프롬프트에 넣어 → LLM이 답을 생성합니다.

    기억할 점

    왼쪽 절반(질문 임베딩 · 검색 · 재순위화)은 "무엇을 읽힐지"를 정하는 검색 문제이고, 오른쪽 절반(프롬프트 조립 · 생성)은 "어떻게 답하게 할지"를 정하는 생성 문제입니다. RAG 품질이 나쁠 때 십중팔구 원인은 왼쪽, 즉 검색이 엉뚱한 문서를 가져온 것입니다.

    04컴포넌트 · 인덱싱

    ① 문서 로더 — 원본 모으기

    모든 것은 "검색할 자료"를 모으는 데서 시작합니다. Document Loader는 PDF, 워드, 마크다운, 웹페이지, Notion, 데이터베이스, S3/MinIO 객체 스토리지 등 흩어진 원본을 읽어 일관된 텍스트 + 메타데이터 형태로 변환합니다.

    • 본문 추출: PDF의 레이아웃, 표, 헤더/푸터를 제거하고 의미 있는 텍스트만 뽑습니다. (스캔 PDF라면 OCR이 필요합니다.)
    • 메타데이터 부착: source(출처 파일), page, 작성일, 부서 등을 함께 저장합니다. 나중에 출처 인용과 필터링에 씁니다.
    • 품질이 곧 천장: 여기서 깨진 텍스트가 들어오면 그 뒤 단계가 아무리 좋아도 답이 좋아질 수 없습니다. "Garbage In, Garbage Out".
    05컴포넌트 · 인덱싱

    ② 청킹 — 문서를 조각내기

    긴 문서를 통째로 다루면 두 가지 문제가 생깁니다. (1) 임베딩 모델·LLM에는 입력 길이 한계가 있고, (2) 한 덩어리에 여러 주제가 섞이면 검색 정확도가 떨어집니다. 그래서 문서를 청크(chunk)라는 작은 조각으로 나눕니다.

    ▹ 핵심 변수 두 개

    • 청크 크기(chunk size): 한 조각의 길이. 너무 크면 한 조각에 잡음이 섞이고, 너무 작으면 문맥이 잘려 의미가 사라집니다.
    • 겹침(overlap): 인접한 조각이 일부 내용을 공유하게 하는 영역. 문장이 경계에서 잘려 맥락이 끊기는 것을 막아 줍니다.

    아래 실습에서 직접 값을 바꿔 보세요. 겹침을 0으로 두면 문장이 어떻게 끊기는지, 크기를 키우면 조각 수가 어떻게 줄어드는지 감이 옵니다.

    🧪 청킹 시뮬레이터

    슬라이더로 청크 크기와 겹침을 조절하면 실제로 어떻게 잘리는지 보여 줍니다. 겹치는 영역은 앰버로 표시됩니다.

    생성된 조각 · 0개
    실무 팁

    아무 데서나 자르지 말고 의미 단위로 자르세요. 문단·문장·마크다운 헤더 단위로 나누는 RecursiveCharacterTextSplitter나, 문장 의미가 바뀌는 지점을 찾는 Semantic Chunking이 단순 글자 수 자르기보다 검색 품질이 좋습니다. 일반 문서는 보통 300~800 토큰 청크에 10~20% 겹침으로 시작합니다.

    06컴포넌트 · 인덱싱

    ③ 임베딩 — 의미를 좌표로

    컴퓨터는 "재택근무"와 "원격근무"가 비슷한 말이라는 걸 글자만 봐서는 모릅니다. 임베딩(Embedding)은 텍스트의 의미를 수백~수천 차원의 숫자 배열(벡터)로 바꿉니다. 핵심 성질은 단 하나입니다.

    임베딩의 단 하나의 약속

    의미가 비슷한 문장은 벡터 공간에서 가까이 놓인다. "강아지"와 "반려견"은 가깝고, "강아지"와 "주식 시장"은 멀다. 그래서 "질문과 가까운 문서 = 질문과 의미가 통하는 문서"를 거리 계산만으로 찾을 수 있습니다.

    이 "가까움"은 보통 코사인 유사도(cosine similarity)로 잽니다. 두 벡터가 가리키는 방향이 비슷할수록 1에 가깝고, 무관하면 0, 반대면 -1입니다. 다음 섹션의 실습에서 직접 눈으로 확인합니다.

    • 같은 모델로 통일: 문서를 임베딩한 모델과 질문을 임베딩하는 모델은 반드시 동일해야 합니다. 자[尺]가 다르면 거리 비교가 무의미합니다.
    • 모델 선택: 한국어가 많다면 다국어 지원 모델(bge-m3, multilingual-e5, OpenAI text-embedding-3 등)을 고릅니다. 차원이 클수록 표현력은 좋지만 저장·검색 비용이 늘어납니다.
    07컴포넌트 · 인덱싱

    ④ 벡터 데이터베이스 — 의미로 검색하는 창고

    수십만 개의 청크를 벡터로 바꿨다면, 이제 "질문 벡터와 가장 가까운 벡터 k개"를 빠르게 찾아야 합니다. 전부 일일이 비교하면 너무 느리므로, 벡터 DB는 근사 최근접 이웃(ANN, Approximate Nearest Neighbor) 알고리즘(HNSW, IVF 등)으로 "거의 정확하게, 아주 빠르게" 찾습니다.

    제품형태특징
    FAISS라이브러리메타가 만든 로컬 라이브러리. 가볍고 빠름. 프로토타입·온프레미스에 적합

    pgvectorPostgreSQL 확장기존 RDB에 벡터 검색을 더함. 운영 부담을 줄이고 싶을 때

    고를 때 보는 것

    장난감이 아니라면 ① 메타데이터 필터링(부서·날짜로 거르기), ② 하이브리드 검색(벡터+키워드), ③ 확장성·운영 편의성을 봅니다. 쿠버네티스 환경이라면 Helm/Operator로 배포·스케일이 쉬운지도 중요합니다.

    08컴포넌트 · 검색

    ⑤ 검색기 — 가까운 것을 골라내다

    여기서부터는 실시간 단계입니다. 검색기(Retriever)는 질문 벡터를 받아 벡터 DB에서 top-k(가장 가까운 k개) 청크를 꺼내옵니다. 아래 실습은 임베딩 공간을 2차원으로 단순화한 지도입니다. 점 하나가 문서 조각이고, ⭐는 질문입니다.

    🧪 벡터 유사도 시각화

    지도를 클릭하거나 아래 예시 질문을 눌러 ⭐ 질문 위치를 옮겨 보세요. 질문과 코사인 유사도가 높은 상위 3개 문서가 민트로 연결됩니다. 가까운 것이 곧 의미가 통하는 것입니다.

    ▹ top-k는 몇이 적당할까

    k가 너무 작으면 정답 문서를 놓치고(재현율↓), 너무 크면 잡음이 섞여 LLM이 헷갈립니다(정밀도↓). 보통 3~10에서 시작해 평가를 보며 조정합니다. 여기서 "일단 넉넉히 가져온 뒤 정밀하게 다시 거르는" 다음 단계가 등장합니다.

    09컴포넌트 · 검색

    ⑥ 재순위화 — 정밀하게 다시 줄 세우기

    벡터 검색은 빠르지만 거칠 수 있습니다. 질문과 문서를 따로따로 벡터로 바꿔 거리만 보기 때문입니다. 재순위화(Reranker)는 질문과 후보 문서를 함께 보는 정밀 모델(Cross-Encoder)로 "이 문서가 이 질문에 정말 답이 되는가"를 다시 채점해 순서를 바로잡습니다.

    전략은 "넓게 → 좁게"입니다. 벡터 검색으로 후보 20~50개를 빠르게 가져온 뒤, 무거운 재순위 모델로 상위 3~5개만 정밀하게 추립니다. 정확도와 비용을 동시에 잡는 표준 패턴입니다.

    🧪 재순위화 효과 보기

    질문: "환불 정책이 어떻게 되나요?"   왼쪽은 벡터 검색만 한 순서, 오른쪽은 재순위 모델이 다시 줄 세운 결과입니다. 버튼을 눌러 순서가 어떻게 바뀌는지 보세요.

    벡터 검색 결과 (top-5)

    10컴포넌트 · 생성

    ⑦ 프롬프트 조립 & 생성 — 근거로 답하기

    이제 검색해 온 청크들을 하나의 프롬프트로 조립(Augment)해 LLM에게 건넵니다. 이 단계의 프롬프트 한 장이 답변의 톤·정확성·출처 표기를 좌우합니다.

    # LLM에게 주는 지시: 근거 안에서만 답하고, 모르면 모른다고 말하라
"""
당신은 사내 정책 안내 도우미입니다.
반드시 아래 [근거]만 사용해 답하세요.
근거에 없는 내용은 추측하지 말고 "자료에서 찾을 수 없습니다"라고 답하세요.
답변 끝에 사용한 근거의 출처를 [출처] 형식으로 표기하세요.

[근거]
{retrieved_chunks}   # ← 검색·재순위로 추린 top-k 청크 삽입

[질문]
{user_question}
"""
    • 근거 한정(grounding): "근거 안에서만 답하라"는 지시가 환각을 크게 줄입니다.
    • 모르면 모른다: 검색이 실패했을 때 지어내지 않도록 명시적으로 허락해 줍니다.
    • 출처 인용: 메타데이터의 source/page를 함께 출력해 검증 가능성을 줍니다.
    11고급

    검색 품질을 끌어올리는 기법

    기본 RAG가 "그럭저럭"이라면, 아래 기법들이 "쓸 만함"으로 바꿔 줍니다. 대부분 검색 단계를 보강하는 데 집중합니다 — 앞서 말했듯 품질 문제의 대부분은 검색에서 옵니다.

    H하이브리드 검색 (Hybrid Search)

    의미 기반 벡터 검색과 키워드 기반 검색(BM25)을 합칩니다. 벡터는 "비슷한 의미"에, 키워드는 "정확한 고유명사·코드·약어"에 강합니다. 두 점수를 합쳐(RRF) 약점을 서로 메웁니다.

    Q쿼리 변환 (Query Rewriting / Multi-Query)

    사용자 질문이 모호하거나 검색어로 부적합할 때, LLM이 질문을 여러 개의 검색 친화적 표현으로 바꿔 각각 검색합니다. HyDE는 한 발 더 나가, "가상의 모범 답안"을 먼저 생성해 그것으로 검색합니다.

    P부모-자식 청킹 (Parent-Document)

    검색은 작고 정밀한 조각으로 하되, LLM에 넘길 때는 그 조각이 속한 더 큰 문단(부모)을 줍니다. "정확히 찾고, 넓게 읽힌다"는 두 마리 토끼를 잡습니다.

    A에이전틱 RAG (Agentic / Self-RAG)

    한 번 검색하고 끝내지 않고, LLM이 스스로 판단합니다. "검색이 필요한가? 결과가 충분한가? 다시 검색할까?"를 반복하며 답을 다듬습니다. 복잡한 다단계 질문에 강력합니다.

    12구현

    직접 만들어 보기

    같은 RAG를 세 가지 방식으로 보여 줍니다. 순수 파이썬(원리 이해용), LangChain(가장 널리 쓰임), LlamaIndex(데이터 인덱싱 특화). 탭을 눌러 비교하세요.

    # ── 원리 그대로: 임베딩 → 코사인 유사도 → 프롬프트 ──
from sentence_transformers import SentenceTransformer
import numpy as np

embedder = SentenceTransformer("BAAI/bge-m3")
docs = ["재택근무는 주 3일까지 가능합니다.",
        "연차는 입사 1년 후 15일 부여됩니다."]

# 1) 인덱싱: 문서를 미리 벡터화
doc_vecs = embedder.encode(docs, normalize_embeddings=True)

def retrieve(query, k=1):
    q = embedder.encode([query], normalize_embeddings=True)[0]
    sims = doc_vecs @ q              # 코사인 유사도(정규화했으므로 내적)
    top = np.argsort(sims)[::-1][:k]
    return [docs[i] for i in top]

# 2) 검색 + 생성
ctx = "\n".join(retrieve("원격근무 며칠 돼요?"))
prompt = f"근거:\n{ctx}\n\n질문: 원격근무 며칠 돼요?\n근거로만 답하세요."
# llm(prompt) → "재택근무는 주 3일까지 가능합니다."
    # ── LangChain: 컴포넌트를 체인으로 연결 ──
from langchain_community.vectorstores import Chroma
from langchain_huggingface import HuggingFaceEmbeddings
from langchain_text_splitters import RecursiveCharacterTextSplitter
from langchain_openai import ChatOpenAI
from langchain import hub

# 1) 청킹 → 임베딩 → 벡터DB 저장 (인덱싱)
splits = RecursiveCharacterTextSplitter(
    chunk_size=500, chunk_overlap=80).split_documents(documents)
emb = HuggingFaceEmbeddings(model_name="BAAI/bge-m3")
vectordb = Chroma.from_documents(splits, emb)

# 2) 검색기 + LLM 을 체인으로
retriever = vectordb.as_retriever(search_kwargs={"k": 4})
prompt = hub.pull("rlm/rag-prompt")
llm = ChatOpenAI(model="gpt-4o-mini")

def rag(question):
    ctx = retriever.invoke(question)
    msg = prompt.invoke({"context": ctx, "question": question})
    return llm.invoke(msg).content
    # ── LlamaIndex: 인덱싱이 가장 간결 ──
from llama_index.core import (VectorStoreIndex,
    SimpleDirectoryReader, Settings)
from llama_index.embeddings.huggingface import HuggingFaceEmbedding
from llama_index.llms.openai import OpenAI

Settings.embed_model = HuggingFaceEmbedding("BAAI/bge-m3")
Settings.llm = OpenAI(model="gpt-4o-mini")

# 1) 폴더 읽기 → 청킹·임베딩·인덱싱이 한 줄에
docs = SimpleDirectoryReader("./policies").load_data()
index = VectorStoreIndex.from_documents(docs)

# 2) 질의 엔진 = 검색 + 생성 통합
engine = index.as_query_engine(similarity_top_k=4)
answer = engine.query("원격근무 며칠 가능해?")
# print(answer)  → 출처(source_nodes)도 함께 제공
    어떤 걸 쓸까

    원리를 배우는 단계라면 순수 Python으로 한 번 손수 짜 보길 권합니다. 실무 프로토타입은 생태계가 넓은 LangChain, 문서 인덱싱·복잡한 데이터 소스가 중심이면 LlamaIndex가 편합니다. 셋 다 컴포넌트(청킹·임베딩·벡터DB·검색·생성)는 동일합니다 — 프레임워크가 아니라 구조를 이해하는 게 핵심입니다.

    13비교

    RAG vs 다른 접근법

    "LLM이 우리 지식을 알게 하려면" 길은 여러 갈래입니다. RAG가 항상 정답은 아니며, 상황에 따라 조합하기도 합니다. 핵심 후보들을 정면으로 비교합니다.

    기준RAG파인튜닝프롬프트 엔지니어링롱 컨텍스트
    방식외부 문서를 검색해 프롬프트에 주입모델 가중치를 추가 학습지시·예시를 정교하게 작성긴 문서를 통째로 입력
    지식 갱신문서만 바꾸면 즉시재학습 필요(느림)매번 수동 입력입력만 바꾸면 됨
    최신·실시간강함약함보통보통
    환각 억제근거 제공으로 강함스타일↑ 사실성은 별개제한적보통(길수록 누락)
    출처 인용가능불가불가제한적
    초기 비용·난이도중간(검색 파이프라인)높음(데이터·GPU)낮음낮음
    운영 비용검색 인프라 유지추론만(저렴)저렴토큰 폭증(비쌈)
    잘하는 일사실·문서 기반 Q&A, 사내 지식말투·형식·도메인 표현 학습간단한 형식·역할 지정한 문서를 깊게 분석
    현장의 결론

    "무엇(지식)"은 RAG, "어떻게(말투·형식)"는 파인튜닝. 사실을 정확히 답하게 하려면 RAG, 특정 도메인 어조나 출력 형식을 몸에 익히려면 파인튜닝이 맞습니다. 둘은 경쟁이 아니라 보완 관계라, 실무에선 파인튜닝한 모델 위에 RAG를 얹는 조합도 흔합니다. 롱 컨텍스트는 RAG를 "대체"한다는 말이 있지만, 비용과 "건초더미 속 바늘(중간 내용 누락)" 문제 때문에 대규모 지식엔 여전히 RAG가 유리합니다.

    14품질

    RAG를 어떻게 평가할까

    "좋아진 것 같다"는 느낌으로는 운영할 수 없습니다. RAG는 검색과 생성 두 축을 나눠서 측정합니다. RAGAS, TruLens 같은 도구가 LLM을 심판으로 써서 자동 채점해 줍니다.

    • Context Precision/Recall (검색 품질): 가져온 문서가 정말 관련 있는가, 필요한 문서를 빠뜨리지 않았는가.
    • Faithfulness (충실성): 답이 검색 근거에 실제로 기반하는가, 즉 환각이 없는가.
    • Answer Relevancy (답변 적합성): 답이 질문에 제대로 응답하는가.
    디버깅 순서

    답이 나쁘면 검색부터 의심하세요. 검색된 청크를 직접 눈으로 봅니다. 근거에 정답이 없으면 → 검색·청킹·임베딩 문제. 근거에 정답이 있는데 답이 틀리면 → 프롬프트·생성 문제. 이 분기만 잡아도 디버깅 시간이 절반으로 줍니다.

    15운영 · LLMOps

    운영 관점에서의 RAG

    프로토타입과 운영은 다릅니다. RAG를 서비스로 굴리려면 검색 인프라와 관측 체계를 함께 설계해야 합니다.

    • 인덱싱 파이프라인 자동화: 문서가 추가·수정될 때 청킹·임베딩·업서트가 자동으로 돌도록(예: Argo Workflows / Kubeflow 배치) 구성합니다.
    • 벡터 DB 운영: 쿠버네티스에 배포 시 영속 스토리지(PV/PVC), 백업, 재인덱싱 전략, 임베딩 모델 교체 시 전면 재계산 계획이 필요합니다.
    • 관측성(Observability): Langfuse 같은 도구로 질문→검색된 청크→프롬프트→답변→지연·비용을 추적합니다. "왜 이 답이 나왔는지" 추적 가능해야 개선이 됩니다.
    • 가드레일·보안: 민감 문서 접근 권한(메타데이터 필터로 부서별 격리), 프롬프트 인젝션 방어, PII 마스킹을 검색·생성 양쪽에 적용합니다.
    • 평가의 상시화: 골든 질문셋으로 배포 전 회귀 테스트(RAGAS)를 CI에 넣어 "조용한 품질 저하"를 잡습니다.
    16정리

    정리 & 이해도 점검

    RAG는 결국 "질문에 맞는 자료를 잘 찾아(검색) LLM에게 잘 읽히는(생성)" 구조입니다. 새 모델 능력이 아니라, 검색이라는 오래된 기술과 LLM을 결합한 시스템 설계입니다. 컴포넌트마다 작은 선택(청크 크기, 임베딩 모델, top-k, 재순위, 프롬프트)이 쌓여 전체 품질을 만듭니다. 아래 퀴즈로 점검해 보세요.

    Q1RAG가 "지식 컷오프 이후의 최신 정보"에 답할 수 있는 가장 정확한 이유는?
    Q2RAG 답변 품질이 나쁠 때 "가장 먼저" 확인해야 할 곳은?
    Q3"사실 지식"을 알려주는 것과 "특정 말투·출력 형식"을 익히게 하는 것의 올바른 짝은?
    Q4청킹에서 "겹침(overlap)"을 두는 주된 이유는?
