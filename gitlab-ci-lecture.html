Chapter One — Foundations

CI/CD란 무엇인가

프로그래밍을 처음 배울 때 우리는 손으로 코드를 짜고, 손으로 빌드 버튼을 누르고, 손으로 서버에 파일을 올렸습니다. 혼자라면 괜찮습니다. 하지만 여러 명이 매일 수십 번 코드를 커밋한다면 어떻게 될까요?

누군가는 빌드를 깜빡하고, 누군가는 테스트를 건너뛰고, 누군가는 잘못된 버전을 배포합니다. 이 모든 반복 작업을 자동화 파이프라인이 대신 해주는 것 — 그것이 바로 CI/CD입니다.

CI · CD를 한 줄로 정의하면

CI

Continuous Integration

지속적 통합. 여러 개발자의 코드 변경을 자주, 자동으로 합치고 곧바로 빌드·테스트해서 문제를 빨리 발견합니다.

CD

Continuous Delivery

지속적 전달. 통합된 코드를 언제든 배포 가능한 상태로 유지합니다. 배포 직전 단계까지 자동화하고, 마지막 승인만 사람이 합니다.

CD+

Continuous Deployment

지속적 배포. 승인 단계마저 자동입니다. 테스트만 통과하면 코드는 곧장 운영(production) 서버까지 올라갑니다.

💡 한 줄 요약

CI는 “코드가 깨졌는지 자동으로 확인”, CD는 “코드를 자동으로 배포 가능 상태로 유지(또는 배포까지 완료)”입니다. 둘은 보통 한 파이프라인에 연결되어 굴러갑니다.

CI/CD가 없으면 어떤 일이 벌어지나

  1. 월요일 — 개발자 A가 main 브랜치를 빌드해 본 적이 없다. 빌드가 깨진 채로 한 주가 시작된다.
  2. 수요일 — QA가 “이거 어떤 커밋의 빌드인가요?”라고 묻지만 아무도 답할 수 없다.
  3. 금요일 — 배포 담당자가 SSH로 서버에 접속해 손으로 파일을 옮긴다. 한 번 실수하면 주말이 사라진다.

CI/CD는 이 모든 시나리오를 “그냥 자동으로 돌아가게” 만들어줍니다. GitLab CI는 그중에서도 저장소·이슈·배포가 한 플랫폼에 통합된 CI/CD 도구입니다.

Chapter Two — Meet GitLab CI

GitLab CI 소개

GitLab CI는 GitLab에 내장된 CI/CD 엔진입니다. 저장소에 .gitlab-ci.yml 파일 하나만 추가하면, 커밋 → 자동 빌드 → 자동 테스트 → 자동 배포가 한 번에 굴러갑니다.

왜 GitLab CI인가

  • 저장소·CI·이슈·레지스트리가 하나 — Jenkins처럼 따로 설치할 도구가 없습니다. 코드와 파이프라인이 같은 UI 안에서 보입니다.
  • YAML 한 파일로 끝 — 파이프라인 전체가 .gitlab-ci.yml 하나에 선언됩니다. 코드와 함께 버전 관리됩니다(=Pipeline as Code).
  • Runner 분리 구조 — 작업은 “GitLab Runner”라는 별도 에이전트가 실행합니다. 회사 서버, GPU 머신, Kubernetes 클러스터 어디든 등록 가능합니다.
  • Self-hosted 옵션 — GitLab을 직접 설치해 사내에서 운영할 수 있어 보안에 민감한 조직(통신·금융·제조)에 적합합니다.

전체 그림: 커밋 한 번에 무슨 일이 벌어지나

아래는 가장 일반적인 흐름입니다. 머릿속에 이 그림을 박아두면 나머지가 쉽습니다.

— Push 한 번에 실행되는 파이프라인 —
build
compile
→
test
unit-test
lint
→
deploy
deploy-staging

위 버튼을 눌러 보세요. 실제 GitLab의 Pipelines 화면도 거의 똑같이 생겼습니다. 각 단계가 끝나야 다음 단계가 시작되고, 한 단계 안의 작업들은 병렬로 돕니다.

⚙️ Runner는 “일꾼”입니다

GitLab UI에서 “Pipeline 실행”을 눌러도 GitLab 본체가 직접 빌드를 돌리진 않습니다. 등록된 Runner(별도 서버, Docker 컨테이너, Kubernetes Pod 등)가 일을 받아 처리합니다. 회사 환경에서는 보통 사내 Runner를 등록해 사용합니다.

Chapter Three — Core Concepts

딱 4가지 개념만 잡으면 끝

GitLab CI의 모든 용어는 결국 다음 네 가지 개념의 조합입니다. 이 넷의 관계를 머릿속에 그려둔다면 어떤 복잡한 파이프라인도 분해할 수 있습니다.

01 · 가장 큰 단위

Pipeline

한 번의 커밋·MR로 실행되는 전체 흐름. 여러 Stage로 구성됩니다. “2024년 5월 27일 14:03의 파이프라인”처럼 시간 단위로 식별됩니다.

02 · 순서 단위

Stage

실행 순서를 정하는 그룹. 같은 Stage 안의 작업은 동시에 돌고, Stage 간에는 앞 단계가 끝나야 다음이 시작됩니다. 예: build → test → deploy.

03 · 실행 단위

Job

실제로 명령어를 실행하는 최소 단위. 어떤 Stage에 속할지, 어떤 Docker 이미지 위에서 돌지, 어떤 셸 스크립트를 돌릴지를 정합니다.

04 · 일꾼

Runner

Job을 실제로 실행해 주는 에이전트. 리눅스 VM, Docker, Kubernetes 등 어디든 설치 가능. 태그(tag)로 “이 Job은 GPU 노드에서” 같은 지정도 됩니다.

관계도

관계 트리 CONCEPT 
# 위에서 아래로 포함 관계
Pipeline          # 한 번의 커밋 = 한 개의 Pipeline
 └── Stage        # 순서대로 실행되는 그룹
      └── Job     # 실제 작업 단위 (스크립트 실행)
            └── Runner가 받아서 실행
Runner는 누가 관리하나요? (Shared vs Specific)

GitLab.com을 쓰면 Shared Runner(GitLab이 제공)를 무료/유료 한도 내에서 쓸 수 있습니다. 사내 GitLab을 쓰는 환경이라면 보통 Specific Runner(우리 회사가 직접 등록한 머신)를 씁니다. ML/LLM 파이프라인에서 GPU 노드나 특정 아키텍처(ARM64, x86_64)가 필요하다면 Runner에 태그를 붙여서 그 머신에서만 실행되게 지정합니다.

Job 한 개의 수명 주기 (Pending → Running → Success/Failed)

Job은 생성되면 pending(Runner 배정 대기) 상태로 시작합니다. Runner가 잡아채면 running으로 바뀌고, 스크립트가 0으로 종료(=성공)하면 success, 그렇지 않으면 failed가 됩니다. allow_failure: true를 주면 실패해도 파이프라인은 멈추지 않습니다.

Chapter Four — Writing the YAML

.gitlab-ci.yml 파헤치기

GitLab CI의 모든 설정은 저장소 루트의 .gitlab-ci.yml 한 파일에 들어갑니다. YAML이라는 형식으로 작성하는데, 처음 보면 어렵지만 들여쓰기 + 키:값 구조만 익히면 끝입니다.

가장 단순한 예시 — Hello, CI!

.gitlab-ci.yml YAML 
hello-job:                          # Job 이름
  script:                              # 실행할 셸 명령어
    - echo "안녕, 첫 번째 CI!"
    - date

이게 끝입니다. 커밋하면 GitLab이 Runner에 “이 명령어들을 실행하라”고 명령합니다. 결과는 GitLab의 Pipelines → Jobs 화면에서 콘솔 로그로 확인할 수 있습니다.

현실적인 예시 — 빌드 → 테스트 → 배포

.gitlab-ci.yml YAML 
stages:                              # 전체 순서를 먼저 선언
  - build
  - test
  - deploy

image: python:3.11-slim          # 기본 실행 환경(Docker 이미지)

build-app:
  stage: build
  script:
    - pip install -r requirements.txt
    - python setup.py sdist

unit-test:
  stage: test
  script:
    - pip install -r requirements.txt
    - pytest tests/

deploy-staging:
  stage: deploy
  script:
    - echo "staging 서버에 업로드…"
    - ./deploy.sh staging
  only:
    - main                          # main 브랜치일 때만 실행

위 코드만 봐도 4가지 개념이 모두 보입니다:

  • stages로 Stage 순서를 정의했고,
  • build-app, unit-test, deploy-staging이 각각 Job이며,
  • 각 Job은 stage 키워드로 어느 단계에 속할지 명시하고,
  • 모두 합쳐 하나의 Pipeline이 됩니다.

자주 쓰는 핵심 키워드

키워드역할예시
stages파이프라인 전체 단계 순서- build / - test
imageJob 실행용 Docker 이미지node:20
script실행할 셸 명령어 (필수)- npm test
before_scriptscript 전에 실행할 준비 명령- npm install
after_scriptscript 후 실행 (성공/실패 무관)- ./cleanup.sh
variables환경 변수 정의ENV: prod
tags특정 Runner 지정- gpu
only / except(구식) 실행 조건- main
rules(권장) 실행 조건if: $CI_COMMIT_BRANCH == "main"
artifactsJob 결과물 보존·전달paths: [dist/]
cache의존성 캐싱paths: [node_modules/]
needs특정 Job만 끝나면 시작 (DAG)- build-app
⚠️ YAML 작성 시 흔한 실수

YAML은 들여쓰기로 구조를 표현합니다. 탭(Tab)을 쓰지 말고 스페이스 2칸으로 통일하세요. 한 파일 안에서 탭과 스페이스를 섞으면 파싱 에러가 나는데, 에러 메시지가 영문이라 초보자가 가장 많이 막히는 지점입니다.

Chapter Five — Data Flow

변수·아티팩트·캐시

Job들 사이에서 데이터를 어떻게 주고받을 것인가는 모든 CI/CD에서 마주치는 문제입니다. GitLab CI는 세 가지 도구로 이 문제를 풀어줍니다.

① Variables — 환경 변수

API 키, 환경 이름, 빌드 옵션 등을 변수로 빼두면 코드와 설정을 깔끔하게 분리할 수 있습니다.

.gitlab-ci.yml YAML 
variables:                            # 전역 변수
  APP_ENV: "production"
  NODE_VERSION: "20"

build-app:
  image: node:$NODE_VERSION
  variables:                          # Job 단위 변수
    LOG_LEVEL: "debug"
  script:
    - echo "$APP_ENV / $LOG_LEVEL"     # production / debug

비밀번호·토큰처럼 노출되면 안 되는 값은 GitLab UI의 Settings → CI/CD → Variables에 등록합니다. 코드에는 키만 적고, 실제 값은 GitLab이 안전하게 주입합니다.

🔐 미리 정의된 변수들

GitLab은 수십 개의 변수를 자동으로 채워줍니다. $CI_COMMIT_BRANCH(현재 브랜치 이름), $CI_PROJECT_NAME(저장소 이름), $CI_PIPELINE_ID(파이프라인 번호), $CI_COMMIT_SHORT_SHA(커밋 해시) 같은 것들입니다. 이걸 알면 스크립트의 90%가 더 짧아집니다.

② Artifacts — Job 결과물

예를 들어 build Job이 만든 dist/ 폴더를 deploy Job이 써야 한다면? 각 Job은 다른 컨테이너에서 돌기 때문에 그냥은 안 됩니다. artifacts로 다음 Job에 전달합니다.

.gitlab-ci.yml YAML 
build:
  stage: build
  script:
    - npm run build                    # dist/ 생성
  artifacts:
    paths:
      - dist/                          # 이 폴더를 보존·전달
    expire_in: 1 week                # 일주일 후 자동 삭제

deploy:
  stage: deploy
  script:
    - ls dist/                         # build가 만든 결과물 사용 가능
    - ./upload.sh dist/

Artifacts는 GitLab 서버에 저장되어 웹 UI에서도 다운로드할 수 있습니다. 테스트 리포트, 빌드 산출물, 로그 파일을 보존하기에 좋습니다.

③ Cache — 다음 파이프라인에서도 재사용

node_modules/나 pip 패키지를 매 빌드마다 다운로드하면 비효율입니다. cache는 여러 파이프라인에 걸쳐 폴더를 캐싱해줍니다.

.gitlab-ci.yml YAML 
cache:
  key: "$CI_COMMIT_REF_SLUG"          # 브랜치별 캐시
  paths:
    - node_modules/
    - .npm/

install-and-test:
  script:
    - npm ci --cache .npm --prefer-offline
    - npm test
🆚 Artifacts vs Cache

Artifacts는 “이 파이프라인의 결과물, 다음 Job·사용자에게 전달”입니다.
Cache는 “시간 절약용 임시 저장, 사라져도 빌드는 정상 동작”입니다.
설치한 의존성 → cache / 빌드 산출물·테스트 리포트 → artifacts로 기억하세요.

Chapter Six — Flow Control

언제·어떻게 실행할지 정밀 제어

실무 파이프라인은 “항상 모든 Job을 다 돌리지” 않습니다. “main에 머지될 때만 배포”, “Dockerfile이 바뀔 때만 이미지 빌드”, “수동 승인을 받아야 운영 배포” 같은 조건이 필요합니다. GitLab CI는 rules와 needs로 이를 표현합니다.

rules — 조건부 실행의 표준

예전엔 only/except를 썼지만 지금은 rules가 표준입니다. 더 강력하고 직관적입니다.

.gitlab-ci.yml YAML 
deploy-prod:
  stage: deploy
  script:
    - ./deploy.sh production
  rules:
    # 1) 태그를 단 커밋이면 실행하고 수동 승인 받기
    - if: '$CI_COMMIT_TAG =~ /^v\d+\.\d+\.\d+$/'
      when: manual
    # 2) main 브랜치 + Dockerfile이 바뀐 경우에만
    - if: '$CI_COMMIT_BRANCH == "main"'
      changes:
        - Dockerfile
        - src/**/*
    # 3) 위 조건에 다 안 걸리면 이 Job은 스킵
    - when: never

위 코드는 세 가지 케이스를 한 번에 처리합니다. 규칙은 위에서 아래로 평가되며, 첫 번째로 매칭되는 조건이 적용됩니다.

needs — Stage를 건너뛰는 DAG

기본적으로 Stage는 순차 실행입니다. 하지만 “테스트가 끝나길 기다리지 말고, 빌드만 끝나면 바로 배포 시작!” 같은 최적화가 필요할 때 needs를 씁니다.

.gitlab-ci.yml YAML 
stages: [build, test, deploy]

build-frontend:
  stage: build
  script: [npm run build]

build-backend:
  stage: build
  script: [./gradlew build]

deploy-frontend:
  stage: deploy
  needs: [build-frontend]    # backend가 늦어도 frontend는 먼저 배포
  script: [./deploy-fe.sh]

이렇게 짜면 GitLab은 파이프라인을 DAG(Directed Acyclic Graph)로 그려서 보여줍니다. Stage라는 줄세우기를 깨고 진짜 의존성만 따라 병렬화하는 거죠. 큰 모노레포에서 빌드 시간을 절반 이하로 줄이는 핵심 기법입니다.

parallel — 같은 Job을 여러 개로

테스트 분할, 매트릭스 빌드(여러 OS·Python 버전 동시 테스트)에 유용합니다.

.gitlab-ci.yml YAML 
test:
  stage: test
  parallel:
    matrix:
      - PYTHON: ["3.10", "3.11", "3.12"]
        OS: ["ubuntu", "alpine"]
  image: $OS:$PYTHON
  script:
    - pytest tests/

이 한 블록이 3 × 2 = 6개의 Job으로 펼쳐져 동시에 실행됩니다.

Chapter Seven — Side by Side

GitHub Actions와 비교

GitHub Actions는 가장 많이 비교되는 경쟁 도구입니다. 둘은 철학적으로는 비슷하고 문법은 완전히 다른 사촌 같은 관계입니다. 같은 일을 양쪽 문법으로 직접 보면 가장 빠르게 이해할 수 있습니다.

같은 파이프라인, 두 가지 문법

“Node 앱을 설치 → 테스트 → main 브랜치라면 배포”라는 동일한 시나리오를 양쪽으로 작성해 봤습니다.

.gitlab-ci.yml YAML · GITLAB 
stages: [install, test, deploy]
image: node:20

install:
  stage: install
  script:
    - npm ci
  cache:
    paths: [node_modules/]

test:
  stage: test
  script:
    - npm test

deploy:
  stage: deploy
  script:
    - ./deploy.sh
  rules:
    - if: '$CI_COMMIT_BRANCH == "main"'
.github/workflows/ci.yml YAML · GITHUB 
name: CI
on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: "20"
          cache: "npm"
      - run: npm ci
      - run: npm test

  deploy:
    needs: test
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: ./deploy.sh

위 두 탭을 번갈아 눌러보세요. 핵심 차이를 한눈에 알 수 있을 겁니다.

핵심 차이를 표로 정리하면

항목 GitLab CI GitHub Actions
설정 파일 저장소 루트 .gitlab-ci.yml 하나 .github/workflows/ 안에 여러 개 가능
실행 단위 Job · Script (셸 명령어 위주) Step · Action (재사용 액션 + 셸)
재사용 컴포넌트 include, extends, !reference Marketplace의 actions/*(생태계 강함)
실행 트리거 기본 push/MR · rules로 세분화 on: 키로 명시 (push, pull_request, schedule…)
실행 환경 Runner (Docker/Shell/K8s/SSH) — Self-host 자연스러움 Hosted Runners 또는 Self-hosted Runner
병렬·매트릭스 parallel: matrix strategy.matrix
의존성/DAG needs: needs:
변수·시크릿 Project/Group/Instance 단위 변수 Repo/Org/Environment Secrets
아티팩트 네이티브 artifacts: actions/upload-artifact
Self-host 친화도 ★★★★★ (사내 운영이 본업) ★★★☆☆ (가능하지만 클라우드 우선)
가격(공용 사용) 월 무료 CI 분 + 유료 플랜 월 무료 CI 분 + 유료 플랜

언제 어떤 걸 쓰면 좋을까

CASE A

사내 GitLab을 쓰고 있다

이미 답이 정해져 있습니다. GitLab CI가 같은 UI 안에 통합되어 있어 학습·운영·보안 모두 가장 깔끔합니다.

CASE B

오픈소스 프로젝트를 GitHub에 호스팅

GitHub Actions가 자연스럽습니다. Marketplace의 액션들로 빌드/배포/릴리스 자동화를 빠르게 조립할 수 있습니다.

CASE C

Self-hosted Runner가 중심

GPU 노드, 폐쇄망, 사내 K8s에서 돌려야 한다면 GitLab CI가 운영 부담이 더 적습니다. Runner 등록·태그·태크닉이 더 성숙합니다.

CASE D

마이크로서비스 다수 + 모노레포

둘 다 됩니다. GitLab은 include로 모듈화, GitHub는 reusable workflow + Marketplace로 모듈화합니다. 팀이 익숙한 쪽이 정답입니다.

Chapter Eight — Recipes

실전 응용 예제

지금까지 배운 개념을 조합한, 현장에서 그대로 갖다 써도 되는 4가지 레시피입니다.

레시피 1 · Docker 이미지 빌드 후 레지스트리에 푸시

.gitlab-ci.yml YAML 
stages: [build]

build-image:
  stage: build
  image: docker:24
  services:
    - docker:24-dind                  # Docker in Docker
  variables:
    IMAGE: "$CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA"
  before_script:
    - docker login -u "$CI_REGISTRY_USER" -p "$CI_REGISTRY_PASSWORD" $CI_REGISTRY
  script:
    - docker build -t $IMAGE .
    - docker push $IMAGE
  rules:
    - if: '$CI_COMMIT_BRANCH == "main"'

$CI_REGISTRY_* 변수들은 GitLab이 자동으로 채워주는 내장 Container Registry 자격증명입니다. 이 한 토막으로 이미지 빌드·푸시 자동화가 완성됩니다.

레시피 2 · 멀티 아키텍처 이미지 빌드 (ARM64 + x86_64)

로컬에 ARM 워크스테이션과 x86 워크스테이션이 섞여 있다면, 이미지도 양쪽 아키텍처로 빌드해야 합니다. docker buildx가 정답입니다.

.gitlab-ci.yml YAML 
build-multiarch:
  stage: build
  image: docker:24
  services: [docker:24-dind]
  before_script:
    - docker buildx create --use --name multi
    - docker login -u "$CI_REGISTRY_USER" -p "$CI_REGISTRY_PASSWORD" $CI_REGISTRY
  script:
    - |
      docker buildx build \
        --platform linux/amd64,linux/arm64 \
        -t $CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA \
        --push .

레시피 3 · Python ML 파이프라인 (학습 → 평가 → 모델 산출물 업로드)

.gitlab-ci.yml YAML 
stages: [prep, train, eval, publish]
image: python:3.11-slim

prep-data:
  stage: prep
  script:
    - pip install -r requirements.txt
    - python scripts/prepare_data.py --out data/processed/
  artifacts:
    paths: [data/processed/]

train:
  stage: train
  tags: [gpu]                       # GPU Runner에서만 실행
  needs: [prep-data]
  script:
    - pip install -r requirements.txt
    - python train.py --data data/processed/ --out model/
  artifacts:
    paths: [model/]
    expire_in: 30 days

evaluate:
  stage: eval
  needs: [train]
  script:
    - pip install -r requirements.txt
    - python evaluate.py --model model/ --report report.json
  artifacts:
    paths: [report.json]
    reports:
      metrics: report.json          # MR에 메트릭 자동 표시

publish-model:
  stage: publish
  needs: [evaluate]
  script:
    - ./scripts/upload_model.sh model/ s3://my-models/
  rules:
    - if: '$CI_COMMIT_BRANCH == "main"'
      when: manual

MLOps 파이프라인의 전형입니다. tags: [gpu]로 학습 단계만 GPU 노드에 떨어뜨리고, 마지막 publish는 수동 승인으로 둬서 의도치 않은 모델 배포를 막습니다.

레시피 4 · include로 파이프라인 모듈화

저장소 여러 개가 비슷한 파이프라인을 쓴다면 공통 부분을 별도 파일·저장소로 분리하고 include 하세요.

.gitlab-ci.yml (각 프로젝트) YAML 
include:
  - project: "devops/ci-templates"
    file: "/templates/python-app.yml"
    ref: "v1.2.0"                  # 버전 고정 권장

variables:
  APP_NAME: "my-api"                # 템플릿이 받는 변수만 정의

이렇게 하면 100개 저장소의 파이프라인을 한 곳에서 업데이트할 수 있습니다. 큰 조직에서 표준화의 핵심 기법입니다.

Chapter Nine — Best Practices

모범 사례와 흔한 함정

해야 할 일 ✅

  • 파이프라인을 빠르게 유지하세요. 10분 넘어가면 개발자가 안 기다립니다. cache·needs·매트릭스를 적극 활용하세요.
  • 실패는 빨리, 비싼 작업은 늦게. lint·unit test 같은 빠른 검증을 앞단에 배치하고, e2e·배포는 뒤로 미루세요.
  • 운영 배포는 수동 승인. when: manual로 휴먼 게이트를 두면 사고가 줄어듭니다.
  • 비밀 값은 코드가 아니라 변수로. GitLab Settings → CI/CD → Variables에 등록하고, “Masked” + “Protected” 옵션을 체크하세요.
  • 이미지를 고정하세요. node:20보다 node:20.11.1처럼 정확한 태그가 재현성에 유리합니다.
  • 로그를 깨끗하게. set -e를 추가하거나 명시적인 에러 처리를 넣어 어디서 깨졌는지 바로 알 수 있게 하세요.

하지 말아야 할 일 ❌

  • 한 Job에 너무 많은 일을 몰아넣지 마세요. 한 Job은 한 가지 책임만. 디버깅과 재실행이 훨씬 쉬워집니다.
  • 모든 브랜치에서 배포 Job을 돌리지 마세요. 비용 낭비이고 사고의 원천입니다. rules로 막으세요.
  • artifacts에 거대한 폴더(node_modules 등)를 넣지 마세요. cache가 맞는 자리입니다.
  • only/except는 새 코드에서 쓰지 마세요. rules로 통일하세요. GitLab도 only/except를 점진적으로 deprecate하고 있습니다.
  • 비밀번호를 echo로 출력하지 마세요. 로그에 그대로 남습니다. 디버깅용이라도 위험합니다.
🚨 가장 흔한 디버깅 포인트

“내 Job이 안 돌아요”의 80%는 다음 셋 중 하나입니다: (1) rules 조건이 안 맞아서 스킵됨, (2) Runner에 적합한 태그가 없어서 pending에 멈춤, (3) Docker 이미지 안에 필요한 도구가 없어서 명령어가 not found. UI의 “CI Lint”와 Job 로그 첫 화면을 먼저 확인하세요.
