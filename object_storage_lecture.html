Object
Storage의
모든 것.

클라우드 시대의 데이터 저장소, Object Storage는 단순한 파일 보관함이 아닙니다. Netflix의 동영상, Instagram의 사진, ChatGPT의 학습 데이터셋, 여러분의 백업 파일까지 — 현대 인터넷의 거의 모든 비정형 데이터가 이 위에서 살아갑니다.

"스토리지는 단지 디스크가 아니다. 데이터를 어떻게 다룰지에 대한 철학이다."

이 강의에서는 Object Storage가 무엇인지, Block Storage / File Storage와는 어떻게 다른지, AWS S3·Azure Blob·GCP Cloud Storage는 어떻게 다른지, 그리고 실제 MLOps·웹 서비스·백업 파이프라인에서 어떻게 활용되는지를 초보자도 이해할 수 있도록 단계별로 풀어봅니다.

Chapter 01

— 정의와 등장 배경Object Storage란 무엇인가

Object Storage(객체 스토리지)는 데이터를 "객체(Object)"라는 단위로 저장하는 방식의 스토리지입니다. 객체란 단순히 파일 그 자체 + 메타데이터 + 고유 식별자(ID)를 하나의 묶음으로 취급하는 데이터 단위를 의미합니다.

한 줄 정의

"파일을 폴더 트리가 아닌, 키-값(Key-Value) 형태로 평면(Flat)하게 저장하는, REST API로 접근하는 무한 확장 가능한 클라우드 저장소."

왜 등장했는가?

2000년대 중반, 인터넷의 데이터는 폭발적으로 증가하기 시작했습니다. 사진, 동영상, 로그, 백업 같은 비정형 데이터(unstructured data)가 전통적인 파일 시스템(NTFS, ext4, NFS)이 감당하기 힘들 정도로 늘어났죠. 기존 파일 시스템의 한계는 명확했습니다:

  • 📁 폴더 계층이 깊어질수록 탐색 속도가 급격히 떨어짐
  • 📁 단일 파일 시스템의 용량 상한(수 PB 수준)이 명확함
  • 📁 메타데이터(파일 속성)가 매우 제한적임 (생성일, 권한 등 몇 가지뿐)
  • 📁 여러 서버에 분산해서 쓸 때 락(Lock), 일관성 문제가 복잡함

그래서 2006년 AWS S3(Simple Storage Service)가 등장하면서 "객체 단위로, HTTP로, 거의 무한히 확장 가능한" 스토리지의 시대가 열렸습니다. 이후 Azure Blob Storage(2010), Google Cloud Storage(2010)가 차례로 등장하며 현대 클라우드의 핵심 인프라가 되었습니다.

객체(Object) 하나는 어떻게 생겼는가?

위 그림에서 보듯, 하나의 객체 = 데이터 + 메타데이터 + 고유 키로 구성됩니다. 특히 메타데이터가 풍부하다는 점이 핵심입니다. 기존 파일 시스템이 "이 파일은 4.2MB, 어제 만들어짐" 정도밖에 못 적는다면, Object Storage는 "이 이미지는 ML 학습용, 프로젝트 v3, 라벨링 완료, 검수자 김XX" 같은 풍부한 정보를 함께 저장할 수 있습니다.

Chapter 02

— 스토리지 3대장 비교Block vs File vs Object

클라우드에는 세 가지 주요 스토리지 패러다임이 있습니다. 각각은 "데이터를 어떻게 나누어 저장하느냐" 가 다릅니다. 이 차이를 이해하는 것이 클라우드 엔지니어 입문의 핵심입니다.

▸ TYPE 01
Block
Storage

데이터를 고정 크기 블록으로 쪼개서 저장. OS가 직접 다루는 raw 디스크에 가까움.

  • 접근: SCSI, iSCSI, NVMe (OS 레벨)
  • 속도: 매우 빠름
  • 용량: 보통 수 GB ~ 수십 TB
  • 예시: AWS EBS, Azure Disk, GCP PD
  • 용도: DB, OS 부트 디스크, VM
▸ TYPE 02
File
Storage

전통적인 계층적 파일 시스템. 디렉토리 트리로 파일을 관리. 여러 서버가 공유 가능.

  • 접근: NFS, SMB, CIFS (마운트)
  • 속도: 빠름
  • 용량: 수 TB ~ 수 PB
  • 예시: AWS EFS, Azure Files, GCP Filestore
  • 용도: 공유 워크스페이스, 미디어 작업
▸ TYPE 03
Object
Storage

데이터를 객체 단위 + Flat 구조로 저장. HTTP REST API로 접근. 거의 무한 확장.

  • 접근: REST API (HTTPS)
  • 속도: 중간 (지연 있음)
  • 용량: 사실상 무제한
  • 예시: AWS S3, Azure Blob, GCP GCS
  • 용도: 백업, 정적웹, ML 데이터, 미디어

비교표로 한눈에 정리

특성 Block File Object

▸ 비유로 이해하기

Block Storage는 빈 노트의 페이지(빠르고 자유롭지만 혼자만 씀).
File Storage는 사무실 공유 캐비닛(여러 명이 폴더로 나눠 씀).
Object Storage는 거대한 도서관의 무한대 책장 시스템(전 세계 어디서나 ISBN으로 책을 빌릴 수 있음).

그럼 Object Storage의 단점은?

Object Storage가 만능은 아닙니다. 몇 가지 명확한 제약이 있습니다:

  • ❌ 부분 수정 불가 — 객체의 일부만 바꿀 수 없음. 전체 재업로드 필요. → DB의 백엔드로 쓰면 안 됨
  • ❌ 레이턴시 — HTTP 기반이라 첫 바이트 도달까지 수십 ms. 트랜잭션 처리에 부적합
  • ❌ OS 마운트가 자연스럽지 않음 — s3fs, goofys 같은 우회 도구가 있지만 진짜 파일 시스템처럼 작동하지 않음
  • ❌ 강한 일관성은 비교적 최근에 도입 — S3는 2020년에야 strong read-after-write consistency 보장
Chapter 03

— 주요 컴포넌트핵심 구성 요소 8가지

Object Storage를 구성하는 핵심 요소들을 하나씩 살펴봅시다. 카드를 클릭하면 자세한 설명이 펼쳐집니다.

버킷은 객체들이 담기는 최상위 컨테이너입니다. 마치 하드디스크의 "드라이브"처럼, 모든 객체는 반드시 어떤 버킷 안에 속해야 합니다. Azure에서는 Container, GCP에서는 Bucket이라고 부릅니다.

주요 특징

  • 버킷 이름은 전 세계적으로 유일해야 함 (URL의 일부가 되기 때문)
  • 버킷마다 리전(Region)이 지정됨 → 데이터가 저장될 물리적 위치 결정
  • 버킷 단위로 접근 권한, 암호화, 버전 관리, 수명주기 등을 설정
  • 한 계정당 만들 수 있는 버킷 수는 제한 (AWS는 기본 100개)
URL 구조 
# AWS S3
https://my-ml-datasets.s3.ap-northeast-2.amazonaws.com/train/img_001.jpg
       ────────────       ──────────────
       버킷 이름           리전

# GCP
https://storage.googleapis.com/my-ml-datasets/train/img_001.jpg

# Azure
https://myaccount.blob.core.windows.net/mycontainer/train/img_001.jpg
       ──────────                            ───────────
       Storage Account                       Container

객체 키는 버킷 안에서 객체를 유일하게 식별하는 문자열입니다. 우리 눈에는 "폴더/파일.확장자" 처럼 보이지만, 실제로는 슬래시(/)를 포함한 하나의 긴 문자열일 뿐입니다. 즉, Object Storage에는 진짜 폴더가 없습니다!

"폴더처럼 보이지만 폴더가 아닌" 평면 구조

아래 예시에서 "datasets/2026/train/img.jpg"는 폴더 3개가 아니라 단일 키 하나입니다. UI에서 폴더처럼 보여주는 건 단지 슬래시를 구분자로 그룹화한 시각적 표현일 뿐입니다. 

# 좋은 키 설계 예시 (Hive 스타일 파티셔닝)
datasets/project=llm-v3/year=2026/month=05/data.parquet
logs/service=api/date=2026-05-21/server01.log
models/arch=qwen2.5/stage=trained/version=v3.2/model.safetensors

# 좋지 않은 키 설계
asdf123.jpg                           # 의미 없음
2026-05-21_김철규_최종_진짜최종_v2.docx  # 비ASCII, 공백, 너무 길음

키 설계 원칙

  • 의미 있는 계층 구조를 사용 (예: {project}/{year}/{type}/{filename})
  • 날짜는 YYYY-MM-DD 형식으로 (정렬에 유리)
  • 특수문자, 공백, 한글은 가급적 피하기
  • 키 prefix를 검색에 활용 (예: logs/2026-05-21/로 시작하는 모든 객체 listing)

메타데이터는 객체에 대한 "데이터에 대한 데이터"입니다. Object Storage가 강력한 이유 중 하나가 바로 이 풍부한 메타데이터입니다.

두 가지 종류

  • 시스템 메타데이터(System Metadata) — Content-Type, Content-Length, Last-Modified, ETag(해시) 등 자동 생성
  • 사용자 정의 메타데이터(User-defined Metadata) — x-amz-meta-*(AWS), x-ms-meta-*(Azure) 접두사로 자유롭게 추가
# Python: 객체 업로드 시 사용자 메타데이터 추가
import boto3
s3 = boto3.client('s3')
s3.put_object(
    Bucket='ml-datasets',
    Key='images/cat_001.jpg',
    Body=open('cat_001.jpg', 'rb'),
    Metadata={
        'project': 'image-classification-v3',
        'label': 'cat',
        'annotator': 'cklee',
        'reviewed': 'true'
    },
    ContentType='image/jpeg'
)
▸ ML 엔지니어 팁

학습 데이터셋의 라벨, 검수 상태, 데이터 소스 등을 메타데이터로 저장하면 별도 DB 없이도 풍부한 데이터 카탈로깅이 가능합니다.

버전 관리를 활성화하면, 동일한 키로 객체를 덮어쓰거나 삭제할 때 이전 버전이 자동으로 보존됩니다. 각 버전은 고유한 Version ID를 가지며, 언제든 복원할 수 있습니다.

유즈케이스

  • 실수로 파일을 덮어쓰거나 삭제했을 때 복구
  • 랜섬웨어로부터 데이터 보호 (Object Lock과 함께)
  • ML 모델의 버전 관리 (모델 파일을 시간 순으로 추적)
  • 규제 요구사항(GDPR, HIPAA 등)에 따른 데이터 이력 보존
▸ 주의

버전 관리를 켜면 모든 이전 버전도 스토리지 비용에 포함됩니다. Lifecycle Policy로 오래된 버전을 자동 정리하는 것이 일반적입니다.

객체의 나이에 따라 자동으로 등급(Storage Class) 이동 또는 삭제를 수행하는 규칙입니다. 비용 최적화의 핵심입니다.

AWS S3 Lifecycle JSON 
{
  "Rules": [{
    "ID": "archive-old-logs",
    "Status": "Enabled",
    "Filter": {"Prefix": "logs/"},
    "Transitions": [
      {"Days": 30,  "StorageClass": "STANDARD_IA"},
      {"Days": 90,  "StorageClass": "GLACIER"},
      {"Days": 365, "StorageClass": "DEEP_ARCHIVE"}
    ],
    "Expiration": {"Days": 2555}  // 7년 후 삭제
  }]
}

위 예시는 logs/ prefix를 가진 모든 로그를 30일 후 IA로, 90일 후 Glacier로, 1년 후 Deep Archive로 자동 이동시키고, 7년 후 영구 삭제합니다.

누가 어떤 객체를 읽고 쓸 수 있는지 통제하는 메커니즘입니다. 보통 세 가지 레이어가 있습니다:

1. IAM Policy (계정/사용자 단위)

AWS IAM, Azure RBAC, GCP IAM처럼 클라우드 사용자/역할에 권한 부여. 가장 권장되는 방식.

2. Bucket Policy (버킷 단위 정책)

버킷 자체에 JSON 정책을 부착. 특정 IP, VPC, 외부 계정 등에 대한 접근 통제.

3. ACL (Access Control List, 객체 단위)

과거에 많이 쓰던 레거시 방식. 현재는 IAM/Bucket Policy 사용을 권장. AWS는 신규 버킷에 ACL을 기본 비활성화.

4. Pre-signed URL (시간 제한 임시 링크)

비공개 객체를 일정 시간 동안만 접근 가능하게 만드는 서명된 URL. 사용자에게 임시 다운로드 링크 제공할 때 유용. 

# 10분간 유효한 다운로드 URL 생성
url = s3.generate_presigned_url(
    'get_object',
    Params={'Bucket': 'private-bucket', 'Key': 'report.pdf'},
    ExpiresIn=600  # seconds
)

데이터를 안전하게 저장하기 위한 암호화. 두 가지 관점이 있습니다:

  • 전송 중 암호화(In-transit) — HTTPS/TLS로 네트워크 구간 암호화
  • 저장 시 암호화(At-rest) — 디스크에 저장될 때 AES-256으로 암호화

키 관리 방식 3가지

  • SSE-S3 / Microsoft-managed key: CSP가 키 관리 (가장 간단, 기본)
  • SSE-KMS / Customer-managed key: KMS의 사용자 키 사용 (감사 로그 가능)
  • SSE-C / Customer-provided: 사용자가 직접 키를 제공 (가장 통제력 높음)

객체 업로드, 삭제 같은 이벤트가 발생했을 때 다른 서비스(Lambda, SQS, EventBridge 등)로 알림을 보내 자동화 파이프라인을 구성할 수 있습니다.

전형적인 MLOps 패턴

# 시나리오: 새 학습 데이터가 업로드되면 자동으로 전처리

s3://datasets/raw/*.csv  업로드
        │
        ▼ [S3 Event Notification]
   AWS Lambda 트리거
        │
        ▼
   전처리 작업 → s3://datasets/preprocessed/
        │
        ▼ [다시 S3 Event]
   Argo Workflow 트리거 → ML 학습 파이프라인 시작
Chapter 04

— 내부 동작 원리Object Storage는 어떻게 동작하는가

Object Storage가 어떻게 PB급 데이터를 저렴하게, 그것도 11 9's(99.999999999%) 내구성으로 저장하는지 그 마법의 비밀을 들여다봅시다.

1. PUT 요청의 여정

  1. 인증 & 권한 검증: 요청에 포함된 Signature(서명)와 IAM 정책 확인
  2. 메타데이터 등록: 분산 메타데이터 스토어에 객체 정보 기록 (DynamoDB, etcd 등 사용)
  3. 청킹(Chunking): 큰 객체를 작은 조각으로 분할
  4. Erasure Coding 적용: 데이터를 N개 조각 + M개 패리티로 나눔 (예: 10+4 → 10개 조각 중 4개까지 잃어도 복원 가능)
  5. 다중 AZ 분산 저장: 최소 3개 이상의 가용 영역에 분산 → 데이터센터 1개가 통째로 사라져도 데이터 살아남음
  6. ACK 응답: 충분한 조각이 안전하게 저장되면 클라이언트에 200 OK 반환
▸ Why 11 Nines?

AWS S3의 99.999999999% 내구성은 마케팅 숫자가 아닙니다. Erasure Coding + 다중 AZ 분산 + 지속적 무결성 검증(scrubbing) + 자동 복구의 조합으로 통계적으로 도출되는 값입니다. "1천만 개의 객체를 저장하면 1만 년에 1개 손실"이라는 의미입니다.

2. 일관성 모델(Consistency Model)

"방금 업로드한 객체를 바로 GET하면 받아질까?" 이게 일관성의 핵심 질문입니다. 현재 주요 CSP들은 모두 Strong Read-after-Write Consistency를 보장합니다:

  • ✅ 새 객체 PUT 직후 GET → 항상 성공
  • ✅ 객체 덮어쓰기 직후 GET → 새 버전 즉시 반환
  • ✅ DELETE 직후 GET → 404 반환

과거(2020년 이전) S3는 Eventually Consistent 모델이었지만, 지금은 강한 일관성이 표준입니다. 단, LIST 작업은 약간의 지연이 있을 수 있습니다.

AWS CLI — 기본 명령어

bash 
# 1. 버킷 생성 (서울 리전)
aws s3 mb s3://my-ml-datasets-cklee --region ap-northeast-2

# 2. 단일 파일 업로드
aws s3 cp ./train.csv s3://my-ml-datasets-cklee/data/train.csv

# 3. 폴더 전체 동기화 (rsync 같은 동작)
aws s3 sync ./local-dataset/ s3://my-ml-datasets-cklee/dataset/

# 4. 버킷 안의 객체 리스트 보기
aws s3 ls s3://my-ml-datasets-cklee/dataset/ --recursive

# 5. 다운로드
aws s3 cp s3://my-ml-datasets-cklee/data/train.csv ./

# 6. 객체 삭제
aws s3 rm s3://my-ml-datasets-cklee/data/old_file.csv

# 7. Storage Class 지정하여 업로드 (자주 안 볼 데이터)
aws s3 cp ./archive.tar.gz s3://my-ml-datasets-cklee/archive/ \
    --storage-class STANDARD_IA

# 8. 버킷 비우고 삭제
aws s3 rb s3://my-ml-datasets-cklee --force

Python (boto3) — ML 워크플로우 예제

python 
import boto3
from botocore.exceptions import ClientError

s3 = boto3.client('s3', region_name='ap-northeast-2')
BUCKET = 'my-ml-datasets-cklee'

# [1] 학습 데이터 업로드 (메타데이터 포함)
def upload_training_data(local_path, key, label):
    s3.upload_file(
        Filename=local_path,
        Bucket=BUCKET,
        Key=key,
        ExtraArgs={
            'Metadata': {
                'label': label,
                'project': 'llm-fine-tuning',
                'uploaded-by': 'cklee'
            },
            'StorageClass': 'STANDARD'
        }
    )
    print(f"✓ uploaded: s3://{BUCKET}/{key}")

# [2] 특정 prefix 아래 객체 모두 listing
def list_dataset(prefix):
    paginator = s3.get_paginator('list_objects_v2')
    for page in paginator.paginate(Bucket=BUCKET, Prefix=prefix):
        for obj in page.get('Contents', []):
            print(f"  {obj['Key']}  ({obj['Size']} B)")

# [3] Pre-signed URL 생성 (외부 사용자에게 임시 다운로드 링크 제공)
def generate_share_link(key, expires=3600):
    return s3.generate_presigned_url(
        'get_object',
        Params={'Bucket': BUCKET, 'Key': key},
        ExpiresIn=expires
    )

# [4] 학습 후 모델 가중치 업로드
upload_training_data(
    './model.safetensors',
    'models/qwen2.5-7b/v3.2/model.safetensors',
    label='production'
)

# [5] PyTorch DataLoader가 직접 S3에서 스트리밍 (s3fs 사용)
import s3fs
fs = s3fs.S3FileSystem()
with fs.open(f's3://{BUCKET}/dataset/train.parquet', 'rb') as f:
    df = pd.read_parquet(f)

Kubernetes 환경 — Pod에서 Object Storage 사용

Kubernetes에서는 보통 세 가지 방법이 있습니다:

  • ① ServiceAccount + IRSA (AWS) / Workload Identity (GCP): Pod이 IAM 역할을 직접 위임받음 (가장 권장)

  • ③ CSI Driver: S3를 PVC처럼 마운트 (s3-csi-driver)

# IRSA로 S3 접근 권한을 부여받은 Pod

  name: train-llm

        resources:

Chapter 09

— Best Practices현업에서 지켜야 할 원칙

1. 명명 규칙

버킷: {회사}-{프로젝트}-{환경}-{용도} 형태로 일관성. 예: kt-llm-prod-datasets
객체 키: Hive 파티셔닝 스타일로 type=value를 활용하면 Athena/Spark 쿼리에 유리.

2. 비용 최적화

- Lifecycle Policy를 반드시 설정 (30일/90일/365일 등)
- 접근 패턴을 모르면 Intelligent-Tiering / Autoclass 사용
- Egress(외부 전송) 비용이 의외로 크다 — CDN과 결합

3. 보안

- Public Access Block 기본 활성화
- IAM 최소 권한 원칙 (필요한 prefix만 허용)
- 저장 시 암호화 항상 ON (SSE-S3 또는 KMS)
- Access Logging 활성화로 감사 추적

4. 성능

- 큰 파일(100MB+)은 멀티파트 업로드 사용
- 동일 prefix에 초당 수천 요청 시 prefix 분산 (예: 해시 prefix)
- Transfer Acceleration / CloudFront로 글로벌 가속

5. 안정성

- 중요한 버킷은 Versioning + Object Lock 활성화
- Cross-Region Replication으로 재해 대비
- 정기적인 무결성 검사 (체크섬)

6. 관측 가능성

- CloudTrail / Azure Monitor / GCP Audit Logs
- S3 Storage Lens로 사용 패턴 분석
- 비용 알람 설정 (Budget, Cost Anomaly Detection)

안티패턴 — 이렇게는 하지 마세요

  • ❌ Object Storage를 DB처럼 사용 — 부분 업데이트 못 함, 트랜잭션 없음. DynamoDB/Postgres 등을 쓰세요.
  • ❌ Public 버킷 + 민감 데이터 — 매년 수많은 데이터 유출 사고의 원인.
  • ❌ 버킷 이름에 회사 비밀 정보 노출 — 버킷 이름은 글로벌 유니크 = 추측 가능.
  • ❌ Access Key를 코드/Git에 하드코딩 — IAM Role / IRSA / Workload Identity 사용.
  • ❌ Lifecycle 없이 무한 적재 — 6개월 뒤 청구서를 보고 후회하게 됨.

Question 01다음 중 Object Storage의 특징이 아닌 것은?
  • REST API를 통해 HTTP/HTTPS로 접근한다
  • 사용자 정의 메타데이터를 풍부하게 저장할 수 있다
  • 객체의 일부분만 in-place로 수정할 수 있다
  • 사실상 무제한 확장 가능하다
✓ 정답: 3번 — Object Storage는 객체 전체를 다시 업로드해야 합니다. 부분 수정이 가능한 것은 Block Storage의 특성입니다. 그래서 DB 백엔드로는 적합하지 않습니다.
Question 02"버킷 안에 폴더를 만든다"는 표현이 기술적으로 정확하지 않은 이유는?
  • 버킷은 폴더를 5개까지만 허용하기 때문
  • Object Storage는 Flat 구조이며, 폴더처럼 보이는 건 슬래시(/)로 그룹화한 키일 뿐이기 때문
  • 폴더는 별도로 만들어야 하기 때문
  • 폴더 안에는 객체를 5개 이상 못 넣기 때문
✓ 정답: 2번 — Object Storage는 평면(Flat) 네임스페이스입니다. logs/2026/05/file.log는 폴더 3개 + 파일이 아니라, 슬래시를 포함한 단일 키 문자열입니다. UI에서 폴더처럼 보여주는 건 시각적 편의일 뿐입니다.
Question 03다음 워크로드 중 Object Storage가 가장 부적합한 것은?
  • ML 학습용 이미지 데이터셋 저장
  • 7년 장기 보관해야 하는 거래 로그
  • 초당 수만 건의 OLTP 트랜잭션 처리 DB
  • 정적 웹사이트의 이미지/CSS/JS 호스팅
✓ 정답: 3번 — OLTP DB는 낮은 레이턴시(μs)와 트랜잭션, 부분 업데이트가 필수입니다. Object Storage는 수십~수백 ms 레이턴시에 객체 전체 재업로드가 필요하므로 부적합합니다. RDS, Aurora, DynamoDB 같은 DB 서비스를 사용하세요.
Question 04AWS S3, Azure Blob, GCP Cloud Storage에서 "객체"를 가리키는 용어가 모두 다릅니다. 올바른 매핑은?
  • S3: Object / Azure: Blob / GCP: Object
  • S3: Blob / Azure: Object / GCP: File
  • S3: File / Azure: Blob / GCP: Item
  • S3: Item / Azure: File / GCP: Blob
✓ 정답: 1번 — AWS와 GCP는 "Object", Azure는 "Blob"(Binary Large OBject의 줄임말)이라고 부릅니다. Azure는 컨테이너도 "Container", AWS/GCP는 "Bucket"이라고 부르는 차이도 기억해두세요.
Question 05Lifecycle Policy의 주된 목적은?
  • 버킷의 보안 강화
  • 접근 빈도가 낮아진 객체를 저렴한 등급으로 자동 이동시켜 비용 최적화
  • 객체 업로드 속도 향상
  • 버킷 이름 자동 변경
✓ 정답: 2번 — Lifecycle Policy는 객체의 나이에 따라 자동으로 Storage Class를 전환(예: 30일 후 IA, 90일 후 Glacier)하거나 영구 삭제하여 비용을 최적화하는 자동화 규칙입니다.
— 강의를 마치며

Object Storage는 처음엔 단순한 "클라우드 파일 보관함"처럼 보이지만, 그 안에는 분산 시스템의 정수, 데이터 엔지니어링의 핵심, 그리고 현대 ML/AI 인프라의 토대가 들어있습니다. 이 강의가 그 첫 걸음에 도움이 되었기를 바랍니다.
