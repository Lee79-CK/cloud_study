Lecture 01 · Foundations

LLMOps란 무엇인가

LLMOps는 대규모 언어 모델(LLM)을 운영 가능한 형태로 만드는 모든 일이다. 모델을 학습시키는 것은 시작에 불과하며, 진짜 어려운 일은 그것을 안정적으로, 예측 가능하게, 그리고 비용 효율적으로 서비스하는 데 있다.

LLMOps의 정의

LLMOps(Large Language Model Operations)는 LLM의 전체 라이프사이클 — 데이터 준비, 사전학습/파인튜닝, 평가, 배포, 모니터링, 개선 — 을 자동화·표준화·관찰가능하게 만드는 엔지니어링 분야입니다. MLOps의 하위 분야로 출발했지만, LLM 고유의 특성 때문에 별도의 학문으로 분화되었습니다.

한 줄 정의

LLMOps = MLOps + 프롬프트 엔지니어링 + RAG + 토큰 경제학 + GPU 인프라 운영

왜 LLMOps인가

전통적 ML 모델과 비교했을 때 LLM이 운영자에게 주는 새로운 도전들은 다음과 같습니다.

  • 크기와 비용 — 7B 모델만 해도 fp16 기준 14GB의 GPU 메모리를 소비합니다. 70B 모델은 단일 GPU로 서빙이 불가능합니다.
  • 비결정성 — 동일한 입력에 다른 출력이 나옵니다. 회귀 테스트, A/B 테스트, 평가 자체가 어려운 문제가 됩니다.
  • 프롬프트 = 코드 — 프롬프트의 작은 변화가 출력 품질을 좌우합니다. 프롬프트 버전 관리는 새로운 형태의 형상관리입니다.
  • 토큰 단위 과금 — 비용이 입출력 토큰 길이에 비례합니다. 컨텍스트 윈도우, 캐시 히트율, 모델 라우팅이 운영 KPI가 됩니다.
  • 환각(Hallucination) — 모델은 자신 있게 거짓말을 합니다. RAG, 가드레일, 사실성 평가가 필수입니다.
  • 지연시간 제약 — 챗봇 UX는 첫 토큰까지 200~500ms를 요구합니다. TTFT(Time-To-First-Token)가 새로운 SLO 지표입니다.
현장의 함정

"LLM API 한 줄 호출"은 데모에는 충분하지만, 사내 시스템에 통합되는 순간 버전 고정, 폴백 모델, 비용 가드, PII 마스킹, 프롬프트 인젝션 방어 등 10가지 이상의 운영 관심사가 한꺼번에 등장합니다. LLMOps는 이를 체계적으로 처리하기 위한 청사진입니다.

LLMOps의 5가지 활동

현장에서 LLMOps 엔지니어가 실제로 하는 일은 다음 다섯 가지로 요약됩니다.

활동 1
모델 서빙 (Serving)
vLLM, TGI, TensorRT-LLM 등으로 추론 엔드포인트를 운영하고, 오토스케일링·배치·캐싱을 관리합니다.
활동 2
파인튜닝 (Fine-tuning)
LoRA, QLoRA, DPO 등으로 도메인 특화 모델을 만들고, 학습 파이프라인을 자동화합니다.
활동 3
검색 증강 (RAG)
벡터 DB, 임베딩 서비스, 청킹 전략을 구축해 모델에 외부 지식을 연결합니다.
활동 4
평가 (Evaluation)
자동 평가 지표(LLM-as-Judge, BLEU, ROUGE), 휴먼 평가, 회귀 테스트 스위트를 운영합니다.
활동 5
관찰성 (Observability)
프롬프트 트레이싱, 토큰 사용량, 지연시간, 비용, 환각률을 측정하고 알람을 설정합니다.
활동 ⊕
거버넌스 (Governance)
EU AI Act, 사내 정책, PII/민감정보 처리, 가드레일, 감사 로그를 책임집니다.

Lecture 02 · Comparison

MLOps vs LLMOps

둘은 사촌이지만 형제는 아니다. MLOps에서 출발한 엔지니어가 LLMOps로 넘어올 때 놓치기 쉬운 관점의 차이를 짚어본다.

두 분야의 동시 비교

차원MLOpsLLMOps
모델 출처주로 자체 학습주로 사전학습 모델 + 파인튜닝/프롬프팅
모델 크기MB ~ 수 GB수 GB ~ 수 백 GB
학습 시간분 ~ 시간일 ~ 주 (사전학습) / 시간 (파인튜닝)
추론 단위요청당 ms 단위토큰당 ms, 응답당 초 단위
비용 모델인스턴스 시간토큰 수 + GPU 메모리 점유
출력 결정성결정적 (분류/회귀)비결정적 (생성)
평가 지표Accuracy, F1, AUCBLEU, ROUGE, LLM-as-Judge, Pass@K, Faithfulness
주요 아티팩트모델 파일, Feature Store모델 + 프롬프트 + 벡터 인덱스 + 평가셋
실패 모드예측 오류, 데이터 드리프트환각, 인젝션, 유해 출력, 비용 폭주
핵심 도구MLflow, Kubeflow, SeldonvLLM, LangChain, LlamaIndex, Langfuse, Ragas

관점의 전환

MLOps 마인드셋

모델은 결과물이다

  • 학습 → 검증 → 배포의 일방향 파이프라인
  • 데이터 품질이 결과를 결정
  • 모델 버전 = 가중치 해시
  • SLO = 정확도 + 지연시간
LLMOps 마인드셋

모델은 컴포넌트다

  • 모델·프롬프트·검색·도구가 결합된 시스템
  • 데이터 + 프롬프트 + 평가셋이 모두 자산
  • 모델 버전 = 모델 + 프롬프트 + 시스템 메시지
  • SLO = TTFT + 비용/요청 + 환각률 + 거부율

중복되는 영역

그럼에도 불구하고 두 분야는 많은 것을 공유합니다. Kubernetes, GPU 스케줄링, 파이프라인 자동화, 컨테이너 이미지 관리, 모니터링 스택, GitOps는 그대로 통용됩니다. KT 환경에서 이미 운영 중인 Kubeflow Training Operator, Argo Workflows, Volcano 같은 도구들은 LLMOps에서도 그대로 쓰입니다. 다만 그 위에 LLM 특화 레이어가 한 겹 더 올라간다고 이해하면 됩니다.

실무 조언

기존 MLOps 플랫폼을 가진 조직이라면 처음부터 새로 짜지 말고 기존 컴포넌트(Argo, MLflow, KServe) 위에 LLM 전용 레이어 (vLLM, Langfuse, 벡터 DB)를 얹는 점진적 진화 전략을 권장합니다.

Checkpoint 02

  • 모델 가중치 해시 하나로 충분하다
  • 모델 + 학습 데이터 버전

각 레이어 깊이 보기

Inference Engine: vLLM이 표준이 된 이유

2023년까지 LLM 추론은 Hugging Face Transformers + FastAPI 조합이 흔했지만, 2024년 이후 사실상의 표준은 vLLM이 되었습니다. 핵심 이유는 두 가지입니다.

  • PagedAttention — KV 캐시를 페이지 단위로 관리해 메모리 단편화를 제거. 동일 GPU에서 2~4배 더 많은 동시 요청 처리.
  • Continuous Batching — 정적 배치 대신 토큰 단위로 배치를 재구성. GPU 활용률 60% → 90%+ 상승.

Retrieval: RAG의 심장

RAG(Retrieval-Augmented Generation)는 LLM의 환각과 지식 한계를 해결하는 가장 보편적 패턴입니다. 파이프라인은 단순합니다: 문서 → 청킹 → 임베딩 → 벡터 인덱스 → 질의 시 검색 → 프롬프트에 주입. 그러나 운영에 들어가면 청크 크기, 오버랩, 임베딩 모델 선택, 리랭커 사용 여부, 인덱스 갱신 주기 등 수십 개의 튜닝 파라미터가 등장합니다.

Observability: LLM은 블랙박스다, 그러나 추적은 가능하다

전통 모니터링이 CPU·메모리·QPS를 봤다면, LLMOps는 거기에 토큰 사용량, TTFT, ITL(Inter-Token Latency), 캐시 히트율, 평가 점수, 프롬프트 변경 이력을 추가합니다. Langfuse·Phoenix 같은 도구는 프롬프트별 트레이스를 저장해 "어떤 프롬프트가 어떤 출력을 냈는지"를 기록합니다.

도구 선택 가이드

오픈소스 우선 스택을 구성한다면: vLLM(서빙) + Milvus 또는 Qdrant(벡터) + Langfuse(관찰성) + Ragas(평가) + Argo Workflows(오케스트레이션)의 조합이 KT 환경에서 가장 자연스럽게 안착합니다.

Lecture 04 · Reference Architecture

참조 아키텍처

실제 운영 환경에서 컴포넌트들이 어떻게 연결되는지, 데이터와 트래픽이 어떤 경로로 흐르는지 한 장의 그림으로 정리한다.

엔드투엔드 데이터 흐름

Reference Architecture · 12-Box View

두 가지 경로

① Serving Path (실시간 추론)

사용자 요청은 Gateway → Orchestrator → Inference Engine 순으로 흐릅니다. Orchestrator는 프롬프트를 조립하면서 필요하면 Vector DB와 Embedder를 호출해 관련 문서를 가져와 컨텍스트에 주입합니다(RAG). Inference Engine(vLLM)은 Model Registry에서 가져온 가중치로 토큰을 생성합니다.

② Offline / Ops Path (학습·평가·관찰)

새 모델은 Training Pipeline에서 학습되어 Registry에 등록되고, Evaluation을 통과한 뒤에야 서빙으로 승격됩니다. 모든 레이어의 텔레메트리는 Observability로 흘러 단일 대시보드에서 조회됩니다.

설계 원칙

"Serving Path는 stateless, Ops Path는 stateful로 설계하라." 서빙 경로는 어떤 노드에서든 동일하게 동작해야 하며, 상태(모델·인덱스·프롬프트)는 외부 저장소에 위임합니다. 이 원칙이 깨지면 카나리 배포·롤백·스케일아웃이 모두 어려워집니다.

노드 라벨링과 테인트

2-노드 환경에서 가장 먼저 해야 할 일은 각 노드의 정체성을 K8s에 알려주는 것입니다. 라벨은 "이 노드는 무엇이다"를, 테인트는 "이 노드에는 아무나 못 들어온다"를 의미합니다.

Bash · Node Labels & Taints
# DGX Spark 노드에 GPU 라벨과 테인트 부여
kubectl label node dgx-spark node-role=gpu workload=llm
kubectl taint node dgx-spark dedicated=gpu:NoSchedule

# ThinkStation은 컨트롤 플레인이지만 워크로드도 받게
kubectl label node thinkstation node-role=control-plane workload=ops
kubectl taint node thinkstation node-role.kubernetes.io/control-plane- # 테인트 제거

# 확인
kubectl get nodes -o custom-columns=\
NAME:.metadata.name,\
ARCH:.status.nodeInfo.architecture,\
ROLE:.metadata.labels.node-role,\
GPU:.status.allocatable.nvidia\\.com/gpu
중요 · 컨트롤 플레인에 워크로드 허용

노드가 두 대뿐이므로 컨트롤 플레인 테인트를 제거해 ThinkStation에서도 일반 워크로드(Argo, MLflow, Prometheus)가 돌도록 합니다. 운영 환경에서는 권장하지 않지만, 실습·소규모 클러스터에서는 표준 패턴입니다.

Lecture 06 · Heterogeneous Strategy

이종 아키텍처 전략

ARM64와 x86_64가 한 클러스터에 섞여 있을 때 가장 자주 깨지는 것은 이미지다. 이 문제를 어떻게 시스템적으로 해결할지 살펴본다.

왜 이종 클러스터가 어려운가

K8s 자체는 노드 아키텍처를 알지만, 컨테이너 이미지는 빌드 시점에 특정 아키텍처로 고정됩니다. python:3.11처럼 보이는 태그도 실제로는 linux/amd64 매니페스트만 있을 수 있고, 그 이미지를 ARM64 노드에서 풀하면 exec format error로 실패합니다.

3가지 흔한 실패 모드

  • Pull은 되지만 실행 실패 — 이미지 매니페스트가 단일 아키텍처. exec /usr/local/bin/python: exec format error.
  • 일부 노드에서만 Pending — nodeSelector가 없어 ARM64 노드에 amd64-only 이미지가 스케줄됨.
  • 성능 저하 — qemu 에뮬레이션으로는 돌지만 GPU 성능을 절반 이하로 사용.

해결책 1 · Multi-Arch 이미지 빌드

Docker Buildx를 사용하면 단일 명령으로 amd64와 arm64 이미지를 동시에 빌드해 하나의 매니페스트로 묶을 수 있습니다. 이후 K8s는 노드 아키텍처에 맞는 변형을 자동으로 선택합니다. 

# buildx 빌더 생성 (한 번만)
docker buildx create --name multi --driver docker-container --use
docker buildx inspect --bootstrap

# 멀티아키 빌드 + 푸시
docker buildx build \
  --platform linux/amd64,linux/arm64 \
  -t registry.kt.local/llmops/preprocess:v1.2 \
  --push .
# 멀티아키를 의식한 Dockerfile
FROM --platform=$BUILDPLATFORM python:3.11-slim AS base
ARG TARGETARCH
RUN echo "Building for $TARGETARCH"

# 아키텍처별 분기가 필요한 경우
RUN if [ "$TARGETARCH" = "arm64" ]; then \
        pip install torch --index-url https://download.pytorch.org/whl/cu124; \
    else \
        pip install torch; \
    fi

COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . /app
WORKDIR /app
CMD ["python", "main.py"]
# GitHub Actions 예시
name: build-multiarch
on: [push]
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: docker/setup-qemu-action@v3

          platforms: linux/amd64,linux/arm64

해결책 2 · 명시적 nodeSelector

멀티아키 이미지가 없거나 GPU 워크로드처럼 특정 노드에만 가야 한다면 nodeSelector와 tolerations로 위치를 강제합니다.

해결책 3 · Affinity로 더 풍부한 표현

조건이 복잡해지면 nodeAffinity가 필요합니다. "ARM64 GPU 노드 우선, 없으면 x86_64 GPU 노드"와 같은 부드러운 선호를 표현할 수 있습니다.

YAML · Node Affinity
spec:
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
          - matchExpressions:
              - key: nvidia.com/gpu.present
                operator: In
                values: ["true"]
      preferredDuringSchedulingIgnoredDuringExecution:
        - weight: 100
          preference:
            matchExpressions:
              - key: kubernetes.io/arch
                operator: In
                values: ["arm64"]
현장 체크리스트

이종 클러스터를 운영할 때 매번 점검할 4가지: ① 이미지 매니페스트가 멀티아키인가 (docker manifest inspect), ② 모든 GPU Pod에 nodeSelector가 있는가, ③ Helm 차트의 image.repository가 멀티아키 레지스트리를 가리키는가, ④ Init/Sidecar 컨테이너도 ARM64를 지원하는가.

Checkpoint 06
Q. ARM64 노드에 amd64-only 이미지를 스케줄했을 때 가장 흔한 증상은?
  • Pod가 ImagePullBackOff로 멈춘다
  • Pull은 성공하지만 컨테이너가 exec format error로 즉시 종료된다
  • K8s가 자동으로 호환 이미지를 찾아준다
  • 네트워크 정책이 차단한다
정답: ② — 매니페스트에 amd64만 있어도 pull은 됩니다(다른 아키 매니페스트가 없으면 그것을 받음). 실행 단계에서 바이너리 형식 불일치로 exec format error가 발생하는 게 전형적인 증상입니다.

Lab 2 · Fine-tuning

Lab 2 · LoRA 파인튜닝 파이프라인

Argo Workflows로 데이터 준비 → 학습 → 평가 → 모델 등록을 한 번에 수행하는 파이프라인을 만든다. 학습은 DGX Spark에서, 메타데이터는 ThinkStation의 MLflow에서 관리한다.

왜 LoRA인가

Full Fine-tuning은 7B 모델 기준으로 약 84GB의 GPU 메모리가 필요합니다(가중치 14GB + 옵티마이저 28GB + 그라디언트 14GB + 활성화 ~28GB). LoRA는 모델 가중치를 동결하고 작은 어댑터(보통 0.1~1%)만 학습하므로 같은 모델을 16~24GB GPU 메모리로 파인튜닝 가능하게 만들어줍니다.

방식메모리(7B)학습 속도품질
Full FT~84GB1×최고
LoRA~24GB1.5~2×Full의 95~99%
QLoRA (4bit)~10GB1×Full의 90~98%

apiVersion: argoproj.io/v1alpha1
kind: Workflow
metadata:
  generateName: qwen-lora-
spec:
  entrypoint: pipeline
  arguments:
    parameters:
      - { name: base-model, value: "Qwen/Qwen2.5-7B-Instruct" }
      - { name: dataset,    value: "s3://datasets/customer-support-ko" }
      - { name: epochs,     value: "3" }

  templates:
    - name: pipeline
      dag:
        tasks:
          - { name: prep,     template: prep-data }
          - { name: train,    template: train-lora,  dependencies: [prep] }
          - { name: merge,    template: merge-adapter, dependencies: [train] }
          - { name: eval,     template: evaluate,    dependencies: [merge] }
          - { name: register, template: mlflow-register, dependencies: [eval],
              when: "{{tasks.eval.outputs.parameters.score}} >= 0.7" }

    # --- 데이터 전처리 (ThinkStation) ---
    - name: prep-data
      nodeSelector: { kubernetes.io/arch: amd64 }
      container:
        image: registry.kt.local/llmops/dataprep:v1.2

        args: [--src, "{{workflow.parameters.dataset}}", --dst, /out]

    # --- LoRA 학습 (DGX Spark) ---
    - name: train-lora
      nodeSelector: { kubernetes.io/arch: arm64, node-role: gpu }
      tolerations:
        - { key: dedicated, operator: Equal, value: gpu, effect: NoSchedule }
      container:
        image: registry.kt.local/llmops/trainer:v1.2
        command: [python, train_lora.py]
        args:
          - --base={{workflow.parameters.base-model}}
          - --data=/data
          - --epochs={{workflow.parameters.epochs}}
          - --rank=16
          - --alpha=32
          - --output=/adapters
        resources:
          limits: { nvidia.com/gpu: "1" }

    # --- 어댑터 머지 + 평가 + 등록은 동일한 패턴 ---

학습 스크립트의 핵심

Hugging Face PEFT 라이브러리를 쓰면 LoRA는 놀랄 만큼 단순해집니다. 실제 학습 코드의 본질은 다음 30줄에 모두 담겨 있습니다.

Python · train_lora.py
from peft import LoraConfig, get_peft_model
from transformers import AutoModelForCausalLM, AutoTokenizer, TrainingArguments, Trainer
from datasets import load_from_disk

base = "Qwen/Qwen2.5-7B-Instruct"
tok = AutoTokenizer.from_pretrained(base)
model = AutoModelForCausalLM.from_pretrained(base, torch_dtype="bfloat16", device_map="auto")

lora = LoraConfig(
    r=16, lora_alpha=32, lora_dropout=0.05,
    target_modules=["q_proj", "k_proj", "v_proj", "o_proj"],
    bias="none", task_type="CAUSAL_LM",
)
model = get_peft_model(model, lora)
model.print_trainable_parameters()  # typical: 0.3% trainable

ds = load_from_disk("/data")
args = TrainingArguments(
    output_dir="/adapters",
    per_device_train_batch_size=2,
    gradient_accumulation_steps=8,
    num_train_epochs=3,
    learning_rate=2e-4,
    bf16=True,
    logging_steps=10,
    save_strategy="epoch",
    report_to=["mlflow"],     # MLflow에 자동 로깅
)
Trainer(model=model, tokenizer=tok, args=args, train_dataset=ds["train"]).train()
model.save_pretrained("/adapters/final")
DGX Spark 활용 팁

GB10의 128GB 통합 메모리는 LoRA 학습에서 진가를 발휘합니다. CPU↔GPU 데이터 복사가 사실상 무료이므로 bitsandbytes 4bit 양자화를 쓰면 13B~14B 모델까지도 무리 없이 파인튜닝 가능합니다. 일반 24GB GPU라면 LoRA여도 7B가 한계입니다.

connections.connect(host="milvus.vector", port=19530)

    FieldSchema("text", DataType.VARCHAR, max_length=2000),
    FieldSchema("source", DataType.VARCHAR, max_length=200),
]
coll = Collection("kt_docs", CollectionSchema(fields))
coll.create_index("vector", {"index_type": "HNSW", "metric_type": "COSINE",
                              "params": {"M": 16, "efConstruction": 200}})

# 임베딩 + 적재
def embed(texts):
    r = requests.post("http://tei-bge.rag:8080/embed", json={"inputs": texts})
    return r.json()

for path in glob.glob("/docs/*.md"):
    chunks = chunk_text(open(path).read(), size=400, overlap=50)

prompt = ChatPromptTemplate.from_template("""
다음 컨텍스트를 활용해 질문에 답하세요. 컨텍스트에 없는 내용은 모른다고 답하세요.
출처를 인용하세요.

[컨텍스트]
{context}

[질문]
{question}
""")

def rag(question):
    qvec = embed([question])[0]
    hits = coll.search([qvec], "vector", {"metric_type": "COSINE"},
                       limit=5, output_fields=["text", "source"])
    ctx = "\n---\n".join(f"[{h.entity.source}] {h.entity.text}" for h in hits[0])
    return llm.invoke(prompt.format(context=ctx, question=question)).content
RAG 운영의 함정

RAG 품질 = 검색 품질입니다. 모델을 아무리 큰 것으로 바꿔도 잘못된 청크가 컨텍스트에 들어오면 답이 망가집니다. 우선순위는 ① 청킹 전략 → ② 임베딩 모델 선택 → ③ 리랭커 도입 → ④ 모델 업그레이드입니다. ④번부터 시도하는 흔한 실수를 피하세요.

Lecture 11 · Best Practices

운영 모범 사례

실습을 넘어 실제 운영 환경에서 LLMOps를 안정적으로 굴리려면 무엇을 더 챙겨야 하는가. 체크리스트로 정리한다.

비용 최적화

  • 모델 라우팅 — 쉬운 질문은 작은 모델(1B~3B), 어려운 질문만 큰 모델로. 평균 비용 60~80% 절감 가능.
  • 프롬프트 캐시 — 시스템 프롬프트가 길면 prefix caching으로 TTFT와 비용을 동시에 줄임.
  • 구조화 출력 — JSON 스키마 강제로 후처리 비용과 재시도 감소.
  • 스트리밍 — 사용자에게 첫 토큰을 빨리 보여줘 체감 지연을 절반으로.

신뢰성

  • 다중 모델 폴백 — 주 모델 실패 시 보조 모델로 자동 전환.
  • 회로 차단기 — 외부 API 호출 시 연속 실패 임계값 도달 시 일시 차단.
  • 타임아웃 명시 — 무한 생성을 막는 max_tokens, 요청 단위 timeout.
  • 모델 워밍업 — 배포 직후 더미 요청으로 KV 캐시·CUDA 그래프 미리 빌드.

보안

  • 프롬프트 인젝션 방어 — 사용자 입력과 시스템 지시를 명확히 분리. NeMo Guardrails로 정책 강제.
  • PII 마스킹 — 입력 단계에서 주민번호·이메일·전화번호 자동 식별·마스킹.
  • 출력 필터 — 유해 콘텐츠 분류기로 응답 사후 검사.
  • 감사 로그 — 누가, 언제, 어떤 프롬프트로, 어떤 출력을 받았는지 90일 이상 보관.

CI/CD for LLM

프롬프트 변경도 코드 변경입니다. PR 단위로 평가셋을 돌려 회귀를 막아야 합니다.

YAML · GitHub Actions · Prompt Eval
name: prompt-regression
on: { pull_request: { paths: ["prompts/**"] } }
jobs:
  eval:
    runs-on: [self-hosted, k8s]
    steps:
      - uses: actions/checkout@v4
      - name: Run eval suite
        run: |
          python -m eval.run \
            --prompts prompts/ \
            --dataset eval/regression-set.jsonl \
            --baseline main \
            --threshold 0.95
      - name: Comment results
        uses: actions/github-script@v7

Lecture 12 · Wrap-up

마무리와 학습 로드맵

12장의 강의를 통해 LLMOps의 개념·아키텍처·실습·운영을 한 번에 훑었다. 이제 어디로 더 깊이 들어가야 할까.

강의 요약 — 한 페이지 정리

기억 1
LLMOps ≠ MLOps
비결정성·토큰 경제·프롬프트·환각이 LLMOps를 별도 분야로 만든다. 기존 MLOps 위에 LLM 레이어를 쌓는 점진적 전략을 권장.
기억 2
7-레이어 스택
Infra → Train → Registry → Inference → Retrieval → Orchestration → App. 관찰성과 평가는 모든 레이어를 관통한다.
기억 3
이종 클러스터
멀티아키 이미지 + nodeSelector/toleration의 두 축으로 ARM64/x86_64 혼재 환경을 안정적으로 운영.
기억 4
vLLM 우선
OpenAI 호환 + PagedAttention + Continuous Batching. 추론 표준으로 자리잡음.
기억 5
LoRA · QLoRA
DGX Spark 128GB 통합 메모리는 LoRA에 이상적. 학습 비용을 1/10로 낮추면서 품질은 95%+ 유지.
기억 6
RAG 품질 = 검색 품질
청킹 → 임베딩 → 리랭커 → 모델 순으로 개선. 모델부터 키우는 실수를 피할 것.
기억 7
3+1 텔레메트리
Metrics·Logs·Traces에 Eval(품질 회귀)을 더한다. LLM에서는 품질이 가장 조용히 망가진다.
기억 8
프롬프트 = 코드
PR마다 평가셋 회귀를 돌려라. 프롬프트 변경은 모델 변경만큼 영향이 크다.
