CHAPTER 00

왜 "아키텍처 설계"부터 배워야 할까?

kubectl 명령어를 외우기 전에, 전체 그림을 그릴 줄 알아야 하는 이유

많은 초보 엔지니어가 쿠버네티스를 kubectl apply와 YAML 문법부터 배웁니다. 하지만 실무에서 마주치는 진짜 어려운 질문은 문법이 아니라 설계 결정입니다.

  • "개발/스테이징/운영 환경을 클러스터로 나눌까, 네임스페이스로 나눌까?"
  • "팀이 10개인데 권한(RBAC)은 어떻게 나눠줘야 사고가 안 날까?"
  • "DB 파드가 죽으면 데이터는 어디에 남아있지?"
  • "파드끼리 통신을 어디까지 허용해야 안전할까?"
  • "장애가 나면 어떤 지표를 보고 원인을 찾지?"

이 질문들에 답하는 능력이 바로 아키텍처 설계 역량이고, 이 강의의 목표입니다.

🏙️
핵심 비유 — 이 강의 전체를 관통합니다 쿠버네티스 클러스터를 대규모 아파트 단지(신도시)라고 생각해 보세요. 이 비유 하나로 거의 모든 개념을 설명할 수 있습니다.

Cluster = 신도시 전체 · Node = 아파트 동(건물) · Namespace = 회사별로 임대한 층/구역 · Pod = 세대(집) · 네트워크 = 도로와 우편 시스템 · 스토리지 = 개인 창고 · RBAC = 출입카드 권한 · Policy = 단지 관리규약 · Observability = CCTV와 관리사무소 상황판 · CI/CD = 이사·입주 자동화 시스템

아키텍트가 설계하는 순서 = 이 강의의 순서

"무엇을 얼마나 분리할 것인가"를 결정합니다. 가장 되돌리기 어려운 결정입니다.

3

CHAPTER 01

Kubernetes 아키텍처 큰 그림

Control Plane과 Worker Node — 두뇌와 손발의 분업 구조

쿠버네티스 클러스터는 크게 두 부분으로 나뉩니다. 결정을 내리는 곳(Control Plane)과 실제로 일하는 곳(Worker Node)입니다.

🏢
아파트 단지 비유 Control Plane은 관리사무소입니다. 어느 동 몇 호에 누가 입주할지 결정하고(스케줄링), 입주자 명부를 관리하고(etcd), 민원 창구를 운영합니다(API Server). Worker Node는 실제 아파트 동이고, 각 동에는 동 관리인(kubelet)이 상주하며 관리사무소의 지시대로 세대(Pod)를 관리합니다.

파드가 생성되는 전체 흐름 — "선언하면 알아서"의 비밀

쿠버네티스의 핵심 철학은 선언적(Declarative) 모델입니다. "파드를 만들어라"(명령)가 아니라 "파드 3개가 떠 있는 상태를 원한다"(선언)고 적으면, 시스템이 알아서 그 상태를 만들고 유지합니다.

1
사용자: kubectl apply

"nginx 파드 3개를 원한다"는 Deployment YAML을 API Server에 제출합니다.

2
API Server: 검증 후 etcd에 기록

인증 → 인가(RBAC) → Admission(Policy) 검사를 통과하면 etcd에 "원하는 상태"로 저장합니다.

3
Controller: 차이를 발견하고 파드 생성

"원하는 것 3개 vs 현재 0개"의 차이를 발견하고 Pod 오브젝트 3개를 만듭니다. (아직 노드 미배정)

4
Scheduler: 노드 배정

자원 여유·제약 조건을 평가해 각 파드에 최적의 노드를 배정합니다.

5
kubelet: 컨테이너 실행

자기 노드에 배정된 파드를 발견한 kubelet이 containerd에게 컨테이너 실행을 지시하고, 상태를 API Server에 보고합니다.

🔑

설계자의 관점: 이 흐름에서 모든 길목이 설계 포인트입니다. 2단계의 인가에서 RBAC(Ch.07)이, Admission에서 Policy(Ch.08)가, 4단계 배치에서 Node 설계(Ch.03)가, 5단계 실행 이후 Observability(Ch.09)가 개입합니다.

Quiz 01

Q. 클러스터의 모든 상태 정보(어떤 파드가 어디에 떠 있는지 등)가 저장되는 유일한 컴포넌트는?

정답: B. API Server는 통로일 뿐 저장하지 않습니다. 모든 상태는 etcd에만 저장되며, 그래서 etcd 백업이 곧 클러스터 백업입니다. etcd는 과반수 합의(quorum)가 필요해 3대 또는 5대의 홀수로 구성합니다.
CHAPTER 02

Cluster 설계 — 몇 개의 도시를 지을 것인가

멀티 클러스터 vs 단일 클러스터, HA 토폴로지, 그리고 되돌리기 어려운 결정들

클러스터 설계는 아키텍처에서 가장 상위의, 가장 되돌리기 어려운 결정입니다. 도시를 하나 지을지 여러 개 지을지는 나중에 바꾸기 매우 어렵기 때문입니다.

결정 1 — 클러스터를 몇 개로 나눌 것인가?

🌆
아파트 단지 비유 단일 클러스터는 하나의 거대한 신도시에 모든 회사와 주민이 함께 사는 것이고, 멀티 클러스터는 용도별로 도시를 따로 짓는 것입니다. 한 도시에 살면 도로·전기 같은 인프라를 공유해 저렴하지만, 정전(장애)이 나면 모두가 함께 피해를 봅니다. 도시를 나누면 비싸지만 서로 완전히 격리됩니다.
기준단일 클러스터 + Namespace 분리멀티 클러스터
격리 수준논리적 격리 (커널·Control Plane 공유)물리적 격리 (완전 분리)
비용낮음 — Control Plane 1세트, 노드 공유높음 — 클러스터당 관리 비용 발생
운영 복잡도낮음 — 업그레이드·모니터링 1회높음 — 클러스터 수만큼 반복 (자동화 필수)
장애 반경 (Blast Radius)큼 — Control Plane 장애 시 전체 영향작음 — 한 클러스터 장애가 다른 곳에 무영향
버전 업그레이드전체가 동시에 (위험)클러스터별 순차 진행 (안전)
규제·컴플라이언스충족 어려움망분리·데이터 주권 요구 충족 용이
실무에서 가장 흔한 정답: "환경별 분리 + 환경 내 네임스페이스 분리"
cluster: dev
개발 클러스터

실험적 워크로드, 느슨한 정책. 여러 팀이 네임스페이스로 나눠 사용. 최신 K8s 버전을 먼저 적용해 검증.

cluster: stg
스테이징 클러스터

운영과 동일한 구성(버전·정책·네트워크)을 유지하는 것이 핵심. "스테이징에서 됐는데 운영에서 안 돼요"를 방지.

cluster: prod
운영 클러스터

엄격한 RBAC·Policy, 변경은 GitOps로만. 사람의 kubectl 직접 접근을 최소화.

⚠️

흔한 실수: 개발과 운영을 하나의 클러스터에서 네임스페이스로만 분리하는 것. 개발 팀의 실수(예: 노드 자원을 다 먹는 파드, CRD 삭제)가 운영 서비스를 죽일 수 있습니다. 운영(prod)은 최소한 별도 클러스터로 분리하는 것이 업계 표준입니다.

결정 2 — Control Plane 고가용성(HA) 토폴로지

운영 클러스터라면 Control Plane이 단일 장애점(SPOF)이 되면 안 됩니다. 관리사무소가 하나뿐인데 불이 나면 단지 전체 행정이 마비되는 것과 같습니다.

🧱
Stacked etcd 토폴로지

Control Plane 노드 3대에 etcd를 함께 배치. 구성이 단순하고 노드 수가 적어 대부분의 조직에 적합. 단, 노드 1대 장애 시 Control Plane과 etcd 멤버를 동시에 잃음.

🏛️
External etcd 토폴로지

etcd 클러스터 3대를 별도로 분리 운영. 장애 격리가 뛰어나지만 노드가 6대 이상 필요. 대규모·미션크리티컬 환경용.

💡

관리형 서비스(EKS/AKS/GKE)를 쓴다면 Control Plane HA는 클라우드 사업자가 책임집니다. 이 경우 설계 포인트는 Control Plane이 아니라 노드 그룹의 다중 AZ 배치(Ch.03)로 이동합니다. 온프레미스(kubeadm, RKE2, OpenShift)라면 위 토폴로지를 직접 결정해야 합니다.

결정 3 — 클러스터 수명주기 전략

  • 버전 정책: 쿠버네티스는 약 4개월마다 마이너 버전이 나오고, 최신 3개 버전만 지원됩니다. "최신-1" 버전 유지가 일반적인 안전선입니다.
  • 업그레이드 순서: dev → stg → prod 순으로 검증하며 올립니다. Control Plane 먼저, 노드는 그다음.
  • Cattle, not Pets: 클러스터 구성 자체를 코드(Terraform, Cluster API)로 관리해, 언제든 동일한 클러스터를 재생성할 수 있어야 합니다.

  eks_managed_node_groups = {
    general = {

    }
  }
}

Quiz 02

Q. 스타트업 A사는 비용을 아끼려고 dev/stg/prod를 하나의 클러스터에 네임스페이스로 분리했습니다. 이 설계의 가장 큰 위험은?

정답: C. 네임스페이스는 논리적 경계일 뿐, 노드 커널·Control Plane·CRD 같은 클러스터 수준 리소스는 공유됩니다. 개발 파드가 limits 없이 메모리를 다 먹으면 같은 노드의 운영 파드가 밀려날 수 있습니다(장애 반경 공유). 그래서 prod는 별도 클러스터가 표준입니다. 참고로 B는 반대입니다 — 네임스페이스 간 통신은 기본적으로 모두 허용됩니다(Ch.05에서 NetworkPolicy로 제어).
CHAPTER 03

Node 설계 — 건물을 용도별로 짓기

노드 풀, Taint/Toleration, Affinity — 워크로드를 올바른 자리에 배치하는 기술

모든 노드를 똑같이 만들면 편할 것 같지만, 실제로는 워크로드 특성별로 노드를 그룹화(Node Pool / Node Group)하는 것이 정석입니다.

🏗️
아파트 단지 비유 단지 안에도 일반 주거동, 상가동, 기계실 동이 따로 있습니다. GPU 노드는 특수 설비가 갖춰진 기계실 동이고, 일반 웹 서비스는 주거동에 삽니다. Taint는 "관계자 외 출입금지" 표지판이고, Toleration은 그 표지판을 통과할 수 있는 출입증입니다. Node Affinity는 입주자가 "나는 꼭 남향 동에 살고 싶어요"라고 요청하는 것입니다.

노드 풀 분리 전략

pool: system
시스템 노드 풀

CoreDNS, Ingress Controller, 모니터링 등 플랫폼 필수 컴포넌트 전용. 애플리케이션과 분리해 플랫폼 안정성을 보장합니다.

pool: general
일반 워크로드 풀

대부분의 stateless 애플리케이션. 오토스케일링을 적극 활용하고 다중 AZ에 분산 배치합니다.

pool: gpu
GPU 노드 풀

ML 추론/학습 전용. 비싸므로 taint로 일반 파드의 침입을 막고, 필요 시에만 스케일 아웃합니다. LLMOps 환경의 핵심 설계 포인트.

pool: spot
Spot/Preemptible 풀

언제든 회수될 수 있는 저가 인스턴스. 배치 작업·재시도 가능한 워크로드 전용으로 비용을 크게 절감합니다.

배치 제어 3종 세트: Label → Taint/Toleration → Affinity

메커니즘방향한 줄 정의비유
nodeSelector / Node Affinity파드 → 노드"나는 이런 노드에 가고 싶다"입주 희망 조건
Taint + Toleration노드 → 파드"이 노드는 아무나 못 들어온다"출입금지 + 출입증
Pod Anti-Affinity파드 ↔ 파드"같은 앱 복제본끼리는 흩어져라"계란을 한 바구니에 담지 않기
topologySpreadConstraints파드 ↔ 토폴로지"AZ별로 고르게 퍼져라"동별 균등 배치
🔑

핵심 구분: Taint만으로는 "GPU 파드가 GPU 노드에만 가는 것"을 보장하지 못합니다. Taint는 일반 파드가 GPU 노드에 못 오게 막을 뿐입니다. GPU 파드가 반드시 GPU 노드로 가게 하려면 Node Affinity(또는 nodeSelector)를 함께 써야 합니다. 둘은 세트입니다.

gpu-workload.yaml — Taint + Toleration + Affinity 세트
# 1) 노드에 taint 부여 (출입금지 표지판)
#    kubectl taint nodes gpu-node-1 gpu=true:NoSchedule

apiVersion: v1
kind: Pod
metadata:
  name: llm-inference
spec:
  # 2) Toleration — 출입증 (taint를 견딜 수 있음)
  tolerations:
  - key: "gpu"
    operator: "Equal"
    value: "true"
    effect: "NoSchedule"
  # 3) Affinity — 반드시 GPU 노드로만 (필수 조건)
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
        - matchExpressions:
          - key: nvidia.com/gpu.present
            operator: In
            values: ["true"]
  containers:
  - name: vllm
    image: vllm/vllm-openai:latest
    resources:
      limits:
        nvidia.com/gpu: 1   # GPU 1장 요청
deployment.yaml — AZ 균등 분산 (topologySpreadConstraints)
spec:
  replicas: 6
  template:
    spec:
      topologySpreadConstraints:
      - maxSkew: 1                # AZ 간 파드 수 차이 최대 1
        topologyKey: topology.kubernetes.io/zone
        whenUnsatisfiable: DoNotSchedule  # 못 지키면 배치 거부
        labelSelector:
          matchLabels:
            app: web
# 결과: 3개 AZ에 2개씩 균등 배치 → AZ 하나가 죽어도 4개 생존
같은 앱 복제본을 같은 노드에 몰아넣지 않기
affinity:
  podAntiAffinity:
    # 가능하면 흩어지되, 불가능하면 배치는 허용 (soft)
    preferredDuringSchedulingIgnoredDuringExecution:
    - weight: 100
      podAffinityTerm:
        labelSelector:
          matchLabels:
            app: web
        topologyKey: kubernetes.io/hostname  # 노드 단위로 분산

# required(hard)로 하면 노드 수 < 복제본 수일 때 Pending 발생.
# 대부분 preferred(soft)가 실용적입니다.

노드 운영 시 알아야 할 개념

  • Cordon: 노드에 새 파드 배치 금지 (기존 파드는 유지) — "신규 입주 중단"
  • Drain: 노드의 파드를 다른 노드로 안전하게 대피 — 노드 점검·업그레이드 전 필수. "이사 후 리모델링"
  • PodDisruptionBudget(PDB): drain 중에도 "최소 N개는 살아있어야 한다"를 보장하는 안전장치. 이게 없으면 노드 업그레이드가 서비스 순단으로 이어질 수 있습니다.
Quiz 03

Q. GPU 노드에 gpu=true:NoSchedule taint를 걸었는데, 일반 웹 파드가 아닌 GPU 파드마저 일반 노드에 배치되는 일이 발생했습니다. 무엇이 빠졌을까요?

정답: B. Toleration은 "GPU 노드에 가는 것을 허용"할 뿐, "GPU 노드로만 가라"고 강제하지 않습니다. 스케줄러 입장에선 일반 노드도 여전히 후보입니다. GPU 파드를 GPU 노드에 고정하려면 nodeSelector나 required Node Affinity를 추가해야 합니다. (단, nvidia.com/gpu 리소스를 요청하면 해당 리소스가 있는 노드로만 가긴 하지만, 명시적 affinity가 설계상 안전합니다.)

Namespace 설계 — 하나의 도시를 나눠 쓰는 법

멀티테넌시의 기본 단위, ResourceQuota와 LimitRange로 공정한 자원 분배

네임스페이스는 하나의 클러스터를 논리적으로 나누는 칸막이입니다. 이름 충돌을 방지하고, RBAC·Quota·NetworkPolicy가 적용되는 기본 경계 단위가 됩니다.

🏬
아파트 단지 비유 네임스페이스는 회사별로 임대한 층(구역)입니다. A사가 임대한 3층과 B사가 임대한 5층에는 각각 "회의실 1호"가 있어도 충돌하지 않습니다(이름 격리). 각 회사는 자기 층 출입카드만 받고(RBAC 경계), 전기 사용량 한도를 배정받습니다(ResourceQuota). 하지만 건물의 벽과 배관은 공유합니다 — 완전한 격리가 아니라는 뜻입니다.

네임스페이스를 나누는 기준

👥
팀 기준

team-payment, team-search — 팀별 자율성과 권한 경계가 명확. 가장 흔한 패턴.

📦
애플리케이션 기준

shop-frontend, shop-backend — 서비스 단위 배포·정책 관리에 유리.

🌱
환경 기준

myapp-dev, myapp-qa — 소규모 조직에서 클러스터 분리 전 임시 단계로 사용.

💡

네이밍 컨벤션을 먼저 정하세요. <팀>-<서비스> 또는 <서비스>-<환경>처럼 규칙을 문서화하고, 필수 레이블(team, cost-center, env)을 Policy(Ch.08)로 강제하면 비용 배분과 거버넌스가 훨씬 쉬워집니다.

공정한 자원 분배 — ResourceQuota & LimitRange

칸막이만 치고 자원 한도를 정하지 않으면, 한 팀이 클러스터 자원을 독식하는 "시끄러운 이웃(Noisy Neighbor)" 문제가 생깁니다.

오브젝트적용 대상역할
ResourceQuota네임스페이스 전체 총량"이 네임스페이스는 CPU 총 20코어, 파드 최대 50개까지" — 층별 전기 총량 계약
LimitRange네임스페이스 내 개별 파드/컨테이너"컨테이너 하나는 최대 CPU 2코어까지, 미지정 시 기본값 부여" — 세대별 차단기 용량
team-ml.yaml — 팀 네임스페이스 + 총량 제한
apiVersion: v1
kind: Namespace
metadata:
  name: team-ml
  labels:
    team: ml-platform
    cost-center: "cc-3021"
    env: prod
---
apiVersion: v1
kind: ResourceQuota
metadata:
  name: team-ml-quota
  namespace: team-ml
spec:
  hard:
    requests.cpu: "40"           # CPU 요청 총합 40코어
    requests.memory: 160Gi
    requests.nvidia.com/gpu: "4"  # GPU도 쿼터 대상!
    limits.cpu: "80"
    limits.memory: 320Gi
    pods: "100"
    persistentvolumeclaims: "20"
limitrange.yaml — 개별 컨테이너 기본값·상한
apiVersion: v1
kind: LimitRange
metadata:
  name: default-limits
  namespace: team-ml
spec:
  limits:
  - type: Container
    default:              # limits 미지정 시 기본값
      cpu: "1"
      memory: 1Gi
    defaultRequest:       # requests 미지정 시 기본값
      cpu: 200m
      memory: 256Mi
    max:                  # 컨테이너 하나의 상한
      cpu: "8"
      memory: 32Gi

# 중요: ResourceQuota가 걸린 NS에서 requests/limits 없는
# 파드는 생성 거부됩니다. LimitRange의 기본값이 이를 구제합니다.
⚠️

네임스페이스가 격리하지 못하는 것: 노드(커널), 클러스터 수준 리소스(CRD, ClusterRole, PV, StorageClass), 그리고 네트워크. 네임스페이스가 달라도 파드끼리는 기본적으로 자유롭게 통신합니다. 네트워크 격리는 다음 챕터의 NetworkPolicy로 직접 설계해야 합니다.

Quiz 04

Q. team-a 네임스페이스에 ResourceQuota를 설정한 후, 팀원들이 "파드 생성이 갑자기 거부된다"고 문의합니다. 파드 스펙에 requests/limits가 없는 것이 원인일 때, 가장 적절한 해결책은?

정답: B. Quota가 걸린 네임스페이스에서는 requests/limits가 없는 파드가 거부됩니다. LimitRange의 default/defaultRequest가 자동으로 값을 채워주므로, 개발자가 스펙을 빠뜨려도 안전하게 생성됩니다. A는 통제를 포기하는 것이고, C는 보안상 최악의 선택입니다.
CHAPTER 05

네트워크 설계 — 도시의 도로망과 우편 시스템

CNI, Service, Ingress/Gateway, DNS, 그리고 NetworkPolicy로 만드는 Zero Trust

쿠버네티스 네트워크는 처음엔 어렵게 느껴지지만, 딱 4개의 질문으로 정리됩니다.

1
파드끼리 어떻게 통신하나? → CNI

모든 파드는 고유 IP를 받고, NAT 없이 서로 직접 통신합니다. 이 "평평한 네트워크"를 만들어주는 것이 CNI 플러그인입니다.

2
죽었다 살아나는 파드를 어떻게 찾나? → Service + DNS

파드 IP는 수시로 바뀝니다. Service가 고정된 가상 IP와 DNS 이름을 제공해 "움직이는 과녁"을 안정적으로 겨냥하게 해줍니다.

3
외부에서 어떻게 들어오나? → Ingress / Gateway API

단지 정문에서 방문객을 도메인·경로 기준으로 올바른 서비스에 안내합니다.

4
통신을 어디까지 허용하나? → NetworkPolicy

기본은 "모두 허용". 운영 환경에서는 "기본 차단 + 필요한 것만 허용"으로 뒤집는 것이 설계의 핵심입니다.

Service 3형제 + Ingress

📮
아파트 단지 비유 ClusterIP는 단지 내부 전용 내선번호(외부에서 못 걺), NodePort는 모든 동 경비실에 뚫어놓은 임시 직통 창구(30000번대 포트), LoadBalancer는 단지 앞에 세운 정식 안내 데스크(클라우드 LB), Ingress는 정문에서 "택배는 A동, 방문객은 B동"처럼 목적지별로 안내하는 똑똑한 컨시어지(L7 라우팅)입니다.
service.yaml — 내부 통신의 기본
apiVersion: v1
kind: Service
metadata:
  name: backend-api
  namespace: shop
spec:
  type: ClusterIP          # 내부 전용 (기본값·가장 많이 사용)
  selector:
    app: backend            # 이 레이블을 가진 파드들에게 분배
  ports:
  - port: 80                # 서비스가 받는 포트
    targetPort: 8080        # 파드 컨테이너 포트

# 다른 파드에서 호출:
#   같은 NS:   http://backend-api
#   다른 NS:   http://backend-api.shop.svc.cluster.local
ingress.yaml — 도메인/경로 기반 외부 라우팅
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: shop-ingress
  namespace: shop
  annotations:
    cert-manager.io/cluster-issuer: letsencrypt-prod  # TLS 자동화
spec:
  ingressClassName: nginx
  tls:
  - hosts: [shop.example.com]
    secretName: shop-tls
  rules:
  - host: shop.example.com
    http:
      paths:
      - path: /api
        pathType: Prefix
        backend: { service: { name: backend-api, port: { number: 80 } } }
      - path: /
        pathType: Prefix
        backend: { service: { name: frontend, port: { number: 80 } } }

# 참고: 신규 설계라면 Ingress의 후계자인 Gateway API
# (HTTPRoute 등)도 검토하세요. 역할 분리·표준화가 개선됨.
networkpolicy.yaml — "기본 차단 + 화이트리스트" 패턴
# 1) 기본 차단: 네임스페이스의 모든 인바운드 거부
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-ingress
  namespace: shop
spec:
  podSelector: {}          # NS의 모든 파드 대상
  policyTypes: [Ingress]
---
# 2) 필요한 통신만 허용: frontend → backend 8080 포트만
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-frontend-to-backend
  namespace: shop
spec:
  podSelector:
    matchLabels: { app: backend }
  policyTypes: [Ingress]
  ingress:
  - from:
    - podSelector:
        matchLabels: { app: frontend }
    ports:
    - protocol: TCP
      port: 8080

# 주의: DNS(53/UDP) egress는 항상 허용해야 합니다.
# default-deny-egress를 걸 때 흔히 빠뜨리는 부분!

설계 원칙 — Zero Trust: 쿠버네티스의 기본값은 "모든 파드 간 통신 허용"입니다. 운영 클러스터에서는 반드시 네임스페이스별 default-deny를 깔고 필요한 통신만 화이트리스트로 여는 것이 원칙입니다. 침해 사고 시 공격자의 횡적 이동(lateral movement)을 막는 최후의 방어선입니다.

🔑

더 나아가기 — Service Mesh: 서비스가 수십 개를 넘고 mTLS 암호화, 세밀한 트래픽 제어(카나리 배포 비율 조절), 서비스 간 관측성이 필요해지면 Istio·Linkerd 같은 서비스 메시를 검토합니다. 단, 운영 복잡도가 크게 오르므로 "필요해질 때" 도입하는 것이 정석입니다.

Quiz 05

Q. 운영 네임스페이스에 default-deny NetworkPolicy(Ingress+Egress)를 적용하자 모든 파드에서 서비스 이름으로 통신이 전부 실패하기 시작했습니다. 가장 유력한 원인은?

정답: B. default-deny-egress는 DNS 조회 트래픽(kube-system의 CoreDNS로 가는 53/UDP·TCP)까지 막습니다. 이름 해석이 안 되니 모든 서비스 호출이 실패하죠. default-deny를 설계할 때는 DNS egress 허용 정책을 반드시 세트로 배포해야 합니다. 실무에서 가장 흔한 NetworkPolicy 사고 1위입니다.
CHAPTER 06

스토리지 설계 — 이사 가도 남는 창고

PV/PVC/StorageClass의 3단 구조, 그리고 Stateful 워크로드를 다루는 법

파드는 언제든 죽고 다시 태어나는 존재입니다. 컨테이너 안에 쓴 데이터는 파드와 함께 사라집니다. 그래서 "데이터를 파드 밖 어딘가에 남기는 체계"가 필요하고, 그것이 스토리지 설계입니다.

📦
아파트 단지 비유 PV(PersistentVolume)는 단지 지하의 실제 창고 공간이고, PVC(PersistentVolumeClaim)는 입주자가 관리사무소에 내는 "10㎡ 창고 하나 주세요"라는 신청서입니다. 입주자는 창고가 지하 몇 층 어느 구역인지 몰라도 됩니다 — 신청서만 내면 배정됩니다. StorageClass는 창고의 등급표(일반/냉장/금고)이며, 신청서에 등급만 적으면 창고가 자동으로 지어집니다(동적 프로비저닝).

3단 구조: 개발자와 인프라의 관심사 분리

1
인프라 관리자: StorageClass 정의

"gp3 SSD 등급", "고성능 io2 등급", "NFS 공유 등급"처럼 스토리지 메뉴판을 만들어 둡니다. 실제 디스크 생성은 CSI 드라이버가 담당합니다.

2
개발자: PVC로 신청

"fast 등급으로 20Gi 주세요"라고 PVC만 작성합니다. 클라우드 콘솔을 만질 필요가 없습니다.

3
시스템: PV 자동 생성 & 바인딩

CSI 드라이버가 실제 볼륨(EBS, Azure Disk 등)을 만들고 PV로 등록, PVC와 연결(bind)합니다. 파드는 PVC 이름만 참조합니다.

Quiz 06

Q. 3개 AZ로 구성된 클러스터에서 PVC를 만들자마자 볼륨이 AZ-a에 생성되었는데, 파드는 AZ-c 노드에 배치되어 volume node affinity conflict로 기동에 실패했습니다. 이를 예방하는 StorageClass 설정은?

정답: C. 기본값 Immediate는 PVC 생성 즉시 임의의 AZ에 볼륨을 만듭니다. WaitForFirstConsumer는 파드가 스케줄된 후 그 노드의 AZ에 볼륨을 생성하므로 AZ 불일치가 원천 차단됩니다. 다중 AZ + 블록 스토리지 조합에서는 사실상 필수 설정입니다.
CHAPTER 07

RBAC & 인증/인가 — 출입카드 시스템 설계

누가(Subject) 무엇을(Resource) 어떻게(Verb) — 최소 권한 원칙의 구현

API Server로 들어오는 모든 요청은 3개의 관문을 통과합니다: 인증(누구세요?) → 인가(그거 해도 돼요?) → Admission(규칙에 맞나요?). 이 챕터는 두 번째 관문, 인가(RBAC)를 다룹니다.

🪪
아파트 단지 비유 Role은 "3층 문서고 열람 가능"처럼 권한 목록이 적힌 출입카드 템플릿입니다. RoleBinding은 그 카드를 특정 사람(User)·팀(Group)·로봇(ServiceAccount)에게 발급하는 행위입니다. ClusterRole은 층이 아니라 단지 전체에 통하는 마스터 카드 템플릿이죠. 카드(Role) 따로, 발급(Binding) 따로 — 이 분리가 RBAC의 핵심 구조입니다.

RBAC 4형제 정리

오브젝트범위정의하는 것
Role네임스페이스특정 NS 안에서의 권한 목록 (pods를 get/list 등)
ClusterRole클러스터 전체전체 범위 권한, 또는 nodes/PV 같은 클러스터 리소스 권한
RoleBinding네임스페이스Role(또는 ClusterRole)을 특정 NS에서 주체에게 부여
ClusterRoleBinding클러스터 전체ClusterRole을 모든 NS에 걸쳐 주체에게 부여 (신중히!)
💡

실무 꿀조합: ClusterRole + RoleBinding. 권한 템플릿(예: "개발자 표준 권한")은 ClusterRole로 한 번만 만들고, 각 팀 네임스페이스에서 RoleBinding으로 발급합니다. 같은 Role을 NS마다 복사할 필요가 없어져 관리가 깔끔해집니다.

주체(Subject) 3종류

👤
User

사람. K8s 내부에 User 오브젝트는 없으며 인증서나 외부 IdP(OIDC — Keycloak, Entra ID 등)가 신원을 증명합니다.

👥
Group

사람의 묶음. IdP의 그룹(예: dev-team)에 바인딩하면 인사이동 시 K8s를 건드릴 필요가 없습니다. 개인이 아닌 그룹에 바인딩이 정석.

🤖
ServiceAccount

파드(프로그램)의 신원. CI/CD 파이프라인, 모니터링 에이전트 등이 API를 호출할 때 사용합니다.

dev-role.yaml — ClusterRole 템플릿 + 팀별 RoleBinding
# 1) 권한 템플릿 (클러스터에 1개만 정의)
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: developer
rules:
- apiGroups: ["", "apps"]
  resources: [pods, deployments, services, configmaps]
  verbs: [get, list, watch, create, update, patch]
- apiGroups: [""]
  resources: [pods/log, pods/exec]   # 로그 조회 + 파드 접속
  verbs: [get, create]
# 주의: secrets는 의도적으로 제외! (별도 절차로)
---
# 2) team-a 네임스페이스에서만 이 권한을 발급
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: team-a-developers
  namespace: team-a
subjects:
- kind: Group
  name: "idp:team-a-dev"       # IdP 그룹에 바인딩 (개인 X)
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: ClusterRole              # 템플릿 재사용
  name: developer
  apiGroup: rbac.authorization.k8s.io
auditor.yaml — 전체 읽기 전용 (보안팀·신입 온보딩용)
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: security-auditors
subjects:
- kind: Group
  name: "idp:security-team"
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: ClusterRole
  name: view       # K8s 내장 읽기전용 역할 (secrets 제외)
  apiGroup: rbac.authorization.k8s.io

# 내장 ClusterRole 4종: view < edit < admin < cluster-admin
# cluster-admin의 ClusterRoleBinding은 극소수 플랫폼팀에게만!
RBAC 디버깅 — "이 권한, 정말 있나?" 확인법
# 내가 team-a에서 파드를 만들 수 있나?
kubectl auth can-i create pods -n team-a

# 특정 사용자/SA로 가장(impersonate)해서 확인
kubectl auth can-i delete deployments -n team-a \
  --as=jane@example.com
kubectl auth can-i get secrets -n team-a \
  --as=system:serviceaccount:team-a:ci-bot

# 특정 사용자가 가진 모든 권한 나열
kubectl auth can-i --list -n team-a --as=jane@example.com

# 특정 리소스에 누가 접근 가능한지 역추적 (krew 플러그인)
kubectl who-can delete pods -n team-a

RBAC 설계 원칙 5가지

  • 최소 권한(Least Privilege): 필요한 verb·resource만. verbs: ["*"]는 금지어에 가깝습니다.
  • 그룹 바인딩: 개인이 아닌 IdP 그룹에 바인딩 → 퇴사·이동 시 IdP에서만 관리.
  • cluster-admin 최소화: 손에 꼽는 인원 + break-glass(비상용) 계정만. 사용 시 감사 로그 알림.
  • ServiceAccount 분리: 워크로드마다 전용 SA. default SA에 권한을 얹지 마세요. 토큰 자동 마운트(automountServiceAccountToken)는 필요 없으면 끕니다.
  • RBAC도 GitOps로: 권한 변경이 PR 리뷰를 거치게 하면, 그 자체가 감사 기록이 됩니다.
🚨

권한 상승(Privilege Escalation) 경로 주의: secrets get 권한은 SA 토큰 탈취로, pods/exec는 파드 내부 침투로, create pods 권한은 강력한 SA를 마운트한 파드 생성으로 이어질 수 있습니다. "읽기만 주면 안전하다"는 착각을 버리고, secrets·exec 권한은 특히 엄격하게 관리하세요.

Quiz 07

Q. "모든 팀 네임스페이스에 동일한 '개발자 표준 권한'을 배포하되, 각 팀은 자기 네임스페이스에서만 그 권한을 갖게" 하려면 가장 관리하기 좋은 조합은?

정답: B. ClusterRole은 "템플릿"으로 한 번만 정의하고, RoleBinding이 적용 범위를 해당 네임스페이스로 한정합니다. A는 Role 수정 시 전체 복사본을 다 고쳐야 하고, C는 모든 NS에 권한이 열려 격리가 깨지며, D는… 설명이 필요 없겠죠. 😅
CHAPTER 08

Policy 설계 — 단지 관리규약 만들기

Pod Security Standards, ValidatingAdmissionPolicy, Kyverno/Gatekeeper — "무엇이 배포될 수 있는가"의 통제

RBAC이 "누가 무엇을 할 수 있는가"라면, Policy는 "무엇이 클러스터에 들어올 수 있는가"입니다. 권한이 있는 사람이라도 규약에 어긋나는 배포(root 컨테이너, latest 태그, limits 누락)는 Admission 단계에서 자동으로 차단합니다.

📜
아파트 단지 비유 출입카드(RBAC)가 있어도 관리규약은 따라야 합니다. "반려동물은 소형견만"(Pod Security), "발코니 확장은 승인된 규격만"(이미지 레지스트리 제한), "전입신고서에 연락처 필수 기재"(필수 레이블 강제) 같은 규칙을 입주 심사 시점(Admission)에 자동 검사하는 것입니다. 규약 위반은 입주 자체가 거부됩니다.

정책이 개입하는 지점: Admission Control

Ch.01의 요청 흐름을 기억하시나요? 인증 → 인가(RBAC) → Admission → etcd 저장. Admission은 요청을 검증(Validating)하거나 변형(Mutating)할 수 있는 마지막 관문입니다.

도구 선택지 3단계

내장 · 1단계
Pod Security Standards

K8s 내장. 네임스페이스 레이블 하나로 privileged / baseline / restricted 3단계 보안 프로파일을 적용. 모든 클러스터의 출발점.

내장 · 2단계
ValidatingAdmissionPolicy

K8s 내장(v1.30+ GA). CEL 표현식으로 커스텀 검증 규칙을 웹훅 없이 작성. 외부 도구 없이 "latest 태그 금지" 같은 규칙 구현 가능.

외부 · 3단계
Kyverno / OPA Gatekeeper

정책 엔진의 완성형. 검증+변형+생성, 정책 리포트, 감사 모드. Kyverno는 YAML 친화적, Gatekeeper는 Rego 언어 기반.

네임스페이스 레이블만으로 보안 기준 강제
apiVersion: v1
kind: Namespace
metadata:
  name: team-a
  labels:
    # restricted 위반 시 파드 생성 거부
    pod-security.kubernetes.io/enforce: restricted
    # 감사 로그·경고는 더 엄격한 기준으로 미리 확인 가능
    pod-security.kubernetes.io/warn: restricted
    pod-security.kubernetes.io/audit: restricted

# 3단계 프로파일:
#  privileged — 제한 없음 (시스템 컴포넌트용)
#  baseline   — 알려진 권한상승 차단 (hostPath, privileged 금지)
#  restricted — 강화 기준 (non-root 강제, seccomp 필수 등)
# 전략: 일반 워크로드 NS는 restricted를 기본으로,
#       예외는 명시적으로 baseline 허용
kyverno-policies.yaml — 실무 필수 정책 예시
# 정책 1: latest 태그 금지 (이미지 버전 고정 강제)

kind: ClusterPolicy

  name: disallow-latest-tag
spec:
  validationFailureAction: Enforce   # Audit로 시작 → Enforce 전환
  rules:
  - name: require-image-tag

CI/CD & GitOps — 이사 자동화 시스템

Push에서 Pull로 — Git을 유일한 진실로 삼는 배포 아키텍처

쿠버네티스 배포의 현대적 표준은 GitOps입니다. 핵심 아이디어는 단순합니다: "클러스터의 원하는 상태를 전부 Git에 선언하고, 에이전트가 Git과 클러스터를 계속 일치시킨다."

🚚
아파트 단지 비유 기존 방식(Push)은 이사업체(CI 서버)가 단지 마스터키를 들고 직접 들어와 짐을 배치하는 것입니다. 열쇠가 유출되면 큰일이죠. GitOps(Pull)는 공증된 도면(Git)을 게시하면, 단지 안에 상주하는 인테리어 관리인(ArgoCD)이 도면을 보고 알아서 집 상태를 도면과 일치시키는 방식입니다. 외부인은 열쇠가 필요 없고, 누가 도면을 언제 바꿨는지 공증 기록(Git 히스토리)이 전부 남습니다.

CI와 CD의 책임 분리

단계담당하는 일도구
CI (Continuous Integration)파이프라인테스트 → 이미지 빌드 → 취약점 스캔 → 레지스트리 푸시 → 매니페스트 저장소의 이미지 태그 업데이트GitHub Actions, GitLab CI, Jenkins, Tekton
CD (Continuous Delivery)클러스터 내 에이전트Git 매니페스트 저장소 감시 → 차이 발견 시 클러스터에 반영(sync) → 드리프트 자동 교정ArgoCD, Flux
💡

저장소 분리 원칙: 애플리케이션 코드 저장소(app repo)와 배포 매니페스트 저장소(config repo)를 분리하세요. CI는 config repo의 이미지 태그만 커밋하고, ArgoCD는 config repo만 바라봅니다. 코드 커밋과 배포 이력이 분리되어 감사와 롤백이 깨끗해집니다.

GitOps 배포 흐름 전체 그림

1
개발자: 코드 푸시 (app repo)

기능 브랜치 → PR → 리뷰 → main 병합.

2
CI: 빌드·스캔·푸시

테스트 통과 → 이미지 빌드 myapp:1.4.2 → Trivy 취약점 스캔 → 레지스트리 푸시 → 이미지 서명(cosign).

3
CI: config repo에 태그 커밋

Kustomize/Helm values의 이미지 태그를 1.4.2로 바꾸는 커밋(또는 PR). 운영 환경은 PR 승인을 배포 게이트로 활용.

4
ArgoCD: 감지 & 동기화

config repo 변경 감지 → 클러스터 상태와 비교(diff) → 자동/수동 sync → 헬스체크 확인.

5
점진적 배포 & 롤백

Argo Rollouts로 카나리(5% → 25% → 100%) 진행, 메트릭 악화 시 자동 롤백. Git revert 한 번이 곧 전체 롤백.

리소스 관리 & 오토스케일링 — 탄력적인 도시

Requests/Limits, QoS, HPA/VPA/Cluster Autoscaler — 낭비 없이 버티는 설계

앞의 모든 설계가 "구조"라면, 이 챕터는 "체력"입니다. 자원을 얼마나 예약하고 언제 늘리고 줄일지가 안정성과 비용을 동시에 결정합니다.

Requests vs Limits — 가장 오해가 많은 개념

⚡
아파트 단지 비유 Requests는 계약 전력입니다 — "우리 집은 3kW를 보장받습니다". 스케줄러는 이 계약량 합계를 보고 어느 동(노드)에 입주시킬지 결정합니다. Limits는 차단기 용량입니다 — 순간적으로 더 써도 되지만 이 선을 넘으면 차단기가 내려갑니다. CPU 초과는 스로틀링(속도 제한)이지만, 메모리 초과는 OOMKill 즉사라는 점이 결정적 차이입니다.
QoS 클래스조건노드 메모리 부족 시
Guaranteed모든 컨테이너 requests = limits가장 늦게 축출 — DB·핵심 서비스용
Burstablerequests < limits중간 순위 — 일반 웹/API에 적합
BestEffortrequests/limits 없음가장 먼저 축출 — 운영에서는 사실상 금지
💡

실무 가이드라인: 메모리는 requests = limits로 맞춰 OOM 동작을 예측 가능하게 하고, CPU limits는 과도하게 조이지 않는 것(스로틀링에 의한 지연 스파이크 방지)이 널리 쓰이는 전략입니다. 초기값은 부하 테스트 + VPA 추천 모드로 잡고, 주기적으로 실사용량과 대조해 조정하세요.

오토스케일링 3총사 — 스케일의 3가지 방향

HPA
Horizontal Pod Autoscaler

파드 수를 늘리고 줄임. CPU/메모리 또는 커스텀 메트릭(큐 길이, RPS) 기반. 트래픽 변동에 대한 1차 대응.

VPA
Vertical Pod Autoscaler

파드 하나의 크기(requests)를 조정. 추천 모드로 적정값 파악에 활용. HPA와 같은 메트릭으로 동시 사용 금지.

CA/Karpenter
Cluster Autoscaler

노드 수를 조정. Pending 파드가 생기면 노드 추가, 여유가 생기면 회수. Karpenter는 워크로드에 맞는 인스턴스 타입까지 지능적으로 선택.

hpa.yaml — CPU 70% 기준 수평 확장
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: backend-hpa
  namespace: shop
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: shop-backend
  minReplicas: 3          # AZ 3개 × 1 이상 유지
  maxReplicas: 30
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70  # requests 대비 70%!
  behavior:
    scaleDown:
      stabilizationWindowSeconds: 300  # 급격한 축소 방지

# 핵심: HPA의 %는 limits가 아니라 requests 기준입니다.
# requests가 없으면 HPA는 동작하지 않습니다!
pdb.yaml — 노드 작업 중에도 서비스 지키기
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: backend-pdb
  namespace: shop
spec:
  minAvailable: 2          # drain 중에도 최소 2개 유지
  selector:
    matchLabels:
      app: backend

# 노드 업그레이드(drain), Cluster Autoscaler의 노드 축소 등
# "자발적 중단"으로부터 최소 가용성을 보장하는 계약서.
# 주의: replicas=2에 minAvailable=2면 drain이 영원히 막힙니다.
probes.yaml — 파드의 건강검진 3종
containers:
- name: app
  image: myapp:1.4.2
  # 기동 완료 전 판정 유예 — 느리게 뜨는 앱(JVM 등)의 보호막
  startupProbe:
    httpGet: { path: /healthz, port: 8080 }
    failureThreshold: 30
    periodSeconds: 5       # 최대 150초까지 기동 대기
  # 살아있는가? 실패 시 컨테이너 재시작
  livenessProbe:
    httpGet: { path: /healthz, port: 8080 }
    periodSeconds: 10
  # 트래픽 받을 준비가 됐는가? 실패 시 Service에서 제외만
  readinessProbe:
    httpGet: { path: /ready, port: 8080 }
    periodSeconds: 5

# 흔한 사고: liveness에 DB 연결 체크를 넣으면
# DB 순단 → 전체 파드 동시 재시작 → 장애 증폭!
# 외부 의존성 체크는 readiness에만 넣으세요.
Quiz 11

Q. HPA를 설정했는데 부하가 늘어도 <unknown>/70%로 표시되며 전혀 확장되지 않습니다. 가장 흔한 원인은?

정답: A. HPA의 Utilization은 requests 대비 사용률입니다. requests가 없으면 분모가 없어 <unknown>이 됩니다(metrics-server 미설치도 같은 증상). Ch.04의 LimitRange로 기본 requests를 강제하면 이 사고를 예방할 수 있습니다 — 챕터들이 이렇게 연결됩니다.

여러 챕터를 가로지르는 실전 시나리오 문제 — 아키텍트처럼 생각해 보세요

Final 01 — 종합 시나리오

Q. 신입 개발자가 실수로 운영 네임스페이스에 limits 없는 파드를 배포했고, 메모리 누수로 노드 전체가 불안정해져 같은 노드의 다른 서비스까지 장애가 났습니다. 이 사고를 구조적으로 예방하는 가장 완전한 조합은?

정답: C. 좋은 아키텍처는 "사람이 실수해도 시스템이 막아주는" 구조입니다. LimitRange(기본값 자동 부여, Ch.04) + Admission Policy(limits 없는 배포 차단, Ch.08) + Quota(설령 통과해도 NS 총량 안에서만 피해, Ch.04)의 3중 방어가 정석입니다. A와 D는 사람에 의존하는 대책이라 반드시 재발합니다.
Final 02 — 보안 다층 방어

Q. 공격자가 취약점을 통해 frontend 파드 하나를 장악했다고 가정합니다. 다음 중 공격자의 다음 행동을 막는 방어선과 그 역할이 잘못 짝지어진 것은?

정답: D. HPA는 보안 도구가 아니라 부하에 따라 파드 수를 조절하는 오토스케일러입니다(오히려 DDoS 시 비용 폭탄이 될 수 있어 maxReplicas 상한이 중요). A·B·C는 실제 다층 방어(Defense in Depth)의 구성요소로, 하나가 뚫려도 다음 층이 피해를 국한시킵니다.
Final 03 — 장애 추적

Q. 새벽 2시, "결제 성공률 급락" 알림이 왔습니다. 가장 효율적인 원인 추적 순서는?

정답: B. 관측성 3신호의 정석적 활용 흐름입니다: 메트릭(무엇이 얼마나 나쁜가) → 트레이스(어디서) → 로그(왜). 그리고 GitOps 덕분에 "최근에 무엇이 바뀌었나"를 Git 히스토리에서 1분 안에 확인하고, 원인이 배포라면 revert 한 번으로 롤백할 수 있습니다. 장애의 상당수는 '직전 변경'이 원인입니다.
Final 04 — 설계 트레이드오프

Q. 다음 중 아키텍처 설계 사고방식으로 가장 올바른 것은?

정답: C. 이 강의의 최종 메시지입니다. 정답인 아키텍처는 없고, 맥락에 맞는 아키텍처가 있을 뿐입니다. A는 운영 복잡도라는 비용을 무시했고(필요할 때 도입), B는 격리의 이득 대비 관리 비용이 과도합니다. 좋은 아키텍트는 "왜 이 선택을 했고, 무엇을 포기했는가"를 항상 설명할 수 있습니다.
