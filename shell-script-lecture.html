CHAPTER 01

셸(Shell)이란 무엇인가?

셸은 사용자와 운영체제 커널 사이의 통역사입니다. 마우스 클릭 대신 키보드로 커널에게 명령을 전달하는 가장 빠르고 정확한 방법, 그것이 셸입니다.

왜 GUI가 아닌 셸인가?

Kubernetes Pod를 100개 재시작하는 작업을 상상해 봅시다. GUI 콘솔에서는 100번 클릭이 필요하지만, 셸에서는 kubectl rollout restart deployment/my-app 한 줄이면 끝납니다. 인프라 엔지니어가 셸을 떠나지 못하는 이유는 단순합니다 — 속도, 자동화, 그리고 재현성.

셸은 사용자의 명령을 커널이 이해할 수 있는 시스템 콜로 변환합니다

셸 vs. 셸 스크립트

두 용어는 자주 혼용되지만, 명확히 구분됩니다:

SHELL
대화형 셸 (Interactive Shell)

터미널을 열고 한 줄씩 입력하며 즉시 결과를 확인하는 환경입니다. 평소 우리가 cd, ls, kubectl 등을 치는 그 자리입니다.

SCRIPT
셸 스크립트 (Shell Script)

여러 명령어를 .sh 파일에 묶어 둔 것입니다. 한 번 작성하면 언제든 재실행할 수 있고, 다른 시스템에 그대로 옮길 수도 있습니다.

인프라 엔지니어가 셸 스크립트를 배워야 하는 5가지 이유

▸ Why Shell Script Matters
  1. 자동화의 기본 단위 — Ansible, Terraform, CI/CD 파이프라인 모두 내부적으로 셸 명령을 실행합니다. Bash를 모르면 디버깅이 불가능합니다.
  2. Container Entrypoint — Docker/Kubernetes의 CMD와 ENTRYPOINT는 결국 셸 명령입니다. 이미지 안에서 일어나는 일을 이해하려면 필수입니다.
  3. 장애 대응의 속도 — 새벽 3시 GPU 노드 장애. nvidia-smi, dmesg, journalctl을 파이프로 묶어 1분 안에 원인을 잡아야 합니다.
  4. 이식성 — Python이나 다른 도구가 깔리지 않은 미니멀 컨테이너(Alpine, distroless 등)에서도 셸은 거의 항상 동작합니다.
  5. Glue Code — 서로 다른 도구(kubectl, helm, aws cli, jq, curl)를 묶어 하나의 워크플로로 만드는 접착제 역할은 셸이 가장 잘합니다.

실제 사례: 1줄의 셸이 만드는 차이

예시로, GPU 노드의 사용 중인 메모리를 1초마다 모니터링하는 작업을 봅시다. 만약 셸을 모른다면 별도 모니터링 도구를 설치해야 하지만, 셸을 안다면:

bash
gpu-memory-watcher.sh
# GPU 메모리 사용량을 1초마다 출력 (간단 모니터링)
while true; do
  nvidia-smi --query-gpu=timestamp,name,memory.used,memory.total \
              --format=csv,noheader
  sleep 1
done

이 4줄은 모든 Linux 시스템에서 동작하며, 별도 설치도 필요 없고, 출력 결과를 그대로 파일로 저장해서 나중에 분석할 수도 있습니다. 이게 셸의 힘입니다.

▸ Check Your Understanding
Q1. 셸과 셸 스크립트의 가장 중요한 차이는 무엇인가요?
셸은 명령을 실시간으로 입력하고 즉시 결과를 받는 대화형 환경이고, 셸 스크립트는 그 명령들을 파일로 묶어 재실행 가능하게 만든 프로그램입니다. 둘 다 같은 문법을 사용하지만, 스크립트는 자동화와 재현성을 보장합니다.
CHAPTER 02

셸의 종류와 비교

Linux에는 여러 셸이 존재합니다. 어떤 셸을 선택하느냐에 따라 스크립트의 문법, 속도, 이식성이 달라집니다. 인프라 엔지니어가 알아야 할 핵심 셸들을 비교해 봅시다.

주요 셸 한눈에 비교

셸 경로 설명 주요 사용처 POSIX
sh /bin/sh Bourne Shell. 모든 Unix의 조상 POSIX 호환 스크립트, 컨테이너 init 100%
bash /bin/bash Bourne Again Shell. 가장 대중적 대부분의 Linux 배포판 기본 셸 확장 多
dash /bin/dash 경량 POSIX 셸. 매우 빠름 Ubuntu/Debian의 /bin/sh 실체 엄격
ash /bin/ash BusyBox에 포함된 경량 셸 Alpine, 임베디드, 미니멀 컨테이너 제한적
zsh /bin/zsh 강력한 대화형 기능, 플러그인 풍부 macOS 기본 셸, 개발자 데일리 드라이버 대부분
ksh /bin/ksh Korn Shell. AT&T 출신 AIX, Solaris 등 엔터프라이즈 Unix 호환
fish /usr/bin/fish Friendly Interactive Shell 대화형 개발 환경 (스크립트엔 부적합) 비호환

각 셸의 특징과 선택 기준

sh — Bourne Shell의 후예

탄생: 1977저자: Stephen Bourne표준: POSIX

sh는 사실 셸이라기보단 표준(specification)에 가깝습니다. POSIX sh 문법으로 작성된 스크립트는 어떤 Unix 시스템에서도 동작해야 합니다.

실제로 /bin/sh는 배포판마다 다른 셸로 링크되어 있습니다 — Ubuntu에서는 dash, RHEL에서는 bash, Alpine에서는 ash. 그래서 최대 이식성을 원하면 sh 문법만 사용해야 합니다.

▸ 언제 sh를 쓰나?

컨테이너 entrypoint 스크립트, 다양한 환경에서 돌아가야 하는 부트스트랩 스크립트, 임베디드 시스템 등.

bash — 사실상의 Linux 표준

탄생: 1989저자: Brian Fox (GNU)표준: bash 확장

대부분의 Linux 배포판에서 기본 셸로 사용됩니다. POSIX sh의 모든 기능에 더해 배열, [[ ]] 조건식, 프로세스 치환 등 강력한 확장을 제공합니다.

인프라 엔지니어의 99%는 bash로 스크립트를 작성합니다. 문서, 예제, Stack Overflow 답변이 가장 많은 셸입니다.

▸ Bashism이란?

bash에만 있고 다른 sh 호환 셸에는 없는 문법을 Bashism이라고 합니다. 예: [[ ]], echo -e, <<< (here-string), 배열 등. #!/usr/bin/env bash로 명시하면 안전합니다.

dash — 빠르고 가벼운 POSIX 셸

기반: NetBSD ash특징: 작고 빠름표준: POSIX 엄격

Debian과 Ubuntu의 /bin/sh는 사실 dash입니다. bash보다 4배 가까이 빠르게 부팅 스크립트를 실행할 수 있어, 시스템 초기화에 사용됩니다.

dash는 bash 확장 문법을 지원하지 않습니다. 따라서 #!/bin/sh로 시작하는 스크립트에서 [[ ]]를 사용하면 Ubuntu에서는 동작하지 않습니다.

▸ 흔한 함정

스크립트가 macOS와 CentOS에서는 동작하는데 Ubuntu에서는 실패할 때, 십중팔구 /bin/sh가 dash라서 그렇습니다. shebang을 #!/bin/bash로 명시하세요.

zsh — 개발자가 사랑하는 대화형 셸

탄생: 1990저자: Paul Falstad특징: 강력한 자동완성

macOS Catalina(2019) 이후 기본 셸이 bash에서 zsh로 변경되었습니다. Oh My Zsh, Powerlevel10k 같은 프레임워크 덕분에 대화형 사용성이 압도적입니다.

스크립트로 작성할 때는 bash와 90% 호환되지만, 인프라 자동화 스크립트는 여전히 bash로 작성하는 것이 표준입니다. (대부분의 서버에 zsh가 설치되어 있지 않기 때문)

ash — Alpine과 BusyBox의 셸

크기: 매우 작음표준: POSIX 일부주요 사용: Alpine Linux

Alpine Linux 기반 컨테이너 이미지(예: node:alpine, python:alpine)에서 /bin/sh는 BusyBox의 ash입니다. 매우 가볍지만 bash 확장은 거의 지원하지 않습니다.

Kubernetes 환경에서 이미지 크기를 줄이려고 Alpine을 자주 쓰는데, 이때 작성된 스크립트가 ash에서 동작하지 않으면 컨테이너가 죽습니다.

▸ Alpine + bash

Alpine에서 bash가 필요하면 apk add --no-cache bash로 명시적 설치해야 합니다. 또는 더 가벼운 ash 호환 스크립트를 작성하세요.

현재 셸 확인하기

지금 내가 어떤 셸을 쓰고 있는지 확인하는 방법입니다.

terminal

# Alpine → /bin/busybox
▸ Check Your Understanding
Q2. 다음 스크립트는 Ubuntu에서 동작할까요? 안 된다면 왜 그럴까요?
#!/bin/sh
if [[ "$1" == "deploy" ]]; then echo "배포 중..."; fi
동작하지 않습니다. Ubuntu에서 /bin/sh는 dash이고, dash는 [[ ]]를 지원하지 않습니다. 해결책은 두 가지: ① shebang을 #!/bin/bash로 변경하거나, ② POSIX 호환인 [ "$1" = "deploy" ]로 바꾸면 됩니다. (단일 등호와 단일 대괄호 주의!)
CHAPTER 03

첫 번째 셸 스크립트 만들기

이제 직접 스크립트를 작성해 봅시다. 단순한 Hello World부터 시작해서, 실행 권한, shebang, 디버깅까지 — 모든 셸 스크립트의 토대가 되는 부분입니다.

Hello, World 스크립트

bash
hello.sh
#!/usr/bin/env bash
# 이름: hello.sh
# 설명: 첫 번째 셸 스크립트

echo "안녕하세요, 셸 스크립트의 세계입니다."
echo "오늘 날짜: $(date)"
echo "현재 사용자: $USER"
echo "현재 디렉토리: $(pwd)"

구성 요소 분석

1. Shebang (#!) — 첫 줄의 마법

셸 스크립트의 첫 줄에 등장하는 #! (sharp + bang = shebang)은 "이 파일을 어떤 인터프리터로 실행할 것인가"를 OS에 알려줍니다. shebang이 없으면 현재 셸로 실행되거나 Permission denied 오류를 만날 수 있습니다.

Shebang 표현의미권장도
#!/bin/shPOSIX 셸로 실행 (가장 이식성 높음)⭐⭐⭐⭐⭐ 이식성 최우선시
#!/bin/bashbash로 실행. 절대 경로⭐⭐⭐ 대부분 OK
#!/usr/bin/env bashPATH에서 bash를 찾아 실행⭐⭐⭐⭐⭐ 가장 권장
#!/usr/bin/python3다른 언어도 가능 (참고)언어별 적용
▸ 왜 /usr/bin/env bash가 더 권장될까?

macOS의 bash는 /bin/bash(매우 오래된 3.2 버전)에 있지만, Homebrew로 설치한 최신 bash는 /opt/homebrew/bin/bash나 /usr/local/bin/bash에 있습니다. env를 거치면 PATH에서 검색하므로, 사용자가 원하는 bash 버전을 자동으로 사용합니다.

2. 주석 (Comments)

#으로 시작하는 줄은 셸이 무시합니다. 단, shebang의 #!만은 예외입니다. 인프라 스크립트에서는 주석이 곧 운영 문서입니다. 6개월 뒤 새벽 3시에 깨어났을 때를 생각하며 작성하세요.

3. 명령어와 인자

echo "..."처럼 명령어 뒤에 공백으로 인자를 구분합니다. 문자열은 큰따옴표 "나 작은따옴표 '로 감쌀 수 있는데, 차이가 큽니다 (4장에서 자세히).

실행하는 3가지 방법

terminal
스크립트 실행 방법
# 방법 1: 인터프리터를 직접 호출 (실행 권한 불필요)
bash hello.sh
sh   hello.sh

# 방법 2: 실행 권한 부여 후 직접 실행 (가장 일반적)
chmod +x hello.sh   # 실행 권한 부여
./hello.sh           # 현재 디렉토리의 hello.sh 실행

# 방법 3: source / dot (.) — 현재 셸에서 실행 (변수 공유)
source hello.sh
. hello.sh           # source의 단축형
▸ 방법 2와 3의 결정적 차이

./hello.sh는 새로운 자식 프로세스에서 스크립트를 실행합니다. 따라서 스크립트 안에서 정의한 변수는 종료 후 사라집니다. 반면 source hello.sh는 현재 셸에서 직접 실행하므로 변수가 그대로 남습니다. .bashrc를 다시 적용할 때 source ~/.bashrc를 쓰는 이유입니다.

실행 권한 (Permission) 깊이 보기

Linux의 모든 파일은 읽기(r), 쓰기(w), 실행(x) 권한을 갖습니다. 스크립트를 ./로 실행하려면 실행 권한이 필요합니다.

terminal
권한 확인과 변경
# 권한 확인
ls -l hello.sh
# -rw-r--r-- 1 cheolkyu staff 156 May 24 10:30 hello.sh
#  ↑↑↑ ↑↑↑ ↑↑↑
#  소유자 그룹 기타 (각 자리: r/w/x)

# 모든 사용자에게 실행 권한 추가
chmod +x hello.sh

# 소유자에게만 실행 권한 추가 (더 엄격)
chmod u+x hello.sh

# 숫자 방식: 754 = rwx r-x r--
#           소유자=7  그룹=5  기타=4
chmod 754 hello.sh

디버깅: -x 옵션의 마법

스크립트가 의도대로 동작하지 않을 때, 한 줄씩 추적하면서 무엇이 실행되는지 보고 싶다면 -x (xtrace) 옵션을 쓰세요.

terminal
디버깅 모드
# 방법 1: 실행 시 -x 옵션
bash -x hello.sh

# 방법 2: shebang에서 -x 추가
#!/bin/bash -x

# 방법 3: 스크립트 일부만 디버깅
set -x   # 추적 시작
# ... 의심스러운 부분 ...

"만약 ~라면" — 모든 자동화의 핵심입니다. 셸의 조건문은 명령의 종료 코드를 기반으로 동작한다는 점만 이해하면 나머지는 어렵지 않습니다.

fi

fi

실전: 재시도 패턴 (Retry with Exponential Backoff)

bash

#!/usr/bin/env bash
# 명령을 최대 N번까지 재시도 (지수 백오프)

MAX_ATTEMPTS=5
INITIAL_DELAY=1
attempt=1
delay=$INITIAL_DELAY

while [ $attempt -le $MAX_ATTEMPTS ]; do
  echo "시도 $attempt/$MAX_ATTEMPTS..."

  if curl -fsS https://api.example.com/health; then
    echo "✓ 성공"
    exit 0
  fi

  echo "실패. ${delay}초 후 재시도..."
  sleep "$delay"
  ((attempt++))
  delay=$((delay * 2))   # 1, 2, 4, 8, 16...
done

echo "✗ $MAX_ATTEMPTS번 모두 실패" >&2
exit 1

매개변수와 반환값

함수의 매개변수는 스크립트 인자와 똑같이 $1, $2, $@로 받습니다. 반환값은 두 가지 방식이 있는데, 다른 언어와 다르니 주의하세요.

bash
매개변수와 반환
#!/usr/bin/env bash

# 방식 1: return은 종료 코드만 (0~255 정수)
is_pod_running() {
  local pod_name="$1"
  local status=$(kubectl get pod "$pod_name" -o jsonpath='{.status.phase}')

  if [ "$status" = "Running" ]; then
    return 0   # 성공 (true)
  else
    return 1   # 실패 (false)
  fi
}

# 사용: if 문에서 자연스럽게
if is_pod_running "my-app"; then
  echo "실행 중"
fi

# 방식 2: 값을 반환하려면 echo로 출력 + $()로 캡처
get_pod_count() {
  local namespace="$1"
  kubectl get pods -n "$namespace" --no-headers | wc -l
}

count=$(get_pod_count "default")
echo "Pod 개수: $count"

지역 변수의 중요성

함수 내부에서 local을 쓰지 않으면 전역 변수가 오염됩니다. 큰 스크립트에서 디버깅하기 가장 어려운 종류의 버그를 만듭니다.

bash
local의 차이 — 실험
count=100   # 전역 변수

# local 없는 함수 (위험)
bad_function() {
  count=5     # 전역 변수를 덮어씀!
  echo $count
}

# local 있는 함수 (안전)
good_function() {
  local count=5   # 이 함수 안에서만 유효
  echo $count
}

bad_function
echo "호출 후 count: $count"   # 5 (덮어쓰임!)

count=100
good_function
echo "호출 후 count: $count"   # 100 (그대로)
▸ 절대 규칙

함수 안의 모든 변수는 예외 없이 local로 선언하세요. 전역 변수가 필요하면 함수 밖에서 명시적으로 선언하고 함수 안에서는 그것만 수정하세요. 이 한 가지 규칙만 지켜도 셸 스크립트 버그의 70%가 사라집니다.

실전: 로깅 함수 패턴

대부분의 인프라 스크립트가 반드시 갖춰야 할 컬러 로그 함수 세트입니다.

log-helpers.sh
#!/usr/bin/env bash

# ANSI 색상 코드
readonly RED='\033[0;31m'
readonly GREEN='\033[0;32m'
readonly YELLOW='\033[1;33m'
readonly BLUE='\033[0;34m'
readonly NC='\033[0m'     # No Color

log_info() {
  echo -e "${BLUE}[INFO]${NC} $(date +'%Y-%m-%d %H:%M:%S') $*"
}

log_warn() {
  echo -e "${YELLOW}[WARN]${NC} $(date +'%Y-%m-%d %H:%M:%S') $*" >&2
}

log_error() {
  echo -e "${RED}[ERROR]${NC} $(date +'%Y-%m-%d %H:%M:%S') $*" >&2
}

log_success() {
  echo -e "${GREEN}[ ✓ ]${NC} $*"
}

# 사용
log_info "배포 시작"
log_success "Pod 생성 완료"
log_warn "replica 수가 권장보다 낮습니다"
log_error "이미지 풀 실패"

CHAPTER 09

입출력과 리다이렉션

유닉스 철학의 핵심은 "모든 것은 파일이다"입니다. 키보드 입력, 화면 출력, 에러 메시지조차도 파일 디스크립터(FD)라는 번호로 다뤄집니다. 리다이렉션과 파이프를 익히면, 작은 명령어들을 조립해 거대한 데이터 파이프라인을 만들 수 있습니다.

▸ 9.1 세 가지 표준 스트림

모든 프로세스는 시작될 때 세 개의 파일 디스크립터를 자동으로 가지고 시작합니다.

FD이름기호설명
0stdin<표준 입력 (보통 키보드)
1stdout>표준 출력 (보통 터미널 화면)
2stderr2>표준 에러 (보통 터미널 화면, 별도 채널)
💡 stdout과 stderr를 왜 나누는가?
화면에 같이 출력되지만 채널이 다르기 때문에 따로 처리할 수 있습니다. 예를 들어 정상 출력은 파일로 저장하고 에러만 화면에 띄우거나, 반대로 에러만 모아서 로그 파일로 보낼 수 있습니다. 이게 가능해야 자동화가 됩니다.

▸ 9.2 출력 리다이렉션

bash
# >  : 덮어쓰기 (파일이 있으면 내용을 비우고 새로 씀)
echo "hello" > output.txt

# >> : 추가 (append, 파일 끝에 덧붙임)
echo "line 2" >> output.txt
date >> output.txt

# 2> : stderr만 파일로
kubectl get pod nonexistent 2> error.log

# 2>&1 : stderr를 stdout과 같은 곳으로 (병합)
# 순서 중요! '> file' 다음에 '2>&1'
./run.sh > all.log 2>&1

# &> : stdout + stderr 동시에 (bash 단축 표현)
./run.sh &> all.log

# >/dev/null : 출력 버리기 (블랙홀)
kubectl apply -f manifest.yaml >/dev/null 2>&1
⚠ 순서가 바뀌면 동작이 다르다
cmd 2>&1 > file ← 이건 잘못됨. stderr가 먼저 stdout(=터미널)에 연결되고 나서 stdout이 file로 바뀌므로, stderr는 여전히 터미널로 나갑니다. 반드시 cmd > file 2>&1 순서로 쓰세요.

▸ 9.4 파이프 (|)

한 명령의 stdout을 다음 명령의 stdin으로 연결합니다. 작은 도구들을 조합해 복잡한 작업을 표현하는 유닉스 철학의 핵심입니다.

bash
# 가장 흔한 패턴: 필터링과 가공의 파이프라인
kubectl get pods --all-namespaces \
  | grep "CrashLoopBackOff" \
  | awk '{print $1, $2}' \
  | sort -u

# GPU 사용률이 80% 이상인 프로세스만 추출
nvidia-smi --query-compute-apps=pid,process_name,used_memory \
  --format=csv,noheader \
  | awk -F',' '$3+0 > 8000 {print}'

# 한 줄에서 여러 도구를 거치는 데이터 변환
cat access.log \
  | grep -v "200" \
  | cut -d' ' -f1 \
  | sort \
  | uniq -c \
  | sort -rn \
  | head -10
# → 에러가 많은 IP 상위 10개
🔧 파이프는 stdout만 흘려보낸다
cmd1 | cmd2에서 cmd1의 stderr는 파이프를 타지 않고 터미널로 직접 나옵니다. 에러까지 다음 명령으로 넘기려면 cmd1 2>&1 | cmd2 또는 cmd1 |& cmd2(bash 4+) 형태를 쓰세요.

▸ 9.5 tee — 분기 출력

tee는 stdin을 받아 stdout과 파일에 동시에 쓰는 'T자 분기관' 같은 도구입니다. 로그를 화면으로도 보면서 파일에도 저장할 때 필수입니다.

▸ 9.6 xargs — 인자로 변환

xargs는 stdin으로 들어온 데이터를 다음 명령의 인자(argument)로 변환합니다. 파이프로 명령 인자를 못 넘긴다는 셸의 한계를 우회합니다.

bash

# 올바른 예: xargs 사용
echo "file1.txt file2.txt" | xargs rm

# Pending 상태 Pod 일괄 삭제

# 병렬 실행 (-P): 4개씩 동시에
cat image-list.txt | xargs -P 4 -I {} docker pull {}

# 안전 모드: 공백/특수문자 있는 파일명 처리 (null 구분자)
find . -name "*.tmp" -print0 | xargs -0 rm

▸ Check Your Understanding
Q7. ./script.sh 2>&1 > output.log와 ./script.sh > output.log 2>&1의 차이는?
리다이렉션은 왼쪽에서 오른쪽으로 순서대로 평가됩니다. 첫 번째는 stderr를 (그 시점의) stdout 위치인 터미널로 복제한 뒤 stdout만 파일로 바꾸므로, stderr는 여전히 터미널에 나옵니다. 두 번째는 stdout을 먼저 파일로 바꾸고 stderr를 그 stdout(=파일)으로 복제하므로, 둘 다 파일에 기록됩니다. 의도가 "전부 파일로"라면 후자가 정답이며, &> output.log로 더 간단히 쓸 수 있습니다.
CHAPTER 10

문자열과 배열

셸의 기본 자료형은 문자열입니다. 자르고, 자르고, 또 자르는 일이 90%죠. 여기에 배열까지 익히면 YAML/JSON 파싱, 다중 클러스터 컨텍스트 관리 같은 실제 인프라 작업을 셸만으로 충분히 처리할 수 있게 됩니다.

▸ 10.1 문자열 길이와 부분 추출

bash
str="kubernetes-cluster-prod-01"

# 길이
echo ${#str}          # 26

# 부분 추출: ${var:offset:length}
echo ${str:0:10}       # kubernetes

▸ 10.2 패턴 제거 (Prefix/Suffix Stripping)

가장 자주 쓰이는 문자열 가공 패턴입니다. 파일 경로, 확장자, URL 분해에 필수입니다.

문법의미예 (path=/var/log/app/error.log)
${var#pat}앞에서 짧게 매칭 제거${path#*/} → var/log/app/error.log
${var##pat}앞에서 길게 매칭 제거${path##*/} → error.log (basename)
${var%pat}뒤에서 짧게 매칭 제거${path%/*} → /var/log/app (dirname)
${var%%pat}뒤에서 길게 매칭 제거${path%%/*} → (빈 문자열)
bash
# 실전: 이미지 태그에서 버전만 분리
image="nvcr.io/nvidia/pytorch:24.05-py3"
name=${image%:*}   # nvcr.io/nvidia/pytorch
tag=${image##*:}   # 24.05-py3
repo=${image%%/*}  # nvcr.io

# 파일 확장자 바꾸기
f="report.txt"
echo "${f%.txt}.bak"   # report.bak

▸ 10.3 치환 (Replace)

bash
s="pod-app-01-pod-app-02"

# 첫 번째 매칭만 치환
echo ${s/pod/POD}     # POD-app-01-pod-app-02

# 모두 치환 (//)
echo ${s//pod/POD}    # POD-app-01-POD-app-02

# 앞쪽만 (#) / 뒤쪽만 (%)
echo ${s/#pod/POD}    # POD-app-01-pod-app-02 (시작이 pod일 때만)
echo ${s/%02/XX}      # pod-app-01-pod-app-XX (끝이 02일 때만)

# 대소문자 변환 (bash 4+)
env="prod"
echo ${env^^}           # PROD
CTX="PROD-CLUSTER"
echo ${CTX,,}           # prod-cluster

▸ 10.4 기본값 (Default Value)

변수가 비어있을 때의 동작을 한 줄로 제어합니다. 설정값 처리에 거의 매일 씁니다.

문법동작
${var:-default}var가 비어있으면 default 반환 (var는 그대로)
${var:=default}var가 비어있으면 default 대입 후 반환
${var:?msg}var가 비어있으면 msg 출력 후 스크립트 종료
${var:+alt}var가 있으면 alt 반환, 없으면 빈 문자열
bash
# 환경변수가 없으면 기본값 사용
NAMESPACE="${NAMESPACE:-default}"
REPLICAS="${REPLICAS:-3}"

# 필수 변수 검증 (없으면 즉시 종료)
: "${KUBECONFIG:?KUBECONFIG 환경변수가 필요합니다}"
: "${GPU_TYPE:?GPU_TYPE을 설정하세요 (a100, h100, ...)}"

# 디버그 플래그가 켜졌을 때만 추가 옵션
DEBUG_FLAG=${DEBUG:+--verbose --debug}

▸ 10.5 인덱스 배열

bash
# 선언과 초기화
nodes=("dgx-spark" "thinkstation" "master-node")

# 요소 추가
nodes+=("worker-04")

# 개별 접근
echo ${nodes[0]}      # dgx-spark
echo ${nodes[-1]}     # worker-04 (bash 4.3+)

# 전체 요소: "${arr[@]}" 가 안전 (각 요소가 따로)
#           "${arr[*]}" 는 IFS로 합쳐진 한 문자열
for n in "${nodes[@]}"; do
  echo "노드: $n"
done

# 길이, 인덱스 목록, 슬라이싱
echo ${#nodes[@]}     # 4 (개수)
echo ${!nodes[@]}     # 0 1 2 3 (인덱스들)
echo "${nodes[@]:1:2}" # thinkstation master-node

# 명령 결과를 배열로
pods=($(kubectl get pod -o name))
echo "총 ${#pods[@]} 개의 Pod"
⚠ "${arr[@]}" vs "${arr[*]}"
공백이 포함된 요소를 안전하게 다루려면 거의 항상 "${arr[@]}"를 쓰세요. "${arr[*]}"는 모든 요소를 IFS 첫 글자(기본은 공백)로 이어붙인 단일 문자열을 만듭니다. 로그 메시지처럼 한 문자열로 합치고 싶을 때만 의도적으로 사용합니다.

▸ 10.6 연관 배열 (Associative Array, bash 4+)

키-값 쌍을 다루는 해시맵입니다. 사용 전 반드시 declare -A로 선언해야 합니다.

bash
# 노드별 GPU 종류 매핑
declare -A gpu_map
gpu_map[dgx-spark]="GB10"
gpu_map[thinkstation]="RTX-A6000"
gpu_map[worker-03]="none"

# 조회
echo ${gpu_map[dgx-spark]}     # GB10

# 모든 키 / 값 순회
for node in "${!gpu_map[@]}"; do
  echo "$node → ${gpu_map[$node]}"
done

# 존재 여부 확인
if [[ -v gpu_map[thinkstation] ]]; then
  echo "키 존재"
fi

# 삭제
unset 'gpu_map[worker-03]'
CHAPTER 11

명령어 치환과 확장

셸이 명령을 실행하기 전에 거치는 단계들을 정확히 이해해야, 변수가 안 풀리거나 따옴표가 이상하게 동작하는 함정에 빠지지 않습니다. 이 챕터는 "왜 내 변수가 안 먹지?"의 해답입니다.

▸ 11.1 명령어 치환: $(...) vs `...`

bash
# 현대 표기법 ($) — 권장
now=$(date +'%Y-%m-%d')
pod_count=$(kubectl get pod --no-headers | wc -l)

# 레거시 표기법 (백틱) — 가독성/중첩 문제로 비추천
now=`date +'%Y-%m-%d'`

# 중첩이 자연스러움 (백틱은 이게 안 됨)
latest_pod=$(kubectl get pod -o name | head -$(echo 1))

▸ 11.2 셸의 확장 순서 (Expansion Order)

셸이 한 줄의 명령을 실행하기 전, 다음 순서로 변환을 거칩니다. 이 순서를 외워두면 디버깅이 훨씬 쉬워집니다.

expansion order
1) 중괄호 확장   : {a,b,c}.txt          → a.txt b.txt c.txt
2) 틸드 확장     : ~/data               → /home/cklee/data
3) 매개변수 확장 : $var, ${var:-def}    → 변수값
4) 명령어 치환   : $(cmd), `cmd`         → 명령 출력
5) 산술 확장     : $((1+2))             → 3
6) 단어 분할     : IFS 기준으로 분리
7) 경로명 확장   : *.txt, ?.log         → 매칭 파일들 (globbing)
8) 따옴표 제거   : 남은 따옴표 제거
💡 단어 분할이 일어나는 시점
"단어 분할(word splitting)"이 매개변수 확장 다음에 일어나기 때문에, $var를 따옴표 없이 쓰면 변수값이 공백 기준으로 쪼개져 여러 단어가 됩니다. 변수에 공백이 포함될 수 있다면 항상 "$var"로 감싸세요. 이게 셸 버그의 80%입니다.

▸ 11.3 중괄호 확장 (Brace Expansion)

bash
# 리스트 확장
echo file_{a,b,c}.log
# → file_a.log file_b.log file_c.log

# 범위 확장
echo node-{01..05}
# → node-01 node-02 node-03 node-04 node-05

# 증분 지정 (bash 4+)
echo {0..100..10}
# → 0 10 20 30 ... 100

# 실전: 디렉터리 일괄 생성
mkdir -p logs/{api,worker,scheduler}/{2025,2026}/{01..12}

# 백업 파일 생성 (확장 후 cp가 두 인자를 받음)
cp deployment.yaml{,.bak}
# ↑ cp deployment.yaml deployment.yaml.bak 으로 확장됨

▸ 11.4 산술 확장 $((...)) vs let vs expr

bash
# 권장: $(())
total=$(( 8 * 1024 ))           # 8192
avg=$(( (a + b) / 2 ))
(( count++ ))                       # 후위 증가

# 비교도 ((...)) 안에서는 자연스러운 C 스타일
if (( gpu_mem > 8000 )); then
  echo "메모리 충분"
fi

# 레거시: let, expr (특별한 이유 없으면 사용 X)
let "total = 8 * 1024"
total=$(expr 8 \* 1024)      # 별표는 이스케이프 필요
⚠ 셸은 정수만 다룬다
$(( 3 / 2 ))는 1입니다 (소수점 버림). 실수가 필요하면 bc나 awk를 쓰세요:
echo "scale=2; 3/2" | bc → 1.50

▸ 11.5 글로브(Globbing) vs 정규식

이 둘은 완전히 다른 시스템입니다. 셸 자체가 처리하는 것은 글로브이고, 정규식은 grep, sed, [[ =~ ]] 같은 도구가 처리합니다.

패턴글로브정규식 (POSIX)
임의의 한 글자?.
임의의 문자열*.*
문자 클래스[abc][abc]
예제*.log^.*\.log$
bash
# Globbing (셸이 처리)
ls *.yaml                      # 현재 디렉터리의 모든 yaml
ls pod-[0-9][0-9].log         # pod-NN.log
rm /tmp/cache-*.tmp

# 확장 글로브 (shopt -s extglob 필요)
shopt -s extglob
ls !(*.bak)                    # .bak가 아닌 모든 파일
rm *.@(log|tmp)               # log 또는 tmp 확장자

# 정규식 (도구가 처리, ERE 사용시 -E)
grep -E "^[0-9]{4}-[0-9]{2}" log.txt
[[ "$ip" =~ ^10\.0\.[0-9]+\.[0-9]+$ ]]

▸ 11.6 따옴표의 세 가지 차이

유형변수 확장명령 치환예
홑따옴표 '...'❌ 아니오❌ 아니오'$HOME' → 리터럴 $HOME
쌍따옴표 "..."✅ 예✅ 예"$HOME" → /home/cklee
따옴표 없음✅ 예 + 단어 분할✅ 예 + 단어 분할위험! 공백/글로브 함정
🔧 따옴표 황금률
의심스러우면 쌍따옴표로 감싸라. 변수, 명령 치환, 파일 경로는 거의 항상 "$var", "$(cmd)", "$file"로. 리터럴 문자열은 홑따옴표로 (이스케이프 신경쓸 일이 없어짐). 두 종류를 이어붙일 수도 있습니다: 'literal '"$var"' more'.
CHAPTER 12

에러 처리와 방어적 스크립팅

"스크립트가 중간에 실패해도 계속 실행돼서 더 큰 사고가 났다" — 운영 자동화에서 가장 흔한 사고 시나리오입니다. 셸은 기본값이 매우 관대해서, 명시적으로 엄격 모드를 켜지 않으면 에러를 그냥 넘깁니다. 프로덕션 스크립트는 반드시 방어적으로 작성해야 합니다.

▸ 12.1 종료 코드 (Exit Code)

모든 명령은 종료 시 0~255 사이의 정수를 반환합니다. 0은 성공, 그 외는 실패. $?로 직전 명령의 종료 코드를 조회합니다.

코드의미 (관습)
0성공
1일반적인 에러
2사용법 오류 (잘못된 인자)
126실행 권한 없음
127명령 없음 (command not found)
128+N시그널 N에 의한 종료 (예: 130=Ctrl+C=SIGINT)
bash
kubectl get pod my-app
echo "종료 코드: $?"

# 명시적 종료 코드 반환
check_gpu() {
  if ! command -v nvidia-smi >/dev/null; then
    echo "nvidia-smi 없음" >&2
    return 127
  fi
  nvidia-smi >/dev/null || return 1
  return 0
}

▸ 12.2 엄격 모드 (Unofficial Bash Strict Mode)

프로덕션 스크립트의 첫 줄에 거의 자동으로 넣는 4종 세트입니다.

bash
#!/usr/bin/env bash
set -euo pipefail
IFS=$'\n\t'

# 각 옵션의 역할:
# -e (errexit)     : 명령이 실패하면 즉시 스크립트 종료
# -u (nounset)     : 정의되지 않은 변수 사용 시 에러
# -o pipefail      : 파이프라인 중 하나라도 실패하면 전체 실패
# IFS=$'\n\t'      : 공백 분할의 함정 회피 (탭/개행만 구분자)

▸ 12.3 set -e의 함정

set -e는 만능이 아닙니다. 다음 경우엔 발동되지 않습니다:

bash
### 함정 1: if/while/until의 조건절
if false; then echo "never"; fi   # 스크립트 안 멈춤 (정상)

### 함정 2: ||, && 뒤에 오는 명령
false || true          # 안 멈춤 (의도된 동작)

### 함정 3: 함수 안에서의 실패가 호출자에 전파되지 않을 수 있음
my_func() { false; echo "이게 출력되면 곤란"; }
my_func && echo "ok"   # 'echo'가 실행되어버림!

### 함정 4: pipefail 없으면 파이프 중간 실패가 묻힘
cat nonexistent.txt | grep error    # cat 실패해도 종료 코드는 grep 기준

### 회피: pipefail 설정 후 다시 시도
set -o pipefail
cat nonexistent.txt | grep error    # 이제 종료 코드 1 (cat 실패 반영)

▸ 12.4 명시적 에러 처리 패턴

bash
# 패턴 1: || 로 폴백
kubectl apply -f manifest.yaml || {
  echo "적용 실패, 롤백 시도" >&2
  kubectl rollout undo deployment/my-app
  exit 1
}

# 패턴 2: 명시적 if
if ! curl -sf "$HEALTH_URL" >/dev/null; then
  echo "헬스체크 실패: $HEALTH_URL" >&2
  exit 2
fi

# 패턴 3: 종료 코드 보존 후 분기
kubectl wait --for=condition=Ready pod/my-app --timeout=60s
rc=$?
case $rc in
  0)  echo "준비 완료" ;;
  1)  echo "조건 불만족" ;;
  *)  echo "타임아웃 또는 오류 (rc=$rc)" ;;
esac

▸ 12.5 trap — 정리(cleanup)와 시그널 처리

trap은 시그널을 받았을 때나 스크립트가 종료될 때 실행할 코드를 등록합니다. 임시 파일 정리, 백그라운드 프로세스 종료 등에 필수입니다.

bash
#!/usr/bin/env bash
set -euo pipefail

TMPDIR=$(mktemp -d)
PORT_FWD_PID=""

cleanup() {
  local rc=$?
  echo "[cleanup] 정리 중 (rc=$rc)"
  
  # 백그라운드 프로세스 종료
  [[ -n "$PORT_FWD_PID" ]] && kill $PORT_FWD_PID 2>/dev/null || true
  
  # 임시 디렉터리 삭제
  [[ -d "$TMPDIR" ]] && rm -rf "$TMPDIR"
  
  exit $rc
}

# EXIT: 어떤 이유로든 스크립트 종료 시
# INT:  Ctrl+C (SIGINT)
# TERM: kill (SIGTERM)
trap cleanup EXIT INT TERM

# 작업 본문
kubectl port-forward svc/api 8080:80 &
PORT_FWD_PID=$!
sleep 2

curl -sf http://localhost:8080/health
🔧 ERR trap으로 디버깅 한 줄 추가
어디서 죽었는지 알고 싶다면 ERR 트랩을 등록하세요: 
trap 'echo "[ERR] line $LINENO: cmd=\"$BASH_COMMAND\" rc=$?" >&2' ERR

▸ 12.6 로깅 헬퍼와 die() 패턴

bash
#!/usr/bin/env bash
set -euo pipefail

log()  { printf '\033[36m[%s]\033[0m %s\n' "$(date +'%H:%M:%S')" "$*"; }
warn() { printf '\033[33m[WARN]\033[0m %s\n'  "$*" >&2; }
die()  { printf '\033[31m[FATAL]\033[0m %s\n' "$*" >&2; exit 1; }

# 사용
[[ -f "$CONFIG" ]] || die "설정 파일이 없습니다: $CONFIG"
log "배포 시작"
kubectl apply -f "$CONFIG" || die "apply 실패"
warn "이전 버전 정리는 5분 뒤"
log "완료"
▸ Check Your Understanding
Q8. set -e가 켜져 있는데도 my_func || handle_error 같은 코드에서 my_func 실패가 스크립트를 종료시키지 않습니다. 왜일까요?
이게 바로 ||를 쓴 목적입니다. set -e는 "실패가 명시적으로 처리되지 않은" 명령에만 발동합니다. ||나 &&로 연결된 명령, if/while의 조건절, !로 반전된 명령은 "실패할 수도 있음"이 코드로 표현된 것이므로 종료시키지 않습니다. 이걸 알아야 의도적으로 실패를 처리하는 패턴(cmd || true로 무시, cmd || die "msg"로 중단)이 자연스럽게 보입니다.
CHAPTER 13

Linux 배포판 비교

같은 "리눅스"라도 컨테이너 이미지 베이스로 무엇을 선택하느냐에 따라 보안성, 이미지 크기, 패키지 가용성, 기본 셸, init 시스템이 전혀 다릅니다. 인프라 엔지니어는 각 배포판의 성격을 알고 워크로드에 맞게 골라 써야 합니다.

▸ 13.1 배포판 계보 — 큰 두 갈래

현대 리눅스 배포판은 크게 Debian 계열과 Red Hat 계열로 나뉘며, 그 외 독자 노선(SUSE, Alpine, Arch 등)이 있습니다.

▸ 13.2 핵심 비교표

배포판 패키지 관리자 기본 셸 init libc 최소 이미지 특징

Debian apt/dpkg bash (sh→dash) systemd glibc ~25MB 안정성 우선, 보수적 패키지

#!/usr/bin/env bash
# gpu-watch.sh — GPU 메모리/사용률 임계치 초과 시 알림
set -euo pipefail

INTERVAL="${INTERVAL:-10}"           # 초
MEM_THRESHOLD="${MEM_THRESHOLD:-90}" # %
UTIL_THRESHOLD="${UTIL_THRESHOLD:-95}"

trap 'echo "[stop] 모니터링 종료"; exit 0' INT TERM

while true; do
  # --query-gpu: name,memory.used,memory.total,utilization.gpu
  while IFS=',' read -r idx name mem_used mem_total util; do
    # 공백 제거
    mem_used=${mem_used// /}
    mem_total=${mem_total// /}
    util=${util// /}

    mem_pct=$(( mem_used * 100 / mem_total ))
    ts=$(date +'%H:%M:%S')

    if (( mem_pct >= MEM_THRESHOLD )); then
      printf '\033[31m[%s] GPU%s MEM ALERT %d%% (%s/%s MiB)\033[0m\n' \
        "$ts" "$idx" "$mem_pct" "$mem_used" "$mem_total"
    elif (( util >= UTIL_THRESHOLD )); then
      printf '\033[33m[%s] GPU%s UTIL %d%%\033[0m\n' "$ts" "$idx" "$util"
    else
      printf '[%s] GPU%s ok (mem=%d%%, util=%d%%)\n' \
        "$ts" "$idx" "$mem_pct" "$util"
    fi
  done < <(nvidia-smi \
      --query-gpu=index,name,memory.used,memory.total,utilization.gpu \
      --format=csv,noheader,nounits)

  sleep "$INTERVAL"
done

#!/usr/bin/env bash
# rotate-logs.sh — N일 이상 된 로그 압축, M일 이상 된 압축본 삭제
set -euo pipefail

LOG_DIR="${LOG_DIR:-/var/log/myapp}"
COMPRESS_AFTER="${COMPRESS_AFTER:-7}"     # 7일 이상 → 압축
DELETE_AFTER="${DELETE_AFTER:-30}"        # 30일 이상 → 삭제

[[ -d "$LOG_DIR" ]] || { echo "로그 디렉터리 없음: $LOG_DIR"; exit 1; }

total_compressed=0
total_deleted=0

# 1단계: 압축 (.log → .log.gz)
while IFS= read -r -d '' f; do
  gzip "$f" && (( ++total_compressed ))
  echo "  압축: $f"
done < <(find "$LOG_DIR" -type f -name "*.log" -mtime +"$COMPRESS_AFTER" -print0)

# 2단계: 오래된 압축본 삭제
while IFS= read -r -d '' f; do
  rm -f "$f" && (( ++total_deleted ))
  echo "  삭제: $f"
done < <(find "$LOG_DIR" -type f -name "*.log.gz" -mtime +"$DELETE_AFTER" -print0)

printf '\n결과: 압축 %d개, 삭제 %d개\n' "$total_compressed" "$total_deleted"
⚠ find -print0과 read -d ''의 짝
파일명에 공백/개행이 있을 가능성이 1%라도 있다면 반드시 find -print0(null 구분)과 read -d '' 짝을 쓰세요. 일반 파이프(find ... | while read)는 공백이 들어간 파일명을 부쉽니다.

#!/usr/bin/env bash
# deploy.sh — 이미지 업데이트, rollout 대기, 실패 시 자동 롤백
set -euo pipefail

NS="${NS:?NS 환경변수 필요}"
DEPLOY="${DEPLOY:?DEPLOY 환경변수 필요}"
IMAGE="${IMAGE:?IMAGE 환경변수 필요}"
CONTAINER="${CONTAINER:-main}"
TIMEOUT="${TIMEOUT:-300s}"

log()  { printf '\033[36m▸\033[0m %s\n' "$*"; }
die()  { printf '\033[31m✗\033[0m %s\n'  "$*" >&2; exit 1; }
ok()   { printf '\033[32m✓\033[0m %s\n'  "$*"; }

rollback() {
  local rc=$?
  if (( rc != 0 )); then
    printf '\033[33m⚠\033[0m 실패 감지, 롤백 중...\n' >&2
    kubectl -n "$NS" rollout undo deployment/"$DEPLOY" || true
  fi
}
trap rollback EXIT

log "이미지 업데이트: $DEPLOY → $IMAGE"
kubectl -n "$NS" set image deployment/"$DEPLOY" "$CONTAINER=$IMAGE" \
  || die "이미지 설정 실패"

log "rollout 대기 (최대 $TIMEOUT)"
kubectl -n "$NS" rollout status deployment/"$DEPLOY" \
  --timeout="$TIMEOUT" || die "rollout 타임아웃"

# 헬스체크 (선택)
if [[ -n "${HEALTH_URL:-}" ]]; then
  log "헬스체크: $HEALTH_URL"
  for i in {1..5}; do
    curl -sf --max-time 5 "$HEALTH_URL" >/dev/null && break
    sleep $(( i * 2 ))     # exponential backoff
    [[ $i -eq 5 ]] && die "헬스체크 5회 연속 실패"
  done
fi

trap - EXIT          # 성공: 롤백 트랩 해제
ok "배포 성공"

# 디버그 출력에서 민감 라인 마스킹
set +x          # 민감 작업 직전에 끔
DB_PASSWORD="$(get_password)"
set -x          # 다시 켜기

▸ 15.4 스크립트 템플릿

새 스크립트를 시작할 때 복사해서 쓰는 표준 골격입니다.

template.sh
#!/usr/bin/env bash
#
# Usage: ./template.sh [-v] [-n NAMESPACE] <target>
#
# Description: 한 줄 요약
#
# Environment:
#   KUBECONFIG  : kubeconfig 경로 (필수)
#   LOG_LEVEL   : info|debug (기본 info)
#
# Exit codes:
#   0: 성공  1: 일반 에러  2: 사용법 오류  127: 의존 명령 없음

set -euo pipefail
IFS=$'\n\t'

### 상수 ----------------------------------------------
readonly SCRIPT_NAME="$(basename "$0")"
readonly SCRIPT_DIR="$(cd "$(dirname "$0")" && pwd)"
readonly VERSION="1.0.0"

### 로깅 ----------------------------------------------
log()   { printf '\033[36m[%s]\033[0m %s\n' "$(date +'%H:%M:%S')" "$*"; }
warn()  { printf '\033[33m[WARN]\033[0m %s\n'  "$*" >&2; }
die()   { printf '\033[31m[FATAL]\033[0m %s\n' "$*" >&2; exit 1; }
debug() { [[ "${LOG_LEVEL:-info}" == "debug" ]] && \
          printf '\033[90m[DEBUG]\033[0m %s\n' "$*" >&2 || true; }

### 사용법 ----------------------------------------------
usage() {
  cat << EOF
$SCRIPT_NAME v$VERSION

사용법:
  $SCRIPT_NAME [-v] [-n NAMESPACE] <target>

옵션:
  -v               verbose 출력
  -n NAMESPACE     대상 네임스페이스 (기본: default)
  -h               도움말
EOF
}

### 정리 ----------------------------------------------
cleanup() {
  local rc=$?
  debug "cleanup (rc=$rc)"
  [[ -n "${TMPDIR_:-}" ]] && rm -rf "$TMPDIR_"
  exit $rc
}
trap cleanup EXIT INT TERM

### 인자 파싱 ----------------------------------------------
NAMESPACE="default"
while getopts ":vn:h" opt; do
  case "$opt" in
    v) LOG_LEVEL=debug ;;
    n) NAMESPACE="$OPTARG" ;;
    h) usage; exit 0 ;;
    \?) die "알 수 없는 옵션: -$OPTARG" ;;
    :) die "-$OPTARG에 인자가 필요합니다" ;;
  esac
done
shift $(( OPTIND - 1 ))

[[ $# -ge 1 ]] || { usage >&2; exit 2; }
TARGET="$1"

### 의존성 검증 ----------------------------------------------
for cmd in kubectl jq; do
  command -v "$cmd" >/dev/null || die "필요한 명령 없음: $cmd"
done

### 본문 ----------------------------------------------
log "시작: target=$TARGET, namespace=$NAMESPACE"
TMPDIR_=$(mktemp -d)

# ... 실제 로직 ...

log "완료"

▸ 15.5 최종 체크리스트

📋 프로덕션 스크립트 체크리스트
  • ✅ #!/usr/bin/env bash로 시작 (이식성)
  • ✅ set -euo pipefail 설정
  • ✅ ShellCheck 통과 (CI에서 자동 실행)
  • ✅ 모든 변수 확장에 쌍따옴표
  • ✅ 필수 환경변수는 ${VAR:?}로 검증
  • ✅ trap cleanup EXIT으로 임시 리소스 정리
  • ✅ 사용법(usage) 함수와 -h 옵션
  • ✅ stderr와 stdout 구분 (사람용 로그는 stderr, 데이터는 stdout)
  • ✅ 종료 코드 의미 부여 (0 성공, 1 일반 실패, 2 사용법 오류 등)
  • ✅ 의존 명령 사전 확인 (command -v)
  • ✅ 비밀 정보는 환경변수/secret에서, 로그에 마스킹
  • ✅ 멱등성(idempotency) 고려 — 두 번 실행해도 안전한가
▸ Check Your Understanding
Q9. local x=$(some_command_that_fails) 한 줄짜리 코드에 ShellCheck SC2155 경고가 뜨는 이유는?
local(또는 declare, readonly, export)와 명령 치환을 한 줄에 같이 쓰면 종료 코드가 손실됩니다. 이 줄에서 셸이 보는 명령은 local이며, local의 종료 코드는 거의 항상 0입니다. some_command_that_fails가 실패해도 set -e가 발동되지 않고, 변수에는 빈 문자열이 들어가 이후 로직이 조용히 잘못된 길로 갈 수 있습니다. 안전한 패턴: 
local x
x=$(some_command_that_fails)   # 이제 set -e가 정상 동작
