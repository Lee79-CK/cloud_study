Lecture 01 · Foundations

파인튜닝이란 무엇인가

LoRA를 이해하려면 먼저 "파인튜닝"이 무엇인지 알아야 한다. 어렵지 않다. 그저 이미 똑똑한 AI를 우리 일에 맞게 살짝 가르치는 것이다.

AI 모델은 거대한 숫자 덩어리

ChatGPT, Claude, Llama 같은 LLM의 내부를 들여다보면 결국 가중치(weights)라고 불리는 거대한 숫자 표(행렬)들의 모음입니다. 모델이 7B(70억) 파라미터라는 건, 이 숫자들이 70억 개 있다는 뜻이죠. 모델은 이 숫자들로 입력 텍스트를 변환해 다음 단어를 예측합니다.

가중치를 거대한 악기의 70억 개 손잡이라고 상상해보세요. 각 손잡이를 어떻게 돌리느냐에 따라 같은 입력에도 다른 음악이 나옵니다. 학습이란 결국 이 손잡이들을 적절한 위치에 맞추는 과정입니다.

사전학습 vs 파인튜닝

모델을 만드는 과정은 보통 두 단계로 나뉩니다.

Stage 1 · Pre-training

사전학습 — 일반 지식 습득

인터넷의 방대한 텍스트(수 TB)를 보면서 언어의 패턴, 세상의 상식, 코드, 수학 등을 익힙니다. 수백 ~ 수천만 달러가 들고 수 주~수 개월이 걸립니다. 보통 우리는 이 단계의 결과물 (예: Qwen2.5-7B, Llama 3.1)을 가져다 씁니다.

Stage 2 · Fine-tuning

파인튜닝 — 특정 작업 적응

이미 똑똑한 모델에게 "이제 KT 사내 매뉴얼 톤으로 답해줘", "의료 상담만 해줘"와 같이 좁은 영역으로 특화시키는 단계입니다. 데이터는 수천~수만 건, 시간은 수 시간~수 일이면 충분합니다.

왜 파인튜닝이 필요한가

"그냥 프롬프트로 시키면 안 되나?"라는 질문을 자연스럽게 하게 됩니다. 가능은 하지만 다음과 같은 한계가 있습니다.

  • 긴 시스템 프롬프트는 비싸다 — 매 호출마다 같은 지시문을 토큰 비용으로 지불해야 합니다.
  • 스타일/톤 학습은 프롬프트로 어렵다 — "친근하게 말해"라고 적어도 모델의 본래 톤은 잘 안 바뀝니다.
  • 긴 도메인 지식은 컨텍스트에 안 들어간다 — 매뉴얼 100페이지를 매번 프롬프트에 붙일 수는 없습니다.
  • 일관성이 떨어진다 — 프롬프트로는 매번 다르게 답할 수 있습니다.

파인튜닝은 이런 도메인 지식과 스타일을 모델의 가중치 자체에 새겨넣어 앞으로의 모든 호출에서 자동으로 반영되도록 만듭니다.

언제 파인튜닝이 필요한가

① 모델이 특정 출력 형식을 일관되게 따라야 할 때 (예: 특정 JSON 스키마) ② 도메인 특화 어휘·말투가 중요할 때 (사내 용어, 의료, 법률) ③ 짧은 응답으로 비용을 줄여야 할 때 ④ 모델의 기본 능력을 유지하면서 좁은 주제에 더 정확해야 할 때

강의의 목적지

이 강의가 끝나면 여러분은 다음 질문에 자신 있게 답할 수 있습니다.

  • 왜 Full Fine-tuning은 비싸고, LoRA는 싸면서도 비슷한 성능을 내는가?
  • LoRA의 rank, alpha, target_modules는 무엇이고 어떻게 정하는가?
  • QLoRA는 LoRA와 무엇이 다른가? 언제 써야 하는가?
  • 실제로 Hugging Face PEFT 라이브러리로 어떻게 학습하고 배포하는가?
  • 파인튜닝이 실패할 때 어떤 신호를 봐야 하는가?

Lecture 02 · The Problem

Full Fine-tuning의 한계

가장 단순한 파인튜닝 방법은 모델의 모든 가중치를 다시 조금씩 학습시키는 것이다. 그런데 이게 왜 그렇게 큰 문제가 되는지, 숫자로 직접 보자.

7B 모델의 메모리 청구서

7B 파라미터 모델을 Full Fine-tuning할 때 GPU에 올려야 할 것들을 차례로 나열해보겠습니다. 모든 숫자는 fp16/bf16(반정밀도) 기준이며, 1 파라미터당 2바이트입니다.

구성용도크기계산
모델 가중치 W예측에 사용14 GB7B × 2 byte
그라디언트 ∇W역전파에서 계산14 GB가중치와 동일 크기
옵티마이저 상태 (Adam: m, v)모멘텀과 분산28 GBfp32로 보관 시 가중치의 4배

합계약 76 ~ 84 GB
현실 점검

7B 모델 Full FT는 일반적인 24GB GPU(RTX 4090, A10) 한 장으로는 불가능합니다. A100 80GB가 있어야 겨우 가능하고, 13B부터는 다중 GPU + 분산 학습이 필요합니다. 70B 모델은 H100 8장으로도 빠듯합니다.

메모리만 문제가 아니다

① 저장 공간 폭발

Full FT를 하면 매 작업마다 별도의 14GB 모델 파일이 생깁니다. 사내에 10개 부서가 각자 모델을 파인튜닝하면 140GB. 100개 작업이면 1.4TB. 모델 레지스트리 비용과 배포 시간이 모두 폭증합니다.

② 학습 시간

모든 70억 개 파라미터의 그라디언트를 계산하고 업데이트하는 데 시간이 걸립니다. 같은 데이터셋이라도 LoRA 대비 1.5~2배 느립니다.

③ 파국적 망각 (Catastrophic Forgetting)

가중치를 전부 움직이다 보니 원래 모델이 잘하던 것까지 망가지는 경우가 흔합니다. 영어 능력을 한국어 데이터로 파인튜닝하다 영어 성능이 떨어지는 식입니다.

우리는 더 똑똑한 방법이 필요하다

문제의 본질은 명확합니다. "좁은 작업에 적응하기 위해 정말로 70억 개 손잡이를 모두 다시 만져야 하는가?" 직관적으로도 그럴 필요는 없을 것 같습니다. 이미 똑똑한 모델인데, 그 위에 작은 변형만 가하면 되지 않을까요?

2021년 Microsoft 연구팀이 발표한 LoRA 논문은 정확히 이 직관에 수학적 답을 제시했습니다. 다음 장에서 그 답을 살펴봅시다.

Checkpoint 02
Q. 7B 모델의 Full FT에서 가장 큰 메모리 차지 항목은?
  • 모델 가중치 W (14GB)
  • 그라디언트 ∇W (14GB)
  • 옵티마이저 상태 m, v (28GB)
  • 활성화 (~20GB)
정답: ③ — Adam 옵티마이저는 각 파라미터에 대해 1차 모멘텀(m)과 2차 모멘텀(v)을 fp32로 저장합니다. 한 파라미터당 8바이트라서 가중치(2바이트)의 4배인 28GB를 차지합니다. LoRA가 절약하는 가장 큰 부분이 바로 이것입니다.

Lecture 03 · Core Intuition

LoRA의 핵심 직관

한 줄로 요약하면 이렇다. "큰 변화는 사실 작은 방향에서 일어난다." 이 한 문장을 이해하면 LoRA의 70%는 끝난 셈이다.

저랭크 가설 (Low-Rank Hypothesis)

LoRA 논문의 출발점은 다음과 같은 관찰입니다. 사전학습된 모델을 특정 작업에 맞게 파인튜닝할 때, 가중치의 변화량 ΔW는 실제로는 매우 단순한 구조를 가진다는 것입니다. 수학적으로 표현하면 ΔW의 "랭크(rank)"가 매우 낮다는 가설입니다.

"랭크"가 낮다는 게 뭔가요?

랭크는 행렬에서 독립적인 정보의 양이라고 생각하면 됩니다. 100×100 행렬이라도 실제로 의미 있는 정보는 5차원에 불과할 수 있습니다.

음악 비유. 한 곡의 노래는 수십 만 개의 샘플 데이터로 구성되지만, 그 노래의 본질은 몇 개의 멜로디 라인과 화성으로 압축할 수 있습니다. LoRA가 말하는 것도 비슷합니다. 수십억 개의 가중치 변화도 사실은 몇 백 개의 "방향성"으로 표현 가능하다.

행렬 분해 비유

LoRA의 핵심 트릭은 큰 행렬을 두 개의 작은 행렬로 분해해서 표현하는 것입니다.

Figure · 큰 행렬을 두 개의 가는 행렬로
16,777,216개 파라미터를 65,536개로 — 99.6% 감소. 그래도 같은 효과를 낸다는 것이 LoRA의 마법입니다.

왜 이 분해가 작동하는가

직관적으로 말하면, B는 입력을 좁은 통로(r차원)로 압축하고, A는 그 압축된 표현을 다시 원래 차원으로 펼칩니다. 이 과정에서 모델이 표현할 수 있는 변화는 r차원에 사는 변화로 제한됩니다. 그런데 위에서 말한 "저랭크 가설"이 맞다면, 대부분의 유용한 변화는 실제로 r차원으로 충분히 표현됩니다.

한 번 더 정리

LoRA = "ΔW 전체를 학습하는 대신, ΔW를 만들어낼 수 있는 작은 두 행렬 A, B만 학습한다"

가장 큰 이점

이 단순한 트릭은 다음과 같은 큰 이점을 줍니다.

  • 학습 파라미터 99% 감소 — 7B 모델에서 보통 7~70M개의 LoRA 파라미터만 학습합니다.
  • 옵티마이저 상태 99% 감소 — Adam의 m, v는 학습 가능한 파라미터에만 필요합니다. 메모리 절약의 진짜 원천.
  • 저장 공간 99% 감소 — 7B 모델당 14GB 대신 LoRA 어댑터 하나당 30~200MB.
  • 원본 보존 — 원본 가중치는 동결되므로 파국적 망각이 거의 없습니다.
  • 여러 작업을 한 모델로 — 같은 베이스 모델 + 작업별 어댑터 조합이 가능합니다.

Lecture 04 · The Math, Gently

수학을 천천히

수식 한두 개를 살펴보자. 절대 어렵지 않다. 곱셈과 덧셈만 알면 된다. 하지만 이 수식이 코드에서 어떻게 등장하는지 안다면 LoRA를 쓸 때 훨씬 자신감이 붙는다.

파인튜닝의 일반 표현

모델의 어떤 한 층의 가중치를 W라고 합시다. 파인튜닝은 이 W를 W' 으로 바꾸는 일입니다. 수식으로는 이렇게 씁니다.

W' = W + ΔW
"새 가중치 = 원래 가중치 + 변화량"

Full Fine-tuning은 이 ΔW 전체를 직접 학습합니다. 만약 W가 4096×4096 크기라면 ΔW도 4096×4096 = 16,777,216개의 숫자를 학습해야 합니다. 한 층만 그렇고, 모델에는 이런 층이 수백 개 있죠.

LoRA의 표현

LoRA는 ΔW를 직접 학습하는 대신, 두 작은 행렬의 곱으로 표현합니다.

W' = W + B · A
B는 (d × r) · A는 (r × k) · 보통 r ≪ d, k

여기서 핵심은 r입니다. r은 "rank"를 의미하며, 보통 4, 8, 16, 32, 64 같은 작은 값을 씁니다. d나 k가 4096이라면 r은 그 1/100 이하입니다.

구체적인 숫자로 비교

방식학습 가능 파라미터비고
Full FT (W가 4096×4096)4096 × 4096 = 16,777,216약 16.8M
LoRA r=84096×8 + 8×4096 = 65,5360.39% (256배 감소)
LoRA r=164096×16 + 16×4096 = 131,0720.78%
LoRA r=644096×64 + 64×4096 = 524,2883.13%

스케일링 — alpha의 역할

실제 LoRA에는 alpha라는 또 하나의 하이퍼파라미터가 있습니다. 진짜 사용되는 수식은 다음과 같습니다.

W' = W + (α / r) · B · A
α는 LoRA 출력의 "강도"를 조절

직관적으로 보면 α/r는 LoRA 분기가 만들어내는 변화에 곱해지는 강도 조절기입니다. α를 높이면 LoRA의 영향력이 커지고, 낮추면 작아집니다. 분모에 r이 있는 이유는, rank를 바꿀 때마다 자동으로 비례 보정이 되도록 하기 위해서입니다.

실전 규칙

현장에서 가장 흔한 조합은 alpha = 2 × rank입니다. 예: r=8 → α=16, r=16 → α=32. 이 비율로 두면 r을 바꿔도 학습 동학이 비슷하게 유지됩니다.

초기화 트릭

한 가지 중요한 디테일이 있습니다. 학습 시작 시 LoRA 분기는 0의 출력을 내야 합니다. 그래야 학습 초기에 모델이 갑자기 망가지지 않고, 원본 모델 그대로의 동작에서 시작할 수 있습니다. 이를 위해 LoRA는 다음과 같이 초기화합니다.

  • A는 가우시안(정규분포)으로 초기화
  • B는 0으로 초기화
  • 따라서 학습 시작 시 B·A = 0 (어떤 A든 0과 곱하면 0)
  • 학습이 진행되면서 B가 0에서 점차 의미 있는 값으로 이동

이 작은 디테일 덕분에 LoRA는 안전하게 출발해서 천천히 적응하는 자연스러운 학습 곡선을 그립니다.

Checkpoint 04
Q. 4096×4096 가중치 행렬을 LoRA r=16으로 학습할 때, 학습 가능 파라미터 수는?
  • 4096 × 4096 = 16,777,216
  • 4096 × 16 + 16 × 4096 = 131,072
  • 4096 × 16 = 65,536
  • 16 × 16 = 256
정답: ② — LoRA는 두 행렬 B(4096×16)와 A(16×4096)를 학습합니다. 파라미터 수는 두 행렬의 합, 즉 4096×16 + 16×4096 = 131,072입니다. 원본 16.7M의 0.78%에 불과합니다.

다음 시나리오들을 직접 슬라이더로 만들어보세요. 각 경우 GPU 메모리가 어떻게 달라지는지 관찰합니다.

  1. "24GB GPU에서 7B를 어떻게 굴릴까?" — 모델: 7B, 방식을 Full → LoRA → QLoRA로 바꿔보세요. Full은 ~80GB로 불가능, LoRA는 ~30GB로 빡빡, QLoRA는 ~12GB로 여유롭습니다.
  2. "rank를 키우면 얼마나 더 무거워지나?" — 7B + LoRA 고정. r=8 → 16 → 32 → 64로 슬라이드. 변화가 크지 않다는 점을 확인하세요. 대부분의 메모리는 모델 가중치 자체입니다.
  3. "target_modules를 늘리면?" — q,k,v,o만 → gate, up, down까지 추가. 학습 파라미터가 약 2.5배 늘어나지만, 메모리는 크게 안 늘어납니다. 어댑터 파일은 커집니다.
  4. "70B를 LoRA로?" — 70B 선택. LoRA여도 가중치만 140GB. QLoRA로 가야 80GB GPU에 겨우 들어갑니다.
관찰 포인트

위 계산기를 만지다 보면 "방식 선택"이 메모리를 결정하고, "rank"는 미세한 차이만 만든다는 점이 보입니다. 이것이 LoRA가 강력한 이유입니다 — rank를 충분히 크게 가져가도 메모리 부담은 거의 없으니, 품질을 위해 r을 키우는 데 주저할 필요가 없습니다.

Lecture 06 · Hyperparameters

하이퍼파라미터 가이드

LoRA를 시작할 때 마주치는 첫 질문은 항상 같다. "숫자들을 뭘로 정해야 해요?" 실전에서 자주 쓰는 값과 그 이유를 정리한다.

핵심 4개 하이퍼파라미터

① rank (r) — 어댑터의 표현력

가장 중요하고도 가장 자주 잘못 설정하는 값입니다. r은 어댑터가 표현할 수 있는 변화의 풍부함을 결정합니다.

r 범위적합한 작업비고
4 ~ 8스타일 변경, 톤 조정, 간단한 분류매우 가벼움. 빠른 실험용.
16 (기본값)대부분의 instruction tuning품질/비용의 스위트 스팟.
32 ~ 64도메인 특화, 복잡한 추론충분한 데이터 필요.
128 +거의 Full FT 수준이 정도면 그냥 Full FT를 고려.

② alpha (α) — 어댑터의 강도

앞에서 본 (α/r) 스케일링 인자입니다. 직관적으로 "LoRA 분기가 얼마나 큰 영향을 줄지"를 결정합니다.

  • 가장 흔한 설정: α = 2r (예: r=16 → α=32)
  • 안전한 설정: α = r (영향력 중립)
  • 과감한 설정: α = 4r (강한 도메인 적응)

③ target_modules — 어디에 어댑터를 붙일까

Transformer 한 층에는 여러 가중치 행렬이 있습니다. 그중 어디에 LoRA를 붙일지 선택합니다.

모듈역할우선순위
q_proj, k_proj, v_proj어텐션의 Query, Key, Value 투영★★★ 필수
o_proj어텐션 출력 투영★★★ 권장
gate_proj, up_proj, down_projFFN(피드포워드) 층★★ 품질 향상 시
embed_tokens, lm_head임베딩, 출력 헤드★ 도메인이 매우 다를 때만
실전 시작 세트

처음이라면 target_modules=["q_proj", "k_proj", "v_proj", "o_proj"]로 시작하세요. 품질이 부족하면 FFN 모듈(gate_proj, up_proj, down_proj)을 추가합니다. 이 7개 모듈을 모두 포함하는 것은 LoRA의 "풀 커버리지" 표준 설정입니다.

④ lora_dropout — 정규화

LoRA 분기에 적용되는 드롭아웃 비율입니다. 데이터셋이 작거나 과적합 신호가 있을 때 사용합니다.

  • 데이터 풍부 (수만 건+): 0.0 ~ 0.05
  • 데이터 보통 (수천 건): 0.05 ~ 0.1
  • 데이터 부족 (수백 건): 0.1 ~ 0.2

학습 자체의 하이퍼파라미터

학습률 (learning rate)

Full FT보다 훨씬 큰 학습률을 쓸 수 있습니다. LoRA는 학습 파라미터가 적고 초기화가 안전하기 때문에 빠르게 적응할 수 있습니다.

  • Full FT 일반: 1e-5 ~ 5e-5
  • LoRA 일반: 1e-4 ~ 3e-4 (10배 큼)
  • QLoRA 일반: 2e-4 (논문 권장)

배치 크기 & gradient accumulation

실제 batch size = per_device_batch × accumulation × num_gpus입니다. 효과적인 batch size는 32~128이 흔합니다. GPU 메모리에 안 들어가면 per_device_batch=2, accumulation=16처럼 누적으로 만듭니다.

에폭 수

instruction tuning은 보통 2~5 에폭이면 충분합니다. 그 이상은 과적합 위험이 큽니다. 도메인 특화는 데이터에 따라 1~3 에폭.

한눈에 보는 시작 템플릿

Python · 안전한 시작 설정
from peft import LoraConfig

config = LoraConfig(
    r=16,                     # 시작은 16. 부족하면 32, 64로
    lora_alpha=32,            # 보통 r의 2배
    lora_dropout=0.05,        # 가벼운 드롭아웃
    target_modules=["q_proj", "k_proj", "v_proj", "o_proj"],
    bias="none",              # bias는 학습하지 않음 (표준)
    task_type="CAUSAL_LM",    # 생성 모델
)

# 학습 시
training_args = TrainingArguments(
    learning_rate=2e-4,        # LoRA 표준
    num_train_epochs=3,
    per_device_train_batch_size=2,
    gradient_accumulation_steps=8,
    warmup_ratio=0.03,
    lr_scheduler_type="cosine",
    bf16=True,                 # DGX Spark는 bf16 추천
)

Lecture 07 · QLoRA

QLoRA · 한 번 더 압축

LoRA가 충분히 가벼웠는데, QLoRA는 한 단계 더 나아갔다. 베이스 모델 자체를 4-bit로 압축한 채로 그 위에 LoRA를 학습하는 마법 같은 기법이다.

아이디어

LoRA 학습 시 베이스 모델 W는 동결되어 있습니다. 즉, 정확한 값으로 보관할 필요 없이 "읽기 전용"으로 쓸 수 있을 만큼만 정확하면 됩니다. QLoRA는 이 점을 이용해 베이스 모델을 4비트로 양자화(quantize)합니다.

비트수와 메모리

형식1 파라미터당7B 모델
fp324 byte28 GB
fp16 / bf162 byte14 GB
int81 byte7 GB
nf4 (QLoRA)0.5 byte3.5 GB

QLoRA의 3가지 핵심 트릭

NF4 (NormalFloat 4-bit)

일반적인 int4가 아닌, 정규분포에 최적화된 새로운 4-bit 자료형을 사용합니다. 신경망 가중치는 정규분포에 가깝다는 사실에 기반해 정보 손실을 최소화합니다.

Double Quantization

양자화 자체에 사용되는 상수(scale)들도 다시 양자화합니다. 추가로 0.4비트/파라미터 절약. 7B 기준 약 350MB의 추가 절약입니다.

Paged Optimizer

옵티마이저 상태를 GPU 메모리가 부족할 때 CPU로 자동 페이징합니다. 메모리 스파이크에 의한 OOM(Out-Of-Memory)을 방지합니다.

LoRA vs QLoRA 어떻게 선택하는가

LoRA를 쓰자

충분한 GPU 메모리가 있을 때

  • 7B 모델 학습 시 32GB+ GPU
  • 학습 속도가 중요한 경우 (QLoRA보다 1.3~1.5배 빠름)
  • 최대 품질이 필요한 production 모델
  • DGX Spark 128GB는 LoRA에 이상적
QLoRA를 쓰자

메모리가 제한적일 때

  • 16GB / 24GB GPU에서 7B 학습
  • 13B 이상 큰 모델로 가야 할 때
  • 여러 실험을 동시에 돌려야 할 때
  • 품질 손실이 거의 없음 (LoRA의 98~99%)

QLoRA 코드 차이점

Hugging Face Transformers + bitsandbytes 조합으로 LoRA를 QLoRA로 바꾸는 데 실제로는 코드 5줄만 더 추가하면 됩니다.

Python · QLoRA 추가 부분
from transformers import AutoModelForCausalLM, BitsAndBytesConfig
from peft import prepare_model_for_kbit_training, LoraConfig, get_peft_model

# ① 양자화 설정
bnb = BitsAndBytesConfig(
    load_in_4bit=True,
    bnb_4bit_quant_type="nf4",             # NF4 사용
    bnb_4bit_compute_dtype="bfloat16",     # 연산은 bf16으로
    bnb_4bit_use_double_quant=True,        # Double Quantization
)

# ② 4-bit로 모델 로드
model = AutoModelForCausalLM.from_pretrained(
    "Qwen/Qwen2.5-7B-Instruct",
    quantization_config=bnb,
    device_map="auto",
)

# ③ k-bit 학습 준비 (gradient checkpointing 등)
model = prepare_model_for_kbit_training(model)

# ④ LoRA 추가 (이후는 일반 LoRA와 동일)
lora_config = LoraConfig(r=16, lora_alpha=32, ...)
model = get_peft_model(model, lora_config)
DGX Spark에서의 선택

DGX Spark의 GB10은 128GB Unified Memory를 가지므로, 7B~13B 모델은 QLoRA 없이 LoRA만으로 충분히 빠르고 안정적으로 학습할 수 있습니다. QLoRA의 진가는 30B+ 모델이나 여러 실험을 한 GPU에 동시에 굴릴 때 드러납니다.

from datasets import load_dataset

# 사내 데이터든 공개 데이터든 형식만 맞추면 됨
ds = load_dataset("json", data_files="train.jsonl", split="train")

# Qwen 채팅 템플릿에 맞는 텍스트로 변환
def format_example(ex):
    msgs = [
        {"role": "user", "content": ex["instruction"]},
        {"role": "assistant", "content": ex["output"]},
    ]
    return {"text": tokenizer.apply_chat_template(msgs, tokenize=False)}

ds = ds.map(format_example)
print(ds[0]["text"][:200])

Step 3 ~ 5 — 토크나이저, 모델, LoRA

Python · 모델과 LoRA 준비
from transformers import AutoTokenizer, AutoModelForCausalLM
from peft import LoraConfig, get_peft_model

MODEL_ID = "Qwen/Qwen2.5-1.5B-Instruct"

# 토크나이저
tokenizer = AutoTokenizer.from_pretrained(MODEL_ID)
if tokenizer.pad_token is None:
    tokenizer.pad_token = tokenizer.eos_token

# 베이스 모델 (bf16, DGX Spark 권장)
model = AutoModelForCausalLM.from_pretrained(
    MODEL_ID,
    torch_dtype="bfloat16",
    device_map="auto",
)

# LoRA 설정
lora_config = LoraConfig(
    r=16,
    lora_alpha=32,
    lora_dropout=0.05,
    target_modules=["q_proj", "k_proj", "v_proj", "o_proj"],
    bias="none",
    task_type="CAUSAL_LM",
)

# LoRA 어댑터를 모델에 부착
model = get_peft_model(model, lora_config)
model.print_trainable_parameters()
# 출력 예: trainable params: 4,194,304 || all params: 1,547,899,392 || trainable%: 0.27
잠깐 · print_trainable_parameters()의 의미

이 한 줄은 LoRA를 처음 만질 때 가장 친한 친구입니다. 학습 가능한 파라미터의 비율(여기서는 0.27%)이 기대 범위(0.1~1%)에 들어 있는지 확인합니다. 만약 50% 같은 큰 값이 나온다면 target_modules 설정이 잘못되어 LoRA가 제대로 안 붙은 신호입니다.

Step 6 — 학습

Python · Trainer 설정과 실행
from transformers import TrainingArguments
from trl import SFTTrainer

args = TrainingArguments(
    output_dir="./qwen-lora-out",
    num_train_epochs=3,
    per_device_train_batch_size=4,
    gradient_accumulation_steps=4,    # 효과적 batch = 16
    learning_rate=2e-4,
    lr_scheduler_type="cosine",
    warmup_ratio=0.03,
    bf16=True,
    logging_steps=10,
    save_strategy="epoch",
    save_total_limit=2,
    report_to="none",                 # MLflow 쓰면 "mlflow"
)

trainer = SFTTrainer(
    model=model,
    tokenizer=tokenizer,
    args=args,
    train_dataset=ds,
    dataset_text_field="text",
    max_seq_length=2048,
)

trainer.train()
# DGX Spark에서 1.5B + 5000 sample × 3 epoch ≈ 약 20분

from peft import PeftModel
from transformers import AutoModelForCausalLM, AutoTokenizer

# 베이스 모델 로드
base = AutoModelForCausalLM.from_pretrained(
    "Qwen/Qwen2.5-1.5B-Instruct",
    torch_dtype="bfloat16", device_map="auto",
)
tok = AutoTokenizer.from_pretrained("./qwen-lora-final")

# LoRA 어댑터 부착
model = PeftModel.from_pretrained(base, "./qwen-lora-final")
model.eval()

# 생성
prompt = tok.apply_chat_template(
    [{"role": "user", "content": "LLMOps를 한 줄로 설명해줘."}],
    tokenize=False, add_generation_prompt=True,
)
inputs = tok(prompt, return_tensors="pt").to(model.device)
out = model.generate(**inputs, max_new_tokens=128, temperature=0.7)
print(tok.decode(out[0], skip_special_tokens=True))

Lecture 09 · Pitfalls

자주 하는 실수

나도 처음에는 다 겪었던 실수들이다. 이 한 페이지를 미리 읽어두면 며칠의 디버깅을 아낄 수 있다.

증상별 진단표

증상가능 원인해결
학습 loss가 안 떨어짐 학습률 너무 작음 / target_modules 잘못됨 lr를 2e-4로, target_modules에 q,k,v,o 모두 포함
학습 loss가 폭발(NaN) 학습률 너무 큼 / fp16 오버플로우 lr를 1e-4로 낮춤. fp16 → bf16으로 변경
학습 loss는 떨어지나 검증 성능 나쁨 과적합 (데이터 부족 / 에폭 과다) 에폭 1~2로 줄이기, dropout 0.1로
모델이 이상한 답을 생성 채팅 템플릿 미적용 / 라벨 마스킹 오류 apply_chat_template 사용 확인
OOM (메모리 부족) 배치 크기 너무 큼 / Full FT로 돌고 있음 per_device_batch=1, accumulation 사용. 또는 QLoRA 전환
trainable %가 50%+ LoRA가 제대로 부착 안 됨 get_peft_model() 호출 확인, target_modules 이름 검증

10가지 흔한 함정

  1. 채팅 템플릿을 안 씀 — Qwen, Llama 등은 각자 고유한 채팅 포맷이 있습니다. apply_chat_template을 안 쓰고 raw 문자열을 넣으면 모델이 이상해집니다.
  2. target_modules 이름 오타 — Llama는 q_proj, GPT-2는 c_attn입니다. 모델별로 다릅니다. print(model)로 실제 이름 확인.
  3. 라벨 마스킹 미설정 — instruction 부분의 토큰은 학습에서 제외해야 합니다. SFTTrainer는 자동으로 처리하지만, 직접 Trainer를 쓸 때는 수동 작업.
  4. 너무 작은 데이터 — instruction tuning은 최소 1000건은 있어야 의미 있는 학습이 됩니다. 500건 미만이면 프롬프트 엔지니어링이나 RAG가 더 나을 수 있습니다.
  5. 검증셋 없이 돌림 — train loss만 보면 과적합을 놓칩니다. 항상 5~10%는 eval로 분리.
  6. 저장 전 모델 머지 — 어댑터를 베이스에 머지한 채로 저장하면 LoRA의 장점이 사라집니다. model.save_pretrained()는 어댑터만 저장하는 게 기본.
  7. 학습 후 베이스 모델 버전 변경 — 어댑터는 특정 베이스 모델 버전에 묶여 있습니다. 베이스 업그레이드 시 재학습 필요.
  8. 여러 데이터셋을 그냥 합침 — 도메인 비중이 다르면 한쪽에 편향됩니다. 가중 샘플링이나 비율 맞추기 필요.
  9. 너무 큰 max_seq_length — 4096이 필요 없는데 4096으로 두면 메모리만 낭비. 데이터의 실제 길이 분포를 먼저 분석.
  10. pad_token을 안 정함 — Qwen, Llama 등 일부 모델은 pad_token이 없어 직접 설정해야 합니다. 이걸 빠뜨리면 배치 학습이 깨집니다.

디버깅 워크플로

의심될 때 순서대로

① print(model)로 모듈 이름 확인 → ② print_trainable_parameters()로 비율 확인 → ③ 첫 배치의 입력 텍스트 출력해서 채팅 포맷 확인 → ④ logging_steps=1로 첫 10 step의 loss 패턴 관찰 → ⑤ 작은 데이터(100건)로 num_epochs=10 돌려서 overfitting 가능한지 확인 (학습 동학 자체가 정상인지 확인하는 sanity check)

Checkpoint 09
Q. 학습 loss는 잘 떨어지는데 모델이 생성하는 답변이 이상합니다. 가장 먼저 의심해야 할 것은?
  • 학습률이 너무 큼
  • 채팅 템플릿이 적용되지 않음
  • rank가 너무 작음
  • 옵티마이저가 잘못됨
정답: ② — loss가 떨어진다는 건 모델이 주어진 텍스트 분포는 잘 학습했다는 뜻입니다. 그런데 그 텍스트가 채팅 템플릿(예: <|im_start|>user...<|im_end|>) 없이 생짜로 들어갔다면, 모델은 추론 시 채팅 템플릿이 들어와도 그것과 다른 분포를 학습한 셈이라 이상하게 답변합니다.

Lecture 10 · Deployment

배포 전략

잘 학습된 LoRA 어댑터를 어떻게 운영 환경에 올릴까. 두 가지 큰 선택지가 있고, 각각 장단점이 명확하다.

전략 A · 어댑터 머지(Merge)

학습된 LoRA 가중치를 베이스 모델에 합쳐서 단일 모델로 만드는 방식입니다. 일단 머지가 끝나면 일반 모델과 똑같이 다룰 수 있습니다.

Python · 머지 코드
from peft import PeftModel
from transformers import AutoModelForCausalLM

base = AutoModelForCausalLM.from_pretrained("Qwen/Qwen2.5-7B-Instruct",
                                            torch_dtype="bfloat16")
model = PeftModel.from_pretrained(base, "./my-lora-adapter")

# 머지 — W' = W + (α/r) · B · A 를 실제로 계산해서 W에 덮어씀
merged = model.merge_and_unload()

# 머지된 모델 저장 (이제 일반 모델, 어댑터 정보 사라짐)
merged.save_pretrained("./qwen-merged", safe_serialization=True)

전략 B · 동적 로딩(Dynamic Loading)

베이스 모델은 한 번만 메모리에 올리고, 요청에 따라 여러 어댑터를 그때그때 활성화하는 방식입니다. 멀티테넌트 서빙에 강력합니다.

Python · 여러 어댑터 동시 보유
from peft import PeftModel

# 베이스 + 첫 어댑터 로드
model = PeftModel.from_pretrained(base, "./adapter-customer-support",
                                  adapter_name="support")

# 추가 어댑터들
model.load_adapter("./adapter-tech-docs", adapter_name="techdoc")
model.load_adapter("./adapter-sales", adapter_name="sales")

# 요청에 따라 활성 어댑터 전환
model.set_adapter("techdoc")
output = model.generate(...)

model.set_adapter("support")
output = model.generate(...)

두 전략 비교

vLLM에서 LoRA 서빙

vLLM은 --enable-lora 플래그로 동적 어댑터 로딩을 지원합니다. 한 베이스 모델 위에 여러 LoRA를 동시에 서빙할 수 있어 KT처럼 다부서 환경에서 강력합니다.

Bash · vLLM with LoRA
python -m vllm.entrypoints.openai.api_server \
  --model Qwen/Qwen2.5-7B-Instruct \
  --enable-lora \
  --lora-modules \
      support=/adapters/customer-support \
      techdoc=/adapters/tech-docs \
      sales=/adapters/sales \
  --max-loras 3 \
  --max-lora-rank 32

요청 시 model 필드에 어댑터 이름을 적으면 해당 어댑터로 추론합니다.

Bash · 어댑터 선택 요청
curl http://localhost:8000/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "techdoc",                  // 어댑터 이름
    "messages": [
      {"role": "user", "content": "Helm 차트 작성법 알려줘"}
    ]
  }'
실전 권장

① 어댑터가 1~2개라면 머지가 단순하고 빠릅니다. ② 어댑터가 3개 이상이거나 부서별/팀별로 늘어날 예정이라면 vLLM의 multi-LoRA를 권장합니다. ③ 머지/동적은 한 번 정하면 끝이 아니라, 실험은 동적으로 → production은 머지로 전환하는 패턴이 흔합니다.

Lecture 11 · Wrap-up

정리와 다음 단계

11개 챕터를 통해 LoRA의 직관·수학·하이퍼파라미터·코드·배포까지 한 바퀴 돌았다. 머릿속에 단단히 남겨야 할 것을 정리한다.

8가지 핵심 기억

기억 1
저랭크 가설
파인튜닝의 가중치 변화는 본질적으로 저차원 구조를 가진다.
기억 2
B·A 분해
큰 ΔW를 두 작은 행렬의 곱으로 표현 → 99% 파라미터 감소.
기억 3
메모리는 옵티마이저가 차지
Full FT의 진짜 부담은 가중치가 아니라 옵티마이저 상태(28GB).
기억 4
r=16, α=32, lr=2e-4
모를 때의 안전한 출발점. 90%는 이 설정으로 충분.
기억 5
target_modules가 핵심
최소 q,k,v,o. 부족하면 FFN(gate, up, down) 추가.
기억 6
QLoRA = LoRA + 4-bit
메모리 부족할 때 첫 번째 카드. 품질 손실은 거의 없음.
기억 7
채팅 템플릿이 첫 번째 의심 대상
학습은 잘 됐는데 답이 이상하다면 90%는 이 문제.
기억 8
머지 vs 동적, 상황 따라
단일 작업은 머지, 멀티테넌트는 vLLM의 multi-LoRA.

마지막 한 마디

LoRA의 매력은 단순함에 있습니다. "큰 변화는 작은 방향에서 일어난다"는 한 줄의 직관과 행렬 두 개의 곱이라는 단순한 트릭이, GPU 비용을 1/10로 줄이면서도 품질은 거의 그대로 유지합니다. 좋은 엔지니어링은 늘 이런 모양이죠 — 복잡해 보이는 문제를 단순한 통찰로 풀어내는 것. 이제 여러분의 차례입니다. 학습을 시작해보세요.
