00 — Motivation

왜 "관측 가능성(Observability)"인가

모니터링이 "정해 둔 것만 본다"면, 관측 가능성은 "예상하지 못한 질문에도 답할 수 있다"는 뜻입니다. 클라우드 네이티브 시스템에서 왜 이 차이가 중요한지부터 시작합니다.

예전 단일 서버 시대에는 CPU 그래프 하나, 로그 파일 하나면 충분했습니다. 하지만 지금은 하나의 사용자 요청이 API 게이트웨이 → 인증 서비스 → 주문 서비스 → DB → 캐시 → 결제 외부 API를 거칩니다. "주문이 느려요"라는 신고가 들어왔을 때, 어느 구간에서 느려졌는지 로그만으로 찾는 건 건초더미에서 바늘 찾기입니다.

모니터링 vs 관측 가능성

모니터링은 "CPU가 80%를 넘으면 알람"처럼 미리 정의한 질문에 답합니다. 관측 가능성은 시스템이 내보내는 신호(텔레메트리)만으로 "왜 19시 10분에 특정 사용자의 결제만 느렸지?" 같은 예상치 못한 질문에도 답할 수 있는 능력입니다.

세 가지 신호, 따로 놀면 추리 게임

관측 가능성의 고전적 세 기둥은 메트릭·로그·트레이스입니다. 문제는 각 신호가 서로 다른 도구·다른 형식·다른 백엔드에 흩어져 있을 때입니다.

Metrics 📈

무엇이 일어났나

"결제 지연이 19:10에 증가" — 전체 추세·SLO·포화도. 저용량·고안정. 하지만 왜인지는 모릅니다.

Logs 📜

무슨 일이 있었나

특정 순간의 상세 기록. 그러나 분당 수천 줄에서 "timeout"을 찾아도 원인인지 노이즈인지 알기 어렵습니다.

Traces 🧭

어디서 일어났나

요청이 거친 경로와 각 구간 소요 시간. 병목 위치를 콕 짚지만, 얼마나 광범위한지는 메트릭이 필요합니다.

세 신호가 같은 맥락(context)으로 연결되어야 비로소 "느린 메트릭 → 그 트레이스 → 그 트레이스의 로그"로 한 번에 점프할 수 있습니다. 이 연결을 표준으로 만드는 것이 바로 OpenTelemetry입니다.

01 — Definition

OpenTelemetry란 무엇인가

줄여서 OTel. CNCF가 관리하는 오픈소스 관측 가능성 프레임워크로, 텔레메트리를 일관되고 · 연결되며 · 이식 가능하게 만드는 것을 목표로 합니다.

OpenTelemetry는 API · SDK · 도구 · Collector의 모음입니다. 핵심 약속은 단 하나입니다.

한 줄 정의

"한 번 계측(instrument)하고, 어떤 백엔드로든 코드 변경 없이 내보낸다." — 애플리케이션은 OTel 표준으로만 신호를 내보내고, 저장·분석 도구는 자유롭게 교체할 수 있습니다.

🔓

벤더 중립 (Vendor-neutral)

특정 APM 업체의 SDK에 코드가 묶이지 않습니다. Datadog → Grafana로 옮겨도 애플리케이션 코드는 그대로입니다. 벤더 락인 탈출이 가장 큰 가치입니다.

🌐

업계 표준

거의 모든 클라우드 제공자와 APM 도구가 OTLP를 지원합니다. 2026년 기준 사실상의 de-facto 표준이며, 신규 백엔드도 OTLP 수신을 기본 제공합니다.

🧩

두 프로젝트의 통합

2019년 OpenTracing(추적 API 표준)과 OpenCensus(구글의 라이브러리)가 합쳐져 탄생했습니다. 두 진영의 모범 사례를 흡수했습니다.

📚

시맨틱 컨벤션

속성 이름을 http.request.method처럼 표준화합니다. 덕분에 Python 서비스와 Java 서비스의 데이터를 같은 백엔드가 동일하게 인식합니다.

CNCF에서의 위치

OpenTelemetry는 Kubernetes에 이어 CNCF에서 두 번째로 활발한 프로젝트입니다. 졸업(Graduated) 단계로, 프로덕션 채택에 대한 성숙도가 검증되었음을 의미합니다.

02 — Signals

네 가지 시그널

전통적 3대 신호(트레이스·메트릭·로그)에 프로파일(Profiles)이 정식 추가되어 이제 4대 시그널 체계입니다. 각각이 답하는 질문이 다릅니다.

Trace · 안정(Stable)

🧭 트레이스 — 요청의 여정

단일 요청이 여러 서비스를 통과하는 전체 경로를 기록합니다. 여러 개의 스팬(Span)으로 구성되며, 분산 시스템의 병목·실패 지점을 찾는 핵심 신호입니다.

Metric · 안정(Stable)

📈 메트릭 — 시간에 따른 수치

처리량·에러율·지연 백분위·자원 사용률 등 집계된 숫자. 저장이 저렴하고 조회가 빨라 대시보드와 실시간 알람에 적합합니다.

Log · 안정(Stable)

📜 로그 — 시점의 사건 기록

특정 시각에 무슨 일이 있었는지 텍스트로 남깁니다. OTel은 로그에 trace_id·span_id를 자동 삽입해 트레이스와 정확히 연결합니다.

Profile · 신규 안정화

🔥 프로파일 — 코드 레벨 자원 소비

함수별 CPU·메모리 사용을 연속 측정(Continuous Profiling). 2026년 정식 안정 시그널이 되었고, eBPF로 무수정 수집이 가능합니다. "느린 요청 → 그 시간대 프로파일 → hot 함수"로 점프합니다.

상관관계(Correlation)가 핵심

네 신호가 같은 trace context를 공유하기 때문에, 느린 메트릭 → 트레이스 → 로그 → 프로파일로 끊김 없이 이동할 수 있습니다. 이것이 단순한 "신호 4개"를 하나의 이야기로 묶는 OTel의 진짜 힘입니다. 이 일관성이 곧 AIOps(자동 상관·근본원인 추정)의 전제 조건이 됩니다.

03 — Core Concepts

핵심 개념 사전

OTel 문서에 자주 등장하는 용어를 한 번에 정리합니다. 이 단어들만 익히면 공식 문서가 갑자기 읽히기 시작합니다.

Span(스팬)

작업의 최소 단위. 시작/종료 시각, 이름, 상태, 속성(Attributes), 이벤트(Events)를 가집니다. "DB 쿼리 1건", "함수 호출 1회"가 각각 하나의 스팬입니다.

Trace(트레이스)

하나 이상의 스팬으로 구성된 트리. 첫 스팬이 루트 스팬이고, 자식 스팬들이 요청의 세부 단계를 나타냅니다. 같은 trace_id로 묶입니다.

Context Propagation

분산 추적의 심장. 서비스 A가 B를 호출할 때 trace_id·span_id를 HTTP 헤더(W3C traceparent)에 실어 전파합니다. 이 덕분에 서로 다른 서비스의 스팬이 한 트레이스로 이어집니다.

Attributes(속성)

스팬·메트릭·로그에 붙는 키-값 메타데이터. http.response.status_code=500, user.tier=premium 같은 식으로 필터·그룹화의 기준이 됩니다.

Resource(리소스)

"이 텔레메트리가 어디서 왔는가"를 설명. service.name, k8s.pod.name, host 정보 등. 모든 신호에 동일하게 붙어 정확한 상관관계를 가능케 합니다.

Semantic Conventions

속성 이름의 공식 명명 규약. 모두가 같은 이름을 쓰므로 언어·서비스가 달라도 백엔드가 데이터를 동일하게 이해합니다. GenAI·DB·HTTP 등 영역별로 정의됩니다.

Baggage(배기지)

trace context와 함께 흐르는 사용자 정의 키-값. 예: 최초 진입점에서 tenant.id를 넣어 두면 하위 모든 서비스에서 꺼내 쓸 수 있습니다.

Sampling(샘플링)

모든 요청을 다 기록하면 비용 폭발. 일부만(또는 에러·고지연만) 선택해 저장합니다. Head-based(요청 시작 시 결정)와 Tail-based(완료 후 판단) 방식이 있습니다.

초보자가 가장 헷갈리는 지점

Context Propagation 없이는 분산 추적이 안 됩니다. 각 서비스가 스팬은 잘 만들어도, trace_id를 헤더로 넘기지 않으면 서로 다른 트레이스로 쪼개져 "요청의 여정"이 끊깁니다. 자동 계측 라이브러리가 이 전파를 대부분 알아서 해 줍니다.

04 — Architecture

전체 아키텍처

OTel은 크게 ① 애플리케이션 안(API·SDK) → ② 전송(OTLP) → ③ Collector → ④ 백엔드 흐름으로 동작합니다. 각 조각의 역할을 분리해 이해하면 설정이 쉬워집니다.

App 내부

API

"스팬을 시작해라" 같은 인터페이스만 정의. 라이브러리 개발자는 API에만 의존하므로, 최종 사용자가 SDK를 안 깔면 아무 동작도 안 하는 안전한 설계입니다.

App 내부

SDK

API의 실제 구현체. 샘플링·배치·내보내기를 담당. 애플리케이션이 SDK를 설정하고 초기화합니다.

전송

OTLP (OpenTelemetry Protocol)

OTel 전용 표준 전송 프로토콜. gRPC(:4317) 또는 HTTP(:4318)로 신호를 보냅니다. 모든 신호가 이 하나의 프로토콜을 공유합니다.

중앙 처리

Collector

애플리케이션과 백엔드 사이의 중간 계층. 받고(receive) → 가공(process) → 내보냅니다(export). 앱은 "어디로 가는지" 몰라도 됩니다.

왜 Collector를 따로 두나요?

애플리케이션에서 백엔드로 직접 보낼 수도 있지만, 중간에 Collector를 두면 다음 두 가지가 결정적으로 좋아집니다.

  • 관심사 분리 — 애플리케이션은 그저 Collector로 보냅니다. 백엔드를 Jaeger에서 Tempo로 바꿔도 앱 재배포가 필요 없습니다. Collector 설정만 고치면 됩니다.
  • 중앙 집중 가공 — 민감정보 마스킹, 속성 추가(예: k8sattributesprocessor로 Pod 정보 자동 부착), 샘플링, 배치, 재시도를 한곳에서 처리합니다.
Span Metrics Connector — 실전 꿀팁

트레이스 계측은 잘 됐는데 메트릭이 부족할 때, Collector의 spanmetrics 커넥터가 스팬에서 지연 히스토그램·요청 수 메트릭을 자동 생성합니다. 메트릭과 트레이스가 같은 데이터에서 나오니 완벽히 일치합니다.

Receiver — 데이터를 받아들이는 입구

여러 소스에서 텔레메트리를 수신합니다. 가장 흔한 건 OTLP receiver이며, Prometheus·Jaeger·Kafka·파일 로그 등 다양한 형식을 받아 OTel 내부 형식으로 변환합니다.

예: otlp, prometheus, jaeger, filelog, kafka

최소 Collector 설정 (YAML)

Collector는 YAML 한 장으로 동작합니다. 구성요소를 정의(define)하고, service.pipelines에서 조립(wire)하는 2단 구조입니다. 정의만 하고 파이프라인에 넣지 않으면 동작하지 않습니다.

receivers:
  otlp:                      # ① 앱에서 OTLP로 수신
    protocols:
      grpc: { endpoint: 0.0.0.0:4317 }
      http: { endpoint: 0.0.0.0:4318 }

processors:
  batch: {}                    # ② 묶어서 효율적으로 전송
  memory_limiter:              # OOM 방지 (강력 권장)
    check_interval: 1s
    limit_percentage: 80

exporters:
  otlp/tempo:                # ③ 트레이스 백엔드로 전송
    endpoint: tempo:4317
  prometheus:                # 메트릭은 Prometheus가 스크랩
    endpoint: 0.0.0.0:8889

service:
  pipelines:                 # ④ 위 부품을 조립!
    traces:
      receivers:  [otlp]
      processors: [memory_limiter, batch]
      exporters:  [otlp/tempo]
    metrics:
      receivers:  [otlp]
      processors: [batch]
      exporters:  [prometheus]
# 로그 파이프라인 추가 + K8s 속성 자동 부착
processors:
  k8sattributes: {}          # Pod/Node 정보 자동 첨부
  batch: {}

exporters:
  otlphttp/loki:
    endpoint: http://loki:3100/otlp

service:
  pipelines:
    logs:
      receivers:  [otlp]
      processors: [k8sattributes, batch]
      exporters:  [otlphttp/loki]
    profiles:                # 프로파일 시그널 (신규)
      receivers:  [otlp]
      exporters:  [otlphttp/loki]
      # ※ 일부 버전에선 feature gate 활성화 필요

배포 패턴: Agent vs Gateway

Agent 모드

노드마다 하나 (DaemonSet)

각 호스트/노드에 Collector를 붙여 로컬에서 빠르게 수집·전처리. 네트워크 부담이 적고 호스트 메타데이터 부착에 유리합니다.

Gateway 모드

중앙 집중 클러스터

여러 Agent가 보낸 데이터를 중앙 Gateway가 모아 tail-based 샘플링·집계·라우팅. 보통 Agent + Gateway를 2단으로 함께 씁니다.

현업 권장 구성

Kubernetes에서는 노드별 Agent(DaemonSet) → 중앙 Gateway(Deployment) 2계층이 정석입니다. Agent가 빠르게 수집하고, Gateway에서 비용이 큰 tail-based 샘플링과 백엔드 분기를 처리합니다.

06 — Instrumentation

계측 방법 — 코드는 어떻게 신호를 낼까

계측(Instrumentation)에는 자동(Auto)과 수동(Manual) 두 갈래가 있습니다. 자동으로 80%를 깔고, 비즈니스 로직만 수동으로 보강하는 것이 정석입니다.

⚡

자동 계측 (Zero-code)

코드를 거의 안 고치고 HTTP·DB·gRPC 등 인기 프레임워크를 자동으로 추적. Python·Java·Node가 강력하며, eBPF 기반 무수정 계측도 성숙했습니다. 시작은 무조건 여기부터.

✍️

수동 계측 (Manual)

비즈니스 로직(예: "쿠폰 검증", "추천 모델 추론")에 직접 스팬·속성을 추가. 자동 계측이 모르는 우리 도메인의 의미를 담습니다.

실제 코드 — 같은 작업, 여러 언어

아래는 결제 처리 함수에 수동으로 스팬을 만들고, 속성·예외·상태를 기록하는 예시입니다. 탭을 눌러 언어별로 비교하세요.

from opentelemetry import trace

tracer = trace.get_tracer("payment-service")

def process_payment(order_id, amount):
    with tracer.start_as_current_span("payment.process") as span:
        # 1) 의미 있는 속성 부착
        span.set_attribute("order.id", order_id)
        span.set_attribute("payment.amount", amount)
        try:
            validate_order(order_id)
            charge_card(amount)
            span.set_status(trace.StatusCode.OK)
        except Exception as e:
            # 2) 예외를 스팬에 기록 → 백엔드에서 빨갛게 표시
            span.record_exception(e)
            span.set_status(trace.StatusCode.ERROR, str(e))
            raise
const { trace, SpanStatusCode } = require('@opentelemetry/api');
const tracer = trace.getTracer('payment-service');

async function processPayment(orderId, amount) {
  const span = tracer.startSpan('payment.process', {
    attributes: { 'order.id': orderId, 'payment.amount': amount }
  });
  try {
    await validateOrder(orderId);
    await chargeCard(amount);
    span.setStatus({ code: SpanStatusCode.OK });
  } catch (err) {
    span.recordException(err);
    span.setStatus({ code: SpanStatusCode.ERROR, message: err.message });
    throw err;
  } finally {
    span.end();   // 수동 시작은 반드시 end() 호출!
  }
}
import (
    "go.opentelemetry.io/otel"
    "go.opentelemetry.io/otel/attribute"
    "go.opentelemetry.io/otel/codes"
)

var tracer = otel.Tracer("payment-service")

func ProcessPayment(ctx context.Context, orderID string, amount float64) error {
    ctx, span := tracer.Start(ctx, "payment.process")
    defer span.End()

    span.SetAttributes(
        attribute.String("order.id", orderID),
        attribute.Float64("payment.amount", amount),
    )
    if err := chargeCard(ctx, amount); err != nil {
        span.RecordError(err)
        span.SetStatus(codes.Error, err.Error())
        return err
    }
    span.SetStatus(codes.Ok, "")
    return nil
}

# 3) 앱 앞에 opentelemetry-instrument 만 붙여 실행

# → Flask/Django, requests, psycopg2 등이 자동으로 추적됨
자주 하는 실수

수동으로 startSpan()으로 시작한 스팬은 반드시 end()/End()로 닫아야 합니다. 닫지 않으면 스팬이 영원히 미완료 상태가 되어 트레이스가 깨집니다. Python의 with, Go의 defer span.End()처럼 언어 관용구를 쓰면 실수를 막을 수 있습니다.

07 — Comparison

다른 수집 도구와의 비교

"그럼 Prometheus·Jaeger·Fluent Bit은 OTel이 대체하나요?" — 아닙니다. OTel은 대부분 경쟁이 아니라 보완입니다. 각 도구의 자리부터 정리합니다.

핵심 오해 바로잡기

가장 흔한 혼동은 OTel을 "또 하나의 백엔드"로 오해하는 것입니다. OTel은 수집·전송·표준화 계층이고, Prometheus/Jaeger/Loki 등은 저장·질의 백엔드입니다. 역할이 다릅니다. 실제로는 "OTel로 수집해서 Prometheus·Jaeger에 저장"하는 조합이 흔합니다.

도구주 역할다루는 신호OTel과의 관계백엔드 포함?
OpenTelemetry계측·수집·전송 표준Trace·Metric·Log·Profile—(자기 자신)없음
Prometheus메트릭 저장·질의(TSDB)MetricOTel이 메트릭을 Prometheus로 export, 또는 Prometheus가 OTLP 수신메트릭 DB
Jaeger트레이스 저장·시각화TraceOTel SDK로 계측 → Jaeger로 전송(네이티브 OTLP 수신)트레이스 DB
Zipkin트레이스 저장·시각화TraceJaeger와 유사. OTel이 Zipkin 형식으로 export 가능트레이스 DB
Grafana Tempo대용량 트레이스 저장TraceOTLP 네이티브 수신. OTel과 궁합 좋음트레이스 DB
Loki로그 저장·질의LogOTel Collector가 로그를 Loki로 export로그 DB
Fluentd / Fluent Bit로그 수집·전달 에이전트Log(주로)역할 일부 겹침. OTel Collector가 점점 대체 중이나 로그 수집은 Fluent Bit이 여전히 강세없음
Logstash로그 수집·변환(ELK)LogCollector의 processor와 역할 유사. ELK 스택 내에서 사용없음
Datadog Agent벤더 통합 수집 에이전트All독자 SDK 사용 시 락인. Datadog도 OTLP 수신 지원 → OTel로 계측 가능SaaS 연동
Prometheus와의 관계

경쟁이 아닙니다. OTel로 메트릭을 만들고 Prometheus에 저장하는 게 표준 조합입니다. Prometheus는 OTLP 수신과 exemplars(메트릭→트레이스 연결)를 지원해 둘이 더 가까워졌습니다.

Fluent Bit과의 관계

로그 수집에서 역할이 겹칩니다. 하지만 Fluent Bit은 경량·고성능 로그 수집에 특화되어 있고, OTel Collector는 4대 신호를 통합한다는 점이 다릅니다. 종종 함께 쓰입니다.

언제 OTel을 선택하나

① 벤더 락인을 피하고 싶을 때(가장 큰 이유), ② 여러 신호를 하나의 맥락으로 연결하고 싶을 때, ③ 다국어 마이크로서비스를 일관되게 계측해야 할 때. 반대로 단일 언어·단일 백엔드 소규모라면 백엔드의 자체 SDK가 더 빠를 수도 있습니다. — 다만 규모가 커질수록 OTel의 이식성이 빛을 발합니다.

08 — Applications

응용 분야

OTel이 실제로 가치를 내는 현장입니다. 마이크로서비스·쿠버네티스는 기본이고, 2026년에는 LLM/AI 관측(GenAI Observability)이 가장 뜨거운 응용 분야입니다.

🕸️

마이크로서비스 분산 추적

요청이 수십 개 서비스를 거치는 환경에서 병목·실패 지점을 워터폴 한 장으로 찾습니다. OTel의 출발점이자 가장 성숙한 영역입니다.

☸️

Kubernetes 관측

DaemonSet Agent + Gateway 구성으로 Pod/Node 메타데이터를 자동 부착. k8sattributesprocessor가 모든 신호에 동일한 K8s 속성을 붙여 정확한 상관관계를 만듭니다.

💸

비용 최적화 · FinOps

tail-based 샘플링으로 의미 있는 트레이스만 저장하고, 프로파일로 요청당 CPU 시간·hot 함수를 찾아 자원 낭비를 줄입니다.

🤖

AIOps 기반

일관되게 명명·연결된 텔레메트리는 알람 중복 제거, 토폴로지 기반 근본원인 후보 추정, 이상 탐지의 전제입니다. "AI가 상관시킬 수 있으려면 데이터가 일관되게 연결돼야 한다."

🔥 LLMOps / MLOps에서의 OpenTelemetry

2026년 OTel 스펙에는 GenAI 시맨틱 컨벤션이 들어와, LLM 애플리케이션의 토큰 수·모델 지연·프롬프트/응답을 표준 속성으로 기록할 수 있게 되었습니다. LLM 파이프라인은 본질적으로 분산 호출(임베딩 → 벡터 검색 → LLM 추론 → 후처리)이라 트레이스가 특히 잘 맞습니다.

GenAI Conventions

표준 속성

gen_ai.request.model, gen_ai.usage.input_tokens, gen_ai.usage.output_tokens 등으로 토큰·비용·모델을 일관 기록.

OpenLLMetry

LLM 자동 계측

OTel 위에서 OpenAI·LangChain·LlamaIndex 등을 자동 계측하는 오픈소스. 프롬프트·체인·툴 호출이 스팬으로 떨어집니다.

Langfuse 연동

LLM 옵저버빌리티

Langfuse는 OTLP 수신을 지원해, OTel로 계측한 LLM 트레이스를 그대로 받습니다. 기존 OTel 파이프라인에 자연스럽게 합류시킬 수 있습니다.

RAG 파이프라인을 트레이스로 본다면

하나의 사용자 질문이 질의 임베딩(120ms) → Milvus 벡터 검색(45ms) → 프롬프트 구성(5ms) → vLLM 추론(1,800ms) → 가드레일 검사(30ms)로 흐른다고 합시다. 워터폴로 보면 "응답이 느린 이유의 90%가 추론 단계"임이 즉시 드러나고, gen_ai.usage.output_tokens 속성으로 토큰당 비용까지 분석할 수 있습니다. 분산 추적의 사고방식이 그대로 LLM 파이프라인에 이식되는 셈입니다.

from opentelemetry import trace
tracer = trace.get_tracer("rag-pipeline")

def answer(question):
    with tracer.start_as_current_span("rag.answer") as root:
        with tracer.start_as_current_span("retrieval.search") as s:
            docs = vector_db.search(question, k=5)
            s.set_attribute("retrieval.top_k", 5)
            s.set_attribute("retrieval.hits", len(docs))

        with tracer.start_as_current_span("llm.generate") as g:
            # GenAI 시맨틱 컨벤션 속성
            g.set_attribute("gen_ai.request.model", "qwen2.5-72b")
            resp = llm.chat(question, context=docs)
            g.set_attribute("gen_ai.usage.input_tokens", resp.usage.prompt)
            g.set_attribute("gen_ai.usage.output_tokens", resp.usage.completion)
        return resp.text
09 — Check Your Understanding

이해도 점검 퀴즈

다섯 문제로 핵심을 점검하세요. 정답을 고르면 해설이 펼쳐집니다.

Q1.OpenTelemetry가 제공하는 가장 큰 가치는 무엇인가?
OTel은 백엔드가 아니라 계측·수집·전송 표준입니다. 핵심 가치는 벤더 락인을 피하고, 백엔드를 자유롭게 교체할 수 있는 이식성입니다.
Q2.분산 시스템에서 서로 다른 서비스의 스팬이 하나의 트레이스로 이어지게 하는 메커니즘은?
서비스 A가 B를 호출할 때 trace_id·span_id를 traceparent 헤더에 실어 전파해야 두 서비스의 스팬이 같은 trace_id로 묶입니다. 이게 없으면 트레이스가 끊깁니다.
Q3.OTel Collector 파이프라인의 올바른 순서는?
받아서(Receiver) → 가공하고(Processor) → 내보냅니다(Exporter). 그리고 정의한 부품은 service.pipelines에서 조립해야 실제로 동작합니다.
Q4.Prometheus와 OpenTelemetry의 관계로 가장 정확한 것은?
경쟁이 아니라 보완입니다. OTel은 수집·표준화 계층, Prometheus는 메트릭 저장·질의 백엔드입니다. Prometheus는 OTLP 수신도 지원합니다.
Q5.2026년 OTel에서 LLM 애플리케이션 관측과 직접 관련된 것은?
OTel 스펙에 GenAI 시맨틱 컨벤션이 들어와 토큰 수·모델·지연을 표준 속성으로 기록합니다. OpenLLMetry·Langfuse(OTLP 수신) 등과 결합해 LLM 파이프라인을 트레이스로 관측합니다.

정리하며. OpenTelemetry는 "또 하나의 도구"가 아니라, 흩어진 텔레메트리를 하나의 언어로 모으는 표준입니다. 자동 계측으로 시작해 비즈니스 로직만 수동으로 보강하고, Collector를 중간에 두어 백엔드를 자유롭게 교체하세요. 그리고 LLM 시대에는 RAG·추론 파이프라인을 트레이스로 들여다보는 것이 곧 새로운 디버깅 표준이 됩니다.
