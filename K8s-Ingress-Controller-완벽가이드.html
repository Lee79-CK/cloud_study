Ingress Controller 완벽 이해

Kubernetes 클러스터 안의 수십, 수백 개 서비스로 외부 트래픽을 어떻게 안전하고 효율적으로 전달할까요? 이 강의는 Service의 한계에서 출발해 Ingress의 개념, 동작 원리, 실제 구현 제품 비교, 그리고 실습까지 — 초보 Cloud 엔지니어가 반드시 알아야 할 모든 것을 다룹니다.

난이도: 입문 ~ 중급 예상 학습 시간: 90분 실습 포함 퀴즈 10문항

01왜 Ingress가 필요한가

모든 기술은 문제에서 출발합니다. Ingress를 이해하려면 먼저 "외부에서 클러스터 안의 앱에 접속하기"가 왜 어려운 문제인지부터 봐야 합니다.

상황극: 쇼핑몰 서비스를 오픈했습니다

여러분이 Kubernetes 위에 쇼핑몰을 구축했다고 상상해 봅시다. 클러스터 안에는 이런 마이크로서비스들이 돌아가고 있습니다.

web-front
메인 웹 페이지 (React SPA)
api-server
주문/상품 REST API
admin-portal
내부 관리자 페이지

사용자는 shop.example.com으로 접속하면 웹 페이지가, shop.example.com/api로 요청하면 API가, admin.example.com으로 가면 관리자 페이지가 나오길 원합니다. 도메인과 URL 경로에 따라 서로 다른 서비스로 트래픽을 나눠줘야 하는 거죠.

그런데 Pod은 클러스터 내부 IP만 가지고 있어 외부에서 직접 접근할 수 없습니다. 게다가 Pod은 언제든 죽고 다시 생성되며 그때마다 IP가 바뀝니다. "그럼 Service를 쓰면 되잖아?"라고 생각하셨다면 정확합니다. 하지만 Service만으로는 부족합니다. 왜 그런지 다음 섹션에서 하나씩 확인해 봅시다.

💡 한 줄 요약

Ingress는 "하나의 진입점으로 들어온 HTTP(S) 트래픽을 호스트명과 URL 경로 기준으로 여러 서비스에 분배하는 L7 라우팅 규칙"입니다. 클러스터의 정문이자 안내 데스크라고 생각하면 됩니다.

02K8s Service 복습과 한계

Ingress를 이해하기 위한 필수 배경지식입니다. Service의 세 가지 타입이 각각 어떤 문제를 풀고, 어떤 문제를 못 푸는지 봅시다.

Service 3형제: ClusterIP, NodePort, LoadBalancer

클러스터 내부 전용 고정 IP. Pod들이 서로를 찾을 때 사용하는 기본 타입입니다. Pod IP가 바뀌어도 Service의 ClusterIP는 유지되고, 뒤의 Pod들로 트래픽을 분산합니다.

  • 해결하는 문제: Pod IP 변경 문제, 내부 로드밸런싱
  • 못 푸는 문제: 외부에서 접근 자체가 불가능
clusterip-svc.yaml
apiVersion: v1
kind: Service
metadata:
  name: api-server
spec:
  type: ClusterIP   # 생략 시 기본값
  selector:
    app: api-server
  ports:
    - port: 80        # Service가 노출하는 포트
      targetPort: 8080 # Pod 컨테이너 포트

모든 노드의 특정 포트(기본 30000~32767)를 열어 외부 접근을 허용. 노드IP:30080 같은 형태로 접속합니다.

  • 해결하는 문제: 외부 접근 가능
  • 못 푸는 문제: 포트 번호가 이상함(30080?), 노드 IP를 알아야 함, 노드 장애 시 대응 필요, 서비스마다 포트 하나씩 소모

실무에서는 주로 테스트 용도나, 외부 LB와 연결하는 중간 다리로 사용합니다.

nodeport-svc.yaml
apiVersion: v1
kind: Service
metadata:
  name: api-server
spec:
  type: NodePort
  selector:
    app: api-server
  ports:
    - port: 80
      targetPort: 8080
      nodePort: 30080  # 모든 노드에서 이 포트가 열림

클라우드 제공자의 L4 로드밸런서를 자동 생성해 외부 IP를 부여. AWS의 NLB/ELB, GCP의 Cloud Load Balancing 같은 실제 LB가 만들어집니다.

  • 해결하는 문제: 제대로 된 외부 진입점(고정 외부 IP, 표준 포트 80/443)
  • 못 푸는 문제: 서비스 1개당 LB 1개 = 비용 폭탄 💸, L4라서 URL 경로/호스트 기반 분기 불가, TLS 인증서를 서비스마다 각각 관리
lb-svc.yaml
apiVersion: v1
kind: Service
metadata:
  name: api-server
spec:
  type: LoadBalancer  # 클라우드 LB 자동 프로비저닝
  selector:
    app: api-server
  ports:
    - port: 80
      targetPort: 8080

비용으로 보는 문제의 심각성

마이크로서비스가 30개인 조직이 모든 서비스를 type: LoadBalancer로 노출한다면? AWS NLB 기준 대당 월 $16~$30 수준이므로 로드밸런서 비용만 월 $500~$900이 나갑니다. 반면 Ingress를 쓰면 LB 1대 + Ingress Controller로 30개 서비스를 전부 처리할 수 있습니다.

📚 L4 vs L7, 30초 정리

L4(전송 계층)는 IP와 포트만 봅니다. "10.0.0.5의 443 포트로 보내"까지만 가능합니다. L7(응용 계층)은 HTTP 요청의 내용 — Host 헤더, URL 경로, 쿠키, 헤더 — 을 읽을 수 있습니다. "Host가 admin.example.com이면 admin 서비스로, 경로가 /api로 시작하면 API 서비스로"가 가능해지는 이유입니다. Ingress가 L7인 이유가 바로 이것입니다.

03Ingress란 무엇인가

이제 본론입니다. Ingress의 정의와 전체 트래픽 흐름을 그림으로 이해해 봅시다.

정의

공식 문서의 정의를 풀어 쓰면 이렇습니다. Ingress는 클러스터 외부에서 내부 Service로 향하는 HTTP/HTTPS 트래픽의 라우팅 규칙을 정의하는 Kubernetes API 오브젝트입니다. 핵심 키워드는 세 가지입니다.

🚪 단일 진입점
외부 LB와 IP는 하나만 두고, 그 뒤에서 수십 개 서비스로 분배
🧭 L7 라우팅
Host 헤더와 URL 경로를 읽어 목적지 서비스를 결정
🔐 TLS 종료
HTTPS 암호화를 진입점에서 한 번에 처리(인증서 중앙 관리)

전체 트래픽 흐름 한 장 정리

그림 1. 외부 사용자 → LB → Ingress Controller → (Ingress 규칙 참조) → Service → Pod 흐름

04Ingress 리소스 vs Ingress Controller

이 강의에서 단 하나만 가져간다면 바로 이 구분입니다. 둘은 이름만 비슷할 뿐 완전히 다른 존재입니다.

📋 Ingress 리소스 (규칙)

  • Kubernetes API 오브젝트 (kind: Ingress)
  • 단순한 선언적 설정 데이터 — YAML로 작성해 etcd에 저장됨
  • "이 호스트/경로로 오면 저 서비스로 보내라"는 희망사항
  • 그 자체로는 트래픽을 단 1바이트도 처리하지 못함
  • K8s에 기본 내장 (별도 설치 불필요)

⚙️ Ingress Controller (실행자)

  • 클러스터에서 실제로 실행되는 프로그램 (Deployment/DaemonSet의 Pod)
  • 내부에 NGINX, Envoy, HAProxy 같은 리버스 프록시를 품고 있음
  • API 서버를 watch하며 Ingress 리소스 변경을 감지
  • 규칙을 프록시 설정으로 변환해 실제 트래픽을 처리
  • 기본 미설치 — 반드시 직접 선택하여 설치해야 함

Kubernetes 프로젝트는 의도적으로 "규칙의 표준 문법"만 정의하고 "실행자"는 생태계에 맡겼습니다. 그 덕분에 NGINX, Traefik, HAProxy, Kong, 클라우드 네이티브 컨트롤러 등 다양한 구현체가 경쟁하며 발전할 수 있었고, 사용자는 같은 Ingress YAML을 유지한 채 컨트롤러만 교체할 수 있습니다(어노테이션 제외 — 뒤에서 설명). 이는 CSI(스토리지), CNI(네트워크), CRI(런타임)와 같은 Kubernetes 특유의 인터페이스-구현 분리 철학입니다.

💡 IngressClass — 컨트롤러가 여러 개라면?

한 클러스터에 NGINX와 ALB 컨트롤러를 동시에 운영할 수도 있습니다. 이때 "이 Ingress 규칙은 누가 처리할 것인가"를 지정하는 것이 IngressClass 리소스와 spec.ingressClassName 필드입니다. 클러스터에 IngressClass가 무엇이 있는지는 kubectl get ingressclass로 확인합니다. 기본 클래스는 ingressclass.kubernetes.io/is-default-class: "true" 어노테이션으로 지정합니다.

ingressclass.yaml — 컨트롤러 설치 시 함께 생성됨
apiVersion: networking.k8s.io/v1
kind: IngressClass
metadata:
  name: nginx
  annotations:
    ingressclass.kubernetes.io/is-default-class: "true"
spec:
  controller: k8s.io/ingress-nginx  # 이 클래스를 처리할 컨트롤러 식별자

05Ingress 리소스 작성법

실제 YAML을 한 줄씩 뜯어봅니다. 호스트 라우팅, 경로 라우팅, pathType, TLS까지 실무에 필요한 문법을 모두 다룹니다.

가장 기본적인 Ingress

basic-ingress.yaml
apiVersion: networking.k8s.io/v1   # v1.19+ 정식 API
kind: Ingress
metadata:
  name: shop-ingress
spec:
  ingressClassName: nginx        # 어떤 컨트롤러가 처리할지 지정
  rules:
  - host: shop.example.com        # Host 헤더 매칭
    http:
      paths:
      - path: /                    # URL 경로 매칭
        pathType: Prefix           # 매칭 방식 (필수!)
        backend:
          service:
            name: web-front        # 목적지 Service 이름
            port:
              number: 80

⚠️ 주의: Ingress는 같은 네임스페이스의 Service만 가리킬 수 있습니다. 다른 네임스페이스의 서비스로 보내려면 ExternalName Service 등의 우회 기법이 필요합니다.

pathType 3종 — 시험에도 실무에도 나옵니다

pathType동작예시 (path: /api)
Prefix / 단위 요소로 나눈 접두사 매칭. 가장 널리 사용. /api ◯, /api/v1/users ◯, /apiserver ✕ (요소 단위라 부분 문자열 매칭 아님)
Exact 대소문자 구분 완전 일치 /api ◯, /api/ ✕, /api/v1 ✕
ImplementationSpecific 매칭 방식을 IngressClass(컨트롤러)에 위임. 컨트롤러마다 다르게 동작 — 정규식을 지원하기도 함 컨트롤러 문서 확인 필요
📚 매칭 우선순위 규칙

여러 경로가 동시에 매칭되면 ① 가장 긴 경로가 우선하고, ② 길이가 같으면 Exact가 Prefix보다 우선합니다. 예를 들어 /api/v2 요청은 /api(Prefix)보다 /api/v2(Prefix)에 매칭됩니다.

실전 예제: 호스트 + 경로 복합 라우팅

섹션 1의 쇼핑몰 요구사항을 그대로 구현한 YAML입니다.

shop-full-ingress.yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: shop-ingress
  annotations:
    # 컨트롤러별 확장 기능은 어노테이션으로 (NGINX 예시)
    nginx.ingress.kubernetes.io/ssl-redirect: "true"
spec:
  ingressClassName: nginx
  tls:                              # HTTPS 설정
  - hosts:
    - shop.example.com
    - admin.example.com
    secretName: shop-tls-cert      # 인증서를 담은 Secret
  rules:
  - host: shop.example.com
    http:
      paths:
      - path: /api              # 더 긴(구체적) 경로가 우선 매칭됨
        pathType: Prefix
        backend:
          service:
            name: api-server
            port: { number: 80 }
      - path: /
        pathType: Prefix
        backend:
          service:
            name: web-front
            port: { number: 80 }
  - host: admin.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: admin-portal
            port: { number: 8080 }
  defaultBackend:                  # 어느 규칙에도 안 걸린 트래픽 처리
    service:
      name: default-404
      port: { number: 80 }

TLS 인증서 준비 방법

terminal
# 인증서(crt)와 개인키(key) 파일로 TLS 타입 Secret 생성
kubectl create secret tls shop-tls-cert \
  --cert=shop.example.com.crt \
  --key=shop.example.com.key

테스트라면 openssl로 자체 서명 인증서를 만들어도 됩니다. 운영에서는 공인 CA 인증서 또는 아래 cert-manager를 사용하세요.

실무 표준은 cert-manager + Let's Encrypt 조합입니다. Ingress에 어노테이션 한 줄만 달면 인증서 발급·갱신이 전부 자동화됩니다.

ingress-with-certmanager.yaml (일부)
metadata:
  annotations:
    cert-manager.io/cluster-issuer: letsencrypt-prod
spec:
  tls:
  - hosts: [shop.example.com]
    secretName: shop-tls-cert  # cert-manager가 자동 생성/갱신

실제 Ingress Controller와 동일한 순서로 판정합니다.

  1. Host 매칭: 요청의 Host 헤더와 일치하는 rules[].host를 찾습니다. 없으면 defaultBackend로.
  2. Path 매칭: 해당 호스트의 paths 중 매칭되는 것을 모두 찾고, 가장 긴 경로 → Exact 우선 순으로 하나를 선택합니다.
  3. Prefix는 경로 요소 단위: /api는 /api/orders에 매칭되지만 /apiserver에는 매칭되지 않습니다.
  4. TLS/리다이렉트: ssl-redirect 어노테이션이 있으면 HTTP 요청은 308로 HTTPS 리다이렉트됩니다.

07Ingress Controller 동작 원리

컨트롤러 내부에서는 무슨 일이 벌어질까요? "watch → 설정 변환 → 리로드"라는 컨트롤 루프를 이해하면 트러블슈팅 능력이 확 올라갑니다.

컨트롤 루프: NGINX Ingress Controller의 예

Watch — API 서버 감시
컨트롤러는 Kubernetes API 서버에 watch 연결을 걸고 Ingress, Service, Endpoint(EndpointSlice), Secret, ConfigMap 리소스의 변경 이벤트를 실시간으로 수신합니다.
Translate — 프록시 설정으로 변환
수신한 Ingress 규칙 + 어노테이션을 자신이 품고 있는 프록시의 설정 언어로 변환합니다. NGINX 컨트롤러라면 Go 템플릿으로 nginx.conf를 생성합니다.
Reload/Update — 무중단 반영
설정 파일이 바뀌면 nginx -s reload로 무중단 리로드합니다. 단, 백엔드 Pod IP 변경(Endpoint 변화)은 리로드 없이 Lua 기반 동적 업데이트로 처리해 잦은 리로드를 피합니다. Envoy 기반 컨트롤러(Contour, Emissary 등)는 xDS API로 아예 리로드 개념 없이 동적 반영합니다.
Proxy — 실제 트래픽 처리
이제 외부 요청이 들어오면 프록시가 Host/경로를 보고 매칭한 뒤, Service를 거치지 않고 Endpoint의 Pod IP로 직접 로드밸런싱하는 경우가 많습니다(NGINX 컨트롤러 기본 동작). kube-proxy를 한 번 덜 거치므로 성능에 유리하고, 세션 어피니티 같은 L7 기능 제어가 가능해집니다.
Status 보고
처리 중인 Ingress 리소스의 status.loadBalancer.ingress 필드에 진입점 IP/호스트명을 기록합니다. kubectl get ingress의 ADDRESS 열이 바로 이 값입니다.

컨트롤러 자체는 어떻게 외부에 노출되나?

여기서 초보자들이 두 번째로 헷갈립니다. Ingress Controller도 결국 Pod이므로, 자기 자신이 외부 트래픽을 받으려면 Service가 필요합니다. 일반적인 패턴은 세 가지입니다.

☁️ LoadBalancer Service
클라우드 환경 표준. 컨트롤러 앞에 클라우드 L4 LB 1대를 두는 구성. "LB는 결국 쓰지만 1대만 쓴다"가 포인트.
🔌 NodePort Service
온프레미스에서 외부 하드웨어 LB(F5 등)나 DNS 라운드로빈과 조합. 온프레미스라면 MetalLB로 LoadBalancer 타입을 흉내낼 수도 있음.
🖥️ hostNetwork / DaemonSet
컨트롤러 Pod이 노드의 80/443 포트를 직접 점유. 홉이 줄어 성능은 좋지만 포트 충돌·보안 고려 필요. 엣지 노드 전용 구성에서 사용.
💡 DGX Spark + ThinkStation 홈랩이라면

철규님처럼 베어메탈 2노드 클러스터를 운영한다면 MetalLB(L2 모드) + NGINX Ingress Controller 조합이 가장 간단합니다. MetalLB가 공유기 대역의 IP를 LoadBalancer Service에 할당해 주면, 클라우드와 동일한 아키텍처를 홈랩에서 재현할 수 있습니다. ARM64(Grace Blackwell)와 x86_64 혼합 클러스터에서는 컨트롤러 이미지가 멀티아치(multi-arch)를 지원하는지 확인하세요 — ingress-nginx, Traefik 모두 arm64 이미지를 제공합니다.

08주요 기능 총정리

Ingress Controller가 제공하는 핵심 기능 8가지를 실무 예제와 함께 정리합니다. 표준 스펙 기능과 어노테이션 확장 기능을 구분해서 익히는 것이 중요합니다.

① L7 라우팅 (표준 스펙)

앞에서 다룬 호스트 기반(가상 호스팅) + 경로 기반 라우팅입니다. Ingress의 존재 이유이자 유일하게 표준 스펙으로 보장되는 라우팅 기능입니다. 헤더·쿠키·메서드 기반 라우팅은 표준 스펙에 없으며 컨트롤러별 확장(어노테이션/CRD)으로 제공됩니다.

② TLS 종료 (TLS Termination, 표준 스펙)

HTTPS 암호화/복호화를 Ingress Controller가 대신 처리하고, 뒤의 서비스에는 평문 HTTP로 전달하는 방식입니다. 인증서를 진입점 한 곳에서 중앙 관리할 수 있고, 각 애플리케이션은 TLS 로직 없이 개발할 수 있습니다.

그림 2. TLS 종료 — 암호화 구간은 클라이언트 ↔ 컨트롤러까지
⚠️ 제로 트러스트 환경이라면

컨트롤러 → 백엔드 구간도 암호화가 필요하면 TLS 재암호화(re-encryption) 또는 서비스 메시(mTLS)를 함께 사용합니다. 금융권/공공 프로젝트에서 자주 요구되는 사항입니다. 반대로 클라이언트 인증서까지 검증하는 mTLS 종료도 대부분의 컨트롤러가 어노테이션으로 지원합니다.

③ 로드밸런싱 & 세션 어피니티

백엔드 Pod들에 대한 분산 알고리즘(라운드로빈, least_conn, EWMA 등)을 제공하고, 쿠키 기반 Sticky Session으로 같은 사용자를 같은 Pod에 고정할 수 있습니다.

sticky-session 어노테이션 (NGINX)
metadata:
  annotations:
    nginx.ingress.kubernetes.io/affinity: "cookie"
    nginx.ingress.kubernetes.io/session-cookie-name: "route"
    nginx.ingress.kubernetes.io/session-cookie-max-age: "3600"

④ URL Rewrite & Redirect

외부 경로와 내부 앱 경로가 다를 때 필수입니다. 예: 외부에서는 /api/orders로 받지만 백엔드 앱은 /orders만 알고 있는 경우.

rewrite-ingress.yaml (NGINX)
metadata:
  annotations:
    # /api/orders → /orders 로 재작성. $2 = 두 번째 캡처 그룹
    nginx.ingress.kubernetes.io/rewrite-target: /$2
spec:
  rules:
  - http:
      paths:
      - path: /api(/|$)(.*)       # 정규식 사용
        pathType: ImplementationSpecific
        backend:
          service: { name: api-server, port: { number: 80 } }

⑤ 카나리 배포 (Canary)

신규 버전에 트래픽의 일부(예: 10%)만 흘려보내 안전하게 검증하는 점진적 배포 기법입니다. NGINX 컨트롤러는 별도의 카나리 Ingress를 만들어 구현합니다.

canary-ingress.yaml (NGINX)
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: shop-canary
  annotations:
    nginx.ingress.kubernetes.io/canary: "true"
    nginx.ingress.kubernetes.io/canary-weight: "10"   # 10%만 v2로
    # weight 외에도 canary-by-header, canary-by-cookie 방식 지원
spec:
  ingressClassName: nginx
  rules:
  - host: shop.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service: { name: web-front-v2, port: { number: 80 } }

본격적인 프로그레시브 딜리버리(자동 분석·롤백)가 필요하면 Argo Rollouts나 Flagger를 Ingress Controller와 연동하는 것이 정석입니다.

⑥ Rate Limiting & 보안

  • 속도 제한: IP당 초당 요청 수 제한 — nginx.ingress.kubernetes.io/limit-rps: "10"
  • IP 허용목록: whitelist-source-range: "10.0.0.0/8,192.168.0.0/16" — 관리자 페이지 보호에 필수
  • 기본 인증(Basic Auth): Secret과 어노테이션 조합으로 간단한 접근 통제
  • WAF 연동: ModSecurity(NGINX), Coraza 등을 활성화해 OWASP 룰셋 기반 공격 차단
  • CORS 처리: enable-cors: "true"로 백엔드 수정 없이 CORS 헤더 부여

⑦ 관측성 (Observability)

모든 외부 트래픽이 지나가는 길목이므로 클러스터에서 가장 가치 있는 메트릭 수집 지점입니다. 대부분의 컨트롤러가 Prometheus 메트릭 엔드포인트를 내장하며, 요청 수·지연시간(p95/p99)·상태코드 분포·액세스 로그를 제공합니다. AIOps 관점에서는 Ingress 메트릭이 이상 탐지의 1차 신호원이 됩니다.

⑧ 어노테이션 — 양날의 검

지금까지 본 것처럼 표준 Ingress 스펙은 의도적으로 최소한(호스트/경로/TLS)만 정의했고, 나머지 고급 기능은 전부 컨트롤러별 어노테이션으로 구현됩니다. 이것이 Ingress의 최대 강점이자 최대 약점입니다.

장점

  • 표준이 가볍고 배우기 쉬움
  • 컨트롤러가 자유롭게 혁신 가능

단점

  • 어노테이션은 이식성이 없음 — NGINX 어노테이션은 Traefik에서 무시됨
  • 타입 검증이 안 되는 문자열이라 오타가 조용히 무시됨
  • 복잡한 설정이 어노테이션 수십 줄로 얽히는 "어노테이션 지옥"

이 한계를 근본적으로 해결하려는 것이 섹션 14의 Gateway API입니다.

Kubernetes Steering/Security 위원회는 "은퇴 후 계속 사용하는 것은 사용자를 공격에 노출시키는 것"이라는 이례적으로 강한 표현의 공동 성명을 냈습니다. 기존 배포는 계속 동작하지만, 학습은 하되 신규 도입은 피하고, 운영 중이라면 Gateway API 구현체(Envoy Gateway, kgateway 등)나 유지보수되는 다른 Ingress Controller로의 마이그레이션을 계획해야 합니다. 마이그레이션 도구로는 ingress2gateway 1.0이 공식 제공됩니다(섹션 14 참조). 이 강의에서 ingress-nginx를 예제로 쓰는 이유는 여전히 가장 많은 클러스터에서 돌아가고 있고, 개념 학습용으로 가장 풍부한 자료를 갖고 있기 때문입니다.

13트러블슈팅 가이드

현장에서 가장 자주 만나는 증상 6가지와 진단 순서입니다. 접었다 펴며 체크리스트로 활용하세요.

증상 1. Ingress를 만들었는데 아무 반응이 없다 (연결 거부/타임아웃)
  1. 컨트롤러가 있긴 한가? → kubectl get pods -n ingress-nginx (없으면 그게 원인. 압도적 1위 원인입니다)
  2. IngressClass가 맞는가? → kubectl get ingressclass와 Ingress의 ingressClassName 비교. 클래스 불일치 시 컨트롤러가 조용히 무시합니다.
  3. ADDRESS가 채워졌는가? → kubectl get ingress의 ADDRESS 열이 비어 있으면 컨트롤러가 아직 처리하지 않은 것.
  4. 컨트롤러 Service의 EXTERNAL-IP는? → <pending> 상태라면 LB 프로비저닝 실패(온프레미스에서 MetalLB 미설치가 흔한 원인).
증상 2. 404 Not Found (기본 백엔드 응답)

컨트롤러까지는 도달했지만 어떤 규칙에도 매칭되지 않은 상태입니다.

  1. Host 헤더 확인: IP로 직접 접속하면(http://1.2.3.4/) Host가 IP라서 host 규칙에 안 걸립니다. curl -H "Host: ..."로 테스트하세요.
  2. 경로/pathType 확인: Exact인데 트레일링 슬래시가 다르거나, Prefix 요소 단위 규칙을 오해한 경우.
  3. 네임스페이스 확인: Ingress와 Service가 같은 네임스페이스인가?
증상 3. 502 Bad Gateway / 503 Service Unavailable

라우팅은 성공했지만 백엔드가 응답하지 못하는 상태입니다.

  1. kubectl get endpoints <svc명> → 엔드포인트가 비어 있으면 Service의 selector와 Pod 라벨 불일치, 또는 Pod이 Not Ready.
  2. Service의 targetPort와 컨테이너 실제 리스닝 포트 불일치 여부.
  3. Pod의 readinessProbe 실패 여부 → kubectl describe pod.
  4. NetworkPolicy가 컨트롤러 네임스페이스 → 앱 네임스페이스 트래픽을 차단하고 있지 않은지.
증상 4. HTTPS 인증서 오류 (Fake Certificate)

브라우저에 "Kubernetes Ingress Controller Fake Certificate"가 보이면 TLS 설정이 매칭되지 않아 컨트롤러의 기본 자체 서명 인증서가 나가는 것입니다.

  1. Secret이 Ingress와 같은 네임스페이스에 있는가?
  2. Secret 타입이 kubernetes.io/tls인가? (Opaque면 안 됨)
  3. spec.tls[].hosts와 rules[].host, 그리고 인증서의 CN/SAN이 일치하는가?
  4. 컨트롤러 로그에서 kubectl logs -n ingress-nginx deploy/ingress-nginx-controller | grep -i cert 확인.
증상 5. 어노테이션이 적용되지 않는다
  1. 접두사 확인: nginx.ingress.kubernetes.io(커뮤니티) vs nginx.org(F5) — 다른 컨트롤러의 어노테이션은 조용히 무시됩니다.
  2. 오타 확인: 어노테이션은 검증 없는 문자열입니다. 오타가 나도 에러가 발생하지 않습니다.
  3. 값은 문자열로: canary-weight: 10이 아니라 "10"처럼 따옴표 필요.
  4. 컨트롤러 로그에서 reject/invalid 메시지 확인.
증상 6. 대용량 업로드 실패 (413) / WebSocket 끊김
  1. 413 Request Entity Too Large: NGINX 기본 본문 제한(1MB) 때문. nginx.ingress.kubernetes.io/proxy-body-size: "50m"로 상향.
  2. WebSocket/SSE 60초 후 끊김: 기본 read timeout 때문. proxy-read-timeout, proxy-send-timeout을 "3600" 등으로 상향. LLM 스트리밍 응답을 Ingress 뒤에 둘 때도 똑같이 겪는 문제입니다.
  3. gRPC: backend-protocol: "GRPC" 어노테이션 필요.

14Gateway API — Ingress의 미래

Ingress의 구조적 한계를 해결하기 위해 Kubernetes SIG-Network가 설계한 차세대 표준입니다. 2026년 현재, "배워두면 좋은 것"이 아니라 "지금 배워야 하는 것"이 되었습니다.

Ingress의 3가지 구조적 한계

  • 표현력 부족: 표준 스펙은 호스트/경로/TLS뿐. 헤더 매칭, 트래픽 분할, 재시도 등은 전부 비표준 어노테이션.
  • 역할 분리 불가: 인프라 관리자와 앱 개발자의 설정이 한 리소스에 뒤엉킴. RBAC로 나눌 수 없음.
  • HTTP 전용: TCP/UDP/gRPC/TLS passthrough를 표준으로 다룰 수 없음.

Gateway API의 3계층 역할 분리

그림 4. Gateway API의 persona 기반 3계층 모델

같은 요구사항, 두 가지 문법

canary — NGINX 전용 어노테이션에 의존
kind: Ingress
metadata:
  annotations:
    nginx.ingress.kubernetes.io/canary: "true"          # 비표준!
    nginx.ingress.kubernetes.io/canary-weight: "10"    # 비표준!
# → Traefik으로 바꾸면 전부 다시 작성해야 함
httproute.yaml — 가중치가 표준 필드
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: shop-route
spec:
  parentRefs:
  - name: main-gateway       # 플랫폼 팀이 만든 Gateway에 연결
  hostnames: [shop.example.com]
  rules:
  - matches:
    - path: { type: PathPrefix, value: / }
    backendRefs:
    - name: web-front-v1
      port: 80
      weight: 90              # 표준 스펙으로 카나리!
    - name: web-front-v2
      port: 80
      weight: 10
# → 어떤 Gateway API 구현체로 바꿔도 그대로 동작
📚 2026년의 마이그레이션 현실

ingress-nginx EOL(2026.3)과 함께 공식 변환 도구 ingress2gateway 1.0이 릴리스되어 30개 이상의 주요 어노테이션을 HTTPRoute로 자동 변환해 줍니다. 다만 변환 불가능한 어노테이션(snippet 등)은 수동 재설계가 필요하므로, "YAML 번역"이 아니라 단계적 컨트롤 플레인 이전 프로젝트로 접근해야 합니다. Ingress API 자체는 K8s 코어에 남아 있으므로 Traefik·Kong 등 유지보수되는 컨트롤러 위의 Ingress는 계속 유효합니다 — 급한 것은 "Ingress API 폐기"가 아니라 "ingress-nginx 컨트롤러 이탈"입니다.

15확인 퀴즈 — 10문항

배운 내용을 점검해 봅시다. 보기를 클릭하면 즉시 정답 여부와 해설이 표시됩니다. 틀려도 괜찮습니다 — 해설이 진짜 공부입니다.

QUIZ 01 / 10
Ingress 리소스를 클러스터에 배포했는데 아무 일도 일어나지 않습니다. 가장 가능성 높은 원인은?
정답: Ingress Controller 미설치. Ingress 리소스는 "규칙 선언문(종이)"일 뿐이고, 이를 읽어 실행하는 것은 Controller(직원)입니다. Controller가 없으면 리소스는 생성되지만 아무 동작도 하지 않습니다. kubectl apply가 성공했다면 문법 오류는 아니며, 백엔드 Service는 ClusterIP면 충분합니다.
QUIZ 02 / 10
Service 타입 중 클라우드 환경에서 서비스 1개당 로드밸런서(와 비용)가 1개씩 생기는 타입은?
정답: LoadBalancer. LoadBalancer 타입은 서비스마다 클라우드 LB를 하나씩 프로비저닝합니다. 서비스가 30개면 LB도 30개 — 비용과 IP가 함께 폭증합니다. Ingress는 LB 1개 뒤에서 L7 라우팅으로 여러 서비스를 나눠주기 때문에 이 문제를 해결합니다.
QUIZ 03 / 10
Ingress가 동작하는 OSI 계층과, 그 계층에서 볼 수 있는 정보의 조합으로 옳은 것은?
정답: L7. Ingress의 핵심 가치는 HTTP를 "이해"한다는 것입니다. Host 헤더(shop.example.com vs admin.example.com)와 경로(/api vs /)를 읽고 서로 다른 Service로 분기할 수 있습니다. L4(Service)는 IP:포트만 볼 수 있어 이런 분기가 불가능합니다.
QUIZ 04 / 10
pathType: Prefix, path: /api 규칙이 있을 때, 매칭되지 않는 요청은?
정답: /apiserver/health. Prefix 매칭은 문자열이 아니라 경로 요소(path element) 단위로 비교합니다. /api는 [api]라는 요소이고 /apiserver는 [apiserver]라는 전혀 다른 요소입니다. 문자열로는 "api"로 시작하지만 요소가 다르므로 매칭되지 않습니다 — 시뮬레이터에서 직접 확인했던 함정이죠.
QUIZ 05 / 10
하나의 클러스터에 NGINX와 Traefik 컨트롤러를 함께 설치했습니다. 특정 Ingress 리소스를 Traefik이 처리하게 하려면?
정답: ingressClassName 지정. IngressClass는 "이 규칙서를 어느 직원(컨트롤러)이 처리할지"를 지정하는 이름표입니다. 여러 컨트롤러 공존은 흔한 패턴이며(예: 내부용/외부용 분리), 각 Ingress는 ingressClassName으로 담당 컨트롤러를 선택합니다.
QUIZ 06 / 10
TLS 종료(TLS Termination)에 대한 설명으로 옳은 것은?
정답: Controller에서 복호화. 인증서 관리를 정문(Ingress) 한 곳으로 모으는 패턴입니다. 개별 앱은 TLS를 신경 쓸 필요가 없어지고, cert-manager와 결합하면 발급/갱신까지 자동화됩니다. 내부 구간까지 암호화가 필요하면 mTLS(서비스 메시)를 추가로 고려합니다.
QUIZ 07 / 10
nginx.ingress.kubernetes.io/canary-weight: "10" 어노테이션의 효과는?
정답: 10% 카나리 배포. 신버전을 소수 트래픽으로 먼저 검증하고 문제 없으면 비율을 올려가는 점진적 배포 전략입니다. 단, 이는 NGINX 전용 어노테이션이라 다른 컨트롤러에선 동작하지 않습니다 — Gateway API의 HTTPRoute weight는 이를 표준 스펙으로 해결합니다.
QUIZ 08 / 10
브라우저에서 502 Bad Gateway가 반복됩니다. 문제의 위치로 가장 가능성 높은 곳은?
정답: 백엔드 문제. 502/503은 "Controller까지는 도달했지만, 그 뒤로 전달할 곳이 없다"는 신호입니다. kubectl get endpoints로 Service에 연결된 Pod IP가 있는지부터 확인하세요. host 오타면 404, 인증서 문제면 TLS 경고가 나타납니다 — 에러 코드가 문제 위치를 알려줍니다.
QUIZ 09 / 10
2026년 3월 EOL을 맞은 ingress-nginx를 운영 중인 팀이 취해야 할 조치로 가장 적절한 것은?
정답: 마이그레이션 계획. EOL된 것은 ingress-nginx 컨트롤러이지 Ingress API가 아닙니다. Traefik·Kong 등 유지보수되는 컨트롤러로 옮기거나, ingress2gateway 도구를 활용해 Gateway API로 전환하는 것이 권장 경로입니다. 인터넷에 노출된 무패치 프록시를 방치하는 것은 심각한 보안 리스크입니다.
QUIZ 10 / 10
Gateway API가 Ingress의 한계를 개선한 방식이 아닌 것은?
정답: "NGINX 엔진 통일"은 사실이 아님. Gateway API는 스펙(인터페이스)의 표준화이지 구현 엔진의 통일이 아닙니다. Envoy Gateway, Cilium, Istio, Traefik, kgateway 등 다양한 엔진이 같은 표준을 구현하며, 덕분에 YAML을 거의 그대로 두고 구현체를 교체할 수 있는 이식성이 생깁니다.
