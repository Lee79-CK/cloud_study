Module 01 · Mental Model

먼저, “무엇이 무엇 안에 있는가”부터

Kubernetes 네트워크가 어려운 가장 큰 이유는 네트워크 장치 하나만 보는 것이 아니라, 물리 네트워크·가상 네트워크·Linux 커널·Kubernetes 추상화가 겹쳐 있기 때문입니다. 아래 한 문장만 잡고 시작하면 훨씬 쉬워집니다.

핵심 문장: Container는 Pod의 네트워크를 공유하고, Pod는 Node에서 실행되며, Node는 Cluster에 속하고, Node의 네트워크 기능은 OS Kernel 위에서 동작하며, OS는 VM 또는 물리 서버 위에서 실행됩니다.

포함 관계

H/W → VM → OS → Node → Cluster는 실행 기반의 관점이고, Cluster가 Node를 묶으며 Node가 Pod를 실행하고 Pod가 Container를 담습니다.

📍

IP의 소유 위치

Node IP는 보통 VM의 vNIC 또는 물리 NIC에 연결됩니다. Pod IP는 Pod Network Namespace에 연결되고, Service ClusterIP는 일반적으로 가상 VIP입니다.

🚚

패킷의 이동

같은 Pod면 localhost, 같은 Node면 커널 내부 연결, 다른 Node면 CNI가 구성한 경로, Service면 Service dataplane을 거칩니다.

주의: “Node = VM”은 자주 맞지만 항상 맞지는 않습니다. Node는 Kubernetes의 작업 머신 역할이며, VM일 수도 있고 Bare Metal 물리 서버일 수도 있습니다.

Module 03 · Cluster Anatomy

Cluster는 “하나의 서버”가 아니다

Kubernetes Cluster는 Control Plane과 Worker Node들의 집합입니다. 관리형 Kubernetes(AKS/EKS/GKE 등)에서는 Control Plane을 Cloud 사업자가 관리하는 경우가 많지만, Worker Node와 워크로드 네트워크는 여전히 사용자가 이해해야 합니다.

Scheduler는 길을 만들지 않는다

Scheduler는 Pod를 배치할 Node를 선택합니다. 실제 Pod 네트워크 인터페이스와 IP 구성은 Node의 Container Runtime과 CNI 연동을 통해 이뤄집니다.

kubelet은 Node의 현장 관리자

API Server에서 PodSpec을 확인하고 Runtime에 Container 생성을 요청합니다. Pod Sandbox와 네트워크 준비가 완료되어야 Container가 정상 실행됩니다.

네트워크 구현은 교체 가능

Kubernetes는 네트워크 모델을 요구하지만 모든 구현을 직접 제공하지는 않습니다. CNI 플러그인과 Cloud 네트워크 구현이 실제 경로를 구성합니다.

Module 04 · Kubernetes Network Model

Kubernetes가 해결해야 하는 4가지 통신 문제

공식 Kubernetes 문서는 Cluster Networking의 핵심 문제를 네 가지로 구분합니다. 초보자는 이 네 가지를 분리해서 이해하면 Service와 Ingress가 섞이지 않습니다.

1

Container ↔ Container

같은 Pod의 Container들은 하나의 Network Namespace를 공유합니다. 서로 localhost:port로 통신합니다.

2

Pod ↔ Pod

같은 Node 또는 다른 Node에 있어도 Pod IP를 통해 통신합니다. CNI가 연결과 경로를 구현합니다.

3

Pod ↔ Service

변하는 Pod 집합 앞에 안정적인 이름과 가상 IP를 제공합니다. EndpointSlice가 실제 backend를 추적합니다.

4

External ↔ Service

LoadBalancer, NodePort, Ingress, Gateway API 등을 통해 Cluster 외부 트래픽을 내부 Service로 전달합니다.

기본 네트워크 모델: 각 Pod는 고유한 Cluster 사설 IP를 가지며, 의도적인 네트워크 분리가 없다면 Pod 간 직접 통신을 기대합니다. 같은 Pod의 Container는 동일한 Network Namespace를 공유하므로 localhost를 사용합니다.

Network Namespace

인터페이스, IP, Route, ARP/Neighbor, 포트 공간 등을 논리적으로 분리하는 Linux Kernel 기능입니다. Pod는 자신만의 네트워크 시야를 갖습니다.

veth pair

가상의 양끝 케이블입니다. 한쪽은 Pod Namespace의 eth0, 다른 쪽은 Node 쪽에 연결되어 패킷을 커널 네트워크로 전달합니다.

CNI

Container Runtime이 Pod Sandbox를 준비할 때 호출하는 네트워크 플러그인 규격입니다. 인터페이스 추가·IP 할당·Route 구성·정리 등을 구현합니다.

중요: CNI는 “특정 제품 이름”이 아니라 인터페이스 규격입니다. Calico, Cilium, Azure CNI, AWS VPC CNI 등은 서로 다른 방식으로 Kubernetes 네트워크를 구현합니다.

Module 06 · Packet Journey Simulator

패킷은 실제로 어떤 길을 지나갈까?

시나리오를 선택하고 ‘다음 단계’를 눌러 패킷이 지나가는 논리 경로를 확인하세요. 실제 구현은 CNI·Cloud·Service dataplane에 따라 달라질 수 있지만, 문제를 분해하는 기본 흐름은 같습니다.

같은 Pod

Pod 내부 Container 간 통신은 별도 Pod Network를 나가지 않습니다. 같은 Namespace의 loopback 또는 동일 인터페이스를 공유합니다.

다른 Node

송신 Pod에서 나온 패킷은 Node 커널과 CNI가 구성한 경로를 따라 Node 간 네트워크를 건너 목적지 Pod로 전달됩니다.

Service 경유

클라이언트는 안정적인 Service 이름/IP를 사용합니다. dataplane은 Service VIP를 실제 EndpointSlice의 Pod IP 중 하나로 연결합니다.

Module 07 · Service, DNS & North-South

Pod는 사라져도, Service는 안정적인 입구를 제공한다

Deployment가 Pod를 교체하면 Pod 이름과 IP는 달라질 수 있습니다. Service는 Label Selector로 backend Pod 집합을 찾고, EndpointSlice에 현재 backend 주소를 반영하여 안정적인 접근 지점을 제공합니다.

Service 연결 사슬: Client → Service DNS/ClusterIP → Service dataplane(kube-proxy 또는 구현별 대체) → EndpointSlice → 선택된 Pod IP:targetPort

Service ClusterIP는 왜 “가상 IP”인가?

일반 서버 IP

보통 NIC에 IP가 설정되고, ARP/Neighbor와 Route를 통해 해당 NIC로 패킷이 도착합니다.

ClusterIP

일반적으로 특정 Pod나 NIC에 고정되어 있지 않습니다. 각 Node의 Service dataplane 규칙이 VIP 목적지 트래픽을 backend로 전달합니다.

구현 차이

iptables, IPVS, nftables, eBPF 기반 구현 등 실제 처리 방식은 환경에 따라 다릅니다. 개념적으로는 안정적 VIP→동적 backend 매핑입니다.

Module 08 · Network Security

“통신 가능”이 기본이라면, “필요한 통신만 허용”은 정책으로

NetworkPolicy는 Pod 중심으로 L3/L4 트래픽 허용 규칙을 선언합니다. 단, 정책 Resource를 만든다고 자동 적용되는 것이 아니라 CNI/네트워크 플러그인이 NetworkPolicy enforcement를 지원해야 합니다.

통신 Matrix 실험

Default deny egress 주의: DNS egress도 차단될 수 있으므로, 필요한 DNS 목적지와 포트를 별도로 허용해야 합니다.

최소 허용 예시

apiVersion: networking.k8s.io/v1 kind: NetworkPolicy metadata: name: allow-frontend-to-api namespace: shop spec: podSelector: matchLabels: app: api policyTypes: [Ingress] ingress: - from: - podSelector: matchLabels: app: frontend ports: - protocol: TCP port: 8080

Namespace 경계

Namespace가 자동 방화벽은 아닙니다. 별도 NetworkPolicy가 없다면 Namespace가 다르더라도 통신 가능한 구현이 일반적입니다.

Ingress vs Egress

Ingress는 선택된 Pod로 들어오는 방향, Egress는 선택된 Pod에서 나가는 방향입니다. 양쪽 정책이 모두 영향을 줄 수 있습니다.

Label 기반

Pod·Namespace Label로 대상을 선택합니다. Label 변경이 곧 보안 경계 변경이 될 수 있으므로 Label 거버넌스가 중요합니다.

Layer 7 정책

기본 NetworkPolicy는 주로 L3/L4입니다. HTTP 메서드·경로 같은 L7 정책은 CNI·Service Mesh 등 구현별 기능을 검토합니다.

Module 09 · Cloud Network Integration

Kubernetes 네트워크는 VPC/VNet 바깥에 떠 있지 않다

Cloud 환경에서는 Kubernetes 네트워크가 VPC/VNet, Subnet, Route Table, NSG/SG, Load Balancer, NAT Gateway, Firewall, Private DNS와 연결됩니다. 장애 원인이 Cluster 안이 아니라 Cloud 네트워크 계층에 있을 수 있습니다.

North-South

Cluster 외부 ↔ 내부 트래픽입니다. Public/Internal LB, Ingress/Gateway, Firewall/WAF, DNS, SNAT/DNAT 경로를 함께 봅니다.

East-West

Pod ↔ Pod, Service ↔ Service, Node ↔ Node 등 내부 수평 트래픽입니다. CNI, NetworkPolicy, Service Mesh, 라우팅이 핵심입니다.

Egress

Pod가 외부 API·Internet·온프레미스로 나가는 경로입니다. Source IP 보존 여부, SNAT 지점, Firewall 허용, 비대칭 라우팅을 확인합니다.

kubectl get pod -o wide kubectl describe pod <pod> kubectl logs <pod> kubectl exec -it <pod> -- sh # Container 내부 ss -lntp # 이미지에 없으면 netstat 또는 애플리케이션 로그 활용 curl -v http://127.0.0.1:8080/health

애플리케이션이 127.0.0.1에만 바인딩되어 있으면 다른 Pod나 Service에서 접근하지 못할 수 있습니다. 일반적으로 외부 접근이 필요하면 0.0.0.0 또는 Pod 인터페이스에 Listen해야 합니다.

kubectl get svc web -o wide kubectl describe svc web kubectl get endpointslice -l kubernetes.io/service-name=web -o wide kubectl get pod -l app=web --show-labels

EndpointSlice가 비어 있으면 Service selector와 Pod label, Pod readiness를 먼저 확인합니다. port와 targetPort 혼동도 매우 흔합니다.

kubectl exec -it <client-pod> -- cat /etc/resolv.conf kubectl exec -it <client-pod> -- nslookup web.default.svc.cluster.local kubectl get pods -n kube-system -l k8s-app=kube-dns kubectl get svc,endpointslice -n kube-system | grep -E 'dns|kube-dns'

Default deny egress를 적용했다면 UDP/TCP 53 또는 환경별 DNS 경로를 허용했는지 확인합니다.

kubectl get networkpolicy -A kubectl describe networkpolicy -n <namespace> kubectl get ns --show-labels kubectl get pod -n <namespace> --show-labels

정책의 podSelector, namespaceSelector, 포트, ingress/egress 방향을 확인합니다. CNI가 해당 정책을 실제로 enforcement하는지도 확인합니다.

kubectl get nodes -o wide kubectl -n kube-system get pods -o wide kubectl describe node <node> # Node 접근이 허용되는 환경에서 ip addr ip route ip link ss -s # CNI별 진단 명령은 제품 문서 확인

Pod가 특정 Node에서만 실패하면 Node별 CNI Agent, Route, conntrack, MTU, NIC 상태를 비교합니다.

kubectl get svc -A | grep LoadBalancer kubectl describe svc <lb-service> kubectl get ingress -A kubectl describe ingress <ingress> kubectl get pods -n <ingress-controller-namespace> kubectl logs -n <ingress-controller-namespace> deploy/<controller>

Ingress Resource만 있고 Controller가 없거나, Controller Service의 External/Internal IP가 Pending이거나, LB Health Probe가 막힌 경우를 확인합니다.

장애 분석 원칙: 한 번에 하나의 경계만 건너 테스트하세요. localhost → Pod IP → Service IP/DNS → Ingress/LB IP → 외부 DNS 순서로 가면 어디부터 실패하는지 명확해집니다.
Module 11 · Hands-on Lab

직접 확인하는 최소 네트워크 실습

테스트 Cluster에서 실행하세요. 운영 환경에서는 Namespace·보안 정책·Image 허용 정책을 검토하고 적용해야 합니다. 아래 실습은 기본 개념 확인 목적입니다.

실습 Namespace 생성

kubectl create namespace net-lab

web Deployment와 ClusterIP Service 생성

kubectl -n net-lab create deployment web --image=nginx:stable kubectl -n net-lab scale deployment web --replicas=2 kubectl -n net-lab expose deployment web --port=80 --target-port=80 kubectl -n net-lab get pod,svc -o wide

운영에서는 Image digest pinning, Private Registry, Resource request/limit, SecurityContext 등을 추가합니다.

client Pod에서 Service DNS와 ClusterIP 접근

kubectl -n net-lab run client \ --image=curlimages/curl:latest \ --restart=Never -- sleep 3600 kubectl -n net-lab exec -it client -- nslookup web kubectl -n net-lab exec -it client -- curl -I http://web

Pod IP 직접 접근과 Service 접근 비교

kubectl -n net-lab get pod -l app=web -o wide # 출력된 Pod IP 중 하나 사용 kubectl -n net-lab exec -it client -- curl -I http://<POD_IP> # 안정적인 Service 이름 사용 kubectl -n net-lab exec -it client -- curl -I http://web

Pod를 삭제하고 재생성한 뒤 Pod IP는 바뀌어도 Service 이름이 계속 동작하는지 확인하세요.

Service → EndpointSlice 매핑 확인

kubectl -n net-lab get svc web -o yaml kubectl -n net-lab get endpointslice \ -l kubernetes.io/service-name=web -o yaml

NetworkPolicy 적용 전 CNI 지원 여부 확인

# 설치된 CNI/네트워크 Agent 확인 예시 kubectl -n kube-system get pods -o wide # 관리형 Kubernetes는 Cloud 문서와 Cluster 네트워크 프로필도 확인

NetworkPolicy 미지원 구현에서는 Resource가 생성되어도 기대한 차단 효과가 없을 수 있습니다.

리소스 정리

kubectl delete namespace net-lab

Appendix B · One-page Summary

이것만 기억해도 큰 그림은 잡힌다

계층 관계

  • H/W는 물리 자원을 제공한다.
  • VM은 물리 자원을 논리 머신으로 분리한다.
  • OS Kernel이 실제 Network Namespace·Route·Filter를 실행한다.
  • Node는 Kubernetes가 Pod를 실행하는 작업 머신이다.
  • Cluster는 Control Plane과 Worker Node 집합이다.
  • Pod가 네트워크 단위이며 Container들이 이를 공유한다.

IP 관계

  • Node IP: VM/물리 NIC의 기반 주소
  • Pod IP: Pod Network Namespace의 주소
  • Service ClusterIP: 안정적인 가상 VIP
  • LoadBalancer IP: Cluster 진입용 LB Frontend
  • DNS: 변하는 IP를 숨기는 안정적 이름
  • Pod CIDR·Service CIDR·연결망 CIDR은 중복 금지

통신 경로

  • 같은 Pod: localhost
  • 같은 Node Pod 간: Node Kernel 내부 경로
  • 다른 Node Pod 간: CNI 경로 + Node Network
  • Pod→Service: VIP→Service dataplane→EndpointSlice→Pod
  • 외부→앱: LB/Ingress/Gateway→Service→Pod
  • Pod→외부: Route·SNAT·Firewall·Return path 확인

장애 분석

  • 앱 Listen → Pod Ready → Pod IP
  • Service Selector·Port·EndpointSlice
  • DNS와 CoreDNS
  • NetworkPolicy ingress/egress
  • CNI Agent·Route·MTU·Node 상태
  • Cloud Route·NSG/SG·LB Health·Firewall
