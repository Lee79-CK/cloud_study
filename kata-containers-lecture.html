01 — The Problem

먼저, "컨테이너 격리"가 왜 문제일까?

일반 컨테이너(Docker, runc)의 가장 큰 장점이자, 동시에 가장 큰 약점은 "호스트의 커널 하나를 모두가 나눠 쓴다"는 점입니다.

가상머신(VM)은 게스트마다 자기만의 운영체제 커널을 갖습니다. 무겁지만 그만큼 단단히 분리되죠. 반면 컨테이너는 커널을 새로 띄우지 않고 호스트 커널을 공유합니다. 그래서 수 초가 아니라 수십 밀리초 만에 뜨고, 메모리도 훨씬 적게 먹습니다. 이 가벼움이 컨테이너가 클라우드를 점령한 이유입니다.

문제는 "커널을 공유한다"는 사실 자체가 보안 경계를 약하게 만든다는 데 있습니다. 한 컨테이너가 커널의 취약점을 찔러 탈출(container escape)에 성공하면, 같은 노드 위의 다른 모든 컨테이너가 함께 위험해집니다.

⚠ 실제 사례 — CVE-2019-5736

runc에서 발견된 이 취약점은 컨테이너 안에서 호스트의 runc 바이너리를 덮어써 호스트 권한을 탈취할 수 있게 했습니다. 커널을 공유하는 모델에서는 이런 한 번의 탈출이 전체 노드의 붕괴로 이어질 수 있습니다. 리눅스 커널에는 매년 수백 건의 CVE가 보고됩니다.

그래서 자연스러운 질문이 생깁니다. "컨테이너의 가벼움은 유지하면서, VM 수준의 격리를 얻을 수는 없을까?" 이 질문에 대한 가장 성숙한 답 중 하나가 바로 Kata Containers입니다.

VM

가상머신

격리는 강하지만 부팅이 느리고 무겁다. 게스트마다 풀 OS.

CONTAINER

일반 컨테이너

가볍고 빠르지만 커널을 공유 → 탈출 시 노드 전체 위험.

KATA

Kata Containers

둘의 장점을 결합 — 컨테이너처럼 다루되 VM으로 격리.

02 — The Concept

Kata Containers란 무엇인가

Kata Containers는 각 컨테이너(정확히는 각 파드)를 "초경량 가상머신(microVM)" 안에서 실행하는 오픈소스 컨테이너 런타임입니다.

핵심 아이디어는 단순합니다. 컨테이너를 호스트 커널 위에 그냥 올리는 대신, 아주 가벼운 전용 VM을 하나 띄우고 그 안에서 컨테이너를 돌립니다. 이 VM에는 컨테이너만을 위한 독립된 게스트 커널이 들어 있습니다. 덕분에 컨테이너가 보는 커널과 호스트 커널이 물리적으로 분리됩니다.

결정적인 점은 이 격리가 소프트웨어 약속이 아니라 CPU의 가상화 기능(Intel VT-x / AMD-V)으로 하드웨어가 강제한다는 것입니다. 설령 공격자가 컨테이너를 탈출하더라도, 그는 여전히 작은 VM 안에 갇혀 있을 뿐 호스트에는 닿지 못합니다. 이것이 Kata가 말하는 "두 번째 방어선(second layer of defense)"입니다.

◆ 한 줄 정의

Kata Containers = 컨테이너처럼 느껴지고 동작하지만, 하드웨어 가상화로 격리되는 컨테이너 런타임. Kubernetes·containerd 같은 기존 생태계에 그대로 끼워 쓸 수 있다.

"기존 생태계에 그대로 끼운다"는 게 핵심

Kata가 실용적인 이유는 새 도구를 배우게 만들지 않기 때문입니다. OCI 컨테이너 이미지를 그대로 쓰고, Kubernetes의 CRI 인터페이스와 containerd·CRI-O 같은 기존 런타임에 자연스럽게 연결됩니다. 개발자는 평소처럼 컨테이너 이미지를 빌드하고 파드를 배포할 뿐, 그 아래에서 VM이 뜨고 있다는 걸 거의 의식하지 않습니다.

  • 표준 호환 — OCI 이미지 포맷, Kubernetes CRI, containerd shim v2 인터페이스 준수
  • 다중 하이퍼바이저 — QEMU, Cloud Hypervisor, AWS Firecracker, Dragonball 중 선택 가능
  • 다중 아키텍처 — x86_64(AMD64), ARM(aarch64), IBM p-series·z-series 지원
  • 오픈 거버넌스 — 2017년 12월 출범(Intel Clear Containers + Hyper.sh runV 통합), 현재 OpenInfra 재단 산하 프로젝트
ℹ 버전 메모 (2026년 기준)

현재 안정판은 3.x 계열입니다. 커뮤니티는 런타임을 Go에서 Rust 기반(runtime-rs)으로 다시 쓴 4.0.0 출시를 준비 중이며, 더 안전하고 빠르며 유지보수하기 쉬운 코드베이스를 목표로 합니다. 동작 방식과 설정은 대체로 유지되지만 일부 구현 차이가 있어 마이그레이션 가이드가 함께 준비되고 있습니다.

03 — Under the Hood

내부 구조 들여다보기

"파드를 VM 안에서 돌린다"는 말이 추상적으로 느껴질 수 있습니다. 실제로 kubectl apply 한 줄을 실행했을 때 어떤 부품들이 맞물려 돌아가는지 따라가 봅시다.

핵심 구성 요소

  • kata 런타임 (shim-v2) — containerd/CRI-O가 "이 파드는 kata로 띄워라"라고 호출하면, VM 생성을 지휘하는 지휘자입니다.
  • 하이퍼바이저(VMM) — QEMU·Cloud Hypervisor·Firecracker 등. KVM을 이용해 실제 microVM을 만듭니다.
  • 게스트 커널 + 미니 게스트 OS — VM 안에서 도는 가볍게 튜닝된 리눅스 커널. 컨테이너에 필요한 최소한만 담겨 있어 빠르게 뜹니다.
  • kata-agent — VM 내부에 상주하며, 바깥의 런타임 명령을 받아 VM 안에서 실제 컨테이너 프로세스를 만들고 관리합니다.
  • virtio-fs / virtio-vsock — 호스트와 VM 사이에서 파일과 통신을 안전하게 중계하는 표준 가상 장치들입니다.

파드가 뜨는 순서 (간단 버전)

  • Kubelet이 CRI를 통해 containerd에게 파드 생성을 요청한다.
  • 파드의 runtimeClassName이 kata이므로, containerd는 일반 runc 대신 kata shim을 호출한다.
  • kata shim이 하이퍼바이저를 띄워 전용 microVM을 부팅한다 (자체 게스트 커널 포함).
  • VM 안의 kata-agent가 컨테이너 이미지를 풀어 실제 컨테이너 프로세스를 시작한다.
  • 이후 로그·exec·네트워킹은 virtio 가상 장치를 통해 호스트와 안전하게 오간다.
◆ 한눈에 보는 비유

일반 컨테이너가 "공용 사무실의 칸막이 책상"이라면, Kata 컨테이너는 "건물 안에 둔 독립 금고실"입니다. 출입은 똑같이 편하지만, 벽 자체가 콘크리트(=하드웨어 가상화)라 옆방으로 넘어갈 수 없습니다.

이 모든 부품이 하는 일은 결국 하나입니다. 개발자에게는 평범한 컨테이너처럼 보이게 하면서, 그 아래에 진짜 VM 경계를 몰래 깔아 두는 것. 다음 섹션의 데모에서 이 경계가 다른 방식들과 어떻게 다른지 직접 비교해 봅니다.

04 — The Comparison

다른 격리 방식과 비교하기

컨테이너 격리에는 여러 접근이 있습니다. 가장 자주 비교되는 셋은 runc(기본), gVisor(유저공간 커널), Kata(microVM)입니다. 차이의 핵심은 딱 하나, "보안 경계가 어디에 그어지는가"입니다. 아래 데모에서 직접 골라 보세요.

DEMO 컨테이너의 시스템 콜은 어디까지 가닿을까?
런타임을 선택하면, 컨테이너 → 하드웨어로 이어지는 계층 구조와 "보안 경계선"이 어떻게 달라지는지 보여줍니다.

세 가지 방식, 한 장 요약

항목 runc (기본) gVisor Kata Containers
격리 방식 네임스페이스 + cgroup 유저공간 커널(Sentry)이 syscall 가로채기 전용 게스트 커널 + 하드웨어 VM
커널 호스트와 공유 유저공간에서 재구현 VM별 독립 커널
보안 강도 가장 약함 중간(공격면 축소) 강함(하드웨어 경계)
성능 오버헤드 거의 없음(기준선) syscall 위주 워크로드에 중간 I/O·네트워크에서 가장 큼
부팅/콜드스타트 가장 빠름 빠름 느림(VM 부팅 ~수백 ms)
집적 밀도 가장 높음 높음 낮음(VM당 ~128MB 추가)
중첩 가상화 불필요 불필요 필요(KVM/베어메탈 권장)

※ 구체적 수치는 워크로드·하드웨어·하이퍼바이저에 따라 크게 달라집니다. 위 표는 일반적 경향을 보여주는 참고용입니다.

Firecracker는 경쟁자가 아니라 "부품"입니다

혼동하기 쉬운 부분입니다. Firecracker는 AWS가 Rust로 만든 초경량 VMM(가상머신 모니터)으로, AWS Lambda·Fargate를 떠받치며 약 100~200ms 만에 microVM을 띄웁니다. 다만 Firecracker는 VMM 그 자체일 뿐이라, Kubernetes에서 직접 운영하려면 상당한 오케스트레이션을 직접 만들어야 합니다.

여기서 Kata의 역할이 분명해집니다. Kata는 Firecracker·Cloud Hypervisor·QEMU 같은 VMM들을 Kubernetes 위에 매끄럽게 얹어 주는 통합 계층입니다. 즉 "Firecracker vs Kata"가 아니라, "Kata가 Firecracker를 하이퍼바이저로 골라 쓸 수 있다"가 맞는 그림입니다.

WHEN gVisor

gVisor가 어울릴 때

중첩 가상화가 불가능한 환경, 더 높은 집적 밀도가 필요하고 syscall이 가벼운 짧은 작업(예: CI 잡), 가장 간단한 도입 경로를 원할 때.

WHEN Kata

Kata가 어울릴 때

VM 수준의 강한 하드웨어 격리가 필요하고, Kubernetes에서 운영 부담을 최소화하며 멀티테넌트·신뢰할 수 없는 워크로드를 돌릴 때.

05 — Hands On

실제로 어떻게 쓰는가

Kata의 가장 큰 매력은 "기존 워크플로를 거의 바꾸지 않는다"는 점입니다. 큰 그림은 세 단계입니다 — ① 노드에 Kata 설치 → ② Kubernetes에 RuntimeClass 등록 → ③ 파드에서 그 클래스를 지정. 아래 탭을 차례로 보세요.

ℹ 사전 조건

워커 노드가 하드웨어 가상화를 지원해야 합니다 — 베어메탈이거나, 클라우드라면 중첩 가상화(nested virtualization)가 켜진 인스턴스여야 합니다. 노드에서 ls /dev/kvm이 보이는지 확인하세요.

# 가장 권장되는 방식: kata-deploy (DaemonSet)로 노드에 자동 설치
# 모든 워커 노드에 Kata 바이너리/커널/하이퍼바이저를 배포한다
kubectl apply -k "github.com/kata-containers/kata-containers/tools/packaging/kata-deploy/kata-deploy/base?ref=stable-3.x"

# 배포 상태 확인 — 각 노드에 kata-deploy 파드가 Running 이어야 함
kubectl -n kube-system get pods -l name=kata-deploy

# 설치가 끝나면 containerd에 kata 핸들러가 자동 등록된다
# RuntimeClass: "kata라는 이름표를 누르면 kata 핸들러를 쓴다"고 선언
apiVersion: node.k8s.io/v1
kind: RuntimeClass
metadata:
  name: kata
# containerd 설정의 [plugins...runtimes.kata] 핸들러 이름과 일치시킨다
handler: kata

# 등록 확인
# $ kubectl get runtimeclass
# NAME   HANDLER   AGE
# kata   kata      10s
# 평소 파드 정의에 딱 한 줄만 추가하면 끝 — runtimeClassName
apiVersion: v1
kind: Pod
metadata:
  name: secure-workload
spec:
  runtimeClassName: kata   # ← 이 한 줄이 "이 파드는 VM 안에서 돌려라"
  containers:
    - name: app
      image: my-app:latest
      resources:
        requests:
          # VM 기본 오버헤드(~128MB)를 자원 계획에 반영하자
          cpu: "500m"
          memory: "384Mi"
        limits:
          cpu: "2"
          memory: "1Gi"
# 정말 별도 커널 안에서 도는지 검증해 보자

# 1) 호스트 커널 버전
$ uname -r
6.8.0-45-generic

# 2) Kata 파드 안의 커널 버전 — 호스트와 다르면 성공!
$ kubectl exec secure-workload -- uname -r
6.1.62   # ← VM 전용 게스트 커널 (호스트와 다름)

# 3) 파드 안에서 본 프로세스 — VM 안에 격리되어 PID 1부터 깔끔
$ kubectl exec secure-workload -- ps aux
# 호스트의 다른 프로세스가 전혀 보이지 않는다
◆ 운영 팁

모든 워크로드를 Kata로 돌릴 필요는 없습니다. 신뢰할 수 없거나 민감한 워크로드만 runtimeClassName: kata로 지정하고, 나머지는 기본 runc로 두는 혼합 운영이 가장 흔하고 합리적인 패턴입니다.

06 — Use Cases

어디에 쓰는가

Kata는 "강한 격리가 가벼움보다 더 중요한" 상황에서 빛납니다. 대표적인 응용 분야를 정리하면 다음과 같습니다.

멀티테넌트 / 매니지드 SaaS

여러 고객의 워크로드가 같은 클러스터에 섞여 도는 환경. 한 테넌트의 침해가 다른 테넌트로 번지지 않도록 테넌트 간 하드웨어 경계를 제공합니다.

신뢰할 수 없는 코드 실행

사용자가 제출한 CI/CD 잡, 빌드 파이프라인, 임의 스크립트 실행 등 "내가 작성하지 않은 코드"를 돌릴 때 탈출 위험을 봉쇄합니다.

AI 에이전트 / LLM 생성 코드

LLM이 만들어 낸 코드를 실행하는 에이전트는 본질적으로 임의 코드를 돌립니다. 적대적 프롬프트로 악성 코드가 생성될 수 있어 강한 샌드박스가 필요합니다.

규제 / 컴플라이언스 워크로드

금융·의료·공공처럼 강한 격리 증빙이 요구되는 분야. VM 경계는 감사·인증 요건을 충족하기에 유리합니다.

Confidential Containers (CoCo)

Kata는 기밀 컴퓨팅의 기반이 됩니다. TEE(신뢰 실행 환경)와 결합해 메모리 내용까지 호스트로부터 보호하는 워크로드를 실현합니다.

엣지 / 이기종 클러스터

x86·ARM 혼합 노드에서도 동작하므로, 다양한 하드웨어가 섞인 환경에서 일관된 격리 정책을 적용할 수 있습니다.

◆ 인프라 엔지니어 관점의 한 줄

"이 워크로드가 뚫렸을 때 옆 워크로드와 호스트가 함께 무너져도 괜찮은가?"라는 질문에 "안 된다"라면, 그 워크로드는 Kata 후보입니다.

07 — Trade-offs

장단점과 고려사항

은탄환은 없습니다. Kata는 보안을 위해 약간의 성능·자원·운영 복잡도를 지불합니다. 도입 전에 저울질해야 할 것들입니다.

✓ 얻는 것

  • 하드웨어가 강제하는 강력한 격리(두 번째 방어선)
  • 커널 취약점 탈출이 VM 안에 봉쇄됨
  • OCI/CRI 표준 호환 — 기존 워크플로 유지
  • 워크로드별 선택 적용(혼합 운영) 가능
  • 기밀 컴퓨팅·멀티테넌시의 탄탄한 기반

⚠ 치르는 비용

  • VM당 메모리 오버헤드(기본 ~128MB)로 집적 밀도 하락
  • VM 부팅 탓에 콜드스타트가 느림
  • I/O·네트워크 집약 워크로드에서 오버헤드 큼
  • 중첩 가상화 또는 베어메탈 노드 필요
  • 하이퍼바이저·게스트 커널 등 운영 요소 증가
ℹ 결정 가이드

모든 것을 Kata로 바꾸려 하지 마세요. 격리가 필수인 소수의 워크로드에만 적용하고, 일반 워크로드는 runc로 두는 것이 성능·비용·보안의 균형점입니다. "보안 강도 ↔ 성능 ↔ 운영 복잡도"는 삼각 트레이드오프이며, Kata는 그중 보안 쪽으로 추를 옮기는 선택입니다.
