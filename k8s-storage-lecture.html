CHAPTER 00

강의 개요 ─ 무엇을, 왜, 어떻게

Kubernetes(이하 K8s)에서 "스토리지"라는 단어가 등장하는 순간, 초보자는 갑자기 PV, PVC, StorageClass, CSI 같은 약어 폭풍에 휩쓸리곤 합니다. 이 강의는 그 모든 것을 "우리가 이미 알고 있는 디스크"와 비교하면서, 한 단계씩 풀어 설명합니다.

이 강의에서 다루는 핵심 질문 6가지

  • Q1. 컨테이너는 왜 데이터를 잃어버리는가? 어디로 사라지는가?
  • Q2. 옛날엔 그냥 /data에 저장하면 끝이었는데, 왜 갑자기 복잡해졌나?
  • Q3. Volume, PV, PVC, StorageClass는 도대체 누가, 언제, 왜 만드는가?
  • Q4. RWO vs RWX, 왜 GPU 학습은 RWX가 절실한가?
  • Q5. CSI(Container Storage Interface)는 도대체 무엇을 하는 물건인가?
  • Q6. MLOps 실무에서 데이터셋, 모델 체크포인트는 어떻게 저장하는가?
💡 학습 팁
K8s Storage는 "관심사의 분리(Separation of Concerns)"를 극단적으로 추구합니다. 인프라 팀은 PV/StorageClass를, 개발자는 PVC만 알면 됩니다. 이 분리의 미학을 이해하는 순간, 절반은 끝난 겁니다.
CHAPTER 01

왜 K8s Storage가 필요한가

먼저 가장 기본적인 질문부터. 그냥 컨테이너 안에 파일 만들면 안 되나요? 이 질문에 답하려면 컨테이너의 "수명(lifecycle)"을 이해해야 합니다.

컨테이너의 파일시스템은 휘발성(ephemeral)이다

컨테이너는 이미지(Image)에서 생성된 읽기 전용 레이어 위에 얇은 쓰기 가능 레이어(writable layer)를 하나 얹어서 동작합니다. 컨테이너가 종료되면? 이 쓰기 레이어는 그대로 폐기됩니다. 저장한 파일도 같이 사라지죠.

▶ 시연: 파드가 죽으면 어떻게 되는가?
--:--:--버튼을 순서대로 눌러 컨테이너 라이프사이클을 시뮬레이션해보세요.
⚠️ 핵심 깨달음
이 시뮬레이션의 결론은 명확합니다. 컨테이너의 파일시스템은 컨테이너의 생명과 함께 사라진다. 노드 장애, 파드 재스케줄링, 무중단 배포(rolling update) ─ 이 모든 일상적인 이벤트에서 데이터가 손실됩니다. 그래서 우리는 "컨테이너 밖에" 데이터를 두는 메커니즘이 필요합니다.

해결책의 본질 ─ 데이터를 "외부 어딘가"에 마운트하기

리눅스의 마운트(mount) 개념을 떠올려보세요. mount /dev/sdb1 /data 처럼, 물리적으로 다른 곳에 있는 디스크를 특정 경로에 "끼워넣을" 수 있죠. K8s의 Volume은 정확히 같은 아이디어입니다 ─ 컨테이너 외부의 저장소를 컨테이너 안의 특정 경로에 끼워넣어서, 컨테이너가 죽어도 데이터가 살아남게 합니다.

CHAPTER 02

전통 인프라 디스크 vs Kubernetes Storage

리눅스 서버 한 대에 디스크를 붙이던 시절과, 수십 대의 노드에 걸쳐 파드가 떠다니는 K8s 시대는 무엇이 다를까요? 모든 차이는 결국 한 단어로 요약됩니다 ─ "동적(dynamic)".

▲ Legacy Infrastructure
전통적인 서버 디스크
  • 📌 서버 한 대에 디스크가 물리적으로 고정
  • 📌 관리자가 SSH로 접속해 fdisk, mkfs, mount를 수동 실행
  • 📌 /etc/fstab으로 부팅 시 자동 마운트
  • 📌 디스크 용량이 부족하면 → 디스크 추가 또는 LVM 확장
  • 📌 애플리케이션은 항상 같은 서버, 같은 경로에서 실행됨
  • 📌 "데이터가 있는 곳에 앱이 있다"
VS
▼ Kubernetes Native
K8s 스토리지
  • 🚀 파드는 어느 노드에든 스케줄링됨 (이동성)
  • 🚀 스토리지는 API로 선언 (PVC YAML 한 장)
  • 🚀 StorageClass가 디스크를 자동 프로비저닝
  • 🚀 용량 부족? → PVC를 수정하면 온라인 확장
  • 🚀 파드가 죽고 다른 노드로 옮겨가도 같은 데이터에 연결
  • 🚀 "앱이 있는 곳에 데이터가 따라온다"

관점의 전환 ─ 누가 무엇을 책임지는가?

관심사 전통 인프라 Kubernetes
디스크 프로비저닝 스토리지 관리자가 LUN 할당, 마운트 StorageClass가 자동으로 처리
앱-디스크 연결 /etc/fstab 수동 편집 PVC를 Pod spec에 선언
장애 시 복구 장애 서버로 디스크 이전 (수동) K8s가 자동으로 다른 노드에 재연결
용량 확장 LVM 또는 RAID 재구성 PVC의 resources.requests.storage 수정
백업/스냅샷 스토리지 벤더 도구 VolumeSnapshot 리소스로 K8s 네이티브 처리
표준 인터페이스 벤더마다 제각각 CSI(Container Storage Interface) 통일
🎯 핵심 패러다임 전환
전통 인프라가 "명령형(imperative)"이라면(이거 해, 저거 해), K8s 스토리지는 "선언형(declarative)"입니다("내가 원하는 상태는 이거야"). YAML로 "10GB SSD 디스크 하나가 필요해"라고 선언하면, K8s가 알아서 만들고 연결합니다.
CHAPTER 03

Volume의 기본 ─ Pod 안에서의 마운트

K8s 스토리지의 가장 작은 단위는 Volume입니다. Pod가 정의될 때 함께 선언되며, Pod 안의 모든 컨테이너가 공유할 수 있습니다. Volume의 수명은 Pod의 수명과 같다는 점을 꼭 기억하세요.

가장 단순한 Volume 예제

YAMLapiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  containers:
  - name: app
    image: nginx
    volumeMounts:              # ⬅ 컨테이너 입장: "어디에 끼워넣을지"
    - name: my-storage
      mountPath: /usr/share/nginx/html
  volumes:                      # ⬅ Pod 입장: "어떤 볼륨을 가져올지"
  - name: my-storage
    emptyDir: {}              # ← 볼륨의 타입

이 YAML은 두 개의 섹션으로 구성됩니다:

  • spec.volumes ─ "이 Pod는 어떤 볼륨들을 가져올 것인가"
  • spec.containers[].volumeMounts ─ "각 컨테이너가 그 볼륨을 어디에 마운트할 것인가"

이 분리는 매우 중요합니다. 하나의 볼륨을 Pod 안의 여러 컨테이너가 공유할 수 있고, 각자 다른 경로에 마운트할 수도 있기 때문이죠.

🔍 리눅스 mount와 1:1 비교
리눅스에서 mount /dev/sdb1 /data를 실행할 때,
• /dev/sdb1이 바로 K8s의 volumes 정의 (어떤 저장소를)
• /data가 K8s의 volumeMounts (어디에 끼워넣을지)
개념적으로 완전히 동일합니다. 다만 K8s는 이 모든 걸 API 객체로 추상화한 것뿐.
CHAPTER 04

Volume의 종류 ─ 용도별 분류

K8s는 다양한 종류의 Volume을 지원합니다. 크게 임시 볼륨(ephemeral)과 영구 볼륨(persistent)로 나뉘며, 용도에 따라 적절히 선택해야 합니다.

Pod와 함께 생성되고 Pod가 삭제되면 함께 사라지는 볼륨들입니다. 캐시, 임시 작업 공간, 컨테이너 간 데이터 공유 등에 사용됩니다.

emptyDir
emptyDir
Pod가 노드에 스케줄링될 때 생성되는 빈 디렉토리. Pod 안 컨테이너끼리 데이터 공유에 최적. 노드의 디스크 또는 메모리(medium: Memory)에 저장.
hostPath
hostPath
노드의 호스트 파일시스템을 직접 마운트. 로그 수집(/var/log), 시스템 메트릭 수집 등 DaemonSet 용도에 한정. 보안 주의
ephemeral / generic
generic ephemeral
Pod와 함께 동적으로 PVC를 생성하지만, Pod가 사라지면 같이 삭제. 큰 임시 스크래치 공간이 필요한 ML 학습 작업에 유용.

예시: emptyDir로 사이드카 패턴 구현

YAMLspec:
  containers:
  - name: producer
    image: data-generator
    volumeMounts:
    - { name: shared, mountPath: /output }
  - name: consumer
    image: data-processor
    volumeMounts:
    - { name: shared, mountPath: /input }
  volumes:
  - name: shared
    emptyDir:
      medium: Memory      # tmpfs 사용 → 매우 빠름
      sizeLimit: 2Gi

설정값, 비밀번호, K8s 메타데이터 등을 파일 형태로 컨테이너에 제공하는 특수 볼륨들입니다.

configMap
configMap
애플리케이션 설정 파일(application.yaml, nginx.conf 등)을 ConfigMap 리소스에서 가져와 컨테이너에 파일로 노출.
secret
secret
비밀번호, API 키, TLS 인증서 등 민감 정보를 Secret 리소스에서 가져옴. base64 인코딩, RBAC로 접근 제어, etcd 암호화 권장.
downwardAPI
downwardAPI
Pod 자신의 메타데이터(이름, IP, 노드명, 리소스 한도 등)를 파일로 노출. 자기 자신을 알아야 하는 앱에 유용.
projected
projected
위 세 가지(configMap, secret, downwardAPI)를 하나의 디렉토리로 합쳐서 마운트. ServiceAccount 토큰 마운트의 표준 방식.

Pod가 삭제되어도 데이터가 보존되는 영구 볼륨. 데이터베이스, 모델 저장소, 사용자 데이터의 핵심.

persistentVolumeClaim
PVC 참조
가장 권장되는 방식. Pod는 PVC만 참조하고, 실제 스토리지는 PV/StorageClass가 처리. 챕터 5에서 깊이 다룸.
nfs (deprecated)
nfs (in-tree)
NFS 서버를 직접 마운트. 과거에는 직접 사용했지만, 이제 NFS CSI 드라이버를 통해 PVC로 사용하는 것이 표준.
csi
CSI Volume
벤더 중립 표준 인터페이스. AWS EBS, GCP PD, Ceph, Longhorn 등 모두 CSI 드라이버 형태로 제공. 챕터 7 참고.
📜 역사적 메모
예전에는 awsElasticBlockStore, gcePersistentDisk처럼 클라우드 벤더별 볼륨 타입이 K8s 코어에 내장(in-tree)되어 있었습니다. 이제는 모두 CSI 드라이버로 외부화(out-of-tree)되어, K8s 1.27부터는 in-tree 드라이버가 제거되거나 CSI Migration을 통해 자동 변환됩니다.
CHAPTER 05

PV / PVC / StorageClass ─ 3대 핵심 객체

이제 K8s 스토리지의 심장부로 들어갑니다. 이 세 가지 객체의 관계만 정확히 이해하면, K8s 스토리지의 70%는 끝났다고 봐도 됩니다. 핵심은 "누가 무엇을 책임지는가"입니다.

3대 객체의 역할 분담

객체 전체 이름 누가 만드는가? 무엇을 의미하는가? 비유
PV PersistentVolume 관리자 또는 자동 실제 물리적 스토리지를 K8s 객체로 표현 매물(부동산)
PVC PersistentVolumeClaim 개발자(앱 배포 시) "내가 필요한 스토리지 사양"을 선언 매물 신청서
SC StorageClass 관리자(1회) "이런 사양 요청이 오면 어떻게 만들지" 템플릿 부동산 중개사

두 가지 프로비저닝 모드

실전 예제: 동적 프로비저닝

① StorageClass (관리자가 1회 정의)

YAMLapiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: fast-ssd
provisioner: ebs.csi.aws.com      # 어떤 CSI 드라이버를 쓸지
parameters:
  type: gp3                       # 벤더별 옵션
  iops: "3000"
  throughput: "125"
reclaimPolicy: Delete             # PVC 삭제 시 PV도 삭제
volumeBindingMode: WaitForFirstConsumer  # Pod가 스케줄링된 노드 근처에 디스크 생성
allowVolumeExpansion: true           # 온라인 확장 허용

② PVC (개발자가 앱 배포 시 생성)

YAMLapiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: my-data
spec:
  storageClassName: fast-ssd     # 위에서 만든 SC 참조
  accessModes: [ReadWriteOnce]
  resources:
    requests:
      storage: 20Gi               # 20GB 요청

③ Pod (PVC만 참조하면 끝)

YAMLapiVersion: v1
kind: Pod
metadata:
  name: db
spec:
  containers:
  - name: postgres
    image: postgres:16
    volumeMounts:
    - { name: data, mountPath: /var/lib/postgresql/data }
  volumes:
  - name: data
    persistentVolumeClaim:
      claimName: my-data           # ⬅ PVC 이름만 적으면 끝!
✨ 동적 프로비저닝의 마법
위 세 YAML을 kubectl apply하면 자동으로:
  1. PVC가 생성되고 Pending 상태가 됨
  2. StorageClass가 트리거되어 AWS API로 EBS 볼륨 생성
  3. PV 객체가 자동 생성되고 PVC와 바인딩됨 (Bound)
  4. Pod가 시작될 때 노드에 마운트되어 컨테이너에 연결

volumeBindingMode 옵션의 중요성

모드 동작 장단점
Immediate PVC가 만들어지자마자 PV 생성/바인딩 빠르지만, AZ가 안 맞으면 Pod가 스케줄링 실패 가능
WaitForFirstConsumer Pod가 스케줄링될 때까지 대기 후 그 노드의 AZ에 디스크 생성 권장 멀티 AZ 환경에서 필수
CHAPTER 06

접근 모드(Access Mode)와 회수 정책(Reclaim Policy)

PVC가 PV를 요청할 때 가장 중요한 두 가지 속성. 이 두 가지를 잘못 설정하면 "왜 내 Pod가 다른 노드에서 안 뜨지?", "왜 데이터가 다 날아갔지?" 같은 악몽이 펼쳐집니다.

4가지 접근 모드

모드 약어 의미 전형적 예시
ReadWriteOnce RWO 단일 노드에서 읽기/쓰기 가능 PostgreSQL, MySQL, 단일 인스턴스 앱
ReadOnlyMany ROX 여러 노드에서 읽기만 가능 공통 설정, 정적 학습 데이터셋(읽기)
ReadWriteMany RWX 여러 노드에서 동시 읽기/쓰기 가능 분산 학습, 공유 워크스페이스, NFS
ReadWriteOncePod RWOP 단일 Pod에서만 읽기/쓰기 (1.22+) 엄격한 단일점 쓰기, leader election
⚠️ 자주 하는 실수
"ReadWriteOnce는 Pod 하나에서만 쓸 수 있다"고 오해하기 쉽지만, 정확히는 "하나의 노드"입니다. 같은 노드에 떠 있는 여러 Pod는 RWO 볼륨을 함께 마운트할 수 있습니다. 진짜 단일 Pod 제한이 필요하면 ReadWriteOncePod를 쓰세요.

스토리지 백엔드별 지원 매트릭스

백엔드 RWO ROX RWX 특징
AWS EBS / GCP PD / Azure Disk ✓ ✗ ✗ 블록 스토리지, 빠른 IOPS
NFS / EFS / Azure Files ✓ ✓ ✓ 파일 스토리지, 공유 가능
CephFS ✓ ✓ ✓ 분산 파일시스템, 자체 관리
Ceph RBD ✓ ✓ ✗ 블록, 높은 성능
Longhorn ✓ ✗ △ (NFS) K8s 네이티브, Rancher

회수 정책(Reclaim Policy)

PVC가 삭제되면 그것이 가리키던 PV(그리고 실제 디스크)는 어떻게 해야 할까요? 이걸 정하는 것이 reclaimPolicy입니다.

SAFE
Retain
PVC가 삭제되어도 PV와 실제 디스크는 보존. 관리자가 수동으로 정리해야 함. 프로덕션 DB에 권장.
CONVENIENT
Delete
PVC 삭제 시 PV와 실제 디스크까지 모두 삭제. 동적 프로비저닝 기본값. 실수로 데이터 날릴 위험.
DEPRECATED
Recycle
rm -rf로 데이터 삭제 후 재사용. 보안/안전성 이슈로 폐기됨. 사용 금지.
🚨 실전 경고
동적 프로비저닝의 기본값이 Delete라는 점은 사고의 1순위 원인입니다. 프로덕션 데이터베이스용 StorageClass는 반드시 reclaimPolicy: Retain으로 별도 만들어 사용하세요. PVC 삭제 한 줄로 회사 데이터가 사라지는 일을 막을 수 있습니다.
CHAPTER 07

CSI ─ Container Storage Interface

K8s가 AWS, GCP, Azure, Ceph, NetApp, 그리고 듣지도 보지도 못한 신규 스토리지까지 모두 지원할 수 있는 이유 ─ 그것이 바로 CSI입니다. K8s 코어와 스토리지 벤더를 분리한 표준 플러그인 인터페이스입니다.

CSI가 등장한 이유

2015~2018년경 K8s에는 코드 안에 직접(in-tree) AWS EBS, GCE PD, NFS, Ceph 등의 드라이버가 박혀 있었습니다. 새 스토리지를 지원하려면 K8s 코어를 수정해서 릴리스해야 했죠. 이건 지속 가능하지 않았습니다. 그래서:

CSI의 핵심 아이디어 (2017년 도입)

  • K8s는 스토리지 작업 표준 인터페이스(gRPC API)만 정의
  • 각 벤더는 그 인터페이스를 구현한 CSI Driver를 K8s와 별개로 배포
  • K8s 업그레이드와 스토리지 드라이버 업그레이드가 완전히 분리
  • 커뮤니티/벤더가 자유롭게 신규 드라이버 개발 가능

CSI 드라이버의 구성

CSI 드라이버는 보통 두 부분으로 나뉘어 K8s 클러스터에 배포됩니다:

MLOps/클라우드에서 자주 쓰는 CSI 드라이버

드라이버 제공자 특징 전형적 용도
ebs.csi.aws.com AWS EBS 볼륨, gp3/io2 EKS 기본, DB, RWO 워크로드
efs.csi.aws.com AWS NFS 기반, RWX 지원 분산 학습 데이터셋, 공유 모델 저장소
s3.csi.aws.com AWS (Mountpoint) S3를 POSIX처럼 마운트 대용량 학습 데이터셋 읽기
pd.csi.storage.gke.io Google Persistent Disk GKE 기본 블록 스토리지
disk.csi.azure.com Azure Managed Disk AKS 기본 블록 스토리지
driver.longhorn.io Rancher/SUSE K8s 네이티브 분산 블록 스토리지 온프레미스 클러스터, 스냅샷
rook-ceph.rbd.csi.ceph.com Ceph + Rook 분산 블록+파일+오브젝트 대규모 온프레미스, 자체 운영
nfs.csi.k8s.io K8s SIG-Storage 기존 NFS 서버 활용 온프레미스 RWX, 레거시 NAS 연동

CSI가 제공하는 추가 기능

  • Volume Snapshot ─ VolumeSnapshot, VolumeSnapshotClass로 백업/복원 표준화
  • Volume Cloning ─ 기존 PVC를 복제하여 새 PVC 생성 (dataSource 필드)
  • Volume Expansion ─ 온라인 확장 (PVC resources.requests.storage 증가)
  • Raw Block Volume ─ 파일시스템 없이 블록 디바이스 그대로 노출 (DB 등 고성능 용도)
  • Generic Ephemeral Volumes ─ Pod 라이프사이클을 따르는 동적 임시 볼륨
CHAPTER 08

StatefulSet과 Storage ─ 영속성의 완성

Deployment는 Pod에 무작위 이름을 붙이고 마음대로 재배치합니다. 그런데 데이터베이스는? 레플리카 셋에서 각 Pod는 자기만의 "정체성"과 "전용 볼륨"을 가져야 합니다. 그래서 등장한 것이 StatefulSet과 volumeClaimTemplates.

volumeClaimTemplates의 마법

YAMLapiVersion: apps/v1
kind: StatefulSet
metadata:
  name: postgres
spec:
  serviceName: postgres-headless
  replicas: 3
  selector:
    matchLabels: { app: postgres }
  template:
    metadata:
      labels: { app: postgres }
    spec:
      containers:
      - name: postgres
        image: postgres:16
        volumeMounts:
        - { name: data, mountPath: /var/lib/postgresql/data }
  volumeClaimTemplates:           # ⬅ 핵심! 각 레플리카마다 PVC를 자동 생성
  - metadata:
      name: data
    spec:
      accessModes: [ReadWriteOnce]
      storageClassName: fast-ssd
      resources:
        requests:
          storage: 100Gi

이 한 장의 YAML이 만드는 결과:

Pod 이름 자동 생성되는 PVC 네트워크 ID 특성
postgres-0 data-postgres-0 postgres-0.postgres-headless 고정 ID, 고정 볼륨
postgres-1 data-postgres-1 postgres-1.postgres-headless 고정 ID, 고정 볼륨
postgres-2 data-postgres-2 postgres-2.postgres-headless 고정 ID, 고정 볼륨
🔑 StatefulSet의 핵심 보장
  • Stable Identity: Pod 이름이 {name}-{ordinal} 형식으로 고정
  • Stable Storage: 같은 ordinal의 Pod는 항상 같은 PVC를 사용
  • Ordered Deployment: 0→1→2 순서로 시작, 역순으로 종료
  • Pod 삭제 ≠ PVC 삭제: 안전을 위해 PVC는 수동 삭제해야 함
⚠️ StatefulSet 삭제의 함정
kubectl delete sts postgres를 하면 Pod는 삭제되지만 PVC는 남습니다. 실수로 데이터가 날아가는 것을 막기 위한 안전장치. 진짜 정리하려면 PVC도 수동 삭제 필요. K8s 1.27+에서는 persistentVolumeClaimRetentionPolicy로 동작 제어 가능.

실전 패턴: 분산 LLM 학습의 스토리지 설계

DESIGN# 시나리오: Qwen2.5-7B를 8x GPU 노드에서 LoRA 학습

▷ 학습 데이터 (수백 GB, 모든 노드가 동시 읽기)
  → StorageClass: efs-csi  (또는 CephFS)
  → AccessMode:   ReadWriteMany
  → 마운트 위치:    /data/dataset

▷ 모델 체크포인트 (학습 중 빈번한 쓰기, 한 곳에서만)
  → StorageClass: fast-ssd  (EBS gp3 또는 Longhorn)
  → AccessMode:   ReadWriteOnce
  → 마운트 위치:    /checkpoints
  → reclaimPolicy: Retain   # 절대 자동 삭제 금지

▷ 임시 스크래치 (캐시, 셔플 버퍼)
  → 타입:         emptyDir (medium: ""=node disk)
  → 마운트 위치:    /scratch
  → sizeLimit:    200Gi

▷ Hugging Face 토큰, WandB API 키
  → 타입:         secret (또는 ExternalSecrets)
  → 마운트 위치:    /etc/credentials

▷ 학습 완료 후 모델 → MLflow 레지스트리(S3/MinIO)로 업로드
💡 이기종 클러스터에서의 주의점
ARM64(예: NVIDIA DGX Spark)와 x86_64 노드가 혼재된 환경에서는 CSI 드라이버 이미지도 multi-arch여야 합니다. docker buildx로 linux/amd64,linux/arm64 이미지를 빌드하고, NodeSelector로 GPU 노드에 적절히 배치하세요. Longhorn, Rook-Ceph 모두 ARM64를 공식 지원합니다.

운영 체크리스트

  • ✅ 프로덕션 StorageClass는 reclaimPolicy: Retain을 별도 만들어 사용
  • ✅ VolumeSnapshot 기반의 정기 백업 자동화 (Velero, Kasten K10)
  • ✅ allowVolumeExpansion: true 설정 후 PVC 확장 절차 문서화
  • ✅ ResourceQuota로 네임스페이스별 PVC 총량/개수 제한
  • ✅ Prometheus + kube-state-metrics로 PV/PVC 상태 모니터링
  • ✅ etcd에 저장되는 Secret은 encryption-at-rest 활성화
  • ✅ 데이터 분실 시나리오 정기 훈련 (PVC 실수 삭제, 노드 장애 등)
