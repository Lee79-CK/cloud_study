Chapter 01

네트워크가 전부다

VM이 아무리 빨라도, AKS 클러스터가 아무리 커도 — 네트워크 설계가 잘못되면 보안 사고, 통신 단절, IP 고갈로 시스템 전체가 멈춥니다. 왜 네트워크부터 배워야 하는지 먼저 짚고 갑니다.

🏙️
Analogy — 신도시 건설

클라우드 네트워크 설계는 신도시를 짓는 일과 같습니다. 건물(VM, AKS)을 올리기 전에 도시 경계(VNet), 구역 분할(Subnet), 도로(라우팅), 검문소(NSG), 톨게이트(Firewall/Gateway)를 먼저 계획해야 합니다. 도로 없이 건물부터 지으면, 나중에 도로를 내기 위해 건물을 부숴야 하는 일이 생깁니다. 클라우드에서도 네트워크는 나중에 바꾸기 가장 어려운 레이어입니다.

왜 "나중에 바꾸기 어려운가"?

  • IP 주소 공간(CIDR)은 사실상 불변 — VNet의 주소 대역은 리소스가 올라간 뒤 축소·변경이 매우 어렵습니다. 처음에 좁게 잡으면 IP 고갈, 온프레미스와 겹치게 잡으면 하이브리드 연결 자체가 불가능해집니다.
  • Public/Private 경계는 보안의 뿌리 — "어디까지 인터넷에 노출할 것인가"라는 결정은 모든 보안 정책의 출발점입니다. 한 번 Public IP로 운영을 시작한 시스템을 Private으로 전환하는 것은 대규모 마이그레이션입니다.
  • 모든 장애의 절반은 네트워크에서 시작 — "연결이 안 돼요"의 원인은 대부분 NSG 규칙, 라우팅 테이블, DNS 설정에 있습니다. 구조를 알면 장애 분석 시간이 1/10로 줄어듭니다.
NOTE

이 강의는 온프레미스 네트워크 경험이 거의 없는 분도 따라올 수 있도록 모든 개념을 비유와 그림으로 먼저 설명한 뒤 Azure 용어를 붙입니다. 이미 AWS를 아신다면 괄호 안의 AWS 대응 개념을 참고하세요.

Chapter 02

핵심 빌딩 블록 — 도시의 부품들

Azure 네트워크는 10여 개의 부품으로 조립됩니다. 각 부품을 도시 비유와 함께 하나씩 살펴봅니다. 이 장의 카드들은 이후 모든 챕터에서 계속 등장하므로, 여기서 확실히 잡고 가세요.

1단계 — 땅과 구역

🗺️
Virtual Network (VNet) 기본 단위
Azure 안에 만드는 나만의 격리된 사설 네트워크 공간. 도시의 행정 경계선입니다. VNet끼리는 기본적으로 통신 불가 — 명시적으로 연결(Peering)해야 합니다. AWS의 VPC에 해당.
예: 10.100.0.0/16 (65,536개 IP)
🏘️
Subnet 구역 분할
VNet 내부를 용도별로 나눈 구역. 주거지구/상업지구/공업지구처럼 웹 서브넷, 앱 서브넷, DB 서브넷으로 나눕니다. 보안 규칙(NSG)과 라우팅은 서브넷 단위로 붙이는 것이 기본입니다.
예: 10.100.1.0/24 (Azure가 5개 IP 예약 → 251개 사용 가능)

2단계 — 검문소와 도로

🛂
NSG (Network Security Group)
서브넷/NIC 입구의 검문소. "어디서 온, 어떤 포트의 트래픽을 허용/거부할지"를 우선순위 번호가 있는 규칙 목록으로 정의합니다. 5장에서 시뮬레이터로 직접 체험합니다. (AWS: Security Group + NACL)
🛣️
Route Table (UDR)
트래픽이 어느 길로 갈지 정하는 도로 표지판. Azure는 기본 경로를 자동 제공하지만, "모든 외부행 트래픽은 방화벽을 경유하라" 같은 사용자 정의 경로(User Defined Route)를 덮어쓸 수 있습니다.
예: 0.0.0.0/0 → Azure Firewall(10.100.2.4)
🔥
Azure Firewall
도시 전체의 중앙 통합 검문소. NSG가 구역별 간이 검문이라면, Firewall은 L3~L7 전체를 검사하는 정식 세관입니다. FQDN 필터링("*.ubuntu.com만 허용"), 위협 인텔리전스 기반 차단을 지원합니다.
⚖️
Load Balancer / Application Gateway
트래픽을 여러 서버로 나누는 교통 경찰. L4(TCP/UDP)는 Load Balancer, L7(HTTP, URL 기반 라우팅, WAF)은 Application Gateway. 둘 다 Public/Internal 두 가지 모드로 만들 수 있다는 점이 핵심입니다.

3단계 — 외부와의 연결

📮
Public IP PUBLIC
인터넷에서 직접 접근 가능한 공인 주소. 이 강의의 앰버색은 전부 이 영역입니다. Public IP가 붙는 순간 그 리소스는 전 세계에서 보이는 간판을 단 셈입니다.
🚪
NAT Gateway 아웃바운드 전용
내부 리소스가 인터넷으로 나갈 때만 쓰는 일방통행 출구. 밖에서 안으로는 절대 못 들어옵니다. "패키지 다운로드는 해야 하지만 외부 노출은 싫다"의 표준 답안.
🔗
VNet Peering
VNet과 VNet을 잇는 전용 다리. Microsoft 백본망을 타기 때문에 인터넷을 거치지 않고 빠르고 안전합니다. Hub-Spoke 아키텍처(7장)의 뼈대입니다.
🚇
VPN Gateway / ExpressRoute
온프레미스 사내망과 잇는 통로. VPN은 인터넷 위 암호화 터널(저렴, ~1Gbps대), ExpressRoute는 통신사 전용 회선(비싸지만 안정적, 최대 100Gbps)입니다.
🔒
Private Endpoint PRIVATE
Azure SQL, Storage 같은 PaaS 서비스에 내 VNet 안의 사설 IP를 부여하는 기술. PaaS를 인터넷이 아닌 내부망으로만 쓰게 만드는, 현대 보안 설계의 핵심 부품입니다. 10장에서 심화.
📖
Azure Private DNS
VNet 내부 전용 전화번호부. Private Endpoint를 쓰면 "mydb.database.windows.net"이 공인 IP가 아닌 사설 IP(10.x.x.x)로 풀리도록 이 DNS 존이 함께 동작해야 합니다.
TIP

암기 요령: VNet=도시, Subnet=구역, NSG=검문소, UDR=표지판, Firewall=세관, LB=교통경찰, Peering=다리, VPN/ER=고속도로, Private Endpoint=관공서 직통 내선전화. 이 비유 체계 하나면 면접 설명도 가능합니다.

Chapter 03

Public 망 vs Private 망 — 이 강의의 심장

클라우드 네트워크 설계의 모든 의사결정은 결국 하나의 질문으로 수렴합니다: "이 리소스를 인터넷에 노출할 것인가, 숨길 것인가?" 두 세계의 차이를 정확히 이해해 봅시다.

🏪
Analogy — 가게의 앞문과 주방

Public 망은 손님이 드나드는 가게 앞문입니다. 누구나 올 수 있어 매출(서비스)이 생기지만, 도둑도 올 수 있으니 CCTV·경비(WAF, DDoS 방어)가 필수입니다. Private 망은 주방과 금고입니다. 직원 전용 통로로만 출입하며, 손님이 주방에 직접 들어올 일은 절대 없어야 합니다. 좋은 설계란 앞문은 단 하나로 좁히고, 나머지는 전부 주방으로 만드는 것입니다.

정의부터 정확히

  • Public 망(공인망) — 리소스가 Public IP를 가지며 인터넷에서 직접 도달 가능한 네트워크 영역. 진입점(웹사이트, API 게이트웨이)만 여기에 둡니다.
  • Private 망(사설망) — RFC 1918 사설 IP(10.x, 172.16~31.x, 192.168.x)만 사용하며, 인터넷에서 직접 도달이 불가능한 영역. VNet 내부는 기본적으로 전부 Private입니다.
NOTE

중요한 직관: Azure에서 VNet을 만들면 기본 상태는 Private입니다. Public이 되는 건 여러분이 의도적으로 Public IP를 붙이거나 Public Load Balancer를 만들 때뿐입니다. 즉 "Private 설계"란 무언가를 추가하는 게 아니라 Public을 최소화하는 절제의 설계입니다.

흔한 오해 바로잡기

❌ "Private이면 인터넷을 아예 못 쓴다?"
아닙니다. Private 리소스도 NAT Gateway나 Firewall을 통해 나가는 것(아웃바운드)은 가능합니다. 막히는 건 인터넷에서 들어오는(인바운드) 직접 접근입니다. 일방통행 회전문을 떠올리세요.
❌ "NSG로 막으면 Public IP 있어도 안전하다?"
절반만 맞습니다. NSG는 강력하지만 규칙 실수 한 줄이면 바로 노출됩니다. Public IP가 없으면 실수해도 도달 자체가 불가능 — 구조적 안전(secure by design)이 규칙 기반 안전보다 우선입니다.
❌ "사내 서비스니까 대충 Public으로 열고 IP 제한하면 된다?"
소스 IP 제한은 우회(IP 스푸핑은 어렵지만 VPN 종단 변경, 협력사 망 경유 등)와 관리 누락에 취약합니다. 사내 전용이라면 VPN/ExpressRoute + Private 망이 정석입니다.
❌ "Public = 나쁨, Private = 좋음?"
아닙니다. 외부 고객 서비스는 Public 진입점이 반드시 필요합니다. 핵심은 선악이 아니라 "Public 노출면을 의도한 한 지점으로 좁히는 통제"입니다.

설계 원칙 — "깔때기 모델"

실무 표준은 아래처럼 요약됩니다. 이 그림을 머리에 새기면 어떤 아키텍처 다이어그램을 봐도 구조가 읽힙니다.

The Funnel — Public 노출은 한 점으로, 내부는 전부 Private으로
  1. 진입점 최소화 — 인터넷에서 들어오는 문은 App Gateway(WAF) 하나로 좁힙니다.
  2. 기본값은 Private — 모든 신규 리소스는 Public IP 없이 만들고, 노출이 필요할 때만 예외 승인합니다.
  3. 출구도 단일화 — 나가는 트래픽은 NAT GW 또는 Firewall 한 곳으로 모아 로그를 남깁니다.
  4. PaaS도 Private으로 — DB·Storage 같은 PaaS는 Private Endpoint로 내부망에 편입합니다.

시뮬레이터가 말해주는 것

PUBLIC 직접 노출 에서 본 것
VM마다 Public IP가 붙어 있어 공격자가 DB 포트까지 직접 노크할 수 있었습니다. NSG가 막아주더라도 "노크 자체가 가능한 구조"는 규칙 한 줄 실수가 곧 침해 사고가 되는 구조입니다.
PRIVATE 설계 에서 본 것

WARN

실무에서 가장 많이 보는 사고 패턴: "테스트하려고 잠깐" VM에 Public IP를 붙이고 NSG에 Any → 22(SSH) 허용을 넣은 뒤 잊는 경우입니다. 봇은 새 Public IP를 수 분 안에 스캔하기 시작합니다. SSH 접속은 반드시 Azure Bastion 또는 VPN을 통하세요.

NSG — 클라우드의 검문소를 운영해보기

NSG는 우선순위(priority) 숫자가 작은 규칙부터 위에서 아래로 평가되며, 처음 일치한 규칙에서 평가가 끝납니다(first match wins). 이 두 문장이 NSG의 전부입니다. 직접 패킷을 만들어 검문소에 통과시켜 보세요.

Analogy — 공항 입국 심사 매뉴얼

NSG 설계 모범 사례

  • 우선순위는 100 단위로 띄워서 — 100, 200, 300… 으로 잡아야 나중에 사이에 규칙을 끼워 넣을 수 있습니다.
  • Source에 Any 대신 구체적 대역 — "WAS 서브넷 → DB 1433"처럼 출발지를 좁힐수록 안전합니다.
  • Service Tag 활용 — Internet, VirtualNetwork, AzureLoadBalancer 같은 태그로 IP를 일일이 나열하지 않고 의미 단위로 제어합니다.
  • ASG(Application Security Group) — VM들을 "web-servers", "db-servers" 같은 논리 그룹으로 묶어 IP가 아닌 역할 기준으로 규칙을 작성합니다.
  • NSG Flow Logs 활성화 — 허용/거부된 모든 흐름을 Storage에 기록해 두면 장애·침해 분석의 1차 자료가 됩니다.

IP 주소 설계 — CIDR, 가장 되돌리기 어려운 결정

VNet의 주소 대역(CIDR)은 한 번 정하면 사실상 바꿀 수 없는 도시의 경계선입니다. 좁으면 IP 고갈, 다른 망과 겹치면 연결 불가. 슬라이더로 직접 크기를 가늠해 보세요.

CIDR 30초 요약

10.100.0.0/16에서 /16은 "앞 16비트가 네트워크 식별자로 고정"이라는 뜻입니다. 숫자가 작을수록 고정 부분이 짧아 더 큰 네트워크(/8 = 1,677만 개), 클수록 작은 네트워크(/29 = 8개)입니다. Azure는 각 서브넷에서 5개 IP를 시스템용으로 예약하므로 /24 서브넷의 실제 가용 IP는 256 − 5 = 251개입니다.

실무 주소 설계 수칙

  • 온프레미스와 절대 겹치지 않게 — 회사 사내망이 10.0.0.0/8의 일부를 쓴다면, Azure는 그와 겹치지 않는 대역(예: 10.100.0.0/14)을 전사 IP 관리대장(IPAM)에서 할당받아 시작합니다.
  • VNet은 넉넉히, 서브넷은 용도별로 — VNet /16, 서브넷 /24를 기본 감각으로. AKS처럼 IP를 많이 먹는 워크로드는 별도 대형 대역을 배정합니다(9장).
  • 특수 서브넷의 고정 이름과 최소 크기 — GatewaySubnet(/27 이상), AzureFirewallSubnet(/26), AzureBastionSubnet(/26)은 이름과 크기가 정해져 있습니다.
  • 성장 여유 2배 법칙 — 현재 필요량의 최소 2배를 잡습니다. IP는 모자라면 재앙, 남으면 그냥 여유입니다.
WARN

여러 팀이 각자 VNet을 만들 때 모두가 기본값 10.0.0.0/16을 쓰는 사고가 정말 흔합니다. 대역이 겹친 VNet끼리는 Peering이 불가능합니다. 조직 차원의 IP 할당 표를 반드시 먼저 만드세요.

Chapter 07

Hub-Spoke — 사실상의 표준 아키텍처

워크로드가 2~3개만 넘어가도 VNet을 어떻게 배치할지가 문제가 됩니다. Azure의 답은 명확합니다: 공용 인프라는 Hub에 한 번만, 워크로드는 Spoke에 격리. 항공사의 허브 공항 모델과 같습니다.

✈️
Analogy — 허브 공항

인천공항(Hub)에 출입국 심사·세관·보안검색을 집중시키고, 김포·제주·부산(Spoke)은 인천을 경유해 해외와 연결됩니다. 모든 도시에 국제공항을 짓는 것보다 훨씬 싸고, 보안 통제가 한 곳에 모입니다. Hub VNet에는 Firewall·VPN Gateway·Bastion·Private DNS 같은 공용 인프라만, Spoke VNet에는 워크로드만 둡니다.

구조와 트래픽 규칙

Hub-Spoke Topology — Spoke 간 통신도 Hub Firewall 경유
  • Spoke ↔ Spoke 직접 Peering은 하지 않습니다 — Spoke의 UDR에 "다른 Spoke 대역 → Firewall" 경로를 넣어 모든 동·서 트래픽이 Hub Firewall에서 검사·로깅되게 합니다.
  • 환경 분리 — 운영/개발/DMZ를 Spoke 단위로 격리하면 폭발 반경(blast radius)이 Spoke 하나로 제한됩니다.
  • 비용 효율 — Firewall(월 수십만 원대)과 VPN Gateway를 Spoke마다 두면 Spoke 수만큼 비용이 곱해집니다. Hub에 한 번만.
  • 확장 — 새 워크로드 = 새 Spoke VNet + Peering 2줄. 기존 구조를 건드리지 않습니다. Spoke가 수십 개로 늘면 관리형 서비스인 Azure Virtual WAN으로 전환을 검토합니다.
# Hub ↔ Spoke 양방향 Peering (반드시 양쪽 모두 생성해야 활성화)
az network vnet peering create --name hub-to-spokeB \
  -g rg-net --vnet-name vnet-hub --remote-vnet vnet-spoke-vm \
  --allow-vnet-access --allow-gateway-transit

az network vnet peering create --name spokeB-to-hub \
  -g rg-net --vnet-name vnet-spoke-vm --remote-vnet vnet-hub \
  --allow-vnet-access --use-remote-gateways

# Spoke의 모든 외부행 트래픽을 Hub Firewall(10.100.2.4)로 강제
az network route-table create -g rg-net -n rt-spokeB
az network route-table route create -g rg-net --route-table-name rt-spokeB \
  -n default-via-fw --address-prefix 0.0.0.0/0 \
  --next-hop-type VirtualAppliance --next-hop-ip-address 10.100.2.4
resource "azurerm_virtual_network_peering" "hub_to_spoke" {
  name                      = "hub-to-spokeB"
  resource_group_name       = azurerm_resource_group.net.name
  virtual_network_name      = azurerm_virtual_network.hub.name
  remote_virtual_network_id = azurerm_virtual_network.spoke_vm.id
  allow_gateway_transit     = true
}

resource "azurerm_route_table" "spoke_b" {
  name                = "rt-spokeB"
  location            = azurerm_resource_group.net.location
  resource_group_name = azurerm_resource_group.net.name

  route {
    name                   = "default-via-fw"
    address_prefix         = "0.0.0.0/0"
    next_hop_type          = "VirtualAppliance"
    next_hop_in_ip_address = "10.100.2.4"  # Azure Firewall 사설 IP
  }
}

  • VM 접속은 Azure Bastion으로 — 어떤 VM에도 Public IP·SSH 직접 노출 없이, 포털/CLI를 통해 브라우저 기반 SSH/RDP를 제공합니다. Hub에 1개만 두면 Peering된 모든 Spoke의 VM에 접속 가능합니다.

AKS 네트워크 설계 — 컨테이너 시대의 도시 계획

AKS는 VM과 차원이 다른 고민을 던집니다: Pod 수백 개가 각자 IP를 요구하기 때문입니다. 네트워크 플러그인 선택, Private Cluster 여부, 들어오고 나가는 길의 설계 — 세 가지 결정을 차례로 해봅니다.

결정 1 — 네트워크 플러그인: Pod에게 어떤 주소를 줄 것인가

🏢
Analogy — 오피스텔 호실 번호

kubenet(레거시)은 건물(노드)에만 도로명 주소를 주고, 입주자(Pod)는 건물 내부 호실 번호만 갖는 방식 — 외부에서 입주자에게 직접 우편을 보내려면 NAT 변환이 필요합니다. Azure CNI는 입주자 전원에게 정식 도로명 주소(VNet IP)를 발급합니다. 누구나 직접 찾아올 수 있지만 주소(IP)가 대량으로 소모됩니다. Azure CNI Overlay는 절충안 — 입주자에게는 사설 우편번호 체계(별도 Overlay 대역)를 주고, 건물 차원에서 변환해 줍니다. VNet IP를 아끼면서 CNI의 성능을 얻는 현재의 권장 기본값입니다.

결정 3 — 들어오는 길(Ingress)과 나가는 길(Egress)

  • Ingress — Service를 type: LoadBalancer로 만들면 기본은 Public LB가 생깁니다. 운영 환경에서는 어노테이션 한 줄(azure-load-balancer-internal: "true")로 Internal LB를 만들고, 그 앞에 App Gateway(WAF)를 세우는 것이 표준입니다(AGIC 또는 App GW for Containers).
  • Egress — 기본값은 AKS가 만든 Public LB로 나가는 것. 통제된 설계에서는 outboundType: userDefinedRouting으로 바꿔 모든 노드의 외부행 트래픽을 Hub의 Azure Firewall로 보냅니다. 이때 AKS가 필요로 하는 FQDN(mcr.microsoft.com, *.azmk8s.io 등)을 Firewall에서 허용해야 클러스터가 정상 동작합니다.
  • 이미지 저장소도 Private으로 — ACR(Azure Container Registry)에 Private Endpoint를 붙여, 이미지 Pull 트래픽조차 인터넷을 거치지 않게 합니다.
# Private + CNI Overlay + Egress는 Firewall 경유(UDR)인 운영형 클러스터
az aks create -g rg-prod -n aks-prod \
  --enable-private-cluster \
  --network-plugin azure --network-plugin-mode overlay \
  --pod-cidr 192.168.0.0/16 \
  --vnet-subnet-id $SUBNET_ID \
  --outbound-type userDefinedRouting \
  --zones 1 2 3 \
  --enable-managed-identity

# Private Cluster에는 인터넷에서 kubectl이 안 되므로,
# VNet 내 점프 호스트가 없을 때는 command invoke로 원격 실행
az aks command invoke -g rg-prod -n aks-prod \
  --command "kubectl get nodes -o wide"

AKS의 CIDR는 3종 세트입니다 — ① 노드 서브넷(VNet 내), ② Pod CIDR(Overlay 대역, 예: 192.168.0.0/16), ③ Service CIDR(클러스터 가상 대역, 예: 10.0.0.0/16). 이 셋과 VNet·온프레미스 대역이 서로 겹치지 않아야 합니다. 겹침은 미묘한 통신 장애로 나타나 디버깅이 매우 어렵습니다.

Chapter 10

Private Endpoint 심화 — PaaS를 내 편으로 만들기

Azure SQL, Storage, Key Vault 같은 PaaS는 태생적으로 공인 엔드포인트를 갖고 태어납니다. 이들을 내부망으로 끌어들이는 두 가지 기술 — Service Endpoint와 Private Endpoint — 의 차이를 정확히 구분하는 것이 중급 엔지니어의 분기점입니다.

☎️
Analogy — 관공서에 연락하는 두 가지 방법

Service Endpoint는 관공서 대표번호(공인 주소)로 전화하되, 발신번호가 우리 회사면 VIP 전용선으로 연결해 주는 것 — 목적지는 여전히 공인 주소이고, "우리 회사에서 온 전화만 받아라"는 방화벽 설정을 관공서 쪽에 해야 합니다. Private Endpoint는 아예 관공서 직원이 우리 사무실 안에 내선 전화기(사설 IP)를 설치해 주는 것 — 공인 주소를 완전히 끊어버릴 수 있고, VPN으로 연결된 온프레미스에서도 그 내선으로 연락할 수 있습니다.

항목Service EndpointPrivate Endpoint (권장)
PaaS가 갖는 주소여전히 공인 IP (경로만 최적화)내 VNet 안의 사설 IP (예: 10.102.3.8)
공인 접근 차단PaaS 방화벽에서 "이 서브넷만 허용" 설정 필요공인 엔드포인트 자체를 완전 차단 가능
온프레미스에서 접근불가 (VNet 출발 트래픽만 인식)가능 — VPN/ER 너머에서도 사설 IP로 도달
DNS 작업불필요Private DNS Zone 연동 필수
비용무료엔드포인트당 시간 요금 + 데이터 처리 요금

Private Endpoint의 동작 — DNS가 절반이다

Private Endpoint에서 가장 많이 실수하는 부분은 네트워크가 아니라 DNS입니다. 애플리케이션은 여전히 mydb.database.windows.net이라는 공인 도메인 이름으로 접속합니다. 마법은 이름풀이 단계에서 일어납니다:

Private Endpoint + Private DNS Zone — 같은 이름, 다른 목적지
  1. VNet 안의 앱이 mydb.database.windows.net의 IP를 질의합니다.
  2. VNet에 연결된 Private DNS Zone(privatelink.database.windows.net)이 가로채 사설 IP 10.102.3.8을 응답합니다.
  3. 트래픽은 VNet을 떠나지 않고 Private Endpoint NIC를 거쳐 PaaS에 도달합니다.
  4. 마지막으로 PaaS의 공인 접근을 Disabled로 잠그면, 인터넷 쪽 문은 완전히 사라집니다.
WARN

전형적 장애: "Private Endpoint를 만들었는데 온프레미스에서는 여전히 공인 IP로 풀려요." 온프레미스 DNS는 Azure Private DNS Zone을 모르기 때문입니다. 해결책은 Azure DNS Private Resolver (또는 VNet 내 DNS 포워더 VM)를 두고, 사내 DNS가 *.database.windows.net 질의를 그쪽으로 전달(conditional forwarding)하게 하는 것입니다.

설계 체크리스트 & 안티패턴

지금까지의 내용을 실무에서 바로 쓰는 점검표로 압축했습니다. 새 프로젝트의 네트워크 설계 리뷰에서 이 목록을 그대로 사용해도 됩니다.

✅ 설계 체크리스트

1. 주소 계획
온프레미스·타 VNet과 겹치지 않는 CIDR인가? 조직 IPAM 대장에 등록했는가? 성장분 2배 여유가 있는가? AKS의 3종 CIDR(노드/Pod/Service)는 상호 비겹침인가?
2. Public 노출면
인터넷 인바운드 진입점이 App GW(WAF) 단 하나로 좁혀져 있는가? Public IP가 붙은 리소스의 전수 목록이 있고, 각각 노출 사유가 승인됐는가?
3. 토폴로지
Hub-Spoke 구조인가? Firewall·Gateway·Bastion·DNS가 Hub에 집중돼 있는가? Spoke 간 트래픽이 Firewall을 경유하도록 UDR이 걸려 있는가?
4. 세분화
계층(Web/WAS/DB)별 서브넷이 분리됐는가? NSG가 모든 서브넷에 부착됐고 Source가 Any가 아닌가? AKS 내부는 NetworkPolicy로 Pod 간 통제를 하는가?
5. PaaS 연결
DB·Storage·Key Vault·ACR에 Private Endpoint가 있는가? 각 서비스의 Public access가 Disabled인가? Private DNS Zone이 모든 관련 VNet에 링크됐는가?
6. 아웃바운드
나가는 트래픽이 NAT GW 또는 Firewall로 단일화됐는가? AKS는 outboundType=UDR인가? Firewall에서 목적지 FQDN 허용 목록을 관리하는가?
7. 관리 접근
VM SSH/RDP는 Bastion 전용인가? AKS는 Private Cluster이고 kubectl 경로(점프 호스트/VPN/command invoke)가 문서화됐는가?
8. 관측성
NSG Flow Logs(또는 VNet Flow Logs)·Firewall 로그가 수집되는가? "어제 이 서버에 누가 접속 시도했나"에 5분 안에 답할 수 있는가?
