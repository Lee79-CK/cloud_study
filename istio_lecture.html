CHAPTER 01

왜 Istio가 필요한가?

모놀리식에서 마이크로서비스로 넘어가면 "분산 시스템의 문제"가 폭발적으로 늘어납니다. Istio가 풀어주는 문제부터 명확히 짚고 갑시다.

마이크로서비스가 만들어내는 5가지 골칫거리

서비스를 잘게 쪼개는 순간, "프로세스 안의 함수 호출"이 "네트워크를 건너는 RPC 호출"로 바뀝니다. 그러면 다음과 같은 문제들이 동시다발적으로 발생합니다.

🚦
트래픽 제어
카나리 배포, A/B 테스트, 가중치 기반 라우팅 — K8S Service는 단순한 라운드 로빈만 지원합니다.
🔐
서비스 간 보안
클러스터 내부 통신은 기본적으로 평문(HTTP)입니다. 서비스마다 TLS를 직접 구현해야 할까요?
📊
관측성 부족
"A → B → C 호출 중 어디서 느려졌지?" 분산 추적, 메트릭, 로그를 일관되게 모으기 어렵습니다.
⚡
복원력(Resilience)
재시도, 타임아웃, 서킷 브레이커, 폴백 — 모든 언어/프레임워크마다 구현이 다릅니다.
🛡️
접근 제어
"주문 서비스만 결제 서비스를 호출하게 하고 싶다" — L7 단위 정책이 필요합니다.
💻
언어 종속성
Java, Go, Python, Node.js… 라이브러리(예: Netflix OSS)로 풀면 언어마다 다시 짜야 합니다.

비유: 회사가 작을 때는 모두가 한 사무실에서 일하니 그냥 옆자리에 말하면 됩니다(모놀리식).

회사가 커져서 여러 빌딩으로 흩어지면 → 전화선 깔고, 경비 시스템 만들고, CCTV 설치하고, 출입증 발급해야 합니다(마이크로서비스).

이걸 각 빌딩 직원이 직접 다 하면 너무 비효율적이죠. 이걸 "건물 인프라"가 자동으로 해결해주는 게 Istio입니다.

해결책의 진화

세대접근 방식장점단점
1세대 각 서비스에 라이브러리 통합 (Netflix Hystrix, Ribbon) 유연함 언어별 구현, 코드 수정 필수, 버전 동기화 지옥
2세대 공통 게이트웨이/프록시 단일 지점 중앙 관리 SPOF, 내부 통신 제어 불가
3세대 Service Mesh (사이드카 패턴) 언어 무관, 코드 무관, 분산 + 중앙 통제 학습 곡선, 리소스 오버헤드
CHAPTER 02

Service Mesh란 무엇인가

Istio 자체보다 먼저, "서비스 메시"라는 개념적 패턴을 이해해야 합니다.

한 줄 정의

📘
Service Mesh

마이크로서비스 간의 모든 통신을 처리하는 전용 인프라 레이어. 애플리케이션 코드와 분리되어 네트워킹, 보안, 관측성을 일관되게 제공합니다.

핵심 아이디어: 사이드카(Sidecar) 패턴

모든 마이크로서비스 Pod 옆에 작은 프록시 컨테이너를 하나 더 붙입니다. 애플리케이션은 http://payment-service를 호출하지만, 실제로는 옆에 붙은 프록시(Envoy)가 가로채서 처리합니다.

Pod: order-service
📦 order-app app container
🔁 istio-proxy sidecar (envoy)
Pod: payment-service
💳 payment-app app container
🔁 istio-proxy sidecar (envoy)

order-app이 payment-service를 호출하면 → 같은 Pod의 istio-proxy가 가로채서 → payment-service Pod의 istio-proxy로 전달 → 그 안의 payment-app이 받습니다.

비유: 개인 비서

각 임원(애플리케이션)에게 비서(사이드카 프록시)가 한 명씩 붙어있다고 상상하세요.

임원이 "B 임원에게 이 서류 보내줘"라고 하면 비서가 알아서: ① 서류 봉투에 도장(암호화), ② B 임원이 받을 자격이 있는지 확인(권한), ③ 보낸 시간/내용 기록(로깅), ④ 못 받으면 다시 시도(재시도)를 합니다.

임원은 비즈니스에만 집중하고, 비서들이 모든 통신 인프라를 처리합니다. 그리고 모든 비서를 한 명의 비서실장(istiod)이 통제합니다.

Service Mesh가 제공하는 3대 기능

🚦
Traffic Management
라우팅, 로드밸런싱, 카나리, 미러링, 서킷 브레이커, 재시도, 타임아웃 등 L4/L7 트래픽 제어
🔒
Security
자동 mTLS, 인증(서비스 ID 기반), 인가(L7 정책), 인증서 자동 발급/갱신
👁️
Observability
모든 요청에 대한 메트릭(RED), 분산 추적, 접근 로그 자동 수집 (Prometheus, Jaeger, Kiali 연동)
CHAPTER 03

Istio 아키텍처: 2개의 평면

Istio는 Control Plane(통제부)과 Data Plane(데이터부)의 2층 구조입니다. 한 번 이해하면 나머지는 다 응용입니다.

CONTROL PLANE — 두뇌 (1개)
istiod
Pilot + Citadel + Galley 통합
Pilot
설정 → Envoy 배포
Citadel
인증서 발급/관리
Galley
설정 검증/배포
⬇️ xDS API로 동적 설정 전송 ⬇️
DATA PLANE — 실제 트래픽 처리 (Pod마다)
Envoy (Sidecar)
앱 Pod 옆에 자동 주입
Envoy (Gateway)
클러스터 입출구
iptables 룰
트래픽 자동 가로채기

Control Plane vs Data Plane 정확히 이해하기

구분Control Plane (istiod)Data Plane (Envoy)
역할 "무엇을 어떻게 할지" 결정 (정책 수립) 실제 트래픽을 받고 처리 (정책 실행)
위치 istio-system 네임스페이스에 Deployment 1개 모든 워크로드 Pod에 사이드카로 주입
처리량 실제 트래픽은 안 거침 모든 요청이 통과
구성 요소 istiod (Pilot+Citadel+Galley 통합) Envoy proxy
통신 xDS gRPC 스트림으로 Envoy에 설정 push 설정 받아서 트래픽 라우팅/필터링

트래픽이 흐르는 전체 과정

  1. ① 호출: service-A 앱이 http://service-b로 요청을 보냅니다. 일반 HTTP 호출처럼 보입니다.
  2. iptables 가로채기: Pod에 적용된 iptables 룰이 outbound 트래픽을 강제로 같은 Pod의 Envoy(:15001)로 보냅니다.
  3. ② Envoy A: istiod에서 받은 라우팅 규칙을 보고, 어디로 보낼지 결정합니다. mTLS 핸드셰이크로 암호화하여 service-B Pod의 Envoy로 전송합니다.
  4. ③ 부가 처리: 도중에 재시도, 타임아웃, 메트릭 기록, 추적 헤더 주입 등이 모두 자동 처리됩니다.
  5. ④ Envoy B → app: 도착한 Envoy가 mTLS를 풀고, 권한 정책을 확인 후 같은 Pod의 앱 컨테이너에 평문으로 전달합니다.
  6. ⑤ 텔레메트리: 모든 양쪽 Envoy가 호출 정보를 Prometheus, Jaeger 등에 자동 송신합니다.
💡
핵심 통찰

애플리케이션은 평범한 HTTP를 호출했을 뿐인데, 인프라(Envoy + istiod)가 자동으로 암호화, 라우팅, 재시도, 메트릭, 추적을 다 처리했습니다. 코드 한 줄도 안 고치고요.

CHAPTER 04

핵심 컴포넌트 상세

Istio 1.5 이후로는 Control Plane이 istiod 하나로 통합되었습니다. 옛 책에 나오는 Pilot/Citadel/Galley도 알아두면 좋습니다.

istiod — 통합된 Control Plane

Istio 1.5 이후의 모든 Control Plane 기능이 통합된 단일 바이너리/Pod입니다. 기존의 Pilot, Citadel, Galley, sidecar injector 역할을 모두 수행합니다.

  • 서비스 디스커버리: K8S API에서 서비스/엔드포인트 정보 수집
  • 설정 변환: Istio CRD(VirtualService 등) → Envoy 설정으로 변환
  • xDS 서버: 모든 Envoy 사이드카에 gRPC 스트림으로 동적 설정 push
  • 인증서 발급: 각 워크로드에 SPIFFE ID 기반 인증서 자동 발급 (Citadel 역할)
  • 사이드카 주입: Mutating Webhook으로 Pod에 Envoy 컨테이너 자동 추가
📦
실제로 어디서 도나요?

istio-system 네임스페이스에 Deployment/istiod로 배포됩니다. HA 구성 시 보통 2~3개의 replica를 운영합니다.

Envoy Proxy — 데이터 평면의 일꾼

Lyft에서 만든 고성능 L7 프록시. Istio는 Envoy를 사이드카로 활용해 모든 트래픽을 가로챕니다.

  • L7 인지: HTTP/1.1, HTTP/2, HTTP/3, gRPC, WebSocket, TCP 모두 처리
  • 동적 설정: 재시작 없이 xDS API로 라우팅 규칙 실시간 업데이트
  • 핵심 기능: 로드밸런싱, 재시도, 서킷 브레이커, 헬스 체크, 율 제한, mTLS 종료
  • 관측성: 모든 요청에 대한 메트릭/추적/로그 자동 생성
  • 리소스: 보통 Pod당 50~100MB 메모리, CPU 적당량 소비 (오버헤드)

애플리케이션은 localhost로 호출하는 것처럼 보이지만 실제로는 iptables가 Envoy로 리다이렉트합니다. Envoy의 listener는 보통 15001(outbound), 15006(inbound), 15090(prometheus), 15021(health) 포트를 사용합니다.

Pilot (현 istiod 내부)

Envoy에 라우팅/디스커버리 설정을 배포하는 컴포넌트. 1.5 버전 이후 istiod 안으로 흡수되었지만 개념적으로 가장 중요합니다.

  • K8S Service, Endpoints, Istio CRD를 watch
  • 이를 Envoy가 이해하는 xDS 형식(Listener, Route, Cluster, Endpoint)으로 변환
  • 각 Envoy 사이드카에 gRPC로 push

비유: 본부에서 "오늘 트래픽은 v2:v1을 8:2로 보내라"라는 명령을 받아, 모든 현장 비서들에게 동시에 무전으로 전파하는 역할입니다.

Citadel (현 istiod 내부)

서비스 메시의 보안 책임자. mTLS를 위한 인증서 발급 및 회전을 담당합니다.

  • 각 워크로드(ServiceAccount)에 SPIFFE 형식의 ID 부여 (spiffe://<cluster>/ns/<ns>/sa/<sa>)
  • X.509 인증서 자동 발급 (기본 24시간 수명)
  • 인증서 자동 회전 (수명의 절반 시점에)
  • CA Root는 클러스터 외부의 사설/공인 CA로도 위임 가능

Galley (현 istiod 내부)

Istio 설정의 검증, 변환, 배포 담당. K8S API와 Istio의 다른 컴포넌트 사이의 추상화 레이어 역할이었습니다.

  • YAML 유효성 검증 (잘못된 VirtualService를 미리 거부)
  • 설정 변경 사항을 Pilot 등에 알림
  • Validating Webhook으로 동작

Ingress / Egress Gateway

클러스터의 출입구 역할을 하는 Envoy 인스턴스들입니다. 사이드카가 아닌 독립 Deployment로 배포됩니다.

  • Ingress Gateway: 외부 → 메시 안으로 들어오는 트래픽 처리 (K8S Ingress와 비슷한 역할)
  • Egress Gateway: 메시 → 외부로 나가는 트래픽 처리 (감사, 정책 적용 가능)

기본 설치 시 istio-ingressgateway Pod가 istio-system에 배포되며, 보통 LoadBalancer 타입의 Service로 노출됩니다.

CHAPTER 05

Istio 리소스 (CRD) 완벽 가이드

Istio는 K8S에 새로운 종류의 오브젝트(CRD)를 추가합니다. 각각이 어떤 일을 하는지 정확히 알아봅시다.

📐
리소스 분류

크게 트래픽 관리(Gateway, VirtualService, DestinationRule, ServiceEntry, Sidecar)와 보안(PeerAuthentication, RequestAuthentication, AuthorizationPolicy)으로 나뉩니다.

1. Gateway — 메시의 입구/출구

클러스터 가장자리(edge)에서 어떤 호스트, 포트, 프로토콜의 트래픽을 받을지 정의합니다. 쉽게 말해 "어느 문을 열어둘지" 정합니다.

YAML · Gateway 예제 
apiVersion: networking.istio.io/v1beta1
kind: Gateway
metadata:
  name: shop-gateway
  namespace: shop
spec:
  selector:
    # istio-ingressgateway Pod에 적용
    istio: ingressgateway
  servers:
  - port:
      number: 80
      name: http
      protocol: HTTP
    hosts:
    - "shop.example.com"
  - port:
      number: 443
      name: https
      protocol: HTTPS
    tls:
      mode: SIMPLE
      credentialName: shop-tls-cert  # Secret 이름
    hosts:
    - "shop.example.com"

2. VirtualService — 트래픽 라우팅 규칙

"이런 요청이 오면 → 어디로 보내라"를 세밀하게 정의합니다. URL 경로, 헤더, 쿠키, 비율 등으로 라우팅 가능합니다. K8S Service의 "라운드 로빈 단순 분산"을 강력하게 확장한 버전입니다.

YAML · VirtualService 예제 (카나리 배포) 
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: reviews-route
spec:
  hosts:
  - reviews  # 메시 내부 호출 시 사용되는 호스트명
  http:
  - match:
    - headers:
        end-user:
          exact: jason  # jason 사용자만 v2로
    route:
    - destination:
        host: reviews
        subset: v2
  - route:  # 나머지는 90:10 카나리
    - destination:
        host: reviews
        subset: v1
      weight: 90
    - destination:
        host: reviews
        subset: v3
      weight: 10

3. DestinationRule — 서비스의 하위 그룹과 정책

VirtualService가 "어떤 트래픽이 어디로 갈지"라면, DestinationRule은 "도착해서 어떻게 행동할지"를 정의합니다. 같은 Service라도 라벨로 subset(v1, v2 등)으로 나누고, 로드밸런싱·서킷 브레이커·연결 풀 설정을 적용합니다.

YAML · DestinationRule 예제 
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: reviews-dr
spec:
  host: reviews
  trafficPolicy:
    loadBalancer:
      simple: LEAST_REQUEST
    connectionPool:
      tcp:
        maxConnections: 100
      http:
        http2MaxRequests: 1000
        maxRequestsPerConnection: 10
    outlierDetection:  # 서킷 브레이커
      consecutive5xxErrors: 5
      interval: 30s
      baseEjectionTime: 60s
  subsets:  # 버전별 그룹 정의
  - name: v1
    labels:
      version: v1
  - name: v2
    labels:
      version: v2
  - name: v3
    labels:
      version: v3
🤝
VirtualService와 DestinationRule은 짝꿍

거의 항상 함께 씁니다. VirtualService에서 subset을 참조하려면 DestinationRule에서 미리 정의해야 합니다.

4. ServiceEntry — 메시 외부 서비스를 등록

외부 API(예: api.openai.com)도 Istio가 관리하도록 메시 안에 "가상으로 등록"합니다. 이렇게 하면 외부 호출에도 라우팅, 재시도, 메트릭, mTLS를 적용할 수 있습니다.

YAML · ServiceEntry 예제 
apiVersion: networking.istio.io/v1beta1
kind: ServiceEntry
metadata:
  name: openai-api
spec:
  hosts:
  - api.openai.com
  ports:
  - number: 443
    name: https
    protocol: HTTPS
  resolution: DNS
  location: MESH_EXTERNAL

5. Sidecar — 사이드카 범위 제한

기본적으로 모든 사이드카는 클러스터의 모든 서비스 정보를 다 가지고 있습니다(메모리 낭비). Sidecar 리소스로 "이 워크로드는 어떤 네임스페이스/서비스만 보면 된다"라고 제한할 수 있습니다.

YAML · Sidecar 리소스 예제 
apiVersion: networking.istio.io/v1beta1
kind: Sidecar
metadata:
  name: default
  namespace: shop
spec:
  egress:
  - hosts:
    - "./*"           # 같은 네임스페이스
    - "istio-system/*"  # istio 시스템
    - "shared/*"        # 공용 서비스

6. PeerAuthentication — 서비스 간 mTLS 정책

서비스끼리 통신할 때 mTLS를 어떻게 적용할지 정의합니다. STRICT(반드시 mTLS), PERMISSIVE(둘 다 허용), DISABLE(미적용) 모드가 있습니다.

YAML · PeerAuthentication 예제 
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: default
  namespace: shop
spec:
  mtls:
    mode: STRICT  # shop 네임스페이스 안에서는 mTLS 강제

7. AuthorizationPolicy — L7 접근 제어

"누가 무엇을 호출할 수 있는지" L7 단위로 제어합니다. K8S NetworkPolicy의 L3/L4를 넘어, HTTP 메서드/경로/헤더 기반 정책이 가능합니다.

YAML · AuthorizationPolicy 예제 
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: payment-allow-order-only
  namespace: shop
spec:
  selector:
    matchLabels:
      app: payment
  action: ALLOW
  rules:
  - from:
    - source:
        principals:
        - "cluster.local/ns/shop/sa/order-sa"  # order 서비스만
    to:
    - operation:
        methods: ["POST"]
        paths: ["/api/charge"]

8. RequestAuthentication — JWT 검증

외부에서 들어오는 요청의 JWT 토큰을 검증합니다. 보통 AuthorizationPolicy와 함께 씁니다.

YAML · RequestAuthentication 예제 
apiVersion: security.istio.io/v1beta1
kind: RequestAuthentication
metadata:
  name: jwt-on-ingress
  namespace: istio-system
spec:
  selector:
    matchLabels:
      istio: ingressgateway
  jwtRules:
  - issuer: "https://accounts.example.com"
    jwksUri: "https://accounts.example.com/.well-known/jwks.json"
CHAPTER 06

K8S 기본 오브젝트 vs Istio 리소스

가장 헷갈리는 부분. "이미 K8S Service가 있는데 왜 또 VirtualService?" 하나하나 짝지어서 비교합니다.

K8S Service
VS
VirtualService + DestinationRule
K8S Service
  • DNS 이름과 ClusterIP 제공
  • 라벨 selector로 Pod 묶음
  • 로드밸런싱: kube-proxy의 iptables 라운드 로빈 정도
  • 고급 라우팅 (헤더, 가중치) 불가능
  • L4 수준 (TCP/UDP 포트)
Istio VirtualService + DestinationRule
  • K8S Service를 기반으로 그 위에 동작
  • L7 라우팅: HTTP 헤더/경로/메서드/쿠키 기반
  • 가중치 분배: 90:10 카나리
  • 로드밸런싱 알고리즘 선택 가능 (LEAST_REQUEST 등)
  • 재시도, 타임아웃, 미러링, fault injection
  • subset(version)으로 같은 서비스 안의 그룹 분리
🎯
한 줄 요약

K8S Service는 그대로 둡니다. Istio는 그 위에 얹어서 더 똑똑한 라우팅 레이어를 제공할 뿐입니다.

K8S Ingress
VS
Istio Gateway + VirtualService
K8S Ingress
  • 외부 → 클러스터 진입점 정의
  • 호스트/경로 기반 라우팅
  • 구현체(Ingress Controller)에 따라 기능 천차만별 (NGINX, Traefik 등)
  • 어노테이션 의존성 큼 (구현체별로 다름)
  • L7이지만 표준 기능 제한적
Istio Gateway + VirtualService
  • Gateway: 진입점 정의 (포트, 프로토콜, TLS)
  • VirtualService: 라우팅 규칙 (Ingress와 별도 객체)
  • 역할 분리가 깔끔 (네트워크팀/앱팀)
  • L7 고급 기능 풍부 (재시도, 미러링, JWT 검증)
  • 같은 VirtualService를 메시 내부/외부에 모두 적용 가능
K8S NetworkPolicy
VS
Istio AuthorizationPolicy
K8S NetworkPolicy
  • L3/L4 방화벽 (IP/포트 기반)
  • CNI 플러그인이 구현 (Calico, Cilium 등)
  • "이 Pod는 저 Pod에 80 포트로 연결 가능" 수준
  • HTTP 경로/메서드 검사 불가
  • 워크로드 ID 개념 없음 (IP 기반)
Istio AuthorizationPolicy
  • L7 인가 (HTTP 메서드, 경로, 헤더, JWT claim)
  • ServiceAccount 기반 워크로드 ID 사용
  • "order-sa만 /api/charge POST 가능" 수준
  • mTLS와 결합하여 strong identity 보장
  • Envoy에서 처리, NetworkPolicy와 보완적 사용 가능

한 눈에 보는 매핑 테이블

역할K8S 기본 오브젝트Istio 리소스관계
서비스 디스커버리 Service VirtualService + DestinationRule Istio는 K8S Service를 그대로 활용하고 위에 라우팅 레이어 추가
외부 노출 Ingress + Controller Gateway + VirtualService Istio는 Ingress 대체 가능. 둘 다 쓸 수도 있음
외부 서비스 호출 직접 호출 or ExternalName Service ServiceEntry Istio가 외부 호출도 메시에 통합 관리
L3/L4 방화벽 NetworkPolicy (없음, NetworkPolicy 그대로 사용) 둘이 보완 — NetworkPolicy는 L3/L4, AuthorizationPolicy는 L7
L7 접근 제어 (없음) AuthorizationPolicy Istio만의 영역
인증 RBAC (kube-api 한정) RequestAuthentication, PeerAuthentication RBAC은 K8S API 호출용, Istio는 서비스 간 호출용
TLS Secret + 앱 코드 자동 mTLS (PeerAuthentication) Istio는 자동/투명, K8S는 수동/명시적
로드밸런싱 kube-proxy iptables (RR) Envoy (RR, LR, Random, Consistent Hash 등) Istio는 더 정교한 알고리즘
관측성 kubectl logs / metrics-server 자동 Prometheus 메트릭 / Jaeger 추적 / 액세스 로그 Istio는 자동, 기본 K8S는 수동
트래픽 정책 없음 VirtualService, DestinationRule 재시도, 타임아웃, 서킷 브레이커는 Istio만의 영역

같이 쓰는 것이지, 대체하는 것이 아니다

⚠️
흔한 오해

"Istio를 쓰면 K8S Service나 Deployment를 안 만들어도 된다" → 아닙니다. Istio는 K8S 위에 얹는 레이어입니다. Deployment, Service, ConfigMap, Secret은 그대로 다 만들고, 그 위에 Istio 리소스를 추가합니다.

CHAPTER 07

트래픽 관리 — Istio의 가장 매력적인 기능

여기서부터 Istio가 빛납니다. 코드 한 줄 안 고치고 배포 전략을 자유자재로.

카나리 배포 — 새 버전에 트래픽 일부만

v1에 90%, v2에 10% 트래픽을 보내고 점진적으로 비율을 늘리는 패턴입니다.

YAML
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: api-canary
spec:
  hosts: [api]
  http:
  - route:
    - destination: {host: api, subset: v1}
      weight: 90
    - destination: {host: api, subset: v2}
      weight: 10

점점 10→25→50→100으로 weight 변경. 문제 발생 시 즉시 0으로 되돌리면 끝.

블루/그린 배포 — 한 번에 100% 스위치

v1(blue)와 v2(green)를 모두 띄워두고, 트래픽만 100% 한쪽으로 보냅니다. 롤백이 매우 빠릅니다.

YAML
spec:
  hosts: [api]
  http:
  - route:
    - destination: {host: api, subset: v2}
      weight: 100
    # 문제 발생시 v1로 weight 변경하면 즉시 롤백

A/B 테스트 — 특정 조건의 사용자만 신버전

헤더, 쿠키, 사용자 ID 등으로 일부 사용자만 v2 경험. 비즈니스 메트릭 비교에 활용합니다.

YAML
http:
- match:
  - headers:
      x-user-tier: {exact: "beta"}
  route:
  - destination: {host: api, subset: v2}
- route:
  - destination: {host: api, subset: v1}

트래픽 미러링 (Shadow) — 안전한 새 버전 검증

실제 응답은 v1에서 받지만, 같은 요청을 v2에도 복사해서 보냅니다. 사용자에게 영향 없이 v2를 검증합니다.

YAML
http:
- route:
  - destination: {host: api, subset: v1}
    weight: 100
  mirror:
    host: api
    subset: v2
  mirrorPercentage:
    value: 100.0
🪞
유즈케이스

DB 마이그레이션 후 신규 쿼리 성능 검증, 머신러닝 모델 A/B 비교, 보안 패치 검증 등.

재시도 & 타임아웃

네트워크 일시 장애를 코드 수정 없이 처리합니다.

YAML
http:
- route:
  - destination: {host: api}
  timeout: 3s
  retries:
    attempts: 3
    perTryTimeout: 1s
    retryOn: "5xx,reset,connect-failure"

서킷 브레이커 (Circuit Breaker)

장애 발생한 인스턴스를 풀에서 자동 제거합니다. DestinationRule의 outlierDetection로 설정합니다.

YAML
trafficPolicy:
  outlierDetection:
    consecutive5xxErrors: 5      # 5번 연속 5xx
    interval: 30s             # 30초마다 검사
    baseEjectionTime: 60s     # 60초간 격리
    maxEjectionPercent: 50    # 최대 50%까지 격리
  connectionPool:
    tcp: {maxConnections: 100}
    http: {http1MaxPendingRequests: 1}

Fault Injection — 카오스 엔지니어링

일부러 지연/에러를 주입해서 시스템 복원력을 테스트합니다.

YAML
http:
- fault:
    delay:
      percentage: {value: 10}    # 10% 요청에
      fixedDelay: 5s             # 5초 지연 주입
    abort:
      percentage: {value: 5}     # 5% 요청은
      httpStatus: 503            # 503 에러 반환
  route:
  - destination: {host: api}
CHAPTER 08

보안 — Zero Trust by Default

"내부망이니까 안전하다"는 가정은 더 이상 통하지 않습니다. Istio는 모든 통신을 의심하고 검증합니다.

1. 자동 mTLS — 서비스 간 통신 자동 암호화

Istio의 가장 강력한 보안 기능. 각 워크로드에 인증서가 자동 발급되고, Envoy끼리 mTLS로 통신합니다. 앱은 그냥 HTTP를 호출했을 뿐인데 실제로는 TLS 1.3으로 암호화되어 흐릅니다.

🔑
자동 인증서 발급
각 ServiceAccount에 SPIFFE ID 부여 후 X.509 인증서 발급. 기본 24시간 수명, 자동 회전.
🤝
Mutual TLS
클라이언트와 서버가 서로 인증서로 신원 확인. 평범한 TLS는 서버만 인증.
🔍
Workload Identity
IP 주소가 아닌 ServiceAccount 기반 ID. Pod가 재시작/이동해도 신원 유지.

2. mTLS 모드 3종

모드동작언제 쓰나
STRICTmTLS만 허용. 평문 거부완전히 메시화된 프로덕션 환경
PERMISSIVEmTLS + 평문 둘 다 허용마이그레이션 중 (기본값)
DISABLEmTLS 미적용레거시 통합 등 특수 케이스

3. 인가 정책 흐름

요청이 Envoy에 도착하면 다음 순서로 검사합니다:

  1. PeerAuthentication — mTLS 강제 여부 확인
  2. RequestAuthentication — JWT 토큰 유효성 검증 (있다면)
  3. AuthorizationPolicy — ALLOW/DENY 규칙 평가
  4. 모두 통과해야 앱으로 전달
🚫
주의: DENY 우선

여러 AuthorizationPolicy가 적용될 때, DENY가 ALLOW보다 우선합니다. 명시적 거부 정책은 즉시 차단됩니다.

4. 실전 예: 네임스페이스 격리

YAML · 기본 거부 후 명시 허용
# 1) 기본은 모두 거부
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: deny-all
  namespace: payment
spec: {}  # 빈 spec = 모두 거부
---
# 2) order 네임스페이스만 허용
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: allow-from-order
  namespace: payment
spec:
  action: ALLOW
  rules:
  - from:
    - source:
        namespaces: ["order"]

관측성 — 들여다보기

Istio는 코드 수정 없이 메트릭, 분산 추적, 로그를 자동으로 수집합니다. 다만 저장소(Prometheus 등)는 별도로 설치해야 합니다.

3대 관측 신호 (Metrics, Traces, Logs)

📊
Metrics (Prometheus)
모든 요청에 대한 RED 지표 자동 생성:
istio_requests_total, istio_request_duration_milliseconds, istio_request_bytes
🔗
Distributed Tracing (Jaeger/Zipkin)
요청이 여러 서비스를 거치는 흐름을 추적. 단, 앱에서 trace 헤더(x-b3-* 또는 W3C trace-context)를 전파해야 함
📝
Access Logs
각 Envoy가 모든 요청을 stdout에 기록. ELK/Loki 등으로 수집 가능

Kiali — Service Mesh 시각화의 끝판왕

Istio 메시의 토폴로지를 실시간 그래프로 보여주는 도구. 어떤 서비스가 어떤 서비스를 호출하는지, 에러율은 어떤지, 트래픽은 얼마나 흐르는지 시각적으로 확인할 수 있습니다.

  • 실시간 서비스 토폴로지 그래프
  • 버전별 트래픽 흐름 표시 (카나리 진행 상황 확인)
  • VirtualService, DestinationRule YAML을 GUI로 편집
  • 인증/인가 정책 검증
  • 분산 추적(Jaeger) 통합 뷰

Telemetry API — 텔레메트리 제어

v1.12+ 부터 Telemetry CRD로 어떤 메트릭/트레이스/로그를 수집할지 세밀하게 제어할 수 있습니다.

YAML · 추적 샘플링 1% 설정
apiVersion: telemetry.istio.io/v1alpha1
kind: Telemetry
metadata:
  name: sample-tracing
  namespace: istio-system
spec:
  tracing:
  - randomSamplingPercentage: 1.0

설치하기 — 30분이면 충분

istioctl로 가장 간단하게 설치하고 첫 사이드카 주입까지 따라해봅시다.

1단계: istioctl 설치

bash
# 최신 stable 다운로드 (Linux/macOS)
curl -L https://istio.io/downloadIstio | sh -
cd istio-*/
export PATH=$PWD/bin:$PATH

# 버전 확인
istioctl version

2단계: 프로파일 선택 후 설치

Istio는 5개의 미리 정의된 프로파일을 제공합니다.

프로파일용도구성
default프로덕션 권장istiod + ingress gateway
demo학습/실습용모든 기능 + addon (logging 많음)
minimal최소 설치istiod만
ambient사이드카 없는 모드 (v1.18+)ztunnel + waypoint
empty커스텀 시작점아무것도 없음
bash
# 데모 프로파일로 설치 (학습용)
istioctl install --set profile=demo -y

# 설치 확인
kubectl get pods -n istio-system

3단계: 네임스페이스에 자동 주입 활성화

이 라벨이 붙은 네임스페이스에 새로 만들어지는 모든 Pod에 자동으로 istio-proxy 사이드카가 들어갑니다.

bash
# 네임스페이스 생성 후 라벨링
kubectl create namespace shop
kubectl label namespace shop istio-injection=enabled

# 확인
kubectl get namespace -L istio-injection

4단계: 첫 워크로드 배포 후 사이드카 확인

bash
kubectl apply -n shop -f - <<EOF
apiVersion: apps/v1
kind: Deployment
metadata: {name: hello}
spec:
  replicas: 1
  selector: {matchLabels: {app: hello}}
  template:
    metadata: {labels: {app: hello}}
    spec:
      containers:
      - {name: hello, image: nginxdemos/hello}
EOF

CHAPTER 11

실습 — Bookinfo로 카나리 배포

Istio 공식 샘플 Bookinfo 앱으로 카나리 배포를 실습합니다.

아키텍처

4개의 마이크로서비스가 통신합니다: productpage → details, reviews → ratings. reviews는 v1, v2, v3 세 버전이 동시 배포되어 있습니다.

STEP 1: 샘플 배포

bash
kubectl create namespace bookinfo
kubectl label namespace bookinfo istio-injection=enabled

kubectl apply -n bookinfo -f samples/bookinfo/platform/kube/bookinfo.yaml
kubectl apply -n bookinfo -f samples/bookinfo/networking/bookinfo-gateway.yaml

# 외부 IP 확인
kubectl get svc istio-ingressgateway -n istio-system

STEP 2: DestinationRule로 subset 정의

YAML
kubectl apply -n bookinfo -f - <<EOF
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata: {name: reviews}
spec:
  host: reviews
  subsets:
  - {name: v1, labels: {version: v1}}
  - {name: v2, labels: {version: v2}}
  - {name: v3, labels: {version: v3}}
EOF

STEP 3: 모든 트래픽을 v1으로

YAML
kubectl apply -n bookinfo -f - <<EOF
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata: {name: reviews}
spec:
  hosts: [reviews]
  http:
  - route:
    - {destination: {host: reviews, subset: v1}, weight: 100}
EOF

브라우저에서 productpage를 새로고침 → 항상 별점 없는 v1만 표시.

STEP 4: 카나리 — 20%만 v2로

YAML
kubectl apply -n bookinfo -f - <<EOF
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata: {name: reviews}
spec:
  hosts: [reviews]
  http:
  - route:
    - {destination: {host: reviews, subset: v1}, weight: 80}
    - {destination: {host: reviews, subset: v2}, weight: 20}
EOF

10번 새로고침하면 약 2번 정도 검은 별이 보일 것입니다.

STEP 5: 정리

bash
kubectl delete namespace bookinfo
CHAPTER 12

현장에서 통하는 베스트 프랙티스

실무에서 자주 부딪히는 함정과 권장 패턴 정리.

도입 전 체크리스트

  • 실제 마이크로서비스 갯수가 5개 이상인가? 3개 이하면 Istio 오버헤드가 더 클 수 있음
  • 관측성 인프라(Prometheus, Jaeger)가 준비되었나? Istio만 깔면 데이터는 다 버려짐
  • 팀이 Envoy/L7 개념에 익숙한가? 디버깅 난이도가 올라감
  • K8S 클러스터의 리소스 여유가 있나? Pod당 50-100MB RAM 추가 필요
  • 점진적 마이그레이션 계획이 있나? 한 번에 모든 네임스페이스에 주입하면 위험

운영 권장 사항

1. PeerAuthentication은 처음엔 PERMISSIVE로 +

처음부터 STRICT으로 가면 사이드카 없는 워크로드(레거시 시스템)와 통신이 깨집니다. PERMISSIVE로 마이그레이션 후, 모든 호출이 mTLS로 보이면 STRICT으로 전환하세요.

2. Sidecar 리소스로 메모리 절약 +

기본적으로 모든 사이드카가 전 클러스터의 서비스 정보를 받기 때문에 메모리 사용량이 증가합니다. Sidecar 리소스로 egress hosts를 제한하면 Envoy 메모리를 30~80%까지 줄일 수 있습니다.

3. istioctl analyze로 사전 검증 +

YAML 적용 전에 istioctl analyze -n <ns>로 구성 충돌, 누락된 DestinationRule 등을 미리 잡으세요. CI 파이프라인에 넣으면 좋습니다.

4. 분산 추적은 앱 협조가 필요 +

Istio는 trace 헤더를 자동 "주입"하지만 "전파"하진 못합니다. 앱이 들어온 요청의 x-b3-traceid, x-b3-spanid, x-request-id 등을 다음 호출에 그대로 복사해야 트레이스가 이어집니다.

5. Resource Request/Limit 설정 필수 +

istio-proxy의 기본 limit이 충분하지 않을 수 있습니다. 트래픽이 많은 워크로드는 어노테이션으로 별도 조정하세요.

YAML
annotations:
  sidecar.istio.io/proxyCPU: "200m"
  sidecar.istio.io/proxyMemory: "256Mi"
  sidecar.istio.io/proxyCPULimit: "1000m"
  sidecar.istio.io/proxyMemoryLimit: "512Mi"
6. Canary 자동화는 Flagger와 함께 +

VirtualService를 수동으로 weight 변경하는 건 위험합니다. Flagger를 함께 쓰면 메트릭(에러율, 지연) 기반으로 자동 progression/rollback이 가능합니다.

7. Ambient Mesh도 고려해 보세요 +

Istio 1.18+에서 GA된 새로운 모드. 사이드카 대신 노드 레벨 ztunnel + 옵션 waypoint proxy로 동작해 메모리 오버헤드가 훨씬 작습니다. 트래픽 정책이 단순한 워크로드에 적합합니다.

흔한 트러블슈팅 명령어

bash
# 1) 사이드카 주입 확인
kubectl get pod -n <ns> -o jsonpath='{.items[*].spec.containers[*].name}'

# 2) Envoy에 적용된 설정 확인
istioctl proxy-config routes <pod-name> -n <ns>
istioctl proxy-config clusters <pod-name> -n <ns>
istioctl proxy-config listeners <pod-name> -n <ns>

# 3) Envoy 로그 보기
kubectl logs <pod-name> -c istio-proxy -n <ns>

# 4) 설정 충돌 검증
istioctl analyze -n <ns>

# 5) mTLS 상태 확인
istioctl x describe pod <pod-name> -n <ns>
